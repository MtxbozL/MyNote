## 一、开篇：事务隔离级别的核心定位

事务隔离级别，是数据库为了平衡 **「数据一致性」和「并发性能」**，定义的**多并发事务之间的隔离规则 **，是 ACID 特性中「隔离性（Isolation）」的具体落地实现。

简单理解：隔离级别就像「事务之间的墙」，墙越厚，隔离性越强，并发事务互相干扰越少，数据一致性越高，但并发性能越低；墙越薄，隔离性越弱，并发性能越高，但数据异常风险越大。

### 前置核心认知

1. 隔离级别是**基于事务**生效的，不同事务可以设置不同的隔离级别；
2. 隔离级别仅对**InnoDB 存储引擎**生效，MyISAM、Memory 等引擎不支持事务，自然也不支持隔离级别；
3. 隔离级别解决的核心问题，是**并发事务同时操作同一份数据时，产生的「读异常」和「更新丢失」问题**。

---

## 二、前置基础：并发事务的异常问题

隔离级别存在的核心意义，就是解决并发事务带来的数据异常问题。我们先把所有异常问题讲透，才能理解不同隔离级别的设计逻辑。

### 2.1 三大读异常（SQL 标准定义）

这是 SQL 标准中明确的、由隔离级别针对性解决的三类读数据异常，也是面试核心考点。

#### 1. 脏读（Dirty Read）

##### 核心定义

一个事务读取到了**另一个事务未提交的修改数据**。

> 核心关键词：未提交、脏数据（随时可能被回滚）。

##### 场景复现（时间线）

|时间|事务 A（转账）|事务 B（查询余额）|
|:--|:--|:--|
|T1|开启事务|开启事务|
|T2|张三账户扣减 1000 元（未提交）|-|
|T3|-|查询张三账户，读到了扣减后的余额（脏数据）|
|T4|转账失败，回滚事务|-|
|T5|-|基于脏数据的业务操作全部出错|

##### 危害

基于未提交的临时数据做业务判断，会导致业务逻辑完全错误，比如财务记账、库存扣减、风控判断失效，是最严重的读异常。

---

#### 2. 不可重复读（Non-Repeatable Read）

##### 核心定义

**同一个事务内**，多次执行同一条查询语句，得到的结果不一致。核心是其他事务对数据做了**UPDATE/DELETE**并提交，导致前后读取结果不同。

> 核心关键词：同一事务、同一条查询、结果不一致、更新 / 删除操作。

##### 场景复现（时间线）

|时间|事务 A（对账）|事务 B（消费）|
|:--|:--|:--|
|T1|开启事务|开启事务|
|T2|第一次查询张三账户余额：1000 元|-|
|T3|-|张三消费 200 元，提交事务|
|T4|第二次查询张三账户余额：800 元|-|
|T5|同一事务内两次查询结果不一致，对账失败|-|

##### 危害

同一个事务内，无法基于固定的查询结果做后续操作，比如统计报表、数据校验、循环更新逻辑出错，核心是破坏了事务内的读取一致性。

---

#### 3. 幻读（Phantom Read）

##### 核心定义

**同一个事务内**，多次执行同一条**范围查询**，得到的行数不一致。核心是其他事务做了**INSERT**并提交，导致第二次查询多了「幻影行」。

> 核心关键词：同一事务、范围查询、行数不一致、插入操作。

##### 场景复现（时间线）

|时间|事务 A（统计）|事务 B（新增）|
|:--|:--|:--|
|T1|开启事务|开启事务|
|T2|第一次查询：余额 > 1000 元的用户，共 5 人|-|
|T3|-|新增一个余额 2000 元的用户，提交事务|
|T4|第二次查询：余额 > 1000 元的用户，共 6 人|-|
|T5|同一事务内两次统计行数不一致，出现「幻影行」|-|

##### 危害

同一个事务内，范围统计、批量更新的结果不稳定，比如批量给符合条件的用户发福利，第二次查询多了一行，导致重复发放 / 漏发，核心是破坏了事务内的范围查询一致性。

---

#### 三大读异常核心区别

| 异常类型  | 核心触发原因         | 操作类型          | 核心特征             |
| :---- | :------------- | :------------ | :--------------- |
| 脏读    | 读取了未提交的数据      | UPDATE/DELETE | 读取到随时可能被回滚的临时数据  |
| 不可重复读 | 读取了已提交的更新 / 删除 | UPDATE/DELETE | 同一行数据，前后读取的值不同   |
| 幻读    | 读取了已提交的新增      | INSERT        | 同一范围查询，前后读取的行数不同 |

---

### 2.2 两类更新丢失（拓展高频考点）

除了读异常，并发事务还会出现更新丢失问题，也是隔离级别需要解决的场景。

#### 1. 第一类更新丢失（回滚丢失）

一个事务的回滚，覆盖了另一个事务已提交的更新。目前绝大多数数据库的所有隔离级别都已经解决了这个问题，无需额外处理。

#### 2. 第二类更新丢失（提交覆盖）

两个事务同时更新同一行数据，后提交的事务覆盖了先提交的事务的更新，是生产环境高频出现的问题。

##### 场景复现

1. 商品库存初始为 100；
2. 事务 A 读取库存 100，准备扣减 10，计算为 90；
3. 事务 B 同时读取库存 100，准备扣减 20，计算为 80；
4. 事务 A 先提交，库存更新为 90；
5. 事务 B 后提交，库存更新为 80；
6. 最终结果：库存 80，正确结果应该是 70，事务 A 的更新被覆盖丢失。

##### 解决方案

- 悲观锁：`SELECT ... FOR UPDATE` 加排他锁；
- 乐观锁：版本号机制 `UPDATE table SET num = 90, version = version+1 WHERE id = 1 AND version = 1`；
- 串行化隔离级别：完全串行执行，避免并发更新。

## 三、SQL 标准四大隔离级别详解

SQL-92 标准定义了 4 个事务隔离级别，从低到高依次为：**读未提交 → 读已提交 → 可重复读 → 串行化**。级别越高，解决的异常问题越多，隔离性越强，但并发性能越低。

### 核心总览表（面试必背）

|隔离级别|英文全称|脏读|不可重复读|幻读|核心特点|
|:--|:--|:--|:--|:--|:--|
|读未提交|Read Uncommitted（RU）|✅ 存在|✅ 存在|✅ 存在|最低级别，可读取未提交数据|
|读已提交|Read Committed（RC）|❌ 解决|✅ 存在|✅ 存在|只能读取已提交数据，Oracle/SQL Server 默认|
|可重复读|Repeatable Read（RR）|❌ 解决|❌ 解决|✅ SQL 标准存在 /❌ InnoDB 解决|同一事务内读取结果始终一致，MySQL 默认|
|串行化|Serializable|❌ 解决|❌ 解决|❌ 解决|最高级别，事务完全串行执行|

> 关键说明：SQL 标准中，RR 级别仅解决了脏读和不可重复读，未解决幻读；但**MySQL InnoDB 引擎在 RR 级别下，通过「MVCC（快照读）+ 临键锁（当前读）」彻底解决了幻读问题**，这是 MySQL 和其他数据库的核心区别，也是面试高频考点。

---

### 3.1 读未提交（Read Uncommitted, RU）

#### 核心定义

最低隔离级别，事务中的修改，**即使未提交，对其他事务也完全可见**。

#### 解决的问题

无，所有读异常都存在。

#### 未解决的问题

脏读、不可重复读、幻读、第二类更新丢失。

#### 底层实现

几乎不加锁，所有读操作不加锁，直接读取数据的最新版本，性能极高，但数据一致性极差。

#### 适用场景

几乎没有生产场景会使用，仅用于理论测试，严禁在核心业务中使用。

#### 实操：复现脏读

```sql
-- 步骤1：会话A，设置隔离级别为读未提交，开启事务
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
BEGIN;
-- 步骤2：会话A，更新数据，不提交
UPDATE account SET balance = balance - 1000 WHERE user_name = '张三';

-- 步骤3：会话B，设置隔离级别为读未提交，开启事务
SET SESSION TRANSACTION ISOLATION LEVEL READ UNCOMMITTED;
BEGIN;
-- 步骤4：会话B，查询数据，读到了会话A未提交的修改（脏读）
SELECT balance FROM account WHERE user_name = '张三';

-- 步骤5：会话A，回滚事务
ROLLBACK;
-- 步骤6：会话B，再次查询，数据恢复，脏读验证完成
```

---

### 3.2 读已提交（Read Committed, RC）

#### 核心定义

只能读取其他事务**已经提交**的修改，未提交的修改对其他事务完全不可见。

#### 解决的问题

彻底解决了**脏读**问题。

#### 未解决的问题

不可重复读、幻读、第二类更新丢失。

#### 底层实现（InnoDB）

1. **MVCC 机制**：事务内**每一次执行快照读（普通 SELECT），都会生成一个全新的 Read View（读视图）**，每次都能读到其他事务已提交的最新数据，因此会出现不可重复读；
2. **锁机制**：仅锁定更新的行，不锁定间隙，间隙锁关闭，因此会出现幻读；
3. **半一致性读**：更新操作时，如果行已被加锁，会先读取该行最新的已提交版本，判断是否符合更新条件，不符合则提前释放锁，减少锁等待，提升并发性能。

#### 适用场景

- 互联网高并发业务场景（很多大厂会把 MySQL 默认级别改为 RC）；
- 对不可重复读不敏感，更看重并发性能的业务；
- 长事务较少，以短事务为主的系统。

#### 实操：复现不可重复读

```sql
-- 步骤1：会话A，设置隔离级别为读已提交，开启事务
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
-- 步骤2：会话A，第一次查询，得到余额1000元
SELECT balance FROM account WHERE user_name = '张三';

-- 步骤3：会话B，设置隔离级别为读已提交，开启事务
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;
BEGIN;
-- 步骤4：会话B，更新余额为800元，提交事务
UPDATE account SET balance = 800 WHERE user_name = '张三';
COMMIT;

-- 步骤5：会话A，第二次查询，得到余额800元，和第一次结果不一致，不可重复读复现完成
SELECT balance FROM account WHERE user_name = '张三';
```

---

### 3.3 可重复读（Repeatable Read, RR）

#### 核心定义

MySQL 默认隔离级别，**同一个事务内，第一次快照读生成的 Read View，会被后续所有快照读复用**，保证同一个事务内，多次读取同一条 / 同一范围的数据，结果完全一致。

#### 解决的问题

- 彻底解决了**脏读、不可重复读**问题；
- InnoDB 引擎下，彻底解决了**幻读**问题。

#### 未解决的问题

第二类更新丢失（需通过悲观锁 / 乐观锁额外解决）。

#### 底层实现（InnoDB，核心考点）

InnoDB 通过两大核心机制，实现了 RR 级别下的完整隔离：

1. **MVCC 多版本并发控制（解决快照读的幻读）**
    
    - 事务内**第一次执行快照读（普通 SELECT）时，生成一个 Read View，后续所有快照读都复用这个 Read View**；
    - 无论其他事务是否提交了新增 / 修改数据，当前事务始终只能读到 Read View 生成时的已提交数据，因此不会出现不可重复读和幻读。
    
2. **临键锁（Next-Key Lock，解决当前读的幻读）**
    
    - 对于当前读（`SELECT ... FOR UPDATE`、`UPDATE`、`DELETE`、`INSERT`），InnoDB 会给查询范围加**临键锁（记录锁 + 间隙锁）**；
    - 锁定查询范围的所有行和间隙，禁止其他事务在这个范围内插入新数据，从根源上杜绝了幻读。
    

#### 适用场景

- MySQL 默认级别，适合绝大多数业务场景；
- 对数据一致性要求较高的场景，如电商订单、金融支付、库存管理；
- 需要在同一个事务内多次读取同一数据，保证读取一致性的业务。

#### 实操：验证 InnoDB RR 级别解决幻读

```sql
-- 步骤1：会话A，设置隔离级别为可重复读，开启事务
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
-- 步骤2：会话A，第一次范围查询，得到5条数据
SELECT * FROM account WHERE balance > 1000;

-- 步骤3：会话B，设置隔离级别为可重复读，开启事务
SET SESSION TRANSACTION ISOLATION LEVEL REPEATABLE READ;
BEGIN;
-- 步骤4：会话B，插入一条符合条件的新数据，提交事务
INSERT INTO account (user_name, balance) VALUES ('新用户', 2000);
COMMIT;

-- 步骤5：会话A，第二次范围查询，还是5条数据，没有出现幻读（快照读）
SELECT * FROM account WHERE balance > 1000;

-- 步骤6：会话A，执行当前读，加临键锁，禁止其他事务插入
SELECT * FROM account WHERE balance > 1000 FOR UPDATE;
-- 此时会话B再次插入符合条件的数据，会被阻塞，彻底杜绝幻读
```

### 3.4 串行化（Serializable）

#### 核心定义

最高隔离级别，**所有事务完全串行执行**，彻底放弃并发，读写互相阻塞。

- 所有普通 SELECT 语句，会被自动加上`LOCK IN SHARE MODE`共享锁；
- 所有 UPDATE/DELETE/INSERT 语句，会被自动加上排他锁；
- 读和读不冲突，读和写、写和写完全冲突，必须排队执行。

#### 解决的问题

彻底解决了脏读、不可重复读、幻读、第二类更新丢失的所有异常问题。

#### 未解决的问题

无，数据一致性拉满，但并发性能极差。

#### 底层实现

完全基于锁机制，放弃 MVCC，所有操作都加锁，事务必须排队执行，不会出现任何并发冲突。

#### 适用场景

- 对数据一致性要求极高，并发量极低的场景，如财务总账、合规审计系统、核心账务系统；
- 几乎没有并发写入的只读场景。

## 四、核心对比：RC vs RR（生产环境选型核心）

RC 和 RR 是生产环境唯二会使用的隔离级别，我们从多个维度做详细对比，帮你明确选型逻辑。

| 对比维度               | 读已提交（RC）                              | 可重复读（RR）                                    |
| :----------------- | :------------------------------------ | :------------------------------------------ |
| **脏读**             | 解决                                    | 解决                                          |
| **不可重复读**          | 存在                                    | 解决                                          |
| **幻读**             | 存在                                    | InnoDB 彻底解决                                 |
| **Read View 生成时机** | 每次快照读都生成新的                            | 事务内第一次快照读生成，后续复用                            |
| **锁机制**            | 仅记录锁，无间隙锁，锁范围小，冲突少                    | 临键锁（记录锁 + 间隙锁），锁范围大，冲突概率高                   |
| **并发性能**           | 高，锁等待少，死锁概率低                          | 中，锁范围大，死锁概率更高                               |
| **半一致性读**          | 支持，减少锁等待                              | 不支持                                         |
| **主从同步**           | binlog 格式必须用 ROW，STATEMENT 格式会导致主从不一致 | 支持 STATEMENT/ROW/MIXED 所有 binlog 格式，主从同步更稳定 |
| **数据一致性**          | 中，仅语句级一致性                             | 高，事务级一致性                                    |
| **长事务影响**          | 较小，长事务不会阻塞其他事务的 DML                   | 较大，长事务持有间隙锁，会阻塞其他事务的插入                      |
| **适用场景**           | 互联网高并发、短事务为主、对不可重复读不敏感的业务             | 金融、电商核心交易、对数据一致性要求高、需要事务内读取一致性的业务           |

### 生产环境选型建议

1. **优先选 RR 的场景**：
    
    - 金融支付、订单交易、库存管理等对数据一致性要求高的核心业务；
    - 需要在同一个事务内多次读取同一数据，保证结果一致的业务；
    - 使用 STATEMENT 格式 binlog 的场景；
    - 业务逻辑复杂，长事务较多的场景。
    
2. **优先选 RC 的场景**：
    
    - 互联网高并发读写业务，对性能要求高，对不可重复读不敏感；
    - 短事务为主，极少有长事务的场景；
    - 业务逻辑简单，同一个事务内不会重复读取同一数据；
    - 大厂微服务架构，单事务仅操作单库单表，无需事务内强一致性。

## 五、隔离级别相关操作（实操必备）

### 5.1 查看隔离级别

```sql
-- MySQL 8.0+ 版本（当前主流）
-- 查看当前会话的隔离级别
SELECT @@transaction_isolation;
-- 查看全局隔离级别
SELECT @@global.transaction_isolation;

-- MySQL 5.7 及之前版本
SELECT @@tx_isolation;
SELECT @@global.tx_isolation;
```

### 5.2 修改隔离级别

```sql
-- 1. 修改当前会话的隔离级别（仅当前连接生效，断开后失效）
-- 可选值：READ UNCOMMITTED / READ COMMITTED / REPEATABLE READ / SERIALIZABLE
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 2. 修改全局隔离级别（所有新创建的连接生效，已存在的连接不生效，MySQL重启后失效）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 3. 永久生效（修改MySQL配置文件my.cnf/my.ini）
[mysqld]
transaction_isolation = REPEATABLE-READ
```

## 六、常见坑与避坑指南

### 6.1 隔离级别设置不生效

- **坑 1**：修改了全局隔离级别，但当前会话查询还是旧值。
    
    - 原因：全局隔离级别仅对新连接生效，已存在的连接不会更新。
    - 解决：修改后重新建立连接，或修改会话级隔离级别。
    
- **坑 2**：修改了配置文件，但重启后还是默认值。
    
    - 原因：配置项写错，MySQL 8.0 用`transaction_isolation`，5.7 用`tx_isolation`，不能混用。
    - 解决：核对版本对应的配置项，放在`[mysqld]`节点下。
    

### 6.2 隔离级别和存储引擎不匹配

- **坑**：设置了隔离级别，但还是出现脏读，发现表用的是 MyISAM 引擎。
    
    - 原因：MyISAM、Memory 等引擎不支持事务，自然也不支持隔离级别，所有操作都是自动提交。
    - 解决：事务相关的表，必须使用 InnoDB 引擎。
    

### 6.3 事务中 DDL 语句导致隐式提交

- **坑**：事务中执行了 ALTER/CREATE/DROP 等 DDL 语句，导致事务被隐式提交，隔离级别失效。
    
    - 原因：MySQL 中执行 DDL 语句会触发隐式提交，无论事务是否结束，都会强制提交之前的操作。
    - 解决：严禁在事务中执行任何 DDL 语句。
    

### 6.4 RR 级别下的幻读误区

- **坑**：很多人以为 RR 级别下绝对不会出现幻读，结果在业务中复现了。
    
    - 原因：InnoDB 的 RR 级别，仅对**快照读（普通 SELECT）** 通过 MVCC 解决幻读；如果在事务中先快照读，再执行当前读，再快照读，可能出现幻读。
    - 示例：
        
        1. 事务 A 第一次快照读，查到 5 条数据；
        2. 事务 B 插入新数据并提交；
        3. 事务 A 执行 UPDATE 更新了这条新数据（当前读，能读到）；
        4. 事务 A 再次快照读，就能读到这条新数据，出现幻读。
        
    - 解决：如果需要在事务中混合快照读和当前读，建议给查询加锁，或使用串行化隔离级别。
    

### 6.5 长事务在 RR 级别下的危害

- **坑**：RR 级别下，一个长事务持有 Read View，会导致 undo log 无法被 purge 线程回收，占用大量磁盘空间，同时长事务持有的间隙锁会阻塞其他事务的插入，导致数据库性能急剧下降。
    
    - 解决：RR 级别下，严禁长事务，事务要尽可能短，执行完立即提交 / 回滚。

## 八、核心要点总结

1. 事务隔离级别是隔离性的具体实现，核心是平衡「数据一致性」和「并发性能」，级别越高，一致性越强，性能越低。
2. 并发事务的三大读异常：脏读（读未提交）、不可重复读（读已提交的更新）、幻读（读已提交的新增），隔离级别从低到高逐步解决这些问题。
3. 生产环境唯二可选的隔离级别：RC（高并发优先）和 RR（一致性优先，MySQL 默认），严禁使用 RU 和串行化（特殊场景除外）。
4. InnoDB RR 级别解决幻读的核心：快照读靠 MVCC 复用 Read View，当前读靠临键锁锁定间隙。
5. RC 和 RR 的核心区别：Read View 生成时机不同、锁机制不同、并发性能不同、数据一致性不同。
6. 实操核心：修改隔离级别要区分会话级和全局级，事务中严禁执行 DDL 语句，RR 级别下严禁长事务。
7. 选型核心：金融核心业务选 RR，互联网高并发业务可选 RC，始终以业务的一致性需求和并发需求为核心。

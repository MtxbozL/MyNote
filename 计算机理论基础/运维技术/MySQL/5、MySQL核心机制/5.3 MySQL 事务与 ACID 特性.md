## 一、开篇：事务的核心定义与价值

### 1. 什么是事务

**事务（Transaction）** 是一组**不可分割的 SQL 操作集合**，是数据库管理系统执行的最小逻辑单元。事务内的所有操作，要么**全部执行成功**，要么**全部执行失败回滚**，不存在中间状态。

简单理解：事务就是把多条 SQL 打包成一个「整体」，一荣俱荣，一损俱损。

### 2. 为什么需要事务？经典场景

最经典的场景是**银行转账**：张三给李四转 1000 元，需要两步操作：

1. 张三的账户扣减 1000 元；
2. 李四的账户增加 1000 元。

如果没有事务，会出现严重问题：第一步执行成功，第二步因为网络 / 系统故障执行失败，最终导致张三的钱扣了，李四的钱没到账，资金凭空消失。

有了事务之后，这两步操作被打包在一个事务中：

- 两步都成功：事务提交，资金转移完成；
- 任意一步失败：事务回滚，张三的钱恢复，李四的钱不变，不会出现数据不一致。

### 3. 事务的核心应用场景

所有对**数据一致性、安全性有强要求**的场景，都必须使用事务：

- 金融交易：转账、支付、充值、提现；
- 电商核心：订单创建、库存扣减、支付核销；
- 企业系统：合同签署、财务记账、库存管理；
- 任何需要「多步操作必须同时成功 / 同时失败」的场景。

## 二、核心基石：ACID 四大特性详解

ACID 是事务必须满足的四大核心特性，是事务的灵魂，也是数据库数据一致性的根本保障。

> 关键认知：**一致性是事务的最终目标，原子性、隔离性、持久性是实现一致性的三大手段**。

### 2.1 原子性（Atomicity）

#### 核心定义

原子性指事务是一个**不可分割的最小执行单元**，事务内的所有操作，要么全部执行完成，要么全部执行失败回滚，不存在「只执行一半」的中间状态。

#### 通俗示例

银行转账的两步操作，就是一个原子单元：

- 成功：扣钱 + 加钱都完成，事务提交；
- 失败：任意一步出错，两步都撤销，回到转账前的状态。

#### InnoDB 实现方式

原子性的底层核心是 **undo log（回滚日志）**：

- 事务执行的每一步修改操作，都会先记录对应的反向操作到 undo log 中；
- 事务执行失败 / 手动回滚时，InnoDB 会根据 undo log 中的反向操作，把数据恢复到事务开始前的状态；
- 事务提交后，undo log 会被后台 purge 线程异步回收。

#### 核心要点

- 原子性保障的是「操作的不可分割性」，解决「部分执行」的问题；
- 事务中任意一条 SQL 执行失败，已执行的 SQL 会全部回滚。

---

### 2.2 一致性（Consistency）

#### 核心定义

一致性指事务执行前后，数据库的**数据完整性约束、业务规则始终保持一致**，数据从一个合法的状态，转移到另一个合法的状态。

这里的「一致性」分为两层：

1. **数据库层面的一致性**：主键、唯一键、外键、非空、检查约束等始终不被破坏；
2. **业务层面的一致性**：业务规则始终成立，比如转账前后两个账户的总金额不变，库存扣减不能为负数。

#### 通俗示例

还是银行转账：

- 转账前，张三账户有 2000 元，李四账户有 1000 元，总金额 3000 元；
- 转账 1000 元后，张三账户 1000 元，李四账户 2000 元，总金额还是 3000 元；
- 总金额始终不变，符合业务规则，这就是一致性。

#### InnoDB 实现方式

一致性是事务的最终目标，由三大特性共同保障，同时配合数据库约束：

- 原子性：避免部分执行破坏业务规则；
- 隔离性：避免并发事务互相干扰导致数据错乱；
- 持久性：避免宕机导致数据丢失；
- 数据库约束：主键、唯一键、外键、CHECK 约束等，从底层保障数据合法性。

#### 核心要点

- 一致性是事务的**最终目的**，其他三大特性都是为了保障一致性；
- 数据库只能保障底层约束的一致性，业务层面的一致性需要开发者在事务中实现（比如转账的总金额校验）。

---

### 2.3 隔离性（Isolation）

#### 核心定义

隔离性指**多个并发执行的事务之间，互相隔离、互不干扰**，一个事务的中间状态，对其他事务是不可见的。

隔离性解决的是「并发事务同时操作同一份数据」带来的脏读、不可重复读、幻读问题，数据库通过**隔离级别**来控制隔离的严格程度。

#### 通俗示例

两个事务同时操作同一个账户：

- 事务 A 给张三的账户加 1000 元，还没提交；
- 事务 B 查询张三的账户余额，必须看不到事务 A 未提交的 1000 元；
- 只有事务 A 提交后，事务 B 才能看到更新后的余额，这就是隔离性。

#### InnoDB 实现方式

隔离性的底层核心是**锁机制 + MVCC 多版本并发控制**：

- 锁机制：写操作加排他锁，避免多个事务同时修改同一份数据；
- MVCC：通过 undo log 数据版本链 + Read View 读视图，实现读写不阻塞，不同事务看到不同的数据版本，互不干扰。

#### 核心要点

- 隔离性保障的是「并发事务之间的互不干扰」；
- 隔离性的严格程度，由「事务隔离级别」控制，级别越高，隔离性越强，并发性能越低。

---

### 2.4 持久性（Durability）

#### 核心定义

持久性指**一个事务一旦提交成功，它对数据库的修改就是永久生效的**，无论后续发生系统宕机、断电、硬件故障，已提交的数据都不会丢失。

#### 通俗示例

张三给李四转账成功，事务提交后，哪怕银行系统突然宕机，重启后张三的账户还是扣了 1000 元，李四的账户还是多了 1000 元，不会因为宕机导致数据恢复到转账前。

#### InnoDB 实现方式

持久性的底层核心是 **redo log（重做日志） + WAL 预写日志机制**：

- WAL 机制：先写日志，再写磁盘数据；
- 事务修改数据时，先把修改记录写入 redo log，再更新内存中的数据；
- 事务提交时，必须把 redo log 刷到磁盘，才算提交成功；
- 数据库宕机重启后，会读取 redo log，把未刷到磁盘的数据恢复回来，保障已提交的数据不丢失。

#### 核心要点

- 持久性保障的是「已提交数据的永久不丢失」；
- 持久性只针对「已提交的事务」，未提交的事务宕机后会全部回滚。

## 三、并发事务带来的三大问题

并发事务同时操作同一份数据时，如果隔离性没有保障，会出现三大经典问题，也是隔离级别要解决的核心问题。

### 3.1 脏读（Dirty Read）

#### 定义

一个事务读取到了**另一个事务未提交的修改数据**。

#### 示例

1. 事务 A：修改张三的账户余额为 2000 元，未提交；
2. 事务 B：查询张三的账户余额，读到了 2000 元（未提交的脏数据）；
3. 事务 A：回滚了修改，余额恢复为 1000 元；
4. 结果：事务 B 读到的 2000 元是无效的脏数据，基于这个数据的操作都会出错。

#### 危害

基于未提交的脏数据做业务判断，会导致业务逻辑完全错误，比如财务记账、库存扣减出错。

---

### 3.2 不可重复读（Non-Repeatable Read）

#### 定义

同一个事务内，多次执行同一条查询语句，得到的结果不一致（核心是**更新 / 删除操作**导致的）。

#### 示例

1. 事务 A：第一次查询张三的账户余额，是 1000 元，事务未结束；
2. 事务 B：修改张三的账户余额为 2000 元，提交成功；
3. 事务 A：第二次查询张三的账户余额，变成了 2000 元；
4. 结果：同一个事务内，两次查询同一条数据，结果不一样，无法重复读取。

#### 危害

同一个事务内，无法基于固定的查询结果做后续操作，比如统计报表、数据校验出错。

---

### 3.3 幻读（Phantom Read）

#### 定义

同一个事务内，多次执行同一条范围查询，得到的行数不一致（核心是**插入操作**导致的）。

#### 示例

1. 事务 A：查询余额大于 1000 元的用户，得到 5 条数据，事务未结束；
2. 事务 B：插入了一个余额 2000 元的新用户，提交成功；
3. 事务 A：再次查询余额大于 1000 元的用户，得到 6 条数据；
4. 结果：同一个事务内，两次范围查询，多了一条「幻影行」。

#### 危害

同一个事务内，范围查询的结果集不稳定，比如批量更新、统计计数出错。

---

### 三大问题核心区别

|问题|核心原因|触发场景|数据变化类型|
|:--|:--|:--|:--|
|脏读|读取了未提交的数据|单条数据查询|未提交的更新|
|不可重复读|读取了已提交的更新数据|同一条数据多次查询|已提交的更新 / 删除|
|幻读|读取了已提交的新增数据|范围查询多次执行|已提交的插入|

## 四、事务的四大隔离级别

SQL 标准定义了四大事务隔离级别，从低到高依次为：读未提交、读已提交、可重复读、串行化，级别越高，隔离性越强，并发性能越低，解决的问题越多。

### 4.1 四大隔离级别详解

|隔离级别|英文全称|脏读|不可重复读|幻读|核心说明|
|:--|:--|:--|:--|:--|:--|
|**读未提交**|Read Uncommitted|✅ 存在|✅ 存在|✅ 存在|最低级别，事务中的修改，即使未提交，对其他事务也可见|
|**读已提交**|Read Committed|❌ 解决|✅ 存在|✅ 存在|只能看到其他事务已提交的修改，解决脏读，Oracle、SQL Server 默认级别|
|**可重复读**|Repeatable Read|❌ 解决|❌ 解决|❌ InnoDB 解决|同一个事务内，多次读取同一条数据结果一致，MySQL 默认级别|
|**串行化**|Serializable|❌ 解决|❌ 解决|❌ 解决|最高级别，所有事务串行执行，完全隔离，并发性能极差|

> 关键说明：SQL 标准中，RR 级别仅解决了脏读和不可重复读，未解决幻读；但 **InnoDB 引擎在 RR 级别下，通过「临键锁（Next-Key Lock）+ MVCC」彻底解决了幻读问题**，这是 MySQL 和其他数据库的核心区别。

---

### 4.2 每个隔离级别的详细说明

#### 1. 读未提交（Read Uncommitted）

- 特点：几乎没有隔离性，事务之间可以互相看到未提交的修改；
- 优点：并发性能极高，几乎没有锁开销；
- 缺点：存在脏读、不可重复读、幻读，所有并发问题都无法解决；
- 适用场景：几乎没有生产场景会使用，仅用于理论测试。

#### 2. 读已提交（Read Committed，RC）

- 特点：只能看到其他事务**已提交**的修改，未提交的修改完全不可见；
- 实现原理：每次执行快照读，都会生成一个新的 Read View；
- 优点：解决了脏读，并发性能较好；
- 缺点：存在不可重复读、幻读；
- 适用场景：对数据一致性要求不高，能接受不可重复读的互联网业务，很多互联网公司会把 MySQL 的默认级别改为 RC，减少锁冲突，提升并发性能。

#### 3. 可重复读（Repeatable Read，RR）

- 特点：同一个事务内，第一次快照读时生成 Read View，后续所有快照读都复用这个 Read View，保证同一个事务内多次读取结果完全一致；
- 实现原理：MVCC 复用 Read View 解决不可重复读，临键锁解决幻读；
- 优点：解决了脏读、不可重复读、幻读，隔离性和并发性能平衡得最好；
- 适用场景：MySQL 默认级别，适合绝大多数业务场景，尤其是对数据一致性有要求的金融、电商核心业务。

#### 4. 串行化（Serializable）

- 特点：最高隔离级别，所有事务串行执行，读加共享锁，写加排他锁，读写互相阻塞；
- 优点：完全解决了所有并发问题，数据一致性最高；
- 缺点：并发性能极差，锁冲突严重，吞吐量极低；
- 适用场景：对数据一致性要求极高，并发量极低的场景，比如财务总账、核心合规系统。

---

### 4.3 MySQL 隔离级别操作

#### 查看当前隔离级别

```sql
-- MySQL 8.0+
SELECT @@transaction_isolation;

-- MySQL 5.7 及之前
SELECT @@tx_isolation;
```

#### 修改隔离级别

```sql
-- 修改当前会话的隔离级别（仅当前连接生效）
SET SESSION TRANSACTION ISOLATION LEVEL READ COMMITTED;

-- 修改全局隔离级别（所有新连接生效，重启后失效）
SET GLOBAL TRANSACTION ISOLATION LEVEL REPEATABLE READ;

-- 永久修改：修改 my.cnf 配置文件
transaction_isolation = REPEATABLE-READ
```

## 五、MySQL 事务的基础语法与实操

### 5.1 核心语法

|操作|语法|说明|
|:--|:--|:--|
|**开启事务**|`START TRANSACTION;` / `BEGIN;`|两种写法等价，开启一个新的事务|
|**提交事务**|`COMMIT;`|提交事务，所有修改永久生效|
|**回滚事务**|`ROLLBACK;`|回滚事务，撤销所有未提交的修改|
|**创建保存点**|`SAVEPOINT 保存点名;`|在事务内创建一个保存点，用于部分回滚|
|**回滚到保存点**|`ROLLBACK TO SAVEPOINT 保存点名;`|回滚到指定保存点，不撤销保存点之前的操作|
|**删除保存点**|`RELEASE SAVEPOINT 保存点名;`|删除指定的保存点|

---

### 5.2 自动提交（autocommit）机制

MySQL 默认开启**自动提交**，即 `autocommit = 1`：

- 开启自动提交时，每一条单独的 SQL 语句都是一个独立的事务，执行完成后自动提交，无法回滚；
- 关闭自动提交时，所有 SQL 都在一个事务内，直到手动执行 `COMMIT` 或 `ROLLBACK` 才会结束。

#### 自动提交相关操作

```sql
-- 查看自动提交状态
SELECT @@autocommit;

-- 关闭当前会话的自动提交
SET SESSION autocommit = 0;

-- 开启当前会话的自动提交
SET SESSION autocommit = 1;
```

> 关键提醒：即使开启了自动提交，执行 `BEGIN/START TRANSACTION` 后，自动提交会临时失效，直到事务提交 / 回滚后恢复。

---

### 5.3 完整实操示例

基于经典的银行转账场景，演示事务的完整使用：

```sql
-- 1. 创建账户表
CREATE TABLE IF NOT EXISTS account (
    id INT PRIMARY KEY AUTO_INCREMENT,
    user_name VARCHAR(50) NOT NULL,
    balance DECIMAL(10,2) NOT NULL DEFAULT 0 COMMENT '账户余额',
    CHECK (balance >= 0) -- 余额不能为负数
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 2. 插入测试数据
INSERT INTO account (user_name, balance) VALUES ('张三', 2000.00), ('李四', 1000.00);

-- 3. 开启事务
BEGIN;

-- 4. 执行转账操作：张三给李四转1000元
-- 4.1 张三账户扣减1000元
UPDATE account SET balance = balance - 1000 WHERE user_name = '张三';
-- 4.2 李四账户增加1000元
UPDATE account SET balance = balance + 1000 WHERE user_name = '李四';

-- 5. 校验数据（可选）
SELECT * FROM account;

-- 6. 提交事务（校验无误后执行）
COMMIT;

-- 7. 回滚事务（如果执行出错，执行这个，不要执行COMMIT）
-- ROLLBACK;
```

---

### 5.4 保存点（SAVEPOINT）实操示例

用于复杂事务中，实现部分回滚，避免一个小错误导致整个事务全部回滚：

```sql
-- 开启事务
BEGIN;

-- 第一步：给张三加500元
UPDATE account SET balance = balance + 500 WHERE user_name = '张三';
-- 创建保存点
SAVEPOINT sp1;

-- 第二步：给李四扣减800元（这里会因为余额不足报错）
UPDATE account SET balance = balance - 800 WHERE user_name = '李四';
-- 报错后，回滚到保存点sp1，第一步的操作不会被撤销
ROLLBACK TO SAVEPOINT sp1;

-- 继续执行其他操作
UPDATE account SET balance = balance - 300 WHERE user_name = '李四';

-- 提交事务
COMMIT;
```

## 六、ACID 四大特性的底层实现原理（InnoDB）

之前的内容讲了 ACID 的定义，这里系统梳理 InnoDB 是如何实现 ACID 的，这是面试高频考点。

| ACID 特性 | 核心实现组件                    | 实现原理                                                                                                |
| :------ | :------------------------ | :-------------------------------------------------------------------------------------------------- |
| **原子性** | undo log（回滚日志）            | 事务的每一步修改，都会记录对应的反向操作到 undo log；事务回滚时，通过 undo log 把数据恢复到事务开始前的状态，保证操作要么全成，要么全败。                      |
| **持久性** | redo log（重做日志） + WAL 预写日志 | 事务修改数据时，先写 redo log 到磁盘，再更新内存数据；事务提交必须等 redo log 刷盘完成；宕机重启后，通过 redo log 恢复已提交但未刷到磁盘的数据，保证数据不丢失。     |
| **隔离性** | 锁机制 + MVCC 多版本并发控制        | 1. 写操作加行级排他锁，避免并发修改冲突；<br><br>2. 读操作通过 MVCC 读取数据的历史版本，无需加锁，实现读写不阻塞；<br><br>3. 临键锁解决幻读问题，保证并发事务互不干扰。 |
| **一致性** | 原子性 + 隔离性 + 持久性 + 数据库约束   | 一致性是最终目标，通过原子性避免部分执行、隔离性避免并发干扰、持久性避免数据丢失，再配合数据库的主键、唯一键、外键、CHECK 约束，共同保障数据从一个合法状态转移到另一个合法状态。         |

## 七、事务的生命周期与状态流转

一个事务从开始到结束，会经历 5 个状态，完整的生命周期如下：

1. **活动状态（Active）**：事务开启后，正在执行 SQL 操作，处于活动状态；
2. **部分提交状态（Partially Committed）**：事务内的所有 SQL 都执行完成，还未执行 COMMIT 提交；
3. **提交状态（Committed）**：执行了 COMMIT，事务提交成功，修改永久生效，生命周期结束；
4. **失败状态（Failed）**：事务执行过程中 SQL 报错，或手动执行 ROLLBACK，事务无法继续执行；
5. **回滚状态（Aborted）**：事务失败后，回滚完成，数据恢复到事务开始前的状态，生命周期结束。

## 八、分布式事务基础（拓展）

### 8.1 什么是分布式事务

传统的事务是「本地事务」，只操作一个数据库；而在微服务架构中，一个业务操作会跨多个服务、多个数据库，比如「订单创建 → 库存扣减 → 支付核销 → 物流创建」，每个步骤都在不同的服务 / 数据库中，需要保证这多个步骤的原子性，这就是**分布式事务**。

### 8.2 常见分布式事务解决方案

| 方案                          | 核心原理                                         | 优点            | 缺点                | 适用场景                |
| :-------------------------- | :------------------------------------------- | :------------ | :---------------- | :------------------ |
| **2PC（两阶段提交）**              | 分为准备阶段和提交阶段，由协调者统一管理所有分支事务，全部准备成功则提交，任意失败则回滚 | 强一致性，实现简单     | 性能差，有阻塞风险，协调者单点故障 | 传统单体应用跨库事务，并发量低     |
| **TCC（Try-Confirm-Cancel）** | 分为尝试、确认、取消三个阶段，在业务代码层面实现事务的提交和回滚             | 性能好，无锁阻塞，灵活性高 | 对业务代码侵入性强，开发成本高   | 微服务高并发场景，核心交易业务     |
| **SAGA**                    | 把大事务拆分为多个本地小事务，通过正向执行和反向补偿实现事务               | 无锁阻塞，长流程适配性好  | 仅保证最终一致性，不支持隔离性   | 长流程业务，对实时一致性要求不高    |
| **本地消息表 + 消息队列**            | 通过本地事务 + 消息队列，保证多个操作的最终一致性                   | 实现简单，无侵入，性能好  | 仅保证最终一致性，不支持隔离性   | 非核心业务，对一致性要求不高的异步场景 |

## 九、实战避坑指南（高频踩坑点）

### 9.1 长事务 / 大事务的危害与避免

#### 危害

- 锁持有时间过长，导致大量并发事务阻塞，数据库连接池被打满；
- undo log 持续膨胀，占用大量磁盘空间；
- 主从同步延迟增大；
- 回滚时间长，故障恢复慢。

#### 避免方案

- 不要在事务中处理非数据库操作（比如 RPC 调用、HTTP 请求、文件读写）；
- 不要在事务中等待用户输入，事务的生命周期要尽可能短；
- 大批量数据操作，拆分为多个小事务执行；
- 避免在事务中执行全表扫描、大表关联查询。

### 9.2 自动提交的坑

- 坑 1：开启自动提交时，误以为多条 SQL 在一个事务中，结果一条执行失败，其他已经自动提交，无法回滚；
- 坑 2：关闭自动提交后，忘记提交事务，导致事务长期不提交，形成长事务，锁不释放；
- 避坑：明确事务边界，需要事务的场景，必须显式用 `BEGIN/COMMIT/ROLLBACK` 控制。

### 9.3 混合使用存储引擎的坑

- 坑：事务中同时操作 InnoDB 表和 MyISAM 表，MyISAM 不支持事务和回滚，事务失败时，MyISAM 表的修改无法回滚，导致数据不一致；
- 避坑：事务中只操作 InnoDB 表，不要混合使用不支持事务的存储引擎。

### 9.4 事务中混用 DDL 语句的坑

- 坑：事务中执行 DDL 语句（CREATE/ALTER/DROP），会触发隐式提交，导致之前的事务被强制提交，无法回滚；
- 避坑：不要在事务中执行任何 DDL 语句。

### 9.5 回滚不彻底的问题

- 坑：事务执行报错后，没有手动执行 ROLLBACK，以为报错会自动回滚，导致事务未结束，锁不释放；
- 避坑：事务执行报错后，必须显式执行 ROLLBACK，彻底结束事务。

### 9.6 隔离级别设置不当的问题

- 坑 1：使用读未提交级别，导致脏读，业务数据出错；
- 坑 2：使用串行化级别，导致并发性能极差，系统卡顿；
- 避坑：绝大多数场景使用默认的 RR 级别，高并发互联网场景可使用 RC 级别，严禁使用读未提交和串行化（特殊场景除外）。

## 十一、核心要点总结

1. 事务是一组不可分割的 SQL 操作集合，核心是「要么全成，要么全败」，是数据一致性的核心保障。
2. ACID 四大特性：原子性是基础，隔离性是并发保障，持久性是安全底线，一致性是最终目标。
3. 并发事务的三大问题：脏读（读未提交）、不可重复读（读已提交的更新）、幻读（读已提交的新增）。
4. 四大隔离级别：读未提交 < 读已提交 < 可重复读（MySQL 默认） < 串行化，级别越高，隔离性越强，并发性能越低。
5. InnoDB 对 ACID 的实现：原子性靠 undo log，持久性靠 redo log，隔离性靠锁 + MVCC，一致性靠前三者 + 约束。
6. 事务实操核心：显式开启 / 提交 / 回滚，避免长事务，不要在事务中执行 DDL、非数据库操作。
7. 生产环境核心原则：事务越短越好，边界越清晰越好，隔离级别适配业务场景，严禁长事务。

## 一、开篇：三大日志的核心定位与整体认知

MySQL 三大日志，指的是 **redo log（重做日志）、undo log（回滚日志）、binlog（二进制归档日志）**，是 MySQL 实现事务特性、数据持久化、崩溃恢复、主从复制的核心基石。

### 核心分层与设计目标

三大日志分属 MySQL 架构的两个层级，解决两大核心问题：

|日志类型|所属层级|核心设计目标|核心作用|
|:--|:--|:--|:--|
|redo log|InnoDB 存储引擎层|解决「事务持久性」，实现崩溃恢复（Crash-Safe）|保证数据库宕机重启后，已提交的事务数据不丢失|
|undo log|InnoDB 存储引擎层|解决「事务原子性」，支撑 MVCC 多版本控制|事务失败时回滚数据，为 MVCC 提供数据历史版本|
|binlog|MySQL 服务层|解决「数据归档与主从同步」，全引擎通用|数据备份恢复、主从架构的数据同步|

> 关键认知：redo log 和 undo log 是 InnoDB 引擎独有的，服务层完全不感知；binlog 是 MySQL 服务层实现的，所有存储引擎都可以生成。

## 二、三大日志核心总览表（面试必背）

|对比维度|redo log（重做日志）|undo log（回滚日志）|binlog（归档日志）|
|:--|:--|:--|:--|
|**所属层级**|InnoDB 存储引擎层|InnoDB 存储引擎层|MySQL 服务层|
|**日志类型**|物理日志（记录数据页的物理修改）|逻辑日志（记录数据修改的反向操作）|逻辑日志（记录 SQL 执行逻辑 / 行数据变更）|
|**写入方式**|固定大小，循环覆盖写入|追加写入，后台异步回收|追加写入，永不覆盖（除非手动删除）|
|**核心作用**|保障事务持久性，崩溃恢复|保障事务原子性，支撑 MVCC|数据备份恢复、主从复制|

## 三、第一部分：redo log（重做日志）

redo log 是 InnoDB 实现 **WAL（Write-Ahead Logging，预写日志）** 机制的核心，是 MySQL 崩溃恢复、事务持久性的根本保障。

### 3.1 核心定义与作用

#### 核心定义

redo log 是 InnoDB 引擎生成的**物理日志**，记录的是「某个数据页在某个偏移量做了什么修改」，而非 SQL 逻辑，记录的是数据修改的最终结果。

#### 核心作用

1. **保障事务持久性**：事务提交时，只需把 redo log 刷入磁盘，就算提交成功，无需等待内存中的脏数据刷入磁盘；
2. **解决随机 IO 性能问题**：修改数据时，直接修改内存中的缓冲池，再顺序写入 redo log（顺序 IO 性能是随机 IO 的上百倍），后续由后台线程异步把脏数据刷入磁盘，大幅提升写入性能；
3. **崩溃恢复（Crash-Safe）**：数据库宕机重启后，可通过 redo log 恢复已提交但未刷入磁盘的数据，保证数据不丢失。

### 3.2 文件结构与写入机制

#### 1. 文件结构

redo log 由一组固定大小的文件组成，默认是 2 个文件（`ib_logfile0`、`ib_logfile1`），每个文件默认 48MB，采用**循环写入**的方式：

- **write pos**：当前日志写入的位置，写满一个文件自动切换到下一个；
- **checkpoint**：当前数据已刷入磁盘的位置，checkpoint 之前的日志是已落盘的，可被覆盖；
- 当 write pos 追上 checkpoint 时，会触发 checkpoint 推进，强制把脏数据刷入磁盘，腾出可写入的空间。

#### 2. 写入链路

redo log 的写入分为三层，链路为：

```txt
事务执行 → 写入 Log Buffer（日志缓冲区，内存） → 写入 OS Page Cache（操作系统缓存） → 刷入磁盘 redo log 文件
```

#### 3. 核心刷盘策略（innodb_flush_log_at_trx_commit）

这是控制 redo log 刷盘行为的核心参数，有 3 个可选值，直接决定数据安全性和写入性能：

| 参数值       | 刷盘规则                                              | 数据安全性                         | 性能  | 适用场景              |
| :-------- | :------------------------------------------------ | :---------------------------- | :-- | :---------------- |
| **1（默认）** | 事务提交时，必须把 Log Buffer 中的 redo log 直接刷入磁盘（跳过 OS 缓存） | 最高，宕机绝对不丢已提交数据                | 最低  | 金融、支付等核心交易场景      |
| **0**     | 事务提交时不刷盘，由后台线程每秒把 Log Buffer 刷入磁盘                 | 最低，宕机丢失 1 秒内的已提交数据            | 最高  | 测试环境、非核心日志场景      |
| **2**     | 事务提交时，把 redo log 写入 OS Page Cache，后台线程每秒刷入磁盘      | 中等，操作系统宕机丢失 1 秒内数据，MySQL 宕机不丢 | 中等  | 互联网普通业务场景，性能与安全折中 |

### 3.3 核心参数配置与优化建议

| 参数名                            | 默认值  | 优化建议               | 说明                                       |
| :----------------------------- | :--- | :----------------- | :--------------------------------------- |
| innodb_log_file_size           | 48MB | 生产环境设为 1GB-4GB     | 单个 redo log 文件大小，太小会导致频繁 checkpoint，性能抖动 |
| innodb_log_files_in_group      | 2    | 保持默认 2 个即可         | redo log 文件组的数量                          |
| innodb_flush_log_at_trx_commit | 1    | 核心场景保持 1，普通场景可设为 2 | redo log 刷盘策略                            |
| innodb_log_buffer_size         | 16MB | 大事务场景设为 64MB-128MB | redo log 内存缓冲区大小                         |

### 3.4 常见问题

1. **为什么 redo log 要设计成固定大小、循环写入？**
    
    - 避免日志文件无限膨胀，占用过多磁盘空间；
    - 固定大小的文件顺序写入，性能更稳定，checkpoint 推进逻辑更简单。
    
2. **redo log 写满了会怎么样？**
    
    - 会触发强制 checkpoint，把内存中对应的脏数据全部刷入磁盘，推进 checkpoint 位置，腾出可写入的空间；
    - 这个过程中所有写入操作会被阻塞，因此 redo log 不能设置太小，避免频繁触发阻塞。

## 四、第二部分：undo log（回滚日志）

undo log 是 InnoDB 实现事务原子性、MVCC 多版本并发控制的核心，是事务回滚和读写不阻塞的根本保障。

### 4.1 核心定义与作用

#### 核心定义

undo log 是 InnoDB 引擎生成的**逻辑日志**，记录的是「数据修改的反向操作」，比如：

- 执行 `INSERT` 语句，undo log 会记录对应的 `DELETE` 操作；
- 执行 `UPDATE` 语句，undo log 会记录修改前的旧值，用于把数据恢复到修改前的状态。

#### 两大核心作用

1. **保障事务原子性**：事务执行失败 / 手动回滚时，InnoDB 会根据 undo log 中的反向操作，把数据恢复到事务开始前的状态，保证事务要么全成、要么全败；
2. **支撑 MVCC 多版本并发控制**：同一行数据的多次修改，会通过 undo log 生成一条**版本链**，不同事务可以通过版本链读取对应的数据历史版本，实现读写不阻塞，无需加锁。

### 4.2 文件结构与写入机制

#### 1. 文件结构

- MySQL 5.6 之前，undo log 只能存储在系统表空间 `ibdata1` 中，无法自动收缩，容易导致文件膨胀；
- MySQL 5.7+ 支持**独立 undo 表空间**，可把 undo log 存放在独立的 ibd 文件中，支持自动截断、收缩，生产环境推荐开启；
- undo log 按类型分为：`insert undo log`（插入操作生成，事务提交后可直接删除）、`update undo log`（更新 / 删除操作生成，事务提交后不能立即删除，需等待 MVCC 不再使用）。

#### 2. 写入链路

undo log 的写入遵循「先写日志，再修改数据」的原则，完整链路：

```txt
事务开启 → 执行修改操作前，先写 undo log 到 Log Buffer → 同时记录 undo log 对应的 redo log（undo log 也需要持久化） → 修改内存中的数据页
```

> 关键说明：undo log 本身也会产生 redo log，因为 undo log 需要持久化，数据库宕机重启时，需要通过 redo log 恢复 undo log，才能完成事务回滚。

#### 3. 生命周期

1. 事务开启后，执行修改操作时生成对应的 undo log；
2. 事务提交后，undo log 不会立即删除，会被放入 purge 队列；
3. 后台 purge 线程会判断 undo log 是否还被 MVCC 使用，如果没有任何事务需要这个历史版本，就会异步清理回收；
4. 如果存在长事务，会导致 undo log 长期无法回收，造成文件膨胀。

### 4.3 核心参数配置与优化建议

|参数名|默认值|优化建议|说明|
|:--|:--|:--|:--|
|innodb_undo_tablespaces|0|生产环境设为 2-4|独立 undo 表空间的数量，0 表示不开启独立表空间|
|innodb_undo_log_truncate|OFF|开启为 ON|自动截断收缩超过阈值的 undo 表空间|
|innodb_max_undo_log_size|1GB|保持默认或设为 2GB|undo 表空间自动截断的阈值|
|innodb_purge_rseg_truncate_frequency|128|保持默认|purge 线程回收 undo log 的频率|

### 4.4 常见问题

1. **事务提交后，undo log 为什么不立即删除？**
    
    - 因为其他事务可能还需要通过 undo log 读取数据的历史版本，实现 MVCC 快照读；
    - 只有当所有事务都不再需要这个 undo log 时，才会被 purge 线程回收。
    
2. **undo log 膨胀怎么办？**
    
    - 根本原因：存在长事务，持有旧的 Read View，导致 undo log 无法回收；
    - 解决方案：避免长事务，开启独立 undo 表空间和自动截断，定期监控长事务并 kill。

## 五、第三部分：binlog（二进制归档日志）

binlog 是 MySQL 服务层生成的二进制日志，记录了所有对数据库数据有修改的操作，是 MySQL 数据备份恢复、主从复制的核心。

### 5.1 核心定义与作用

#### 核心定义

binlog 是 MySQL 服务层生成的**逻辑日志**，记录的是数据修改的逻辑，不记录数据页的物理变化，所有存储引擎都可以生成。

#### 两大核心作用

1. **数据备份与恢复**：误删数据 / 库后，可通过全量备份 + binlog 增量日志，把数据恢复到任意时间点；
2. **主从复制**：MySQL 主从架构的核心，主库把 binlog 发送给从库，从库读取并重放 binlog 中的操作，实现主从数据同步。

### 5.2 三种日志格式详解

binlog 有三种格式，生产环境必须根据场景选择，否则会出现主从不一致、数据恢复失败等问题。

|格式类型|全称|核心说明|优点|缺点|适用场景|
|:--|:--|:--|:--|:--|:--|
|**STATEMENT**|基于语句的日志|记录执行的 SQL 语句本身|日志文件小，占用磁盘少|存在大量不确定函数（如 NOW ()、RAND ()），会导致主从不一致|几乎淘汰，不推荐生产使用|
|**ROW（推荐）**|基于行的日志|记录每一行数据的修改前后的值，不记录 SQL 语句|绝对不会出现主从不一致，是生产环境默认格式|日志文件大，尤其是批量更新 / 删除时|所有生产场景，尤其是主从架构|
|**MIXED**|混合模式|自动在 STATEMENT 和 ROW 之间切换，简单语句用 STATEMENT，不确定语句用 ROW|兼顾文件大小和数据一致性|复杂场景下仍有主从不一致风险|不推荐核心场景使用|

> 关键说明：MySQL 5.7.7 之后，binlog 格式默认为 ROW，生产环境必须使用 ROW 格式，避免主从不一致。

### 5.3 文件结构与写入机制

#### 1. 文件结构

binlog 采用**追加写入**的方式，不会覆盖旧文件，文件结构分为两部分：

- **binlog 日志文件**：命名格式为 `binlog.000001`、`binlog.000002`，文件写满后自动生成新的序号文件，序号递增；
- **binlog.index 索引文件**：纯文本文件，记录了当前所有有效的 binlog 日志文件列表，用于 MySQL 快速定位日志文件。

#### 2. 写入链路

每个事务都有独立的 binlog cache，事务提交时一次性把缓存中的 binlog 刷入磁盘，链路为：

```txt
事务执行 → 写入 binlog cache（线程私有内存） → 写入 OS Page Cache → 刷入磁盘 binlog 文件
```

#### 3. 核心刷盘策略（sync_binlog）

控制 binlog 刷盘行为的核心参数，直接决定主从同步和数据恢复的可靠性：

| 参数值        | 刷盘规则                              | 可靠性 | 性能  | 说明                                                      |
| :--------- | :-------------------------------- | :-- | :-- | :------------------------------------------------------ |
| **1**      | 事务提交时，必须把 binlog 直接刷入磁盘（跳过 OS 缓存） | 最高  | 最低  | 双 1 配置的核心，和 innodb_flush_log_at_trx_commit=1 配合，保障数据零丢失 |
| **0**      | 事务提交时不刷盘，由操作系统决定何时刷盘              | 最低  | 最高  | 宕机丢失大量 binlog，主从同步断裂，生产环境严禁使用                           |
| **N（N>1）** | 每提交 N 个事务，统一刷一次盘                  | 中等  | 中等  | 宕机丢失 N 个事务的 binlog，非核心场景可使用                             |

### 5.4 常用实操命令

#### 1. 基础查看命令

```sql
-- 查看binlog是否开启
SHOW VARIABLES LIKE 'log_bin';

-- 查看当前binlog格式
SHOW VARIABLES LIKE 'binlog_format';

-- 查看所有binlog文件列表
SHOW BINARY LOGS;

-- 查看当前正在写入的binlog文件
SHOW MASTER STATUS;

-- 查看binlog文件内容（文本格式，仅STATEMENT格式可读）
SHOW BINLOG EVENTS IN 'binlog.000001';
```

#### 2. 离线查看 binlog（mysqlbinlog 工具）

ROW 格式的 binlog 是二进制的，需要用 mysqlbinlog 工具解析：

```bash
# 基础解析binlog文件
mysqlbinlog /var/lib/mysql/binlog.000001

# 解析ROW格式binlog，显示行数据变更
mysqlbinlog -v --base64-output=decode-rows /var/lib/mysql/binlog.000001

# 按时间范围解析
mysqlbinlog --start-datetime="2026-03-01 00:00:00" --stop-datetime="2026-03-10 23:59:59" /var/lib/mysql/binlog.000001

# 按位置范围解析
mysqlbinlog --start-position=107 --stop-position=1000 /var/lib/mysql/binlog.000001
```

#### 3. 管理命令

```sql
-- 手动刷新binlog，生成新的日志文件（备份前必执行）
FLUSH LOGS;

-- 删除指定binlog之前的所有文件
PURGE BINARY LOGS TO 'binlog.000005';

-- 删除指定时间之前的所有binlog
PURGE BINARY LOGS BEFORE '2026-03-01 00:00:00';

-- 清空所有binlog（重置，主从架构严禁使用）
RESET MASTER;
```

### 5.5 核心参数配置与优化建议

| 参数名              | 默认值 | 优化建议                      | 说明                            |
| :--------------- | :-- | :------------------------ | :---------------------------- |
| log_bin          | OFF | 必须开启为 ON                  | 开启 binlog，主从架构和数据恢复必须开启       |
| server_id        | 1   | 主从架构必须设置唯一值（1-2^32-1）     | 主从同步的节点唯一标识，主从不能重复            |
| binlog_format    | ROW | 保持 ROW 格式                 | binlog 日志格式，生产环境必须用 ROW       |
| sync_binlog      | 1   | 核心场景保持 1，普通场景可设为 100-1000 | binlog 刷盘策略                   |
| expire_logs_days | 0   | 设为 7-30 天                 | binlog 自动过期时间，0 表示永久保留，避免磁盘占满 |
| max_binlog_size  | 1GB | 保持默认 1GB                  | 单个 binlog 文件的最大大小，写满自动生成新文件   |

### 5.6 常见问题

1. **binlog 和 redo log 的核心区别？**
    
    - 所属层级不同：binlog 是服务层，redo log 是 InnoDB 引擎层；
    - 日志类型不同：binlog 是逻辑日志，redo log 是物理日志；
    - 写入方式不同：binlog 追加写入，redo log 循环写入；
    - 作用不同：binlog 用于主从复制和数据恢复，redo log 用于崩溃恢复和持久化。
    
2. **开启 binlog 会影响性能吗？**
    
    - 会有轻微的性能损耗，因为多了一次日志刷盘的开销，但双 1 配置下，性能损耗在 10% 以内，换来的数据安全性是完全值得的。
    
3. **怎么通过 binlog 恢复误删的数据？**
    
    - 核心步骤：先恢复最近的全量备份，再通过 mysqlbinlog 工具，把备份时间点到误删时间点之间的 binlog 重放，跳过误删的 SQL 语句，即可恢复数据。

## 六、三大日志的协同工作：两阶段提交（2PC）

两阶段提交是 MySQL 为了保证 **redo log 和 binlog 两份日志的数据一致性** 设计的核心机制，是事务提交的完整流程，也是面试必考点。

### 6.1 为什么需要两阶段提交？

redo log 是引擎层日志，binlog 是服务层日志，两者是完全独立的。如果没有两阶段提交，可能出现：

- redo log 写成功，binlog 写失败：数据库重启后，通过 redo log 恢复了数据，但 binlog 没有记录，主从同步时从库会丢失这条数据，主从不一致；
- binlog 写成功，redo log 写失败：数据库重启后，redo log 没有记录，事务回滚，但 binlog 已经记录了操作，主从同步时从库会执行这条操作，主从不一致。

两阶段提交就是为了保证：**只有 redo log 和 binlog 都完整写入磁盘，事务才算真正提交成功**，两者要么都成功，要么都失败。

### 6.2 两阶段提交完整流程

以一条 `UPDATE user_info SET age = 26 WHERE user_id = 1;` 语句为例，完整的事务提交流程如下：

1. **执行器调用引擎接口**：执行器先把 SQL 语句传给 InnoDB 引擎，引擎把 user_id=1 的数据行从磁盘加载到 Buffer Pool 中；
2. **写 undo log**：记录更新的反向操作到 undo log，同时把 undo log 对应的 redo log 写入 Log Buffer；
3. **修改内存数据**：修改 Buffer Pool 中的数据行，生成脏页；
4. **写 redo log 到 Log Buffer**：记录本次修改的物理变更到 redo log，标记为 **prepare（准备）状态**；
5. **写 binlog 到 cache**：执行器生成本次更新的 binlog，写入线程私有的 binlog cache；
6. **第一阶段：redo log 刷盘**：事务提交，先把 Log Buffer 中的 redo log 刷入磁盘，保持 prepare 状态；
7. **第二阶段：binlog 刷盘**：把 binlog cache 中的 binlog 完整刷入磁盘；
8. **事务提交完成**：把磁盘中的 redo log 标记为 **commit（提交）状态**，执行器返回操作成功给客户端；
9. **后台异步刷脏页**：InnoDB 后台线程在合适的时机，把 Buffer Pool 中的脏页刷入磁盘，完成数据的最终持久化。

### 6.3 崩溃恢复的判断规则

MySQL 宕机重启后，会通过 redo log 和 binlog 做崩溃恢复，判断规则如下：

1. 如果 redo log 已经是 commit 状态：直接提交事务，数据恢复；
2. 如果 redo log 只有 prepare 状态：
    
    - 检查对应的 binlog 是否完整存在且已经刷盘：如果存在，提交事务；
    - 如果 binlog 不存在 / 不完整：回滚事务，保证两份日志一致。
    

### 6.4 双 1 配置

生产环境核心交易场景，必须开启**双 1 配置**，保障数据零丢失：

- `innodb_flush_log_at_trx_commit = 1`：事务提交时，redo log 必须刷盘；
- `sync_binlog = 1`：事务提交时，binlog 必须刷盘。

> 双 1 配置下，每个事务提交都要做两次磁盘刷盘，是 MySQL 能保证数据不丢失的最高级别配置。

## 七、常见坑与避坑指南

1. **redo log 设置太小，导致性能频繁抖动**
    
    - 坑：redo log 设为默认 48MB，频繁写满触发 checkpoint，强制刷脏页，导致写入操作频繁阻塞；
    - 避坑：生产环境把单个 redo log 文件设为 1GB-4GB，总大小不超过服务器内存的一半。
    
2. **binlog 未开启，误删数据无法恢复**
    
    - 坑：测试环境 / 非核心业务未开启 binlog，误删数据后无法恢复；
    - 避坑：所有生产环境必须开启 binlog，设置合理的过期时间，定期备份 binlog。
    
3. **binlog 格式用 STATEMENT，导致主从不一致**
    
    - 坑：使用 STATEMENT 格式，SQL 中使用 RAND ()、NOW ()、UUID () 等不确定函数，主库和从库执行结果不同，主从不一致；
    - 避坑：生产环境必须使用 ROW 格式的 binlog。
    
4. **长事务导致 undo log 膨胀，磁盘占满**
    
    - 坑：长事务持有旧的 Read View，导致 undo log 无法回收，文件膨胀到几十 GB，占用大量磁盘空间；
    - 避坑：严禁长事务，把大事务拆分为小事务，开启独立 undo 表空间和自动截断。
    
5. **双 1 配置未开启，宕机丢失数据**
    
    - 坑：为了提升性能，把 innodb_flush_log_at_trx_commit 设为 0/2，sync_binlog 设为 0，宕机后丢失已提交的事务数据；
    - 避坑：核心交易场景必须开启双 1 配置，普通场景可折中设置，但严禁设为 0。
    
6. **binlog 未设置过期时间，磁盘占满**
    
    - 坑：expire_logs_days 设为 0，binlog 永久保留，运行一段时间后磁盘被占满，数据库无法写入；
    - 避坑：设置 binlog 过期时间为 7-30 天，定期监控磁盘使用率。

## 九、核心要点总结

1. **三大日志分层**：redo log 和 undo log 是 InnoDB 引擎层日志，分别保障事务持久性和原子性；binlog 是服务层日志，负责主从复制和数据恢复。
2. **WAL 机制**：先写日志，再写磁盘，把随机 IO 转为顺序 IO，大幅提升写入性能，同时保障崩溃恢复能力。
3. **日志类型**：redo log 是物理日志，记录数据页修改；undo log 和 binlog 是逻辑日志，分别记录反向操作和 SQL 执行逻辑。
4. **写入方式**：redo log 固定大小循环写入，binlog 追加写入永不覆盖，undo log 追加写入异步回收。
5. **两阶段提交**：事务提交分为 prepare 和 commit 两个阶段，保证 redo log 和 binlog 数据一致，是崩溃恢复和主从同步的基础。
6. **双 1 配置**：innodb_flush_log_at_trx_commit=1 和 sync_binlog=1，是 MySQL 数据零丢失的最高保障，核心场景必须开启。
7. **生产避坑**：合理设置 redo log 大小，开启 binlog 并使用 ROW 格式，避免长事务，设置 binlog 过期时间，根据业务场景选择刷盘策略。

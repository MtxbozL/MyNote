## 一、开篇：为什么要学习 MySQL 架构？

### 1. 核心学习价值

学习 MySQL 架构，本质是搞懂「一条 SQL 语句从客户端发出，到 MySQL 返回结果，中间到底经历了什么」，解决三大核心问题：

- **性能定位**：SQL 执行慢、超时，能快速定位是哪个环节出问题（语法错误？优化器选错索引？存储引擎 IO 瓶颈？）；
- **原理理解**：搞懂之前学的事务、索引、锁、视图、存储过程、触发器，到底在 MySQL 的哪个模块生效；
- **优化落地**：从架构层面做数据库优化，而不是只停留在「加索引」的表层操作；
- **面试刚需**：MySQL 架构、SQL 执行流程、日志体系是后端 / 运维 / DBA 岗位的面试高频考点。

### 2. MySQL 架构的核心特点

MySQL 最核心的设计是**插件式分层架构**，核心分为「连接层、服务层、存储引擎层」三大层，其中**服务层与存储引擎层完全解耦**：

- 服务层：封装了 MySQL 的核心能力，所有存储引擎共用；
- 存储引擎层：负责数据的存储和读取，可插拔式替换，不同存储引擎提供不同的能力（如事务、锁、索引类型）。

### 3. MySQL 整体架构全景

MySQL 8.0 完整架构分为三大核心层，从外到内依次为：

plaintext

```
客户端 → 【连接层】 → 【服务层】 → 【存储引擎层】 → 磁盘文件系统
```

> 关键说明：本笔记基于**MySQL 8.0** 版本（当前主流稳定版），会明确标注与 5.7 版本的核心差异。

## 二、第一层：连接层（网络接入层）

### 2.1 核心作用

连接层是 MySQL 的「大门」，负责处理客户端的网络连接、身份认证、权限校验、连接管理，是客户端与 MySQL 服务端交互的第一站。

### 2.2 核心模块拆解

#### 1. 连接协议支持

MySQL 支持多种客户端连接协议：

- **TCP/IP 协议**：最常用，支持跨机器远程连接，如 `mysql -h 127.0.0.1 -P 3306 -u root -p`；
- **Unix Socket 协议**：仅支持本机连接，无需经过网络，性能更高，如 `mysql -S /tmp/mysql.sock -u root -p`；
- 共享内存、命名管道：仅 Windows 环境支持，极少使用。

#### 2. 连接器（Connector）

连接器是连接层的核心，负责两件事：

- **身份认证**：校验客户端传入的用户名、密码、主机地址，认证不通过直接拒绝连接；
- **权限分配**：认证通过后，读取该用户的权限表，后续该连接的所有操作，都基于这个权限上下文校验。

#### 3. 连接池（Connection Pool）

MySQL 为每个连接创建一个独立线程，频繁创建 / 销毁连接会消耗大量服务器资源，连接池的核心作用是：

- 管理长连接，实现连接复用，避免频繁创建销毁连接的开销；
- 限制最大连接数，避免服务器连接数爆满宕机（默认最大连接数 151，可通过 `max_connections` 配置）。

### 2.3 连接的生命周期

1. 客户端发起 TCP 连接请求，与 MySQL 3306 端口建立连接；
2. 连接器进行身份认证，认证通过后读取用户权限；
3. 连接建立完成，客户端发送 SQL 语句；
4. MySQL 处理 SQL，返回结果；
5. 客户端断开连接，MySQL 回收连接资源。

### 2.4 常见问题

- **长连接内存溢出**：长连接累计的大查询结果会占用 MySQL 内存，长时间不释放会导致 OOM，解决方案：定期断开长连接、MySQL 8.0+ 可通过 `mysql_reset_connection` 重置连接资源；
- **连接数爆满**：大量空闲长连接占用连接数，导致新连接无法建立，解决方案：设置 `wait_timeout` 自动释放空闲连接、优化业务代码避免连接泄露。

## 三、第二层：服务层（MySQL 核心功能层）

服务层是 MySQL 的「大脑」，封装了 MySQL 的所有核心能力，**所有存储引擎共用这一层的功能**，我们之前学的绝大多数 SQL 能力，都在这一层实现。

### 3.1 核心作用

负责 SQL 语句的接收、解析、优化、执行，封装了视图、存储过程、触发器、函数、权限控制、日志归档等核心功能。

### 3.2 核心模块拆解（重点）

#### 1. 接口模块

对接连接层，接收客户端发送的 SQL 语句，同时将执行结果返回给客户端，是服务层的入口和出口。

#### 2. 查询解析器（Parser）

SQL 语句进入 MySQL 后，第一站就是解析器，核心做 3 件事：

- **词法分析**：把 SQL 语句拆分成一个个关键字（如 `SELECT`、`FROM`、`WHERE`）、表名、字段名、值；
- **语法分析**：根据 MySQL 语法规则，校验 SQL 语法是否正确，语法错误会直接返回 `You have an error in your SQL syntax` 报错；
- **语义检查**：校验表、字段是否存在，别名是否有歧义，权限是否合法，校验不通过直接报错。

最终，解析器会生成一棵**语法解析树**，交给后续的优化器处理。

#### 3. 查询优化器（Optimizer）

优化器是 MySQL 的「军师」，核心作用是：**基于解析器生成的语法树，生成成本最低的执行计划**。

- 这里的「成本」，指的是 IO 成本（磁盘读写）+ CPU 成本（数据计算），MySQL 会选择成本最低的方案执行。

优化器的核心工作包括：

- 索引选择：决定使用哪个索引，或者是否全表扫描；
- 连接顺序优化：多表关联查询时，决定表的连接顺序；
- 语句重写：优化子查询、合并条件、简化表达式，选择更高效的执行路径；
- 执行计划生成：最终生成一个可执行的执行计划，交给执行器执行。

> 衔接之前的知识点：我们用 `EXPLAIN` 查看的执行计划，就是优化器生成的。

#### 4. 执行器（Executor）

执行器是 MySQL 的「执行者」，核心作用是：**根据优化器生成的执行计划，调用存储引擎的接口，完成 SQL 的执行，并返回结果**。

执行器的核心工作：

- **权限二次校验**：再次校验当前连接对操作的表 / 字段是否有权限，避免权限绕过；
- **调用存储引擎接口**：根据执行计划，调用存储引擎提供的统一读写接口，和存储引擎层交互；
- **结果处理与返回**：读取存储引擎返回的数据，进行聚合、排序、过滤等处理，最终返回给客户端。

#### 5. 核心功能组件

我们之前学的绝大多数高级功能，都封装在这一层：

- 视图：解析视图定义，转换为对基表的查询；
- 存储过程 / 函数：执行预编译的存储程序逻辑；
- 触发器：监听表的 DML 事件，触发预设逻辑；
- 权限控制：细粒度的表、字段、操作权限校验；
- 锁管理：服务层的表级锁管理（行级锁由存储引擎实现）。

#### 6. 日志模块：binlog（归档日志）

binlog 是**服务层独有的日志**，所有存储引擎都可以生成，核心作用：

- 数据归档：记录所有对数据库数据有修改的 SQL 语句（DDL/DML），是 MySQL 数据备份、恢复的核心；
- 主从复制：MySQL 主从架构的核心，主库通过 binlog 把数据变更同步给从库，实现数据复制。

> 关键说明：MySQL 8.0 彻底移除了「查询缓存」模块，5.7 及之前版本的查询缓存，因为命中率极低、锁冲突严重，已经被废弃，无需再学习。

## 四、第三层：存储引擎层（数据存储与读写层）

### 4.1 核心作用

存储引擎层是 MySQL 的「手脚」，负责**数据的物理存储和读取**，直接和磁盘文件系统交互，基于服务层的执行计划，完成数据的增删改查。

MySQL 的存储引擎是**插件式可插拔**的，服务层通过统一的接口和存储引擎交互，不同存储引擎之间完全独立，提供不同的能力。

### 4.2 主流存储引擎对比

表格

|对比维度|InnoDB（MySQL 5.5+ 默认引擎）|MyISAM（5.5 之前默认）|Memory（内存引擎）|
|:--|:--|:--|:--|
|**事务支持**|完整支持 ACID 事务|不支持|不支持|
|**锁级别**|行级锁（默认）+ 表级锁|仅表级锁|表级锁|
|**崩溃恢复**|支持，通过 redo log/undo log 实现|不支持，崩溃后易损坏数据|不支持，数据全在内存，重启丢失|
|**外键支持**|支持|不支持|不支持|
|**索引类型**|聚簇索引 + 辅助索引|非聚簇索引|哈希索引 + B 树索引|
|**MVCC**|支持（多版本并发控制）|不支持|不支持|
|**核心特点**|高并发、高可靠、支持事务|只读场景性能高、占用空间小|极致性能，数据存在内存|
|**适用场景**|互联网业务、核心交易、高并发读写|静态数据、只读报表、历史归档|临时表、热点数据缓存、计算中间结果|

### 4.3 InnoDB 核心架构（重点）

InnoDB 是 MySQL 默认存储引擎，也是 99% 业务场景的选择，其架构分为**内存结构**和**磁盘结构**两大部分：

#### 1. 内存结构

核心是**缓冲池（Buffer Pool）**，是 InnoDB 的性能核心，本质是一块内存区域，用来缓存磁盘上的数据页和索引页，避免每次查询都访问磁盘：

- **Change Buffer**：写缓冲，缓存对非唯一辅助索引的修改操作，减少随机 IO，提升写入性能；
- **自适应哈希索引**：InnoDB 自动为热点数据页创建哈希索引，提升查询效率；
- **Log Buffer**：日志缓冲区，缓存 redo log 日志，减少磁盘刷盘次数。

#### 2. 磁盘结构

- **表空间**：
    
    - 独立表空间：每张表对应一个 `.ibd` 文件，存储表的数据和索引，推荐开启；
    - 系统表空间：存储 InnoDB 的元数据、undo log、数据字典等；
    
- **redo log**：重做日志，InnoDB 独有，用于崩溃恢复，保证事务的持久性；
- **undo log**：回滚日志，InnoDB 独有，用于事务回滚、MVCC 多版本控制。

## 五、核心实战：一条 SQL 的完整执行流程

把前面的架构分层串联起来，彻底搞懂 SQL 的执行全流程，这是面试高频必考题。

### 5.1 一条 SELECT 查询语句的执行全流程

sql

```
SELECT user_name, age FROM user_info WHERE user_id = 1;
```

执行步骤：

1. **连接层**：客户端与 MySQL 建立 TCP 连接，连接器完成身份认证、权限校验，连接建立完成；
2. **服务层 - 接口模块**：接收 SQL 语句，传递给解析器；
3. **服务层 - 解析器**：做词法分析、语法分析、语义检查，生成语法解析树，校验表、字段是否存在；
4. **服务层 - 优化器**：基于语法树，判断可用索引，生成成本最低的执行计划（这里选择 user_id 主键索引）；
5. **服务层 - 执行器**：做权限二次校验，根据执行计划，调用 InnoDB 存储引擎的接口；
6. **存储引擎层**：InnoDB 通过主键索引，找到 user_id=1 的数据行，返回给执行器；
7. **服务层 - 执行器**：读取存储引擎返回的数据，处理后返回给客户端；
8. **连接层**：客户端接收查询结果，本次 SQL 执行完成。

### 5.2 一条 UPDATE 更新语句的执行全流程（含事务）

sql

```
UPDATE user_info SET age = 26 WHERE user_id = 1;
```

更新语句比查询多了**事务、日志刷盘、两阶段提交**的核心流程，步骤如下：

1. 连接层建立连接，解析器解析 SQL，优化器选择主键索引执行计划，执行器校验权限；
2. **执行器调用 InnoDB 接口**，查询 user_id=1 的数据行，如果数据在 Buffer Pool 中，直接返回；否则从磁盘加载到 Buffer Pool；
3. **InnoDB 加锁**：给该行数据加排他行锁，防止并发修改；
4. **InnoDB 写 undo log**：记录修改前的旧数据，用于事务回滚；
5. **InnoDB 修改 Buffer Pool 中的数据**：把 age 更新为 26，此时数据是内存中的脏页，还没刷到磁盘；
6. **InnoDB 写 redo log 到 Log Buffer**：记录本次修改的物理变更；
7. **服务层写 binlog**：记录本次更新的 SQL 逻辑，写入 binlog 文件；
8. **两阶段提交**：
    
    - 第一阶段（prepare）：redo log 刷盘到磁盘，标记为准备状态；
    - 第二阶段（commit）：binlog 刷盘到磁盘，把 redo log 标记为提交状态，事务正式提交；
    
9. **InnoDB 后台 IO 线程**：异步把 Buffer Pool 中的脏页刷到磁盘，完成物理数据的持久化；
10. 执行器返回更新结果给客户端，SQL 执行完成。

## 六、MySQL 核心日志体系（架构核心）

MySQL 的日志体系是保证数据一致性、崩溃恢复、主从复制的核心，分为服务层日志和存储引擎层日志两大类。

### 6.1 三大核心日志对比

表格

|日志类型|所属层级|核心作用|写入方式|生命周期|
|:--|:--|:--|:--|:--|
|**binlog**|服务层|数据归档、主从复制|逻辑日志（记录 SQL 语句 / 数据变更逻辑）|追加写入，文件满了生成新文件，可长期保留|
|**redo log**|InnoDB 存储引擎层|崩溃恢复、保证事务持久性|物理日志（记录数据页的物理修改）|循环写入，固定大小，写满后覆盖旧日志|
|**undo log**|InnoDB 存储引擎层|事务回滚、MVCC 多版本控制|逻辑日志（记录数据修改的反向操作）|事务提交后可被 purge 线程回收|

### 6.2 binlog vs redo log 核心区别（面试高频）

表格

|对比维度|binlog|redo log|
|:--|:--|:--|
|**所属层级**|服务层，所有存储引擎通用|InnoDB 存储引擎独有|
|**日志类型**|逻辑日志，记录 SQL 执行逻辑|物理日志，记录数据页的物理变更|
|**写入方式**|追加写入，无大小限制|循环写入，固定大小，写满覆盖|
|**核心作用**|数据备份恢复、主从复制|崩溃恢复，防止 MySQL 宕机后数据丢失|
|**提交时机**|事务提交时一次性刷盘|事务执行中就会写入，分阶段刷盘|

### 6.3 两阶段提交

两阶段提交是 MySQL 为了保证**binlog 和 redo log 的数据一致性**设计的核心机制：

- 如果只有 redo log，MySQL 宕机重启后可以恢复数据，但 binlog 没有记录，主从同步时会丢失数据；
- 如果只有 binlog，宕机后事务没有持久化，binlog 记录的变更无法生效，导致主从数据不一致；
- 两阶段提交保证了：只有 redo log 和 binlog 都完整刷盘，事务才会正式提交，任何一个环节失败，事务都会回滚，保证两份日志的一致性。

## 七、学习架构的核心应用：性能问题定位

### 7.1 SQL 执行慢，怎么通过架构定位？

1. **语法报错**：直接定位到**解析器**环节，检查 SQL 语法、表 / 字段是否存在；
2. **执行时间长，返回结果少**：大概率是**优化器**环节选错了索引，用`EXPLAIN`查看执行计划，优化索引；
3. **执行计划正确，还是慢**：大概率是**存储引擎层**问题，Buffer Pool 命中率低、磁盘 IO 瓶颈、锁等待；
4. **连接建立慢**：定位到**连接层**，检查网络、DNS 解析、连接数是否爆满；
5. **更新语句慢**：检查日志刷盘配置、锁冲突、redo log/binlog 的 IO 性能。

### 7.2 架构层面的核心优化方向

1. **连接层**：使用长连接 + 连接池，避免频繁创建连接，合理设置最大连接数；
2. **服务层**：避免大 SQL、复杂子查询，减少优化器的执行计划生成成本，关闭不必要的日志；
3. **存储引擎层**：合理设计索引，优化 Buffer Pool 大小（建议设置为服务器内存的 50%-70%），优化 redo log/binlog 的刷盘策略，使用 SSD 磁盘提升 IO 性能。

## 八、核心要点总结

1. MySQL 是**插件式分层架构**，核心分为连接层、服务层、存储引擎层，层与层之间解耦，职责清晰；
2. **连接层**：负责网络连接、身份认证、连接管理，是客户端与 MySQL 的交互入口；
3. **服务层**：是 MySQL 的核心，解析器、优化器、执行器都在这一层，视图、存储过程、触发器、binlog 也在这一层实现；
4. **存储引擎层**：负责数据的存储和读写，InnoDB 是默认引擎，支持事务、行级锁、崩溃恢复，核心是 Buffer Pool 和 redo/undo log；
5. **SQL 执行流程**：连接层 → 解析器 → 优化器 → 执行器 → 存储引擎 → 返回结果；
6. **日志体系**：binlog（服务层，归档 / 主从）、redo log（InnoDB，崩溃恢复）、undo log（InnoDB，回滚 / MVCC），两阶段提交保证两份日志的一致性；
7. 学习架构的核心价值，是从「会写 SQL」升级为「懂 SQL 的执行原理」，能快速定位和解决数据库的核心问题。

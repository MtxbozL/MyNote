排序（`ORDER BY`）和分页（`LIMIT`）是日常开发中最常用的查询操作，但在**数据量大、偏移量高**的场景下极易出现性能问题。以下是从「索引优化、查询逻辑、业务层面」三个维度整理的核心优化方案，基于 `user_info`（用户表，`user_id` 为主键）和 `order_info`（订单表，`order_id` 为主键，`create_time` 为下单时间）演示。

---

## 一、索引优化：最基础、最有效的手段

索引是排序分页性能的核心，**合理的索引能让查询效率提升 10-100 倍**。

### 1. 给排序字段加索引（避免文件排序）

#### 原理

MySQL 排序有两种方式：

- **索引排序（Using index）**：利用索引的有序性直接返回结果，无需额外排序，效率极高；
- **文件排序（Using filesort）**：在内存或磁盘中对结果集排序，效率极低（数据量大时会严重拖慢查询）。

给排序字段加索引，可让 MySQL 直接走「索引排序」。

#### 示例

```sql
-- 需求：按用户创建时间倒序分页查询
-- 优化前：无索引，走文件排序
SELECT * FROM user_info ORDER BY create_time DESC LIMIT 100000, 10;

-- 优化：给 create_time 加索引
ALTER TABLE user_info ADD INDEX idx_create_time (create_time);

-- 优化后：走索引排序，效率大幅提升
SELECT * FROM user_info ORDER BY create_time DESC LIMIT 100000, 10;
```

---

### 2. 给「WHERE 条件 + ORDER BY 排序」加联合索引（最左前缀原则）

#### 原理

如果查询同时有 `WHERE` 条件和 `ORDER BY` 排序，需创建**联合索引**，且索引顺序需符合「最左前缀原则」：

- 先放 `WHERE` 条件的字段，再放 `ORDER BY` 排序的字段；
- 这样 MySQL 能先通过 `WHERE` 字段快速过滤数据，再利用索引的有序性排序。

#### 示例

```sql
-- 需求：查询「金额大于100的订单」，按「下单时间倒序」分页
-- 优化前：无联合索引，可能走文件排序
SELECT * FROM order_info WHERE order_amount > 100 ORDER BY create_time DESC LIMIT 10000, 10;

-- 优化：创建联合索引（WHERE字段在前，ORDER BY字段在后）
ALTER TABLE order_info ADD INDEX idx_amount_time (order_amount, create_time);

-- 优化后：走联合索引，先过滤再排序
SELECT * FROM order_info WHERE order_amount > 100 ORDER BY create_time DESC LIMIT 10000, 10;
```

## 二、查询逻辑优化：解决「大偏移量」性能痛点

当偏移量很大时（如 `LIMIT 1000000, 10`），MySQL 需要扫描前 1000010 条数据，然后丢弃前 1000000 条，效率极低。以下是针对大偏移量的优化方案。

### 1. 用「WHERE 子句 + 索引」代替 LIMIT 偏移量（推荐）

#### 原理

利用**自增主键**或**有序索引字段**记录上一页的最后一条数据的位置，下一页直接从该位置开始查询，避免扫描前面的大量数据。

#### 适用场景

- 排序字段是**自增主键**或**连续有序的索引字段**；
- 业务允许「不跳页」（如 APP 的无限滚动、电商的「下一页」）。

#### 示例

```sql
-- 需求：按订单ID自增顺序分页，每页10条
-- 传统写法（大偏移量时慢）：查询第10001页
SELECT * FROM order_info ORDER BY order_id ASC LIMIT 100000, 10;

-- 优化写法（利用WHERE子句）：
-- 第1页：直接查前10条
SELECT * FROM order_info ORDER BY order_id ASC LIMIT 10;
-- 假设第1页最后一条的order_id是10，第2页从order_id > 10开始查
SELECT * FROM order_info WHERE order_id > 10 ORDER BY order_id ASC LIMIT 10;
-- 假设第10000页最后一条的order_id是100000，第10001页从order_id > 100000开始查
SELECT * FROM order_info WHERE order_id > 100000 ORDER BY order_id ASC LIMIT 10;
```

#### 关键优势

- 无论偏移量多大，查询效率都稳定（利用主键索引快速定位）；
- 避免了扫描前面的大量无效数据。

---

### 2. 延迟关联（Deferred Join）：先查主键，再关联原表

#### 原理

如果查询需要返回大量字段（如 `SELECT *`），且偏移量很大，可先通过索引查询出**主键 ID**，再通过主键 ID 关联原表获取完整数据，减少「回表」次数（回表：通过索引查到主键后，再去原表查完整数据的过程）。

#### 示例

```sql
-- 需求：按下单时间倒序分页查询订单的所有字段，偏移量100000
-- 传统写法（慢）：
SELECT * FROM order_info ORDER BY create_time DESC LIMIT 100000, 10;

-- 优化写法（延迟关联）：
-- 第一步：先通过索引查主键ID（只查ID，无需回表）
SELECT order_id FROM order_info ORDER BY create_time DESC LIMIT 100000, 10;
-- 第二步：通过主键ID关联原表，获取完整数据（主键查询极快）
SELECT o.* FROM order_info o
INNER JOIN (SELECT order_id FROM order_info ORDER BY create_time DESC LIMIT 100000, 10) temp
ON o.order_id = temp.order_id;
```

#### 关键优势

- 第一步只查主键，利用索引快速完成，无需回表；
- 第二步通过主键关联，回表次数仅为「每页条数」（如 10 次），远低于传统写法的 100010 次。

---

### 3. 覆盖索引（Covering Index）：避免回表

#### 原理

如果查询的字段**全部包含在索引中**，MySQL 可直接从索引中获取数据，无需回表，这种索引称为「覆盖索引」。

#### 示例

```sql
-- 需求：按下单时间倒序分页，只查询订单ID、订单号、下单时间
-- 优化前：查询需要回表
SELECT order_id, order_no, create_time FROM order_info ORDER BY create_time DESC LIMIT 100000, 10;

-- 优化：创建覆盖索引（包含所有查询字段 + 排序字段）
ALTER TABLE order_info ADD INDEX idx_time_no (create_time, order_no, order_id);

-- 优化后：直接从索引中获取数据，无需回表
SELECT order_id, order_no, create_time FROM order_info ORDER BY create_time DESC LIMIT 100000, 10;
```

#### 关键优势

- 完全避免回表，查询效率极高；
- 索引文件通常比原表小，IO 开销更低。

## 三、其他优化技巧

### 1. 避免 SELECT *，只查需要的字段

- `SELECT *` 会查询所有字段，增加回表开销和网络传输量；
- 只查需要的字段，更容易利用「覆盖索引」。

---

### 2. 合理设置每页条数

- 每页条数不宜过大（如不要超过 100 条），否则会返回过多数据，增加 IO 和网络开销；
- 建议每页 10-50 条，根据业务场景调整。

---

### 3. 分库分表（数据量极大时）

- 当单表数据量超过 1000 万时，即使优化索引，性能也会下降；
- 可通过「分库分表」（如按 `user_id` 哈希分表、按 `create_time` 范围分表）将数据分散到多个表中，减少单表数据量。

---

### 4. 业务层面优化

- **限制跳页**：不允许用户直接跳转到非常靠后的页面（如 “第 10000 页”），引导用户通过筛选条件缩小范围；
- **缓存热点数据**：对前几页的热点数据（如电商首页的商品列表）进行缓存（如 Redis），减少数据库查询。

---

## 四、核心优化方案总结

|优化方案|适用场景|核心原理|性能提升|
|:--|:--|:--|:--|
|**给排序字段加索引**|所有排序查询|避免文件排序，走索引排序|⭐⭐⭐⭐⭐|
|**给 WHERE+ORDER BY 加联合索引**|有筛选条件的排序查询|先过滤再排序，符合最左前缀|⭐⭐⭐⭐⭐|
|**WHERE 子句代替 LIMIT 偏移量**|不跳页的分页（如无限滚动）|利用索引快速定位，避免大偏移量|⭐⭐⭐⭐⭐|
|**延迟关联**|大偏移量、需返回大量字段的查询|先查主键，再关联原表，减少回表|⭐⭐⭐⭐|
|**覆盖索引**|只查询少量字段的查询|直接从索引取数，完全避免回表|⭐⭐⭐⭐|

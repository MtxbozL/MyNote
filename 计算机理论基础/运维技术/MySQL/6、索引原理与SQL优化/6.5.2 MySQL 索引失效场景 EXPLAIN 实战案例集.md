## 前置准备：统一测试表与数据

所有案例基于以下统一的测试表结构，方便你直接复制执行验证：

```sql
-- 1. 创建测试表
CREATE TABLE IF NOT EXISTS user_test (
    id INT PRIMARY KEY AUTO_INCREMENT COMMENT '主键',
    name VARCHAR(50) NOT NULL COMMENT '姓名',
    phone VARCHAR(11) NOT NULL COMMENT '手机号',
    age INT COMMENT '年龄',
    gender CHAR(1) COMMENT '性别（男/女）',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    INDEX idx_name (name),
    INDEX idx_phone (phone),
    INDEX idx_age (age),
    INDEX idx_gender (gender),
    INDEX idx_create_time (create_time),
    INDEX idx_name_age_gender (name, age, gender)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='索引失效测试表';

-- 2. 插入测试数据（10000行，区分度场景用）
DELIMITER //
CREATE PROCEDURE insert_test_data()
BEGIN
    DECLARE i INT DEFAULT 1;
    WHILE i <= 10000 DO
        INSERT INTO user_test (name, phone, age, gender, create_time)
        VALUES (
            CONCAT('用户', i),
            CONCAT('138', LPAD(FLOOR(RAND()*100000000), 8, '0')),
            FLOOR(18 + RAND()*60),
            IF(RAND() > 0.5, '男', '女'),
            DATE_SUB(NOW(), INTERVAL FLOOR(RAND()*365) DAY)
        );
        SET i = i + 1;
    END WHILE;
END //
DELIMITER ;

-- 执行插入（约10秒）
CALL insert_test_data();

-- 3. 插入少量数据（数据量极小场景用）
CREATE TABLE IF NOT EXISTS user_test_small LIKE user_test;
INSERT INTO user_test_small (name, phone, age, gender, create_time)
VALUES 
('张三', '13800138000', 25, '男', NOW()),
('李四', '13800138001', 30, '女', NOW()),
('王五', '13800138002', 28, '男', NOW());
```

## 第一大类：索引有序性被破坏（占 80% 失效场景）

### 案例 1：索引列使用函数 / 表达式 / 算术运算

#### 前置条件

- 表：`user_test`
- 索引：`idx_name (name)`、`idx_age (age)`、`PRIMARY KEY (id)`

#### 错误 SQL（索引失效）

sql

```
-- 1. 索引列使用函数
EXPLAIN SELECT * FROM user_test WHERE LEFT(name, 2) = '用户';

-- 2. 索引列使用算术运算
EXPLAIN SELECT * FROM user_test WHERE id + 1 = 100;

-- 3. 索引列使用表达式
EXPLAIN SELECT * FROM user_test WHERE CONCAT(name, '_test') = '用户1_test';
```

#### EXPLAIN 输出结果（以第一个为例）

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|ALL|NULL|NULL|NULL|NULL|10000|10.00|Using where|

#### 结果分析

- `type=ALL`：全表扫描，索引完全失效；
- `key=NULL`：没有使用任何索引；
- `rows=10000`：扫描了表的所有行；
- 原因：`LEFT(name,2)` 对索引列每一行进行计算，破坏了索引的有序性，无法利用 B + 树二分查找。

#### 正确 SQL（索引生效）

sql

```
-- 1. 函数放在值的一侧，或用范围查询替代
EXPLAIN SELECT * FROM user_test WHERE name LIKE '用户%';

-- 2. 算术运算放在值的一侧
EXPLAIN SELECT * FROM user_test WHERE id = 99;

-- 3. 表达式放在值的一侧
EXPLAIN SELECT * FROM user_test WHERE name = '用户1';
```

#### 正确 SQL 的 EXPLAIN 分析（以第一个为例）

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|range|idx_name|idx_name|202|NULL|1000|100.00|Using where|

- `type=range`：索引范围查询，生效；
- `key=idx_name`：使用了 name 索引；
- `rows=1000`：仅扫描了索引过滤后的少量行。

### 案例 3：联合索引违反最左前缀原则

#### 前置条件

- 表：`user_test`
- 索引：`idx_name_age_gender (name, age, gender)`
- 索引总长度：name (202) + age (5) + gender (5) = 212

#### 错误 SQL（索引失效 / 部分失效）

sql

```
-- 1. 跳过最左列，索引完全失效
EXPLAIN SELECT * FROM user_test WHERE age = 25 AND gender = '男';

-- 2. 范围查询截断，后续列失效（仅用到name+age）
EXPLAIN SELECT * FROM user_test WHERE name = '用户1' AND age > 20 AND gender = '男';

-- 3. 跳过中间列，仅用到最左列
EXPLAIN SELECT * FROM user_test WHERE name = '用户1' AND gender = '男';
```

#### EXPLAIN 输出结果（以第二个为例）

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|range|idx_name_age_gender|idx_name_age_gender|207|NULL|1|100.00|Using where|

#### 结果分析

- `type=range`：索引范围查询，部分生效；
- `key_len=207`：仅用到了 name (202) + age (5)，gender 列失效；
- 原因：`age > 20` 是范围查询，中断了后续 gender 列的匹配。

#### 正确 SQL（索引全量生效）

sql

```
-- 1. 严格遵循最左前缀，连续匹配
EXPLAIN SELECT * FROM user_test WHERE name = '用户1' AND age = 25 AND gender = '男';

-- 2. 调整索引顺序，范围查询列放在最后
ALTER TABLE user_test DROP INDEX idx_name_age_gender;
ALTER TABLE user_test ADD INDEX idx_name_gender_age (name, gender, age);
EXPLAIN SELECT * FROM user_test WHERE name = '用户1' AND gender = '男' AND age > 20;

-- 3. IN查询不会截断，三列全生效
EXPLAIN SELECT * FROM user_test WHERE name = '用户1' AND age IN (20,25) AND gender = '男';
```

#### 正确 SQL 的 EXPLAIN 分析（以第一个为例）

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|ref|idx_name_age_gender|idx_name_age_gender|212|const,const,const|1|100.00|Using where|

- `type=ref`：非唯一索引等值查询，最优；
- `key_len=212`：三列索引全量生效，利用率 100%。

## 第二大类：优化器成本计算选择全表 / 全索引

### 案例 5：索引列区分度极低

#### 前置条件

- 表：`user_test`
- 索引：`idx_gender (gender)`（gender 只有男 / 女两个值，区分度≈0.0002）

#### 错误 SQL（索引失效）

sql

```
-- gender 区分度极低，优化器选择全表扫描
EXPLAIN SELECT * FROM user_test WHERE gender = '男';
```

#### EXPLAIN 输出结果

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|ALL|idx_gender|NULL|NULL|NULL|10000|50.00|Using where|

#### 结果分析

- `type=ALL`：全表扫描，索引完全失效；
- `possible_keys=idx_gender`：优化器认为可能用到索引，但最终没选；
- 原因：gender 区分度极低，索引查询需要扫描 5000 行索引 + 回表 5000 行数据，IO 成本比全表扫描（扫描 10000 行数据）更高。

#### 正确 SQL（优化 / 避免）

sql

```
-- 1. 建立联合索引，利用高区分度列的过滤效果
ALTER TABLE user_test ADD INDEX idx_gender_name (gender, name);
EXPLAIN SELECT * FROM user_test WHERE gender = '男' AND name LIKE '用户%';

-- 2. 建立覆盖索引，避免回表
ALTER TABLE user_test ADD INDEX idx_gender_id (gender, id);
EXPLAIN SELECT id FROM user_test WHERE gender = '男';
```

#### 正确 SQL 的 EXPLAIN 分析（以第二个为例）

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|ref|idx_gender_id|idx_gender_id|6|const|5000|100.00|Using index|

- `type=ref`：索引生效；
- `Extra=Using index`：覆盖索引生效，无需回表，IO 成本比全表扫描低。

### 案例 8：使用 OR 连接非索引列

#### 前置条件

- 表：`user_test`
- 索引：`idx_name (name)`（address 列无索引，需先添加）

sql

```
ALTER TABLE user_test ADD COLUMN address VARCHAR(100) COMMENT '地址';
UPDATE user_test SET address = CONCAT('地址', id);
```

#### 错误 SQL（索引失效）

sql

```
-- OR 连接了非索引列 address，索引失效
EXPLAIN SELECT * FROM user_test WHERE name = '用户1' OR address = '地址1';
```

#### EXPLAIN 输出结果

表格

|id|select_type|table|partitions|type|possible_keys|key|key_len|ref|rows|filtered|Extra|
|---|---|---|---|---|---|---|---|---|---|---|---|
|1|SIMPLE|user_test|NULL|ALL|idx_name|NULL|NULL|NULL|10000|19.00|Using where|

#### 结果分析

- `type=ALL`：全表扫描，索引完全失效；
- 原因：OR 连接的 address 列无索引，MySQL 无法只通过索引查询所有符合条件的数据，必须全表扫描。

#### 正确 SQL（索引生效）

sql

```

-- 2. 用 UNION ALL 拆分查询，每个子查询单独使用索引
EXPLAIN SELECT * FROM user_test WHERE name = '用户1'
UNION ALL
SELECT * FROM user_test WHERE address = '地址1';
```

|---|---|---|---|---|---|---|---|---|---|---|---|
|1|PRIMARY|user_test|NULL|ref|idx_name|idx_name|202|const|1|100.00|Using where|
|2|UNION|user_test|NULL|ALL|NULL|NULL|NULL|NULL|10000|10.00|Using where|
|NULL|UNION RESULT|<union1,2>|NULL|ALL|NULL|NULL|NULL|NULL|NULL|NULL|Using temporary|

- 第一个子查询：`type=ref`，使用了 name 索引；
- 第二个子查询：address 无索引，全表扫描（需给 address 加索引优化）。

## 总结：EXPLAIN 索引失效判断标准

拿到一条 SQL 的 EXPLAIN 结果，按以下 3 步判断索引是否失效：

1. **看 `type`**：出现 `ALL` 或 `index`（全索引扫描），代表索引完全 / 部分失效；
2. **看 `key`**：`key=NULL`，代表索引完全失效；`key` 显示索引名但 `key_len` 远小于索引总长度，代表索引部分失效；
3. **看 `Extra`**：出现 `Using filesort`、`Using temporary`、`Using join buffer`，代表索引虽然可能用于 WHERE 条件，但排序 / 分组 / 关联没有利用索引，性能差。

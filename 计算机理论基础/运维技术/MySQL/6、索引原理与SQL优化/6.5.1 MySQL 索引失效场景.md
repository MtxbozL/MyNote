## 一、开篇：索引失效的核心本质与价值

### 1. 什么是索引失效

索引失效，不是指索引 “坏了” 或 “被删除了”，而是指：

> MySQL 优化器**认为全表扫描（ALL）或全索引扫描（index）的执行成本更低**，或者**索引的有序性被破坏，无法利用 B + 树的二分查找快速定位数据**，最终选择不使用索引，或仅使用索引的部分前缀列。

### 2. 学习索引失效的核心价值

表格

|核心价值|具体说明|
|:--|:--|
|**避免踩坑**：提前知道哪些操作会破坏索引，写 SQL 时主动规避，从源头减少慢查询||
|**快速定位**：遇到慢查询时，结合 EXPLAIN 快速定位索引失效的根因，精准优化||
|**举一反三**：理解失效的本质（有序性破坏、成本计算），而不是死记硬背场景，能应对所有新的 SQL 写法||
|**面试刚需**：后端 / 运维 / DBA 岗位面试必考，是 SQL 优化能力的核心体现||

### 3. 前置核心认知

1. 索引失效的本质只有两个：**有序性被破坏**、**优化器成本计算选择全表 / 全索引**；
2. 所有失效场景，最终都可以用 **EXPLAIN 执行计划** 验证（type=ALL/index、key=NULL、key_len 远小于索引总长度）；
3. 优化器的成本计算是动态的，**数据量变化、统计信息过期**，也可能导致索引失效（需定期更新统计信息）。

---

## 二、前置基础：优化器的成本计算逻辑

优化器选择索引还是全表扫描，核心依据是**执行成本**，成本越低越优先。成本主要由两部分组成：

表格

|成本类型|核心含义|计算逻辑|
|:--|:--|:--|
|**IO 成本**|磁盘读写的次数，是成本的核心（磁盘 IO 速度是内存的百万倍）|全表扫描：扫描表的所有数据页；<br><br>索引查询：扫描索引页 + 回表的数据页（如果需要）|
|**CPU 成本**|内存中数据计算、比较的次数|全表扫描：比较所有行的 WHERE 条件；<br><br>索引查询：比较索引页的索引值 + 回表后的 WHERE 条件（如果需要）|

### 为什么优化器会选择全表扫描？

即使有可用索引，优化器也可能选择全表扫描，核心原因是：

1. **索引过滤效果差**：索引列区分度极低（如性别、状态只有 2-3 个值），索引查询需要扫描大量索引页 + 回表大量数据页，IO 成本反而比全表扫描高；
2. **数据量极小**：表中只有几百 / 几千行数据，全表扫描的 IO 成本比索引查询（索引页 + 回表）更低；
3. **统计信息过期**：MySQL 的统计信息（表的总行数、索引的区分度等）是定期更新的，过期后优化器会误判成本；
4. **使用了 `FORCE INDEX` 以外的强制索引，但索引确实不合适**。

## 三、核心失效场景（分大类 + 原理 + 示例 + EXPLAIN 验证）

### 3.1 第一大类：索引的有序性被破坏（最常见，占 80% 以上）

索引的核心优势是**B + 树的有序性**，一旦有序性被破坏，MySQL 无法利用二分查找快速定位数据，只能全表 / 全索引扫描。

#### 3.1.1 场景 1：索引列使用函数 / 表达式 / 算术运算

##### 失效原理

索引存储的是**索引列的原始值**，如果在索引列上使用函数（如 `LEFT(name,1)`）、表达式（如 `id+1`）、算术运算（如 `age*2`），MySQL 需要先对**每一行数据**的索引列进行计算，再比较 WHERE 条件，**索引的有序性完全失效**，只能全表扫描。

##### 错误示例（索引失效）

sql

```
-- 表结构：user 表，name 列有普通索引 idx_name
-- 错误1：索引列使用函数
EXPLAIN SELECT * FROM user WHERE LEFT(name,1) = '张';

-- 错误2：索引列使用算术运算
EXPLAIN SELECT * FROM user WHERE id + 1 = 100;

-- 错误3：索引列使用表达式
EXPLAIN SELECT * FROM user WHERE CONCAT(name,'_test') = '张三_test';
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows = 表的总行数。

##### 正确写法（索引生效）

将**计算 / 函数 / 表达式放在值的一侧**，保持索引列的原始值不变，利用索引的有序性：

sql

```
-- 正确1：函数放在值的一侧，或用范围查询替代
EXPLAIN SELECT * FROM user WHERE name LIKE '张%';

-- 正确2：算术运算放在值的一侧
EXPLAIN SELECT * FROM user WHERE id = 99;

-- 正确3：表达式放在值的一侧
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

##### EXPLAIN 验证结果

- type=range/ref（索引生效）；
- key=idx_name/PRIMARY；
- rows = 预估扫描的少量行数。

---

#### 3.1.2 场景 2：索引列使用隐式类型转换

##### 失效原理

隐式类型转换是指**查询条件的类型与索引列的类型不一致**，MySQL 会自动对**索引列的每一行数据**进行类型转换，**索引的有序性被破坏**，只能全表扫描。

##### 高频隐式转换场景

表格

|索引列类型|查询条件类型|隐式转换方向|失效情况|
|:--|:--|:--|:--|
|VARCHAR/CHAR|INT/BIGINT|索引列的字符串 → 数字|100% 失效|
|INT/BIGINT|VARCHAR/CHAR|查询条件的字符串 → 数字|部分生效（如果查询条件是纯数字字符串）|
|DATE/DATETIME|VARCHAR/CHAR|查询条件的字符串 → 日期|部分生效（如果查询条件是标准日期格式）|

##### 错误示例（索引失效）

sql

```
-- 表结构：user 表，phone 列是 VARCHAR(11)，有普通索引 idx_phone
-- 错误：phone 是 VARCHAR，查询条件是 INT，隐式转换索引列
EXPLAIN SELECT * FROM user WHERE phone = 13800138000;
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows = 表的总行数。

##### 正确写法（索引生效）

保持**查询条件的类型与索引列的类型完全一致**：

sql

```
-- 正确：phone 是 VARCHAR，查询条件用字符串
EXPLAIN SELECT * FROM user WHERE phone = '13800138000';
```

##### EXPLAIN 验证结果

- type=ref（索引生效）；
- key=idx_phone；
- rows = 预估扫描的少量行数。

---

#### 3.1.3 场景 3：索引列使用隐式字符集转换

##### 失效原理

隐式字符集转换是指**JOIN 查询时，两个表的关联字段字符集不一致**，MySQL 会自动对**索引列的每一行数据**进行字符集转换，**索引的有序性被破坏**，只能全表扫描。

##### 错误示例（索引失效）

sql

```
-- 表结构：
-- user 表：user_id INT PRIMARY KEY, name VARCHAR(50)，字符集 utf8mb4
-- order_info 表：order_id INT PRIMARY KEY, user_id VARCHAR(50)，字符集 utf8，有普通索引 idx_userid
-- 错误：关联字段 user_id 字符集不一致（utf8mb4 vs utf8），隐式转换 order_info 的 user_id
EXPLAIN SELECT u.name, o.order_no 
FROM user u
INNER JOIN order_info o ON u.user_id = o.user_id
WHERE u.age > 18;
```

##### EXPLAIN 验证结果

- order_info 表的 type=ALL（全表扫描）；
- order_info 表的 key=NULL（索引完全失效）；
- Extra=Using join buffer (Block Nested Loop)（无索引关联，性能极差）。

##### 正确写法（索引生效）

保持**JOIN 关联字段的字符集、类型、长度完全一致**：

sql

```
-- 正确：修改 order_info 表的 user_id 为 INT，字符集 utf8mb4
ALTER TABLE order_info MODIFY COLUMN user_id INT NOT NULL;
ALTER TABLE order_info CONVERT TO CHARACTER SET utf8mb4;

-- 再次查询，索引生效
EXPLAIN SELECT u.name, o.order_no 
FROM user u
INNER JOIN order_info o ON u.user_id = o.user_id
WHERE u.age > 18;
```

##### EXPLAIN 验证结果

- order_info 表的 type=eq_ref（关联查询最优）；
- order_info 表的 key=idx_userid；
- Extra 无 Using join buffer。

---

#### 3.1.4 场景 4：索引列使用左模糊 / 全模糊匹配

##### 失效原理

B + 树的索引是**按前缀有序排列**的，只有前缀匹配（`LIKE '张%'`）才能利用索引的有序性快速定位数据；左模糊（`LIKE '%三'`）、全模糊（`LIKE '%张三%'`）无法确定前缀，**索引的有序性被破坏**，只能全表扫描。

##### 错误示例（索引失效）

sql

```
-- 表结构：user 表，name 列有普通索引 idx_name
-- 错误1：左模糊匹配
EXPLAIN SELECT * FROM user WHERE name LIKE '%三';

-- 错误2：全模糊匹配
EXPLAIN SELECT * FROM user WHERE name LIKE '%张三%';
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows = 表的总行数。

##### 正确写法（索引生效 / 优化）

1. **前缀匹配**：如果业务允许，优先使用前缀匹配：
    
    sql
    
    ```
    EXPLAIN SELECT * FROM user WHERE name LIKE '张%';
    ```
    
2. **全文索引**：如果必须使用全模糊匹配，给文本字段建立全文索引，用 `MATCH() AGAINST()` 替代 `LIKE`：
    
    sql
    
    ```
    -- 建立全文索引
    ALTER TABLE user ADD FULLTEXT INDEX idx_fulltext_name (name);
    -- 全文索引查询
    EXPLAIN SELECT * FROM user WHERE MATCH(name) AGAINST('张三');
    ```
    
3. **搜索引擎**：如果全文索引也无法满足需求（如复杂的分词、模糊匹配），使用 Elasticsearch、Solr 等搜索引擎。

---

#### 3.1.5 场景 5：联合索引违反最左前缀原则

##### 失效原理

联合索引的排序规则是：**先按最左列排序，最左列值相同时再按第二列排序，以此类推**，只有连续的最左前缀列才能利用索引的有序性；如果跳过中间列、遇到范围查询（除 IN 外），后续列的有序性被破坏，索引匹配中断。

##### 前置表结构

sql

```
-- 表结构：user 表，联合索引 idx_name_age_gender (name, age, gender)
```

##### 错误示例（索引失效 / 部分失效）

sql

```
-- 错误1：跳过最左列，索引完全失效
EXPLAIN SELECT * FROM user WHERE age = 20 AND gender = '男';

-- 错误2：范围查询截断，后续列失效（仅用到name+age列）
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age > 20 AND gender = '男';

-- 错误3：跳过中间列，仅用到最左列
EXPLAIN SELECT * FROM user WHERE name = '张三' AND gender = '男';
```

##### EXPLAIN 验证结果

- 错误 1：type=ALL，key=NULL；
- 错误 2：type=range，key=idx_name_age_gender，key_len=name+age 的长度之和；
- 错误 3：type=ref，key=idx_name_age_gender，key_len=name 的长度。

##### 正确写法（索引全量生效）

1. **严格遵循最左前缀原则**：查询条件从最左列开始连续匹配；
2. **范围查询列放在最后**：调整联合索引的列顺序，将范围查询列放在最后，避免截断后续列：
    
    sql
    
    ```
    -- 调整联合索引为 idx_name_gender_age (name, gender, age)
    ALTER TABLE user DROP INDEX idx_name_age_gender;
    ALTER TABLE user ADD INDEX idx_name_gender_age (name, gender, age);
    -- 再次查询，三列索引全量生效
    EXPLAIN SELECT * FROM user WHERE name = '张三' AND gender = '男' AND age > 20;
    ```
    
3. **IN 查询不会截断**：IN 属于等值查询，不会中断后续列的匹配：
    
    sql
    
    ```
    -- 三列索引全量生效
    EXPLAIN SELECT * FROM user WHERE name = '张三' AND age IN (20,25) AND gender = '男';
    ```

### 3.2 第二大类：优化器成本计算选择全表 / 全索引

即使索引的有序性没有被破坏，优化器也可能因为成本计算选择全表 / 全索引扫描。

#### 3.2.1 场景 6：索引列区分度极低

##### 失效原理

索引列的**区分度 = 列的唯一值数量 / 表的总行数**，区分度越高，索引过滤效果越好，IO 成本越低；如果区分度极低（如性别、状态只有 2-3 个值），索引查询需要扫描大量索引页 + 回表大量数据页，IO 成本反而比全表扫描高，优化器会选择全表扫描。

##### 错误示例（索引失效）

sql

```
-- 表结构：user 表，gender 列（只有男/女两个值）有普通索引 idx_gender
-- 错误：gender 区分度极低，优化器选择全表扫描
EXPLAIN SELECT * FROM user WHERE gender = '男';
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows = 表的总行数。

##### 正确写法（优化 / 避免）

1. **避免给低区分度列单独建索引**：除非是覆盖索引，否则单独建索引没有意义；
2. **建立联合索引**：将低区分度列和高区分度列一起建联合索引，利用高区分度列的过滤效果：
    
    sql
    
    ```
    -- 建立联合索引 idx_gender_name (gender, name)
    ALTER TABLE user ADD INDEX idx_gender_name (gender, name);
    -- 查询时同时使用 gender 和 name，索引生效
    EXPLAIN SELECT * FROM user WHERE gender = '男' AND name LIKE '张%';
    ```
    
3. **覆盖索引**：如果查询的字段都在低区分度列的索引中，建立覆盖索引，避免回表，IO 成本可能比全表扫描低：
    
    sql
    
    ```
    -- 建立覆盖索引 idx_gender_id (gender, id)
    ALTER TABLE user ADD INDEX idx_gender_id (gender, id);
    -- 查询的字段都在索引中，覆盖索引生效
    EXPLAIN SELECT id FROM user WHERE gender = '男';
    ```

#### 3.2.2 场景 7：数据量极小

##### 失效原理

如果表中只有几百 / 几千行数据，全表扫描的 IO 成本（扫描少量数据页）比索引查询（扫描索引页 + 回表数据页）更低，优化器会选择全表扫描。

##### 示例（索引失效）

sql

```
-- 表结构：user 表，只有100行数据，name 列有普通索引 idx_name
-- 优化器选择全表扫描
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows=100。

##### 说明

这种情况**无需优化**，数据量极小，全表扫描和索引查询的性能差异可以忽略不计；当数据量增长到几万 / 几十万行时，优化器会自动选择索引。

---

#### 3.2.3 场景 8：统计信息过期

##### 失效原理

MySQL 的统计信息（表的总行数、索引的区分度、数据页的数量等）是**定期自动更新**的，或者通过 `ANALYZE TABLE` 手动更新；如果统计信息过期，优化器会误判执行成本，选择错误的执行计划（如全表扫描）。

##### 示例（索引失效）

sql

```
-- 表结构：user 表，name 列有普通索引 idx_name，之前只有100行数据，优化器选择全表扫描；
-- 现在插入了100万行数据，但统计信息未更新，优化器仍选择全表扫描
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows=100（统计信息过期，显示的是旧的总行数）。

##### 正确写法（更新统计信息）

手动更新表的统计信息：

sql

```
-- 更新单表的统计信息
ANALYZE TABLE user;
-- 更新所有表的统计信息（谨慎使用，大库会耗时较长）
ANALYZE TABLE;
```

##### EXPLAIN 验证结果

- type=ref（索引生效）；
- key=idx_name；
- rows = 预估扫描的少量行数。

---

#### 3.2.4 场景 9：使用了 `OR` 连接非索引列

##### 失效原理

如果 `OR` 连接的两个条件中，**有一个条件的列没有索引**，MySQL 无法只通过索引查询所有符合条件的数据，必须全表扫描所有行，检查两个条件是否满足，**索引的有序性被破坏**，只能全表扫描。

##### 错误示例（索引失效）

sql

```
-- 表结构：user 表，name 列有普通索引 idx_name，address 列无索引
-- 错误：OR 连接了非索引列 address，索引失效
EXPLAIN SELECT * FROM user WHERE name = '张三' OR address = '北京';
```

##### EXPLAIN 验证结果

- type=ALL（全表扫描）；
- key=NULL（索引完全失效）；
- rows = 表的总行数。

##### 正确写法（索引生效）

1. **给所有 OR 条件的列建联合索引**：如果业务允许，给所有 OR 条件的列建联合索引：
    
    sql
    
    ```
    -- 建立联合索引 idx_name_address (name, address)
    ALTER TABLE user ADD INDEX idx_name_address (name, address);
    -- 再次查询，索引生效
    EXPLAIN SELECT * FROM user WHERE name = '张三' OR address = '北京';
    ```
    
2. **用 `UNION ALL` 拆分查询**：如果无法建联合索引，用 `UNION ALL` 拆分查询，每个子查询单独使用索引：
    
    sql
    
    ```
    -- 拆分查询，两个子查询都使用索引
    EXPLAIN SELECT * FROM user WHERE name = '张三'
    UNION ALL
    SELECT * FROM user WHERE address = '北京';
    ```
    
    > 注意：如果需要去重，用 `UNION`，但 `UNION` 会创建临时表，性能比 `UNION ALL` 差。

### 3.3 第三大类：其他特殊场景

#### 3.3.1 场景 10：使用了 `IS NULL`/`IS NOT NULL`（部分失效）

##### 失效原理

InnoDB 对 `NULL` 值的处理比较特殊，`NULL` 值在索引中是**有序排列的**（放在索引的最前面），但 `IS NOT NULL` 可能会扫描大量索引页 + 回表大量数据页，IO 成本比全表扫描高，优化器可能选择全表扫描；`IS NULL` 通常可以使用索引。

##### 示例

sql

```
-- 表结构：user 表，name 列允许 NULL，有普通索引 idx_name
-- 示例1：IS NULL，索引生效
EXPLAIN SELECT * FROM user WHERE name IS NULL;

-- 示例2：IS NOT NULL，可能失效（如果大部分行的 name 都不为 NULL）
EXPLAIN SELECT * FROM user WHERE name IS NOT NULL;
```

##### EXPLAIN 验证结果

- 示例 1：type=ref，key=idx_name；
- 示例 2：如果大部分行的 name 都不为 NULL，type=ALL，key=NULL。

##### 正确写法（优化）

1. **避免给允许 NULL 的列单独建索引**：除非是覆盖索引，否则单独建索引意义不大；
2. **将列设为 NOT NULL**：如果业务允许，将列设为 NOT NULL，给默认值（如空字符串、0），索引的稳定性更高；
3. **覆盖索引**：如果查询的字段都在允许 NULL 的列的索引中，建立覆盖索引，避免回表。

---

#### 3.3.2 场景 11：使用了 `ORDER BY`/`GROUP BY` 但索引不匹配

##### 失效原理

如果 `ORDER BY`/`GROUP BY` 的字段**不在索引中**，或者**索引的顺序与 ORDER BY/GROUP BY 的顺序不一致**，MySQL 无法利用索引的有序性完成排序 / 分组，需要在内存 / 磁盘中进行文件排序（Using filesort）/ 创建临时表（Using temporary），性能极差，但**索引本身可能用于 WHERE 条件的过滤**，不是完全失效。

##### 错误示例（索引部分失效，性能差）

sql

```
-- 表结构：user 表，name 列有普通索引 idx_name
-- 错误：WHERE 条件使用了 name（索引生效），但 ORDER BY 的 age 不在索引中，需要文件排序
EXPLAIN SELECT * FROM user WHERE name = '张三' ORDER BY age;
```

##### EXPLAIN 验证结果

- type=ref，key=idx_name（WHERE 条件的索引生效）；
- Extra=Using where; Using filesort（需要文件排序，性能差）。

##### 正确写法（索引全量生效，无文件排序）

给 `WHERE` 条件和 `ORDER BY`/`GROUP BY` 的字段建立**联合索引**，利用索引的有序性避免文件排序 / 临时表：

sql

```
-- 建立联合索引 idx_name_age (name, age)
ALTER TABLE user ADD INDEX idx_name_age (name, age);
-- 再次查询，索引全量生效，无文件排序
EXPLAIN SELECT * FROM user WHERE name = '张三' ORDER BY age;
```

##### EXPLAIN 验证结果

- type=ref，key=idx_name_age；
- Extra=Using where（无 Using filesort，无 Using temporary）。

## 四、进阶避坑：索引失效的其他注意事项

### 4.1 注意事项 1：不要过度依赖 `FORCE INDEX`

`FORCE INDEX` 可以强制 MySQL 使用指定的索引，但如果索引确实不合适（如区分度极低、回表成本过高），强制使用索引反而会导致性能更差；**只有在统计信息过期、优化器明显误判时，才使用 `FORCE INDEX`**。

### 4.2 注意事项 2：定期更新统计信息

MySQL 的统计信息会随着数据的插入、更新、删除而逐渐过期，建议**定期（如每周）更新核心表的统计信息**：

sql

```
-- 更新核心表的统计信息
ANALYZE TABLE user, order_info, product;
```

### 4.3 注意事项 3：避免给大字段建索引

大字段（如 TEXT、BLOB、LONGTEXT）的索引会占用大量磁盘空间，且查询效率低；如果必须给大字段建索引，建议使用**前缀索引**（只索引字段的前 N 个字符）：

sql

```
-- 给 content 字段（TEXT）建立前缀索引，只索引前20个字符
ALTER TABLE article ADD INDEX idx_content_prefix (content(20));
```

> 注意：前缀索引的长度要根据字段的区分度确定，区分度越高，前缀长度可以越短。

### 4.4 注意事项 4：避免频繁更新索引列

频繁更新索引列会导致索引的 B + 树频繁分裂、合并，维护成本极高，性能下降；**尽量将索引列设为静态字段**（如 user_id、create_time），避免频繁更新。

## 六、核心要点总结

1. **索引失效的两大本质**：索引的有序性被破坏、优化器成本计算选择全表 / 全索引扫描。
2. **80% 的失效场景**：索引列使用函数 / 表达式 / 算术运算、隐式类型 / 字符集转换、左模糊 / 全模糊匹配、联合索引违反最左前缀原则。
3. **优化的核心思路**：保持索引列的原始值不变、严格遵循最左前缀原则、给 JOIN/ORDER BY/GROUP BY 的字段建联合索引、避免给低区分度列单独建索引。
4. **验证的唯一工具**：EXPLAIN 执行计划，重点看 type、key、key_len、Extra 四个字段。
5. **进阶避坑**：不要过度依赖 FORCE INDEX、定期更新统计信息、避免给大字段建索引、避免频繁更新索引列。

## 一、开篇：核心定义与价值

### 1. 什么是联合索引

联合索引（也叫复合索引、组合索引），是**对表中 2 个及以上的列共同创建的索引**，属于非聚簇索引的一种，底层依然是 B + 树结构。

与单值索引（仅对单个列建索引）不同，联合索引的排序规则是：**先按最左侧的第一列排序，第一列值相同时，再按第二列排序，第二列值相同时，再按第三列排序，以此类推**。

示例：对 `(a, b, c)` 创建联合索引，索引的排序逻辑为：

1. 先按 `a` 列的值升序排序；
2. `a` 列值相同的行，再按 `b` 列的值升序排序；
3. `a`、`b` 列值都相同的行，最后按 `c` 列的值升序排序。

### 2. 为什么用联合索引，而不是多个单值索引？

这是新手最容易踩的误区：给每个查询列都建一个单值索引，不如建一个合理的联合索引，核心原因如下：

表格

|对比维度|联合索引|多个单值索引|
|:--|:--|:--|
|**查询效率**|一次索引查询即可匹配多条件，无需回表 / 多次索引扫描|MySQL 查询时通常只会选择一个最优的单值索引，其他条件需回表过滤，效率低|
|**索引维护成本**|仅需维护 1 个 B + 树，插入 / 更新 / 删除的开销低|需维护多个 B + 树，写操作开销成倍增加|
|**磁盘占用**|仅存储 1 份索引数据，占用空间小|多份索引数据，占用大量磁盘空间|
|**覆盖索引能力**|可通过合理设计列顺序，轻松实现覆盖索引，避免回表|单值索引很难覆盖多字段查询，几乎无法避免回表|

> 核心结论：**多条件联合查询场景下，一个设计合理的联合索引，远优于多个单值索引**。

### 3. 核心前置认知

1. 联合索引的底层依然是 B + 树结构，仅支持 InnoDB/MyISAM 等支持 B + 树索引的引擎；
2. 联合索引的生效规则，完全由**最左前缀匹配原则**决定，这是联合索引的灵魂；
3. 联合索引的列顺序，直接决定了索引的利用率和查询效率，是设计的核心；
4. 联合索引的最大列数，MySQL 限制为**最多 16 个列**。

## 二、联合索引的底层存储结构（理解最左前缀的前提）

要彻底搞懂最左前缀原则，必须先理解联合索引在 B + 树中是如何存储的，这是规则的底层根源。

我们以 `user` 表的联合索引 `idx_name_age_gender (name, age, gender)` 为例，拆解其存储结构：

### 2.1 联合索引的 B + 树存储逻辑

1. **非叶子节点**：仅存储索引列的值和子节点指针，先按 `name` 排序，`name` 相同按 `age` 排序，`age` 相同按 `gender` 排序，用于引导查询方向；
2. **叶子节点**：存储完整的索引列值 + 聚簇索引的主键值，依然严格遵循上述排序规则，且叶子节点之间通过双向指针串联；
3. **核心特点**：**只有第一列是全局有序的，第二列及后续列，仅在第一列值相同时，才是局部有序的**。

举个直观的例子，索引的叶子节点数据排序如下：

表格

|name|age|gender|主键 id|
|:--|:--|:--|:--|
|李四|20|男|2|
|张三|18|女|5|
|张三|20|男|1|
|张三|25|男|3|
|王五|30|女|4|

可以清晰看到：

- `name` 列是全局有序的（李四→张三→王五）；
- `age` 列仅在 `name='张三'` 时是有序的（18→20→25），全局是无序的；
- `gender` 列仅在 `name` 和 `age` 都相同时，才是有序的。

### 2.2 最左前缀原则的底层根源

正是因为联合索引的排序规则：**只有最左列是全局有序的，后续列仅在前序列值相同时局部有序**，所以 MySQL 查询联合索引时，**必须从最左列开始匹配，才能利用索引的有序性快速定位数据**。

如果跳过最左列，直接查询第二列（如 `WHERE age=20`），由于 `age` 列全局无序，无法利用索引的排序特性，只能全表扫描，索引完全失效。

这就是最左前缀原则的底层本质，不是 MySQL 的硬性规定，而是由 B + 树的排序结构决定的。

## 三、核心重点：最左前缀匹配原则

### 3.1 什么是最左前缀原则

最左前缀原则，是 MySQL 联合索引的匹配核心规则，定义为：

> MySQL 查询联合索引时，**会从索引的最左侧列开始匹配，遇到范围查询、非等值匹配、列缺失时，会中断后续列的匹配**；只要保证最左前缀列连续匹配，就能利用索引加速查询。

这里的「前缀」包含两层含义：

1. **联合索引的列前缀**：必须从最左列开始，不能跳过中间列；
2. **字符串的前缀匹配**：对于字符串类型的列，支持前缀匹配（如 `name LIKE '张%'`），不支持后缀匹配（`name LIKE '%三'`）。

### 3.2 核心匹配规则详解（分场景 + 示例）

我们以联合索引 `idx_name_age_gender (name, age, gender)` 为例，分场景拆解所有匹配规则，所有示例均通过 `EXPLAIN` 验证生效情况。

#### 3.2.1 规则 1：连续最左匹配，索引全量生效

只要查询条件**从最左列开始，连续匹配索引的前缀列**，就能完整利用匹配到的索引列，无匹配中断。

##### 生效场景（索引全量 / 部分生效）

sql

```
-- 1. 匹配全部3列，索引完全生效，key_len=所有列的长度之和
SELECT * FROM user WHERE name = '张三' AND age = 20 AND gender = '男';

-- 2. 匹配前2列，索引生效（name+age），key_len=name+age的长度之和
SELECT * FROM user WHERE name = '张三' AND age = 20;

-- 3. 仅匹配第1列，索引生效（仅name），key_len=name的长度
SELECT * FROM user WHERE name = '张三';

-- 4. 字符串前缀匹配，索引生效（仅name）
SELECT * FROM user WHERE name LIKE '张%';
```

##### 失效场景（索引完全失效）

sql

```
-- 跳过最左列，直接匹配后续列，索引完全失效，type=ALL全表扫描
SELECT * FROM user WHERE age = 20 AND gender = '男';
SELECT * FROM user WHERE gender = '男';
SELECT * FROM user WHERE age = 20;
```

#### 3.2.2 规则 2：匹配中断规则 —— 跳过中间列，仅前缀列生效

如果查询条件**跳过了中间的索引列**，则仅能匹配到跳过列之前的索引列，跳过列及后续列的索引完全失效。

##### 示例

sql

```
-- 仅name列索引生效，gender列无法利用索引，需回表过滤
SELECT * FROM user WHERE name = '张三' AND gender = '男';
```

- 原因：跳过了中间列 `age`，匹配在 `name` 列后中断，`gender` 列全局无序，无法利用索引排序特性；
- 验证：`EXPLAIN` 的 `key_len` 仅为 `name` 列的长度，而非 `name+gender` 的长度之和。

#### 3.2.3 规则 3：范围查询截断规则 —— 范围查询后，后续列索引失效

这是最核心、最高频的规则，也是面试必考点：

> 当查询条件中出现**范围查询**（`>`、`<`、`>=`、`<=`、`BETWEEN AND`、`LIKE '前缀%'`）时，范围列可以匹配索引，但**范围列之后的所有列，索引匹配完全中断，无法利用索引**。

##### 示例

sql

```
-- name+age列索引生效，gender列索引失效
SELECT * FROM user WHERE name = '张三' AND age > 20 AND gender = '男';
```

- 原因：`age` 是范围查询，匹配在 `age` 列后中断，`gender` 列仅在 `age` 固定值时有序，范围查询后 `gender` 无序，无法利用索引；
- 验证：`EXPLAIN` 的 `key_len` 为 `name+age` 的长度之和，而非三列总长度。

##### 优化方案

将**等值查询列前置，范围查询列放在最后**，调整索引顺序为 `idx_name_gender_age (name, gender, age)`，此时查询：

sql

```
-- 三列索引全部生效，因为范围查询age在最后一列，无后续列
SELECT * FROM user WHERE name = '张三' AND gender = '男' AND age > 20;
```

#### 3.2.4 规则 4：IN 等值查询不会中断匹配

很多人会混淆「IN 查询」和「范围查询」，这里明确：

> `IN (xxx,xxx)` 属于**等值查询**，不是范围查询，不会中断后续列的索引匹配，MySQL 会自动优化 IN 查询的顺序，保证索引匹配。

##### 示例

sql

```
-- 三列索引全部生效，IN是等值查询，不会中断gender列的匹配
SELECT * FROM user WHERE name = '张三' AND age IN (20,25,30) AND gender = '男';
```

- 验证：`EXPLAIN` 的 `key_len` 为三列总长度，索引完全生效。

#### 3.2.5 规则 5：索引跳跃扫描（MySQL 8.0+ 新特性）

MySQL 8.0.13 版本引入了**索引跳跃扫描（Index Skip Scan）**，打破了传统的最左前缀限制，解决了「跳过最左列，索引失效」的部分场景。

##### 核心逻辑

当最左列的唯一值数量很少、区分度极低时，MySQL 会自动对最左列的每个唯一值，执行一次子查询，拼接结果，从而利用索引，避免全表扫描。

##### 示例

联合索引 `idx_gender_name (gender, name)`，gender 列只有男 / 女两个值，区分度极低：

sql

```
-- MySQL 5.7：索引完全失效，全表扫描
-- MySQL 8.0+：触发索引跳跃扫描，索引生效，type=range
SELECT * FROM user WHERE name = '张三';
```

##### 限制条件

1. 仅 MySQL 8.0.13+ 版本支持；
2. 仅适用于最左列区分度极低的场景；
3. 无法替代最左前缀原则，仅作为补充优化，不能作为索引设计的依据。

### 3.3 匹配规则的核心本质

所有最左前缀规则，最终都回归到联合索引的 B + 树排序结构：

- 只有连续的最左前缀列，才能保证索引的有序性，MySQL 才能利用 B + 树的二分查找快速定位数据；
- 一旦跳过中间列、遇到范围查询，后续列的有序性被打破，MySQL 无法利用索引，只能回表过滤。

## 四、联合索引的列顺序设计黄金原则

联合索引的性能，90% 取决于列顺序的设计，遵循以下 5 个黄金原则，就能设计出高效的联合索引。

### 4.1 原则 1：高频查询列前置

将**查询频率最高的列**，放在联合索引的最左侧，保证最左前缀能覆盖最多的查询场景。

示例：业务中 90% 的查询都会带 `name` 条件，50% 的查询带 `age` 条件，10% 的查询带 `gender` 条件，索引应设计为 `(name, age, gender)`，可覆盖：

- `name`
- `name+age`
- `name+age+gender`
    
    三个查询场景，最大化索引利用率。

### 4.2 原则 2：高区分度列前置

将**区分度最高的列**，放在联合索引的左侧，区分度 = 列的唯一值数量 / 总数据量，区分度越高，索引过滤效果越好，扫描的行数越少。

示例：`name` 列的唯一值有 10 万个，`gender` 列只有 2 个唯一值，`name` 列区分度远高于 `gender`，应放在最左侧。

> 例外情况：如果低区分度列是高频等值查询列，且符合原则 1，依然优先高频列前置。

### 4.3 原则 3：等值查询列前置，范围查询列后置

这是高频优化原则：**所有等值查询列，放在范围查询列的前面**，避免范围查询截断后续列的索引匹配。

反例：索引 `(name, age, gender)`，查询 `name='张三' AND age>20 AND gender='男'`，gender 列索引失效；

正例：调整为 `(name, gender, age)`，查询条件不变，三列索引全部生效。

### 4.4 原则 4：最小化索引列数量

联合索引不是列越多越好，**只将查询、排序、分组需要的列，加入索引**，避免冗余列。

原因：索引列越多，索引 B + 树越大，磁盘占用越高，插入 / 更新 / 删除的维护成本越高，查询时的索引扫描开销也越大。

### 4.5 原则 5：适配业务场景，优先覆盖核心查询

索引设计永远以业务查询场景为核心，不要为了覆盖所有场景，设计超大的联合索引，优先保证核心高频查询的索引效率。

#### 设计案例实操

**业务场景**：电商订单表，核心查询场景如下：

1. 高频：`WHERE user_id = ? AND order_status = ?`
2. 高频：`WHERE user_id = ? AND create_time BETWEEN ? AND ?`
3. 低频：`WHERE order_status = ? AND create_time BETWEEN ? AND ?`

**索引设计**：

- 核心索引：`idx_userid_status_createtime (user_id, order_status, create_time)`
    
    - 覆盖场景 1：`user_id+order_status` 等值查询，索引全生效；
    - 覆盖场景 2：`user_id` 等值查询，`create_time` 范围查询，索引生效到 `create_time`；
    
- 补充索引：`idx_status_createtime (order_status, create_time)`，覆盖低频场景 3。

## 五、联合索引的进阶优化：覆盖索引

覆盖索引是联合索引最核心的优化手段，能彻底避免回表，将查询性能提升一个量级。

### 5.1 联合索引与覆盖索引的关系

覆盖索引的定义：**查询语句中需要的所有字段（SELECT、WHERE、ORDER BY、GROUP BY），都包含在联合索引的列中**，MySQL 无需通过聚簇索引回表查询整行数据，直接从索引中获取所有需要的信息。

联合索引是实现覆盖索引的最佳方式：单值索引很难覆盖多字段查询，而联合索引可以通过合理设计列，将查询需要的所有字段都包含进来，实现覆盖索引。

### 5.2 如何通过联合索引实现覆盖索引

核心方法：**将查询语句中 SELECT 的字段，加入到联合索引的后缀列中**，保证查询的所有字段都在索引里。

#### 反例（需要回表）

sql

```
-- 联合索引：idx_name_age (name, age)
SELECT name, age, phone FROM user WHERE name = '张三' AND age = 20;
```

- 问题：`phone` 字段不在索引中，MySQL 需要通过主键回表查询，多一次磁盘 IO；
- 验证：`EXPLAIN` 的 `Extra` 字段无 `Using index`。

#### 正例（覆盖索引，无需回表）

sql

```
-- 调整联合索引为：idx_name_age_phone (name, age, phone)
SELECT name, age, phone FROM user WHERE name = '张三' AND age = 20;
```

- 优化：查询的所有字段都在索引中，无需回表，直接从索引获取数据；
- 验证：`EXPLAIN` 的 `Extra` 字段出现 `Using index`，表示覆盖索引生效。

### 5.3 实操案例：覆盖索引优化排序

联合索引的有序性，还可以优化 `ORDER BY` 排序，避免文件排序（Using filesort）。

#### 反例（文件排序）

sql

```
-- 联合索引：idx_name (name)
SELECT name, age FROM user WHERE name = '张三' ORDER BY age;
```

- 问题：`age` 列不在索引中，无法利用索引的有序性，需要在内存中排序，`Extra` 出现 `Using filesort`。

#### 正例（索引排序，无文件排序）

sql

```
-- 联合索引：idx_name_age (name, age)
SELECT name, age FROM user WHERE name = '张三' ORDER BY age;
```

- 优化：`name` 等值匹配后，`age` 列在索引中是有序的，直接利用索引排序，无需文件排序；
- 验证：`EXPLAIN` 的 `Extra` 字段无 `Using filesort`，同时出现 `Using index`。

## 六、实操验证：用 EXPLAIN 判断联合索引是否生效

判断联合索引是否生效、用到了哪些列，唯一的方法是使用 `EXPLAIN` 执行计划，重点关注 3 个核心字段。

### 6.1 核心判断字段

表格

|字段|核心作用|生效判断标准|
|:--|:--|:--|
|**type**|查询类型，代表索引的使用效率|优先级：system > const > eq_ref > ref > range > ALL；<br><br>出现 ALL 代表全表扫描，索引完全失效；<br><br>出现 ref/range 代表索引生效|
|**key**|实际使用的索引名称|显示我们创建的联合索引名称，代表索引被选中；<br><br>为 NULL 代表索引未生效|
|**key_len**|索引使用的字节长度|核心：通过字节长度，判断用到了联合索引的哪些列；<br><br>长度越长，用到的索引列越多|
|**Extra**|额外信息|出现 `Using index` 代表覆盖索引生效；<br><br>出现 `Using where` 代表需要回表过滤；<br><br>出现 `Using filesort` 代表无法利用索引排序|

### 6.2 索引生效案例实操

**表结构**：`user` 表，联合索引 `idx_name_age_gender (name varchar(50), age int, gender char(1))`

- name 列：varchar (50)，utf8mb4 字符集，非空，key_len=50*4=200；
- age 列：int，非空，key_len=4；
- gender 列：char (1)，utf8mb4，非空，key_len=4；
- 三列总 key_len=208。

sql

```
-- 案例1：三列全匹配，索引完全生效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age = 20 AND gender = '男';
-- 结果：type=ref，key=idx_name_age_gender，key_len=208，索引全量生效

-- 案例2：匹配前两列，索引部分生效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age = 20;
-- 结果：type=ref，key=idx_name_age_gender，key_len=204，用到name+age列

-- 案例3：覆盖索引生效，无需回表
EXPLAIN SELECT name, age FROM user WHERE name = '张三' AND gender = '男';
-- 结果：type=ref，key=idx_name_age_gender，key_len=200，Extra=Using index，仅name列生效，覆盖索引生效

-- 案例4：IN等值查询，索引全量生效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age IN (20,25) AND gender = '男';
-- 结果：type=range，key=idx_name_age_gender，key_len=208，三列索引全生效
```

### 6.3 索引失效案例实操

sql

```
-- 案例1：跳过最左列，索引完全失效
EXPLAIN SELECT * FROM user WHERE age = 20 AND gender = '男';
-- 结果：type=ALL，key=NULL，全表扫描

-- 案例2：范围查询截断，后续列失效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age > 20 AND gender = '男';
-- 结果：type=range，key=idx_name_age_gender，key_len=204，仅name+age列生效，gender列失效

-- 案例3：索引列使用函数，索引失效
EXPLAIN SELECT * FROM user WHERE LEFT(name,1) = '张' AND age = 20;
-- 结果：type=ALL，key=NULL，全表扫描

-- 案例4：隐式类型转换，索引失效
EXPLAIN SELECT * FROM user WHERE name = 123 AND age = 20;
-- 结果：type=ALL，key=NULL，name列是varchar，用数字查询，隐式转换，索引失效

-- 案例5：左模糊匹配，索引失效
EXPLAIN SELECT * FROM user WHERE name LIKE '%三' AND age = 20;
-- 结果：type=ALL，key=NULL，全表扫描
```

## 七、高频踩坑与避坑指南

### 7.1 坑 1：列顺序设计错误，索引完全失效

- **坑**：把低频、低区分度的列放在最左侧，高频查询列放在后面，导致绝大多数查询无法匹配最左前缀，索引失效。
- **避坑**：严格遵循「高频列前置、高区分度列前置、等值列前置、范围列后置」的设计原则。

### 7.2 坑 2：索引列使用函数 / 运算，匹配中断

- **坑**：在索引列上使用函数、算术运算，导致索引的有序性被破坏，MySQL 无法利用索引匹配，索引失效。
- **避坑**：禁止在索引列上使用任何函数、运算，将运算放在值的一侧。
    
    sql
    
    ```
    -- 错误写法
    SELECT * FROM user WHERE YEAR(create_time) = 2026;
    -- 正确写法
    SELECT * FROM user WHERE create_time BETWEEN '2026-01-01' AND '2026-12-31';
    ```
    

### 7.3 坑 3：隐式类型 / 字符集转换，索引失效

- **坑 1**：varchar 类型的列，用数字类型查询，触发隐式类型转换，索引失效；
- **坑 2**：JOIN 查询时，两个表的关联字段字符集不同，触发隐式转换，索引失效；
- **避坑**：保证查询条件的类型、字符集与索引列完全一致，建表时统一字段类型和字符集。

### 7.4 坑 4：范围查询列前置，后续列索引失效

- **坑**：把范围查询列放在联合索引的中间位置，导致后续列的索引完全失效，索引利用率低。
- **避坑**：所有范围查询列，统一放在联合索引的最后面，避免截断后续列的匹配。

### 7.5 坑 5：使用 OR 连接条件，索引失效

- **坑**：用 `OR` 连接索引列和非索引列，导致整个查询索引失效。
    
    sql
    
    ```
    -- 索引失效，OR后面的gender无索引，只能全表扫描
    SELECT * FROM user WHERE name = '张三' OR gender = '男';
    ```
    
- **避坑**：避免用 `OR` 连接不同列的条件，改用 `UNION ALL` 拆分查询，或给所有 OR 条件的列建联合索引。

### 7.6 坑 6：过度创建联合索引，维护成本爆炸

- **坑**：为每个查询场景都建一个联合索引，导致一张表有十几个甚至几十个索引，插入 / 更新性能急剧下降。
- **避坑**：遵循「最少索引原则」，通过调整列顺序，让一个联合索引覆盖尽可能多的查询场景，避免冗余索引。

## 九、核心要点总结

1. **联合索引本质**：对多列共同创建的 B + 树索引，先按最左列排序，前序列值相同时再按后续列排序，这是最左前缀原则的底层根源。
2. **最左前缀核心规则**：从最左列开始连续匹配，遇到范围查询中断后续列匹配，跳过中间列仅前缀列生效，IN 等值查询不会中断匹配。
3. **索引设计黄金原则**：高频列前置、高区分度列前置、等值列前置范围列后置、最小化列数量、适配业务场景。
4. **进阶优化**：通过联合索引实现覆盖索引，避免回表，同时可优化 ORDER BY 排序，避免文件排序。
5. **生效验证**：通过 EXPLAIN 的 type、key、key_len 字段，判断索引是否生效、用到了哪些列，key_len 是判断索引匹配列数的核心依据。
6. **避坑核心**：避免索引列使用函数 / 隐式转换、范围列前置、OR 连接非索引列、过度创建索引，严格遵循最左前缀原则设计索引。

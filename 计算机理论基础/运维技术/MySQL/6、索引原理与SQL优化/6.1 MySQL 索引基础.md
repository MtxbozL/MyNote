# 一、开篇：索引的核心定位与价值

索引（Index）是 MySQL 中用于**加速数据查询**的核心数据结构，本质是对表中一列或多列数据进行排序后的“目录”，类比书籍的目录——通过目录可快速定位到目标章节，无需逐页翻阅；通过索引可快速定位到表中目标数据，无需全表扫描，大幅提升查询效率。

## 核心认知（必记）

1. 索引的核心作用：**减少磁盘I/O次数**，降低查询耗时（全表扫描需遍历所有数据，索引可直接定位数据位置）；
    
2. 索引的本质：一种**排好序的数据结构**（如B+树、哈希表），存储索引列的值和对应数据的物理地址；
    
3. 索引的代价：索引会占用额外的磁盘空间，且插入、更新、删除数据时，需同步维护索引（索引越多，维护成本越高）；
    
4. 核心原则：**索引不是越多越好**，需结合查询场景合理创建，平衡查询性能和写操作性能。
    

## 索引解决的核心问题

没有索引时，MySQL 查询数据只能通过“全表扫描”：从表的第一条数据开始，逐行遍历，直到找到目标数据，适用于数据量极小的表；当表中数据量达到万级、十万级以上时，全表扫描的效率会急剧下降（遍历 thousands/millions 条数据，磁盘I/O次数暴增）。

索引通过对数据排序、建立“目录”，让查询语句能快速定位到数据所在的物理位置，避免全表扫描，核心解决“大数据量下查询变慢”的问题，是高并发、大数据量场景的必备优化手段。

# 二、索引的核心分类（按不同维度）

MySQL 索引可从多个维度分类，核心分类包括“按数据结构”“按功能用途”“按索引列数量”，掌握分类是理解索引用法的基础。

## 2.1 按数据结构分类（最核心）

MySQL 支持多种索引数据结构，不同存储引擎支持的类型不同，核心常用的是 B+树索引，其他类型仅适用于特殊场景。

### 2.1.1 B+树索引（InnoDB 默认，核心重点）

B+树是一种平衡树结构，所有数据都存储在叶子节点，非叶子节点仅存储索引值，用于引导查询方向，是 InnoDB 存储引擎的默认索引结构，也是生产环境中最常用的索引类型。

#### 核心特点

- 平衡结构：树的高度极低（通常3-4层），即使是千万级数据，也只需3-4次磁盘I/O就能定位到数据；
    
- 叶子节点有序且连续：叶子节点按索引列排序，且通过指针串联，支持范围查询（如 BETWEEN、>、<）；
    
- 非叶子节点不存储数据：仅存储索引值，节省磁盘空间，提升查询效率；
    
- 聚簇索引特殊：InnoDB 中，聚簇索引的叶子节点存储的是整行数据，而非聚簇索引存储的是数据的物理地址。
    

适用场景：绝大多数查询场景（单值查询、范围查询、排序查询），是 MySQL 索引的核心。

### 2.1.2 哈希索引（Memory 引擎默认）

哈希索引基于哈希表实现，通过计算索引列的哈希值，将哈希值与数据物理地址关联，查询时直接通过哈希值定位数据。

#### 核心特点

- 查询速度极快：单值查询时，时间复杂度接近 O(1)，比 B+树索引更快；
    
- 不支持范围查询：哈希值是无序的，无法通过哈希索引实现 BETWEEN、>、< 等范围查询；
    
- 不支持排序：哈希值无序，无法用于 ORDER BY 排序；
    
- 存在哈希冲突：不同索引值可能计算出相同的哈希值，需额外处理冲突，影响性能。
    

适用场景：仅适用于 Memory 引擎，且查询场景为“单值精确匹配”（如 WHERE id = 1），生产环境中极少使用。

### 2.1.3 其他索引结构（了解即可）

- R树索引：主要用于空间数据类型（如 GIS 地理信息），支持空间范围查询，MySQL 中较少使用；
    
- Full-Text 全文索引：用于文本内容的模糊查询（如文章内容、评论），支持关键词匹配，替代 LIKE %xxx% 的低效查询；
    
- T树索引：主要用于 MyISAM 引擎的某些场景，性能不如 B+树，已逐渐被淘汰。
    

## 2.2 按功能用途分类（实操常用）

按索引的功能的用途，可分为聚簇索引、非聚簇索引、唯一索引、普通索引等，是日常建表、优化查询时最常接触的分类。

|索引类型|核心说明|特点|适用场景|
|---|---|---|---|
|聚簇索引（Clustered Index）|InnoDB 核心索引，索引与数据存储在一起，叶子节点存储整行数据|一张表只能有1个，查询效率最高，主键默认是聚簇索引|主键查询、高频单值查询|
|非聚簇索引（Secondary Index）|索引与数据分离，叶子节点存储聚簇索引的值（而非物理地址）|一张表可有多个，查询需“回表”（通过聚簇索引找数据）|非主键的高频查询、联合查询|
|唯一索引（Unique Index）|索引列的值唯一（允许 NULL，但只能有1个 NULL）|可用于主键（主键是特殊的唯一索引），防止数据重复|唯一标识列（如身份证号、手机号）|
|普通索引（Normal Index）|最基础的索引，无任何约束，仅用于加速查询|无唯一性要求，维护成本低|高频查询的非唯一列（如姓名、性别）|
|全文索引（Full-Text Index）|用于文本内容的关键词匹配，支持模糊查询|不支持 %xxx% 语法，需用 MATCH() AGAINST() 函数|文章内容、评论、日志等文本查询|

### 关键补充：聚簇索引 vs 非聚簇索引（面试高频）

InnoDB 中，聚簇索引和非聚簇索引的区别是核心考点，重点掌握“回表”概念：

- 聚簇索引：叶子节点存储整行数据，查询时直接获取数据，无需回表，效率最高；
    
- 非聚簇索引：叶子节点存储的是“聚簇索引的值”（如主键值），查询时需先通过非聚簇索引找到主键值，再通过聚簇索引找到整行数据，这个过程称为“回表”，比聚簇索引多一次磁盘I/O。
    

示例：表 user（id 主键，name 普通索引），执行 SELECT * FROM user WHERE name = '张三'：

1. 通过 name 非聚簇索引，找到对应的主键 id（如 100）；
    
2. 通过 id 聚簇索引，找到 id=100 的整行数据（回表）；
    
3. 返回查询结果。
    

## 2.3 按索引列数量分类

- **单值索引**：仅对单一列创建的索引（如 id、name），最常用，适用于单条件查询（WHERE name = '张三'）；
    
- **联合索引（复合索引）**：对多列组合创建的索引（如 (name, age)），适用于多条件查询（WHERE name = '张三' AND age = 20）；
    
- **覆盖索引**：特殊的联合索引，索引列包含查询语句中所有需要的字段（如查询 SELECT name, age FROM user WHERE name = '张三'，创建 (name, age) 联合索引，无需回表，直接从索引中获取数据）。
    

关键：联合索引遵循“最左前缀原则”（后续详细讲解），这是联合索引使用的核心。

# 三、索引的底层实现（核心重点：B+树）

生产环境中，99% 的 MySQL 索引都是 B+树索引（InnoDB 默认），因此重点掌握 B+树的结构和工作原理，是理解索引查询效率的关键。

## 3.1 B+树的结构（简化理解）

B+树是一种“平衡多路查找树”，结构分为3层，每层的作用不同，核心是“叶子节点存储数据，非叶子节点引导查询”：

1. **根节点**：最顶层节点，存储索引值的范围，用于引导查询方向（如索引值 1-100 指向左子节点，101-200 指向右子节点）；
    
2. **非叶子节点**：中间层节点，仅存储索引值和子节点指针，不存储数据，继续引导查询到叶子节点；
    
3. **叶子节点**：最底层节点，存储索引值和对应的数据（聚簇索引存储整行数据，非聚簇索引存储聚簇索引值），且叶子节点之间通过指针串联，形成有序链表，支持范围查询。
    

## 3.2 B+树索引的查询流程（以聚簇索引为例）

假设表 user 有 1000 万条数据，主键 id 是聚簇索引（B+树高度为3），执行查询 SELECT * FROM user WHERE id = 123456：

1. 查询请求先到达 B+树的根节点，根节点判断 123456 属于哪个范围，指向对应的非叶子节点；
    
2. 非叶子节点再次判断 123456 的范围，指向对应的叶子节点；
    
3. 叶子节点找到 id=123456 的数据，直接返回（无需回表）；
    
4. 整个过程仅需 3 次磁盘I/O，而全表扫描可能需要数万次磁盘I/O，效率差距极大。
    

## 3.3 B+树索引的优势（为什么 InnoDB 选择 B+树）

- 平衡结构，高度低：无论数据量多大，查询次数都稳定（3-4次），性能稳定；
    
- 叶子节点有序连续：支持范围查询、排序查询，适配绝大多数业务场景；
    
- 非叶子节点不存储数据：节省磁盘空间，可存储更多索引值，减少磁盘I/O；
    
- 支持聚簇索引：将索引与数据结合，减少回表次数，提升查询效率。
    

# 四、索引的实操语法（建表、增删改查）

掌握索引的实操语法，是日常开发和优化的基础，重点掌握普通索引、唯一索引、联合索引的创建、删除和查询。

## 4.1 索引的创建（3种方式）

### 方式1：建表时创建索引

```sql
-- 1. 普通索引
CREATE TABLE user (
    id INT PRIMARY KEY, -- 主键，默认创建聚簇索引
    name VARCHAR(50),
    age INT,
    INDEX idx_name (name) -- 普通索引（单值）
);

-- 2. 唯一索引
CREATE TABLE user (
    id INT PRIMARY KEY,
    phone VARCHAR(11) UNIQUE, -- 唯一索引（单列）
    name VARCHAR(50),
    UNIQUE INDEX idx_phone (phone) -- 等价写法
);

-- 3. 联合索引
CREATE TABLE user (
    id INT PRIMARY KEY,
    name VARCHAR(50),
    age INT,
    INDEX idx_name_age (name, age) -- 联合索引（name在前，age在后）
);

-- 4. 全文索引
CREATE TABLE article (
    id INT PRIMARY KEY,
    title VARCHAR(100),
    content TEXT,
    FULLTEXT INDEX idx_content (content) -- 全文索引
);
```

### 方式2：建表后添加索引（ALTER TABLE）

```sql
-- 普通索引
ALTER TABLE user ADD INDEX idx_name (name);

-- 唯一索引
ALTER TABLE user ADD UNIQUE INDEX idx_phone (phone);

-- 联合索引
ALTER TABLE user ADD INDEX idx_name_age (name, age);

-- 全文索引
ALTER TABLE article ADD FULLTEXT INDEX idx_content (content);
```

### 方式3：建表后添加索引（CREATE INDEX）

```sql
-- 普通索引
CREATE INDEX idx_name ON user (name);

-- 唯一索引
CREATE UNIQUE INDEX idx_phone ON user (phone);

-- 联合索引
CREATE INDEX idx_name_age ON user (name, age);

-- 全文索引
CREATE FULLTEXT INDEX idx_content ON article (content);
```

## 4.2 索引的删除

```sql
-- 方式1：ALTER TABLE 删除
ALTER TABLE user DROP INDEX idx_name;

-- 方式2：DROP INDEX 删除
DROP INDEX idx_phone ON user;

-- 注意：主键索引不能直接删除（需先删除主键约束）
ALTER TABLE user DROP PRIMARY KEY;
```

## 4.3 索引的查询（查看表中所有索引）

```sql
-- 方式1：查看表的索引详情（推荐）
SHOW INDEX FROM user;

-- 方式2：查看表的创建语句，包含索引信息
SHOW CREATE TABLE user;
```

## 4.4 全文索引的查询（特殊语法）

全文索引不能用 LIKE %xxx% 查询，需使用 MATCH() AGAINST() 函数：

```sql
-- 全文索引查询（匹配 content 中包含“MySQL”的记录）
SELECT * FROM article WHERE MATCH(content) AGAINST('MySQL');
```

# 五、索引的核心使用规则（避坑重点）

创建索引后，若使用不当，索引会失效（查询仍会全表扫描），因此必须掌握索引的使用规则，避免踩坑。

## 5.1 联合索引的“最左前缀原则”（核心）

联合索引（如 (a, b, c)）的查询，必须从最左边的列开始匹配，否则索引失效，具体规则：

- 有效查询（触发索引）：WHERE a = 1、WHERE a = 1 AND b = 2、WHERE a = 1 AND b = 2 AND c = 3；
    
- 无效查询（索引失效）：WHERE b = 2、WHERE c = 3、WHERE b = 2 AND c = 3（未匹配最左列 a）；
    
- 部分有效：WHERE a = 1 AND c = 3（仅 a 列触发索引，c 列不触发，仍比全表扫描快）。
    

注意：联合索引的列顺序很重要，需将**查询频率最高、区分度最高**的列放在最左边。

## 5.2 导致索引失效的常见场景（必记）

1. **索引列使用函数或运算**：如 WHERE SUBSTR(name, 1, 3) = '张'、WHERE id + 1 = 100，索引会失效（无法使用索引排序的特性）；
    
2. **索引列使用模糊查询（左模糊）**：如 WHERE name LIKE '%张三'，索引失效；右模糊（WHERE name LIKE '张三%'）索引有效；
    
3. **索引列使用 NULL/IS NULL/IS NOT NULL**：多数情况下索引失效（InnoDB 对 NULL 值的处理特殊，无法有效排序）；
    
4. **查询条件使用 OR**：若 OR 两边的列都没有索引，索引失效；若两边都有索引，索引有效；
    
5. **数据量极小的表**：索引反而会增加查询开销（索引查询需先查索引，再查数据，比全表扫描慢）；
    
6. **索引列区分度极低**：如性别（男/女），创建索引无意义（索引过滤效果差，不如全表扫描）。
    

## 5.3 索引的设计原则（实操重点）

1. **优先给高频查询列创建索引**：如 WHERE、JOIN、ORDER BY 中常用的列；
    
2. **避免过度创建索引**：索引越多，插入、更新、删除的维护成本越高（每次写操作需同步更新所有索引）；
    
3. **联合索引优先于多个单值索引**：如查询 WHERE a = 1 AND b = 2，创建 (a, b) 联合索引，比创建两个单值索引更高效；
    
4. **选择区分度高的列创建索引**：区分度 = 不同值的数量 / 总数据量，区分度越高，索引过滤效果越好（如身份证号区分度100%，性别区分度极低）；
    
5. **避免给大字段创建索引**：如 TEXT、BLOB 类型，索引会占用大量磁盘空间，且查询效率低；
    
6. **利用覆盖索引减少回表**：查询语句中需要的字段，尽量包含在索引列中，避免回表（如查询 SELECT name, age FROM user，创建 (name, age) 联合索引）。
    

# 六、索引的性能分析（实操工具）

创建索引后，需验证索引是否生效、查询是否使用了索引，核心使用 EXPLAIN 命令分析查询计划。

## 6.1 EXPLAIN 命令使用方法

```sql
-- 在查询语句前加 EXPLAIN，查看查询计划
EXPLAIN SELECT * FROM user WHERE name = '张三';
```

## 6.2 核心字段解读（判断索引是否生效）

重点关注 EXPLAIN 结果中的 3 个字段，判断索引是否生效：

1. **type**：查询类型，用于判断查询效率，从好到差依次是：system > const > eq_ref > ref > range > ALL；
    
    1. ALL：全表扫描，索引失效；
        
    2. range：范围查询，索引有效（如 BETWEEN、>）；
        
    3. ref：单值查询，索引有效（如 WHERE name = '张三'）；
        
    4. const：主键/唯一索引查询，效率最高（如 WHERE id = 1）。
        
2. **key**：显示查询使用的索引名称，若为 NULL，说明未使用索引（索引失效）；
    
3. **rows**：显示查询扫描的行数，行数越少，查询效率越高（索引生效时，行数会大幅减少）。
    

## 6.3 实操示例：判断索引是否生效

```sql
-- 示例1：索引生效（type=ref，key=idx_name）
EXPLAIN SELECT * FROM user WHERE name = '张三';

-- 示例2：索引失效（type=ALL，key=NULL），因为使用左模糊
EXPLAIN SELECT * FROM user WHERE name LIKE '%张三';

-- 示例3：联合索引生效（type=ref，key=idx_name_age）
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age = 20;
```

# 八、核心要点总结

1. 索引的核心是“排好序的目录”，作用是减少磁盘I/O，加速查询，代价是占用磁盘空间、增加写操作维护成本；
    
2. InnoDB 默认使用 B+树索引，核心优势是平衡结构、支持范围查询、查询性能稳定，是生产环境的核心索引类型；
    
3. 核心索引分类：聚簇索引（1个/表，主键默认）、非聚簇索引（多个/表，需回表）、联合索引（遵循最左前缀原则）；
    
4. 索引失效是高频坑，需避免索引列使用函数、左模糊、OR 等场景，通过 EXPLAIN 验证索引是否生效；
    
5. 索引设计的核心：高频查询列、区分度高、联合索引优先、避免过度创建，平衡查询和写操作性能；
    
6. 核心关联：索引与 InnoDB 锁机制、MVCC 密切相关（如行锁基于索引实现），后续学习需重点关联。

索引类型是 MySQL 索引学习的核心基础，不同类型的索引适配不同的查询场景、底层实现不同、性能表现也存在差异。本笔记将聚焦 MySQL 核心索引类型，按“数据结构”“功能用途”“索引列数量”三大核心维度分类讲解，结合实操场景和面试重点，帮你彻底分清各类索引的特点、用法及区别，规避使用误区。

# 一、索引类型的核心分类维度（总览）

MySQL 索引的分类维度有多种，其中最常用、最核心的三大维度的是：**按数据结构分类**、**按功能用途分类**、**按索引列数量分类**。

核心原则：没有“最好”的索引类型，只有“最适配”的场景——需结合查询方式、数据量、业务需求，选择对应的索引类型，平衡查询性能和写操作（插入/更新/删除）维护成本。

# 二、按数据结构分类（最核心，决定底层性能）

索引的本质是“排好序的数据结构”，不同的数据结构决定了索引的查询效率、适用场景，MySQL 支持多种索引数据结构，其中 InnoDB 存储引擎默认使用 B+树索引，也是生产环境中最常用的类型。

## 2.1 B+树索引（InnoDB 默认，核心重点）

B+树是一种“平衡多路查找树”，是 MySQL 索引的核心，几乎适配所有日常查询场景，也是面试考查的重点。

### 核心结构（简化理解）

- 分层存储：分为根节点、非叶子节点、叶子节点三层，树高通常为3-4层（即使千万级数据，也能快速定位）；
    
- 非叶子节点：仅存储索引值和子节点指针，不存储实际数据，作用是“引导查询方向”；
    
- 叶子节点：存储索引值和对应数据（聚簇索引存储整行数据，非聚簇索引存储聚簇索引值），且叶子节点通过指针串联，形成有序链表。
    

### 核心特点（必记）

1. 平衡结构，性能稳定：树高极低，查询时磁盘I/O次数固定（3-4次），不受数据量影响；
    
2. 支持范围查询：叶子节点有序连续，可快速定位范围区间（如 BETWEEN、>、<、ORDER BY）；
    
3. 支持聚簇存储：InnoDB 中可将索引与数据结合（聚簇索引），减少回表，提升查询效率；
    
4. 非叶子节点不存数据：节省磁盘空间，单个节点可存储更多索引值，进一步减少磁盘I/O。
    

### 适用场景

绝大多数日常查询场景：单值查询（WHERE id = 1）、范围查询（WHERE age BETWEEN 20 AND 30）、排序查询（ORDER BY name）、联合查询（WHERE name = '张三' AND age = 20），是生产环境的首选索引类型。

## 2.2 哈希索引（Memory 引擎默认，了解即可）

哈希索引基于哈希表实现，核心是通过计算索引列的哈希值，将哈希值与数据的物理地址关联，查询时直接通过哈希值定位数据。

### 核心特点

1. 单值查询极快：时间复杂度接近 O(1)，比 B+树索引的单值查询更快；
    
2. 不支持范围查询：哈希值是无序的，无法通过哈希索引实现 BETWEEN、>、< 等范围操作；
    
3. 不支持排序：哈希值无序，无法用于 ORDER BY 排序，也无法用于 GROUP BY 分组；
    
4. 存在哈希冲突：不同的索引值可能计算出相同的哈希值，需额外处理冲突（如链表法），冲突过多会影响性能；
    
5. 仅支持精确匹配：无法用于模糊查询（如 LIKE）。
    

### 适用场景

仅适用于 Memory 存储引擎（InnoDB 不支持原生哈希索引），且查询场景为“单值精确匹配”（如 WHERE id = 100），生产环境中极少使用（Memory 引擎数据存于内存，重启丢失）。

## 2.3 其他索引结构（了解即可，无需深入）

- R树索引：主要用于空间数据类型（如 GIS 地理信息、经纬度），支持空间范围查询，MySQL 中使用极少；
    
- Full-Text 全文索引：专门用于文本内容的关键词匹配（如文章内容、评论），替代 LIKE %xxx% 的低效模糊查询，需使用 MATCH() AGAINST() 语法；
    
- T树索引：主要用于 MyISAM 引擎的部分场景，性能不如 B+树，目前已逐渐被淘汰。
    

# 三、按功能用途分类（实操常用，建表必选）

按功能用途分类是日常建表、优化查询时最常接触的分类，核心是根据“索引的功能的约束”划分，不同类型的索引有明确的适用场景，需重点掌握。

## 3.1 聚簇索引（Clustered Index）—— InnoDB 核心

聚簇索引是 InnoDB 存储引擎的核心索引，也是 MySQL 性能优化的关键，本质是“索引与数据存储在一起”，叶子节点直接存储整行数据。

### 核心特点（必记）

1. 一张表只能有1个聚簇索引：InnoDB 会默认将主键作为聚簇索引；若表没有主键，会选择唯一非空列作为聚簇索引；若既没有主键，也没有唯一非空列，InnoDB 会自动生成一个隐藏的聚簇索引；
    
2. 查询效率最高：叶子节点存储整行数据，查询时无需回表，直接从索引中获取所有数据；
    
3. 数据物理有序：表中数据的物理存储顺序，与聚簇索引的排序顺序一致，进一步提升查询效率。
    

### 适用场景

主键查询、高频单值查询（如 WHERE id = 100），是所有查询的“基础索引”——非聚簇索引查询最终都需依赖聚簇索引回表。

## 3.2 非聚簇索引（Secondary Index）—— 辅助查询

非聚簇索引又称“辅助索引”，与聚簇索引相对，本质是“索引与数据分离”，叶子节点不存储整行数据，仅存储聚簇索引的值（如主键值）。

### 核心特点（必记）

1. 一张表可以有多个非聚簇索引：日常建表中创建的普通索引、唯一索引、联合索引，本质都是非聚簇索引；
    
2. 查询需“回表”：通过非聚簇索引查询时，先找到聚簇索引的值（如主键），再通过聚簇索引找到整行数据，这个过程称为“回表”，比聚簇索引多一次磁盘I/O；
    
3. 维护成本低于聚簇索引：插入、更新、删除数据时，仅需维护非聚簇索引的排序，无需调整数据的物理存储。
    

### 适用场景

非主键的高频查询、多条件联合查询（如 WHERE name = '张三'），用于辅助聚簇索引，提升查询效率。

### 关键示例（理解回表）

表 user（id 主键，聚簇索引；name 普通索引，非聚簇索引），执行查询 SELECT * FROM user WHERE name = '张三'：

1. 通过 name 非聚簇索引，找到对应的主键 id（如 100）；
    
2. 通过 id 聚簇索引，找到 id=100 的整行数据（回表操作）；
    
3. 返回查询结果。
    

## 3.3 唯一索引（Unique Index）—— 保证数据唯一性

唯一索引的核心作用是“保证索引列的值唯一”，可用于防止数据重复，同时兼具加速查询的功能。

### 核心特点（必记）

1. 索引列的值唯一：不允许重复值，但允许 NULL 值（且只能有1个 NULL 值）；
    
2. 主键是特殊的唯一索引：主键索引本质是“唯一非空索引”，既保证唯一，又不允许 NULL；
    
3. 查询效率接近聚簇索引：唯一索引的查询效率仅次于聚簇索引，适合作为高频唯一字段的索引。
    

### 适用场景

唯一标识列的查询和去重，如手机号、身份证号、邮箱（如 WHERE phone = '13800138000'），既保证数据不重复，又加速查询。

## 3.4 普通索引（Normal Index）—— 基础加速查询

普通索引是最基础、最常用的索引类型，无任何约束条件，仅用于加速查询，维护成本最低。

### 核心特点

1. 无任何约束：索引列的值可以重复，也可以为 NULL；
    
2. 维护成本最低：插入、更新、删除数据时，维护普通索引的开销最小；
    
3. 仅用于加速查询：无去重、无排序约束，核心作用就是减少磁盘I/O，提升查询效率。
    

### 适用场景

高频查询的非唯一列，如姓名、性别、地址（如 WHERE name = '张三'），适合作为辅助查询的基础索引。

## 3.5 全文索引（Full-Text Index）—— 文本查询专用

全文索引是专门用于文本内容关键词匹配的索引，替代 LIKE %xxx% 的低效模糊查询，适用于长文本查询场景。

### 核心特点

1. 支持关键词匹配：可匹配文本中的关键词，而非精确匹配或前缀匹配；
    
2. 不支持 LIKE 语法：需使用 MATCH(索引列) AGAINST(关键词) 语法查询；
    
3. 适用于长文本：如文章内容、评论、日志等 TEXT、VARCHAR 类型的长文本字段。
    

### 适用场景

文本内容的模糊查询，如查询文章中包含“MySQL 索引”的记录（SELECT * FROM article WHERE MATCH(content) AGAINST('MySQL 索引')）。

## 按功能用途分类总结（表格对比，一目了然）

|   |   |   |   |
|---|---|---|---|
|**索引类型**|**核心约束**|**核心优势**|**适用场景**|
|聚簇索引|1个/表，默认主键|查询效率最高，无需回表|主键、高频单值查询|
|非聚簇索引|多个/表，无约束|维护成本低，辅助查询|非主键高频查询、联合查询|
|唯一索引|索引列唯一，允许1个NULL|去重+加速查询|手机号、身份证号等唯一列|
|普通索引|无约束，可重复、可NULL|维护成本最低，基础加速|姓名、性别等非唯一高频列|
|全文索引|仅适用于文本字段|文本关键词查询高效|文章、评论等长文本查询|

# 四、按索引列数量分类（实操进阶）

按索引列数量分类，核心是根据“索引包含的列数”划分，主要分为单值索引、联合索引、覆盖索引，其中联合索引是面试和实操的重点。

## 4.1 单值索引（Single-Column Index）

单值索引是仅对“单一列”创建的索引，是最基础的索引形式，对应前面的普通索引、唯一索引（单列）。

### 核心特点

- 仅关联1列数据：如 idx_name（name）、idx_phone（phone）；
    
- 适配单条件查询：仅能加速“单个列”的查询（如 WHERE name = '张三'）；
    
- 创建简单，维护成本低：无需考虑列顺序，适合简单查询场景。
    

## 4.2 联合索引（Composite Index）—— 重点

联合索引又称“复合索引”，是对“多列组合”创建的索引（如 (name, age, gender)），核心用于多条件联合查询，是优化多条件查询的关键。

### 核心特点（必记）

1. 多列组合排序：索引按“最左边的列”优先排序，再按后续列排序（如 (name, age) 先按 name 排序，name 相同再按 age 排序）；
    
2. 遵循“最左前缀原则”：查询时必须从最左边的列开始匹配，否则索引失效（核心考点）；
    
3. 替代多个单值索引：1个联合索引可适配多个查询场景（如 (name, age) 可适配 WHERE name = '张三'、WHERE name = '张三' AND age = 20），比创建多个单值索引更高效、维护成本更低。
    

### 最左前缀原则（示例说明）

联合索引 idx_name_age_gender（name, age, gender），查询场景适配情况：

- 有效查询（索引生效）：
    
    - WHERE name = '张三'（匹配最左列 name）；
        
    - WHERE name = '张三' AND age = 20（匹配 name + age，连续最左前缀）；
        
    - WHERE name = '张三' AND age = 20 AND gender = '男'（匹配全部列）。
        
- 无效查询（索引失效）：
    
    - WHERE age = 20（未匹配最左列 name）；
        
    - WHERE gender = '男'（未匹配最左列）；
        
    - WHERE age = 20 AND gender = '男'（未匹配最左列 name）。
        
- 部分有效：WHERE name = '张三' AND gender = '男'（仅 name 列触发索引，gender 列不触发，仍比全表扫描快）。
    

### 适用场景

多条件联合查询场景，如 WHERE name = '张三' AND age = 20、WHERE name = '张三' AND age BETWEEN 20 AND 30，需将“查询频率最高、区分度最高”的列放在最左边。

## 4.3 覆盖索引（Covering Index）—— 优化回表

覆盖索引是一种“特殊的联合索引”，核心特点是“索引列包含查询语句中所有需要的字段”，无需回表，查询效率极高。

### 核心特点

1. 索引列覆盖查询字段：查询语句中 SELECT 的字段，全部包含在索引列中；
    
2. 无需回表：查询时直接从索引中获取所有需要的数据，无需通过聚簇索引回表，减少磁盘I/O；
    
3. 本质是联合索引：需根据查询字段设计，如查询 SELECT name, age FROM user WHERE name = '张三'，创建 (name, age) 联合索引，即为覆盖索引。
    

### 适用场景

高频的多字段查询场景，如查询用户姓名和年龄（SELECT name, age FROM user WHERE name = '张三'），通过覆盖索引可大幅提升查询效率。

# 五、核心区别与面试高频对比（必记）

索引类型的区别是 MySQL 面试高频考点，重点掌握以下3组核心对比，避免混淆。

## 5.1 聚簇索引 vs 非聚簇索引（核心对比）

- 存储方式：聚簇索引“索引+数据”存储在一起，非聚簇索引“索引与数据分离”；
    
- 数量限制：聚簇索引1个/表，非聚簇索引多个/表；
    
- 查询效率：聚簇索引无需回表，效率更高；非聚簇索引需回表，效率略低；
    
- 数据顺序：聚簇索引数据物理有序，非聚簇索引数据物理无序。
    

## 5.2 唯一索引 vs 普通索引

- 约束不同：唯一索引保证索引列值唯一，普通索引无约束；
    
- NULL 值：唯一索引允许1个 NULL，普通索引允许多个 NULL；
    
- 查询效率：唯一索引略高于普通索引（MySQL 可快速定位唯一值）；
    
- 维护成本：唯一索引维护成本略高（需判断值是否唯一）。
    

## 5.3 B+树索引 vs 哈希索引

- 数据结构：B+树是平衡树，哈希索引是哈希表；
    
- 查询类型：B+树支持单值、范围、排序查询；哈希索引仅支持单值精确查询；
    
- 查询效率：单值查询哈希索引更快，范围查询 B+树更高效；
    
- 适用引擎：B+树支持 InnoDB、MyISAM；哈希索引仅支持 Memory。
    

# 六、索引类型选择原则（实操重点）

1. 优先选择 B+树索引：无论什么场景，B+树索引都是首选，适配绝大多数查询需求；
    
2. 主键必用聚簇索引：InnoDB 会默认将主键作为聚簇索引，无需手动创建；
    
3. 唯一字段用唯一索引：如手机号、身份证号，既保证数据唯一，又加速查询；
    
4. 多条件查询用联合索引：按“查询频率+区分度”排序列顺序，遵循最左前缀原则；
    
5. 文本查询用全文索引：避免使用 LIKE %xxx%，用全文索引提升效率；
    
6. 避免过度创建索引：索引越多，写操作维护成本越高，按需创建即可。

# 八、核心要点总结

1. 索引类型的三大核心分类：按数据结构（B+树、哈希等）、按功能用途（聚簇、非聚簇等）、按索引列数量（单值、联合等）；
    
2. 核心重点：B+树索引是生产环境首选，聚簇索引是 InnoDB 核心，联合索引遵循最左前缀原则，覆盖索引可优化回表；
    
3. 关键区别：聚簇与非聚簇的核心是“是否存储数据”，唯一与普通的核心是“是否有唯一约束”；
    
4. 选择原则：结合查询场景、数据约束，平衡查询性能和写操作维护成本，不盲目创建索引；
    
5. 关联学习：索引类型与底层实现、查询优化、锁机制密切相关，后续需结合 EXPLAIN 工具，验证索引是否生效、是否选对类型。

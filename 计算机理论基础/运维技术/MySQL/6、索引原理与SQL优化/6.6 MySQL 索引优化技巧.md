## 一、开篇：索引优化的核心定位与底层逻辑

### 1. 什么是索引优化

索引优化是 MySQL SQL 优化中**成本最低、见效最快、覆盖范围最广**的核心手段，本质是：

> 基于 MySQL 索引的 B + 树 底层特性，通过合理的索引设计、SQL 语句适配、索引运维，最大化利用索引的有序性，**减少磁盘 IO 次数、避免全表扫描、减少回表操作、消除文件排序 / 临时表**，同时最小化索引的维护成本与磁盘开销。

### 2. 索引优化的两大核心目标

1. **性能最大化**：让所有高频查询都能命中合适的索引，彻底避免全表扫描、索引失效、回表次数过多、文件排序、临时表等性能杀手；
2. **成本最小化**：用最少的索引数量，支撑最多的业务查询场景，平衡查询性能与写操作（插入 / 更新 / 删除）的维护开销，避免冗余、无效索引。

### 3. 前置核心认知（优化的底层逻辑）

所有优化技巧都围绕 B + 树索引的核心特性展开，必须先牢记 3 个底层逻辑：

1. **索引的核心优势是有序性**：B + 树的叶子节点是有序排列的双向链表，所有优化都要围绕「保护索引的有序性」展开，避免破坏有序性导致索引失效；
2. **性能的核心瓶颈是磁盘 IO**：MySQL 数据和索引都存储在磁盘，一次磁盘 IO 的耗时是内存操作的百万倍，优化的核心是**减少磁盘 IO 次数**；
3. **回表是高频性能损耗点**：非聚簇索引查询需要通过主键回表获取完整数据，优化的终极目标之一是**通过覆盖索引彻底避免回表**。

### 4. 索引优化的黄金四原则

- **等值优先**：等值查询列优先放在联合索引的最左侧；
- **高频优先**：查询频率越高的列，越优先放在联合索引左侧；
- **区分度优先**：列的区分度越高，越优先放在联合索引左侧；
- **覆盖优先**：优先设计覆盖索引，避免回表操作。

## 二、第一部分：索引设计阶段优化（事前优化，根本之策）

索引优化的核心在「设计」，90% 的索引性能问题，都是设计阶段埋下的隐患。这部分是索引优化的根基，决定了后续优化的上限。

### 2.1 主键索引设计优化（聚簇索引，重中之重）

InnoDB 中，主键索引就是聚簇索引，叶子节点存储整行数据，所有二级索引的叶子节点都存储主键值，主键设计直接影响整张表的查询和写入性能。

|优化规则|核心说明|错误示例|正确示例|
|:--|:--|:--|:--|
|**必须用有序递增的数值类型**|自增主键是顺序写入，不会导致 B + 树页分裂，写入性能拉满；随机值会导致频繁页分裂、索引碎片|UUID、随机字符串、长字符串做主键|自增 INT / BIGINT 类型做主键|
|**禁止用过长的字段做主键**|二级索引的叶子节点都会存储主键值，主键过长会导致所有二级索引膨胀，磁盘占用翻倍|VARCHAR (50) 字符串做主键|BIGINT UNSIGNED 做主键|
|**必须保证主键非空唯一**|InnoDB 会给无主键的表自动生成隐藏的 6 字节行 ID，无法控制，且无法用于查询|无主键的表|显式设置 PRIMARY KEY 主键约束|
|**禁止频繁更新主键**|主键更新会导致聚簇索引重构、所有二级索引同步更新，开销极大|用业务可修改字段（如订单号）做主键|用与业务无关的自增 ID 做主键，业务唯一字段建唯一索引|

> 核心结论：**99% 的场景下，自增 BIGINT UNSIGNED 是主键的最优选择**。

### 2.2 联合索引列顺序设计优化（最核心的设计技巧）

联合索引是生产环境中使用最多的索引类型，列顺序直接决定了索引的利用率和查询性能，必须严格遵循以下优先级排序规则：

#### 排序优先级（从高到低）

1. **等值查询列 > 范围查询列**：等值查询列必须全部前置，范围查询列统一放在联合索引的最后。
    
    - 原因：范围查询会截断后续列的索引匹配，放在最后不会影响其他等值列的索引生效；
    - 反例：`idx_age_name (age, name)`，查询 `WHERE age>20 AND name='张三'`，仅 age 列生效；
    - 正例：`idx_name_age (name, age)`，查询 `WHERE name='张三' AND age>20`，两列索引全生效。
    
2. **高频查询列 > 低频查询列**：查询频率越高的列，越靠左，让一个联合索引覆盖更多的查询场景。
    
    - 示例：90% 的查询带`name`条件，50% 带`age`，10% 带`gender`，索引应设计为 `idx_name_age_gender (name, age, gender)`，可覆盖 `name`、`name+age`、`name+age+gender` 三个场景。
    
3. **高区分度列 > 低区分度列**：区分度 = 列的唯一值数量 / 表总行数，越接近 1，区分度越高，过滤效果越好。
    
    - 高区分度列：手机号、身份证号、用户 ID、订单号（区分度≈1）；
    - 低区分度列：性别、状态、是否删除（区分度≈0.001）；
    - 规则：高区分度列优先前置，低区分度列**禁止单独建索引**，仅可作为联合索引的后缀列。
    - 示例：`WHERE gender='男' AND name LIKE '张%'`，索引应设计为 `idx_name_gender (name, gender)`，而非 `idx_gender_name`。
    

#### 反例避坑

- 错误：把范围列、低频列、低区分度列放在联合索引左侧，导致索引利用率极低；
- 错误：为每个查询场景单独建联合索引，导致索引数量爆炸，写入性能骤降。

### 2.3 覆盖索引设计优化（性能提升的终极杀器）

覆盖索引是指：**查询语句中需要的所有字段（SELECT、WHERE、ORDER BY、GROUP BY），都包含在联合索引的列中**，MySQL 无需通过主键回表查询整行数据，直接从索引中获取所有需要的信息。

#### 覆盖索引的核心优势

1. 彻底避免回表操作，减少 50% 以上的磁盘 IO，查询性能提升数倍；
2. 索引文件远小于数据文件，可全部加载到内存，减少磁盘访问；
3. 避免随机 IO，索引是顺序存储的，范围查询效率更高；
4. 可利用索引的有序性，避免文件排序（Using filesort）和临时表（Using temporary）。

#### 设计技巧

1. **后缀追加原则**：在联合索引的等值查询列后，追加 SELECT 需要的字段，无需回表。
    
    - 示例：查询 `SELECT name, age FROM user WHERE name='张三'`，索引 `idx_name (name)` 需要回表，优化为 `idx_name_age (name, age)`，覆盖索引生效，Extra 出现`Using index`。
    
2. **避免过度追加**：不要为了覆盖索引，把所有字段都塞进索引，导致索引膨胀，维护成本升高，仅追加查询需要的字段。
3. **大字段禁止加入覆盖索引**：TEXT、BLOB、VARCHAR (255+) 等大字段，禁止加入覆盖索引，会导致索引文件急剧膨胀。

### 2.4 前缀索引设计优化（大字符串字段专属）

对于 VARCHAR (255)、TEXT 等长字符串字段，建全量索引会占用大量磁盘空间，查询效率低，此时应使用**前缀索引**：仅索引字段的前 N 个字符，平衡索引占用空间和区分度。

#### 设计步骤

1. **计算最优前缀长度**：找到最小的 N，让前缀的区分度接近全字段的区分度。

```sql
    -- 1. 全字段区分度
    SELECT COUNT(DISTINCT name)/COUNT(*) FROM user;
    -- 2. 前3个字符的区分度
    SELECT COUNT(DISTINCT LEFT(name,3))/COUNT(*) FROM user;
    -- 3. 前5个字符的区分度
    SELECT COUNT(DISTINCT LEFT(name,5))/COUNT(*) FROM user;
```
    
- 选择区分度接近全字段的最小 N，比如前 5 个字符区分度已达到 0.99，就无需用前 10 个。
    
2. **创建前缀索引**

```sql
    -- 给name字段创建前5个字符的前缀索引
    ALTER TABLE user ADD INDEX idx_name_prefix (name(5));
```

#### 适用场景与限制

- 适用场景：长字符串字段，且仅用于前缀等值 / 模糊查询；
- 限制：无法用于 ORDER BY、GROUP BY，无法用于覆盖索引，无法用于全模糊匹配。

### 2.5 索引数量控制优化（避免过度建索引）

新手最容易踩的坑：给每个查询列都建索引，认为索引越多查询越快。实际上，索引数量过多会带来严重的问题：

1. 写入性能骤降：插入 / 更新 / 删除数据时，需要同步更新所有相关索引，索引越多，开销越大；
2. 磁盘空间暴涨：每个索引都是一个独立的 B + 树，占用大量磁盘空间；
3. 优化器选择混乱：索引过多，MySQL 优化器可能误判，选错执行计划；
4. 运维成本升高：索引碎片、统计信息更新、冗余索引清理的成本成倍增加。

#### 优化规则

1. **单表索引数量控制在 5 个以内**：核心高频查询用联合索引覆盖，禁止为每个场景单独建索引；
2. **禁止创建冗余索引**：已有联合索引`(a,b,c)`，则`(a)`、`(a,b)`都是冗余索引，无需创建（最左前缀可匹配）；
3. **禁止创建重复索引**：已有主键索引`PRIMARY KEY(id)`，禁止再给 id 建唯一索引 / 普通索引；
4. **及时清理无效索引**：从未被使用过的索引，直接删除，释放磁盘空间。

### 2.6 唯一索引 vs 普通索引选型优化

|索引类型|核心优势|核心劣势|适用场景|
|:--|:--|:--|:--|
|**唯一索引**|1. 保证数据唯一性，从数据库层面避免脏数据；<br><br>2. 等值查询时，找到匹配行后立即停止扫描，性能略高|1. 无法使用 Change Buffer，批量写入性能差；<br><br>2. 插入 / 更新时需要校验唯一性，额外开销|业务上必须保证唯一的字段：手机号、身份证号、订单号、用户唯一标识|
|**普通索引**|1. 可使用 Change Buffer，批量写入 / 更新性能提升 50% 以上；<br><br>2. 无唯一性校验开销，写入更快|1. 无法保证数据唯一性；<br><br>2. 等值查询时，会扫描到第一个不匹配的行才停止|非唯一的高频查询列：姓名、年龄、创建时间、状态|

> 核心选型原则：**业务上有唯一性要求的字段，必须建唯一索引；无唯一性要求的字段，统一用普通索引**。

## 三、第二部分：SQL 语句索引适配优化（事中优化，避免索引失效）

即使索引设计得再合理，SQL 写法不当也会导致索引完全失效，这部分重点讲解如何改写 SQL，最大化利用索引，避免失效。

### 3.1 避免索引列上的任何操作，保护索引有序性

索引列上的函数、算术运算、表达式、类型转换，都会破坏索引的有序性，导致优化器无法使用索引，必须把操作移到值的一侧。

|错误写法（索引失效）|正确写法（索引生效）|核心说明|
|:--|:--|:--|
|`WHERE LEFT(name,2) = '张'`|`WHERE name LIKE '张%'`|避免索引列使用字符串函数|
|`WHERE YEAR(create_time) = 2026`|`WHERE create_time BETWEEN '2026-01-01' AND '2026-12-31'`|避免索引列使用日期函数|
|`WHERE id + 1 = 100`|`WHERE id = 99`|避免索引列使用算术运算|
|`WHERE CONCAT(name,'_test') = '张三_test'`|`WHERE name = '张三'`|避免索引列使用表达式|
|`WHERE phone = 13800138000`（phone 是 VARCHAR）|`WHERE phone = '13800138000'`|避免隐式类型转换，保持查询条件与索引列类型一致|

### 3.2 模糊查询优化

1. **绝对禁止左模糊 / 全模糊匹配**：`LIKE '%张三'`、`LIKE '%张三%'` 会完全破坏索引的有序性，导致全表扫描；
2. **优先使用前缀匹配**：`LIKE '张%'` 可正常使用索引，type=range；
3. **全模糊场景优化方案**：
    
    - 方案 1：使用 MySQL 全文索引，替代 LIKE 全模糊匹配；
    - 方案 2：使用 Elasticsearch 等搜索引擎，专门处理全文检索场景；
    - 方案 3：使用前缀索引 + 反向存储，比如把手机号反向存储，查询尾号 1234 时，用`reverse_phone LIKE '4321%'`。

### 3.3 OR 条件查询优化

`OR` 连接的条件中，只要有一个列没有索引，就会导致全表扫描，优化方案如下：

1. **最优方案：用 UNION ALL 拆分查询**：每个子查询单独使用索引，避免全表扫描。

```sql
    -- 错误写法（OR连接非索引列，索引失效）
    SELECT * FROM user WHERE name = '张三' OR address = '北京';
    -- 正确写法（UNION ALL拆分，两个子查询都用索引）
    SELECT * FROM user WHERE name = '张三'
    UNION ALL
    SELECT * FROM user WHERE address = '北京';
```

2. **备选方案：给所有 OR 条件的列建联合索引**：如果业务允许，给 OR 连接的所有列建联合索引，让优化器选择索引合并。
3. **禁止方案：给每个 OR 列单独建单值索引**：MySQL 优化器大概率不会选择索引合并，依然会全表扫描，还会增加索引维护成本。

### 3.4 范围查询优化

1. **范围列统一放在联合索引最后**：避免范围查询截断后续列的索引匹配，最大化索引利用率；
2. **用 IN 等值查询替代范围查询**：IN 属于等值查询，不会中断后续列的索引匹配，比如`age IN (20,21,22,23,24)` 替代 `age BETWEEN 20 AND 24`；
3. **避免过大的范围扫描**：`WHERE id>0`、`WHERE create_time < NOW()` 这种全表范围，优化器会直接选择全表扫描，应缩小范围区间。

### 3.5 NULL 值查询优化

1. **优先将列设为 NOT NULL，给默认值**：NULL 值会导致索引统计、比较运算更复杂，优化器更难选择执行计划，且 IS NOT NULL 大概率会导致索引失效；
2. **IS NULL 可正常使用索引**：`WHERE name IS NULL` 可正常命中索引，无需担心；
3. **IS NOT NULL 优化方案**：如果必须使用 IS NOT NULL，建立覆盖索引，避免回表，让优化器选择索引。

```sql
    -- 优化前：IS NOT NULL全表扫描
    SELECT * FROM user WHERE age IS NOT NULL;
    -- 优化后：覆盖索引生效，无需回表
    SELECT id, age FROM user WHERE age IS NOT NULL; -- 索引idx_age_id (age, id)
```

## 四、第三部分：高频业务场景专项优化技巧

这部分是生产环境实战核心，针对 JOIN、排序、分组、分页、聚合等高频场景，给出针对性的索引优化方案。

### 4.1 JOIN 查询索引优化

JOIN 查询的性能瓶颈 90% 都出在关联字段无索引，优化核心原则：**小表驱动大表，被驱动表的关联字段必须建索引**。

#### 核心优化规则

1. **驱动表选择**：MySQL 优化器会自动选择小表作为驱动表，小表是指「经过 WHERE 条件过滤后，结果集更小的表」，而非表的总行数；
2. **被驱动表的关联字段必须建索引**：这是 JOIN 优化的核心，驱动表的每一行数据，都会去被驱动表中匹配，关联字段无索引会导致嵌套循环全表扫描，出现`Using join buffer`；
3. **多表 JOIN 的关联字段类型、字符集必须完全一致**：避免隐式类型 / 字符集转换，导致关联字段索引失效；
4. **禁止超过 3 张表的 JOIN**：多表 JOIN 会导致优化器选择执行计划的难度指数级升高，且索引维护成本高，业务上应尽量拆分。

#### 示例

```sql
-- 表：user（小表，1万行）、order_info（大表，100万行）
-- 关联字段：user.user_id = order_info.user_id
-- 优化：给被驱动表order_info的user_id建索引
ALTER TABLE order_info ADD INDEX idx_userid (user_id);
-- 优化后JOIN，被驱动表使用索引，type=eq_ref，无Using join buffer
SELECT u.name, o.order_no 
FROM user u
INNER JOIN order_info o ON u.user_id = o.user_id
WHERE u.age > 18;
```

### 4.2 ORDER BY 排序优化

排序优化的核心目标是**消除 Using filesort（文件排序）**，利用索引的有序性，直接从索引中获取有序的数据，无需在内存 / 磁盘中排序。

#### 核心优化规则

1. **WHERE 条件字段 + ORDER BY 字段，建立联合索引**，且顺序严格遵循最左前缀原则。
    
    - 反例：索引`idx_name (name)`，查询`WHERE name='张三' ORDER BY age`，出现 Using filesort；
    - 正例：索引`idx_name_age (name, age)`，查询同上，利用索引有序性，无文件排序。
    
2. **ORDER BY 的字段顺序必须和联合索引的列顺序一致**，排序方向必须全部一致（全 ASC / 全 DESC），混合排序方向无法利用索引。
    
    - 正例：`ORDER BY name ASC, age ASC` 匹配索引`(name, age)`；
    - 反例：`ORDER BY age ASC, name ASC`、`ORDER BY name ASC, age DESC` 无法利用索引。
    
3. **禁止在 ORDER BY 的字段上使用函数 / 表达式**，会破坏索引有序性，导致文件排序。
4. **避免 SELECT 大字段**：SELECT * 会导致排序时需要加载大量数据，内存放不下就会触发磁盘文件排序，性能骤降，只 SELECT 需要的字段。

### 4.3 GROUP BY 分组优化

分组优化的核心目标是**消除 Using temporary（临时表）和 Using filesort**，MySQL 执行 GROUP BY 时，默认会先排序，再分组，很容易触发临时表和文件排序。

#### 核心优化规则

1. **WHERE 条件字段 + GROUP BY 字段，建立联合索引**，利用索引的有序性，避免排序和临时表。
    
    - 示例：查询`WHERE create_time BETWEEN '2026-01-01' AND '2026-03-01' GROUP BY user_id`，建立索引`idx_createtime_userid (create_time, user_id)`，无临时表和文件排序。
    
2. **禁止 SELECT 非分组、非聚合的字段**：比如`SELECT name, age, COUNT(*) FROM user GROUP BY age`，name 字段无意义，且会导致临时表膨胀。
3. **用 ORDER BY NULL 禁止默认排序**：如果业务不需要分组结果有序，加上`ORDER BY NULL`，避免 MySQL 默认排序，消除 Using filesort。

```sql
    -- 优化后，禁止默认排序，消除文件排序
    SELECT age, COUNT(*) FROM user GROUP BY age ORDER BY NULL;
```

### 4.4 深分页查询优化

MySQL 分页查询`LIMIT offset, size`的性能，会随着 offset 的增大急剧下降，比如`LIMIT 100000, 20`，需要先扫描 100020 行，再丢弃前 100000 行，开销极大。

#### 核心优化方案

1. **覆盖索引 + 延迟关联优化**：先在索引中完成分页，找到符合条件的主键，再回表查询完整数据，避免全表扫描后回表。

```sql
    -- 传统写法（深分页性能极差，全表扫描+多次回表）
    SELECT * FROM user WHERE age > 18 LIMIT 100000, 20;
    -- 优化写法（覆盖索引分页，仅20次回表）
    SELECT a.* FROM user a
    INNER JOIN (SELECT id FROM user WHERE age > 18 LIMIT 100000, 20) b
    ON a.id = b.id;
```

2. **主键范围过滤优化**：如果分页是按主键递增排序的，可记录上一页的最大主键 ID，下一页用主键范围过滤，避免扫描前面的行。

```sql
    -- 上一页最大id是100000，下一页直接从100000开始扫描
    SELECT * FROM user WHERE id > 100000 LIMIT 20;
```

3. **禁止使用前端传参的无限制分页**：禁止用户跳转到 1000 页以后，业务上限制最大分页偏移量，从源头避免深分页。

### 4.5 聚合函数优化

1. **MAX ()/MIN () 优化**：给对应的字段建索引，MySQL 可直接从索引的最左 / 最右节点获取最大值 / 最小值，无需扫描表。

```sql
    -- 优化前：全表扫描
    SELECT MAX(create_time) FROM order_info;
    -- 优化后：索引idx_create_time (create_time)，直接从索引获取最大值，无需扫描
```
    
2. **COUNT () 优化**：
    
    - 禁止使用`COUNT(非索引列)`、`COUNT(*)` 是最优写法，MySQL 优化器会自动选择最小的索引进行扫描，性能最高；
    - 禁止使用`COUNT(1)`，和`COUNT(*)`性能几乎无差异，`COUNT(*)`是 SQL 标准写法；
    - 大数据量的实时计数，建议用 Redis 等缓存存储，避免每次查询全表扫描。
    
3. **SUM ()/AVG () 优化**：给计算的字段建索引，用覆盖索引避免回表，直接从索引中计算聚合结果。

```sql
    -- 优化：覆盖索引idx_userid_amount (user_id, amount)，无需回表
    SELECT SUM(amount) FROM order_info WHERE user_id = 100;
```

## 五、第四部分：索引运维与长期健康优化（事后优化）

索引不是建完就一劳永逸的，随着数据的插入、更新、删除，索引会出现碎片、失效、冗余等问题，需要定期运维优化，保证索引长期健康。

### 5.1 冗余 / 无效索引清理

#### 1. 查看冗余索引

```sql
-- MySQL 5.7+ 用sys库查看冗余索引
SELECT 
  table_schema,
  table_name,
  redundant_index_name,
  dominant_index_name,
  sql_drop_index
FROM sys.schema_redundant_indexes;
```

- 冗余索引定义：已有索引`(a,b)`，则`(a)`是冗余索引，因为最左前缀可完全覆盖；
- 清理规则：冗余索引直接删除，不会影响任何查询性能，还能降低写入开销。

#### 2. 查看无效索引（从未被使用过的索引）

```sql
-- 查看未使用的索引（MySQL启动后从未被使用）
SELECT 
  table_schema,
  table_name,
  index_name,
  rows_unused,
  sql_drop_index
FROM sys.schema_unused_indexes
WHERE table_schema NOT IN ('mysql', 'sys', 'information_schema', 'performance_schema');
```

- 清理规则：业务上线 3 个月以上，从未被使用过的索引，直接删除；
- 注意：低频业务场景的索引（如月末统计），需确认业务周期后再删除。

### 5.2 索引碎片整理

频繁的插入、更新、删除操作，会导致 B + 树出现页分裂、页合并，产生大量索引碎片，索引文件膨胀，查询性能下降。

#### 1. 查看索引碎片率

```sql
-- 查看表的索引碎片率
SELECT 
  table_name,
  index_name,
  ROUND((data_free / (data_length + index_length)) * 100, 2) AS fragment_rate
FROM information_schema.tables t
JOIN information_schema.statistics s ON t.table_schema = s.table_schema AND t.table_name = s.table_name
WHERE t.table_schema NOT IN ('mysql', 'sys', 'information_schema')
  AND (data_length + index_length) > 0;
```

- 碎片率 < 5%：无需处理；
- 碎片率 5%-30%：可整理碎片；
- 碎片率 > 30%：必须整理碎片，否则查询性能会严重下降。

#### 2. 碎片整理方法

```sql
-- 方法1：OPTIMIZE TABLE（适用于InnoDB独立表空间）
OPTIMIZE TABLE user;

-- 方法2：ALTER TABLE重建表（彻底整理碎片，推荐大表使用）
ALTER TABLE user ENGINE=InnoDB;
```

#### 注意事项

1. 整理碎片会锁表，必须在业务低峰期执行；
2. 大表整理碎片会占用大量磁盘空间，需提前确认磁盘剩余空间；
3. 整理碎片后，需更新表的统计信息。

### 5.3 统计信息更新

MySQL 优化器选择索引的核心依据是表的统计信息（索引基数、总行数、数据分布等），统计信息过期会导致优化器误判，选错执行计划，甚至全表扫描。

#### 1. 手动更新统计信息

```sql
-- 更新单表统计信息
ANALYZE TABLE user;

-- 更新全库统计信息（谨慎使用，大库会耗时较长，低峰期执行）
ANALYZE TABLE;
```

#### 2. 优化规则

1. 大表批量插入 / 更新 / 删除后，必须手动更新统计信息；
2. 定期（每周）更新核心业务表的统计信息；
3. 发现优化器选错索引时，第一时间更新统计信息。

### 5.4 大表索引创建优化

生产环境中，给千万级以上的大表建索引，很容易导致锁表、业务阻塞，必须遵循以下优化规则：

1. **使用 Online DDL**：MySQL 5.6+ 支持 Online DDL，建索引时不会锁全表，仅在最后阶段加短暂的元数据锁；

```sql
    -- Online DDL建索引，ALGORITHM=INPLACE表示原地修改，不拷贝全表，LOCK=NONE表示不锁表
    ALTER TABLE user ADD INDEX idx_name_age (name, age), ALGORITHM=INPLACE, LOCK=NONE;
```
    
2. **使用第三方工具**：超大型表推荐使用`pt-online-schema-change`、`gh-ost`等工具，无锁建索引，避免主从延迟；
3. **业务低峰期执行**：建索引会占用大量 IO 和 CPU 资源，必须在凌晨等业务低峰期执行；
4. **禁止批量建索引**：大表多个索引合并成一条语句执行，避免多次扫描全表。

## 六、索引优化避坑指南（高频踩坑）

1. **过度建索引**：一张表建十几个索引，认为索引越多查询越快，最终导致写入性能骤降，主从延迟严重。
    
    - 避坑：单表索引控制在 5 个以内，用联合索引覆盖多个查询场景。
    
2. **索引列顺序设计错误**：把范围列、低区分度列、低频列放在联合索引左侧，导致索引利用率极低。
    
    - 避坑：严格遵循「等值优先、高频优先、区分度优先、范围列后置」的排序规则。
    
3. **给小表建索引**：几百行的小表，全表扫描比索引查询更快，建索引反而增加了优化器的选择开销和维护成本。
    
    - 避坑：行数 < 1000 的小表，无需建索引，全表扫描即可。
    
4. **给大字段建全量索引**：给 TEXT、VARCHAR (500) 等大字段建全量索引，导致索引文件膨胀，查询和写入性能下降。
    
    - 避坑：大字符串字段用前缀索引，或用全文索引，禁止建全量索引。
    
5. **用 UUID / 随机字符串做主键**：导致聚簇索引频繁页分裂，碎片严重，所有二级索引膨胀，写入和查询性能都骤降。
    
    - 避坑：用自增 BIGINT 做主键，业务唯一字段建唯一索引。
    
6. **频繁更新索引列**：导致 B + 树频繁分裂合并，锁竞争激烈，写入性能下降，索引碎片快速增加。
    
    - 避坑：索引列尽量选择静态字段，避免频繁更新。
    
7. **忽略联合索引，滥用单值索引**：给每个查询列都建单值索引，MySQL 查询时只能选一个最优的，不如一个合理的联合索引。
    
    - 避坑：多条件查询优先建联合索引，而非多个单值索引。

## 七、索引优化实操工具与验证方法

所有优化效果，都必须通过工具验证，避免主观判断，核心工具如下：

1. **EXPLAIN 执行计划**：索引优化的核心工具，验证索引是否生效、是否有回表、文件排序、临时表等问题，重点看`type`、`key`、`key_len`、`rows`、`Extra`5 个字段。
2. **SHOW INDEX**：查看索引的基数（Cardinality）、区分度、字段长度，判断索引设计是否合理。

```sql
    -- 查看表的所有索引详情
    SHOW INDEX FROM user;
```
    
3. **MySQL sys 库**：查看冗余索引、无效索引、索引使用统计、碎片率，是索引运维的核心工具。
4. **慢查询日志**：开启慢查询日志，定位慢 SQL，找到需要优化的索引，是优化的源头。
5. **OPTIMIZER_TRACE**：跟踪优化器的执行计划选择过程，分析为什么没选预期的索引，解决优化器误判问题。

## 九、核心要点总结

1. **索引优化的核心在设计**：事前的合理设计，远胜于事后的 SQL 改写和运维优化，设计阶段就要遵循主键设计、联合索引排序、覆盖索引等核心规则。
2. **优化的本质是利用 B + 树特性**：所有优化技巧都围绕「保护索引的有序性、减少磁盘 IO、避免回表」展开，脱离 B + 树特性的优化都是空谈。
3. **平衡是关键**：索引优化不是追求极致的查询性能，而是在查询性能和写入维护成本之间找到平衡，用最少的索引支撑最多的业务场景。
4. **验证是唯一标准**：所有优化都必须通过 EXPLAIN 执行计划验证，避免主观判断，上线前必须做性能测试。
5. **长期运维是保障**：索引不是建完就结束了，需要定期巡检冗余 / 无效索引、整理碎片、更新统计信息，保证索引长期健康。

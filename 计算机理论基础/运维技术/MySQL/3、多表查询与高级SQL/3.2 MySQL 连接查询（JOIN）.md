## 一、开篇：连接查询的核心定义与作用

### 1. 什么是连接查询

连接查询（JOIN）是**基于多张表的关联关系，联合查询数据**的 SQL 语法，是 DQL（数据查询语言）的核心能力，也是多表设计后获取关联数据的唯一手段。

简单来说：数据拆分到多张表后，想要一次性查出「用户信息 + 他的订单信息」「商品信息 + 所属分类信息」这类跨表数据，就必须用连接查询。

### 2. 为什么需要连接查询

对应之前的多表设计，单表拆分后解决了数据冗余、更新异常等问题，但带来了新的需求：**跨表获取关联数据**。

- 反例：不用连接查询，只能分两次查：先查用户表，再根据用户 ID 去订单表查对应订单，效率极低，代码复杂度高；
- 正解：用连接查询，一条 SQL 就能直接获取「用户 + 订单」的关联结果，高效简洁。

---

## 二、前置核心：笛卡尔积（必须先掌握）

### 1. 核心定义

笛卡尔积，指的是**两张表不加任何关联条件时，查询结果是两张表所有行的全排列组合**。

- 示例：A 表有 5 行数据，B 表有 10 行数据，不加条件的连接查询，结果会返回 5×10=50 行数据，绝大多数都是无效的垃圾数据。

### 2. 产生原因

- 忘记写连接条件（ON 子句）；
- 连接条件无效、不匹配；
- 多表关联时，中间表的关联条件缺失。

### 3. 危害与规避

- 危害：数据量稍大就会产生百万、千万级的无效结果，直接打满数据库 CPU 和内存，导致服务宕机；
- 规避：**所有连接查询必须加有效的 ON 关联条件**，严禁无关联条件的多表查询。

---

## 三、连接查询的分类与核心语法

MySQL 连接查询分为 4 大类，其中**内连接、左外连接**是日常开发中 99% 场景会用到的，必须重点掌握。

> 语法说明：`OUTER` 关键字可省略，`LEFT OUTER JOIN` 等价于 `LEFT JOIN`，日常开发中通常省略`OUTER`；**推荐所有连接查询都给表加别名**，简化代码，避免字段歧义。

### 3.1 内连接（INNER JOIN）：最常用的等值连接

#### 核心定义

内连接只返回**两张表中关联条件完全匹配**的数据，相当于取两张表的「交集」。

- 主表中没有匹配到从表的数据，不会返回；
- 从表中没有匹配到主表的数据，也不会返回。

#### 核心语法

sql

```
-- 显式写法（推荐，可读性强，语义清晰）
SELECT 字段列表
FROM 表1 别名1
INNER JOIN 表2 别名2
ON 别名1.关联字段 = 别名2.关联字段; -- 关联条件，必须写

-- 隐式写法（不推荐，容易忘记条件产生笛卡尔积）
SELECT 字段列表
FROM 表1 别名1, 表2 别名2
WHERE 别名1.关联字段 = 别名2.关联字段;
```

#### 示例（基于用户表 + 订单表）

sql

```
-- 需求：查询有订单的用户，以及他们的订单信息
SELECT 
  u.user_id, u.user_name, u.phone,
  o.order_id, o.order_no, o.order_amount, o.order_status
FROM user_info u
INNER JOIN order_info o
ON u.user_id = o.user_id; -- 关联条件：用户ID匹配
```

> 结果说明：只会返回「有对应订单的用户」和「有对应用户的订单」，没有订单的用户、没有对应用户的异常订单，都不会出现在结果中。

#### 适用场景

- 只需要两张表中完全匹配的关联数据；
- 核心业务的关联查询，比如「订单 + 支付流水」「商品 + 所属分类」。

---

### 3.2 外连接（OUTER JOIN）：保留全量数据的连接

外连接的核心特点：**会保留某一张表的全部数据，另一张表匹配不到的数据用 NULL 填充**，分为左外连接、右外连接、全外连接。

#### 3.2.1 左外连接（LEFT JOIN）：开发中第二常用

##### 核心定义

以`LEFT JOIN`**左边的表为主表**，保留主表的**全部数据**，右边的从表只返回匹配到的数据，匹配不到的字段全部用`NULL`填充。

- 无论主表的数据在从表有没有匹配到，都会出现在结果中；
- 从表只有匹配到主表的数据才会返回，匹配不到的用 NULL 填充。

##### 核心语法

sql

```
SELECT 字段列表
FROM 主表 主表别名
LEFT JOIN 从表 从表别名
ON 主表别名.关联字段 = 从表别名.关联字段;
```

##### 示例（基于用户表 + 订单表）

sql

```
-- 需求：查询所有用户，以及他们的订单信息（包括没有订单的用户）
SELECT 
  u.user_id, u.user_name, u.phone,
  o.order_id, o.order_no, o.order_amount
FROM user_info u
LEFT JOIN order_info o
ON u.user_id = o.user_id;
```

> 结果说明：所有用户都会出现在结果中，有订单的用户会显示对应的订单信息，没有订单的用户，订单相关的字段全部为 NULL。

##### 适用场景

- 需要保留主表的全部数据，无论从表有没有匹配记录；
- 统计类场景，比如「统计所有用户的订单数量，包括 0 单的用户」「统计所有商品的销量，包括 0 销量的商品」。

#### 3.2.2 右外连接（RIGHT JOIN）

##### 核心定义

以`RIGHT JOIN`**右边的表为主表**，保留主表的全部数据，左边的表只返回匹配到的数据，匹配不到的字段用 NULL 填充。

> 注意：所有右连接都可以等价转换为左连接，**日常开发中优先使用左连接**，语法统一，可读性更高。

##### 语法与示例

sql

```
-- 右连接写法
SELECT 
  u.user_id, u.user_name,
  o.order_id, o.order_no
FROM user_info u
RIGHT JOIN order_info o
ON u.user_id = o.user_id;

-- 等价转换为左连接（推荐）
SELECT 
  u.user_id, u.user_name,
  o.order_id, o.order_no
FROM order_info o
LEFT JOIN user_info u
ON o.user_id = u.user_id;
```

#### 3.2.3 全外连接（FULL OUTER JOIN）

##### 核心定义

保留两张表的**全部数据**，无论有没有匹配到，匹配不到的字段用 NULL 填充，相当于取两张表的「并集」。

> 关键说明：**MySQL 原生不支持全外连接**，可以通过 `LEFT JOIN UNION RIGHT JOIN` 的方式等价实现。

##### 实现示例

sql

```
-- MySQL实现全外连接
SELECT * FROM user_info u LEFT JOIN order_info o ON u.user_id = o.user_id
UNION
SELECT * FROM user_info u RIGHT JOIN order_info o ON u.user_id = o.user_id;
```

> 说明：`UNION` 会自动去重，避免两张表匹配到的重复数据出现两次。

---

### 3.3 交叉连接（CROSS JOIN）

#### 核心定义

交叉连接不加任何关联条件，返回的结果就是两张表的**笛卡尔积**，日常开发中几乎不会用到，仅适用于极少数需要生成全量组合的场景（比如生成所有商品 + 所有标签的组合）。

#### 语法

sql

```
SELECT 字段列表
FROM 表1
CROSS JOIN 表2;
```

---

### 3.4 自连接：自己和自己连接

#### 核心定义

自连接不是一种新的连接类型，而是**把一张表当成两张表来用，自己和自己做连接查询**，核心用于处理「同一张表内有父子级、层级关联关系」的数据。

#### 适用场景

- 员工表：员工和上级领导都存在同一张表，通过`manager_id`关联员工 ID；
- 商品分类表：一级分类、二级分类都存在同一张表，通过`parent_id`关联父分类 ID；
- 区域表：省、市、区都存在同一张表，通过父级 ID 关联。

#### 示例 1：员工与上级领导查询

sql

```
-- 员工表
CREATE TABLE employee (
    emp_id INT PRIMARY KEY AUTO_INCREMENT COMMENT '员工ID',
    emp_name VARCHAR(50) NOT NULL COMMENT '员工姓名',
    manager_id INT COMMENT '上级领导ID，关联同表的emp_id'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 自连接查询：查询员工姓名和对应的上级领导姓名
SELECT 
  e.emp_id, e.emp_name AS '员工姓名',
  m.emp_name AS '上级领导姓名'
FROM employee e
LEFT JOIN employee m -- 同一张表，起不同的别名，当成两张表用
ON e.manager_id = m.emp_id;
```

#### 示例 2：商品分类父子级查询

sql

```
-- 分类表
CREATE TABLE category (
    category_id INT PRIMARY KEY AUTO_INCREMENT COMMENT '分类ID',
    category_name VARCHAR(50) NOT NULL COMMENT '分类名称',
    parent_id INT COMMENT '父分类ID，0表示一级分类'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 自连接查询：查询二级分类和对应的一级分类名称
SELECT 
  c.category_id AS '二级分类ID',
  c.category_name AS '二级分类名称',
  p.category_name AS '一级分类名称'
FROM category c
LEFT JOIN category p
ON c.parent_id = p.category_id
WHERE c.parent_id != 0;
```

## 四、多表连接（3 张及以上表关联）

连接查询不局限于两张表，可扩展到 3 张、4 张甚至更多表，核心规则是：**每多关联一张表，就加一个 JOIN...ON 子句，指定关联条件**。

### 核心语法

sql

```
SELECT 字段列表
FROM 表1 别名1
LEFT JOIN 表2 别名2 ON 别名1.关联字段 = 别名2.关联字段
LEFT JOIN 表3 别名3 ON 别名2.关联字段 = 别名3.关联字段
INNER JOIN 表4 别名4 ON 别名3.关联字段 = 别名4.关联字段;
```

### 示例（电商场景 4 表关联）

需求：查询所有订单，包含下单用户信息、订单明细、商品信息

sql

```
SELECT 
  o.order_id, o.order_no, o.create_time,
  u.user_name, u.phone,
  od.product_num, od.product_price,
  p.product_name, p.category_id
FROM order_info o
-- 关联用户表
LEFT JOIN user_info u ON o.user_id = u.user_id
-- 关联订单明细表
LEFT JOIN order_detail od ON o.order_id = od.order_id
-- 关联商品表
LEFT JOIN product p ON od.product_id = p.product_id;
```

## 五、高频易错点：ON 与 WHERE 的区别（重中之重）

这是新手使用左连接时，90% 会踩的坑，必须彻底搞懂！

### 核心区别

表格

|关键字|作用阶段|核心逻辑|左连接中的效果|
|:--|:--|:--|:--|
|**ON**|连接时|两张表**生成临时关联结果集**的匹配条件|即使 ON 条件不满足，主表的数据依然会保留，从表字段填充 NULL|
|**WHERE**|连接后|对**已经生成的关联结果集**进行过滤|会过滤掉所有不满足条件的行，包括主表的数据|

### 正反示例对比

需求：查询所有用户，以及他们的「已支付」订单信息（没有订单的用户也要显示）

sql

```
-- 正确写法：订单状态过滤写在ON里
SELECT 
  u.user_id, u.user_name,
  o.order_id, o.order_status
FROM user_info u
LEFT JOIN order_info o
ON u.user_id = o.user_id AND o.order_status = '已支付'; -- 过滤条件写在ON中
```

> 正确结果：所有用户都会显示，只有已支付的订单会匹配，没有订单 / 未支付订单的用户，订单字段为 NULL。

sql

```
-- 错误写法：订单状态过滤写在WHERE里
SELECT 
  u.user_id, u.user_name,
  o.order_id, o.order_status
FROM user_info u
LEFT JOIN order_info o
ON u.user_id = o.user_id
WHERE o.order_status = '已支付'; -- 过滤条件写在WHERE中
```

> 错误结果：只会显示有已支付订单的用户，没有订单 / 未支付订单的用户会被 WHERE 过滤掉，相当于变成了内连接，完全不符合需求。

### 核心结论

- 左连接中，**对从表的过滤条件，必须写在 ON 子句里**；
- 对主表的过滤条件，写在 WHERE 子句里；
- 内连接中，条件写在 ON 和 WHERE 里，结果是一样的（因为内连接只保留匹配的行），但推荐写在 ON 里，语义更清晰。

## 六、连接查询的核心对比与选型

表格

|连接类型|核心特点|结果集范围|推荐使用场景|
|:--|:--|:--|:--|
|**INNER JOIN 内连接**|只返回匹配的行|两张表的交集|只需要完全匹配的关联数据|
|**LEFT JOIN 左连接**|保留左表全部数据，右表匹配不到填 NULL|左表全集 + 匹配的右表数据|需要保留主表全量数据，无论从表是否有匹配|
|**RIGHT JOIN 右连接**|保留右表全部数据，左表匹配不到填 NULL|右表全集 + 匹配的左表数据|可转换为左连接，不推荐优先使用|
|**FULL JOIN 全外连接**|保留两张表全部数据，匹配不到填 NULL|两张表的并集|极少场景使用，MySQL 需用 UNION 实现|
|**CROSS JOIN 交叉连接**|无关联条件，返回笛卡尔积|两张表的全排列组合|几乎不用，仅用于生成全量组合|

## 七、连接查询的性能优化与避坑指南

### 1. 性能优化核心要点

- **关联字段必须加索引**：JOIN 的关联字段（通常是主键、外键）必须建立索引，否则会触发全表扫描，数据量大时性能极差；
- **小表驱动大表**：左连接中，用数据量小的表当主表，驱动大表，减少循环匹配的次数；
- ** 避免 SELECT ***：只查询需要的字段，减少数据传输和回表开销；
- **控制连接表的数量**：不要过度关联表，通常不建议超过 5 张表关联，否则查询优化器会很难生成最优执行计划，性能急剧下降；
- **避免在关联字段上做函数运算**：比如 `ON YEAR(u.create_time) = YEAR(o.create_time)`，会导致索引失效。

### 2. 常见避坑指南

1. **忘记加 ON 条件，产生笛卡尔积**：所有 JOIN 必须加有效的 ON 关联条件；
2. **左连接的过滤条件写在 WHERE 里**：导致主表数据被过滤，变成内连接；
3. **字段歧义**：两张表有同名字段时，必须加表别名指定字段所属的表，否则会报错；
    
    - 错误示例：`SELECT user_id, user_name FROM user_info u LEFT JOIN order_info o ON u.user_id = o.user_id;`（user_id 在两张表都有，歧义）
    - 正确示例：`SELECT u.user_id, u.user_name FROM user_info u LEFT JOIN order_info o ON u.user_id = o.user_id;`
    
4. **滥用连接查询**：能单表查的就不要多表关联，避免不必要的性能开销；
5. **用逗号分隔的隐式内连接**：可读性差，容易忘记条件，推荐用显式的 INNER JOIN...ON 写法。

## 八、综合实操演练

基于之前的 `user_info`、`order_info`、`product`、`category` 表，完成以下查询需求：

1. 查询所有商品，以及所属的分类名称（包括没有分类的商品）；
2. 查询每个用户的订单总金额，包括没有订单的用户（总金额显示 0）；
3. 查询已支付的订单，包含下单用户姓名、订单明细、商品名称、商品分类名称；
4. 查询所有一级分类，以及对应的二级分类列表。

### 参考答案

sql

```
-- 1. 查询所有商品及所属分类名称
SELECT 
  p.product_id, p.product_name, p.price,
  c.category_name
FROM product p
LEFT JOIN category c ON p.category_id = c.category_id;

-- 2. 查询每个用户的订单总金额，无订单显示0
SELECT 
  u.user_id, u.user_name,
  IFNULL(SUM(o.order_amount), 0) AS '总订单金额'
FROM user_info u
LEFT JOIN order_info o ON u.user_id = o.user_id
GROUP BY u.user_id, u.user_name;

-- 3. 查询已支付订单的完整信息
SELECT 
  o.order_id, o.order_no, o.create_time,
  u.user_name, u.phone,
  od.product_num, od.product_price,
  p.product_name, c.category_name
FROM order_info o
INNER JOIN user_info u ON o.user_id = u.user_id
INNER JOIN order_detail od ON o.order_id = od.order_id
INNER JOIN product p ON od.product_id = p.product_id
LEFT JOIN category c ON p.category_id = c.category_id
WHERE o.order_status = '已支付';

-- 4. 查询一级分类及对应的二级分类
SELECT 
  p.category_id AS '一级分类ID',
  p.category_name AS '一级分类名称',
  c.category_id AS '二级分类ID',
  c.category_name AS '二级分类名称'
FROM category p
LEFT JOIN category c ON p.category_id = c.parent_id
WHERE p.parent_id = 0;
```

## 九、核心要点总结

1. 连接查询的核心是**通过关联条件，联合多张表获取数据**，是多表设计后必备的查询能力；
2. 必须先理解笛卡尔积的危害，**所有连接查询必须加有效的 ON 关联条件**；
3. 日常开发 99% 的场景，只用**INNER JOIN（内连接）** 和 **LEFT JOIN（左连接）**，必须重点掌握；
4. 左连接的核心坑点：**从表的过滤条件必须写在 ON 里，不能写在 WHERE 里**，否则会变成内连接；
5. 自连接用于处理同表内的层级关系，核心是给同一张表起不同的别名，当成两张表用；
6. 性能优化的核心：**关联字段必须加索引，小表驱动大表，避免过度关联**。

## 一、基础定义与核心背景

### 1. 核心定义

**关系型数据库（Relational Database Management System，简称 RDBMS）**，是建立在**关系模型**基础上的数据库管理系统，借助集合代数等数学理论处理数据，核心特征是采用**二维表格（关系）** 组织数据，通过表与表之间的关联关系维护数据完整性，是目前工业界最成熟、应用最广泛的数据库类型。

### 2. 发展背景

- 1970 年，IBM 研究员埃德加・弗兰克・科德（Edgar F. Codd）发表论文《大型共享数据库的关系模型》，首次提出关系模型理论，奠定了关系型数据库的理论基石。
- 配套的**SQL（结构化查询语言）** 成为关系型数据库的标准通用语言，实现了数据库操作的标准化。

---

## 二、核心基础概念（必掌握）

关系型数据库的所有能力都基于以下核心概念，是理解其运行逻辑的核心：

表格

|概念|通俗释义|核心规则 / 作用|
|:--|:--|:--|
|**关系（Relation）**|一张二维数据表，是数据存储的核心单元|每个关系有唯一表名，表结构由行列组成，同一张表内列名不可重复|
|**元组（Tuple）**|数据表中的**一行**，对应一条业务记录|同一张表内的元组结构完全一致，通过主键唯一标识|
|**属性（Attribute）**|数据表中的**一列**，对应一个字段|每个属性有唯一名称、固定数据类型，定义了该列数据的存储规则|
|**主键（Primary Key, PK）**|唯一标识一条元组的属性 / 属性组合|非空、不重复，一张表只能有一个主键，是数据定位和关联的核心|
|**外键（Foreign Key, FK）**|用来建立两张表关联关系的字段|在当前表中为外键，在关联表中必须是主键，核心作用是维护**参照完整性**，避免无效关联数据|
|**域（Domain）**|属性的取值范围|限制字段的合法值，比如性别只能是「男 / 女 / 未知」，年龄在 0-150 之间，保障数据有效性|
|**约束（Constraint）**|对数据的强制规则限制|除主键、外键外，还包括非空约束（NOT NULL）、唯一约束（UNIQUE）、检查约束（CHECK）、默认值约束（DEFAULT），核心是保障数据完整性|

---

## 三、核心理论基石：ACID 事务特性

ACID 是关系型数据库区别于绝大多数非关系型数据库的核心特征，是保障数据强一致性的核心，**事务**是数据库中一组不可分割的 SQL 操作集合，是 ACID 的执行单元。

表格

|特性|全称|核心含义|通俗示例|
|:--|:--|:--|:--|
|**原子性（Atomicity）**|Atomicity|事务是最小执行单元，操作要么**全部执行成功**，要么**全部失败回滚**，不存在中间状态|银行转账：A 账户扣钱、B 账户加钱，两个操作要么都完成，要么都撤销，不会出现扣了钱没到账的情况|
|**一致性（Consistency）**|Consistency|事务执行前后，数据库的完整性约束、业务规则不会被破坏|转账前后，A+B 两个账户的总金额始终保持不变，不会出现资金凭空增减|
|**隔离性（Isolation）**|Isolation|多个并发执行的事务之间相互隔离，互不干扰，避免并发操作导致的数据异常|两个事务同时操作同一账户，不会出现互相覆盖修改的问题，数据库通过隔离级别解决脏读、不可重复读、幻读问题|
|**持久性（Durability）**|Durability|事务一旦提交成功，对数据库的修改就是**永久生效**的，无论系统崩溃、断电、硬件故障，已提交的数据都不会丢失|转账成功后，哪怕银行系统宕机，账户余额的修改也会永久保留|

## 四、关系型数据库的核心特性

1. **结构化存储，逻辑清晰**
    
    采用固定 schema 的二维表结构，数据格式统一，行列定义明确，易于理解、维护和迭代，业务逻辑映射直观。
2. **强数据一致性保障**
    
    依托 ACID 事务特性，可实现数据的强一致性，是金融、政务、核心交易等对数据准确性要求极高场景的首选。
3. **标准化查询语言**
    
    统一支持 SQL 结构化查询语言，语法标准化程度高，跨数据库迁移成本低，学习门槛低，生态极其完善，支持复杂的关联查询、子查询、聚合统计等操作。
4. **完善的数据完整性保障**
    
    通过主键、外键、非空、唯一等多重约束，从根源上避免脏数据、冗余数据，维护表与表之间的关联关系，保障数据的准确性和有效性。
5. **成熟稳定的生态与安全能力**
    
    发展超 50 年，技术体系极其成熟，具备完善的权限管理、备份恢复、容灾机制、性能优化方案，企业级服务能力完善。

## 六、优缺点与适用场景

### 核心优点

1. 强事务、强一致性，完美适配核心交易类业务；
2. SQL 语言标准化，学习和迁移成本低，通用能力强；
3. 完善的约束机制，保障数据完整性，减少数据冗余；
4. 支持复杂的多表关联查询、聚合统计，适配复杂业务需求；
5. 技术成熟，配套工具、文档、解决方案完善，运维体系成熟。

### 核心缺点

1. **横向扩展能力弱**：传统架构以单机为主，分布式扩展难度大、成本高，弹性扩容能力远弱于非关系型数据库；
2. **海量数据下性能瓶颈**：单表数据量过大时，查询性能会显著下降，高并发写入场景下，受事务、约束、索引维护的影响，性能上限低于非关系型数据库；
3. **数据结构灵活性差**：表结构（schema）固定，一旦定义后修改成本高，不适合半结构化、非结构化数据（图片、视频、JSON 文档、日志等）的存储；
4. **分布式场景复杂度高**：分布式部署时，需要解决分库分表、分布式事务等复杂问题，运维和开发成本大幅上升。

### 适用 & 不适用场景

表格

|推荐适用场景|不推荐适用场景|
|:--|:--|
|金融支付、银行转账、订单系统等需要强数据一致性的核心交易业务|海量非结构化 / 半结构化数据存储（日志、图片、视频、文档等）|
|企业 ERP、CRM、OA 等需要复杂关联查询和数据完整性的管理系统|超高并发读写场景（秒杀、热搜榜单、实时日志采集等）|
|数据结构固定、业务逻辑稳定的结构化数据存储场景|数据结构频繁变动、需求快速迭代的敏捷开发场景|
|需要严格的权限管控、合规审计的政务、国企业务系统|需弹性横向扩展的 PB 级海量数据分布式存储场景|

## 七、与非关系型数据库（NoSQL）的核心区别

表格

|对比维度|关系型数据库（RDBMS）|非关系型数据库（NoSQL）|
|:--|:--|:--|
|数据模型|二维表结构，固定 schema，结构化存储|无固定表结构，支持键值、文档、列族、图等多种模型，适配非结构化数据|
|事务支持|完整支持 ACID 事务，强一致性|大多不支持强事务，仅保证最终一致性，部分产品支持弱事务|
|查询语言|标准化 SQL 语言，功能强大，支持复杂查询|无统一标准语言，多为专属 API，复杂查询能力弱|
|扩展性|纵向扩展为主，横向分布式扩展难度高|原生支持分布式架构，横向弹性扩展能力极强|
|数据一致性|优先保障强一致性|优先保障高可用、高性能，大多为最终一致性|
|核心优势|数据准确、稳定可靠、逻辑清晰、复杂查询能力强|高并发、高性能、灵活扩展、适配非结构化数据|

## 八、入门实操示例（MySQL）

### 1. 基础建库建表（含主键、外键、约束）

sql

```
-- 1. 创建数据库
CREATE DATABASE study_rdbms DEFAULT CHARACTER SET utf8mb4;
USE study_rdbms;

-- 2. 创建用户主表（含主键、非空、唯一、检查、默认值约束）
CREATE TABLE user_info (
    user_id INT PRIMARY KEY AUTO_INCREMENT COMMENT '用户ID，主键',
    user_name VARCHAR(50) NOT NULL COMMENT '用户名，非空约束',
    phone VARCHAR(11) UNIQUE NOT NULL COMMENT '手机号，唯一约束',
    age TINYINT CHECK (age >= 0 AND age <= 150) COMMENT '年龄，检查约束',
    create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间，默认值约束'
) COMMENT '用户信息表';

-- 3. 创建订单从表（含外键，关联用户表）
CREATE TABLE order_info (
    order_id INT PRIMARY KEY AUTO_INCREMENT COMMENT '订单ID，主键',
    order_no VARCHAR(32) UNIQUE NOT NULL COMMENT '订单编号',
    user_id INT NOT NULL COMMENT '下单用户ID',
    order_amount DECIMAL(10,2) NOT NULL COMMENT '订单金额',
    order_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '下单时间',
    -- 外键约束：关联用户表的主键user_id，维护参照完整性
    FOREIGN KEY (user_id) REFERENCES user_info(user_id)
) COMMENT '订单信息表';
```

### 2. 基础数据操作与关联查询

sql

```
-- 插入用户数据
INSERT INTO user_info (user_name, phone, age) VALUES ('张三', '13800138000', 25);

-- 插入订单数据
INSERT INTO order_info (order_no, user_id, order_amount) VALUES ('ORD20260309001', 1, 99.99);

-- 多表关联查询：查询用户对应的订单信息
SELECT 
  u.user_id, u.user_name, u.phone,
  o.order_no, o.order_amount, o.order_time
FROM user_info u
LEFT JOIN order_info o ON u.user_id = o.user_id
WHERE u.user_id = 1;
```

## 九、核心学习要点总结

1. 关系型数据库的核心是**关系模型**，以二维表为存储单元，通过行列组织数据，通过主键、外键维护数据关联与完整性。
2. **ACID 事务特性**是关系型数据库的灵魂，是其能支撑核心交易业务的核心原因。
3. **SQL 语言**是关系型数据库的通用操作语言，标准化的语法使其具备极强的通用性和可移植性。
4. 关系型数据库的核心优势是**强一致性、数据完整性、复杂查询能力**，短板是**弹性扩展能力弱、海量数据性能瓶颈、非结构化数据适配差**。
5. 技术选型核心：**强一致性、结构化、复杂查询**优先选关系型数据库；**高并发、海量非结构化数据、弹性扩展**优先选非关系型数据库。

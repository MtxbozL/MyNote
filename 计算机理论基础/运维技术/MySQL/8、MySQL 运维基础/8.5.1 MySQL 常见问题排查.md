## 一、开篇：排查核心原则与标准化流程

### 1.1 笔记定位与核心目标

本笔记是生产环境 MySQL 故障排查的**实战落地手册**，覆盖从基础连接、资源占用、SQL 性能、锁问题、主从复制到服务崩溃的全场景常见问题，所有内容均遵循「**现象定位→根因分析→分步排查→落地解决方案→避坑指南**」的结构，可直接对应线上故障直接套用。

本笔记完全联动之前的库表设计、索引优化、慢查询优化、主从复制、备份恢复、读写分离 / MHA 全体系知识，形成完整的 MySQL 运维闭环。

### 1.2 故障排查黄金原则

1. **先保业务，再查根因**：生产环境故障第一优先级是恢复业务，而非死磕根因。比如 CPU 打满先 kill 异常 SQL，主从故障先切换主库，业务恢复后再复盘根因；
2. **先整体，后局部**：先看服务器整体资源（CPU / 内存 / IO / 网络），再看数据库状态，最后定位到具体 SQL / 表 / 事务；
3. **先监控，后命令**：先通过监控平台看故障时间点的指标变化，锁定故障范围，再用命令深入排查，避免无目的的乱查；
4. **先系统，后数据库**：先排除操作系统、硬件、网络问题，再排查数据库本身的问题，80% 的故障都和系统资源相关；
5. **操作可回滚**：所有排查操作必须可回滚，禁止在生产环境执行无备份的修改操作（如改配置、删数据、强制恢复）。

### 1.3 通用标准化排查流程（所有故障通用）

```c
graph LR
A[故障现象确认] --> B[业务影响范围评估]
B --> C[系统资源排查：CPU/内存/IO/网络/磁盘]
C --> D[数据库整体状态排查：连接数/进程/错误日志]
D --> E[细分场景根因定位：SQL/锁/主从/配置等]
E --> F[应急处理：恢复业务]
F --> G[根因深度分析与优化]
G --> H[长效防控：监控/规范/预案]
```

### 1.4 前置必备：核心日志与排查工具

#### 1.4.1 必须开启的核心日志

|日志类型|核心作用|查看配置命令|
|:--|:--|:--|
|**错误日志（error log）**|记录 MySQL 启动、运行、崩溃的所有错误信息，是 90% 故障的第一排查点|`SHOW VARIABLES LIKE 'log_error';`|
|**慢查询日志（slow log）**|记录所有执行超时的 SQL，是性能问题的核心排查依据|`SHOW VARIABLES LIKE 'slow_query_log%';`|
|**二进制日志（binlog）**|记录所有数据修改操作，用于数据恢复、主从复制问题排查|`SHOW VARIABLES LIKE 'log_bin%';`|
|**InnoDB 死锁日志**|记录 InnoDB 死锁的详细信息，用于死锁问题排查|`SHOW VARIABLES LIKE 'innodb_print_all_deadlocks';`|
|**通用查询日志（general log）**|记录所有执行的 SQL，仅临时排查开启，生产环境禁止长期开启|`SHOW VARIABLES LIKE 'general_log%';`|

#### 1.4.2 核心排查命令 / 工具

|工具 / 命令|核心作用|
|:--|:--|
|`SHOW PROCESSLIST` / `SHOW FULL PROCESSLIST`|查看当前所有数据库连接，定位正在执行的异常 SQL、阻塞的连接|
|`SHOW ENGINE INNODB STATUS`|InnoDB 引擎状态详情，核心用于锁等待、死锁、事务、IO 问题排查|
|`EXPLAIN`|分析 SQL 执行计划，定位索引失效、全表扫描等性能问题|
|`SHOW STATUS`|查看 MySQL 运行状态指标，连接数、锁、缓存、IO 等统计信息|
|`SHOW SLAVE STATUS\G`|主从复制状态排查，核心用于主从延迟、复制中断问题|
|`pt-query-digest`|慢查询日志分析工具，定位累计耗时最高的异常 SQL|
|`top/htop`|系统 CPU / 内存占用排查，定位 MySQL 进程资源占用|
|`iostat/vmstat`|系统磁盘 IO / 内存状态排查，定位 IO 瓶颈|
|`netstat/ss`|网络连接状态排查，定位连接数、端口监听问题|

## 二、第一大类：连接类问题排查（最常见入门级故障）

### 2.1 问题 1：客户端无法连接 MySQL 服务

#### 故障现象

客户端连接报错：`Can't connect to MySQL server on 'xxx' (10060/10061/10067)`，无法建立数据库连接。

#### 分步排查与解决方案

| 排查顺序 | 排查动作                         | 异常根因                            | 解决方案                                                                           |
| :--- | :--------------------------- | :------------------------------ | :----------------------------------------------------------------------------- |
| 1    | 服务器本地登录验证                    | MySQL 服务未启动 / 启动失败              | 查看错误日志，修复启动问题，重新启动 MySQL 服务                                                    |
| 2    | 检查端口监听状态 `ss -tulnp          | grep 3306`                      | 端口未监听 / 绑定了 127.0.0.1修改`my.cnf`的`bind-address=0.0.0.0`，重启服务                    |
| 3    | 检查服务器防火墙 / SELinux           | 防火墙未开放 3306 端口，SELinux 拦截       | 关闭 SELinux，防火墙开放 3306 端口：`firewall-cmd --add-port=3306/tcp --permanent`        |
| 4    | 检查客户端与服务器网络连通性 `ping/telnet` | 网络不通 / 跨网段访问限制                  | 联系运维开通网络权限，保证客户端能 telnet 通 3306 端口                                             |
| 5    | 检查 MySQL 用户主机限制              | 用户仅允许本地 / 指定 IP 登录，客户端 IP 不在白名单 | 重新创建用户，指定允许访问的 IP / 网段：`CREATE USER 'user'@'192.168.1.%' IDENTIFIED BY 'xxx';` |

#### 避坑指南

- 生产环境禁止给用户设置`@'%'`全主机权限，仅开放必要的 IP / 网段；
- 禁止直接关闭防火墙，仅开放必要的端口，避免安全风险。

### 2.2 问题 2：Too many connections 连接数打满

#### 故障现象

连接报错：`ERROR 1040 (08004): Too many connections`，新连接无法建立，业务完全不可用。

#### 核心根因

1. 业务突发流量，连接数超过`max_connections`上限；
2. 长连接未释放，连接池配置不合理，连接泄露；
3. 慢 SQL / 锁等待导致连接堆积，大量连接处于 Sleep / 执行状态；
4. `wait_timeout`/`interactive_timeout`设置过大，空闲连接未及时回收。

#### 分步排查与解决方案

1. **应急处理：紧急恢复业务**

```sql
    -- 1. 用root账号登录（root有额外的1个预留连接，即使连接数打满也能登录）
    mysql -u root -p
    
    -- 2. 查看当前连接数与上限
    SHOW VARIABLES LIKE 'max_connections';
    SHOW STATUS LIKE 'Threads_connected';
    
    -- 3. 批量kill空闲的Sleep连接，释放连接数（仅应急用）
    -- 生成kill命令
    SELECT CONCAT('KILL ', id, ';') FROM information_schema.PROCESSLIST WHERE COMMAND = 'Sleep' AND TIME > 300;
    -- 执行生成的kill命令，释放空闲超过5分钟的连接
```
    
2. **根因定位**

```sql
    -- 1. 查看所有连接详情，定位堆积的连接来源、执行的SQL
    SHOW FULL PROCESSLIST;
    
    -- 2. 查看连接数相关配置
    SHOW VARIABLES LIKE 'max_connections'; -- 默认151，生产环境建议设为500-2000
    SHOW VARIABLES LIKE 'wait_timeout'; -- 空闲连接超时时间，建议设为300秒（5分钟）
    SHOW VARIABLES LIKE 'interactive_timeout';
```
    
3. **长效解决方案**
    
    - 调整合理的连接数上限：`my.cnf`中设置`max_connections=1000`，根据服务器配置调整；
    - 缩短空闲连接超时时间：`wait_timeout=300`，`interactive_timeout=300`，及时回收空闲连接；
    - 优化业务连接池：配置合理的最小 / 最大连接数，开启连接池空闲连接回收，避免连接泄露；
    - 优化慢 SQL / 锁等待：从根源解决连接执行时间过长导致的堆积问题。

#### 避坑指南

- 禁止把`max_connections`设为 10000+，过大的连接数会导致内存占用过高，数据库 OOM；
- 禁止批量 kill 正在执行事务的连接，会导致事务回滚，引发数据一致性问题；
- 连接数打满时，不要直接重启数据库，会导致业务中断时间更长，优先释放空闲连接。

### 2.3 问题 3：Access denied 权限拒绝问题

#### 故障现象

连接报错：`ERROR 1045 (28000): Access denied for user 'xxx'@'xxx' (using password: YES/NO)`，无法登录或执行 SQL 时报权限不足。

#### 分步排查与解决方案

1. **登录时权限拒绝**
    
    - 根因 1：密码错误，大小写 / 特殊字符输入错误；
        
        解决方案：重置用户密码：`ALTER USER 'user'@'host' IDENTIFIED BY '新密码';`
    - 根因 2：用户主机不匹配，比如用户仅允许`192.168.1.100`登录，客户端 IP 是`192.168.1.200`；
        
        解决方案：查看用户主机：`SELECT user, host FROM mysql.user WHERE user='xxx';`，创建对应主机的用户。
    - 根因 3：用户不存在，或被删除；
        
        解决方案：重新创建用户并授权。
    
2. **执行 SQL 时权限拒绝**
    
    - 根因：用户没有对应操作的权限，比如只有 SELECT 权限，执行 INSERT 语句；
        
        解决方案：查看用户权限：`SHOW GRANTS FOR 'user'@'host';`，授予对应权限：`GRANT SELECT,INSERT ON db.* TO 'user'@'host';`

#### 避坑指南

- 禁止给业务用户授予`ALL PRIVILEGES`超级权限，遵循最小权限原则；
- 生产环境禁止使用 root 用户执行业务 SQL，必须使用专用业务用户。

## 三、第二大类：系统资源占用过高问题排查（生产环境核心故障）

### 3.1 问题 1：CPU 使用率飙升 / 100% 打满

#### 故障现象

服务器 CPU 使用率持续 90%+，甚至 100%，数据库响应极慢，业务接口超时，是生产环境最高发的性能故障。

#### 核心根因（按出现概率排序）

1. **高频慢 SQL / 索引失效**：占比 90%，无索引、索引失效的 SQL 高频执行，全表扫描 / 全索引扫描，占用大量 CPU；
2. **大量排序 / 分组 / 聚合操作**：`ORDER BY/GROUP BY/DISTINCT/COUNT(*)` 无索引，导致`Using filesort/Using temporary`，CPU 计算开销极大；
3. **锁等待 / 事务堆积**：高并发下锁等待导致大量连接重试执行 SQL，CPU 飙升；
4. **MySQL 参数配置不合理**：`innodb_buffer_pool_size`设置过小，频繁磁盘 IO；`thread_concurrency`设置不当，线程切换频繁；
5. **硬件瓶颈**：服务器 CPU 核心数过少，无法支撑业务并发。

#### 分步排查与解决方案

1. **应急处理：快速恢复业务**

```sql
    -- 1. 查看正在执行的SQL，定位CPU消耗最高的长执行SQL
    SHOW FULL PROCESSLIST;
    -- 重点看：TIME（执行时间）、STATE（执行状态）、INFO（执行的SQL）
    
    -- 2. 批量kill长时间执行的异常SQL，快速降低CPU（仅应急）
    SELECT CONCAT('KILL ', id, ';') FROM information_schema.PROCESSLIST WHERE COMMAND = 'Query' AND TIME > 10;
```
    
2. **根因定位**

```sql
    -- 1. 开启慢查询日志，捕获所有慢SQL
    SET GLOBAL slow_query_log = 'ON';
    SET GLOBAL long_query_time = 0.5; -- 捕获超过0.5秒的SQL
    
    -- 2. 用pt-query-digest分析慢日志，按累计耗时排序，找到TOP10消耗CPU的SQL
    pt-query-digest /var/lib/mysql/slow.log > slow_analysis.txt
    
    -- 3. 用EXPLAIN分析异常SQL的执行计划，确认是否全表扫描、索引失效、文件排序
    EXPLAIN SELECT * FROM user_info WHERE name LIKE '%张三%';
```
    
3. **长效解决方案**
    
    - **核心优化：索引优化**：给高频 SQL 建立合适的联合索引、覆盖索引，消除全表扫描、文件排序、临时表；
    - **SQL 改写优化**：优化不合理的 SQL 写法，避免索引列函数运算、左模糊匹配、大表 JOIN、深分页等问题；
    - **业务优化**：高频查询加 Redis 缓存，避免频繁访问数据库；大事务拆分为小事务；离线统计放到从库执行；
    - **参数优化**：设置合理的`innodb_buffer_pool_size`（服务器内存的 50%-70%），减少磁盘 IO；
    - **硬件升级**：CPU 核心数不足时，升级服务器配置。
    

#### 避坑指南

- 禁止直接重启数据库解决 CPU 飙升，重启后业务流量进来，问题会立刻复现，必须从根源优化 SQL 和索引；
- 禁止在业务高峰期执行大表 DDL、批量更新操作，会导致 CPU/IO 飙升；
- 不要盲目调大线程数，过多的线程会导致 CPU 上下文切换开销剧增。

### 3.2 问题 2：内存占用过高 / OOM 宕机

#### 故障现象

MySQL 进程内存占用持续增长，最终触发系统 OOM（Out of Memory），进程被系统 kill，数据库宕机。

#### 核心根因

1. **`innodb_buffer_pool_size`设置过大**：超过服务器内存的 70%，加上其他内存开销，导致内存溢出；
2. **连接数过多**：每个连接都会占用内存，`max_connections=10000`时，即使空闲连接也会占用大量内存；
3. **大内存操作**：大表全表查询、超大事务、批量排序操作，占用大量临时内存；
4. **内存泄漏**：MySQL 版本 bug，或第三方插件内存泄漏；
5. **系统配置问题**：swap 分区未开启 / 过小，OOM killer 优先级设置不当。

#### 分步排查与解决方案

1. **应急处理**：立即重启 MySQL 服务，恢复业务，同时调整`max_connections`限制，避免连接数过多。
2. **根因定位**
    
    - 查看 MySQL 内存配置：`SHOW VARIABLES LIKE 'innodb_buffer_pool_size';`，确认是否超过服务器内存的 70%；
    - 查看当前连接数：`SHOW STATUS LIKE 'Threads_connected';`，确认是否连接数过多；
    - 查看系统内存状态：`free -m`、`top`，确认 MySQL 进程内存占用；
    - 查看系统 OOM 日志：`dmesg | grep -i oom`，确认是否被 OOM killer kill。
    
3. **长效解决方案**
    
    - 合理设置缓冲池：`innodb_buffer_pool_size` 不超过服务器内存的 70%，预留足够的系统内存和连接内存；
    - 控制连接数：`max_connections` 设为合理值（500-2000），避免过大；
    - 优化大内存操作：避免大表全表查询，超大事务拆分，排序分组加索引；
    - 开启 swap 分区：设置 2-4G 的 swap 分区，避免直接触发 OOM；
    - 升级 MySQL 版本：修复已知的内存泄漏 bug。

### 3.3 问题 3：磁盘 IO 使用率 100%

#### 故障现象

服务器磁盘 IO 使用率持续 100%，`iowait`极高，数据库读写响应极慢，SQL 执行超时。

#### 核心根因

1. **大量随机 IO**：无索引的 SQL 频繁执行，全表扫描导致大量磁盘随机读；
2. **redo log/binlog 刷盘频繁**：`sync_binlog=1`、`innodb_flush_log_at_trx_commit=1` 双 1 配置下，高并发小事务导致频繁刷盘；
3. **大事务 / 批量写入**：大批量 INSERT/UPDATE/DELETE，导致大量 IO 写入；
4. **缓冲池设置过小**：`innodb_buffer_pool_size` 太小，热点数据无法缓存，频繁磁盘 IO；
5. **磁盘硬件瓶颈**：机械硬盘（HDD）性能不足，无法支撑高并发 IO。

#### 分步排查与解决方案

1. **根因定位**
    
    - 用`iostat -x 1`查看磁盘 IO 状态，确认`%util`是否 100%，`rMB/s/wMB/s`读写情况；
    - 用`SHOW FULL PROCESSLIST`定位正在执行的大量读写 SQL；
    - 查看慢查询日志，找到大量扫描行的慢 SQL。
    
2. **解决方案**
    
    - 索引优化：给高频读写 SQL 建立合适的索引，消除全表扫描，减少随机 IO；
    - 缓冲池优化：调大`innodb_buffer_pool_size`，缓存更多热点数据，减少磁盘 IO；
    - 写入优化：大事务拆分为小事务，批量写入合并，减少刷盘次数；
    - 硬件升级：将机械硬盘更换为 SSD/NVMe 硬盘，IO 性能提升 10 倍以上；
    - 架构优化：读写分离，读请求分散到从库，减少主库 IO 压力。

### 3.4 问题 4：磁盘空间爆满

#### 故障现象

服务器磁盘使用率 100%，MySQL 无法写入，甚至无法启动，业务写入失败。

#### 核心根因

1. **binlog 日志未清理**：binlog 开启后未设置过期时间，长期积累占用大量磁盘空间；
2. **慢查询日志 / 错误日志过大**：日志长期未清理，文件过大；
3. **大表数据量过大**：单表数据量上亿，冷热数据未分离，历史数据未归档；
4. **临时文件过大**：大查询 / 排序操作生成的临时文件未及时清理；
5. **ibdata1 系统表空间膨胀**：开启了共享表空间，大量数据写入导致 ibdata1 文件过大，无法收缩。

#### 分步排查与解决方案

1. **应急处理：快速释放磁盘空间**

```bash
    # 1. 查看磁盘占用情况，定位大文件
    df -h
    du -sh /var/lib/mysql/* | sort -hr
    
    # 2. 清理过期binlog（MySQL内执行，禁止直接rm删除binlog文件！）
    PURGE BINARY LOGS BEFORE '2026-04-01 00:00:00'; -- 清理指定时间前的binlog
    PURGE BINARY LOGS TO 'mysql-bin.000100'; -- 清理指定文件前的binlog
```
    
⚠️ 绝对禁止直接用 rm 命令删除 binlog 文件，会导致主从复制崩溃，必须用 PURGE 命令清理。
    
2. **长效解决方案**
    
    - 设置 binlog 自动过期：`my.cnf`中设置`expire_logs_days=7`，自动清理 7 天前的 binlog；
    - 冷热数据分离：历史数据归档到归档表 / 数据仓库，删除主表的无效数据；
    - 日志轮转：配置 logrotate，自动切割、压缩、清理慢查询日志、错误日志；
    - 共享表空间优化：新库必须开启`innodb_file_per_table=ON`，每个表独立表空间，避免 ibdata1 膨胀；
    - 临时文件优化：设置合理的`tmpdir`，大查询放到空间充足的分区，自动清理临时文件。

## 四、第三大类：SQL 执行与性能问题排查

### 4.1 问题 1：SQL 执行慢 / 慢查询频发

#### 故障现象

SQL 执行耗时超过 1 秒，接口响应超时，慢查询日志中大量慢 SQL 记录。

#### 标准化排查流程

1. **捕获慢 SQL**：通过慢查询日志、`SHOW FULL PROCESSLIST` 拿到完整的慢 SQL 语句；
2. **分析执行计划**：用`EXPLAIN`分析 SQL 执行计划，重点看 6 个核心字段：
    
    - `type`：是否出现`ALL`全表扫描，是否达到`range/ref`级别以上；
    - `key`：是否命中了预期的索引，`key=NULL`代表索引失效；
    - `key_len`：索引利用率是否足够；
    - `rows`：扫描行数是否远大于返回行数；
    - `Extra`：是否出现`Using filesort/Using temporary/Using where`；
    
3. **定位根因**：90% 的慢 SQL 根因是无索引、索引失效、联合索引顺序设计错误、回表次数过多；
4. **优化方案**：
    
    - 索引优化：设计合适的联合索引、覆盖索引，消除全表扫描和回表；
    - SQL 改写：避免索引列函数运算、隐式类型转换、左模糊匹配、OR 连接非索引列；
    - 业务优化：深分页优化、大事务拆分、冷热数据分离、缓存兜底。
    

详细优化方案可参考之前的《索引优化技巧》《慢查询优化流程》笔记。

---

### 4.2 问题 2：索引失效，查询走全表扫描

#### 故障现象

已经给字段建了索引，但 EXPLAIN 显示`type=ALL`全表扫描，`key=NULL`，索引完全失效。

#### 高频失效场景与解决方案

|索引失效场景|示例|解决方案|
|:--|:--|:--|
|索引列使用函数 / 表达式|`WHERE YEAR(create_time) = 2026`|改写 SQL，把操作移到值一侧：`WHERE create_time BETWEEN '2026-01-01' AND '2026-12-31'`|
|隐式类型转换|`WHERE phone = 13800138000`（phone 是 VARCHAR）|保持类型一致：`WHERE phone = '13800138000'`|
|左模糊 / 全模糊匹配|`WHERE name LIKE '%张三%'`|前缀匹配`LIKE '张%'`，或用全文索引 / 搜索引擎|
|OR 连接非索引列|`WHERE name='张三' OR address='北京'`（address 无索引）|用 UNION ALL 拆分查询，或给 address 建联合索引|
|联合索引不满足最左前缀原则|索引`idx_name_age (name,age)`，查询`WHERE age=20`|调整联合索引顺序，或补充查询条件，满足最左前缀|
|索引列使用 IS NOT NULL|`WHERE name IS NOT NULL`|设计覆盖索引，或避免用 IS NOT NULL 做主要查询条件|
|MySQL 优化器误判|索引区分度过低，优化器认为全表扫描比索引查询更快|强制索引`FORCE INDEX(idx_name)`，或更新表统计信息`ANALYZE TABLE 表名;`|

## 五、第四大类：锁等待与死锁问题排查

### 5.1 问题 1：Lock wait timeout exceeded 行锁等待超时

#### 故障现象

执行 INSERT/UPDATE/DELETE 时报错：`ERROR 1205 (HY000): Lock wait timeout exceeded; try restarting transaction`，行锁等待超时。

#### 核心根因

1. 长事务未提交，持有行锁不释放，其他事务等待超时；
2. 高并发更新同一行，导致锁等待堆积；
3. 无索引导致全表扫描，升级为表锁，所有行都被锁定；
4. 事务隔离级别过高（如 SERIALIZABLE），锁范围过大。

#### 分步排查与解决方案

1. **查看锁等待信息**

```bash
    -- 1. 查看InnoDB事务状态，找到持有锁的长事务
    SHOW ENGINE INNODB STATUS;
    
    -- 2. 查看当前正在执行的事务，找到长时间未提交的事务
    SELECT * FROM information_schema.INNODB_TRX;
    
    -- 3. 查看锁等待详情
    SELECT * FROM information_schema.INNODB_LOCK_WAITS;
```
    
2. **应急处理**：kill 持有锁的长事务，释放锁：`KILL 事务对应的线程ID;`
3. **长效解决方案**
    
    - 禁止长事务：所有事务必须快速提交，禁止事务内包含接口调用、用户等待等长耗时操作；
    - 索引优化：给 WHERE 条件的字段建索引，避免全表扫描导致的表锁；
    - 业务优化：高并发更新同一行的场景，用分布式锁串行化，或合并更新操作；
    - 合理设置锁超时：`innodb_lock_wait_timeout` 默认 50 秒，可根据业务调整为 5-10 秒。

---

### 5.2 问题 2：Deadlock found when trying to get lock 死锁

#### 故障现象

执行 SQL 时报错：`ERROR 1213 (40001): Deadlock found when trying to get lock; try restarting transaction`，事务被回滚。

#### 死锁核心原理

两个事务互相持有对方需要的锁，且都不释放自己的锁，形成循环等待，InnoDB 会自动检测死锁，回滚代价更小的事务。

#### 分步排查与解决方案

1. **开启死锁日志**

```sql
    -- 开启死锁日志，永久生效需写到my.cnf
    SET GLOBAL innodb_print_all_deadlocks = 'ON';
```
    
2. **分析死锁日志**
    
    - 死锁信息会记录到 MySQL 错误日志中，或通过`SHOW ENGINE INNODB STATUS`查看；
    - 重点分析：两个事务分别持有什么锁、等待什么锁、执行的什么 SQL，找到循环等待的根源。
    
3. **高频死锁场景与解决方案**

| 死锁场景              | 解决方案                           |
| :---------------- | :----------------------------- |
| 两个事务按相反顺序更新同一批行   | 统一事务的更新顺序，所有事务都按主键从小到大更新       |
| 高并发下唯一键冲突导致的间隙锁死锁 | 降低事务隔离级别为 READ COMMITTED，关闭间隙锁 |
| 无索引导致的表锁，多事务互相等待  | 给查询条件建索引，避免全表扫描和表锁             |
| 长事务持有锁，导致死锁概率升高   | 拆分长事务，保证事务快速提交                 |

---

## 六、第五大类：主从复制问题排查

### 6.1 问题 1：主从延迟过大 / 持续增长

#### 故障现象

`SHOW SLAVE STATUS\G` 中 `Seconds_Behind_Master` 数值持续增长，从库数据落后主库，读写分离场景下读到旧数据。

#### 核心根因与解决方案

|根因|解决方案|
|:--|:--|
|主库大事务 / 批量写入，binlog 过大，从库重放慢|大事务拆分为小事务，批量写入分批执行|
|从库 SQL 线程单线程，重放速度跟不上主库写入|MySQL 5.7 + 开启并行复制：`slave_parallel_type=LOGICAL_CLOCK`，`slave_parallel_workers=8`|
|从库硬件配置比主库差，CPU/IO 性能不足|从库配置与主库保持一致，或升级从库硬件|
|从库有大查询 / 慢 SQL，占用大量资源，阻塞 SQL 线程|从库开启只读，大查询放到专用的只读从库执行，优化慢 SQL|
|主库 DDL 操作（大表加字段），从库重放阻塞|业务低峰期执行 DDL，用 Online DDL/gh-ost 工具执行大表 DDL|
|从库表无主键 / 唯一索引，更新时全表扫描|所有表必须显式设置自增主键，避免无主键表|

详细排查方案可参考之前的《MySQL 主从复制》笔记。

---

### 6.2 问题 2：Slave_IO_Running: No

#### 故障现象

`SHOW SLAVE STATUS\G` 中 `Slave_IO_Running: No`，从库无法接收主库的 binlog，复制中断。

#### 核心根因与解决方案

1. **网络不通 / 主库端口无法访问**：检查主从网络连通性，防火墙开放 3306 端口；
2. **主从 server-id 重复**：主从库的 server-id 必须全局唯一，修改 my.cnf 的 server-id，重启服务；
3. **复制用户权限错误 / 密码错误**：重新给复制用户授权，确认从库`CHANGE MASTER TO`中的密码正确；
4. **主库 binlog 文件已删除**：主库的 binlog 已过期清理，从库找不到对应的 binlog，重新搭建主从复制；
5. **主库 binlog 格式不兼容**：主从库 binlog 格式不一致，统一设置为 ROW 格式。

---

### 6.3 问题 3：Slave_SQL_Running: No

#### 故障现象

`SHOW SLAVE STATUS\G` 中 `Slave_SQL_Running: No`，从库无法重放 relay log，复制中断。

#### 核心根因与解决方案

1. **主从数据不一致**：从库手动写入了数据，主库的操作在从库执行时出现主键冲突 / 唯一键冲突；
    
    - 应急方案：跳过错误（测试环境用，生产环境不推荐）：

```sql
        STOP SLAVE;
        SET GLOBAL sql_slave_skip_counter = 1;
        START SLAVE;
```
        
- 生产环境方案：重新搭建主从复制，保证主从数据一致，开启从库只读。
    
2. **主库执行了从库不支持的操作**：主库创建了从库不存在的库 / 表，或执行了从库没有权限的操作；
    
    - 解决方案：在从库手动创建对应的库 / 表，恢复复制。
    
3. **从库 relay log 损坏**：服务器宕机导致 relay log 损坏；
    
    - 解决方案：重新搭建主从复制，或重新同步主库的 binlog。

## 七、第六大类：MySQL 服务启停与崩溃问题排查

### 7.1 问题 1：MySQL 服务启动失败

#### 故障现象

`systemctl start mysqld` 执行失败，MySQL 无法启动，业务完全不可用。

#### 分步排查流程

1. **第一步：查看错误日志（必做）**
    
    错误日志是启动失败的唯一排查依据，先找到错误日志路径：`cat /etc/my.cnf | grep log_error`，查看日志中的报错信息。
2. **高频启动失败根因与解决方案**

| 报错根因                     | 解决方案                                                                       |
| :----------------------- | :------------------------------------------------------------------------- |
| 配置文件 my.cnf 语法错误 / 参数不兼容 | 修正 my.cnf 中的错误配置，删除不兼容的参数，重启服务                                             |
| 数据目录 / 日志目录权限错误          | 修正目录权限：`chown -R mysql:mysql /var/lib/mysql`，`chmod -R 750 /var/lib/mysql` |
| 端口 3306 被占用              | 查看占用进程：`ss -tulnp \| grep 3306`，kill 占用进程，或修改 MySQL 端口                     |
| 磁盘空间满，无法创建 pid 文件 / 日志文件 | 清理磁盘空间，释放存储空间                                                              |
| 数据文件 / 系统表空间损坏           | 用 InnoDB 强制恢复模式启动，备份数据后重建实例                                                |
| 内存不足，无法分配缓冲池             | 调小`innodb_buffer_pool_size`，或增加服务器内存                                       |

---

### 7.2 问题 2：MySQL 异常崩溃 / 自动重启

#### 故障现象

MySQL 服务运行中突然崩溃，自动重启，业务短暂中断，错误日志中有崩溃信息。

#### 核心根因与解决方案

1. **系统 OOM kill**：MySQL 内存占用过大，被系统 OOM killer kill；
    
    - 解决方案：调小`innodb_buffer_pool_size`，控制连接数，开启 swap 分区，调整 OOM killer 优先级。
    
2. **MySQL 版本 bug**：已知的内核 bug 导致崩溃；
    
    - 解决方案：升级到稳定的 MySQL 版本，如 5.7.44、8.0.36+。
    
3. **硬件故障**：磁盘损坏、内存故障、CPU 异常；
    
    - 解决方案：检查服务器硬件日志，更换故障硬件。
    
4. **数据文件 /redo log 损坏**：磁盘 IO 异常导致文件损坏；
    
    - 解决方案：用备份恢复数据，重建实例。

## 八、生产环境故障应急处理黄金法则

1. **先止损，再排查**：第一优先级恢复业务，而非死磕根因。比如 CPU 打满先 kill 异常 SQL，主库故障先切换到从库，业务恢复后再复盘；
2. **操作可回滚**：所有操作必须有回滚方案，禁止在生产环境执行不可逆操作（如 rm 删除文件、DROP 表、强制恢复数据）；
3. **禁止盲目重启**：重启会丢失现场（进程列表、锁信息、事务状态），导致无法定位根因，除非业务完全不可用，否则不要第一时间重启；
4. **保留现场**：故障发生后，第一时间备份错误日志、慢查询日志、binlog、系统监控数据，便于后续根因分析；
5. **分级处理**：根据业务影响范围分级处理，核心业务故障优先处理，非核心业务可先限流再排查；
6. **事后复盘**：故障恢复后，必须做完整的根因分析，制定优化方案和应急预案，避免同类故障再次发生。

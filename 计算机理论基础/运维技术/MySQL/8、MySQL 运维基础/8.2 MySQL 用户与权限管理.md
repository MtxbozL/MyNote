## 一、开篇：用户与权限的核心定位与前置说明

### 1. 为什么用户与权限管理如此重要

用户与权限管理是 MySQL 数据安全的**第一道防线**，核心目标是：

- **数据安全**：防止未授权访问、误操作、数据泄露，保证数据的保密性、完整性、可用性；
- **权限最小化**：遵循「最小权限原则」，用户只拥有完成工作所需的最小权限，避免过度授权；
- **责任追溯**：不同业务 / 角色使用不同的用户，出现问题时可通过用户追溯责任；
- **资源隔离**：不同用户的操作隔离，避免相互影响。

### 2. 前置说明

- **版本差异**：MySQL 5.7 和 8.0 在用户管理上有较大差异，本笔记会分别说明，默认基于 MySQL 5.7，同时标注 8.0 的变化；
- **安全红线**：生产环境绝对禁止使用 root 用户直接操作业务，绝对禁止过度授权，绝对禁止弱密码；
- **权限生效**：授权 / 撤销权限后，通常需要执行 `FLUSH PRIVILEGES` 刷新权限（部分场景自动生效，手动刷新更稳妥）。

## 二、第一部分：用户管理

### 2.1 创建用户

#### 核心语法

```sql
-- 通用语法
CREATE USER '用户名'@'主机' IDENTIFIED BY '密码';
```

|参数|说明|示例|
|:--|:--|:--|
|`用户名`|登录 MySQL 的用户名|`dev_user`、`read_user`|
|`主机`|允许登录的主机地址，`%` 表示任意主机，`192.168.1.100` 表示指定 IP，`localhost` 表示本地|`%`、`192.168.1.%`（指定网段）、`localhost`|
|`密码`|登录密码，生产环境必须使用强密码（大小写 + 数字 + 特殊字符，长度≥12 位）|`Dev@123456`|

#### 完整示例

```sql
-- 1. 创建允许任意主机登录的用户（生产环境谨慎使用%）
CREATE USER 'dev_user'@'%' IDENTIFIED BY 'Dev@123456';

-- 2. 创建仅允许指定IP登录的用户（更安全，推荐）
CREATE USER 'dev_user'@'192.168.1.100' IDENTIFIED BY 'Dev@123456';

-- 3. 创建仅允许本地登录的用户（适合运维/备份）
CREATE USER 'backup_user'@'localhost' IDENTIFIED BY 'Backup@123456';

-- 4. MySQL 5.7 兼容写法：用GRANT同时创建用户和授权（⚠️ MySQL 8.0 已废弃，必须先CREATE USER再GRANT）
-- MySQL 5.7 可用
GRANT SELECT ON my_database.* TO 'read_user'@'%' IDENTIFIED BY 'Read@123456';
```

#### 安全提示

⚠️ **生产环境绝对禁止使用 `%` 作为主机**，除非是测试环境，否则会允许任意 IP 访问，存在极大的安全风险；必须指定具体的 IP 或网段。

---

### 2.2 查看用户

```sql
-- 1. 查看所有用户（MySQL 5.7+，用户信息存储在mysql.user表）
SELECT user, host, authentication_string FROM mysql.user;

-- 2. 查看当前登录用户
SELECT USER();
SELECT CURRENT_USER();

-- 3. 查看用户的详细权限（见第三部分）
SHOW GRANTS FOR 'dev_user'@'%';
```

---

### 2.3 修改用户

#### 2.3.1 修改用户密码

```sql
-- 方式1：推荐（MySQL 5.7+）
ALTER USER 'dev_user'@'%' IDENTIFIED BY 'NewDev@123456';

-- 方式2：SET PASSWORD（兼容旧版本）
SET PASSWORD FOR 'dev_user'@'%' = PASSWORD('NewDev@123456');

-- 修改当前用户密码
ALTER USER USER() IDENTIFIED BY 'NewDev@123456';
```

#### 2.3.2 修改用户的主机

```sql
-- 方式1：RENAME USER（推荐，MySQL 5.7+）
RENAME USER 'dev_user'@'%' TO 'dev_user'@'192.168.1.100';

-- 方式2：直接更新mysql.user表（不推荐，需FLUSH PRIVILEGES）
UPDATE mysql.user SET host = '192.168.1.100' WHERE user = 'dev_user' AND host = '%';
FLUSH PRIVILEGES;
```

#### 2.3.3 修改用户名

```sql
-- RENAME USER（推荐）
RENAME USER 'dev_user'@'%' TO 'new_dev_user'@'%';
```

#### 2.3.4 锁定 / 解锁用户（MySQL 5.7+）

```sql
-- 锁定用户（禁止登录）
ALTER USER 'dev_user'@'%' ACCOUNT LOCK;

-- 解锁用户
ALTER USER 'dev_user'@'%' ACCOUNT UNLOCK;
```

---

### 2.4 删除用户

⚠️ **危险操作**：删除用户后，该用户的所有权限会被一并删除，无法恢复！

```sql
-- 基础删除
DROP USER 'dev_user'@'%';

-- 安全删除（避免用户不存在报错）
DROP USER IF EXISTS 'dev_user'@'%';
```

## 三、第二部分：权限管理（核心重点）

### 3.1 权限的层级

MySQL 权限按范围从大到小分为 5 个层级，不同层级的权限存储在不同的系统表中：

| 权限层级     | 存储表                  | 说明                   | 示例                                     |
| :------- | :------------------- | :------------------- | :------------------------------------- |
| **全局权限** | `mysql.user`         | 对所有库、所有表生效，权限最大，谨慎授权 | `*.*`                                  |
| **库级权限** | `mysql.db`           | 对指定库的所有表生效           | `my_database.*`                        |
| **表级权限** | `mysql.tables_priv`  | 对指定库的指定表生效           | `my_database.user_info`                |
| **列级权限** | `mysql.columns_priv` | 对指定库的指定表的指定列生效       | `my_database.user_info(id, user_name)` |
| **例程权限** | `mysql.procs_priv`   | 对存储过程、函数生效           | 较少使用                                   |

> 核心原则：**权限层级越小越好**，能用表级就不用库级，能用库级就不用全局，严格遵循最小权限原则。

---

### 3.2 常用权限说明

|权限|说明|适用场景|
|:--|:--|:--|
|`SELECT`|查询数据|只读用户、开发用户|
|`INSERT`|插入数据|开发用户、业务用户|
|`UPDATE`|更新数据|开发用户、业务用户|
|`DELETE`|删除数据|开发用户、业务用户（谨慎授权）|
|`CREATE`|创建库、表|开发用户（测试环境）|
|`ALTER`|修改表结构|开发用户（测试环境）|
|`DROP`|删除库、表|谨慎授权，生产环境禁止|
|`INDEX`|创建、删除索引|开发用户、运维用户|
|`REFERENCES`|创建外键|较少使用|
|`CREATE TEMPORARY TABLES`|创建临时表|开发用户|
|`LOCK TABLES`|锁表|开发用户、备份用户|
|`EXECUTE`|执行存储过程、函数|较少使用|
|`PROCESS`|查看所有进程|运维用户|
|`RELOAD`|刷新权限、日志等|运维用户|
|`FILE`|读写服务器文件|⚠️ 危险权限，禁止授权|
|`SUPER`|超级权限（修改全局变量、终止进程等）|⚠️ 危险权限，禁止授权|
|`ALL PRIVILEGES`|所有权限（除了 GRANT OPTION）|⚠️ 谨慎授权，仅 root 或运维用户可用|
|`GRANT OPTION`|授权权限（可以把自己的权限授予其他用户）|⚠️ 危险权限，禁止授权|

---

### 3.3 授权（GRANT）

#### 核心语法

```sql
-- 通用语法
GRANT 权限列表 ON 权限层级 TO '用户名'@'主机' [WITH GRANT OPTION];
-- 授权后必须刷新权限
FLUSH PRIVILEGES;
```

#### 完整示例

```sql
-- 1. 全局权限（⚠️ 生产环境谨慎授权，仅运维用户可用）
GRANT ALL PRIVILEGES ON *.* TO 'admin_user'@'%' WITH GRANT OPTION;
FLUSH PRIVILEGES;

-- 2. 库级权限（推荐，开发/业务用户常用）
-- 给dev_user授予my_database库的所有表的SELECT、INSERT、UPDATE、DELETE权限
GRANT SELECT, INSERT, UPDATE, DELETE ON my_database.* TO 'dev_user'@'192.168.1.100';
FLUSH PRIVILEGES;

-- 3. 表级权限（更安全，最小权限原则）
-- 给read_user授予my_database库的user_info表的SELECT权限
GRANT SELECT ON my_database.user_info TO 'read_user'@'%';
FLUSH PRIVILEGES;

-- 4. 列级权限（最精细，极少使用）
-- 给read_user授予my_database库的user_info表的id、user_name列的SELECT权限
GRANT SELECT(id, user_name) ON my_database.user_info TO 'read_user'@'%';
FLUSH PRIVILEGES;
```

#### 安全提示

⚠️ **绝对禁止使用 `WITH GRANT OPTION`**，除非是 root 用户，否则用户可以把自己的权限授予其他用户，导致权限失控。

---

### 3.4 查看权限

```sql
-- 1. 查看当前登录用户的权限
SHOW GRANTS;

-- 2. 查看指定用户的权限
SHOW GRANTS FOR 'dev_user'@'192.168.1.100';

-- 3. 从系统表查看用户权限（更详细）
SELECT * FROM mysql.user WHERE user = 'dev_user'; -- 全局权限
SELECT * FROM mysql.db WHERE user = 'dev_user'; -- 库级权限
SELECT * FROM mysql.tables_priv WHERE user = 'dev_user'; -- 表级权限
```

---

### 3.5 撤销权限（REVOKE）

#### 核心语法

```sql
-- 通用语法
REVOKE 权限列表 ON 权限层级 FROM '用户名'@'主机';
-- 撤销后必须刷新权限
FLUSH PRIVILEGES;
```

#### 完整示例

```sql
-- 1. 撤销全局权限
REVOKE ALL PRIVILEGES, GRANT OPTION FROM 'admin_user'@'%';
FLUSH PRIVILEGES;

-- 2. 撤销库级权限
REVOKE INSERT, UPDATE, DELETE ON my_database.* FROM 'dev_user'@'192.168.1.100';
FLUSH PRIVILEGES;

-- 3. 撤销表级权限
REVOKE SELECT ON my_database.user_info FROM 'read_user'@'%';
FLUSH PRIVILEGES;
```

---

### 3.6 刷新权限（FLUSH PRIVILEGES）

#### 什么时候需要刷新权限

1. 直接修改 `mysql.user`、`mysql.db` 等系统表后，必须刷新；
2. 使用 `GRANT`、`REVOKE`、`CREATE USER`、`DROP USER`、`ALTER USER` 后，MySQL 5.7+ 通常会自动刷新，但手动刷新更稳妥；
3. 权限不生效时，优先刷新权限。

#### 语法

```sql
FLUSH PRIVILEGES;
```

## 四、第三部分：权限管理核心原则

### 4.1 最小权限原则

这是权限管理的**第一原则**，用户只拥有完成工作所需的最小权限，不多不少：

- 只读用户只给 `SELECT` 权限；
- 开发用户只给指定库的 `SELECT`、`INSERT`、`UPDATE`、`DELETE`、`INDEX` 权限，不给 `CREATE`、`ALTER`、`DROP` 权限（测试环境可适当放宽）；
- 业务用户只给指定库的指定表的必要权限；
- 绝对不给 `FILE`、`SUPER`、`ALL PRIVILEGES`、`GRANT OPTION` 等危险权限。

### 4.2 禁止 root 用户远程登录

root 用户是 MySQL 的超级管理员，拥有所有权限，一旦泄露后果不堪设想：

- 生产环境必须禁止 root 用户远程登录，只允许 `localhost` 本地登录；
- 业务操作、开发操作、运维操作都使用专门的用户，禁止使用 root。

```sql
-- 修改root用户的主机为localhost，禁止远程登录
RENAME USER 'root'@'%' TO 'root'@'localhost';
FLUSH PRIVILEGES;
```

### 4.3 严格限制登录主机

不要使用 `%` 作为主机，必须指定具体的 IP 或网段：

- 开发用户只允许公司内网 IP 登录；
- 业务用户只允许应用服务器 IP 登录；
- 运维用户只允许运维堡垒机 IP 登录。

### 4.4 密码安全原则

1. **强密码**：所有用户必须使用强密码，包含大小写字母、数字、特殊字符，长度≥12 位；
2. **定期修改**：定期（如每 3 个月）修改用户密码；
3. **密码加密**：MySQL 5.7+ 默认使用 `mysql_native_password`，8.0 默认使用 `caching_sha2_password`，不要使用旧的密码插件；
4. **禁止密码泄露**：不要在代码、配置文件中明文存储密码，使用加密配置中心。

### 4.5 定期审计原则

1. **定期查看用户列表**：清理不再使用的用户；
2. **定期查看权限列表**：撤销过度授权的权限；
3. **定期查看登录日志**：发现异常登录（如陌生 IP、非工作时间登录）及时处理；
4. **定期修改密码**：按周期修改所有用户的密码。

## 五、第四部分：常用业务场景权限示例

### 5.1 开发用户（测试环境）

```sql
-- 创建用户
CREATE USER 'dev_user'@'192.168.1.%' IDENTIFIED BY 'Dev@123456';
-- 授权：指定库的SELECT、INSERT、UPDATE、DELETE、CREATE、ALTER、INDEX权限
GRANT SELECT, INSERT, UPDATE, DELETE, CREATE, ALTER, INDEX ON my_database.* TO 'dev_user'@'192.168.1.%';
FLUSH PRIVILEGES;
```

### 5.2 只读用户（报表 / 数据分析）

```sql
-- 创建用户
CREATE USER 'read_user'@'192.168.1.200' IDENTIFIED BY 'Read@123456';
-- 授权：指定库的所有表的SELECT权限
GRANT SELECT ON my_database.* TO 'read_user'@'192.168.1.200';
FLUSH PRIVILEGES;
```

### 5.3 业务用户（生产环境应用服务器）

```sql
-- 创建用户
CREATE USER 'app_user'@'10.0.0.10' IDENTIFIED BY 'App@123456';
-- 授权：指定库的指定表的SELECT、INSERT、UPDATE、DELETE权限（最小权限，只给需要的表）
GRANT SELECT, INSERT, UPDATE, DELETE ON my_database.user_info TO 'app_user'@'10.0.0.10';
GRANT SELECT, INSERT, UPDATE, DELETE ON my_database.order_info TO 'app_user'@'10.0.0.10';
FLUSH PRIVILEGES;
```

### 5.4 备份用户

```sql
-- 创建用户
CREATE USER 'backup_user'@'localhost' IDENTIFIED BY 'Backup@123456';
-- 授权：全局SELECT、LOCK TABLES、RELOAD权限
GRANT SELECT, LOCK TABLES, RELOAD ON *.* TO 'backup_user'@'localhost';
FLUSH PRIVILEGES;
```

### 5.5 运维用户

```sql
-- 创建用户
CREATE USER 'ops_user'@'192.168.1.250' IDENTIFIED BY 'Ops@123456';
-- 授权：全局PROCESS、RELOAD、SELECT权限（不给SUPER、ALL PRIVILEGES）
GRANT PROCESS, RELOAD, SELECT ON *.* TO 'ops_user'@'192.168.1.250';
FLUSH PRIVILEGES;
```

## 六、第五部分：MySQL 5.7 vs 8.0 用户与权限差异

### 6.1 创建用户与授权分离

- **MySQL 5.7**：可以用 `GRANT` 同时创建用户和授权；
- **MySQL 8.0**：必须先 `CREATE USER` 创建用户，再 `GRANT` 授权，否则报错。

```sql
-- MySQL 5.7 可用
GRANT SELECT ON my_database.* TO 'read_user'@'%' IDENTIFIED BY 'Read@123456';

-- MySQL 8.0 必须分开写
CREATE USER 'read_user'@'%' IDENTIFIED BY 'Read@123456';
GRANT SELECT ON my_database.* TO 'read_user'@'%';
FLUSH PRIVILEGES;
```

### 6.2 密码插件变化

- **MySQL 5.7**：默认密码插件是 `mysql_native_password`；
- **MySQL 8.0**：默认密码插件是 `caching_sha2_password`，安全性更高，但旧版本客户端可能不兼容，如需兼容可修改为 `mysql_native_password`。

```sql
-- MySQL 8.0 修改用户密码插件为mysql_native_password
ALTER USER 'dev_user'@'%' IDENTIFIED WITH mysql_native_password BY 'Dev@123456';
```

### 6.3 角色功能（MySQL 8.0 新增）

MySQL 8.0 新增了角色功能，可以把权限授予角色，再把角色授予用户，简化权限管理：

```sql
-- 1. 创建角色
CREATE ROLE 'dev_role';

-- 2. 给角色授权
GRANT SELECT, INSERT, UPDATE, DELETE ON my_database.* TO 'dev_role';

-- 3. 创建用户
CREATE USER 'dev_user1'@'%' IDENTIFIED BY 'Dev1@123456';
CREATE USER 'dev_user2'@'%' IDENTIFIED BY 'Dev2@123456';

-- 4. 把角色授予用户
GRANT 'dev_role' TO 'dev_user1'@'%', 'dev_user2'@'%';

-- 5. 用户激活角色（登录后执行）
SET ROLE 'dev_role';

-- 6. 查看用户的角色
SHOW GRANTS FOR 'dev_user1'@'%';
```

## 八、核心要点总结

1. **安全第一**：用户与权限管理是数据安全的第一道防线，必须严格遵守安全规范；
2. **最小权限原则**：用户只拥有完成工作所需的最小权限，不多不少，严格控制权限层级；
3. **禁止 root 远程登录**：生产环境 root 只允许本地登录，所有操作使用专门用户；
4. **严格限制主机**：不要使用 `%`，必须指定具体的 IP 或网段；
5. **密码安全**：强密码、定期修改、禁止明文存储；
6. **版本差异**：MySQL 8.0 必须先创建用户再授权，新增角色功能；
7. **定期审计**：定期查看用户、权限、登录日志，清理不再使用的用户和权限；
8. **授权后刷新**：授权 / 撤销权限后，手动执行 `FLUSH PRIVILEGES` 刷新权限。

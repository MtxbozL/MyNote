本教程基于生产环境实战标准编写，重点解决**大库全量备份耗时长、占用空间大**的痛点，实现高效、低占用的增量备份方案。

## 一、前置核心认知（必看，否则 90% 概率踩坑）

### 1.1 版本兼容铁律

Percona XtraBackup（简称 PXB）的版本与 MySQL 版本强绑定，**版本不匹配会直接导致备份 / 恢复失败**，严格遵循以下对应关系：

|PXB 版本|支持的 MySQL 版本|核心说明|
|:--|:--|:--|
|**PXB 8.0.x**|MySQL 8.0.x|目前主流版本，仅支持 MySQL 8.0，移除了旧版 `innobackupex` 脚本，仅保留 `xtrabackup` 核心命令|
|**PXB 2.4.x**|MySQL 5.6/5.7|兼容 MySQL 5.7 及以下版本，生产环境存量库常用|

> 本教程以 **PXB 8.0 + MySQL 8.0** 为基准，5.7 版本仅标注差异点，核心逻辑完全通用。

### 1.2 增量备份核心原理：LSN 日志序列号

增量备份的底层完全依赖 InnoDB 引擎的 **LSN（Log Sequence Number，日志序列号）**，这是理解增量备份的核心：

1. LSN 是一个 64 位的递增整数，是 InnoDB 数据修改的唯一标识，每一次数据写入 / 更新，LSN 都会全局递增；
2. InnoDB 的每个数据页（默认 16KB）都记录了该页最后一次修改的 LSN；
3. 全量备份会记录备份结束时的全局 LSN（`last_lsn`）；
4. **增量备份的本质**：只备份「上次备份结束 LSN 之后发生变化的数据页」，而非全量数据，因此备份速度极快、占用空间极小。

### 1.3 增量备份 vs 差异备份

增量备份分为两种模式，生产环境需根据恢复复杂度选择：

|备份类型|核心定义|优点|缺点|适用场景|
|:--|:--|:--|:--|:--|
|**连续增量备份**|每次增量备份，都基于**上一次增量备份**的结束 LSN|备份速度最快，备份文件最小，占用磁盘空间最少|恢复流程复杂，必须依次恢复全量 + 所有增量，任意一个增量损坏，后续全部失效|超大型库（>500GB），磁盘空间紧张|
|**差异备份**|每次增量备份，都基于**最初的全量基准备份**的结束 LSN|恢复简单，只需恢复全量 + 最后一次差异备份，安全性高|备份文件随时间逐渐变大，备份速度比连续增量慢|绝大多数生产环境（推荐首选）|

### 1.4 适用场景

- 单库数据量 **>10GB**，全量备份耗时超过 1 小时；
- 业务对停机时间要求极高，不允许每天长时间全量备份；
- 磁盘空间有限，无法每天存储完整的全量备份文件；
- 需要实现按小时 / 按半天的高频备份，同时控制备份对业务的性能影响。

### 2.2 创建备份专用用户（最小权限原则）

遵循之前的权限管理规范，禁止直接使用 root 用户备份，创建专用备份用户，仅授予备份所需的最小权限：

```bash
-- 登录 MySQL 创建备份用户
CREATE USER 'backup_user'@'localhost' IDENTIFIED BY 'Backup@20260401';

-- 授予备份所需权限（MySQL 8.0）
GRANT BACKUP_ADMIN, RELOAD, LOCK TABLES, PROCESS, REPLICATION CLIENT ON *.* TO 'backup_user'@'localhost';

-- MySQL 5.7 权限授予（无 BACKUP_ADMIN 权限）
-- GRANT RELOAD, LOCK TABLES, PROCESS, REPLICATION CLIENT, CREATE TABLESPACE, SUPER ON *.* TO 'backup_user'@'localhost';

-- 刷新权限
FLUSH PRIVILEGES;
```

### 2.3 环境前置检查

1. 确认 MySQL 数据目录权限正常（默认 `/var/lib/mysql`，属主为 `mysql:mysql`）；
2. 确认备份目录有足够的磁盘空间，且属主为 `mysql:mysql`；
3. 确认 MySQL 服务正常运行，`innodb_file_per_table=ON`（MySQL 8.0 默认开启）；
4. 关闭 SELinux（避免权限拦截）：`setenforce 0`，永久关闭需修改 `/etc/selinux/config`。

## 三、第一步：创建增量备份的基准（全量备份）

**增量备份必须基于一个完整的全量备份（基准备份）**，没有全量基准，增量备份无法独立恢复，这是铁律。

### 3.1 执行全量基准备份

```bash
# 1. 创建备份目录（基准全量备份目录）
mkdir -p /backup/xtrabackup/full_base

# 2. 执行全量备份（核心命令）
xtrabackup \
--user=backup_user \
--password='Backup@20260401' \
--backup \
--target-dir=/backup/xtrabackup/full_base/20260401_full \
--parallel=4 \  # 并行备份线程数，根据CPU核心数设置
--compress \     # 开启压缩，大幅减少备份文件大小（可选）
--compress-threads=2

# 命令执行结束，末尾输出 completed OK! 代表备份成功
```

|核心参数|说明|
|:--|:--|
|`--backup`|执行备份操作，固定参数|
|`--target-dir`|备份文件存储目录，必须是空目录|
|`--parallel`|并行备份线程数，提升大库备份速度|
|`--compress`|开启备份压缩，推荐大库使用|

### 3.2 全量备份目录关键文件说明

备份成功后，进入备份目录，重点关注 2 个核心文件，这是增量备份的关键：

```bash
cd /backup/xtrabackup/full_base/20260401_full
cat xtrabackup_checkpoints
```

`xtrabackup_checkpoints` 文件内容示例：

```c
backup_type = full-backuped
from_lsn = 0                  # 备份起始LSN，全量备份为0
to_lsn = 19283746            # 备份结束LSN，增量备份的基准
last_lsn = 19283755          # 备份结束时的全局LSN
compact = 0
compressed = 1
```

- `to_lsn`：全量备份的结束 LSN，后续增量备份都基于这个值；
- 这个文件是判断备份是否完整、增量备份基准的核心依据。

### 3.3 全量备份的预准备（Prepare）基础说明

备份完成后，备份文件里包含了未提交的事务和已提交但未刷入数据页的 redo log，必须经过 **Prepare（预准备）** 步骤，才能让备份数据处于一致性状态，用于恢复。

增量备份的 Prepare 步骤和全量备份有本质区别，**基准全量备份的第一次 Prepare 必须加 `--apply-log-only` 参数**，这是增量恢复成功的核心！

> `--apply-log-only`：仅应用 redo log，不回滚未提交的事务，保证后续增量备份可以正常叠加到当前备份上。如果不加这个参数，后续增量备份无法合并，恢复会直接失败。

## 四、第二步：增量备份实操

我们以**生产环境最常用的连续增量备份**为例，演示完整流程：

- 基准全量备份：周日凌晨 2 点执行；
- 第一次增量：周一凌晨 2 点执行，基于周日的全量备份；
- 第二次增量：周二凌晨 2 点执行，基于周一的增量备份；
- 以此类推，周六执行最后一次增量，周日重新生成新的全量基准。

### 4.1 第一次增量备份（基于全量基准）

```bash
# 1. 创建增量备份目录
mkdir -p /backup/xtrabackup/increment/inc_20260402

# 2. 执行增量备份，核心参数 --incremental-basedir 指定上一次备份的目录
xtrabackup \
--user=backup_user \
--password='Backup@20260401' \
--backup \
--target-dir=/backup/xtrabackup/increment/inc_20260402 \
--incremental-basedir=/backup/xtrabackup/full_base/20260401_full \
--parallel=4
```

- `--incremental-basedir`：增量备份的基准目录，这里填**上一次全量备份的目录**；
- 执行结束输出 `completed OK!` 代表备份成功。

查看增量备份的 `xtrabackup_checkpoints`：

```c
backup_type = incremental
from_lsn = 19283746          # 等于全量备份的 to_lsn，完全衔接
to_lsn = 19289901            # 本次增量备份的结束LSN
last_lsn = 19289910
```

### 4.2 第二次增量备份（基于上一次增量）

```bash
# 1. 创建增量备份目录
mkdir -p /backup/xtrabackup/increment/inc_20260403

# 2. 执行增量备份，--incremental-basedir 指定上一次增量备份的目录
xtrabackup \
--user=backup_user \
--password='Backup@20260401' \
--backup \
--target-dir=/backup/xtrabackup/increment/inc_20260403 \
--incremental-basedir=/backup/xtrabackup/increment/inc_20260402 \
--parallel=4
```

- 核心变化：`--incremental-basedir` 不再指向全量备份，而是**上一次增量备份的目录**，实现连续增量。

查看增量备份的 `xtrabackup_checkpoints`：

```c
backup_type = incremental
from_lsn = 19289901          # 等于上一次增量备份的 to_lsn，完全衔接
to_lsn = 19295000
last_lsn = 19295009
```

> 差异备份的操作区别：如果是差异备份，每次的 `--incremental-basedir` 都指向最初的全量基准备份目录，其他参数完全一致。

## 五、第三步：增量备份完整恢复流程（重点，步骤绝对不能错）

### 场景说明

我们需要恢复：周日的全量基准备份 + 周一的第一次增量 + 周二的第二次增量，恢复到周二备份结束时的数据状态。

### 5.1 恢复前必做准备

1. **停止 MySQL 服务**：恢复前必须关闭 MySQL，禁止在运行状态下恢复数据；

```bash
    systemctl stop mysqld
```
    
2. **备份当前数据目录**：避免恢复失败导致现有数据丢失，必须先备份！

```bash
    mv /var/lib/mysql /var/lib/mysql_bak_$(date +%Y%m%d)
```
    
3. **创建空的数据目录**：恢复需要空的目标目录，MySQL 8.0 不允许在非空目录恢复；

```bash
    mkdir -p /var/lib/mysql
```
    
4. **确认备份文件完整**：检查全量 + 所有增量备份的目录都存在，`xtrabackup_checkpoints` 文件完整。

---

### 5.2 恢复步骤 1：预准备基准全量备份

⚠️ **核心重点**：必须加 `--apply-log-only` 参数，禁止省略！

```bash
xtrabackup \
--prepare \
--apply-log-only \
--target-dir=/backup/xtrabackup/full_base/20260401_full
```

- `--prepare`：执行预准备操作，固定参数；
- 执行结束输出 `completed OK!` 代表预准备成功。

---

### 5.3 恢复步骤 2：依次预准备每一次增量备份

⚠️ **核心重点**：除了最后一次增量备份，其他所有增量备份都必须加 `--apply-log-only` 参数！

#### 合并第一次增量备份（周一）

```bash
xtrabackup \
--prepare \
--apply-log-only \
--target-dir=/backup/xtrabackup/full_base/20260401_full \
--incremental-dir=/backup/xtrabackup/increment/inc_20260402
```

- `--target-dir`：始终指向基准全量备份目录（所有增量都合并到这里）；
- `--incremental-dir`：当前要合并的增量备份目录；
- 执行结束输出 `completed OK!` 代表合并成功。

#### 合并第二次增量备份（周二，最后一次增量）

⚠️ **最后一次增量备份，必须去掉 `--apply-log-only` 参数**！

```bash
xtrabackup \
--prepare \
--target-dir=/backup/xtrabackup/full_base/20260401_full \
--incremental-dir=/backup/xtrabackup/increment/inc_20260403
```

- 去掉 `--apply-log-only` 的目的：回滚未提交的事务，让整个备份数据处于完全一致的状态，可直接用于恢复。

---

### 5.4 恢复步骤 3：将备份恢复到 MySQL 数据目录

```bash
# --copy-back 会将预准备完成的备份文件，复制到 MySQL 的数据目录
xtrabackup \
--copy-back \
--target-dir=/backup/xtrabackup/full_base/20260401_full \
--datadir=/var/lib/mysql
```

- `--datadir`：MySQL 的数据目录，必须和 `my.cnf` 中的 `datadir` 配置一致；
- 执行结束输出 `completed OK!` 代表恢复成功。

---

### 5.5 恢复步骤 4：修复数据目录权限

恢复后的文件属主是执行备份命令的用户（如 root），必须修改为 `mysql:mysql`，否则 MySQL 无法启动：

```bash
chown -R mysql:mysql /var/lib/mysql
chmod -R 750 /var/lib/mysql
```

---

### 5.6 恢复步骤 5：启动 MySQL 并验证数据

```bash
# 启动 MySQL 服务
systemctl start mysqld

# 验证服务状态
systemctl status mysqld

# 登录 MySQL，验证数据完整性
mysql -u root -p
# 查看库表、数据行数，确认恢复成功
SHOW DATABASES;
SELECT COUNT(*) FROM my_database.user_info;
```

## 六、差异备份的备份与恢复（生产环境推荐首选）

### 6.1 差异备份核心特点

差异备份是增量备份的特殊形式，**每次备份都基于最初的全量基准**，而非上一次增量，恢复时只需「全量基准 + 最后一次差异备份」，无需依次合并所有增量，安全性更高，恢复更简单，是绝大多数生产环境的首选。

### 6.2 差异备份实操

```bash
# 第一次差异备份（周一，基于全量基准）
xtrabackup \
--user=backup_user \
--password='Backup@20260401' \
--backup \
--target-dir=/backup/xtrabackup/diff/diff_20260402 \
--incremental-basedir=/backup/xtrabackup/full_base/20260401_full

# 第二次差异备份（周二，依然基于全量基准，而非上一次差异）
xtrabackup \
--user=backup_user \
--password='Backup@20260401' \
--backup \
--target-dir=/backup/xtrabackup/diff/diff_20260403 \
--incremental-basedir=/backup/xtrabackup/full_base/20260401_full
```

### 6.3 差异备份恢复流程

恢复流程比连续增量简单得多，只需 2 步合并：

```bash
# 1. 预准备全量基准备份
xtrabackup --prepare --apply-log-only --target-dir=/backup/xtrabackup/full_base/20260401_full

# 2. 合并最后一次差异备份（无需--apply-log-only）
xtrabackup --prepare --target-dir=/backup/xtrabackup/full_base/20260401_full --incremental-dir=/backup/xtrabackup/diff/diff_20260403

# 3. 后续恢复步骤和增量备份完全一致：copy-back → 修复权限 → 启动MySQL
```

## 七、进阶：增量备份 + Binlog 实现任意时间点恢复

增量备份只能恢复到备份结束时的状态，如果需要恢复到**误操作前的任意时间点**（比如周三上午 10 点误删了表，需要恢复到 10 点前），需要配合 MySQL 的 Binlog 实现。

### 7.1 核心原理

1. 增量备份恢复到最近一次备份结束的状态；
2. 通过 Binlog 重放「备份结束时间 → 误操作前时间」的所有数据修改，实现时间点恢复。

### 7.2 实操步骤

1. **先完成增量备份恢复**：恢复到周二凌晨 2 点的备份结束状态；
2. **找到备份结束时的 Binlog 位置**：全量备份目录的 `xtrabackup_binlog_info` 文件记录了备份结束时的 Binlog 文件名和位置；

```bash
    cat /backup/xtrabackup/full_base/20260401_full/xtrabackup_binlog_info
    # 输出示例：mysql-bin.000005        1200
```
    
3. **找到误操作的 Binlog 位置 / 时间**：通过 `mysqlbinlog` 工具查看 Binlog，找到误操作的 SQL，记录其前一个位置 / 时间；
4. **重放 Binlog 到误操作前**：

```bash
    mysqlbinlog \
    --start-position=1200 \
    --stop-datetime="2026-04-03 10:00:00" \
    /var/lib/mysql/mysql-bin.000005 | mysql -u root -p
```
    
5. **验证数据**：确认误操作的数据已恢复，业务正常。

## 八、生产环境自动化备份脚本

### 8.1 全量备份脚本（每周日凌晨 2 点执行）

```bash
#!/bin/bash
# 全量备份脚本
# 配置项
BACKUP_USER="backup_user"
BACKUP_PASSWORD="Backup@20260401"
BASE_DIR="/backup/xtrabackup"
FULL_DIR="${BASE_DIR}/full_base/$(date +%Y%m%d)_full"
LOG_FILE="${BASE_DIR}/backup.log"
RETENTION_DAYS=14
PARALLEL=4

# 创建目录
mkdir -p ${FULL_DIR}
echo "[$(date +%Y-%m-%d %H:%M:%S)] 开始全量备份，目录：${FULL_DIR}" >> ${LOG_FILE}

# 执行全量备份
xtrabackup \
--user=${BACKUP_USER} \
--password=${BACKUP_PASSWORD} \
--backup \
--target-dir=${FULL_DIR} \
--parallel=${PARALLEL} \
--compress \
--compress-threads=2 >> ${LOG_FILE} 2>&1

# 备份结果判断
if [ $? -eq 0 ]; then
    echo "[$(date +%Y-%m-%d %H:%M:%S)] 全量备份成功" >> ${LOG_FILE}
else
    echo "[$(date +%Y-%m-%d %H:%M:%S)] 全量备份失败！" >> ${LOG_FILE}
    exit 1
fi

# 删除14天前的备份
echo "[$(date +%Y-%m-%d %H:%M:%S)] 删除${RETENTION_DAYS}天前的备份" >> ${LOG_FILE}
find ${BASE_DIR} -type d -mtime +${RETENTION_DAYS} -exec rm -rf {} \;

echo "[$(date +%Y-%m-%d %H:%M:%S)] 全量备份脚本执行完成" >> ${LOG_FILE}
```

### 8.2 增量备份脚本（周一到周六凌晨 2 点执行）

```bash
#!/bin/bash
# 增量备份脚本
# 配置项
BACKUP_USER="backup_user"
BACKUP_PASSWORD="Backup@20260401"
BASE_DIR="/backup/xtrabackup"
INCREMENT_DIR="${BASE_DIR}/increment/inc_$(date +%Y%m%d)"
LOG_FILE="${BASE_DIR}/backup.log"
PARALLEL=4

# 找到上一次备份的目录（最新的备份目录）
LAST_BACKUP_DIR=$(find ${BASE_DIR} -type d -name "*_full" -o -name "inc_*" | sort -r | head -n 1)

if [ ! -d "${LAST_BACKUP_DIR}" ]; then
    echo "[$(date +%Y-%m-%d %H:%M:%S)] 未找到上一次备份目录，增量备份失败！" >> ${LOG_FILE}
    exit 1
fi

# 创建增量目录
mkdir -p ${INCREMENT_DIR}
echo "[$(date +%Y-%m-%d %H:%M:%S)] 开始增量备份，基准目录：${LAST_BACKUP_DIR}，目标目录：${INCREMENT_DIR}" >> ${LOG_FILE}

# 执行增量备份
xtrabackup \
--user=${BACKUP_USER} \
--password=${BACKUP_PASSWORD} \
--backup \
--target-dir=${INCREMENT_DIR} \
--incremental-basedir=${LAST_BACKUP_DIR} \
--parallel=${PARALLEL} >> ${LOG_FILE} 2>&1

# 备份结果判断
if [ $? -eq 0 ]; then
    echo "[$(date +%Y-%m-%d %H:%M:%S)] 增量备份成功" >> ${LOG_FILE}
else
    echo "[$(date +%Y-%m-%d %H:%M:%S)] 增量备份失败！" >> ${LOG_FILE}
    exit 1
fi

echo "[$(date +%Y-%m-%d %H:%M:%S)] 增量备份脚本执行完成" >> ${LOG_FILE}
```

## 九、核心避坑指南（90% 的人踩过的坑）

1. **坑 1：全量备份 Prepare 时不加 --apply-log-only**
    
    - 后果：后续增量备份无法合并，恢复直接失败；
    - 避坑：除了最后一次增量的 Prepare，其他所有步骤都必须加 `--apply-log-only`。
    
2. **坑 2：版本不匹配**
    
    - 后果：备份失败、备份文件损坏、恢复后 MySQL 无法启动；
    - 避坑：严格遵循 PXB 版本与 MySQL 版本的对应关系，禁止跨大版本备份 / 恢复。
    
3. **坑 3：恢复前不备份当前数据目录**
    
    - 后果：恢复失败，现有数据也丢失，无法回滚；
    - 避坑：恢复前必须 mv 备份当前数据目录，确认恢复成功后再删除。
    
4. **坑 4：恢复后不修改数据目录权限**
    
    - 后果：MySQL 启动失败，提示权限拒绝；
    - 避坑：恢复后必须执行 `chown -R mysql:mysql /var/lib/mysql`。
    
5. **坑 5：增量备份的基准目录错误**
    
    - 后果：增量备份不完整，恢复时数据丢失；
    - 避坑：连续增量的基准目录必须是上一次备份的目录，差异备份的基准目录必须是全量基准目录。
    
6. **坑 6：MyISAM 引擎使用增量备份**
    
    - 后果：MyISAM 引擎没有 LSN，增量备份会全量拷贝 MyISAM 表，无法实现增量；
    - 避坑：MyISAM 引擎只能使用全量备份，生产环境统一使用 InnoDB 引擎。
    
7. **坑 7：只备份不测试恢复**
    
    - 后果：真要恢复时，才发现备份文件损坏、步骤错误，数据丢失；
    - 避坑：每月至少测试一次增量备份的完整恢复流程，确认备份可用。

## 十、常见问题排查

1. **备份失败，提示权限 denied**
    
    - 排查：备份用户的权限是否正确，是否授予了所有必要权限；SELinux 是否关闭；备份目录是否有写入权限。
    
2. **恢复后 MySQL 启动失败，提示 Table 'xxx' doesn't exist**
    
    - 排查：恢复时是否遗漏了增量备份的合并；数据目录权限是否正确；备份文件是否完整。
    
3. **增量备份文件和全量备份一样大**
    
    - 排查：`--incremental-basedir` 是否正确；表是否为 MyISAM 引擎；是否开启了压缩。
    
4. **合并增量时提示 LSN 不匹配**
    
    - 排查：增量备份的顺序是否正确；是否跳过了中间的增量备份；全量备份是否执行过不带 `--apply-log-only` 的 Prepare。

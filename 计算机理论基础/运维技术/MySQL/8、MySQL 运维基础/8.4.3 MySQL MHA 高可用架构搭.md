## 一、开篇：MHA 核心定位与原理

### 1.1 什么是 MHA

MHA（Master High Availability）是日本 DeNA 公司开源的 MySQL 主从高可用管理工具，是一主多从架构下**最经典、最成熟的自动故障切换方案**，专门解决 MySQL 主库单点故障问题。

MHA 能在**30 秒内完成主库故障的自动检测与切换**，同时保证切换过程中数据零丢失（基于 binlog 补全），是金融、电商等核心业务生产环境的首选高可用方案。

### 1.2 核心价值

1. **自动故障检测与切换**：主库故障时，无需人工干预，自动完成故障切换，大幅缩短停机时间；
2. **数据零丢失**：切换时自动补全主库故障前未同步的 binlog 到从库，保证数据一致性；
3. **无侵入性**：无需修改现有主从复制架构，只需安装 MHA 组件即可，对业务无影响；
4. **成本极低**：开源免费，无需额外付费，适配所有 MySQL 5.6/5.7/8.0 版本；
5. **适配性强**：支持一主一从、一主多从架构，支持半同步复制、GTID 复制。

### 1.3 MHA 核心组件

MHA 由两个核心组件组成，分别部署在不同的节点：

| 组件                    | 部署节点              | 核心作用                                                                                                                            |
| :-------------------- | :---------------- | :------------------------------------------------------------------------------------------------------------------------------ |
| **MHA Manager（管理节点）** | 独立的管理服务器          | 1. 监控主库的健康状态；<br><br>2. 主库故障时，协调所有从库完成数据补全和主从切换；<br><br>3. 发送告警通知、执行自定义脚本；<br><br>4. 管理多个 MySQL 集群。                             |
| **MHA Node（数据节点）**    | 所有 MySQL 主库、从库服务器 | 1. 提供 binlog 复制、relay log 解析等核心工具；<br><br>2. 故障切换时，Manager 通过 Node 组件拉取主库的 binlog、对比从库的数据差异；<br><br>3. 所有 MySQL 节点必须安装 Node 组件。 |

### 1.4 故障切换完整流程（核心原理）

主库故障时，MHA 会按以下步骤完成自动切换，全程保证数据一致性：

1. **故障检测**：Manager 节点通过心跳检测主库状态，连续多次连接失败，确认主库故障；
2. **SSH 检查**：Manager 检查所有节点的 SSH 免密是否正常，确认可远程执行命令；
3. **保存故障主库的 binlog**：如果故障主库还能通过 SSH 连接，Manager 通过 Node 组件拉取主库故障前未同步的 binlog，保证数据不丢失；
4. **选举新主库**：Manager 对比所有从库的 relay log 位置，选择数据最新的从库作为候选新主库；
5. **补全数据差异**：Manager 对比所有从库的数据差异，将缺失的 binlog/relay log 补全到每个从库，保证所有从库数据完全一致；
6. **提升新主库**：将候选新主库提升为正式主库，开启读写权限，重置主从复制配置；
7. **其他从库指向新主库**：所有其他从库重新配置 `CHANGE MASTER TO`，指向新主库，开启复制；
8. **告警通知**：执行自定义脚本，发送邮件 / 短信告警，通知运维人员主库故障切换完成；
9. **VIP 漂移**：执行 VIP 切换脚本，将虚拟 IP 从故障主库漂移到新主库，业务无感知切换。

### 2.2 前置环境要求（所有节点必须完成）

#### 1. 关闭防火墙和 SELinux

```bash
# 所有节点执行
systemctl stop firewalld
systemctl disable firewalld
setenforce 0
sed -i 's/^SELINUX=.*/SELINUX=disabled/' /etc/selinux/config
```

#### 2. 配置主机名与 hosts 解析

```bash
# 所有节点执行，修改对应主机名
hostnamectl set-hostname mysql-master  # 主库执行
hostnamectl set-hostname mysql-slave1  # 从库1执行
hostnamectl set-hostname mysql-slave2  # 从库2执行
hostnamectl set-hostname mha-manager   # Manager节点执行

# 所有节点配置hosts解析
cat >> /etc/hosts << EOF
192.168.1.100 mysql-master
192.168.1.101 mysql-slave1
192.168.1.102 mysql-slave2
192.168.1.200 mha-manager
EOF
```

#### 3. 配置 SSH 免密登录（核心前提）

MHA Manager 需要通过 SSH 免密登录所有 MySQL 节点，所有 MySQL 节点之间也需要 SSH 免密登录，用于拉取 binlog。

```bash
# 1. 所有节点生成SSH密钥（一路回车即可）
ssh-keygen -t rsa

# 2. Manager节点将公钥分发到所有节点（包括自己）
ssh-copy-id root@mysql-master
ssh-copy-id root@mysql-slave1
ssh-copy-id root@mysql-slave2
ssh-copy-id root@mha-manager

# 3. 所有MySQL节点将公钥分发到其他所有节点
# mysql-master执行
ssh-copy-id root@mysql-slave1
ssh-copy-id root@mysql-slave2
ssh-copy-id root@mha-manager

# mysql-slave1执行
ssh-copy-id root@mysql-master
ssh-copy-id root@mysql-slave2
ssh-copy-id root@mha-manager

# mysql-slave2执行
ssh-copy-id root@mysql-master
ssh-copy-id root@mysql-slave1
ssh-copy-id root@mha-manager

# 4. 验证免密登录，所有节点之间互相ssh，无需输入密码即可登录即为成功
ssh mysql-master date
ssh mysql-slave1 date
ssh mysql-slave2 date
```

#### 4. 搭建稳定的主从复制架构

参考之前的主从复制笔记，搭建一主两从架构，要求：

- 主从复制正常，`Slave_IO_Running` 和 `Slave_SQL_Running` 均为 `Yes`；
- 主库开启 `binlog_format=ROW`，`log_bin=mysql-bin`，`server-id` 全局唯一；
- 从库开启 `read_only=1`，`relay_log=mysql-relay-bin`；
- 所有节点关闭 `skip-name-resolve`，避免 DNS 解析导致的连接超时。

#### 5. 创建 MHA 专用数据库账号

在主库创建 MHA 管理账号，主从复制会自动同步到从库：

```sql
-- 主库执行
CREATE USER 'mha_user'@'%' IDENTIFIED BY 'Mha@20260401';
GRANT ALL PRIVILEGES ON *.* TO 'mha_user'@'%';
FLUSH PRIVILEGES;
```

#### 6. 安装依赖包（所有节点）

```bash
# 所有节点执行，安装MHA依赖
yum install -y perl-DBD-MySQL perl-Config-Tiny perl-Log-Dispatch perl-Parallel-ForkManager perl-Time-HiRes perl-ExtUtils-MakeMaker
```

### 3.1 所有节点安装 MHA Node

MHA Node 是 Manager 的依赖，所有节点（包括 Manager 节点）必须先安装 Node 组件。

```bash
# 所有节点执行

# 安装MHA Node
yum install -y mha4mysql-node-0.58-0.el7.centos.noarch.rpm
```

### 3.2 Manager 节点安装 MHA Manager

仅在 Manager 节点安装：

```bash
# Manager节点执行
# 下载MHA Manager安装包
wget https://github.com/yoshinorim/mha4mysql-manager/releases/download/v0.58/mha4mysql-manager-0.58-0.el7.centos.noarch.rpm

# 安装MHA Manager
yum install -y mha4mysql-manager-0.58-0.el7.centos.noarch.rpm

# 验证安装成功
masterha_check_ssh --version
masterha_check_repl --version
```

### 3.3 配置 MHA 配置文件

在 Manager 节点创建 MHA 工作目录和配置文件：

```bash
# Manager节点执行
# 创建工作目录
mkdir -p /etc/mha
mkdir -p /var/log/mha/app1

# 创建配置文件
cat > /etc/mha/app1.cnf << EOF
[server default]
# MHA管理账号
user=mha_user
password=Mha@20260401
# SSH登录用户
ssh_user=root
# 主库binlog存放目录
master_binlog_dir=/var/lib/mysql
# MHA工作目录
manager_workdir=/var/log/mha/app1
# MHA日志文件
manager_log=/var/log/mha/app1/manager.log
# 远程MySQL端口
port=3306
# 故障切换后执行的脚本（VIP漂移）
master_ip_failover_script=/etc/mha/master_ip_failover
# 切换完成后执行的告警脚本
report_script=/etc/mha/send_report.sh
#  ping检测间隔，3次失败判定主库故障
ping_interval=3

# 主库节点配置
[server1]
hostname=mysql-master
candidate_master=1  # 标记为候选主库，故障时优先提升为新主库
check_repl_delay=0   # 不检查复制延迟，即使延迟也优先提升

# 从库1节点配置
[server2]
hostname=mysql-slave1
candidate_master=1
check_repl_delay=0

# 从库2节点配置
[server3]
hostname=mysql-slave2
no_master=1  # 标记为永远不能提升为主库，仅作为从库
EOF
```

|核心参数|说明|
|:--|:--|
|`candidate_master=1`|标记为候选主库，主库故障时优先选择该节点作为新主库|
|`no_master=1`|该节点永远不能提升为主库，适合配置低的从库、只读从库|
|`ping_interval=3`|每 3 秒 ping 一次主库，3 次失败判定主库故障|
|`master_ip_failover_script`|VIP 漂移脚本，故障切换时自动执行|

### 3.4 配置 VIP 漂移脚本（生产环境必备）

业务连接数据库使用虚拟 IP（VIP），主库故障时，MHA 自动将 VIP 漂移到新主库，业务无需修改连接地址，完全无感知。

```bash
# Manager节点执行
# 创建VIP漂移脚本
cat > /etc/mha/master_ip_failover << EOF
#!/usr/bin/perl
use strict;
use warnings FATAL => 'all';
use Getopt::Long;

my (
    \$command,        \$ssh_user,        \$orig_master_host, \$orig_master_ip,
    \$orig_master_port, \$new_master_host, \$new_master_ip,    \$new_master_port
);

# 虚拟IP配置
my \$vip = '192.168.1.150/24';
# 网卡名称，根据服务器实际网卡修改（如eth0/ens33）
my \$nic = 'ens33';
my \$key = "1";
my \$ssh_start_vip = "/sbin/ifconfig \$nic:\$key \$vip";
my \$ssh_stop_vip = "/sbin/ifconfig \$nic:\$key down";

GetOptions(
    'command=s'             => \$command,
    'ssh_user=s'            => \$ssh_user,
    'orig_master_host=s'    => \$orig_master_host,
    'orig_master_ip=s'      => \$orig_master_ip,
    'orig_master_port=i'    => \$orig_master_port,
    'new_master_host=s'     => \$new_master_host,
    'new_master_ip=s'       => \$new_master_ip,
    'new_master_port=i'     => \$new_master_port,
);

exit &main();

sub main {
    if ( \$command eq "stop" || \$command eq "stopssh" ) {
        my \$exit_code = 1;
        eval {
            print "Disabling the VIP on old master: \$orig_master_host \n";
            &ssh_exec(\$ssh_user, \$orig_master_host, \$ssh_stop_vip);
            \$exit_code = 0;
        };
        if (\$@) {
            warn "Got Error: \$@\n";
            exit \$exit_code;
        }
        exit \$exit_code;
    }
    elsif ( \$command eq "start" ) {
        my \$exit_code = 1;
        eval {
            print "Enabling the VIP on new master: \$new_master_host \n";
            &ssh_exec(\$ssh_user, \$new_master_host, \$ssh_start_vip);
            \$exit_code = 0;
        };
        if (\$@) {
            warn "Got Error: \$@\n";
            exit \$exit_code;
        }
        exit \$exit_code;
    }
    elsif ( \$command eq "status" ) {
        print "Checking the Status of the script.. OK \n";
        exit 0;
    }
    else {
        &usage();
        exit 1;
    }
}

sub ssh_exec(\$\$) {
    my ( \$ssh_user, \$host, \$cmd ) = @_;
    my \$ssh_cmd = "ssh \$ssh_user\@\$host \"\$cmd\"";
    print "Executing: \$ssh_cmd\n";
    return system(\$ssh_cmd);
}

sub usage {
    print "Usage: master_ip_failover --command=start|stop|stopssh|status --ssh_user=user --orig_master_host=host --orig_master_ip=ip --orig_master_port=port --new_master_host=host --new_master_ip=ip --new_master_port=port\n";
}
EOF

# 给脚本添加执行权限
chmod +x /etc/mha/master_ip_failover

# 给主库绑定VIP（初始状态）
ssh root@mysql-master "ifconfig ens33:1 192.168.1.150/24"
```

### 3.5 配置告警脚本

```bash
# Manager节点执行
cat > /etc/mha/send_report.sh << EOF
#!/bin/bash
# 故障切换告警脚本，可对接邮件、企业微信、钉钉告警

    # 钉钉机器人webhook地址

# 执行告警
send_to_dingtalk \$1 \$2

# 给脚本添加执行权限
chmod +x /etc/mha/send_report.sh
```

## 五、MHA 避坑指南

1. **坑 1：SSH 免密配置失败**
    
    - 后果：MHA 无法远程执行命令，故障切换失败；
    - 避坑：所有节点之间必须双向免密，包括 Manager 节点自己，切换前必须用 `masterha_check_ssh` 验证。
    
2. **坑 2：MySQL 8.0 版本兼容问题**
    
    - 后果：MHA 0.58 官方版本不支持 MySQL 8.0 的 caching_sha2_password 认证；
    - 避坑：MySQL 8.0 需使用 MHA 0.58 的社区修改版，或修改 MySQL 用户认证方式为 mysql_native_password。
    
3. **坑 3：主库 binlog 目录配置错误**
    
    - 后果：主库故障时，无法拉取 binlog，导致数据丢失；
    - 避坑：配置文件中的 `master_binlog_dir` 必须和主库 my.cnf 中的 `log_bin` 目录完全一致。
    
4. **坑 4：从库开启了中继日志自动清理**
    
    - 后果：MHA 无法通过从库的 relay log 补全数据，导致数据丢失；
    - 避坑：从库 my.cnf 中配置 `relay_log_purge=0`，禁止自动清理中继日志，MHA 会自动清理。
    
5. **坑 5：故障切换后未重启 MHA**
    
    - 后果：MHA 完成一次故障切换后，会自动停止，不会继续监控新主库；
    - 避坑：故障切换完成后，修复原主库，修改配置文件，重新启动 MHA Manager 服务。
    
6. **坑 6：防火墙 / SELinux 未关闭**
    
    - 后果：MHA 心跳检测失败，误判主库故障，或远程执行命令失败；
    - 避坑：所有节点必须关闭防火墙和 SELinux，或开放对应的端口和权限。

## 六、核心要点总结

### 读写分离核心总结

1. 读写分离的核心是「写主库、读从库」，基于主从复制实现，是扩展读性能的核心方案；
2. 三大方案选型：Java 生态首选 Sharding-JDBC，生产环境多语言架构首选 ProxySQL，简单场景用 MySQL Router；
3. 最大痛点是主从延迟，核心解决方案是强制主库读、半同步复制、并行复制、缓存兜底；
4. 生产环境必须保障代理层高可用，避免单点故障，同时做好从库延迟自动剔除。

### MHA 高可用核心总结

1. MHA 是一主多从架构下最成熟的自动故障切换方案，可在 30 秒内完成切换，保证数据零丢失；
2. 核心前提是 SSH 免密登录、稳定的主从复制、正确的 MHA 账号权限；
3. 生产环境必须配合 VIP 漂移，实现业务无感知切换，同时配置告警通知；
4. 搭建完成后必须做故障模拟测试，验证切换流程正常，避免主库真故障时切换失败；
5. 核心避坑点是 SSH 免密、binlog 目录配置、中继日志自动清理、版本兼容问题。

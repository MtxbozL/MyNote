## 一、总则

### 1.1 故障分级标准（生产环境通用）

根据业务影响范围、影响用户比例、恢复难度，将故障分为 4 个等级，明确响应优先级：

|故障等级|定义|核心场景示例|响应时限|处理负责人|
|:--|:--|:--|:--|:--|
|**P0 致命故障**|核心业务完全不可用，全量用户受影响，造成重大资损 / 舆情风险|1. MySQL 主库宕机，集群完全不可用<br><br>2. 数据库 OOM 崩溃，无法启动<br><br>3. 磁盘爆满，数据库无法写入，业务全停<br><br>4. 核心交易库数据被误删 / 篡改|5 分钟内响应，30 分钟内必须完成业务恢复|DBA 负责人 + 业务技术负责人 + 运维负责人|
|**P1 严重故障**|核心业务严重受损，≥50% 用户受影响，业务体验极差|1. 数据库 CPU/IO 100% 打满，接口大面积超时<br><br>2. 连接数打满，新业务无法连接数据库<br><br>3. 主从复制完全中断，读写分离架构失效<br><br>4. 大面积锁等待，业务写入 / 更新完全阻塞|10 分钟内响应，1 小时内完成业务恢复|DBA + 业务开发负责人|
|**P2 一般故障**|核心业务部分受损，<50% 用户受影响，非核心功能不可用|1. 慢查询频发，部分接口超时<br><br>2. 死锁频发，部分业务操作失败<br><br>3. 主从延迟持续 > 10s，读写分离读到旧数据<br><br>4. 磁盘空间告警（使用率 > 85%）|30 分钟内响应，4 小时内完成修复|DBA + 对应业务开发|
|**P3 轻微故障**|非核心业务异常，无用户感知，不影响线上业务|1. 监控指标异常告警（如连接数突增、缓存命中率下降）<br><br>2. 非核心表慢查询告警<br><br>3. 日志异常，无业务影响|2 小时内响应，24 小时内完成修复|DBA 值班人员|

### 1.2 应急处理核心黄金原则

1. **先止损，再查根因**：第一优先级恢复业务可用性，而非死磕根因。业务恢复后，再做完整的根因分析与复盘；
2. **操作可回滚**：所有应急操作必须有明确的回滚方案，禁止在生产环境执行不可逆操作（如直接 rm 删除文件、DROP 表、无备份修改数据）；
3. **禁止盲目重启**：重启会丢失故障现场（进程列表、锁信息、事务状态、内存堆栈），导致无法定位根因。除非业务完全不可用且无其他恢复方案，否则禁止第一时间重启数据库；
4. **保留故障现场**：故障发生后，第一时间备份错误日志、慢查询日志、binlog、系统监控数据、进程快照，便于后续根因分析；
5. **分级通报升级**：P0/P1 故障必须立即向上级通报，同步业务、运维、风控等相关团队，禁止隐瞒故障；
6. **最小化影响**：应急操作尽量选择对业务影响最小的方案，比如优先 kill 异常 SQL，而非重启数据库；优先限流降级，而非全量停服。

## 二、分场景标准化应急 SOP

### 2.1 P0 级故障：MySQL 服务完全不可用 / 宕机

#### 故障现象

1. 业务完全无法连接数据库，接口 100% 超时；
2. 服务器本地无法登录 MySQL，`mysql -u root -p` 连接失败；
3. MySQL 进程不存在 / 自动退出，`ps -ef | grep mysqld` 无进程；
4. 端口无监听，`ss -tulnp | grep 3306` 无结果。

#### 应急处理步骤（按优先级排序）

1. **业务止损（10 分钟内完成）**
    
    - 若为一主多从架构，立即执行主从切换，将业务流量切到从库，恢复业务（切换流程见 2.2 SOP）；
    - 若无主从架构，立即对业务做降级 / 限流，暂停非核心功能，避免用户请求持续报错。
    
2. **快速定位与恢复**
    
    bash
    
    运行
    
    ```
    # 步骤1：查看MySQL错误日志（唯一核心依据，必须先看）
    cat /var/log/mysqld.log | grep -i 'error\|fatal' | tail -100
    # 若日志路径未知，查看my.cnf配置：cat /etc/my.cnf | grep log_error
    
    # 步骤2：检查进程状态，确认是否崩溃/被OOM kill
    ps -ef | grep mysqld
    dmesg | grep -i 'oom\|kill' | grep mysqld  # 确认是否被系统OOM kill
    
    # 步骤3：检查基础环境
    df -h  # 检查磁盘是否100%爆满
    free -m # 检查内存是否耗尽
    ss -tulnp | grep 3306 # 检查端口是否被占用
    ls -ld /var/lib/mysql # 检查数据目录权限是否为mysql:mysql
    ```
    
3. **针对性恢复操作**
    
    表格
    
    |根因|恢复操作|
    |:--|:--|
    |被 OOM kill，内存耗尽|1. 调小 my.cnf 中`innodb_buffer_pool_size`（不超过内存 70%）；<br><br>2. 限制`max_connections`，避免连接数过多；<br><br>3. 启动 MySQL 服务。|
    |磁盘 100% 爆满，无法启动|1. 用 PURGE 命令清理过期 binlog，禁止直接 rm 删除；<br><br>2. 清理过大的日志文件（慢查询日志、错误日志）；<br><br>3. 磁盘空间释放后，启动 MySQL 服务。|
    |配置文件 my.cnf 语法 / 参数错误|1. 注释掉新增的错误配置，恢复到上一个可用的配置；<br><br>2. 启动 MySQL 服务。|
    |数据目录权限错误|1. 修复权限：`chown -R mysql:mysql /var/lib/mysql && chmod -R 750 /var/lib/mysql`；<br><br>2. 启动 MySQL 服务。|
    |端口被占用|1. 结束占用 3306 端口的进程，或修改 MySQL 端口；<br><br>2. 启动 MySQL 服务。|
    |数据文件损坏，无法启动|1. 开启 InnoDB 强制恢复模式，启动数据库备份数据；<br><br>2. 用最近的全量备份 + binlog 恢复数据，重建实例。|
    
4. **业务验证**
    
    - 数据库启动后，验证主从复制、业务账号权限、核心表数据完整性；
    - 业务流量切回，验证接口可用性，确认业务恢复。
    

#### 红线禁止操作

1. 无备份的情况下，直接执行强制恢复、重建实例操作；
2. 直接 rm 删除 MySQL 数据文件、binlog 文件；
3. 未定位根因的情况下，反复重启 MySQL 服务；
4. 业务未切走的情况下，在故障实例上执行数据恢复操作。

### 2.2 P0 级故障：主库宕机，主从架构故障切换

#### 故障现象

1. 主库服务器宕机 / MySQL 进程崩溃，无法连接、无法重启；
2. 业务写入完全失败，读请求可通过从库正常执行；
3. 从库复制中断，`Slave_IO_Running: No`。

#### 应急处理步骤（手动切换，MHA 自动切换失败时备用）

1. **业务止损**：立即暂停业务写入功能，开启读降级，避免用户写入请求报错。
    
2. **主库存活确认**
    
    bash
    
    运行
    
    ```
    # 多次ping/ssh主库，确认主库完全不可恢复，避免脑裂
    ping 主库IP
    ssh root@主库IP "systemctl status mysqld"
    # 确认主库无法恢复后，再执行切换，绝对禁止主库还存活时执行切换，会导致脑裂、数据不一致
    ```
    
3. **新主库选举与提升**
    
    sql
    
    ```
    -- 步骤1：登录所有从库，确认复制状态，选择数据最完整的从库作为新主库
    SHOW SLAVE STATUS\G
    -- 选择标准：Exec_Master_Log_Pos最大，Relay_Master_Log_File最新，数据最完整
    
    -- 步骤2：登录候选新主库，执行提升操作
    STOP SLAVE;
    RESET MASTER;
    RESET SLAVE ALL;
    -- 关闭只读，开启读写
    SET GLOBAL read_only = 0;
    SET GLOBAL super_read_only = 0;
    -- 验证新主库状态
    SHOW MASTER STATUS;
    ```
    
4. **其他从库切换到新主库**
    
    sql
    
    ```
    -- 登录其他从库，执行切换
    STOP SLAVE;
    RESET SLAVE ALL;
    CHANGE MASTER TO
    MASTER_HOST='新主库IP',
    MASTER_PORT=3306,
    MASTER_USER='复制用户',
    MASTER_PASSWORD='复制用户密码',
    MASTER_AUTO_POSITION = 1; -- GTID模式用，非GTID模式填MASTER_LOG_FILE和MASTER_LOG_POS
    START SLAVE;
    
    -- 验证复制状态
    SHOW SLAVE STATUS\G
    -- 确认Slave_IO_Running和Slave_SQL_Running均为Yes
    ```
    
5. **业务切换与恢复**
    
    - 修改业务代码 / 数据库中间件 / ProxySQL 的数据库连接地址，指向新主库；
    - 恢复业务写入功能，验证读写接口可用性，确认业务恢复。
    
6. **故障主库修复**
    
    - 原主库修复后，重新配置为新主库的从库，加入集群。
    

#### 红线禁止操作

1. 主库还存活的情况下，执行主从切换，导致脑裂、数据双写、数据不一致；
2. 未确认从库数据完整性的情况下，提升新主库，导致数据丢失；
3. 切换后未关闭新主库的只读模式，导致业务写入失败；
4. 切换后未验证从库复制状态，导致二次故障。

### 2.3 P1 级故障：CPU 使用率 100% 打满

#### 故障现象

1. 服务器 CPU 使用率持续 95%+，甚至 100%；
2. 数据库响应极慢，业务接口大面积超时；
3. 慢查询数量突增，连接数持续上涨。

#### 应急处理步骤

1. **应急止损（5 分钟内完成）**
    
    sql
    
    ```
    -- 步骤1：登录数据库，查看正在执行的异常SQL
    SHOW FULL PROCESSLIST;
    -- 重点关注：COMMAND=Query、TIME>10、STATE=Sending data/Using filesort/Using temporary的SQL
    
    -- 步骤2：批量kill长时间执行的异常SQL，快速降低CPU（仅kill查询语句，禁止kill事务内的写入语句）
    -- 生成kill命令
    SELECT CONCAT('KILL ', id, ';') FROM information_schema.PROCESSLIST WHERE COMMAND = 'Query' AND TIME > 10 AND INFO NOT LIKE '%COMMIT%' AND INFO NOT LIKE '%ROLLBACK%';
    -- 执行生成的kill命令，快速释放CPU资源
    ```
    
2. **根因快速定位**
    
    bash
    
    运行
    
    ```
    # 步骤1：分析慢查询日志，定位累计CPU消耗最高的TOP10 SQL
    pt-query-digest /var/lib/mysql/slow.log --since '1h' > slow_analysis.log
    
    # 步骤2：用EXPLAIN分析异常SQL的执行计划，确认是否全表扫描、索引失效、文件排序
    EXPLAIN SELECT xxx FROM xxx WHERE xxx;
    ```
    
    - 90% 的场景根因：高频慢 SQL、索引失效、全表扫描、大量排序分组操作。
    
3. **快速恢复优化**
    
    - 临时方案：给异常 SQL 紧急加合适的索引，或下线异常业务接口；
    - 长效方案：SQL 改写优化、业务加缓存、读写分离分散读压力。
    
4. **业务验证**
    
    - 确认 CPU 使用率下降到正常范围，业务接口响应时间恢复正常。
    

#### 红线禁止操作

1. 业务高峰期直接重启数据库，重启后流量进来 CPU 会再次打满，问题复现；
2. 批量 kill 正在执行事务的写入语句，导致大量事务回滚，数据不一致；
3. 未验证索引有效性的情况下，在业务高峰期给大表加索引，导致锁表、业务阻塞。

### 2.4 P1 级故障：Too many connections 连接数打满

#### 故障现象

1. 新连接报错`ERROR 1040 (08004): Too many connections`，业务无法建立新连接；
2. `Threads_connected` 达到`max_connections`上限；
3. 已有连接执行缓慢，业务接口超时。

#### 应急处理步骤

1. **应急止损（5 分钟内完成）**
    
    sql
    
    ```
    -- 步骤1：用root账号登录（root有1个预留连接，即使连接数打满也能登录）
    mysql -u root -p
    
    -- 步骤2：查看连接数配置与当前连接数
    SHOW VARIABLES LIKE 'max_connections';
    SHOW STATUS LIKE 'Threads_connected';
    
    -- 步骤3：批量kill空闲时间过长的Sleep连接，释放连接数（优先kill空闲>300秒的连接）
    SELECT CONCAT('KILL ', id, ';') FROM information_schema.PROCESSLIST WHERE COMMAND = 'Sleep' AND TIME > 300;
    -- 执行生成的kill命令，快速释放连接
    ```
    
2. **根因快速定位**
    
    sql
    
    ```
    -- 步骤1：查看连接来源，定位是哪个业务/IP的连接堆积
    SELECT HOST, COUNT(*) FROM information_schema.PROCESSLIST GROUP BY HOST ORDER BY COUNT(*) DESC;
    
    -- 步骤2：查看连接相关配置
    SHOW VARIABLES LIKE 'wait_timeout'; -- 空闲连接超时时间，默认8小时，建议设为300秒
    SHOW VARIABLES LIKE 'interactive_timeout';
    ```
    
    - 常见根因：业务突发流量、连接池配置不合理、慢 SQL 导致连接堆积、空闲连接超时时间过长、连接泄露。
    
3. **快速恢复与长效优化**
    
    - 临时方案：紧急调大`max_connections`（不超过 2000，避免内存溢出）；
    - 长效方案：
        
        1. 缩短空闲连接超时时间：`wait_timeout=300`，`interactive_timeout=300`；
        2. 优化业务连接池配置，设置合理的最大连接数、空闲连接回收机制；
        3. 优化慢 SQL，从根源解决连接执行时间过长导致的堆积。
        
    
4. **业务验证**
    
    - 确认`Threads_connected`下降到正常范围，业务可正常建立连接，接口恢复正常。
    

#### 红线禁止操作

1. 把`max_connections`设为 10000+，导致内存占用过高，数据库 OOM 宕机；
2. 批量 kill 正在执行事务的连接，导致大量事务回滚，数据不一致；
3. 未定位根因的情况下，反复重启数据库，问题很快复现。

### 2.5 P1 级故障：磁盘空间 100% 爆满

#### 故障现象

1. 磁盘使用率 100%，数据库无法写入，业务更新 / 插入操作失败；
2. MySQL 无法启动，binlog 无法写入，主从复制中断；
3. 服务器无法创建新文件，日志无法写入。

#### 应急处理步骤

1. **应急止损（5 分钟内完成）**
    
    sql
    
    ```
    -- 步骤1：登录数据库，用PURGE命令安全清理过期binlog，绝对禁止直接rm删除！
    -- 清理7天前的binlog
    PURGE BINARY LOGS BEFORE DATE_SUB(NOW(), INTERVAL 7 DAY);
    -- 清理指定binlog文件之前的所有文件
    PURGE BINARY LOGS TO 'mysql-bin.000100';
    
    -- 步骤2：清理过大的慢查询日志、错误日志（先备份再清理）
    cp /var/lib/mysql/slow.log /data/backup/slow.log.bak
    echo "" > /var/lib/mysql/slow.log
    ```
    
2. **根因快速定位**
    
    bash
    
    运行
    
    ```
    # 步骤1：定位大文件/大目录
    df -h
    du -sh /var/lib/mysql/* | sort -hr
    du -sh /var/log/* | sort -hr
    
    # 步骤2：查看binlog保留策略
    mysql -u root -p -e "SHOW VARIABLES LIKE 'expire_logs_days';"
    ```
    
    - 常见根因：binlog 未设置自动过期、大表数据量过大、日志文件长期未清理、临时文件过大、ibdata1 系统表空间膨胀。
    
3. **长效优化**
    
    - 设置 binlog 自动过期：`my.cnf`中设置`expire_logs_days=7`，自动清理 7 天前的 binlog；
    - 配置 logrotate，自动切割、压缩、清理慢查询日志、错误日志；
    - 冷热数据分离，历史数据归档，清理无效数据；
    - 开启`innodb_file_per_table=ON`，避免共享表空间膨胀。
    
4. **业务验证**
    
    - 确认磁盘使用率下降到安全范围（<85%），业务写入操作恢复正常，主从复制恢复。
    

#### 红线禁止操作

1. 直接用 rm 命令删除正在使用的 binlog 文件，会导致主从复制崩溃、数据丢失，必须用 PURGE 命令清理；
2. 直接 rm 删除正在使用的日志文件，会导致磁盘空间无法释放，必须先 cp 备份，再用 echo 清空；
3. 无备份的情况下，直接删除业务表数据；
4. 清理 binlog 时，删除从库还未同步的 binlog，导致主从复制中断。

## 三、事后复盘与长效防控

### 3.1 故障复盘标准流程

故障恢复后，必须在 24 小时内组织复盘会，完成以下内容：

1. **故障还原**：完整还原故障发生时间、现象、处理过程、每一步操作的时间线；
2. **根因分析**：明确故障的根本原因，而非表面现象（比如 CPU 打满的根因是索引缺失，而非慢 SQL）；
3. **影响评估**：统计故障影响时长、影响用户数、资损情况、业务受损范围；
4. **优化方案**：制定短期应急优化方案和长期架构优化方案，明确负责人和完成时间；
5. **预案完善**：完善同类故障的应急处理预案，补充到 SOP 手册中；
6. **复盘沉淀**：复盘记录归档，组织团队培训，避免同类故障再次发生。

### 3.2 长效防控机制

1. **监控告警体系**：搭建全维度监控，覆盖系统资源（CPU / 内存 / IO / 磁盘）、数据库指标（连接数 / 慢查询 / 锁 / 主从延迟）、业务指标（接口响应时间 / 错误率），设置多级告警阈值，提前发现隐患；
2. **变更管控**：生产环境 SQL 上线、表结构变更、配置修改必须经过评审、测试、灰度发布，禁止未经评审的变更；
3. **容灾演练**：每季度开展一次主从切换、故障恢复演练，验证应急预案的有效性，提升团队应急处理能力；
4. **定期巡检**：每周对数据库做健康巡检，检查索引健康度、表空间、配置规范、主从状态，提前修复隐患；
5. **备份验证**：每周验证一次全量备份的恢复有效性，确保故障时备份可用，避免 “备份了但恢复不了” 的问题。

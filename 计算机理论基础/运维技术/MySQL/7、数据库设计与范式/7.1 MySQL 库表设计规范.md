## 一、开篇：库表设计的核心定位与价值

### 1. 为什么库表设计规范如此重要

库表设计是系统架构的**根基**，是所有业务逻辑和数据操作的载体，其重要性远高于事后的 SQL 优化和索引优化：

- **改表成本极高**：线上业务库表一旦上线，修改表结构（如加字段、改类型、调整索引）风险大、耗时长，可能导致锁表、业务阻塞、数据丢失；
- **性能影响深远**：不合理的库表设计（如字段类型错误、无主键、单表数据量过大）会导致全表扫描、索引失效、锁竞争加剧，是慢查询的核心根源；
- **可维护性决定生命周期**：规范的库表设计（统一命名、清晰注释、合理结构）能大幅降低后续开发、运维的沟通成本，延长系统的生命周期。

### 2. 库表设计的核心黄金原则

1. **设计先行，开发在后**：库表设计必须在业务开发前完成，禁止边开发边改表；
2. **规范为要，统一标准**：命名、字符集、存储引擎、字段类型等必须统一规范，避免个性化设计；
3. **性能兼顾，避免过度**：在保证业务需求的前提下，优先考虑查询和写入性能，但避免过度设计（如预留大量无用字段）；
4. **冷热分离，可扩展性**：提前规划数据增长，冷热数据分离，预留分库分表的空间；
5. **注释清晰，文档同步**：所有库、表、字段必须有清晰的注释，变更时同步更新文档。

## 二、第一部分：数据库设计规范

### 2.1 数据库命名规范

统一的命名规范是库表可维护性的基础，必须严格遵守，禁止随意命名。

| 命名对象     | 规范要求                                                                                                                                                                                      | 正确示例                                                              | 错误示例                                                            |
| :------- | :---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | :---------------------------------------------------------------- | :-------------------------------------------------------------- |
| **数据库名** | 1. 全部小写字母，用下划线分隔；<br><br>2. 格式：`业务名_环境`；<br><br>3. 禁止使用 MySQL 关键字；<br><br>4. 见名知意，长度不超过 30 字符                                                                                             | `ecommerce_prod`（电商生产库）、`user_center_dev`（用户中心开发库）                | `ECommerce_Prod`、`userCenter`、`db123`、`order`（关键字）              |
| **表名**   | 1. 全部小写字母，用下划线分隔；<br><br>2. 格式：`模块名_表功能名`；<br><br>3. 禁止使用 MySQL 关键字；<br><br>4. 见名知意，长度不超过 50 字符；<br><br>5. 中间表 / 关联表格式：`a表_b表_rel`                                                        | `user_info`（用户信息表）、`order_detail`（订单详情表）、`user_role_rel`（用户角色关联表） | `UserInfo`、`orderDetail`、`order`（关键字）、`table1`                  |
| **字段名**  | 1. 全部小写字母，用下划线分隔；<br><br>2. 禁止使用 MySQL 关键字；<br><br>3. 见名知意，避免缩写（除非是通用缩写如 id、ctime、utime）；<br><br>4. 布尔类型格式：`is_xxx`、`has_xxx`；<br><br>5. 时间类型格式：`create_time`、`update_time`、`delete_time` | `user_id`、`user_name`、`is_deleted`、`create_time`、`order_amount`   | `UserID`、`userName`、`name`（歧义）、`id`（无业务前缀，关联表时易混淆）、`order`（关键字） |

### 2.2 字符集与排序规则规范

字符集选择错误会导致乱码、存储异常，排序规则选择错误会导致查询、排序结果不符合预期，必须统一规范。

|规范项|推荐配置|说明|
|:--|:--|:--|
|**数据库字符集**|`utf8mb4`|1. 支持完整的 Unicode 字符，包括 emoji 表情；<br><br>2. 避免了 `utf8`（仅支持 3 字节）的存储异常；<br><br>3. 是 MySQL 8.0 的默认字符集。|
|**表字符集**|继承数据库字符集（`utf8mb4`）|无需单独设置，统一继承数据库即可，避免表与表字符集不一致。|
|**字段字符集**|继承表字符集（`utf8mb4`）|特殊场景（如存储二进制数据）可单独设置，否则统一继承。|
|**排序规则**|`utf8mb4_unicode_ci`（推荐）/ `utf8mb4_general_ci`|1. `utf8mb4_unicode_ci`：排序更准确，支持多语言，性能略低；<br><br>2. `utf8mb4_general_ci`：排序速度快，性能高，适合纯中文 / 英文场景；<br><br>3. 禁止使用 `utf8mb4_bin`（区分大小写，易导致查询异常）。|

### 2.3 存储引擎规范

MySQL 支持多种存储引擎，但生产环境 99% 的场景只需要用 InnoDB，必须统一规范。

|存储引擎|推荐场景|核心优势|核心劣势|
|:--|:--|:--|:--|
|**InnoDB**|生产环境默认，所有业务表|1. 支持事务（ACID），保证数据一致性；<br><br>2. 支持行级锁，并发性能高；<br><br>3. 支持聚簇索引，查询性能优；<br><br>4. 支持外键（可选）；<br><br>5. 支持崩溃恢复，数据安全性高。|无明显劣势，是生产环境的唯一选择。|
|**MyISAM**|禁止使用|1. 不支持事务；<br><br>2. 仅支持表级锁，并发性能极差；<br><br>3. 崩溃后无法安全恢复，数据易丢失；<br><br>4. 已被 MySQL 官方逐步淘汰。|仅适用于只读的临时表 / 测试表，生产环境禁止使用。|
|**Memory**|禁止使用|数据存储在内存，重启丢失，仅适用于临时缓存，生产环境禁止使用。||

> 核心规范：**所有生产环境业务表，统一使用 InnoDB 存储引擎**，建表时显式指定（或默认继承），避免使用其他引擎。

## 三、第二部分：表结构设计规范

### 3.1 表设计的核心原则

1. **单表数据量控制原则**：
    
    - 推荐单表数据量控制在 **1000 万行以内**，超过 500 万行需提前规划分库分表；
    - 原因：InnoDB B + 树高度超过 4 层后，查询性能会急剧下降，1000 万行以内 B + 树高度通常为 3 层，查询性能稳定。
    
2. **冷热数据分离原则**：
    
    - 历史数据（如 1 年前的订单、日志）必须与热数据（当前业务数据）分离，归档到历史表 / 数据仓库；
    - 原因：热数据表数据量小，查询性能高，避免历史数据影响当前业务。
    
3. **避免过度设计原则**：
    
    - 只保留业务当前需要的字段，禁止预留大量 “未来可能用” 的字段；
    - 原因：预留字段会占用磁盘空间，增加维护成本，未来真的需要时再加字段即可（Online DDL 很方便）。
    
4. **表必须有注释原则**：
    
    - 所有表必须有 `COMMENT` 注释，说明表的业务用途、所属模块、数据量级、归档策略等。
    

### 3.2 主键设计规范（重中之重）

主键是表的灵魂，InnoDB 中主键就是聚簇索引，主键设计直接影响整张表的查询和写入性能，必须严格遵守以下规范：

|规范要求|正确示例|错误示例|核心说明|
|:--|:--|:--|:--|
|**必须有显式主键**|`id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT`|无主键的表|InnoDB 会自动生成隐藏的 6 字节行 ID，无法控制，且无法用于查询，必须显式设置主键。|
|**用自增数值类型**|`BIGINT UNSIGNED AUTO_INCREMENT`|UUID、随机字符串、业务字段（如订单号）|1. 自增主键是顺序写入，不会导致 B + 树页分裂，写入性能拉满；<br><br>2. UUID / 随机字符串是随机写入，会导致频繁页分裂、索引碎片，写入和查询性能都骤降；<br><br>3. 业务字段可能修改，禁止做主键。|
|**禁止用复合主键**|单字段主键|`(user_id, role_id)` 做主键|1. 复合主键会导致所有二级索引膨胀（二级索引叶子节点存储复合主键值）；<br><br>2. 仅中间表 / 关联表可考虑复合主键，但优先用自增 ID 做主键，复合字段建唯一索引。|
|**主键类型选择**|`BIGINT UNSIGNED`（通用）、`INT UNSIGNED`（小表，数据量 < 1000 万）|`TINYINT`、`VARCHAR`|1. `BIGINT UNSIGNED` 范围是 0~18446744073709551615，足够覆盖所有业务场景；<br><br>2. `INT UNSIGNED` 范围是 0~4294967295，小表可用；<br><br>3. 禁止用字符串做主键，索引膨胀严重。|

> 核心结论：**99% 的场景下，自增 BIGINT UNSIGNED 是主键的最优选择**，业务唯一字段（如订单号、手机号）建唯一索引即可，不要做主键。

### 3.3 表字段数量控制规范

- 推荐单表字段数量控制在 **50 个以内**，超过 30 个需评估是否合理；
- 超过 50 个字段的表，必须进行**垂直拆分**：将核心字段（如用户 ID、姓名、手机号）保留在主表，大字段 / 低频字段（如用户简介、扩展信息）拆分到详情表（`user_info_ext`），通过主键关联；
- 原因：字段过多会导致数据行过大，B + 树每页存储的行数减少，磁盘 IO 次数增加，查询性能下降。

### 3.4 表的时间字段规范

所有表必须包含以下 3 个时间字段，统一命名和类型：

|字段名|类型|默认值|说明|
|:--|:--|:--|:--|
|`create_time`|`DATETIME`|`CURRENT_TIMESTAMP`|记录创建时间，插入时自动设置。|
|`update_time`|`DATETIME`|`CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP`|记录更新时间，插入和更新时自动设置。|
|`delete_time`|`DATETIME`|`NULL`|逻辑删除时间，NULL 表示未删除，有值表示已删除（软删除）。|

> 核心规范：**禁止用字符串存储时间**，必须用 `DATETIME`/`TIMESTAMP`，避免隐式类型转换导致索引失效。

## 四、第三部分：字段设计规范（最容易踩坑的部分）

### 4.1 字段类型选择的核心原则

1. **用小不用大**：在满足业务需求的前提下，选择最小的数据类型（如用 `TINYINT` 代替 `INT`，用 `VARCHAR(20)` 代替 `VARCHAR(255)`）；
2. **用简单不用复杂**：用整数类型代替字符串类型（如用 `TINYINT` 代替 `ENUM`，用 `INT` 存储 IP 地址）；
3. **避免 NULL 值**：所有字段尽量设为 `NOT NULL`，给合理的默认值，避免 NULL 值带来的索引失效、统计复杂等问题；
4. **见名知意**：字段类型要与业务含义匹配（如金额用 `DECIMAL`，不用 `DOUBLE`）。

### 4.2 整数类型设计规范

|类型|字节数|有符号范围|无符号范围|推荐场景|避坑提示|
|:--|:--|:--|:--|:--|:--|
|`TINYINT`|1|-128~127|0~255|性别、状态、是否删除（0/1/2）|禁止用 `TINYINT(1)` 做布尔类型以外的用途，无符号范围更大。|
|`SMALLINT`|2|-32768~32767|0~65535|年龄、数量（小范围）|仅在 `TINYINT` 不够用时使用。|
|`INT`|4|-2147483648~2147483647|0~4294967295|普通表的主键（数据量 < 1000 万）、订单数量|无符号范围更大，推荐用 `INT UNSIGNED`。|
|`BIGINT`|8|-9223372036854775808~9223372036854775807|0~18446744073709551615|通用主键、大表主键、用户 ID、订单 ID|无符号范围足够覆盖所有业务，推荐用 `BIGINT UNSIGNED` 做主键。|

> 核心规范：**优先用无符号整数**，范围更大，避免负数的业务歧义；布尔类型统一用 `TINYINT UNSIGNED NOT NULL DEFAULT 0`，0 表示 false，1 表示 true。

### 4.3 字符串类型设计规范

|类型|字节数|特点|推荐场景|避坑提示|
|:--|:--|:--|:--|:--|
|`CHAR(n)`|n× 字符集字节数|固定长度，存储效率高，查询速度快|固定长度的字符串：手机号（11 位）、身份证号（18 位）、性别（1 位）|n 必须是实际长度，不要浪费空间，如手机号用 `CHAR(11)`，不用 `CHAR(20)`。|
|`VARCHAR(n)`|实际长度 + 1/2 字节|可变长度，节省空间|可变长度的字符串：姓名、地址、商品名称|1. n 必须根据实际需求设置，不要随便设 255；<br><br>2. 超过 5000 字符用 `TEXT`，不要用 `VARCHAR(10000)`；<br><br>3. 禁止用 `VARCHAR` 存储日期、金额、IP 地址。|
|`TEXT`|实际长度 + 2 字节|大文本类型，不能有默认值，不能建全量索引|长文本：文章内容、用户简介、评论|1. 禁止在 `TEXT` 字段上建全量索引，用前缀索引或全文索引；<br><br>2. 避免在主表中放 `TEXT` 字段，拆分到详情表；<br><br>3. 禁止用 `TEXT` 存储短字符串，浪费空间。|
|`MEDIUMTEXT`/`LONGTEXT`|更大的存储范围|超大文本类型|仅用于存储超大文本（如小说、文档）|生产环境尽量避免使用，超大文件建议存对象存储（如 OSS、S3），数据库只存路径。|

> 核心规范：**VARCHAR 长度按需设置**，不要随便设 255；**禁止用字符串存储非字符串数据**（日期、金额、IP、整数），避免隐式类型转换导致索引失效。

### 4.4 日期时间类型设计规范

|类型|字节数|范围|特点|推荐场景|避坑提示|
|:--|:--|:--|:--|:--|:--|
|`DATETIME`|5|1000-01-01 00:00:00 ~ 9999-12-31 23:59:59|与时区无关，范围大|通用时间字段：创建时间、更新时间、订单时间|1. 推荐用 `DATETIME`，范围大，无时区问题；<br><br>2. 禁止用字符串存储时间，必须用 `DATETIME`/`TIMESTAMP`。|
|`TIMESTAMP`|4|1970-01-01 00:00:01 UTC ~ 2038-01-19 03:14:07 UTC|与时区有关，自动转换时区|仅用于需要时区转换的场景|1. 2038 年问题，范围小，不推荐通用场景；<br><br>2. 时区转换可能导致时间不一致。|
|`DATE`|3|1000-01-01 ~ 9999-12-31|仅存储日期，不存储时间|仅需要日期的场景：生日、下单日期|不需要时间时用 `DATE`，节省空间。|
|`TIME`|3|-838:59:59 ~ 838:59:59|仅存储时间，不存储日期|仅需要时间的场景：营业时间|生产环境使用较少。|

> 核心规范：**通用时间字段统一用 DATETIME NOT NULL**，设置合理的默认值（如 `CURRENT_TIMESTAMP`）；禁止用 `VARCHAR` 存储时间，避免隐式类型转换导致索引失效。

### 4.5 金额 / 小数类型设计规范

金额、小数类型的选择必须非常谨慎，否则会导致精度丢失，引发严重的业务问题。

|类型|字节数|特点|推荐场景|避坑提示|
|:--|:--|:--|:--|:--|
|`DECIMAL(m, d)`|可变|精确小数类型，不会丢失精度|金额、价格、税率、百分比等需要高精度的场景|1. m 是总位数，d 是小数位数，如金额用 `DECIMAL(10, 2)`（总 10 位，小数 2 位）；<br><br>2. 禁止用 `DOUBLE`/`FLOAT` 存储金额，会丢失精度；<br><br>3. 也可以用 `BIGINT` 存储分（如 10.00 元存为 1000），避免小数运算。|
|`DOUBLE`/`FLOAT`|8/4|浮点数类型，会丢失精度|仅用于不需要高精度的科学计算|生产环境禁止用于金额、价格等业务数据。|

> 核心规范：**金额统一用 DECIMAL (m, d) 或 BIGINT 存储分**，绝对禁止用 `DOUBLE`/`FLOAT`，避免精度丢失。

### 4.6 NULL 值设计规范

NULL 值是很多性能问题和业务问题的根源，必须严格控制：

|规范要求|说明|
|:--|:--|
|**尽量设为 NOT NULL**|所有字段尽量设为 `NOT NULL`，给合理的默认值：<br><br>- 整数类型默认 0；<br><br>- 字符串类型默认空字符串 `''`；<br><br>- 日期时间类型默认 `CURRENT_TIMESTAMP` 或 `'1970-01-01 00:00:00'`。|
|**NULL 值的危害**|1. 索引统计、比较运算更复杂，优化器更难选择执行计划；<br><br>2. `IS NOT NULL` 大概率会导致索引失效；<br><br>3. NULL 值与任何值比较都返回 NULL，易导致业务逻辑错误；<br><br>4. NULL 值会占用额外的存储空间。|
|**必须用 NULL 的场景**|仅在 “未知状态” 与 “空状态” 有业务区别时使用，如 “用户未填写生日” 用 NULL，“用户生日为空” 用默认日期，否则统一用默认值代替 NULL。|

### 4.7 字段注释规范

所有字段必须有 `COMMENT` 注释，注释要清晰、完整，包含以下信息：

1. 字段的业务含义；
2. 字段的取值范围（如果是枚举 / 状态值）；
3. 字段的单位（如果是金额、数量、时间）；
4. 字段的特殊说明（如是否加密、是否关联其他表）。

#### 正确示例

```sql
CREATE TABLE user_info (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    user_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '用户姓名',
    phone CHAR(11) NOT NULL DEFAULT '' COMMENT '手机号（唯一标识）',
    age TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '年龄（单位：岁）',
    gender TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '性别：0-未知，1-男，2-女',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户信息表：存储用户的基本信息';
```

## 五、第四部分：索引设计规范（简化版，详见索引优化笔记）

索引设计的详细规范已在之前的《MySQL 索引优化技巧完整学习笔记》中讲解，这里仅列设计阶段的核心规范，避免重复：

1. **优先设计联合索引**：用一个联合索引覆盖多个查询场景，避免多个单值索引；
2. **遵循最左前缀原则**：等值查询列前置，范围查询列后置，高频列前置，高区分度列前置；
3. **覆盖索引优先**：将 SELECT 需要的字段追加到联合索引后缀，避免回表；
4. **单表索引不超过 5 个**：避免索引过多导致写入性能骤降；
5. **低区分度列不单独建索引**：如性别、状态，仅可作为联合索引的后缀列；
6. **索引字段设为 NOT NULL**：避免 NULL 值导致索引失效；
7. **大字段用前缀索引**：长字符串字段用前缀索引，平衡区分度和索引大小。

## 六、第五部分：库表运维与变更规范

### 6.1 库表变更前的准备

1. **测试环境验证**：所有库表变更必须先在测试环境执行，验证表结构、数据、业务逻辑是否正常；
2. **影响评估**：评估变更的影响范围（锁表时间、数据量、业务影响），大表变更必须做详细评估；
3. **回滚方案**：准备完整的回滚 SQL 脚本，变更失败时能快速回滚；
4. **备份数据**：变更前必须备份相关表的数据，避免数据丢失。

### 6.2 库表变更的执行规范

1. **变更时间**：必须在**业务低峰期**执行（如凌晨 2-4 点），避免影响线上业务；
2. **变更方式**：
    
    - 小表（<100 万行）：直接用 `ALTER TABLE`，MySQL 5.6+ 支持 Online DDL，不会锁全表；
    - 大表（>100 万行）：用第三方工具（`pt-online-schema-change`、`gh-ost`）执行，避免锁表和主从延迟；
    
3. **变更监控**：变更过程中监控数据库的 CPU、IO、连接数、慢查询，出现异常立即停止；
4. **变更后验证**：变更完成后，验证表结构、数据、业务接口是否正常，确认无误后再结束。

### 6.3 库表的定期巡检规范

1. **每周巡检**：
    
    - 检查慢查询日志，定位因库表设计导致的慢 SQL；
    - 检查表的索引健康度，清理冗余 / 无效索引；
    
2. **每月巡检**：
    
    - 检查表的数据量增长情况，评估是否需要分库分表 / 归档；
    - 检查表的碎片率，碎片率超过 30% 整理碎片；
    - 更新表的统计信息，避免优化器误判；
    
3. **每季度巡检**：
    
    - 检查库表规范的执行情况，整改不规范的表；
    - 归档历史数据，清理无用表。

## 七、库表设计避坑指南（高频踩坑）

1. **坑 1：用 UUID / 随机字符串做主键**
    
    - 后果：B + 树频繁页分裂，索引碎片严重，写入和查询性能骤降，所有二级索引膨胀；
    - 避坑：统一用自增 BIGINT UNSIGNED 做主键。
    
2. **坑 2：用字符串存储日期 / 金额 / IP**
    
    - 后果：隐式类型转换导致索引失效，查询性能差，金额精度丢失，IP 存储浪费空间；
    - 避坑：日期用 DATETIME，金额用 DECIMAL/BIGINT，IP 用 INT UNSIGNED（INET_ATON/INET_NTOA 转换）。
    
3. **坑 3：字段随便设为 NULL**
    
    - 后果：索引失效，优化器误判，业务逻辑错误；
    - 避坑：所有字段设为 NOT NULL，给合理的默认值。
    
4. **坑 4：VARCHAR 长度随便设 255**
    
    - 后果：浪费磁盘空间，索引膨胀，查询性能下降；
    - 避坑：VARCHAR 长度按需设置，根据实际业务需求确定。
    
5. **坑 5：单表字段过多 / 数据量过大**
    
    - 后果：数据行过大，B + 树每页存储行数少，磁盘 IO 多，查询性能差；
    - 避坑：字段过多垂直拆分，数据量过大冷热分离 / 分库分表。
    
6. **坑 6：没有主键 / 用复合主键**
    
    - 后果：InnoDB 生成隐藏行 ID，无法控制，复合主键导致二级索引膨胀；
    - 避坑：所有表必须有显式自增主键，禁止复合主键。
    
7. **坑 7：用 ENUM 类型**
    
    - 后果：修改枚举值需要 ALTER TABLE，成本高，扩展性差；
    - 避坑：用 TINYINT UNSIGNED 代替，代码里加注释说明取值范围。
    
8. **坑 8：大字段（TEXT/BLOB）存主表**
    
    - 后果：数据行过大，查询性能差，备份 / 恢复慢；
    - 避坑：大字段拆分到详情表，或存对象存储，数据库只存路径。
    
9. **坑 9：库表变更不测试 / 不回滚 / 不在低峰期**
    
    - 后果：锁表导致业务阻塞，数据丢失，无法回滚；
    - 避坑：严格遵守变更规范，测试→评估→备份→低峰期执行→验证。
    
10. **坑 10：没有注释 / 注释不清晰**
    
    - 后果：后续开发 / 运维看不懂表结构，沟通成本高，易出错；
    - 避坑：所有库、表、字段必须有清晰的注释，变更时同步更新。

## 九、核心要点总结

1. **库表设计是根基**：事前规范的库表设计，远胜于事后的 SQL 优化和索引优化，改表成本极高，必须设计先行。
2. **统一规范是基础**：命名、字符集、存储引擎、字段类型、主键、时间字段必须统一规范，避免个性化设计。
3. **主键设计是灵魂**：统一用自增 BIGINT UNSIGNED 做主键，禁止用 UUID、随机字符串、业务字段、复合主键。
4. **字段设计是核心**：用小不用大、用简单不用复杂、避免 NULL、见名知意、清晰注释，每个字段都要仔细斟酌。
5. **性能与可扩展性兼顾**：单表数据量控制、冷热数据分离、预留分库分表空间，提前规划数据增长。
6. **变更与运维是保障**：严格遵守变更规范，定期巡检库表健康度，建立长效运维机制。
7. **避坑是关键**：避免 UUID 做主键、字符串存非字符串数据、字段为 NULL、VARCHAR 随便设 255 等高频坑点。

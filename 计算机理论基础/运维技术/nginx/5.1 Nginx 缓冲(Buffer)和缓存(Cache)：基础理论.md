## 一、核心概念定义与本质区别

### 1.1 缓冲（Buffer）

#### 核心定义

Buffer 是一块用于**暂存数据流**的连续内存区域，核心作用是解决**上下游设备 / 程序之间处理速度不匹配**的问题，通过批量处理减少频繁 IO 交互，实现流量整形，是**以写优化为核心**的性能机制。

#### 核心原理

基于生产者 - 消费者模型，在速度不对等的两个组件之间搭建中间暂存区：生产者将数据先写入 Buffer，攒到指定大小 / 阈值后，一次性批量发送给消费者；消费者从 Buffer 中批量读取数据，避免单字节 / 小包频繁交互带来的系统开销。

#### 典型应用场景

1. **磁盘 IO Buffer**：操作系统写文件时，不会逐字节写入磁盘，先将数据写入 Page Buffer，攒满 4KB/16KB 等磁盘块大小后，一次性写入磁盘，大幅减少磁盘寻道和 IO 次数。
2. **TCP Socket Buffer**：TCP 协议的发送 / 接收缓冲区，发送端应用先把数据写入 Socket Buffer，内核再批量封装发送；接收端内核将收到的数据包暂存到接收 Buffer，应用程序再批量读取，解决网络传输和应用处理的速度差。
3. **Nginx Proxy Buffer**：反向代理场景中，Nginx 从上游服务器快速接收响应数据，先暂存到 Buffer，再适配客户端的网络速度缓慢发送，避免上游服务器因客户端慢而长期占用连接，提升上游并发能力。
4. **文件上传 Buffer**：客户端上传文件时，先将文件流暂存到内存 Buffer，达到阈值后再批量写入磁盘，避免频繁触发磁盘 IO。

---

### 1.2 缓存（Cache）

#### 核心定义

Cache 是一块用于**存储高频访问的热点数据**的高速存储区域（内存 / 高速 SSD），核心作用是**减少重复的计算 / IO 开销**，基于局部性原理，让后续相同请求直接从高速缓存中获取数据，无需重新走完整的处理流程，是**以读优化为核心**的性能机制。

#### 核心原理

基于计算机系统的**局部性原理**：

- 时间局部性：刚被访问过的数据，大概率在短时间内会被再次访问；
- 空间局部性：访问某个数据时，其相邻的数据大概率也会被访问。

将热点数据存储在比原始存储介质（机械硬盘、远程数据库、上游服务）速度快几个数量级的介质中，后续请求直接命中缓存，降低访问延迟，同时大幅减轻后端服务 / 存储的压力。

#### 典型应用场景

1. **CPU Cache**：CPU 的 L1/L2/L3 三级缓存，缓存 CPU 刚用过 / 即将用到的指令和数据，速度远超内存，大幅减少 CPU 访问内存的等待时间。
2. **浏览器 HTTP 缓存**：将静态资源（JS、CSS、图片、字体）存储在本地磁盘 / 内存，后续访问相同 URL 时，直接读取本地缓存，无需重新请求服务器，提升页面加载速度。
3. **Nginx Proxy Cache**：反向代理场景中，将上游服务器返回的静态资源、固定接口响应存储在本地磁盘 / 内存，后续相同请求直接从 Nginx 返回，无需转发到上游服务。
4. **分布式缓存（Redis/Memcached）**：将数据库的热点查询结果、高频访问的业务数据存储在内存中，应用请求先查缓存，命中则直接返回，无需访问数据库，降低数据库负载。
5. **CDN 缓存**：将站点静态资源缓存到全球边缘节点，用户访问时从最近的节点获取资源，减少跨地域网络传输延迟。

---

### 1.3 Buffer 与 Cache 的核心区别与共性

#### 核心区别对照表

|对比维度|Buffer（缓冲）|Cache（缓存）|
|:--|:--|:--|
|**核心目标**|解决上下游速度不匹配，减少 IO 次数，平滑数据流|减少重复计算 / 回源开销，降低访问延迟，提升读性能|
|**核心偏向**|写优化为主，面向数据传输 / 写入过程|读优化为主，面向数据重复读取场景|
|**数据生命周期**|一次性使用，数据被读取 / 转发完成后立即释放，生命周期极短|可重复多次使用，数据会被长期保留，直到过期 / 被清理|
|**数据复用性**|无复用性，仅作为数据传输的临时中转站|高复用性，核心价值就是让热点数据被重复读取|
|**数据新鲜度**|必须保证数据完整、顺序一致，不允许丢数据|允许返回过期数据（降级场景），可牺牲一定新鲜度换性能|
|**失效逻辑**|无过期时间，数据处理完成即释放，满了就触发刷写|有明确的过期策略、淘汰算法（LRU/LFU）、验证机制|
|**存储内容**|正在传输 / 处理的、不完整的数据流|完整的、可直接返回给请求方的响应 / 数据副本|

#### 核心共性

- 均利用内存的高速读写特性，作为两个速度不对等组件的中间层；
- 最终目标都是提升系统整体性能，减少不必要的 IO、CPU、网络开销；
- 均有容量限制，超过阈值后会触发刷盘、丢弃、淘汰等兜底逻辑。

## 二、HTTP 协议中的缓存设置

HTTP 缓存是 Web 场景中最常用的缓存实现，核心目标是减少网络传输、降低服务器负载、提升前端页面加载速度，分为**强缓存**和**协商缓存**两大类。

### 2.1 强缓存（本地缓存）

#### 核心逻辑

浏览器请求资源时，先校验本地缓存是否过期，未过期则**完全不发送 HTTP 请求到服务器**，直接从本地磁盘 / 内存读取资源，性能最高。

#### 核心响应头字段

1. **Expires（HTTP/1.0）**
    
    - 作用：设置缓存的绝对过期时间，格式为 GMT 标准时间。
    - 示例：`Expires: Wed, 21 Oct 2026 07:28:00 GMT`
    - 缺点：依赖客户端本地时间，若客户端时间被修改，会导致缓存判断失效，已被 HTTP/1.1 的 Cache-Control 替代。
    
2. **Cache-Control（HTTP/1.1，主流方案）**
    
- 优先级：高于 Expires，采用相对时间，不受客户端本地时间影响，配置更灵活。

- 核心常用指令：

| 指令                | 作用说明                                                   |
| :---------------- | :----------------------------------------------------- |
| `max-age=xxx`     | 缓存的最大有效时间，单位为秒，从资源返回的时间开始计算，例：`max-age=86400` 表示缓存 1 天 |
| `public`          | 响应可被任何中间节点（浏览器、CDN、代理服务器）缓存，适用于公共静态资源                  |
| `private`         | 响应仅能被客户端浏览器缓存，禁止公共缓存节点缓存，适用于用户私有数据                     |
| `no-cache`        | 并非不缓存，而是强制跳过强缓存，每次使用缓存前必须发请求到服务器验证资源有效性（走协商缓存）         |
| `no-store`        | 完全禁止任何形式的缓存，所有内容都不存储到本地，每次都必须从服务器获取完整响应，适用于敏感数据        |
| `s-maxage=xxx`    | 仅针对共享缓存（CDN、代理服务器）生效，优先级高于 max-age 和 Expires           |
| `must-revalidate` | 缓存一旦过期，必须回源服务器验证，禁止使用过期缓存                              |

---

### 2.2 协商缓存（条件缓存）

#### 核心逻辑

强缓存失效后，浏览器发送**带条件的 HTTP 请求**到服务器，服务器对比资源是否更新：

- 资源无更新：返回`304 Not Modified`，无响应体，浏览器继续使用本地缓存；
- 资源有更新：返回`200 OK`和完整的资源内容，浏览器更新本地缓存。

#### 两组核心头字段

##### 第一组：Last-Modified / If-Modified-Since（基于资源修改时间）

- 执行流程：
    
    1. 服务器首次返回资源时，响应头添加`Last-Modified`，值为资源在服务器上的最后修改时间（GMT 格式），例：`Last-Modified: Tue, 10 Mar 2026 08:00:00 GMT`；
    2. 浏览器缓存资源和该时间，强缓存失效后，再次请求时，请求头添加`If-Modified-Since`，值为之前存储的 Last-Modified 时间；
    3. 服务器收到请求后，对比 If-Modified-Since 的时间和资源当前的最后修改时间：
        
        - 时间一致：资源无更新，返回 304，浏览器复用本地缓存；
        - 时间不一致：资源已更新，返回 200 和新资源，同时更新 Last-Modified。
        
    
- 核心缺点：
    
    1. 时间精度仅到秒级，无法识别 1 秒内的多次资源修改；
    2. 资源仅重新生成、内容无变化时，修改时间会更新，导致缓存无效触发不必要的重传。
    

##### 第二组：ETag / If-None-Match（基于资源内容唯一标识）

- 优先级：高于 Last-Modified/If-Modified-Since，解决了基于时间的缓存缺陷。
- 核心逻辑：ETag 是服务器根据资源内容生成的唯一标识（通常为内容哈希值），资源内容不变，ETag 就不变，与修改时间无关。
- 执行流程：
    
    1. 服务器首次返回资源时，响应头添加`ETag`，值为资源的唯一哈希标识，例：`ETag: "abc123def456ghi789"`；
    2. 浏览器缓存资源和 ETag，强缓存失效后，再次请求时，请求头添加`If-None-Match`，值为之前存储的 ETag；
    3. 服务器收到请求后，对比 If-None-Match 的 ETag 和当前资源的 ETag：
        
        - 标识一致：资源无更新，返回 304，浏览器复用本地缓存；
        - 标识不一致：资源已更新，返回 200 和新资源，同时更新 ETag。
        
    
- 核心优势：精准匹配资源内容，不受修改时间影响，秒级修改可识别，内容不变则不会触发重传。

---

### 2.3 HTTP 缓存完整执行流程

1. 浏览器首次发起资源请求，无任何本地缓存，直接发送请求到服务器；
2. 服务器返回 200 OK 和完整资源，同时返回缓存规则响应头（Cache-Control/Expires、Last-Modified/ETag）；
3. 浏览器将资源和缓存规则存储到本地缓存；
4. 浏览器再次发起相同资源请求：
    
    - 第一步：校验强缓存，判断缓存是否在 max-age/Expires 有效期内，有效则直接读取本地缓存，不发请求，请求状态为`200 OK (from memory/disk cache)`；
    - 第二步：强缓存失效，发起协商缓存请求，携带 If-Modified-Since/If-None-Match 请求头；
    - 第三步：服务器校验资源有效性，无更新则返回 304，浏览器复用本地缓存；有更新则返回 200 和新资源，更新本地缓存规则。

## 三、Nginx 中 Buffer 与 Cache 的配置详解

Nginx 作为主流的反向代理、Web 服务器，Buffer 和 Cache 是其性能调优的核心模块，分别对应数据流传输优化和请求响应缓存优化。

### 3.1 Nginx 中 Buffer 核心配置

Nginx Buffer 的核心价值是解决上下游速度不匹配问题，减少连接占用，提升并发能力，分为客户端请求 Buffer、反向代理 Buffer、FastCGI Buffer 三大类。

#### 3.1.1 客户端请求相关 Buffer

该类配置用于处理客户端到 Nginx 的请求数据，在`http`块全局生效，也可在`server`/`location`块单独调整。

1. **client_body_buffer_size**
    
    - 作用：设置客户端请求体（POST 表单、文件上传等）的内存缓冲区大小。
    - 默认值：32 位系统 8k，64 位系统 16k。
    - 配置说明：请求体小于该值时，直接存储在内存中；超过该值时，会写入磁盘临时文件（`client_body_temp_path`指定目录）。
    - 调优示例：`client_body_buffer_size 10m;`（适用于文件上传、大表单提交场景）。
    
2. **client_header_buffer_size**
    
    - 作用：设置客户端请求头的内存缓冲区大小。
    - 默认值：1k。
    - 配置说明：常规请求头 1k 足够，若存在超长 Cookie、自定义请求头，需调大该值。
    - 调优示例：`client_header_buffer_size 4k;`。
    
3. **large_client_header_buffers**
    
    - 作用：设置超大请求头的缓冲区数量和单缓冲区大小。
    - 格式：`large_client_header_buffers 数量 单缓冲区大小;`
    - 默认值：`4 8k;`
    - 配置说明：请求头超过`client_header_buffer_size`时，会启用该缓冲区；若请求行 / 请求头超过单缓冲区大小，返回 414/400 错误。
    - 调优示例：`large_client_header_buffers 8 16k;`（适用于超长 GET 参数、超长 Cookie 场景）。
    

#### 3.1.2 反向代理 Proxy Buffer（核心常用）

该类配置用于处理 Nginx 到上游服务的反向代理数据流，是 Nginx 性能调优的核心，在`http`/`server`/`location`块均可配置。

1. **proxy_buffering**
    
    - 作用：代理缓冲区总开关，开启 / 关闭缓冲功能。
    - 默认值：`on`（开启）。
    - 配置说明：
        
        - 开启后：Nginx 尽可能将上游响应全部读取到 Buffer 中，再适配客户端速度发送，上游连接读取完成后立即释放，大幅提升上游并发能力；
        - 关闭后：Nginx 收到上游响应会同步转发给客户端，上游连接会持续占用，仅适用于直播、实时消息推送等低延迟要求场景。
        
    - 注意：关闭后，`proxy_buffer_size`仍生效，仅用于存储响应头。
    
2. **proxy_buffer_size**
    
    - 作用：设置上游响应头的缓冲区大小。
    - 默认值：等于系统页大小（通常 4k/8k）。
    - 配置说明：仅用于存储响应头，与响应体无关，若上游存在超长 Set-Cookie、自定义响应头，需调大该值。
    - 调优示例：`proxy_buffer_size 16k;`。
    
3. **proxy_buffers**
    
    - 作用：设置上游响应体的缓冲区数量和单缓冲区大小。
    - 格式：`proxy_buffers 数量 单缓冲区大小;`
    - 默认值：`8 4k/8k;`
    - 配置说明：总缓冲区大小 = 数量 × 单缓冲区大小，响应体小于总大小则全部存储在内存中；超过总大小则写入磁盘临时文件（`proxy_temp_path`指定目录）。
    - 调优建议：根据业务平均响应体大小设置，总大小需覆盖 80% 以上的响应体，避免频繁写磁盘。
    - 调优示例：`proxy_buffers 32 16k;`（总大小 512k，适用于常规接口响应）。
    
4. **proxy_busy_buffers_size**
    
    - 作用：设置处于 “忙碌状态” 的缓冲区最大总大小。
    - 默认值：`proxy_buffers`单缓冲区大小的 2 倍。
    - 配置说明：缓冲区中数据未完全发送给客户端时，处于忙碌状态，该参数限制同时忙碌的缓冲区总大小，保证不会占用全部缓冲区，预留空间用于从上游读取数据。
    - 注意：该值必须小于`proxy_buffers总大小 - proxy_buffer_size`，否则 Nginx 启动报错。
    - 调优示例：`proxy_busy_buffers_size 64k;`。
    
5. **proxy_temp_path**
    
    - 作用：设置代理临时文件的存放路径。
    - 格式：`proxy_temp_path 路径 [层级1] [层级2] [层级3];`
    - 默认值：Nginx 安装目录下的 proxy_temp。
    - 配置说明：响应体超过内存缓冲区大小时，会写入该目录的临时文件，建议部署在 SSD 磁盘上，避免机械硬盘 IO 瓶颈。
    - 调优示例：`proxy_temp_path /var/nginx/proxy_temp 1 2;`。
    

#### 3.1.3 FastCGI Buffer

用于 LNMP 架构中 Nginx 与 PHP-FPM 的交互，逻辑与 Proxy Buffer 完全一致，仅指令前缀为`fastcgi_`，核心指令包括：

- `fastcgi_buffering`：缓冲总开关，默认 on
- `fastcgi_buffer_size`：响应头缓冲区
- `fastcgi_buffers`：响应体缓冲区
- `fastcgi_busy_buffers_size`：忙碌缓冲区大小
- `fastcgi_temp_path`：临时文件路径

### 3.2 Nginx 中 Cache（Proxy Cache）核心配置

Nginx Proxy Cache 是服务端 HTTP 缓存实现，将上游服务的响应存储在本地，后续相同请求直接从 Nginx 返回，无需回源，大幅降低上游负载，提升响应速度。

#### 3.2.1 缓存配置核心步骤

1. 在`http`块中全局定义缓存的基础配置（存储路径、内存 zone、大小、清理规则）；
2. 在`server`/`location`块中开启缓存，配置缓存规则、失效时间、命中逻辑等。

#### 3.2.2 全局缓存定义（http 块）

核心指令为`proxy_cache_path`，**必须在 http 块中配置**，是缓存生效的基础。

```nginx
# 完整的缓存全局配置示例
proxy_cache_path /var/nginx/proxy_cache
   levels=1:2
   keys_zone=common_cache:200m
   max_size=20g
   inactive=7d
   use_temp_path=off;
```

核心参数详解：

|参数|作用说明|
|:--|:--|
|路径（必选）|缓存文件的磁盘存放根目录，建议部署在 SSD 磁盘|
|`levels`|缓存目录的层级，最多 3 级，每级 1-2 个字符，用于分散缓存文件，避免单目录文件过多导致性能下降，例：`levels=1:2` 表示一级目录 1 字符，二级目录 2 字符|
|`keys_zone`|定义内存 zone 的名称和大小，该 zone 用于存储缓存 key 和元数据（失效时间、命中次数），查找缓存时直接查内存，无需遍历磁盘；100m 内存约可存储 80 万条缓存 key，例：`keys_zone=common_cache:200m`|
|`max_size`|缓存占用的最大磁盘空间，超过该值后，Nginx 会通过 LRU 算法清理最少使用的缓存文件|
|`inactive`|非活动失效时间，若一个缓存文件在该时间内未被访问，无论是否在有效期内，都会被强制清理，优先级高于缓存失效时间|
|`use_temp_path`|建议设置为`off`，直接将缓存写入目标目录，避免跨磁盘文件拷贝，提升写入性能|

#### 3.2.3 缓存规则配置（server/location 块）

1. **proxy_cache**
    
    - 作用：开启缓存，指定使用的内存 zone 名称（`proxy_cache_path`中定义的 zone 名）。
    - 示例：`proxy_cache common_cache;`（开启缓存，使用 common_cache zone）；`proxy_cache off;`（关闭当前块的缓存）。
    
2. **proxy_cache_key**
    
    - 作用：定义缓存的唯一 key，Nginx 根据该 key 生成哈希值，作为缓存的唯一标识。
    - 默认值：`$scheme$proxy_host$request_uri;`（协议 + 代理主机 + 完整请求 URI，包含参数）。
    - 自定义示例：

```nginx
# 区分请求方法的缓存key
proxy_cache_key "$request_method$scheme$proxy_host$request_uri";
# 仅缓存URI，忽略所有参数（仅适用于静态资源）
proxy_cache_key "$scheme$proxy_host$uri";

```

3. **proxy_cache_valid**
    
    - 作用：设置不同响应状态码的缓存失效时间。
    - 格式：`proxy_cache_valid [状态码] 失效时间;`
    - 示例：

```nginx
   # 200/301/302响应缓存1小时
   proxy_cache_valid 200 301 302 1h;
   # 404响应缓存1分钟
   proxy_cache_valid 404 1m;
   # 所有状态码缓存5分钟
   proxy_cache_valid any 5m;

```

4. **proxy_cache_min_uses**
    
    - 作用：设置请求被访问多少次后，才会被缓存，默认 1 次。
    - 示例：`proxy_cache_min_uses 3;`（请求被访问 3 次后才缓存，避免低频资源占用磁盘空间）。
    
5. **proxy_cache_methods**
    
    - 作用：设置哪些 HTTP 请求方法会被缓存。
    - 默认值：`GET HEAD;`（仅缓存 GET 和 HEAD 请求）。
    - 示例：`proxy_cache_methods GET HEAD POST;`（同时缓存 POST 请求，仅适用于参数固定的幂等接口）。
    
6. **proxy_cache_bypass & proxy_no_cache**
    
    - `proxy_cache_bypass`：设置条件，条件不为空 / 0 时，跳过缓存，直接回源，即使缓存中有对应内容；
    - `proxy_no_cache`：设置条件，条件不为空 / 0 时，不缓存本次请求的响应，即使符合缓存规则；
    - 常用示例：

```nginx
   # 请求带nocache参数时，跳过缓存且不缓存响应
   proxy_cache_bypass $arg_nocache;
   proxy_no_cache $arg_nocache;
   # 管理员用户跳过缓存
   proxy_cache_bypass $cookie_user_role;
   proxy_no_cache $cookie_user_role;
```

7. **proxy_cache_lock**
    
    - 作用：开启缓存锁，高并发场景下，多个客户端同时请求同一个未缓存的资源时，仅允许第一个请求回源，其他请求等待缓存生成后直接读取，避免缓存击穿。
    - 默认值：`off`，高并发场景强烈建议开启。
    - 配套参数：
        
        - `proxy_cache_lock_timeout`：锁超时时间，默认 5s，超时后请求直接回源；
        - `proxy_cache_lock_age`：锁最大持有时间，默认 5s，超时后释放锁。
        
    - 配置示例：

```nginx
   proxy_cache_lock on;
   proxy_cache_lock_timeout 5s;
```

8. **缓存状态调试配置**
    
    - 作用：给响应头添加缓存命中状态，方便调试和监控。
    - 配置示例：`add_header X-Cache-Status $upstream_cache_status;`
    - 核心状态值：
        
        - `HIT`：缓存命中，直接从 Nginx 缓存返回；
        - `MISS`：缓存未命中，请求回源，响应写入缓存；
        - `EXPIRED`：缓存过期，请求回源；
        - `BYPASS`：跳过缓存，直接回源；
        - `STALE`：返回了过期缓存（降级场景）。

#### 3.2.4 缓存进阶高可用配置

1. **proxy_cache_revalidate**
    
    - 作用：开启缓存重新验证，缓存过期时，Nginx 会携带 If-Modified-Since/If-None-Match 请求头回源验证，资源无更新时，仅更新缓存失效时间，无需重新下载完整响应体，节省带宽。
    - 默认值：`off`，建议开启：`proxy_cache_revalidate on;`。
    
2. **proxy_cache_use_stale**
    
    - 作用：设置异常场景下，Nginx 可返回过期缓存，提升服务可用性，避免上游故障时服务不可用。
    - 常用配置示例：

```nginx
# 上游报错、超时、更新缓存时，返回过期缓存
proxy_cache_use_stale error timeout updating http_500 http_502 http_503 http_504;
```

### 3.3 完整的 Nginx Buffer+Cache 配置示例

#### http 块全局配置

```nginx
http {
    # 客户端请求Buffer全局配置
    client_header_buffer_size 4k;
    large_client_header_buffers 8 16k;
    client_body_buffer_size 10m;

    # 反向代理Buffer全局配置
    proxy_buffering on;
    proxy_buffer_size 16k;
    proxy_buffers 32 16k;
    proxy_busy_buffers_size 64k;
    proxy_temp_file_write_size 128k;
    proxy_temp_path /var/nginx/proxy_temp 1 2;

    # 代理缓存全局定义
    proxy_cache_path /var/nginx/proxy_cache
        levels=1:2
        keys_zone=common_cache:200m
        max_size=20g
        inactive=7d
        use_temp_path=off;
    
    # 缓存锁全局配置
    proxy_cache_lock on;
    proxy_cache_lock_timeout 5s;

    # 其他基础配置
    include mime.types;
    default_type application/octet-stream;
    sendfile on;
    tcp_nopush on;
    keepalive_timeout 65;

    include /etc/nginx/conf.d/*.conf;
}
```

#### server 块业务配置

```nginx
server {
    listen 80;
    server_name example.com;
    root /usr/share/nginx/html;
    index index.html index.htm;

    # 静态资源：开启Buffer+Cache，配置HTTP缓存
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
        proxy_pass http://upstream_server;
        
        # 缓存配置
        proxy_cache common_cache;
        proxy_cache_key "$scheme$proxy_host$request_uri";
        proxy_cache_valid 200 301 302 7d;
        proxy_cache_valid 404 1m;
        proxy_cache_min_uses 2;
        proxy_cache_revalidate on;
        proxy_cache_use_stale error timeout updating http_500 http_502 http_503 http_504;
        proxy_cache_bypass $arg_nocache;
        proxy_no_cache $arg_nocache;
        
        # 给客户端设置HTTP强缓存
        expires 7d;
        add_header Cache-Control "public, max-age=604800";
        
        # 缓存状态头
        add_header X-Cache-Status $upstream_cache_status;
    }

    # 动态接口：仅开启Buffer，关闭缓存
    location ~* \.(php|jsp|do|api)$ {
        proxy_pass http://upstream_server;
        
        # 关闭缓存
        proxy_cache off;
        # 接口专属Buffer调优
        proxy_buffer_size 16k;
        proxy_buffers 16 16k;
        proxy_busy_buffers_size 32k;
    }

    # 实时数据流：关闭Buffer+缓存，低延迟转发
    location /live/ {
        proxy_pass http://upstream_live;
        
        # 关闭缓冲，实时转发
        proxy_buffering off;
        # 关闭缓存
        proxy_cache off;
        # 关闭TCP_NODELAY，降低延迟
        tcp_nodelay on;
    }
}
```

## 四、常见误区与最佳实践

### 4.1 Buffer 相关误区与最佳实践

1. **误区**：Buffer 开得越大越好
    
    - 正解：Buffer 过大会占用过多内存，导致 Nginx 触发 OOM，同时会增加客户端等待时间；最佳实践是根据业务平均响应体大小设置，总 Buffer 大小覆盖 80% 的响应体即可。
    
2. **误区**：所有场景都开启 proxy_buffering
    
    - 正解：直播、websocket、实时消息推送等低延迟场景，必须关闭 proxy_buffering，否则会出现数据延迟；常规 Web 站点、接口开启后可大幅提升上游并发能力。
    
3. **误区**：忽略临时文件的磁盘性能
    
    - 正解：响应体超过 Buffer 大小时会写入临时文件，建议将临时文件目录部署在 SSD 上，或调大 Buffer 避免写临时文件，防止机械硬盘 IO 瓶颈。
    

### 4.2 Cache 相关误区与最佳实践

1. **误区**：所有内容都可以缓存
    
    - 正解：仅缓存幂等、不频繁更新的内容（静态资源、固定接口、列表页），禁止缓存用户私有数据、实时数据、非幂等 POST 接口、个性化内容，避免数据错乱和信息泄露。
    
2. **误区**：缓存有效期设置过长 / 过短
    
    - 正解：带哈希后缀的静态资源可设置长缓存（30 天以上）；频繁更新的内容设置短缓存（1-5 分钟），配合 revalidate 和 stale 配置，平衡新鲜度和性能。
    
3. **误区**：忽略 inactive 参数
    
    - 正解：inactive 是强制清理，优先级高于缓存有效期，建议设置为缓存有效期的 2-3 倍，避免无效缓存长期占用磁盘空间。
    
4. **误区**：高并发场景不开启缓存锁
    
    - 正解：热点缓存失效时，大量并发请求会同时回源，打垮上游服务，高并发场景必须开启 proxy_cache_lock。
    
5. **误区**：不配置 stale 降级策略
    
    - 正解：proxy_cache_use_stale 是提升服务可用性的核心配置，上游故障、超时时可返回过期内容，避免用户看到 5xx 错误，强烈建议开启。
    

### 4.3 核心区分总结

- 当你需要解决**上下游速度不匹配，减少 IO 次数，平滑数据流，提升写入 / 传输效率**时，使用 Buffer；
- 当你需要解决**重复请求的开销，减少回源 / 计算，降低读延迟，提升读性能**时，使用 Cache；
- Buffer 中的数据是一次性的，处理完成即释放；Cache 中的数据是可复用的，会被长期保留、多次读取。

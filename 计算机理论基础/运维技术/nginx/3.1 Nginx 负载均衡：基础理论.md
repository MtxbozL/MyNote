## 一、Upstream 模块

`upstream` 是 Nginx 实现负载均衡的核心模块，用于定义一组后端服务器，并指定请求分发的规则。

### 基本语法

```nginx
http {
    upstream backend_servers {
        # 定义后端服务器及参数
        server 10.0.0.1:8080 weight=5;
        server 10.0.0.2:8080 backup;
        # ...
    }

    server {
        location / {
            proxy_pass http://backend_servers;
        }
    }
}
```

## 二、负载均衡策略

### 1. 最小连接（least_conn）

- **原理**：将请求优先分配给当前活跃连接数最少的后端服务器。
- **适用场景**：后端服务器处理能力差异较大，或请求处理时间差异较大的场景。
- **配置**：

```nginx
upstream backend_servers {
    least_conn;
    server 10.0.0.1:8080;
    server 10.0.0.2:8080;
}
```


### 2. 随机（random）

- **原理**：随机选择一台后端服务器处理请求。
- **适用场景**：后端服务器配置和负载相近，且对请求分发顺序无特殊要求。
- **配置**：

```nginx
    upstream backend_servers {
        random;
        server 10.0.0.1:8080;
        server 10.0.0.2:8080;
    }
```

### 3. 轮询（round-robin，默认）

- **原理**：按顺序将请求逐一分配到不同的后端服务器。
- **适用场景**：后端服务器配置相同，且请求处理时间均匀的场景。
- **特点**：无需额外配置，默认启用。

### 4. IP 哈希（ip_hash）

- **原理**：根据客户端 IP 地址的哈希值，将请求固定分配到某一台后端服务器。
- **适用场景**：需要保持会话一致性（如购物车、登录状态）的应用。
- **注意**：后端服务器下线后，对应客户端的会话会丢失。
- **配置**：

```nginx
    upstream backend_servers {
        ip_hash;
        server 10.0.0.1:8080;
        server 10.0.0.2:8080;
    }
```

### 5. 一致性哈希（consistent_hash）

- **原理**：使用一致性哈希算法，将请求映射到后端服务器。当后端服务器数量变化时，仅影响少量请求的映射关系。
- **适用场景**：缓存服务器集群（如 Redis、Memcached），或需要减少服务器上下线影响的场景。
- **依赖**：需要 Nginx 官方的 `ngx_http_upstream_consistent_hash` 模块或 OpenResty。
- **配置**：

```nginx
    upstream backend_servers {
        consistent_hash $request_uri;  # 按请求 URI 哈希
        server 10.0.0.1:8080;
        server 10.0.0.2:8080;
    }
```

### 6. 权重（weight）

- **原理**：在轮询或其他策略基础上，为不同后端服务器设置权重，权重越高，被分配的请求越多。
- **适用场景**：后端服务器硬件配置或处理能力不同的场景。
- **配置**：

```nginx
    upstream backend_servers {
        server 10.0.0.1:8080 weight=5;  # 权重 5
        server 10.0.0.2:8080 weight=2;  # 权重 2
        server 10.0.0.3:8080 weight=3;  # 权重 3
    }
```

## 三、后端服务器参数

在 `upstream` 中，还可以为每个后端服务器配置额外参数，以控制其行为：

|参数|含义|
|:--|:--|
|`backup`|标记为备份服务器，仅在其他服务器不可用时才会被使用。|
|`down`|标记为永久下线，Nginx 不会向其分发请求。|
|`max_fails`|在 `fail_timeout` 时间内，最大失败次数，超过则标记为不可用。|
|`fail_timeout`|服务器被标记为不可用的时间，之后会重新尝试连接。|
|`max_conns`|该服务器的最大并发连接数，超过则排队。|

### 示例

```nginx
upstream backend_servers {
    server 10.0.0.1:8080 weight=5 max_fails=3 fail_timeout=30s;
    server 10.0.0.2:8080 weight=2 max_fails=3 fail_timeout=30s;
    server 10.0.0.3:8080 backup;  # 备份服务器
}
```

## 四、健康检查

Nginx 提供了被动和主动两种健康检查方式，确保请求只分发到健康的后端服务器。

### 1. 被动健康检查（原生支持）

- **原理**：根据请求的失败次数（如 502/503 错误、连接超时）来判断服务器是否健康。
- **配置**：通过 `max_fails` 和 `fail_timeout` 实现，已在上文示例中展示。

### 2. 主动健康检查（需额外模块）

- **原理**：Nginx 定期向后端服务器发送健康检查请求（如 HTTP GET），根据响应判断服务器状态。
- **依赖**：`ngx_http_healthcheck_module` 或商业版 Nginx Plus。
- **配置示例**：

```nginx
    upstream backend_servers {
        server 10.0.0.1:8080;
        server 10.0.0.2:8080;
        healthcheck interval=5s rise=2 fall=3 timeout=2s type=http;
        healthcheck_request uri=/health method=GET;
        healthcheck_status 200;
    }
```

## 五、完整配置示例

```nginx
http {
    # 1. 定义上游服务池：负载均衡的后端节点集合
    upstream my_app {
        least_conn;  # 负载均衡策略：最小连接数 + 权重
        server 192.168.1.10:8080 weight=5 max_fails=3 fail_timeout=30s;
        server 192.168.1.11:8080 weight=3 max_fails=3 fail_timeout=30s;
        server 192.168.1.12:8080 backup;  # 备份节点
    }

    # 2. 定义对外服务：监听 80 端口，接收客户端请求
    server {
        listen 80;
        server_name app.example.com;  # 仅处理该域名的请求

        # 3. 转发规则：所有请求转发到上游服务池
        location / {
            proxy_pass http://my_app;  # 转发到 my_app 上游池
            proxy_set_header Host $host;  # 传递真实域名给后端
            proxy_set_header X-Real-IP $remote_addr;  # 传递客户端真实IP
        }
    }
}
```

## 六、策略选择建议

|业务场景|推荐策略|核心原因|
|:--|:--|:--|
|通用 Web 服务，后端配置相同|轮询（round-robin）|简单高效，无需额外配置|
|后端处理能力差异大|最小连接（least_conn）+ 权重（weight）|避免性能差的服务器过载|
|需要保持会话一致性|IP 哈希（ip_hash）|同一客户端始终访问同一服务器|
|缓存服务器集群|一致性哈希（consistent_hash）|减少服务器上下线对缓存命中率的影响|
|后端服务器负载波动大|随机（random）|

## 一、HTTPS 基础与 Nginx 配置核心标识

### 1.1 基础概念

HTTPS（Hyper Text Transfer Protocol Secure）= HTTP + TLS/SSL 协议，基于**非对称加密 + 对称加密**实现传输层全程加密，**默认端口 443**，彻底解决 HTTP 明文传输的窃听、篡改、伪造三大核心风险。

Nginx 实现 HTTPS 依赖 `http_ssl_module`模块 ，编译 Nginx 时必须添加`--with-http_ssl_module`参数开启（主流 Linux 发行版的二进制安装包默认已自带），可通过`nginx -V`命令查看是否已编译该模块。

### 1.2 核心配置标识：端口开启 ssl

Nginx 中启用 HTTPS 服务的**核心必填标识**，是在`listen`监听端口指令后添加`ssl`参数，这是当前主流版本（Nginx 1.15.0+）的标准开启方式，旧版本废弃的`ssl on;`指令不推荐使用。

基础语法示例：

```nginx
server {
    # 核心：443端口开启ssl，启用HTTPS服务
    listen 443 ssl;
    # 绑定域名，必须与SSL证书的域名保持一致
    server_name example.com www.example.com;
}
```

## 二、HTTPS 证书准备与完整可落地配置

### 2.1 SSL 证书类型与文件准备

要配置 HTTPS，必须先获取 CA 机构签发的 SSL/TLS 证书，按验证等级分为三类：

|证书类型|特点|适用场景|
|:--|:--|:--|
|DV 域名验证型|免费（Let's Encrypt、云厂商免费证书），仅验证域名所有权，分钟级签发|个人站点、测试环境、非核心业务|
|OV 组织验证型|付费，验证企业主体身份，浏览器展示企业信息|企业官网、中小业务站点|
|EV 扩展验证型|高价付费，最高级实名验证，浏览器地址栏直接展示企业名称|金融支付、政务、强安全需求场景|

#### Nginx 所需核心证书文件

- **证书文件**：后缀通常为`.crt`/`.pem`，包含公钥、域名信息、证书链、CA 颁发机构信息，可公开分发
- **私钥文件**：后缀通常为`.key`，与证书公钥配对，必须严格保密，不可泄露，文件权限建议设置为`600`（仅所有者可读可写）

### 2.2 完整基础配置（开箱即用）

```nginx
# http全局块配置，可全局复用SSL基础规则
http {
    # 全局SSL基础配置（也可在server块单独配置）
    # 禁用不安全的SSLv3、TLSv1.0、TLSv1.1，仅保留安全协议版本
    ssl_protocols TLSv1.2 TLSv1.3;
    # 优先使用高安全、高性能的加密套件
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305:DHE-RSA-AES128-GCM-SHA256:DHE-RSA-AES256-GCM-SHA384;
    ssl_prefer_server_ciphers on; # 优先使用服务端加密套件顺序

    # HTTP强制跳转HTTPS（必配，消除HTTP明文访问）
    server {
        listen 80;
        server_name example.com www.example.com;
        # 301永久跳转，保留原请求uri，SEO友好
        return 301 https://$host$request_uri;
    }

    # HTTPS核心服务块
    server {
        # 核心标识：443端口开启SSL，同时开启HTTP/2提升性能
        listen 443 ssl http2;
        server_name example.com www.example.com;

        # ========== 核心必填：证书与私钥配置 ==========
        ssl_certificate /etc/nginx/ssl/example.com.pem; # 证书文件绝对路径
        ssl_certificate_key /etc/nginx/ssl/example.com.key; # 私钥文件绝对路径

        # ========== 站点基础配置 ==========
        root /usr/share/nginx/html;
        index index.html index.htm;

        # 静态资源/前端站点配置
        location / {
            try_files $uri $uri/ /index.html;
        }

        # 反向代理场景配置（后端接口转发常用）
        location /api/ {
            proxy_pass http://127.0.0.1:8080;
            proxy_set_header Host $host;
            proxy_set_header X-Real-IP $remote_addr;
            proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
            # 传递HTTPS协议标识给后端，避免后端协议识别错误
            proxy_set_header X-Forwarded-Proto $scheme;
        }
    }
}
```

### 2.3 配置生效与验证

1. **语法校验**：执行`nginx -t`，必须提示`syntax is ok`和`test is successful`，否则先修复配置错误
2. **重载生效**：执行`nginx -s reload`，平滑重载配置，不影响在线业务
3. **访问验证**：浏览器访问`https://你的域名`，查看地址栏是否出现安全小锁，点击小锁可查看证书详情，确认生效

## 三、Nginx 配置 HTTPS 的核心优点

### 3.1 基础安全价值

1. **全程加密防窃听**：TLS 协议对 HTTP 传输内容全程加密，避免运营商劫持、中间人窃听，杜绝账号密码、用户敏感数据明文泄露。
2. **防篡改防伪造**：通过证书身份校验和哈希完整性校验，确保传输内容不被篡改，同时验证服务端真实身份，彻底防范钓鱼站点仿冒。
3. **合规必备**：等保 2.0、金融监管、小程序 / APP 上架、支付接口对接，均强制要求使用 HTTPS 加密传输，纯 HTTP 站点会被限制访问或拒绝上架。

### 3.2 业务与用户体验优势

1. **SEO 排名提升**：谷歌、百度等主流搜索引擎均将 HTTPS 作为排名权重因子，优先收录 HTTPS 站点，提升自然搜索流量。
2. **消除浏览器安全警告**：Chrome、Edge、Firefox 等主流浏览器会对纯 HTTP 站点标记「不安全」红色警告，HTTPS 站点消除警告，大幅提升用户信任度。
3. **支持现代 Web 特性**：HTTP/2、HTTP/3、Service Worker、WebRTC、地理位置 API 等所有现代浏览器高级特性，均要求站点必须使用 HTTPS，纯 HTTP 站点无法使用。

### 3.3 架构核心优势：降低后端服务压力

这是反向代理架构下的核心价值，也就是 **SSL 卸载 / SSL 终结** 能力：

1. **加解密运算集中卸载**：将 HTTPS 的 TLS 握手、非对称加密、对称加解密等 CPU 密集型操作，全部在 Nginx 反向代理层完成，后端服务只需处理纯 HTTP 请求，无需消耗 CPU 资源进行加解密运算，大幅降低后端负载。
2. **连接复用降开销**：Nginx 与客户端建立 HTTPS 长连接，与后端服务使用长连接池复用连接，大幅减少后端服务的 TCP 连接数、进程 / 线程资源占用，提升后端并发能力。
3. **统一证书与运维管理**：多后端服务集群场景下，只需在 Nginx 层统一配置、更新 SSL 证书，无需每个后端服务单独部署，大幅降低运维成本，避免证书过期风险。
4. **恶意请求前置拦截**：HTTPS 解密后，Nginx 可直接实现 WAF 防护、CC 攻击拦截、访问控制、黑白名单过滤，恶意请求直接在边缘层拦截，不会到达后端服务，进一步降低后端压力。

## 四、Nginx HTTPS 深度性能与安全优化

HTTPS 的性能损耗主要来自 **TLS 握手（非对称加密）** 和 **加解密运算**，优化核心是减少握手次数、降低 CPU 开销、提升安全等级，同时兼顾兼容性。

### 4.1 性能优化：极致降低 HTTPS 访问延迟

#### 1. 开启 TLS 会话复用，消除重复握手开销

TLS 完整握手需要 2 次 RTT 往返，是 HTTPS 最大的性能开销，会话复用可让重复访问的客户端跳过完整握手，直接复用已有会话，握手耗时降为 0，性能提升 50% 以上。

```nginx
# 方案1：SSL会话缓存（兼容性最好，主流方案）
# 开辟10M共享内存存储会话，1M内存约可存储4000个会话
ssl_session_cache shared:SSL_SESSION:10m;
ssl_session_timeout 1d; # 会话有效期1天

# 方案2：TLS会话票证（无状态复用，多机集群必备）
ssl_session_tickets on;
# 多机集群需统一key文件，确保不同节点会话可复用
ssl_session_ticket_key /etc/nginx/ssl/session_ticket.key;
```

#### 2. 开启 OCSP 装订，消除证书校验延迟

客户端访问 HTTPS 时，会主动访问 CA 机构验证证书是否吊销，海外访问延迟极高。OCSP 装订（OCSP Stapling）是 Nginx 提前获取 OCSP 验证结果，随证书一起发送给客户端，客户端无需单独访问 CA 服务器，大幅减少握手延迟。

- 什么是 OCSP？ ----> [[OCSP 装订（OCSP Stapling）完整详解]]

```nginx
# OCSP装订核心配置
ssl_stapling on;
ssl_stapling_verify on;
# 根证书/中间证书链，用于验证OCSP响应合法性
ssl_trusted_certificate /etc/nginx/ssl/ca_chain.pem;
# DNS服务器，用于解析CA的OCSP地址
resolver 223.5.5.5 114.114.114.114 valid=300s;
resolver_timeout 5s;
```

#### 3. 启用 HTT2 / HTTP3，提升传输性能

HTTP/2 基于二进制帧、多路复用，彻底解决 HTTP/1.1 的队头阻塞问题，HTTPS 场景下所有主流浏览器均已支持，开启后页面加载速度提升 30% 以上；HTTP/3 基于 QUIC 协议，弱网环境性能更强。

```nginx
# 开启HTTP/2，直接在listen指令添加http2参数即可
listen 443 ssl http2;

# 进阶：Nginx 1.25.0+ 支持HTTP/3（QUIC），需编译对应模块
listen 443 quic reuseport;
# 通知浏览器当前站点支持HTTP/3
add_header Alt-Svc 'h3=":443"; ma=86400' always;
```

#### 4. 加密算法与套件优化，降低 CPU 开销

优先选择**椭圆曲线加密（ECDHE）+ GCM 认证加密**套件，椭圆曲线加密运算量远低于 RSA，GCM 加密性能和安全性双优，同时禁用低性能、不安全的加密套件。

```nginx
# 按优先级排序，优先使用高性能高安全的加密套件
ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256:ECDHE-ECDSA-AES256-GCM-SHA384:ECDHE-RSA-AES256-GCM-SHA384:ECDHE-ECDSA-CHACHA20-POLY1305:ECDHE-RSA-CHACHA20-POLY1305;
# 优先使用服务端的加密套件顺序
ssl_prefer_server_ciphers on;
# 椭圆曲线优化，优先使用高性能的X25519曲线
ssl_ecdh_curve X25519:prime256v1:secp384r1;
```

### 4.2 安全优化：提升站点安全等级，防范漏洞

#### 1. 禁用不安全的 TLS 协议版本

SSLv3、TLSv1.0、TLSv1.1 均存在严重安全漏洞（如 POODLE、BEAST 攻击），必须禁用，仅保留安全的 TLSv1.2 和 TLSv1.3：

```nginx
ssl_protocols TLSv1.2 TLSv1.3;
```

#### 2. 开启 HSTS，防止降级攻击

HSTS（HTTP Strict Transport Security）强制浏览器在本地只使用 HTTPS 访问站点，即使输入 HTTP 地址，也会在浏览器内部直接跳转到 HTTPS，防止中间人降级攻击，同时消除 HTTP 到 HTTPS 的 301 跳转开销。

```nginx
# 有效期1年，包含所有子域名，支持浏览器预加载
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains; preload" always;
```

⚠️ 注意：`always`参数必须添加，确保错误响应也会返回 HSTS 头；配置前需确保所有子域名都支持 HTTPS，否则会导致子域名无法访问。

#### 3. 安全响应头配置，防范 Web 攻击

```nginx
# 防XSS跨站脚本攻击
add_header X-XSS-Protection "1; mode=block" always;
# 防iframe嵌套点击劫持
add_header X-Frame-Options SAMEORIGIN always;
# 禁用MIME类型嗅探，防文件上传漏洞
add_header X-Content-Type-Options nosniff always;
# 内容安全策略，限制资源加载，从根源防范XSS
add_header Content-Security-Policy "default-src 'self'; script-src 'self' 'unsafe-inline'; style-src 'self' 'unsafe-inline'" always;
```

#### 4. 禁用不安全特性

```nginx
# 禁用SSL压缩，防止CRIME攻击
ssl_compression off;
# 禁用不安全的TLS重协商，防止中间人攻击
ssl_renegotiation off;
```

## 五、证书自动续期方案

免费 DV 证书（如 Let's Encrypt）有效期仅 90 天，手动续期运维成本高，推荐使用`certbot`工具实现全自动申请、续期、配置更新。

### 核心操作步骤

1. 安装 certbot 与 Nginx 插件

```bash
    # CentOS/RHEL系统
    yum install certbot python3-certbot-nginx -y
    
    # Ubuntu/Debian系统
    apt update && apt install certbot python3-certbot-nginx -y
```
    
2. 一键申请证书并自动生成 Nginx 配置

```bash
    certbot --nginx -d example.com -d www.example.com
```
    
3. 配置自动续期定时任务

```bash
    # 编辑定时任务
    crontab -e
    # 添加任务：每天凌晨3点检查证书续期，续期成功自动重载Nginx
    0 3 * * * certbot renew --quiet && nginx -s reload
```

## 六、常见问题与排查方案

### 6.1 浏览器提示「您的连接不是私密连接」

常见原因与解决方案：

1. 证书与域名不匹配：检查证书绑定的域名是否与`server_name`一致，泛域名证书是否覆盖当前子域名
2. 证书过期：点击浏览器地址栏小锁，查看证书有效期，及时更新证书
3. 证书链不完整：证书文件未包含中间证书，导致浏览器无法验证 CA 机构，需补全完整证书链
4. 客户端系统时间错误：系统时间偏差过大，导致证书有效期校验失败

### 6.2 HTTPS 可访问，HTTP 无法访问 / 不跳转

常见原因：

1. 服务器防火墙 / 云厂商安全组未开放 80 端口
2. Nginx 未配置 80 端口的监听 server 块，或跳转规则语法错误
3. HSTS 已生效，浏览器内部直接完成 HTTPS 跳转，属于正常现象

### 6.3 配置重载报错「SSL: error:02001002:system library:fopen:No such file or directory」

原因：证书 / 私钥文件路径错误，或 Nginx 进程无读取权限

解决方案：

1. 使用绝对路径配置证书文件，检查路径是否正确
2. 给证书文件设置`644`权限，私钥文件设置`600`权限，所有者改为 Nginx 运行用户（nginx/www-data）
3. 关闭 SELinux，或给证书文件设置正确的 SELinux 上下文

### 6.4 反向代理后，后端无法识别 HTTPS 协议

原因：后端未收到协议标识，导致跳转、链接生成错误

解决方案：在反向代理配置中添加`proxy_set_header X-Forwarded-Proto $scheme;`，后端通过该请求头判断当前协议类型。

### 6.5 浏览器提示「混合内容」警告

原因：HTTPS 页面中加载了 HTTP 协议的资源（图片、JS、CSS、接口），主流浏览器会拦截这些资源，导致页面显示异常

解决方案：将页面中所有 HTTP 资源改为 HTTPS 协议，或使用相对协议`//`，彻底消除混合内容。

## 一、高可用核心概述

Nginx 作为业务流量的统一入口（反向代理、负载均衡、静态资源服务），是整个业务架构的第一道关口，一旦 Nginx 节点故障，会导致全业务不可用。

Nginx 高可用架构的核心目标：**彻底消除流量入口的单点故障，实现故障秒级自动切换、业务无感知，保障服务 7×24 小时连续可用，同时支持水平扩展应对高并发流量冲击**。

主流高可用实现分为两大核心方向，完全对应大纲内容：

1. **主从高可用**：基于`Keepalived + Nginx`实现，解决单台 Nginx 的单点故障问题，是中小规模业务、单机房架构的首选方案；
2. **集群化高可用**：基于多节点 Nginx 集群部署，配合上层四层负载均衡实现流量分发，支持自动扩缩容，解决大规模业务、高并发流量的性能与可用性问题，是互联网核心业务的标准架构。

## 二、Keepalived + Nginx 主从高可用（生产主流方案）

### 2.1 核心实现原理

#### 1. 基础协议支撑：VRRP 协议

VRRP（Virtual Router Redundancy Protocol，虚拟路由冗余协议）是该方案的核心，其设计目标是解决静态网关的单点故障问题：

- 将多台服务器（Nginx 节点）组成一个**虚拟路由器组**，对外提供一个固定的**虚拟 IP（VIP）**；
- 同一时间，只有一台主节点（Master）持有并对外宣告 VIP，处理所有业务流量；
- 主节点故障时，备节点（Backup）会在毫秒级内完成选举，接管 VIP，实现流量无缝切换，用户完全无感知。

#### 2. Keepalived 核心组件

Keepalived 是 VRRP 协议的开源实现，专为 Linux 服务器高可用设计，三大核心模块支撑 Nginx 高可用：

|模块|核心作用|
|---|---|
|VRRP 协议栈|实现主备节点的选举、VIP 管理、心跳包收发，是核心调度层|
|健康检查模块|支持自定义脚本检测 Nginx 服务状态，不仅能感知服务器宕机，还能感知 Nginx 进程异常、服务不可用，触发主备切换|
|通知告警模块|主备切换时可触发自定义脚本、邮件 / 短信 / 企业微信告警，及时通知运维人员|

#### 3. 核心解决的痛点

- 解决 Nginx 服务器宕机导致的业务中断；
- 解决 Nginx 进程崩溃、服务异常但服务器正常运行的场景（默认 VRRP 仅感知服务器宕机，需自定义脚本补充）；
- 实现秒级故障切换，RTO（恢复时间目标）可控制在 1-3 秒内，用户无感知。

### 2.2 环境准备

|节点角色|服务器 IP|核心配置|
|---|---|---|
|Master 主节点|192.168.1.10|安装 Nginx、Keepalived，优先级高于备节点|
|Backup 备节点|192.168.1.11|安装 Nginx、Keepalived，优先级低于主节点|
|虚拟 IP（VIP）|192.168.1.200|同网段未被占用的固定 IP，对外提供服务|

前置要求：

1. 两台服务器同网段互通，关闭 SELinux，防火墙放行 VRRP 协议（IP 协议号 112）；
2. 两台服务器均完成 Nginx 编译安装 / 二进制安装，配置文件一致，服务可正常启动；
3. 服务器时间同步，避免日志、心跳异常；
4. 配置好 yum/apt 源，可直接安装 Keepalived。

### 2.3 完整可落地配置步骤

#### 步骤 1：两台服务器均安装 Keepalived

```bash
# CentOS/RHEL系统
yum install keepalived -y
# Ubuntu/Debian系统
apt update && apt install keepalived -y

# 设置开机自启
systemctl enable keepalived
```

#### 步骤 2：编写 Nginx 健康检查脚本（核心）

默认 Keepalived 仅检测服务器是否宕机，无法感知 Nginx 服务是否正常，必须通过自定义脚本实现 Nginx 服务状态检测，故障时触发切换。

**两台服务器均创建相同的脚本**：`/etc/keepalived/check_nginx.sh`

```bash
#!/bin/bash
# 检测Nginx进程是否存在
nginx_process=$(ps -C nginx --no-header | wc -l)
# 进程不存在，尝试重启Nginx
if [ $nginx_process -eq 0 ]; then
    systemctl restart nginx
    sleep 2
    # 重启后仍不存在，停止Keepalived，触发VIP漂移
    nginx_process=$(ps -C nginx --no-header | wc -l)
    if [ $nginx_process -eq 0 ]; then
        systemctl stop keepalived
    fi
fi
```

给脚本添加执行权限：

```bash
chmod +x /etc/keepalived/check_nginx.sh
```

脚本逻辑说明：

1. 定时检测 Nginx 进程是否存活；
2. 进程不存在时，尝试重启 Nginx；
3. 重启失败，直接停止 Keepalived 服务，主节点释放 VIP，备节点自动接管。

#### 步骤 3：Master 主节点 Keepalived 核心配置

编辑主节点配置文件：`/etc/keepalived/keepalived.conf`

```nginx
! 全局配置
global_defs {
    router_id nginx_master  # 节点标识，主备节点需不同
    script_user root        # 执行健康检查脚本的用户
    enable_script_security  # 开启脚本安全校验
}

! 健康检查脚本定义
vrrp_script check_nginx {
    script "/etc/keepalived/check_nginx.sh"  # 脚本路径
    interval 2                                 # 检测间隔，单位秒，每2秒执行一次
    weight -20                                 # 检测失败，节点优先级降低20
    fall 2                                     # 连续2次检测失败，判定为故障
    rise 1                                     # 连续1次检测成功，判定为恢复
}

! VRRP实例配置
vrrp_instance VI_1 {
    state MASTER              # 角色：MASTER主节点 / BACKUP备节点
    interface ens33           # 网卡名称，需和服务器实际网卡一致（ifconfig/ip addr查看）
    virtual_router_id 51      # 虚拟路由ID，0-255，主备节点必须完全一致
    priority 100              # 优先级，主节点必须高于备节点，建议主100，备80
    advert_int 1              # 心跳包发送间隔，单位秒，每1秒发送一次VRRP心跳

    ! 认证配置，主备节点必须完全一致
    authentication {
        auth_type PASS
        auth_pass 123456
    }

    ! 虚拟IP（VIP）配置，可配置多个
    virtual_ipaddress {
        192.168.1.200/24 dev ens33 label ens33:0
    }

    ! 绑定健康检查脚本
    track_script {
        check_nginx
    }

    ! 切换通知脚本（可选），主节点切换为备节点时执行
    notify_backup "/etc/keepalived/notify_backup.sh"
    ! 备节点切换为主节点时执行
    notify_master "/etc/keepalived/notify_master.sh"
    ! 节点故障时执行
    notify_fault "/etc/keepalived/notify_fault.sh"
}
```

#### 步骤 4：Backup 备节点 Keepalived 核心配置

编辑备节点配置文件：`/etc/keepalived/keepalived.conf`，**仅需修改 3 个核心参数，其余与主节点完全一致**：

```nginx
! 全局配置
global_defs {
    router_id nginx_backup  # 节点标识，与主节点区分
    script_user root
    enable_script_security
}

! 健康检查脚本，与主节点完全一致
vrrp_script check_nginx {
    script "/etc/keepalived/check_nginx.sh"
    interval 2
    weight -20
    fall 2
    rise 1
}

! VRRP实例配置
vrrp_instance VI_1 {
    state BACKUP              # 角色改为BACKUP备节点
    interface ens33           # 与主节点同网卡名
    virtual_router_id 51      # 与主节点完全一致
    priority 80               # 优先级低于主节点（主100，备80）
    advert_int 1

    ! 认证配置，与主节点完全一致
    authentication {
        auth_type PASS
        auth_pass 123456
    }

    ! VIP配置，与主节点完全一致
    virtual_ipaddress {
        192.168.1.200/24 dev ens33 label ens33:0
    }

    ! 绑定健康检查脚本
    track_script {
        check_nginx
    }

    ! 通知脚本可选，与主节点一致
}
```

#### 步骤 5：配置同步与防火墙放行

1. **Nginx 配置同步**：主备节点的 Nginx 配置必须完全一致，避免切换后业务异常，推荐使用`rsync+inotify`实现配置自动同步，或 Ansible 批量管理。
2. **防火墙放行 VRRP 协议**：

  ```bash
    # firewalld放行
    firewall-cmd --permanent --add-rich-rule='rule protocol value="vrrp" accept'
    firewall-cmd --reload
    
    # iptables放行
    iptables -A INPUT -p vrrp -j ACCEPT
    service iptables save
```

#### 步骤 6：启动服务与验证

1. 先启动主节点、再启动备节点的 Keepalived 服务：

  ```bash
    systemctl start keepalived
```

2. **验证 VIP 绑定**：主节点执行`ip addr`，可看到 VIP 已绑定到对应网卡；备节点无 VIP。
3. **业务验证**：浏览器 / 客户端访问 VIP，可正常访问 Nginx 服务，与直接访问主节点 IP 一致。
4. **故障切换验证**：
    
    - 停止主节点 Nginx 服务，验证脚本是否能自动重启，重启失败是否触发 VIP 漂移；
    - 停止主节点 Keepalived 服务，查看备节点是否自动接管 VIP，业务访问是否正常；
    - 恢复主节点服务，验证 VIP 是否自动切回主节点（抢占模式）。

### 2.4 核心故障切换流程

#### 1. 正常运行状态

- Master 节点持有 VIP，对外提供服务，每 1 秒向备节点发送 VRRP 心跳包；
- Backup 节点监听心跳，处于备用状态，不持有 VIP，不处理业务流量。

#### 2. 故障触发场景

|故障场景|切换逻辑|
|---|---|
|Master 服务器宕机|Keepalived 进程停止，心跳包中断；Backup 节点 3 秒内未收到心跳，优先级最高的备节点升级为 Master，接管 VIP，发送 ARP 广播更新 MAC 地址，流量无缝切换|
|Master 节点 Nginx 服务崩溃|健康检查脚本检测到故障，重启失败后停止 Master 的 Keepalived，触发上述 VIP 漂移流程|
|Master 节点网络中断|心跳包无法发送到备节点，备节点超时后自动接管 VIP|

#### 3. 故障恢复流程

- 抢占模式（默认）：Master 节点恢复正常，Keepalived 启动，因优先级高于备节点，会重新夺回 Master 身份，VIP 漂移回主节点；
- 非抢占模式：Master 节点恢复后，不会主动夺回 VIP，直到当前持有 VIP 的备节点故障，避免二次切换影响业务，推荐生产环境使用。

---

### 2.5 核心避坑指南：脑裂问题

#### 1. 什么是脑裂

主备节点**同时持有 VIP**，导致流量被拆分到两台服务器，出现业务异常、数据不一致，是 Keepalived 最常见的严重故障。

#### 2. 核心原因

- 防火墙拦截了 VRRP 心跳包，主备节点无法通信，都认为对方故障，同时宣告自己为 Master；
- 主备节点网络链路故障，二层网络不通，心跳包无法送达；
- 主备节点`virtual_router_id`、认证密码不一致，心跳包无法识别；
- 服务器多网卡场景，心跳包网卡与 VIP 网卡不一致。

#### 3. 解决方案

1. 防火墙严格放行 VRRP 协议，确保主备节点之间 UDP 112 端口、VRRP 协议互通；
2. 配置双链路心跳，避免单链路故障导致心跳中断；
3. 编写脑裂检测脚本，备节点检测到主节点存活但自己持有 VIP 时，自动停止 Keepalived，触发告警；
4. 生产环境推荐使用**非抢占模式**，减少不必要的主备切换；
5. 配置 VIP 归属监控，出现双节点持有 VIP 时立即触发告警。

---

### 2.6 生产环境最佳实践

1. **非抢占模式配置**：主备节点均设置`state BACKUP`，添加`nopreempt`指令，仅通过优先级决定主节点，避免故障恢复后的二次切换：

  ```nginx
    vrrp_instance VI_1 {
        state BACKUP        # 主备节点都设为BACKUP
        nopreempt           # 开启非抢占模式
        priority 100        # 主节点优先级仍高于备节点
        # 其余配置不变
    }
```

2. **配置一致性保障**：使用`rsync+inotify`实现主节点 Nginx 配置变更后，自动同步到备节点，避免切换后配置不一致导致业务异常；
3. **权限最小化**：健康检查脚本禁止给 777 权限，严格控制脚本修改权限，避免恶意代码执行；
4. **监控告警全覆盖**：监控主备节点 Keepalived 状态、VIP 归属、Nginx 服务状态、主备切换事件，切换时立即触发企业微信 / 短信告警；
5. **定期故障演练**：每月执行主备切换演练，验证高可用方案的有效性，避免故障发生时方案失效；
6. **多备节点架构**：核心业务配置 1 主 2 备，避免备节点故障导致无节点可切换。

## 三、多节点集群部署与自动扩缩容

Keepalived 主从高可用解决了单点故障问题，但仍受限于单台 Nginx 的性能瓶颈，无法应对百万级 QPS 的高并发流量。**多节点 Nginx 集群 + 自动扩缩容**是大规模业务的终极高可用方案，彻底解决可用性与性能瓶颈。

### 3.1 集群架构核心设计

#### 1. 标准分层架构

```plaintext

用户流量 → 四层负载均衡（LVS/HAProxy/云厂商SLB） → Nginx集群（多台节点） → 后端业务服务
```

- **四层负载均衡**：作为集群的统一入口，基于 TCP 协议实现流量分发，将用户请求均匀转发到下层多台 Nginx 节点，同时实现故障节点自动剔除；
- **Nginx 集群**：多台对等的 Nginx 节点，配置完全一致，实现七层反向代理、负载均衡、静态资源处理、缓存加速，任意节点故障不影响整体服务；
- **核心优势**：
    
    1. 彻底消除单点故障：任意 Nginx 节点故障，四层负载会自动剔除，流量分发到健康节点；
    2. 无限水平扩展：Nginx 节点可按需横向扩容，突破单台性能瓶颈，支撑百万级 QPS；
    3. 无感知发布：可平滑上下线 Nginx 节点，实现灰度发布、蓝绿部署，业务零中断；
    4. 异地多活：可跨机房部署 Nginx 节点，实现多活容灾，单机房故障不影响全业务。
    

#### 2. 集群核心配置要点

1. **配置一致性**：所有 Nginx 节点必须使用完全一致的配置文件，推荐方案：
    
    - 配置中心：Nacos/Consul/Apollo，集中管理 Nginx 配置，变更后自动同步到所有节点；
    - 批量管理：Ansible/SaltStack，批量下发配置、重载 Nginx 服务；
    - 镜像化：将 Nginx 配置打包到容器镜像，所有节点使用相同镜像部署。
    
2. **健康检查**：
    
    - 四层负载对 Nginx 节点做**端口 + HTTP 健康检查**，比如每 3 秒访问 Nginx 的`/health`接口，连续 2 次失败则剔除节点，连续 1 次成功则恢复；
    - 配合 Nginx 自身的被动健康检查，实现后端服务的故障剔除，形成全链路健康检查体系。
    
3. **流量分发策略**：
    
    - 常规场景：加权轮询，根据 Nginx 节点的配置分配权重；
    - 会话保持场景：源 IP 哈希，同一个客户端 IP 的请求始终转发到同一台 Nginx 节点；
    - 低延迟场景：最小连接数 / 最短响应时间，将请求转发到负载最低的节点。
    
4. **可观测性**：
    
    - 统一日志收集：通过 ELK/PLG 栈收集所有 Nginx 节点的访问日志、错误日志，统一检索分析；
    - 全链路监控：Prometheus+Grafana 监控所有节点的 QPS、CPU、内存、连接数、错误率、响应时间，统一大盘展示；
    - 分布式追踪：配合 Jaeger/SkyWalking 实现请求全链路追踪，定位故障节点。
    

---

### 3.2 自动扩缩容核心方案

自动扩缩容的核心目标：**根据业务流量的波峰波谷，自动增减 Nginx 集群节点数量，流量高峰时扩容保障服务稳定，流量低谷时缩容降低资源成本**。

根据部署环境分为两大主流方案，覆盖传统架构与云原生架构。

#### 方案 1：云原生 Kubernetes 架构（生产首选）

基于 K8s 容器编排平台，使用 Nginx Ingress Controller 替代传统物理机 Nginx，是目前互联网业务的标准方案，原生支持自动扩缩容。

##### 核心实现原理

- Nginx Ingress 以 Pod 形式部署在 K8s 集群中，所有 Pod 配置完全一致，统一接收集群南北向流量；
- 通过 HPA（Horizontal Pod Autoscaler，水平 Pod 自动扩缩容）实现 Pod 副本数的自动调整；
- 配合 Cluster Autoscaler 实现集群节点的自动扩缩容，底层资源按需伸缩。

##### 核心扩缩容指标

|指标类型|常用指标|扩缩容阈值示例|
|---|---|---|
|资源指标|CPU 使用率、内存使用率|CPU 连续 5 分钟超过 70% 扩容，低于 30% 缩容|
|业务指标|QPS、每秒请求数、连接数|单 Pod QPS 超过 1000 扩容，低于 200 缩容|
|性能指标|平均响应时间、错误率|平均响应时间超过 500ms 扩容，错误率超过 1% 扩容|

##### 完整可落地 HPA 配置示例

基于 CPU 使用率和 QPS 自定义指标的自动扩缩容：

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: nginx-ingress-hpa
  namespace: ingress-nginx
spec:
  # 扩缩容目标：Nginx Ingress Controller
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: ingress-nginx-controller
  # 副本数范围：最小2个，最大20个
  minReplicas: 2
  maxReplicas: 20
  # 扩缩容规则
  metrics:
    # 规则1：CPU使用率
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70
    # 规则2：内存使用率
    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80
    # 规则3：自定义指标QPS，通过Prometheus Adapter接入
    - type: Pods
      pods:
        metric:
          name: nginx_ingress_qps
        target:
          type: AverageValue
          averageValue: 1000
  # 扩缩容行为控制，避免频繁抖动
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60  # 扩容稳定窗口60秒
      policies:
        - type: Percent
          value: 50
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300 # 缩容稳定窗口5分钟
      policies:
        - type: Percent
          value: 10
          periodSeconds: 120
```

##### 配套进阶能力

1. **VPA 垂直扩缩容**：自动调整 Nginx Pod 的 CPU、内存申请值，适配业务负载，避免资源浪费；
2. **金丝雀发布 / 蓝绿部署**：通过 Ingress 配置实现流量切分，平滑发布新版本，无业务中断；
3. **跨可用区部署**：Nginx Pod 分散部署在不同可用区，单可用区故障不影响服务可用性；
4. **自动配置同步**：通过 Ingress 资源自动更新 Nginx 配置，无需手动修改配置文件，零人工干预。

#### 方案 2：传统物理机 / 虚拟机架构自动扩缩容

针对未容器化的传统架构，基于监控系统 + 自动化工具实现自动扩缩容。

##### 核心实现流程

1. **监控采集**：Prometheus 采集所有 Nginx 节点的 CPU、内存、QPS、连接数等指标；
2. **告警触发**：Alertmanager 配置扩缩容告警规则，指标达到阈值时触发告警，调用扩缩容脚本；
3. **自动扩缩容执行**：
    
    - 扩容：通过 Ansible/terraform 批量创建虚拟机，初始化 Nginx 环境、同步配置，将新节点加入四层负载均衡的节点池；
    - 缩容：先将目标节点从负载池下线，等待连接排空，再销毁虚拟机，释放资源；
    
4. **通知告警**：扩缩容完成后，发送通知到企业微信 / 钉钉，记录操作日志。

##### 核心注意事项

1. **缩容保护**：缩容前必须先将节点从负载池下线，等待长连接排空，再销毁节点，避免业务请求中断；
2. **冷却时间**：设置扩容冷却时间 5 分钟，缩容冷却时间 10 分钟，避免流量抖动导致的频繁扩缩容；
3. **最小副本数**：设置最小节点数≥2，避免缩容到单节点出现单点故障；
4. **配置校验**：扩容节点时，必须校验 Nginx 配置一致性，配置校验通过后再加入负载池。

#### 方案 3：云厂商托管方案

直接使用云厂商的托管负载均衡 + 弹性伸缩服务，开箱即用，无需自己维护底层组件：

1. 阿里云 SLB / 腾讯云 CLB / 华为云 ELB 作为四层统一入口；
2. 搭配弹性伸缩 ESS，根据监控指标自动扩缩容 ECS 实例，自动挂载到负载均衡；
3. 云厂商托管的 Nginx Ingress 服务，原生支持自动扩缩容、高可用部署，无需运维底层节点。

### 3.3 集群部署最佳实践

1. **无状态化设计**：Nginx 节点必须完全无状态，不存储任何业务数据、会话信息，所有节点配置一致，可随时扩容、销毁；
2. **全链路健康检查**：从四层负载到 Nginx，再到后端服务，层层配置健康检查，故障节点秒级剔除；
3. **流量调度优化**：核心业务跨机房部署，通过 DNS 智能解析实现就近接入，单机房故障时自动切流到其他机房；
4. **故障演练常态化**：定期执行混沌工程，模拟 Nginx 节点故障、机房故障，验证集群的容灾能力；
5. **限流熔断兜底**：在 Nginx 层配置全局限流、单 IP 限流，避免突发流量打垮集群，同时配置降级兜底策略；
6. **配置灰度发布**：Nginx 配置变更时，先灰度发布到少量节点，验证无误后全量下发，避免全集群配置错误导致业务中断。

## 四、高可用架构选型指南

|业务规模|推荐架构|核心优势|
|---|---|---|
|中小业务、QPS＜1 万|Keepalived+Nginx 1 主 1 备|部署简单、运维成本低、完全满足可用性需求|
|中大型业务、QPS 1 万 - 10 万|Keepalived 主从 + 多节点 Nginx 集群|兼顾可用性与性能，可平滑扩容，运维成本可控|
|大型业务、QPS＞10 万|四层负载 + Nginx 集群 + 云原生自动扩缩容|无限水平扩展，高可用能力拉满，适配流量波动|
|金融 / 政务核心业务|异地多活架构，多机房多集群部署|机房级故障不影响业务，可用性达到 99.99% 以上|

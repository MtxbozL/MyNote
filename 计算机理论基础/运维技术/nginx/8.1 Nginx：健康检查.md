## 一、健康检查核心概述

Nginx 健康检查是**负载均衡高可用架构的核心能力**，核心作用是自动检测上游（upstream）后端服务器的可用性，自动剔除故障节点、恢复可用节点，避免请求转发到故障服务器，从转发层杜绝单点故障，保障业务连续性与用户无感知。

Nginx 健康检查分为两大类：

1. **被动健康检查**：开源版 Nginx 原生自带，无需额外模块，核心依赖`max_fails`和`fail_timeout`两个参数（笔记大纲核心内容），随业务请求同步检测；
2. **主动健康检查**：Nginx Plus 商业版原生支持，开源版需通过第三方模块实现，独立于业务流量主动检测，故障发现无感知，是生产高可用场景的核心方案。

## 二、原生被动健康检查（开源版默认支持）

### 2.1 核心原理

被动健康检查是 Nginx 在转发业务请求的过程中，根据请求的响应结果同步判断后端节点是否健康，无独立的健康检查进程，无额外的检测请求开销，开箱即用。

核心逻辑：在指定时间窗口内，某节点的请求失败次数达到阈值，Nginx 就会判定该节点为故障节点，触发熔断（暂时剔除出负载均衡池），暂停转发请求；熔断时间结束后，自动尝试恢复转发，重新检测节点状态。

### 2.2 核心参数详解（大纲核心内容）

两个参数必须成对配置在`upstream`块的`server`指令中，属于上游节点的核心配置项。

#### 1. max_fails（最大失败次数）

- **核心定义**：在`fail_timeout`指定的时间窗口内，允许请求后端节点失败的最大次数，超过该阈值，Nginx 立即判定节点为故障，触发熔断。
- **默认值**：1 次
- **关键细节**：
    
    - 「失败判定规则」由`proxy_next_upstream`（反向代理）、`fastcgi_next_upstream`（PHP-FPM）等指令定义，默认计入失败的场景包括：网络连接错误、连接超时、读取响应超时、后端返回 500/502/503/504 状态码；
    - 404/403 等业务状态码默认不计入失败，不可随意添加，避免业务正常场景导致节点被误判；
    - 核心作用：设置故障判定的容错阈值，避免单次偶发网络波动就误踢节点，导致负载不均。
    

#### 2. fail_timeout（失败超时时间）

- **核心定义**：一个参数承担两个核心作用，是健康检查的时间标尺。
    
    1. **故障判定的时间窗口**：连续的`fail_timeout`时间内，节点失败次数达到`max_fails`，才会触发熔断；
    2. **故障节点的熔断时长**：节点被判定为故障后，接下来的`fail_timeout`时间内，Nginx 不会转发任何业务请求给该节点，直到熔断时间结束。
    
- **默认值**：10 秒
- **关键细节**：
    
    - 熔断时间结束后，Nginx 会自动恢复转发少量请求给该节点，重新检测可用性：如果再次失败达到`max_fails`，继续触发新一轮熔断；如果请求成功，节点恢复为健康状态，重新加入负载均衡池。
    - 该参数同时平衡「故障判定准确性」和「故障恢复速度」，不可设置过长或过短。
    

### 2.3 完整可落地配置示例

```nginx
# 上游负载均衡池配置
upstream backend_api {
    # 核心：健康检查参数配置，30秒内失败3次，熔断30秒
    server 192.168.1.101:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:8080 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:8080 max_fails=3 fail_timeout=30s;
}

server {
    listen 80;
    server_name api.example.com;

    location / {
        proxy_pass http://backend_api;
        # 核心联动：定义哪些场景计入失败次数，触发健康检查计数
        proxy_next_upstream error timeout http_500 http_502 http_503 http_504;
        # 限制最大重试次数，避免重试风暴
        proxy_next_upstream_tries 2;
        # 超时配置，和健康检查联动，超时即计入失败
        proxy_connect_timeout 5s;  # 后端连接超时
        proxy_read_timeout 10s;    # 后端响应读取超时
        proxy_send_timeout 10s;    # 请求发送超时

        # 基础请求头传递
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    }
}
```

**配置逻辑说明**：30 秒的时间窗口内，单个节点的请求失败次数达到 3 次，立即触发 30 秒熔断；30 秒内不转发任何请求给该节点，时间结束后自动尝试恢复检测。

### 2.4 被动健康检查优缺点

|优点|缺点|
|:--|:--|
|开源版 Nginx 原生支持，无需编译额外模块，开箱即用|被动检测，必须有业务请求转发才能发现故障，故障发现有延迟|
|无额外独立检测请求，不占用后端服务器额外资源|故障检测期间，会有部分用户请求转发到故障节点，影响用户体验|
|配置极简，仅需 2 个核心参数，运维成本极低|检测维度单一，仅能通过业务请求响应判断，无法自定义检测规则|

## 三、主动健康检查（生产高可用核心方案）

### 3.1 核心原理

主动健康检查与业务流量完全解耦，Nginx 启动独立的健康检查进程，按照预设的频率、规则，主动向后端节点发送专门的健康检查请求，根据响应结果提前判断节点可用性。故障节点会在用户请求到达前就被剔除，完全不影响业务，是生产环境高可用架构的首选方案。

- 开源版 Nginx：需编译淘宝开源的`nginx_upstream_check_module`第三方模块实现；
- Nginx Plus 商业版：原生自带`health_check`指令，无需额外模块。

### 3.2 完整配置示例

```nginx
# 上游负载均衡池
upstream backend_api {
    server 192.168.1.101:8080;
    server 192.168.1.102:8080;
    server 192.168.1.103:8080;

    # 主动健康检查核心配置
    # 每3秒检测1次，连续3次失败判定为故障，连续2次成功判定为恢复，单次检测超时2秒
    check interval=3000 rise=2 fall=3 timeout=2000 type=http;
    # 健康检查HTTP请求内容，指定检测接口和Host头
    check_http_send "GET /health HTTP/1.1\r\nHost: api.example.com\r\nConnection: close\r\n\r\n";
    # 定义健康判定规则：2xx状态码即为健康
    check_http_expect_alive http_2xx;
}

server {
    listen 80;
    server_name api.example.com;

    # 业务接口转发
    location / {
        proxy_pass http://backend_api;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
    }

    # 健康检查状态监控页面（可选）
    location /nginx_status {
        check_status;
        access_log off;
        # 访问控制：仅允许内网IP访问，禁止公网暴露
        allow 192.168.1.0/24;
        deny all;
    }
}
```

### 3.3 核心参数详解

|参数|作用说明|
|:--|:--|
|`interval`|健康检查的间隔时间，单位毫秒；示例 3000 = 每 3 秒给每个后端节点发送 1 次检测请求|
|`fall`|连续失败多少次，判定节点为故障，剔除出负载池；对应被动检查的`max_fails`|
|`rise`|故障节点连续成功多少次，判定为恢复健康，重新加入负载池|
|`timeout`|单次健康检查请求的超时时间，单位毫秒；超时即判定为本次检查失败，必须小于`interval`|
|`type`|健康检查类型：<br><br>- `tcp`：仅检测端口是否连通，最简单，无法校验业务可用性<br><br>- `http`：发送 HTTP 请求校验状态码，可精准检测业务可用性，生产最常用<br><br>- `ssl_hello`：HTTPS 端口 SSL 握手检测|
|`check_http_send`|自定义 HTTP 健康检查的请求内容，必须携带 Host 头，否则后端可能返回 400 错误|
|`check_http_expect_alive`|定义健康的响应状态码，常用`http_2xx`，可搭配`http_3xx`使用|

### 3.4 主动健康检查优缺点

|优点|缺点|
|:--|:--|
|与业务流量解耦，提前发现并剔除故障节点，用户完全无感知|开源版需手动编译第三方模块，有一定运维成本|
|可自定义检测接口、频率、判定规则，精准校验业务可用性（如数据库 / 缓存连接状态）|独立检测请求会占用后端少量资源，可通过调整间隔控制影响|
|故障发现和恢复时间完全可控，延迟极低|需后端配套开发轻量健康检查接口`/health`|
|自带状态监控页面，可对接 Prometheus 等监控系统，实现可视化告警|配置相对复杂，需适配业务场景调整参数|

## 四、健康检查最佳实践与避坑指南

### 4.1 生产环境最佳实践

1. **被动检查基础配置规范**
    
    - 阈值配置：生产环境建议`max_fails=2~3`，`fail_timeout=10s~30s`，平衡误判率和故障恢复速度；
    - 失败规则：`proxy_next_upstream`仅添加`error timeout http_500 http_502 http_503 http_504`，禁止添加 404/403 等业务状态码；
    - 重试防护：必须配置`proxy_next_upstream_tries`限制最大重试次数，避免所有节点故障时引发重试风暴，导致服务雪崩。
    
2. **主动检查生产配置规范**
    
    - 检测接口：必须使用能真实反映业务可用性的接口，而非仅返回 200 的空接口，需覆盖数据库、缓存、核心依赖的健康状态；
    - 频率控制：核心业务检测间隔 1~2 秒，非核心业务 3~5 秒，禁止低于 1 秒，避免给后端造成压力；
    - 超时控制：`timeout`必须小于`interval`，建议设置为间隔的 1/2，避免检测请求堆积；
    - 访问控制：健康检查状态页和检测接口，必须限制仅内网 IP 访问，禁止公网暴露，避免信息泄露和恶意攻击。
    
3. **高可用配套规范**
    
    - 核心业务建议同时配置「被动 + 主动」双重健康检查，兜底保障；
    - 熔断时长最长不超过 60 秒，避免节点恢复后无法及时加入负载池；
    - 必须配置监控告警：节点故障、健康状态变化、负载池可用节点数低于阈值时，立即触发告警通知。
    

### 4.2 常见避坑指南

1. **坑 1：`max_fails=1`+ 超长`fail_timeout`**
    
    - 问题：单次偶发网络波动就会误踢节点，导致负载不均，极端情况所有节点被踢，服务不可用；
    - 解决：生产环境`max_fails`至少设置为 2，避免误判。
    
2. **坑 2：健康检查仅检测 TCP 端口连通性**
    
    - 问题：端口通但业务进程卡死、数据库连接断开，Nginx 仍判定节点健康，转发请求导致用户业务失败；
    - 解决：必须使用 HTTP 接口检测，校验业务真实可用性。
    
3. **坑 3：健康检查超时时间大于检测间隔**
    
    - 问题：上一次检测还未超时，下一次检测已发起，导致请求堆积，后端压力飙升；
    - 解决：`timeout`必须小于`interval`，严格控制单次检测的超时时间。
    
4. **坑 4：健康检查 Host 头缺失**
    
    - 问题：后端虚拟主机配置下，无 Host 头会导致检测请求返回 400/404，误判节点故障；
    - 解决：`check_http_send`必须携带与业务一致的 Host 头。
    
5. **坑 5：被动检查无重试次数限制**
    
    - 问题：所有节点故障时，单个请求会重试所有节点，请求量放大 N 倍，引发雪崩；
    - 解决：配置`proxy_next_upstream_tries 2~3`，限制最大重试次数。

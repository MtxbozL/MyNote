## 一、核心概述

Redis 事务是 Redis 原生提供的**批量命令原子性执行单元**，核心能力是将多个 Redis 命令打包，按入队顺序串行化执行，执行期间不会被其他客户端的命令打断，是 Redis 实现并发控制、批量操作原子性的核心方案之一。

### 核心特性

1. **顺序执行**：事务内的命令严格按入队顺序串行执行，执行期间不会插入其他客户端的命令
2. **隔离性**：事务执行全程独占 Redis 主线程，其他客户端的命令必须等待事务执行完成后才能执行，完全无并发干扰
3. **部分原子性**：仅保证「入队错误全不执行」，**执行阶段的命令错误不支持回滚**，正确命令会继续执行（与关系型数据库事务的核心差异）
4. **乐观锁支持**：通过`WATCH`命令实现 CAS（比较并交换）乐观锁，解决并发场景下的竞态问题
5. **轻量无额外开销**：基于 Redis 单线程模型实现，无锁、无事务日志的额外性能开销，执行效率极高

### 与关系型数据库 ACID 事务的核心差异

|特性|Redis 事务|MySQL/PostgreSQL 等关系型数据库事务|
|---|---|---|
|原子性|仅支持入队错误的全量放弃，执行时错误不回滚，不满足完整原子性|完整支持原子性，要么全执行成功，要么全部回滚|
|一致性|仅能保证数据符合基础约束，依赖业务层保证业务一致性|完整支持一致性，通过约束、事务日志保证数据从一个一致状态到另一个一致状态|
|隔离性|天然串行化隔离级别，无脏读、不可重复读、幻读问题|支持 4 种隔离级别，可按需配置|
|持久性|依赖 Redis 持久化配置，默认不保证持久性|完整支持持久性，事务提交后数据永久落盘|

## 二、底层实现原理

Redis 事务的底层实现极简且高效，核心依赖**命令队列**和**CAS 监控机制**两大核心模块，完整执行流程分为 3 个阶段。

### 2.1 事务执行的三大核心阶段

#### 阶段 1：事务开启（MULTI）

客户端执行`MULTI`命令后，Redis 会将该客户端的状态从「非事务状态」切换为「事务状态」，后续除事务专属命令外，所有常规命令都不会立即执行。

#### 阶段 2：命令入队

事务状态下，Redis 会对收到的命令做分类处理：

- **立即执行的命令**：事务专属命令（`MULTI/EXEC/DISCARD/WATCH/UNWATCH`），会立即执行并返回结果
- **入队等待的命令**：所有常规读写命令（`GET/SET/INCR/LPUSH`等），不会立即执行，而是进入客户端专属的**先进先出（FIFO）事务命令队列**，服务端返回`QUEUED`确认入队成功

#### 阶段 3：事务执行 / 放弃

- **执行事务（EXEC）**：客户端执行`EXEC`命令后，Redis 会按入队顺序串行执行队列中的所有命令，执行完成后：
    
    1. 按顺序返回所有命令的执行结果
    2. 清空事务命令队列
    3. 客户端切换回非事务状态
    4. 自动取消所有`WATCH`的 key 监控
    
- **放弃事务（DISCARD）**：客户端执行`DISCARD`命令后，Redis 会：
    
    1. 直接清空事务命令队列，放弃所有入队的命令
    2. 客户端切换回非事务状态
    3. 自动取消所有`WATCH`的 key 监控

### 2.2 WATCH 乐观锁的底层实现

`WATCH`是 Redis 事务实现并发控制的核心，本质是**CAS（Compare And Swap）比较并交换**机制，用于监控一个或多个 key 的变化，保证事务执行的前提是「被监控的 key 未被其他客户端修改」。

#### 核心执行逻辑

1. 客户端执行`WATCH key [key ...]`后，Redis 会将该客户端加入到数据库全局`watched_keys`字典的监听列表中：
    
    - 字典 key：被监控的 Redis key
    - 字典 value：监听该 key 的所有客户端列表
    
2. 当任何客户端执行修改 key 的命令（`SET/INCR/DEL`等）后，Redis 会遍历监听该 key 的所有客户端，给客户端状态打上`REDIS_DIRTY_CAS`（CAS 脏标记）
3. 客户端执行`EXEC`时，Redis 会先检查客户端是否有`REDIS_DIRTY_CAS`标记：
    
    - 有标记：说明被监控的 key 已被修改，直接放弃整个事务，返回`nil`
    - 无标记：正常执行事务队列中的所有命令

#### 关键特性

- 监控有效期：`WATCH`的监控仅在**当前事务周期内有效**，`EXEC/DISCARD`执行后，所有监控会自动取消，下次事务需要重新`WATCH`
- 无过期感知：被监控的 key 过期 / 内存淘汰，**不会触发脏标记**，事务不会失效，这是高频踩坑点
- 集群限制：Redis Cluster 中，`WATCH`的 key 必须和事务内的 key 在同一个哈希槽，否则会报错

## 三、核心命令大全

按功能分为 2 大类，标注核心作用、语法、示例与关键注意事项，适配 Redis 6.0 + 稳定版本。

### 3.1 事务基础命令

|命令|语法|核心作用|示例|关键注意事项|
|---|---|---|---|---|
|MULTI|`MULTI`|开启一个事务，将客户端切换为事务状态|`MULTI`|多次执行 MULTI 不会报错，但无额外效果；执行后除事务专属命令外，所有常规命令均进入队列|
|EXEC|`EXEC`|执行事务队列中的所有命令，按顺序返回执行结果|`EXEC`|若被 WATCH 的 key 被修改，或入队时有语法错误，事务直接放弃，返回 nil|
|DISCARD|`DISCARD`|放弃事务，清空命令队列，客户端切换回非事务状态|`DISCARD`|执行后所有 WATCH 监控自动取消；EXEC 执行后无法再执行 DISCARD|

### 3.2 乐观锁控制命令

|命令|语法|核心作用|示例|关键注意事项|
|---|---|---|---|---|
|WATCH|`WATCH key [key ...]`|监控一个或多个 key，若 EXEC 执行前 key 被其他客户端修改，事务直接放弃|`WATCH goods:stock:1001 user:balance:1001`|必须在 MULTI 之前执行，事务开启后执行 WATCH 会报错；仅监控 key 的修改，不监控过期 / 淘汰|
|UNWATCH|`UNWATCH`|手动取消当前客户端所有 WATCH 的 key 监控|`UNWATCH`|仅在 WATCH 之后、MULTI 之前执行有效；EXEC/DISCARD 会自动取消监控，无需手动执行 UNWATCH|

## 四、Redis 事务的 ACID 特性深度解析

这是面试最高频考点，必须明确 Redis 事务对 ACID 四大特性的支持程度与底层原因。

### 4.1 原子性（Atomicity）

**仅支持部分原子性，不支持完整原子性，无回滚能力**，分两种场景：

1. **入队阶段错误：满足原子性**
    
    若命令入队时出现语法错误、参数数量错误、命令名错误等问题，Redis 会记录错误，执行`EXEC`时直接放弃整个事务，所有命令都不会执行，保证「全不执行」的原子性。

```bash
    127.0.0.1:6379> MULTI
    OK
    127.0.0.1:6379> SET a 1
    QUEUED
    127.0.0.1:6379> SET a  # 语法错误，入队时直接报错
    (error) ERR wrong number of arguments for 'set' command
    127.0.0.1:6379> EXEC
    (error) EXECABORT Transaction discarded because of previous errors.
    # 整个事务被放弃，SET a 1 未执行
```
    
2. **执行阶段错误：不满足原子性，无回滚**
    
    若命令入队时无错误，但执行时出现运行时错误（如对 String 类型执行 List 命令、key 类型不匹配），Redis 会**跳过错误命令，继续执行后续所有正确的命令**，不会回滚已执行的命令，无法保证原子性。

```
    127.0.0.1:6379> MULTI
    OK
    127.0.0.1:6379> SET a 1
    QUEUED
    127.0.0.1:6379> LPOP a  # 对String执行List命令，入队无报错，执行时报错
    QUEUED
    127.0.0.1:6379> SET b 2
    QUEUED
    127.0.0.1:6379> EXEC
    1) OK
    2) (error) WRONGTYPE Operation against a key holding the wrong kind of value
    3) OK
    # 错误命令跳过，SET a 1 和 SET b 2 均执行成功，无回滚
```

#### 为什么 Redis 不支持事务回滚？

Redis 官方给出了明确的设计原因：

1. **极致性能优先**：回滚机制需要额外的事务日志、锁与回滚逻辑，会大幅增加 Redis 的复杂度与性能开销，违背 Redis「高性能、低延迟」的核心设计目标
2. **错误场景可预判**：执行时的运行时错误，几乎都是开发阶段的语法、类型使用错误，生产环境极少出现，无需通过回滚机制兜底
3. **单线程模型限制**：Redis 单线程模型无法实现复杂的回滚逻辑，且回滚无法解决并发竞态问题

### 4.2 一致性（Consistency）

**Redis 事务满足数据一致性**，这里的一致性指「数据符合 Redis 的内置约束与规则」：

1. 入队错误：事务全量放弃，数据保持原有状态，符合一致性
2. 执行错误：错误命令被跳过，正确命令正常执行，数据符合 Redis 的类型与语法约束，无脏数据
3. 进程崩溃：若开启了 RDB/AOF 持久化，Redis 重启后会通过持久化文件恢复数据，保证数据一致性；无持久化时，重启后数据清空，也符合空库的一致性约束

> 注意：业务层面的一致性（如转账时 A 扣钱、B 加钱必须同时成功），需要业务层通过前置校验、幂等设计保证，Redis 事务本身无法保证业务一致性。

### 4.3 隔离性（Isolation）

**Redis 事务完全满足隔离性，天然达到串行化隔离级别**。

Redis 是单线程模型，事务执行期间会独占主线程，所有命令串行执行，其他客户端的命令必须等待整个事务执行完成后才能执行，不会出现脏读、不可重复读、幻读等并发问题，是最高级别的隔离性。

### 4.4 持久性（Durability）

**Redis 事务的持久性完全依赖持久化配置，默认不满足持久性**。

1. **无持久化**：Redis 重启后所有数据丢失，完全不满足持久性
2. **RDB 持久化**：快照是周期性执行的，事务执行后到下一次快照之间发生崩溃，数据会丢失，不满足持久性
3. **AOF 持久化**：
    
    - `appendfsync=always`：每条命令执行后立即同步刷盘，事务执行后数据永久落盘，满足持久性
    - `appendfsync=everysec`：每秒刷盘一次，最多丢失 1 秒数据，不满足持久性
    - `appendfsync=no`：完全依赖操作系统刷盘，数据丢失风险极高，不满足持久性

## 五、典型应用场景

Redis 事务仅适用于**简单批量操作、低并发乐观锁控制**场景，高并发复杂场景优先使用 Lua 脚本。

### 5.1 批量命令串行化执行

核心价值是保证多个命令按顺序执行，期间不会被其他客户端的命令打断，避免并发修改导致的数据异常。

- 典型场景：批量配置更新、多 key 同时设置、数据批量初始化
- 示例：

```bash
    # 批量更新用户的多个配置项，保证全部执行，无并发干扰
    MULTI
    SET user:1001:status active
    SET user:1001:vip_level 1
    EXPIRE user:1001:token 86400
    EXEC
```

### 5.2 基于乐观锁的并发库存扣减（秒杀场景）

通过`WATCH`监控库存 key，保证只有库存未被修改时，扣减操作才会执行，避免超卖问题。

- 示例：

```bash
    # 1. 监控库存key，防止并发修改
    WATCH goods:stock:1001
    # 2. 查询当前库存
    GET goods:stock:1001
    # 3. 业务层判断库存>0，开启事务
    MULTI
    # 4. 库存扣减1
    DECR goods:stock:1001
    # 5. 记录用户下单记录
    SADD order:goods:1001 user:1001
    # 6. 执行事务：若库存被其他用户修改，返回nil，扣减失败
    EXEC
```

### 5.3 转账场景的并发控制

保证 A 账户扣钱、B 账户加钱的操作串行执行，配合`WATCH`防止两个账户被并发修改，避免资金异常。

- 示例：

```bash
    # 监控两个账户的余额，防止并发修改
    WATCH user:balance:1001 user:balance:1002
    # 查询余额，判断是否足够转账
    GET user:balance:1001
    # 开启事务
    MULTI
    DECRBY user:balance:1001 100
    INCRBY user:balance:1002 100
    # 执行事务，若余额被修改，事务放弃
    EXEC
```

## 六、常见坑与最佳实践

### 6.1 高频踩坑点

1. **执行时错误不回滚，导致数据不一致**
    
    线上绝对禁止出现执行时的运行时错误，必须在事务开启前完成所有类型、参数、语法的前置校验，避免出现类型不匹配、命令使用错误等问题。
    
2. **WATCH 监控失效场景**
    
    - 被监控的 key 过期 / 内存淘汰，不会触发脏标记，事务仍会执行，导致业务逻辑异常
    - `EXEC/DISCARD`执行后，WATCH 监控自动失效，下次事务必须重新执行 WATCH
    - 集群模式下，WATCH 的 key 与事务 key 不在同一个哈希槽，会直接报错
    
3. **大事务阻塞 Redis 主线程**
    
    事务内的命令过多、执行耗时过长，会独占 Redis 单线程，导致整个实例卡顿、请求超时。线上事务内的命令数量建议控制在 100 个以内，绝对禁止在事务内执行`KEYS *`、`SMEMBERS`大集合、`HGETALL`大哈希等耗时命令。
    
4. **Redis Cluster 集群事务限制**
    
    集群模式下，事务内的**所有 key 必须落在同一个哈希槽**，否则会报错无法执行。解决方案是使用**哈希标签（Hash Tag）**，给所有 key 加上相同的`{xxx}`前缀，保证哈希槽一致。

```bash
    # 正确示例：通过{order}哈希标签保证两个key落在同一个哈希槽
    MULTI
    SET {order}:1001:status paid
    INCR {order}:total:paid
    EXEC
```

5. **高并发下 WATCH 导致的事务频繁失败**
    
    秒杀等高并发场景下，大量客户端同时监控同一个库存 key，只要有一个客户端修改成功，其他所有客户端的事务都会失败，导致大量请求无效，吞吐量极低。这种场景**禁止使用 WATCH + 事务**，优先使用 Lua 脚本实现。

### 6.2 线上最佳实践

1. **优先使用 Lua 脚本替代事务**
    
    Lua 脚本是 Redis 2.6 + 提供的更强大的原子性方案，整个脚本作为一个整体串行执行，中间不会被其他命令打断，支持条件判断、循环、分支逻辑，可完全替代事务，且功能更强大、性能更优。

| 对比维度 | Redis 事务             | Lua 脚本                       |
| ---- | -------------------- | ---------------------------- |
| 原子性  | 仅部分原子性，执行时错误不回滚      | 完整原子性，脚本要么全执行，要么全不执行         |
| 灵活性  | 仅支持批量命令执行，无逻辑判断      | 支持条件判断、循环、分支等复杂逻辑            |
| 并发控制 | 依赖 WATCH 乐观锁，高并发下易失败 | 脚本内可直接做条件判断，无需 WATCH，高并发下更稳定 |
| 性能   | 多次网络往返，开销较大          | 一次网络请求完成所有操作，网络开销更小          |
| 集群支持 | 仅支持单哈希槽内的 key        | 仅支持单哈希槽内的 key，与事务一致          |

选型建议：
- 简单批量命令执行，无复杂逻辑，可使用事务
- 高并发场景、需要条件判断、并发控制的场景，**优先使用 Lua 脚本**
- 秒杀、库存扣减等核心场景，必须使用 Lua 脚本，避免 WATCH 导致的事务频繁失败

2. **严格控制事务大小**
    
    事务内的命令数量建议控制在 100 个以内，单条命令执行耗时控制在 1ms 以内，绝对禁止大事务阻塞主线程。
    
3. **提前完成所有业务校验**
    
    所有业务判断、类型校验、参数校验必须在事务开启前完成，事务内仅保留最终的读写命令，避免执行时出现错误。
    
4. **禁止在事务内执行耗时命令**
    
    绝对禁止在事务内执行全量遍历、大 key 操作、阻塞命令，避免主线程长时间阻塞。
    
5. **集群环境下必须使用哈希标签**
    
    集群模式下使用事务，必须给所有 key 加上相同的哈希标签，保证落在同一个哈希槽，否则事务无法执行。

## 一、核心概述

Redis Stream 是**Redis 5.0 版本引入的核心数据类型**，是 Redis 官方为解决「消息队列」场景专门设计的持久化、多消费者、高可靠数据结构，彻底弥补了之前 Pub/Sub（消息丢失）、List（无法多消费者并行消费同一条消息）做消息队列的核心缺陷。

### 核心定位

Redis Stream 是**轻量级专业消息队列**，兼具 Redis 的高性能、低延迟特性，同时具备专业 MQ 的核心能力：

- 消息持久化：基于 Redis 内存 + RDB/AOF 持久化，消息不丢失
- 多消费者组：支持多个消费者组独立消费同一条流，互不干扰
- 消息 ACK 确认：消费者处理完成后需手动确认，未确认消息可重试
- 消息回溯：支持按 ID / 时间回溯历史消息，重新消费
- 消息堆积：支持海量消息堆积，内存可控
- 原子性保证：所有操作均为原子性，基于 Redis 单线程模型，并发无竞态

### 核心数据结构概念

| 概念                        | 作用                                    | 类比专业 MQ                 |
| ------------------------- | ------------------------------------- | ----------------------- |
| Stream（流）                 | 消息的存储载体，一个 Stream 对应一个消息队列            | Topic（主题）               |
| Message（消息）               | Stream 中的单条数据，包含唯一 ID 和键值对内容          | Message（消息）             |
| Message ID（消息 ID）         | 消息的唯一标识，格式为`timestamp-sequence`，按时间递增 | Offset（偏移量）             |
| Consumer Group（消费者组）      | 独立的消费单元，一个 Stream 可对应多个消费者组           | Consumer Group（消费者组）    |
| Consumer（消费者）             | 消费者组内的具体消费节点，一个组可包含多个消费者，负载均衡消费       | Consumer（消费者）           |
| PEL（Pending Entries List） | 消费者组内的待确认消息列表，存储已读取但未 ACK 的消息         | Unacked Offset（未确认偏移量）  |
| Last Delivered ID         | 消费者组最后一次投递的消息 ID，用于记录消费进度             | Committed Offset（提交偏移量） |

## 二、底层实现原理

Redis Stream 的底层设计兼顾了高性能、低内存占用与功能完整性，核心依赖**radix tree（基数树）+ listpack（紧凑列表）** 双结构，消费者组则依赖独立的 PEL 结构。

### 2.1 Stream 消息存储的核心结构

Stream 的消息存储分为两层：

1. **radix tree（基数树）**：索引层，key 为消息 ID 的时间戳部分，value 为指向 listpack 的指针，用于快速按时间范围查找消息
2. **listpack（紧凑列表）**：数据层，存储实际的消息内容，一个 listpack 对应一个时间窗口内的多条消息，极致优化内存占用

#### 核心结构细节

```plaintext
Stream整体结构
├── radix tree（基数树索引）
│   ├── key: 1741440000（时间戳）
│   │   └── value: 指向listpack1的指针
│   ├── key: 1741440001（时间戳）
│   │   └── value: 指向listpack2的指针
│   └── ...
└── listpack（紧凑列表数据）
    ├── listpack1: [消息1内容, 消息2内容, ...]
    ├── listpack2: [消息3内容, 消息4内容, ...]
    └── ...
```

#### 消息 ID 的格式与生成规则

消息 ID 是 Stream 的核心标识，格式为`timestamp-sequence`，例如`1741440000000-0`：

- `timestamp`：毫秒级时间戳，生成消息时的 Redis 服务器时间
- `sequence`：同一毫秒内的序列号，从 0 开始递增
- 生成规则：
    
    1. 默认由 Redis 自动生成，保证全局递增
    2. 支持手动指定 ID，但必须大于 Stream 中已存在的最大 ID
    3. 特殊 ID：`0-0`代表 Stream 的起始位置，`$`代表 Stream 的最新位置（仅用于消费）
    

#### 核心优势

1. **基数树索引**：O (logN) 时间复杂度按时间范围查找消息，适合消息回溯
2. **紧凑列表存储**：连续内存存储，无指针开销，内存占用比纯哈希表低 70% 以上
3. **批量存储**：同一时间窗口的多条消息存入同一个 listpack，减少索引开销，提升读写性能

### 2.2 消费者组（Consumer Group）的核心结构

消费者组是 Stream 实现多消费者、ACK、消息重试的核心，每个消费者组独立维护以下元数据：

|元数据|类型|核心作用|
|---|---|---|
|`last_delivered_id`|消息 ID|消费者组最后一次投递的消息 ID，用于记录消费进度，新消费者从该 ID 之后开始消费|
|`pel`|哈希表|Pending Entries List，待确认消息列表：key 为消息 ID，value 为消费者 ID + 投递次数，用于消息重试|
|`consumers`|哈希表|组内的消费者列表：key 为消费者 ID，value 为该消费者的 PEL（仅该消费者读取但未 ACK 的消息）|

#### PEL 的核心作用

PEL 是 Stream 消息可靠性的核心保障：

1. 消费者通过`XREADGROUP`读取消息后，消息会自动加入该消费者组的 PEL 和该消费者的 PEL
2. 消费者处理完成后，通过`XACK`确认消息，消息会从 PEL 中移除
3. 若消费者宕机，未 ACK 的消息仍保留在 PEL 中，可通过`XPENDING`查询，通过`XCLAIM`转移给其他消费者重试

## 四、典型应用场景

Stream 的核心价值是**高可靠、多消费者的专业消息队列**，以下为经典落地场景：

### 4.1 核心业务异步队列（最核心场景）

Stream 是 Redis 中唯一适合核心业务异步队列的方案，保证消息不丢失、不重复、可重试，适配订单处理、支付回调、数据同步等核心场景。

- 实现示例：
```bash
    # 1. 创建Stream和消费者组，从第一条消息开始消费
    XGROUP CREATE order:queue order_group 0 MKSTREAM
    # 2. 订单服务生产消息
    XADD order:queue * order_id 1001 user_id 1001 amount 99 pay_time 2026-03-08
    # 3. 消费者1负载均衡消费消息
    XREADGROUP GROUP order_group consumer1 COUNT 10 BLOCK 0 STREAMS order:queue >
    # 4. 处理完成后手动ACK，确认消息
    XACK order:queue order_group 1741440000000-0
    # 5. 定时任务扫描PEL，将空闲超过60秒的消息转移给其他消费者重试
    XPENDING order:queue order_group - + 10
    XCLAIM order:queue order_group consumer2 60000 1741440000000-0
```

### 4.2 多消费者组并行消费

一个 Stream 可对应多个消费者组，每个组独立消费，互不干扰，适合同一条消息需要被多个业务并行处理的场景。

- 典型场景：订单支付成功后，订单组处理订单状态、库存组扣减库存、积分组增加积分、物流组创建物流单，四个组并行消费同一条支付成功消息。
- 实现示例：
```bash
    # 1. 创建多个独立的消费者组
    XGROUP CREATE order:pay:success order_group 0 MKSTREAM
    XGROUP CREATE order:pay:success stock_group 0 MKSTREAM
    XGROUP CREATE order:pay:success point_group 0 MKSTREAM
    # 2. 支付服务生产一条消息
    XADD order:pay:success * order_id 1001 amount 99
    # 3. 四个组的消费者分别独立消费，互不干扰，同一条消息被四个组各消费一次
```

### 4.3 消息回溯与重放

Stream 支持按 ID / 时间重置消费者组的起始 ID，重新消费历史消息，适合数据修复、业务回滚、新消费者组初始化等场景。

- 实现示例：
```bash
    # 1. 查询需要回溯的起始消息ID
    XRANGE order:queue 1741440000000-0 + COUNT 1
    # 2. 重置消费者组的起始ID，从指定ID开始重新消费
    XGROUP SETID order:queue order_group 1741440000000-0
    # 3. 消费者重新读取消息，完成回溯
    XREADGROUP GROUP order_group consumer1 COUNT 10 BLOCK 0 STREAMS order:queue >
```

### 4.4 海量日志收集与聚合

Stream 支持海量消息堆积，内存可控，适合日志收集、监控数据聚合、事件溯源等场景，可按时间范围快速查询历史日志。

- 实现示例：
    
    bash
    
    运行
    
```bash
    # 1. 各服务节点向Stream写入日志
    XADD log:service * service_name user-service level ERROR msg "数据库连接失败" time 2026-03-08
    # 2. 日志聚合服务消费日志，写入Elasticsearch
    XREADGROUP GROUP log_group consumer1 COUNT 100 BLOCK 0 STREAMS log:service >
    # 3. 按时间范围查询历史日志，排查问题
    XRANGE log:service 1741440000000-0 1741443600000-0 COUNT 100
    # 4. 定期裁剪Stream，保留最近7天的日志
    XTRIM log:service MINID ~ 1740835200000-0
```

### 4.5 延时队列（配合 Sorted Set 实现）

Stream 原生不支持延时队列，但可配合 Sorted Set 实现：

1. 将延时任务存入 Sorted Set，score 为任务执行的时间戳
2. 定时任务扫描 Sorted Set，将到期任务转移到 Stream
3. 消费者从 Stream 中消费任务，保证可靠性

## 五、常见坑与最佳实践

### 5.1 核心坑点与避坑方案

1. **消息堆积导致内存膨胀，Redis OOM**
    
    问题本质：Stream 的消息默认永久保留，若消费速度跟不上生产速度，或忘记裁剪，会导致消息无限堆积，内存占用持续增长，最终触发 Redis OOM。
    
    避坑方案：
    
    - 生产消息时，直接在`XADD`中指定`MAXLEN`或`MINID`参数，自动裁剪，避免手动管理
    - 优先使用`~`近似裁剪，避免一次性删除太多消息阻塞主线程
    - 配合`LIMIT`参数（Redis 7.0+），限制单次裁剪的最大数量
    - 监控 Stream 的长度，设置阈值告警，提前处理堆积
    - 示例：`XADD order:queue MAXLEN ~ 100000 * order_id 1001`，自动保留最新的约 10 万条消息
    
2. **消费者组的 PEL 积压，内存泄漏**
    
    问题本质：消费者读取消息后，未调用`XACK`确认，消息会一直保留在 PEL 中，即使 Stream 已裁剪，PEL 中的消息也不会被删除，导致内存泄漏。
    
    避坑方案：
    
    - 严格规范消费逻辑：**消息处理成功后必须调用 XACK**，处理失败则不 ACK，等待重试
    - 定时任务扫描 PEL：通过`XPENDING`定期查询待确认消息，通过`XCLAIM`转移给其他消费者，或手动确认 / 删除
    - 监控 PEL 的大小，设置阈值告警，提前处理积压
    - 避免使用`NOACK`参数，除非业务允许消息丢失
    
3. **XTRIM 一次性删除太多消息，阻塞 Redis 主线程**
    
    问题本质：使用`XTRIM key MAXLEN = 10000`精确裁剪时，若需要删除的消息数量过多，会一次性遍历并删除大量 listpack，阻塞 Redis 单线程，导致实例卡顿。
    
    避坑方案：
    
    - 线上**绝对禁止使用`=`精确裁剪**，必须使用`~`近似裁剪，Redis 会保证裁剪后的数量不小于阈值，但不会一次性删除太多，避免阻塞
    - Redis 7.0 + 配合`LIMIT`参数，限制单次裁剪的最大数量，例如`XTRIM order:queue MAXLEN ~ 100000 LIMIT 1000`，单次最多删除 1000 条
    - 优先在`XADD`时自动裁剪，而非手动调用`XTRIM`
    
4. **消费者宕机导致消息重复消费**
    
    问题本质：消费者读取消息后，处理过程中宕机，消息未 ACK，会保留在 PEL 中，后续被其他消费者重试，导致重复消费。
    
    避坑方案：
    
    - 业务层必须实现**幂等性处理**，通过消息 ID 或业务唯一标识判断是否已处理，避免重复执行业务逻辑
    - 合理设置`XCLAIM`的`min-idle-time`，避免消息被过早重试，给原消费者足够的处理时间
    - 监控消息的投递次数，投递次数过多的消息移入死信队列，人工处理
    
5. **集群模式下的 Stream 分片问题**
    
    问题本质：Redis Cluster 中，一个 Stream 只会落在一个哈希槽上，无法分片，若 Stream 的消息量过大，会导致集群数据倾斜，单分片负载过高。
    
    避坑方案：
    
    - 超大 Stream 按业务维度拆分，比如按时间（`order:queue:202603`、`order:queue:202604`）、按地区、按用户分群拆分，分散到多个哈希槽
    - 使用哈希标签（`{order}`）保证相关 Stream 落在同一个分片，或分散到不同分片
    - 监控集群各分片的内存与负载，提前发现数据倾斜
    

### 5.2 线上最佳实践

1. **消息内容控制**：单条消息的键值对数量建议控制在 10 个以内，单条消息大小控制在 1KB 以内，最大不超过 10KB。超大内容不要直接放在消息体中，仅存储事件 ID，内容存储在 Redis Hash 或数据库中，消费者收到消息后自行拉取。
2. **消费者组命名规范**：消费者组名建议包含业务含义，如`order_process_group`、`stock_deduct_group`，方便管理与监控。
3. **消费者 ID 规范**：消费者 ID 建议包含主机名 / IP + 进程 ID，如`consumer_192.168.1.10_12345`，方便排查问题时定位到具体的消费者节点。
4. **阻塞超时设置**：`XREADGROUP`的`BLOCK`参数建议设置为 30~60 秒，避免永久阻塞，超时后重新发起读取即可，防止网络波动导致的假死。
5. **批量消费与批量 ACK**：优先使用`COUNT`参数批量读取消息，处理完成后批量 ACK，减少网络 RTT，提升消费性能，但批量大小建议控制在 100 以内，避免处理时间过长导致消息被重试。
6. **死信队列处理**：设置消息的最大投递次数，超过次数的消息通过`XADD`移入死信队列（`order:queue:dlq`），停止自动重试，人工介入排查问题，避免无限重试占用资源。
7. **监控告警**：重点监控 Stream 的长度、消息生产 / 消费速度、消费者组的 PEL 大小、消息投递次数、消费者在线状态，设置阈值告警，提前发现风险。

## 六、核心选型对比

### 6.1 Stream vs Pub/Sub vs List 消息队列

表格

| 对比维度   | Redis Stream             | Redis Pub/Sub    | Redis List             |
| ------ | ------------------------ | ---------------- | ---------------------- |
| 消息持久化  | 支持，基于 Redis 内存 + RDB/AOF | 不支持，消息发布即丢弃      | 支持，消息存储在 List 中        |
| 多消费者   | 支持多消费者组，组内负载均衡，组间并行消费    | 支持多消费者，一对多广播     | 支持多消费者，但一条消息仅能被一个消费者消费 |
| 消息 ACK | 支持，手动确认，未确认可重试           | 不支持              | 不支持，消费即删除              |
| 消息回溯   | 支持，按 ID / 时间重置消费进度       | 不支持              | 不支持                    |
| 消息堆积   | 支持海量堆积，内存可控              | 不支持堆积            | 支持堆积，但 List 过大性能下降     |
| 消息可靠性  | 极高，不丢不重可重试               | 极低，极易丢失          | 中等，不丢但可能重复             |
| 核心适用场景 | 核心业务异步队列、高可靠场景           | 实时广播、允许消息丢失的简单场景 | 简单任务队列、低可靠场景           |

### 6.2 Stream vs 专业消息队列（Kafka/RabbitMQ）

表格

| 对比维度   | Redis Stream                                 | 专业消息队列（Kafka/RabbitMQ）           |
| ------ | -------------------------------------------- | -------------------------------- |
| 消息可靠性  | 高，支持持久化、ACK、重试                               | 极高，支持多副本、事务消息、死信队列               |
| 消息堆积能力 | 支持百万级堆积，内存存储，成本较高                            | 支持亿级堆积，磁盘存储，成本极低                 |
| 集群能力   | Redis Cluster 中 Stream 无法分片，单 Stream 仅落在一个分片 | 原生支持分布式分片、多副本、高可用、水平扩展           |
| 功能丰富度  | 支持基础的消息队列功能，无高级特性                            | 支持延迟消息、事务消息、顺序消息、消息轨迹、权限控制、监控告警等 |
| 性能与延迟  | 极高，延迟在毫秒级，吞吐可达 10 万 + QPS                    | 高，延迟在毫秒级到秒级，吞吐可达百万级 QPS          |
| 部署成本   | 零成本，Redis 原生支持                               | 需独立部署、运维，有一定的学习与运维成本             |
| 核心适用场景 | 中小规模核心业务、低延迟、高可靠场景，已有 Redis 基础设施             | 大规模核心业务、海量消息堆积、复杂功能需求场景          |

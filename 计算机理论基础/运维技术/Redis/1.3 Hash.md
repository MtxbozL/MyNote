## 一、核心概述

Hash 是 Redis 专门为**结构化对象存储**设计的核心数据类型，本质是「键中键」的映射结构：一个 Redis 顶级 key 对应一个哈希表，哈希表内包含多组`field（字段）-value（值）`键值对，field 和 value 均为 String 类型。

- 核心特性：**field 唯一、操作原子性、细粒度读写、内存占用极低**，完美适配对象类数据的存储与修改
- 原子性保证：所有 field 级别的操作均为原子性，基于 Redis 单线程模型，并发场景无竞态问题
- 容量限制：单个 Hash 最多支持`2^32 - 1`个 field-value 对（约 42 亿），单个 field/value 最大容量 512MB
- 是 Redis 中替代「String+JSON 序列化」存储对象的最优方案，支持单字段独立读写，无全量序列化开销

## 二、底层实现原理

Redis Hash 会根据 field 数量、单个 field/value 的大小，自动选择最优编码格式，平衡内存占用与读写性能，编码转换**不可逆**（仅能从低内存开销编码向高内存开销编码转换）。

Redis 7.0 版本前，Hash 底层支持`ziplist（压缩列表）`和`hashtable（哈希表/字典）`两种编码；Redis 7.0 + 用`listpack（紧凑列表）`完全替代 ziplist，解决了 ziplist 连锁更新的缺陷，核心逻辑与配置规则保持兼容。

### 2.1 紧凑型编码：ziplist/listpack（默认优先使用）

ziplist/listpack 是一段连续的内存块，无额外指针开销，是小体量 Hash 的默认编码，极致优化内存占用。

#### 触发条件（同时满足）

表格

|版本|触发规则|默认阈值|
|---|---|---|
|Redis 6.x 及之前（ziplist）|1. field-value 总数量 ≤ `hash-max-ziplist-entries`<br><br>2. 每个 field/value 的字节长度 ≤ `hash-max-ziplist-value`|entries=512，value=64 字节|
|Redis 7.0+（listpack）|1. field-value 总数量 ≤ `hash-max-listpack-entries`<br><br>2. 每个 field/value 的字节长度 ≤ `hash-max-listpack-value`|entries=512，value=64 字节|

#### 存储结构

ziplist/listpack 中，Hash 的 field 和 value**成对连续存储**，按插入顺序排列。例如执行`HSET user:1001 name zhangsan age 18`，存储结构为：`[name, zhangsan, age, 18]`，无任何额外元数据开销。

#### 核心优劣势

- 优势：连续内存存储，内存占用比 hashtable 低 80% 以上，缓存命中率高，无内存碎片，适合小体量、高频访问的结构化数据
- 缺陷：查找、插入、删除需遍历内存，时间复杂度 O (n)，field 过多或 value 过大时性能急剧下降；ziplist 存在连锁更新风险（listpack 已彻底解决）

### 2.2 高性能编码：hashtable（哈希表 / 字典）

当 Hash 不满足 ziplist/listpack 的触发条件时，会自动不可逆地转为 hashtable 编码，底层是 Redis 核心的字典结构，保证海量 field 下的稳定 O (1) 操作性能。

#### 核心结构

hashtable 本质是一个支持渐进式 rehash 的哈希字典，核心结构简化如下：

c

运行

```
// 哈希字典核心结构
struct dict {
    dictType *type;
    dictht ht[2];       // 双哈希表，ht[0]为主表，ht[1]仅用于rehash
    long rehashidx;     // rehash进度标记，-1代表未进行rehash
    unsigned long iterators; // 迭代器数量
};

// 单哈希表结构
struct dictht {
    dictEntry **table;  // 哈希桶数组
    unsigned long size; // 数组总长度（2的整数次幂）
    unsigned long sizemask; // 哈希索引掩码，固定为size-1
    unsigned long used; // 已存储的field-value总数
};

// 哈希桶节点（存储单个field-value对）
struct dictEntry {
    void *key;          // 哈希表的field
    void *val;          // 哈希表的value
    struct dictEntry *next; // 哈希冲突链表指针
};
```

#### 核心特性

1. **O (1) 级别的操作性能**：通过哈希函数计算 field 的桶索引，平均时间复杂度 O (1)，最坏 O (n)（哈希冲突严重时），性能不随 field 数量增加而衰减
2. **链地址法解决哈希冲突**：哈希值相同的 field 会存入同一个桶的单向链表中，避免哈希冲突导致的数据丢失
3. **渐进式 rehash 机制**：解决哈希表扩容 / 缩容时的主线程阻塞问题，是 Redis 哈希表的核心设计亮点
    
    - 触发时机：负载因子（`used/size`）≥1 时触发扩容，负载因子≤0.1 时触发缩容，扩容后新大小为原大小的 2 倍
    - 执行过程：
        
        1. 为`ht[1]`分配新的内存空间，rehashidx 置为 0，标记 rehash 开始
        2. 每次对该哈希表执行增删改查操作时，顺带将`ht[0]`中 rehashidx 位置的桶全量迁移到`ht[1]`，rehashidx+1
        3. 所有元素迁移完成后，释放`ht[0]`，将`ht[1]`赋值给`ht[0]`，`ht[1]`置空，rehashidx 重置为 - 1
        
    - 核心优势：将全量 rehash 的耗时分摊到每一次操作中，避免一次性迁移海量数据导致的主线程阻塞
    

#### 核心优劣势

- 优势：支持海量 field，增删改查性能稳定，无遍历开销，适合大体量、频繁修改的哈希表
- 缺陷：指针与哈希桶的额外内存开销大，内存占用远高于 ziplist/listpack，易产生内存碎片

## 三、核心命令大全

按功能分为 6 大类，标注时间复杂度、高频标记、示例与核心注意事项，适配 Redis 6.0 + 稳定版本。

### 3.1 基础读写命令（高频核心）

表格

|命令|语法|时间复杂度|核心作用|示例|注意事项|
|---|---|---|---|---|---|
|HSET|`HSET key field value [field value ...]`|O (n)，n 为本次设置的 field 数量|为哈希表设置一个 / 多个 field-value 对，key 不存在自动创建，field 已存在则覆盖值|`HSET user:1001 name zhangsan age 25 phone 13800138000`|Redis 4.0 + 支持多 field 批量写入，**完全替代废弃的 HMSET 命令**；返回本次新创建的 field 数量|
|HGET|`HGET key field`|O(1)|获取哈希表中指定 field 对应的 value|`HGET user:1001 name`|key/field 不存在返回`nil`；key 非 Hash 类型直接报错|
|HSETNX|`HSETNX key field value`|O(1)|仅当 field**不存在**时，才设置其 value，原子性执行|`HSETNX article:1001:like user:1001 1`|成功返回 1，失败返回 0；是幂等控制、防重复操作的核心命令|
|HDEL|`HDEL key field [field ...]`|O (n)，n 为本次删除的 field 数量|删除哈希表中一个 / 多个 field，返回实际删除的 field 数量|`HDEL cart:1001 goods:2001 goods:2002`|field 不存在自动忽略；key 不存在返回 0；删除最后一个 field 后，key 会自动删除|

表格

|命令|语法|时间复杂度|核心作用|示例|注意事项|
|---|---|---|---|---|---|
|HMGET|`HMGET key field [field ...]`|O (n)，n 为本次查询的 field 数量|批量获取哈希表中多个 field 的 value，按输入顺序返回|`HMGET user:1001 name age phone`|不存在的 field 返回`nil`；替代多次循环 HGET，减少网络 RTT|
|HMSET|`HMSET key field value [field value ...]`|O (n)，n 为本次设置的 field 数量|批量设置多个 field-value 对|`HMSET user:1001 name lisi age 26`|**Redis 4.0 + 已废弃**，完全被 HSET 替代，新业务禁止使用|

### 3.3 数值原子操作命令（高频）

针对存储整数 / 浮点数的 field，提供原子性加减操作，是多维度计数场景的核心能力。

表格

|命令|语法|时间复杂度|核心作用|示例|
|---|---|---|---|---|
|HINCRBY|`HINCRBY key field 步长`|O(1)|对指定 field 的整数值原子增加指定整数步长，key/field 不存在默认从 0 开始，返回增加后的值|`HINCRBY article:1001 read_count 1`|
|HINCRBYFLOAT|`HINCRBYFLOAT key field 浮点步长`|O(1)|对指定 field 的数值原子增加指定浮点数，返回增加后的值|`HINCRBYFLOAT goods:price 0.5`|

### 3.4 查询与判断命令

|命令|语法|时间复杂度|核心作用|示例|注意事项|

### 3.5 全量遍历命令（高危 + 安全替代）

表格

|命令|语法|时间复杂度|核心作用|高危警告|替代方案|
|---|---|---|---|---|---|
|HGETALL|`HGETALL key`|O (n)，n 为哈希表总 field 数|一次性返回哈希表中所有的 field-value 对|**线上长哈希绝对禁用**，field 数量过大会直接阻塞 Redis 主线程，引发实例雪崩|仅 field 数量明确 < 100 时可使用，长哈希必须用 HSCAN 替代|
|HKEYS|`HKEYS key`|O (n)，n 为哈希表总 field 数|一次性返回哈希表中所有的 field|长哈希禁用，风险同 HGETALL|HSCAN 配合 MATCH 参数过滤|
|HVALS|`HVALS key`|O (n)，n 为哈希表总 field 数|一次性返回哈希表中所有的 value|长哈希禁用，风险同 HGETALL|HSCAN 配合 WITHVALUES 使用|
|HSCAN|`HSCAN key cursor [MATCH pattern] [COUNT count]`|单次调用 O (1)，全量遍历 O (n)|增量式迭代遍历哈希表，每次仅返回少量数据，无主线程阻塞风险|无高危风险，是线上长哈希遍历的唯一合规方案|示例：`HSCAN user:fans:1001 0 MATCH * COUNT 100`，循环迭代直到 cursor 返回 0|

## 四、典型应用场景

### 4.1 结构化对象存储（核心场景）

Hash 最核心的用途，替代「String+JSON 序列化」方案，存储用户信息、商品详情、文章信息、设备信息等结构化对象，完美解决单字段修改的全量读写开销问题。

- 核心优势：无需序列化 / 反序列化，可独立读写 / 修改单个字段，网络开销小，无并发竞态问题，内存占用更低
- 实现示例：
    
    bash
    
    运行
    
    ```
    # 存储用户完整信息，无需JSON序列化
    HSET user:1001 id 1001 name zhangsan age 25 phone 13800138000 avatar /avatar/1001.jpg vip_level 1
    # 单独读取用户姓名，无需拉取全量数据
    HGET user:1001 name
    # 单独修改用户年龄，无并发覆盖风险
    HSET user:1001 age 26
    # 批量获取核心字段，减少网络请求
    HMGET user:1001 name phone vip_level
    ```
    

### 4.2 多维度原子计数器

基于 HINCRBY 的原子性，实现多维度的计数统计，用一个 Hash key 管理同一主体的多个计数维度，无需创建大量 String key，管理更便捷，内存更节省。

- 典型场景：文章阅读 / 点赞 / 评论 / 收藏数、商品销量 / 浏览量、用户行为统计、接口多维度调用量统计
- 实现示例：
    
    bash
    
    运行
    
    ```
    # 初始化文章统计维度
    HSET article:1001 read_count 0 like_count 0 comment_count 0 collect_count 0
    # 用户阅读文章，阅读量+1
    HINCRBY article:1001 read_count 1
    # 用户点赞，点赞量+1
    HINCRBY article:1001 like_count 1
    # 批量获取全量统计数据
    HMGET article:1001 read_count like_count comment_count collect_count
    ```
    

### 4.3 电商购物车

Hash 是实现电商购物车的最优方案，用用户 ID 作为 Redis 顶级 key，商品 ID 作为 field，商品购买数量 / 规格作为 value，实现购物车的增删改查全量操作，粒度极细，性能极高。

- 实现示例：
    
    bash
    
    运行
    
    ```
    # 用户1001加入购物车：商品2001买2件，商品2002买1件
    HSET cart:1001 goods:2001 2 goods:2002 1
    # 修改商品2001的数量，+1件
    HINCRBY cart:1001 goods:2001 1
    # 删除购物车中的商品2002
    HDEL cart:1001 goods:2002
    # 获取购物车商品总数量
    HLEN cart:1001
    # 分页遍历购物车商品（商品数量多时用HSCAN）
    HSCAN cart:1001 0 COUNT 10
    ```
    

### 4.4 细粒度权限 / 配置管理

用于存储用户权限、角色权限、系统配置、租户配置等数据，每个 field 对应一个权限项 / 配置项，value 对应权限值 / 配置值，支持细粒度的权限校验、配置修改，无需全量读写。

- 实现示例：
    
    bash
    
    运行
    
    ```
    # 存储管理员角色的权限配置
    HSET role:admin user:add 1 user:delete 1 user:query 1 user:update 1 order:query 1
    # 存储访客角色的权限配置
    HSET role:guest user:query 1 order:query 1 user:add 0
    # 权限校验：判断访客是否有用户删除权限
    HGET role:guest user:delete
    # 单独修改管理员的订单管理权限
    HSET role:admin order:delete 1
    ```
    

### 4.5 热点数据分桶缓存

针对超热点的 String 大 key，为了避免热 key 问题、缓存击穿和单 key 分片倾斜，将大 key 拆分为 Hash 的多个 field，存储在同一个 Redis key 下，既解决热 key 问题，又降低内存开销，方便统一管理。

- 典型场景：超热点商品库存、秒杀活动库存、高频访问的配置项
- 实现示例：将超热点的商品库存拆分为 10 个分桶，存储在同一个 Hash key 中，读写时随机选择分桶，分散热 key 压力

### 4.6 幂等性控制与防重复提交

基于 HSETNX 的原子性，实现分布式场景下的幂等控制，无需加锁，即可防止重复提交、重复点赞、重复下单等问题。

- 实现示例：用户点赞文章，用用户 ID 作为 field，仅当 field 不存在时设置成功，代表首次点赞；返回 0 代表已点赞，拒绝重复操作
    
    bash
    
    运行
    
    ```
    HSETNX article:1001:like user:1001 1
    ```

## 五、常见坑与最佳实践

1. **绝对禁止对长哈希使用 HGETALL/HKEYS/HVALS**
    
    这是 Hash 最常见的线上事故！HGETALL 会一次性拉取全量 field-value 对，时间复杂度 O (n)，若哈希表有上万个 field，会直接阻塞 Redis 单线程，导致整个实例卡顿、雪崩。
    
    - 线上规范：仅当 field 数量明确小于 100 个时，才可使用 HGETALL；长哈希遍历必须使用 HSCAN 增量迭代，每次返回少量数据，无阻塞风险
    - 反例：循环执行 HGETALL 获取百万级 field 的哈希表，直接导致 Redis 实例宕机
    
2. **严格控制 Hash 的 field 数量与单个 value 大小**
    
    线上单个 Hash 的 field 总数建议控制在**1000 个以内**，单个 field 的 value 字节大小控制在**1KB 以内**。
    
    - field 过多会触发编码转换，从 ziplist 转为 hashtable，内存占用飙升，同时范围操作性能急剧下降
    - 单个 value 过大，会直接触发编码转换，同时形成大 key，导致网络 IO 阻塞、主从同步延迟
    - 优化方案：超大型对象按业务维度拆分，比如用户信息拆分为`user:1001:base`（基础信息）、`user:1001:ext`（扩展信息）两个 Hash key
    
3. **Hash 仅支持 key 级别的过期时间，不支持 field 级别过期**
    
    这是最高频的业务坑！Redis 的过期时间只能作用于整个 Hash 顶级 key，无法给单个 field 设置过期时间。
    
    - 合规解决方案：
        
        1. 方案 1：value 中附加过期时间戳，读取时判断是否过期，配合定时任务清理过期 field
        2. 方案 2：将需要单独过期的 field 拆分为独立的 String key，单独设置过期时间
        3. 方案 3：配合 Sorted Set 存储 field 的过期时间，定时扫描清理
        
    - 反例：试图给单个 field 设置过期时间，导致业务逻辑失效
    
4. **Hash vs String（JSON） 明确选型规范**
    
    很多业务场景会纠结选型，这里给出明确的判断标准，避免错误使用：
    
    表格
    
    |业务场景|推荐类型|核心原因|
    |---|---|---|
    |频繁修改 / 读取单个字段，很少全量读写|Hash|无需全量序列化 / 反序列化，O (1) 操作单字段，无并发竞态问题，网络开销极小|
    |几乎全量读写，极少修改单个字段|String（JSON）|全量读取一次即可，序列化开销可忽略，支持嵌套复杂对象，结构更灵活|
    |对象字段多，大部分字段很少访问|Hash|按需读取字段，减少无效数据传输，降低内存与网络开销|
    |对象字段少、结构固定，无嵌套|优先 Hash|内存占用更低，操作更灵活|
    |需要存储嵌套复杂对象（如对象里包含数组 / 子对象）|String（JSON）|Hash 仅支持扁平化的 field-value 结构，不支持嵌套|
    
5. **优先使用批量操作，减少网络 RTT**
    
    多次循环执行单 field 的 HGET/HSET，会产生大量网络往返，性能极差，甚至会打满 Redis 连接数。
    
    - 最佳实践：
        
        1. 批量写入：用`HSET key field1 value1 field2 value2...`替代多次循环 HSET
        2. 批量读取：用`HMGET key field1 field2...`替代多次循环 HGET
        3. 单次批量操作的 field 数量建议控制在 100 个以内，避免单命令执行时间过长阻塞主线程
        
    
6. **合理调整编码配置，最大化内存优化**
    
    可根据业务场景调整 ziplist/listpack 的阈值参数，让更多 Hash 使用紧凑型编码，降低内存占用。
    
    - 若业务 Hash 均为小 field、小 value，且 field 数量不多，可适当调大`hash-max-ziplist-entries`（如 1000），让更多 Hash 使用 ziplist 编码
    - 若业务 Hash 经常有超过 64 字节的 value，可适当调大`hash-max-ziplist-value`（如 128），避免频繁触发编码转换
    - 红线：entries 参数不建议超过 2000，否则 ziplist 的遍历性能会急剧下降
    
7. **禁止用 Hash 存储超扁平的海量数据**
    
    不要把 Hash 当成全局 key-value 存储，比如把用户 ID 作为 field，用户全量信息作为 value，存入一个`all_user`大 Hash key 中。
    
    - 核心缺陷：
        
        1. 极易形成超大 key，field 数量百万级，导致 Redis 主线程阻塞
        2. 无法给单个用户设置过期时间
        3. 集群环境下，大 key 只会落在一个分片上，导致数据倾斜、分片负载不均
        
    - 正确做法：每个用户用独立的 Hash/String key，如`user:1001`、`user:1002`，分散到集群各个分片

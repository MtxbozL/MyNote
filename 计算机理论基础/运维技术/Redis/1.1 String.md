## 一、核心概述

String 是 Redis 最基础、最核心、使用频率最高的数据类型，本质是**二进制安全的字节序列**，可存储文本、数字、序列化对象、二进制文件（图片 / 音视频）等任意格式的数据。

- 单 key 对应单 value，key 和 value 的最大容量均为**512MB**
- 所有操作均为**原子性**，基于 Redis 单线程模型，并发场景下无竞争问题
- 是 Redis 其他所有数据类型的底层基础

---

## 二、底层实现原理

### 2.1 核心存储结构：SDS（简单动态字符串）

Redis String 并未直接使用 C 语言原生字符串，而是自研了 SDS（Simple Dynamic String）结构作为底层存储，解决了 C 字符串的性能与安全缺陷。

#### SDS 核心结构（以 sdshdr8 为例）

```C
struct sdshdr8 {
    uint8_t len;    // 已使用的字节长度
    uint8_t alloc;  // 已分配的总字节数（不包含头部和结束符\0）
    unsigned char flags; // 标志位，区分SDS类型
    char buf[];     // 实际存储数据的字节数组
};
```

#### SDS 核心优势

1. **O (1) 时间复杂度获取长度**：通过`len`字段直接获取，无需遍历，远优于 C 字符串的 O (n) 遍历
2. **二进制安全**：不依赖`\0`作为字符串结束标志，通过`len`判断数据边界，可存储包含`\0`的任意二进制数据
3. **减少内存重分配**：内置空间预分配 + 惰性空间释放机制，追加 / 缩短字符串时避免频繁的内存申请与释放
4. **兼容 C 字符串函数**：buf 数组末尾默认追加`\0`，可直接复用 C 语言标准库的字符串函数

### 2.2 String 的三种编码格式

Redis 会根据 value 的类型和长度，自动选择最优的编码格式，最大化节省内存、提升性能，编码可自动转换（不可逆）。

|编码类型|触发条件|核心特点|
|---|---|---|
|int|存储的值为可转换为 64 位有符号整数的字符串，范围`[-9223372036854775808, 9223372036854775807]`|直接将 value 存储在 redisObject 的指针字段，无需额外内存分配，内存占用极小|
|embstr|存储的字符串长度 **≤44 字节 **（Redis 3.2+，旧版本为 39 字节），且非 int 编码|只读编码，一次内存分配，redisObject 与 SDS 结构连续存储，缓存命中率高，内存碎片少|
|raw|存储的字符串长度 **>44 字节 **|两次内存分配，分别为 redisObject 和 SDS 分配独立内存空间，支持修改，无长度限制（上限 512MB）|

#### 编码转换规则（不可逆）

1. int 编码的 value，若修改为非整数格式，自动转为 embstr/raw 编码
2. embstr 编码为只读，任何修改操作（APPEND/SET 等）都会直接转为 raw 编码
3. 编码转换仅会从小内存占用向大内存占用转换，无法反向转换（除非重新 SET 值）

## 三、核心命令大全

按功能分为 5 大类，标注高频使用命令与核心注意事项。

### 3.1 基础读写命令（高频）

|命令|语法|作用|示例|注意事项|
|---|---|---|---|---|
|SET|`SET key value [参数]`|为 key 设置 value，是 Redis 最核心的命令|`SET user:1001 zhangsan EX 3600 NX`|核心参数：<br><br>- `EX seconds`：设置秒级过期时间<br><br>- `PX milliseconds`：设置毫秒级过期时间<br><br>- `NX`：仅 key 不存在时才设置（分布式锁核心）<br><br>- `XX`：仅 key 存在时才设置（更新专用）<br><br>- `GET`：设置新值并返回旧值|
|GET|`GET key`|获取 key 对应的 value|`GET user:1001`|key 不存在返回`nil`；key 非 String 类型报错|
|GETSET|`GETSET key value`|为 key 设置新值，同时返回旧值|`GETSET user:1001 lisi`|已被`SET key value GET`参数替代，不推荐新业务使用|
|APPEND|`APPEND key value`|向 key 的 value 末尾追加内容|`APPEND user:1001 _vip`|key 不存在则新建，返回追加后的总字节长度|
|STRLEN|`STRLEN key`|获取 value 的**字节长度**|`STRLEN user:1001`|注意是字节长度，非字符数（如 UTF-8 中文占 3 字节）|

### 3.2 数值原子操作命令（高频）

针对存储整数 / 浮点数的 String，提供原子性加减操作，是高并发计数场景的核心能力。

|命令|语法|作用|示例|
|---|---|---|---|
|INCR|`INCR key`|对 key 的 value 原子 + 1，key 不存在默认从 0 开始|`INCR article:read:1001`|
|INCRBY|`INCRBY key 步长`|对 key 的 value 原子增加指定整数步长|`INCRBY order:count 10`|
|INCRBYFLOAT|`INCRBYFLOAT key 浮点步长`|对 key 的 value 原子增加指定浮点数|`INCRBYFLOAT goods:price 0.5`|
|DECR|`DECR key`|对 key 的 value 原子 - 1，key 不存在默认从 0 开始|`DECR stock:1001`|
|DECRBY|`DECRBY key 步长`|对 key 的 value 原子减少指定整数步长|`DECRBY stock:1001 5`|

> 核心特性：所有数值操作均为原子性，单线程串行执行，高并发场景下无竞态问题，无需加锁。

### 3.3 批量操作命令（性能优化核心）

批量操作可显著减少网络 RTT（往返时间），提升 Redis 访问性能，所有批量命令均为原子操作。

表格

|命令|语法|作用|示例|
|---|---|---|---|
|MSET|`MSET key1 value1 key2 value2 ...`|批量设置多个 key-value，全部成功才生效|`MSET user:1001 zhangsan user:1002 lisi`|
|MGET|`MGET key1 key2 ...`|批量获取多个 key 的 value，按输入顺序返回|`MGET user:1001 user:1002`|
|MSETNX|`MSETNX key1 value1 key2 value2 ...`|仅当所有 key 都不存在时，批量设置成功|`MSETNX lock:1 lock_val lock:2 lock_val`|

> 最佳实践：单次批量操作的 key 数量建议控制在 100 个以内，避免单命令执行时间过长，阻塞 Redis 单线程。

### 3.4 位操作（BitMap）命令

String 本质是字节数组，支持对单个二进制位（bit）进行操作，1 个 key 最大支持 2^32 个 bit（对应 512MB），是海量数据统计的低成本方案。

表格

|命令|语法|作用|示例|
|---|---|---|---|
|SETBIT|`SETBIT key offset 0/1`|设置指定偏移量的 bit 位值，返回旧值|`SETBIT sign:202603:1001 7 1`|
|GETBIT|`GETBIT key offset`|获取指定偏移量的 bit 位值|`GETBIT sign:202603:1001 7`|
|BITCOUNT|`BITCOUNT key [start end]`|统计指定范围内 bit 值为 1 的个数|`BITCOUNT sign:202603:1001`|
|BITPOS|`BITPOS key 0/1 [start end]`|查找指定范围内第一个 0/1 的 bit 偏移量|`BITPOS sign:202603:1001 0`|
|BITOP|`BITOP 操作符 目标key key1 key2 ...`|对多个 key 执行位运算（AND/OR/XOR/NOT），结果存入目标 key|`BITOP AND active:user sign:0301 sign:0302`|

## 四、典型应用场景

### 4.1 常规数据缓存

最核心的使用场景，用于缓存高频读、低频写的数据，降低数据库压力。

- 示例：用户信息、商品详情、系统配置、静态页面等
- 用法：`SET user:1001 '{"id":1001,"name":"zhangsan","phone":"13xxxxxxxxx"}' EX 86400`

### 4.2 分布式计数器

基于 INCR/DECR 的原子性，实现高并发场景下的计数功能，无锁安全。

- 示例：文章阅读量、点赞数、接口调用次数、库存计数、UV 统计
- 用法：用户每次访问文章，执行`INCR article:read:1001`，实时更新阅读量

### 4.3 分布式锁

基于 SET 命令的 NX+EX 参数，实现轻量级分布式锁，是业界最通用的 Redis 分布式锁实现方案。

- 核心用法：`SET lock:order:1001 random_uuid NX EX 30`
    
    - NX：保证只有一个客户端能加锁成功
    - EX：设置锁过期时间，避免客户端宕机导致死锁
    - random_uuid：解锁时校验，防止误释放其他客户端的锁
    

### 4.4 接口限流

基于 INCR + 过期时间，实现简单高效的限流方案。

- 示例：限制用户 1 分钟内最多调用接口 100 次
- 用法：
    
    1. 每次请求执行`INCR api:limit:user:1001:202603081500`
    2. 第一次调用时设置过期时间 60 秒
    3. 若返回值超过 100，拒绝本次请求
    

### 4.5 位图（BitMap）海量数据统计

基于位操作，用极低的内存成本实现海量数据的统计，1 亿条数据仅需约 12MB 内存。

- 示例：用户签到、日活 / 月活用户统计、用户在线状态、布隆过滤器底层实现
- 用法：用户 3 月 8 日签到，执行`SETBIT sign:202603:1001 7 1`（偏移量 7 对应当月第 8 天），用`BITCOUNT`统计当月签到天数

### 4.6 分布式 Session 共享

分布式系统中，将用户登录后的 Session 信息统一存储在 Redis 中，实现多服务节点的 Session 共享。

- 用法：`SET session:xxxxxx 用户信息序列化数据 EX 1800`，所有服务节点通过 sessionId 从 Redis 中获取用户登录状态

## 五、常见坑与最佳实践

1. **严禁大 key**
    
    线上环境单个 String 的 value 建议控制在**10KB 以内**，绝对禁止存储超过 100KB 的大 key。大 key 会导致网络阻塞、Redis 主线程卡顿、主从同步异常、内存碎片激增。大对象建议拆分存储或使用 Hash 类型。
    
2. **必须设置合理的过期时间**
    
    所有非永久 key 必须设置过期时间，避免内存泄漏。禁止用`DEL+SET`替代`SET EX`，前者非原子，并发场景下会出现过期时间丢失问题，直接使用 SET 命令自带的过期参数。
    
3. **避免用 String 存储复杂结构**
    
    不要将列表、映射等复杂结构序列化为 JSON 存入 String，修改单个字段需要全量读写，性能极差。应根据数据结构选择 Hash、List、Set 等对应类型。
    
4. **数值操作注意边界**
    
    INCR/DECR 仅支持 64 位有符号整数，最大值为 9223372036854775807，超出范围会报错。超大数值建议拆分为多个 key 或用字符串存储。
    
5. **位操作避免超大偏移量**
    
    SETBIT 的 offset 不要直接设置超大值（如 1 亿），会导致 Redis 一次性分配大量内存，阻塞主线程，甚至触发 OOM。
    
6. **优先使用批量操作**
    
    多次 GET/SET 操作，优先替换为 MGET/MSET，减少网络往返次数，提升性能，同时注意控制单次批量操作的 key 数量。

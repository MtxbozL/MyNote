## 一、核心概述

Redis Sentinel（哨兵模式）是**Redis 官方原生的高可用解决方案**，基于主从复制架构实现，彻底解决了主从复制模式下「主节点故障需人工手动切换」的核心痛点，实现了 Redis 服务的自动化故障检测、故障转移、配置推送与告警通知。

### 核心定位

主从复制解决了**数据备份与读能力扩展**的问题，而哨兵模式解决了**主节点单点故障导致的服务不可用**问题，二者结合构成了 Redis 生产环境的基础高可用架构。

### 核心特性

1. **自动化监控**：持续监控主节点、从节点、其他哨兵节点的健康状态，自动判定节点是否下线
2. **自动故障转移**：主节点故障后，无需人工干预，自动完成从节点升主、其他从节点切换新主、配置更新全流程
3. **配置中心能力**：客户端无需直连 Redis 节点，仅需连接哨兵集群，即可获取当前主节点地址；故障切换后，哨兵会主动向客户端推送最新主节点地址
4. **事件通知与告警**：节点下线、故障切换完成、配置异常等事件，可主动推送至客户端、管理员邮箱 / 短信，实现故障告警
5. **集群自管理**：哨兵节点之间可自动发现，自动维护集群拓扑，支持动态扩缩容
6. **兼容主从复制所有能力**：完全兼容主从复制的读写分离、数据同步、级联复制等特性，无侵入式改造

### 核心解决的痛点

1. 主节点单点故障，人工切换效率低、故障恢复时间长
2. 主节点地址变更后，客户端需手动修改配置，运维成本高
3. 主从节点健康状态需人工巡检，故障发现不及时
4. 人工切换主从易出现操作失误，导致数据不一致、服务中断

## 二、核心架构与角色职责

### 2.1 核心架构组成

哨兵模式架构由三类节点组成，所有节点基于 TCP 长连接通信，完整架构如下：

```bash
客户端 ←→ 哨兵集群（3+奇数节点） ←→ Redis主从集群（1主N从）
```

|节点类型|数量要求|核心职责|端口|
|---|---|---|---|
|哨兵节点（Sentinel）|生产环境必须**3 个及以上奇数个**|负责监控、故障判定、领导者选举、故障切换、配置管理|默认 26379|
|Redis 主节点（Master）|1 个|处理写请求，同步写命令到从节点，是哨兵的核心监控对象|默认 6379|
|Redis 从节点（Replica）|至少 1 个，生产环境建议 2+|处理读请求，同步主节点数据，主节点故障后可被选举为新主节点|默认 6379|

### 2.2 关键部署原则

1. **奇数节点原则**：哨兵集群必须部署奇数个节点，最少 3 个，禁止 2 节点部署。核心原因是哨兵的故障判定、领导者选举依赖**过半投票（Quorum）** 机制，奇数节点可避免平票导致的选举失败。
2. **物理隔离原则**：哨兵节点禁止与 Redis 主 / 从节点部署在同一台物理机 / 虚拟机 / 容器内，避免服务器宕机导致 Redis 节点与哨兵节点同时失效，失去监控能力。
3. **跨机房部署原则**：核心业务场景下，哨兵节点需跨机房 / 可用区部署，避免单机房故障导致整个哨兵集群失效。
4. **Quorum 过半原则**：判定主节点客观下线的票数（Quorum）必须设置为「哨兵节点总数 / 2 + 1」，即过半票数，避免脑裂问题。

## 三、四大核心功能详解

### 3.1 监控（Monitoring）

监控是哨兵的基础能力，哨兵集群会持续对所有节点做健康状态检测：

1. **监控范围**：所有哨兵节点会同时监控配置的主节点、主节点下的所有从节点、集群内其他所有哨兵节点
2. **检测方式**：哨兵节点每隔 1 秒向所有监控的节点发送`PING`命令，根据节点的回复判断健康状态
3. **状态同步**：哨兵节点之间会通过发布订阅机制，实时同步节点的健康状态、配置信息，保证所有哨兵的视图一致

### 3.2 自动故障转移（Automatic Failover）

故障转移是哨兵的核心能力，也是生产环境高可用的核心保障：

1. **触发条件**：主节点被哨兵集群判定为**客观下线（ODOWN）** 后，立即触发故障转移流程
2. **核心执行动作**：
    
    1. 从故障主节点的所有从节点中，选举出一个合格的从节点，升级为新的主节点
    2. 让剩余所有从节点，切换为新主节点的从节点，开始从新主节点同步数据
    3. 将故障的旧主节点标记为新主节点的从节点，恢复上线后自动从新主节点同步数据
    4. 更新集群配置，向客户端推送最新的主节点地址
    
3. **核心特性**：整个故障转移流程完全自动化，无需人工干预，故障恢复时间可控制在秒级

### 3.3 配置中心（Configuration Provider）

哨兵集群是 Redis 客户端的统一配置入口，彻底解决了主节点地址变更后的客户端适配问题：

1. 客户端启动时，无需配置 Redis 主节点地址，仅需配置哨兵集群的地址列表和主节点名称
2. 客户端连接哨兵集群后，通过哨兵命令获取当前可用的主节点地址，直连主节点进行读写
3. 故障切换完成后，哨兵集群会主动向客户端推送新的主节点地址，客户端自动切换连接，无需重启、无需修改配置

### 3.4 事件通知（Notification）

哨兵提供了完整的事件推送能力，可实现故障告警与自动化运维：

1. **可推送的核心事件**：主节点主观下线、主节点客观下线、故障转移开始、故障转移完成、从节点切换主节点、哨兵集群配置变更、节点不可达等
2. **推送方式**：通过脚本回调、发布订阅频道推送两种方式，可对接监控系统、告警平台、自动化运维平台
3. **典型应用**：节点下线时触发短信 / 邮件告警、故障切换完成后触发配置同步、异常事件触发运维工单

## 四、底层核心原理

### 4.1 节点下线判定机制

这是哨兵的核心基础，分为**主观下线（SDOWN）** 和**客观下线（ODOWN）** 两个阶段，彻底避免了网络波动导致的故障误判。

#### 4.1.1 主观下线（Subjective Down，SDOWN）

- **定义**：单个哨兵节点视角下，判定某个节点不可达，标记为「主观下线」
- **触发条件**：哨兵节点向目标节点发送`PING`命令后，超过`down-after-milliseconds`配置的超时时间，仍未收到有效回复
- **核心特性**：主观下线是**单个哨兵的单边判断**，不代表节点真的故障，不会触发故障转移，仅作为客观下线的前置条件
- **适用范围**：主节点、从节点、其他哨兵节点，均可被标记为主观下线

#### 4.1.2 客观下线（Objective Down，ODOWN）

- **定义**：多个哨兵节点达成共识，共同判定主节点不可达，标记为「客观下线」，是触发故障转移的唯一条件
- **触发条件（必须同时满足）**：
    
    1. 哨兵节点已将主节点标记为主观下线
    2. 该哨兵通过`SENTINEL is-master-down-by-addr`命令，向其他哨兵节点询问主节点的状态
    3. 收集到的「判定主节点下线」的哨兵票数，达到了配置的`quorum`阈值
    
- **核心特性**：客观下线是**集群共识结果**，仅针对主节点生效；从节点、哨兵节点不会触发客观下线判定
- **Quorum 配置原则**：生产环境必须设置为「哨兵节点总数 / 2 + 1」，即过半票数，例如 3 个哨兵节点 quorum=2，5 个哨兵节点 quorum=3

#### SDOWN vs ODOWN 核心对比

|对比维度|主观下线（SDOWN）|客观下线（ODOWN）|
|---|---|---|
|判定主体|单个哨兵节点|多个哨兵节点集群共识|
|触发条件|单哨兵 PING 超时|达到 quorum 票数的哨兵共同判定下线|
|生效对象|主节点、从节点、哨兵节点|仅主节点|
|后续动作|向其他哨兵询问节点状态|触发哨兵领导者选举，启动故障转移流程|
|误判概率|高，易受网络波动影响|极低，集群共识避免误判|

### 4.2 哨兵领导者选举机制

主节点被标记为客观下线后，哨兵集群需要先选举出一个**哨兵领导者**，由该领导者全权负责后续的故障转移流程，避免多个哨兵同时执行切换导致的集群混乱。

#### 核心选举规则

哨兵的领导者选举基于**Raft 分布式一致性算法的简化版**，核心规则如下：

1. **发起选举**：每个将主节点标记为客观下线的哨兵节点，都会向其他哨兵节点发起投票请求，要求其他哨兵选举自己为领导者
2. **先到先得原则**：每个哨兵节点在一轮选举中，只能给第一个向自己发起投票请求的哨兵投票，不可重复投票
3. **过半当选原则**：发起选举的哨兵节点，必须获得**超过半数哨兵节点的投票**，才能当选为领导者
4. **平票重试原则**：一轮选举中没有任何哨兵获得过半票数，等待`failover-timeout`超时后，发起新一轮选举，直到选出领导者
5. **唯一领导者原则**：一轮选举只会选出一个领导者，由该领导者唯一执行故障转移流程

#### 为什么哨兵节点必须是奇数个？

1. **避免平票死循环**：偶数个哨兵节点（如 2 个、4 个）极易出现平票，导致多轮选举无法选出领导者，故障转移流程无法启动
2. **过半原则生效**：奇数个节点才能稳定实现「过半投票」，3 个节点最少 2 票过半，5 个节点最少 3 票过半，不会出现票数对半的情况
3. **容灾能力更强**：3 个节点允许 1 个节点故障，5 个节点允许 2 个节点故障，相同容灾能力下，奇数节点部署成本更低

### 4.3 新主节点选举规则

哨兵领导者当选后，第一步是从故障主节点的所有从节点中，选举出一个合格的从节点作为新的主节点，选举规则按优先级从高到低排序：

|优先级|选举规则|规则说明|
|---|---|---|
|1（最高）|从节点优先级（replica-priority）|优先选择`replica-priority`配置值最小的从节点；值为 0 的从节点永远不会被选举为新主节点|
|2|复制偏移量（Replication Offset）|优先级相同的情况下，优先选择复制偏移量最大的从节点，即与旧主节点数据同步最完整、数据丢失最少的从节点|
|3（最低）|运行 ID（Run ID）|前两个条件都相同的情况下，优先选择运行 ID 字典序最小的从节点，保证选举结果唯一|

#### 选举过滤规则

选举前，哨兵领导者会先过滤掉不合格的从节点，只有符合以下条件的从节点才能进入选举池：

1. 与哨兵节点之间网络连通，未被标记为主观下线
2. 最近 5 秒内回复过哨兵的 INFO 命令，节点状态健康
3. 与旧主节点断连时间不超过`down-after-milliseconds * 10`，避免数据过于陈旧的从节点当选
4. `replica-priority`不为 0

### 4.4 完整故障转移执行流程

主节点故障后，哨兵集群会按以下流程完成自动化故障转移，全程无人工干预：

1. **故障检测阶段**：哨兵节点持续 PING 主节点，超过超时时间未收到回复，将主节点标记为**主观下线（SDOWN）**
2. **集群共识阶段**：该哨兵向其他哨兵询问主节点状态，收集到的下线票数达到 quorum 阈值，将主节点标记为**客观下线（ODOWN）**
3. **领导者选举阶段**：哨兵集群发起领导者选举，选出唯一的哨兵领导者，负责执行故障转移
4. **新主节点选举阶段**：哨兵领导者从合格的从节点中，按优先级规则选出新的主节点
5. **主节点提升阶段**：哨兵领导者向当选的从节点发送`REPLICAOF NO ONE`命令，将其升级为新的主节点，并持续确认其状态正常
6. **从节点切换阶段**：哨兵领导者向剩余所有从节点发送`REPLICAOF 新主节点IP 新主节点端口`命令，让所有从节点切换到新主节点，开始数据同步
7. **旧主节点处理阶段**：哨兵将故障的旧主节点标记为新主节点的从节点，待其恢复上线后，自动从新主节点同步数据
8. **配置更新与通知阶段**：哨兵集群更新集群配置，向客户端推送新的主节点地址，推送故障转移完成事件，整个流程结束

### 4.5 哨兵集群自动发现机制

哨兵集群无需手动配置其他哨兵节点的地址，仅需配置主节点地址，即可自动发现集群内所有其他哨兵节点和所有从节点，核心实现原理：

1. **从节点自动发现**：哨兵通过向主节点发送`INFO REPLICATION`命令，获取主节点下的所有从节点地址，自动将这些从节点加入监控列表
2. **哨兵节点自动发现**：
    
    1. 每个哨兵节点都会向监控的主节点的`__sentinel__:hello`发布订阅频道，定期推送自己的 IP、端口、运行 ID 等信息
    2. 所有监控同一个主节点的哨兵节点，都会订阅这个频道，从而收到其他哨兵的信息，自动发现彼此
    3. 发现新的哨兵节点后，会自动与其建立 TCP 长连接，用于后续的投票、状态同步

## 五、核心配置与命令

### 5.1 核心配置项详解

哨兵有独立的配置文件`sentinel.conf`，以下为生产环境核心配置项，适配 Redis 5.0 + 版本：

|配置项|语法格式|核心作用|生产环境推荐值|注意事项|
|---|---|---|---|---|
|`sentinel monitor`|`sentinel monitor <master-name> <master-ip> <master-port> <quorum>`|定义要监控的主节点，设置主节点名称、地址、客观下线票数阈值|按需填写，quorum 必须为过半票数|主节点名称只能包含字母、数字、.-_；一个哨兵集群可监控多个主节点，配置多条即可|
|`sentinel auth-pass`|`sentinel auth-pass <master-name> <password>`|主节点设置了密码时，配置主节点的认证密码|与主节点`requirepass`一致|主从节点必须使用相同的密码，否则哨兵无法监控从节点|
|`sentinel down-after-milliseconds`|`sentinel down-after-milliseconds <master-name> <毫秒数>`|配置 PING 超时时间，超过该时间未收到回复，标记为主观下线|3000~5000（3~5 秒）|值过小易因网络波动误判，值过大会导致故障检测不及时|
|`sentinel failover-timeout`|`sentinel failover-timeout <master-name> <毫秒数>`|故障转移超时时间，控制选举、切换的整体超时|180000（3 分钟）|同时用于平票选举重试间隔、从节点同步超时判定|
|`sentinel parallel-syncs`|`sentinel parallel-syncs <master-name> <数量>`|故障切换后，同时向新主节点发起同步的从节点数量|1|值越大，新主节点的同步压力越大；值为 1 时，从节点逐个同步，避免新主节点带宽 / CPU 打满|
|`sentinel notification-script`|`sentinel notification-script <master-name> <脚本路径>`|配置事件通知脚本，节点状态变更、故障切换时自动执行|按需填写|脚本必须有可执行权限，用于对接告警平台|
|`sentinel client-reconfig-script`|`sentinel client-reconfig-script <master-name> <脚本路径>`|配置故障切换完成后的回调脚本，主节点地址变更时自动执行|按需填写|用于客户端配置更新、DNS 刷新等自动化操作|
|`port`|`port 26379`|哨兵节点的服务端口|26379|禁止与 Redis 节点端口 6379 冲突|
|`daemonize yes`|`daemonize yes`|哨兵节点以守护进程后台运行|yes|生产环境必须开启|

### 5.2 核心操作命令

所有哨兵命令均需连接哨兵节点（默认 26379 端口）执行，核心命令如下：

|命令|语法|核心作用|示例|
|---|---|---|---|
|SENTINEL masters|`SENTINEL masters`|列出所有被监控的主节点信息，包括状态、配置、从节点数量等|`SENTINEL masters`|
|SENTINEL master|`SENTINEL master <master-name>`|查看指定主节点的详细信息|`SENTINEL master mymaster`|
|SENTINEL slaves|`SENTINEL slaves <master-name>`|查看指定主节点的所有从节点信息，包括状态、复制偏移量等|`SENTINEL slaves mymaster`|
|SENTINEL sentinels|`SENTINEL sentinels <master-name>`|查看监控指定主节点的所有其他哨兵节点信息|`SENTINEL sentinels mymaster`|
|SENTINEL get-master-addr-by-name|`SENTINEL get-master-addr-by-name <master-name>`|获取指定主节点的当前 IP 和端口，客户端核心命令|`SENTINEL get-master-addr-by-name mymaster`|
|SENTINEL failover|`SENTINEL failover <master-name>`|手动触发强制故障转移，用于运维演练、主节点计划性下线|`SENTINEL failover mymaster`|
|SENTINEL reset|`SENTINEL reset <master-name-pattern>`|重置匹配的主节点监控状态，清空历史状态，重新发现从节点和哨兵|`SENTINEL reset mymaster*`|
|SENTINEL ckquorum|`SENTINEL ckquorum <master-name>`|检查当前哨兵集群是否能达到 quorum 票数，用于验证集群可用性|`SENTINEL ckquorum mymaster`|
|PING|`PING`|测试哨兵节点是否正常运行，正常返回 PONG|`PING`|

## 六、典型应用场景

1. **核心业务 Redis 高可用部署**
    
    生产环境核心业务的 Redis 集群标准部署方案，基于哨兵 + 主从复制，实现主节点故障自动切换，保证服务 99.99% 可用性，避免单点故障导致的业务中断。
    
2. **读写分离架构的高可用保障**
    
    读多写少业务的读写分离架构中，哨兵集群不仅监控主节点，同时监控所有从节点，可自动剔除故障从节点，向客户端推送健康的从节点地址，保证读服务的高可用。
    
3. **自动化运维与故障告警**
    
    基于哨兵的事件通知能力，对接监控告警平台，实现节点故障实时告警、故障切换自动化通知、异常事件闭环处理，大幅降低运维成本。
    
4. **多机房容灾部署**
    
    跨机房部署哨兵集群与 Redis 主从集群，主机房故障时，哨兵集群可自动将备用机房的从节点升级为主节点，实现机房级容灾，避免单机房故障导致的服务完全不可用。
    
5. **微服务架构的统一配置中心**
    
    微服务架构中，所有服务通过哨兵集群获取 Redis 地址，无需在每个服务中配置 Redis 地址，主节点变更时无需重启服务，实现配置统一管理、动态更新。

## 七、常见坑与线上最佳实践

### 7.1 高频踩坑点与避坑方案

#### 1. 哨兵节点部署不规范，导致集群失效

**踩坑场景**：

- 仅部署 2 个哨兵节点，quorum=2，一个哨兵节点故障后，无法达成 quorum，无法判定主节点客观下线，故障转移无法触发
- 哨兵节点与 Redis 节点部署在同一台服务器，服务器宕机后，Redis 节点与哨兵节点同时失效，失去监控能力
    
    **避坑方案**：
- 生产环境必须部署**3 个及以上奇数个**哨兵节点，禁止 2 节点部署
- 哨兵节点必须与 Redis 节点物理隔离，跨服务器 / 跨机房部署
- 核心业务场景部署 5 个哨兵节点，允许 2 个节点同时故障，容灾能力更强

#### 2. 脑裂问题，导致数据丢失

**问题本质**：主节点因网络分区与哨兵集群、从节点失联，但主节点仍在正常运行，客户端继续向该主节点写入数据；网络恢复后，该主节点被降级为从节点，清空本地数据，全量同步新主节点，导致网络分区期间的写数据全部丢失。

**避坑方案**：

- 在 Redis 主节点配置防脑裂参数：

```bash
    min-replicas-to-write 1       # 至少有1个健康从节点，主节点才允许写
    min-replicas-max-lag 10       # 从节点最大延迟不超过10秒
```

- 哨兵 quorum 必须设置为过半票数，避免少数哨兵分区后独立发起故障转移
- 合理设置`down-after-milliseconds`，避免网络波动导致的误判

#### 3. 故障切换时，从节点全量同步导致业务阻塞

**踩坑场景**：`parallel-syncs`设置过大，故障切换后，所有从节点同时向新主节点发起全量同步，导致新主节点的 CPU、带宽被打满，新主节点阻塞，业务读写异常。

**避坑方案**：

- 生产环境`parallel-syncs`默认设置为 1，让从节点逐个同步，避免新主节点压力过大
- 大内存实例场景下，可调整为 2，禁止设置超过 3 的值
- 主从节点开启复制积压缓冲区，减少全量同步的概率

#### 4. 客户端直连主节点，故障切换后服务中断

**踩坑场景**：客户端配置中直接写死主节点的 IP 和端口，故障切换后，主节点地址变更，客户端无法自动切换，导致服务持续报错，业务中断。

**避坑方案**：

- 客户端必须通过哨兵集群获取主节点地址，禁止直连 Redis 节点
- 使用支持哨兵模式的 Redis 客户端（如 Jedis、Lettuce、Redisson），这些客户端原生支持哨兵事件监听，故障切换后自动切换连接
- 客户端必须配置完整的哨兵节点地址列表，不能只配置一个哨兵地址

#### 5. 主观下线超时时间设置不合理，导致误判或故障发现不及时

**踩坑场景**：

- `down-after-milliseconds`设置过小（如 100 毫秒），网络轻微波动就导致主节点被标记为主观下线，频繁触发故障切换
- `down-after-milliseconds`设置过大（如 30 秒），主节点真故障后，长时间无法检测到，故障恢复时间过长
    
    **避坑方案**：
- 生产环境推荐设置为 3000~5000 毫秒（3~5 秒），平衡误判率与故障检测速度
- 网络不稳定的跨机房场景，可适当调大至 10 秒，避免跨机房网络波动导致的误判

### 7.2 线上通用最佳实践

1. **哨兵集群与 Redis 集群分离部署**：哨兵节点使用独立的服务器，不与 Redis 节点共享资源，避免 Redis 节点的资源占用影响哨兵的监控与选举
2. **统一主从节点配置**：主从节点必须使用相同的 Redis 版本、相同的密码、相同的内存配置、相同的数据淘汰策略，避免故障切换后出现数据不一致、同步失败等问题
3. **关闭从节点只读配置**：生产环境必须保持从节点`replica-read-only yes`，禁止从节点写入，避免主从数据不一致
4. **监控全覆盖**：重点监控哨兵集群的节点状态、主节点健康状态、主从同步延迟、故障切换事件、主观 / 客观下线事件，设置阈值告警
5. **定期故障演练**：每季度至少执行一次故障切换演练，手动模拟主节点故障，验证哨兵集群的故障检测、自动切换能力，同时验证客户端的自动切换能力
6. **禁止手动修改主从配置**：哨兵模式下，禁止手动执行`REPLICAOF`命令修改主从关系，所有主从变更必须通过哨兵执行，否则会导致哨兵配置混乱，集群失效
7. **合理设置故障切换超时时间**：`failover-timeout`推荐设置为 3 分钟，避免超时时间过短导致切换失败，超时时间过长导致故障恢复延迟

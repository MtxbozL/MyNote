## 一、核心概述

List 是 Redis 中基于**插入顺序排序**的字符串元素集合，底层是双向链表结构的优化实现，是 Redis 最常用的多元素数据类型之一。

- 核心特性：**有序可重复**、支持双端 O (1) 原子操作、元素为 String 类型、单 key 最大支持`2^32 - 1`个元素（约 42 亿）
- 所有单元素操作均为**原子性**，范围操作按执行粒度保证原子性，并发场景下无竞态问题
- 是 Redis 中唯一同时支持高效队列、栈、双向遍历的基础数据类型

---

## 二、底层实现原理

Redis List 的底层编码经历了两次核心迭代，从 Redis 3.2 版本开始，统一使用**quicklist（快速列表）** 作为默认底层实现，替代了旧版本的`ziplist（压缩列表）`和`linkedlist（双端链表）`双编码方案。

### 2.1 旧版双编码方案（Redis 3.2 之前）

旧版本会根据列表元素的大小和数量，自动切换两种编码格式，平衡内存占用与读写性能：

|编码类型|触发条件|核心特点|核心缺陷|
|---|---|---|---|
|ziplist（压缩列表）|列表所有元素字节长度均小于`list-max-ziplist-value`（默认 64 字节），且元素总数小于`list-max-ziplist-entries`（默认 512 个）|连续内存存储，无指针开销，内存占用极低，适合小体量列表|插入 / 删除操作会引发内存重分配，元素过多时性能急剧下降；存在连锁更新风险|
|linkedlist（双端链表）|不满足 ziplist 触发条件的列表|每个元素为独立节点，前后节点通过指针关联，双端插入 / 删除 O (1)，无元素数量限制|指针内存开销大，节点内存不连续，缓存命中率低，易产生内存碎片|

### 2.2 新版核心实现：quicklist（快速列表）

quicklist 是 ziplist 与 linkedlist 的结合体，完美平衡了内存占用、读写性能与操作稳定性，是 Redis List 的最终优化方案。

#### 核心结构

quicklist 本质是**一个由 ziplist 作为节点的双向链表**：

1. 整个 quicklist 是双向链表，链表中的每个节点称为`quicklistNode`
2. 每个`quicklistNode`内部对应一个独立的 ziplist 压缩列表，用于实际存储列表元素
3. 通过配置项控制每个 ziplist 的大小，兼顾连续内存的优势与操作性能

```c
// 核心结构简化版
struct quicklist {
    quicklistNode *head; // 头节点
    quicklistNode *tail; // 尾节点
    unsigned long len;   // 列表总元素个数
    int fill;            // 单个ziplist的最大大小/元素个数（对应配置list-max-listpack-size）
    int compress;        // 首尾不压缩的节点数（对应配置list-compress-depth）
    // ... 其他字段
};

struct quicklistNode {
    quicklistNode *prev; // 前一个节点
    quicklistNode *next; // 后一个节点
    unsigned char *zl;   // 指向当前节点的ziplist
    unsigned int sz;     // ziplist的字节大小
    // ... 其他字段
};
```

#### 核心优势

1. **双端操作极致性能**：头尾节点的插入 / 删除均为 O (1) 时间复杂度，继承了 linkedlist 的优势
2. **内存开销极低**：通过 ziplist 连续存储减少指针开销，相比纯 linkedlist 内存占用降低 80% 以上
3. **规避连锁更新风险**：将大 ziplist 拆分为多个小 ziplist，单个节点的修改仅影响当前节点，不会引发全列表连锁更新
4. **支持内存压缩**：通过`list-compress-depth`配置，可对中间非热点节点进行 LZF 压缩，进一步降低内存占用（适合长列表冷数据存储）
5. **高效遍历**：节点内连续内存支持高效顺序遍历，节点间双向指针支持双向遍历，兼顾范围操作性能

#### 关键配置说明

|配置项|作用|推荐值|
|---|---|---|
|`list-max-listpack-size`|控制单个 quicklistNode 内 ziplist 的最大大小，负数代表字节限制，正数代表元素个数限制|默认 - 2（单个 ziplist 最大 8KB），线上建议 - 1~-5，避免单个 ziplist 过大|
|`list-compress-depth`|列表首尾不压缩的节点数量，0 = 不压缩，1 = 首尾 1 个节点不压缩，中间节点全部压缩|长列表冷数据场景建议设为 1，热点列表建议设为 0|

## 三、核心命令大全

按功能分为 6 大类，标注时间复杂度、高频使用标记与核心注意事项，适配 Redis 6.0 + 稳定版本。

### 3.1 双端插入 / 弹出命令（高频核心）

List 的核心特色命令，所有双端操作均为**O (1) 时间复杂度**，原子性执行。

|命令|语法|作用|示例|注意事项|
|---|---|---|---|---|
|LPUSH|`LPUSH key element [element ...]`|从列表**头部（左侧）** 批量插入一个 / 多个元素，返回插入后的列表总长度|`LPUSH user:msg:1001 "订单通知" "登录提醒"`|支持批量插入，key 不存在则自动创建；插入顺序为从左到右，最终列表头部为最后一个插入的元素|
|RPUSH|`RPUSH key element [element ...]`|从列表**尾部（右侧）** 批量插入一个 / 多个元素，返回插入后的列表总长度|`RPUSH order:queue 1001 1002 1003`|最常用的队列生产命令，key 不存在则自动创建|
|LPUSHX|`LPUSHX key element [element ...]`|仅当 key**存在**时，从列表头部插入元素，返回列表总长度|`LPUSHX user:msg:1001 "活动推送"`|key 不存在时返回 0，不创建新 key|
|RPUSHX|`RPUSHX key element [element ...]`|仅当 key**存在**时，从列表尾部插入元素，返回列表总长度|`RPUSHX order:queue 1004`|常用于队列兜底校验，避免无效 key 创建|
|LPOP|`LPOP key [count]`|从列表**头部（左侧）** 弹出并返回一个 / 多个元素，count 为弹出数量|`LPOP order:queue 2`|列表为空返回`nil`；弹出多个元素时，按从左到右顺序返回|
|RPOP|`RPOP key [count]`|从列表**尾部（右侧）** 弹出并返回一个 / 多个元素，count 为弹出数量|`RPOP user:msg:1001`|最常用的队列消费命令，列表为空返回`nil`|

### 3.2 列表基础查询命令

|命令|语法|时间复杂度|作用|示例||
|---|---|---|---|---|---|
|LLEN|`LLEN key`|O(1)|获取列表的总元素个数|`LLEN order:queue`|key 不存在返回 0，非 List 类型报错|
|LINDEX|`LINDEX key index`|O (n)，n 为到 index 的遍历距离|获取列表指定下标位置的元素，下标从 0 开始|`LINDEX user:msg:1001 0`|支持负下标，`-1`代表最后一个元素；下标超出范围返回`nil`|
|LRANGE|`LRANGE key start end`|O (n)，n 为返回元素总数|获取列表指定下标范围内的所有元素，闭区间返回|`LRANGE order:queue 0 9`|高频分页查询命令；`LRANGE key 0 -1`返回全量元素，**线上严禁使用**；支持负下标|

### 3.3 列表元素修改与裁剪命令

|命令|语法|时间复杂度|作用|示例|注意事项|
|---|---|---|---|---|---|
|LSET|`LSET key index new_element`|O (n)，n 为列表总长度，头尾节点为 O (1)|将列表指定下标位置的元素替换为新值|`LSET user:msg:1001 0 "新订单通知"`|下标超出范围直接报错，key 不存在或非 List 类型也报错|
|LTRIM|`LTRIM key start end`|O (n)，n 为被裁剪的元素数量|裁剪列表，仅保留指定下标范围内的元素，其余全部删除|`LTRIM article:comment:1001 0 99`|高频配合 LPUSH 使用，实现固定长度的最新列表；支持负下标，超出范围的下标会自动修正为列表边界|
|LINSERT|`LINSERT key BEFORE/AFTER pivot element`|O (n)，n 为 pivot 到列表头部的距离|在指定基准元素 pivot 的前 / 后插入新元素，返回插入后的列表长度|`LINSERT order:queue BEFORE 1002 1005`|pivot 不存在返回 - 1，key 不存在返回 0|

### 3.5 原子性元素移动命令

核心用于实现安全队列、任务备份等场景，**全程原子性执行**，无并发竞态问题。

表格

|命令|语法|时间复杂度|作用|示例|版本说明|
|---|---|---|---|---|---|
|RPOPLPUSH|`RPOPLPUSH source_key destination_key`|O(1)|原子性操作：从 source 列表尾部弹出 1 个元素，插入到 destination 列表头部，返回该元素|`RPOPLPUSH order:queue order:backup`|Redis 6.2.0 起被`LMOVE`命令替代，兼容但不推荐新业务使用|
|LMOVE|`LMOVE source_key destination_key LEFT/RIGHT LEFT/RIGHT`|O(1)|原子性操作：从 source 列表的指定端弹出元素，插入到 destination 列表的指定端，返回该元素|`LMOVE order:queue order:backup RIGHT LEFT`|完全兼容 RPOPLPUSH，支持自定义弹出 / 插入端，灵活性更高|

### 3.6 阻塞式弹出命令（队列核心）

List 最具特色的命令，是实现阻塞队列、消息消费的核心，**无元素时阻塞客户端，有元素时立即唤醒返回**，全程原子性。

表格

|命令|语法|时间复杂度|作用|示例|核心特性|
|---|---|---|---|---|---|
|BLPOP|`BLPOP key [key ...] timeout`|O(1)|阻塞式从列表头部弹出元素，支持监听多个 key，timeout 为阻塞超时时间（秒）|`BLPOP order:queue msg:queue 0`|1. 按 key 顺序检查，返回第一个有元素的列表的弹出值<br><br>2. timeout=0 代表永久阻塞，直到有元素可弹出<br><br>3. 多个客户端同时监听，元素到达时按阻塞顺序唤醒（先阻塞先消费）|
|BRPOP|`BRPOP key [key ...] timeout`|O(1)|阻塞式从列表尾部弹出元素，其余特性与 BLPOP 完全一致|`BRPOP order:queue 30`|最常用的阻塞队列消费命令，超时时间建议设置合理值，避免永久阻塞|
|BRPOPLPUSH|`BRPOPLPUSH source destination timeout`|O(1)|阻塞版 RPOPLPUSH，source 无元素时阻塞，有元素时原子执行弹出 + 插入，返回元素|`BRPOPLPUSH order:queue order:backup 30`|Redis 6.2.0 起被`BLMOVE`替代|
|BLMOVE|`BLMOVE source destination LEFT/RIGHT LEFT/RIGHT timeout`|O(1)|阻塞版 LMOVE，支持自定义弹出 / 插入端，是安全阻塞队列的最优实现|`BLMOVE order:queue order:backup RIGHT LEFT 30`|完美解决消费者宕机导致的消息丢失问题，消费成功后再删除备份队列的元素|

## 四、典型应用场景

### 4.1 消息队列（核心场景）

基于 List 的 FIFO（先进先出）特性，实现轻量级消息队列，支持普通消费与阻塞消费，是中小流量场景的最优解。

- 普通队列实现：
    
    1. 生产者：`RPUSH order:queue 1001 1002 1003`（尾部批量插入订单消息）
    2. 消费者：`LPOP order:queue`（头部弹出消费）
    
- 阻塞队列优化（避免消费者轮询浪费 CPU）：
    
    消费者：`BRPOP order:queue 30`（无消息时阻塞 30 秒，有消息立即唤醒消费）
- 安全队列实现（解决消费者宕机消息丢失）：
    
    1. 消费者：`BLMOVE order:queue order:backup:1001 RIGHT LEFT 30`（原子性将消息移到备份队列）
    2. 消费成功后：`LREM order:backup:1001 1 1001`（删除备份消息）
    3. 定时任务：扫描备份队列，将超时未消费的消息重新放回主队列，实现消息重试
    

### 4.2 栈结构实现

基于 List 的 LIFO（后进先出）特性，一行命令实现栈结构，常用于任务回溯、操作记录等场景。

- 入栈：`LPUSH operate:stack "新增用户" "修改订单"`
- 出栈：`LPOP operate:stack`

### 4.3 最新列表 / 动态流

基于`LPUSH + LTRIM`组合，实现固定长度的最新列表，永远保留最新插入的 N 条数据，内存占用固定，性能极高。

- 典型场景：朋友圈动态、商品评论、用户操作日志、文章最新回复
- 实现示例：
    
    1. 用户发布评论：`LPUSH article:comment:1001 '{"user_id":1002,"content":"好评","time":"2026-03-08"}'`
    2. 裁剪列表，仅保留最新 100 条评论：`LTRIM article:comment:1001 0 99`
    3. 分页查询最新评论：`LRANGE article:comment:1001 0 9`（查询第 1 页，每页 10 条）
    

### 4.4 分布式任务队列 / 延时任务

基于阻塞命令与双端操作，实现分布式环境下的任务调度，支持多消费者并发消费，无重复消费问题。

- 典型场景：订单超时关闭、异步邮件发送、数据同步任务
- 实现示例：
    
    1. 任务生产：`RPUSH task:email:queue '{"email":"xxx@xx.com","content":"验证码"}'`
    2. 多消费者并发消费：`BRPOP task:email:queue 0`（多个消费者同时监听，同一任务只会被一个消费者获取）
    

### 4.5 排行榜 / 热门列表

基于 List 的有序特性，配合`LINSERT`与`LRANGE`，实现简单的排行榜功能，适合数据量不大、更新频率低的场景。

- 典型场景：商品销量榜、直播间人气榜、热门搜索词
- 实现示例：
    
    1. 初始化榜单：`RPUSH hot:search "Java" "Redis" "Python"`
    2. 调整榜单顺序：`LINSERT hot:search BEFORE "Redis" "MySQL"`
    3. 查询 TOP3 热词：`LRANGE hot:search 0 2`
    

### 4.6 循环列表 / 轮询调度

基于`RPOPLPUSH/LMOVE`命令，实现循环列表，元素弹出后自动回到列表头部，循环往复，适合轮询调度、负载均衡场景。

- 典型场景：服务节点轮询、定时任务轮询执行、资源分配
- 实现示例：
    
    1. 初始化服务节点：`RPUSH server:node "192.168.1.1" "192.168.1.2" "192.168.1.3"`
    2. 轮询获取节点：`LMOVE server:node server:node RIGHT LEFT`（原子性将尾部节点移到头部，返回该节点，实现循环轮询）

## 五、常见坑与最佳实践

1. **严禁全量遍历长列表**
    
    线上绝对禁止使用`LRANGE key 0 -1`全量获取长列表元素，长列表全量遍历会导致 Redis 主线程阻塞、网络 IO 飙升。分页查询必须控制单次`LRANGE`的范围，建议单次返回元素不超过 100 个。
    
2. **严格控制 List 长度与元素大小**
    
    线上单个 List 的元素总数建议控制在**1 万以内**，单个元素字节大小控制在**1KB 以内**。超长列表会导致内存占用过高、范围操作性能急剧下降、主从同步延迟，超大列表建议按业务维度拆分（如按时间、按用户 ID 分 key）。
    
3. **慎用 O (n) 复杂度的命令**
    
    线上长列表场景尽量避免使用`LINDEX`、`LINSERT`、`LREM`、`LSET`（非头尾节点）等 O (n) 复杂度的命令，尤其是`LREM key 0 element`全量匹配删除，长列表会直接阻塞 Redis 主线程。如需频繁修改中间元素，建议更换 Hash 或 Sorted Set 类型。
    
4. **阻塞命令必须设置合理的超时时间**
    
    使用`BLPOP/BRPOP/BLMOVE`等阻塞命令时，严禁无脑设置`timeout=0`永久阻塞。永久阻塞会导致客户端连接闲置占用，Redis 服务端连接数溢出，同时网络波动时会出现客户端假死。建议超时时间设置为 30~60 秒，超时后重新发起监听即可。
    
5. **避免多个消费者监听同一个非阻塞队列**
    
    普通`LPOP/RPOP`无排他性，多个消费者同时消费同一个队列时，会出现重复消费问题。多消费者场景必须使用阻塞命令，或通过分布式锁控制，推荐使用`BLPOP/BRPOP`，Redis 原生保证同一消息只会被一个消费者获取。
    
6. **禁止给单个元素设置过期时间**
    
    Redis 的过期时间是**key 级别**的，无法给 List 中的单个元素设置过期时间。如需实现元素级别的过期，建议配合 Sorted Set（按过期时间排序），或拆分 key 按时间维度设置过期时间。
    
7. **批量插入优先使用 LPUSH/RPUSH 批量语法**
    
    多次循环执行`LPUSH key element`，会产生大量网络 RTT，性能极差。批量插入元素时，直接使用`LPUSH key element1 element2 element3...`批量语法，单次请求完成插入，性能提升 10 倍以上，建议单次批量插入元素不超过 100 个。
    
8. **长列表开启节点压缩优化内存**
    
    对于存储冷数据的长列表，建议调整`list-compress-depth=1`配置，对中间非热点节点进行压缩，可降低 50% 以上的内存占用，同时不影响头尾节点的双端操作性能。

### 3.3 列表元素修改与裁剪命令

表格

|命令|语法|时间复杂度|作用|示例|注意事项|

|LSET|`LSET key index new_element`|O (n)，n 为列表总长度，头尾节点为 O (1)|将列表指定下标位置的元素替换为新值|`LSET user:msg:1001 0 "新订单通知"`|下标超出范围直接报错，key 不存在或非 List 类型也报错|

### 3.5 原子性元素移动命令

核心用于实现安全队列、任务备份等场景，**全程原子性执行**，无并发竞态问题。

表格

|命令|语法|时间复杂度|作用|示例|版本说明|
|---|---|---|---|---|---|
|RPOPLPUSH|`RPOPLPUSH source_key destination_key`|O(1)|原子性操作：从 source 列表尾部弹出 1 个元素，插入到 destination 列表头部，返回该元素|`RPOPLPUSH order:queue order:backup`|Redis 6.2.0 起被`LMOVE`命令替代，兼容但不推荐新业务使用|

### 3.6 阻塞式弹出命令（队列核心）

List 最具特色的命令，是实现阻塞队列、消息消费的核心，**无元素时阻塞客户端，有元素时立即唤醒返回**，全程原子性。

表格

|命令|语法|时间复杂度|作用|示例|核心特性|
|---|---|---|---|---|---|
|BLPOP|`BLPOP key [key ...] timeout`|O(1)|阻塞式从列表头部弹出元素，支持监听多个 key，timeout 为阻塞超时时间（秒）|`BLPOP order:queue msg:queue 0`|1. 按 key 顺序检查，返回第一个有元素的列表的弹出值<br><br>2. timeout=0 代表永久阻塞，直到有元素可弹出<br><br>3. 多个客户端同时监听，元素到达时按阻塞顺序唤醒（先阻塞先消费）|
|BRPOP|`BRPOP key [key ...] timeout`|O(1)|阻塞式从列表尾部弹出元素，其余特性与 BLPOP 完全一致|`BRPOP order:queue 30`|最常用的阻塞队列消费命令，超时时间建议设置合理值，避免永久阻塞|
|BRPOPLPUSH|`BRPOPLPUSH source destination timeout`|O(1)|阻塞版 RPOPLPUSH，source 无元素时阻塞，有元素时原子执行弹出 + 插入，返回元素|`BRPOPLPUSH order:queue order:backup 30`|Redis 6.2.0 起被`BLMOVE`替代|
|BLMOVE|`BLMOVE source destination LEFT/RIGHT LEFT/RIGHT timeout`|O(1)|阻塞版 LMOVE，支持自定义弹出 / 插入端，是安全阻塞队列的最优实现|`BLMOVE order:queue order:backup RIGHT LEFT 30`|完美解决消费者宕机导致的消息丢失问题，消费成功后再删除备份队列的元素|

## 四、典型应用场景

### 4.1 消息队列（核心场景）

基于 List 的 FIFO（先进先出）特性，实现轻量级消息队列，支持普通消费与阻塞消费，是中小流量场景的最优解。

- 普通队列实现：
    
    1. 生产者：`RPUSH order:queue 1001 1002 1003`（尾部批量插入订单消息）
    2. 消费者：`LPOP order:queue`（头部弹出消费）
    
- 阻塞队列优化（避免消费者轮询浪费 CPU）：
    
    消费者：`BRPOP order:queue 30`（无消息时阻塞 30 秒，有消息立即唤醒消费）
- 安全队列实现（解决消费者宕机消息丢失）：
    
    1. 消费者：`BLMOVE order:queue order:backup:1001 RIGHT LEFT 30`（原子性将消息移到备份队列）
    2. 消费成功后：`LREM order:backup:1001 1 1001`（删除备份消息）
    3. 定时任务：扫描备份队列，将超时未消费的消息重新放回主队列，实现消息重试
    

### 4.2 栈结构实现

基于 List 的 LIFO（后进先出）特性，一行命令实现栈结构，常用于任务回溯、操作记录等场景。

- 入栈：`LPUSH operate:stack "新增用户" "修改订单"`
- 出栈：`LPOP operate:stack`

### 4.3 最新列表 / 动态流

基于`LPUSH + LTRIM`组合，实现固定长度的最新列表，永远保留最新插入的 N 条数据，内存占用固定，性能极高。

- 典型场景：朋友圈动态、商品评论、用户操作日志、文章最新回复
- 实现示例：
    
    1. 用户发布评论：`LPUSH article:comment:1001 '{"user_id":1002,"content":"好评","time":"2026-03-08"}'`
    2. 裁剪列表，仅保留最新 100 条评论：`LTRIM article:comment:1001 0 99`
    3. 分页查询最新评论：`LRANGE article:comment:1001 0 9`（查询第 1 页，每页 10 条）
    

### 4.4 分布式任务队列 / 延时任务

基于阻塞命令与双端操作，实现分布式环境下的任务调度，支持多消费者并发消费，无重复消费问题。

- 典型场景：订单超时关闭、异步邮件发送、数据同步任务
- 实现示例：
    
    1. 任务生产：`RPUSH task:email:queue '{"email":"xxx@xx.com","content":"验证码"}'`
    2. 多消费者并发消费：`BRPOP task:email:queue 0`（多个消费者同时监听，同一任务只会被一个消费者获取）
    

### 4.5 排行榜 / 热门列表

基于 List 的有序特性，配合`LINSERT`与`LRANGE`，实现简单的排行榜功能，适合数据量不大、更新频率低的场景。

- 典型场景：商品销量榜、直播间人气榜、热门搜索词
- 实现示例：
    
    1. 初始化榜单：`RPUSH hot:search "Java" "Redis" "Python"`
    2. 调整榜单顺序：`LINSERT hot:search BEFORE "Redis" "MySQL"`
    3. 查询 TOP3 热词：`LRANGE hot:search 0 2`
    

### 4.6 循环列表 / 轮询调度

基于`RPOPLPUSH/LMOVE`命令，实现循环列表，元素弹出后自动回到列表头部，循环往复，适合轮询调度、负载均衡场景。

- 典型场景：服务节点轮询、定时任务轮询执行、资源分配
- 实现示例：
    
    1. 初始化服务节点：`RPUSH server:node "192.168.1.1" "192.168.1.2" "192.168.1.3"`
    2. 轮询获取节点：`LMOVE server:node server:node RIGHT LEFT`（原子性将尾部节点移到头部，返回该节点，实现循环轮询）

## 五、常见坑与最佳实践

1. **严禁全量遍历长列表**
    
    线上绝对禁止使用`LRANGE key 0 -1`全量获取长列表元素，长列表全量遍历会导致 Redis 主线程阻塞、网络 IO 飙升。分页查询必须控制单次`LRANGE`的范围，建议单次返回元素不超过 100 个。
    
2. **严格控制 List 长度与元素大小**
    
    线上单个 List 的元素总数建议控制在**1 万以内**，单个元素字节大小控制在**1KB 以内**。超长列表会导致内存占用过高、范围操作性能急剧下降、主从同步延迟，超大列表建议按业务维度拆分（如按时间、按用户 ID 分 key）。
    
3. **慎用 O (n) 复杂度的命令**
    
    线上长列表场景尽量避免使用`LINDEX`、`LINSERT`、`LREM`、`LSET`（非头尾节点）等 O (n) 复杂度的命令，尤其是`LREM key 0 element`全量匹配删除，长列表会直接阻塞 Redis 主线程。如需频繁修改中间元素，建议更换 Hash 或 Sorted Set 类型。
    
4. **阻塞命令必须设置合理的超时时间**
    
    使用`BLPOP/BRPOP/BLMOVE`等阻塞命令时，严禁无脑设置`timeout=0`永久阻塞。永久阻塞会导致客户端连接闲置占用，Redis 服务端连接数溢出，同时网络波动时会出现客户端假死。建议超时时间设置为 30~60 秒，超时后重新发起监听即可。
    
5. **避免多个消费者监听同一个非阻塞队列**
    
    普通`LPOP/RPOP`无排他性，多个消费者同时消费同一个队列时，会出现重复消费问题。多消费者场景必须使用阻塞命令，或通过分布式锁控制，推荐使用`BLPOP/BRPOP`，Redis 原生保证同一消息只会被一个消费者获取。
    
6. **禁止给单个元素设置过期时间**
    
    Redis 的过期时间是**key 级别**的，无法给 List 中的单个元素设置过期时间。如需实现元素级别的过期，建议配合 Sorted Set（按过期时间排序），或拆分 key 按时间维度设置过期时间。
    
7. **批量插入优先使用 LPUSH/RPUSH 批量语法**
    
    多次循环执行`LPUSH key element`，会产生大量网络 RTT，性能极差。批量插入元素时，直接使用`LPUSH key element1 element2 element3...`批量语法，单次请求完成插入，性能提升 10 倍以上，建议单次批量插入元素不超过 100 个。
    
8. **长列表开启节点压缩优化内存**
    
    对于存储冷数据的长列表，建议调整`list-compress-depth=1`配置，对中间非热点节点进行压缩，可降低 50% 以上的内存占用，同时不影响头尾节点的双端操作性能。

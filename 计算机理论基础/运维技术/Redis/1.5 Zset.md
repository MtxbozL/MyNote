## 一、核心概述

Zset（Sorted Set，有序集合）是 Redis 中**唯一同时支持元素唯一性、权重排序、范围查询**的核心数据类型。每个元素由两部分组成：

- `member`（成员）：String 类型，**不可重复**，是元素的唯一标识
- `score`（分值）：64 位双精度浮点数，是排序的核心依据，支持重复

核心特性：

- 排序规则：默认按`score`升序排列，score 相同时按`member`字典序排序
- 性能优势：单元素操作 O (1)，排序 / 范围查询平均 O (logN)，性能不随元素数量线性衰减
- 原子性保证：所有操作均为原子性，基于 Redis 单线程模型，并发场景无竞态问题
- 容量限制：单 key 最大支持`2^32 - 1`个元素（约 42 亿），单个 member 最大容量 512MB
- 差异化核心价值：同时具备 Set 的去重能力、List 的有序能力，且支持按权重灵活排序与范围筛选，是排行榜、延时队列等场景的最优解

## 二、底层实现原理

Redis Zset 会根据元素数量、单个 member 的大小，自动选择最优编码格式，编码转换**不可逆**。Redis 7.0 + 用`listpack`替代了旧版本的`ziplist`，核心逻辑与配置规则保持兼容。

### 2.1 紧凑型编码：ziplist/listpack

小体量有序集合的默认编码，一段连续内存块，无额外指针开销，极致优化内存占用。

#### 触发条件（同时满足）

1. 集合元素总数 ≤ `zset-max-ziplist-entries`（默认 128）
2. 每个 member 的字节长度 ≤ `zset-max-ziplist-value`（默认 64 字节）

#### 存储结构

ziplist/listpack 中，Zset 的`member`和`score`**成对连续存储**，且整体按`score`从小到大升序排列。例如执行`ZADD rank 10 a 20 b`，存储结构为`[a, 10, b, 20]`，无冗余元数据开销。

#### 核心优劣势

- 优势：连续内存存储，内存占用比跳表编码低 70% 以上，缓存命中率高，无内存碎片
- 缺陷：插入 / 删除 / 查找需遍历内存，时间复杂度 O (n)，元素过多时性能急剧下降，ziplist 存在连锁更新风险（listpack 已彻底解决）

### 2.2 高性能编码：跳表 (skiplist)+ 哈希表 (hashtable) 双结构

当 Zset 不满足 ziplist/listpack 触发条件时，会自动不可逆地转为双结构编码，这是 Zset 的核心高性能实现，完美平衡了单元素查询与范围排序的性能需求。

#### 2.2.1 双结构设计的核心原因

双结构共享`member`和`score`数据，无冗余存储，各自解决不同场景的性能需求：

1. **hashtable（哈希表）**：以`member`为 key，`score`为 value，保证**O (1) 时间复杂度的单 member 查询**（如 ZSCORE、ZISMEMBER、ZINCRBY）
2. **skiplist（跳表）**：按`score`有序存储所有`member`，保证**O (logN) 时间复杂度的排序、范围查询、排名计算**（如 ZRANGE、ZRANK）

#### 2.2.2 跳表 (skiplist) 核心结构与原理

跳表是一种多层有序链表结构，是平衡树的轻量化替代方案，Redis 跳表的核心设计如下：

1. **分层结构**：
    
    - 最底层（Level 0）是包含全量元素的**双向有序链表**，按`score`升序排列，支持正序 / 倒序双向遍历
    - 上层是下层的稀疏索引层，每一层都是有序链表，节点数量随层数升高指数级减少
    - Redis 跳表默认最大层数为 32 层，新节点的层数通过随机算法生成（每层晋升概率 1/4）
    
2. **核心结构简化版**

```c
    // 跳表节点结构
    typedef struct zskiplistNode {
        sds ele; // 存储member
        double score; // 存储score
        struct zskiplistNode *backward; // 后退指针，指向底层链表的前一个节点
        // 层结构数组，每个元素对应一层
        struct zskiplistLevel {
            struct zskiplistNode *forward; // 前进指针，指向当前层的下一个节点
            unsigned long span; // 跨度，记录当前节点到forward指向节点的元素个数，用于快速计算排名
        } level[];
    } zskiplistNode;
    
    // 跳表整体结构
    typedef struct zskiplist {
        struct zskiplistNode *header, *tail; // 头尾节点
        unsigned long length; // 元素总个数
        int level; // 当前跳表的最大层数
    } zskiplist;
    
    // Zset最终结构（双结构组合）
    typedef struct zset {
        dict *dict; // 哈希表，member到score的映射
        zskiplist *zsl; // 跳表，有序存储元素
    } zset;
```
    
3. **核心执行逻辑**
    
    - 查找：从最高层开始遍历，若当前节点的 forward 指针指向的节点 score 大于目标值，则向下一层，直到底层链表，时间复杂度平均 O (logN)，最坏 O (n)
    - 排名计算：通过每层的 span 跨度累加，直接计算出元素的排名，无需遍历全表，O (logN) 时间复杂度
    - 插入 / 删除：先通过哈希表判断元素是否存在，再通过跳表找到对应位置，修改链表指针，无需移动大量内存，O (logN) 时间复杂度
    

#### 2.2.3 为什么 Redis 选择跳表，而非平衡树（B + 树 / 红黑树）？

这是面试最高频考点，核心原因如下：

1. **范围查询更友好**：跳表的底层双向链表可直接遍历范围数据，无需平衡树的中序遍历，实现更简单，性能更稳定
2. **实现与维护成本极低**：跳表代码实现远简单于平衡树，无复杂的旋转、变色、节点分裂合并逻辑，bug 率更低，更易适配 Redis 的单线程模型
3. **内存开销可控**：跳表的层数可通过晋升概率灵活调整，平均每个节点仅 1.33 个指针，内存开销低于红黑树
4. **并发场景更友好**：跳表的修改仅影响相邻节点，锁的粒度远小于平衡树，更适合并发读写场景（Redis 虽为单线程，但持久化、主从同步场景下有优势）
5. **排名计算原生支持**：跳表的 span 跨度字段可直接实现 O (logN) 的排名计算，平衡树需要额外存储元数据，实现复杂

#### 2.2.4 核心优劣势

- 优势：海量元素下性能稳定，增删改查、范围查询、排名计算均为 O (logN)，无性能衰减，支持灵活的正序 / 倒序操作
- 缺陷：内存开销高于 ziplist/listpack，有指针与索引层的额外开销

## 三、核心命令大全

按功能分为 7 大类，标注时间复杂度、高频标记、示例与核心注意事项，适配 Redis 6.0 + 稳定版本，标注 Redis 6.2 + 新增的整合语法。

### 3.1 基础增删改查命令（高频核心）

|命令|语法|时间复杂度|核心作用|示例|关键注意事项|
|---|---|---|---|---|---|
|ZADD|`ZADD key [NX/XX] [CH] [INCR] score member [score member ...]`|O (logN)，N 为集合元素总数，批量添加为 O (M*logN)，M 为本次添加元素数|向有序集合添加一个 / 多个元素，member 已存在则更新 score|`ZADD article:like 100 1001 200 1002`|核心参数：<br><br>- `NX`：仅 member 不存在时才添加，不更新已存在元素<br><br>- `XX`：仅 member 存在时才更新 score，不新增元素<br><br>- `CH`：返回本次发生变化的元素总数（新增 + 更新）<br><br>- `INCR`：对 member 的 score 执行增量操作，等价于 ZINCRBY<br><br>Redis 4.0 + 支持以上参数|
|ZINCRBY|`ZINCRBY key 步长 member`|O(logN)|对指定 member 的 score 原子性增减指定步长，member 不存在则从 0 开始创建，返回更新后的 score|`ZINCRBY article:like 1 1001`|步长支持负数，实现 score 递减；原子性保证，高并发点赞 / 投票场景无竞态问题|
|ZREM|`ZREM key member [member ...]`|O (M*logN)，M 为本次删除元素数|删除有序集合中一个 / 多个 member，返回实际删除的元素数量|`ZREM article:like 1001`|不存在的 member 自动忽略；删除最后一个元素后，key 自动删除|
|ZSCORE|`ZSCORE key member`|O(1)|获取指定 member 对应的 score 值|`ZSCORE article:like 1001`|member/key 不存在返回`nil`；基于哈希表实现，性能极高|
|ZMSCORE|`ZMSCORE key member [member ...]`|O (M)，M 为本次查询元素数|批量获取多个 member 的 score 值，按输入顺序返回|`ZMSCORE article:like 1001 1002 1003`|Redis 6.2.0 + 新增，替代多次循环 ZSCORE，大幅减少网络 RTT|

### 3.2 排名查询命令（高频核心）

|命令|语法|时间复杂度|核心作用|示例|核心区别|
|---|---|---|---|---|---|
|ZRANK|`ZRANK key member [WITHSCORE]`|O(logN)|获取指定 member 的**正序排名**（score 从小到大），排名从 0 开始|`ZRANK article:like 1001`|score 越小，排名越靠前；member 不存在返回`nil`|
|ZREVRANK|`ZREVRANK key member [WITHSCORE]`|O(logN)|获取指定 member 的**倒序排名**（score 从大到小），排名从 0 开始|`ZREVRANK article:like 1001`|score 越大，排名越靠前，是排行榜场景的核心命令；Redis 7.0 + 支持 WITHSCORE 参数|

### 3.3 范围查询命令（核心特色，最高频）

Redis 6.2.0 起，`ZRANGE`命令整合了所有范围查询能力，支持 BYRANK、BYSCORE、BYLEX 三种模式，替代了旧版本的 ZRANGEBYSCORE、ZRANGEBYLEX 等命令，新业务优先使用 ZRANGE。

|命令|语法|时间复杂度|核心作用|示例|关键注意事项|
|---|---|---|---|---|---|
|ZRANGE|`ZRANGE key start end [BYSCORE/BYLEX] [REV] [LIMIT offset count] [WITHSCORES]`|O (logN + M)，M 为返回的元素数|按排名 /score/ 字典序获取指定范围的元素，支持正序 / 倒序|`ZRANGE article:like 0 9 REV WITHSCORES`|核心参数：<br><br>- 默认按排名查询，start/end 为排名下标，支持负下标（-1 为最后一个元素）<br><br>- `BYSCORE`：按 score 范围查询，start/end 为 score 值，`-inf`负无穷，`+inf`正无穷<br><br>- `BYLEX`：按 member 字典序查询，仅当所有元素 score 完全相同时使用<br><br>- `REV`：倒序查询（从大到小）<br><br>- `LIMIT`：限制返回的元素数量，分页查询必备<br><br>- `WITHSCORES`：同时返回元素对应的 score|
|ZREVRANGE|`ZREVRANGE key start end [WITHSCORES]`|O(logN + M)|按倒序排名获取指定范围的元素，等价于`ZRANGE key start end REV`|`ZREVRANGE article:like 0 9 WITHSCORES`|排行榜 TOP N 查询的经典命令，Redis 6.2 + 推荐用 ZRANGE+REV 替代|
|ZRANGEBYSCORE|`ZRANGEBYSCORE key min max [WITHSCORES] [LIMIT offset count]`|O(logN + M)|按 score 升序获取 min 到 max 范围内的元素，等价于`ZRANGE key min max BYSCORE`|`ZRANGEBYSCORE article:like 100 200 WITHSCORES`|支持开区间：`(100`代表大于 100，`(200`代表小于 200；Redis 6.2 + 已被 ZRANGE 整合，兼容但不推荐新业务使用|
|ZREVRANGEBYSCORE|`ZREVRANGEBYSCORE key max min [WITHSCORES] [LIMIT offset count]`|O(logN + M)|按 score 倒序获取 max 到 min 范围内的元素，等价于`ZRANGE key max min BYSCORE REV`|`ZREVRANGEBYSCORE article:like 200 100 WITHSCORES`|注意参数顺序是 max 在前，min 在后，极易写反；Redis 6.2 + 已被 ZRANGE 整合|

### 3.5 集合运算命令（带权重特色）

Zset 原生支持交 / 并 / 差集运算，且支持权重设置与聚合规则，是带权重标签筛选、用户分群的核心能力，所有运算均为原子性执行。

|命令|语法|时间复杂度|核心作用|核心参数|
|---|---|---|---|---|
|ZINTERSTORE|`ZINTERSTORE destination numkeys key [key ...] [WEIGHTS weight [weight ...]] [AGGREGATE SUM/MIN/MAX]`|O(N_K) + O(M_logM)，N 为最小集合元素数，K 为集合数，M 为交集元素数|计算多个集合的交集，将结果存入 destination 集合，返回交集元素总数|- `WEIGHTS`：为每个输入集合设置权重，运算时每个元素的 score 会乘以对应权重<br><br>- `AGGREGATE`：结果元素的 score 聚合规则，默认 SUM（求和），可选 MIN（取最小值）、MAX（取最大值）|
|ZUNIONSTORE|`ZUNIONSTORE destination numkeys key [key ...] [WEIGHTS weight [weight ...]] [AGGREGATE SUM/MIN/MAX]`|O (N) + O (M*logM)，N 为所有集合元素总数，M 为并集元素数|计算多个集合的并集，将结果存入 destination 集合，返回并集元素总数|参数与 ZINTERSTORE 完全一致|
|ZINTER/ZUNION/ZDIFF|`ZINTER numkeys key [key ...] [WITHSCORES] [WEIGHTS ...] [AGGREGATE ...]`|与对应 STORE 命令一致|计算多个集合的交 / 并 / 差集，直接返回结果，不存入 Redis|Redis 6.2.0 + 新增，适合结果集小、无需复用的场景|
|ZDIFFSTORE|`ZDIFFSTORE destination numkeys key [key ...]`|O (L)，L 为第一个集合的元素数|计算第一个集合与其他所有集合的差集，将结果存入 destination，返回差集元素数|严格遵循顺序，`ZDIFF A B`返回 A 有但 B 没有的元素；Redis 6.2.0 + 新增|

### 3.6 随机操作与安全遍历命令

|命令|语法|时间复杂度|核心作用|注意事项|
|---|---|---|---|---|
|ZRANDMEMBER|`ZRANDMEMBER key [count [WITHSCORES]]`|O (N)，N 为返回的元素数|随机获取集合中 count 个元素，不删除元素|Redis 6.2.0 + 新增；count 正数返回不重复元素，负数返回允许重复的元素|
|ZSCAN|`ZSCAN key cursor [MATCH pattern] [COUNT count]`|单次调用 O (1)，全量遍历 O (N)|增量式迭代遍历有序集合，每次仅返回少量数据，无主线程阻塞风险|线上大集合遍历的唯一合规方案，**绝对禁止对大集合使用 ZRANGE 0 -1 全量遍历**；示例：`ZSCAN user:rank 0 MATCH *10* COUNT 100`，循环迭代直到 cursor 返回 0|

## 四、典型应用场景

### 4.1 各类排行榜系统（核心经典场景）

Zset 是实现排行榜的最优方案，完美支持按权重排序、TOP N 查询、排名计算、增量更新，适配所有排行榜场景。

- 典型场景：文章点赞 / 阅读量排行、商品销量榜、游戏战力榜、直播间人气榜、热搜榜
- 实现示例：

```bash
    # 1. 初始化文章点赞数据
    ZADD article:like 150 1001 230 1002 89 1003 310 1004
    # 2. 用户点赞，原子性+1
    ZINCRBY article:like 1 1001
    # 3. 获取点赞量TOP10的文章（倒序，score越大排名越靠前）
    ZRANGE article:like 0 9 REV WITHSCORES
    # 4. 获取文章1001的点赞排名（第1名排名为0）
    ZREVRANK article:like 1001
    # 5. 获取点赞量100~300之间的文章
    ZRANGE article:like 100 300 BYSCORE WITHSCORES
    # 6. 仅保留点赞TOP100的文章，删除其余数据
    ZREMRANGEBYRANK article:like 100 -1
```

### 4.2 延时队列 / 定时任务

基于 Zset 实现轻量级延时队列，无专业 MQ 的部署成本，支持分布式环境，是中小流量延时任务的最优解。

- 核心原理：将**任务执行的时间戳作为 score**，任务唯一标识作为 member，定时扫描 score 小于当前时间戳的元素（到期任务），原子性取出并执行
- 实现示例：

```bash
    # 1. 生产延时任务：订单30分钟后自动关闭，时间戳=当前时间+30分钟
    ZADD order:delay 1741441800 order:20260308001
    # 2. 消费端定时拉取到期任务：原子性获取并删除1条到期任务
    # 方案1：先查询再删除（非原子，适合低并发）
    ZRANGE order:delay 0 1741440000 BYSCORE LIMIT 0 1
    ZREM order:delay order:20260308001
    # 方案2：Lua脚本实现原子性查询+删除（高并发场景必备，无重复消费）
    # 方案3：Redis 6.2+用ZRANGE + DELETE参数（原子性）
    ZRANGE order:delay 0 1741440000 BYSCORE LIMIT 0 1 WITHSCORES DELETE
```

### 4.3 滑动窗口限流 / 统计

基于 Zset 实现分布式滑动窗口限流，精度高、灵活性强，是接口限流、UV 统计的经典方案。

- 核心原理：将**请求时间戳作为 score**，用户唯一标识 + 时间戳作为 member，统计当前时间窗口内的请求数，超过阈值则限流，同时删除窗口外的过期数据
- 实现示例：限制用户 1 分钟内最多调用接口 100 次

```bash
    # 1. 用户请求时，添加请求记录
    ZADD api:limit:user:1001 1741440001 1741440001:user:1001
    # 2. 删除1分钟前的过期数据
    ZREMRANGEBYSCORE api:limit:user:1001 0 1741439940
    # 3. 统计当前窗口内的请求数
    ZCOUNT api:limit:user:1001 1741439940 1741440001
    # 4. 若请求数>100，触发限流；否则允许请求
```
    

### 4.4 带权重的标签筛选与用户分群

基于 Zset 的带权重集合运算，实现多标签的组合筛选，支持标签权重设置，是用户画像、精准推送的核心方案。

- 实现示例：筛选同时有「数码」「游戏」标签，且按标签权重总和排序的用户

```bash
    # 1. 存储用户标签权重
    ZADD tag:数码 10 user:1001 8 user:1002 5 user:1003
    ZADD tag:游戏 9 user:1001 7 user:1003 6 user:1004
    # 2. 计算两个标签的交集，score求和，存入目标集合
    ZINTERSTORE target:user 2 tag:数码 tag:游戏 AGGREGATE SUM
    # 3. 按权重倒序获取TOP100目标用户
    ZRANGE target:user 0 99 REV WITHSCORES
```

### 4.5 优先级队列

基于 Zset 的 score 排序能力，实现分布式优先级队列，score 越高优先级越高，优先被消费。

- 典型场景：订单分级处理、消息优先级分发、任务调度
- 实现示例：

```bash
    # 1. 生产任务：score为优先级，数字越大优先级越高
    ZADD task:priority 10 "普通订单" 100 "VIP订单" 50 "企业订单"
    # 2. 消费端按优先级从高到低获取任务
    ZRANGE task:priority 0 0 REV WITHSCORES
```

### 4.6 地理位置范围查询

Redis GEO 地理位置类型的底层就是 Zset，其核心原理是将地理位置的经纬度转换为 52 位的 Geohash 整数作为 score，member 为地理位置 ID，通过 Zset 的范围查询实现附近的人、地理位置范围筛选功能。

## 五、常见坑与最佳实践

1. **绝对禁止对大 Zset 执行全量范围查询**
    
    线上严禁使用`ZRANGE key 0 -1 WITHSCORES`全量遍历大 Zset，时间复杂度 O (N)，若集合有上万个元素，会直接阻塞 Redis 主线程，引发实例雪崩。
    
    - 线上规范：仅当元素数量明确小于 100 时可全量查询；大集合遍历必须使用 ZSCAN 增量迭代，分页查询必须使用 LIMIT 参数限制单次返回元素数量，建议单次返回不超过 100 个。
    
2. **score 浮点数精度丢失问题**
    
    score 是 64 位双精度浮点数，**仅能精确表示 - 2^53 ~ 2^53 范围内的整数**，超出该范围的超大整数（如雪花 ID、19 位订单号）会出现精度丢失，导致 member 排序异常、查询失败。
    
    - 避坑方案：
        
        1. 不要用超大整数作为 score，时间戳、优先级等常规数值无此问题
        2. 必须用超大唯一标识时，将其作为 member，而非 score
        3. 超大数值可拆分为两部分，避免超出精度范围
        
    
3. **大集合的范围查询与集合运算性能优化**
    
    大 Zset（10 万 + 元素）的大范围查询、交并差运算会消耗大量 CPU，阻塞 Redis 主线程，是线上高频故障点。
    
    - 优化规范：
        
        1. 范围查询必须加 LIMIT 限制，避免返回大量元素
        2. 集合运算优先用最小的集合作为基准，减少计算量
        3. 用 STORE 系列命令将运算结果存入 Redis，避免重复计算
        4. 超大集合的运算必须放在从节点执行，绝对禁止在主节点执行
        5. 单次运算的集合数量建议不超过 3 个
        
    
4. **score 相同时的排序规则避坑**
    
    当多个元素的 score 相同时，Redis 会按**member 的字典序**排序，而非插入顺序。很多业务场景会忽略这个规则，导致排名结果不符合预期。
    
    - 避坑方案：若需要按插入顺序排序，可在 score 中拼接时间戳后缀，保证 score 唯一且按插入顺序排序。
    
5. **不支持 member 级别的过期时间**
    
    Redis 的过期时间仅作用于整个 Zset 顶级 key，无法给单个 member 设置过期时间，这是最高频的业务坑。
    
    - 合规解决方案：
        
        1. 方案 1：定时任务扫描，用 ZREMRANGEBYSCORE 删除过期的元素（延时队列 / 滑动窗口场景通用）
        2. 方案 2：按时间维度拆分 Zset，比如按天拆分，每天的 key 单独设置过期时间，到期自动删除
        3. 方案 3：member 中附加过期时间戳，读取时判断是否过期，配合定时清理
        
    
6. **合理调整 ziplist 阈值，优化内存占用**
    
    可根据业务场景调整`zset-max-ziplist-entries`和`zset-max-ziplist-value`参数，让更多小体量 Zset 使用紧凑型编码，降低内存占用。
    
    - 若业务 Zset 均为小 member、元素数量不多，可适当调大 entries 阈值（如 256），降低内存占用
    - 红线：entries 阈值不建议超过 512，否则 ziplist 的插入 / 删除性能会急剧下降，反而不如跳表编码
    
7. **ZADD 参数的正确使用，避免并发竞态问题**
    
    高并发场景下，必须正确使用 ZADD 的 NX/XX 参数，避免业务逻辑异常：
    
    - 新增元素且禁止更新已存在的 score：必须加 NX 参数，避免并发覆盖已有的 score
    - 仅更新已存在元素的 score，禁止新增：必须加 XX 参数，避免无效元素新增
    - 原子性增量操作：优先使用 ZINCRBY，而非 ZADD+INCR 参数，语义更清晰
    
8. **超大 Zset 拆分，避免集群数据倾斜**
    
    线上单个 Zset 的元素总数建议控制在**10 万以内**，超大 Zset 会导致集群数据倾斜（单 key 仅落在一个分片上），引发分片负载不均、主从同步延迟。
    
    - 优化方案：按业务维度拆分，比如全国排行榜拆分为省份排行榜，年度排行榜拆分为月度排行榜，分散到多个 key，均衡集群负载。

## 一、核心概述

Redis 主从复制是 Redis 实现**高可用、高性能、数据容灾**的核心基础能力，本质是通过数据单向同步，实现一台 Redis 主节点（Master）的数据实时同步到多台 Redis 从节点（Replica，Redis5.0 前称 Slave），主从节点之间形成稳定的复制拓扑。

### 核心角色定义

表格

|角色|核心职责|读写权限|
|---|---|---|
|Master 主节点|负责处理所有写请求，同步写命令到所有从节点|默认可读可写，生产环境建议仅负责写|
|Replica 从节点|负责处理读请求，实时同步主节点的写数据，保持与主节点数据一致|默认只读（replica-read-only=yes），禁止写操作|

### 核心特性

1. **数据单向同步**：仅支持主节点→从节点的单向复制，不支持反向同步
2. **一主多从架构**：一个主节点可挂载多个从节点，一个从节点仅能归属一个主节点
3. **级联复制支持**：从节点可挂载次级从节点，形成树状级联拓扑，减轻主节点压力
4. **非阻塞复制**：全量 / 增量复制过程中，主节点仍可正常处理读写请求，无长时间业务阻塞
5. **自动重连与断点续传**：主从网络断连后，从节点会自动重连，优先尝试增量复制，无需全量同步
6. **读写分离原生支持**：主写从读，水平扩展读性能，解决单机读性能瓶颈

### 核心解决的问题

1. **单点故障风险**：主节点宕机后，从节点可快速切换为主节点，实现故障恢复，避免数据丢失
2. **读性能瓶颈**：单机读能力有限，通过多从节点水平扩展读吞吐量，适配读多写少的业务场景
3. **数据容灾备份**：从节点持久化数据，避免主节点磁盘故障导致的数据完全丢失
4. **运维操作无感知**：数据备份、全量扫描等耗时操作可在从节点执行，不影响主节点业务

## 二、核心底层原理与关键概念

主从复制的核心分为**全量复制**和**增量（部分）复制**两大模式，Redis2.8 版本之前仅支持全量复制，2.8 + 版本引入增量复制，彻底解决了网络断连后频繁全量同步的性能问题。

### 2.1 增量复制的三大核心要素

这是主从复制的底层核心，也是面试最高频考点，三者共同实现了断点续传的增量复制能力。

表格

|核心要素|定义与作用|核心特性|
|---|---|---|
|**运行 ID（Run ID）**|每个 Redis 实例启动时自动生成的 40 位十六进制唯一标识，用于识别实例身份|1. 实例重启后 Run ID 会重新生成<br><br>2. 从节点必须与主节点 Run ID 一致，才能进行增量复制<br><br>3. 主节点 Run ID 变更，会触发从节点全量复制|
|**复制偏移量（Replication Offset）**|用于标记主从节点的数据同步进度，是一个持续递增的数字|1. 主节点每处理 1 字节的写命令，自身偏移量 + 1<br><br>2. 主节点同步写命令给从节点时，会携带当前偏移量<br><br>3. 从节点每成功执行 1 字节的同步命令，自身偏移量 + 1<br><br>4. 主从偏移量一致，代表数据完全同步|
|**复制积压缓冲区（Replication Backlog Buffer）**|主节点维护的一个**固定长度的环形 FIFO 缓冲区**，用于缓存最近执行的写命令|1. 主节点的写命令会同时同步给从节点、写入该缓冲区<br><br>2. 主从断连重连后，主节点通过对比偏移量，从缓冲区中提取断连期间的增量命令，实现增量复制<br><br>3. 固定长度，写满后会覆盖最旧的命令，默认大小 1MB<br><br>4. 所有从节点共享同一个缓冲区，不会为每个从节点单独创建|

### 2.2 全量复制（Full Resynchronization）

全量复制是主从首次建立复制关系时的同步方式，核心是主节点将全量内存数据生成 RDB 快照，发送给从节点，从节点清空本地数据后加载 RDB，最终完成全量数据同步。

#### 触发场景

1. 从节点首次与主节点建立复制关系
2. 主从断连重连后，从节点的偏移量不在复制积压缓冲区范围内
3. 主节点 Run ID 发生变更（如主节点重启）
4. 从节点执行`REPLICAOF`命令切换主节点

#### 核心执行流程

1. **握手认证阶段**：从节点向主节点发送`PING`命令，主节点返回`PONG`，确认网络连通；若主节点设置了密码，从节点通过`AUTH`命令完成认证。
2. **信息协商阶段**：从节点向主节点发送自身的 Run ID 和复制偏移量，主节点判断需要执行全量复制，向从节点返回自身的 Run ID 和当前全量同步的基准偏移量。
3. **RDB 生成与传输**：主节点执行`BGSAVE`命令，fork 子进程生成当前内存的全量 RDB 快照文件；生成完成后，主节点将 RDB 文件发送给从节点。
4. **从节点加载 RDB**：从节点收到完整的 RDB 文件后，清空本地所有数据，加载 RDB 文件到内存，完成全量基准数据的恢复。
5. **增量命令同步**：RDB 生成、传输、加载期间，主节点收到的新写命令会存入**复制缓冲区**（注意：区别于复制积压缓冲区）；从节点加载完 RDB 后，主节点将缓冲区中的增量命令全部发送给从节点，从节点执行后，主从偏移量达成一致。
6. **复制完成**：主从进入持续命令传播阶段，主节点每执行一条写命令，就异步同步给所有从节点，保持数据一致。

### 2.3 增量 / 部分复制（Partial Resynchronization）

增量复制是主从网络断连重连后的优化同步方式，核心是仅同步断连期间主节点新增的写命令，无需全量 RDB 同步，大幅降低同步开销。

#### 触发条件（必须同时满足）

1. 从节点重连后，携带的主节点 Run ID 与当前主节点的 Run ID 完全一致（主节点未重启）
2. 从节点断开连接时的复制偏移量，**存在于主节点的复制积压缓冲区中**（断连期间的增量命令未被缓冲区覆盖）

#### 核心执行流程

1. 主从网络恢复后，从节点重新连接主节点，发送自身的 Run ID 和最后一次同步的复制偏移量。
2. 主节点校验 Run ID 一致后，检查偏移量是否在复制积压缓冲区范围内，确认符合增量复制条件。
3. 主节点从复制积压缓冲区中，提取从偏移量之后的所有增量写命令，发送给从节点。
4. 从节点执行收到的增量命令，更新自身的复制偏移量，最终与主节点偏移量一致，增量复制完成。
5. 主从恢复持续的命令传播阶段。

## 三、主从复制完整生命周期流程

主从复制从建立到持续运行，分为 6 个完整阶段，全程无业务阻塞：

plaintext

```
1. 复制发起 → 2. 握手认证 → 3. 同步模式判断 → 4. 数据同步（全量/增量） → 5. 命令持续传播 → 6. 断连自动重连
```

1. **复制发起**：从节点通过配置 / 命令 / 启动参数设置主节点地址，发起复制请求。
2. **握手认证**：从节点与主节点建立 TCP 连接，通过`PING/PONG`确认连通性，完成密码认证。
3. **同步模式判断**：从节点发送自身 Run ID 和偏移量，主节点判断执行全量复制还是增量复制。
4. **数据同步**：根据判断结果，执行全量复制或增量复制，完成基准数据对齐。
5. **命令持续传播**：同步完成后，进入长连接持续同步阶段，主节点每执行一条写命令，就异步将命令广播给所有从节点，从节点执行命令并更新偏移量，保持数据一致。
6. **断连自动重连**：主从网络断连后，从节点会按`repl-ping-replica-period`配置的频率（默认 10 秒）持续重试连接主节点，重连成功后优先尝试增量复制。

## 四、主从复制的主流拓扑结构

表格

|拓扑类型|结构说明|适用场景|核心优势|核心缺陷|
|---|---|---|---|---|
|一主一从|1 台主节点 + 1 台从节点，最简拓扑|小体量业务、数据热备、测试环境|结构简单、运维成本低、无数据冗余|读扩展能力有限，从节点宕机后无备用读节点|
|一主多从|1 台主节点 + N 台从节点，所有从节点直接挂载到主节点|读多写少业务、读写分离场景、读压力大的系统|读能力线性扩展、配置简单、故障切换灵活|从节点数量过多时，主节点复制压力大，易触发复制风暴；全量复制时主节点带宽 / CPU 占用高|
|树状级联复制|主节点→一级从节点→次级从节点，形成树状层级结构，一级从节点同时作为次级从节点的主节点|大规模从节点场景、跨机房部署、主节点带宽有限的场景|极大减轻主节点的复制压力，跨机房同步仅需主节点同步到机房内一级从节点，其余从节点在机房内同步|拓扑复杂，运维成本高；层级越深，主从同步延迟越大；中间节点故障会导致下游所有从节点同步中断|

## 五、核心命令与配置详解

### 5.1 核心操作命令

Redis 5.0 版本将原`SLAVEOF`命令重命名为`REPLICAOF`，向下兼容旧命令，新业务优先使用`REPLICAOF`。

表格

|命令|语法|核心作用|示例|注意事项|
|---|---|---|---|---|
|REPLICAOF|`REPLICAOF master_ip master_port`|运行时为当前节点设置主节点，开启主从复制|`REPLICAOF 192.168.1.10 6379`|执行后会立即断开旧主节点连接，与新主节点建立复制关系，触发全量复制|
|REPLICAOF|`REPLICAOF NO ONE`|关闭主从复制，当前节点从从节点切换为主节点|`REPLICAOF NO ONE`|执行后不会清空已同步的数据，节点转为独立主节点|
|INFO REPLICATION|`INFO REPLICATION`|查看当前节点的复制状态、主从信息、偏移量等核心数据|`INFO REPLICATION`|排查复制问题的核心命令，主从节点执行返回的内容不同|
|PING|`PING`|测试主从节点之间的网络连通性|`PING`|复制握手阶段的核心命令，主节点正常返回 PONG|

#### INFO REPLICATION 关键字段解读

- 主节点执行核心字段：
    
    - `role:master`：当前节点角色为主节点
    - `connected_slaves:3`：已连接的从节点数量
    - `repl_offset:174200`：当前主节点的复制偏移量
    - `repl_backlog_active:1`：复制积压缓冲区已开启
    - `repl_backlog_size:1048576`：复制积压缓冲区大小（默认 1MB）
    
- 从节点执行核心字段：
    
    - `role:slave`：当前节点角色为从节点
    - `master_host:192.168.1.10`：主节点 IP
    - `master_port:6379`：主节点端口
    - `master_link_status:up`：主从连接状态，up = 正常，down = 断开
    - `slave_repl_offset:174200`：当前从节点的复制偏移量，与主节点一致代表数据同步完成
    - `master_link_down_since_seconds:0`：主从连接断开的时长，0 代表正常连接
    

### 5.2 核心配置项详解

主从复制的核心配置均在`redis.conf`中，可按需调整，以下为生产环境高频配置：

表格

|配置项|作用|推荐值|注意事项|
|---|---|---|---|
|`replicaof <master_ip> <master_port>`|配置主节点的 IP 和端口，开启主从复制|按需填写|重启后永久生效，运行时可通过 REPLICAOF 命令临时修改|
|`masterauth <master_password>`|主节点设置了密码时，从节点配置该参数完成认证|与主节点 requirepass 一致|主节点密码变更后，从节点必须同步修改该配置，否则认证失败|
|`replica-read-only yes`|设置从节点为只读模式|yes（生产环境强制开启）|关闭后从节点可执行写命令，会导致主从数据不一致，复制中断，生产环境绝对禁止关闭|
|`repl-backlog-size <size>`|设置复制积压缓冲区的大小|高写量场景建议 16~64MB，默认 1MB|缓冲区越大，支持的主从断连时间越长，越不容易触发全量复制；需根据业务写量调整|
|`repl-backlog-ttl <seconds>`|所有从节点断开连接后，复制积压缓冲区的保留时长|默认 3600（1 小时）|超时后会释放缓冲区内存，重连后会触发全量复制|
|`repl-ping-replica-period <seconds>`|从节点向主节点发送 PING 心跳的频率|默认 10 秒|用于主从保活、偏移量同步，网络不稳定场景可适当调小|
|`repl-timeout <seconds>`|主从复制的超时时间|默认 60 秒|超过该时间未收到心跳 / 数据，会判定连接断开，触发重连；必须大于 repl-ping-replica-period|
|`min-replicas-to-write <number>`|主节点可正常执行写操作的最小从节点数量|高可用场景建议 1~2|核心防脑裂配置，当健康从节点数量小于该值时，主节点拒绝所有写请求，避免脑裂数据丢失|
|`min-replicas-max-lag <seconds>`|从节点最大允许的延迟时长，配合 min-replicas-to-write 使用|默认 10 秒|从节点延迟超过该值，会被判定为不健康，不计入健康从节点数量|
|`repl-disable-tcp-nodelay no`|是否关闭 TCP_NODELAY|生产环境默认 no|关闭后会合并 TCP 包，减少带宽占用，但会增加主从同步延迟；开启后包数量增多，带宽占用高，但延迟低|

## 六、典型应用场景

### 6.1 读写分离

读多写少业务的核心方案，主节点仅负责处理写请求，所有读请求分发到多个从节点，实现读能力的线性扩展。

- 典型场景：电商商品详情查询、资讯内容阅读、用户信息查询等读占比超 90% 的业务
- 核心收益：单机读瓶颈突破，可通过增加从节点无限扩展读吞吐量，主节点仅需处理写请求，性能大幅提升

### 6.2 数据热备与故障恢复

主节点实时同步数据到从节点，从节点开启持久化，实现数据异地热备。主节点宕机后，从节点可通过`REPLICAOF NO ONE`快速切换为主节点，恢复业务写入，避免长时间服务不可用。

- 典型场景：核心业务数据容灾、机房级故障备份
- 核心收益：避免单点故障导致的数据丢失和服务中断，故障恢复时间可缩短到秒级

### 6.3 无感知运维操作

所有耗时的运维操作均可在从节点执行，不影响主节点的业务读写：

- 数据备份：在从节点执行 BGSAVE 生成 RDB 快照，避免主节点 fork 阻塞
- 全量数据扫描：在从节点执行 KEYS、SCAN、SMEMBERS 等耗时命令，不阻塞主节点
- 数据迁移、升级：先升级从节点，验证无误后切换主从，实现无感知版本升级

### 6.4 跨机房数据同步

跨机房部署场景下，主节点部署在核心机房，从节点部署在备用机房，实现跨机房数据实时同步。核心机房故障时，备用机房的从节点可快速切换为主节点，实现机房级容灾。

### 6.5 Redis 高可用集群的底层基础

Redis Sentinel（哨兵）、Redis Cluster 集群的高可用能力，均基于主从复制实现。哨兵的故障自动切换、集群的分片数据同步，底层都依赖主从复制的能力。

## 七、常见坑与线上最佳实践

### 7.1 高频踩坑点与避坑方案

#### 1. 频繁触发全量复制，导致主节点性能雪崩

**触发原因**：

- 复制积压缓冲区设置过小，主从断连后偏移量已被缓冲区覆盖
- 主节点频繁重启，Run ID 变更，强制触发全量复制
- 从节点频繁重启、网络频繁闪断，导致复制中断
    
    **避坑方案**：
- 高写量场景调大`repl-backlog-size`至 16~64MB，延长断连后增量复制的窗口期
- 优化主从节点的网络稳定性，避免频繁闪断
- 主节点重启前，先手动切换主从，避免 Run ID 变更触发全量复制

#### 2. 复制风暴，导致主节点资源耗尽

**问题本质**：多个从节点同时向主节点发起全量复制，主节点需要多次 fork 子进程生成 RDB，多次传输大体积 RDB 文件，导致 CPU、内存、带宽被打满，主节点阻塞。

**触发场景**：

- 主节点重启后，所有从节点同时发起全量复制
- 多个从节点同时启动，同时连接主节点
    
    **避坑方案**：
- 优先使用**树状级联复制**，减少直接挂载到主节点的从节点数量
- 错开从节点的启动时间，分批建立复制关系，避免同时发起全量复制
- 主节点重启前，先将一个从节点切换为主节点，其余从节点挂载到新主节点，避免全量复制风暴

#### 3. 主从数据不一致

**触发原因**：

- 从节点关闭了`replica-read-only`，手动执行写命令，导致数据冲突
- 主从节点的配置不一致，如 maxmemory、数据淘汰策略不同，主节点未淘汰的数据，从节点因内存不足被淘汰
- 主从节点的 Redis 版本不一致，存在命令兼容性问题
- 网络延迟导致增量命令同步不及时
    
    **避坑方案**：
- 生产环境强制开启`replica-read-only yes`，禁止从节点写入
- 主从节点必须使用相同版本的 Redis、完全一致的配置（除了端口、IP、持久化策略）
- 监控主从复制偏移量差值，设置阈值告警，及时发现同步异常

#### 4. 主从复制延迟过高

**问题本质**：主节点已执行的写命令，从节点长时间未同步，导致读写分离场景下，从节点读到旧数据。

**核心原因**：

- 从节点性能不足，CPU / 内存 / 磁盘 IO 瓶颈，无法及时执行同步命令
- 主节点写量过大，从节点单线程执行同步命令，追不上主节点的写速度
- 网络带宽不足，主节点同步命令传输延迟高
- 从节点开启了 AOF 持久化，`appendfsync=always`导致 IO 阻塞，同步命令执行延迟
    
    **避坑方案**：
- 从节点使用与主节点相同规格的服务器，避免性能短板
- 高写量场景使用级联复制，分散主节点的同步压力
- 优化从节点持久化策略，使用`appendfsync=everysec`，避免 IO 阻塞
- 监控主从延迟（偏移量差值、主从数据对比），设置阈值告警
- 读写分离场景下，强一致性读请求直接走主节点，最终一致性读请求走从节点

#### 5. 脑裂问题，导致数据丢失

**问题本质**：主节点因网络分区，与从节点、哨兵失联，但主节点仍在正常运行，客户端继续向主节点写入数据；网络恢复后，原主节点会被降级为从节点，清空本地数据，全量同步新主节点，导致网络分区期间的写数据全部丢失。

**避坑方案**：

- 开启主节点的防脑裂配置：`min-replicas-to-write 1`、`min-replicas-max-lag 10`，当健康从节点数量不足时，主节点拒绝写请求
- 合理设置哨兵的心跳检测、故障判定阈值，避免误判主节点故障
- 优化机房网络，减少网络分区的概率

#### 6. fork 阻塞导致复制超时

**问题本质**：全量复制时，主节点执行 BGSAVE 需要 fork 子进程，Redis 内存越大，fork 阻塞时间越长，导致主从复制超时中断。

**避坑方案**：

- 控制 Redis 单实例内存在 8~32G，最大不超过 64G
- 调整 Linux 内核参数`vm.overcommit_memory=1`，避免 fork 失败
- 关闭透明大页（THP），减少 fork 时的页表复制开销
- 避免在业务高峰期执行全量复制

### 7.2 线上通用最佳实践

1. **主从节点同规格部署**：从节点使用与主节点相同的 CPU、内存、磁盘、网络规格，避免性能短板导致的同步延迟
2. **强制开启从节点只读**：生产环境必须设置`replica-read-only yes`，禁止从节点写入，避免数据不一致
3. **合理调整复制积压缓冲区**：根据业务写量调整`repl-backlog-size`，高写量场景建议 16~64MB，减少全量复制触发概率
4. **开启防脑裂配置**：核心业务必须设置`min-replicas-to-write`和`min-replicas-max-lag`，避免脑裂数据丢失
5. **主从节点同版本部署**：主从节点使用相同的 Redis 大版本，避免命令兼容性问题导致的复制中断
6. **监控全覆盖**：重点监控主从连接状态、复制偏移量差值、复制延迟、全量复制触发次数、缓冲区使用率，设置阈值告警
7. **避免主节点持久化**：主节点关闭 RDB/AOF 持久化，仅在从节点开启持久化，避免主节点 fork 阻塞影响业务
8. **定期演练故障切换**：定期执行主从切换演练，验证复制的可用性和故障恢复流程，避免线上故障时操作失误

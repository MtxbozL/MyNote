## 一、核心概述

Set 是 Redis 中**无序、不可重复**的字符串元素集合，每个元素均为 String 类型，无重复值、无固定排序，是 Redis 专门用于去重、集合运算的核心数据类型。

- 核心特性：**元素唯一性**、单元素操作 O (1) 时间复杂度、所有操作原子性、原生支持交 / 并 / 差集运算、单 key 最大支持`2^32 - 1`个元素（约 42 亿）
- 原子性保证：所有单元素操作均为原子性，基于 Redis 单线程模型，并发场景无竞态问题
- 核心差异化优势：天然去重能力 + 高性能集合运算，是社交关系、标签系统、去重统计等场景的最优解

---

## 二、底层实现原理

Redis Set 会根据元素类型、元素数量自动选择最优编码格式，平衡内存占用与读写性能，编码转换**不可逆**（仅能从低内存开销编码向高内存开销编码转换）。

Set 底层支持两种编码：`intset（整数集合）`和`hashtable（哈希表）`，默认优先使用 intset 编码，不满足触发条件时自动转为 hashtable 编码。

### 2.1 紧凑型编码：intset（整数集合）

intset 是 Redis 为纯整数元素集合设计的紧凑型存储结构，一段连续内存块，无额外指针开销，极致优化小体量整数集合的内存占用。

#### 触发条件（同时满足）

1. 集合中**所有元素均为 64 位有符号整数**
2. 集合元素总数 ≤ `set-max-intset-entries`（默认值 512）

#### 核心结构

```c
typedef struct intset {
    uint32_t encoding;  // 编码类型：int16/int32/int64
    uint32_t length;    // 集合元素总个数
    int8_t contents[];  // 柔性数组，按从小到大有序存储实际元素
} intset;
```

#### 核心特性

1. **有序存储 + 二分查找**：contents 数组按整数大小升序排列，查找操作通过二分查找实现，时间复杂度 O (logn)，远优于无序遍历
2. **编码自动升级机制**：当新增的整数超出当前编码的取值范围时，会自动升级整个集合的编码（int16→int32→int64），重新分配内存并迁移所有元素
    
    - 核心优势：按需分配编码，无需一开始就使用高字节编码，最大化节省内存
    - 不可逆规则：编码升级后，即使删除了超出范围的大整数，编码也不会降级
    
3. **内存占用极低**：无指针、无冗余元数据，相同数量的整数元素，intset 内存占用比 hashtable 低 80% 以上

#### 操作性能说明

- 查找：O (logn)（二分查找）
- 插入 / 删除：O (n)（需移动连续内存块，元素越多性能衰减越明显）

### 2.2 高性能编码：hashtable（哈希表）

当 Set 不满足 intset 的触发条件时，会自动不可逆地转为 hashtable 编码，底层复用 Redis 核心字典结构，保证海量元素下的稳定 O (1) 操作性能。

#### 触发条件（满足其一即可）

1. 集合中存在非整数类型的元素
2. 集合元素总数超过`set-max-intset-entries`阈值（默认 512）

#### 核心实现逻辑

hashtable 编码的底层结构与 Hash 类型的哈希字典完全一致，核心设计如下：

- 用哈希表的**key 存储集合元素**，保证元素唯一性（哈希表 key 天然不可重复）
- 哈希表的**value 统一为 NULL**，无额外存储开销
- 采用链地址法解决哈希冲突，支持渐进式 rehash，避免扩容 / 缩容时阻塞主线程

#### 核心优劣势

- 优势：增删改查平均时间复杂度 O (1)，性能不随元素数量增加而衰减，支持海量元素存储，无连续内存移动开销
- 缺陷：指针与哈希桶的额外内存开销大，内存占用远高于 intset，易产生内存碎片

### 3.4 集合运算命令（Set 核心特色，高频）

Set 的核心差异化能力，原生支持交 / 并 / 差集运算，所有运算均为原子性执行，是社交关系、标签筛选等场景的核心支撑。

> 性能警告：大集合的集合运算会消耗大量 CPU，阻塞主线程，线上需严格遵循优化规范。

| 命令          | 语法                                      | 时间复杂度                       | 核心作用                                    | 示例                                                                   | 注意事项                                                                  |

## 四、典型应用场景

### 4.1 数据去重与幂等性控制（核心基础场景）

Set 的元素唯一性天然适配所有去重场景，无需业务层做重复判断，O (1) 性能支撑高并发场景。

- 典型场景：用户 UV 统计、接口幂等性控制、内容标签去重、订单去重
- 实现示例：

```bash
    # 1. 网站UV统计：每日用户访问ID存入Set，SCARD直接获取去重后的UV数
    SADD uv:20260308 user:1001 user:1002 user:1001
    SCARD uv:20260308 # 返回2，自动去重重复访问的用户
    
    # 2. 接口幂等性控制：请求前判断唯一标识是否存在，不存在则处理

    SISMEMBER order:idempotent 20260308001 # 不存在返回0，允许处理
    SADD order:idempotent 20260308001 # 处理完成后添加，防止重复提交
```

### 4.2 社交关系链（经典核心场景）

基于 Set 的交 / 并 / 差集运算，完美实现社交场景的关注、粉丝、共同好友、好友推荐等功能，是 Set 最具特色的应用场景。

- 实现示例：

```bash
    # 1. 关注/取关功能
    SADD user:1001:focus 1002 1003 1004 # 用户1001关注3个用户
    SREM user:1001:focus 1003 # 用户1001取关1003
    SADD user:1002:fans 1001 # 用户1002的粉丝列表添加1001
    
    # 2. 共同关注/共同好友：你们共同关注了XX
    SINTER user:1001:focus user:1002:focus
    
    # 3. 好友推荐：给1001推荐1002关注了、但自己没关注的人
    SDIFF user:1002:focus user:1001:focus
    
    # 4. 共同群聊：两个群的共同成员
    SINTER group:1001:user group:1002:user
```

### 4.3 黑白名单与权限校验

基于 SISMEMBER 的 O (1) 高性能判断，实现高并发场景下的黑白名单、接口权限校验，无需查询数据库，性能提升 10 倍以上。

- 典型场景：IP 黑名单、接口访问白名单、敏感词过滤、操作权限校验
- 实现示例：

```bash
    # 1. IP黑名单校验
    SADD ip:blacklist 192.168.1.1 192.168.1.2 192.168.1.3
    SISMEMBER ip:blacklist 192.168.1.1 # 存在返回1，直接拒绝请求
    
    # 2. 管理员接口白名单校验
    SADD api:admin:whitelist user:1001 user:1002
    SISMEMBER api:admin:whitelist user:1003 # 不存在返回0，拒绝访问
```

### 4.4 随机抽奖与随机推荐

基于 SPOP 和 SRANDMEMBER，原生实现高性能的随机逻辑，无需业务层处理随机算法，原子性执行无并发问题。

- 典型场景：活动抽奖、随机点名、商品随机推荐、内容随机分发
- 实现示例：

```bash
    # 1. 不可重复抽奖：抽完即从奖池中移除，不会重复中奖
    SADD lottery:pool user:1001 user:1002 user:1003 user:1004 user:1005
    SPOP lottery:pool 3 # 随机弹出3个中奖用户，奖池中不再保留
    
    # 2. 可重复随机商品推荐：随机返回10个热门商品，不删除原数据
    SRANDMEMBER goods:hot 10
    
    # 3. 课堂随机点名：允许重复点名
    SRANDMEMBER student:class:1 1
```


### 4.5 标签系统与用户画像

基于 Set 的集合运算，实现用户标签管理、兴趣画像、精准用户筛选，是电商、内容平台的核心应用场景。

- 实现示例：

```bash
    # 1. 给用户打标签
    SADD user:1001:tag 数码 游戏 运动 摄影
    SADD user:1002:tag 数码 美妆 旅行 摄影
    SADD tag:数码 user:1001 user:1002 user:1003
    SADD tag:游戏 user:1001 user:1004 user:1005
    
    # 2. 精准推送：筛选所有同时喜欢「数码」+「游戏」的目标用户
    SINTER tag:数码 tag:游戏
    
    # 3. 兴趣匹配：计算两个用户的共同兴趣标签
    SINTER user:1001:tag user:1002:tag
```

### 4.6 分布式任务去重

基于 Set 的唯一性，实现分布式环境下的任务去重，避免多个服务节点重复执行同一个任务，无需加锁，原子性保证。

- 实现示例：

```bash
    # 任务生产：将待执行的任务ID存入待执行集合
    SADD task:pending 1001 1002 1003
    
    # 任务消费：原子性将任务从待执行集合移动到执行中集合，避免重复消费
    SMOVE task:pending task:running 1001
    
    # 任务完成：从执行中集合删除任务
    SREM task:running 1001
```

## 五、常见坑与最佳实践

1. **绝对禁止对大集合使用 SMEMBERS 全量遍历**
    
    这是 Set 最常见的线上事故！SMEMBERS 会一次性拉取全量元素，时间复杂度 O (n)，若集合有上万个元素，会直接阻塞 Redis 单线程，导致实例卡顿、雪崩。
    
    - 线上规范：仅当元素数量明确小于 100 个时，才可使用 SMEMBERS；大集合遍历必须使用 SSCAN 增量迭代，每次返回少量数据，无阻塞风险。
    - 反例：循环执行 SMEMBERS 获取百万级元素的集合，直接导致 Redis 主库阻塞，主从切换。
    
2. **严格控制集合的元素数量与单个元素大小**
    
    线上单个 Set 的元素总数建议控制在**1 万以内**，单个元素的字节大小控制在**1KB 以内**。
    
    - 元素过多会触发编码转换，从 intset 转为 hashtable，内存占用飙升，同时集合运算性能急剧下降。
    - 单个元素过大，会形成大 key，导致网络 IO 阻塞、主从同步延迟、内存碎片激增。
    - 优化方案：超大集合按业务维度拆分，比如按时间、地区、用户分群拆分，如`uv:202603`拆分为`uv:20260301`、`uv:20260302`，避免单集合过大，同时均衡集群分片负载。
    
3. **大集合的交并差运算必须做性能优化，严禁在主库执行超大集合运算**
    
    集合运算的时间复杂度随集合大小和数量线性增长，超大集合（10 万 + 元素）的交并差运算会消耗大量 CPU，直接阻塞 Redis 主线程，是线上高频故障点。
    
    - 优化规范：
        
        1. 优先用最小的集合作为运算基准，减少遍历次数，比如交集运算，Redis 会自动以最小集合为基准遍历，降低计算量。
        2. 用 STORE 系列命令（SINTERSTORE 等）将运算结果存入 Redis，避免重复计算和大结果集的网络传输。
        3. 超大集合的运算必须放在从节点执行，绝对禁止在主节点执行，避免阻塞业务写入。
        4. 单次运算的集合数量建议不超过 3 个，避免复杂度指数级增长。
        
    
4. **合理调整 intset 编码阈值，最大化内存优化**
    
    intset 编码的内存占用远低于 hashtable，可根据业务场景调整`set-max-intset-entries`参数，让更多集合使用紧凑型编码。
    
    - 若业务 Set 均为整数元素，且元素数量不大，可适当调大阈值（如 1000），降低内存占用。
    - 红线：阈值不建议超过 2000，否则 intset 的插入 / 删除需要移动大量连续内存，二分查找性能也会急剧下降，反而不如 hashtable。
    
5. **分清 SPOP 与 SRANDMEMBER 的使用场景，避免业务逻辑错误**
    
    这两个命令是高频用错的点，核心区别必须明确：
    
    - SPOP：随机取出**并删除**元素，取出后元素从集合中永久消失，适合不可重复的场景（如抽奖、名额分配）。
    - SRANDMEMBER：随机取出**不删除**元素，元素仍保留在集合中，适合可重复的场景（如随机推荐、随机点名）。
    - 反例：抽奖场景用 SRANDMEMBER，导致同一个用户被多次抽中，引发业务事故。
    
6. **Set 仅支持 key 级别的过期时间，不支持元素级别过期**
    
    Redis 的过期时间只能作用于整个 Set 顶级 key，无法给单个元素设置过期时间，这是最高频的业务坑。
    
    - 合规解决方案：
        
        1. 方案 1：按时间维度拆分 Set，比如 UV 按天拆分，每天的 Set 单独设置过期时间，到期自动删除。
        2. 方案 2：配合 Sorted Set，将元素存入 Set 的同时，将元素和过期时间戳存入 ZSet，定时扫描 ZSet，删除过期元素。
        3. 方案 3：元素中附加过期时间戳，读取时判断是否过期，配合定时任务清理。
        
    
7. **禁止用 Set 存储超大规模的扁平全局数据**
    
    不要把 Set 当成全局存储，比如把所有用户 ID 存入一个`all_user`大 Set key 中。
    
    - 核心缺陷：
        
        1. 极易形成超大 key，导致 Redis 主线程阻塞。
        2. 集群环境下，大 key 只会落在一个分片上，导致数据倾斜、分片负载不均，成为集群性能瓶颈。
        3. 无法设置元素级别的过期时间，运维难度极大。
        
    - 正确做法：按业务维度拆分 key，分散到多个 key，均衡集群负载。

#### 核心标准答案

##### 一、协程的核心概念

协程（Coroutine），也叫用户态轻量级线程，是**完全在用户态实现的、由用户程序自行调度的执行单元**，无需内核参与调度。协程依附于线程运行，一个线程内可以运行多个协程，同一时间一个线程内只有一个协程在运行，协程的切换由用户程序控制，在用户态完成，无需陷入内核，切换开销几乎为零。

协程的核心特性：

1. **用户态调度**：调度完全由用户程序控制，无需内核参与，没有内核态和用户态的切换开销；
2. **极低的切换开销**：协程切换只需保存少量的上下文（程序计数器、栈指针），无需切换地址空间、刷新 TLB，切换耗时是纳秒级，远低于线程的微秒级切换；
3. **极小的资源占用**：协程的初始栈大小通常只有几 KB（Go 协程初始栈 2KB），可动态扩缩容，而线程默认栈大小是 8MB，一个线程的内存可以运行上千个协程，可轻松创建数十万级别的协程；
4. **协作式调度**：传统线程是抢占式调度，由内核决定调度时机；协程是协作式调度，协程主动让出 CPU 执行权，切换到其他协程，避免了抢占式调度的不确定性；
5. **共享线程资源**：同一个线程内的多个协程，共享该线程的地址空间和资源，协程间通信和线程内通信一样简单，直接读写共享内存。

##### 二、协程和线程的核心区别

表格

|对比维度|协程|线程|
|:--|:--|:--|
|**调度主体**|用户态调度，由用户程序 / 语言 runtime 控制，内核无感知|内核态调度，由操作系统内核完全控制|
|**切换开销**|极低，用户态切换，仅保存少量上下文，纳秒级耗时|极高，内核态切换，需要切换上下文、地址空间，微秒级耗时|
|**资源占用**|极小，初始栈仅几 KB，可动态扩缩容，支持数十万级并发|极大，默认栈 8MB，资源占用固定，最多支持上千级并发|
|**调度模式**|协作式调度，协程主动让出执行权，无强制抢占|抢占式调度，内核强制分配时间片，超时自动切换|
|**并发能力**|极高，单机可轻松支持数十万协程并发|较低，单机最多支持几千个线程并发|
|**多核利用**|单个协程只能使用一个线程的 CPU，需要多线程 + 多协程才能利用多核 CPU|可直接利用多核 CPU，内核会将线程调度到不同的核心执行|
|**稳定性**|单个协程阻塞，会导致整个线程阻塞（除非有异步调度）|单个线程阻塞，不会影响进程内的其他线程|
|**锁竞争**|同一线程内的协程串行执行，无并行执行，无需加锁，锁竞争极少|多线程并行执行，共享资源需要加锁，高并发下锁竞争激烈|

#### 专家级拓展

- 主流协程实现模型：
    
    1. **N:1 模型**：多个协程映射到一个内核线程，切换开销极小，但无法利用多核 CPU，一个协程阻塞会导致整个线程阻塞，早期的协程库大多采用这个模型；
    2. **1:1 模型**：一个协程对应一个内核线程，本质就是线程，失去了协程的优势，极少使用；
    3. **M:N 模型**：多个协程映射到多个内核线程，是目前最主流的模型，Go 语言的 GMP 调度模型就是典型的 M:N 模型，兼顾了协程的低开销和多核 CPU 的利用，同时实现了异步抢占式调度，解决了协程阻塞导致线程阻塞的问题。
    
- 生产环境最佳实践：
    
    1. 高并发 IO 密集型场景，优先使用协程模型，比如 Go、Python 的 asyncio、Java 的 Project Loom，可大幅提升服务的并发能力，降低资源占用；
    2. CPU 密集型场景，协程的优势不明显，需要配合多线程 / 多进程，才能充分利用多核 CPU；
    3. 协程中避免执行阻塞的同步 IO 操作，否则会导致线程阻塞，其他协程无法执行，必须使用异步 IO，配合协程的调度器，实现 IO 阻塞时自动切换协程；
    4. 控制协程的并发数量，即使协程资源占用小，也不能无限制创建，避免内存占用过高、调度开销增大。
    

#### 面试避坑指南

- 严禁说「协程比线程快」，协程的优势是切换开销小、并发能力强，不是执行速度快，CPU 密集型任务，协程和线程的执行速度没有区别；
- 避免混淆协程和线程的调度主体，协程是用户态调度，线程是内核态调度，这是二者最核心的区别，必须说清楚；
- 不要只说协程的优势，不说协程的局限性，比如无法直接利用多核、阻塞操作会影响其他协程，体现你对协程的全面理解。

**职级适配**：中级必问、高级进阶

#### 【核心考点】

考察负载均衡算法的底层原理理解、生产环境选型与落地能力，判断候选人能否根据业务场景匹配最优策略，而非仅会背诵配置。

#### 【专家级标准答案】

Nginx 负载均衡基于`ngx_http_upstream_module`模块实现，核心是将客户端请求按照预设规则分发至多台后端服务器，实现系统水平扩容、并发能力提升与服务高可用。策略分为 **原生内置策略（默认编译集成）** 和 **第三方扩展策略（需额外编译安装）** 两大类，核心策略、原理与选型如下：

| 策略分类  | 策略名称              | 算法核心原理                                                           | 生产级选型标准                                        |
| :---- | :---------------- | :--------------------------------------------------------------- | :--------------------------------------------- |
| 原生内置  | 轮询（默认策略）          | 基于平滑加权轮询算法，默认权重均为 1，请求按顺序逐一、平滑分配到后端服务器；后端服务器宕机自动剔除，恢复后自动重新加入     | 后端服务器硬件性能完全一致，请求处理耗时均匀，无会话保持要求的通用场景            |
| 原生内置  | 加权轮询              | 基于权重分配请求流量，权重越高的服务器，被分配的请求概率与数量越多，Nginx 默认采用平滑加权轮询，避免权重不均导致的流量突刺 | 后端服务器硬件性能不均，需给高性能服务器分配更多流量的场景                  |
| 原生内置  | IP 哈希（ip_hash）    | 基于客户端 IP 地址做哈希计算，同一个 IP 的请求固定转发到同一台后端服务器，后端宕机后需手动剔除              | 需保持用户会话一致性，无分布式 session 方案，需解决 session 共享问题的场景 |
| 原生内置  | 最少连接数（least_conn） | 优先将请求转发到当前活跃连接数最少的后端服务器，动态适配后端负载                                 | 请求处理耗时差异大（如长请求、大文件上传、接口耗时不均）的场景，避免单台服务器过载      |
| 第三方扩展 | 响应时间优先（fair）      | 基于后端服务器的响应时间做负载均衡，响应时间越短的服务器，被分配的请求优先级越高                         | 对接口响应速度要求极高，需优先保障用户访问体验的核心业务场景                 |
| 第三方扩展 | URL 哈希（url_hash）  | 基于客户端请求的 URL 做哈希计算，同一个 URL 的请求固定转发到同一台后端服务器                      | 后端有缓存服务（如 CDN、静态资源缓存、分布式缓存）的场景，大幅提升缓存命中率       |

#### 【深度拓展（面试加分项）】

1. 平滑加权轮询算法的核心优势：Nginx 默认的加权轮询并非简单的按权重集中分配，而是平滑加权轮询，可避免高权重服务器被集中分配请求导致的流量突刺，实现流量的均匀、平滑分发。
2. 会话保持的进阶方案：ip_hash 存在客户端出口 IP 变化导致会话失效的问题，生产环境进阶方案为基于 cookie、用户 ID 的哈希策略，或通过后端分布式 session（Redis 存储）彻底解耦，无需依赖负载均衡的会话保持。
3. 健康检查机制：Nginx 开源版仅支持被动健康检查，通过`proxy_next_upstream`配置，当后端返回 502/504 / 超时等错误时，自动剔除故障节点；Nginx Plus 商业版支持主动健康检查，定期发送探测请求，提前剔除故障节点，从根源减少业务错误。
4. 高级流量调度能力：基于 upstream 模块可实现灰度发布、蓝绿发布、A/B 测试，通过权重配比实现流量的逐步切流，降低发布风险。

#### 【面试避坑提示】

1. 禁止仅罗列策略名称，不讲解算法原理与选型标准，面试官核心考察的是场景化落地能力，而非配置背诵。
2. 禁止认为 ip_hash 能解决所有会话保持问题，忽略客户端出口 IP 变化、NAT 网络环境下的会话失效问题。
3. 禁止忽略后端节点的宕机剔除与恢复机制，生产环境必须配置`max_fails`（最大失败次数）和`fail_timeout`（故障剔除时长），避免请求转发到已宕机的节点。

**职级适配**：高级必问、中级核心

#### 【核心考点】

考察生产环境故障的体系化排查思维、全链路问题定位能力、应急响应能力，是面试中最能体现候选人实战经验的题目，80% 的线上 Nginx 故障都围绕这两个错误码。

#### 【专家级标准答案】

##### 一、核心本质区别

二者均为 Nginx 作为反向代理时，与后端服务通信出现异常返回的错误码，**核心区别在于异常发生的链路阶段不同**：

- **502 Bad Gateway（网关错误）**：Nginx 已向后端服务发起连接请求，但**连接失败，或收到了后端服务返回的无效、异常、被重置的响应**，核心是「连不上、响应无效」。
- **504 Gateway Time-out（网关超时）**：Nginx 已成功与后端服务建立 TCP 连接，且已将请求完整转发给后端，但**在配置的超时时间内，未收到后端服务返回的任何响应**，核心是「连得上、等不到响应」。

##### 二、502 Bad Gateway 全链路排查方法论与解决方案

1. **高频根因分类**
    
    表格
    
| 根因分类         | 具体故障场景                                        |
| :----------- | :-------------------------------------------- |
| 后端服务不可用      | 后端服务进程崩溃、未启动、被 OOM 杀死，端口未监听，服务宕机              |
| 网络连通性故障      | 服务器防火墙 / 安全组拦截了 Nginx 到后端的端口，服务器之间网络不通，端口被封禁  |
| 后端服务过载拒绝     | 后端服务的最大连接数、backlog 队列满了，无法处理新的 TCP 连接，拒绝连接请求  |
| FastCGI 服务异常 | PHP 场景下，PHP-FPM 进程未启动、进程耗尽、请求队列满了，无法处理请求      |
| Nginx 配置错误   | proxy_pass 配置的后端地址、端口错误，或后端 upstream 节点配置全部失效 |

1. **标准化排查步骤（从易到难，从高频到低频）**
    
    1. **第一步：验证后端服务可用性，定位故障边界**
        
        在 Nginx 服务器上，通过`curl -i 后端IP:端口/接口`直接访问后端服务，若无法正常响应，说明故障在后端服务或网络链路；若能正常响应，说明故障在 Nginx 配置层面。
    2. **第二步：检查后端服务状态**
        
        登录后端服务器，检查服务进程是否正常运行（`ps -ef | grep 服务名`）、端口是否正常监听（`netstat -tunlp | grep 端口`）、服务日志是否有崩溃、报错信息，重启异常服务。
    3. **第三步：排查网络连通性与访问控制**
        
        检查后端服务器防火墙、云服务器安全组是否放行 Nginx 服务器的 IP 与端口，通过`telnet 后端IP 端口`、`traceroute`命令验证网络连通性，关闭不必要的防火墙规则，放行对应访问权限。
    4. **第四步：检查 Nginx 错误日志，定位精准根因**
        
        查看 Nginx error.log 错误日志（默认路径`/var/log/nginx/error.log`），根据日志中的报错信息精准定位：
        
        - `connect() failed (111: Connection refused)`：后端服务端口未监听、服务未启动，或被防火墙拒绝；
        - `connect() failed (113: No route to host)`：网络不通，IP 地址错误、服务器宕机，或路由不可达；
        - `upstream prematurely closed connection`：后端服务在处理请求过程中，主动关闭了连接，通常是后端服务过载、进程崩溃。
        
    5. **第五步：核对 Nginx 配置，验证配置合法性**
        
        核对 upstream 模块中的后端地址、端口是否正确，proxy_pass 配置是否有误，执行`nginx -t`验证配置文件合法性，重载配置`nginx -s reload`验证。
    
2. **生产级根治方案**
    
    - 配置被动健康检查，通过`max_fails`和`fail_timeout`自动剔除故障节点，避免请求转发到已宕机的服务；
    - 配置后端服务监控告警，实时感知服务宕机、端口异常、进程退出等问题，提前介入处理；
    - 配置后端服务的自动重启机制，服务崩溃后自动拉起，减少故障时长。
    

##### 三、504 Gateway Time-out 全链路排查方法论与解决方案

1. **高频根因分类**
    
    表格
    
| 根因分类          | 具体故障场景                                    |
| :------------ | :---------------------------------------- |
| 后端接口性能瓶颈      | 接口存在慢 SQL、复杂业务逻辑、循环调用、第三方接口超时，导致接口执行耗时过长  |
| 服务器资源瓶颈       | 后端服务器 CPU、内存、磁盘 IO、网络带宽满载，无法及时处理请求，响应延迟   |
| Nginx 超时配置不合理 | Nginx 超时阈值设置过低，小于后端接口的正常执行耗时，提前断开连接返回 504 |
| 后端服务阻塞        | 后端服务出现死锁、线程池耗尽、数据库连接池耗尽，无法处理请求，无响应返回      |

1. **标准化排查步骤（从易到难，从高频到低频）**
    
    1. **第一步：直接访问后端接口，验证接口性能**
        
        在 Nginx 服务器上通过`curl -o /dev/null -s -w "%{time_total}\n" 后端IP:端口/接口`，测试接口的响应耗时，若耗时超过 Nginx 的超时阈值，说明故障核心是后端接口性能问题。
    2. **第二步：检查 Nginx 超时配置，核对阈值合理性**
        
        查看 Nginx 配置中的超时参数，核心配置如下，需与后端服务的超时阈值对齐，避免 Nginx 提前断开连接：

    ```nginx
        proxy_connect_timeout 60s; # 与后端建立TCP连接的超时时间，默认60s
        proxy_read_timeout 120s; # 后端服务响应的超时时间，核心参数，两次读操作的间隔超时
        proxy_send_timeout 120s; # 向后端发送请求的超时时间，两次写操作的间隔超时
    ```

    若接口正常执行耗时超过配置阈值，可临时调高超时阈值，避免业务中断，同时推进接口性能优化。
    3. **第三步：排查后端服务器资源瓶颈**
        
        登录后端服务器，通过`top`、`free -m`、`iostat`、`iftop`命令，检查 CPU 使用率、内存占用、磁盘 IO 负载、网络带宽占用，定位资源瓶颈，扩容对应资源，优化资源占用过高的进程。
    4. **第四步：全链路排查后端接口性能瓶颈**
        
        查看后端服务的业务日志、慢日志，数据库的慢查询日志，定位耗时的业务逻辑、慢 SQL、第三方接口调用；检查后端服务的线程池、数据库连接池是否耗尽，是否存在死锁、阻塞问题，针对性优化。
    5. **第五步：检查网络链路质量**
        
        通过`ping`、`mtr`命令检查 Nginx 与后端服务器之间的网络延迟、丢包率，若存在网络抖动、丢包，会导致响应数据传输超时，需解决网络链路问题。
    
2. **生产级根治方案**
    
    - 核心是优化后端接口性能，解决慢 SQL、复杂业务逻辑、第三方接口依赖等问题，从根源降低接口响应耗时；
    - 配置全链路超时对齐，Nginx、网关、后端服务、数据库的超时阈值需逐层收敛，避免出现超时断层；
    - 配置慢请求监控告警，实时感知接口耗时异常，提前介入优化，避免触发 504 超时；
    - 针对长耗时接口，采用异步化处理方案，前端轮询查询结果，避免同步长请求导致的超时问题。
    

#### 【深度拓展（面试加分项）】

1. 故障根因精准定位进阶：通过`tcpdump`抓包工具，抓取 Nginx 与后端服务之间的 TCP 数据包，精准分析是连接被拒绝、还是连接建立后无响应，彻底区分 502 与 504 的根因，解决疑难杂症。
2. 故障自愈方案：生产环境可配置`proxy_next_upstream`，当出现 502/504 / 超时等错误时，自动将请求重试转发到其他健康的后端节点，实现故障自动容灾，用户无感知。
3. 全链路监控体系：搭建 Nginx + 后端服务 + 数据库的全链路监控体系，通过 Prometheus+Grafana 实时监控 Nginx 的错误码、请求耗时、后端服务状态、服务器资源指标，实现故障提前预警、快速定位。

#### 【面试避坑提示】

1. 禁止混淆 502 与 504 的核心区别，导致排查方向完全错误，这是面试中最常见的扣分点，必须先明确二者的本质差异，再展开排查思路。
2. 禁止盲目调高超时参数，掩盖后端接口的性能问题，调高超时阈值仅为临时应急方案，核心根治方案是优化后端接口性能。
3. 禁止仅排查 Nginx 层面，忽略后端服务、数据库、网络链路的全链路排查，线上故障通常是全链路的问题，而非单一节点的问题。
4. 禁止忽略 Nginx 错误日志，错误日志中已明确记录了故障的精准根因，是排查问题的第一优先级，而非盲目猜测。

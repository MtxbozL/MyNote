**职级适配**：中级必问、高级打底

#### 【核心考点】

考察对 Nginx 高并发、高可用底层架构的深度理解，进程模型的设计思想，判断候选人能否从底层解释 Nginx 的性能与稳定性优势，是进阶能力的核心分水岭。

#### 【专家级标准答案】

Nginx 采用**单 Master 多 Worker 的多进程单线程架构**，是其高并发、高可用、支持热部署的核心底座，默认启动两类核心进程，辅以可选的缓存管理进程，进程间相互独立，通过共享内存实现通信。

1. **核心进程职责**
    
    - **Master 进程（主进程，1 个）**：核心定位是**管理与调度中枢**，不处理任何客户端请求。核心职责：读取并校验配置文件的合法性；管理 Worker 进程的生命周期，启动、停止、重启 Worker 进程；接收运维信号（reload/stop/quit），实现配置平滑重载、服务热部署、平滑停止；监控 Worker 进程状态，Worker 异常崩溃时自动拉起新进程，保障服务不中断。
    - **Worker 进程（工作进程，多个）**：核心定位是**请求处理实体**，数量可配置，生产环境默认等于服务器 CPU 物理核心数。核心职责：全量处理客户端的网络连接、请求读写、内容转发、响应返回；采用 epoll 异步非阻塞事件模型处理并发请求，单进程单线程无锁设计，避免线程切换开销；进程间相互独立，单个 Worker 崩溃不影响整体服务。
    - 可选辅助进程：Cache Manager（缓存管理进程）、Cache Loader（缓存加载进程），用于管理磁盘缓存的索引、过期与预热，不处理客户端请求。
    
2. **该架构的核心优势**
    
    - 极致的高可用性：Worker 进程相互隔离，单进程异常不会导致服务整体宕机，Master 可快速拉起新进程，配合热部署能力，实现配置变更、版本升级不中断业务。
    - 最大化 CPU 性能利用率：Worker 进程数可与 CPU 核心一一绑定（worker_cpu_affinity），避免进程在多核间切换的开销，充分利用多核 CPU 的并行处理能力，无锁设计杜绝了多线程的资源竞争问题。
    - 高并发低资源消耗：单 Worker 采用单线程 + 异步非阻塞模型，无需为每个连接创建新的进程 / 线程，内存占用极低，可在低配服务器上支撑数万级并发连接。
    - 运维友好与风险可控：所有配置变更、服务操作均通过 Master 进程管控，无需修改业务流量链路，重载配置时仅增量替换 Worker 进程，操作风险极低，可回滚性强。

#### 【深度拓展（面试加分项）】

1. **惊群问题**的解决方案：Nginx 通过`accept_mutex`互斥锁解决多 Worker 进程的 TCP 连接惊群问题，仅持有锁的 Worker 进程会处理新的连接请求，避免多个 Worker 同时竞争同一个连接，造成资源浪费；Nginx 1.11.3 版本后默认关闭该锁，采用内核级的 SO_REUSEPORT 特性解决惊群问题，性能更优。
2. **热部署的底层实现**：Master 进程收到 reload 信号后，重新加载配置文件，启动新的 Worker 进程，向旧 Worker 进程发送退出信号；旧 Worker 进程停止接收新连接，处理完当前所有请求后自动退出，实现无感知平滑重载。
3. 单线程设计的核心逻辑：Worker 进程采用单线程，是因为异步非阻塞模型已能充分利用 CPU 性能，多线程会引入锁竞争、线程切换的额外开销，反而降低性能；仅在磁盘 IO 等阻塞操作中，会采用多线程池处理，不阻塞主事件循环。

#### 【面试避坑提示】

1. 禁止将 Nginx 架构说成 “多线程模型”，这是根本性概念错误，Nginx Worker 进程为单线程设计，核心是多进程架构。
2. 禁止混淆 Master 与 Worker 进程的职责，出现 “Master 进程处理客户端请求” 这类错误，直接判定为底层架构认知缺失。
3. 禁止认为 “Worker 进程数越多性能越好”，最优配置为等于 CPU 物理核心数，超过核心数会导致进程频繁切换，反而降低性能。

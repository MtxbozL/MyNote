**职级适配**：高级必问、中级进阶

#### 【核心考点】

考察高可用架构的流量治理与风险防控能力，对限流算法的底层理解，生产环境恶意攻击、流量突刺的防护能力，是资深工程师与初级工程师的核心区分点之一。

#### 【专家级标准答案】

Nginx 限流是生产环境流量治理的核心手段，核心作用是**控制客户端请求的速率与并发连接数，防止恶意爬虫、CC 攻击、流量突刺导致后端服务过载宕机**，原生基于两个核心模块实现：`ngx_http_limit_req_module`（请求速率限流）、`ngx_http_limit_conn_module`（连接数限流）。

1. **核心限流方案与标准化配置**
    
    #### 方案 1：请求速率限流（核心常用，基于漏桶算法）
    
    生产环境最主流的限流方案，**严格控制单位时间内客户端的请求数**，基于漏桶算法实现，可应对突发流量，核心是控制请求处理的速率，避免后端服务被突发请求打垮。

```nginx
    # 1. http块中定义限流规则（必须定义在http块，全局生效）
    # 限流维度：基于客户端IP $binary_remote_addr，比$remote_addr节省内存
    # zone：定义共享内存区域，名称req_limit，大小10m，可存储约16万个IP的限流状态
    # rate：限流速率，10r/s表示每秒最多处理10个请求，支持1r/m（每分钟1个请求）
    http {
        limit_req_zone $binary_remote_addr zone=req_limit:10m rate=10r/s;
    
        # 2. server/location块中启用限流，针对业务接口配置
        server {
            server_name example.com;
    
            # 针对核心业务接口限流
            location /api/ {
                # burst：漏桶容量，允许20个突发请求排队；nodelay：突发请求立即处理，不延迟，超过阈值直接拒绝
                limit_req zone=req_limit burst=20 nodelay;
                # 限流响应码，默认503，推荐使用429（Too Many Requests），符合HTTP规范
                limit_req_status 429;
                # 反向代理后端服务
                proxy_pass http://backend_server;
            }
        }
    }
```

核心适用场景：接口防刷、CC 攻击防护、后端服务的请求速率管控，覆盖 90% 的生产限流场景。

#### 方案 2：连接数限流（基于并发连接数）

控制单个客户端同时与 Nginx 建立的并发连接数，**核心用于限制大文件下载、长连接、爬虫高频并发请求**。

```nginx
    # 1. http块中定义连接数限流规则
    http {
        limit_conn_zone $binary_remote_addr zone=conn_limit:10m;
    
        # 2. 启用限流，针对下载场景
        server {
            location /download/ {
                # 限制单个IP最多同时建立10个连接
                limit_conn conn_limit 10;
                limit_conn_status 429;
                # 下载限速，单连接限速100k/s
                limit_rate 100k;
            }
        }
    }
```
    
    核心适用场景：大文件下载、视频点播、长连接 API、高频爬虫防护。
    
2. **生产级限流核心原则**
    
    - 限流维度可扩展：除基于 IP 限流外，可基于（接口地址）、server_name（域名）、用户 ID（请求头 /cookie）等多维度限流，适配不同业务场景；
    - 白名单机制：生产环境必须配置白名单，对内网 IP、监控节点、业务内部服务等放行，避免误拦截；
    - 阈值合理设置：限流阈值必须基于压测数据设置，不可盲目配置，需预留 30% 以上的冗余量；
    - 可观测性：必须开启限流日志，配置监控告警，实时感知限流触发情况，排查误拦截问题。
    

#### 【深度拓展（面试加分项）】

1. 限流算法进阶：Nginx 原生 limit_req 模块基于漏桶算法，核心是 “削峰填谷”，严格控制请求速率；若需支持 “允许流量突发，同时控制平均速率” 的场景，可通过第三方模块实现令牌桶算法，适配秒杀、活动等突发流量场景。
2. 分布式限流：单机 Nginx 限流仅能控制单节点流量，集群环境下可通过 Redis+Lua 模块实现分布式限流，统一控制全集群的请求速率，避免单机限流阈值分散导致后端服务过载。
3. 限流降级策略：生产环境限流触发后，并非直接拒绝请求，可配置降级策略，如返回静态兜底页面、跳转到降级接口、放入队列异步处理，提升用户体验。
4. 多层级限流防护：采用「CDN 边缘限流 + Nginx 接入层限流 + 后端服务接口限流」的三级限流架构，在流量入口处拦截恶意请求，避免恶意流量到达源站。

#### 【面试避坑提示】

1. 禁止使用代替binary_remote_addr，为字符串格式，占用内存更大，binary_remote_addr 为二进制格式，10m 内存可存储约 16 万个 IP 的限流状态，生产环境标准用法。
2. 禁止忽略 burst 与 nodelay 的配合使用，仅配置 burst 不配置 nodelay，会导致突发请求被排队延迟处理，接口响应耗时大幅增加，用户体验极差。
3. 禁止盲目设置过高的限流阈值，阈值必须基于后端服务的压测承载能力设置，否则限流失去意义，无法起到防护作用。
4. 禁止针对静态资源配置限流，静态资源请求不会打到后端服务，限流反而会增加 Nginx 的性能开销。

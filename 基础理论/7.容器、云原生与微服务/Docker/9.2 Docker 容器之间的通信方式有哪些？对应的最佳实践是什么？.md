### 【面试官标准提问】

Docker 容器之间怎么实现互相通信？有哪些方式？生产环境的最佳实践是什么？

### 【专家级作答话术】

Docker 容器之间的通信，核心分为**单主机内的容器通信**和**跨主机的容器通信**两大场景，所有通信方式的底层，都是基于 Linux Network Namespace 的隔离、veth pair 虚拟网卡、bridge 网桥、iptables 路由转发实现的。我分场景给你讲清楚所有的通信方式，以及我们生产环境严格执行的最佳实践。

首先是**单主机内的容器通信（最常用的场景）**，有 4 种实现方式，按官方推荐的优先级排序：

第一种，**自定义 bridge 网络（官方首选最佳实践）**

- 实现方式：创建一个自定义的 bridge 网络，把所有需要互相通信的容器，都加入这个网络。Docker 会为这个网络提供内置的 DNS 域名解析服务，同一个网络内的容器，可以直接通过**容器名 / 服务名**互相访问，不需要配置固定的 IP 地址，也不需要做端口映射。
- 示例：创建一个名为 app-net 的自定义网络，把后端服务 backend 和数据库 mysql 都加入这个网络，backend 服务里可以直接用`mysql:3306`访问数据库，不需要写 IP 地址。
- 核心优势：隔离性好，只有加入同一个网络的容器才能互相通信，不同网络的容器默认隔离；配置简单，不需要维护 IP 地址；安全性高，不需要把服务的端口暴露到宿主机上，减少攻击面；支持容器的动态启停，IP 变化不影响通信。
- 最佳实践：**单主机内的容器通信，必须使用自定义 bridge 网络**，这是我们生产环境的强制规范，绝对不允许用默认 bridge 网络通信。

第二种，**host 网络模式**

- 实现方式：所有使用 host 模式的容器，都和宿主机共享同一个网络栈，直接使用宿主机的 IP 和端口，容器之间可以通过`localhost+端口`直接互相访问。
- 适用场景：仅适用于对网络性能要求极高的单体应用，或者边车代理场景，普通业务容器之间的通信，绝对不推荐使用这个方式，因为隔离性极差，端口冲突风险极高，安全性差。

第三种，**container 网络模式**

- 实现方式：多个容器共享同一个 Network Namespace，共享同一个 IP 和网络栈，容器之间可以通过`localhost`直接互相访问，不需要经过网桥转发。
- 适用场景：仅适用于 Sidecar 边车模式，比如主业务容器和日志采集、监控代理容器共享网络栈，普通的多服务通信，不推荐使用这个方式，配置繁琐，可维护性差。

第四种，**默认 bridge 网络（官方不推荐）**

- 实现方式：所有容器默认都加入 docker0 网桥，容器之间只能通过**IP 地址**互相通信，没有内置的 DNS 解析，需要手动查看容器的 IP 地址，配置到应用里。
- 劣势：没有隔离性，所有容器都在同一个网段，互相都能访问；IP 地址会随容器重启变化，维护成本极高；需要把端口暴露到宿主机上，攻击面大。
- 最佳实践：生产环境绝对禁止使用默认 bridge 网络进行容器间通信，仅用于临时测试场景。

然后是**跨主机的容器通信（集群场景）**，有 3 种主流实现方式，按生产环境的使用频率排序：

第一种，**overlay 网络模式（Docker 原生跨主机方案）**

- 实现方式：创建一个 overlay 覆盖网络，把集群内的所有 Docker 节点都加入这个网络，不同宿主机上的容器，都在同一个虚拟网段内，可以直接通过容器名 / 服务名互相通信，底层通过 VXLAN 隧道实现跨主机的数据转发，不需要做端口映射。
- 适用场景：Docker Swarm 集群，小规模的跨主机容器部署，配置简单，原生支持，不需要额外的插件。

第二种，**CNI 网络插件（生产环境主流方案）**

- 实现方式：使用 Calico、Flannel 等成熟的 CNI（容器网络接口）插件，实现跨主机容器的三层路由通信，Calico 基于 BGP 路由协议，Flannel 基于 VXLAN 隧道，都支持大规模集群，提供了网络策略、访问控制、隔离性等企业级能力。
- 适用场景：Kubernetes 大规模容器集群，这是现在生产环境的主流方案，我们的 K8s 集群用的就是 Calico 插件，支持跨主机容器通信，同时可以配置网络策略，实现细粒度的访问控制。

第三种，**端口映射 + 宿主机 IP 通信（小规模场景）**

- 实现方式：把容器的端口映射到宿主机上，容器之间通过「宿主机 IP + 映射端口」互相通信，不管是不是在同一个宿主机上。
- 优势：配置简单，不需要额外的网络配置；劣势：管理繁琐，需要维护所有宿主机的 IP 和端口映射，端口冲突风险高，安全性差，攻击面大。
- 适用场景：只有 2-3 台宿主机的超小规模场景，生产环境大规模集群绝对不推荐使用。

最后总结一下核心最佳实践：

- 单主机内的容器通信：无脑用**自定义 bridge 网络**，通过服务名 / 容器名通信
- 跨主机 Swarm 集群：用**overlay 网络**
- 跨主机 K8s 集群：用**Calico/Flannel 等 CNI 插件**
- 其他所有场景，都不推荐使用

### 【面试避坑提醒】

❌ 绝对不要说「单主机通信用默认 bridge 网络就可以」，这是严重违背官方最佳实践的，会被认为经验不足

❌ 不要说跨主机通信只能用端口映射，这是非常初级的认知，面试官会直接追问更优的方案

❌ 不要混淆自定义 bridge 和 overlay 网络，bridge 是单主机的，overlay 是跨主机的

### 【面试官高频追问 & 应答要点】

1. 追问：容器里怎么访问宿主机上的服务？
    
    应答要点：有 3 种可靠的方式，第一种，Docker 18.03 + 版本，直接使用内置的 DNS 域名`host.docker.internal`，这个域名会自动解析到宿主机的 IP，这是最推荐的方式，跨平台兼容；第二种，使用宿主机的网关 IP，容器里通过`ip route show`查看 default 网关，就是宿主机的 IP；第三种，容器使用 host 网络模式，直接用[localhost](https://localhost)访问宿主机服务，不推荐这种方式，隔离性差。
2. 追问：两个容器分别在两个不同的自定义 bridge 网络里，怎么让它们互相通信？
    
    应答要点：有两种合规的方式，第一种，把两个容器都加入到对方的网络里，一个容器可以加入多个网络，这样两个容器就可以在同一个网络里通信；第二种，创建一个第三方的公共网络，把两个需要通信的容器都加入这个公共网络，同时保留它们原来的网络，实现隔离和互通的平衡，绝对不要直接把两个网桥打通，会破坏网络隔离。

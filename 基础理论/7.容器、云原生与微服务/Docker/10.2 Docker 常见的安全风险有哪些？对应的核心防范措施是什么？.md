### 【面试官标准提问】

Docker 容器有哪些常见的安全风险？生产环境你怎么加固 Docker，防范这些安全问题？

### 【专家级作答话术】

Docker 容器的安全，核心遵循**最小权限原则、最小攻击面原则、纵深防御原则**，所有的安全风险，本质都是打破了容器的隔离性，或者扩大了攻击面。我把生产环境最常见的安全风险，分为 5 大类，每一类都对应我们已经落地的核心防范措施，同时补充我们踩过的坑和行业内的真实案例。

第一大类，也是最严重的风险：**容器逃逸风险**

- 风险本质：攻击者通过容器内的漏洞或者错误配置，突破了 Namespace 的隔离，获取了宿主机的 root 权限，控制整个宿主机，甚至整个集群，这是 Docker 最致命的安全风险。
- 常见的风险点：
    
    1. 使用`--privileged`特权容器启动，容器内的 root 用户拥有和宿主机 root 完全一样的权限，直接可以挂载宿主机磁盘，修改宿主机配置，甚至加载内核模块
    2. 挂载了宿主机的根目录、/etc、/root、/dev、/var/run/docker.sock 等敏感目录，攻击者可以直接修改宿主机的 ssh 配置、写入定时任务，甚至通过 docker.sock 控制整个宿主机的 Docker daemon
    3. Linux Capabilities 权限配置过大，默认的容器已经禁用了大部分高危权限，但如果手动添加了 SYS_ADMIN、NET_ADMIN 等高危权限，就可能导致逃逸
    4. 宿主机内核、Docker 版本存在已知的逃逸漏洞，比如脏牛、CVE-2024-21626 等
    
- 核心防范措施（生产环境强制规范）：
    
    1. **绝对禁止使用 --privileged 特权容器**，没有任何例外，这是我们的红线规范，之前行业内有大量的挖矿事件，都是因为使用了特权容器，被攻击者逃逸后植入了挖矿程序
    2. 严格控制目录挂载，仅挂载业务必需的目录，绝对禁止挂载宿主机的根目录、敏感系统目录、docker.sock 套接字，生产环境的挂载目录必须经过安全审核
    3. 最小化 Capabilities 权限，使用`--cap-drop=ALL`默认禁用所有权限，只添加业务运行必需的最小权限，比如只给需要绑定 80 端口的容器添加 NET_BIND_SERVICE 权限，其他所有权限全部禁用
    4. 定期更新宿主机内核、Docker 版本，及时修复已知的高危安全漏洞，建立漏洞扫描和应急响应机制
    

第二大类风险：**镜像安全风险**

- 风险本质：使用了包含恶意代码、高危漏洞的镜像，导致容器被植入后门、挖矿程序、木马，甚至被攻击者控制，这是生产环境最高频的安全风险。
- 常见的风险点：
    
    1. 使用了来源不明的第三方镜像，比如 Docker Hub 上个人上传的镜像，可能被植入了恶意代码
    2. 镜像存在高危漏洞，比如 Log4j2、OpenSSL 等组件的远程代码执行漏洞，攻击者可以利用漏洞入侵容器
    3. 镜像没有做签名验证，被中间人攻击篡改，植入恶意代码
    4. 镜像使用 root 用户构建，包含大量不必要的工具，攻击面极大
    
- 核心防范措施：
    
    1. 仅使用 Docker 官方镜像、软件厂商官方镜像等可信来源的基础镜像，绝对禁止使用来源不明的第三方镜像
    2. 建立镜像全生命周期安全管控：在 CI/CD 流水线中，强制使用 Trivy、Clair 等镜像扫描工具，对镜像进行漏洞扫描、恶意代码检测，存在高危漏洞的镜像，不允许合并代码，不允许推送到生产仓库
    3. 开启镜像签名验证，使用 Docker Content Trust（DCT），只允许运行经过签名的可信镜像，防止镜像被篡改
    4. 最小化镜像体积，使用 distroless、alpine 等极简基础镜像，剔除镜像里的 shell、包管理工具、不必要的系统命令，减少攻击面
    

第三大类风险：**权限过大风险**

- 风险本质：容器内的应用以 root 用户运行，一旦容器被入侵，攻击者就拥有了容器内的 root 权限，更容易进行提权、逃逸操作，风险极大。
- 常见的风险点：
    
    1. 容器默认以 root 用户运行，没有指定非 root 用户
    2. 容器内的 root 用户没有做权限限制，拥有完整的 root 权限
    
- 核心防范措施：
    
    1. 严格遵循最小权限原则，在 Dockerfile 中通过 USER 指令，创建普通用户，指定应用以非 root 用户运行，这是我们生产环境的强制规范，所有镜像必须使用非 root 用户
    2. 开启 rootless Docker 模式，Docker daemon 以非 root 用户运行，即使容器被入侵，也无法获取宿主机的 root 权限，进一步提升安全性
    3. 使用用户命名空间（User Namespace），把容器内的 root 用户，映射到宿主机上的普通用户，容器内的 root 权限，在宿主机上只有普通用户的权限，即使逃逸，也无法获取宿主机的 root 权限
    

第四大类风险：**资源耗尽拒绝服务风险**

- 风险本质：没有对容器设置资源限制，单个容器可以占用宿主机所有的 CPU、内存、磁盘 IO、文件句柄资源，导致宿主机宕机，所有业务中断，也就是 DoS 拒绝服务攻击。
- 常见的风险点：
    
    1. 没有给容器设置 CPU、内存限制，容器被攻击后，恶意程序占用所有资源，导致宿主机 OOM
    2. 没有限制容器的文件句柄数、进程数，攻击者通过 fork 炸弹，耗尽宿主机的资源
    
- 核心防范措施：
    
    1. 生产环境所有容器，必须设置 CPU、内存资源限制，通过`--cpus`限制 CPU 核心数，`--memory`限制内存上限，防止容器占用过多资源，我们的规范是，单个容器的资源限制，不能超过宿主机资源的 30%
    2. 限制容器的最大进程数`--pids-limit`，防止 fork 炸弹；限制文件句柄数、磁盘 IO 权重，防止资源耗尽
    3. 监控容器的资源使用情况，设置告警阈值，超过阈值自动告警、隔离容器
    

第五大类风险：**网络安全风险**

- 风险本质：容器端口过度暴露、网络隔离不足，导致容器被扫描、攻击，攻击者可以横向渗透，控制整个集群。
- 常见的风险点：
    
    1. 把容器的管理端口、数据库端口直接暴露到公网，被攻击者扫描爆破
    2. 没有做网络隔离，所有容器都在同一个网络里，一个容器被入侵，攻击者可以横向访问所有容器
    3. 容器之间的通信没有加密，数据明文传输，被中间人窃听
    
- 核心防范措施：
    
    1. 最小化端口暴露，仅暴露业务必需的端口，绝对禁止把数据库、缓存、管理端口暴露到公网，公网只暴露 80/443 等业务端口
    2. 按业务线、安全级别创建独立的自定义网络，实现网络隔离，不同安全级别的容器，不能在同一个网络里；通过 iptables、网络策略，限制容器的南北向、东西向访问，只允许必要的通信
    3. 容器之间的敏感通信，使用 TLS 加密，比如数据库连接、API 调用，防止数据明文传输
    4. 开启 Docker daemon 的 TLS 远程访问认证，禁止 Docker daemon 监听公网端口，只允许本地访问，必须远程访问的话，必须配置双向 TLS 认证，禁止无密码的远程访问
    

最后补充一个纵深防御的最佳实践：我们生产环境会开启 AppArmor/SELinux 安全模块，给容器配置强制访问控制策略，限制容器内的进程能执行的操作；同时使用 Falco 等运行时安全监控工具，实时监控容器内的异常行为，比如特权操作、敏感文件访问、恶意进程启动，发现异常立即告警、隔离容器，实现全生命周期的安全防护。

### 【面试避坑提醒】

❌ 绝对不要说「生产环境可以用特权容器，方便操作」，这是严重的安全红线，会被面试官直接否定

❌ 不要说「容器和虚拟机一样安全」，容器的隔离性比虚拟机弱，必须正视安全风险，不能盲目乐观

❌ 不要只讲风险，不讲对应的防范措施，面试官问这个问题，核心是看你有没有生产环境的安全加固能力，而不是看你能不能背出风险点

### 【面试官高频追问 & 应答要点】

1. 追问：挂载 /var/run/docker.sock 有什么风险？
    
    应答要点：docker.sock 是 Docker daemon 的 Unix 套接字，拥有这个套接字的读写权限，就等于拥有了宿主机 Docker daemon 的完全控制权，可以创建、删除、修改所有容器，挂载宿主机的任意目录，甚至获取宿主机的 root 权限，直接导致容器逃逸。生产环境绝对禁止把 docker.sock 挂载到容器内，除非是必须的监控、运维工具，而且必须严格控制权限，做安全审计。
2. 追问：怎么实现容器的运行时安全监控？
    
    应答要点：可以通过 3 个层面实现，第一，内核层面，用 eBPF 工具监控容器内的系统调用，发现异常的特权操作、文件访问；第二，用户态层面，用 Falco 等开源工具，实时监控容器内的进程启动、文件修改、网络连接等行为，配置规则，发现恶意行为立即告警；第三，监控层面，通过 Prometheus+Grafana 监控容器的资源使用、网络流量、进程数，发现异常的资源占用、网络连接，及时告警和隔离。

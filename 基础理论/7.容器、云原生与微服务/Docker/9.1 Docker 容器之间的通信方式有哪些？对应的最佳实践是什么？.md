**标准答案**：

Docker 容器通信分为单主机内通信与跨主机通信两大类，具体方式与最佳实践如下：

1. **单主机内的容器通信（最常用）**
    
    - 方式 1：**自定义 bridge 网络（官方推荐最佳实践）**：将所有需要通信的容器加入同一个自定义 bridge 网络，Docker 内置 DNS 服务，容器之间可直接通过**容器名 / 服务名**互相访问，无需配置 IP，隔离性好、配置简单、安全性高。
    - 方式 2：host 网络模式：所有使用 host 模式的容器共享宿主机网络，直接通过[localhost](https://localhost)+ 端口互相访问，性能高，但隔离性差，端口冲突风险高。
    - 方式 3：container 网络模式：多个容器共享同一个容器的网络栈，通过[localhost](https://localhost)互相通信，适用于边车模式的容器组合。
    - 方式 4：默认 bridge 网络：容器间只能通过 IP 地址通信，无 DNS 解析，隔离性差，官方不推荐生产环境使用。
    
2. **跨主机的容器通信**
    
    - 方式 1：**overlay 网络模式**：Docker 原生跨主机网络，所有集群节点的容器加入同一个 overlay 网络，可直接通过容器名 / 服务名通信，无需端口映射，适配 Docker Swarm 集群。
    - 方式 2：端口映射 + 宿主机 IP 通信：将容器端口映射到宿主机，通过宿主机的 IP + 端口实现跨主机容器通信，配置简单，但管理繁琐，适合小规模场景。
    - 方式 3：第三方网络插件：配合 Calico、Flannel 等 CNI 网络插件，实现跨主机容器的网络互通，适配 Kubernetes 等大规模容器集群。
    

**核心最佳实践**：单主机通信优先使用自定义 bridge 网络，跨主机集群通信优先使用 overlay 网络或成熟的 CNI 插件，禁止使用默认 bridge 网络进行容器间通信。

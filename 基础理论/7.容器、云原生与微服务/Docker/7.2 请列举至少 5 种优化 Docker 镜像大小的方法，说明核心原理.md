### 【面试官标准提问】

你怎么优化 Docker 镜像的大小？有哪些方法？核心原理是什么？

### 【专家级作答话术】

Docker 镜像优化的核心目标有两个：**一是缩小镜像体积，提升镜像拉取、推送的速度，减少存储占用；二是减少镜像的攻击面，提升镜像的安全性**。优化的核心底层逻辑，是基于 Docker 镜像的**UnionFS 联合文件系统、分层存储、写时复制机制**，核心原则是：减少镜像层数、剔除不必要的文件、最大化利用构建缓存、最小化基础镜像体积。

我们生产环境有严格的镜像构建规范，优化后的镜像体积，比优化前平均缩小了 80% 以上，我给你讲 6 种最核心、可落地的优化方法，从核心到次要排序：

第一种，也是**最核心、效果最明显的优化方法：多阶段构建（Multi-stage builds）**

- 核心原理：把镜像的「构建环境」和「运行环境」完全分离，在 Dockerfile 里定义多个构建阶段。第一阶段是构建阶段，使用包含完整构建工具的基础镜像（比如 Go SDK、JDK、Node.js），完成代码的编译、依赖安装、打包构建；第二阶段是运行阶段，只使用极简的运行时基础镜像，只把第一阶段构建完成的、最终可运行的产物（比如二进制文件、编译后的代码、运行依赖）复制到运行镜像里，彻底剔除构建工具、中间产物、源码、临时文件、测试代码等所有冗余内容。
- 实战效果：比如 Go 语言应用，用多阶段构建后，镜像体积从 300MB + 缩小到 10MB 以内，缩小了 97%；Java SpringBoot 应用，从 800MB + 缩小到 200MB 以内，缩小了 75%。这是我们生产环境所有应用必须使用的构建方式，没有例外。

第二种，**选用极简的基础镜像**

- 核心原理：基础镜像占了镜像体积的 90% 以上，替换大体积的全功能基础镜像，用极简的基础镜像，可以直接把镜像体积降低 90% 以上。
- 主流的极简基础镜像选型，按体积从小到大排序：
    
    1. scratch 镜像：空镜像，体积为 0，只能存放静态编译的二进制文件，比如 Go 语言编译后的二进制文件，没有任何额外的系统工具，攻击面最小，安全性最高
    2. distroless 镜像：Google 出品的无发行版镜像，只包含应用运行所需的最核心依赖，没有 shell、包管理工具、系统命令，体积比 alpine 还小，攻击面极小，适合 Java、Node.js 等需要运行时的应用
    3. alpine 镜像：基于 Alpine Linux 的精简镜像，体积只有 5MB 左右，自带精简的包管理工具 apk，兼容性好，是现在最常用的极简基础镜像
    
- 最佳实践：绝对不要用 ubuntu、centos 这种全功能镜像作为生产环境的运行基础镜像，它们的体积超过 200MB，包含大量不必要的工具，不仅体积大，攻击面也大，我们生产环境已经全面淘汰了 centos/ubuntu 作为运行基础镜像。

第三种，**合并 RUN 指令，减少镜像层数**

- 核心原理：Dockerfile 里，每一条 RUN、COPY、ADD 指令，都会生成一个新的镜像层，镜像层是只读的，一旦生成就无法修改，后续的指令只能在上面叠加新的层，无法删除之前层里的文件。所以，镜像层数越多，镜像体积越大。
- 实战方法：把多个连续的 RUN 命令，通过`&&`合并成一条指令，同时在同一条 RUN 指令里，完成「安装依赖→使用依赖→清理缓存」的全流程，避免缓存文件被提交到镜像层里。
- 反例和正例对比：
    
    反例（3 层镜像，缓存无法清理）：
    
    dockerfile
    
    ```
    RUN apt update
    RUN apt install -y nginx
    RUN rm -rf /var/lib/apt/lists/*
    ```
    
    正例（1 层镜像，缓存彻底清理）：
    
    dockerfile
    
    ```
    RUN apt update && \
        apt install -y nginx && \
        rm -rf /var/lib/apt/lists/*
    ```
    
- 补充：Docker 现在已经放开了镜像层数的限制，但我们生产环境的规范是，Dockerfile 的指令越少越好，层数越少越好，避免不必要的层叠加。

第四种，**清理冗余依赖和缓存，剔除不必要的文件**

- 核心原理：镜像里的包管理缓存、临时文件、无用依赖、文档、测试代码、debug 符号，都会占用大量的体积，必须在构建的同一条 RUN 指令里彻底清理。
- 实战落地方法：
    
    1. 安装依赖时，使用`--no-install-recommends`参数，只安装必须的依赖，不安装推荐的可选依赖，比如`apt install --no-install-recommends -y nginx`
    2. 安装完成后，立即清理包管理缓存，比如 apt 的`/var/lib/apt/lists/*`，yum 的`yum clean all`，npm 的`npm cache clean --force`，pip 的`pip cache purge`
    3. 剔除镜像里的文档、man 手册、debug 符号、测试代码、示例文件，这些完全不会在运行时使用
    4. 前端应用，构建完成后只保留 dist 目录，剔除 node_modules 目录，这是前端镜像体积大的核心原因
    

第五种，**使用.dockerignore 文件，优化构建上下文**

- 核心原理：Docker 构建镜像时，会把当前目录下的所有文件，作为构建上下文，发送给 Docker Daemon，如果有大量不需要的文件（比如 node_modules、.git、本地日志、临时文件、测试数据），会导致构建上下文过大，构建速度变慢，甚至会被 COPY 指令复制到镜像里，增大镜像体积。
- 最佳实践：每个项目必须有.dockerignore 文件，排除所有构建时不需要的文件，比如.git、node_modules、.vscode、*.log、tmp 目录、测试代码、本地配置文件，和.gitignore 配合使用，我们生产环境要求，构建上下文必须控制在 10MB 以内，否则不允许合并代码。

第六种，**优化 Dockerfile 指令顺序，最大化利用构建缓存**

- 核心原理：Docker 构建镜像时，会按顺序执行 Dockerfile 的指令，如果当前指令的内容和之前的构建没有变化，就会复用之前的缓存层，不会重新构建。如果某一层的缓存失效了，后面所有层的缓存都会全部失效。
- 实战方法：把**不常变动的指令放在前面，频繁变动的指令放在后面**。比如，先复制依赖配置文件（package.json、pom.xml、go.mod），然后安装依赖，最后再复制应用代码。这样，只有当依赖配置文件变化时，才会重新安装依赖，代码变动时，只会重新构建最后一层，最大化利用缓存，同时避免不必要的层重建。

### 【面试避坑提醒】

❌ 绝对不要说「把所有指令都合并成一条，层数越少越好」，过度合并会导致构建缓存完全失效，每次构建都要重新执行所有指令，构建速度会变得极慢，要平衡层数和缓存的关系

❌ 不要说 alpine 镜像适合所有场景，alpine 用的是 musl libc，和 glibc 有兼容性问题，比如 Java 应用用 alpine 可能会出现性能问题，要讲清楚选型的注意事项

❌ 不要忽略.dockerignore 的作用，这是面试中高频的扣分点，很多候选人完全不知道这个文件的作用

### 【面试官高频追问 & 应答要点】

1. 追问：怎么查看 Docker 镜像的分层内容？怎么知道镜像里的大文件在哪里？
    
    应答要点：用`docker history 镜像名`可以查看镜像的每一层的大小和对应的指令，定位大体积的层；用`dive`这个开源工具，可以交互式地查看镜像的每一层的文件内容，定位大文件和冗余文件，这是我们生产环境优化镜像的常用工具。
2. 追问：镜像构建速度很慢，你会怎么优化？
    
    应答要点：第一，优化.dockerignore，缩小构建上下文；第二，优化 Dockerfile 指令顺序，最大化利用构建缓存；第三，使用多阶段构建，分离构建和运行环境；第四，使用国内的镜像源，加快依赖下载速度；第五，使用构建缓存挂载，比如 RUN --mount=type=cache，缓存依赖包，避免每次构建都重新下载。

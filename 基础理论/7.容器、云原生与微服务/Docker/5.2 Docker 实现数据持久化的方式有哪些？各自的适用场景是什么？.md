### 【面试官标准提问】

Docker 容器是无状态的，你怎么实现容器的数据持久化？有几种方式？分别适合什么场景？生产上你怎么选？

### 【专家级作答话术】

首先，Docker 数据持久化的核心本质，是**把容器内的数据目录，挂载到宿主机的文件系统上，实现数据与容器生命周期的解耦**，容器销毁、重建，数据都不会丢失。Docker 官方提供了 3 种核心的持久化方案，按照官方推荐的优先级排序是：**Volumes（卷）> Bind Mounts（绑定挂载）> tmpfs Mounts（临时文件系统）**，我分别讲一下每种方案的特点、适用场景，以及我们生产上的选型规范。

第一种，也是**官方首选的方案：Docker Volumes（Docker 卷）**

- 核心特点：由 Docker daemon 完全管理，存储在宿主机的 Docker 专属目录`/var/lib/docker/volumes/`下，非 Docker 进程无法修改这个目录，生命周期完全独立于容器，容器删除后，卷依然存在，不会被清理。它分为具名卷和匿名卷，具名卷有固定的名字，方便管理和复用，匿名卷是随机生成的名字，通常用于临时场景。
- 核心优势：第一，完全由 Docker 管理，不会和宿主机的系统目录产生耦合，不会出现权限问题；第二，支持多种驱动，比如本地驱动、NFS 网络驱动、云存储驱动，支持跨主机共享数据；第三，支持备份、恢复、迁移，操作非常简单；第四，和 Docker Desktop、Linux、Windows 都完全兼容，移植性最好。
- 适用场景：**生产环境的首选方案**，所有需要持久化的业务数据，比如 MySQL、PostgreSQL 数据库的数据目录、应用的日志文件、用户上传的文件、需要跨容器共享的数据，全部用 Volumes。我们生产环境所有有状态的容器，全部用的是具名卷，没有例外。

第二种，**Bind Mounts（绑定挂载）**

- 核心特点：把宿主机上的任意文件、目录，直接挂载到容器内，宿主机和容器可以双向读写，权限完全开放，宿主机上的修改会实时同步到容器内，容器内的修改也会实时同步到宿主机上。
- 核心优势：灵活性极高，用户可以完全控制挂载的目录，开发环境下非常方便，比如把本地的代码目录挂载到容器内，本地修改代码，容器内的应用可以实时生效，不需要重新构建镜像。
- 核心劣势：第一，和宿主机的目录强耦合，移植性差，不同的宿主机目录结构不一样，挂载命令就不一样；第二，权限问题非常多，比如容器内的用户 UID/GID 和宿主机的不匹配，会出现读写权限拒绝的问题，我们生产上踩过很多次这个坑；第三，有安全风险，比如挂载了宿主机的`/etc`、`/root`等敏感目录，可能会导致容器逃逸。
- 适用场景：**仅推荐开发环境使用**，比如本地开发、代码热更新、配置文件实时修改；生产环境只有一种场景可以用，就是宿主机上的固定配置文件，需要挂载到容器内，比如 nginx 的配置文件，除此之外，生产环境禁止使用绑定挂载。

第三种，**tmpfs Mounts（临时文件系统挂载）**

- 核心特点：把数据挂载到宿主机的内存中，不会写入宿主机的磁盘文件系统，容器停止后，数据会立即丢失，完全无法恢复。
- 核心优势：读写性能极高，延迟极低，因为是直接操作内存，同时不会在宿主机上留下任何数据，安全性高。
- 适用场景：**不需要持久化的临时数据、敏感数据**，比如高并发场景的临时缓存、会话数据、加密密钥、临时计算结果，不需要落地到磁盘，容器停止后自动清除，保证数据安全。

最后补充一个生产上的最佳实践：我们会给所有的卷做定期备份，用`docker run --rm -v 卷名:/source -v 宿主机备份目录:/target alpine tar -czf /target/backup.tar.gz -C /source .`这个命令，定期把卷的数据备份到对象存储里，防止数据误删。

### 【面试避坑提醒】

❌ 绝对不要说「绑定挂载比卷更好用，生产上都用绑定挂载」，这是严重违背 Docker 官方最佳实践的，会直接扣分

❌ 不要说 tmpfs 可以持久化数据，tmpfs 是临时存储，容器停止后数据就丢失了

❌ 不要忽略绑定挂载的权限问题，这是生产上最高频的坑，面试官一定会追问

### 【面试官高频追问 & 应答要点】

1. 追问：具名卷和匿名卷有什么区别？生产上怎么选？
    
    应答要点：具名卷有用户指定的固定名称，生命周期独立，不会随容器删除而自动清理，方便管理、复用、备份，生产环境必须用具名卷；匿名卷是 Docker 随机生成的名称，没有指定名字，默认随容器删除而清理，不方便管理，仅用于临时测试场景，生产环境禁止使用。
2. 追问：容器内的应用写入数据很慢，你会怎么排查？
    
    应答要点：第一步，先看数据目录是不是用了绑定挂载，宿主机的磁盘 IO 是不是有瓶颈；第二步，看容器的 Cgroup 有没有限制 IO，导致写入被限流；第三步，看存储驱动是不是用了 overlay2，有没有用不推荐的 devicemapper；第四步，看是不是容器内的日志写入量太大，导致磁盘占满。

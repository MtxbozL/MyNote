### 【面试官标准提问】

讲一下 Docker 的核心架构，有哪些核心组件，分别是做什么的，它们之间是怎么协同工作的？

### 【专家级作答话术】

Docker 采用的是标准的**C/S（客户端 - 服务端）架构**，核心组件分为客户端、守护进程、容器运行时、镜像、仓库 5 大部分，同时现在的 Docker 已经适配了 OCI（开放容器标准），底层依赖 containerd 和 runc，整个架构是解耦的。

我先讲每个核心组件的作用，再给你讲一个`docker run`命令的完整执行流程，你就能清楚它们之间的协同关系：

1. **Docker Client（客户端）**：就是我们日常用的 docker CLI 命令行工具，是用户和 Docker 交互的唯一入口。它的核心作用是接收用户输入的 docker 命令，然后把请求转换成 REST API，发送给 Docker Daemon 守护进程。它和守护进程可以在同一台机器，也可以远程连接生产环境的守护进程。
2. **Docker Daemon（dockerd，守护进程）**：运行在宿主机后台的常驻服务，是 Docker 的大脑。它的核心作用是持续监听 Docker API 请求，然后管理宿主机上的容器、镜像、网络、数据卷所有核心资源，同时它会和底层的 containerd 交互，完成容器的生命周期管理。
3. **Containerd（容器运行时管理器）**：现在已经捐给 CNCF 托管，是 Docker 架构中负责容器生命周期管理的核心组件，承接来自 dockerd 的请求，负责镜像的拉取解压、容器的创建启停、网络和存储的管理，同时它会调用 runc 来创建和运行容器。
4. **Runc（OCI 运行时规范实现）**：是 OCI 容器运行时标准的参考实现，本质是一个轻量级的 CLI 工具，不常驻后台。它的核心作用是直接调用 Linux 内核的 Namespace、Cgroup、Capability 等系统接口，完成容器的创建、启动、销毁等底层操作，是真正和内核交互的组件。
5. **Docker Image（镜像）**：是容器的只读模板，包含了应用运行所需的所有内容：代码、运行环境、依赖库、配置文件、环境变量、启动命令。它采用 UnionFS 联合文件系统，是分层存储的，支持分层复用和缓存，是容器创建的基础。
6. **Docker Container（容器）**：是镜像的运行实例，本质是宿主机上一个被 Namespace 隔离、Cgroup 限制资源的独立进程，拥有自己的根文件系统、网络栈、进程空间，是应用真正的运行载体。
7. **Docker Registry（镜像仓库）**：是镜像的存储和分发中心，分为公共仓库（比如 Docker Hub、阿里云 ACR）和私有仓库，负责镜像的版本管理、共享和分发，我们构建的镜像可以推送到仓库，其他机器可以直接拉取运行。

举个最常用的`docker run nginx`命令，整个协同流程是：

- 我在客户端输入命令，Docker Client 把请求发送给 Docker Daemon
- Daemon 先检查本地有没有 nginx 镜像，没有的话就通过 containerd 从 Registry 拉取镜像到本地
- 镜像拉取完成后，Daemon 把创建容器的请求发给 containerd
- containerd 创建对应的容器配置，调用 runc，runc 调用内核接口创建 Namespace 和 Cgroup，启动容器内的 nginx 进程
- 最终把容器启动成功的结果返回给客户端

### 【核心加分亮点】

1. 补充了 containerd 和 runc 两个核心底层组件，符合现在 Docker 的最新架构，而不是停留在老旧的架构认知，这是 90% 的候选人都会忽略的点
2. 用一个完整的`docker run`流程把所有组件串联起来，而不是孤立地讲每个组件，证明你真的懂整个架构的运行逻辑
3. 提到了 OCI 开放容器标准、UnionFS 分层存储、Namespace/Cgroup 等核心概念，体现技术深度

### 【面试避坑提醒】

❌ 不要把 containerd 和 runc 的作用搞反，runc 是负责和内核交互的底层运行时，containerd 是上层的管理器

❌ 不要说 Docker Daemon 直接创建容器，现在的 Docker 已经把容器运行时的能力解耦给了 containerd 和 runc

❌ 不要混淆镜像和容器的关系，镜像只读，容器是镜像的运行实例，有可写层

### 【面试官高频追问 & 应答要点】

1. 追问：什么是 OCI 标准？为什么要有这个标准？
    
    应答要点：OCI 是开放容器标准，由 Linux 基金会主导，核心分为镜像规范和运行时规范，目的是统一容器的格式和运行标准，避免厂商锁定，让不同的容器 runtime（比如 Docker、containerd、CRI-O）都能运行符合标准的容器镜像，实现生态的兼容。
2. 追问：Docker Daemon 是单进程的，会不会有单点故障问题？
    
    应答要点：确实存在这个问题，Docker Daemon 挂掉之后，宿主机上所有的容器都会失去管理，这也是生产环境大规模容器编排会用 K8s 替代原生 Docker 的核心原因之一，K8s 的 kubelet 是节点级的，不会有全局的单点故障。

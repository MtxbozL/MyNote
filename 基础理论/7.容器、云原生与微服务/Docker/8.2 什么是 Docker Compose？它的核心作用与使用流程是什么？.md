## ### 【面试官标准提问】

什么是 Docker Compose？它是用来做什么的？你在生产环境中怎么用它？

### 【专家级作答话术】

首先，Docker Compose 是 Docker 官方推出的**单机多容器应用的定义与编排工具**，核心解决的问题是：一个完整的业务应用，通常由多个互相依赖的服务组成（比如前端、后端、数据库、缓存、消息队列），如果用原生的 docker 命令，需要逐个创建容器、配置网络、设置依赖关系、管理端口映射，操作非常繁琐，而且无法保证环境一致性。而 Docker Compose 可以通过一个**单一的 YAML 配置文件（compose.yaml）**，把应用所需的所有服务、网络、数据卷全部定义好，然后用一条命令，完成所有服务的创建、启动、停止、销毁全生命周期管理，真正实现「一键部署」。

它的核心作用，我结合我们生产环境的用法，总结为 4 点：

1. **简化多容器应用的部署与管理**：把多个容器的复杂配置，收敛到一个 YAML 文件里，不需要写复杂的 shell 脚本，一条`docker compose up`命令就能启动整个应用，一条`docker compose down`就能停止并清理所有资源，运维成本极低。
2. **解决服务依赖管理问题**：可以通过`depends_on`配置服务的启动顺序，比如先启动数据库和缓存，再启动后端应用，避免应用启动时因为依赖的服务没就绪而启动失败。
3. **保证多环境的配置一致性**：开发、测试、生产环境使用同一个 compose.yaml 文件，通过环境变量、覆盖配置文件（compose.override.yaml）区分不同环境的配置，彻底解决「开发环境能跑，测试 / 生产环境跑不起来」的问题，我们的开发环境和测试环境，就是用同一个 Compose 配置，保证环境 100% 一致。
4. **实现单机环境的服务隔离与网络互通**：Compose 会为每个项目创建独立的自定义 bridge 网络，同一个项目的所有服务，都加入这个网络，通过服务名直接互相通信，不同项目的服务之间默认隔离，不会互相干扰，安全性和隔离性都非常好。

接下来是 Docker Compose 的标准使用流程，我们生产环境有严格的规范，分为 4 步：

1. **编写每个服务的 Dockerfile**：为应用的每个服务（前端、后端）编写对应的 Dockerfile，定义服务的运行环境、构建流程、启动命令，保证每个服务的镜像可重复构建。
2. **编写 compose.yaml 配置文件**：在文件里定义组成应用的所有服务，包括每个服务的镜像构建规则、端口映射、数据卷挂载、环境变量、依赖关系、网络配置、重启策略、健康检查等所有运行参数。
3. **配置环境区分**：通过`.env`文件配置环境变量，通过`compose.override.yaml`文件覆盖不同环境的配置，比如开发环境挂载本地代码目录实现热更新，生产环境使用固定的镜像版本、配置资源限制。
4. **执行命令管理应用生命周期**：
    
    - 开发环境：`docker compose up --build`，构建镜像并启动所有服务，实时查看日志
    - 测试 / 生产环境：`docker compose up -d`，后台启动所有服务，不占用终端
    - 停止应用：`docker compose stop`
    - 销毁应用：`docker compose down`，停止并删除所有容器、网络，保留数据卷
    - 查看服务状态：`docker compose ps`
    - 查看服务日志：`docker compose logs -f 服务名`
    

最后补充两个生产环境的最佳实践和踩过的坑：

第一个坑：`depends_on`只能保证服务的启动顺序，不能保证服务的就绪状态。比如数据库容器启动了，但数据库还没完成初始化，后端应用就启动了，会导致连接数据库失败。我们的解决方案是，给所有服务配置`healthcheck`健康检查，同时用`condition: service_healthy`配置依赖，只有当依赖的服务健康检查通过后，才会启动当前服务，彻底解决这个问题。

第二个最佳实践：生产环境必须给每个服务配置 CPU 和内存的资源限制，通过`deploy.resources.limits`配置，避免单个服务占用宿主机所有资源，导致宿主机宕机；同时必须配置重启策略`restart: always`，保证服务异常退出、宿主机重启时，能自动重启，提升业务可用性。

### 【面试避坑提醒】

❌ 绝对不要说「Docker Compose 可以用于大规模集群的容器编排」，Compose 是单机编排工具，大规模集群必须用 Kubernetes，这是面试的高频考点

❌ 不要说 depends_on 可以保证服务就绪，它只能保证启动顺序，这是非常常见的错误认知

❌ 不要混淆 docker-compose（旧版 V1，Python 编写）和 docker compose（新版 V2，Go 编写），现在官方已经废弃了 V1 版本，生产环境必须用 V2 版本

### 【面试官高频追问 & 应答要点】

1. 追问：Docker Compose 怎么实现零停机更新？
    
    应答要点：可以通过`docker compose up -d --force-recreate --no-deps 服务名`命令，只更新指定的服务，不重启依赖的服务；同时配合`--scale`参数实现服务的多实例扩容，先启动新版本的实例，健康检查通过后，再停止旧版本的实例，实现滚动更新，零停机。对于更复杂的零停机更新，可以配合反向代理，实现流量的平滑切换。
2. 追问：Docker Compose 和 Kubernetes 的区别是什么？分别适合什么场景？
    
    应答要点：Docker Compose 是单机多容器编排工具，优点是简单、轻量、上手快，适合开发环境、测试环境、单机部署的小型应用；Kubernetes 是分布式容器集群编排平台，提供了自动扩缩容、滚动更新、服务发现、负载均衡、故障自愈、存储编排等企业级能力，适合生产环境大规模、分布式的微服务应用，两者的定位完全不同，不是替代关系，而是互补关系，我们开发环境用 Compose，生产环境用 Kubernetes。

## ### 【面试官标准提问】

讲一下 Docker 容器的完整生命周期，有哪些状态，对应的操作命令是什么？

### 【专家级作答话术】

Docker 容器的完整生命周期，是容器从创建到彻底销毁的全流程状态流转，核心分为**创建、运行、暂停、停止、删除**5 个核心状态，同时还有中间的过渡状态，比如重启中、退出状态，每个状态都有对应的操作命令，而且每个状态的底层实现逻辑是不同的，我结合生产上的用法给你讲清楚。

首先是完整的生命周期流转，以及每个状态的核心定义和对应命令：

1. **创建状态（Created）**
    
    核心定义：容器已经被创建，对应的配置、文件系统、网络、Namespace 都已经初始化完成，但容器内的进程还没有启动，没有占用 CPU 和内存资源。
    
    对应命令：`docker create`，这个命令我们生产上用得很多，比如提前创建好容器，配置好所有参数，等到需要的时候直接启动，避免启动时的配置耗时。
2. **运行状态（Running）**
    
    核心定义：容器内的主进程已经正常启动并运行，拥有独立的 Namespace、Cgroup 资源限制，网络、存储都已经正常挂载，是容器的正常工作状态。
    
    对应命令：`docker run`（创建 + 启动，一步到位，开发环境最常用）、`docker start`（启动已创建 / 已停止的容器，生产环境最常用）。
    
    底层逻辑：start 命令会调用 runc，启动容器内的主进程，同时把容器的状态更新为 Running。
3. **暂停状态（Paused）**
    
    核心定义：容器内的所有进程被临时冻结，暂停运行，不会释放已经占用的 CPU 和内存资源，只是暂停了进程的调度，恢复之后可以直接回到之前的运行状态，不会丢失数据。
    
    对应命令：`docker pause`，恢复命令是`docker unpause`。
    
    底层逻辑：这个命令的底层不是靠信号实现的，而是用了 Linux Cgroup 的 freezer 子系统，直接冻结进程的调度，比 kill 信号更安全，不会导致进程退出。我们生产上会用这个命令来给容器做热快照、备份数据，避免备份过程中数据被修改。
4. **停止状态（Exited/Stopped）**
    
    核心定义：容器内的主进程已经终止运行，释放了 CPU、内存等运行资源，但容器的配置、文件系统、日志都还保留在宿主机上，可以重新启动，也可以查看退出日志排查问题。
    
    对应命令：`docker stop`（优雅停止，生产环境首选）、`docker kill`（强制终止，紧急场景使用）。
    
    底层逻辑：`docker stop`会先给容器内的主进程发送 SIGTERM 信号，给应用 10 秒的时间优雅退出，比如保存数据、关闭连接，超时之后再发送 SIGKILL 信号强制终止；而`docker kill`是直接发送 SIGKILL 信号，立即终止进程，会导致业务中断、数据丢失，非紧急情况绝对不用。
5. **删除状态（Deleted）**
    
    核心定义：容器被彻底销毁，对应的配置、文件系统、网络、日志、进程全部被清理，释放所有占用的宿主机资源，无法恢复。
    
    对应命令：`docker rm`，只能删除已停止的容器，强制删除运行中的容器需要加`-f`参数，生产环境禁止直接强制删除运行中的容器。

补充两个过渡状态：

- 重启状态（Restarting）：容器正在重启过程中，通常是设置了重启策略，容器退出后自动触发重启
- 异常状态（Dead）：容器启动失败，无法正常运行，通常是配置错误、依赖缺失、内核不兼容导致的

最后补充一个生产上的最佳实践：我们会给所有业务容器设置重启策略`--restart=always`，保证宿主机重启、容器异常退出时，能自动重启，提升业务可用性。

### 【核心加分亮点】

1. 不仅讲了状态和命令，还补充了每个状态的底层实现逻辑，比如 pause 的 Cgroup freezer、stop 和 kill 的信号差异，体现底层原理的深度理解
2. 补充了生产环境的用法和最佳实践，比如 create 提前创建容器、pause 做备份、stop 优雅停止、重启策略，证明有线上运维经验
3. 区分了开发环境和生产环境的命令选型，体现了生产环境的规范意识

### 【面试避坑提醒】

❌ 不要说`docker kill`和`docker stop`是一样的，两者的优雅退出逻辑完全不同，这是面试高频考点

❌ 不要说暂停状态会释放资源，暂停状态只是冻结进程，不会释放资源，和停止状态有本质区别

❌ 不要忽略 Exited 状态，面试官经常会追问「容器退出了怎么排查问题」，提前准备好

### 【面试官高频追问 & 应答要点】

1. 追问：容器启动后立即变成 Exited 状态，你会怎么排查？
    
    应答要点：标准排查流程是：第一步，用`docker ps -a`查看容器的退出码，定位退出原因；第二步，用`docker logs 容器ID`查看容器的启动日志，看有没有报错信息；第三步，用`docker inspect 容器ID`查看容器的配置，比如启动命令、挂载目录、端口映射有没有错误；第四步，如果还是不行，用`docker run --rm -it 镜像名 /bin/bash`进入容器，手动执行启动命令，看有没有报错。
2. 追问：docker stop 的超时时间可以修改吗？怎么改？
    
    应答要点：可以，用`docker stop -t 30 容器ID`，-t 参数指定超时时间，单位是秒，默认是 10 秒。对于 Java 应用这种启动慢、退出需要时间保存数据的应用，我们会把超时时间设置为 30-60 秒，保证应用能优雅退出。

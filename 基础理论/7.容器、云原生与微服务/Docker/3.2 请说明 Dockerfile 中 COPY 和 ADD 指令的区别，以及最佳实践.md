### 【面试官标准提问】

Dockerfile 里的 COPY 和 ADD 都可以复制文件，它们有什么区别？你在生产环境中是怎么选择的？

### 【专家级作答话术】

首先，我先给一个我们生产环境严格执行的**最佳实践结论：99% 的场景优先使用 COPY 指令，只有一个场景可以用 ADD，就是需要自动解压本地压缩包到镜像里的场景**，这是 Docker 官方明确推荐的规范。

接下来我讲两者的核心区别，以及我们踩过的坑：

COPY 和 ADD 的基础能力是一致的：都是将宿主机构建上下文中的文件、目录，复制到镜像内的目标路径中。核心差异有 2 点：

第一，**ADD 支持额外的两个特殊能力，COPY 不支持**：

1. ADD 可以自动解压本地的 tar、gzip、bzip2 等压缩包，直接解压到镜像的目标路径里，比如`ADD ./app.tar.gz /app/`，会自动把 tar 包解压到 /app 目录下，而 COPY 只能原封不动地复制压缩包，不会解压。
2. ADD 支持从远程 URL 下载文件，复制到镜像的目标路径里，比如`ADD https://example.com/app.tar.gz /app/`，而 COPY 只支持复制本地构建上下文中的文件，不支持远程下载。
    
    第二，**Docker 构建缓存的处理逻辑不同**：
    
    COPY 的缓存逻辑非常简单，只有当本地复制的源文件的内容、权限、修改时间发生变化时，才会失效缓存，重新构建这一层；而 ADD 如果使用了远程 URL 下载文件，Docker 不会缓存这个文件，每次构建都会重新下载，不管文件有没有变化，这会导致镜像构建速度变慢，镜像体积不可控，这也是官方不推荐用 ADD 做远程下载的核心原因。

然后我补充两个我们生产上踩过的坑：

第一个坑：之前有同事用 ADD 复制远程的压缩包，以为会自动解压，结果 ADD 只解压本地的压缩包，远程下载的压缩包不会解压，导致镜像里只有压缩包，没有解压后的文件，线上应用启动失败。

第二个坑：用 ADD 复制了大量的本地小文件，没有配合.dockerignore，导致构建上下文过大，构建速度极慢，后来换成 COPY，同时优化了.dockerignore，构建速度提升了 70%。

最后再明确一下选型规则：

- 只复制本地文件 / 目录，无脑用 COPY，这是最安全、最可控的选择
- 只有需要自动解压本地压缩包的时候，才用 ADD
- 远程下载文件，绝对不要用 ADD，应该用 RUN 指令 + wget/curl 下载，同时在同一条 RUN 里完成下载、解压、清理安装包，避免多余的文件占用镜像分层，增大镜像体积

### 【面试官高频追问 & 应答要点】

1. 追问：Dockerfile 里的 RUN、CMD、ENTRYPOINT 有什么区别？
    
    应答要点：RUN 是在镜像构建阶段执行的命令，会提交到镜像层里；CMD 和 ENTRYPOINT 是容器启动阶段执行的命令，CMD 是容器的默认启动命令，会被 docker run 后面的命令覆盖，ENTRYPOINT 是容器的入口命令，不会被覆盖，通常用来做容器启动前的初始化脚本，和 CMD 配合使用，ENTRYPOINT 写固定的启动命令，CMD 写默认的参数。
2. 追问：你写 Dockerfile 的时候，有哪些规范来保证镜像的安全和稳定？
    
    应答要点：可以从使用官方基础镜像、指定固定版本、非 root 用户运行、合并 RUN 指令、清理缓存、使用多阶段构建、设置健康检查、暴露必要端口等方面回答，刚好可以衔接第 7 道题的镜像优化内容。

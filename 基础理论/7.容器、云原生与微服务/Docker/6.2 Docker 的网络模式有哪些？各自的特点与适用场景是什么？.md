### 【面试官标准提问】

Docker 有哪几种网络模式？分别有什么特点？适合什么场景？生产上你怎么选？

### 【专家级作答话术】

Docker 网络的底层核心，是通过**Linux Network Namespace 实现网络隔离，通过 veth pair 虚拟网卡、bridge 网桥、iptables 实现网络互通和端口转发**，原生提供了 5 种核心网络模式，我按照生产环境的使用频率，分别讲清楚每种模式的特点、适用场景，以及我们的选型规范。

首先先明确一个核心前提：Docker 默认的 bridge 网络，官方是不推荐生产环境使用的，生产环境必须用**自定义 bridge 网络**，这是我们的核心规范。

第一种，**bridge 模式（桥接模式）**，分为默认 bridge 和自定义 bridge，是 Docker 的默认模式，也是最常用的模式

- 核心特点：Docker 会为每个容器创建独立的 Network Namespace，分配独立的 IP 地址，所有容器都接入到一个虚拟的 docker0 网桥，通过网桥实现容器之间、容器和宿主机之间的通信，对外提供服务需要通过 iptables 做端口映射，把容器的端口映射到宿主机的端口上。
- 自定义 bridge 和默认 bridge 的核心区别（也是生产环境必须用自定义 bridge 的原因）：
    
    1. 自定义 bridge 支持 Docker 内置的 DNS 域名解析，同一个网络内的容器，可以直接通过容器名 / 服务名互相通信，不需要写固定的 IP 地址；而默认 bridge 只能通过 IP 地址通信，没有 DNS 解析
    2. 自定义 bridge 的隔离性更好，只有加入同一个网络的容器才能互相通信，不同网络的容器默认隔离；而默认 bridge 的所有容器都在同一个网段，没有隔离
    3. 自定义 bridge 可以灵活配置子网、网关、MTU 等参数，默认 bridge 只能用全局配置，无法自定义
    
- 适用场景：**单宿主机上的业务容器，生产环境首选自定义 bridge 模式**，我们生产环境上，所有单机部署的业务，都会按业务线创建独立的自定义 bridge 网络，同一个业务的容器加入同一个网络，实现网络隔离和互通，安全性和可维护性都非常好。

第二种，**host 模式（主机模式）**

- 核心特点：容器和宿主机共享同一个 Network Namespace，直接使用宿主机的 IP 地址、网卡、端口，没有任何网络隔离，容器内的应用可以直接占用宿主机的端口，不需要做端口映射，网络性能没有任何损耗，和宿主机原生进程的网络性能完全一致。
- 核心优势：网络性能极高，没有 NAT 转发的开销，延迟最低；核心劣势：隔离性极差，端口冲突风险极高，容器可以直接操作宿主机的网络栈，有安全风险。
- 适用场景：**对网络性能要求极高的场景**，比如网关代理、压测工具、监控采集组件、高并发的业务转发服务，我们生产环境的 Nginx 网关容器，就是用的 host 模式，保证转发性能。除此之外，普通业务容器禁止使用 host 模式。

第三种，**none 模式（无网络模式）**

- 核心特点：容器没有任何网络接口，只有一个 lo 本地回环接口，完全关闭了网络，无法和外界通信，隔离性和安全性是最高的。
- 适用场景：**高安全要求的离线计算、敏感数据处理、不需要网络的加密解密任务**，比如离线的数据分析、密码学计算，完全不需要网络，防止数据泄露，我们生产环境的敏感数据处理容器，会用这个模式。

第四种，**container 模式（容器共享模式）**

- 核心特点：新创建的容器，和指定的已有容器共享同一个 Network Namespace，两个容器共享同一个 IP 地址、端口、网络栈，容器之间可以通过[localhost](https://localhost)直接通信，不需要经过网桥转发。
- 适用场景：**边车（Sidecar）模式**，这是现在微服务架构里非常常用的模式，比如主业务容器 + 日志采集容器、主业务容器 + 监控代理容器、主业务容器 + 网络代理容器，两个容器共享网络栈，边车容器可以直接采集主容器的网络流量、日志，不需要额外的网络配置，我们生产环境的 APM 链路追踪代理，就是用的这个模式。

第五种，**overlay 模式（覆盖网络模式）**

- 核心特点：Docker 原生的跨主机网络模式，构建在宿主机底层网络之上的虚拟二层网络，实现不同宿主机上的容器，在同一个虚拟网段内，直接通过 IP / 容器名互相通信，不需要做端口映射，底层通过 VXLAN 隧道实现跨主机的数据转发。
- 适用场景：**Docker Swarm、Kubernetes 等容器集群场景**，跨多台宿主机部署的分布式应用，需要跨主机容器直接通信的场景，我们之前的 Swarm 集群，就是用的 overlay 网络实现跨主机容器互通。

### 【面试避坑提醒】

❌ 绝对不要说「生产环境用默认 bridge 网络就可以」，这是严重违背官方最佳实践的，会被认为经验不足

❌ 不要混淆 container 模式和 host 模式，host 是和宿主机共享网络栈，container 是和另一个容器共享网络栈

❌ 不要说 overlay 模式可以用于单主机容器通信，它是专门用于跨主机集群的

### 【面试官高频追问 & 应答要点】

1. 追问：两个运行在不同宿主机上的 Docker 容器，怎么实现互相通信？
    
    应答要点：有 3 种主流方案，第一种，用 Docker 原生的 overlay 网络，适合 Swarm 集群；第二种，把容器的端口映射到宿主机，通过宿主机的 IP + 端口通信，适合小规模场景；第三种，用 Calico、Flannel 等成熟的 CNI 网络插件，实现跨主机容器的三层路由通信，适合 Kubernetes 大规模集群，这也是现在生产环境的主流方案。
2. 追问：容器内无法访问外网，你会怎么排查？
    
    应答要点：标准排查流程：第一步，先看容器的 DNS 配置是不是正确，`cat /etc/resolv.conf`，能不能 ping 通公网 IP，比如 114.114.114.114，如果能 ping 通 IP，ping 不通域名，就是 DNS 问题；第二步，看宿主机的 iptables FORWARD 链是不是 ACCEPT，Docker 默认会设置 iptables 规则，如果被清空了，容器就无法转发流量；第三步，看宿主机有没有开启 IP 转发，`cat /proc/sys/net/ipv4/ip_forward`，必须是 1，否则无法转发；第四步，看宿主机的防火墙、安全组有没有禁止容器的网段访问外网。

#### 核心标准答案

K8s 中用于管理业务负载的核心资源是**工作负载（Workloads）**，也就是和 Deployment 平行的负载类型，均内置在 K8s 中，无需额外安装，默认提供的核心负载类型共 5 种，分别是：

1. **Deployment**：最常用的无状态负载类型，用于管理无状态应用，提供滚动更新、版本回滚、弹性扩缩容、副本数管理能力，保证应用的高可用，适用于前端服务、后端 API 服务等无状态应用。
2. **StatefulSet**：有状态负载类型，用于管理有状态应用，提供固定的 Pod 名称、固定的 DNS 域名、有序的部署 / 更新 / 删除、持久化存储绑定能力，保证应用的状态一致性，适用于 MySQL、Redis、Kafka 等有状态的中间件、数据库服务。
3. **DaemonSet**：守护进程负载类型，用于在集群的每个 Node 节点上运行一个 Pod 副本，新增节点时自动在新节点创建 Pod，删除节点时自动清理对应 Pod，适用于节点级的监控、日志采集、网络插件等，比如 node-exporter、filebeat、Calico。
4. **Job**：一次性任务负载类型，用于管理一次性运行的任务，创建 Pod 运行任务，任务完成后 Pod 自动退出，不会持续运行，适用于数据备份、数据计算、批量处理等一次性任务。
5. **CronJob**：定时任务负载类型，基于 Job 实现，用于管理定时运行的任务，按照预设的 Cron 表达式定时创建 Job，执行任务，适用于定时备份、定时报表、定时巡检等周期性任务。

补充：ReplicaSet 是 Deployment 的底层资源，由 Deployment 自动管理，用于控制 Pod 的副本数，不直接用于业务部署，不属于和 Deployment 平行的业务负载类型。

#### 专家级拓展

- 不同负载类型的核心差异与选型标准：
    
    表格
    
    |负载类型|核心特性|适用场景|不可用场景|
    |:--|:--|:--|:--|
    |Deployment|无状态、副本灵活、滚动更新|无状态 API 服务、前端服务|有状态应用、节点级守护进程|
    |StatefulSet|有状态、固定标识、有序部署|数据库、中间件、分布式存储|无状态应用、一次性任务|
    |DaemonSet|节点级单副本、自动扩缩容|监控、日志、网络 / 存储插件|业务应用、多副本服务|
    |Job|一次性运行、完成即退出|批量处理、数据备份、一次性计算|长期运行的服务|
    |CronJob|定时触发、周期性运行|定时备份、定时巡检、定时报表|实时业务服务|
    
- 进阶知识点：Deployment 的滚动更新策略，可通过`maxSurge`和`maxUnavailable`控制滚动更新的速度，保证更新过程中业务不中断；StatefulSet 的更新策略分为滚动更新、按需更新、金丝雀发布，可实现有状态应用的灰度更新。

#### 面试避坑指南

- 严禁把 ReplicaSet、Pod、Service 说成和 Deployment 平行的负载类型，Service 是服务发现资源，不是负载类型，ReplicaSet 是 Deployment 的底层资源，不能直接用于业务部署；
- 避免只说负载类型的名称，不说核心作用和适用场景，面试问这个问题，核心是考察你会不会根据业务场景做选型。

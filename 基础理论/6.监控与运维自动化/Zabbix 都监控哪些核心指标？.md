#### 核心标准答案

Zabbix 作为全栈监控系统，覆盖从基础设施到业务层的全维度监控，核心监控指标按层级分为 6 大类，覆盖生产环境 99% 的监控场景：

1. **主机硬件与基础状态监控**
    
    服务器硬件健康状态：CPU 型号 / 核心数、内存总容量、磁盘总容量、电源状态、风扇转速、主板温度、RAID 卡状态、硬盘 SMART 信息；服务器开机时长、硬件故障告警。
2. **操作系统层核心指标**
    
    - CPU：整体使用率、用户态 / 内核态 / IO 等待 / 空闲占比、负载平均值（1/5/15 分钟）、上下文切换次数、中断次数
    - 内存：总内存、已用内存、可用内存、缓存 / 缓冲占用、swap 使用率、swap 换入换出速率
    - 磁盘：分区使用率、inode 使用率、磁盘读写吞吐量（MB/s）、IOPS、IO 等待时间（await）、磁盘使用率（% util）
    - 网络：网卡入 / 出带宽、数据包收发量、错包 / 丢包率、TCP 连接数（ESTABLISHED/TIME_WAIT 等状态）
    
3. **应用服务层监控**
    
    服务进程状态、端口监听状态、服务启动 / 运行时长、CPU / 内存占用、进程打开的文件描述符数；
    
    覆盖常见服务：Nginx、Apache、Tomcat、MySQL、Redis、Kafka、Elasticsearch、Docker、K8s 组件等，监控其 QPS、连接数、请求耗时、错误率等专属指标。
4. **网络设备与链路监控**
    
    交换机 / 路由器 / 防火墙的端口状态、带宽使用率、错包 / 丢包率、设备 CPU / 内存使用率；网络链路连通性、延迟、丢包率、抖动。
5. **日志与安全监控**
    
    系统日志、应用日志的错误关键字匹配、告警；SSH 登录异常、权限变更、端口扫描、暴力破解等安全事件监控。
6. **业务层自定义监控**
    
    基于业务场景的自定义指标：接口 QPS、成功率、平均响应时间、订单量、支付成功率、用户登录数、消息队列生产 / 消费速率、任务执行成功率等。

#### 专家级拓展

- 监控落地核心能力：
    
    1. 自动发现：通过 Zabbix 的 LLD（低级别发现），自动发现服务器的磁盘分区、网卡、监听端口、K8s Pod、数据库实例等，自动创建监控项，无需人工逐个配置，适配大规模集群场景。
    2. 触发器与告警分级：按业务影响程度设置告警级别（信息 / 警告 / 严重 / 灾难），配置合理的阈值与告警抑制，避免告警风暴；支持邮件、钉钉、企业微信、短信、电话等多渠道告警。
    3. 分布式监控：针对跨机房、多区域的大规模集群，使用 Zabbix Proxy 分布式部署，Proxy 负责采集区域内的监控数据，统一上报给 Zabbix Server，降低网络带宽占用，提升监控稳定性。
    
- 与 Prometheus 的选型差异：Zabbix 适合中小规模、全栈一体化监控，开箱即用，支持硬件、操作系统、应用、网络的全场景覆盖，无需额外组件；Prometheus 适合云原生、K8s 环境，时序数据处理能力更强，配合 Grafana 可视化效果更好，适合大规模云原生集群监控。
- 生产最佳实践：核心业务指标必须配置**趋势告警**，而非仅阈值告警，比如磁盘使用率一周内增长超过 20% 提前告警，避免磁盘突然打满；同时配置监控项的可用性检测，避免监控失效导致故障漏报。

#### 面试避坑指南

- 严禁只零散罗列 CPU、内存等基础指标，不做层级分类，会被面试官判定为「只会基础监控，无体系化监控设计能力」；
- 避免遗漏业务层自定义监控，面试问这个问题，核心是考察你是否有业务视角，而不只是会搭监控、看系统指标；
- 不要只说监控指标，不说触发器、告警、自动发现等落地能力，无法体现生产环境的实际监控运维经验。

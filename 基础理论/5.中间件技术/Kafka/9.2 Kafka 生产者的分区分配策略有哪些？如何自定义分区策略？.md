### 专家级标准答案

1. **核心结论先行**
    
    分区分配策略（Partitioner）是 Kafka 生产者的核心组件，决定了生产者发送的消息会被路由到 Topic 的哪一个分区，直接影响消息的顺序性、分区负载均衡与集群吞吐性能；Kafka 内置了 3 种核心分区策略，同时提供了标准化的扩展接口，支持自定义分区策略，适配不同的业务场景。
    
2. **底层原理与内置分区策略拆解**
    
    分区分配策略的核心执行时机：生产者发送消息时，先获取 Topic 的元数据（分区总数），然后根据分区策略，计算出目标分区号，将消息发送给该分区的 Leader 副本所在的 Broker。
    
    内置核心分区策略如下：
    
    #### （1）按 Key 哈希分区策略（DefaultPartitioner 的核心逻辑）
    
    - 触发条件：当消息指定了 Key 时，默认触发该策略；
    - 底层实现逻辑：对消息的 Key 进行 MurmurHash2 哈希运算（非加密哈希，性能高，分布均匀），将哈希结果对 Topic 的分区数取模，最终得到目标分区号；
    - 核心优势：**保证同一个 Key 的消息，永远被路由到同一个分区**，而 Kafka 单分区内的消息严格有序，从而实现业务层面的消息顺序性（如同一用户、同一订单的操作记录，按用户 ID / 订单号作为 Key，保证操作顺序）；
    - 核心劣势：若 Key 分布不均匀，会出现数据倾斜，部分分区数据量过大，导致分区负载不均衡；Topic 扩容分区后，Key 与分区的映射关系会发生变化，同一 Key 的消息会被路由到不同分区，破坏顺序性。
    
    #### （2）粘性分区策略（Sticky Partitioner，Kafka 2.4 + 默认无 Key 策略）
    
    - 触发条件：当消息没有指定 Key 时，默认触发该策略；
    - 底层实现逻辑：解决了旧版本轮询策略的小批次频繁发送问题，核心逻辑是「**尽量将消息粘在同一个分区，攒够一个完整的批次（达到 batch.size）后，再切换到下一个分区**」，同时保证长期来看，消息在所有分区上均匀分布；
    - 核心优势：大幅提升生产者的攒批效率，减少网络请求次数，降低 Broker 的请求处理开销，提升集群整体吞吐性能，同时保证分区的负载均衡；
    - 核心劣势：短时间内，消息在分区上的分布不如轮询策略均匀，但长期来看完全均衡。
    
    #### （3）轮询分区策略（Round-Robin Partitioner，旧版本无 Key 默认策略）
    
    - 底层实现逻辑：将消息按顺序，轮流均匀分配到 Topic 的所有分区，每发送一条消息，切换一个分区；
    - 核心优势：保证消息在分区上的绝对均匀分布，分区负载完全均衡；
    - 核心劣势：无法实现消息攒批，每条消息都可能被路由到不同分区，导致每个分区的批次都无法攒满，频繁发送小批次消息，网络 IO 次数过多，吞吐性能极差，Kafka 2.4 + 版本已被粘性分区策略替代。
    
3. **生产级落地规范**
    
    - 顺序性场景规范：需要保证业务顺序性的场景，必须使用业务唯一 Key 作为消息 Key，采用默认的哈希分区策略，禁止使用无 Key 的粘性 / 轮询策略；
    - 高吞吐场景规范：无顺序性要求的高吞吐场景，使用无 Key 的粘性分区策略，优化`batch.size`与`linger.ms`，最大化攒批效率，提升吞吐；
    - 分区扩容规范：使用哈希分区策略的 Topic，扩容分区前，必须评估对业务顺序性的影响，扩容后会导致 Key 与分区的映射关系变化，若要保证顺序性，需新建 Topic，重新迁移数据。
    
4. **自定义分区策略实现方案（生产级标准实现）**
    
    自定义分区策略适用于特殊业务场景，比如按业务优先级路由、按机房就近路由、按用户等级路由、白名单分区隔离等场景，标准实现步骤如下：
    
    #### （1）实现标准化接口
    
    实现`org.apache.kafka.clients.producer.Partitioner`接口，重写 3 个核心方法：
    
    java
    
    运行
    
    ```
    public interface Partitioner extends Configurable, Closeable {
        // 核心方法：计算目标分区号，必须实现
        public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster);
        
        // 配置初始化方法，获取生产者配置，可选实现
        public void configure(Map<String, ?> configs);
        
        // 资源释放方法，关闭生产者时调用，可选实现
        public void close();
    }
    ```
    
    #### （2）生产级自定义实现示例（按业务优先级路由）
    
    场景：核心高优先级消息路由到前 3 个分区，普通消息路由到其他分区，实现资源隔离，避免普通消息影响高优先级消息的处理。
    
    java
    
    运行
    
    ```
    public class PriorityPartitioner implements Partitioner {
        // 高优先级分区数
        private static final int HIGH_PRIORITY_PARTITIONS = 3;
    
        @Override
        public int partition(String topic, Object key, byte[] keyBytes, Object value, byte[] valueBytes, Cluster cluster) {
            // 获取Topic的总分区数
            List<PartitionInfo> partitions = cluster.partitionsForTopic(topic);
            int totalPartitions = partitions.size();
            
            // 校验分区数
            if (totalPartitions < HIGH_PRIORITY_PARTITIONS) {
                throw new IllegalArgumentException("Topic分区数不能小于高优先级分区数");
            }
            
            // 从消息Key中获取优先级（也可从消息头/消息体中获取）
            String priority = new String(keyBytes, StandardCharsets.UTF_8);
            // 高优先级消息，路由到前3个分区
            if ("HIGH".equals(priority)) {
                return Math.abs(Utils.murmur2(keyBytes)) % HIGH_PRIORITY_PARTITIONS;
            }
            // 普通消息，路由到剩余分区
            else {
                int normalPartitionBase = HIGH_PRIORITY_PARTITIONS;
                int normalPartitionCount = totalPartitions - HIGH_PRIORITY_PARTITIONS;
                return normalPartitionBase + Math.abs(Utils.murmur2(keyBytes)) % normalPartitionCount;
            }
        }
    
        @Override
        public void configure(Map<String, ?> configs) {
            // 初始化配置，可从生产者配置中读取自定义参数
        }
    
        @Override
        public void close() {
            // 释放资源，无资源可空实现
        }
    }
    ```
    
    #### （3）生产者配置生效
    
    在生产者配置中，通过`partitioner.class`参数，指定自定义分区策略类的全限定名，即可生效：
    
    java
    
    运行
    
    ```
    Properties props = new Properties();
    props.put(ProducerConfig.BOOTSTRAP_SERVERS_CONFIG, "kafka-brokers:9092");
    props.put(ProducerConfig.KEY_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());
    props.put(ProducerConfig.VALUE_SERIALIZER_CLASS_CONFIG, StringSerializer.class.getName());
    // 指定自定义分区策略
    props.put(ProducerConfig.PARTITIONER_CLASS_CONFIG, "com.xxx.kafka.PriorityPartitioner");
    KafkaProducer<String, String> producer = new KafkaProducer<>(props);
    ```
    
5. **高频风险点与避坑指南**
    
    - 风险 1：自定义分区策略中，返回的分区号超出了 Topic 的分区数范围，导致发送失败，必须先获取 Topic 的分区总数，做边界校验；
    - 风险 2：自定义分区策略的逻辑过于复杂，或存在阻塞操作，导致生产者发送线程阻塞，吞吐下降，分区计算逻辑必须是轻量、无阻塞、无 IO 的；
    - 风险 3：自定义分区策略导致数据倾斜，部分分区数据量过大，出现负载不均，必须保证消息在分区上的分布相对均匀；
    - 风险 4：Topic 扩容分区后，自定义分区策略的映射逻辑失效，导致消息路由异常，扩容前必须评估对自定义策略的影响。
    
6. **进阶加分延伸**
    
    分区策略的设计，本质是在**消息顺序性、分区负载均衡、吞吐性能**三者之间做平衡：哈希分区策略保证顺序性，但可能出现数据倾斜；轮询策略保证负载均衡，但吞吐性能差；粘性分区策略平衡了负载均衡与吞吐性能，但无法保证顺序性。生产环境中，需根据业务的核心诉求，选择合适的分区策略，只有当内置策略无法满足业务需求时，才需要自定义分区策略，避免过度设计。同时，Kafka 的分区策略与消费者端的分区分配策略是两个完全不同的概念，前者是生产者端决定消息路由到哪个分区，后者是消费者端决定分区分配给哪个消费者，面试中需严格区分，避免混淆。

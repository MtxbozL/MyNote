**参考答案**：

重复消费的**根本原因**：消息已经被业务成功处理，但消费位移 Offset 没有成功提交到 Kafka，消费者重启 / 重平衡后，会从上一次成功提交的 Offset 重新拉取消息，导致重复消费。

#### 常见触发场景

1. 自动提交 Offset 场景：消费者开启自动提交，在消息处理完成后、Offset 自动提交前，消费者宕机 / 被强制关闭，下次启动会重复消费。
2. 手动提交 Offset 场景：消息处理完成后，提交 Offset 之前，消费者发生崩溃或触发重平衡，导致 Offset 提交失败。
3. 消费者组重平衡（Rebalance）：分区被重新分配给其他消费者，未提交的 Offset 会导致新消费者重新消费该分区的消息。
4. 生产者重试：网络抖动时，生产者已发送的消息 Broker 已写入，但未返回成功响应，生产者重试会导致 Broker 中写入重复消息，最终被重复消费。

#### 解决方案

核心思路是**实现消费逻辑的幂等性**，无论消息被消费多少次，最终业务结果只生效一次，主流方案如下：

1. 基于唯一键去重：给每条消息生成唯一的 Message ID / 业务唯一键，消费前通过 Redis / 数据库的唯一索引记录已消费的 ID，已存在的消息直接跳过。
2. 数据库唯一约束：对于数据库写入操作，基于业务主键 / 唯一索引实现去重，重复插入会触发唯一键冲突，直接丢弃即可。
3. 业务逻辑幂等化：将消费逻辑设计为天然幂等，比如使用 UPDATE 更新操作（而非 INSERT）、状态机控制（只有待处理状态的记录才会被处理），多次执行结果完全一致。

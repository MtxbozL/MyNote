#### 【核心得分点】

主从复制的核心作用是**数据热备份、读写分离分摊读压力、为哨兵 / 集群的故障转移提供数据基础**；主从复制分为全量复制（首次同步、无法增量同步时触发）和增量复制（网络闪断重连后触发），核心基于复制 ID、复制偏移量、复制积压缓冲区实现。

#### 【专家级完整作答】

##### 一、主从复制的核心作用

Redis 主从复制采用「主节点负责写，从节点负责读」的架构，是 Redis 高可用、高并发架构的基础，核心作用有 3 点：

1. **数据热备份**：主节点的数据实时同步到从节点，实现数据多副本热备份，主节点宕机后，从节点仍有完整数据，可快速恢复；
2. **读写分离**：写请求全部打到主节点，读请求分摊到多个从节点，大幅提升 Redis 集群的读并发能力，适配读多写少的业务场景；
3. **高可用基础**：哨兵、Cluster 集群的自动故障转移能力，完全基于主从复制实现，是 Redis 实现高可用的核心前提。

##### 二、全量复制完整流程

全量复制是主节点将内存全量数据同步给从节点的过程，**首次建立主从关系、从节点复制偏移量不在主节点复制积压缓冲区范围内时触发**，完整流程如下：

1. **建立连接与握手**：从节点执行 SLAVEOF 命令，与主节点建立 TCP 连接，完成身份认证、端口通信等握手流程，向主节点发送同步请求；
2. **主节点生成快照**：主节点执行 BGSAVE 命令，fork 子进程生成当前内存的 RDB 快照文件，同时开辟**复制缓冲区**，记录快照生成期间收到的所有写命令；
3. **快照传输与加载**：主节点将生成的 RDB 快照文件发送给从节点；从节点清空本地所有数据，加载 RDB 快照文件到内存；
4. **增量命令同步**：主节点将复制缓冲区中记录的写命令，发送给从节点；从节点执行这些写命令，完成快照期间增量数据的同步；
5. **持续命令传播**：全量同步完成后，进入持续命令传播阶段，主节点执行的每一条写命令，都会实时同步给从节点，主从数据保持一致。

##### 三、增量复制（部分重同步）完整流程

增量复制是主从网络闪断重连后，仅同步断连期间的增量数据，而非全量同步，大幅降低同步开销，**核心基于复制 ID、复制偏移量、复制积压缓冲区三大核心要素实现**，完整流程如下：

1. **断连期间数据缓存**：主从网络断连期间，主节点仍会将写命令写入**复制积压缓冲区**（固定长度的环形缓冲区），同时记录每个命令对应的复制偏移量；
2. **重连与同步信息上报**：从节点网络恢复后，重新与主节点建立连接，向主节点上报自身的主节点复制 ID、已同步的复制偏移量；
3. **同步校验**：主节点校验两个核心条件：① 从节点上报的复制 ID 与自身的复制 ID 匹配；② 从节点上报的复制偏移量，在复制积压缓冲区的范围内；
4. **增量数据同步**：若两个条件均满足，主节点将复制积压缓冲区中，从偏移量之后的所有写命令，发送给从节点；
5. **同步完成**：从节点执行收到的增量写命令，完成数据同步，重新进入持续命令传播阶段。
    
    若校验不通过，则触发全量复制流程。

#### 【加分项延伸】

1. 生产优化实践：复制积压缓冲区 repl_backlog_size 建议调大（默认 1MB），生产环境建议配置为 100MB-1GB，大幅降低网络闪断后触发全量复制的概率；
2. 无盘复制优化：Redis 支持无盘复制，主节点生成快照时，直接通过网络发送给从节点，无需写入本地磁盘，适合磁盘 IO 性能差的场景；
3. 数据一致性保障：生产环境建议配置 min-replicas-to-write、min-replicas-max-lag 参数，保证主节点至少有 N 个从节点延迟在阈值内，才允许写入，避免主从数据不一致。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能说主从复制只能全量同步；绝对不能说错复制积压缓冲区是环形缓冲区；
- 高频追问预判：主从复制数据不一致的原因和解决方案？复制 ID 的作用是什么？

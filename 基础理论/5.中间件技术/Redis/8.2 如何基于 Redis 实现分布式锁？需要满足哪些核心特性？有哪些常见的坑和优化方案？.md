#### 【核心得分点】

Redis 分布式锁的核心是**跨进程、跨节点的资源互斥访问**，必须满足互斥性、防死锁、可重入性、高可用、操作原子性五大核心特性；标准实现是**SET key unique_value NX PX expire_time** 原子加锁 + Lua 脚本原子解锁；生产环境优先使用 Redisson 分布式锁，解决锁续期、可重入、高可用等问题。

#### 【专家级完整作答】

##### 一、分布式锁必须满足的核心特性

分布式锁是解决分布式系统中，多个进程 / 节点对共享资源的互斥访问问题，必须满足 5 大核心特性，缺一不可：

1. **互斥性**：同一时间，只有一个客户端能持有锁，这是分布式锁的最基本要求；
2. **防死锁**：即使持有锁的客户端宕机，锁最终也能被释放，不会导致其他客户端永远无法获取锁；
3. **可重入性**：同一个线程持有锁后，可重复加锁，不会被自己的锁阻塞；
4. **操作原子性**：加锁、解锁操作必须保证原子性，避免并发场景下的逻辑异常；
5. **高可用**：锁服务需具备高可用能力，少数 Redis 节点宕机，不会导致锁服务不可用。

##### 二、基于 Redis 的分布式锁标准实现

###### 1. 基础原子加锁实现

必须使用 Redis 的**SET 命令单条原子操作**实现加锁，命令如下：

plaintext

```
SET lock_key unique_client_value NX PX 30000
```

- 命令参数说明：
    
    - lock_key：锁的唯一标识，对应共享资源；
    - unique_client_value：客户端唯一标识（如 UUID + 线程 ID），用于解锁时校验，避免误删其他客户端的锁；
    - NX：Only if Not Exists，仅当 key 不存在时才设置成功，保证互斥性；
    - PX 30000：设置 key 的过期时间为 30 秒，保证客户端宕机后，锁会自动释放，避免死锁。
    

###### 2. 安全解锁实现

解锁必须保证「校验客户端唯一标识→删除锁」的原子性，**必须使用 Lua 脚本实现**，避免并发场景下误删锁，Lua 脚本如下：

lua

```
if redis.call('get', KEYS[1]) == ARGV[1] then
    return redis.call('del', KEYS[1])
else
    return 0
end
```

- 核心逻辑：仅当锁的 value 与当前客户端的唯一标识匹配时，才执行删除操作，保证只有锁的持有者才能解锁，完全避免误删锁。

##### 三、生产环境常见坑与优化方案

1. **坑 1：锁过期释放，业务逻辑还未执行完**
    
    - 根因：加锁时设置的过期时间，无法精准匹配业务执行时间，时间过短导致锁提前释放，业务还在执行，引发并发问题；
    - 优化方案：引入**看门狗（WatchDog）** 机制，客户端加锁成功后，启动一个后台定时任务，锁快过期时（如过期时间的 1/3），自动给锁续期，业务执行完成后，停止续期并解锁；Redisson 分布式锁已原生实现该机制。
    
2. **坑 2：主从切换导致锁丢失**
    
    - 根因：加锁成功后，主节点还未将锁同步到从节点，主节点宕机，哨兵触发主从切换，新主节点没有该锁数据，其他客户端可再次加锁成功，导致互斥性失效；
    - 优化方案：核心场景使用**RedLock 红锁**方案，向多个独立的 Redis 主节点加锁，超过半数节点加锁成功，才算加锁成功，即使少数节点宕机，也不会导致锁丢失；非核心场景可通过调整主从同步参数、增加重试机制降低风险。
    
3. **坑 3：不可重入，同一线程重复加锁导致死锁**
    
    - 根因：基础加锁方案不支持可重入，同一个线程持有锁后，再次加锁会失败，导致死锁；
    - 优化方案：基于 Hash 数据结构 + Lua 脚本实现可重入锁，Hash 结构中记录客户端标识和重入次数，加锁时若客户端已持有锁，仅增加重入次数，解锁时减少重入次数，重入次数为 0 时才真正删除锁。
    
4. **坑 4：非原子操作导致的逻辑异常**
    
    - 根因：加锁使用 SETNX + EXPIRE 分两条命令执行，若 SETNX 成功后客户端宕机，EXPIRE 未执行，锁永远不会过期，导致死锁；解锁使用 GET + DEL 分两条命令执行，GET 判断完成后，锁刚好过期被其他客户端持有，DEL 会误删其他客户端的锁；
    - 优化方案：加锁必须使用 SET 单条原子命令，解锁必须使用 Lua 脚本保证原子性，绝对禁止分两次命令执行。
    

#### 【加分项延伸】

1. 生产选型建议：业务系统禁止手动实现分布式锁，优先使用 Redisson 等成熟框架，原生支持可重入锁、公平锁、读写锁、联锁、红锁等全场景，解决了所有常见坑，生产稳定性极高；
2. 性能优化：分布式锁的过期时间建议设置为业务平均执行时间的 3 倍，预留足够的缓冲；高并发场景下，可使用分段锁，降低锁的粒度，提升并发性能；
3. 兜底方案：加锁时必须设置超时时间，避免无限等待；业务代码必须使用 try-finally 结构，保证锁最终一定会被释放。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能使用 SETNX + EXPIRE 分两次命令加锁；绝对不能解锁时不校验 value 直接执行 DEL；绝对不能忽略锁续期问题；

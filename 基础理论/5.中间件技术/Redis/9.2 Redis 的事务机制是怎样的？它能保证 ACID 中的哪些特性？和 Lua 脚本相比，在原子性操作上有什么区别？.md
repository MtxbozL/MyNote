#### 【核心得分点】

Redis 事务基于**MULTI、EXEC、DISCARD、WATCH**四个命令实现，将多个命令打包入队，EXEC 时一次性顺序执行；事务仅保证**一致性、隔离性**，不保证强原子性、持久性；Lua 脚本是强原子性，支持复杂逻辑判断，是生产环境实现原子操作的首选方案。

#### 【专家级完整作答】

##### 一、Redis 事务机制的核心实现

Redis 事务的核心目标是将多个命令打包，保证命令按顺序执行，执行期间不会被其他客户端的命令打断，核心基于 4 个命令实现：

1. **MULTI**：开启事务，客户端进入事务状态，后续执行的命令不会立即执行，而是放入事务队列中，返回 QUEUED；
2. **EXEC**：提交事务，按顺序执行事务队列中的所有命令，返回每条命令的执行结果，事务结束；
3. **DISCARD**：放弃事务，清空事务队列，客户端退出事务状态，事务结束；
4. **WATCH**：乐观锁机制，监控一个或多个 key，在 EXEC 提交事务前，若被监控的 key 被其他客户端修改，事务会自动放弃执行，保证数据一致性。

##### 二、Redis 事务对 ACID 四大特性的支持情况

ACID 是关系型数据库事务的四大核心特性，Redis 事务对其支持情况如下：

1. **原子性（Atomicity）：弱原子性，不保证**
    
    原子性要求事务内的所有操作要么全部成功，要么全部失败，支持回滚。
    
    Redis 事务不支持回滚：事务执行过程中，若某一条命令执行报错，后续的命令依然会继续执行，不会回滚已执行的命令；仅在命令入队时出现语法错误，才会放弃整个事务。因此 Redis 事务仅保证命令批量执行，不保证强原子性。
    
2. **一致性（Consistency）：完全保证**
    
    一致性要求事务执行前后，数据的完整性、约束性保持不变。
    
    Redis 事务执行前后，数据的约束（如过期时间、数据类型）始终保持一致；同时通过 WATCH 机制，可避免并发修改导致的数据不一致，完全满足一致性要求。
    
3. **隔离性（Isolation）：完全保证**
    
    隔离性要求并发事务之间互不干扰，事务执行期间的中间状态，不会被其他事务看到。
    
    Redis 核心命令执行是单线程模型，事务队列中的命令在 EXEC 提交后，会一次性顺序执行完成，执行期间绝对不会被其他客户端的命令打断，完全满足隔离性要求。
    
4. **持久性（Durability）：不保证**
    
    持久性要求事务提交成功后，数据会被持久化到磁盘，宕机后数据不会丢失。
    
    Redis 的持久性取决于 RDB/AOF 配置，即使开启 AOF，默认刷盘策略是 everysec，最多会丢失 1 秒数据，事务提交成功后，数据不一定会立即持久化到磁盘，因此不保证持久性。
    

##### 三、Redis 事务与 Lua 脚本的核心区别

Redis 中实现原子操作有两种方案：事务、Lua 脚本，其中 Lua 脚本是生产环境的首选，两者核心区别如下：

表格

|核心维度|Redis 事务|Lua 脚本|
|:--|:--|:--|
|原子性|弱原子性，仅保证命令顺序执行，单条命令报错不回滚，无事务回滚能力|强原子性，整个 Lua 脚本内的所有命令，要么全部执行成功，要么全部失败，执行期间绝对不会被其他命令打断|
|逻辑能力|仅支持简单的命令打包，无逻辑判断、循环、分支等能力，无法实现复杂业务逻辑|支持完整的 Lua 语法，可实现复杂的条件判断、循环、分支、变量计算，适配复杂的业务原子操作|
|执行效率|命令入队和执行分两次网络交互，多条命令多次网络 IO，效率较低|整个脚本一次性发送给 Redis，一次网络 IO 完成所有操作，减少网络往返开销，执行效率更高|
|可复用性|无复用能力，每次执行都需要重新发送所有命令|支持脚本缓存，通过 SHA1 校验码调用已缓存的脚本，无需重复发送脚本内容，复用性极强|
|生产适用场景|极简单的批量命令执行，无逻辑判断的场景|生产环境绝大多数原子操作场景，尤其是需要条件判断、复杂逻辑的场景|

#### 【加分项延伸】

1. 生产最佳实践：生产环境优先使用 Lua 脚本实现原子操作，而非 Redis 事务；Lua 脚本需控制长度，避免执行时间过长阻塞主进程，建议单脚本执行时间不超过 1ms；
2. WATCH 机制的适用场景：WATCH 仅适用于并发量较低的乐观锁场景，高并发场景下，大量并发修改会导致事务频繁执行失败，优先使用 Lua 脚本替代；
3. 脚本安全：Redis Lua 脚本支持超时中断，生产环境需配置 lua-time-limit 参数，避免脚本死循环导致 Redis 服务阻塞。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能说 Redis 事务支持回滚；绝对不能说 Redis 事务是强原子性；绝对不能混淆事务和 Lua 脚本的原子性差异；
- 高频追问预判：Redis 为什么不支持事务回滚？Lua 脚本的执行原理？

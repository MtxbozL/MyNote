#### 【核心得分点】

Redis 5 种基础数据类型为**String、Hash、List、Set、Sorted Set(ZSet)**；每种类型对应专属的底层编码结构，Redis 会根据数据长度、元素个数自动切换编码以优化内存和性能，不同类型适配不同的业务场景，是 Redis 区别于其他 KV 数据库的核心优势。

#### 【专家级完整作答】

以下为 5 种基础数据类型的全维度拆解，基于 Redis 7.0 + 最新稳定版本的底层实现：

1. **String（字符串）**
    
    - 核心特点：二进制安全，单 value 最大支持 512MB，支持整数增减、位操作，是 Redis 最基础的类型；
    - 底层实现：3 种编码自动切换，int（存储 8 字节长整型）、embstr（存储≤44 字节的短字符串，内存连续分配）、raw（存储 > 44 字节的长字符串）；
    - 典型场景：通用 KV 缓存、分布式锁、用户 Session 存储、实时计数器、接口限流令牌桶。
    
2. **Hash（哈希）**
    
    - 核心特点：字段 - 值的映射结构，适合存储结构化对象，支持单字段精准读写，无需全量序列化 / 反序列化，内存开销远低于 String 序列化存储对象；
    - 底层实现：2 种编码自动切换，listPack（Redis7.0 + 默认，存储小体量哈希）、哈希表（hashtable，存储大体量哈希）；
    - 典型场景：商品 / 用户信息存储、购物车、多维度计数器。
    
3. **List（列表）**
    
    - 核心特点：有序可重复，支持双向头尾 O (1) 复杂度的插入 / 弹出操作，可模拟队列、栈、双向链表结构；
    - 底层实现：quickList（快速列表，由多个 listPack 组成的双向链表，平衡内存开销和读写性能）；
    - 典型场景：轻量消息队列、用户 Timeline 动态流、消息分页、任务队列。
    
4. **Set（集合）**
    
    - 核心特点：无序不可重复，原生支持交集、并集、差集的原子操作，支持随机弹出元素；
    - 底层实现：2 种编码自动切换，intSet（整数集合，存储全整型、小体量集合）、哈希表（存储非整型、大体量集合）；
    - 典型场景：点赞 / 收藏 / 关注、共同好友 / 共同关注、用户标签、随机抽奖。
    
5. **Sorted Set (ZSet，有序集合)**
    
    - 核心特点：元素不可重复，每个元素关联一个 double 类型的 score 权重，按 score 自动排序，支持范围查询、排名查询、权重更新；
    - 底层实现：2 种编码自动切换，listPack（存储小体量有序集合）、跳表 (skiplist)+ 哈希表（存储大体量有序集合，哈希表保证元素唯一性，跳表保证排序和范围查询的高性能）；
    - 典型场景：实时排行榜、延时队列、带权重的任务调度、范围筛选场景。
    

#### 【加分项延伸】

1. 进阶数据类型延伸：除 5 种基础类型外，Redis 原生支持 Bitmap、HyperLogLog、Geo、Stream、Bitfield 等特殊类型，分别适配海量基数统计、地理位置、消息队列等专业场景；
2. 生产选型技巧：存储对象优先用 Hash 而非 String 序列化，可实现单字段更新，减少网络和序列化开销；小体量数据优先用压缩编码，可降低 50% 以上的内存开销；
3. 性能边界：避免对 List 执行中间元素的插入 / 删除操作（O (n) 复杂度），避免对 ZSet 执行全量范围查询，生产环境需限制单次操作的元素数量。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能说 List 底层是 zipList（Redis7.0 已完全废弃 zipList，替换为 listPack）；绝对不能混淆 Set 和 ZSet 的底层实现；
- 高频追问预判：ZSet 为什么用跳表而不用红黑树？Hash 类型的扩容机制？

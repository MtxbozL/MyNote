#### 【核心得分点】

Redis 是一款**基于内存的开源键值型 NoSQL 数据库**，原生支持数据持久化、丰富数据结构、分布式高可用能力；核心优势是极致性能、多数据模型、原生原子性、弹性扩展能力；与关系型数据库是互补而非替代关系，典型场景覆盖分布式缓存、分布式锁、实时计数、排行榜等。

#### 【专家级完整作答】

首先，Redis 的核心定位是**内存优先的高性能数据存储系统**，采用 C 语言开发，核心命令执行采用单线程模型（避免多线程锁竞争开销），单机 QPS 可达 10 万 +，同时通过 RDB/AOF 机制解决内存数据库的宕机数据丢失问题。

和传统关系型数据库（如 MySQL）相比，核心差异与优势体现在 4 个维度：

1. **存储介质与性能差异**：Redis 核心操作基于内存，延迟在微秒级；关系型数据库基于磁盘，延迟在毫秒级，性能差距可达上千倍；
2. **数据模型差异**：Redis 采用键值模型，原生支持 String、Hash、List、Set、ZSet 等 10 + 种数据结构，无需建表，可直接适配复杂业务逻辑；关系型数据库采用关系模型，需提前设计表结构、约束、索引，灵活性弱；
3. **并发模型差异**：Redis 单线程命令执行模型天然保证命令原子性，无锁竞争；关系型数据库需通过事务、锁机制保证并发安全，高并发场景下开销极大；
4. **扩展能力差异**：Redis 原生支持主从复制、哨兵、Cluster 分片集群，可实现读写分离、水平分片扩展，适配海量数据与高并发场景；关系型数据库的分布式扩展能力较弱，通常仅支持主从架构，水平分片需依赖中间件。

基于以上特性，Redis 的典型生产业务场景包括：分布式缓存、分布式锁、接口限流、实时计数器、用户会话存储、实时排行榜、延时队列、共同关注 / 好友推荐、地理位置服务等。

#### 【加分项延伸】

1. 版本特性延伸：Redis 6.0 + 引入多线程 IO 模型，进一步提升网络 IO 处理能力；Redis 7.0 优化了 listPack 编码、AOF 增量重写、函数功能，进一步提升性能和稳定性；
2. 选型边界延伸：和 Memcached 相比，Redis 支持更丰富的数据结构、持久化、主从高可用，仅纯简单 KV 缓存场景可考虑 Memcached，其余场景 Redis 均为首选；
3. 架构实践延伸：生产环境中，Redis 核心定位是「加速读、抗并发」，核心数据仍需落地关系型数据库，形成「数据库持久化 + Redis 加速」的经典分层架构。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能说「Redis 是单线程的」（正确表述：Redis 核心命令执行是单线程，IO、持久化等操作有独立子线程）；绝对不能说「Redis 可以替代 MySQL」；
- 高频追问预判：Redis 为什么用单线程执行核心命令？Redis 和 MongoDB 的选型区别？

#### 【核心得分点】

RDB 是快照式全量持久化，通过 fork 子进程生成内存数据的二进制快照，优点是恢复快、体积小，缺点是存在数据丢失风险；AOF 是日志式增量持久化，记录每一条写命令，优点是数据安全性高，缺点是恢复慢、文件体积大；生产环境 Redis 4.0 + 优先选择**RDB+AOF 混合持久化**，兼顾性能和数据安全。

#### 【专家级完整作答】

Redis 是内存数据库，持久化的核心目标是解决「宕机后内存数据丢失」的问题，官方提供 RDB、AOF 两种持久化机制，以及两者结合的混合持久化方案，具体如下：

##### 一、RDB（Redis DataBase）快照持久化

1. **实现原理**
    
    RDB 是某一时间点内存全量数据的二进制快照，核心基于 fork 子进程 + Copy-On-Write（写时复制）机制实现：
    
    - 触发持久化时，主进程 fork 出一个子进程，fork 瞬间主进程会短暂阻塞，子进程共享主进程的内存地址空间；
    - 子进程遍历内存数据，将其写入临时 RDB 文件，写入完成后，原子替换旧的 RDB 文件；
    - 写时复制机制：fork 之后，主进程修改数据时，会复制对应内存页，子进程仍使用 fork 时的内存快照，保证快照数据的一致性。
    
2. **核心优缺点**
    
    - 优点：RDB 是二进制压缩文件，体积小，宕机后全量数据恢复速度远快于 AOF；对主进程的性能影响极小，适合灾备、数据迁移场景；
    - 缺点：无法做到实时持久化，两次快照之间宕机，会丢失最后一次快照后的所有数据；数据量极大时，fork 子进程的耗时会显著增加，可能导致主进程阻塞。
    

##### 二、AOF（Append Only File）日志持久化

1. **实现原理**
    
    AOF 以日志形式，记录 Redis 的每一条写命令，服务重启时，通过重放 AOF 文件中的所有写命令恢复数据，核心分为 3 个阶段：
    
    - 命令写入：写命令执行完成后，写入 AOF 缓冲区，而非直接写入磁盘；
    - 磁盘刷盘：根据配置的刷盘策略，将缓冲区的数据写入磁盘，核心 3 种策略：
        
        - always：每条命令都刷盘，数据零丢失，性能极差，生产禁用；
        - everysec：每秒刷盘一次，默认策略，最多丢失 1 秒数据，平衡性能和安全；
        - no：由操作系统控制刷盘时机，不可控，生产禁用；
        
    - AOF 重写：当 AOF 文件体积达到阈值时，fork 子进程对 AOF 文件进行重写，合并冗余命令，压缩文件体积，同时保证重写期间的新写命令不丢失。
    
2. **核心优缺点**
    
    - 优点：数据安全性极高，默认策略最多丢失 1 秒数据；AOF 重写可自动压缩文件体积；支持误操作数据恢复（如误执行 FLUSHALL，可修改 AOF 文件删除该命令后重放恢复）；
    - 缺点：AOF 文件体积远大于 RDB 文件；宕机后数据恢复速度远慢于 RDB；高频写入场景下，AOF 刷盘会带来磁盘 IO 压力，可能影响主进程性能。
    

##### 三、生产环境选型逻辑

1. **核心业务，不允许数据丢失**：开启**RDB+AOF 混合持久化**（Redis4.0 + 支持，Redis7.0 默认开启），AOF 文件以 RDB 快照为头部，增量命令为尾部，兼顾 RDB 的恢复速度和 AOF 的数据安全性；
2. **纯缓存场景，可接受部分数据丢失**：仅开启 RDB，配置合理的快照触发条件，最大化服务性能；
3. **数据安全性要求极高，零容忍数据丢失**：仅开启 AOF，刷盘策略使用 everysec，同时定期备份 RDB 文件用于灾备。

#### 【加分项延伸】

1. 生产参数优化：RDB 快照触发频率建议业务低峰期执行，避免高峰频繁 fork；repl-backlog-size 配置合理值，减少主从全量复制；AOF 重写自动触发阈值建议调大，避免高峰频繁重写；
2. 阻塞优化：fork 子进程的耗时与内存使用量正相关，单实例内存建议控制在 10GB 以内，避免 fork 阻塞超过 1 秒；关闭 Linux 透明大页（THP），可大幅降低 fork 耗时；
3. 灾备实践：生产环境禁止将持久化文件存储在系统盘，需挂载独立的数据盘；定期将 RDB 文件备份到异地存储，用于极端场景的数据恢复。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能说「RDB fork 子进程会全程阻塞主进程」（仅 fork 瞬间阻塞，后续子进程执行不阻塞主进程）；绝对不能说错 AOF 默认刷盘策略是 always；
- 高频追问预判：AOF 重写的完整流程？混合持久化的实现原理？fork 阻塞的优化手段？

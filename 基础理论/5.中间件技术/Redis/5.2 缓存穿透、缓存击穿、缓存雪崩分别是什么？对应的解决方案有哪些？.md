#### 【核心得分点】

三者的核心根因都是**缓存层失效，请求直接打到数据库，导致数据库压力骤增甚至宕机**；穿透是请求不存在的 key，击穿是单热点 key 过期，雪崩是大量 key 同时过期或 Redis 集群宕机；解决方案需从全链路分层设计，从接入层到缓存层再到数据库层层层兜底。

#### 【专家级完整作答】

这三个问题是 Redis 缓存架构设计中必须解决的三大经典问题，也是生产环境最常见的数据库故障诱因，具体定义、根因和解决方案如下：

##### 一、缓存穿透

1. **定义与根因**：大量请求查询**既不存在于 Redis 缓存，也不存在于后端数据库**的 key，导致每一次请求都穿透缓存层，直接打到数据库，数据库瞬间被打满。
    
    典型触发场景：恶意攻击、爬虫批量遍历不存在的 ID、业务参数校验异常。
2. **分层解决方案**
    
    - 前置拦截：接口层增加参数合法性校验，拦截非法 ID、格式错误的请求；接入层增加限流、黑白名单，拦截恶意攻击流量；
    - 缓存层兜底：对数据库查询为空的结果，设置短过期时间（如 30-60 秒）的空缓存，避免同一非法 key 重复打库；
    - 精准拦截：使用布隆过滤器，将所有存在的 key 提前写入布隆过滤器，请求先经过布隆过滤器校验，不存在的 key 直接拦截，无需访问缓存和数据库。
    

##### 二、缓存击穿

1. **定义与根因**：某一个**高并发访问的热点 key 突然过期**，瞬间海量并发请求绕过缓存，直接打到数据库，导致数据库压力暴增甚至宕机。
    
    典型触发场景：爆款商品详情缓存、秒杀活动缓存过期。
2. **分层解决方案**
    
    - 热点 key 永不过期：核心热点 key 不设置过期时间，由后台异步任务定时更新缓存，从根源上避免过期问题；
    - 互斥锁控制：热点 key 过期时，仅允许一个线程去查询数据库并更新缓存，其余线程等待缓存更新完成后重试，避免大量请求同时打库；
    - 提前预热：秒杀、大促等活动前，提前将热点数据写入缓存，设置合理的过期时间，避免活动期间缓存过期；
    - 多级缓存：增加本地缓存（如 Caffeine）作为一级缓存，Redis 作为二级缓存，热点 key 先命中本地缓存，大幅降低 Redis 过期带来的风险。
    

##### 三、缓存雪崩

1. **定义与根因**：**大量缓存 key 在同一时间集中过期**，或 Redis 集群整体宕机，导致所有请求全部打到数据库，引发数据库连锁宕机，整个服务雪崩。
    
    典型触发场景：缓存初始化时给所有 key 设置相同的过期时间、Redis 集群主节点全量宕机。
2. **分层解决方案**
    
    - 过期时间打散：给所有 key 的过期时间增加随机偏移量（如基础过期时间 + 0-300 秒随机值），避免大量 key 同时过期；
    - 高可用架构：搭建 Redis 哨兵 / Cluster 集群，实现主从自动故障转移，避免 Redis 单点故障；核心机房部署多活架构，避免单机房宕机导致缓存整体不可用；
    - 服务层保护：接入层增加限流、熔断、降级机制，当数据库负载达到阈值时，触发熔断，拒绝非核心请求，保护核心业务；
    - 多级缓存兜底：本地缓存 + Redis 分布式缓存的多级架构，即使 Redis 集群宕机，本地缓存仍可承接大部分热点请求，避免全量请求打库；
    - 缓存预热：系统重启、大促前，提前预热核心缓存数据，避免启动后大量缓存 miss 集中打库。
    

#### 【加分项延伸】

1. 生产优先级建议：三大问题的解决方案有优先级之分，优先做好「过期时间打散、参数校验、限流熔断」等低成本高收益的方案，再考虑布隆过滤器、多级缓存等进阶方案；
2. 热点 key 治理：生产环境可通过 Redis 监控、客户端埋点实时发现热点 key，自动将热点 key 同步到本地缓存，实现自动化的热点防护；
3. 极端场景兜底：数据库层需做好读写分离，读请求优先打到从库，避免主库被打挂，同时设置数据库连接池上限，避免连接被打满导致服务不可用。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能混淆缓存击穿和缓存雪崩（击穿是单热点 key，雪崩是大量 key 或集群整体故障）；绝对不能说布隆过滤器可以 100% 解决穿透（布隆过滤器存在误判率）；
- 高频追问预判：布隆过滤器的实现原理和误判率优化？多级缓存架构的一致性如何保证？

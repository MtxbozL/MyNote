#### 【核心得分点】

Redis 过期 key 采用**惰性删除 + 定期删除**的组合策略，平衡 CPU 和内存开销；内存达到 maxmemory 上限时触发内存淘汰，Redis 4.0 + 共 8 种淘汰策略，核心分为 LRU/LFU/ 随机 / 不淘汰四大类，通用缓存场景首选 allkeys-lru，核心数据场景首选 volatile-lru，默认策略为 noeviction。

#### 【专家级完整作答】

##### 一、key 过期删除策略

Redis 的过期删除策略，核心是解决「如何高效清理过期 key，同时最小化对服务性能的影响」，最终采用**惰性删除 + 定期删除**的组合方案：

1. **惰性删除**
    
    - 实现原理：key 被客户端访问时，才会检查是否过期，若过期则立即删除并返回空；
    - 优缺点：最大化节省 CPU 资源，完全不做额外的过期扫描；缺点是大量过期 key 未被访问，会持续占用内存，造成内存浪费。
    
2. **定期删除**
    
    - 实现原理：Redis 每隔固定时间（默认 100ms），从过期 key 字典中随机抽取一定数量的 key，检查并删除过期 key；若本次抽取的 key 中，过期占比超过 25%，则重复抽取扫描，同时限制单次扫描的最大时长，避免阻塞主进程；
    - 优缺点：平衡 CPU 和内存开销，通过随机扫描限制过期 key 的内存占用；缺点是无法清理所有过期 key，仍会有部分过期 key 残留。
    

两种策略互补，既避免了 CPU 的过度消耗，也最大程度限制了过期 key 的内存浪费。

##### 二、内存淘汰机制

当 Redis 使用的内存达到配置的 maxmemory 上限时，会触发内存淘汰机制，按照预设的策略淘汰 key，腾出内存供新写入操作使用。Redis 4.0 + 共 8 种淘汰策略，可分为 4 大类：

1. **不淘汰策略**
    
    - noeviction：默认策略，不淘汰任何 key，新写入命令直接返回报错，读操作正常执行；
    - 适用场景：核心数据不允许丢失、不允许自动淘汰的场景，如分布式锁、核心配置存储。
    
2. **LRU 系列策略（最近最少使用）**
    
    - allkeys-lru：对所有 key，淘汰最近最少使用的 key；
    - volatile-lru：仅对设置了过期时间的 key，淘汰最近最少使用的 key；
    - 适用场景：allkeys-lru 是通用缓存场景的首选，适配绝大多数热点数据缓存场景；volatile-lru 适用于需保留持久化核心数据、仅淘汰临时缓存数据的场景。
    
3. **LFU 系列策略（使用频率最低）**
    
    - allkeys-lfu：对所有 key，淘汰使用频率最低的 key；
    - volatile-lfu：仅对设置了过期时间的 key，淘汰使用频率最低的 key；
    - 适用场景：热点数据稳定、长期高频访问的场景，如爆款商品缓存、核心接口缓存，比 LRU 更贴合真实的热点数据访问规律。
    
4. **随机淘汰策略**
    
    - allkeys-random：对所有 key，随机淘汰；
    - volatile-random：仅对设置了过期时间的 key，随机淘汰；
    - 适用场景：key 访问概率完全均等的场景，如轮询任务存储，生产环境极少使用。
    

#### 【加分项延伸】

1. 生产配置最佳实践：maxmemory 建议配置为机器内存的 50%-70%，预留内存给 fork 子进程、缓冲区、系统内核；通用缓存场景强制配置 allkeys-lru 策略，避免默认 noeviction 导致写入失败；
2. 算法优化：Redis 的 LRU 是近似 LRU 算法，通过随机采样 key 实现，而非全局排序，避免全局排序的性能开销，可通过配置调整采样数量，平衡淘汰精度和性能；
3. 业务优化：给 key 的过期时间增加随机偏移量，避免大量 key 同时过期，导致定期扫描阻塞主进程，同时减少缓存雪崩风险。

#### 【避坑 & 追问预判】

- 致命避坑：绝对不能说 Redis 采用「定时删除」策略（定时扫描全量过期 key，CPU 开销极大，Redis 未采用）；绝对不能混淆 volatile 和 allkeys 系列策略的适用范围；
- 高频追问预判：LRU 和 LFU 的核心区别？Redis 的近似 LRU 算法实现原理？

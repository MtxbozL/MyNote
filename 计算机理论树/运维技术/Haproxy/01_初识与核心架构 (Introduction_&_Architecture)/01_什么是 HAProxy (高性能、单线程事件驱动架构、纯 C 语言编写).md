# 第一章 HAProxy 核心概述与基础架构原理

## 1.1 HAProxy 核心定义与技术定位

HAProxy（High Availability Proxy）是一款开源、高性能、高可靠的**TCP/HTTP 反向代理与负载均衡器**，严格遵循 OSI 网络模型，可稳定工作在**传输层（4 层）**与**应用层（7 层）**，核心设计目标为极致的高并发吞吐、亚毫秒级延迟、无中断服务可用性与精细化的流量管控能力。

其核心技术边界明确：

- 核心能力：反向代理、流量负载均衡、4/7 层流量转发与规则管控
- 非核心能力：不具备 Web 服务器功能、不支持正向代理常规场景、无完整的静态资源服务能力、仅提供有限的 HTTP 缓存能力
- 核心应用场景：互联网业务入口流量调度、微服务集群流量治理、数据库 / 中间件 TCP 服务高可用、跨区域服务容灾与流量调度

## 1.2 HAProxy 发展历程与版本基线

HAProxy 由 Willy Tarreau 于 2000 年首次发布，核心演进始终围绕高性能、高可靠与协议兼容性展开，关键版本的技术里程碑如下：

表格

|版本系列|核心技术突破|版本定位|
|---|---|---|
|1.4.x|引入 HTTP Keep-Alive、TCP 连接复用、基础 ACL 规则引擎|稳定基础版，奠定 4 层核心能力|
|1.5.x|首次支持 SSL/TLS 卸载、IPv6、SPDY 协议、七层健康检查|里程碑版本，完整支持 7 层核心能力|
|1.8.x|引入多线程模型、HTTP/2 全栈支持、细粒度限流、进程间通信|性能飞跃版本，适配多核架构|
|2.0.x|原生支持 gRPC 代理、端到端 TLS、Prometheus 原生指标、L7 重试|云原生适配版本|
|2.4.x|引入 QUIC 与 HTTP/3 支持、动态配置增强、eBPF 集成、硬件加速优化|协议全栈版本|
|2.8.x|LTS 长期支持版本，完善 HTTP/3 稳定性、动态 SSL 配置、安全加固、可观测性增强|生产环境推荐基线版本|

**学习与生产环境基线建议**：采用 2.8.x LTS 版本，本文档所有技术特性与配置均基于该版本展开，避免跨版本特性差异导致的学习偏差。

## 1.3 HAProxy 核心架构模型

HAProxy 的高性能与高可用性，完全由其底层架构设计决定，核心为**控制平面与数据平面完全分离**的架构，配合事件驱动的 I/O 模型实现极致的流量处理能力。

### 1.3.1 进程与线程模型

HAProxy 采用**Master-Worker 主从架构**，进程角色与职责完全解耦，无交叉流量处理：

1. **Master 进程（控制平面）**
    
    - 单实例唯一，以 root 权限运行，不处理任何业务流量
    - 核心职责：配置文件解析与校验、Worker 进程生命周期管理、信号处理、热重载 / 热升级调度、权限管理、运维接口管控
    - 核心特性：支持零停机配置热重载、无缝二进制升级，全程不中断业务流量
    
2. **Worker 进程（数据平面）**
    
    - 由 Master 进程 fork 生成，数量可配置，默认与 CPU 核心数绑定，以非 root 权限运行
    - 核心职责：全量处理客户端流量、事件循环调度、转发规则执行、健康检查、指标统计
    - 核心设计：Worker 进程间完全无共享状态、无锁竞争，每个 Worker 拥有独立的事件循环、文件描述符池、内存池与连接队列，彻底避免多核架构下的锁开销
    
3. **线程模型配置规范**
    
    - `nbproc`：多进程配置，每个 Worker 为独立进程，适用于 NUMA 架构、高隔离性需求场景
    - `nbthread`：单进程多线程配置，单个 Worker 进程内启动多个线程，共享文件描述符，适用于通用多核服务器，配置更简洁，性能损耗更低
    - 生产环境最佳实践：单进程多线程模型，线程数与物理 CPU 核心数 1:1 绑定，关闭超线程调度，避免上下文切换开销
    

### 1.3.2 I/O 事件驱动模型

HAProxy 完全采用**异步非阻塞事件驱动 I/O 模型**，是其支撑数十万并发连接的核心基础：

1. **I/O 多路复用机制**
    
    - Linux 平台默认使用`epoll`系统调用，BSD/macOS 平台使用`kqueue`，兼容`poll/select`作为兜底
    - 默认采用边缘触发（ET）模式，仅在状态变化时触发事件通知，大幅减少内核态与用户态的数据拷贝次数，降低事件触发的 CPU 开销
    
2. **零拷贝技术**
    
    - 原生支持`splice()`、`sendfile()`系统调用，实现 TCP 流量在内核态的直接转发，避免用户态与内核态的冗余数据拷贝，大幅提升 4 层流量转发吞吐
    - 针对 HTTP 响应，支持静态内容零拷贝发送，降低 CPU 占用
    
3. **内存管理机制**
    
    - 采用固定大小内存池 + 分级内存分配机制，针对连接、请求、缓冲区等高频对象实现预分配与复用，彻底避免频繁内存申请 / 释放导致的内存碎片与系统调用开销
    - 严格的内存边界管控，无内存泄漏风险，可支持连续数月无重启稳定运行
    

## 1.4 HAProxy OSI 层级工作模型与转发逻辑

HAProxy 的核心转发能力分为**4 层 TCP 代理**与**7 层 HTTP 代理**两大模式，二者的工作机制、能力边界与性能特性存在本质差异，是所有配置与规则设计的核心基础。

### 1.4.1 四层（传输层）TCP 代理模式

TCP 模式下，HAProxy 工作在 OSI 第四层，仅基于 TCP 协议头部信息完成流量转发，不解析任何应用层数据。

1. **核心转发流程**
    
    1. 客户端与 HAProxy 完成 TCP 三次握手，建立客户端侧 TCP 连接
    2. HAProxy 基于预设的负载均衡算法，选择后端上游服务器
    3. HAProxy 与选中的上游服务器完成 TCP 三次握手，建立服务端侧 TCP 连接
    4. 建立双向 TCP 流管道，HAProxy 仅做二进制数据的双向透传，不修改、不解析应用层报文
    5. 连接终止时，HAProxy 负责双向连接的正常关闭与资源回收
    
2. **核心特性**
    
    - 极低的转发延迟与极高的吞吐性能，CPU 占用仅为 7 层模式的 1/3~1/2
    - 协议无感知，支持所有基于 TCP 的应用层协议（MySQL、Redis、SSH、RTSP、SMTP 等）
    - 基于 TCP 五元组（源 IP、源端口、目的 IP、目的端口、传输层协议）实现会话绑定与访问控制
    
3. **能力边界**
    
    - 无法解析应用层数据，不支持基于 HTTP 请求头、URL 等应用层信息的规则匹配
    - 无法实现 SSL/TLS 卸载、HTTP 请求改写、内容路由等 7 层能力
    

### 1.4.2 七层（应用层）HTTP 代理模式

HTTP 模式下，HAProxy 工作在 OSI 第七层，会完整解析 HTTP/HTTPS 协议报文，基于应用层信息实现精细化的流量管控，是 HAProxy 的核心竞争力所在。

1. **核心转发流程**
    
    1. 客户端与 HAProxy 完成 TCP 三次握手，HTTPS 场景下同步完成 TLS 握手与证书校验
    2. HAProxy 完整接收并解析客户端 HTTP 请求报文（请求行、请求头、请求体）
    3. 基于 ACL 规则引擎匹配请求特征，执行预设的流量管控动作，选择对应的后端服务器集群
    4. 基于负载均衡算法，选中集群内的上游服务器，与服务器完成 TCP 握手（HTTPS 场景下可完成后端 TLS 握手）
    5. 向服务器转发处理后的 HTTP 请求，接收服务器的 HTTP 响应报文
    6. 对响应报文执行预设的改写、过滤规则，将响应返回给客户端
    7. 完成请求后，基于 HTTP Keep-Alive 配置管理连接复用
    
2. **核心特性**
    
    - 完整的应用层报文解析能力，支持 HTTP/1.0、HTTP/1.1、HTTP/2、HTTP/3 全协议栈
    - 支持精细化内容路由、请求 / 响应改写、SSL/TLS 全生命周期管理、会话保持、限流限速、访问控制等能力
    - 可实现基于域名、URL、请求头、Cookie、客户端 IP 等多维度的流量调度
    
3. **能力边界**
    
    - 必须解析应用层协议，仅支持 HTTP/HTTPS/gRPC 等兼容 HTTP 语义的应用层协议
    - 相比 4 层模式，存在一定的解析开销，延迟与 CPU 占用更高
    

### 1.4.3 4 层与 7 层代理模式核心技术差异

表格

|对比维度|4 层 TCP 代理模式|7 层 HTTP 代理模式|
|---|---|---|
|OSI 工作层级|传输层（L4）|应用层（L7）|
|报文处理深度|仅解析 TCP 头部|完整解析 HTTP 全报文|
|核心转发依据|TCP 五元组|应用层请求特征|
|协议兼容性|所有 TCP 协议|仅 HTTP/HTTPS/ 兼容 HTTP 语义的协议|
|转发性能|极高，亚毫秒级延迟|高，存在固定解析开销|
|并发承载能力|数十万级并发连接|十万级并发请求|
|精细化管控能力|弱，仅支持传输层维度管控|极强，支持全应用层维度管控|

## 1.5 HAProxy 核心技术特性体系

HAProxy 的特性体系围绕**高可用、高性能、精细化管控、安全防护、可观测性**五大核心维度构建，无冗余功能设计，所有特性均面向生产环境流量调度场景。

### 1.5.1 高可用核心特性

- **多层级健康检查**：支持 4 层 TCP 端口探测、7 层 HTTP 状态码 / 响应内容探测、脚本化自定义健康检查，可配置探测间隔、阈值、超时时间，自动隔离故障节点，故障恢复后自动上线
- **零停机服务能力**：支持配置热重载、二进制程序热升级，全程不中断现有 TCP 连接与 HTTP 请求，满足 7×24 小时无间断服务要求
- **故障转移与容灾**：支持主备节点集群、备份服务器池、跨区域容灾调度、请求重试、故障节点熔断机制，避免单点故障导致的业务不可用
- **会话保持能力**：支持基于源 IP、Cookie、请求头的会话保持，确保同一客户端的请求始终转发至同一上游服务器，适配有状态业务场景
- **优雅停机**：支持 Worker 进程优雅关闭，完成现有请求处理后再终止进程，避免业务中断

### 1.5.2 高性能核心特性

- 异步非阻塞事件驱动 I/O 模型，内核态多路复用机制，支持单实例数十万级并发连接
- 内核态零拷贝转发技术，最小化用户态 / 内核态数据拷贝开销
- 无锁化 Worker 进程设计，多核 CPU 线性性能扩展
- TCP 协议优化：支持窗口缩放、拥塞控制算法自定义、连接复用、超时精细化管控，最小化网络传输开销
- SSL/TLS 硬件加速：支持 Intel QAT、AES-NI 硬件加密加速，大幅降低 TLS 握手与加解密的 CPU 开销
- HTTP 连接复用：支持客户端与服务端侧的 HTTP Keep-Alive，减少 TCP 握手开销，提升请求处理效率

### 1.5.3 精细化流量管控特性

- **ACL 规则引擎**：全功能的访问控制列表引擎，支持基于客户端 IP、端口、域名、URL、请求头、Cookie、响应状态码、时间窗口等数百个维度的特征匹配，是所有流量管控规则的核心基础
- **内容路由**：基于 ACL 规则实现多维度流量分发，支持域名路由、路径路由、用户分群路由、灰度发布、蓝绿部署等核心业务场景
- **请求 / 响应全生命周期管控**：支持 HTTP 请求头 / 响应头增删改、URL 重写、请求体截断、响应过滤、重定向等全报文处理能力
- **限流限速**：支持基于客户端 IP、会话、域名、接口的多维度限流，支持固定窗口、滑动窗口、漏桶、令牌桶等限流算法，可配置峰值限流、平均速率限制、连接数限制
- **负载均衡算法**：内置轮询、加权轮询、最小连接数、加权最小连接数、源 IP 哈希、URI 哈希、请求头哈希、随机等十余种负载均衡算法，适配不同业务场景

### 1.5.4 安全防护特性

- 四层 DDoS 防护：支持 SYN 洪水防护、ACK 洪水防护、UDP 洪水防护、连接数限制，可抵御常见的网络层攻击
- 七层应用防护：支持 HTTP 请求格式校验、恶意请求拦截、爬虫防护、接口频率限制、HTTP 安全头自动注入
- 访问控制：支持基于 IP 地址段、端口、协议的黑白名单管控，支持 TCP Wrappers 集成
- SSL/TLS 安全管控：支持全版本 TLS 协议、自定义加密套件、证书自动管理、OCSP stapling、客户端证书校验，可实现全链路 HTTPS 加密
- 权限隔离：Master-Worker 架构实现权限最小化，Worker 进程以非 root 权限运行，避免权限溢出风险

### 1.5.5 可观测性特性

- 原生内置统计页面，提供实时连接数、请求数、响应时长、健康检查状态、错误率等数百项核心指标
- 原生支持 Prometheus 指标输出，可无缝对接监控告警体系
- 全维度日志系统，支持系统日志、访问日志、错误日志、健康检查日志，可自定义日志格式与输出维度，支持 TCP/UDP/Syslog 日志输出
- 实时状态监控接口，支持 CLI 命令行、API 接口实时查询实例运行状态、配置信息、上游节点状态

## 1.7 学习环境准备与基线校验

本章学习完成后，需完成基础环境搭建，为后续章节的配置学习提供基础：

1. **环境要求**：Linux 操作系统（推荐 CentOS 7+/Ubuntu 20.04+），内核版本 3.10+，具备基础网络环境
2. **安装方式**
    
    - 包管理器安装（快速学习）：
        
        bash
        
        运行
        
        ```
        # Ubuntu/Debian
        apt update && apt install -y haproxy
        # CentOS/RHEL
        yum install -y haproxy
        ```
        
    - 源码编译安装（生产环境 / 自定义特性）：从 HAProxy 官方源码仓库下载 2.8.x LTS 版本，编译时启用 SSL、Zlib、Prometheus 等核心特性
    
3. **基线校验命令**
    
    bash
    
    运行
    
    ```
    # 查看版本信息，确认版本符合基线要求
    haproxy -v
    # 配置文件语法校验，核心运维命令
    haproxy -c -f /etc/haproxy/haproxy.cfg
    # 查看编译启用的特性
    haproxy -vv
    ```

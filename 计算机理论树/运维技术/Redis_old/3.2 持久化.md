## 一、核心概述

Redis 事务是 Redis 原生提供的**批量命令原子性执行单元**，核心能力是将多个 Redis 命令打包，按入队顺序串行化执行，执行期间不会被其他客户端的命令打断，是 Redis 实现并发控制、批量操作原子性的核心方案之一。

### 核心特性

1. **顺序执行**：事务内的命令严格按入队顺序串行执行，执行期间不会插入其他客户端的命令
2. **隔离性**：事务执行全程独占 Redis 主线程，其他客户端的命令必须等待事务执行完成后才能执行，完全无并发干扰
3. **部分原子性**：仅保证「入队错误全不执行」，**执行阶段的命令错误不支持回滚**，正确命令会继续执行（与关系型数据库事务的核心差异）
4. **乐观锁支持**：通过`WATCH`命令实现 CAS（比较并交换）乐观锁，解决并发场景下的竞态问题
5. **轻量无额外开销**：基于 Redis 单线程模型实现，无锁、无事务日志的额外性能开销，执行效率极高

### 与关系型数据库 ACID 事务的核心差异

|特性|Redis 事务|MySQL/PostgreSQL 等关系型数据库事务|
|---|---|---|
|原子性|仅支持入队错误的全量放弃，执行时错误不回滚，不满足完整原子性|完整支持原子性，要么全执行成功，要么全部回滚|
|一致性|仅能保证数据符合基础约束，依赖业务层保证业务一致性|完整支持一致性，通过约束、事务日志保证数据从一个一致状态到另一个一致状态|
|隔离性|天然串行化隔离级别，无脏读、不可重复读、幻读问题|支持 4 种隔离级别，可按需配置|
|持久性|依赖 Redis 持久化配置，默认不保证持久性|完整支持持久性，事务提交后数据永久落盘|

## 二、底层实现原理

Redis 事务的底层实现极简且高效，核心依赖**命令队列**和**CAS 监控机制**两大核心模块，完整执行流程分为 3 个阶段。

### 2.1 事务执行的三大核心阶段

#### 阶段 1：事务开启（MULTI）

客户端执行`MULTI`命令后，Redis 会将该客户端的状态从「非事务状态」切换为「事务状态」，后续除事务专属命令外，所有常规命令都不会立即执行。

#### 阶段 2：命令入队

事务状态下，Redis 会对收到的命令做分类处理：

- **立即执行的命令**：事务专属命令（`MULTI/EXEC/DISCARD/WATCH/UNWATCH`），会立即执行并返回结果
- **入队等待的命令**：所有常规读写命令（`GET/SET/INCR/LPUSH`等），不会立即执行，而是进入客户端专属的**先进先出（FIFO）事务命令队列**，服务端返回`QUEUED`确认入队成功

#### 阶段 3：事务执行 / 放弃

- **执行事务（EXEC）**：客户端执行`EXEC`命令后，Redis 会按入队顺序串行执行队列中的所有命令，执行完成后：
    
    1. 按顺序返回所有命令的执行结果
    2. 清空事务命令队列
    3. 客户端切换回非事务状态
    4. 自动取消所有`WATCH`的 key 监控
    
- **放弃事务（DISCARD）**：客户端执行`DISCARD`命令后，Redis 会：
    
    1. 直接清空事务命令队列，放弃所有入队的命令
    2. 客户端切换回非事务状态
    3. 自动取消所有`WATCH`的 key 监控

### 2.2 WATCH 乐观锁的底层实现

`WATCH`是 Redis 事务实现并发控制的核心，本质是**CAS（Compare And Swap）比较并交换**机制，用于监控一个或多个 key 的变化，保证事务执行的前提是「被监控的 key 未被其他客户端修改」。

#### 核心执行逻辑

1. 客户端执行`WATCH key [key ...]`后，Redis 会将该客户端加入到数据库全局`watched_keys`字典的监听列表中：
    
    - 字典 key：被监控的 Redis key
    - 字典 value：监听该 key 的所有客户端列表
    
2. 当任何客户端执行修改 key 的命令（`SET/INCR/DEL`等）后，Redis 会遍历监听该 key 的所有客户端，给客户端状态打上`REDIS_DIRTY_CAS`（CAS 脏标记）
3. 客户端执行`EXEC`时，Redis 会先检查客户端是否有`REDIS_DIRTY_CAS`标记：
    
    - 有标记：说明被监控的 key 已被修改，直接放弃整个事务，返回`nil`
    - 无标记：正常执行事务队列中的所有命令
#### 关键特性

- 监控有效期：`WATCH`的监控仅在**当前事务周期内有效**，`EXEC/DISCARD`执行后，所有监控会自动取消，下次事务需要重新`WATCH`
- 无过期感知：被监控的 key 过期 / 内存淘汰，**不会触发脏标记**，事务不会失效，这是高频踩坑点
- 集群限制：Redis Cluster 中，`WATCH`的 key 必须和事务内的 key 在同一个哈希槽，否则会报错

## 三、核心命令大全

按功能分为 2 大类，标注核心作用、语法、示例与关键注意事项，适配 Redis 6.0 + 稳定版本。

### 3.1 事务基础命令

|命令|语法|核心作用|示例|关键注意事项|
|---|---|---|---|---|
|MULTI|`MULTI`|开启一个事务，将客户端切换为事务状态|`MULTI`|多次执行 MULTI 不会报错，但无额外效果；执行后除事务专属命令外，所有常规命令均进入队列|
|EXEC|`EXEC`|执行事务队列中的所有命令，按顺序返回执行结果|`EXEC`|若被 WATCH 的 key 被修改，或入队时有语法错误，事务直接放弃，返回 nil|
|DISCARD|`DISCARD`|放弃事务，清空命令队列，客户端切换回非事务状态|`DISCARD`|执行后所有 WATCH 监控自动取消；EXEC 执行后无法再执行 DISCARD|

### 3.2 乐观锁控制命令

|命令|语法|核心作用|示例|关键注意事项|
|---|---|---|---|---|
|WATCH|`WATCH key [key ...]`|监控一个或多个 key，若 EXEC 执行前 key 被其他客户端修改，事务直接放弃|`WATCH goods:stock:1001 user:balance:1001`|必须在 MULTI 之前执行，事务开启后执行 WATCH 会报错；仅监控 key 的修改，不监控过期 / 淘汰|
|UNWATCH|`UNWATCH`|手动取消当前客户端所有 WATCH 的 key 监控|`UNWATCH`|仅在 WATCH 之后、MULTI 之前执行有效；EXEC/DISCARD 会自动取消监控，无需手动执行 UNWATCH|

## 四、Redis 事务的 ACID 特性深度解析

这是面试最高频考点，必须明确 Redis 事务对 ACID 四大特性的支持程度与底层原因。

### 4.1 原子性（Atomicity）

**仅支持部分原子性，不支持完整原子性，无回滚能力**，分两种场景：

1. **入队阶段错误：满足原子性**
    
    若命令入队时出现语法错误、参数数量错误、命令名错误等问题，Redis 会记录错误，执行`EXEC`时直接放弃整个事务，所有命令都不会执行，保证「全不执行」的原子性。

```bash
    127.0.0.1:6379> MULTI
    OK
    127.0.0.1:6379> SET a 1
    QUEUED
    127.0.0.1:6379> SET a  # 语法错误，入队时直接报错
    (error) ERR wrong number of arguments for 'set' command
    127.0.0.1:6379> EXEC
    (error) EXECABORT Transaction discarded because of previous errors.
    # 整个事务被放弃，SET a 1 未执行
```
    
2. **执行阶段错误：不满足原子性，无回滚**
    
    若命令入队时无错误，但执行时出现运行时错误（如对 String 类型执行 List 命令、key 类型不匹配），Redis 会**跳过错误命令，继续执行后续所有正确的命令**，不会回滚已执行的命令，无法保证原子性。

```bash
    127.0.0.1:6379> MULTI
    OK
    127.0.0.1:6379> SET a 1
    QUEUED
    127.0.0.1:6379> LPOP a  # 对String执行List命令，入队无报错，执行时报错
    QUEUED
    127.0.0.1:6379> SET b 2
    QUEUED
    127.0.0.1:6379> EXEC
    1) OK
    2) (error) WRONGTYPE Operation against a key holding the wrong kind of value
    3) OK
    # 错误命令跳过，SET a 1 和 SET b 2 均执行成功，无回滚
```

#### 为什么 Redis 不支持事务回滚？

Redis 官方给出了明确的设计原因：

1. **极致性能优先**：回滚机制需要额外的事务日志、锁与回滚逻辑，会大幅增加 Redis 的复杂度与性能开销，违背 Redis「高性能、低延迟」的核心设计目标
2. **错误场景可预判**：执行时的运行时错误，几乎都是开发阶段的语法、类型使用错误，生产环境极少出现，无需通过回滚机制兜底
3. **单线程模型限制**：Redis 单线程模型无法实现复杂的回滚逻辑，且回滚无法解决并发竞态问题

### 4.2 一致性（Consistency）

**Redis 事务满足数据一致性**，这里的一致性指「数据符合 Redis 的内置约束与规则」：

1. 入队错误：事务全量放弃，数据保持原有状态，符合一致性
2. 执行错误：错误命令被跳过，正确命令正常执行，数据符合 Redis 的类型与语法约束，无脏数据
3. 进程崩溃：若开启了 RDB/AOF 持久化，Redis 重启后会通过持久化文件恢复数据，保证数据一致性；无持久化时，重启后数据清空，也符合空库的一致性约束

> 注意：业务层面的一致性（如转账时 A 扣钱、B 加钱必须同时成功），需要业务层通过前置校验、幂等设计保证，Redis 事务本身无法保证业务一致性。

### 4.3 隔离性（Isolation）

**Redis 事务完全满足隔离性，天然达到串行化隔离级别**。

Redis 是单线程模型，事务执行期间会独占主线程，所有命令串行执行，其他客户端的命令必须等待整个事务执行完成后才能执行，不会出现脏读、不可重复读、幻读等并发问题，是最高级别的隔离性。

### 4.4 持久性（Durability）

**Redis 事务的持久性完全依赖持久化配置，默认不满足持久性**。

1. **无持久化**：Redis 重启后所有数据丢失，完全不满足持久性
2. **RDB 持久化**：快照是周期性执行的，事务执行后到下一次快照之间发生崩溃，数据会丢失，不满足持久性
3. **AOF 持久化**：
    
    - `appendfsync=always`：每条命令执行后立即同步刷盘，事务执行后数据永久落盘，满足持久性
    - `appendfsync=everysec`：每秒刷盘一次，最多丢失 1 秒数据，不满足持久性
    - `appendfsync=no`：完全依赖操作系统刷盘，数据丢失风险极高，不满足持久性

## 五、典型应用场景

Redis 事务仅适用于**简单批量操作、低并发乐观锁控制**场景，高并发复杂场景优先使用 Lua 脚本。

### 5.1 批量命令串行化执行

核心价值是保证多个命令按顺序执行，期间不会被其他客户端的命令打断，避免并发修改导致的数据异常。

- 典型场景：批量配置更新、多 key 同时设置、数据批量初始化
- 示例：

```bash
    # 批量更新用户的多个配置项，保证全部执行，无并发干扰
    MULTI
    SET user:1001:status active
    SET user:1001:vip_level 1
    EXPIRE user:1001:token 86400
    EXEC
```

### 5.2 基于乐观锁的并发库存扣减（秒杀场景）

通过`WATCH`监控库存 key，保证只有库存未被修改时，扣减操作才会执行，避免超卖问题。

- 示例：

```bash
    # 1. 监控库存key，防止并发修改
    WATCH goods:stock:1001
    # 2. 查询当前库存
    GET goods:stock:1001
    # 3. 业务层判断库存>0，开启事务
    MULTI
    # 4. 库存扣减1
    DECR goods:stock:1001
    # 5. 记录用户下单记录
    SADD order:goods:1001 user:1001
    # 6. 执行事务：若库存被其他用户修改，返回nil，扣减失败
    EXEC
```

### 5.3 转账场景的并发控制

保证 A 账户扣钱、B 账户加钱的操作串行执行，配合`WATCH`防止两个账户被并发修改，避免资金异常。

- 示例：

```bash
    # 监控两个账户的余额，防止并发修改
    WATCH user:balance:1001 user:balance:1002
    # 查询余额，判断是否足够转账
    GET user:balance:1001
    # 开启事务
    MULTI
    DECRBY user:balance:1001 100
    INCRBY user:balance:1002 100
    # 执行事务，若余额被修改，事务放弃
    EXEC
```

## 六、常见坑与最佳实践

### 6.1 高频踩坑点

1. **执行时错误不回滚，导致数据不一致**
    
    线上绝对禁止出现执行时的运行时错误，必须在事务开启前完成所有类型、参数、语法的前置校验，避免出现类型不匹配、命令使用错误等问题。
    
2. **WATCH 监控失效场景**
    
    - 被监控的 key 过期 / 内存淘汰，不会触发脏标记，事务仍会执行，导致业务逻辑异常
    - `EXEC/DISCARD`执行后，WATCH 监控自动失效，下次事务必须重新执行 WATCH
    - 集群模式下，WATCH 的 key 与事务 key 不在同一个哈希槽，会直接报错
    
3. **大事务阻塞 Redis 主线程**
    
    事务内的命令过多、执行耗时过长，会独占 Redis 单线程，导致整个实例卡顿、请求超时。线上事务内的命令数量建议控制在 100 个以内，绝对禁止在事务内执行`KEYS *`、`SMEMBERS`大集合、`HGETALL`大哈希等耗时命令。
    
4. **Redis Cluster 集群事务限制**
    
    集群模式下，事务内的**所有 key 必须落在同一个哈希槽**，否则会报错无法执行。解决方案是使用**哈希标签（Hash Tag）**，给所有 key 加上相同的`{xxx}`前缀，保证哈希槽一致。

```bash
    # 正确示例：通过{order}哈希标签保证两个key落在同一个哈希槽
    MULTI
    SET {order}:1001:status paid
    INCR {order}:total:paid
    EXEC
```

5. **高并发下 WATCH 导致的事务频繁失败**
    
    秒杀等高并发场景下，大量客户端同时监控同一个库存 key，只要有一个客户端修改成功，其他所有客户端的事务都会失败，导致大量请求无效，吞吐量极低。这种场景**禁止使用 WATCH + 事务**，优先使用 Lua 脚本实现。

### 6.2 线上最佳实践

1. **优先使用 Lua 脚本替代事务**
    
    Lua 脚本是 Redis 2.6 + 提供的更强大的原子性方案，整个脚本作为一个整体串行执行，中间不会被其他命令打断，支持条件判断、循环、分支逻辑，可完全替代事务，且功能更强大、性能更优。

| 对比维度 | Redis 事务             | Lua 脚本                       |
| ---- | -------------------- | ---------------------------- |
| 原子性  | 仅部分原子性，执行时错误不回滚      | 完整原子性，脚本要么全执行，要么全不执行         |
| 灵活性  | 仅支持批量命令执行，无逻辑判断      | 支持条件判断、循环、分支等复杂逻辑            |
| 并发控制 | 依赖 WATCH 乐观锁，高并发下易失败 | 脚本内可直接做条件判断，无需 WATCH，高并发下更稳定 |
| 性能   | 多次网络往返，开销较大          | 一次网络请求完成所有操作，网络开销更小          |
| 集群支持 | 仅支持单哈希槽内的 key        | 仅支持单哈希槽内的 key，与事务一致          |

选型建议：

- 简单批量命令执行，无复杂逻辑，可使用事务
- 高并发场景、需要条件判断、并发控制的场景，**优先使用 Lua 脚本**
- 秒杀、库存扣减等核心场景，必须使用 Lua 脚本，避免 WATCH 导致的事务频繁失败

2. **严格控制事务大小**
    
    事务内的命令数量建议控制在 100 个以内，单条命令执行耗时控制在 1ms 以内，绝对禁止大事务阻塞主线程。
    
3. **提前完成所有业务校验**
    
    所有业务判断、类型校验、参数校验必须在事务开启前完成，事务内仅保留最终的读写命令，避免执行时出现错误。
    
4. **禁止在事务内执行耗时命令**
    
    绝对禁止在事务内执行全量遍历、大 key 操作、阻塞命令，避免主线程长时间阻塞。
    
5. **集群环境下必须使用哈希标签**
    
    集群模式下使用事务，必须给所有 key 加上相同的哈希标签，保证落在同一个哈希槽，否则事务无法执行。

# Redis 持久化 学习笔记

## 一、核心概述

Redis 是**内存数据库**，所有数据存储在内存中，一旦进程退出、服务器宕机，内存数据会全部丢失。持久化的核心目标，就是将内存中的数据异步写入磁盘，实现数据的持久化存储，保证 Redis 重启后能通过磁盘文件恢复完整数据。

Redis 提供了三种成熟的持久化方案，可按需组合使用：

1. **RDB 快照持久化**：某一时间点内存全量数据的二进制快照，体积小、恢复快，适合灾备
2. **AOF 日志持久化**：记录所有写命令的追加式日志，数据安全性高，适合核心业务
3. **RDB-AOF 混合持久化**：Redis 4.0 + 引入，结合两者优势，是当前线上默认推荐方案

### 核心设计原则

- 持久化操作尽可能不阻塞 Redis 主线程，保证业务读写性能
- 平衡**数据安全性**与**性能开销**，不同方案对应不同的安全与性能平衡点
- 所有持久化操作均通过子进程完成，避免占用主线程资源

---

## 二、RDB（Redis DataBase）快照持久化

RDB 是 Redis 默认开启的持久化方案，本质是**某一时刻内存全量数据的二进制快照文件**，记录了该时间点 Redis 所有数据的完整状态，文件默认名为`dump.rdb`。

### 2.1 核心触发方式

RDB 触发分为**手动触发**和**自动触发**两类，生产环境优先使用自动触发，手动触发仅用于运维场景。

#### 1. 手动触发

|命令|核心特性|适用场景|注意事项|
|---|---|---|---|
|`SAVE`|同步阻塞执行：直接在主线程生成 RDB 文件，执行期间 Redis 完全阻塞，拒绝所有客户端请求|线下运维、停机维护、数据冷备|**生产环境绝对禁止使用**，会导致业务长时间不可用|
|`BGSAVE`|异步非阻塞执行：fork 子进程生成 RDB 文件，仅 fork 瞬间阻塞主线程，fork 完成后主线程继续处理业务请求|线上手动触发快照、灾备备份|同一时间仅允许一个 BGSAVE 子进程执行，避免内存与 IO 过载|

#### 2. 自动触发

满足以下任意条件时，Redis 会自动执行 BGSAVE 生成 RDB 快照：

1. **配置文件规则触发**：`redis.conf`中通过`save`参数配置触发规则，格式为`save <秒数> <修改次数>`，默认配置：

```ini
    save 900 1    # 900秒内至少1次修改，触发BGSAVE
    save 300 10   # 300秒内至少10次修改，触发BGSAVE
    save 60 10000 # 60秒内至少10000次修改，触发BGSAVE
```

> 注意：所有`save`规则是**或**的关系，满足任意一条就会触发；注释掉所有`save`规则会关闭 RDB 自动触发。

2. **服务正常关闭触发**：执行`SHUTDOWN`命令正常关闭 Redis 时，若未开启 AOF，会自动执行 BGSAVE 生成 RDB 快照。
3. **主从全量复制触发**：主从架构中，从节点发起全量复制时，主节点会自动执行 BGSAVE 生成 RDB 文件，发送给从节点完成数据同步。
4. **debug reload 触发**：执行 debug 级别的内存重载命令时，会自动触发 RDB 快照。

### 2.2 BGSAVE 完整执行流程（核心原理）

BGSAVE 是 RDB 持久化的核心，全程通过子进程完成，仅 fork 瞬间阻塞主线程，执行流程如下：

1. 客户端执行`BGSAVE`命令，Redis 首先检查是否存在正在运行的子进程（BGSAVE/BGREWRITEAOF），若存在则直接返回，避免重复 fork 导致资源过载。
2. 父进程执行`fork()`系统调用创建子进程，**fork 瞬间父进程会阻塞**，阻塞时长与 Redis 内存大小正相关（内存越大，页表复制耗时越长）。
3. fork 完成后，父进程立即解除阻塞，继续处理客户端的所有读写请求；子进程继承父进程的内存视图，开始遍历内存数据，生成临时 RDB 快照文件。
4. 子进程完成全量数据写入后，用临时 RDB 文件原子替换旧的 RDB 文件，向父进程发送信号通知执行完成。
5. 父进程更新 RDB 相关元数据，本次 BGSAVE 执行结束。

### 2.3 核心机制：写时复制（COW, Copy-On-Write）

这是 RDB 实现非阻塞快照的核心底层原理，也是面试最高频考点。

- 核心逻辑：fork 子进程后，父子进程**共享同一块物理内存**，内核会将父进程的内存页设置为**只读**；只有当父进程需要修改某一个内存页的数据时，才会触发内核的缺页异常，复制该内存页生成一个新的副本，父进程在新副本上修改，子进程仍使用原内存页。
- 核心价值：保证子进程生成的快照，是**fork 瞬间的内存一致性视图**，父进程后续的修改不会影响子进程的快照生成，同时避免了 fork 时全量内存复制的巨大开销。

### 2.4 核心优劣势

|优势|劣势|
|---|---|
|1. 二进制压缩格式存储，文件体积极小，相同数据集远小于 AOF 文件，适合异地灾备<br><br>2. 数据恢复速度极快，直接加载二进制快照，大数据量下恢复速度远超 AOF<br><br>3. 对主线程性能影响极小，仅 fork 瞬间阻塞，后续子进程操作完全不影响主线程<br><br>4. 适合全量备份、冷备、跨机房数据同步场景|1. 数据安全性低：快照是间隔性生成的，宕机后会丢失最后一次快照之后的所有写数据<br><br>2. fork 阻塞风险：内存越大，fork 阻塞时长越长，大内存实例（32G+）fork 可能阻塞秒级，影响业务<br><br>3. 不适合高频持久化：全量快照开销大，频繁 BGSAVE 会导致 CPU、磁盘 IO、内存资源过载<br><br>4. 写量极大的场景下，COW 机制会导致内存占用翻倍，有 OOM 风险|

---

## 三、AOF（Append Only File）日志持久化

AOF 是 Redis 的增量日志持久化方案，核心是**记录 Redis 执行的每一条写命令**，以 Redis 专属的 RESP 协议格式追加到日志文件末尾，Redis 重启时通过重放 AOF 文件中的所有命令，恢复完整数据。

AOF 默认关闭，需在配置文件中通过`appendonly yes`开启，默认文件名为`appendonly.aof`。

### 3.1 AOF 完整执行流程

AOF 执行分为 4 个核心阶段，全程保证命令执行与日志写入的一致性：

1. **命令执行**：客户端发送写命令，Redis 先校验并执行该命令，执行成功后才会进入日志写入流程（区别于 MySQL 的 WAL 写前日志，Redis 是写后日志，避免写入无效 / 错误命令）。
2. **命令追加**：执行成功的写命令，以 RESP 格式追加到 Redis 的`aof_buf`（AOF 内存缓冲区），避免每次写命令都直接刷盘，提升性能。
3. **缓冲区刷盘**：根据配置的刷盘策略，将`aof_buf`中的内容同步刷入磁盘的 AOF 文件，这是保证数据不丢失的核心环节。
4. **重启恢复**：Redis 重启时，会加载 AOF 文件，按顺序重放所有写命令，恢复内存数据。

### 3.2 核心刷盘策略（appendfsync）

刷盘策略是 AOF 平衡数据安全性与性能的核心，`redis.conf`中通过`appendfsync`参数配置，有 3 个可选值：

|策略|执行逻辑|数据安全性|性能影响|生产环境推荐度|
|---|---|---|---|---|
|`always`|每执行一条写命令，就立即将`aof_buf`内容刷入磁盘，调用`fsync`强制同步|最高，宕机最多丢失 1 条命令|极差，每一条命令都触发磁盘 IO，吞吐量极低，CPU/IO 占用高|仅金融级核心强一致场景使用|
|`everysec`|每秒执行一次刷盘，调用`fsync`强制同步，是 Redis 默认配置|中等，宕机最多丢失 1 秒内的写数据|优秀，性能与安全性完美平衡，IO 压力可控|**生产环境默认推荐**|
|`no`|不主动触发刷盘，完全交由操作系统控制刷盘时机（Linux 默认 30 秒刷一次）|极差，宕机丢失大量不可控数据|最好，但安全性完全不可控|**生产环境绝对禁止使用**|

### 3.3 AOF 重写机制

AOF 是追加式写入，随着运行时间增长，文件体积会持续膨胀，大量冗余命令（如对同一个 key 多次 INCR）会导致文件过大、恢复时间过长。AOF 重写就是为了解决这个问题，核心是**生成一个最小命令集的新 AOF 文件，替换旧的大体积 AOF 文件**。

#### 重写核心原理

AOF 重写**不会读取、分析旧的 AOF 文件**，而是直接读取 Redis 当前内存中的数据，为每个 key 生成对应的最小写入命令，写入新的 AOF 文件。例如对同一个 key 执行 100 次 INCR，旧 AOF 会记录 100 条命令，重写后仅保留 1 条`SET key 最终值`命令，可压缩 90% 以上的文件体积。

#### 重写触发方式

1. **手动触发**：执行`BGREWRITEAOF`命令，fork 子进程异步执行 AOF 重写，仅 fork 瞬间阻塞主线程。
2. **自动触发**：`redis.conf`中配置两个参数，同时满足时自动触发重写：

```ini
    auto-aof-rewrite-min-size 64mb  # AOF文件最小体积，小于该值不会触发重写
    auto-aof-rewrite-percentage 100 # AOF文件当前体积，比上一次重写后的体积增长100%（翻倍）时触发
```

#### 重写完整执行流程

AOF 重写全程通过子进程完成，同时通过重写缓冲区保证重写期间的新命令不丢失：

1. 客户端执行`BGREWRITEAOF`，Redis 检查是否有正在运行的子进程，若有则直接返回。
2. 父进程 fork 子进程，fork 瞬间阻塞主线程，完成后父进程继续处理业务请求。
3. 子进程继承父进程的内存视图，遍历内存数据，生成新的临时 AOF 文件。
4. **重写缓冲区处理**：fork 完成后，父进程收到的新写命令，除了写入`aof_buf`正常刷盘，还会同时写入`aof_rewrite_buf`（AOF 重写缓冲区），保证重写期间的增量命令不丢失。
5. 子进程完成临时 AOF 文件写入后，向父进程发送信号通知完成。
6. 父进程将重写缓冲区中的所有增量命令，追加到临时 AOF 文件末尾。
7. 父进程用临时 AOF 文件原子替换旧的 AOF 文件，本次重写完成。

### 3.4 核心优劣势

|优势|劣势|
|---|---|
|1. 数据安全性极高，`everysec`策略最多丢失 1 秒数据，`always`几乎不丢失数据<br><br>2. 追加式写入，无磁盘寻址开销，写入性能优秀，无文件碎片化问题<br><br>3. 有完善的重写机制，可自动压缩文件体积<br><br>4. 命令为文本格式，可读性强，支持手动修改，可用于误操作恢复（如误删数据可删除对应命令后重启恢复）|1. 相同数据集下，AOF 文件体积远大于 RDB 文件<br><br>2. 数据恢复速度慢，需要重放所有命令，大数据量下恢复时间远超 RDB<br><br>3. 每秒刷盘会带来持续的磁盘 IO 压力，高写量场景下 IO 开销较大<br><br>4. AOF 重写时同样有 fork 阻塞风险，高写量场景下重写缓冲区会占用额外内存|

## 四、RDB-AOF 混合持久化

混合持久化是**Redis 4.0 版本引入，Redis 5.0 + 默认开启**的方案，完美结合了 RDB 和 AOF 的优势，解决了两者的核心痛点，是当前线上环境的首选方案。

### 4.1 核心原理

混合持久化基于 AOF 重写机制实现，开启后（`aof-use-rdb-preamble yes`），AOF 重写时会执行以下逻辑：

1. 子进程将重写这一刻的内存全量数据，以 RDB 二进制格式写入新 AOF 文件的开头
2. 将重写期间的增量写命令，以 AOF 文本格式追加到 AOF 文件的末尾
3. 最终的 AOF 文件，前半部分是 RDB 格式的全量快照，后半部分是 AOF 格式的增量命令

### 4.2 数据恢复流程

Redis 重启加载混合持久化的 AOF 文件时：

1. 先检查 AOF 文件开头是否为 RDB 格式标识，若是则先加载 RDB 全量快照，快速恢复绝大部分数据
2. 再重放文件末尾的 AOF 增量命令，恢复重写期间的增量数据
3. 恢复速度接近纯 RDB，同时数据安全性接近纯 AOF

### 4.3 核心优劣势

|优势|劣势|
|---|---|
|1. 兼顾 RDB 的极速恢复能力和 AOF 的数据安全性，完美平衡<br><br>2. 文件体积远小于纯 AOF，重写和加载性能大幅提升<br><br>3. 兼容 AOF 的所有能力，增量命令可手动编辑，支持误操作恢复|1. 文件可读性差，前半部分是二进制 RDB 格式，后半部分是文本 AOF 格式<br><br>2. 兼容性差，Redis 4.0 之前的版本无法识别混合持久化的 AOF 文件|

---

## 五、Redis 7.0+ AOF 持久化增强

Redis 7.0 对 AOF 持久化做了架构级优化，核心是**Multi-Part AOF（分片式 AOF）**，解决了旧版本重写的性能与可靠性问题：

1. **分片存储**：将 AOF 文件拆分为三类：
    
    - 基础文件：重写生成的文件，可为 RDB 或 AOF 格式，一个实例最多一个
    - 增量文件：重写完成后追加的增量命令文件，多个小文件存储
    - 清单文件：管理所有 AOF 文件的元数据，保证文件一致性
    
2. **重写优化**：重写时无需再重写全量数据，仅需生成新的增量文件，修改清单文件即可，避免了重写时的文件替换风险，大幅降低重写的 IO 开销
3. **恢复加速**：重启时可并行加载多个 AOF 文件，恢复速度大幅提升
4. **内存优化**：重写缓冲区的内存占用大幅降低，高写量场景下内存更稳定

## 六、核心选型对比与加载优先级

### 6.1 三种持久化方案选型对比

|对比维度|RDB 快照|纯 AOF 日志|RDB-AOF 混合持久化|
|---|---|---|---|
|数据安全性|低，丢失快照间隔数据|高，最多丢失 1 秒数据|高，最多丢失 1 秒数据|
|恢复速度|极快，二进制直接加载|慢，需重放所有命令|快，先加载 RDB 再重放增量|
|文件体积|极小，二进制压缩|大，文本格式存储|中等，远小于纯 AOF|
|性能影响|小，仅 fork 瞬间阻塞|中等，持续 IO 刷盘|中等，兼顾两者开销|
|运维复杂度|低，单文件易备份|中，需关注重写与文件体积|低，自动化程度高|
|适用场景|冷备、灾备、全量数据同步|核心业务、强数据安全要求场景|**Redis 4.0 + 线上通用首选**|

### 6.2 持久化文件加载优先级

Redis 重启时，会严格按照以下优先级加载持久化文件恢复数据：

1. 首先检查 AOF 是否开启（`appendonly yes`），且 AOF 文件存在，**优先加载 AOF 文件**，即使 RDB 文件存在也不会加载
2. 若 AOF 关闭，或 AOF 文件不存在，才会加载 RDB 文件
3. 若 AOF 开启但 AOF 文件加载失败，Redis 会直接启动失败，保证数据一致性

## 七、常见坑与线上最佳实践

### 7.1 高频踩坑点

1. **fork 阻塞主线程，导致业务超时**
    
    问题本质：fork 子进程时需要复制父进程的内存页表，Redis 内存越大，页表体积越大，fork 阻塞时间越长（32G 内存实例 fork 阻塞可达数百毫秒，64G 可达秒级）。
    
    避坑方案：
    
    - 严格控制 Redis 单实例内存，建议 8-24G，最大不超过 32G
    - 调整 Linux 内核参数`vm.overcommit_memory = 1`，避免 fork 时因内存不足失败
    - 避免在业务高峰期执行 BGSAVE/BGREWRITEAOF，调整自动触发规则到低峰期
    - 关闭透明大页（THP），减少 fork 时的页表复制开销
    
2. **刷盘策略配置错误，导致数据丢失或性能雪崩**
    
    问题本质：生产环境使用`appendfsync no`导致宕机丢失大量数据；或核心场景使用`appendfsync always`导致 IO 打满，Redis 性能雪崩。
    
    避坑方案：
    
    - 99% 的业务场景使用默认的`appendfsync everysec`，平衡安全与性能
    - 金融级强一致场景使用`appendfsync always`，需搭配高性能 SSD 磁盘
    - 绝对禁止生产环境使用`appendfsync no`
    
3. **内存预留不足，导致 fork 时 OOM**
    
    问题本质：Redis 内存使用率过高，fork 后 COW 机制复制大量修改的内存页，导致内存占用翻倍，触发系统 OOM，Redis 进程被杀死。
    
    避坑方案：
    
    - 设置`maxmemory`，预留至少 30% 的系统空闲内存，用于 fork 和 COW 机制
    - 开启 swap 分区，防止极端场景下 OOM
    - 高写量场景下，避免同时执行 BGSAVE 和 BGREWRITEAOF
    
4. **持久化文件与实例同盘，导致数据完全丢失**
    
    问题本质：持久化文件与 Redis 实例、系统日志存在同一块磁盘，磁盘故障 / 满盘时，持久化文件损坏或丢失，Redis 无法恢复。
    
    避坑方案：
    
    - 持久化文件写入独立的高性能数据盘，与系统盘、日志盘分离
    - 定期（每日）将 RDB 文件备份到异地存储，避免单机房故障
    - 监控磁盘使用率，设置 80% 阈值告警，避免满盘导致持久化失败
    
5. **频繁触发持久化，导致资源持续过载**
    
    问题本质：`save`规则配置过于频繁，或`auto-aof-rewrite-percentage`设置过小，导致频繁 fork 子进程，主线程频繁阻塞，CPU/IO 持续过载。
    
    避坑方案：
    
    - 合理设置 RDB 自动规则，如`save 3600 1`、`save 300 100`，避免高频触发
    - 调整`auto-aof-rewrite-percentage`为 100-200，避免频繁重写
    - RDB 仅用于灾备，不要作为高频持久化方案
    

### 7.2 线上通用最佳实践

1. **Redis 4.0 + 版本，默认开启混合持久化**，兼顾性能、安全与恢复速度，是通用最优解
2. **主从架构分工**：主节点负责业务读写，关闭 RDB，仅开启 AOF everysec 策略；从节点负责持久化与灾备，开启 RDB 定期快照，避免主节点 fork 阻塞影响业务
3. **监控告警全覆盖**：重点监控 fork 阻塞时长、持久化文件大小、磁盘 IO 使用率、AOF 重写频率、RDB 快照成功率，设置阈值告警
4. **定期备份与恢复演练**：每日备份 RDB 文件到异地存储，定期执行恢复演练，验证持久化文件的可用性
5. **关闭自动 RDB 触发，手动定时执行**：线上业务关闭`save`自动规则，在业务低峰期通过定时任务手动执行 BGSAVE，避免高峰期意外触发阻塞主线程
6. **误操作快速恢复预案**：提前制定误操作恢复流程，如误执行 FLUSHALL 后，立即隔离 Redis，修改 AOF 文件删除有害命令，重启恢复数据

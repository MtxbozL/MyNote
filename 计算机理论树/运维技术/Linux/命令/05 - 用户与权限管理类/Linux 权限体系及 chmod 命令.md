## 一、Linux 权限核心基础（chmod 前置必备知识）

Linux 是多用户多任务操作系统，权限体系是系统安全的核心基石，所有文件 / 目录都有严格的访问权限控制，chmod 命令的核心作用就是修改这些权限。

![[Pasted image 20260310221101.png]]

### 1.1 权限核心模型：UGO 归属体系

Linux 权限基于**UGO 三维归属模型**设计，所有文件 / 目录都绑定三类访问身份：

| 标识  | 全称    | 含义                                |
| --- | ----- | --------------------------------- |
| u   | User  | 文件 / 目录的所有者，默认是创建者，仅 root 可变更所有者  |
| g   | Group | 文件 / 目录的所属用户组，同组内用户可按组权限访问        |
| o   | Other | 其他用户，既不是所有者、也不在所属组内的所有用户          |
| a   | All   | 所有用户（u+g+o，chmod 默认不指定身份时，默认代表 a） |

### 1.2 3 种基础权限：r/w/x 及对文件 / 目录的核心区别

基础权限分为读、写、执行三类，**对普通文件和目录的作用完全不同**，这是新手最易踩坑的核心点。

表格

|权限标识|全称|八进制值|对普通文件的作用|对目录的作用|
|---|---|---|---|---|
|r|Read（读）|4|可读取文件内容（cat、vim、more 等操作）|可查看目录内的文件列表（ls 命令）|
|w|Write（写）|2|可修改、新增、删除文件内容|可在目录内新建 / 删除 / 移动 / 重命名文件（**与文件自身权限无关**）|
|x|Execute（执行）|1|可执行文件（二进制程序、shell/python 脚本等）|可进入该目录（cd 命令），是操作目录内文件的基础权限（无 x 权限时，r/w 权限全部失效）|
|-|无权限|0|无对应操作权限|无对应操作权限|

### 1.3 权限的 2 种标准表示方式

1. **符号表示法**：用`rwx`9 个字符分 3 组表示，对应 u、g、o 的权限，无权限用`-`填充。
    
    示例：`rwxr-xr--` → 所有者 rwx，所属组 r-x，其他用户 r--
2. **八进制数字表示法**：用 3 位数字分别对应 u、g、o 的权限，每一位是该身份权限值的和（r=4、w=2、x=1）。
    
    示例：`755` → 7=4+2+1 (rwx)、5=4+1 (r-x)、5=4+1 (r-x)，对应符号`rwxr-xr-x`

### 1.4 权限查看：ls -l 命令输出全解析

使用`ls -l 文件名/目录`可查看权限详情，以下是标准输出拆解：

plaintext

```
-rwxr-xr--  1 alice dev  4096 Mar 10 10:00 test.sh
```

表格

|字段|含义说明|
|---|---|
|第 1 位 `-`|文件类型：`-`= 普通文件、`d`= 目录、`l`= 软链接、`c`= 字符设备、`b`= 块设备|
|第 2-4 位 `rwx`|所有者（u）的权限|
|第 5-7 位 `r-x`|所属组（g）的权限|
|第 8-10 位 `r--`|其他用户（o）的权限|
|数字 1|文件硬链接数|
|alice|文件所有者|
|dev|文件所属用户组|
|4096|文件大小（字节）|
|Mar 10 10:00|最后修改时间|
|test.sh|文件名|

### 1.5 补充：umask 默认权限控制

umask 用于控制**新建文件 / 目录的默认权限**，系统会根据 umask 值自动分配初始权限，无需手动 chmod。

- 核心规则：
    
    - 目录默认最大权限：777，默认权限 = 777 - umask 值
    - 文件默认最大权限：666（系统默认不给文件执行权限，保障安全），默认权限 = 666 - umask 值
    
- 常用默认值：root 用户 umask=0022，普通用户 umask=0002
    
    - umask=022 时，新建目录默认 755，新建文件默认 644（生产环境最常用的默认权限）
    
- 常用命令：
    
    bash
    
    运行
    
    ```
    umask        # 查看当前umask值
    umask 027    # 临时修改umask值（重启终端失效）
    ```
    

## 二、chmod 命令核心详解

### 2.1 命令基础语法与作用

- 核心作用：修改文件 / 目录的访问权限
- 标准语法：
    
    bash
    
    运行
    
    ```
    chmod [选项] 权限模式 文件/目录路径
    ```
    
- 权限修改前提：仅**root 超级用户**和**文件所有者**有权限修改文件 / 目录的权限，普通用户无法修改他人文件的权限。

### 2.2 用法一：符号模式（字母模式，灵活增量修改）

符号模式支持增量修改权限，无需覆盖原有全部权限，适合单权限调整场景。

#### 语法格式

bash

运行

```
chmod [ugoa][+-=][rwxXst] 文件/目录
```

#### 字段拆解

表格

|字段分类|可选值|含义说明|
|---|---|---|
|身份位|u/g/o/a|指定要修改权限的用户身份，不指定默认是 a（所有用户）|
|操作符|+|给指定身份**添加**对应权限，不影响原有其他权限|
||-|给指定身份**移除**对应权限，不影响原有其他权限|
||=|给指定身份**覆盖设置**权限，清空原有权限，仅保留设置的权限|
|权限位|r/w/x|基础读、写、执行权限|
||X|特殊执行权限：仅给目录、以及已有执行权限的文件添加 x 权限，避免给普通文本文件误加执行权限|
||s|SUID/SGID 特殊权限（进阶内容）|
||t|粘滞位特殊权限（进阶内容）|

#### 常用实例

bash

运行

```
# 1. 给脚本所有者添加执行权限（最常用）
chmod u+x test.sh

# 2. 给所属组和其他用户移除写权限
chmod go-w test.txt

# 3. 给所有用户设置只读权限（覆盖原有所有权限）
chmod a=r-- test.txt

# 4. 给目录递归添加所属组的写+执行权限
chmod -R g+wx data/

# 5. 给所有用户添加读权限，同时给所有者添加执行权限
chmod a+r,u+x deploy.sh

# 6. 安全给目录下所有内容加执行权限（仅目录和可执行文件生效）
chmod -R a+X project/
```

### 2.3 用法二：八进制数字模式（全量覆盖，生产环境最常用）

数字模式直接用 3 位 / 4 位八进制数全量设置权限，简洁高效，是运维、脚本开发的首选方式。

#### 核心规则

- 3 位数字：分别对应**所有者、所属组、其他用户**的权限值，每一位是 r=4、w=2、x=1 的和
- 4 位数字：第一位为特殊权限值（SUID=4、SGID=2、粘滞位 = 1），后三位为基础 UGO 权限

#### 常用实例

bash

运行

```
# 1. 普通文件标准权限：所有者读写，组和其他只读（生产环境通用）
chmod 644 test.txt

# 2. 目录/可执行脚本标准权限：所有者全权限，组和其他读+执行
chmod 755 data/ test.sh

# 3. 私密文件权限：仅所有者可读写，其他人无任何权限（ssh私钥强制要求）
chmod 600 ~/.ssh/id_rsa

# 4. 私密目录权限：仅所有者可进入、读写
chmod 700 ~/private/

# 5. 递归修改目录下所有内容权限（谨慎使用）
chmod -R 755 /data/web/
```

### 2.4 常用核心选项

表格

|选项|全称|作用说明|
|---|---|---|
|-R|--recursive|递归修改目录及其下所有子目录、文件的权限（**高危选项，严禁对根目录 / 使用**）|
|-v|--verbose|显示所有文件的权限修改详情，便于调试|
|-c|--changes|仅显示权限发生实际变更的文件信息，精简输出|
|-f|--silent|屏蔽权限修改中的错误信息，静默执行|

## 三、Linux 特殊权限（SUID/SGID/ 粘滞位）

除了基础 rwx 权限，Linux 还有 3 种特殊权限，用于解决特定场景的权限问题，对应 chmod 的 4 位八进制数第一位。

### 3.1 SetUID（SUID）

- 核心作用：**仅对二进制可执行文件生效**，普通用户执行该文件时，临时以**文件所有者**的身份运行（而非执行者本人身份）。
- 权限标识：所有者权限位的 x 变为`s`（有 x 权限时为`s`，无 x 权限时为`S`，表示无效）
- 设置方式：
    
    bash
    
    运行
    
    ```
    # 符号模式
    chmod u+s /usr/bin/passwd
    # 数字模式（4为SUID标识，放在第一位）
    chmod 4755 /usr/bin/passwd
    ```
    
- 典型场景：系统`/usr/bin/passwd`命令，权限为 4755。普通用户执行时可临时获得 root 权限，修改只有 root 能写的`/etc/shadow`密码文件。
- 安全提醒：SUID 有巨大的提权风险，禁止随意给自定义文件、脚本设置 SUID，对目录设置 SUID 无效。

### 3.2 SetGID（SGID）

- 核心作用：
    
    1. 对文件：和 SUID 类似，执行文件时临时以**文件所属组**的身份运行；
    2. 对目录（最常用）：目录下新建的文件 / 目录，会自动继承该目录的所属组，而非创建者的默认组，完美解决团队共享目录的权限协同问题。
    
- 权限标识：所属组权限位的 x 变为`s`（有 x 权限时为`s`，无 x 权限时为`S`，表示无效）
- 设置方式：
    
    bash
    
    运行
    
    ```
    # 符号模式
    chmod g+s /data/team
    # 数字模式（2为SGID标识，放在第一位）
    chmod 2775 /data/team
    ```
    
- 典型场景：团队共享目录，设置 SGID 后，所有成员在目录内新建的文件都继承目录的所属组，避免出现同组成员无法编辑他人文件的问题。

### 3.3 粘滞位（Sticky Bit）

- 核心作用：**仅对目录生效**，目录内的文件 / 目录，只有**文件所有者、root 用户**能删除、移动、重命名，其他用户即使有 w 权限也无法操作他人文件。
- 权限标识：其他用户权限位的 x 变为`t`（有 x 权限时为`t`，无 x 权限时为`T`，表示无效）
- 设置方式：
    
    bash
    
    运行
    
    ```
    # 符号模式
    chmod o+t /tmp
    # 数字模式（1为粘滞位标识，放在第一位）
    chmod 1777 /tmp
    ```
    
- 典型场景：系统临时目录`/tmp`，权限为 1777。所有人都可在目录内新建文件，但只能删除自己的文件，避免公共目录被恶意篡改。

## 四、高频避坑指南 & 最佳实践

1. **严禁滥用 777 权限**
    
    777 代表所有用户拥有读写执行权限，生产环境绝对禁止使用，尤其是 web 根目录、系统配置目录，会导致任意用户篡改、执行文件，引发严重的安全漏洞。替代方案：目录用 755，文件用 644，按需给特定用户 / 组分配权限。
    
2. **递归 - R 操作的高危红线**
    
    绝对禁止执行`chmod -R 777 /` 或 `chmod -R 权限 /`，会直接破坏系统文件的默认权限，导致系统崩溃、无法远程登录。递归修改时，建议用 find 命令分别设置目录和文件权限，避免给普通文件误加执行权限。
    
3. **软链接权限规则**
    
    chmod 无法修改软链接文件的权限，软链接的权限始终显示为 777，实际生效的是源文件的权限，修改必须针对源文件操作。
    
4. **目录权限的核心前提**
    
    目录必须有 x 权限才能进入，没有 x 权限时，即使有 r 权限也无法查看目录内文件列表，有 w 权限也无法新建 / 删除文件。
    
5. **ssh 密钥权限强制要求**
    
    ssh 私钥文件必须设置 600 权限，公钥文件设置 644 权限，权限过宽会导致 ssh 服务拒绝连接，防止私钥被其他用户读取。
    

## 五、实战常用场景示例

### 1. 批量规范权限（生产环境高频操作）

批量将当前目录下所有目录设为 755，所有普通文件设为 644，避免一刀切递归修改：

bash

运行

```
# 批量修改目录权限
find . -type d -exec chmod 755 {} \;
# 批量修改文件权限
find . -type f -exec chmod 644 {} \;
```

### 2. 配置团队共享目录

创建 dev 组共享目录，所有组成员可读写，新建文件自动继承目录所属组：

bash

运行

```
# 创建共享目录
mkdir /data/dev-team
# 设置目录所属组为dev
chown root:dev /data/dev-team
# 设置SGID+775权限，保障协同
chmod 2775 /data/dev-team
```

### 3. 配置公共上传目录

创建公共上传目录，所有人可上传文件，但只能删除自己的文件：

bash

运行

```
mkdir /data/public-upload
chmod 1777 /data/public-upload
```

### 4. 修复系统命令 SUID 权限

解决普通用户无法修改密码的问题，修复 passwd 命令的 SUID 权限：

bash

运行

```
chmod 4755 /usr/bin/passwd
```

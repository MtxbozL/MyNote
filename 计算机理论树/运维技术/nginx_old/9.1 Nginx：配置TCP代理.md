## 一、TCP 代理核心概述

### 1.1 基础定义

Nginx TCP 代理是**四层传输层代理能力**，基于 TCP 协议实现数据包转发，不解析 HTTP 等应用层内容，仅处理 TCP 连接的建立、转发与释放，可实现任意基于 TCP 协议的服务的端口转发、负载均衡、流量管控与高可用保障。

与七层 HTTP 代理的核心区别：

| 对比维度 | TCP 四层代理                            | HTTP 七层代理                     |
| ---- | ----------------------------------- | ----------------------------- |
| 工作层级 | OSI 第四层（传输层）                        | OSI 第七层（应用层）                  |
| 核心能力 | 转发 TCP 数据包，不解析应用层内容                 | 解析 HTTP 协议，可修改请求头、响应体、路径重写等   |
| 适用场景 | MySQL、Redis、SSH、MQ、TCP 接口等非 HTTP 服务 | Web 站点、HTTP 接口、前后端服务等 HTTP 场景 |
| 性能   | 极高，资源占用低，转发延迟极小                     | 中等，需解析应用层内容，开销更高              |
| 配置块  | stream 块（与 http 块平级）                | http 块内的 server/location 块    |

### 1.2 核心依赖模块

Nginx TCP 代理能力依赖 **`ngx_stream_core_module`核心模块 **，该模块**默认不会被编译进 Nginx**，编译安装时必须手动添加编译参数开启：

```bash
# 编译Nginx时添加stream模块
./configure --with-stream --with-stream_ssl_module --with-stream_realip_module
make && make install
```

- 验证是否已开启：执行`nginx -V`，查看输出中是否包含`--with-stream`参数
- 配套可选模块：
    
    - `--with-stream_ssl_module`：支持 TCP 代理的 SSL/TLS 加密
    - `--with-stream_realip_module`：支持源 IP 透传
    - `--with-stream_health_check_module`：商业版主动健康检查，开源版需第三方模块

## 二、TCP 代理核心配置结构

TCP 代理的所有配置必须写在 **`stream {}`块 ** 中，该块与`http {}`块平级，禁止写在 http 块 /server 块内，否则配置不生效。

基础配置框架：

```nginx
# 全局主配置文件nginx.conf
# 与http块平级的stream块，TCP代理所有配置均写在这里
stream {
    # 全局日志配置
    log_format tcp_proxy '$remote_addr [$time_local] $protocol $status $bytes_sent $bytes_received $session_time "$upstream_addr"';
    access_log /var/log/nginx/tcp_access.log tcp_proxy;
    error_log /var/log/nginx/tcp_error.log warn;

    # 1. 上游服务池：定义后端TCP服务节点，实现负载均衡、长连接管理
    upstream backend_tcp {
        # 上游服务地址，支持权重、健康检查等参数
        server 192.168.1.101:3306 weight=1;
        server 192.168.1.102:3306 weight=1;

        # 核心长连接配置（大纲核心内容）
        keepalive 32;
        keepalive_timeout 120s;
    }

    # 2. 代理服务块：定义监听端口、转发规则
    server {
        # 监听本地端口，TCP协议
        listen 3306;
        # 核心：转发到上游服务池/指定地址
        proxy_pass backend_tcp;

        # 超时配置
        proxy_connect_timeout 5s;
        proxy_timeout 300s;
    }
}

# 原有的HTTP配置块，与stream块平级，互不影响
http {
    # 原有HTTP站点配置...
}
```

## 三、Upstream 模块（TCP 代理核心）

Upstream 模块用于定义后端 TCP 服务的节点池，是实现 TCP 负载均衡、长连接复用、健康检查的核心，与 HTTP 的 Upstream 模块逻辑一致，但部分参数仅适用于 TCP 场景。

### 3.1 基础配置与负载均衡策略

#### 基础语法

```nginx
upstream 自定义池名称 {
    # 上游节点配置，语法：server 地址:端口 [参数];
    server 192.168.1.101:6379 weight=2 max_fails=3 fail_timeout=30s;
    server 192.168.1.102:6379 weight=1 max_fails=3 fail_timeout=30s;
    server 192.168.1.103:6379 backup; # 备用节点
}
```

#### 核心节点参数

|参数|作用说明|
|---|---|
|`weight=数字`|节点权重，权重越高，被分配的请求越多，默认 1|
|`max_fails=数字`|失败次数阈值，指定时间内失败次数达到该值，节点被临时剔除|
|`fail_timeout=时间`|失败判定的时间窗口 + 节点熔断时长，默认 10s|
|`backup`|标记为备用节点，主节点全部故障时才会启用|
|`down`|标记节点永久下线，不参与负载均衡|

#### 常用负载均衡策略

1. **轮询（默认）**：按顺序依次将请求分配给上游节点，配合 weight 实现加权轮询，适合节点配置一致的场景
2. **ip_hash**：基于客户端源 IP 的哈希结果分配节点，同一个客户端 IP 始终转发到同一个上游节点，实现 TCP 会话保持，适合 Redis、MySQL 等有状态服务

  ```nginx
    upstream redis_cluster {
        ip_hash;
        server 192.168.1.101:6379;
        server 192.168.1.102:6379;
    }
    ```

3. **least_conn**：将请求分配给当前活跃连接数最少的上游节点，适合节点处理能力不均的场景
4. **least_time**：Nginx Plus 商业版支持，将请求分配给响应时间最短的节点

### 3.2 核心长连接参数（大纲核心内容）

TCP 连接的建立（三次握手）和释放（四次挥手）会产生大量开销，高并发场景下，通过长连接复用可大幅降低延迟、减少系统资源占用，核心参数为`keepalive`和`keepalive_timeout`。

#### 1. keepalive

- **语法**：`keepalive 连接数;`
- **核心定义**：设置 Nginx 每个 worker 进程，与上游 TCP 服务之间**可保留的空闲长连接的最大数量**
- **核心作用**：开启与上游服务的长连接复用能力，TCP 连接处理完请求后不会立即关闭，而是放入空闲连接池，后续新请求可直接复用该连接，避免频繁的 TCP 握手 / 挥手开销，大幅提升高并发下的转发性能
- **关键细节**：
    
    - 该数值不是 Nginx 与上游的最大总连接数，是**空闲长连接的上限**，超过该数值的空闲连接会被主动关闭；
    - 数值需根据业务并发量设置，常规业务建议设置为 16~64，高并发核心业务可设置为 128~256，不可设置过大，避免空闲连接占用过多系统文件句柄；
    - 必须配置在 upstream 块内，仅对当前上游池生效；
    - 开启后，Nginx 会自动禁用 TCP 连接的 "短连接" 行为，保持连接复用。
    
- **示例**：`keepalive 32;` 每个 worker 进程最多保留 32 个与上游的空闲长连接

#### 2. keepalive_timeout

- **语法**：`keepalive_timeout 时间;`
- **默认值**：60s
- **核心定义**：设置空闲长连接的最大存活时间，超过该时间未被复用的空闲长连接，会被 Nginx 主动关闭
- **核心作用**：控制长连接的生命周期，平衡长连接复用率和系统资源占用，避免无效的空闲连接长期占用文件句柄、内存资源
- **关键细节**：
    
    - 超时时间从连接进入空闲状态开始计算，每次复用连接会重置计时；
    - 数值需大于上游服务的空闲超时时间，避免上游先关闭连接，导致 Nginx 复用已失效的连接；
    - 常规业务建议设置为 60s~300s，长连接复用需求高的场景可设置为 10min。
    
- **示例**：`keepalive_timeout 120s;` 空闲超过 120 秒的长连接会被关闭

#### 配套长连接优化参数

- `keepalive_requests`：Nginx Plus 商业版 / 高版本开源版支持，设置单个长连接最多可处理的请求数，超过该数值后，连接会在空闲时关闭，默认 100，高并发场景可调大至 1000。

## 四、TCP 代理核心配置指令

### 4.1 监听指令 listen

定义 Nginx 监听的本地端口，是 TCP 代理的入口，核心语法：

```nginx
listen 端口 [参数];
```

常用参数：

- `tcp`：默认值，监听 TCP 协议，无需显式声明
- `udp`：监听 UDP 协议，实现 UDP 代理
- `ssl`：开启该端口的 SSL/TLS 加密
- `reuseport`：开启端口复用，每个 worker 进程独立监听端口，大幅提升高并发下的性能
- `backlog=数字`：TCP 半连接队列大小，高并发场景建议设置为 65535

示例：

```nginx
# 高并发场景监听配置
listen 6379 reuseport backlog=65535;
```

### 4.2 核心转发指令 proxy_pass

定义请求转发的目标地址，是 TCP 代理的核心指令，语法：

```nginx
proxy_pass 上游地址;
```

两种使用方式：

1. 直接转发到固定地址：`proxy_pass 192.168.1.101:3306;`
2. 转发到 upstream 定义的服务池：`proxy_pass backend_tcp;`

⚠️ 注意：TCP 代理的 proxy_pass**不能加路径**，仅支持`地址:端口`或 upstream 池名称，与 HTTP 代理的 proxy_pass 有本质区别。

### 4.3 超时控制指令

超时控制是 TCP 代理稳定性的核心，避免无效连接长期占用资源：

|指令|作用说明|默认值|推荐配置|
|---|---|---|---|
|`proxy_connect_timeout`|Nginx 与上游服务建立 TCP 连接的超时时间|60s|3s~10s，避免长时间阻塞|
|`proxy_timeout`|客户端与上游之间无数据传输的超时时间，超过则关闭连接|10m|60s~300s，根据业务场景调整|
|`proxy_upload_rate`|限制客户端到 Nginx 的上传速率，0 为不限制|0|按需配置，限制大流量传输|
|`proxy_download_rate`|限制 Nginx 到客户端的下载速率，0 为不限制|0|按需配置|

### 4.4 源 IP 透传配置

默认情况下，上游服务只能获取到 Nginx 的 IP，无法获取客户端真实源 IP，需通过 Proxy Protocol 协议实现透传：

1. Nginx 侧配置：在 server 块添加`proxy_protocol on;`
2. 上游服务侧：必须开启 Proxy Protocol 协议支持（MySQL、Redis、Nginx 均支持）

完整示例：

```nginx
stream {
    upstream mysql_cluster {
        server 192.168.1.101:3306;
        keepalive 32;
        keepalive_timeout 120s;
    }

    server {
        listen 3306;
        proxy_pass mysql_cluster;
        # 开启Proxy Protocol，透传客户端真实IP
        proxy_protocol on;
    }
}
```

## 五、可落地的完整配置示例

### 示例 1：基础 TCP 端口转发（单上游，MySQL 代理）

将 Nginx 本地 3306 端口的 TCP 请求，转发到远程 MySQL 服务器，适合数据库端口映射、内网穿透场景：

```nginx
stream {
    log_format tcp_proxy '$remote_addr [$time_local] $protocol $status $bytes_sent $bytes_received $session_time "$upstream_addr"';
    access_log /var/log/nginx/mysql_proxy_access.log tcp_proxy;
    error_log /var/log/nginx/mysql_proxy_error.log warn;

    server {
        # 监听本地3306端口
        listen 3306 reuseport backlog=65535;
        # 转发到远程MySQL服务器
        proxy_pass 192.168.1.101:3306;

        # 超时配置
        proxy_connect_timeout 5s;
        proxy_timeout 300s;

        # 源IP透传（需MySQL开启Proxy Protocol支持）
        proxy_protocol on;
    }
}
```

### 示例 2：TCP 负载均衡 + 长连接优化（Redis 集群代理）

实现 Redis 集群的负载均衡、会话保持、长连接复用，适合缓存集群高可用场景：

```nginx
stream {
    log_format tcp_proxy '$remote_addr [$time_local] $protocol $status $bytes_sent $bytes_received $session_time "$upstream_addr"';
    access_log /var/log/nginx/redis_proxy_access.log tcp_proxy;
    error_log /var/log/nginx/redis_proxy_error.log warn;

    # Redis上游池，加权轮询+被动健康检查+长连接优化
    upstream redis_cluster {
        # 加权轮询，101节点权重更高
        server 192.168.1.101:6379 weight=2 max_fails=3 fail_timeout=30s;
        server 192.168.1.102:6379 weight=1 max_fails=3 fail_timeout=30s;
        server 192.168.1.103:6379 weight=1 max_fails=3 fail_timeout=30s backup;

        # 核心长连接配置
        keepalive 64;
        keepalive_timeout 180s;
        keepalive_requests 1000;

        # 会话保持：同一个客户端IP固定访问同一个节点
        ip_hash;
    }

    server {
        listen 6379 reuseport backlog=65535;
        proxy_pass redis_cluster;

        # 超时配置
        proxy_connect_timeout 3s;
        proxy_timeout 300s;
    }
}
```

### 示例 3：SSH 端口转发

将 Nginx 本地 2222 端口的请求，转发到内网 SSH 服务器，实现跳板机功能：

```nginx
stream {
    server {
        listen 2222;
        proxy_pass 192.168.1.200:22;
        proxy_connect_timeout 10s;
        proxy_timeout 1h; # SSH长连接超时设置为1小时
    }
}
```

## 六、TCP 代理性能优化方案

### 6.1 长连接核心优化

1. 必须开启`keepalive`长连接复用，根据并发量设置合理的空闲连接数，常规业务 16~64，高并发业务 128~256；
2. `keepalive_timeout`设置需大于上游服务的空闲超时时间，避免上游先关闭连接，导致连接复用失败；
3. 高并发场景调大`keepalive_requests`，提升单个长连接的复用次数，减少连接新建开销。

### 6.2 Nginx 配置优化

1. 开启`reuseport`端口复用，每个 worker 进程独立监听端口，避免锁竞争，高并发场景性能可提升 30% 以上；
2. 调大`backlog`半连接队列，避免高并发下 TCP 连接丢失；
3. 合理设置超时时间，`proxy_connect_timeout`不宜过长，避免无效连接阻塞；`proxy_timeout`根据业务场景设置，避免空闲连接长期占用资源；
4. worker 进程数设置为 CPU 核心数，绑定 CPU 核心，减少上下文切换：

 ```nginx
    worker_processes auto;
    worker_cpu_affinity auto;
    ```
    
5. 调大最大文件句柄数，TCP 代理每个连接占用一个文件句柄，高并发场景需足够的句柄支持：

  ```nginx
    worker_rlimit_nofile 65535;
    ```

### 6.3 Linux 内核参数优化

在`/etc/sysctl.conf`中配置，优化 TCP 协议栈，提升转发性能：

```ini
# TCP端口范围，扩大可用端口数
net.ipv4.ip_local_port_range = 1024 65535
# TCP连接快速回收
net.ipv4.tcp_tw_reuse = 1
net.ipv4.tcp_tw_recycle = 0 # 公网场景关闭，避免NAT环境异常
# TCP半连接队列与全连接队列
net.core.somaxconn = 65535
net.ipv4.tcp_max_syn_backlog = 65535
# 系统最大文件句柄数
fs.file-max = 1048576
# TCP缓冲区优化
net.ipv4.tcp_rmem = 4096 87380 16777216
net.ipv4.tcp_wmem = 4096 65536 16777216
net.core.rmem_max = 16777216
net.core.wmem_max = 16777216
```

配置后执行`sysctl -p`生效。

## 七、常见踩坑与最佳实践

### 7.1 高频踩坑与解决方案

|常见坑|根因|解决方案|
|---|---|---|
|配置后不生效，启动报错|1. 未编译 stream 模块；2. stream 块写在了 http 块内|1. 重新编译 Nginx 添加 --with-stream 参数；2. stream 块必须与 http 块平级|
|长连接不生效，频繁新建 TCP 连接|1. keepalive 未配置在 upstream 块内；2. 上游服务空闲超时时间小于 Nginx 的 keepalive_timeout|1. 把 keepalive 配置到 upstream 块；2. 调大 Nginx 的 keepalive_timeout，大于上游的空闲超时|
|上游服务无法获取客户端真实 IP|TCP 代理是四层转发，默认源 IP 会被替换为 Nginx 的 IP|开启 Proxy Protocol 协议，Nginx 侧配置 proxy_protocol on，上游服务同步开启该协议支持|
|高并发下出现端口耗尽|Nginx 作为客户端与上游建立连接，端口号耗尽|扩大 ip_local_port_range 端口范围，开启 tcp_tw_reuse，开启长连接复用减少连接新建|
|配置重载后，原有 TCP 连接断开|错误使用 nginx -s restart 重启服务，而非 reload 重载|配置修改后使用 nginx -t 校验语法，再用 nginx -s reload 平滑重载，不影响现有 TCP 连接|

### 7.2 生产环境最佳实践

1. **模块开启规范**：编译 Nginx 时必须添加`--with-stream`，配套开启`--with-stream_ssl_module`、`--with-stream_realip_module`，满足加密、源 IP 透传需求；
2. **长连接必配**：所有 TCP 代理场景，只要上游服务支持长连接，必须配置 keepalive 长连接复用，大幅降低系统开销；
3. **健康检查必配**：所有多节点负载均衡场景，必须配置`max_fails`和`fail_timeout`被动健康检查，核心业务可搭配第三方模块实现主动健康检查；
4. **日志规范**：必须配置 TCP 代理专属的日志格式，记录客户端 IP、上游地址、会话时长、传输字节数，方便问题排查与监控；
5. **安全规范**：监听端口必须配置防火墙 / 安全组规则，仅允许可信 IP 访问，避免代理端口暴露到公网被恶意攻击；
6. **高可用规范**：核心 TCP 代理服务，需搭建 Nginx 主备集群（Keepalived），避免 Nginx 单点故障；
7. **测试规范**：配置修改后，必须先执行`nginx -t`校验语法，确认无误后再平滑重载，避免服务中断。

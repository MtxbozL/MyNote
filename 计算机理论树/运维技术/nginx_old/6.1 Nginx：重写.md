## 一、重写核心概述

Nginx 重写是其核心流量管理能力，通过规则修改请求 URI、跳转地址或直接返回响应，实现**URL 标准化、SEO 优化、业务路径兼容、访问控制、站点跳转**等核心需求，是反向代理、Web 站点运维的必备技能。

核心实现分为两大指令，覆盖所有重写场景：

- `return`：轻量、高性能的简单跳转 / 响应指令，Nginx 官方优先推荐，覆盖 80% 日常场景
- `rewrite`：基于正则的复杂重写指令，灵活度高，支持内部路径改写、伪静态等复杂场景
- 可配置上下文：`server`块、`location`块、`if`条件判断块

---

## 二、核心指令一：return（优先推荐）

### 2.1 指令定位

`return`是 Nginx 官方推荐的轻量重写指令，直接向客户端返回状态码、跳转地址或固定响应内容，**匹配命中后立即终止当前块的所有后续指令**，无正则解析开销，性能远超`rewrite`，适合所有简单跳转、状态返回场景。

### 2.2 完整语法格式

```nginx
# 格式1：仅返回HTTP状态码
return 状态码;

# 格式2：返回状态码+响应文本
return 状态码 响应文本;

# 格式3：返回状态码+跳转URL（最常用）
return 状态码 跳转URL;

# 格式4：仅返回跳转URL，默认302临时跳转
return 跳转URL;
```

### 2.3 核心跳转状态码详解

跳转场景必须明确状态码，避免浏览器缓存、请求方法丢失、SEO 权重异常等问题：

|状态码|类型|核心特性|适用场景|
|---|---|---|---|
|301|永久重定向|浏览器长期缓存，SEO 权重完全传递，POST 请求会转为 GET|HTTP→HTTPS、旧域名→新域名、路径永久变更|
|302|临时重定向|浏览器不缓存，POST 请求会转为 GET|临时活动跳转、站点维护、临时路径切换|
|307|临时重定向|浏览器不缓存，严格保留原请求方法和 Body（POST 不会转 GET）|接口临时跳转、表单提交跳转|
|308|永久重定向|浏览器长期缓存，严格保留原请求方法和 Body|接口域名永久变更、API 路径永久调整|

### 2.4 常用场景与可落地配置示例

#### 示例 1：HTTP 强制跳转 HTTPS（最常用）

```nginx
server {
    listen 80;
    server_name example.com www.example.com;
    # 301永久跳转，保留原请求URI和参数，SEO友好
    return 301 https://$host$request_uri;
}
```

#### 示例 2：旧域名永久跳转新域名

```nginx
server {
    listen 80;
    listen 443 ssl;
    server_name old-example.com www.old-example.com;
    # HTTPS跳转必须配置对应域名的证书
    ssl_certificate /etc/nginx/ssl/old-example.com.pem;
    ssl_certificate_key /etc/nginx/ssl/old-example.com.key;
    # 永久跳转到新域名，保留原路径和参数
    return 301 https://www.new-example.com$request_uri;
}
```

#### 示例 3：不带 www 域名统一跳转带 www

```nginx
server {
    listen 443 ssl;
    server_name example.com;
    ssl_certificate /etc/nginx/ssl/example.com.pem;
    ssl_certificate_key /etc/nginx/ssl/example.com.key;
    return 301 https://www.example.com$request_uri;
}
```

#### 示例 4：访问控制，直接返回状态码

```nginx
# 禁止访问敏感后台路径
location /admin/ {
    # 仅允许内网IP访问，其他返回403
    allow 192.168.1.0/24;
    deny all;
    return 403;
}

# 废弃接口直接返回404
location ~* ^/api/v1/old/ {
    return 404;
}
```

#### 示例 5：轻量健康检查接口，直接返回内容

```nginx
location /health {
    access_log off;
    return 200 "ok\n";
    default_type text/plain;
}
```

## 三、核心指令二：rewrite（复杂重写）

### 3.1 指令定位

`rewrite`基于 PCRE 正则表达式匹配请求 URI，实现灵活的地址替换、重定向、内部转发，功能比`return`更强大，支持正则捕获、条件匹配、**内部无感知重写**（地址栏 URL 不变），适合复杂 URL 规则、伪静态、业务路径兼容等场景。

### 3.2 完整语法与核心参数

```nginx
rewrite 正则表达式 替换内容 [标志位];
```

| 参数    | 详细说明                                                      |
| ----- | --------------------------------------------------------- |
| 正则表达式 | 基于 PCRE 正则，匹配请求的**URI 路径部分**（不含域名、query 参数），支持括号捕获分组      |
| 替换内容  | 匹配成功后的目标 URI/URL，支持引用正则捕获组（$1、）、内置变量（host、$args 等）        |
| 标志位   | 核心控制项，决定 rewrite 的执行行为，必选参数（无标志位默认按顺序执行，完成后继续匹配 location） |

### 3.3 核心标志位详解（避坑核心）

|标志位|执行逻辑|地址栏变化|核心适用场景|
|---|---|---|---|
|`last`|终止当前 location 块的后续 rewrite 指令，用新的 URI 重新匹配所有 location 块|不变，用户无感知|内部路径重写、伪静态、入口文件隐藏，需要重新匹配 location 的场景|
|`break`|终止当前 location 块的所有后续 rewrite 指令，直接在当前 location 块处理新 URI，不会重新匹配 location|不变，用户无感知|固定路径内部转发、无需重新匹配 location 的场景，避免循环重写|
|`redirect`|302 临时重定向，替换内容必须是完整 http/https 开头的 URL|变为替换后的地址|临时跳转、不希望浏览器缓存的场景|
|`permanent`|301 永久重定向，替换内容必须是完整 http/https 开头的 URL|变为替换后的地址|永久跳转、SEO 优化、希望浏览器缓存的场景|

⚠️ 关键区分：`last` vs `break`

- 相同点：都是内部重写，地址栏 URL 不变，用户无感知
- 核心区别：`last`会**重新匹配所有 location**，`break`会**停在当前 location**，不再重新匹配
- 示例对比：

```nginx
# 示例1：last标志位
location /test/ {
    rewrite ^/test/(.*)$ /new/$1 last;
    # 后续指令不会执行
}
location /new/ {
    return 200 "匹配到new location，URI: $uri\n";
}
# 访问/test/abc → 重写为/new/abc → 重新匹配到第二个location → 返回内容

# 示例2：break标志位
location /test/ {
    rewrite ^/test/(.*)$ /new/$1 break;
    # 直接在当前location查找文件，根目录为root指定路径
    root /usr/share/nginx/html;
}
location /new/ {
    return 200 "不会匹配到这里";
}
# 访问/test/abc → 重写为/new/abc → 停在当前location，查找/usr/share/nginx/html/new/abc文件
```

### 3.4 常用场景与可落地配置示例

#### 示例 1：URL 伪静态（CMS / 博客系统核心需求）

把动态地址`/index.php?category=1&id=100`转为静态地址`/category/1/100.html`，SEO 友好

```nginx
server {
    listen 80;
    server_name example.com;
    root /usr/share/nginx/html;
    index index.php;

    # 伪静态规则
    location / {
        # 匹配/category/分类ID/文章ID.html
        rewrite ^/category/([0-9]+)/([0-9]+)\.html$ /index.php?category=$1&id=$2 last;
        # 匹配/文章ID.html
        rewrite ^/([0-9]+)\.html$ /index.php?id=$1 last;
        # 兜底，找不到文件走index.php
        try_files $uri $uri/ /index.php?$args;
    }

    # PHP-FPM处理
    location ~ \.php$ {
        fastcgi_pass 127.0.0.1:9000;
        fastcgi_index index.php;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        include fastcgi_params;
    }
}
```

#### 示例 2：隐藏 PHP 入口文件（Laravel/ThinkPHP 框架常用）

```nginx
location / {
    # 所有请求重写到index.php，携带原参数
    rewrite ^(.*)$ /index.php?s=$1 last;
}
```

#### 示例 3：旧接口路径兼容，内部重写不影响前端

业务升级，前端调用`/api/v1/user`，内部转发到新路径`/api/v2/user`，地址栏不变，前端无感知

```nginx
location /api/v1/ {
    # 把v1路径重写为v2，break标志位，在当前location处理
    rewrite ^/api/v1/(.*)$ /api/v2/$1 break;
    # 转发到后端服务
    proxy_pass http://backend_api;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
}
```

#### 示例 4：重写时去除 / 保留 query 参数

```nginx
# 场景1：默认自动保留原query参数
rewrite ^/article/(.*)$ /detail/$1 permanent;
# 访问/article/100?from=wechat → 重定向到/detail/100?from=wechat

# 场景2：去除原query参数，替换内容末尾加?
rewrite ^/article/(.*)$ /detail/$1? permanent;
# 访问/article/100?from=wechat → 重定向到/detail/100，无参数
```

#### 示例 5：防盗链重写，盗链请求跳转到默认图片

```nginx
location ~* \.(jpg|jpeg|png|gif)$ {
    # 校验referer，非本站域名则重定向到防盗链图片
    valid_referers none blocked server_names *.example.com;
    if ($invalid_referer) {
        rewrite ^/ https://example.com/static/forbid.jpg redirect;
    }
    root /usr/share/nginx/html;
    expires 7d;
}
```

## 四、return vs rewrite 核心区别对比

|对比维度|return|rewrite|
|---|---|---|
|性能|极高，无正则解析开销，命中即终止|中等，需要正则匹配解析，有额外开销|
|适用场景|简单跳转、状态码返回、固定内容响应，覆盖 80% 日常场景|复杂 URL 规则、正则匹配、内部重写、伪静态等灵活场景|
|执行逻辑|命中后立即终止当前块所有后续指令，无后续处理|按标志位决定执行逻辑，可能触发重新匹配 location，逻辑更复杂|
|功能复杂度|简单，仅支持固定规则，无正则捕获能力|强大，支持正则捕获、条件匹配、内部重写、地址改写|
|优先级|更高，同一块内 return 先执行，命中后 rewrite 不会执行|更低，同一块内 return 未命中才会执行 rewrite|
|地址改写|仅支持跳转，地址栏必然变化（4xx/200 返回除外）|支持内部重写（地址栏不变）和外部跳转（地址栏变化）|

## 五、重写指令执行优先级与完整流程

Nginx 重写指令有严格的执行顺序，规则冲突时按此流程处理：

1. 先执行**server 块**中的所有 return/rewrite 指令，按书写顺序执行，return 命中则直接终止，不会进入 location 匹配
2. server 块指令执行完成后，用最终的 URI 匹配对应的**location 块**
3. 执行匹配到的 location 块中的 return/rewrite 指令，return 命中则直接终止
4. 若 location 中 rewrite 使用了`last`标志位，则用重写后的 URI 重新回到第 2 步，重新匹配 location（默认最多循环 10 次，超过返回 500 错误）
5. 若 location 中 rewrite 使用了`break`标志位，则终止所有 rewrite 指令，在当前 location 完成后续处理

⚠️ 关键注意：同一块内，return 的优先级永远高于 rewrite，不要在 return 后面写 rewrite，永远不会执行。

## 六、生产环境最佳实践与避坑指南

### 6.1 核心最佳实践

1. **优先使用 return，非必要不用 rewrite**：简单跳转场景用 return，性能更高、逻辑更清晰、不易出错，Nginx 官方也推荐优先使用 return
2. **正则表达式精准化**：避免贪婪匹配，用`^`和`$`限定匹配开头和结尾，减少不必要的匹配，提升性能，避免误匹配
3. **301 跳转先测试再上线**：301 会被浏览器长期缓存，测试阶段先用 302，确认规则无误后再改为 301，避免缓存导致的线上问题
4. **避免循环重写**：谨慎使用 last 标志位，避免重写后的 URI 再次匹配到同一个规则，导致无限循环，Nginx 会返回 500 错误
5. **POST 跳转选对状态码**：POST 表单 / 接口跳转，不要用 301/302，会导致请求方法转为 GET、Body 丢失，必须用 307（临时）/308（永久）
6. **配置前必做语法校验**：每次修改配置后，必须执行`nginx -t`校验语法，确认无误后再执行`nginx -s reload`重载，避免服务中断
7. **复杂规则加注释**：正则规则可读性差，必须添加注释说明规则用途、匹配场景，方便后续维护

### 6.2 常见踩坑与解决方案

|常见坑|原因|解决方案|
|---|---|---|
|rewrite 匹配不到 query 参数|rewrite 默认只匹配 URI 路径，不匹配？后的 query 参数|用`$args`变量获取完整参数，用`$arg_xxx`获取单个参数，通过 if 判断实现参数匹配|
|301 跳转后浏览器一直缓存，修改规则不生效|301 是永久重定向，浏览器会长期缓存|测试用 302，上线再改 301；出现问题让用户清浏览器缓存 / 用无痕模式测试|
|POST 请求跳转后 Body 丢失，请求变成 GET|301/302 会强制把 POST 转为 GET，丢弃 Body|POST 跳转使用 307/308 状态码，严格保留原请求方法和 Body|
|last 标志位导致 500 错误|重写后的 URI 再次匹配到同一个规则，触发无限循环|改用 break 标志位，或调整正则规则，避免循环匹配|
|重写后 CSS/JS/ 图片资源 404|相对路径资源，重写后路径层级变化，导致资源地址错误|资源使用绝对路径，或添加专门的静态资源 location 块，不参与重写|
|HTTPS 跳转后出现证书错误|80 端口跳转 HTTPS，但 443 端口的 server_name 和证书域名不匹配|确保跳转后的域名和证书绑定的域名一致，server_name 配置正确|

## 七、生产环境综合配置示例

```nginx
# HTTP强制跳转HTTPS
server {
    listen 80;
    server_name example.com www.example.com;
    return 301 https://www.example.com$request_uri;
}

# 主HTTPS服务块
server {
    listen 443 ssl http2;
    server_name example.com www.example.com;
    root /var/www/example.com;
    index index.html index.php;

    # SSL证书配置
    ssl_certificate /etc/nginx/ssl/example.com.pem;
    ssl_certificate_key /etc/nginx/ssl/example.com.key;
    ssl_protocols TLSv1.2 TLSv1.3;
    ssl_ciphers ECDHE-ECDSA-AES128-GCM-SHA256:ECDHE-RSA-AES128-GCM-SHA256;
    ssl_prefer_server_ciphers on;

    # 1. 不带www跳转带www
    if ($host = example.com) {
        return 301 https://www.example.com$request_uri;
    }

    # 2. 健康检查接口
    location /health {
        access_log off;
        return 200 "ok\n";
        default_type text/plain;
    }

    # 3. 静态资源处理，不参与重写
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2)$ {
        expires 7d;
        add_header Cache-Control "public, max-age=604800";
    }

    # 4. 旧接口兼容，内部重写
    location /api/v1/ {
        rewrite ^/api/v1/(.*)$ /api/v2/$1 break;
        proxy_pass http://127.0.0.1:8080;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Forwarded-Proto $scheme;
    }

    # 5. 伪静态规则，PHP框架处理
    location / {
        # 文章详情页伪静态
        rewrite ^/article/([0-9]+)\.html$ /index.php?m=article&id=$1 last;
        # 分类页伪静态
        rewrite ^/category/([a-z]+)/([0-9]+)$ /index.php?m=category&code=$1&page=$2 last;
        # 框架兜底规则
        try_files $uri $uri/ /index.php?$args;
    }

    # 6. 禁止访问敏感路径
    location ~* ^/\.env|^/\.git|^/database/ {
        return 403;
    }

    # 7. PHP-FPM处理
    location ~ \.php$ {
        fastcgi_pass 127.0.0.1:9000;
        fastcgi_index index.php;
        fastcgi_param SCRIPT_FILENAME $document_root$fastcgi_script_name;
        include fastcgi_params;
    }
}
```

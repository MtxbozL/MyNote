## 一、开篇：索引设计的核心定位与底层逻辑

### 1. 什么是索引设计

索引设计不是「给表字段随便加索引」，而是**基于 MySQL B + 树索引的底层特性，结合业务查询场景，用最小的空间占用和写入维护成本，最大化查询性能**的系统化工程。

它是之前所有索引相关知识（最左前缀、索引失效、EXPLAIN 执行计划、慢查询优化）的**落地方法论**，也是线上业务数据库性能的核心决定因素 ——80% 的慢查询问题，根源都是不合理的索引设计。

### 2. 索引设计的两大核心平衡目标

所有设计原则都围绕这两个目标展开，缺一不可：

1. **查询性能最大化**：让所有高频业务查询都能命中最优索引，彻底避免全表扫描、回表操作、文件排序、临时表四大性能杀手；
2. **维护成本最小化**：用最少的索引数量覆盖最多的业务场景，最小化索引带来的磁盘空间占用、写入（插入 / 更新 / 删除）维护开销、锁竞争风险。

### 3. 所有设计原则的底层根源

所有原则都基于 InnoDB B + 树索引的三大核心特性，脱离底层特性的设计都是空谈：

1. **有序性是索引的灵魂**：B + 树的核心优势是叶子节点的有序性，所有设计都要「保护索引的有序性」，避免破坏有序性导致索引失效；
2. **回表是高频性能损耗点**：非聚簇索引查询需要通过主键回表获取完整数据，回表次数越多，磁盘 IO 开销越大，设计核心目标之一就是减少 / 避免回表；
3. **索引有明确的写入代价**：每一个索引都是独立的 B + 树，数据插入 / 更新 / 删除时，需要同步维护所有相关索引的 B + 树结构，频繁的页分裂 / 合并会导致写入性能骤降、索引碎片激增。

## 二、第一部分：索引设计前置原则（设计前必须遵守）

这是索引设计的前提，先明确「什么能做、什么不能做」，再进入具体的列顺序、场景化设计，避免从源头就踩坑。

### 2.1 业务驱动原则：索引为查询场景服务，而非为表服务

#### 核心规则

索引的唯一价值是支撑业务查询，**永远不要为表的字段建索引，要为业务的 SQL 语句建索引**。

- 错误认知：给表的每个字段都建单值索引，认为「索引越多，查询越快」；
- 正确做法：先梳理业务的高频查询场景，写出对应的 SQL，再基于 SQL 的查询条件设计索引。

#### 示例

业务高频查询场景：

1. `WHERE name = ? AND age = ?`（90% 的查询）
2. `WHERE name = ?`（50% 的查询）
3. `WHERE age = ?`（1% 的低频查询）

- 错误设计：给`name`和`age`分别建单值索引`idx_name`、`idx_age`；
- 正确设计：建 1 个联合索引`idx_name_age (name, age)`，覆盖前两个高频场景，低频场景无需单独建索引（全表扫描即可）。

### 2.2 SQL 先行原则：先写 SQL，再设计索引

#### 核心规则

索引设计必须在 SQL 语句定稿后进行，**禁止先建索引，再写业务 SQL**。

- 原因：MySQL 优化器对索引的选择，完全依赖 SQL 的写法（WHERE 条件、JOIN 关联、排序分组），先建索引再写 SQL，大概率会出现索引失效、无法命中的问题；
- 落地要求：业务开发流程中，必须先完成 SQL 编写、评审，再进入索引设计、上线环节。

### 2.3 最小化原则：一索引多用，杜绝冗余索引

#### 核心规则

能用 1 个索引覆盖的场景，绝不建 2 个索引；能用联合索引覆盖的场景，绝不建多个单值索引。

- 核心收益：减少索引数量，降低写入维护成本和磁盘空间占用；
- 落地依据：联合索引的最左前缀原则，1 个合理的联合索引，可覆盖多个查询场景。

#### 反例与正例

- 反例：为 3 个查询场景分别建 3 个索引：`idx_name`、`idx_name_age`、`idx_name_age_gender`；
- 正例：仅建 1 个索引`idx_name_age_gender (name, age, gender)`，通过最左前缀原则，覆盖 3 个场景，无任何冗余。

### 2.4 冷热分离原则：热数据优先优化，冷数据精简索引

#### 核心规则

- 热数据（高频访问的当前业务数据）：优先设计高性能索引，保证查询效率；
- 冷数据（归档的历史数据、低频访问数据）：尽量精简索引，甚至不建索引，仅保留核心查询字段的索引。

#### 原因

冷数据几乎不会写入，查询频率极低，过多的索引会占用大量磁盘空间，且归档、迁移时开销极大；冷数据的查询通常是离线统计，对实时性要求不高，全表扫描也可接受。

### 2.5 单表索引数量控制原则：单表索引不超过 5 个

#### 核心规则

单表的索引数量严格控制在**5 个以内**，绝对禁止超过 10 个。

#### 为什么要限制数量

1. 写入性能骤降：插入 / 更新 / 删除数据时，需要同步维护所有索引的 B + 树，索引越多，写入开销越大，TPS 吞吐量越低；
2. 优化器选择混乱：索引过多，MySQL 优化器计算每个索引的执行成本时，容易出现误判，选错执行计划；
3. 运维成本升高：索引碎片、统计信息更新、冗余索引清理的成本成倍增加。

## 三、第二部分：联合索引列顺序设计核心原则（重中之重）

联合索引是生产环境中使用最多的索引类型，列顺序直接决定了索引的利用率和查询性能，是索引设计的核心。所有排序原则的优先级从高到低，严格遵循。

### 3.1 第一优先级：等值查询优先原则

#### 核心规则

**所有等值查询的列，必须放在联合索引的最左侧**，等值查询列之间的顺序可按后续原则调整，范围查询列必须放在所有等值查询列的后面。

#### 底层逻辑

等值查询（`=`、`IN`）可以精准锁定索引的范围，只有先通过等值列锁定范围后，后续列的有序性才能被利用；如果范围列前置，会直接截断后续列的索引匹配。

#### 反例与正例

- 查询 SQL：`WHERE name = '张三' AND age > 20 AND gender = '男'`
- 反例：索引`idx_name_age_gender (name, age, gender)`，`age`是范围列，放在中间，`gender`列索引完全失效；
- 正例：索引`idx_name_gender_age (name, gender, age)`，等值列`name、gender`全部前置，范围列`age`放在最后，三列索引全量生效。

### 3.2 第二优先级：高频查询优先原则

#### 核心规则

在等值查询列中，**查询频率越高的列，越靠左放置**，让一个联合索引覆盖更多的查询场景，最大化索引利用率。

#### 示例

业务查询场景：

1. `WHERE name = ? AND age = ? AND gender = ?`（高频）
2. `WHERE name = ? AND age = ?`（高频）
3. `WHERE name = ?`（高频）
4. `WHERE age = ? AND gender = ?`（低频）

- 最优索引设计：`idx_name_age_gender (name, age, gender)`，覆盖 3 个高频场景，索引利用率 100%；
- 错误设计：`idx_gender_age_name (gender, age, name)`，仅能覆盖 1 个场景，索引利用率极低。

### 3.3 第三优先级：高区分度优先原则

#### 核心规则

在等值查询列中，**列的区分度越高，越靠左放置**。

- 区分度计算公式：`区分度 = COUNT(DISTINCT 列名) / COUNT(*)`，取值范围 0-1，越接近 1，区分度越高；
- 区分度越高，索引的过滤效果越好，扫描的行数越少，查询性能越高。

#### 示例

- 高区分度列：手机号、身份证号、用户 ID、订单号（区分度≈1）；
- 中区分度列：姓名、年龄、创建时间；
- 低区分度列：性别、状态、是否删除（区分度≈0.001）。

查询 SQL：`WHERE gender = '男' AND name = '张三'`

- 反例：索引`idx_gender_name (gender, name)`，低区分度列前置，索引先过滤出 50% 的行，再过滤 name，扫描行数多；
- 正例：索引`idx_name_gender (name, gender)`，高区分度列前置，索引先过滤出 1-2 行，再过滤 gender，扫描行数极少，性能提升数十倍。

#### 例外情况

如果低区分度列是高频等值查询列，且符合「等值优先、高频优先」原则，依然优先高频列前置。比如 90% 的查询都带`gender = '男'`条件，即使区分度低，也可放在左侧。

### 3.4 第四优先级：范围查询后置原则

#### 核心规则

**所有范围查询列（`>`、`<`、`>=`、`<=`、`BETWEEN AND`、`LIKE '前缀%'`），必须统一放在联合索引的最后面**。

#### 底层逻辑

范围查询会中断后续列的索引匹配，MySQL 只能利用范围列之前的索引列，范围列之后的列无法利用索引的有序性，只能回表过滤。

#### 关键补充

`IN()` 属于**等值查询**，不是范围查询，不会中断后续列的匹配，可放在等值查询列中，无需后置。

### 3.5 第五优先级：排序分组适配原则

#### 核心规则

当查询中有`ORDER BY`/`GROUP BY`时，**排序 / 分组的列要跟在等值查询列的后面，且顺序与索引列顺序、排序方向完全一致**，利用索引的有序性，彻底避免文件排序（Using filesort）和临时表（Using temporary）。

#### 示例

- 查询 SQL：`WHERE name = '张三' ORDER BY age ASC`
- 反例：索引`idx_name (name)`，WHERE 条件命中索引，但 ORDER BY 的 age 不在索引中，出现`Using filesort`文件排序；
- 正例：索引`idx_name_age (name, age)`，name 等值匹配后，age 在索引中是有序的，直接利用索引排序，无文件排序。

#### 进阶规则

1. 排序方向必须全部一致（全 ASC / 全 DESC），混合排序无法利用索引；
2. GROUP BY 的列必须和索引列顺序一致，才能避免临时表；
3. 排序 / 分组列不能跨范围查询列，否则无法利用索引有序性。

## 四、第三部分：场景化索引设计原则（生产环境实战）

针对生产环境高频的业务查询场景，给出专属的索引设计原则，直接落地可用。

### 4.1 单表查询场景设计原则

1. **单条件等值查询**：给查询列建普通单值索引，唯一值列建唯一索引；
2. **多条件联合查询**：按「等值优先、高频优先、高区分度优先、范围后置」建联合索引；
3. **分页查询场景**：设计覆盖索引，避免深分页的全表扫描和多次回表，比如`LIMIT 100000, 20`，用`idx_create_time_id (create_time, id)`覆盖索引优化；
4. **模糊查询场景**：前缀匹配查询建普通索引，全模糊匹配用全文索引 / 搜索引擎，绝对禁止左模糊匹配；
5. **聚合查询场景**：给`WHERE`条件 + 聚合函数字段建覆盖索引，避免回表，比如`SELECT SUM(amount) FROM order WHERE user_id = ?`，建索引`idx_userid_amount (user_id, amount)`。

### 4.2 JOIN 关联查询场景设计原则

1. **核心原则：被驱动表的关联字段必须建索引**：这是 JOIN 优化的第一原则，驱动表的每一行数据，都会去被驱动表中匹配，关联字段无索引会导致嵌套循环全表扫描；
2. **关联字段一致性原则**：JOIN 关联字段的类型、长度、字符集必须完全一致，避免隐式类型转换导致索引失效；
3. **小表驱动大表原则**：MySQL 优化器会自动选择小表（WHERE 过滤后结果集更小的表）作为驱动表，索引设计要适配这个逻辑，优先给大表（被驱动表）的关联字段建索引；
4. **禁止超过 3 张表的 JOIN**：多表 JOIN 会导致优化器选择执行计划的难度指数级升高，索引设计难度大，业务上尽量拆分，或用冗余字段避免多表关联。

### 4.3 大字段场景设计原则

1. **长字符串字段用前缀索引**：对于`VARCHAR(255+)`、`TEXT`等长字符串字段，禁止建全量索引，用前缀索引替代，平衡区分度和索引大小；
    
    - 设计方法：找到最小的前缀长度 N，让前缀的区分度接近全字段的区分度，比如`ALTER TABLE user ADD INDEX idx_name_prefix (name(5))`；
    
2. **大字段禁止加入联合索引**：`TEXT`、`BLOB`等大字段，绝对禁止加入联合索引，会导致索引文件急剧膨胀，查询和写入性能骤降；
3. **大字段查询用覆盖索引**：如果必须查询大字段，尽量用主键查询，或设计覆盖索引，避免回表。

### 4.4 分库分表场景设计原则

1. **分片键必须建索引**：分片键是分库分表的核心，必须建索引，且优先放在联合索引的最左侧；
2. **禁止跨库 JOIN**：跨库 JOIN 是分布式数据库的大忌，通过字段冗余避免跨库关联，冗余字段必须建索引；
3. **非分片键查询尽量收敛**：非分片键的查询尽量包含分片键，避免全表扫描，非分片键的索引要结合分片键设计联合索引。

### 4.5 唯一约束场景设计原则

1. **业务唯一字段必须建唯一索引**：手机号、身份证号、订单号等业务上必须唯一的字段，必须建唯一索引，从数据库层面保证数据唯一性，避免脏数据；
2. **唯一索引优先于业务代码校验**：唯一索引是数据唯一性的最后一道防线，即使业务代码做了校验，数据库层面也必须建唯一索引；
3. **联合唯一索引的顺序设计**：按高频查询优先、高区分度优先设计列顺序，让联合唯一索引同时兼顾唯一性约束和查询加速。

## 五、第四部分：索引设计避坑红线原则（绝对禁止）

这些是索引设计的红线，一旦违反，必然会导致索引失效、性能下降、数据异常等问题，必须严格遵守。

1. **禁止给低区分度列单独建索引**
    
    - 比如性别、状态、是否删除、是否上架等只有 2-3 个枚举值的列，单独建索引没有任何意义，优化器不会选择，反而增加写入开销；
    - 正确做法：仅可作为联合索引的后缀列，和高区分度列一起建联合索引。
    
2. **禁止给小表建索引**
    
    - 行数 < 1000 的小表，全表扫描的性能远高于索引查询（索引查询需要扫描索引页 + 回表，全表扫描仅需扫描 1-2 个数据页）；
    - 正确做法：小表无需建索引，全表扫描即可。
    
3. **禁止冗余索引**
    
    - 已有联合索引`(a,b,c)`，则`(a)`、`(a,b)`都是冗余索引，最左前缀原则可完全覆盖，无需重复创建；
    - 正确做法：定期巡检冗余索引，直接删除，不影响任何查询性能。
    
4. **禁止给频繁更新的列建索引**
    
    - 频繁更新的列，每次更新都需要同步维护索引的 B + 树，导致频繁的页分裂、页合并，写入性能骤降，索引碎片激增；
    - 正确做法：尽量给静态字段建索引，频繁更新的字段非必要不建索引。
    
5. **禁止用 UUID / 随机字符串做主键 / 聚簇索引**
    
    - UUID / 随机字符串是无序的，插入时会导致聚簇索引频繁页分裂，写入性能骤降，索引碎片严重，且所有二级索引都会存储主键值，导致索引膨胀；
    - 正确做法：统一用自增`BIGINT UNSIGNED`做主键，业务唯一字段建唯一索引。
    
6. **禁止用函数 / 表达式操作索引列**
    
    - 设计阶段就要避免这种场景，比如不要把日期存成字符串，避免查询时用`DATE()`函数转换；不要给手机号做算术运算，避免隐式类型转换；
    - 正确做法：保持索引列的原始值不变，把函数 / 运算放在值的一侧。
    
7. **禁止给 NULL 值列建核心索引**
    
    - NULL 值会导致索引统计、比较运算更复杂，优化器更难选择执行计划，`IS NOT NULL`大概率会导致索引失效；
    - 正确做法：所有索引列尽量设为`NOT NULL`，给合理的默认值。

## 六、第五部分：索引全生命周期运维原则

索引不是建完就一劳永逸的，需要全生命周期的运维管理，保证索引长期健康、高效。

### 6.1 上线前验证原则

索引上线前，必须完成两项验证，否则禁止上线：

1. **EXPLAIN 执行计划验证**：用 EXPLAIN 查看目标 SQL 的执行计划，确认索引命中、`type`达到 range 及以上、无文件排序 / 临时表 / 全表扫描；
2. **性能压测验证**：在测试环境模拟线上流量，验证索引上线后的 SQL 执行耗时、数据库 CPU/IO 负载，确认性能符合预期。

### 6.2 定期巡检原则

每周 / 每月对核心业务表的索引进行巡检，重点完成：

1. **冗余 / 无效索引清理**：用 MySQL sys 库查看冗余索引、从未使用过的无效索引，确认无业务使用后直接删除；

```sql
    -- 查看冗余索引
    SELECT * FROM sys.schema_redundant_indexes;
    -- 查看未使用的索引
    SELECT * FROM sys.schema_unused_indexes;
```
    
2. **索引碎片检查**：查看索引碎片率，碎片率超过 30% 的索引，在业务低峰期整理碎片；
3. **统计信息更新**：大表批量数据插入 / 更新 / 删除后，手动更新表的统计信息，避免优化器选错索引。

### 6.3 大表索引变更规范

千万级以上的大表，索引变更必须遵守以下规范，避免锁表、业务阻塞：

1. **使用 Online DDL**：MySQL 5.6+ 用`ALGORITHM=INPLACE, LOCK=NONE`执行索引变更，不会锁全表，仅在最后阶段加短暂的元数据锁；
2. **第三方工具兜底**：超大型表推荐用`gh-ost`、`pt-online-schema-change`等工具执行变更，无锁表风险，避免主从延迟；
3. **低峰期执行**：索引变更必须在凌晨等业务低峰期执行，全程监控数据库 CPU/IO/ 连接数，出现异常立即停止；
4. **禁止批量建索引**：多个索引合并成一条语句执行，避免多次扫描全表。

### 6.4 索引下线规范

索引下线必须遵循「先观察、再下线、可回滚」的原则，避免影响线上业务：

1. 先关闭索引（MySQL 8.0 + 支持`ALTER TABLE ... ALTER INDEX ... INVISIBLE`），观察 1-3 天，确认业务无异常、无慢查询；
2. 确认无影响后，再执行删除操作；
3. 下线前必须备份表结构和建索引语句，出现问题可快速回滚。

## 七、索引设计效果验证标准流程

索引设计完成后，必须通过 EXPLAIN 执行计划验证效果，按以下 6 步执行，确认索引符合预期：

1. **看`type`**：确认索引生效，达到`range`及以上级别，最优是`ref/eq_ref/const`，绝对禁止`ALL`全表扫描；
2. **看`key`**：确认实际命中了我们设计的索引，`key`不为 NULL；
3. **看`key_len`**：确认索引的利用率，`key_len`越接近索引总长度，利用率越高，避免索引仅用到前缀列；
4. **看`rows`**：确认扫描行数大幅下降，越接近返回行数越好，过滤效果达标；
5. **看`Extra`**：确认无`Using filesort`、`Using temporary`、`Using join buffer`，出现`Using index`（覆盖索引）为最优；
6. **实际执行验证**：执行 SQL，确认执行耗时、扫描行数符合预期，无性能问题。

## 九、核心要点总结

1. **底层逻辑是根基**：所有索引设计原则都源于 B + 树的有序性、回表代价、写入维护成本，脱离底层特性的设计都是空谈。
2. **四大核心排序原则**：等值优先、高频优先、高区分度优先、范围查询后置，是联合索引设计的灵魂，决定了索引的利用率。
3. **业务驱动是前提**：索引永远为业务查询场景服务，先写 SQL 再设计索引，禁止为字段建索引。
4. **平衡是核心**：索引设计不是追求极致的查询性能，而是在查询性能和写入维护成本之间找到平衡，用最少的索引覆盖最多的场景。
5. **红线绝对不能碰**：禁止给低区分度列单独建索引、禁止冗余索引、禁止 UUID 做主键、禁止频繁更新列建索引，从源头避免性能问题。
6. **全生命周期管理**：索引不是建完就结束了，上线前验证、定期巡检、变更规范、下线流程，保证索引长期健康高效。
7. **验证是唯一标准**：所有索引设计的效果，最终都要通过 EXPLAIN 执行计划和实际性能测试验证，避免主观判断。

## 一、开篇：主从复制的核心定位与前置说明

### 1.1 为什么主从复制如此重要

主从复制是 MySQL 高可用架构的**基石**，是生产环境的标配，核心价值包括：

- **读写分离**：主库负责写（INSERT/UPDATE/DELETE），从库负责读（SELECT），分散主库压力，提升系统整体吞吐量；
- **数据备份**：从库作为主库的实时备份，主库故障时可快速切换，保证数据不丢失；
- **高可用支撑**：配合 MHA、Orchestrator 等高可用工具，实现主库故障的自动切换，缩短停机时间；
- **负载均衡**：一主多从架构下，读请求可分发到多个从库，支撑更高的并发读；
- **业务支撑**：从库可用于数据分析、报表生成、测试环境搭建，不影响主库业务。

### 1.2 前置核心说明

1. **版本兼容**：主从库的 MySQL 版本尽量一致，高版本从库可兼容低版本主库，但低版本从库无法兼容高版本主库；
2. **网络要求**：主从库之间网络必须稳定，延迟尽量 < 10ms，否则会导致主从延迟严重；
3. **server-id 唯一**：主从库的 `server-id` 必须全局唯一，不能重复；
4. **binlog 必须开启**：主库必须开启 binlog（二进制日志），这是主从复制的基础；
5. **数据一致性前提**：搭建主从前，主从库的数据必须一致，否则复制会报错。

## 二、第一部分：主从复制核心原理（必须理解，否则 90% 概率踩坑）

### 2.1 主从复制的三个核心线程

主从复制的本质是**从库通过 binlog 重放主库的数据修改**，整个过程由 3 个核心线程协作完成：

|线程|所在库|核心作用|
|:--|:--|:--|
|**Binlog Dump 线程**|主库|当从库连接主库时，主库启动该线程，读取主库的 binlog 事件，发送给从库的 I/O 线程。|
|**I/O 线程**|从库|连接主库，请求 binlog 事件，接收后写入从库的 **relay log（中继日志）**，并记录当前读取到的主库 binlog 位置。|
|**SQL 线程**|从库|读取并解析 relay log 中的事件，在从库上重放（执行）这些事件，使从库的数据与主库保持一致。|

### 2.2 主从复制完整流程（6 步走）

1. **主库记录 binlog**：主库执行写操作后，将数据修改记录到 binlog（二进制日志）中；
2. **从库连接主库**：从库通过 `CHANGE MASTER TO` 配置主库信息，启动 I/O 线程，连接主库；
3. **主库发送 binlog**：主库的 Binlog Dump 线程读取 binlog，发送给从库的 I/O 线程；
4. **从库写入 relay log**：从库的 I/O 线程接收 binlog 事件，写入本地的 relay log（中继日志）；
5. **从库重放 relay log**：从库的 SQL 线程读取 relay log，解析并执行其中的事件，在从库上重放主库的修改；
6. **从库更新复制位置**：从库记录当前重放的 relay log 位置和主库的 binlog 位置，下次复制从该位置继续。

### 2.3 binlog 的三种格式（核心选择）

binlog 是主从复制的核心，有三种格式，生产环境必须选择合适的格式：

|binlog 格式|核心说明|优点|缺点|适用场景|
|:--|:--|:--|:--|:--|
|**STATEMENT**|记录执行的 SQL 语句（如 `UPDATE user SET age=26 WHERE id=1`）|1. binlog 文件小；<br><br>2. 可读性强；<br><br>3. 兼容性好|1. 数据一致性风险：如 `NOW()`、`RAND()` 等函数，主从执行结果可能不一致；<br><br>2. 无法正确复制触发器、存储过程|已淘汰，生产环境禁止使用|
|**ROW**|记录数据行的变化（如哪一行的哪一列从什么值改成了什么值）|1. 数据一致性极高：主从数据完全一致；<br><br>2. 支持触发器、存储过程、函数的正确复制；<br><br>3. 是 MySQL 8.0 的默认格式|1. binlog 文件大（尤其是批量更新）；<br><br>2. 可读性差|**生产环境首选，推荐**|
|**MIXED**|混合模式，默认用 STATEMENT，遇到特殊情况（如 `NOW()`、`RAND()`）自动切换为 ROW|兼顾 STATEMENT 的小文件和 ROW 的一致性|仍有一致性风险，不如纯 ROW 可靠|不推荐，直接用 ROW|

> 核心规范：**生产环境统一使用 ROW 格式的 binlog**，保证数据一致性。

## 三、第二部分：主从复制环境准备

### 3.1 服务器规划（以一主一从为例）

|角色|IP 地址|主机名|server-id|
|:--|:--|:--|:--|
|主库（Master）|192.168.1.100|mysql-master|1|
|从库（Slave）|192.168.1.101|mysql-slave|2|

### 3.2 主库配置（my.cnf）

编辑主库的 MySQL 配置文件（`/etc/my.cnf` 或 `/etc/mysql/my.cnf`），添加以下配置：

```bash
[mysqld]
# 基础配置
server-id = 1                              # 全局唯一，主库设为1
datadir = /var/lib/mysql
socket = /var/lib/mysql/mysql.sock
port = 3306

# Binlog 核心配置（主从复制必须）
log_bin = mysql-bin                      # 开启 binlog，文件名前缀为 mysql-bin
binlog_format = ROW                       # binlog 格式，统一用 ROW
binlog_do_db = my_database                # 可选，只复制指定库（不推荐，用默认全库复制更稳妥）
binlog_ignore_db = mysql,information_schema,performance_schema  # 可选，忽略系统库
expire_logs_days = 7                      # binlog 保留天数，避免磁盘空间不足
max_binlog_size = 1G                      # 单个 binlog 文件最大大小

# 主从复制优化配置
sync_binlog = 1                            # 每次事务提交都刷盘 binlog，保证数据一致性（双1配置之一）
innodb_flush_log_at_trx_commit = 1        # 每次事务提交都刷盘 redo log，保证数据一致性（双1配置之二）
```

配置完成后，**重启主库 MySQL 服务**：

```bash
systemctl restart mysqld
```

### 3.3 从库配置（my.cnf）

编辑从库的 MySQL 配置文件，添加以下配置：

```bash
[mysqld]
# 基础配置
server-id = 2                              # 全局唯一，不能和主库重复，从库设为2
datadir = /var/lib/mysql
socket = /var/lib/mysql/mysql.sock
port = 3306

# Relay log 核心配置（从库必须）
relay_log = mysql-relay-bin                # 开启 relay log，文件名前缀为 mysql-relay-bin
read_only = 1                               # 从库设为只读（超级用户除外），避免误写
super_read_only = 1                         # MySQL 8.0+，超级用户也只读（可选，更安全）

# 主从复制优化配置
log_slave_updates = 0                       # 从库是否将重放的事件写入自己的 binlog（双主/级联复制时设为1，一主一从设为0）
relay_log_purge = 1                         # 自动删除已重放的 relay log，避免磁盘空间不足
```

配置完成后，**重启从库 MySQL 服务**：

```bash
systemctl restart mysqld
```

### 3.4 主库创建复制专用用户

遵循最小权限原则，创建专门用于主从复制的用户，仅授予复制所需的权限：

```sql
-- 登录主库 MySQL
mysql -u root -p

-- 创建复制用户，仅允许从库 IP 登录
CREATE USER 'repl_user'@'192.168.1.101' IDENTIFIED BY 'Repl@20260401';

-- 授予复制权限
GRANT REPLICATION SLAVE ON *.* TO 'repl_user'@'192.168.1.101';

-- 刷新权限
FLUSH PRIVILEGES;
```

## 四、第三部分：主从复制搭建完整步骤（实战核心）

### 4.1 步骤 1：主库锁表并记录 binlog 位置

⚠️ **注意**：锁表期间主库无法写入，必须在业务低峰期执行！

```sql
-- 登录主库 MySQL
mysql -u root -p

-- 锁表，禁止写入（保证备份期间数据一致）
FLUSH TABLES WITH READ LOCK;

-- 记录主库的 binlog 文件名和位置（从库配置时需要，必须记录！）
SHOW MASTER STATUS;
```

`SHOW MASTER STATUS` 输出示例（重点记录前两列）：

| File             | Position | Binlog_Do_DB | Binlog_Ignore_DB         | Executed_Gtid_Set |
| :--------------- | :------- | :----------- | :----------------------- | :---------------- |
| mysql-bin.000005 | 1200     |              | mysql,information_schema |                   |

- `File`：当前的 binlog 文件名（`mysql-bin.000005`）；
- `Position`：当前的 binlog 位置（`1200`）；
- **这两个值必须记录，从库配置时必须用到！**

### 4.2 步骤 2：主库备份数据（保持锁表状态）

在主库锁表的状态下，用 mysqldump 备份数据，保证备份数据与 binlog 位置一致：

```bash
# 在主库服务器执行，备份所有库（或指定库）
mysqldump -u root -p \
--single-transaction \
--master-data=2 \
--routines \
--triggers \
--events \
--all-databases > /backup/master_backup_20260401.sql
```

- `--master-data=2`：会在备份文件中记录主库的 binlog 文件名和位置，从库导入后可直接查看。

备份完成后，**解锁主库**：

```sql
-- 登录主库 MySQL，解锁表，恢复写入
UNLOCK TABLES;
```

### 4.3 步骤 3：从库导入主库备份数据

将主库的备份文件传输到从库服务器，然后导入从库：

```bash
# 在从库服务器执行，导入备份数据
mysql -u root -p < /backup/master_backup_20260401.sql
```

### 4.4 步骤 4：从库配置主库信息

登录从库 MySQL，配置主库的连接信息、binlog 位置：

```sql
-- 登录从库 MySQL
mysql -u root -p

-- 停止从库复制（如果之前开启过）
STOP SLAVE;

-- 配置主库信息（核心步骤！）
CHANGE MASTER TO
MASTER_HOST='192.168.1.100',       -- 主库 IP
MASTER_USER='repl_user',              -- 复制用户名
MASTER_PASSWORD='Repl@20260401',     -- 复制用户密码
MASTER_PORT=3306,                     -- 主库端口
MASTER_LOG_FILE='mysql-bin.000005',  -- 步骤1记录的主库 binlog 文件名
MASTER_LOG_POS=1200;                  -- 步骤1记录的主库 binlog 位置
```

### 4.5 步骤 5：启动从库复制并验证

```sql
-- 启动从库复制
START SLAVE;

-- 查看从库复制状态（核心验证步骤！）
SHOW SLAVE STATUS\G
```

#### 验证成功的核心标志（必须同时满足）

`SHOW SLAVE STATUS\G` 输出中，重点看以下 3 个参数：

1. **`Slave_IO_Running: Yes`**：从库 I/O 线程正常运行，正在接收主库的 binlog；
2. **`Slave_SQL_Running: Yes`**：从库 SQL 线程正常运行，正在重放 relay log；
3. **`Seconds_Behind_Master: 0`**：从库延迟为 0，主从数据完全一致（刚搭建时为 0，后续可能有小延迟，正常）。

如果这 3 个参数都正常，恭喜你，主从复制搭建成功！

### 4.6 步骤 6：测试主从复制

在主库执行写操作，验证从库是否同步：

```sql
-- 1. 主库执行写操作
USE my_database;
INSERT INTO user_info (user_name, phone) VALUES ('测试用户', '13900000000');

-- 2. 从库查询，验证数据是否同步
USE my_database;
SELECT * FROM user_info WHERE phone = '13900000000';
```

如果从库能查到刚插入的数据，说明主从复制正常工作！

## 五、第四部分：主从复制运维管理

### 5.1 查看主从状态

#### 主库状态查看

```sql
-- 查看主库的 binlog 状态
SHOW MASTER STATUS;

-- 查看主库的 binlog 文件列表
SHOW BINARY LOGS;

-- 查看连接到主库的从库列表
SHOW SLAVE HOSTS;
```

#### 从库状态查看

```sql
-- 查看从库复制状态（最常用，\G 表示垂直显示，更清晰）
SHOW SLAVE STATUS\G

-- 查看从库的 relay log 状态
SHOW RELAYLOG EVENTS;
```

### 5.2 主从复制启停

#### 从库复制启停

```sql
-- 停止从库复制
STOP SLAVE;

-- 启动从库复制
START SLAVE;

-- 仅停止 I/O 线程
STOP SLAVE IO_THREAD;

-- 仅停止 SQL 线程
STOP SLAVE SQL_THREAD;
```

### 5.3 主从切换（手动切换，主库故障时用）

假设主库（192.168.1.100）故障，需要将从库（192.168.1.101）提升为新主库，步骤如下：

1. **停止从库复制**：

```sql
    -- 登录从库
    STOP SLAVE;
    RESET SLAVE ALL;  -- 清除从库配置
```
    
2. **从库设为读写模式**：

```sql
    -- 关闭只读
    SET GLOBAL read_only = 0;
    SET GLOBAL super_read_only = 0;
```
    
3. **修改从库配置为新主库**：
    
    - 编辑从库的 `my.cnf`，开启 binlog，设置 `server-id=1`（或保持 2，只要唯一即可）；
    - 重启从库 MySQL。
    
4. **其他从库（如果有）指向新主库**：
    
    - 其他从库执行 `CHANGE MASTER TO`，指向新主库的 IP 和 binlog 位置。
    
5. **业务切换**：
    
    - 修改业务代码的数据库连接地址，指向新主库。

## 六、第五部分：常见问题排查（90% 的问题都在这里）

### 6.1 问题 1：Slave_IO_Running: No

#### 常见原因

1. **网络问题**：主从库之间网络不通，防火墙未开放 3306 端口；
2. **主库 binlog 位置错误**：`CHANGE MASTER TO` 时 `MASTER_LOG_FILE` 或 `MASTER_LOG_POS` 填错；
3. **复制用户权限错误**：复制用户没有 `REPLICATION SLAVE` 权限，或主机限制错误；
4. **主库 binlog 已删除**：主库的 binlog 已过期删除，从库找不到对应的 binlog。

#### 排查步骤

```sql
-- 1. 查看从库错误日志（最直接）
SHOW SLAVE STATUS\G
-- 重点看 Last_IO_Error 字段，会显示具体错误原因

-- 2. 测试主从网络连通性（从库服务器执行）
telnet 192.168.1.100 3306
# 或
mysql -h 192.168.1.100 -u repl_user -p

-- 3. 重新配置主库信息（如果是 binlog 位置错误）
STOP SLAVE;
CHANGE MASTER TO MASTER_LOG_FILE='正确的文件名', MASTER_LOG_POS=正确的位置;
START SLAVE;
```

### 6.2 问题 2：Slave_SQL_Running: No

#### 常见原因

1. **数据不一致**：主从数据不一致，从库重放时出现主键冲突、唯一键冲突；
2. **SQL 执行错误**：从库重放的 SQL 在从库执行失败（如表不存在、字段不存在）；
3. **从库误写**：从库开启了读写，手动写入了数据，导致与主库冲突。

#### 排查步骤

```sql
-- 1. 查看从库错误日志
SHOW SLAVE STATUS\G
-- 重点看 Last_SQL_Error 字段，显示具体错误 SQL

-- 2. 方案1：跳过错误（仅用于测试环境，生产环境不推荐，会导致数据不一致）
STOP SLAVE;
SET GLOBAL sql_slave_skip_counter = 1;  -- 跳过1个错误
START SLAVE;

-- 3. 方案2：重新搭建主从（生产环境推荐，保证数据一致）
-- 停止从库，删除数据，重新从主库备份导入，配置主从

-- 4. 方案3：用 pt-table-checksum + pt-table-sync 修复数据不一致（生产环境推荐）
-- 先检查数据不一致，再同步修复
```

### 6.3 问题 3：Seconds_Behind_Master 很大（主从延迟严重）

#### 常见原因

1. **主库大事务**：主库执行了大事务（如批量更新 100 万行数据），binlog 很大，从库重放慢；
2. **从库性能差**：从库的 CPU、内存、磁盘性能比主库差，重放速度跟不上主库写入速度；
3. **从库锁等待**：从库有长事务、锁等待，SQL 线程被阻塞；
4. **binlog 格式问题**：用了 STATEMENT 格式，或 ROW 格式但批量更新导致 binlog 过大；
5. **从库 SQL 线程单线程**：MySQL 5.6 之前 SQL 线程是单线程的，重放速度慢。

#### 解决方法

1. **优化主库大事务**：将大事务拆分为小事务，避免一次性写入大量数据；
2. **提升从库性能**：从库配置与主库一致，或更好；
3. **开启从库并行复制**：MySQL 5.6+ 支持并行复制，提升 SQL 线程重放速度；

```sql
    -- MySQL 5.7+ 并行复制配置（my.cnf）
    slave_parallel_type = LOGICAL_CLOCK
    slave_parallel_workers = 4  -- 并行线程数，根据CPU核心数设置
```
    
4. **优化 binlog 格式**：统一用 ROW 格式，避免批量更新导致的大 binlog；
5. **从库只读**：确保从库开启 `read_only`，避免误写导致锁等待。

## 七、第六部分：主从复制高级特性

### 7.1 半同步复制（Semi-Synchronous Replication）

#### 核心价值

默认的主从复制是**异步复制**：主库执行完事务，提交后立即返回成功，不管 binlog 是否发送到从库 —— 如果主库故障，binlog 还没发送到从库，会导致数据丢失。

**半同步复制**：主库执行完事务，必须等待至少一个从库确认收到 binlog 并写入 relay log，才返回成功 —— 保证数据至少在两个节点（主库 + 一个从库）存在，数据一致性更高。

#### 配置方法

1. **主从库都安装半同步插件**：

```sql
    -- 主库安装
    INSTALL PLUGIN rpl_semi_sync_master SONAME 'semisync_master.so';
    -- 从库安装
    INSTALL PLUGIN rpl_semi_sync_slave SONAME 'semisync_slave.so';
```
    
2. **主从库都开启半同步**：

```
    -- 主库配置
    SET GLOBAL rpl_semi_sync_master_enabled = 1;
    SET GLOBAL rpl_semi_sync_master_timeout = 1000;  -- 超时时间1秒，超时后自动降级为异步复制
    
    -- 从库配置
    SET GLOBAL rpl_semi_sync_slave_enabled = 1;
    STOP SLAVE IO_THREAD;
    START SLAVE IO_THREAD;
```
    
3. **验证半同步**：

```sql
    -- 主库查看
    SHOW VARIABLES LIKE 'rpl_semi_sync%';
    SHOW STATUS LIKE 'Rpl_semi_sync%';
```

### 7.2 GTID 复制（Global Transaction ID）

#### 核心价值

GTID 是 MySQL 5.6+ 新增的特性，给每个事务分配一个全局唯一的 ID，简化主从切换，不需要手动记录 binlog 文件名和位置，主从切换更简单、更安全。

#### 核心原理

- GTID 格式：`server_uuid:transaction_id`，如 `3e11fa47-0d4d-11ef-9a0b-000c29b6e8f1:1-100`；
- 从库通过 GTID 自动识别哪些事务已经重放，哪些还没重放，不需要手动指定 binlog 位置。

#### 配置方法

1. **主从库都配置 GTID（my.cnf）**：

```bash
    [mysqld]
    gtid_mode = ON
    enforce_gtid_consistency = ON
    log_bin = mysql-bin
    binlog_format = ROW
    server-id = 1  # 主库1，从库2
```
    
2. **重启主从库 MySQL**；
3. **从库配置主库（GTID 模式，不需要 binlog 位置）**：

```sql
    STOP SLAVE;
    CHANGE MASTER TO
    MASTER_HOST='192.168.1.100',
    MASTER_USER='repl_user',
    MASTER_PASSWORD='Repl@20260401',
    MASTER_PORT=3306,
    MASTER_AUTO_POSITION = 1;  -- 开启 GTID 自动定位
    START SLAVE;
```

## 八、核心避坑指南

1. **坑 1：server-id 重复**
    
    - 后果：主从复制无法启动，I/O 线程报错；
    - 避坑：主从库的 server-id 必须全局唯一，不能重复。
    
2. **坑 2：binlog 格式用 STATEMENT**
    
    - 后果：主从数据不一致，如 `NOW()`、`RAND()` 函数主从执行结果不同；
    - 避坑：生产环境统一用 ROW 格式。
    
3. **坑 3：从库没开启 read_only**
    
    - 后果：从库误写数据，导致主从数据不一致，SQL 线程报错；
    - 避坑：从库开启 `read_only=1`，MySQL 8.0+ 开启 `super_read_only=1`。
    
4. **坑 4：主库没开启双 1 配置**
    
    - 后果：主库宕机时，binlog 没刷盘，数据丢失；
    - 避坑：主库开启 `sync_binlog=1` 和 `innodb_flush_log_at_trx_commit=1`。
    
5. **坑 5：主从库数据不一致就搭建复制**
    
    - 后果：从库 SQL 线程报错，复制中断；
    - 避坑：搭建主从前，必须保证主从数据一致，用主库的备份导入从库。
    
6. **坑 6：只搭建不监控**
    
    - 后果：主从延迟严重、复制中断，业务受影响才发现；
    - 避坑：搭建监控（如 Prometheus+Grafana），监控 `Seconds_Behind_Master`、`Slave_IO_Running`、`Slave_SQL_Running`。

## 十、核心要点总结

1. **原理是根基**：理解三个核心线程、binlog 格式、复制流程，是排查问题的基础；
2. **配置是关键**：server-id 唯一、binlog 开启、ROW 格式、双 1 配置、从库只读，这些是搭建成功的前提；
3. **步骤要严谨**：锁表→记录 binlog 位置→备份→解锁→导入→配置主库→启动复制→验证，一步都不能错；
4. **监控是保障**：监控主从状态、延迟，及时发现问题；
5. **问题要会排**：Slave_IO_Running/Slave_SQL_Running 报错、主从延迟，这些常见问题要会排查；
6. **高级特性要会用**：半同步复制保证数据一致性，GTID 简化主从切换，并行复制提升从库速度；
7. **数据一致性是底线**：双 1 配置、半同步复制、从库只读、定期检查数据一致性，保证主从数据一致。

## 一、开篇：触发器的核心定义与作用

### 1. 什么是触发器

触发器（Trigger）是一种**特殊的存储程序**，它不需要手动调用，而是**基于表的事件自动触发执行**。当表上发生 `INSERT`（插入）、`UPDATE`（更新）、`DELETE`（删除）事件时，MySQL 会自动执行触发器中定义的逻辑。

简单理解：触发器是「表的监听器」，当表上发生指定操作时，自动执行预设的 SQL 逻辑，无需人工干预。

### 2. 触发器的核心特点

- **自动触发**：无需手动调用，事件发生时自动执行；
- **事件驱动**：基于 `INSERT`/`UPDATE`/`DELETE` 事件触发；
- **行级触发**：MySQL 仅支持「行级触发器」（`FOR EACH ROW`），即每一行数据发生变化时都会触发一次；
- **事务关联**：触发器和触发它的 SQL 语句在同一个事务中，任一失败都会回滚。

### 3. 为什么要用触发器

| 优势            | 核心说明                              |
| :------------ | :-------------------------------- |
| **自动维护数据一致性** | 比如订单创建后自动扣减库存、用户注册后自动创建账户记录       |
| **自动记录操作日志**  | 比如记录数据的插入、更新、删除历史，便于审计和追溯         |
| **数据校验**      | 在数据插入 / 更新前，自动校验数据的合法性，不符合规则则拒绝操作 |
| **自动计算衍生字段**  | 比如更新订单金额后，自动计算订单的折扣后金额            |

## 二、前置核心概念（重中之重）

### 2.1 触发三要素

触发器的定义必须包含以下三个要素：

| 要素       | 可选值                            | 说明                                                           |
| :------- | :----------------------------- | :----------------------------------------------------------- |
| **触发时间** | `BEFORE` / `AFTER`             | `BEFORE`：事件发生前触发（如插入前校验数据）；<br><br>`AFTER`：事件发生后触发（如插入后记录日志） |
| **触发事件** | `INSERT` / `UPDATE` / `DELETE` | 触发触发器的表操作类型                                                  |
| **触发频率** | `FOR EACH ROW`                 | MySQL 仅支持行级触发，每一行数据变化都触发一次                                   |

### 2.2 NEW 和 OLD 虚拟表（核心中的核心）

触发器中有两个**临时虚拟表**，用于访问触发事件涉及的数据，是触发器的灵魂：

| 虚拟表     | 触发事件                | 作用                                                 | 可修改性                            |
| :------ | :------------------ | :------------------------------------------------- | :------------------------------ |
| **NEW** | `INSERT` / `UPDATE` | - `INSERT`：表示即将插入的新数据；<br><br>- `UPDATE`：表示更新后的新数据 | `BEFORE` 触发器中可修改，`AFTER` 触发器中只读 |
| **OLD** | `UPDATE` / `DELETE` | - `UPDATE`：表示更新前的旧数据；<br><br>- `DELETE`：表示即将删除的旧数据 | 只读，不可修改                         |

#### 快速记忆

- `INSERT`：只有 `NEW`（新数据）；
- `DELETE`：只有 `OLD`（旧数据）；
- `UPDATE`：既有 `NEW`（新数据），又有 `OLD`（旧数据）。

## 三、创建触发器的核心语法

```sql
DELIMITER //
CREATE TRIGGER 触发器名
触发时间 触发事件
ON 表名
FOR EACH ROW -- MySQL 仅支持行级触发
BEGIN
  -- 触发器的主体逻辑：SQL 语句、变量、流程控制
  -- 可使用 NEW 和 OLD 访问数据
END //
DELIMITER ;
```

### 命名建议

为了便于识别，触发器名建议遵循格式：`trig_表名_触发时间_触发事件`，例如：

- `trig_user_info_before_insert`：用户表插入前触发；
- `trig_order_info_after_update`：订单表更新后触发。

## 四、触发器的实战创建与使用

基于之前的 `user_info`（用户表）、`order_info`（订单表）、`product_stock`（商品库存表）演示常见场景。

### 4.1 案例 1：BEFORE INSERT - 数据自动填充与校验

#### 需求

在用户表插入数据前，自动填充创建时间，并校验年龄必须在 18-100 之间，不符合则拒绝插入。

#### 准备表

```sql
CREATE TABLE IF NOT EXISTS user_info (
    user_id INT PRIMARY KEY AUTO_INCREMENT,
    user_name VARCHAR(50) NOT NULL,
    phone VARCHAR(11) UNIQUE NOT NULL,
    age INT,
    create_time DATETIME,
    update_time DATETIME
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 创建触发器

```sql
DELIMITER //
CREATE TRIGGER trig_user_info_before_insert
BEFORE INSERT
ON user_info
FOR EACH ROW
BEGIN
  -- 1. 自动填充创建时间和更新时间
  SET NEW.create_time = NOW();
  SET NEW.update_time = NOW();
  
  -- 2. 校验年龄：必须在 18-100 之间，不符合则报错
  IF NEW.age < 18 OR NEW.age > 100 THEN
    SIGNAL SQLSTATE '45000' 
    SET MESSAGE_TEXT = '年龄必须在 18-100 之间';
  END IF;
END //
DELIMITER ;
```

#### 测试触发器

```sql
-- 测试1：年龄符合，插入成功
INSERT INTO user_info (user_name, phone, age) VALUES ('张三', '13800138000', 25);
-- 查看数据，create_time 已自动填充
SELECT * FROM user_info;

-- 测试2：年龄不符合（17岁），插入失败，报错
INSERT INTO user_info (user_name, phone, age) VALUES ('李四', '13900139000', 17);
```

#### 关键说明

- `SIGNAL SQLSTATE '45000'`：MySQL 用于主动抛出错误的语法，`45000` 是用户自定义错误的通用状态码；
- `SET MESSAGE_TEXT`：设置错误提示信息。

### 4.2 案例 2：BEFORE UPDATE - 自动更新时间戳

#### 需求

在用户表更新数据时，自动更新 `update_time` 为当前时间，无需手动维护。

#### 创建触发器

```sql
DELIMITER //
CREATE TRIGGER trig_user_info_before_update
BEFORE UPDATE
ON user_info
FOR EACH ROW
BEGIN
  -- 自动更新 update_time 为当前时间
  SET NEW.update_time = NOW();
END //
DELIMITER ;
```

#### 测试触发器

```sql
-- 更新用户数据
UPDATE user_info SET age = 26 WHERE user_id = 1;
-- 查看数据，update_time 已自动更新
SELECT * FROM user_info;
```

### 4.3 案例 3：AFTER INSERT - 自动记录操作日志

#### 需求

在订单表插入数据后，自动将操作记录到日志表中，便于审计。

#### 准备表

```sql
-- 订单表
CREATE TABLE IF NOT EXISTS order_info (
    order_id INT PRIMARY KEY AUTO_INCREMENT,
    order_no VARCHAR(32) UNIQUE NOT NULL,
    user_id INT NOT NULL,
    order_amount DECIMAL(10,2) NOT NULL,
    order_status VARCHAR(20) DEFAULT '待支付',
    create_time DATETIME DEFAULT NOW()
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 操作日志表
CREATE TABLE IF NOT EXISTS operation_log (
    log_id INT PRIMARY KEY AUTO_INCREMENT,
    table_name VARCHAR(50) NOT NULL COMMENT '操作的表名',
    operation_type VARCHAR(20) NOT NULL COMMENT '操作类型：INSERT/UPDATE/DELETE',
    operation_data TEXT COMMENT '操作的数据（JSON格式）',
    operation_time DATETIME DEFAULT NOW() COMMENT '操作时间'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 创建触发器

```sql
DELIMITER //
CREATE TRIGGER trig_order_info_after_insert
AFTER INSERT
ON order_info
FOR EACH ROW
BEGIN
  -- 插入日志：记录新订单的数据
  INSERT INTO operation_log (table_name, operation_type, operation_data)
  VALUES (
    'order_info',
    'INSERT',
    CONCAT(
      '{"order_id":', NEW.order_id,
      ',"order_no":"', NEW.order_no,
      '","user_id":', NEW.user_id,
      ',"order_amount":', NEW.order_amount,
      ',"order_status":"', NEW.order_status,
      '"}'
    )
  );
END //
DELIMITER ;
```

#### 测试触发器

```sql
-- 插入订单
INSERT INTO order_info (order_no, user_id, order_amount) 
VALUES ('ORD20260309001', 1, 99.99);

-- 查看日志表，已自动记录
SELECT * FROM operation_log;
```

### 4.4 案例 4：AFTER INSERT - 订单创建后自动扣减库存

#### 需求

在订单明细表插入数据后，自动扣减对应商品的库存，并校验库存是否充足。

#### 准备表

```sql
-- 商品库存表
CREATE TABLE IF NOT EXISTS product_stock (
    product_id INT PRIMARY KEY,
    product_name VARCHAR(100) NOT NULL,
    stock INT NOT NULL DEFAULT 0 COMMENT '库存数量'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 插入测试库存
INSERT INTO product_stock (product_id, product_name, stock) VALUES 
(1, '商品A', 100),
(2, '商品B', 50);

-- 订单明细表
CREATE TABLE IF NOT EXISTS order_detail (
    detail_id INT PRIMARY KEY AUTO_INCREMENT,
    order_id INT NOT NULL,
    product_id INT NOT NULL,
    product_num INT NOT NULL COMMENT '购买数量',
    product_price DECIMAL(10,2) NOT NULL
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

#### 创建触发器

```sql
DELIMITER //
CREATE TRIGGER trig_order_detail_after_insert
AFTER INSERT
ON order_detail
FOR EACH ROW
BEGIN
  DECLARE current_stock INT DEFAULT 0;
  
  -- 1. 查询当前商品库存
  SELECT stock INTO current_stock FROM product_stock WHERE product_id = NEW.product_id;
  
  -- 2. 校验库存是否充足
  IF current_stock < NEW.product_num THEN
    SIGNAL SQLSTATE '45000'
    SET MESSAGE_TEXT = '库存不足，无法创建订单';
  END IF;
  
  -- 3. 扣减库存
  UPDATE product_stock 
  SET stock = stock - NEW.product_num 
  WHERE product_id = NEW.product_id;
END //
DELIMITER ;
```

#### 测试触发器

```sql
-- 测试1：库存充足，扣减成功
INSERT INTO order_detail (order_id, product_id, product_num, product_price) 
VALUES (1, 1, 2, 99.99);
-- 查看库存，商品A的库存从100变成98
SELECT * FROM product_stock;

-- 测试2：库存不足（商品B只有50，买60），报错
INSERT INTO order_detail (order_id, product_id, product_num, product_price) 
VALUES (1, 2, 60, 59.99);
```

## 五、触发器的查看、修改、删除

### 5.1 查看触发器

```sql
-- 1. 查看当前数据库的所有触发器
SHOW TRIGGERS;

-- 2. 查看指定数据库的触发器
SHOW TRIGGERS FROM 数据库名;

-- 3. 查看触发器的创建语句（核心，可看完整代码）
SHOW CREATE TRIGGER 触发器名;
```

### 5.2 修改触发器

MySQL **不支持直接修改触发器的逻辑**，只能先删除再重新创建。

```sql
-- 先删除
DROP TRIGGER IF EXISTS 触发器名;
-- 再重新创建
DELIMITER //
CREATE TRIGGER 触发器名
触发时间 触发事件
ON 表名
FOR EACH ROW
BEGIN
  -- 新的逻辑
END //
DELIMITER ;
```

### 5.3 删除触发器

```sql
DROP TRIGGER IF EXISTS 触发器名;
```

## 六、触发器的优缺点总结

### 6.1 优点

1. **自动执行**：无需手动调用，事件发生时自动触发，保证逻辑不遗漏；
2. **数据一致性**：自动维护关联数据（如扣库存、更新统计），保证数据一致性；
3. **审计追溯**：自动记录操作日志，便于数据审计和问题追溯；
4. **数据校验**：在数据操作前自动校验，防止非法数据入库。

### 6.2 缺点

1. **性能问题**：触发器会增加数据库的负担，尤其是频繁触发的表，可能导致性能下降；
2. **调试困难**：触发器是自动触发的，出错时难以定位和调试；
3. **维护成本高**：触发器的逻辑隐藏在数据库中，容易被忽略，业务逻辑分散，维护难度大；
4. **死锁风险**：触发器中操作其他表，可能导致死锁；
5. **移植性差**：不同数据库的触发器语法差异大，移植困难。

## 七、常见坑与避坑指南

### 7.1 NEW 和 OLD 的使用错误

- **坑**：在 `INSERT` 触发器中用 `OLD`，或在 `DELETE` 触发器中用 `NEW`，报错 `Unknown column 'OLD' in 'NEW/OLD table'`。
- **避坑**：
    
    - `INSERT`：只用 `NEW`；
    - `DELETE`：只用 `OLD`；
    - `UPDATE`：同时用 `NEW`（新数据）和 `OLD`（旧数据）。
    

---

### 7.2 在触发器中修改本表导致无限循环

- **坑**：在 `AFTER UPDATE` 触发器中又 `UPDATE` 本表，导致触发器再次触发，无限循环，最终报错 `Recursive limit exceeded`。
- **避坑**：
    
    - 如果要修改本表的数据，用 `BEFORE` 触发器，直接修改 `NEW` 的值，不要用 `UPDATE` 语句；
    - 避免在 `AFTER` 触发器中修改本表。
    

---

### 7.3 触发器中做复杂操作导致性能问题

- **坑**：在触发器中写复杂的多表关联、子查询、调用存储过程，导致触发操作极慢。
- **避坑**：
    
    - 触发器的逻辑要尽量简单，只做核心的、必须自动执行的操作；
    - 复杂逻辑建议放在应用代码或存储过程中，不要放在触发器中。
    

---

### 7.4 事务问题

- **坑**：触发器和触发语句在同一个事务中，触发器中的错误会导致触发语句回滚，但容易被忽略。
- **避坑**：
    
    - 明确触发器和触发语句在同一事务中，任一失败都会全部回滚；
    - 用 `SIGNAL` 主动抛出错误时，会触发事务回滚。
    

---

### 7.5 权限问题

- **坑**：创建触发器时报错 `Access denied; you need (at least one of) the TRIGGER privilege(s)`。
- **避坑**：确保当前用户有 `TRIGGER` 权限（创建 / 删除）、`ALTER` 权限（修改表）、`EXECUTE` 权限（执行）。

---

### 7.6 忽略 FOR EACH ROW

- **坑**：创建触发器时忘记写 `FOR EACH ROW`，报错 `You have an error in your SQL syntax; check the manual that corresponds to your MySQL server version`。
- **避坑**：MySQL 仅支持行级触发器，`FOR EACH ROW` 必须写，不能省略。

## 八、综合实战案例：电商订单全流程触发器

整合前面的案例，实现订单创建的完整触发器体系：

1. 订单表插入前自动生成订单号；
2. 订单明细表插入后自动扣减库存；
3. 订单表更新后自动记录日志；
4. 库存不足时主动报错回滚。

### 8.1 创建订单表插入前触发器（自动生成订单号）

```sql
DELIMITER //
CREATE TRIGGER trig_order_info_before_insert
BEFORE INSERT
ON order_info
FOR EACH ROW
BEGIN
  -- 自动生成订单号：ORD + 年月日时分秒 + 3位随机数
  SET NEW.order_no = CONCAT('ORD', DATE_FORMAT(NOW(), '%Y%m%d%H%i%s'), LPAD(FLOOR(RAND()*1000), 3, '0'));
  SET NEW.create_time = NOW();
END //
DELIMITER ;
```

### 8.2 测试完整流程

```sql
-- 1. 插入订单（无需手动写订单号）
INSERT INTO order_info (user_id, order_amount, order_status) 
VALUES (1, 199.98, '待支付');

-- 2. 查看订单，订单号已自动生成
SELECT * FROM order_info;

-- 3. 插入订单明细，自动扣减库存
INSERT INTO order_detail (order_id, product_id, product_num, product_price) 
VALUES (LAST_INSERT_ID(), 1, 2, 99.99);

-- 4. 查看库存，已扣减
SELECT * FROM product_stock;

-- 5. 更新订单状态，自动记录日志
UPDATE order_info SET order_status = '已支付' WHERE order_id = LAST_INSERT_ID();

-- 6. 查看日志
SELECT * FROM operation_log;
```

## 九、核心要点总结

1. **触发器的本质**：事件驱动的自动存储程序，基于表的 `INSERT`/`UPDATE`/`DELETE` 自动触发。
2. **触发三要素**：触发时间（`BEFORE`/`AFTER`）、触发事件（`INSERT`/`UPDATE`/`DELETE`）、触发频率（`FOR EACH ROW`，必须写）。
3. **NEW 和 OLD 虚拟表**：
    
    - `INSERT`：只有 `NEW`；
    - `DELETE`：只有 `OLD`；
    - `UPDATE`：`NEW`（新）+ `OLD`（旧）。
    
4. **主动抛错**：用 `SIGNAL SQLSTATE '45000' SET MESSAGE_TEXT = '错误信息'`。
5. **优缺点**：优点是自动、一致、审计；缺点是性能、调试、维护。
6. **避坑核心**：
    
    - 正确使用 NEW 和 OLD；
    - 避免在触发器中修改本表导致无限循环；
    - 触发器逻辑尽量简单，不要做复杂操作；
    - 明确触发器和触发语句在同一事务中。
    
7. **选型建议**：
    - 必须自动执行、保证数据一致性的场景（如扣库存、自动填充时间）：用触发器；
    - 复杂业务逻辑、性能敏感的场景：优先用应用代码或存储过程，不要滥用触发器。

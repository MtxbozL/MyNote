## 一、开篇：联合查询的核心定义与作用

### 1. 什么是联合查询

联合查询（UNION）是指**将多个 SELECT 查询的结果集，纵向拼接成一个完整的结果集**的 SQL 语法，核心是「合并同类数据」。

简单对比：

- **连接查询（JOIN）**：横向拼接表，从多张表中取字段，结果集的列数变多；
- **联合查询（UNION）**：纵向拼接结果集，合并多个查询的行，结果集的行数变多。

### 2. 为什么需要联合查询

很多场景需要合并「结构相同但来源不同」的数据，比如：

- 场景 1：合并「用户表」和「管理员表」的人员信息（两张表结构相同，分别存普通用户和管理员）；
- 场景 2：合并「今年的订单表」和「去年的订单表」的历史数据（按年份分表，需合并查询）；
- 场景 3：合并「待支付订单」和「已支付订单」的查询结果（不同条件的同类数据，需统一展示）。

---

## 二、核心语法与基础要求

### 2.1 核心语法

```sql
-- 语法1：UNION（自动去重）
SELECT 字段1, 字段2, ... FROM 表1 [WHERE 条件]
UNION
SELECT 字段1, 字段2, ... FROM 表2 [WHERE 条件];

-- 语法2：UNION ALL（不去重，保留所有数据）
SELECT 字段1, 字段2, ... FROM 表1 [WHERE 条件]
UNION ALL
SELECT 字段1, 字段2, ... FROM 表2 [WHERE 条件];
```

### 2.2 联合查询的 3 个基础要求（必须满足）

这是新手最容易踩的坑，必须严格遵守：

1. **字段数量必须一致**：每个 SELECT 查询的字段数量必须完全相同；
2. **字段类型必须兼容**：对应位置的字段数据类型必须兼容（如 INT 和 DECIMAL 兼容，INT 和 VARCHAR 不兼容）；
3. **列名取第一个查询的**：最终结果集的列名，由第一个 SELECT 查询的列名决定。

---

## 三、UNION vs UNION ALL：核心区别（重中之重）

这是联合查询最核心的知识点，90% 的场景都要在这两者之间做选择，核心区别如下：

| 对比维度     | UNION                    | UNION ALL                                                                   |
| :------- | :----------------------- | :-------------------------------------------------------------------------- |
| **去重规则** | 自动对合并后的结果集**去重**，只保留唯一的行 | **不去重**，保留所有查询的所有行（包括重复的）                                                   |
| **执行效率** | 低（需要额外做去重操作，数据量大时性能差）    | 高（直接合并，无需去重，性能通常比 UNION 快 10-100 倍）                                         |
| **适用场景** | 需要合并数据且必须去重的场景           | 1. 确定数据不会重复的场景；<br><br>2. 允许重复数据的场景；<br><br>3. 性能优先的场景（绝大多数场景优先选 UNION ALL） |

### 示例对比

假设有两张表，分别存普通用户和管理员：

```sql
-- 普通用户表
CREATE TABLE user_normal (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    type VARCHAR(20) DEFAULT '普通用户'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 管理员表
CREATE TABLE user_admin (
    id INT PRIMARY KEY AUTO_INCREMENT,
    name VARCHAR(50) NOT NULL,
    type VARCHAR(20) DEFAULT '管理员'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 插入测试数据
INSERT INTO user_normal (name) VALUES ('张三'), ('李四'), ('王五');
INSERT INTO user_admin (name) VALUES ('王五'), ('赵六');
```

#### UNION 示例（自动去重）

```sql
SELECT id, name, type FROM user_normal
UNION
SELECT id, name, type FROM user_admin;
```

> 结果：「王五」只出现一次，因为 UNION 自动去重了。

#### UNION ALL 示例（不去重）

```sql
SELECT id, name, type FROM user_normal
UNION ALL
SELECT id, name, type FROM user_admin;
```

> 结果：「王五」出现两次，保留了所有数据，性能更高。

## 四、联合查询的进阶用法

### 4.1 联合查询 + 排序（ORDER BY）

如果需要对**合并后的最终结果集**排序，必须给每个 SELECT 查询加括号，然后在最后统一加 ORDER BY。

#### 错误示例（语法报错）

```sql
-- 错误：ORDER BY 只对第二个查询生效，且语法不规范
SELECT id, name, type FROM user_normal
UNION ALL
SELECT id, name, type FROM user_admin
ORDER BY id DESC;
```

#### 正确示例（统一排序）

```sql
-- 正确：给每个查询加括号，最后统一排序
(SELECT id, name, type FROM user_normal)
UNION ALL
(SELECT id, name, type FROM user_admin)
ORDER BY id DESC;
```

#### 进阶：对单个查询排序（极少用）

如果需要对**每个查询的结果先排序，再合并**，需给单个查询加括号和 ORDER BY + LIMIT（不加 LIMIT 的话，单个查询的 ORDER BY 会被优化器忽略）。

```sql
(SELECT id, name, type FROM user_normal ORDER BY id DESC LIMIT 10)
UNION ALL
(SELECT id, name, type FROM user_admin ORDER BY id DESC LIMIT 10);
```

---

### 4.2 联合查询 + 分页（LIMIT）

和排序类似，对**最终结果集分页**，需在最后统一加 LIMIT。

```sql
(SELECT id, name, type FROM user_normal)
UNION ALL
(SELECT id, name, type FROM user_admin)
ORDER BY id DESC
LIMIT 0, 5; -- 第1页，每页5条
```

---

### 4.3 联合查询 + 临时字段

如果两张表的字段不完全一致，可以通过「临时字段」补全，满足联合查询的字段数量要求。

```sql
-- 假设 user_admin 表没有 type 字段，用临时字段补全
SELECT id, name, type FROM user_normal
UNION ALL
SELECT id, name, '管理员' AS type FROM user_admin; -- 临时字段补全
```

## 五、联合查询的常见坑与避坑指南

### 1. 字段数量不一致

#### 错误示例

```sql
-- 错误：第一个查询3个字段，第二个查询2个字段
SELECT id, name, type FROM user_normal
UNION ALL
SELECT id, name FROM user_admin;
```

#### 解决方法

- 确保每个 SELECT 查询的字段数量完全相同；
- 用临时字段补全缺失的字段（如 `'默认值' AS 字段名`）。

---

### 2. 字段类型不兼容

#### 错误示例

```sql
-- 错误：第一个查询的 id 是 INT，第二个查询的 id 是 VARCHAR
SELECT id, name FROM user_normal -- id 是 INT
UNION ALL
SELECT 'A001' AS id, name FROM user_admin; -- id 是 VARCHAR
```

#### 解决方法

- 确保对应位置的字段类型兼容；
- 用 `CAST()` 或 `CONVERT()` 函数转换类型：

```sql
    SELECT CAST(id AS CHAR) AS id, name FROM user_normal -- 把 INT 转成 CHAR
    UNION ALL
    SELECT 'A001' AS id, name FROM user_admin;
```

---

### 3. 滥用 UNION 代替 UNION ALL

#### 问题

很多新手不管场景都用 UNION，导致性能极差，尤其是数据量大时。

#### 解决方法

- **90% 的场景优先用 UNION ALL**，只有明确需要去重时才用 UNION；
- 如果确定数据不会重复（比如分表查询，不同表的数据不会重叠），必须用 UNION ALL。

---

### 4. 列名取第一个查询的

#### 示例

```sql
-- 第一个查询的列名是 id, name, type
SELECT id, name, type FROM user_normal
UNION ALL
-- 第二个查询的列名是 user_id, user_name, user_type，但最终结果集的列名取第一个的
SELECT id AS user_id, name AS user_name, type AS user_type FROM user_admin;
```

#### 注意

最终结果集的列名由第一个 SELECT 查询决定，后续查询的列别名会被忽略。

## 六、综合实操演练

基于之前的 `user_normal`、`user_admin` 和 `order_info` 表，完成以下需求：

1. 合并普通用户和管理员的信息，保留所有数据，按 id 降序排列；
2. 合并「待支付订单」和「已支付订单」的信息，去重后按订单金额降序排列；
3. 合并今年和去年的订单表（假设分表为 `order_2025` 和 `order_2026`），查询总订单数。

### 参考答案

```sql
-- 1. 合并普通用户和管理员，保留所有数据，按id降序
(SELECT id, name, type FROM user_normal)
UNION ALL
(SELECT id, name, type FROM user_admin)
ORDER BY id DESC;

-- 2. 合并待支付和已支付订单，去重后按金额降序
(SELECT * FROM order_info WHERE order_status = '待支付')
UNION
(SELECT * FROM order_info WHERE order_status = '已支付')
ORDER BY order_amount DESC;

-- 3. 合并分表订单，查询总订单数
SELECT COUNT(*) AS '总订单数' FROM (
  SELECT order_id FROM order_2025
  UNION ALL
  SELECT order_id FROM order_2026
) temp;
```

## 七、核心要点总结

1. 联合查询的核心是**纵向拼接多个 SELECT 的结果集**，和连接查询（横向拼接）完全不同；
2. 3 个基础要求：**字段数量一致、类型兼容、列名取第一个查询的**；
3. UNION vs UNION ALL：
    
    - UNION：自动去重，性能差；
    - UNION ALL：不去重，性能高，**90% 的场景优先选 UNION ALL**；
    
4. 排序和分页：需给每个查询加括号，最后统一加 ORDER BY 和 LIMIT；
5. 常见坑：字段数量 / 类型不一致、滥用 UNION、忽略列名规则。

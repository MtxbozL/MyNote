# 一、开篇：MVCC 的核心定位与价值

MVCC（Multi-Version Concurrency Control，多版本并发控制），是 InnoDB 存储引擎实现「读写不阻塞、并发性能优化」的核心机制，也是事务隔离性（Isolation）和一致性（Consistency）的底层支撑。

简单理解：MVCC 给数据库中的每一行数据，维护了多个版本的历史数据，不同事务在访问数据时，会根据自身的隔离级别，读取到对应版本的数据——**读操作无需加锁，写操作仅锁定当前版本**，从而实现「读写并行」，彻底解决了传统锁机制中「读写互斥」的性能瓶颈。

## 核心认知（必记）

1. MVCC 是 **InnoDB 独有的机制**，MyISAM、Memory 等引擎不支持（因不支持事务和行锁）；
    
2. MVCC 的核心目标：**在保证数据一致性的前提下，最大化提升并发读写性能**；
    
3. MVCC 仅对「快照读」生效，对「当前读」不生效（当前读仍需加锁）；
    
4. MVCC 依赖 undo log（回滚日志）和 Read View（读视图）实现，与之前学习的三大日志、锁机制深度关联。

## MVCC 解决的核心问题

在 MVCC 出现之前，数据库解决并发读写的方式是「加锁」：读加共享锁，写加排他锁，导致「读写互斥」——读操作会阻塞写操作，写操作也会阻塞读操作，并发性能极差。

MVCC 彻底解决了这个问题：

- 读操作（快照读）：无需加锁，直接读取数据的历史版本，不阻塞写操作；
    
- 写操作（当前读）：仅锁定当前数据版本，不阻塞其他读操作（其他读操作读取历史版本）；
    
- 最终实现「读读共享、读写不互斥、写写互斥」，大幅提升高并发场景的性能。

# 二、前置基础：快照读与当前读（核心区分）

MVCC 的核心是「区分快照读和当前读」，两种读取方式的行为、触发场景完全不同，这是理解 MVCC 的关键前提。

## 2.1 快照读（Snapshot Read）

又称「一致性读」，是 MVCC 生效的读取方式，读取的是数据的「历史版本」，无需加锁，不阻塞任何操作。

### 触发场景（默认触发）

普通的 SELECT 语句（不包含 FOR UPDATE、LOCK IN SHARE MODE）：

```sql
-- 快照读（MVCC 生效）
SELECT * FROM user WHERE id = 1;
SELECT name FROM user WHERE age > 20;
```

## 2.2 当前读（Current Read）

又称「锁定读」，读取的是数据的「最新版本」，会加锁（共享锁/排他锁），阻塞其他写操作，不依赖 MVCC。

### 触发场景

```sql
-- 1. 加排他锁（写锁），触发当前读
SELECT * FROM user WHERE id = 1 FOR UPDATE;
UPDATE user SET name = '张三' WHERE id = 1;
DELETE FROM user WHERE id = 1;
INSERT INTO user (id, name) VALUES (1, '张三');

-- 2. 加共享锁（读锁），触发当前读
SELECT * FROM user WHERE id = 1 LOCK IN SHARE MODE;
```

## 2.3 快照读与当前读核心对比

|对比维度|快照读|当前读|
|---|---|---|
|读取版本|数据的历史版本|数据的最新版本|
|是否加锁|不加锁（无锁读）|加锁（共享锁/排他锁）|
|是否阻塞|不阻塞任何操作（读/写）|阻塞其他写操作（排他锁）/ 写操作（共享锁）|
|是否依赖 MVCC|是（核心依赖）|否（依赖锁机制）|
|触发场景|普通 SELECT 语句|SELECT ... FOR UPDATE / LOCK IN SHARE MODE、UPDATE、DELETE、INSERT|

# 三、MVCC 底层实现原理（核心考点）

MVCC 的实现依赖三大核心组件，三者协同工作，完成历史版本的生成、读取和管理：

**核心组件**：隐藏列 + undo log（回滚日志） + Read View（读视图）

## 3.1 组件1：InnoDB 的隐藏列（版本标识基础）

InnoDB 会给表中的每一行数据，自动添加 3 个隐藏列（无需手动定义），用于标记数据的版本和事务信息，是 MVCC 版本管理的基础：

1. **DB_TRX_ID（6字节）**：当前行数据的「最后修改事务ID」，记录修改该行的事务唯一标识；
    
    1. 插入一行数据时，记录当前事务ID；
        
    2. 更新一行数据时，生成新的版本，记录当前事务ID；
        
    3. 删除一行数据时，视为「特殊更新」，记录当前事务ID，同时标记为删除。
        
2. **DB_ROLL_PTR（7字节）**：「回滚指针」，指向当前行数据的上一个历史版本（存储在 undo log 中），通过该指针，可串联起一行数据的所有历史版本，形成「版本链」。
    
3. **DB_ROW_ID（6字节）**：行唯一标识，当表没有主键或唯一索引时，InnoDB 会自动生成该列，用于唯一标识行数据（有主键时可忽略）。
    

### 示例：版本链的生成过程

假设存在事务 ID 为 100、200、300 的三个事务，依次修改同一行数据，版本链的生成过程如下：

1. 事务100 插入数据：id=1，name=「李四」，此时 DB_TRX_ID=100，DB_ROLL_PTR=NULL（无历史版本）；
    
2. 事务200 更新数据：将 name 改为「张三」，生成新的版本，DB_TRX_ID=200，DB_ROLL_PTR 指向事务100生成的版本；
    
3. 事务300 更新数据：将 name 改为「王五」，生成新的版本，DB_TRX_ID=300，DB_ROLL_PTR 指向事务200生成的版本；
    
4. 最终形成版本链：当前版本（300）→ 历史版本1（200）→ 历史版本2（100），每个版本都通过回滚指针关联。

## 3.2 组件2：undo log（回滚日志）—— 历史版本存储载体

之前学习过，undo log 的核心作用是「事务回滚」，而在 MVCC 中，它的另一个核心作用是「存储数据的历史版本」，是版本链的物理存储载体。

### MVCC 中 undo log 的分类与作用

undo log 按操作类型分为两类，均用于存储历史版本：

1. **insert undo log（插入回滚日志）**：
    
    1. 触发场景：INSERT 操作生成；
        
    2. 特点：事务提交后，该日志可直接被 purge 线程回收（因为插入的行，其他事务的快照读不会用到它的历史版本）；
        
    3. 作用：仅用于事务回滚，不参与 MVCC 版本链。
        
2. **update undo log（更新回滚日志）**：
    
    1. 触发场景：UPDATE、DELETE 操作生成；
        
    2. 特点：事务提交后，不会立即回收，需等待所有依赖该历史版本的事务结束后，才会被 purge 线程回收；
        
    3. 作用：既用于事务回滚，也用于 MVCC 版本链，存储数据的历史版本，供快照读读取。

### 核心关联

行数据的 DB_ROLL_PTR 回滚指针，指向的就是 undo log 中对应的历史版本——也就是说，版本链的物理存储在 undo log 中，逻辑上通过回滚指针串联。

## 3.3 组件3：Read View（读视图）—— 版本筛选规则

Read View（读视图）是 MVCC 的「核心筛选器」，本质是一个「事务可见性规则集合」。当事务执行快照读时，InnoDB 会生成一个 Read View，用于判断当前事务能看到哪些版本的数据（哪些版本对当前事务可见）。

### Read View 的核心组成（4个关键参数）

每个 Read View 包含 4 个核心参数，用于判断版本的可见性：

1. **m_ids**：当前活跃的所有事务 ID 集合（正在执行，未提交的事务）；
    
2. **min_trx_id**：m_ids 集合中最小的事务 ID（当前活跃事务中，最早开始的事务）；
    
3. **max_trx_id**：当前系统中，下一个将要分配的事务 ID（不是 m_ids 中的最大值，是全局自增的下一个 ID）；
    
4. **creator_trx_id**：生成当前 Read View 的事务 ID（当前执行快照读的事务 ID）。

### Read View 的可见性判断规则（核心）

对于版本链中的某一个历史版本（其 DB_TRX_ID = trx_id），通过以下规则判断是否对当前事务可见：

1. 如果 trx_id = creator_trx_id：当前版本是当前事务自己修改的，可见；
    
2. 如果 trx_id < min_trx_id：修改该版本的事务，在当前 Read View 生成前已经提交，可见；
    
3. 如果 trx_id > max_trx_id：修改该版本的事务，在当前 Read View 生成后才开始执行，不可见；
    
4. 如果 min_trx_id ≤ trx_id ≤ max_trx_id：
    
    1. 若 trx_id 在 m_ids 中（事务仍活跃，未提交）：不可见；
        
    2. 若 trx_id 不在 m_ids 中（事务已提交）：可见。

总结：**只有「已提交事务」修改的版本，才可能对当前事务可见；未提交事务修改的版本，永远不可见**（这也是 MVCC 解决脏读的核心原因）。

### Read View 的生成时机（面试高频）

Read View 的生成时机，直接决定了不同隔离级别下的读取一致性，这是 RC 和 RR 隔离级别最核心的区别之一：

1. **读已提交（RC）**：**每次执行快照读时，都会生成一个新的 Read View**；
    
    1. 特点：每次查询，都会重新获取当前活跃事务集合，因此可能出现「不可重复读」（同一事务内，两次查询同一数据，结果不一致）；
        
    2. 原因：两次快照读生成了不同的 Read View，可能读取到不同版本的数据。
        
2. **可重复读（RR）**：**事务内第一次执行快照读时，生成一个 Read View，后续所有快照读都复用这个 Read View**；
    
    1. 特点：同一事务内，所有快照读都使用同一个筛选规则，因此能保证「可重复读」（两次查询同一数据，结果一致）；
        
    2. 原因：复用 Read View，不会因为其他事务提交而改变可见性规则。

注意：串行化隔离级别不使用 MVCC，读操作会加共享锁；读未提交隔离级别也不使用 MVCC，直接读取最新版本（会出现脏读）。

# 四、MVCC 完整工作流程（实操理解）

结合三大组件，以「可重复读（RR）」隔离级别为例，完整梳理 MVCC 的工作流程（快照读场景）：

## 前提条件

- 表：user（id INT PRIMARY KEY，name VARCHAR(50)）；
    
- 初始数据：id=1，name=「李四」，DB_TRX_ID=100，DB_ROLL_PTR=NULL；
    
- 事务A（ID=200）：执行更新操作，未提交；
    
- 事务B（ID=300）：执行快照读，查看 id=1 的数据。

## 完整流程

1. **步骤1：事务A执行更新，生成历史版本**
    
    1. 事务A（ID=200）执行：UPDATE user SET name = 「张三」 WHERE id = 1；
        
    2. InnoDB 生成新的行版本：name=「张三」，DB_TRX_ID=200，DB_ROLL_PTR 指向原版本（DB_TRX_ID=100）；
        
    3. 原版本（name=「李四」）被写入 update undo log 中，形成版本链：新版本（200）→ 原版本（100）；
        
    4. 事务A未提交，因此事务A（ID=200）属于活跃事务。
        
2. **步骤2：事务B执行快照读，生成 Read View**
    
    1. 事务B（ID=300）执行：SELECT * FROM user WHERE id = 1（快照读）；
        
    2. 因是事务B内第一次快照读（RR级别），生成 Read View：
        
        - m_ids = {200}（当前活跃事务只有A）；
            
        - min_trx_id = 200；
            
        - max_trx_id = 301（下一个将要分配的事务ID）；
            
        - creator_trx_id = 300（当前事务B的ID）。

3. **步骤3：通过 Read View 筛选版本链**
    
    1. 先读取版本链的最新版本（DB_TRX_ID=200），判断可见性：
        
        - trx_id=200 ≠ creator_trx_id（300）；
            
        - 200 ≥ min_trx_id（200）且 ≤ max_trx_id（301）；
            
        - trx_id=200 在 m_ids 中（事务A活跃，未提交）→ 不可见。

    2. 通过 DB_ROLL_PTR 回滚指针，读取上一个历史版本（DB_TRX_ID=100），判断可见性：
        
        - trx_id=100 ≠ creator_trx_id（300）；
            
        - 100 < min_trx_id（200）→ 事务100在 Read View 生成前已提交 → 可见。
            
4. **步骤4：返回可见版本**
    
    1. 事务B读取到可见的历史版本（name=「李四」），返回给客户端；
        
    2. 即使后续事务A提交，事务B再次执行快照读（RR级别），仍复用之前的 Read View，还是读取到 name=「李四」，实现「可重复读」。
        

# 五、MVCC 与事务隔离级别的关联（核心重点）

MVCC 仅在「读已提交（RC）」和「可重复读（RR）」两个隔离级别中生效，不同隔离级别下，MVCC 的行为差异，决定了隔离级别解决的并发问题不同。

## 5.1 各隔离级别下 MVCC 的行为

|隔离级别|是否使用 MVCC|Read View 生成时机|核心行为|解决的并发问题|
|---|---|---|---|---|
|读未提交（RU）|否|不生成|直接读取数据最新版本，不加锁|无（脏读、不可重复读、幻读都存在）|
|读已提交（RC）|是|每次快照读都生成新的 Read View|只能看到已提交事务的版本，避免脏读；但同一事务内多次快照读可能看到不同版本（不可重复读）|解决脏读，存在不可重复读、幻读|
|可重复读（RR）|是|事务内第一次快照读生成，后续复用|同一事务内多次快照读看到同一版本（可重复读）；配合临键锁解决幻读|解决脏读、不可重复读、幻读（InnoDB 中）|
|串行化（Serializable）|否|不生成|读加共享锁，写加排他锁，事务完全串行执行|解决所有并发问题，性能极差|

## 5.2 RC vs RR 级别 MVCC 核心区别（面试必背）

|对比维度|读已提交（RC）|可重复读（RR）|
|---|---|---|
|Read View 生成时机|每次快照读都生成新的|事务内第一次快照读生成，后续复用|
|可重复读|不支持（同一事务内两次查询结果可能不同）|支持（同一事务内两次查询结果一致）|
|幻读（快照读）|存在（可能读取到其他事务提交的新增数据）|不存在（复用 Read View，看不到新增数据）|
|undo log 回收时机|较快（每次快照读生成新 Read View，旧版本更快不被依赖）|较慢（复用 Read View，旧版本可能被长期依赖，易导致 undo log 膨胀）|
|并发性能|较高（锁范围小，undo log 回收快）|中等（undo log 可能膨胀，间隙锁增加锁冲突）|

# 六、MVCC 常见问题与生产避坑

## 6.1 问题1：undo log 膨胀（高频坑）

### 坑点

RR 级别下，长事务会持有 Read View，导致 undo log 中的历史版本长期无法被 purge 线程回收，进而导致 undo log 文件无限膨胀，占用大量磁盘空间。

### 原因

长事务的 Read View 会一直依赖旧的历史版本，purge 线程无法判断这些版本是否还会被使用，因此不会回收，导致 undo log 不断累积。

### 解决方法

- 严禁长事务，将大事务拆分为小事务，执行完立即提交/回滚；
    
- 开启 InnoDB 独立 undo 表空间（innodb_undo_tablespaces > 0），并开启自动截断（innodb_undo_log_truncate = ON）；
    
- 定期监控 undo log 大小，超过阈值时，手动触发 purge 回收；
    
- 高并发场景可考虑将隔离级别改为 RC，加快 undo log 回收。

## 6.2 问题2：MVCC 无法解决幻读？（认知误区）

### 误区

很多人认为 MVCC 只能解决脏读和不可重复读，无法解决幻读，但这是不全面的——MVCC 仅解决「快照读」的幻读，「当前读」的幻读需要靠锁机制（临键锁）解决。

### 正确认知

- 快照读（普通 SELECT）：RR 级别下，MVCC 通过复用 Read View，避免读取到其他事务提交的新增数据，解决幻读；
    
- 当前读（SELECT ... FOR UPDATE 等）：MVCC 不生效，InnoDB 通过「临键锁」锁定范围，避免其他事务插入数据，解决幻读；
    
- 综上：InnoDB 在 RR 级别下，通过「MVCC（快照读）+ 临键锁（当前读）」，彻底解决了幻读。

## 6.3 问题3：快照读读取的是「最新已提交版本」吗？

不一定，分隔离级别：

- RC 级别：是，每次快照读生成新的 Read View，能读取到所有已提交事务的最新版本；
    
- RR 级别：不是，复用 Read View，只能读取到「事务内第一次快照读时，已提交的版本」，后续其他事务提交的版本，无法读取到（保证可重复读）。

## 6.4 问题4：MVCC 与锁机制的关系

MVCC 和锁机制是 InnoDB 解决并发问题的「两大核心手段」，二者协同工作，互不冲突：

- MVCC 负责「快照读」：无锁，提升并发性能，解决脏读、不可重复读（RR 下还解决快照读幻读）；
    
- 锁机制负责「当前读」：加锁，保证数据一致性，解决写写冲突、当前读幻读（临键锁）；
    
- 总结：MVCC 优化读性能，锁机制保证写一致性，共同实现「高并发 + 强一致」。

# 八、核心要点总结

1. MVCC 是 InnoDB 独有的多版本并发控制机制，核心目标是「读写不阻塞，提升并发性能」，仅对快照读生效。
    
2. MVCC 底层依赖三大组件：隐藏列（版本标识）、undo log（历史版本存储）、Read View（版本筛选）。
    
3. 版本链由行数据的 DB_ROLL_PTR 串联，物理存储在 undo log 中，是历史版本的载体。
    
4. Read View 的生成时机，决定了 RC 和 RR 级别的核心差异：RC 每次快照读生成新的，RR 事务内复用一个。
    
5. MVCC 解决了脏读、不可重复读，RR 级别下结合临键锁，彻底解决幻读。
    
6. 生产环境核心避坑点：避免长事务导致 undo log 膨胀，根据并发需求选择 RC/RR 隔离级别。
    
7. MVCC 与锁机制协同工作：MVCC 优化读性能，锁机制保证写一致性，共同支撑 InnoDB 的高并发能力。

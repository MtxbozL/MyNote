# 第六章 Rewrite 重写与路由控制

## 04 Return 指令

### 1. 官方定义与核心定位

`return` 指令隶属于 **ngx_http_rewrite_module** 标准 HTTP 模块，无需额外编译即可启用，是 Nginx 重写体系中唯一的**请求终止类核心指令**。其核心能力是立即终止当前请求的全流程处理，直接向客户端返回标准化 HTTP 响应，无任何后续流程分支，具备零歧义、高性能、无循环风险的特性，是 Nginx 实现请求拦截、快速跳转、异常响应的首选方案。

与`rewrite`指令的核心定位差异：`rewrite`的核心是 URI 重写与流程调度，允许请求继续后续处理；`return`的核心是请求终止与直接响应，执行后无任何后续处理流程。

### 2. 标准化语法与合法作用域

#### 2.1 官方完整语法

Nginx 官方定义了 3 种合法语法形式，覆盖所有使用场景：

nginx

```
# 形式1：仅返回HTTP状态码
return code;
# 形式2：返回HTTP状态码 + 响应体文本
return code [text];
# 形式3：返回HTTP状态码 + 重定向URL / 仅返回重定向URL
return code [URL];
return URL;
```

#### 2.2 核心约束

- **默认配置**：无
- **合法作用域**：`server`、`location`、`if`，禁止在`main`、`events`、`http`全局块中配置
- **执行特性**：指令一旦匹配执行，立即终止当前作用域内所有后续指令（含`rewrite`、`set`、`if`等所有 rewrite 模块指令），同时跳过请求处理的所有后续阶段，直接进入响应发送阶段。

### 3. 语法分支深度解析

#### 3.1 仅返回 HTTP 状态码

##### 3.1.1 执行规则

仅向客户端返回指定的 HTTP 状态码，响应体使用 Nginx 对应状态码的默认页面，无自定义内容。

- **合法状态码范围**：支持所有符合 HTTP/1.1 规范的状态码，以及 Nginx 自定义状态码，核心合法值包括：
    
    1. 标准 HTTP 状态码：200、400、401、403、404、405、500、502、503、504 等
    2. 重定向状态码：301、302、303、304、307、308
    3. Nginx 自定义状态码：444（特殊状态码，直接关闭 TCP 连接，不返回任何响应头与响应体）
    
- **非法状态码约束**：禁止使用非规范状态码，配置后会触发`nginx -t`语法校验失败。

##### 3.1.2 标准示例

nginx

```
# 拦截非法请求方法，返回405状态码
if ($request_method !~ ^(GET|POST|HEAD)$) {
    return 405;
}

# 封禁恶意IP，返回403禁止访问
location /admin/ {
    allow 192.168.1.0/24;
    deny all;
    return 403;
}

# 特殊场景：直接关闭TCP连接，无任何响应，用于CC攻击防护
if ($http_user_agent ~* "malicious-scanner") {
    return 444;
}
```

#### 3.2 返回状态码 + 响应体文本

##### 3.2.1 执行规则

向客户端返回指定状态码与自定义响应体文本，适用于自定义错误提示、接口响应、健康检查等场景。

- **核心特性**：
    
    1. 响应体文本支持纯文本、HTML、JSON 等格式，默认`Content-Type`为`text/plain`，可通过`add_header`自定义响应头；
    2. 文本包含空格、换行、特殊字符时，必须用双引号包裹；
    3. 支持引用 Nginx 内置变量与正则捕获组变量，实现动态响应；
    4. 仅支持非重定向类状态码，重定向状态码使用该形式会导致客户端无法正常跳转。
    

##### 3.2.2 标准示例

nginx

```
# 健康检查接口，返回200状态码与JSON格式响应
location /health {
    return 200 '{"status":"ok","code":200,"timestamp":"$time_iso8601"}';
    add_header Content-Type application/json;
}

# 自定义404错误提示，返回HTML内容
location / {
    root /var/www/html;
    error_page 404 @custom_404;
}

location @custom_404 {
    return 404 "<html><body><h1>页面不存在</h1><p>请求URI：$request_uri</p></body></html>";
    add_header Content-Type text/html;
}
```

#### 3.3 返回状态码 + 重定向 URL

##### 3.3.1 执行规则

向客户端返回重定向状态码与`Location`响应头，强制客户端跳转到指定 URL，是 Nginx 实现 URL 跳转的官方推荐方案，性能优于`rewrite`重定向。

- **核心规则**：
    
    1. 仅支持重定向类状态码：301、302、303、307、308，其中 301/302 为最常用场景；
    2. 简写形式`return URL;` 等价于`return 302 URL;`，默认返回 302 临时重定向；
    3. URL 支持完整协议地址（`http://`/`https://`开头），也支持相对路径，相对路径会基于当前`$host`自动补全完整地址；
    4. 支持引用 Nginx 内置变量（`$scheme`、`$host`、`$uri`、`$query_string`等），实现动态跳转，完整保留原始请求的路径与参数；
    5. 执行后立即终止请求，无任何循环匹配风险，行为完全可预测。
    

##### 3.3.2 标准示例

nginx

```
# 简写形式：默认302临时重定向
location /old-page.html {
    return /new-page.html;
}

# 301永久重定向，完整保留原始URI与查询参数
server {
    listen 80;
    server_name old-domain.com;
    return 301 https://new-domain.com$request_uri;
}

# 302临时跳转，保留原始请求参数
location /activity {
    return 302 https://$host/promotion?from=$request_uri;
}
```

### 4. 执行优先级与生命周期

1. **阶段定位**：`return`指令固定在**NGX_HTTP_REWRITE_PHASE**（重写阶段）执行，与`rewrite`、`if`指令处于同一阶段，按配置文件的书写顺序自上而下串行执行。
2. **终止特性**：`return`指令一旦执行，会立即终止当前作用域内所有后续 rewrite 模块指令，同时跳过请求处理的所有后续阶段（包括 location 二次匹配、内容处理、权限校验等），直接进入响应发送阶段，仅保留 access_log 日志记录流程。
3. **优先级对比**：同作用域内，`return`与`rewrite`指令按书写顺序执行，先匹配到的指令先执行；若`return`执行，后续所有指令均不会生效。

### 5. `return` 与 `rewrite` 重定向能力核心差异

表格

|对比维度|`return` 指令|`rewrite` 指令（重定向 flag）|
|---|---|---|
|执行效率|极高，无正则匹配开销，直接终止请求|中等，需完成正则匹配与 URI 替换，存在额外开销|
|循环风险|无，执行即终止全流程，无任何后续分支|存在，配置不当可能触发循环匹配，受 10 次循环上限约束|
|行为确定性|完全确定，无歧义，符合 HTTP 规范|存在歧义性，不同 flag、作用域下行为差异极大|
|参数保留|原生支持`$request_uri`完整保留原始路径与参数，无需正则捕获|需通过正则捕获组保留路径，默认自动追加查询参数，易出现参数丢失|
|官方推荐度|重定向场景官方首选方案|仅用于内部 URI 重写，不推荐用于纯重定向场景|
|适用场景|固定跳转、请求拦截、状态码返回、全局重定向|动态 URI 重写、伪静态、内部路由调度|

### 6. 常见陷阱与配置禁忌

1. **444 状态码误用**：`return 444;` 禁止附带任何文本或 URL，该状态码的核心是直接关闭 TCP 连接，附带内容会完全失效，无任何意义。
2. **重定向 URL 协议缺失**：重定向 URL 若未以`http://`/`https://`开头，会被 Nginx 当作响应体文本处理，导致客户端收到文本内容而非跳转响应，引发业务故障。
3. **非法状态码配置**：非重定向状态码禁止搭配 URL 使用，重定向状态码禁止搭配响应文本使用，否则会出现非预期行为，且语法校验不会报错。
4. **查询参数丢失陷阱**：重定向时若仅使用`$uri`，会丢失原始请求的查询字符串，必须使用`$request_uri`或显式追加`$query_string`才能完整保留参数。
5. **if 块内执行边界误解**：`if`块内的`return`指令执行后，会终止整个请求的处理流程，而非仅终止`if`块内的指令，后续同作用域的所有指令均不会执行。
6. **响应体编码问题**：自定义响应文本包含中文时，必须通过`add_header Content-Type "text/plain; charset=utf-8";`指定编码，否则会出现中文乱码。

### 7. 配置最佳实践

1. **重定向场景优先使用 return**：所有纯 URL 跳转需求，优先使用`return`指令实现，而非`rewrite permanent/redirect`，符合 Nginx 官方推荐，性能更高、风险更低。
2. **全局跳转优先在 server 块配置**：HTTP 转 HTTPS、域名迁移等全局跳转，直接在独立 server 块内使用`return`指令实现，无需`if`判断，性能最优。
3. **响应体显式指定 Content-Type**：自定义响应文本时，必须显式设置`Content-Type`响应头，避免不同客户端的解析差异。
4. **严格校验状态码合法性**：配置前确认状态码的使用场景，禁止混用重定向与非重定向状态码的语法形式。
5. **全场景测试验证**：配置修改后，通过`curl -I`命令验证响应状态码、`Location`头、响应体内容是否符合预期，覆盖所有边界场景。

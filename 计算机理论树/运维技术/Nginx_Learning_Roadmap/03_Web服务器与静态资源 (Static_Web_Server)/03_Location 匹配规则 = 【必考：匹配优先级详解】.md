
## 3.3 Location 匹配规则（必考：匹配优先级详解）

`location`是 Nginx 配置中**根据请求 URI 实现请求路由与差异化处理**的核心指令，通过**修饰符**定义不同的匹配模式，且不同模式存在严格的优先级顺序。理解 Location 匹配规则是实现精准路由、避免配置冲突的关键，也是 Nginx 面试与实际应用中的必考知识点。

### 3.3.1 匹配的核心前提

Nginx 用于 Location 匹配的 URI 为**客户端请求的原始 URI 去除查询参数后的部分**，不包含域名、端口，且为解码后的路径。

示例：请求`http://example.com/path/to/page?name=test&age=20`，参与匹配的 URI 为`/path/to/page`。

### 3.3.2 匹配模式与修饰符

Location 的匹配模式由**修饰符**与**匹配字符串**组成，共分为 5 类，其中 4 类带修饰符，1 类无修饰符，各类模式的语法与特性如下：

1. **精确匹配（=）**
    
    - 语法：`location = /uri { ... }`
    - 特性：请求 URI 与`/uri`**完全一致**时才匹配成功，无任何模糊匹配，是优先级最高的匹配模式；
    - 示例：`location = /`仅匹配`http://example.com/`，不匹配`http://example.com/index.html`或`http://example.com/path/`。
    
2. **优先前缀匹配（^~）**
    
    - 语法：`location ^~ /uri { ... }`
    - 特性：请求 URI**以 /uri 为前缀**即匹配，若该模式为**最长前缀匹配**，则直接生效且**停止后续所有正则匹配**，优先级仅次于精确匹配；
    - 示例：`location ^~ /img/`匹配`/img/a.jpg`、`/img/sub/b.png`，不匹配`/bimg/a.jpg`。
    
3. *_正则匹配（~ 区分大小写 / ~_ 不区分大小写）**
    
    - 语法：`location ~ /uri { ... }`（区分大小写）、`location ~* /uri { ... }`（不区分大小写）；
    - 特性：基于 PCRE 正则表达式匹配请求 URI，**按配置顺序匹配第一个成功的规则**，优先级低于精确匹配与优先前缀匹配；
    - 示例：`location ~ ^/Api/`仅匹配`/Api/`，不匹配`/api/`；`location ~* ^/Api/`同时匹配`/Api/`与`/api/`。
    
4. **标准前缀匹配（无修饰符）**
    
    - 语法：`location /uri { ... }`
    - 特性：请求 URI**以 /uri 为前缀**即匹配，Nginx 会在所有前缀匹配中选择**最长的匹配规则**，若该规则非`^~`模式，则会继续执行后续正则匹配，优先级低于正则匹配；
    - 示例：配置`location /doc/`与`location /documents/`，请求`/documents/file.pdf`会匹配`/documents/`（更长前缀）。
    
5. **通用匹配（/）**
    
    - 语法：`location / { ... }`
    - 特性：所有未被上述模式匹配的请求都会匹配该规则，是**最低优先级**的兜底规则；
    - 作用：处理所有非精准匹配的请求，如动态请求转发、404 统一处理等。
    

### 3.3.3 严格的匹配优先级顺序

Nginx 的 Location 匹配遵循**固定的优先级顺序**，并非按配置顺序生效，核心匹配流程与优先级如下（从高到低）：

1. 首先查找**精确匹配（=）** 规则，若找到则立即生效，终止所有后续匹配；
2. 若未找到精确匹配，查找**所有前缀匹配（^~ 与 无修饰符）** 规则，筛选出**最长前缀匹配**的规则；
3. 若该最长前缀匹配规则为 **^~ 模式 **，则立即生效，终止所有后续正则匹配；
4. 若该最长前缀匹配规则为**无修饰符模式**，则继续查找**所有正则匹配（~ / ~*）** 规则；
5. 正则匹配按**配置顺序**执行，找到第一个匹配成功的规则即生效，终止后续匹配；
6. 若未找到任何正则匹配规则，则使用步骤 2 中筛选出的**最长无修饰符前缀匹配**规则；
7. 若上述所有规则均未匹配，则使用**通用匹配（/）** 规则兜底。

### 3.3.4 经典匹配案例解析

nginx

```
# 规则1：精确匹配
location = / { return 200 "精确匹配/"; }
# 规则2：优先前缀匹配
location ^~ /img/ { return 200 "优先前缀匹配/img/"; }
# 规则3：标准前缀匹配
location /img/file/ { return 200 "标准前缀匹配/img/file/"; }
# 规则4：正则匹配（区分大小写）
location ~ /Img/ { return 200 "正则匹配/Img/"; }
# 规则5：通用匹配
location / { return 200 "通用匹配/"; }
```

- 请求`/`：匹配规则 1（精确匹配），返回 "精确匹配 /"；
- 请求`/img/a.jpg`：匹配规则 2（优先前缀匹配），返回 "优先前缀匹配 /img/"，终止正则匹配；
- 请求`/img/file/b.png`：规则 2 与规则 3 均为前缀匹配，规则 3 为更长前缀，且非 ^~ 模式，继续查找正则，无匹配后使用规则 3，返回 "标准前缀匹配 /img/file/"；
- 请求`/Img/c.jpg`：无精确与 ^~ 匹配，正则匹配规则 4，返回 "正则匹配 / Img/"；
- 请求`/path/test.html`：无任何匹配，使用规则 5（通用匹配），返回 "通用匹配 /"。

### 3.3.5 配置最佳实践

1. 高频访问的固定 URI（如首页`/`）使用**精确匹配（=）**，提升匹配效率；
2. 静态资源路径（如`/img/`、`/css/`）使用**优先前缀匹配（^~）**，避免被正则匹配覆盖，同时提升匹配效率；
3. 正则匹配规则按**匹配范围从窄到宽**配置，避免宽范围规则优先匹配导致窄范围规则失效；
4. 每个 server 块必须配置**通用匹配（/）** 作为兜底，处理未匹配的请求，避免返回 404；
5. 尽量减少正则匹配的使用，正则匹配效率低于前缀匹配，高并发场景下会影响性能。

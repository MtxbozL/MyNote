
## 5.3 节点状态参数

节点状态参数配置在 upstream 块的 server 指令末尾，用于定义后端节点的运行状态、故障阈值、流量控制规则，是实现服务故障隔离、容灾备份、过载保护的核心配置，生产环境所有节点均需配置核心状态参数。

### 5.3.1 down（永久下线）

- **核心作用**：标记该节点为**永久不可用**，不参与任何负载调度，Nginx 不会将任何请求分发至该节点。
- **适用场景**：节点下线维护、硬件故障、服务永久下线的场景。
- **配置示例**：`server 192.168.1.103:8080 down;`
- **注意事项**：down 标记的节点不会参与哈希算法的计算，可避免 ip_hash/url_hash 的哈希值波动。

### 5.3.2 backup（备用节点）

- **核心作用**：标记该节点为**备用容灾节点**，仅当所有主节点全部故障或不可用时，Nginx 才会将请求分发至该节点，实现服务的容灾备份与降级兜底。
- **核心特性**：主节点正常时，备用节点不接收任何请求；所有主节点不可用时，自动切换至备用节点；主节点恢复后，自动切回主节点；ip_hash 算法中，backup 节点不参与哈希计算。
- **配置示例**：
    
    nginx
    
    ```
    upstream backend_app {
        server 192.168.1.101:8080;
        server 192.168.1.102:8080;
        # 备用节点，主节点全故障时启用
        server 192.168.1.200:8080 backup;
    }
    ```
    
- **适用场景**：服务容灾备份、降级服务、主备架构的容灾场景。
- **最佳实践**：备用节点需提前做好服务预热，避免切换时因冷启动导致性能不足。

### 5.3.3 max_fails 与 fail_timeout（故障阈值与隔离超时）

- **核心作用**：两个参数配合使用，定义后端节点的**故障判定与自动隔离规则**：在`fail_timeout`时间内，节点请求失败次数达到`max_fails`阈值，Nginx 将该节点标记为不可用，在接下来的`fail_timeout`时间内，不会将请求分发至该节点；超时时间结束后，Nginx 会自动探测节点是否恢复，恢复则重新加入调度，否则继续隔离。
- **核心特性**：默认值：`max_fails=1`，`fail_timeout=10s`；失败判定规则由`proxy_next_upstream`指令定义（如连接超时、502/503 错误等）；实现故障自动隔离与自动恢复，无需人工干预。
- **配置示例**：
    
    nginx
    
    ```
    upstream backend_app {
        # 10秒内连续失败3次，隔离节点10秒
        server 192.168.1.101:8080 max_fails=3 fail_timeout=10s;
        server 192.168.1.102:8080 max_fails=3 fail_timeout=10s;
    }
    ```
    
- **适用场景**：所有生产环境的负载均衡配置，必配核心参数。
- **最佳实践**：生产环境推荐`max_fails=3~5`，`fail_timeout=10s~30s`，避免偶发超时导致的节点误隔离；配合`proxy_next_upstream`明确失败判定条件。

### 5.3.4 生产级补充参数

- **max_conns**：限制单个后端节点的最大并发连接数，默认值为 0（无限制），用于保护后端节点，避免请求过载，配置示例：`server 192.168.1.101:8080 max_conns=1000;`
- **slow_start**：节点恢复后，流量在指定时间内缓慢增加至权重值，避免节点刚恢复就被大量请求压垮，仅商业版 Nginx Plus 支持，开源版需第三方模块实现，配置示例：`server 192.168.1.101:8080 slow_start=30s;`

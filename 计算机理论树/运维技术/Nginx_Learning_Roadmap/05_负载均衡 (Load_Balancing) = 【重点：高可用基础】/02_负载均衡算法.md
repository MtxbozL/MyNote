
## 5.2 负载均衡算法

Nginx 的负载均衡算法分为**内置默认算法**（无需额外编译）、**内置可选算法**（需配置启用）、**第三方扩展算法**（需编译第三方模块）三类，一个 Upstream 服务池仅能配置一种核心调度算法，未配置时默认启用**加权轮询算法**。本节完整覆盖目录指定的 5 类核心算法，明确其原理、配置、适用场景与局限性。

### 5.2.1 轮询（Round Robin - 默认）

- **核心原理**：Nginx 默认的调度算法，将客户端请求**按顺序依次均匀分发**至后端服务池的每个节点，无状态调度，适用于后端所有节点硬件配置、处理能力完全一致的场景。
- **核心特性**：节点故障时自动剔除，恢复后自动加入调度；无需额外配置，开箱即用；请求分发完全均匀，长期运行下各节点接收请求数偏差小于 1%。
- **配置示例**：
    
    nginx
    
    ```
    upstream backend_app {
        # 未指定算法，默认启用轮询
        server 192.168.1.101:8080;
        server 192.168.1.102:8080;
        server 192.168.1.103:8080;
    }
    ```
    
- **适用场景**：后端节点配置一致的无状态服务（如 RESTful API、静态资源服务）、无需会话保持的通用场景。
- **局限性**：无法适配后端节点性能差异的场景，性能弱的节点易被压满；无法解决无共享存储场景下的 Session 会话共享问题。

### 5.2.2 权重（weight - 适配后端服务器性能差异）

- **核心原理**：轮询算法的扩展，即**加权轮询**，通过`weight`参数为每个节点分配正整数权重值，权重与请求分发比例正相关，权重越高的节点被分配的请求数越多，完美适配后端节点性能不一致的场景。
- **核心特性**：权重默认值为 1；权重比例决定请求分发比例，如 3 个节点权重为 3:2:1，则 6 个连续请求中，节点分别分配 3、2、1 个；支持动态调整权重，`reload`热重载即可生效，无需重启服务。
- **配置示例**：
    
    nginx
    
    ```
    upstream backend_app {
        # 加权轮询，按节点性能分配权重
        server 192.168.1.101:8080 weight=3; # 高性能节点，权重最高
        server 192.168.1.102:8080 weight=2; # 中等性能节点
        server 192.168.1.103:8080 weight=1; # 低性能节点
    }
    ```
    
- **适用场景**：后端服务器硬件 / 性能存在差异的异构集群、服务灰度发布、按比例分配流量的场景。
- **最佳实践**：权重值不宜设置过大，避免单节点压力过载；灰度发布时逐步调整权重比例，实现流量平滑切换。

### 5.2.3 IP 哈希（ip_hash - 解决 Session 共享问题）

- **核心原理**：基于客户端的**真实 IP 地址（$remote_addr）** 计算哈希值，将哈希值与后端节点数取模，固定将同一个客户端 IP 的所有请求分发至**同一台后端节点**，实现会话粘性，彻底解决无分布式 Session 共享方案下的会话保持问题。
- **核心特性**：强会话粘性，同一客户端 IP 始终访问同一节点；节点故障时，客户端 IP 会重新哈希至其他节点；自动剔除故障节点，保证服务可用性；`down`标记的节点不参与哈希计算。
- **配置示例**：
    
    nginx
    
    ```
    upstream backend_app {
        # 启用IP哈希算法
        ip_hash;
        server 192.168.1.101:8080;
        server 192.168.1.102:8080;
        server 192.168.1.103:8080 down; # 下线节点，不参与哈希计算
    }
    ```
    
- **适用场景**：后端服务基于 Session 存储用户状态、无分布式 Session 共享方案、需要固定客户端访问节点的场景（如后台管理系统、IM 服务）。
- **局限性**：大量客户端来自同一出口 IP（如企业内网、NAT 网络）时，会导致流量分发严重不均；节点上下线会导致全量客户端哈希重分配，会话批量丢失。
- **优化方案**：使用`hash $http_x_forwarded_for consistent;`替代 ip_hash，基于 X-Forwarded-For 头计算哈希，同时启用一致性哈希，减少节点变更导致的哈希波动。

### 5.2.4 最少连接（least_conn）

- **核心原理**：自适应动态调度算法，Nginx 将客户端请求分发至**当前活跃连接数最少**的后端节点，解决了轮询算法中 “请求处理耗时不均导致节点负载失衡” 的问题，适用于请求处理时长差异大的场景。
- **核心特性**：结合权重参数实现**加权最少连接**调度，权重越高的节点，可承载的最大连接数越高；自动规避负载过高的节点，提升集群整体稳定性；长连接场景下适配性远优于轮询算法。
- **配置示例**：
    
    nginx
    
    ```
    upstream backend_app {
        # 启用最少连接算法，结合权重实现加权调度
        least_conn;
        server 192.168.1.101:8080 weight=3;
        server 192.168.1.102:8080 weight=2;
        server 192.168.1.103:8080 weight=1;
    }
    ```
    
- **适用场景**：请求处理耗时差异大的服务（如文件上传、大数据查询、复杂计算接口）、WebSocket 长连接服务、后端节点负载波动大的场景。

### 5.2.5 URL 哈希（hash $request_uri）

- **核心原理**：基于客户端请求的**完整 URI（$request_uri）** 计算哈希值，将相同 URI 的请求固定分发至同一后端节点，核心价值是提升后端缓存服务的命中率，适用于静态资源、缓存集群场景。
- **核心特性**：支持一致性哈希（添加`consistent`参数），节点上下线仅影响约 1/N 的缓存（N 为节点数），避免普通哈希导致的全量缓存失效与缓存雪崩；可自定义哈希键值，如`$args`、`$cookie_userid`等。
- **配置示例**：
    
    nginx
    
    ```
    # 场景1：普通URL哈希，用于静态资源缓存服务
    upstream cache_server {
        hash $request_uri;
        server 192.168.1.201:80;
        server 192.168.1.202:80;
    }
    
    # 场景2：一致性URL哈希，生产环境推荐，避免缓存雪崩
    upstream cache_cluster {
        hash $request_uri consistent; # 核心：启用一致性哈希
        server 192.168.1.201:80 weight=3;
        server 192.168.1.202:80 weight=2;
        server 192.168.1.203:80 weight=1;
    }
    ```
    
- **适用场景**：后端为缓存服务器（Varnish、Squid）、CDN 回源服务、静态资源分发、相同 URI 请求需固定节点处理的场景。

### 5.2.6 核心算法对比表

表格

|算法类型|核心调度依据|会话粘性|核心优势|核心局限性|适配场景|
|---|---|---|---|---|---|
|轮询（默认）|请求顺序|无|配置简单、分发均匀、无额外开销|无法适配性能差异、请求耗时不均场景|节点性能一致的无状态 API 服务|
|加权轮询|节点权重|无|适配性能差异、流量可控、平滑灰度|无法解决会话共享、请求耗时不均问题|异构集群、灰度发布、通用业务场景|
|IP 哈希|客户端 IP|有|解决会话共享、配置简单|同出口 IP 导致流量不均、节点变更会话丢失|无分布式 Session 的会话保持场景|
|最少连接|节点实时活跃连接数|无|自适应负载、规避过载节点、长连接适配性强|调度开销略高于轮询、无会话粘性|长连接服务、请求处理耗时差异大的场景|
|URL 哈希|请求 URI|无|大幅提升缓存命中率、支持一致性哈希|无会话粘性、不适用于动态 API 服务|缓存集群、CDN 回源、静态资源服务|

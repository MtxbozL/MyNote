
Ansible 的核心性能瓶颈集中在**SSH 连接开销**与**任务执行往返次数**，本节严格覆盖大纲指定的四大调优手段，同时补充配套优化方案，实现大规模集群下的执行效率数十倍提升，是高级架构师必备的性能调优能力。

### 8.5.1 核心性能瓶颈分析

Ansible 默认配置的性能瓶颈主要来自 4 个方面：

1. **SSH 连接开销**：默认每个任务都会新建 SSH 连接，TCP 握手、密钥认证的开销占比超过 60%，大规模集群下尤为明显；
2. **并发数限制**：默认并发数`forks=5`，百台以上节点执行时，需要分多批次执行，整体执行时间线性增长；
3. **模块执行往返**：默认执行模块时，需要多次 SSH 往返传输文件、执行命令，增加了执行耗时；
4. **Python 解释器开销**：默认每个任务都会在被控节点启动一个新的 Python 解释器进程，重复的进程启动开销巨大。

### 8.5.2 四大核心调优手段

#### 1. 增加并发数：forks 参数

**核心原理**：`forks`参数定义了 Ansible 同时并行执行的最大被控节点数量，默认值为 5，提升该值可让 Ansible 同时管理更多节点，大幅缩短大规模集群的整体执行时间。

**配置方式**：

1. **全局配置**：`ansible.cfg`的`[defaults]`段设置：

    ```bash
    [defaults]
    # 最大并发数设置为50，根据控制节点CPU核心数调整
    forks = 50
    ```
    
2. **命令行临时指定**：通过`-f/--forks`参数覆盖全局配置：

    ```bash
    ansible-playbook deploy.yml -f 100
    ```
    

**最佳实践**：

- 并发数上限建议为控制节点 CPU 核心数的 2-4 倍，避免设置过高导致控制节点 CPU、内存耗尽；
- 2 核控制节点建议设置为 10-20，4 核设置为 30-50，8 核以上可设置为 50-100；
- 网络 IO 密集型任务可适当提高，CPU 密集型任务需降低并发数；
- 生产环境需逐步提升并发数，同时监控控制节点的负载，避免影响其他业务。

#### 2. 开启 SSH Pipelining

**核心原理**：Pipelining 是 Ansible 的 SSH 性能优化核心特性，开启后，Ansible 将模块执行代码通过 SSH 管道直接传递给远程 Python 解释器，无需多次 SSH 往返传输临时文件，将默认的 5 次 SSH 往返减少为 1 次，**可提升执行效率 30%-50%**。

**配置方式**：`ansible.cfg`的`[ssh_connection]`段设置：

```bash
[ssh_connection]
# 开启SSH管道传输
pipelining = True
```

**前置依赖配置**：

开启 Pipelining 需要关闭被控节点 sudoers 配置中的`requiretty`选项，否则会执行失败。需在被控节点的`/etc/sudoers`中配置：

```bash
# 注释或删除该行
# Defaults    requiretty
# 为Ansible管理用户添加免TTY配置
Defaults:ops !requiretty
```

该配置可通过 Ansible 剧本批量下发，适配大规模集群。

#### 3. ControlMaster / SSH 连接复用

**核心原理**：基于 OpenSSH 的 ControlMaster 功能，为同一个被控节点的所有 SSH 会话复用同一个 TCP 连接，避免每次任务都重新进行 TCP 握手、密钥认证、加密协商，**可将 SSH 连接开销降低 90% 以上**，是大规模集群最有效的优化手段。

**配置方式**：`ansible.cfg`的`[ssh_connection]`段设置：

```bash
[ssh_connection]
# SSH连接复用配置，ControlPersist=60s表示连接空闲60秒后关闭
ssh_args = -o ControlMaster=auto -o ControlPersist=60s -o StrictHostKeyChecking=no
# 控制套接字文件存储路径
control_path = /tmp/ansible-ssh-%%h-%%p-%%r
```

**关键参数说明**：

- `ControlMaster=auto`：自动创建 / 复用控制连接；
- `ControlPersist=60s`：控制连接的空闲保持时间，长剧本建议设置为 300s（5 分钟），避免频繁重建连接；
- `%%h/%%p/%%r`：分别对应目标主机、端口、远程用户，保证套接字文件的唯一性。

**最佳实践**：

- 长耗时、多任务的 Playbook，设置`ControlPersist=300s`；
- 短平快的 Ad-Hoc 命令，设置`ControlPersist=60s`即可；
- 控制节点需定期清理 /tmp 下的闲置套接字文件，避免文件句柄泄露。

#### 4. Mitogen 插件提速

**核心原理**：Mitogen 是第三方开源的 Ansible 执行引擎，通过高效的 Python 进程复用、序列化优化、连接池技术，彻底重构了 Ansible 的连接与执行机制，**可将执行速度提升 2-10 倍**，同时大幅降低控制节点的 CPU、内存占用，是超大规模集群的终极优化方案。

**部署与配置方式**：

1. **安装 Mitogen**：

    ```bash
    # pip安装
    pip install mitogen
    
    # 或源码安装
    wget https://github.com/mitogen-hq/mitogen/archive/refs/tags/v0.3.7.tar.gz
    tar -zxf mitogen-0.3.7.tar.gz -C /opt/
    ```
    
2. **Ansible 配置集成**：在`ansible.cfg`中配置策略插件：

    ```bash
    [defaults]
    # 启用Mitogen策略
    strategy_plugins = /opt/mitogen-0.3.7/ansible_mitogen/plugins/strategy
    strategy = mitogen_linear
    # 可选mitogen_free策略，对应原生free策略
    ```
    

**核心优势**：

- 被控节点复用同一个 Python 进程，避免每个任务都启动新进程，大幅降低开销；
- 优化的序列化机制，减少 SSH 传输的数据量；
- 内置连接池与连接复用，无需依赖 SSH ControlMaster；
- 完全兼容原生 Ansible 的所有语法、模块、剧本，无需修改现有代码。

**注意事项**：

- Mitogen 与部分小众模块、自定义插件可能存在兼容性问题，生产环境需先在测试环境验证；
- 需匹配 Ansible 版本，Mitogen 0.3.7 支持 Ansible 2.10-2.16 版本；
- 开启 Mitogen 后，原生的 pipelining、ControlMaster 优化仍可叠加使用，进一步提升性能。

### 8.5.3 补充配套优化方案

1. **Facts 采集优化**：第五章已详解，无需 Facts 的剧本设置`gather_facts: no`，需要时开启 Facts 缓存，避免重复采集；
2. **优化 Python 解释器**：被控节点统一使用 Python3，禁用 Python2，提升执行效率；
3. **减少不必要的任务**：合并重复任务，使用循环替代多个重复 Task，减少 SSH 往返次数；
4. **使用更快的 JSON 解析器**：安装`python-ujson`，替代原生 JSON 解析器，提升大变量的解析速度；
5. **关闭不必要的日志输出**：生产环境关闭`-v`详细输出，减少日志打印的 CPU 开销；
6. **控制节点网络优化**：控制节点与被控节点同网段部署，降低网络延迟，提升 SSH 连接速度。

### 8.5.4 调优最佳实践

1. **循序渐进**：先开启基础优化（Pipelining、ControlMaster、提升 forks），再根据需求启用 Mitogen；
2. **基准测试**：优化前后通过`time`命令测试执行时间，量化优化效果，避免盲目调优；
3. **适配场景**：小规模集群基础优化即可，百台以上大规模集群启用 Mitogen；
4. **兼容性验证**：所有优化必须先在测试环境验证，确保剧本执行正常，无兼容性问题；
5. **监控控制节点**：调优后持续监控控制节点的 CPU、内存、网络、文件句柄使用率，避免资源耗尽；
6. **标准化配置**：将优化后的`ansible.cfg`纳入项目版本控制，保证所有执行环境的一致性。

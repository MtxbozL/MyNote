### 8.4.1 核心定义

执行策略（Strategy）是 Ansible 的核心调度机制，定义了 Ansible 在多台被控节点上的任务执行顺序、并发逻辑与调度规则，直接决定了 Playbook 的执行效率与行为模式。Ansible 官方提供多种执行策略，大纲重点指定了**Linear 线性执行**与**Free 自由并发执行**两大核心策略。

执行策略通过 Play 级的`strategy`关键字指定，未指定时默认使用`linear`线性策略，也可在`ansible.cfg`的`[defaults]`段通过`strategy`参数设置全局默认策略。

### 8.4.2 Linear 线性执行策略（默认）

#### 核心定义

Linear 线性策略是 Ansible 的默认执行策略，核心规则是：**所有被控节点必须全部完成当前 Task 后，才会进入下一个 Task 的执行**，形成严格的任务同步屏障。

执行流程示例（3 台被控节点，3 个 Task）：

1. 所有 3 台节点同时执行 Task1，直到最慢的节点完成 Task1；
2. 所有 3 台节点同时进入 Task2，直到全部完成 Task2；
3. 所有 3 台节点同时进入 Task3，直到全部完成 Task3；
4. Play 执行结束。

#### 标准语法示例

```yaml
---
- name: Linear线性执行策略示例
  hosts: webservers
  strategy: linear  # 显式指定线性策略，默认可省略
  become: yes
  tasks:
    - name: Task1：安装软件包
      yum:
        name: nginx
        state: present

    - name: Task2：配置服务
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf

    - name: Task3：启动服务
      service:
        name: nginx
        state: started
        enabled: yes
```

#### 核心特性与适用场景

|优势|局限|
|:--|:--|
|执行逻辑清晰，可预测性强，便于调试与问题定位|执行效率受最慢的节点拖累，整体执行时间长|
|严格保证任务执行顺序，适配强依赖场景|大规模集群场景下，大量时间消耗在等待慢节点|
|便于实现全集群同步变更，避免版本不一致|节点故障会导致整个 Playbook 阻塞|

**核心适用场景**：

- 强依赖、强一致性要求的场景，如数据库集群变更、分布式系统同步升级；
- 先全集群安装软件，再统一配置，最后统一启动的标准化流程；
- 测试环境、小规模集群，需要清晰的执行日志与调试能力；
- 滚动发布、蓝绿发布等需要严格控制执行节奏的场景。

### 8.4.3 Free 自由并发执行策略

#### 核心定义

Free 自由并发策略的核心规则是：**每个被控节点独立执行完整的任务列表，不受其他节点的执行进度影响，执行速度快的节点会优先完成所有任务，无需等待其他节点**，彻底打破线性策略的同步屏障，最大化执行效率。

执行流程示例（3 台被控节点，3 个 Task）：

1. 节点 1、2、3 同时开始执行 Task1；
2. 节点 1 率先完成 Task1，立即进入 Task2，无需等待节点 2、3；
3. 节点 1 完成 Task2，立即进入 Task3，同时节点 2 完成 Task1 进入 Task2；
4. 节点 1 完成所有 Task，结束执行，节点 2、3 继续独立执行；
5. 所有节点完成各自的任务列表后，Play 执行结束。

#### 标准语法示例

```yaml
---
- name: Free自由并发执行策略示例
  hosts: all
  strategy: free  # 显式指定自由并发策略
  become: yes
  tasks:
    - name: 系统基线初始化
      yum:
        name: ['epel-release', 'htop', 'iftop', 'git']
        state: present

    - name: 配置系统内核参数
      sysctl:
        name: "{{ item.key }}"
        value: "{{ item.value }}"
        state: present
      loop: "{{ sysctl_config }}"

    - name: 配置时间同步
      service:
        name: chronyd
        state: started
        enabled: yes
```

#### 核心特性与适用场景

|优势|局限|
|:--|:--|
|执行效率极高，整体执行时间由最慢的单个节点决定，无等待损耗|执行顺序不可预测，调试难度高，日志分散|
|单节点故障不影响其他节点的执行进度|无法保证全集群的同步变更，可能出现版本不一致的时间窗口|
|大规模集群场景下，大幅缩短整体执行时间|不适用于强依赖、强一致性要求的场景|

**核心适用场景**：

- 无依赖的批量初始化操作，如系统基线配置、监控 Agent 安装、安全补丁下发；
- 大规模集群（百台以上）的无状态变更，最大化执行效率；
- 节点之间无业务依赖、无执行顺序要求的场景；
- 故障修复、批量配置更新等无需严格同步的场景。

### 8.4.4 补充核心执行策略

1. **Host Pinned 策略**：每个被控节点绑定固定的执行进程，完整执行所有任务后，进程才会释放，适用于需要保持会话状态的场景，避免多次 SSH 连接开销；
2. **Debug 策略**：专门用于调试，任务执行失败时自动进入交互式调试终端，可查看变量、执行命令、定位问题；
3. **Serial 串行策略**：严格来说是执行批次控制，配合`serial`参数将被控节点分为多个批次，每个批次内使用 linear/free 策略，适用于滚动发布、零停机升级场景，在第六章已详解。

### 8.4.5 最佳实践

1. **策略选型原则**：强一致性、强依赖场景用 Linear 线性策略，无依赖、大规模批量操作用 Free 自由策略；
2. **避免混用策略**：同一个项目内保持策略统一，避免无规则混用导致的执行逻辑混乱；
3. **串行 + 线性组合**：滚动发布场景必须使用`serial`+`linear`组合，严格控制执行批次与节奏，保障业务可用性；
4. **大规模集群优化**：百台以上节点的无状态操作，优先使用`free`策略，配合`forks`提升并发数，执行效率提升可达 50% 以上；
5. **调试规范**：调试阶段统一使用`linear`策略，保证执行逻辑清晰，问题定位完成后，再根据场景切换为`free`策略。

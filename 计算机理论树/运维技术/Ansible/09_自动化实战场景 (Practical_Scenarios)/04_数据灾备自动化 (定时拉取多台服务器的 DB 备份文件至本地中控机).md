### 9.4.1 场景核心目标

本场景实现企业级数据灾备的全流程自动化，核心解决多服务器业务数据、数据库备份的批量管理、集中存储、完整性校验、生命周期管理问题，通过 Ansible 实现「被控节点本地备份→中控机批量拉取→完整性校验→过期清理」的全闭环灾备流程，完全覆盖大纲指定的「定时拉取多台服务器的 DB 备份文件至本地中控机」核心需求。

### 9.4.2 核心实现方案

灾备流程分为两大核心阶段：

1. **被控节点本地备份**：在目标服务器执行数据库 / 业务数据的本地备份、压缩、加密、权限控制；
2. **中控机集中拉取**：通过`synchronize`/`fetch`模块，将所有被控节点的备份文件批量拉取到中控机的集中存储目录，按主机名分类存储，避免同名文件覆盖；
3. **配套能力**：备份完整性校验、过期备份自动清理、备份结果告警、定时任务自动配置。

### 9.4.3 完整灾备 Playbook 实现

#### 1. 主 Playbook：db_backup_auto.yml

```yaml
---
- name: 数据库批量备份与中控机集中拉取
  hosts: db_servers
  become: yes
  gather_facts: yes
  vars:
    # 数据库配置
    db_type: mysql
    db_user: root
    db_password: "{{ mysql_root_password }}"
    # 备份配置
    backup_local_dir: /data/backup/mysql
    backup_retention_days: 7
    backup_encrypt: yes
    # 中控机配置
    backup_center_dir: /data/backup_center/mysql
    pull_backup_to_center: yes

  pre_tasks:
    - name: 创建本地备份目录
      file:
        path: "{{ backup_local_dir }}"
        state: directory
        mode: 0700
        owner: mysql
        group: mysql

    - name: 生成备份文件名
      set_fact:
        backup_file_name: "{{ inventory_hostname }}_mysql_full_{{ ansible_date_time.date }}_{{ ansible_date_time.hour }}{{ ansible_date_time.minute }}.sql.gz"
        backup_full_path: "{{ backup_local_dir }}/{{ backup_file_name }}"

  tasks:
    - name: 执行MySQL全量备份并压缩
      shell: |
        mysqldump -u{{ db_user }} -p{{ db_password }} --all-databases --single-transaction --routines --triggers | gzip > {{ backup_full_path }}
      no_log: yes
      async: 3600
      poll: 30
      register: backup_result

    - name: 备份文件权限设置
      file:
        path: "{{ backup_full_path }}"
        mode: 0600
        owner: root
        group: root
      when: backup_result is succeeded

    - name: 计算备份文件MD5校验值
      shell: md5sum {{ backup_full_path }} | awk '{print $1}'
      register: backup_md5
      changed_when: no
      when: backup_result is succeeded

    - name: 生成MD5校验文件
      copy:
        content: "{{ backup_md5.stdout }}  {{ backup_file_name }}"
        dest: "{{ backup_full_path }}.md5"
        mode: 0600
      when: backup_result is succeeded

    - name: 清理超过保留期的本地备份文件
      find:
        path: "{{ backup_local_dir }}"
        patterns: "*.sql.gz"
        age: "{{ backup_retention_days }}d"
      register: expired_backup_files

    - name: 删除过期备份文件
      file:
        path: "{{ item.path }}"
        state: absent
      loop: "{{ expired_backup_files.files }}"
      when: expired_backup_files.matched > 0

  post_tasks:
    - name: 批量拉取备份文件到中控机
      synchronize:
        src: "{{ backup_local_dir }}/"
        dest: "{{ backup_center_dir }}/{{ inventory_hostname }}/"
        mode: pull
      delegate_to: localhost
      when: pull_backup_to_center | bool and backup_result is succeeded

    - name: 中控机校验备份文件完整性
      shell: md5sum -c {{ backup_file_name }}.md5
      args:
        chdir: "{{ backup_center_dir }}/{{ inventory_hostname }}/"
      delegate_to: localhost
      register: md5_check_result
      when: pull_backup_to_center | bool and backup_result is succeeded
      failed_when: md5_check_result.rc != 0

    - name: 输出备份结果
      debug:
        msg: "节点 {{ inventory_hostname }} 备份完成，文件：{{ backup_file_name }}，MD5校验：{{ '成功' if md5_check_result is succeeded else '失败' }}"
```

#### 2. 定时任务配置：backup_cron.yml

通过 Ansible 批量配置定时任务，实现每日自动备份，无需手动登录服务器配置：

```yaml
---
- name: 配置数据库每日自动备份定时任务
  hosts: db_servers
  become: yes
  tasks:
    - name: 创建备份脚本目录
      file:
        path: /opt/scripts/backup
        state: directory
        mode: 0700

    - name: 生成备份执行脚本
      template:
        src: mysql_backup.sh.j2
        dest: /opt/scripts/backup/mysql_backup.sh
        mode: 0700

    - name: 配置每日凌晨2点自动备份定时任务
      cron:
        name: mysql_daily_backup
        job: /bin/bash /opt/scripts/backup/mysql_backup.sh > /var/log/mysql_backup.log 2>&1
        minute: "0"
        hour: "2"
        user: root
        state: present
```

### 9.4.4 最佳实践与避坑指南

1. **异步执行保障**：大数据库备份必须使用`async`异步执行，避免 SSH 会话超时导致备份中断；
2. **敏感信息保护**：数据库密码必须通过 Ansible Vault 加密，备份脚本中的密码需严格控制权限，禁止全局可读；
3. **完整性校验**：备份完成后必须进行 MD5/SHA256 校验，拉取到中控机后二次校验，确保备份文件完整可用；
4. **分级存储**：本地备份 + 中控机集中存储双副本，重要数据需额外配置异地备份，避免单点故障导致备份丢失；
5. **生命周期管理**：通过保留期自动清理过期备份，避免磁盘空间耗尽，同时保障合规要求的备份留存周期；
6. **备份恢复验证**：定期执行备份恢复测试，验证备份文件的可用性，避免备份成功但无法恢复的致命问题；
7. **权限最小化**：备份目录、脚本、备份文件均设置最小权限（0700/0600），避免非授权用户访问敏感数据；
8. **告警机制**：备份失败时必须配置告警通知，通过 Callback 插件对接钉钉 / 邮件，及时发现备份异常。

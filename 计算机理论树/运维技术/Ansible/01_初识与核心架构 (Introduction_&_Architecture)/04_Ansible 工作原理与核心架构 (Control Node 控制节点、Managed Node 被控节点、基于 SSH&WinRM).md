### 1.4.1 核心架构组件

Ansible 采用去中心化的极简架构，核心分为两大节点类型与一套原生通信体系，无强制的中心化服务端组件。

#### 1.4.1.1 控制节点（Control Node）

**定义**：Ansible 的核心控制中枢，是执行 Ansible 命令、运行 Playbook 的主机，所有任务编排、逻辑计算、配置渲染、结果汇总均在此节点完成。

- **系统要求**：仅支持 Linux/Unix/macOS 系统（Windows 仅可通过 WSL2 作为控制节点），需预装 Python 3.8+（Ansible 2.15 + 最低要求 Python 3.9）与 Ansible 核心程序。
- **核心职责**：
    
    1. 加载全局配置文件，解析执行参数；
    2. 解析 Inventory 资产清单，获取被控节点列表、连接参数与变量；
    3. 解析 Playbook/Ad-Hoc 命令，完成任务编排、变量渲染、逻辑预处理；
    4. 与被控节点建立安全远程会话，推送任务模块与执行逻辑；
    5. 接收被控节点返回的执行结果，完成日志输出、状态汇总与异常处理；
    6. 调度任务流转、触发器执行等高级逻辑。
    

#### 1.4.1.2 被控节点（Managed Node）

**定义**：被 Ansible 管理的目标主机，也称为受控节点、目标节点，是自动化任务的最终执行对象。

- **系统要求**：Linux/Unix 系统需预装 Python 2.7/3.5+，默认开启 SSH 服务；Windows 系统需开启 WinRM 服务，预装 PowerShell 3.0+。
- **核心职责**：
    
    1. 接收控制节点推送的模块代码与执行指令；
    2. 在本地完成状态校验与任务执行；
    3. 将执行结果、状态信息、日志序列化为 JSON 格式，返回至控制节点。
    

> 核心要点：被控节点**无需安装 Ansible 核心程序**，这是无代理架构的核心体现。

#### 1.4.1.3 核心通信协议

Ansible 完全依托操作系统原生远程管理协议实现无代理通信，无专属通信协议：

1. **SSH 协议**：Linux/Unix 类被控节点的默认通信协议，支持密码认证、密钥认证、证书认证等多种安全认证方式，默认使用系统原生 OpenSSH 客户端，兼容 Paramiko Python SSH 库；
2. **WinRM 协议**：Windows 被控节点的官方通信协议，基于 HTTP/HTTPS 传输，支持 NTLM、Kerberos 等认证方式，需提前在 Windows 节点配置监听与权限。

### 1.4.2 Ansible 完整任务执行工作流程

从用户输入指令到任务完成，Ansible 遵循固定的 8 步执行流程，全链路无黑盒逻辑：

1. **指令解析阶段**：用户在控制节点输入 Ad-Hoc 命令或执行 Playbook，Ansible 优先加载配置文件（按优先级规则），读取全局配置参数；
2. **资产加载阶段**：解析 Inventory 资产清单，获取目标被控节点列表、主机分组、连接参数、主机 / 组变量，锁定任务执行的目标范围；
3. **任务预处理阶段**：解析任务逻辑，完成 Jinja2 变量渲染、条件判断、循环逻辑预处理，确定最终待执行的任务列表与执行顺序；
4. **会话建立阶段**：控制节点基于连接参数，通过 SSH/WinRM 协议与目标被控节点建立安全远程会话，完成身份认证；
5. **模块推送与执行阶段**：Ansible 将任务对应的模块代码推送至被控节点临时目录，在被控节点本地启动 Python 解释器子进程执行模块代码；模块执行前先完成状态校验，基于幂等性逻辑判断是否执行变更；
6. **结果返回阶段**：被控节点执行完成后，将执行状态（成功 / 失败 / 跳过）、变更信息、标准输出 / 错误日志序列化为 JSON 格式，同步返回至控制节点；
7. **结果汇总与后续处理阶段**：控制节点接收所有被控节点的返回结果，完成汇总与日志输出，同时基于执行结果处理任务流转、触发器调度、异常捕获等逻辑；
8. **会话清理阶段**：所有任务执行完成后，关闭远程会话，清理被控节点临时目录中的模块文件，完成全流程。

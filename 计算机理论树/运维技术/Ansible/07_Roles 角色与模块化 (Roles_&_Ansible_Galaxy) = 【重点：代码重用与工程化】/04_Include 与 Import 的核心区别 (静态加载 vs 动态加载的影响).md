`include_*`系列（include_role、include_tasks）与`import_*`系列（import_role、import_tasks）的核心本质区别，在于**加载时机的不同**，所有特性差异、适用场景、限制规则均源于此，是 Ansible 进阶的必考核心知识点。

### 7.4.1 核心本质差异

|特性|import_* 系列（静态导入）|include_* 系列（动态包含）|
|:--|:--|:--|
|**加载时机**|Playbook**解析阶段**（执行前）预加载，完全解析并融入 Play 结构|Playbook**执行阶段**，运行到该任务时才加载，解析时机晚|
|**加载类型**|静态加载，预编译处理|动态加载，运行时处理|

### 7.4.2 全维度特性对比

|对比维度|import_* 静态导入|include_* 动态包含|
|:--|:--|:--|
|**循环支持**|不支持 loop 循环，解析阶段无法处理运行时循环逻辑|完美支持 loop 循环，可基于列表批量包含角色 / 任务|
|**条件判断支持**|支持 when 条件，条件会自动继承给导入的所有任务，每个任务都会判断条件|支持 when 条件，仅在包含时判断一次，条件满足才会加载角色 / 任务|
|**变量解析**|解析阶段就完成变量解析，不支持运行时动态生成的变量|执行时才解析变量，完美支持运行时动态生成的变量、Register 注册变量|
|**语法错误检测**|解析阶段就会检测并抛出语法错误，提前发现问题，执行前即可校验|只有执行到该任务时才会检测语法错误，错误发现时机晚|
|**标签（Tags）继承**|给 import 任务设置的 tags，会完全继承给导入的所有任务，标签筛选时会匹配所有子任务|给 include 任务设置的 tags，仅作用于 include 任务本身，不会继承给包含的子任务，需通过`apply`参数传递标签|
|**执行列表展示**|`--list-tasks`会列出导入的所有子任务，可提前看到完整执行列表|`--list-tasks`仅会列出 include 任务本身，不会展示包含的子任务，因为运行时才加载|
|**执行效率**|预解析完成，执行效率更高|运行时动态加载，执行效率略低|
|**容错性**|语法错误会导致 Playbook 完全无法启动，容错性低|仅当执行到该任务时才会报错，不影响前面的任务执行，容错性高|

### 7.4.3 核心适用场景与最佳实践

#### import_* 静态导入适用场景

1. 固定的、确定要执行的角色 / 任务，无需基于运行时结果动态判断；
2. 需要提前做完整的语法校验、任务列表预览的生产环境剧本；
3. 需要给所有子任务统一继承条件、标签的场景；
4. 对执行效率有要求的大规模集群场景。

#### include_* 动态包含适用场景

1. 需要基于循环批量执行多个角色 / 任务的场景；
2. 需要基于运行时结果（Register 变量、Facts）动态判断是否加载的场景；
3. 复杂的嵌套逻辑、动态流程编排场景；
4. 需要按需加载，提升剧本执行灵活性的场景。

#### 核心最佳实践

1. **优先使用静态导入**：无动态需求的场景，优先使用`import_*`，提前发现语法错误，执行更稳定，可维护性更强；
2. **动态场景专用动态包含**：仅当需要循环、运行时动态判断时，才使用`include_*`，避免滥用动态加载导致的调试困难；
3. **禁止混合使用**：避免在同一个剧本中无规则混用两种方式，导致执行逻辑混乱，调试成本上升；
4. **标签传递注意事项**：使用`include_role`时，如需给子任务传递标签，需使用`apply`参数：

    ```yaml
    - name: 动态包含角色并传递标签
      include_role:
        name: nginx
        apply:
          tags:
            - nginx
            - web
      tags:
        - always
    ```

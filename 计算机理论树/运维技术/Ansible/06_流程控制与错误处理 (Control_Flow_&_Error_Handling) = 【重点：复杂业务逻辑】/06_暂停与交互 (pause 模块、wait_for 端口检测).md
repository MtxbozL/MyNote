### 6.6.1 严格定义

暂停与交互机制，用于控制剧本的执行节奏，实现定时等待、人工确认审批、依赖服务就绪检测、交互式参数输入等功能，适配需要等待外部依赖就绪、人工介入审批、服务启动等待的复杂业务场景，核心通过`pause`模块与`wait_for`模块实现。

### 6.6.2 pause 模块：暂停与交互式确认

`pause`模块用于暂停剧本的执行，支持两种核心模式：**定时暂停**（等待固定时长后自动继续）、**交互式暂停**（等待人工确认后继续执行）。

#### 核心参数

|参数|核心作用|可选值|
|:--|:--|:--|
|`seconds`|暂停的秒数，定时暂停模式|整数|
|`minutes`|暂停的分钟数，定时暂停模式|整数|
|`prompt`|交互式暂停的提示文本，等待用户输入确认|字符串|
|`echo`|交互式输入时是否回显用户输入，默认 yes|yes/no|

#### 标准示例

```yaml
---
- name: pause模块示例
  hosts: all
  tasks:
    - name: 定时暂停30秒，等待服务初始化
      pause:
        seconds: 30

    - name: 交互式暂停，等待人工确认
      pause:
        prompt: "数据库备份已完成，请确认是否继续执行升级操作，按回车继续，Ctrl+C终止"

    - name: 敏感信息交互式输入，关闭回显
      pause:
        prompt: "请输入数据库管理员密码"
        echo: no
      register: db_password_input
      no_log: yes  # 禁止密码输出到日志
```

### 6.6.3 wait_for 模块：条件等待与就绪检测

`wait_for`模块用于等待指定条件满足后，再继续执行剧本，替代固定时长的定时暂停，实现更精准的就绪检测，避免因服务启动时间不确定导致的后续任务失败，是生产环境的核心常用模块。

#### 核心参数

| 参数             | 核心作用                  | 适用场景                                                                   |
| :------------- | :-------------------- | :--------------------------------------------------------------------- |
| `host`         | 待检测的主机地址，默认 127.0.0.1 | 端口连通性检测                                                                |
| `port`         | 待检测的 TCP 端口           | 服务端口就绪检测                                                               |
| `path`         | 待检测的文件路径              | 等待文件生成 / 删除                                                            |
| `state`        | 等待的目标状态               | 端口：`started`（端口可访问，默认）/`stopped`（端口关闭）；文件：`present`/`absent`/`drained` |
| `delay`        | 开始检测前的延迟等待时间，单位秒，默认 0 | 服务重启后先等待再检测                                                            |
| `timeout`      | 检测超时时间，单位秒，默认 300     | 最长等待时间，超时则任务失败                                                         |
| `search_regex` | 匹配文件内容的正则表达式          | 等待日志文件中出现指定启动成功关键字                                                     |
| `sleep`        | 两次检测之间的间隔时间，单位秒，默认 1  | 检测频率控制                                                                 |

#### 全场景实战示例

```yaml
---
- name: wait_for模块就绪检测示例
  hosts: webservers
  become: yes
  tasks:
    - name: 重启Nginx服务
      service:
        name: nginx
        state: restarted

    - name: 等待Nginx 80端口可访问，超时60秒
      wait_for:
        host: 127.0.0.1
        port: 80
        state: started
        timeout: 60
        delay: 2  # 重启后先等待2秒再开始检测

    - name: 等待应用启动成功日志出现
      wait_for:
        path: /var/log/app/app.log
        search_regex: "Application started successfully"
        timeout: 120

    - name: 等待旧进程完全退出，端口关闭
      wait_for:
        port: 8080
        state: stopped
        timeout: 30

    - name: 等待PID文件生成
      wait_for:
        path: /var/run/nginx.pid
        state: present
        timeout: 30
```

### 6.6.4 补充模块：wait_for_connection

专门用于等待被控节点的 Ansible 网络连接就绪，适用于服务器重启后，等待 SSH 服务恢复、被控节点可连接的场景，替代`wait_for`端口检测，更贴合 Ansible 的连接逻辑：

```yaml
tasks:
  - name: 重启服务器
    reboot:

  - name: 等待服务器重启完成，Ansible可正常连接
    wait_for_connection:
      delay: 30
      timeout: 300
```

### 6.6.5 最佳实践

1. **优先使用条件等待**：生产环境优先使用`wait_for`模块做条件就绪检测，替代固定时长的`pause`定时暂停，避免无效等待或等待时间不足导致的任务失败；
2. **超时时间合理设置**：必须设置合理的`timeout`超时时间，避免剧本无限等待；
3. **敏感信息保护**：交互式输入密码等敏感信息时，必须设置`echo: no`关闭回显，同时配合`no_log: yes`禁止日志输出；
4. **审批场景适配**：生产环境的高危操作（如数据库升级、配置回滚），必须添加交互式确认步骤，避免误执行；
5. **前置延迟优化**：服务重启、服务器重启后，通过`delay`参数设置前置等待，避免无效的高频检测。

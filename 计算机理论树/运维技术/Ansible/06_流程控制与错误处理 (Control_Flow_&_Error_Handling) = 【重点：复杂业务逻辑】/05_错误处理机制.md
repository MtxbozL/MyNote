### 6.5.1 严格定义

Ansible 默认执行规则为「单任务执行失败，立即终止当前主机的后续任务执行」，错误处理机制用于打破该规则，实现异常捕获、容错处理、强制逻辑执行、失败条件自定义，提升剧本的稳定性、容错能力与可观测性，是生产环境高可用剧本的核心必备能力。本节严格覆盖大纲指定的`ignore_errors`、`force_handlers`、`Block/Rescue/Always`三大核心能力。

### 6.5.2 忽略错误：ignore_errors

#### 核心定义

通过`ignore_errors: yes`参数，强制 Ansible 忽略当前 Task 的执行失败，继续执行后续任务，适用于非核心、允许失败的任务，避免非关键任务失败导致整个剧本中断。

#### 标准示例

```yaml
---
- name: ignore_errors示例
  hosts: all
  tasks:
    - name: 检查系统是否安装特定软件
      command: which custom_software
      register: software_check
      ignore_errors: yes  # 命令执行失败（返回码非0）也不会中断剧本
      changed_when: no

    - name: 软件存在时输出版本信息
      command: custom_software --version
      when: software_check is succeeded
```

#### 注意事项与最佳实践

1. **禁止滥用**：仅用于非核心、允许失败的任务，禁止在核心业务任务中使用，避免掩盖关键错误；
2. **配合条件判断**：忽略错误后，必须通过`when`语句基于任务执行状态做分支处理，避免后续任务执行异常；
3. **范围控制**：仅在需要的单个 Task 上使用，禁止在 Play 全局开启忽略错误。

### 6.5.3 强制触发器执行：force_handlers

#### 核心定义

解决 Handlers 的核心痛点：Play 中任意 Task 执行失败，默认会中断 Play，已 notify 但未执行的 Handlers 不会被执行。通过`force_handlers: yes`参数，可强制 Ansible 即使 Play 执行失败，也会执行所有已被 notify 的 Handlers，确保配置变更后对应的服务重启 / 重载操作一定会执行。

#### 标准示例

```yaml
---
- name: force_handlers强制触发器执行示例
  hosts: webservers
  become: yes
  force_handlers: yes  # 全局开启强制触发器执行
  tasks:
    - name: 更新Nginx配置文件
      template:
        src: nginx.conf.j2
        dest: /etc/nginx/nginx.conf
      notify: 重载Nginx  # 配置变更后触发Handler

    - name: 执行后续校验任务（该任务会执行失败）
      command: /bin/false  # 强制返回非0退出码，模拟任务失败

  handlers:
    - name: 重载Nginx
      service:
        name: nginx
        state: reloaded
```

> 执行说明：即使后续校验任务执行失败，Play 中断，已 notify 的「重载 Nginx」Handler 仍会被执行，确保配置变更生效。

#### 配置方式

1. **Play 级全局配置**：在 Play 中定义`force_handlers: yes`，对当前 Play 所有 Handler 生效；
2. **全局默认配置**：在`ansible.cfg`中设置`force_handlers = True`，对所有 Playbook 全局生效。

### 6.5.4 结构化异常处理：Block/Rescue/Always

#### 核心定义

Block 是 Ansible 的任务分组机制，将多个 Task 打包为一个逻辑块，配合`Rescue`和`Always`实现完整的结构化异常处理，完全对应编程范式中的`Try-Catch-Finally`逻辑：

- **Block**：对应`Try`块，正常执行的核心任务列表，块内任意 Task 执行失败，会立即跳转到 Rescue 块；
- **Rescue**：对应`Catch`块，仅当 Block 内的任务执行失败时才会执行，用于异常处理、故障回滚、错误上报；
- **Always**：对应`Finally`块，无论 Block 执行成功还是失败，都会无条件执行，用于清理环境、释放资源、收尾操作。

#### 标准语法与实战示例

```yaml
---
- name: Block/Rescue/Always结构化异常处理示例
  hosts: dbservers
  become: yes
  tasks:
    - name: 数据库备份与升级流程
      block:
        # Try块：核心业务流程，任意步骤失败则跳转到Rescue
        - name: 停止数据库服务
          service:
            name: mysqld
            state: stopped

        - name: 执行数据库备份
          shell: mysqldump -u root --all-databases > /data/backup/mysql_full_backup.sql
          args:
            creates: /data/backup/mysql_full_backup.sql

        - name: 升级数据库软件包
          yum:
            name: mysql-community-server
            state: latest

        - name: 启动数据库服务
          service:
            name: mysqld
            state: started
            enabled: yes

      rescue:
        # Catch块：Block执行失败时执行，用于故障回滚
        - name: 数据库升级失败，执行回滚操作
          debug:
            msg: "数据库升级失败，开始回滚，恢复备份数据"

        - name: 恢复数据库备份
          shell: mysql -u root < /data/backup/mysql_full_backup.sql

        - name: 恢复原版本数据库服务
          service:
            name: mysqld
            state: started
            enabled: yes

      always:
        # Finally块：无论成功失败都会执行，用于收尾清理
        - name: 清理临时备份文件
          file:
            path: /tmp/mysql_backup.tmp
            state: absent

        - name: 输出流程执行结果
          debug:
            msg: "数据库升级流程执行完成，最终状态：{{ '成功' if ansible_failed_task is undefined else '失败回滚' }}"
```

#### 进阶特性

1. **Block 级批量配置**：Block 块支持批量定义`when`条件、`become`提权、`tags`标签、`loop`循环，块内所有 Task 自动继承，减少冗余代码；
2. **异常信息获取**：Rescue 块中可通过`ansible_failed_task`获取失败的 Task 信息，`ansible_failed_result`获取失败的执行结果，用于错误上报与定位；
3. **嵌套 Block**：支持 Block 块嵌套，实现多层级的异常处理，适配复杂业务流程。

### 6.5.5 补充错误处理核心参数

|参数|核心作用|适用场景|
|:--|:--|:--|
|`failed_when`|自定义任务失败的条件，基于返回码、输出内容判断任务是否失败|命令返回码为 0 但实际执行失败的场景，自定义失败规则|
|`changed_when`|自定义任务变更状态的条件，第四章已详解，配合错误处理实现状态精准控制|无变更的查询类任务，避免无意义的 changed 状态|
|`any_errors_fatal`|任意主机的任务执行失败，立即终止所有主机的 Play 执行|强一致性要求的集群部署场景，单节点失败则全流程终止|
|`max_fail_percentage`|允许的任务失败主机比例，超过该比例才终止 Play 执行|大规模集群场景，允许少量节点失败不影响整体流程|
|`delegate_to`|任务失败时，委派到其他节点执行告警上报操作|故障时触发钉钉 / 邮件告警|

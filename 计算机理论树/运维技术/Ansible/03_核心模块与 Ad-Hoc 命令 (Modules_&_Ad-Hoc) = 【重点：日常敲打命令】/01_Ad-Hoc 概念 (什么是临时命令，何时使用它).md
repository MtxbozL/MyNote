## 3.1 Ad-Hoc 命令核心概念

### 3.1.1 严格定义

Ad-Hoc 命令是 Ansible 提供的**单任务、一次性、无持久化脚本**的自动化执行指令，通过终端直接调用，无需编写 YAML 格式的 Playbook 文件，底层完全依托 Ansible 模块体系实现，是快速完成批量运维操作的核心工具。

其核心设计定位是：解决简单、临时、高频的批量操作需求，与 Playbook 形成互补 ——Playbook 面向复杂、多步骤、可复用、需版本控制的自动化流程，Ad-Hoc 面向单次、单任务、快速落地的批量操作。

### 3.1.2 适用与不适用场景

|适用场景|不适用场景|
|:--|:--|
|批量主机状态查询（磁盘、内存、进程、负载）|多步骤复杂业务流程编排|
|紧急批量故障处理与配置修复|需要条件判断、循环、异常处理的逻辑|
|单任务批量配置下发（软件安装、服务启停）|需要长期复用、版本控制的标准化流程|
|模块功能快速验证与测试|需配合触发器、变量渲染的动态配置|
|批量文件分发与收集|企业级合规审计、可追溯的自动化发布流程|

### 3.1.3 标准语法结构与核心参数

Ad-Hoc 命令的标准语法格式如下，所有参数均有严格的层级与优先级规则：

```bash
ansible <host-pattern> [核心选项]
```

#### 1. 核心必填项：<host-pattern> 目标主机匹配模式


用于指定 Ad-Hoc 命令的执行目标范围，基于 Inventory 资产清单匹配，支持全场景匹配规则：

- 全量匹配：`all`，匹配 Inventory 中所有主机
- 分组匹配：`webservers`，匹配指定分组的所有主机
- 单主机匹配：`192.168.1.101`、`web01.example.com`，匹配单个主机
- 多组组合匹配：
    
    - 并集：`webservers:dbservers`，匹配两个分组的所有主机
    - 差集：`webservers:!dbservers`，匹配 webservers 中不在 dbservers 的主机
    - 交集：`webservers:&dbservers`，匹配同时属于两个分组的主机
    
- 通配符匹配：`192.168.1.*`、`*.example.com`，匹配符合通配符规则的主机
- 正则匹配：`~web\d+\.example\.com`，以`~`开头标识正则表达式

#### 2. 核心选项参数

|参数|全称|核心作用|
|:--|:--|:--|
|`-m`|`--module-name`|指定调用的 Ansible 模块，**默认值为 command 模块**|
|`-a`|`--args`|传递给模块的参数列表，以空格分隔的键值对格式|
|`-i`|`--inventory`|指定 Inventory 资产清单文件 / 目录 / 动态脚本路径|
|`-b`|`--become`|开启权限提升（提权），对应 become: yes 配置|
|`-u`|`--user`|指定远程连接被控节点的系统用户名|
|`-f`|`--forks`|指定并行执行的最大并发数，默认值为 5|
|`-C`|`--check`|干跑 / 模拟执行，不实际修改被控节点系统状态，用于变更预校验|
|`-v/-vvv/-vvvv`|`--verbose`|详细输出模式，-v 显示基础结果，-vvv 显示完整执行细节，-vvvv 开启 SSH 连接调试|
|`--list-hosts`|-|仅列出匹配的目标主机列表，不执行任何实际操作，用于范围校验|

#### 3. 标准示例与拆解

bash

运行

```
ansible webservers -m command -a "df -h" -i ./inventory/hosts -u ops -b -f 10
```

- 目标范围：Inventory 中 webservers 分组的所有主机
- 调用模块：command 命令执行模块
- 模块参数：执行`df -h`系统命令
- 资产清单：指定当前目录下的 inventory/hosts 文件
- 连接用户：远程连接用户为 ops
- 权限配置：开启 sudo 提权
- 并发配置：最大并发数为 10

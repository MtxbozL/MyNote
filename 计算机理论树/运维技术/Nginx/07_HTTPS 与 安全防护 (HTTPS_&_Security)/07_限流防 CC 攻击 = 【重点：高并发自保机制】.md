
CC 攻击（Challenge Collapsar）是一种应用层 DDoS 攻击，攻击者通过控制大量肉鸡 / 代理 IP，向服务端发起高频 HTTP 请求，耗尽服务端的 CPU、内存、连接数资源，导致正常用户无法访问。Nginx 的限流能力是抵御 CC 攻击的核心自保机制，基于标准内置模块实现，分为**请求速率限流**与**并发连接数限流**两大类，是本章的核心重点。

### 7.1 限制请求速率：limit_req 模块

请求速率限流由**ngx_http_limit_req_module**标准模块提供，基于**漏桶算法**实现，核心作用是限制单个客户端 IP（或其他维度）的 HTTP 请求处理速率，将请求频率控制在设定的阈值内，抵御高频 CC 攻击。

#### 7.1.1 漏桶算法核心原理

漏桶算法的核心逻辑是：请求像水一样进入漏桶，漏桶以固定的速率匀速出水（处理请求），当进水速率超过出水速率，且漏桶缓存满时，多余的水（请求）会被直接丢弃。该算法可严格限制请求的处理速率，平滑突发流量，防止服务被突发高频请求打垮。

#### 7.1.2 核心指令与语法

##### 1. limit_req_zone 指令

用于在 http 全局块定义限流的共享内存区域、限流维度、速率阈值，是限流的基础配置，仅可在 http 块中配置。

**官方语法**：`limit_req_zone key zone=name:size rate=rate [sync];`

**核心参数解析**：

|参数|功能说明|
|---|---|
|`key`|限流维度，即基于什么维度进行限流，最常用的是客户端真实 IP`$binary_remote_addr`，也可使用`$server_name`、`$request_uri`等变量|
|`zone=name:size`|定义共享内存区域的名称与大小，名称自定义，大小决定可存储的限流记录数量；1MB 内存可存储约 16000 个 32 位 IP 的限流状态|
|`rate=rate`|定义允许的最大请求速率，支持单位`r/s`（每秒请求数）、`r/m`（每分钟请求数）|

##### 2. limit_req 指令

用于在 server、location 块中启用限流规则，引用`limit_req_zone`定义的共享内存区域，配置突发流量处理策略。

**官方语法**：`limit_req zone=name [burst=number] [nodelay | delay=number];`

**核心参数解析**：

|参数|功能说明|
|---|---|
|`zone=name`|引用 http 块中定义的限流共享内存区域名称|
|`burst=number`|定义突发流量的最大缓存数量，超过速率阈值但未超过 burst 的请求会被放入缓存队列，等待处理|
|`nodelay`|配合 burst 使用，缓存队列中的请求会被立即处理，无需排队等待，超过 burst 的请求直接返回 503|
|`delay=number`|配置延迟处理的阈值，超过该数量的请求会被延迟处理，默认值等于 burst 值|

#### 7.1.3 标准配置示例

##### 1. 基础限流配置

```nginx
http {
    # 定义限流规则：基于客户端二进制IP，共享内存10MB，速率限制为10r/s
    limit_req_zone $binary_remote_addr zone=REQ_LIMIT:10m rate=10r/s;

    server {
        listen 443 ssl http2;
        server_name www.example.com;

        # 全站启用限流，突发缓存20个请求，nodelay立即处理
        location / {
            limit_req zone=REQ_LIMIT burst=20 nodelay;
            root /var/www/html;
            index index.html;
        }
    }
}
```

> 配置说明：单个 IP 每秒最多处理 10 个请求，突发流量最多缓存 20 个请求，超过的请求直接返回 503 Service Unavailable 状态码。

##### 2. 接口精准限流配置

```nginx
http {
    # 登录接口限流：单IP每分钟最多5次请求，防止暴力破解
    limit_req_zone $binary_remote_addr zone=LOGIN_LIMIT:5m rate=5r/m;
    # 短信接口限流：单IP每小时最多10次请求
    limit_req_zone $binary_remote_addr zone=SMS_LIMIT:5m rate=10r/h;
    # 全局API限流：单IP每秒20次请求
    limit_req_zone $binary_remote_addr zone=API_LIMIT:10m rate=20r/s;

    server {
        listen 443 ssl http2;
        server_name api.example.com;

        # 登录接口严格限流
        location /api/user/login {
            limit_req zone=LOGIN_LIMIT burst=1 nodelay;
            proxy_pass http://user_backend;
        }

        # 短信接口严格限流
        location /api/sms/send {
            limit_req zone=SMS_LIMIT burst=1 nodelay;
            proxy_pass http://sms_backend;
        }

        # 通用API接口限流
        location /api/ {
            limit_req zone=API_LIMIT burst=30 nodelay;
            proxy_pass http://api_backend;
        }
    }
}
```

##### 3. 白名单配置

```nginx
http {
    limit_req_zone $binary_remote_addr zone=REQ_LIMIT:10m rate=10r/s;
    # 定义白名单变量，默认值为0
    map $binary_remote_addr $limit_whitelist {
        default 0;
        # 内网IP段加入白名单
        192.168.1.0/24 1;
        # 办公网公网IP加入白名单
        223.xxx.xxx.xxx/32 1;
    }
    # 白名单IP不执行限流
    limit_req_status 429;
}

server {
    listen 443 ssl http2;
    server_name www.example.com;

    location / {
        # 非白名单IP启用限流
        if ($limit_whitelist = 0) {
            limit_req zone=REQ_LIMIT burst=20 nodelay;
        }
        root /var/www/html;
    }
}
```

### 7.2 限制并发连接数：limit_conn 模块

并发连接数限流由**ngx_http_limit_conn_module**标准模块提供，核心作用是限制单个客户端 IP（或其他维度）与服务端建立的并发连接数量，抵御慢速连接攻击、TCP 连接耗尽攻击。

#### 7.2.1 核心指令与语法

##### 1. limit_conn_zone 指令

用于在 http 全局块定义并发限流的共享内存区域、限流维度，仅可在 http 块中配置。

**官方语法**：`limit_conn_zone key zone=name:size;`

**核心参数解析**：

- `key`：限流维度，最常用的是`$binary_remote_addr`（客户端真实 IP），也可使用`$server_name`限制单个域名的总并发连接数；
- `zone=name:size`：定义共享内存区域的名称与大小，1MB 内存可存储约 32000 个 32 位 IP 的连接状态。

##### 2. limit_conn 指令

用于在 server、location 块中启用并发限流规则，引用`limit_conn_zone`定义的共享内存区域，配置最大并发连接数阈值。

**官方语法**：`limit_conn zone number;`

**核心参数**：

- `zone`：引用 http 块中定义的并发限流共享内存区域名称；
- `number`：允许的最大并发连接数，超过该阈值的连接会被直接拒绝，返回 503 状态码。

#### 7.2.2 标准配置示例

```nginx
http {
    # 定义单IP并发连接数限制：共享内存10MB
    limit_conn_zone $binary_remote_addr zone=CONN_LIMIT:10m;
    # 定义单个域名的总并发连接数限制
    limit_conn_zone $server_name zone=SERVER_CONN_LIMIT:10m;

    # 配置限流拒绝的状态码，默认503，推荐使用429 Too Many Requests
    limit_conn_status 429;
    # 限流请求记录到error_log，级别为notice
    limit_conn_log_level notice;

    server {
        listen 443 ssl http2;
        server_name www.example.com;

        # 限制整个站点单IP最多同时建立50个并发连接
        limit_conn CONN_LIMIT 50;
        # 限制整个站点总并发连接数不超过10000
        limit_conn SERVER_CONN_LIMIT 10000;

        # 下载资源单独限制并发连接数
        location ~* \.(zip|tar.gz|mp4|iso)$ {
            # 单IP最多同时建立2个下载连接
            limit_conn CONN_LIMIT 2;
            # 限制单连接下载速度1MB/s
            limit_rate 1m;
            root /var/www/download;
        }
    }
}
```

### 7.3 关键注意事项与最佳实践

1. ** 限流维度必须使用：禁止使用remote_addr`作为限流key，`是二进制格式的地址，占用字节（）字节（），远小于字符串格式的remote_addr`，可大幅节省共享内存空间，提升限流效率。
2. **反向代理场景真实 IP 适配**：当 Nginx 处于 CDN、负载均衡之后时，必须通过`real_ip`模块获取客户端真实 IP，否则限流 key 为代理节点 IP，导致限流失效，误伤正常用户。
3. **限流阈值合理设置**：需根据业务正常访问量、服务器性能设置阈值，避免阈值过低导致正常用户被拦截，阈值过高无法抵御攻击；建议通过压测确定服务的最大承载能力，设置阈值为最大承载的 70%。
4. **状态码规范**：限流拒绝的状态码推荐使用`429 Too Many Requests`，而非默认的 503，符合 HTTP 协议规范，便于客户端区分限流与服务端异常。
5. **白名单配置**：必须将内网 IP、监控节点、办公网 IP、合作方 IP 加入限流白名单，避免正常业务请求被拦截。
6. **日志与监控**：开启限流日志记录，配合监控系统统计限流触发次数，及时发现攻击行为与异常流量；同时配置自定义限流提示页面，提升用户体验。
7. **多层限流防护**：生产环境建议同时配置请求速率限流与并发连接数限流，配合 IP 黑白名单、基础认证等防护措施，构建多层防护体系，提升抗攻击能力。


Nginx 采用**Master-Worker 多进程 + 事件驱动**的异步非阻塞架构，整体架构分为核心进程体系与事件处理模型两大核心部分，是其高并发、高可用能力的底层支撑。

### 1.2.1 Master-Worker 多进程模型

Nginx 启动后会生成两类核心进程，辅以缓存管理相关的辅助进程，各进程职责完全解耦，具体如下：

1. **Master 主进程**：特权进程，以 root 权限启动（如需绑定 1024 以下端口），核心职责为：读取并校验配置文件合法性；管理所有 Worker 进程，负责 Worker 进程的创建、销毁与信号转发；接收外部管理信号，实现配置热加载、二进制热升级、服务启停等核心操作；不处理任何客户端请求，仅负责集群管控与高可用保障。
2. **Worker 工作进程**：普通用户权限进程，由 Master 进程 fork 生成，核心职责为：处理实际的客户端网络请求；单进程内采用单线程模型，通过 IO 多路复用接口管理所有并发连接；进程间相互独立，单个 Worker 进程崩溃不会影响其他进程与整体服务可用性；Worker 进程数量通常配置为等于或略大于服务器 CPU 物理核心数，最大化利用多核 CPU 性能。
3. **缓存辅助进程**：包含 Cache Loader 进程与 Cache Manager 进程。Cache Loader 进程仅在 Nginx 启动时运行，负责将磁盘上的缓存元数据加载至内存；Cache Manager 进程周期运行，负责缓存内容的过期淘汰、磁盘空间校验、缓存索引维护，保障缓存体系的稳定运行。

### 1.2.2 事件驱动与异步非阻塞 IO 模型

Nginx 的事件驱动模型，将客户端连接的建立、数据读写、连接关闭、异常处理等全流程操作均抽象为独立事件，通过 IO 多路复用接口统一管理所有事件，核心逻辑如下：

1. Worker 进程启动后，会向 IO 多路复用接口注册监听套接字的读事件；
2. 当有客户端发起连接请求，监听套接字产生就绪事件，Worker 进程接收连接并生成客户端套接字，向 IO 多路复用接口注册该套接字的读写事件；
3. 当客户端套接字产生就绪事件，Worker 进程立即处理对应请求；对于磁盘 IO、上游代理转发等阻塞操作，Worker 进程发起异步 IO 请求后立即释放 CPU，继续处理其他就绪事件，无需阻塞等待 IO 完成；
4. 异步 IO 操作完成后，产生就绪事件通知 Worker 进程，执行后续回调处理，完成整个请求生命周期。

### 1.2.3 epoll IO 多路复用机制

epoll 是 Linux 平台下 Nginx 默认使用的 IO 多路复用接口，是其高并发能力的核心底层支撑，区别于传统 select/poll 接口，核心优势如下：

1. **事件管理机制**：采用红黑树管理所有待监听的文件描述符，文件描述符的增删改查时间复杂度为 O (log n)，突破了 select 单进程最大文件描述符限制；
2. **就绪事件获取**：采用双向链表存储就绪事件，仅返回已触发的就绪事件，无需像 select/poll 一样线性扫描全量文件描述符，事件获取时间复杂度稳定为 O (1)，高并发场景下性能无显著衰减；
3. **工作模式**：支持 LT（水平触发，默认）与 ET（边缘触发）两种模式，ET 模式可进一步减少事件触发次数，提升 IO 处理效率。

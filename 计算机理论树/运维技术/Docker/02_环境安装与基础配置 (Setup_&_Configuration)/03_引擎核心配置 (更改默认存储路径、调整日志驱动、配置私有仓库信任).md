
本节针对 Docker 引擎的生产级核心配置进行讲解，覆盖存储路径、日志驱动、私有仓库信任三大核心场景，所有配置均通过`daemon.json`文件实现，是生产环境部署必须完成的合规性配置。

### 3.1 更改默认存储路径

#### 3.1.1 配置背景与原理

Docker 默认存储根目录为`/var/lib/docker`，所有镜像、容器、数据卷、网络配置、元数据等全量数据均存储在该目录下。生产环境中，系统盘通常容量有限，Docker 数据持续增长会导致系统盘占满，引发主机故障，因此必须将存储路径修改至独立的数据盘分区。

Docker 存储根目录通过`data-root`字段控制，**注意**：老版本 Docker 使用的`graph`字段已被废弃，新版本必须使用`data-root`字段，避免配置失效。

#### 3.1.2 配置步骤（Linux 生产环境）

1. **停止 Docker 服务，确保数据一致性**

    ```bash
    sudo systemctl stop docker
    sudo systemctl stop docker.socket
    ```
    
2. **创建新的存储目录，同步原有数据**
    
    假设新的存储路径为`/data/docker`，需确保该路径所在分区为 ext4/xfs 文件系统，支持 Overlay2 驱动：

    ```bash
    # 创建新存储目录
    sudo mkdir -p /data/docker
    # 同步原有数据至新目录，避免数据丢失
    sudo rsync -av /var/lib/docker/ /data/docker/
    ```
    
3. **修改 daemon.json 配置文件**

    ```json
    {
      "data-root": "/data/docker"
    }
    ```
    
4. **重启 Docker 服务，验证配置生效**

    ```bash
    sudo systemctl daemon-reload
    sudo systemctl start docker
    ```
    

#### 3.1.3 注意事项

- 新存储路径的文件系统必须支持 Overlay2 驱动，生产环境严禁使用 NTFS、FAT32 等非 Linux 原生文件系统；
- 若主机开启 SELinux，需为新存储目录配置正确的 SELinux 上下文，否则会导致 Docker 服务启动失败；
- 配置生效且验证无误后，可删除原有`/var/lib/docker`目录，释放系统盘空间。

### 3.2 调整日志驱动

#### 3.2.1 配置背景与原理

Docker 通过**日志驱动（Logging Driver）** 收集容器的标准输出（stdout）与标准错误（stderr）日志，默认日志驱动为`json-file`，其默认配置无日志轮转限制，会导致日志文件无限增长，最终占满磁盘空间，是生产环境最高频的故障点之一。

Docker 支持多种日志驱动，包括`json-file`（默认）、`local`、`syslog`、`journald`、`fluentd`、`gelf`等，其中`json-file`与`local`适用于本地日志存储，其他驱动适用于对接集中化日志平台。

#### 3.2.2 生产级日志轮转配置

在`daemon.json`中通过`log-driver`指定全局默认日志驱动，通过`log-opts`配置日志轮转规则，示例如下：

```json
{
  "log-driver": "json-file",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3",
    "compress": "true"
  }
}
```

参数说明：

- `max-size`：单个日志文件的最大容量，超过该阈值则触发日志轮转；
- `max-file`：最多保留的日志文件数量，超过该数量则自动删除最旧的日志文件；
- `compress`：是否自动压缩轮转后的旧日志文件，开启后可大幅降低磁盘占用。

#### 3.2.3 注意事项

- 该配置为 Docker 引擎的全局默认配置，仅对新建容器生效，已有容器需重建后才能应用新的日志配置；
- 容器启动时可通过`--log-driver`与`--log-opt`参数单独配置日志规则，覆盖全局默认配置；
- 生产环境必须配置日志轮转，无论使用哪种日志驱动，严禁使用无限制的默认配置。

### 3.3 配置私有仓库信任

#### 3.3.1 配置背景与原理

Docker 默认采用 TLS 加密传输，仅信任使用受信任 SSL 证书的 HTTPS 镜像仓库，对于 HTTP 协议的内部私有仓库，会直接拒绝访问并抛出`http: server gave HTTP response to HTTPS client`错误。

通过`insecure-registries`字段可配置私有仓库的信任列表，Docker 会对列表内的仓库跳过 TLS 证书验证，支持 HTTP 协议访问，是内部测试环境搭建私有仓库的必备配置。

#### 3.3.2 配置规范

在`daemon.json`中通过`insecure-registries`字段配置私有仓库地址，格式为 JSON 数组，示例如下：

```json
{
  "insecure-registries": [
    "192.168.1.100:5000",
    "harbor.internal.example.com"
  ]
}
```

配置完成后，重启 Docker Daemon 即可生效。

#### 3.3.3 安全警示

- `insecure-registries`配置会跳过 TLS 证书验证，存在中间人攻击的安全风险，**仅可用于内部隔离的测试环境**；
- 企业生产环境必须为私有仓库配置受信任的 SSL 证书，启用 HTTPS 协议，严禁使用该配置绕过 TLS 验证；
- 严禁将公网地址加入信任列表，避免数据泄露与安全攻击。

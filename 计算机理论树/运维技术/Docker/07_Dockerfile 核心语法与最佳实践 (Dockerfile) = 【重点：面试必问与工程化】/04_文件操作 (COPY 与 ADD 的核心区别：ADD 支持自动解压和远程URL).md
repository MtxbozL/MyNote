
COPY 与 ADD 均用于将构建上下文中的文件 / 目录复制到镜像的文件系统中，是向镜像中注入应用代码、配置文件、依赖文件的核心指令，二者的核心区别是面试高频考点。

### 4.1 COPY 指令（官方优先推荐）

COPY 是功能专一的文件复制指令，仅实现文件复制，无额外特性，行为可预测，无隐性副作用，是常规场景的唯一推荐方案。

- 标准语法：

    ```dockerfile
    COPY [--chown=<user>:<group>] [--chmod=<mode>] [--from=<阶段名|镜像>] <源路径>... <目标路径>
    ```
    
- 核心参数详解：
    
    1. `--chown`：设置复制到容器内的文件 / 目录的所属用户与用户组，默认是`root:root`，生产环境降权运行时必须配置；
    2. `--chmod`：设置复制到容器内的文件权限，如`--chmod=0755`，避免复制后权限不符合要求；
    3. `--from`：指定复制源，可从多阶段构建的前序阶段、或远程镜像中复制文件，是多阶段构建的核心语法，无需完整拉取镜像，仅复制指定文件。
    
- 核心路径规则：
    
    1. 源路径必须是构建上下文内的相对路径，不能是上下文外的路径（如`../xxx`），因为 Docker Daemon 只能访问构建上下文内的文件；
    2. 目标路径必须是容器内的绝对路径，或相对于 WORKDIR 的相对路径；若目标路径不存在，会自动创建所有层级的目录；
    3. 源路径是目录时，会复制目录内的所有内容到目标路径，而非复制目录本身；需保留目录本身时，需在目标路径末尾指定目录名。
    

### 4.2 ADD 指令

ADD 具备 COPY 的所有文件复制能力，同时额外支持两个核心特性，也是其隐性副作用的来源。

- 标准语法：

    ```dockerfile
    ADD [--chown=<user>:<group>] <源路径>... <目标路径>
    ```
    
- 两个核心扩展特性：
    
    1. **本地压缩包自动解压**：若源路径是本地的 tar、gzip、bzip2、xz 等格式的压缩归档文件，ADD 会自动将其解压到目标路径，行为与`tar -x`一致；**注意：远程 URL 的压缩包不会自动解压，仅会下载复制**；
    2. **远程 URL 资源下载**：源路径支持 HTTP/HTTPS 协议的远程 URL，ADD 会自动下载该 URL 的资源，复制到目标路径，并设置文件权限为 600。
    
- 官方禁用规范：
    
    1. 除了**需要本地压缩包自动解压**的场景外，**所有其他场景必须优先使用 COPY 指令**，避免 ADD 的隐性特性导致的不可预测行为；
    2. **严禁使用 ADD 下载远程 URL 资源**：远程 URL 的资源会永久保留在镜像分层中，即使后续 RUN 指令删除，仍会保留在历史分层中，导致镜像体积膨胀；同时无法校验资源的完整性与安全性，存在供应链攻击风险。
        
        正确替代方案：

    ```dockerfile
    RUN curl -fSL https://example.com/software.tar.gz -o /tmp/software.tar.gz && \
        echo "xxx 校验和" /tmp/software.tar.gz | sha256sum -c - && \
        tar -xzf /tmp/software.tar.gz -C /usr/local/bin && \
        rm -rf /tmp/software.tar.gz
    ```
    

### 4.3 核心差异对比（面试必问）

|对比维度|COPY|ADD|
|---|---|---|
|核心功能|仅实现构建上下文到镜像的文件复制|完整支持 COPY 的所有功能，额外支持本地压缩包自动解压、远程 URL 下载|
|行为可预测性|极高，无任何隐性副作用，仅执行复制|中，存在自动解压、下载的隐性行为，易出现非预期结果|
|构建缓存适配|优秀，仅当源文件内容变更时才会失效|一般，远程 URL 资源的内容变更无法被缓存识别，每次构建都会重新下载|
|官方推荐等级|最高，所有常规场景优先使用|低，仅本地压缩包自动解压场景使用|
|多阶段构建支持|支持`--from`参数，可从其他阶段 / 镜像复制|不支持`--from`参数，无法用于多阶段构建产物复制|

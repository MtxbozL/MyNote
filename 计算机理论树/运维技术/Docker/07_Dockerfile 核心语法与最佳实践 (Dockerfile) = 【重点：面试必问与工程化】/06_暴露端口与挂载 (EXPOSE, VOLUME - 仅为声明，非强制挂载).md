
本节核心纠正两个高频认知误区：EXPOSE 与 VOLUME 均为**元数据声明指令，无强制执行效果**，仅作为文档化提示与默认行为补充，完全承接第四章端口映射、第五章数据持久化的核心内容。

### 6.1 EXPOSE 指令

用于**声明容器内服务监听的端口号与协议类型**，属于文档化的元数据声明，无任何实际的网络放行、端口映射效果。

- 标准语法：

    ```dockerfile
    EXPOSE <端口> [<端口>/<协议>...]
    ```
    
- 核心特性：
    
    1. 仅为元数据声明，不会自动创建宿主机到容器的端口映射，外部网络无法通过该声明直接访问容器端口，必须通过`docker run -p`参数显式配置端口映射才能实现外部访问；
    2. 支持 TCP/UDP 协议，默认是 TCP 协议，示例：`EXPOSE 80/tcp 53/udp`；
    3. 声明的端口会写入镜像元数据，`docker run -P`随机端口映射时，会基于 EXPOSE 声明的端口自动创建映射。
    
- 生产规范：**必须为镜像内服务监听的所有端口添加 EXPOSE 声明**，作为标准化的文档化说明，便于运维人员识别镜像的端口监听需求。

### 6.2 VOLUME 指令

用于**声明容器内的持久化数据路径**，属于元数据声明，提示用户该路径需要挂载数据卷实现持久化，避免用户忘记挂载导致数据丢失，**不会自动创建命名卷，不会强制绑定宿主机路径**。

- 标准语法：

    ```dockerfile
    VOLUME ["<容器内路径1>", "<容器内路径2>..."]
    ```
    
- 核心特性：
    
    1. 仅为元数据声明，当容器启动时未为该路径挂载任何卷，Docker 会自动为该路径创建一个匿名 Volume，保证该路径的写入不会进入容器可写层；
    2. **核心陷阱（高频踩坑点）**：一旦在 Dockerfile 中声明了 VOLUME，后续所有 Dockerfile 指令对该路径的修改都不会生效。因为 Docker 在构建时，遇到 VOLUME 指令会创建一个临时匿名卷，执行后续指令时，对该路径的修改会写入临时卷，构建完成后临时卷会被丢弃，最终镜像中不会保留该修改。
        
        反例：

    ```dockerfile
    FROM mysql:8.0
    # 声明VOLUME
    VOLUME /var/lib/mysql
    # 该修改不会生效，写入了临时卷，构建完成后被丢弃
    RUN echo "自定义配置" > /var/lib/mysql/my.cnf
    ```
    
- 生产规范：
    
    1. **必须为有状态应用的核心数据路径声明 VOLUME**，如 MySQL 的`/var/lib/mysql`、Redis 的`/data`，避免用户忘记挂载卷导致数据随容器销毁丢失；
    2. **严禁在 VOLUME 声明之后对该路径执行修改操作**，VOLUME 指令应放在 Dockerfile 的末尾，所有对该路径的修改操作之前；
    3. 生产环境启动容器时，必须为 VOLUME 声明的路径显式挂载命名 Volume，避免匿名卷的产生。


ENV 与 ARG 是 Dockerfile 中用于参数化构建、环境配置的两大核心指令，二者的生命周期、生效范围、安全特性存在本质差异，是面试高频考点。

### 2.1 ENV 环境变量

用于定义容器的环境变量，**在镜像构建阶段的所有后续指令中生效，同时在容器运行时持久保留**，是应用读取配置的核心方式之一。

- 标准语法：
    
    1. 单行单变量：`ENV <key> <value>`
    2. 键值对格式（生产推荐）：`ENV <key>=<value> <key2>=<value2> ...`
    
- 核心特性：
    
    1. 生命周期：构建时 + 运行时双阶段生效，容器内的应用可直接读取 ENV 定义的环境变量；
    2. 持久化：ENV 定义的变量会永久写入镜像元数据，可通过`docker inspect`查看，无法在运行时移除，仅可覆盖；
    3. 指令继承：后续的 RUN、CMD、ENTRYPOINT、COPY 等指令均可通过`${key}`格式引用 ENV 变量。
    
- 安全红线：**严禁在 ENV 中存储密码、密钥、Token 等敏感信息**，ENV 变量会永久保留在镜像历史中，可通过`docker history`轻易提取，引发敏感信息泄露。

### 2.2 ARG 构建时参数

用于定义**仅在镜像构建阶段生效的构建时参数**，不会带入容器运行时环境，是构建过程中动态传参的唯一标准化方式。

- 标准语法：

    ```dockerfile
    ARG <name>[=<default value>]
    ```
    
- 核心特性：
    
    1. 生命周期：仅在定义它的构建阶段内生效，多阶段构建中需在每个阶段重新定义；
    2. 动态传参：构建时可通过`docker build --build-arg <name>=<value>`覆盖默认值，无需修改 Dockerfile；
    3. 不持久化：ARG 参数不会写入最终镜像的元数据，无法在运行时容器中查看，安全性高于 ENV；
    4. 预定义 ARG：Docker 内置了一系列预定义 ARG 变量，如`HTTP_PROXY`、`HTTPS_PROXY`，构建时可直接传参，无需在 Dockerfile 中定义。
    
- 安全红线：**严禁通过 ARG 传递敏感信息**，即使 ARG 不会写入最终镜像，仍会保留在镜像的构建历史缓存中，可通过`docker history`查看，存在泄露风险；生产环境敏感信息传递，必须使用 Docker 18.09 + 支持的`--secret`构建密钥挂载机制，仅在构建时临时加载，不会写入任何镜像分层与历史。

### 2.3 核心差异对比（面试必问）

|对比维度|ENV|ARG|
|---|---|---|
|生效周期|构建阶段全流程 + 容器运行时|仅定义它的构建阶段内，运行时完全失效|
|持久化|永久写入镜像元数据，可通过`docker inspect`查看|不写入最终镜像元数据，仅保留在构建历史中|
|动态传参|运行时可通过`docker run -e`覆盖，构建时无法直接传参|构建时通过`--build-arg`覆盖，运行时无法修改|
|引用范围|所有后续指令均可引用，包括 RUN、CMD、ENTRYPOINT|仅构建阶段的指令可引用，CMD/ENTRYPOINT 运行时无法引用|
|敏感信息风险|极高，永久保留在镜像中|中，保留在构建历史中，严禁传递敏感信息|

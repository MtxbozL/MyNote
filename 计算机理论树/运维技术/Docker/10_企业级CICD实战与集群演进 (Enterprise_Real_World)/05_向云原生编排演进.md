
单主机 Docker 与 Docker Compose 完美适配开发测试环境与小规模单主机部署场景，但面对企业级大规模生产环境，存在无法解决的核心痛点：**无高可用故障自愈能力、无弹性扩缩容能力、无跨主机容器调度、无内置服务发现与负载均衡、无滚动更新与灰度发布能力**。

容器编排系统是解决上述痛点的核心方案，实现了大规模集群环境下容器的全生命周期自动化管理，是 Docker 技术向云原生架构演进的核心方向。

### 5.1 Docker Swarm（Docker 原生集群编排，了解即可）

Docker Swarm 是 Docker 官方原生的集群编排方案，与 Docker Engine 深度集成，原生兼容 Docker Compose 配置文件，学习成本极低，是 Docker 原生的集群解决方案。

#### 5.1.1 核心架构与特性

- **核心架构**：分为 Manager 节点（控制平面，负责集群状态管理、调度决策、API 服务）与 Worker 节点（数据平面，负责运行容器负载），支持多 Manager 节点高可用部署；
- **核心特性**：
    
    1. 与 Docker Engine 深度集成，原生 Docker CLI 即可操作，无需额外安装组件；
    2. 原生兼容 Docker Compose 文件，通过`docker stack deploy`命令一键部署多服务应用；
    3. 内置服务发现、DNS 解析、四层负载均衡、Overlay 跨主机网络；
    4. 支持声明式服务定义、副本数自动调度、故障自愈、滚动更新、滚动回滚；
    5. 原生支持 TLS 加密通信，节点间通信默认加密，安全开箱即用。
    

#### 5.1.2 市场现状与适用场景

Docker Swarm 目前已进入**维护模式**，Docker 官方已停止新功能迭代，市场占有率极低，仅适用于以下场景：

1. 小规模、简单的集群场景，团队已熟练掌握 Docker 与 Compose，希望以最低的学习成本实现集群化部署；
2. 轻量级边缘计算场景，资源受限，无法运行 Kubernetes 集群；
3. 对功能要求简单，仅需要基础的容器调度、故障自愈能力，无需复杂的云原生能力。

**生产环境规范**：大型企业、复杂微服务架构、大规模集群场景，不推荐使用 Docker Swarm，应选择 Kubernetes 作为容器编排平台。

### 5.2 Kubernetes (K8s)（容器编排的事实标准）

Kubernetes（简称 K8s）是 Google 开源、CNCF 毕业的容器编排系统，是目前云原生领域容器编排的**事实标准**，已成为企业级容器化部署的唯一主流选择，也是 Docker 学习的最终进阶目标。

#### 5.2.1 核心定位与 Docker 的关系

Kubernetes 的核心定位是**云原生应用的自动化编排平台**，负责容器的调度、部署、运维、扩缩容全生命周期管理，与 Docker 是互补而非竞争关系：

- Docker 专注于容器镜像的构建、打包与本地运行，是 Kubernetes 的底层容器运行时；
- Kubernetes 专注于大规模集群环境下的容器编排与调度，是 Docker 企业级大规模落地的核心载体；
- 技术演进：Kubernetes 已废弃 Docker Engine 作为容器运行时，转而使用符合 CRI（容器运行时接口）标准的 containerd（Docker 拆分出的容器运行时，前序章节已讲解），但完全兼容 Docker 构建的 OCI 标准镜像，Docker 依然是云原生环境下镜像构建与开发测试的首选工具。

#### 5.2.2 核心架构与核心概念

Kubernetes 采用控制平面 - 节点的分布式架构，与前序 Docker 知识点存在明确的对应关系，可实现学习路径的平滑过渡：

|Kubernetes 核心概念|对应 Docker 知识点|核心作用|
|---|---|---|
|Pod|容器 / 容器组|K8s 的最小调度单元，由一个或多个容器组成，共享 Network Namespace 与存储卷，对应 Docker 的容器运行环境|
|Deployment|docker run / Docker Compose 服务|无状态应用的声明式编排控制器，定义副本数、更新策略、容器模板，实现应用的滚动更新、扩缩容、故障自愈|
|StatefulSet|有状态容器服务|有状态应用的编排控制器，支持固定的网络标识、稳定的持久化存储、有序的部署与更新，适配数据库、消息队列等有状态服务|
|Service|docker port 映射 + 负载均衡|为一组 Pod 提供固定的访问入口、内置服务发现与四层负载均衡，解决 Pod 动态 IP 变化的访问问题|
|Ingress|Nginx 反向代理|七层 HTTP/HTTPS 流量入口，实现域名路由、SSL 终止、流量切分等能力|
|ConfigMap/Secret|环境变量 / 绑定挂载配置文件|配置与敏感信息的管理组件，实现配置与镜像的解耦，无需修改镜像即可调整配置|
|PersistentVolume (PV/PVC)|Docker Volume|持久化存储的抽象模型，实现存储与应用生命周期解耦，支持多种存储后端|
|Namespace|Docker Namespace|集群内的资源隔离单元，实现多租户、多环境的资源隔离|

#### 5.2.3 为什么 K8s 成为事实标准

1. **极致的可扩展性**：通过 CRD、Operator、CSI、CNI 等扩展机制，可无限扩展能力，适配任何业务场景；
2. **完整的云原生生态**：拥有 CNCF 庞大的生态体系，覆盖服务网格、可观测性、安全、CI/CD、数据库中间件等全场景，是云原生技术的事实标准载体；
3. **企业级高可用能力**：原生支持多可用区部署、故障自愈、滚动更新、蓝绿发布、灰度发布、弹性扩缩容等企业级核心能力，满足大规模生产环境的高可用要求；
4. **声明式 API 与不可变基础设施**：完全遵循声明式配置理念，与 IaC、GitOps 等现代 DevOps 理念深度契合，实现基础设施的标准化、可追溯、可重复部署；
5. **跨环境一致性**：支持在私有云、公有云、边缘环境实现统一的部署与管理，彻底解决厂商锁定问题，实现混合云、多云架构的统一管控。

#### 5.2.4 Docker 到 Kubernetes 的学习路径

前序章节掌握的所有 Docker 核心知识，都是学习 Kubernetes 的必备基础，平滑进阶路径如下：

1. 扎实掌握 Docker 核心基础：镜像、容器、存储、网络、Dockerfile、底层原理；
2. 掌握 Docker Compose 多容器编排，理解声明式配置、服务依赖、多服务协同的核心逻辑；
3. 学习 Kubernetes 核心概念与架构，理解 Pod、Deployment、Service 等核心资源的作用与 Docker 知识点的对应关系；
4. 掌握 Kubernetes 的应用部署、配置管理、存储管理、网络管理核心能力；
5. 进阶学习 Kubernetes 的高级编排能力、可观测性、服务网格、安全体系，完成向云原生架构师的进阶。

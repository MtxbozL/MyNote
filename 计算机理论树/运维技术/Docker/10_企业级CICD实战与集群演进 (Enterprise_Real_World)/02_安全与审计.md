
容器安全是企业级 Docker 落地的核心前提，遵循**最小权限原则、默认拒绝、纵深防御、不可变基础设施**四大核心安全理念，所有安全机制均基于前序章节讲解的 Linux Namespace、Cgroups、Capabilities 等内核原生能力实现，覆盖镜像全生命周期、容器运行时、集群管控全链路的安全防护与审计。

### 2.1 镜像漏洞扫描 (Trivy)

容器镜像的安全漏洞是供应链攻击的核心入口，80% 以上的容器安全事件源于基础镜像、依赖包中的已知漏洞，镜像漏洞扫描是容器安全左移的第一道防线。

#### 2.1.1 Trivy 核心定位与能力

Trivy 是 Aqua Security 开源、CNCF 毕业的**容器与制品安全扫描工具**，是目前容器生态中应用最广泛的漏洞扫描引擎，Harbor、GitLab、Jenkins 等平台均原生集成，核心能力包括：

1. **全场景漏洞扫描**：支持操作系统包漏洞（Debian、Ubuntu、RHEL、Alpine 等）、编程语言依赖漏洞（Java、Go、Python、Node.js 等）、IaC 配置文件安全风险、Dockerfile 合规性检测；
2. **OCI 标准完全兼容**：支持所有符合 OCI 规范的容器镜像、镜像仓库、tar 包，与 Docker 原生完全适配；
3. **高性能与低资源占用**：扫描速度远快于同类工具，增量扫描仅检测变更内容，适合嵌入 CI/CD 流水线作为门禁卡点；
4. **SBOM 与合规支持**：支持生成软件物料清单（SBOM），兼容 SPDX、CycloneDX 标准，满足供应链安全合规要求。

#### 2.1.2 核心使用与生产级集成

- 基础扫描命令：

    ```bash
    # 扫描本地Docker镜像
    trivy image nginx:1.27.0-alpine
    # 仅扫描高危、严重漏洞，输出精简结果
    trivy image --severity HIGH,CRITICAL nginx:1.27.0-alpine
    # 扫描Dockerfile，检测配置风险
    trivy config Dockerfile
    # 扫描结果输出JSON格式，用于自动化流水线解析
    trivy image --format json --output scan-result.json nginx:1.27.0-alpine
    ```
    
- 生产级 CI/CD 集成规范：
    
    1. 在镜像构建完成后、推送仓库前，执行 Trivy 扫描，设置漏洞等级阈值，存在高危 / 严重漏洞时直接阻断流水线，禁止镜像推送；
    2. 对私有仓库中的镜像执行定期全量扫描，发现新披露的漏洞时触发告警，通知相关团队修复；
    3. 建立基础镜像定期更新机制，及时修复基础镜像中的已知漏洞，从源头降低安全风险。
    

### 2.2 容器特权与降权

容器的核心安全风险源于权限过度开放，默认情况下，容器内的 root 用户与宿主机 root 用户存在权限隔离边界，但不当的特权配置会彻底突破 Namespace 隔离，导致容器逃逸、宿主机被接管。

#### 2.2.1 --privileged 特权模式的致命风险

`--privileged`参数会为容器授予宿主机的完整 root 权限，彻底突破容器的安全隔离边界，**生产环境严禁使用**，其底层风险包括：

1. 绕过 Linux Capabilities 的所有限制，获得内核的完整权限；
2. 可直接挂载宿主机的所有磁盘设备，读写宿主机的完整文件系统；
3. 可修改宿主机的内核参数、加载内核模块、控制宿主机的网络栈；
4. 可直接访问宿主机的所有进程、Namespace，实现容器逃逸，接管整个宿主机。

#### 2.2.2 生产级权限降权方案

1. **容器内非 root 用户运行**
    
    这是容器安全的基础规范，通过 Dockerfile 的`USER`指令，为业务进程创建非 root 用户，禁止使用 root 用户运行业务进程，即使容器被突破，攻击者也仅拥有普通用户权限，无法实现提权与逃逸。
    
    标准化 Dockerfile 示例：

    ```dockerfile
    FROM eclipse-temurin:17-jre-alpine
    WORKDIR /app
    COPY app.jar .
    # 创建非root用户与用户组
    RUN addgroup -S appgroup && adduser -S appuser -G appgroup
    # 切换为非root用户
    USER appuser
    ENTRYPOINT ["java", "-jar", "app.jar"]
    ```
    
2. **User Namespace Remap**
    
    基于第九章讲解的 USER Namespace 机制，实现容器内 root 用户到宿主机非特权普通用户的 UID/GID 映射，容器内的 root 用户在宿主机上仅拥有普通用户权限，即使容器被突破，也无法对宿主机进行特权操作，是容器安全的核心加固手段。
    
    生产级配置：在`/etc/docker/daemon.json`中添加以下配置，重启 Docker Daemon 生效：

    ```json
    {
      "userns-remap": "default"
    }
    ```
    
    Docker 会自动创建`dockremap`用户，实现容器内 UID 0 到宿主机高 UID 范围的映射，彻底隔离 root 权限。
    
3. **Rootless 容器**
    
    更进一步的安全方案，整个 Docker Daemon 与容器均运行在宿主机的非 root 用户下，即使 Docker Daemon 被接管，攻击者也无法获得宿主机 root 权限，适用于对安全要求极高的场景。

### 2.3 限制容器能力 (--cap-drop, --cap-add)

Linux Capabilities 机制将传统 UNIX 系统中 root 用户的完整特权，拆分为多个细粒度的权限单元，实现权限的最小化管控。Docker 为容器默认提供了 14 项必要的 Capabilities 白名单，移除了绝大多数高危权限，生产环境需在此基础上进一步收紧权限。

#### 2.3.1 核心操作语法

- `--cap-drop`：移除 Docker 默认授予的 Capabilities，生产环境推荐默认移除所有权限；
- `--cap-add`：仅添加业务运行必需的最小权限，遵循最小权限原则。
- 生产级标准配置示例：
    
    ```bash
    # 默认移除所有Capabilities，仅添加业务必需的NET_BIND_SERVICE（绑定1024以下端口）
    docker run -d --cap-drop ALL --cap-add NET_BIND_SERVICE nginx:1.27.0-alpine
    ```

#### 2.3.2 高危 Capabilities 管控

以下 Capabilities 存在极高的容器逃逸与提权风险，**生产环境严禁为容器添加**：

- `SYS_ADMIN`：最危险的权限，可执行绝大多数内核管理操作，极易实现容器逃逸；
- `SYS_MODULE`：可加载 / 卸载内核模块，直接控制宿主机内核；
- `SYS_PTRACE`：可跟踪进程，读取其他进程的内存数据，突破 Namespace 隔离；
- `NET_ADMIN`：可修改网络栈、路由表、iptables 规则，存在网络攻击风险；
- `DAC_READ_SEARCH`/`DAC_OVERRIDE`：可绕过文件系统权限检查，读写宿主机任意文件。

### 2.4 安全审计体系

生产环境必须建立完整的容器安全审计体系，覆盖全生命周期的操作可追溯：

1. **Docker Daemon 审计**：通过 Linux Auditd 工具监控 Docker Daemon 的配置文件、二进制文件、套接字文件的所有操作，记录未授权的修改行为；
2. **容器生命周期审计**：记录所有容器的创建、启动、停止、销毁、exec 进入等操作，关联操作人、时间、操作内容，实现全链路可追溯；
3. **镜像审计**：记录所有镜像的构建、推送、拉取、扫描操作，建立镜像版本与代码提交的关联关系，实现供应链全链路可追溯；
4. **运行时安全审计**：监控容器内的异常行为，如特权操作、敏感文件访问、异常网络连接、进程异常行为，及时发现入侵行为。

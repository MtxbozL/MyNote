
容器日志是生产环境可观测性体系的核心组成部分，是业务故障排查、性能分析、安全审计的核心数据来源。容器的生命周期短暂性特性，决定了传统的本地文件日志存储方案无法适配容器化场景：容器销毁后，本地日志会同步丢失，单主机日志分散无法统一检索，多主机集群场景下日志管理难度指数级上升。

Docker 日志体系完全遵循**12 因素应用规范**：应用应将日志直接输出到标准输出（stdout）与标准错误（stderr），由运行环境负责日志的收集、存储与处理，而非写入容器内的本地文件。

### 3.1 Logging Driver 配置

Docker 通过**Logging Driver（日志驱动）** 实现容器日志的收集与转发，是 Docker 日志体系的核心抽象层，可通过`daemon.json`配置全局默认日志驱动，也可在容器创建时通过`--log-driver`参数单独配置，覆盖全局默认值。

#### 3.1.1 核心日志驱动与生产选型

|日志驱动|核心特性|生产适用场景|
|---|---|---|
|`json-file`|Docker 默认驱动，将容器日志以 JSON 格式写入本地文件，支持日志轮转，兼容`docker logs`命令|单主机开发测试环境、小规模部署场景|
|`local`|优化的本地存储驱动，采用自定义二进制格式，性能更高，默认开启日志轮转，兼容`docker logs`命令|生产环境本地持久化的推荐驱动，替代 json-file|
|`journald`|将日志转发到宿主机的 systemd-journald 服务，支持系统级日志统一管理，兼容`docker logs`命令|适配 systemd 体系的 Linux 发行版，系统级日志统一管控场景|
|`syslog`|将日志转发到 syslog 服务，支持远程 syslog 服务器|企业现有 syslog 日志体系的适配场景|
|`fluentd`|将日志直接转发到 Fluentd/Fluent Bit 日志聚合服务，原生支持标签与结构化数据|生产环境集中化日志架构的首选驱动，适配 EFK/ELK 栈|
|`gelf`|将日志以 GELF 格式转发到 Graylog、Logstash 等服务|适配 Graylog 日志平台的场景|

#### 3.1.2 生产级全局配置

在`/etc/docker/daemon.json`中配置全局日志驱动与轮转规则，彻底解决日志无限增长导致的磁盘占满问题，承接第二章的引擎配置规范：

```json
{
  "log-driver": "local",
  "log-opts": {
    "max-size": "100m",
    "max-file": "3",
    "compress": "true"
  }
}
```

- 核心规范：生产环境必须配置日志轮转规则，无论使用哪种驱动，严禁使用无限制的默认配置；
- 注意事项：全局配置仅对新建容器生效，已有容器需重建后才能应用新的配置。

### 3.2 生产级日志集中化架构

企业级生产环境主流采用**容器标准输出 → 日志采集组件 → 日志存储引擎 → 可视化检索平台**的四层架构，分为两种主流实现方案：

#### 3.2.1 驱动直传方案（Docker 原生推荐）

基于 Docker Logging Driver 直接将容器日志转发到日志聚合组件，无需额外的采集代理，架构最简，性能损耗最低。

- 典型架构：**Docker + Fluent Bit + Elasticsearch + Kibana（EFK）**
    
    1. 容器将日志输出到 stdout/stderr；
    2. Docker 通过`fluentd`日志驱动，将日志直接转发到 Fluent Bit；
    3. Fluent Bit 对日志进行清洗、结构化、脱敏处理后，转发到 Elasticsearch 存储；
    4. 运维人员通过 Kibana 实现日志的检索、可视化、告警与分析。
    
- 核心优势：架构简单，无额外 Sidecar 开销，与 Docker 原生深度集成，性能损耗极低；
- 适用场景：单主机集群、中小规模生产环境、标准化容器输出场景。

#### 3.2.2 Sidecar 采集方案（云原生推荐）

为每个 Pod / 容器部署一个 Sidecar 日志采集代理，采集容器内挂载的日志文件，适用于无法修改应用日志输出方式的遗留系统。

- 核心优势：适配性强，支持容器内文件日志采集，与容器生命周期绑定，适合 Kubernetes 集群场景；
- 适用场景：大规模 Kubernetes 集群、遗留应用容器化、多租户场景。

### 3.3 生产级日志最佳实践

1. **强制标准化日志输出**：所有业务应用必须将日志输出到 stdout/stderr，严禁写入容器内的本地文件，避免日志随容器销毁丢失，同时避免容器内日志文件无限增长导致磁盘占满；
2. **结构化日志规范**：应用日志必须采用 JSON 等结构化格式输出，包含时间、服务名、TraceID、日志级别、线程名、异常堆栈等关键字段，便于后续的检索、过滤与分析；
3. **日志分级与脱敏**：严格遵循 DEBUG、INFO、WARN、ERROR、FATAL 五级日志分级，生产环境关闭 DEBUG 日志；对日志中的密码、手机号、身份证号等敏感信息进行脱敏处理，符合合规要求；
4. **日志生命周期管理**：制定日志存储与保留策略，热日志存储在高性能节点，冷日志归档到低成本存储，超过保留期限的日志自动清理，满足合规留存要求的同时控制存储成本；
5. **全链路追踪集成**：日志中必须包含 TraceID、SpanID 等全链路追踪字段，实现日志、指标、链路的可观测性数据打通，提升故障排查效率。

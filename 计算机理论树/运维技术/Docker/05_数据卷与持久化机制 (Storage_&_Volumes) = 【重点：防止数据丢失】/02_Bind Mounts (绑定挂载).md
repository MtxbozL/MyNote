
Bind Mounts 是 Docker 最早提供的持久化方案，基于 Linux 内核原生 bind mount 机制实现，核心是将宿主机上已存在的目录 / 文件，直接绑定挂载到容器内的指定路径，实现宿主机与容器之间的文件系统直通。

### 2.1 核心语法与用法

Docker 提供两种挂载语法，分别为简写的`-v/--volume`语法与官方推荐的`--mount`完整语法，生产环境优先使用`--mount`语法，避免参数歧义与误操作。

#### 2.1.1 简写语法 `-v/--volume`

- 标准格式：`[宿主机源路径]:[容器内目标路径]:[权限选项]`
- 典型示例：

    ```bash
    # 基础读写挂载
    docker run -d -v /data/nginx/html:/usr/share/nginx/html nginx:1.27.0
    # 只读挂载
    docker run -d -v /data/nginx/nginx.conf:/etc/nginx/nginx.conf:ro nginx:1.27.0
    ```
    
- 核心权限选项：
    
    - `rw`：默认值，容器内对挂载路径拥有读写权限；
    - `ro`：只读模式，容器内仅可读取挂载内容，无法修改，生产环境安全加固推荐；
    - `z/Z`：SELinux 上下文标签配置，适配 RHEL/CentOS 等开启 SELinux 的系统，`z`为共享标签（多容器可访问），`Z`为私有标签（仅当前容器可访问）。
    

#### 2.1.2 完整语法 `--mount`

`--mount`语法参数明确、无歧义，支持高级配置，是生产环境的官方推荐方案，可严格区分绑定挂载与数据卷挂载。

- 标准格式：`type=bind,source=[宿主机源路径],target=[容器内目标路径],[可选参数]`
- 典型示例：

    ```bash
    # 基础读写挂载
    docker run -d --mount type=bind,source=/data/nginx/html,target=/usr/share/nginx/html nginx:1.27.0
    # 只读挂载
    docker run -d --mount type=bind,source=/data/nginx/nginx.conf,target=/etc/nginx/nginx.conf,readonly nginx:1.27.0
    ```
    
- 核心特性：若宿主机源路径不存在，`--mount`会直接报错并终止容器创建，而`-v`会自动在宿主机上创建空目录，可有效避免路径拼写错误导致的误操作，这是生产环境优先选择`--mount`的核心原因。

### 2.2 底层实现原理

Bind Mounts 完全基于 Linux 内核原生的**bind mount 机制**实现，该机制允许将一个已存在的目录 / 文件，挂载到另一个路径下，本质上是对同一个 inode 的硬链接引用。

- 核心特性：宿主机源路径与容器内目标路径指向完全相同的文件系统 inode，所有修改实时双向同步，无任何中间缓存层；
- 架构逻辑：完全绕过容器的 Overlay2 可写层，容器内对挂载路径的所有读写操作，直接作用于宿主机的源文件系统，不会写入容器可写层，彻底规避 Overlay2 的性能损耗与生命周期绑定问题；
- 隔离边界：仅挂载指定的路径，容器无法访问宿主机的其他目录，权限由挂载选项与 Linux 文件系统权限共同控制。

### 2.3 核心特点

1. **性能最优**：直接访问宿主机原生文件系统，无 Docker 存储层的额外开销，IO 性能接近裸盘，是所有存储方案中性能最高的；
2. **生命周期完全自主**：挂载的文件 / 目录由用户完全管理，Docker 不做任何干预，容器销毁不会影响宿主机源数据，生命周期与 Docker 完全解耦；
3. **灵活性极强**：支持目录与单个文件的挂载，可灵活配置读写 / 只读权限，适配精细化的挂载需求；
4. **可移植性极差**：强依赖宿主机的目录结构、文件系统、权限配置、SELinux 策略，不同宿主机的路径差异会导致部署失败，无法跨环境复用；
5. **安全风险高**：若挂载宿主机敏感目录（如`/`、`/etc`、`/var/run/docker.sock`），容器内进程可获得宿主机 root 权限，引发严重的提权漏洞。

### 2.4 适用场景与生产规范

#### 合法适用场景

1. **开发环境代码热更新**：将本地开发目录挂载到容器内，修改本地代码后容器内实时生效，无需重新构建镜像，大幅提升开发效率；
2. **宿主机与容器的配置文件共享**：将宿主机上的标准化配置文件（如`nginx.conf`、`my.cnf`）挂载到容器内，修改宿主机配置后重启容器即可生效，无需重新构建镜像；
3. **宿主机日志集中采集**：将容器内的日志目录挂载到宿主机的统一日志目录，便于日志采集组件统一收集，避免容器内日志文件无限增长；
4. **需要宿主机直接访问容器生成数据的场景**：如容器生成的备份文件、报表数据，直接写入宿主机目录，便于后续处理。

#### 生产环境严格规范

1. **生产环境非必要场景严禁使用 Bind Mounts**，强依赖宿主机目录结构的特性，与容器编排、集群调度的架构理念冲突，可移植性、可管理性远低于 Docker Volume；
2. **最小权限原则**：必须使用时，优先配置`readonly`只读权限，仅对必须写入的路径开放读写权限；
3. **敏感目录挂载禁令**：严禁挂载宿主机的`/`、`/etc`、`/dev`、`/root`、`/var/run/docker.sock`等敏感目录，避免容器提权攻击；
4. **路径标准化**：必须使用时，宿主机路径需遵循企业统一的标准化规范，避免不同宿主机路径不一致导致的部署故障；
5. **SELinux 合规**：RHEL/CentOS 系统开启 SELinux 时，必须正确配置`z/Z`标签，避免容器内无法访问挂载路径。

# 3.3 镜像的导入导出：打破物理隔离的“离线交付” (save / load)

在现代化的互联网公司，服务器大多能够畅通无阻地访问外网，`docker pull` 足以解决 99% 的镜像分发问题。然而，在金融机构、国防军工、医疗系统以及部分政企的生产环境中，服务器通常受到极其严格的网络管控，处于**物理隔离的纯内网环境（Air-Gapped Environment）**。

在无法连接公网，也无法搭建私有内部 Registry 的情况下，我们如何将开发好的软件镜像部署到这些内网服务器上呢？

答案就是 Docker 提供的“物理打包”方案：**`docker save` 与 `docker load`**。它们允许我们将虚拟的镜像资产，具象化为一个压缩包文件，从而通过 U 盘、移动硬盘或堡垒机等物理/跳板媒介进行离线摆渡交付。

### 1. `docker save`：将镜像封存为物理文件

`docker save` 的作用是将宿主机上已经存在的一个或多个镜像，完整地打包导出为一个标准的归档文件（通常是 `.tar` 格式）。

*   **基础语法**：
    `docker save -o [导出路径/文件名.tar] [镜像名:Tag]`
    *(注：`-o` 代表 output，指定输出文件的位置)*
*   **工程实战示例**：
    假设你在本地（有外网环境）拉取并配置好了最新的 Nginx 镜像。现在你要把它打包准备送往内网机房。
    执行：`docker save -o /tmp/nginx-offline.tar nginx:latest`
*   **底层机制剖析（连根拔起的完整度）**：
    与传统的虚拟机导出不同，`docker save` 导出的不仅仅是应用代码，而是包含了该镜像**完整的 UnionFS 层级结构、元数据配置以及所有的基础环境依赖**。这个生成的 `.tar` 文件是自包含（Self-contained）的。只要对方装了 Docker 引擎，无论底层是什么 Linux 发行版，都能保证 100% 还原运行。
*   **👨‍💻 高阶玩法（打包多个镜像）**：
    对于一个包含微服务的复杂项目，你可以一次性将 MySQL、Redis 和你的业务代码镜像打入同一个压缩包中：
    `docker save -o /tmp/my-project-all.tar mysql:8.0 redis:6 alpine-app:1.0`

### 2. `docker load`：在离线环境中解封镜像

当装有 `.tar` 文件的 U 盘被插入内网服务器，或文件被拷入系统后，我们只需通过 `docker load` 命令将其“解封”回 Docker 引擎中。

*   **基础语法**：
    `docker load -i [本地文件路径.tar]`
    *(注：`-i` 代表 input，指定输入文件的位置)*
*   **工程实战示例**：
    在内网服务器上执行：`docker load -i /tmp/nginx-offline.tar`
    执行后，控制台会依次打印出该镜像所包含的层（Layer）被逐个解压并恢复的提示：`Loaded image: nginx:latest`。
*   **验证还原**：
    导入完成后，直接输入 `docker images`，你就能看到那个熟悉的 `nginx:latest` 镜像已经静静地躺在内网服务器的资产列表里了，就像它是通过网络 pull 下来的一样。

### 3. 极客辨析：`save/load` 与 `export/import` 到底有什么区别？

在 Docker 命令集中，还存在一对极易混淆的命令：`export` 和 `import`。很多面试官喜欢拿这道题来考察候选人对 Docker 底层文件系统的理解深度。

*   **操作对象的差异**：
    *   `save/load` 操作的对象是 **镜像 (Image)**。
    *   `export/import` 操作的对象是 **容器 (Container)**（即正在运行或停止状态的运行时实例）。
*   **底层数据的差异（核心重点！）**：
    *   **`save` 是完整的资产保留**：它会保存镜像所有的分层历史（History）和元数据。你可以把它看作是“带有完整演进过程的高清原图”。
    *   **`export` 是运行状态的快照**：当你对一个容器执行 `export` 时，Docker 会将容器当时的文件系统打个包。**但它会丢失所有的镜像分层信息和元数据历史**（相当于把所有的层压成了一个扁平的平面层）。你可以把它看作是“截图”。
*   **应用场景定界**：
    *   绝大多数的**软件交付、离线部署部署**，必须使用 **`save/load`**。
    *   如果只是想做一个轻量级的基础 Linux 根文件系统，或者只想备份容器在某个特定时刻的数据切片，不关心它的构建历史，可以使用 `export/import`。

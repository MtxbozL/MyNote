
UnionFS（联合文件系统）是 Docker 镜像分层存储的核心底层技术，是 Docker 镜像不可变性、分层复用、写时复制机制的技术基石，为本章核心重点内容，需完全掌握其架构与运行原理。

### 4.1 UnionFS 核心定义

UnionFS 是一种为 Linux 系统设计的堆叠式文件系统，核心能力是将多个不同的目录（称为分支 / Branch）透明地叠加在一起，形成一个单一、统一的文件系统视图。其核心特性包括：

1. **分支权限分离**：支持为每个分支设置只读（Read-Only）或读写（Read-Write）权限，底层分支通常设置为只读，上层分支设置为读写；
2. **透明合并视图**：用户仅能看到最终的合并视图，无需感知文件所在的底层分支，所有读写操作均针对合并视图执行；
3. **增量变更存储**：所有文件的修改、删除操作仅会作用于读写分支，不会修改只读分支的内容，实现增量变更的隔离存储。

Docker 镜像的分层存储机制，完全基于 UnionFS 的核心特性实现，镜像的每一个分层对应 UnionFS 的一个只读分支，容器的可写层对应 UnionFS 的读写分支。

### 4.2 Docker 存储驱动演进与 Overlay2 规范

Docker 通过存储驱动（Storage Driver）实现 UnionFS 的能力适配，存储驱动的演进历程如下：

- 早期版本：使用 AUFS 驱动，未合入 Linux 内核主线，兼容性差；
- 过渡版本：支持 DeviceMapper、Btrfs、ZFS 等驱动，均存在场景限制，无法通用；
- 当前标准：**Overlay2** 是 Docker 官方唯一推荐的生产级存储驱动，已合入 Linux 内核主线（3.18 版本以上支持，4.0 版本以上完善），是所有主流 Linux 发行版的默认存储驱动，本章仅针对 Overlay2 驱动进行深度讲解。

#### Overlay2 生产环境前置要求

1. Linux 内核版本≥4.15 LTS，推荐 5.4 以上稳定版本；
2. 底层文件系统必须为 ext4 或 xfs，其中 xfs 文件系统必须在格式化时开启`ftype=1`功能，否则 Overlay2 无法正常工作；
3. 禁用 SELinux 强制模式，或为 Overlay2 目录配置正确的 SELinux 上下文。

### 4.3 Overlay2 核心架构

Overlay2 基于 Linux 内核原生 OverlayFS 实现，其核心架构由四个关键目录构成，完整对应 Docker 镜像与容器的分层结构：

|架构组件|权限|对应 Docker 结构|核心功能|
|---|---|---|---|
|lowerdir|只读|Docker 镜像的所有只读分层|可包含多个堆叠的只读目录，按顺序从下到上排列，最底层为基础镜像层，往上依次为增量镜像分层；所有内容永久只读，永远不会被修改，是镜像不可变性的底层实现|
|upperdir|读写|Docker 容器的可写层|唯一的读写目录，容器运行过程中所有的文件创建、修改、删除操作，均仅作用于该目录，不会修改任何 lowerdir 的内容；生命周期与容器完全绑定，容器销毁时同步删除|
|mergeddir|合并视图|容器内的根文件系统|OverlayFS 的挂载点，是 lowerdir 与 upperdir 的透明合并视图，容器内的进程仅能看到该合并目录，无法感知底层的分层结构，所有文件操作均基于该目录执行|
|workdir|内部读写|OverlayFS 工作目录|内核内部使用的工作目录，用于保障文件操作的原子性，执行写时复制操作时，先在该目录完成中间操作，再原子替换至 upperdir，避免文件损坏；对用户与容器完全透明|

### 4.4 镜像分层的核心实现机制

Docker 镜像的每一条 Dockerfile 构建指令，都会生成一个新的只读分层，每一个分层仅包含相对于上一个分层的增量文件变更，所有分层堆叠在一起，通过 Overlay2 的 lowerdir 实现联合挂载，形成完整的镜像文件系统。

#### 典型镜像分层示例

以 Nginx 镜像的 Dockerfile 构建过程为例，其分层结构如下：

1. `FROM ubuntu:22.04`：生成基础镜像层，作为最底层的 lowerdir；
2. `RUN apt-get update && apt-get install -y nginx`：生成第二个只读分层，仅包含安装 Nginx 后的文件增量变更；
3. `COPY nginx.conf /etc/nginx/nginx.conf`：生成第三个只读分层，仅包含配置文件的变更；
4. `EXPOSE 80`：生成第四个只读分层，仅包含端口声明的元数据变更；
5. `CMD ["nginx", "-g", "daemon off;"]`：生成第五个只读分层，仅包含启动命令的元数据变更。

所有分层均为只读状态，构建完成后永久不可修改，若需变更镜像内容，必须重新构建，生成新的增量分层，这就是 Docker 镜像**不可变性**的底层实现。

#### 镜像分层的核心优势

1. **存储空间高效复用**：不同镜像可共享相同的底层分层，例如 10 个基于 ubuntu:22.04 构建的镜像，仅需存储一份 ubuntu 基础层，大幅降低磁盘空间占用；
2. **传输效率大幅提升**：镜像拉取与推送时，仅需传输本地 / 仓库不存在的分层，无需传输完整镜像，大幅节省网络带宽与传输耗时；
3. **构建缓存复用**：镜像构建时，未发生变更的分层可直接复用缓存，无需重新构建，大幅提升镜像构建速度；
4. **变更可追溯与可审计**：每一个分层对应一条构建指令，可通过`docker image history`命令完整查看镜像的构建历史，实现变更的全链路追溯，符合合规性要求。

### 4.5 写时复制（Copy-on-Write, CoW）核心机制

写时复制是 OverlayFS 的核心运行机制，也是 Docker 容器可写层的核心实现原理，其核心逻辑是：**仅当需要修改一个文件时，才会将该文件从只读的 lowerdir 复制到读写的 upperdir 中，后续所有修改均针对 upperdir 中的副本执行，lowerdir 中的原始文件永远不会被修改**。

#### 三类核心文件操作的底层实现

1. **文件读取操作**
    
    - 容器内进程读取文件时，首先在 upperdir 中查找该文件，若找到则直接读取；
    - 若 upperdir 中不存在该文件，则依次向下在各个 lowerdir 中查找，找到后直接读取该只读文件；
    - 整个过程对容器进程完全透明，无需感知文件所在的分层。
    
2. **文件修改操作（写时复制核心流程）**
    
    - 第一次修改一个存在于 lowerdir 中的文件时，OverlayFS 触发**copy-up**操作：将该文件从 lowerdir 完整复制到 upperdir 中，复制完成后，对 upperdir 中的副本进行修改，lowerdir 中的原始文件保持不变；
    - 后续再次修改该文件时，直接修改 upperdir 中的副本，无需再次执行 copy-up 操作；
    - 性能优化：copy-up 操作仅在第一次修改文件时触发，且仅复制被修改的单个文件，而非整个分层，Overlay2 支持最多 128 个 lowerdir 堆叠，copy-up 性能远优于旧版 AUFS 驱动，性能损耗可忽略不计。
    
3. **文件删除操作**
    
    - 当容器内进程删除一个存在于 lowerdir 中的文件时，OverlayFS 不会直接删除 lowerdir 中的只读文件，而是在 upperdir 中创建一个名为`.whiteout`的特殊隐藏文件，该文件会在 mergeddir 的合并视图中屏蔽 lowerdir 中的原始文件，使容器进程感知到该文件已被删除；
    - 当删除 upperdir 中创建的文件时，直接从 upperdir 中物理删除即可，无需创建 whiteout 文件；
    - 容器销毁时，upperdir 中的 whiteout 文件同步删除，底层镜像中的原始文件完全不受影响，这就是容器重启后，删除的文件会自动恢复的底层原因。
    

#### 目录操作的实现逻辑

目录操作采用合并机制：若 lowerdir 中存在目录 A，包含文件 a，upperdir 中存在同名目录 A，包含文件 b，则 mergeddir 的合并视图中，目录 A 会同时包含文件 a 与文件 b；删除目录时，会在 upperdir 中创建对应的 whiteout 文件，屏蔽整个 lowerdir 中的目录。

# Docker 学习文档 第三章 镜像管理与操作

## 本章导读

本章是 Docker 核心资产 —— 镜像的全生命周期管理章节，核心目标是建立对 Docker 镜像分层存储本质的深度认知，掌握镜像的全流程操作命令、生命周期管理规范、底层存储驱动原理，以及生产环境镜像管理的最佳实践。本章内容是后续 Dockerfile 镜像构建、容器运行、CI/CD 流水线设计的核心基础，其中**UnionFS 联合文件系统与镜像分层原理**为核心重点，需完全掌握其底层机制，不可遗漏任何知识点。

## 01 获取与推送

镜像的获取与推送是 Docker 镜像生命周期的核心操作，所有操作均基于 Docker 镜像的标准化命名规范与 Registry 认证体系实现，本节完整讲解核心命令的语法、底层逻辑与生产级使用规范。

### 1.1 Docker 镜像标准化命名规范

Docker 镜像的唯一完整引用格式为：

```plaintext
[Registry地址]/[命名空间]/[仓库名]:[标签]
```

各字段说明：

- **Registry 地址**：镜像仓库的服务地址，默认为 Docker Hub 官方地址`docker.io`；私有仓库必须显式指定该地址，否则会默认推送至 Docker Hub；
- **命名空间**：用于区分仓库的归属，默认为官方镜像命名空间`library`；个人 / 企业仓库需指定对应的用户名或组织名；
- **仓库名**：镜像的唯一标识名称，通常对应应用名称或业务模块；
- **标签**：用于标识镜像的版本、架构等元数据，默认为`latest`，是镜像版本管理的核心载体。

示例：`docker.io/library/nginx:1.27.0` 可简写为 `nginx:1.27.0`，分别对应默认 Registry、默认命名空间与指定版本标签。

### 1.2 核心操作命令

#### 1.2.1 docker login /docker logout 仓库认证

该组命令用于完成镜像仓库的身份认证与登出，是拉取私有镜像、推送镜像的前置操作。

- **核心语法**

    ```bash
    # 登录指定仓库，不指定Registry则默认登录Docker Hub
    docker login [OPTIONS] [REGISTRY_ADDRESS]
    # 登出指定仓库
    docker logout [REGISTRY_ADDRESS]
    ```
    
- **核心参数**
    
    - `--username/-u`：指定认证用户名；
    - `--password-stdin`：通过标准输入读取密码，**生产环境强制推荐使用**，避免密码明文出现在 Shell 历史记录中，引发安全风险。
    
- **底层逻辑**：登录成功后，认证凭证会加密存储在`~/.docker/config.json`文件中，后续与该 Registry 的所有交互均会自动携带认证信息，无需重复登录。

#### 1.2.2 docker search 镜像仓库搜索

用于在指定镜像仓库中搜索匹配的镜像仓库列表，仅适用于镜像快速检索，生产环境严禁直接基于搜索结果拉取镜像，需先验证镜像的官方性与安全性。

- **核心语法**

    ```bash
    docker search [OPTIONS] IMAGE_NAME
    ```
    
- **核心参数**
    
    - `--filter/-f`：过滤搜索结果，常用过滤规则：`stars=NUM`（收藏数不低于指定值）、`is-official=true`（仅显示官方镜像）、`is-automated=true`（仅显示自动构建的镜像）；
    - `--limit`：限制返回结果的最大数量，默认为 25 条。
    

#### 1.2.3 docker pull 镜像拉取

用于从镜像仓库拉取指定镜像至本地，是获取镜像的核心命令，底层基于镜像分层机制实现增量拉取。

- **核心语法**

    ```bash
    docker pull [OPTIONS] IMAGE_NAME[:TAG|@DIGEST]
    ```
    
- **核心参数**
    
    - `--all-tags/-a`：拉取指定仓库下的所有标签镜像，非特殊场景严禁使用，会导致大量无用镜像占用磁盘空间；
    - `--platform`：指定拉取镜像的系统架构，如`linux/amd64`、`linux/arm64`，用于多架构环境镜像拉取；
    - `--quiet/-q`：静默拉取，仅输出镜像 ID，适用于自动化脚本。
    
- **底层逻辑与核心规范**
    
    1. 拉取操作基于镜像分层实现，仅拉取本地不存在的镜像分层，已存在的分层直接复用，大幅降低网络带宽占用与拉取耗时；
    2. 可通过`@DIGEST`指定镜像的哈希摘要拉取，摘要为镜像内容的唯一标识，可保证拉取的镜像与预期完全一致，**生产环境推荐使用摘要替代标签，避免标签覆盖导致的版本漂移**；
    3. `latest`标签陷阱：`latest`仅为镜像的默认标签，不代表镜像的最新版本，由镜像构建者推送时指定，存在版本覆盖、不可复现的风险，**生产环境必须指定固定的版本标签或摘要，严禁使用 latest 标签部署业务**。
    

#### 1.2.4 docker tag 镜像标签管理

用于为本地镜像创建新的标签，是镜像推送前的必备操作，底层基于镜像只读分层的硬链接实现，不会创建镜像副本，不占用额外磁盘空间。

- **核心语法**

    ```bash
    docker tag SOURCE_IMAGE[:TAG] TARGET_IMAGE[:TAG]
    ```
    
- **核心逻辑**：标签是镜像的引用指针，一个镜像可拥有多个标签，所有标签均指向同一个镜像 ID，仅当所有标签被删除后，镜像才会被纳入清理范围。
- **典型使用场景**：为本地镜像添加私有仓库地址标签，用于后续推送，示例：

    ```bash
    # 为nginx:1.27.0添加私有仓库标签
    docker tag nginx:1.27.0 harbor.internal.example.com/library/nginx:1.27.0
    ```
    

#### 1.2.5 docker push 镜像推送

用于将本地镜像推送至指定镜像仓库，是镜像分发、共享与 CI/CD 流水线的核心环节。

- **核心语法**

    ```bash
    docker push [OPTIONS] IMAGE_NAME[:TAG]
    ```
    
- **核心参数**
    
    - `--all-tags/-a`：推送指定仓库下的所有标签镜像；
    - `--quiet/-q`：静默推送，仅输出镜像摘要，适用于自动化脚本。
    
- **前置条件与底层逻辑**
    
    1. 必须已完成目标仓库的登录认证，且对目标仓库拥有写入权限；
    2. 镜像标签必须包含目标仓库的 Registry 地址，否则会默认推送至 Docker Hub；
    3. 推送操作基于分层增量实现，仅推送目标仓库不存在的镜像分层，已存在的分层直接跳过，大幅提升推送效率。

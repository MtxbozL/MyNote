
Linux Control Groups（简称 Cgroups）是内核提供的**进程组资源管控机制**，核心作用是对一组进程进行统一的资源限制、优先级控制、审计与生命周期管理。

Namespace 解决了「进程能看到什么资源」的视图隔离问题，但无法限制进程对资源的使用量；Cgroups 解决了「进程能用多少资源」的配额管控问题，二者协同工作，才能实现完整的容器隔离与可控性，是防止容器资源抢占宿主机、保障多容器稳定运行的核心基石。

### 2.1 Cgroups 版本演进与核心架构

Cgroups 分为两个大版本，生产环境需明确版本边界与适配性：

|版本|内核支持|核心架构|生产适配性|
|---|---|---|---|
|Cgroups v1|Linux 2.6.24+|多子系统分离架构，每个资源类型对应独立的子系统与层级树，功能分散，兼容性强|传统企业级发行版（RHEL 7、CentOS 7）默认使用，Docker 全版本兼容|
|Cgroups v2|Linux 4.5+，5.0 + 稳定|统一的层级树架构，所有资源类型共享同一个层级结构，设计更简洁、安全、高效，支持更精细的资源管控|现代发行版（RHEL 8+/CentOS Stream 8+、Ubuntu 22.04+）默认使用，Docker 20.10 + 原生支持|

Docker 完全兼容两个版本的 Cgroups，会自动适配宿主机内核的默认版本，本章核心讲解 Docker 高频使用的资源管控能力，同时覆盖两个版本的核心实现。

#### Cgroups 核心工作原理

1. Cgroups 以层级树的结构组织，每个节点为一个控制组，控制组内可以定义一组资源限制规则；
2. 每个进程可以加入一个且仅一个控制组，进程会继承该控制组的所有资源限制规则，进程创建的子进程会默认加入父进程的控制组；
3. 内核通过对应的资源子系统，强制执行控制组内的资源限制规则，当进程的资源使用量超过配额时，内核会触发限流、OOM 杀死、IO 阻塞等管控动作；
4. Docker 为每个容器创建独立的控制组，将容器内的所有进程加入该控制组，应用用户配置的资源限制规则，实现容器级的资源配额管控。

### 2.2 核心资源子系统与 Docker 实现

Docker 默认启用 Cgroups 的核心资源子系统，覆盖容器运行的所有资源管控需求，对应`docker run`命令的资源限制参数，完全匹配目录内的 CPU、内存限制核心内容。

#### 2.2.1 CPU 子系统：CPU 资源限制

CPU 子系统用于管控进程组的 CPU 时间、优先级、核心绑定，是防止容器 CPU 资源抢占的核心手段，Docker 提供三个层级的 CPU 管控能力：

##### 1. 绝对 CPU 配额限制：--cpus

- **核心作用**：限制容器最多能使用的 CPU 核心数，是 Docker 最常用的 CPU 限制参数，对 Cgroups v1 与 v2 均兼容；
- **底层实现**：基于 Linux 内核 CFS（完全公平调度器）实现，通过`cpu.cfs_period_us`（调度周期，默认 100000 微秒 = 100ms）与`cpu.cfs_quota_us`（每个周期内可使用的 CPU 时间）两个参数实现配额管控；
- **示例说明**：`--cpus=0.5`，表示每个调度周期内，容器最多可使用 50000 微秒的 CPU 时间，等价于最多使用 0.5 个 CPU 核心的算力；`--cpus=4`表示最多使用 4 个 CPU 核心的算力；
- **生产规范**：生产环境所有业务容器必须显式配置`--cpus`参数，避免单个容器占用宿主机所有 CPU 资源，导致其他服务卡顿。

##### 2. 相对 CPU 权重：--cpu-shares

- **核心作用**：设置容器的 CPU 相对优先级，仅当宿主机 CPU 资源满负载时生效，权重越高的容器，分配到的 CPU 时间越多；
- **底层实现**：通过`cpu.shares`参数实现，默认值为 1024，权重值为相对值，无绝对上限；
- **示例说明**：两个容器 A（权重 2048）与 B（权重 1024），当 CPU 满负载时，A 能分配到 2/3 的 CPU 时间，B 能分配到 1/3 的 CPU 时间；当 CPU 资源空闲时，权重不生效，容器可使用空闲的所有 CPU 资源；
- **生产规范**：核心业务容器可设置更高的权重，保障核心业务的 CPU 资源优先分配，非核心业务降低权重。

##### 3. CPU 核心绑定：--cpuset-cpus

- **核心作用**：将容器绑定到宿主机的指定 CPU 核心上运行，容器内的所有进程仅能在绑定的 CPU 核心上调度，无法使用其他核心；
- **底层实现**：通过`cpuset`子系统实现，可精准控制容器的 CPU 调度范围；
- **示例说明**：`--cpuset-cpus=0,2`表示容器仅能在 0 号和 2 号 CPU 核心上运行；`--cpuset-cpus=0-3`表示绑定 0-3 号核心；
- **生产价值**：可避免 CPU 核心之间的上下文切换开销，提升 CPU 缓存命中率，大幅提升高并发、低延迟业务的性能，同时实现 CPU 资源的硬隔离。

#### 2.2.2 memory 子系统：内存资源限制

memory 子系统用于管控进程组的内存使用量，包括物理内存、交换分区、内核内存、页缓存等，是防止容器内存溢出（OOM）导致宿主机故障的核心手段。

##### 1. 物理内存硬限制：-m/--memory

- **核心作用**：设置容器可使用的物理内存（RSS）最大上限，是 Docker 最核心的内存限制参数；
- **底层实现**：通过`memory.limit_in_bytes`（Cgroups v1）/`memory.max`（Cgroups v2）实现，当容器的物理内存使用量超过该阈值时，内核会触发 OOM killer，强制终止容器内的进程；
- **示例说明**：`-m 1g`表示容器最多可使用 1GB 的物理内存；
- **生产规范**：生产环境所有业务容器必须显式配置内存硬限制，避免单个容器内存泄漏导致宿主机 OOM，引发系统级故障。

##### 2. 交换分区限制：--memory-swap

- **核心作用**：设置容器可使用的「物理内存 + 交换分区（Swap）」总上限，控制容器的交换分区使用量；
- **底层实现与规则**：
    
    1. 若未设置`--memory-swap`，默认值为`--memory`的 2 倍，即容器可使用与物理内存等量的交换分区；
    2. 若设置`--memory-swap`等于`--memory`，则容器无法使用任何交换分区，生产环境推荐该配置，避免交换分区导致的性能衰减；
    3. 若设置`--memory-swap=-1`，则容器的交换分区使用无限制；
    
- **示例说明**：`-m 1g --memory-swap 1g`表示容器仅能使用 1GB 物理内存，无法使用交换分区；`-m 1g --memory-swap 2g`表示容器最多可使用 1GB 物理内存 + 1GB 交换分区；
- **生产规范**：生产环境建议禁用容器的交换分区，设置`--memory-swap`等于`--memory`，避免内存泄漏的容器占用大量交换分区，导致宿主机性能严重下降。

##### 3. OOM 行为管控：--oom-kill-disable

- **核心作用**：禁用内核的 OOM killer 对容器的终止行为，当容器内存超过限制时，不会被内核杀死，而是会被阻塞等待内存释放；
- **风险警示**：**生产环境严禁全局禁用 OOM killer**，禁用后，内存泄漏的容器会持续占用宿主机内存，可能导致宿主机触发系统级 OOM，引发更严重的故障；仅可用于核心数据库等不可随意终止的服务，同时必须配合严格的内存监控告警。

#### 2.2.3 其他核心子系统（Docker 默认启用）

除 CPU 与内存外，Docker 还默认启用以下 Cgroups 子系统，实现完整的资源管控：

1. **blkio 子系统**：块设备 IO 限制，通过`--device-read-bps`、`--device-write-bps`等参数，限制容器对磁盘的读写带宽与 IOPS，防止容器占用大量磁盘 IO，影响宿主机性能；
2. **pids 子系统**：进程数量限制，通过`--pids-limit`参数限制容器内最多可创建的进程 / 线程数量，防止 fork 炸弹攻击，是生产环境安全加固的必备配置；
3. **devices 子系统**：设备访问权限控制，限制容器可访问的宿主机设备文件，默认仅允许容器访问少数必要的虚拟设备，禁止访问宿主机的磁盘、显卡等物理设备，防止容器通过设备文件提权；
4. **hugetlb 子系统**：大页内存限制，管控容器对大页内存的使用量，适配数据库等需要大页内存优化的高性能应用。

### 2.3 生产级最佳实践

1. **强制资源配额**：生产环境所有业务容器必须显式配置 CPU、内存的硬限制，禁止无限制运行的容器，避免单个容器故障影响整个宿主机；
2. **合理设置资源超配**：根据业务特性设置合理的超配比，CPU 超配建议不超过 200%，内存严禁超配，避免 OOM 风险；
3. **禁用交换分区**：生产环境建议禁用容器的 Swap 使用，设置`--memory-swap`等于`--memory`，保障业务性能稳定；
4. **进程数量限制**：所有容器必须配置`--pids-limit`，防止 fork 炸弹攻击，默认值可设置为 1024，根据业务需求调整；
5. **动态资源调整**：可通过`docker update`命令动态修改运行中容器的资源限制，无需重启容器，适配业务流量波动场景。

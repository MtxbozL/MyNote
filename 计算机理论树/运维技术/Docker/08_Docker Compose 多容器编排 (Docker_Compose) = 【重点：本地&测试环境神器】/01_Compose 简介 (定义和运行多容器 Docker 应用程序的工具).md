## 本章导读

本章是 Docker 单主机多容器应用编排的核心章节，解决单容器`docker run`命令无法高效管理多容器依赖、网络互通、配置统一、生命周期协同的核心痛点，是本地开发、测试环境的标准化部署神器，也是向云原生集群编排（Kubernetes）过渡的核心基础。

本章内容完全承接前序章节的 Dockerfile 构建、容器生命周期管理、自定义网络、数据卷持久化等核心知识，完整覆盖 Docker Compose 的核心定位、标准化语法、依赖管理、高频命令与全栈实战演练，严格遵循 Compose 官方规范，兼顾学术严谨性与工程实用性，无遗漏覆盖目录内所有知识点，同时补充生产级最佳实践与高频踩坑避坑指南。

## 01 Compose 简介

### 1.1 核心定位与解决的痛点

Docker Compose 是 Docker 官方推出的**用于定义和运行多容器 Docker 应用程序的工具**，通过单一的 YAML 配置文件（`docker-compose.yml`）声明应用的所有服务、网络、数据卷等资源，再通过一条命令完成所有资源的创建、启动、停止、销毁等全生命周期管理，彻底替代手动执行多条`docker run`、`docker network create`、`docker volume create`命令的繁琐操作。

传统多容器手动部署的核心痛点，也是 Compose 的核心解决目标：

1. **依赖管理混乱**：微服务架构下，应用由多个容器组成（前端、后端、数据库、缓存等），容器间存在严格的启动依赖与网络互通需求，手动操作极易出现顺序错误、网络不通等问题；
2. **配置碎片化**：每个容器的端口映射、数据卷、环境变量、网络配置分散在多条命令中，无法统一管理、版本化追溯，不符合基础设施即代码（IaC）理念；
3. **环境一致性无法保障**：开发、测试、生产环境的部署命令不一致，极易出现「本地能跑，测试环境跑不起来」的环境漂移问题；
4. **生命周期协同困难**：多容器应用的启动、停止、重启、扩容需要逐一对容器操作，效率极低，无法实现批量协同管理。

### 1.2 核心设计理念与两大核心概念

Compose 的核心设计理念是**声明式配置、项目级隔离、标准化全生命周期管理**，其整个体系围绕两大核心概念构建：

1. **项目（Project）**
    
    Compose 的顶级管理单元，默认以`docker-compose.yml`文件所在的目录名作为项目名，也可通过`-p`参数手动指定。Compose 会为每个项目创建独立的资源隔离空间，不同项目之间的服务、网络、数据卷默认完全隔离，彻底避免不同应用之间的资源冲突。
2. **服务（Service）**
    
    项目内的最小部署单元，对应一组功能相同的容器副本，是对容器运行配置的声明式抽象。一个服务定义了该组容器的镜像、构建规则、端口映射、数据卷、环境变量、网络、依赖关系、副本数量等所有运行配置，Compose 会根据服务定义自动创建并管理对应的容器实例。

### 1.3 版本演进与生产环境选型

Compose 分为两个大版本，生产环境必须明确版本边界，避免兼容性问题：

1. **Compose V1（已废弃）**
    
    基于 Python 开发的独立二进制工具，命令格式为`docker-compose`（带横杠），最终版本为 v2.20.3，Docker 官方已在 2023 年停止维护，生产环境严禁使用。
2. **Compose V2（当前官方唯一推荐）**
    
    基于 Go 语言开发的 Docker CLI 原生插件，完全兼容 V1 的配置语法，命令格式为`docker compose`（空格分隔，无横杠），与 Docker Engine 深度集成，性能、兼容性、安全性均优于 V1，是当前生产环境的唯一标准选型。

本章所有内容均基于 Compose V2 官方规范编写，所有命令均采用`docker compose`标准格式。

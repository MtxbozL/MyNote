
Bridge 模式是 Docker 最基础、最核心的网络模式，分为**默认网桥 docker0**与**用户自定义网桥**，二者底层实现架构完全一致，但功能、安全性、可管理性存在本质差异。本节先讲解 Bridge 模式的通用底层原理，再说明默认网桥的特性与核心缺陷。

### 2.1 默认网桥 docker0 的核心架构

Docker 安装后，会自动在宿主机上创建名为`docker0`的 Linux 网桥，作为所有默认 bridge 模式容器的虚拟二层交换机，其核心架构与初始化规则如下：

1. **地址分配**：docker0 网桥默认分配私有 IPv4 网段（通常为`172.17.0.1/16`），网桥自身占用网段内第一个 IP 作为网关地址，是所有默认 bridge 容器的默认网关；
2. **veth pair 绑定**：每创建一个默认 bridge 模式的容器，Docker 会自动生成一对 veth pair：一端放入容器的 Network Namespace，命名为`eth0`，分配 docker0 网段内的唯一 IP，设置默认网关为 docker0 的 IP；另一端接入 docker0 网桥，作为网桥的端口，生命周期与容器完全绑定；
3. **二层转发**：所有接入 docker0 网桥的容器，同属一个广播域，可通过网桥实现二层数据转发，直接通过 IP 地址互相通信；
4. **三层路由**：docker0 网桥作为宿主机的虚拟网卡，被纳入宿主机的路由表，实现容器网段与宿主机网段的三层互通，是容器访问外部网络的核心路由节点。

### 2.2 核心数据包流转流程

Bridge 模式下，所有网络通信均遵循 Linux 网络协议栈的转发规则，核心分为三类通信场景，完整对应第四章端口映射的底层实现：

#### 2.2.1 同网桥容器间通信

容器 A（172.17.0.2）→ 容器 A 的 eth0 → veth pair 对 A 端 → docker0 网桥（MAC 地址学习后转发）→ veth pair 对 B 端 → 容器 B 的 eth0 → 容器 B（172.17.0.3）

- 核心特性：同网段二层转发，无需经过路由与 NAT 转换，转发效率极高。

#### 2.2.2 容器访问外部公网

容器 → 容器 eth0 → docker0 网桥 → 宿主机路由表匹配 → 宿主机物理网卡 → iptables NAT 表 POSTROUTING 链 SNAT 转换 → 公网

- 核心实现：Docker 默认在 iptables 中添加 SNAT 规则，将源地址为 docker0 网段的容器流量，源地址转换为宿主机物理网卡的 IP 地址，实现容器对公网的访问。

#### 2.2.3 外部网络访问容器（端口映射）

公网客户端 → 宿主机物理网卡：映射端口 → iptables NAT 表 PREROUTING 链 DNAT 转换 → 目标地址改为容器 IP: 容器端口 → 宿主机路由表 → docker0 网桥 → 容器 eth0 → 容器

- 核心实现：Docker 执行`-p`端口映射参数时，自动创建对应的 DNAT 规则，将宿主机端口的流量转发到对应容器的端口，是外部访问容器服务的核心机制。

### 2.3 默认网桥 docker0 的核心缺陷

默认网桥仅适用于临时测试场景，**生产环境严禁使用**，核心缺陷完全违背容器化部署的工程化与安全规范：

1. **无内置 DNS 解析能力**：同网桥内的容器仅能通过 IP 地址互相访问，无法通过容器名 / 主机名通信，仅能通过废弃的`--link`机制实现静态名称解析，容器 IP 变更后通信链路直接失效，完全无法适配容器的动态生命周期特性；
2. **隔离性极差**：所有使用默认网桥的容器共享同一个广播域，无任何网络隔离，任意容器均可访问其他容器的所有端口，存在横向渗透的严重安全风险，不符合最小权限原则；
3. **配置变更影响全局**：docker0 网桥的配置变更（网段、MTU、转发规则等）需要重启 Docker Daemon，且会影响所有使用默认网桥的容器，无法实现精细化的业务级配置管理；
4. **无精细化流量管控能力**：无法针对单个容器或业务组配置独立的防火墙规则、带宽限制、流量监控，可管理性极差，无法满足生产环境的运维与合规要求。

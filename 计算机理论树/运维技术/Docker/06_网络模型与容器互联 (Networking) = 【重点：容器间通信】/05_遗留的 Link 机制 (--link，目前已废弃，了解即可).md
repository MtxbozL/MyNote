
`--link`是 Docker 早期提供的容器互联机制，用于在默认网桥中实现容器名到 IP 的静态解析，解决默认网桥无 DNS 解析的问题，目前已被 Docker 官方**完全废弃**，仅保留兼容性支持，生产环境严禁使用。

### 5.1 基础用法

`--link`参数在容器创建时使用，格式为`--link 目标容器名:别名`，实现当前容器与目标容器的静态互联，示例：

```bash
# 启动MySQL容器
docker run -d --name mysql -e MYSQL_ROOT_PASSWORD=123456 mysql:8.0
# 启动应用容器，通过--link与MySQL容器互联
docker run -d --name app --link mysql:db nginx:1.27.0
```

- 实现效果：app 容器内可通过`mysql`或别名`db`直接访问 MySQL 容器，Docker 会自动在 app 容器的`/etc/hosts`文件中添加 MySQL 容器的 IP 映射，同时通过环境变量传递目标容器的配置信息。

### 5.2 核心缺陷与废弃原因

1. **静态绑定，无法动态更新**：`--link`仅在容器创建时生效，目标容器重启、重建后 IP 地址变更，`/etc/hosts`中的映射不会自动更新，导致通信链路失效，完全无法适配容器的动态生命周期；
2. **强依赖启动顺序**：被 link 的目标容器必须先启动，才能创建当前容器，无法实现容器的并行启动，适配性极差；
3. **敏感信息泄露风险**：`--link`会将目标容器的所有环境变量传递到当前容器，包括密码、密钥等敏感信息，存在严重的信息泄露风险；
4. **隔离性破坏**：link 机制会自动在 iptables 中添加放行规则，打通两个容器的网络，破坏默认网桥的隔离性，存在安全风险；
5. **完全被自定义网络替代**：自定义网络的内置 DNS 解析机制，彻底解决了`--link`的所有缺陷，支持动态更新、无需顺序启动、无敏感信息泄露，是官方指定的替代方案。

### 5.3 生产规范

**生产环境严禁使用`--link`机制**，所有容器互联需求必须通过自定义网络的内置 DNS 解析实现，该机制已被废弃多年，未来可能被完全移除，无任何合规使用场景。

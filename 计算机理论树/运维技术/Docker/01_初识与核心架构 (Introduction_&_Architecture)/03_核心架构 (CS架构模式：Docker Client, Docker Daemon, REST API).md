# 1.3 Docker 核心系统架构与基本实体

要真正掌握 Docker 的运行机制，必须透彻理解其底层的拓扑架构以及贯穿整个生命周期的核心抽象概念。Docker 采用了典型的 **C/S（Client-Server，客户端-服务端）架构模式**，并围绕三大核心实体构建其软件交付的生态系统。

### 1. C/S 架构模型分析

Docker 系统并非一个单一的单体程序，而是由多个协同工作的组件构成的分布式架构。其核心交互组件如下：

*   **Docker Daemon (守护进程)**：
    系统架构中的 Server 端，通常以后台服务（`dockerd`）的形式运行于宿主机。它负责接收和处理来自客户端的请求，并全面接管 Docker 对象的生命周期管理（包括构建、运行、分发镜像与容器，以及管理网络和数据卷）。
*   **Docker Client (客户端)**：
    用户与 Docker 交互的主要接口界面（通常为 CLI 命令行工具，如 `docker` 命令）。客户端自身不执行容器运行逻辑，而是将用户指令封装为请求发送至守护进程。
*   **REST API (通信接口)**：
    Client 与 Daemon 之间的通信桥梁。两者通过 UNIX Socket（默认，适用于同主机通信）或网络接口（TCP，适用于远程控制）利用 RESTful API 进行交互。这种解耦设计使得我们可以轻易地使用第三方工具或远程客户端来调度 Docker Daemon。

### 2. 三大核心实体抽象

在 Docker 的理论体系中，所有的资源组织形式最终都会映射到以下三个核心实体：

#### 2.1 镜像 (Image)：静态的只读模板
*   **学术定义**：镜像是一个极度精简的、包含应用程序及其所有运行时依赖（包括基础操作系统层、第三方库、环境变量及配置文件）的**只读模板**。
*   **底层机制**：镜像采用分层构建技术（基于 UnionFS 联合文件系统）。每一条构建指令（如安装软件、复制文件）都会生成一个新的只读层。这种增量结构极大提升了存储效率与拉取速度。
*   **面向对象类比**：在软件工程中，镜像等同于面向对象编程（OOP）中的 **“类 (Class)”**，它是构建可执行实例的基础蓝图。

#### 2.2 容器 (Container)：动态的运行时实例
*   **学术定义**：容器是镜像的**可运行实例**。它提供了一个被隔离的、拥有独立生命周期的执行环境。
*   **底层机制**：当基于镜像启动一个容器时，Docker 引擎会在镜像栈的顶部附加一个**可写层 (Read-Write Layer)**。容器内产生的一切状态变更（新建文件、修改数据）均发生在此可写层中，而底层的镜像层保持绝对的只读状态。一旦容器被销毁，该可写层亦随之消失（除非采用数据卷进行持久化）。
*   **面向对象类比**：容器等同于面向对象编程中由“类”实例化而来的 **“对象 (Object)”**。一个镜像可以同时派生出无数个相互隔离的容器。

#### 2.3 仓库 (Repository / Registry)：中心化分发系统
*   **学术定义**：用于集中存储、版本管理和分发 Docker 镜像的服务系统。
*   **概念辨析**：
    *   **Registry（注册服务器）**：管理仓库的实际服务器基础设施，例如官方公共平台 Docker Hub，或企业内部署的私有仓库（如 Harbor）。
    *   **Repository（仓库）**：Registry 中用于区分不同软件的具体集合。例如 `nginx` 是一个仓库，内部包含了 `nginx:1.21`、`nginx:latest` 等不同标签（Tag，即版本）的镜像。

### 3. 架构协同工作流示例：`docker run`

为了直观理解上述组件的协同配合，我们解析输入一行指令背后的完整学术路径：
当用户在客户端输入 `docker run nginx:latest` 时：
1.  **Client** 将指令转化为 REST API 请求，发送至宿主机的 **Daemon**。
2.  **Daemon** 检查本地是否存在 `nginx:latest` **Image**。
3.  若本地无此镜像，Daemon 会向远程 **Registry** 发起请求，下载该镜像。
4.  下载完成后，Daemon 基于该镜像分配隔离的网络与进程空间，在其顶层附加可写层，最终实例化并启动一个 **Container**。

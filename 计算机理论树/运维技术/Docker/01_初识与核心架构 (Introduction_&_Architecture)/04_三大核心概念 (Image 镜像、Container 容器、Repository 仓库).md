在 Docker 的工程实践中，无论是构建、分发还是运行，本质上都是围绕着 **镜像（Image）**、**容器（Container）** 和 **仓库（Repository）** 这三大核心实体展开的逻辑状态转换。

### 1. 镜像 (Image)：不可变的基础设施蓝图

如果说 Docker 的目标是标准化交付，那么“镜像”就是这个标准化的载体。

*   **学术定义**：镜像是一个极度精简的、包含应用程序及其所有运行时依赖（包括基础操作系统层、第三方库、环境变量及配置文件）的**只读（Read-Only）模板**。
*   **核心机制：分层架构与 UnionFS**
    镜像并非一个单一的庞大文件，而是通过**联合文件系统（Union File System, UnionFS）** 堆叠而成的一组相互独立的文件系统层。
    *   **基础层（Base Layer）**：通常是极简的操作系统环境（如 Alpine、Ubuntu 基础环境）。
    *   **堆叠层（Stacked Layers）**：每一次环境变更（如 `执行 apt-get 安装 Nginx`，或 `拷贝应用代码到指定目录`）都会在基础层之上生成一个新的只读层。
*   **工程优势**：这种分层设计带来了极高的复用率。如果两个不同的镜像都基于相同的 Ubuntu 基础层，宿主机的磁盘和内存中只需保留一份该基础层的数据，极大地节省了存储空间并加速了网络传输。

### 2. 容器 (Container)：隔离的运行时实例

镜像是静态的图纸，而容器则是将图纸变为现实的动态实体。

*   **学术定义**：容器是基于镜像实例化而来的、拥有独立生命周期和隔离运行环境的**进程级实例**。
*   **核心机制：可写层（Read-Write Layer）与写时复制（CoW）**
    *   **可写层的附加**：当 Docker 引擎基于镜像启动容器时，会在该镜像的只读层栈的顶部，动态附加一个**薄的“可写层”（Container Layer）**。
    *   **写时复制（Copy-on-Write）**：容器运行期间产生的所有状态变更（新建文件、修改配置）均发生在此可写层中。如果容器需要修改底层只读镜像中的文件，Docker 会先将该文件从只读层**复制**到最顶部的可写层中，再进行修改。底层镜像始终保持“不可变（Immutable）”特性。
*   **生命周期约束**：由于数据存储在顶部的可写层中，**一旦容器被销毁，该可写层连同内部的数据会随之永久丢失**。这也是为什么在后续章节中，我们必须引入“数据卷（Volumes）”来实现持久化的原因。

### 3. 仓库 (Repository)：资产的集中化管理与分发网络

镜像构建完成后，需要通过网络进行分发，这就引入了仓库机制。

*   **学术定义**：仓库是用于集中存储、进行版本控制，并提供 Docker 镜像上传与下载服务的**分发中枢**。
*   **层级概念辨析（严格区分）**：
    在日常交流中，很多人会将 Registry 和 Repository 混为一谈，但在学术严谨的语境中，它们存在清晰的层级关系：
    1.  **Registry（注册服务器/平台）**：它是提供服务的物理或逻辑节点。例如官方的公共平台 **Docker Hub**，或者企业内部署的私有 Registry（如 **Harbor**）。一个 Registry 中可以包含成千上万个 Repository。
    2.  **Repository（镜像仓库）**：它是某一个具体软件镜像的逻辑集合，通常对应一个特定的应用。例如 `nginx` 就是一个 Repository。
    3.  **Tag（标签/版本）**：在一个 Repository 内部，通过 Tag 来区分该软件的不同版本（如 `nginx:1.21`，`nginx:latest`）。
*   **寻址规范**：当我们执行 `docker pull ubuntu:20.04` 时，其完整的隐式路由其实是：`<Registry域名>/<命名空间>/<Repository>:<Tag>`（即从默认的 Docker Hub 下载 library 下的 ubuntu 仓库中的 20.04 标签镜像）。

### 4. 三大概念的流转闭环

这三者共同构成了一套完美的 DevOps 流水线闭环：
1.  开发者通过 `docker build` 将代码打包为静态的 **Image**。
2.  通过 `docker push` 将该 Image 上传至远程的 **Repository**。
3.  运维环境通过 `docker pull` 从 Repository 将 Image 拉取到本地服务器。
4.  通过 `docker run` 基于 Image 启动隔离的动态 **Container** 提供服务。

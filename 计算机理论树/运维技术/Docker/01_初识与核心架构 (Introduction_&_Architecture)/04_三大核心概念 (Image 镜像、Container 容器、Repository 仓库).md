
Docker 的整个技术体系、操作逻辑与工程化实践，均围绕 **镜像（Image）、容器（Container）、仓库（Repository）** 三大核心概念构建，三者构成了 Docker 应用交付的完整闭环，是必须完全掌握的核心知识点。

### 1.4.1 镜像（Image）

Docker 镜像是 **只读的、标准化的分层文件系统模板** ，是容器运行的静态基础，包含了应用运行所需的所有内容：应用代码、运行时环境、系统依赖库、环境变量、启动配置、默认参数等。

#### 核心特性

1. **只读不可变性**：镜像一旦构建完成，其分层内容与哈希值完全固定，无法直接修改，任何内容变更均需通过重新构建镜像实现，不可变性是镜像跨环境一致性的核心保障；
2. **分层存储结构**：基于联合文件系统（UnionFS）实现，镜像由多个只读的增量分层叠加而成，每一层仅记录相对于上一层的文件变更，不同镜像可共享相同的底层分层，极大降低镜像的存储占用与网络传输带宽；
3. **唯一标识**：镜像通过 64 位十六进制哈希值（Image ID）唯一标识，该哈希值由镜像的内容计算生成，内容不变则 ID 不变，可用于镜像的完整性校验与版本管理。

### 1.4.2 容器（Container）

容器是 **镜像运行时的实例** ，是一个独立的、被严格隔离与资源限制的进程运行环境，对应宿主机上一个或多个被 Linux 内核 Namespace、Cgroups 机制管控的进程。

#### 核心特性

1. **完整生命周期可管理**：具备创建、启动、停止、暂停、重启、销毁的全生命周期管理能力，与普通进程的生命周期管理逻辑一致；
2. **读写分层机制**：容器基于镜像的只读分层，在顶部创建一个专属的可写层，容器运行过程中的所有文件写入、数据修改、日志存储等操作，均保存在该可写层中；可写层的生命周期与容器完全绑定，容器销毁时，可写层同步删除，且不会对底层只读镜像产生任何修改；
3. **强隔离与资源限制**：通过 Namespace 实现进程、网络、挂载点、主机名、用户、进程间通信的六大环境隔离，通过 Cgroups 实现 CPU、内存、磁盘 IO、网络带宽等资源的硬性限制，保证容器间互不干扰，避免资源抢占；
4. **轻量级运行**：共享宿主机操作系统内核，无完整 Guest OS 的启动开销，启动仅需加载应用进程，启动速度可达毫秒级，资源占用远低于虚拟机。

#### 与镜像的核心关系

镜像与容器的关系，完全等价于面向对象编程中**类（Class）与实例（Instance）** 的关系：镜像是静态的定义模板，容器是基于该模板生成的动态运行实例；一个镜像可创建任意多个相互独立、互不干扰的容器实例。

### 1.4.3 仓库（Repository）

Docker 仓库是**集中存储、分发、版本管理 Docker 镜像的标准化服务**，是镜像跨主机、跨环境共享与交付的核心载体，是 Docker「一次构建，到处运行」能力的关键支撑。

#### 核心分类与概念

1. **仓库分类**：
    
    - 公共仓库：面向全网开放的镜像托管服务，主流包括 Docker 官方的 Docker Hub、阿里云镜像仓库、腾讯云镜像仓库等，提供大量官方维护的基础镜像（如 Ubuntu、Nginx、MySQL、Python 等）与社区贡献镜像；
    - 私有仓库：企业或个人自行搭建的镜像托管服务，用于内部业务镜像的安全存储、权限管控与分发，主流实现包括 Docker 官方 Registry、Harbor 企业级仓库等。
    
2. **核心关联概念**：
    
    - Registry（注册中心）：仓库的底层托管服务，一个 Registry 可包含多个 Repository；
    - Tag（标签）：用于标识同一仓库内镜像的不同版本，格式为`仓库名:标签名`，如`nginx:1.27.0`、`ubuntu:22.04`，默认标签为`latest`，是镜像版本管理的核心方式。
    

#### 核心价值

仓库实现了镜像的版本化管理、跨环境分发与团队共享，是 DevOps 持续集成 / 持续部署流水线的核心环节，支撑从代码提交到镜像构建、推送、部署的全自动化流程。


OCI（Open Container Initiative，开放容器倡议）是 2015 年由 Docker、CoreOS、Google、RedHat 等容器领域核心厂商联合发起，隶属于 Linux 基金会的开源项目，核心目标是制定容器技术的开放行业标准，解决容器技术碎片化、厂商锁定问题，保证容器的跨平台可移植性与跨引擎互操作性。

Docker 是 OCI 标准的核心发起方与核心贡献者，其核心技术体系已完全遵循 OCI 标准实现，是容器技术行业标准化的核心基石。

### 1.5.1 OCI 两大核心规范

OCI 标准定义了容器技术的两大核心规范，覆盖了容器从构建到运行的全生命周期标准化要求：

1. **运行时规范（Runtime Specification）**：定义了容器运行时的标准化行为与接口，包括容器生命周期管理、隔离环境配置、Namespace 与 Cgroups 的实现标准、容器文件系统 Bundle 格式，以及容器创建、启动、停止、销毁等操作的标准化流程，保证符合规范的容器可在任何兼容 OCI 标准的运行时中正常运行。
2. **镜像规范（Image Specification）**：定义了容器镜像的标准化格式，包括镜像的分层结构、内容哈希计算规则、配置文件格式、镜像分发协议等标准，保证符合规范的镜像可在任何兼容 OCI 标准的容器引擎中正常拉取、解压与运行，彻底解决镜像的跨引擎、跨平台兼容性问题。

### 1.5.2 OCI 标准核心实现组件演进

#### 1.5.2.1 runc

runc 是 OCI 运行时规范的官方参考实现，是一个轻量级的 CLI 工具，可直接基于 OCI 规范创建和运行容器，底层直接调用 Linux 内核的 Namespace、Cgroups、SELinux 等原生接口，是容器运行时的最底层实现。

- 演进历史：2015 年 Docker 将自身的底层容器运行时代码捐赠给 OCI，成为 runc 项目的初始代码基础；
- 核心定位：runc 不提供常驻守护进程，仅负责容器的底层创建、运行与销毁，目前所有兼容 OCI 标准的容器引擎（Docker、containerd、Kubernetes 等）均基于 runc 实现容器的底层管控。

#### 1.5.2.2 containerd

containerd 是工业级的容器运行时守护进程，负责管理容器的完整生命周期，包括镜像拉取与存储、容器创建与运行、网络与存储管理、镜像元数据管理等核心能力。

- 演进历史：最初为 Docker 架构中的内部组件，作为 Docker Daemon 与 runc 之间的中间层，负责容器的底层生命周期管理；2017 年 Docker 将 containerd 捐赠给 CNCF（云原生计算基金会），成为独立的开源项目，目前已毕业成为 CNCF 顶级项目，是容器运行时的行业事实标准；
- 核心定位：Docker Daemon 通过调用 containerd 完成容器的全生命周期管理，同时 Kubernetes 可通过 CRI（容器运行时接口）直接对接 containerd 作为集群的容器运行时，替代 Docker Engine；containerd 向上承接上层引擎的管控指令，向下对接 runc 完成容器的底层运行操作，是现代容器架构中的核心中间层组件。

# 1.5 OCI 标准：Docker 的架构解耦与 runc、containerd 演进简史

在我们即将结束第一章“宏观架构”的学习之前，必须探讨一段深刻影响现代云原生生态的历史：**开放容器标准（OCI）的诞生与 Docker 的架构解耦**。

这不仅是面试中考察架构视野的高频考点，更是理解“为什么 Kubernetes 后来宣布抛弃 Docker，但我们依然还在用 Docker 镜像”这一经典误解的核心钥匙。

### 1. 历史背景：从“一家独大”到“标准之争”

在 2013-2015 年间，Docker 是一个庞大的**单体架构（Monolithic）**。从接收客户端 API 请求、下载镜像、解压镜像，到调用 Linux 底层的 Namespace 和 Cgroups 启动容器，所有的逻辑全部强耦合在一个名叫 `dockerd` 的守护进程中。

随着 Docker 几乎垄断了容器市场，整个业界（包括 Google、RedHat 等巨头）开始感到不安。如果容器的底层技术完全由一家商业公司（Docker Inc.）掌控，这将对云计算的未来产生巨大的生态绑定风险。期间，CoreOS 公司甚至推出了竞争产品 `rkt` 来挑战 Docker 的霸主地位。

为了平息“容器战争”，防止技术分裂，2015 年，在 Linux 基金会的牵头下，Docker 公司与各大巨头握手言和，共同成立了 **开放容器倡议组织（Open Container Initiative, OCI）**。

### 2. OCI 核心规范：统一“集装箱”的尺寸

OCI 成立的唯一目的，就是制定一套厂商中立的**容器开源标准**。这就如同制定现实世界中“标准集装箱”的长宽高尺寸一样，只要符合标准，任何船只、港口都可以装卸。

OCI 主要定义了两大核心规范：
1.  **镜像规范 (Image Specification)**：规定了容器镜像的文件格式、打包方式和元数据配置。只要符合该标准的镜像（即 OCI 镜像），任何容器引擎都必须能识别并运行它。我们今天常说的“Docker 镜像”，实际上早就已经是 OCI 镜像了。
2.  **运行时规范 (Runtime Specification)**：规定了容器引擎应该如何解压镜像，以及如何与操作系统内核交互来控制容器的生命周期（创建、启动、停止等）。

### 3. Docker 的架构“大拆解”与核心组件

为了响应 OCI 标准，Docker 公司展现了极高的格局，将其庞大的单体架构进行了**彻底的解耦拆分**，并将其底层的核心组件捐献给了开源基金会。

现代 Docker 引擎的底层架构由上至下被拆分为以下几个核心独立组件：

*   **`dockerd` (Docker Daemon)**
    *   **角色**：最上层的管理者。
    *   **职责**：负责处理 REST API 请求、构建镜像 (Build)、管理网络和数据卷等高级功能。
*   **`containerd` (高级容器运行时)**
    *   **角色**：承上启下的调度核心（已由 Docker 捐赠给 CNCF 基金会）。
    *   **职责**：它不再负责网络和构建逻辑，而是专职负责**镜像的传输与存储管理**，并向下调度执行容器运行指令。它是目前整个云原生生态中最核心的通用容器运行时。
*   **`runc` (低级容器运行时)**
    *   **角色**：OCI 运行时规范的官方**参考实现**（由 Docker 抽取并捐献）。
    *   **职责**：它是一个轻量级的命令行工具，**专门负责与操作系统内核打交道**。它接收文件系统和配置，直接调用 Linux Namespace 和 Cgroups 来真正地启动和隔离出那个“容器进程”。
*   **`containerd-shim` (垫片进程)**
    *   **角色**：进程解耦器。
    *   **职责**：每次 `runc` 启动完一个真正的容器进程后，`runc` 自身就会退出。此时，`containerd-shim` 会接管该容器进程，成为它的父进程。
    *   **工程意义**：因为有了这个垫片进程，即使你升级或重启了宿主机上的 `dockerd` 或 `containerd` 服务，正在运行的业务容器也**绝对不会受到任何影响而宕机**。

### 4. 学术总结与对现代架构的启发

通过上述拆解，现代 Docker 启动一个容器的完整调用链变为了：
**`Docker Client` -> `dockerd` -> `containerd` -> `containerd-shim` -> `runc` -> 启动容器进程。**

**解答世纪误解：Kubernetes 抛弃 Docker 了吗？**
准确地说，Kubernetes 只是在内部去掉了对庞大的 `dockerd` 的直接依赖（因为 K8s 不需要 Docker 的网络和构建功能），转而直接调用更加轻量、专职的底层引擎 **`containerd`**。
由于 `containerd` 和 Docker 本质上使用的是完全相同的 OCI 标准，所以**你用 Docker 构建的镜像，依然可以完美地跑在 K8s 上，一切都没有改变**。

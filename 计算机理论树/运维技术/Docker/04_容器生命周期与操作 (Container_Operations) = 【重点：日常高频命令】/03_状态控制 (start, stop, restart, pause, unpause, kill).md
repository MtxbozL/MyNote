
状态控制命令用于实现容器生命周期各状态之间的转换，是日常运维的高频操作，需严格区分各命令的底层实现、信号机制与适用场景，避免业务异常。

### 3.1 docker start：启动容器

- 作用：启动已创建（Created）或已停止（Exited）的容器，使其进入 Running 状态；
- 语法：`docker start [OPTIONS] CONTAINER [CONTAINER...]`
- 核心参数：
    
    - `-i/--interactive`：以交互模式启动，附加容器的标准输入到当前终端；
    - `-a/--attach`：附加容器的标准输出与错误流到当前终端，实时显示启动日志；
    
- 底层逻辑：重新加载容器的原有 Overlay2 文件系统、Namespace 隔离环境与 Cgroups 资源配置，启动容器的 1 号进程，不会重建容器，可复用容器可写层的所有历史数据；
- 适用场景：已停止的容器重新启动、配置变更后恢复业务运行。

### 3.2 docker stop：优雅停止容器

- 作用：优雅终止运行中的容器，使其进入 Exited 状态，是生产环境停止容器的唯一推荐方式；
- 语法：`docker stop [OPTIONS] CONTAINER [CONTAINER...]`
- 核心参数：`-t/--time`，设置优雅退出的超时等待时间，单位为秒，默认 10 秒；
- 底层实现（Linux 信号机制）：
    
    1. 首先向容器的 1 号进程发送**SIGTERM（15）** 信号，通知业务进程优雅退出，业务进程可捕获该信号，完成连接关闭、数据持久化、资源释放等收尾操作；
    2. 若超过设置的超时时间，进程仍未退出，再向 1 号进程发送**SIGKILL（9）** 信号，强制终止进程；
    
- 生产规范：停止核心业务容器时，需根据业务特性设置合理的超时时间，避免业务未完成收尾就被强制终止，引发数据丢失或事务异常。

### 3.3 docker kill：强制终止容器

- 作用：立即强制终止运行中的容器，使其进入 Exited 状态，仅适用于紧急故障场景；
- 语法：`docker kill [OPTIONS] CONTAINER [CONTAINER...]`
- 核心参数：`-s/--signal`，指定发送给容器 1 号进程的信号，默认发送**SIGKILL（9）** 信号；
- 底层逻辑：直接向容器 1 号进程发送指定信号，SIGKILL 信号无法被进程捕获，会立即终止进程运行，无任何优雅退出的过程；
- 风险警示：**生产环境非紧急情况严禁使用 docker kill**，强制终止会导致业务进程未完成数据持久化、连接异常断开、事务回滚失败等问题，仅当容器卡死、无法通过 docker stop 正常停止时使用。

### 3.4 docker restart：重启容器

- 作用：重启运行中或已停止的容器，等价于`docker stop + docker start`的复合操作；
- 语法：`docker restart [OPTIONS] CONTAINER [CONTAINER...]`
- 核心参数：`-t/--time`，设置停止前的优雅退出超时时间，默认 10 秒；
- 底层逻辑：先按优雅停止规则终止容器，再重新加载容器配置并启动 1 号进程，重启后容器的 IP 地址、端口映射、数据卷挂载等配置保持不变，仅容器内的进程状态重置；
- 适用场景：容器内配置变更后生效、业务异常无响应时的自愈操作。

### 3.5 docker pause /docker unpause：暂停 / 恢复容器

- 作用：冻结 / 恢复容器内的所有进程，实现运行状态的暂停与恢复，不会终止进程；
- 语法：`docker pause CONTAINER [CONTAINER...]` / `docker unpause CONTAINER [CONTAINER...]`
- 底层原理：基于 Linux 内核的**Cgroups freezer 子系统**实现，将容器内的所有进程冻结，暂停 CPU 调度，进程的内存数据、文件句柄、网络连接完全保留，仅释放 CPU 资源；unpause 操作会解除冻结，恢复进程的正常调度；
- 核心特性：暂停操作是瞬时完成的，恢复速度远快于容器重启，不会中断已建立的网络连接；
- 适用场景：容器运行状态快照、故障现场冻结、业务临时暂停调试、数据备份时的 IO 冻结。

## 容器生命周期核心状态机

容器的本质是被 Linux Namespace 隔离、Cgroups 资源限制的进程集合，其生命周期完全绑定容器内 1 号进程的运行状态，核心分为 5 个稳定状态，所有操作均围绕状态转换实现：

1. **Created（已创建）**：容器隔离环境、配置、文件系统已初始化完成，但进程未启动，未占用 CPU 与内存资源；
2. **Running（运行中）**：容器 1 号进程已启动，Namespace 隔离生效，Cgroups 资源限制已加载，正常执行业务逻辑；
3. **Paused（已暂停）**：容器内所有进程被冻结，暂停 CPU 调度，内存数据与运行状态完整保留，仅释放 CPU 资源；
4. **Exited（已停止）**：容器 1 号进程已退出，隔离环境已关闭，仅保留可写层数据与元数据，可重新启动；
5. **Deleted（已销毁）**：容器已被永久删除，可写层、元数据、隔离配置全部释放，数据无法恢复。

## 01 创建与运行

容器的创建与运行是生命周期的起点，核心命令为`docker create`（仅创建）与`docker run`（创建 + 启动，复合命令，日常最高频使用）。

### 1.1 命令底层执行逻辑

`docker run`是 Docker 最核心的复合命令，执行时会按顺序完成以下全链路操作，完整对应 Docker C/S 架构与底层隔离机制：

1. 解析指令参数，通过 REST API 将请求发送至 Docker Daemon；
2. 校验本地是否存在指定镜像，无本地镜像则按配置的镜像源执行`docker pull`；
3. 基于镜像创建容器的只读分层（lowerdir）与可写层（upperdir），通过 Overlay2 构建合并文件系统视图；
4. 创建六大 Namespace 隔离环境，构建容器独立的运行视图；
5. 配置 Cgroups 规则，设置容器的 CPU、内存等资源限制；
6. 配置网络栈、端口映射、数据卷挂载等环境参数；
7. 启动容器，在隔离环境中执行镜像预设的启动命令，创建容器 1 号进程；
8. 按参数配置决定是否附加终端至当前 Shell，返回容器执行结果。

### 1.2 核心语法与必用参数

```bash
docker run [OPTIONS] IMAGE [COMMAND] [ARG...]
```

本节重点讲解核心参数的底层逻辑、适用场景与生产规范，覆盖原目录指定的所有核心参数，同时补充生产环境必用的高可用配置。

#### 1.2.1 容器唯一标识：--name

- 作用：为容器指定全局唯一的名称，同一宿主机内不可重复，替代随机生成的容器 ID 与名称，作为容器的运维标识；
- 底层逻辑：名称会写入 Docker Daemon 的容器元数据库，作为容器的唯一别名，所有运维命令均可通过名称替代容器 ID；
- 生产规范：**所有业务容器必须显式指定 --name**，禁止使用随机名称，便于运维识别、监控与排障，避免容器管理混乱。

补充关联参数`--hostname`：设置容器 UTS Namespace 内的主机名，默认与容器名称一致，仅在容器内生效，不影响宿主机与外部网络访问。

#### 1.2.2 运行模式：-d/--detach 后台运行

- 作用：容器启动后进入后台运行模式，释放当前终端，仅返回容器 ID，容器生命周期与终端无关；
- 底层逻辑：Docker Daemon 启动容器后，将容器的标准流与当前终端解绑，容器在后台作为守护进程运行；
- 生产规范：**生产环境业务容器必须使用 - d 参数**，避免终端关闭导致容器退出，仅临时测试场景可使用前台运行模式。

#### 1.2.3 交互模式：-it 伪终端分配

`-it`是`-i/--interactive`与`-t/--tty`的组合参数，仅适用于需要与容器内进行交互的场景，二者必须配合使用。

- `-i`：保持容器的标准输入（stdin）开启，即使未附加终端，也可接收输入指令；
- `-t`：为容器分配一个伪 TTY 终端，绑定容器 1 号进程的标准输入输出流，提供交互式 Shell 的终端环境；
- 底层逻辑：`-t`在容器的 PID Namespace 中创建伪终端设备，`-i`将宿主机终端的输入流与容器伪终端绑定，实现与容器内环境的交互；
- 典型用法：`docker run -it ubuntu:22.04 /bin/bash`，启动容器并进入交互式 Shell 终端；
- 注意事项：`-it`不可与`-d`同时用于常规业务容器，后台运行的非交互式进程无法接收终端输入，会导致容器启动后立即退出。

#### 1.2.4 自动清理：--rm 退出即销毁

- 作用：容器退出后，自动执行容器销毁操作，删除容器的可写层、元数据、网络配置与匿名数据卷；
- 底层逻辑：Docker Daemon 监听容器的退出事件，一旦容器进入 Exited 状态，立即触发销毁流程，无需手动执行`docker rm`；
- 适用场景：仅适用于临时测试、一次性任务、命令行工具等场景，例如`docker run --rm alpine ping baidu.com`；
- 风险警示：**生产环境严禁使用 --rm 参数**，容器故障退出后会被立即销毁，无法保留运行现场、日志与可写层数据，彻底失去故障排查的核心依据。

#### 1.2.5 生产级高可用配置：--restart 重启策略

- 作用：定义容器退出后的自动重启规则，是生产环境保障业务可用性的核心配置，弥补容器进程级故障的自愈能力；
- 底层逻辑：Docker Daemon 持续监听容器的退出状态，符合重启条件时，自动执行容器重启操作，无需人工干预；
- 重启策略与适用场景：

| 策略值                        | 核心规则                                                          | 生产适用场景                            |
| -------------------------- | ------------------------------------------------------------- | --------------------------------- |
| `no`                       | 默认值，容器退出后永不自动重启                                               | 临时测试容器、一次性任务容器                    |
| `on-failure[:max-retries]` | 仅当容器退出码非 0（异常退出）时自动重启，可设置最大重试次数                               | 业务应用容器，可避免无限重启死循环，便于故障定位          |
| `always`                   | 无论容器退出码是什么，始终自动重启；即使容器被手动停止，仅当 Docker Daemon 重启时也会自动启动        | 核心基础设施容器（如 Nginx、MySQL），需始终保持运行   |
| `unless-stopped`           | 除非容器被手动停止，否则无论退出状态如何，始终自动重启；Docker Daemon 重启后也会自动启动，手动停止的容器除外 | 生产环境绝大多数业务容器的**推荐策略**，兼顾可用性与运维可控性 |

### 1.3 容器 1 号进程的核心约束

容器的生命周期完全绑定其 1 号进程（PID 1），1 号进程启动成功则容器进入 Running 状态，1 号进程退出则容器立即进入 Exited 状态，这是容器运行的核心底层规则。

Linux 系统中，1 号进程负责回收僵尸进程、处理信号，容器内的 1 号进程同样具备该特性：若业务进程不作为 1 号进程运行，或 1 号进程无法正确转发 SIGTERM 信号，会导致业务进程无法优雅退出，引发数据丢失、连接异常断开等问题。生产环境需保证容器 1 号进程为业务主进程，而非 Shell 进程，避免信号转发失效。


容器的销毁与清理是生命周期的终点，核心用于释放磁盘空间、清理无用容器，需严格遵循操作规范，避免误删业务容器与数据丢失。

### 7.1 docker rm：容器删除

`docker rm`用于永久销毁容器，释放容器的可写层、元数据、Namespace 隔离环境、网络配置等资源，容器销毁后，可写层内的所有未持久化数据会被永久删除，无法恢复。

#### 7.1.1 核心语法与参数

```bash
docker rm [OPTIONS] CONTAINER [CONTAINER...]
```

- 核心参数：
    
    - `-f/--force`：强制删除运行中的容器，底层会先发送 SIGKILL 信号强制终止容器进程，再执行销毁操作；
    - `-v/--volumes`：同时删除与容器关联的**匿名数据卷**，注意：不会删除命名数据卷；
    - `-l/--link`：删除容器的关联链接配置。
    

#### 7.1.2 底层逻辑与风险规范

- 底层逻辑：执行删除操作时，Docker Daemon 会先校验容器状态，默认仅允许删除已停止（Exited）的容器；销毁时会删除容器的 Overlay2 可写层、元数据、网络配置、Namespace 资源，仅保留持久化到命名数据卷中的数据；
- 风险警示：
    
    1. **生产环境严禁使用 docker rm -f 强制删除运行中的业务容器**，应先通过`docker stop`优雅停止容器，确认数据已备份后，再执行删除操作，避免业务异常中断与数据丢失；
    2. 容器删除后，可写层内的所有未持久化数据会被永久删除，无法恢复，删除前必须确认业务数据已持久化到命名数据卷，并完成备份；
    3. 未加`-v`参数时，匿名数据卷不会被删除，会成为孤儿卷，持续占用磁盘空间，生产环境推荐使用命名数据卷，便于生命周期管理。
    

### 7.2 docker container prune：批量清理无用容器

`docker container prune`用于批量清理宿主机上的无用容器，默认清理所有处于 Exited 状态的容器，是生产环境磁盘空间管控、环境清理的核心工具。

#### 7.2.1 核心语法与参数

```bash
docker container prune [OPTIONS]
```

- 核心参数：
    
    - `--filter`：按条件过滤清理范围，精准控制清理对象，常用过滤规则：
        
        - `until=24h`：仅清理 24 小时之前停止的容器；
        - `label!=key=value`：排除带有指定标签的容器；
        - `status=exited`：仅清理指定状态的容器；
        
    - `-f/--force`：强制清理，不弹出确认提示，适用于自动化定时清理脚本。
    

#### 7.2.2 生产环境最佳实践

1. 配置定时任务执行 prune 命令，避免大量停止的容器占用磁盘空间，同时必须添加过滤条件，**严禁无差别清理**，避免误删需要保留的故障现场容器；
2. 核心业务容器添加固定的业务标签，通过`label`过滤规则排除在清理范围外，保障业务与排障数据的安全性；
3. 清理前先通过`docker container ls -a`查看所有容器的状态与名称，确认清理范围无误后，再执行 prune 操作，避免误删。


## 04_Pipeline (流水线)

### 01_原理 (RTT 往返时间优化、批量处理提升吞吐量)

#### 一、背景：RTT 性能瓶颈

Redis 是典型的 CS 架构，单条命令的完整执行流程为：

`客户端发送命令 -> 服务器接收命令 -> 执行命令 -> 返回结果给客户端`

该流程的网络耗时为**RTT（Round Trip Time，往返时间）**，即使 Redis 单条命令执行仅需几微秒，但若客户端循环执行 N 条命令，会产生 N 次 RTT，性能瓶颈完全集中在网络开销上。

例如：RTT 为 1ms 时，单线程循环执行 1000 条命令，仅 RTT 就耗时 1 秒，而 Redis 执行 1000 条命令仅需几十微秒，99% 以上的时间都浪费在网络往返上。

#### 二、Pipeline 核心原理

Pipeline 是**客户端实现的批量命令打包机制**，核心逻辑是：将多条 Redis 命令打包，一次性发送给 Redis 服务器，服务器按顺序执行完所有命令后，一次性将所有结果返回给客户端，将 N 次 RTT 减少为 1 次 RTT，极大降低网络开销，提升吞吐量。

- 核心特性：Pipeline 是客户端侧的优化，Redis 服务器端无需任何特殊支持，仅需按顺序执行收到的命令，批量返回结果，与普通命令执行无差异；
- 与原生批量命令（MSET/MGET）的核心差异：

| 特性   | Pipeline               | 原生批量命令               |
| ---- | ---------------------- | -------------------- |
| 原子性  | 不保证，命令逐条执行，期间可能被其他命令打断 | 保证，单条命令原子性执行，期间不会被打断 |
| 命令类型 | 支持任意类型的命令混合打包          | 仅支持同类型的批量命令          |
| 实现侧  | 客户端实现                  | 服务器端原生实现             |
| 灵活性  | 高，支持任意逻辑的命令组合          | 低，仅支持固定批量操作          |

#### 三、核心优势

1. **极致降低 RTT 开销，提升吞吐量**：批量执行 N 条命令，RTT 从 N 次降为 1 次，高延迟网络环境下，吞吐量可提升几十倍甚至上百倍；
2. **降低系统调用开销**：客户端每次发送 / 接收命令都需调用操作系统的`read/write`系统调用，涉及用户态 / 内核态切换，开销极大；Pipeline 将多次系统调用合并为 1 次，大幅降低 CPU 开销；
3. **灵活性高**：支持任意类型命令的混合打包，适配所有批量操作场景。

#### 四、生产规范与限制

1. Pipeline 打包的命令数量不可无限大，单次打包建议控制在 1000 条以内，否则会导致数据包过大，占用过多网络带宽，同时服务器端需缓存大量执行结果，占用过多内存；
2. Pipeline 不保证原子性，需要原子性的场景优先使用 Lua 脚本；
3. Pipeline 仅能用于同一个 Redis 连接，不可跨连接使用；Redis Cluster 集群模式下，Pipeline 内的所有 Key 必须位于同一个节点，否则无法执行；
4. 生产环境批量数据导入、批量查询场景，必须使用 Pipeline，禁止循环执行单条命令，避免 RTT 成为性能瓶颈。

Redis Pub/Sub（发布订阅）是内置的消息通信模式，基于频道（Channel）实现消息的广播通信，解耦消息生产者（发布者）与消费者（订阅者）：发布者向指定频道发布消息，所有订阅该频道的消费者都会实时收到消息，是典型的广播模式。

#### 一、核心机制与频道订阅

1. **核心概念**
    
    - 频道（Channel）：消息的载体，无需提前创建，发布 / 订阅时自动创建，无订阅者时自动销毁；
    - 发布者（Publisher）：向指定频道发送消息的客户端；
    - 订阅者（Subscriber）：订阅一个或多个频道，接收频道消息的客户端。
    
2. **核心命令**

|命令|语法|核心功能|
|---|---|---|
|SUBSCRIBE|`SUBSCRIBE channel [channel ...]`|订阅一个或多个频道，执行后客户端进入订阅状态，仅可执行订阅相关命令，收到消息时返回三元组：`message`、频道名、消息内容|
|PUBLISH|`PUBLISH channel message`|向指定频道发布消息，返回值为收到该消息的订阅者数量|
|UNSUBSCRIBE|`UNSUBSCRIBE [channel ...]`|取消订阅指定频道，无参数则取消所有频道的订阅|
|PSUBSCRIBE|`PSUBSCRIBE pattern [pattern ...]`|模式匹配订阅，支持通配符匹配多个频道，如`PSUBSCRIBE news.*`订阅所有以`news.`开头的频道|
|PUNSUBSCRIBE|`PUNSUBSCRIBE [pattern ...]`|取消模式匹配订阅，无参数则取消所有模式订阅|

3. **底层实现原理**
    
    Redis Pub/Sub 底层基于字典实现，维护一个**频道 -> 订阅者链表**的映射表：
    
    - 发布者向频道发布消息时，Redis 查找该频道对应的订阅者链表，遍历链表将消息发送给所有订阅者；
    - 模式订阅单独维护一个模式链表，发布消息时同时匹配所有模式，匹配成功则发送给对应的订阅者。
    

#### 二、核心局限与消息丢失问题

Pub/Sub 的核心缺陷是**极高的消息丢失风险**，无任何消息可靠性保证，具体场景如下：

1. **无持久化能力，发布即丢弃**
    
    Pub/Sub 的消息是实时转发的，不会持久化到磁盘，也不会缓存：若频道无订阅者，发布的消息会直接丢弃；若订阅者离线，离线期间发布的消息会永久丢失，订阅者重新上线后无法回溯。
    
2. **无 ACK 确认机制，无重试能力**
    
    Pub/Sub 无消息 ACK 确认机制，Redis 发送消息后，无论消费者是否成功接收、处理，都不会重试；消费者处理消息时崩溃、网络断开，消息会直接丢失，无法保证消息至少消费一次。
    
3. **消费者处理能力不足导致消息丢失**
    
    Redis 向消费者发送消息时，若消费者的 TCP 缓冲区满了，会导致消息发送失败，直接丢弃；同时消费者处理速度过慢，会导致客户端缓冲区溢出，连接断开，全量消息丢失。
    
4. **无消息堆积能力**
    
    Redis 不会为消费者缓存未消费的消息，发布的消息仅会实时转发，消费者处理不过来，消息直接丢弃，无法堆积，完全不适合高并发异步消息场景。
    
5. **集群模式适配性极差**
    
    Redis Cluster 集群模式下，发布者向某个节点发布消息，该节点会将消息广播给集群内所有节点，所有节点的订阅者都会收到消息，会导致集群内大量的跨节点消息广播，占用极高的网络带宽，性能极差，不适合集群大规模使用。
    

#### 三、与 Stream 的对比

Stream 是 Redis 5.0 + 新增的流数据类型，专门解决 Pub/Sub 的所有核心缺陷，是 Redis 官方推荐的消息队列解决方案，核心对比如下：

|特性|Pub/Sub|Stream|
|---|---|---|
|消息持久化|无，消息实时转发，不持久化，丢失后无法恢复|支持，消息持久化到 RDB/AOF，重启后不会丢失，支持任意时间回溯消费|
|消息堆积能力|无，无订阅者直接丢弃，消费者处理不过来也丢弃|有，消息会一直保留，直到手动删除，支持海量消息堆积|
|消费确认机制|无，发送后不管，丢失不重试|有，原生支持 ACK 确认机制，未确认的消息会保留，支持重试消费|
|消费者组|无，仅支持广播模式，所有订阅者都收到全量消息|有，原生支持消费者组，实现消息负载均衡、广播、重试、死信队列等企业级能力|
|消息丢失风险|极高，离线、网络波动、处理慢都会导致消息丢失|极低，支持持久化、ACK、重试，可保证消息至少消费一次|
|集群模式适配|差，发布消息会广播到所有集群节点，带宽占用高，性能差|好，消息存储在对应 Key 的节点，无需全集群广播，性能稳定|
|适用场景|简单的实时广播通知、低可靠的消息推送场景|企业级消息队列、异步任务、可靠消息传输、事件溯源等所有消息场景|

#### 生产规范

1. 仅适合简单的实时广播通知场景（如服务配置更新通知、实时聊天消息推送），**绝对不能用于需要保证消息不丢失的业务场景**（订单、支付、异步任务等）；
2. 生产环境需要保证消息可靠性的场景，必须使用 Stream 类型，或专业的消息队列（Kafka、RabbitMQ），禁止使用 Pub/Sub；
3. 禁止在 Redis Cluster 集群模式下大规模使用 Pub/Sub，避免全集群广播导致的网络带宽耗尽；
4. 订阅者必须保证处理速度足够快，避免 TCP 缓冲区溢出导致连接断开、消息丢失。

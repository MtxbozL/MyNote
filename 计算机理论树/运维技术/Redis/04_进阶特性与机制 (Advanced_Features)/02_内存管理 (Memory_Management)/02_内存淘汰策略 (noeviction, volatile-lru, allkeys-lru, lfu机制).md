内存淘汰策略的触发条件：当 Redis 的内存使用量达到`maxmemory`配置的上限时，Redis 会根据配置的淘汰策略，筛选并删除符合规则的键，释放内存，保证新的写命令可正常执行。

Redis 提供 8 种淘汰策略，分为**不淘汰、仅针对设置过期时间的键、针对所有键**三大类，核心差异为淘汰规则与适用范围。

#### 一、淘汰策略全解析

|策略分类|策略名称|核心淘汰规则|适用场景|
|---|---|---|---|
|不淘汰策略|`noeviction`|默认策略，不淘汰任何键，内存达到上限时，所有写命令均返回错误，仅只读命令可正常执行|绝对不允许数据丢失的持久化场景，数据量固定无持续增长的场景；**生产环境纯缓存场景绝对禁止使用**，会导致写入直接不可用|
|仅针对设置了过期时间的键|`volatile-lru`|对设置了过期时间的键，采用 LRU 算法，优先淘汰最近最少使用的键；未设置过期时间的键永远不会被淘汰|实例同时存储永久持久化数据与临时缓存数据，需保证永久数据不被淘汰的场景|
|仅针对设置了过期时间的键|`volatile-lfu`|对设置了过期时间的键，采用 LFU 算法，优先淘汰访问频率最低的键；未设置过期时间的键永远不会被淘汰（Redis 4.0 + 新增）|需保留永久数据，同时需区分键的访问频率，避免偶尔访问的键淘汰高频热点键的场景|
|仅针对设置了过期时间的键|`volatile-ttl`|对设置了过期时间的键，优先淘汰剩余 TTL（生存时间）最短的键，即将过期的键优先淘汰|需要优先清理即将过期的键，保留长有效期键的场景|
|仅针对设置了过期时间的键|`volatile-random`|对设置了过期时间的键，随机淘汰|极少使用，仅适合所有键访问概率完全一致的场景|
|针对所有键|`allkeys-lru`|对所有键（无论是否设置过期时间），采用 LRU 算法，优先淘汰最近最少使用的键|绝大多数纯缓存场景，数据冷热差异明显，热点数据频繁访问，冷数据自动淘汰，**生产环境缓存场景首选**|
|针对所有键|`allkeys-lfu`|对所有键（无论是否设置过期时间），采用 LFU 算法，优先淘汰访问频率最低的键（Redis 4.0 + 新增）|访问频率差异大的高并发缓存场景，如电商大促热点商品，避免临时访问的键淘汰长期高频访问的热点数据，**高并发场景推荐**|
|针对所有键|`allkeys-random`|对所有键，随机淘汰|极少使用，仅适合所有键访问概率完全一致的场景|

#### 二、核心淘汰算法原理

##### 1. 近似 LRU 算法

LRU（Least Recently Used，最近最少使用）算法的核心是：优先淘汰最长时间未被访问的键。

- Redis 未采用传统精准 LRU（需维护双向链表，每次访问都调整链表，内存与 CPU 开销极大），而是实现了**近似 LRU 算法**：
    
    1. 每次淘汰时，根据`maxmemory-samples`配置（默认 5），随机抽取 N 个键；
    2. 从采样的键中，选出最近最少使用的键淘汰；
    3. 若内存仍未降到`maxmemory`以下，重复上述流程。
    
- 优化效果：`maxmemory-samples`设置为 10 时，淘汰效果已接近精准 LRU，同时内存与 CPU 开销仅为精准 LRU 的几十分之一；生产环境可根据需求调整采样数，平衡淘汰精度与 CPU 开销。

##### 2. LFU 算法

LFU（Least Frequently Used，最不经常使用）算法的核心是：优先淘汰访问频率最低的键，Redis 4.0 + 新增，解决 LRU 算法的缺陷：偶尔访问一次的键会被判定为热点，导致真正长期高频访问的键被淘汰。

- Redis LFU 实现原理：每个键的对象头中，用 24 位存储 LFU 信息：
    
    1. 高 16 位：`ldt`，记录上次计数器递减的时间（分钟级），用于定期衰减计数器，避免历史高频键永远不被淘汰；
    2. 低 8 位：`counter`，访问计数器，取值 0-255，并非简单递增，而是基于概率递增：访问频率越高，计数器递增的概率越低，避免计数器快速饱和；
    
- 核心优势：基于长期访问频率淘汰，而非单次访问时间，更贴合真实的热点数据场景，高并发下缓存命中率远高于 LRU。

#### 三、生产环境选择规范

1. 纯缓存场景，数据冷热差异明显：优先`allkeys-lru`；Redis 4.0 + 优先`allkeys-lfu`，缓存命中率更高；
2. 实例同时存储缓存数据与永久持久化数据：优先`volatile-lru`/`volatile-lfu`，保证永久数据不会被淘汰；
3. 绝对不允许数据丢失的持久化场景：使用`noeviction`，但必须做好内存监控，避免内存满导致写入不可用；
4. 禁止使用 random 系列策略，无明确淘汰规则，极易导致热点数据被淘汰，缓存命中率暴跌。

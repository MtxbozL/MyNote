
混合持久化是 Redis 4.0 + 新增的持久化方案，**结合了 RDB 的快速恢复能力与 AOF 的高数据安全性**，是当前生产环境官方推荐的首选持久化方案，通过`aof-use-rdb-preamble yes`配置开启。

#### 一、核心执行原理

混合持久化的核心逻辑在 AOF 重写阶段实现，将 RDB 全量快照与 AOF 增量日志合并到同一个 AOF 文件中：

1. 触发 AOF 重写时，子进程将 fork 瞬间的全量内存数据，以 RDB 二进制格式写入新 AOF 文件的开头；
2. 重写期间的增量写命令，以 AOF 协议格式追加到新 AOF 文件的 RDB 内容之后；
3. 主线程完成增量数据追加后，用新 AOF 文件原子性替换旧文件，重写完成。

最终生成的 AOF 文件结构为：**[RDB 全量快照] + [AOF 增量日志]**。

#### 二、重启恢复流程

Redis 重启加载持久化文件时，遵循以下规则：

1. 优先检测 AOF 是否开启，开启则加载 AOF 文件；
2. 读取 AOF 文件开头，检测是否为 RDB 格式，是则先加载 RDB 全量快照，再重放后续的增量 AOF 日志；
3. 若文件开头无 RDB 格式，则按纯 AOF 文件全量重放。

#### 三、核心优势与生产规范

1. **兼顾恢复速度与数据安全**：全量数据用 RDB 格式存储，恢复速度与纯 RDB 一致，远快于纯 AOF；增量数据用 AOF 格式存储，最多丢失 1 秒数据，安全性与纯 AOF 一致；
2. **文件体积更小**：全量数据采用 RDB 二进制压缩格式，比纯 AOF 文件体积降低 50% 以上，大幅减少磁盘占用；
3. **重写效率更高**：全量数据直接生成 RDB 快照，比生成大量 AOF 命令效率提升数倍，降低重写对系统的 IO 与 CPU 开销。

**生产规范**：Redis 4.0 + 版本必须开启混合持久化，同时配置低频 RDB 自动快照作为兜底灾备，兼顾性能、数据安全、故障恢复速度。

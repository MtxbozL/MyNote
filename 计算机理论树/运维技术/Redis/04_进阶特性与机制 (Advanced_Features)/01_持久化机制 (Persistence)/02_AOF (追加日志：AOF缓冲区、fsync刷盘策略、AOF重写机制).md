AOF（Append Only File）是**增量日志持久化**，核心是记录 Redis 的每一条写命令，以 Redis 协议格式追加到日志文件末尾，故障重启时通过重放 AOF 文件中的所有写命令，恢复全量数据。

#### 一、AOF 执行核心流程

AOF 持久化的完整执行分为 4 个阶段，全程保证写命令不丢失：

1. 命令写入：客户端执行写命令后，Redis 将命令以协议格式写入**AOF 缓冲区（aof_buf）**，而非直接写入磁盘，避免频繁磁盘 IO 拖慢性能；
2. 缓冲区刷盘：根据配置的 fsync 刷盘策略，将 AOF 缓冲区的数据刷入磁盘的 AOF 文件；
3. 文件重写：当 AOF 文件体积达到阈值时，触发 AOF 重写，压缩文件体积，去除冗余命令；
4. 重启恢复：Redis 重启时，检测到 AOF 文件开启，则优先加载 AOF 文件，重放所有写命令完成数据恢复。

#### 二、AOF 缓冲区

AOF 缓冲区是 Redis 内存中的一块固定缓冲区，用于临时存储待落盘的写命令，是平衡性能与数据安全的核心设计。

- 设计原因：磁盘 IO 的速度比内存低 3-4 个数量级，若每次写命令都直接触发磁盘 IO，会导致 Redis 性能暴跌，QPS 下降 90% 以上；
- 执行逻辑：写命令写入缓冲区由主线程完成，耗时可忽略；缓冲区到磁盘的刷盘操作由后台线程 / 操作系统控制，不阻塞主线程；
- 风险点：若缓冲区数据未刷盘时服务器宕机，对应写命令会永久丢失，丢失量由 fsync 刷盘策略决定。

#### 三、fsync 刷盘策略

fsync 是 Linux 系统调用，用于强制将内核缓冲区的数据刷入物理磁盘，是保证 AOF 数据不丢失的核心，Redis 提供 3 种可配置的刷盘策略，覆盖不同的性能与安全需求：

|策略|执行规则|数据安全性|性能表现|生产适用场景|
|---|---|---|---|---|
|`always`|每次写命令写入 AOF 缓冲区后，立即调用 fsync 强制刷盘，刷盘完成后主线程才返回响应|最高，宕机最多丢失 1 条写命令|极差，每次写命令都伴随磁盘 IO，单节点 QPS 通常不超过 1 万|金融、支付等强数据安全、零数据丢失要求的场景|
|`everysec`|默认策略，每秒由后台子线程调用 1 次 fsync，强制刷盘，主线程不阻塞|高，极端宕机场景最多丢失 1 秒内的写命令|优秀，性能接近无持久化状态，QPS 损失小于 5%|绝大多数生产环境的首选策略，平衡性能与数据安全|
|`no`|不主动调用 fsync，由操作系统控制刷盘时机，Linux 默认每 30 秒刷盘一次|极低，宕机丢失数据量完全不可控|最高，完全无刷盘开销|生产环境绝对禁止使用，仅可用于数据可完全丢失的临时缓存场景|

#### 四、AOF 重写机制

AOF 采用追加写模式，长期运行会导致文件体积无限膨胀，不仅占用大量磁盘空间，还会导致重启恢复时间呈指数级增长。AOF 重写的核心目标是压缩 AOF 文件体积，去除冗余命令。

1. **重写核心原理**
    
    重写不是读取旧 AOF 文件分析并精简，而是直接读取 Redis 当前内存中的全量数据，将其转化为对应的最小写命令集，写入新的 AOF 文件，完全替代旧的膨胀文件。冗余命令去除规则：
    
    - 对同一 Key 的多次写操作，仅保留最后一次生效的命令（如多次 SET 同一 Key，仅保留最后一次 SET）；
    - 对同一 Key 的多次增减操作，转化为最终值的 SET 命令（如 100 次 INCR，转化为 SET key 最终值）；
    - 批量操作合并，将多条单命令合并为批量命令（如多条 SADD 合并为一条批量 SADD）。
    
2. **BGREWRITEAOF 后台重写执行流程**
    
    生产环境 AOF 重写的唯一触发方式，基于 fork 子进程实现，全程几乎不阻塞主线程：
    
    1. 触发重写：手动执行`BGREWRITEAOF`命令，或达到自动触发阈值（`auto-aof-rewrite-percentage`与`auto-aof-rewrite-min-size`配置）；
    2. 主线程检查是否存在正在运行的 RDB/AOF 重写子进程，存在则等待完成后再执行；
    3. 主线程 fork 创建子进程，fork 瞬间短暂阻塞，完成后主线程恢复响应；
    4. 双缓冲区机制：重写期间，主线程的写命令除了写入原 AOF 缓冲区（保证原 AOF 正常落盘），同时写入**AOF 重写缓冲区**，避免新 AOF 丢失重写期间的增量数据；
    5. 子进程基于 fork 瞬间的内存快照，生成新的 AOF 文件写入磁盘；
    6. 子进程完成后向主线程发送信号，主线程将 AOF 重写缓冲区的增量数据追加到新 AOF 文件末尾；
    7. 主线程用新 AOF 文件原子性替换旧文件，重写流程完成。
    

#### 五、AOF 持久化优缺点

| 维度  | 核心特性                                                                                                                                                                                                 |
| --- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 优点  | 1. 数据安全性高，默认`everysec`策略最多丢失 1 秒数据，远优于 RDB；<br><br>2. 追加写模式，无磁盘寻址开销，写入性能稳定，无文件损坏风险；<br><br>3. 日志文件为明文 Redis 协议格式，可手动编辑、修复，适合误操作后的数据恢复；<br><br>4. 重写机制自动压缩文件体积，同时保证数据完整性。                             |
| 缺点  | 1. 相同数据集下，AOF 文件体积远大于 RDB，占用更多磁盘空间；<br><br>2. 数据恢复速度慢，需要重放所有写命令，TB 级数据恢复时间可达小时级；<br><br>3. 高并发写场景下，fsync 刷盘会带来一定的性能开销，`everysec`策略虽影响极小，但仍略低于无持久化状态；<br><br>4. AOF 重写期间会产生双倍的磁盘 IO 压力，同时双缓冲区会占用额外内存。 |

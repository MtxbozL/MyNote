Redis 作为内存数据库，数据全量存储于内存中，进程退出或服务器宕机将导致数据完全丢失。持久化机制的核心目标是将内存中的数据异步落盘到磁盘，实现数据持久化存储与故障后恢复，Redis 提供 RDB 快照、AOF 追加日志、混合持久化三种方案。

### 01_RDB (快照: save&bgsave 原理、COW 写时复制技术、优缺点)

RDB（Redis Database）是**全量快照持久化**，将某一时刻 Redis 内存中的全量数据集生成二进制压缩文件（默认`dump.rdb`）落盘，故障重启时可直接加载 RDB 文件恢复全量数据。

#### 一、save 与 bgsave 执行原理

RDB 快照生成提供两种触发命令，核心差异为是否阻塞主线程：

1. **SAVE 同步快照**
    
    - 执行流程：客户端执行 SAVE 后，Redis 主线程直接执行 RDB 文件生成逻辑，遍历全量内存数据并写入磁盘。
    - 核心特性：全程阻塞主线程，执行期间所有客户端命令均会排队等待，快照完成后主线程才恢复响应。
    - 生产规范：**生产环境绝对禁止使用**，仅可用于停机维护、数据冷备份等无业务请求的场景。
    
2. **BGSAVE 后台异步快照**
    
    生产环境 RDB 快照的唯一合规触发方式，基于 fork 子进程实现异步快照，全程几乎不阻塞主线程，执行流程如下：
    
    1. 客户端执行 BGSAVE，主线程先检查是否存在正在运行的 RDB/AOF 重写子进程，存在则直接拒绝执行，避免并发 IO 冲突；
    2. 主线程调用`fork()`系统调用创建子进程，fork 瞬间主线程会短暂阻塞（内存越大，阻塞时间越长），fork 完成后主线程立即恢复响应客户端请求；
    3. 子进程共享主线程 fork 瞬间的内存数据，独立遍历内存生成 RDB 临时文件，写入完成后原子性替换旧的 RDB 文件；
    4. 子进程完成快照后向主线程发送信号，主线程更新 RDB 元数据（如最后一次快照时间、文件大小等），快照流程结束。
    

#### 二、COW 写时复制技术

写时复制（Copy-On-Write, COW）是 BGSAVE 实现的核心底层机制，由操作系统提供，完美平衡了快照数据一致性与性能开销。

- 核心原理：fork 子进程时，操作系统不会复制全量父进程内存，而是让父子进程共享同一块物理内存页，仅将内存页的权限设为只读；当父进程（主线程）对某一内存页执行写操作时，操作系统触发缺页异常，复制该内存页生成独立副本，父子进程各自持有独立的内存页，互不影响。
- 核心价值：
    
    1. 保证快照一致性：子进程看到的内存数据永远是 fork 瞬间的快照，不受父进程后续写操作的影响；
    2. 极致降低内存开销：避免全量内存复制，fork 瞬间内存占用几乎无增长，仅在写操作发生时才复制对应内存页；
    3. 最小化主线程阻塞：fork 完成后主线程可正常处理写请求，不会被快照逻辑阻塞。
    
- 生产风险点：高并发写场景下，fork 后大量内存页被修改，会触发频繁的内存页复制，导致内存占用飙升（极端场景可达原内存的 2 倍），可能触发操作系统 OOM Killer 终止 Redis 进程，生产环境必须预留至少 50% 的空闲内存，同时避免在业务高峰触发 BGSAVE。

#### 三、RDB 持久化优缺点

表格

|维度|核心特性|
|---|---|
|优点|1. 单二进制压缩文件，体积小，非常适合全量数据备份、灾备迁移、跨机房数据同步；<br><br>2. 数据恢复速度极快，TB 级数据恢复速度远快于 AOF，适合大规模数据的故障快速恢复；<br><br>3. 对主线程性能影响极小，仅 fork 瞬间短暂阻塞，子进程 IO 操作不影响主线程正常服务；<br><br>4. 对磁盘 IO 压力小，适合高并发、低延迟敏感的业务场景。|
|缺点|1. 数据安全性低：快照为间隔性触发，宕机时会丢失最后一次快照之后的所有数据，无法做到秒级数据不丢失；<br><br>2. fork 开销不可控：实例内存占用越大，fork 瞬间主线程阻塞时间越长，大内存实例（10GB+）阻塞时间可达百毫秒级；<br><br>3. 不适合实时持久化：频繁触发 BGSAVE 会导致磁盘 IO 与 CPU 开销飙升，同时加剧内存页复制的风险；<br><br>4. 版本兼容性问题：不同 Redis 版本的 RDB 格式存在差异，低版本 Redis 无法加载高版本生成的 RDB 文件。|

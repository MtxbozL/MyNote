
### 02_局限性 (不保证强一致性、无回滚机制)

Redis 事务与关系型数据库的 ACID 事务存在本质差异，存在诸多核心局限性，无法满足强事务场景的需求。

#### 一、无回滚机制

关系型数据库事务执行出错时，会回滚所有已执行的操作，保证原子性；而 Redis 事务**完全不支持回滚**，分两种场景：

1. **入队时错误**：命令语法错误、参数数量错误、命令名不存在等，Redis 会在入队时直接报错，执行 DISCARD 放弃事务，队列中的所有命令都不会执行；
2. **执行时错误**：命令语法正确，但执行时出现运行时错误（如对 String 类型的 Key 执行 HGET 命令），Redis 会跳过该错误命令，继续执行事务内的后续命令，不会回滚已经执行的命令，导致事务内部分成功、部分失败，无法保证 “全有或全无” 的原子性。

Redis 官方不支持回滚的核心理由：

- 命令执行错误均为编程错误，仅会出现在开发环境，不会出现在生产环境，无需回滚机制；
- 回滚机制会大幅增加 Redis 的复杂度，与 Redis 高性能、简洁的设计理念相悖。

#### 二、不保证强一致性，无隔离级别

1. **不支持 ACID 中的一致性与隔离性**：关系型数据库提供 4 种隔离级别，解决脏读、不可重复读、幻读问题；Redis 事务无隔离级别的概念，WATCH 仅能保证事务提交时 Key 未被修改，无法避免事务执行期间其他客户端的修改，无法解决脏读问题。
2. **不保证持久性**：事务执行完成后，数据仅存在于内存中，若未开启持久化，宕机后数据全部丢失；即使开启持久化，RDB/AOF 也无法保证事务提交后立即刷盘，宕机仍会丢失数据，无法保证 ACID 中的持久性。

#### 三、其他核心局限性

1. 事务内无法嵌套 MULTI，不支持子事务；
2. 事务内的命令无法使用前一个命令的返回结果，无法在事务内做逻辑判断，所有命令仅能入队排队，EXEC 时才批量执行，复杂逻辑需配合 WATCH 或 Lua 脚本实现；
3. Redis Cluster 集群模式下，事务内的所有 Key 必须位于同一个哈希槽，否则无法执行跨槽事务，使用限制极大。

#### 生产规范

Redis 事务仅适合简单的无逻辑判断的批量命令执行场景；复杂的原子性操作、并发控制场景，**优先使用 Lua 脚本实现**，Lua 脚本可保证原子性，同时支持条件判断、循环等复杂逻辑，功能与可靠性远强于 Redis 事务。

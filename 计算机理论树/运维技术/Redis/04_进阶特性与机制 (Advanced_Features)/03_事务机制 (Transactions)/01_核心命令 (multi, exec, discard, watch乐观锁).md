Redis 事务是将多个命令打包，按入队顺序串行执行的机制，核心特性是**执行期间不会被其他客户端的命令打断**，保证事务内命令的连续执行，是 Redis 原生提供的批量原子操作方案。

---

### 01_核心命令 (multi, exec, discard, watch 乐观锁)

#### 一、事务核心执行命令

1. **MULTI 开启事务**
    
    - 语法：`MULTI`
    - 核心功能：开启事务，客户端进入事务状态，后续发送的所有命令不会立即执行，而是进入事务队列排队，返回`QUEUED`。
    - 时间复杂度：O (1)
    
2. **EXEC 提交事务**
    
    - 语法：`EXEC`
    - 核心功能：提交事务，按入队顺序串行执行事务队列中的所有命令，执行期间不会被其他客户端命令打断，执行完成后返回所有命令的执行结果数组，事务结束。
    - 时间复杂度：O (N)，N 为事务内的命令数量
    
3. **DISCARD 放弃事务**
    
    - 语法：`DISCARD`
    - 核心功能：清空事务队列，退出事务状态，客户端回到正常模式，队列中的所有命令都不会执行。
    - 时间复杂度：O (N)，N 为队列中的命令数量
    

#### 二、WATCH 乐观锁机制

WATCH 是 Redis 事务的并发控制机制，用于解决并发场景下的竞态问题，实现乐观锁。

- 语法：`WATCH key [key ...]`
- 核心功能：监控一个或多个 Key，在 EXEC 执行前，若被监控的 Key 被其他客户端修改过，则整个事务会被放弃，EXEC 返回`nil`，保证事务的一致性。
- 配套命令：`UNWATCH`，取消所有 Key 的监控，时间复杂度 O (1)，事务执行完成或 DISCARD 后会自动取消监控，无需手动调用。
- 典型使用场景（原子性转账，避免超卖）：

    ```bash
    WATCH account:A account:B
    MULTI
    DECRBY account:A 100
    INCRBY account:B 100
    EXEC
    ```
    
    若 WATCH 之后、EXEC 之前，`account:A`或`account:B`被其他客户端修改，事务会直接放弃执行，避免脏写。

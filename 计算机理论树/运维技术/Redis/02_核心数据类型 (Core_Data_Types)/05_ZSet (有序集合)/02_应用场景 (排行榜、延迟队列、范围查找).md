#### 一、业务排行榜系统

- **核心实现**：ZSet 的核心场景，覆盖商品销量榜、用户积分榜、文章热度榜、热搜榜等场景。以榜单唯一标识为 Key，业务对象 ID 为元素，排序指标（销量、积分、热度）为分数；数据更新通过 ZINCRBY/ZADD 原子性执行，Top N 榜单通过 ZREVRANGE 获取，用户排名通过 ZREVRANK 查询，分页榜单通过 ZRANGE/LIMIT 实现，固定长度榜单通过 ZREMRANGEBYRANK 定期裁剪。
- **核心命令**：`ZADD`、`ZINCRBY`、`ZREVRANGE`、`ZREVRANK`、`ZRANGE`
- **生产规范**：
    
    1. 高并发热点榜单需通过本地缓存 + Redis 多级缓存架构，降低 Redis 查询压力；
    2. 榜单分页单次查询不超过 100 条，禁止全量查询大集合；
    3. 多维度榜单通过 ZINTERSTORE/ZUNIONSTORE 实现聚合排名。
    

#### 二、延迟队列 / 定时任务

- **核心实现**：以延迟任务类型为 Key，任务 ID 为元素，任务执行时间的 Unix 时间戳为分数；生产者通过 ZADD 添加延迟任务，消费者定时通过 ZRANGEBYSCORE 查询分数小于当前时间戳的任务（已到执行时间），获取后通过 ZREM 原子性删除任务并执行，避免重复消费。
- **核心命令**：`ZADD`、`ZRANGEBYSCORE`、`ZREM`
- **生产规范**：
    
    1. 必须通过 Lua 脚本保证任务查询与删除的原子性，避免分布式场景下重复消费；
    2. 高并发延迟队列需分片存储，避免单 Key 热点问题；
    3. 任务执行失败需重试机制，重新写入 ZSet 设置新的执行时间。
    

#### 三、滑动窗口限流

- **核心实现**：以限流维度（用户 ID / 接口 IP）为 Key，请求唯一 ID 为元素，请求时间戳为分数；每次请求到来时，先通过 ZREMRANGEBYSCORE 删除窗口外的历史请求，再通过 ZADD 添加当前请求，最后通过 ZCARD 统计窗口内的请求总数，超过阈值则触发限流。
- **核心命令**：`ZADD`、`ZREMRANGEBYSCORE`、`ZCARD`
- **生产规范**：必须通过 Lua 脚本保证整个限流逻辑的原子性，避免并发场景下的限流穿透；限流 Key 需设置过期时间，避免内存无限增长。

#### 四、带权重的范围查找与筛选

- **核心实现**：如附近的人按距离排序、商品按价格 / 评分筛选排序、用户按活跃度范围查询等场景，将筛选指标（距离、价格、活跃度）作为分数，业务对象 ID 为元素，通过 ZRANGEBYSCORE 实现指定范围的筛选，同时按分数排序，无需业务层二次排序。
- **核心命令**：`ZRANGEBYSCORE`、`ZREVRANGEBYSCORE`、`ZCOUNT`

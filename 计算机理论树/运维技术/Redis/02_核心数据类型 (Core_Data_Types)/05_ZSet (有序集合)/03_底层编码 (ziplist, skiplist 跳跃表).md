ZSet 的底层编码分为**ziplist（压缩列表）**与**skiplist（跳跃表）+ dict（哈希表）双结构**两种，Redis 根据元素数量与元素长度自动切换编码，转换后不可逆。

#### 一、ziplist 压缩列表编码

- **适用场景**：当 ZSet 同时满足以下两个条件时，使用 ziplist 编码：
    
    1. 有序集合的元素总数量不超过`zset-max-ziplist-entries`（redis.conf 配置，默认 128）；
    2. 集合中每个元素的字节长度不超过`zset-max-ziplist-value`（默认 64 字节）。
    
- **底层存储规则**：ziplist 中，每两个连续的 entry 成对存储元素与分数，即`member-entry`紧跟`score-entry`，所有元素按分数从小到大升序紧凑存储在同一段连续内存中。
- **核心优势**：
    
    1. 内存占用极低，无指针冗余，内存利用率远高于跳跃表结构；
    2. 连续内存空间 CPU 缓存命中率高，小元素、短集合场景下性能优异。
    
- **核心缺陷**：
    
    1. 元素查找需遍历 ziplist，时间复杂度 O (N)，元素数量增多时性能急剧下降；
    2. 插入 / 删除操作会触发内存重分配，存在连锁更新问题，大元素场景性能极差；
    3. 不支持高效的范围查询，仅适合小集合场景。
    

#### 二、skiplist 跳跃表 + dict 哈希表双结构

- **适用场景**：当 ZSet 不满足 ziplist 的任一条件时，自动从 ziplist 转换为**skiplist+dict 双结构**编码，转换后不可逆。
- **双结构设计核心原理**：
    
    Redis 为 ZSet 同时维护跳跃表与哈希表两个结构，二者共享元素与分数的内存，无数据冗余：
    
    1. **dict 哈希表**：维护元素到分数的映射，实现 O (1) 时间复杂度的元素分数查询（ZSCORE）、元素存在性检查；
    2. **skiplist 跳跃表**：维护元素按分数的有序排序，实现 O (logN) 时间复杂度的排名查询、范围查询、元素增删。
    

##### 1. 跳跃表（skiplist）核心结构与原理

跳跃表是一种有序的链式数据结构，通过在原始有序链表上增加多层索引，实现平均 O (logN)、最坏 O (N) 的时间复杂度，性能可比拟平衡树，且实现更简单、内存开销更灵活。

- **底层结构**：
    
    1. 跳跃表管理结构`zskiplist`：包含头节点、尾节点、长度（元素总数）、level（当前最大层数）；
    2. 跳跃表节点`zskiplistNode`：
        
        - 元素`member`与分数`score`；
        - 后退指针`backward`，指向当前节点的前一个节点，实现倒序遍历；
        - 层数组`level[]`，每个层包含前进指针`forward`与跨度`span`，前进指针指向后续节点，跨度记录指针跨越的元素数量，用于快速计算排名。
        
    
- **核心特性**：
    
    1. 随机层数生成：每个新节点的层数通过幂次定律随机生成，最大层数 32，层数越高，出现概率越低；
    2. 多层索引：高层索引跨越多个节点，查找时从最高层开始，逐层向下，快速跳过大量不符合条件的节点，大幅降低查找次数；
    3. 无旋转操作：相比红黑树等平衡树，跳跃表的增删操作无需复杂的旋转平衡，实现更简单，并发场景下锁粒度更小。
    
- **Redis 选择跳跃表而非平衡树的核心原因**：
    
    1. 范围查询性能更优：平衡树的范围查询需通过中序遍历实现，而跳跃表仅需找到范围起点后，沿底层链表遍历即可，实现更简单、效率更高；
    2. 实现与维护成本极低：跳跃表代码实现远简单于红黑树，无复杂的平衡逻辑，bug 率更低，可维护性更强；
    3. 内存开销更灵活：跳跃表的层数可随机调整，平均每个节点的指针开销为 1.33 个，低于红黑树的 2 个指针开销；
    4. 并发操作更友好：跳跃表的增删操作仅影响相邻节点，锁粒度远小于平衡树，高并发场景下性能更优。
    

##### 2. dict 哈希表的核心作用

- 实现 O (1) 时间复杂度的元素分数查询（ZSCORE 命令），无需遍历跳跃表；
- 实现 O (1) 时间复杂度的元素存在性检查，为 ZADD、ZREM 等命令提供快速校验；
- 与跳跃表共享元素与分数的内存，无数据冗余，仅存储指针映射，内存开销极低。

#### 生产环境编码优化规范

1. 合理设置`zset-max-ziplist-entries`与`zset-max-ziplist-value`，默认 128 与 64 字节为性能与内存的最优平衡，小元素场景可适当调高，避免频繁编码转换；
2. 单个 ZSet 的元素数量建议控制在 100 万以内，超大集合需拆分存储，避免跳跃表层数过多、内存占用过大；
3. 大集合必须使用 ZSCAN 增量遍历，绝对禁止使用全量 ZRANGE 查询，避免阻塞主线程；
4. 排行榜场景优先使用 ZREVRANGE/LIMIT 分页查询，禁止全量拉取后业务层分页，减少网络开销与阻塞风险。

Redis List 的底层编码经历了三代演进：Redis 3.2 之前使用**ziplist（压缩列表）**和**linkedlist（双向链表）**双编码自动切换；Redis 3.2 + 引入**quicklist（快速列表）**，作为默认编码完全替代前两者，平衡了内存占用与操作性能。

#### 一、ziplist 压缩列表

- **适用场景**：Redis 3.2 之前，当列表同时满足以下两个条件时，使用 ziplist 编码：
    
    1. 列表元素总数量不超过`list-max-ziplist-entries`（redis.conf 配置，默认 512）；
    2. 列表中单个元素的字节长度不超过`list-max-ziplist-value`（默认 64 字节）。
    
- **底层结构**：ziplist 是一段**连续紧凑的内存空间**，无冗余指针，整体结构如下：

| 字段      | 长度   | 核心作用                                 |
| ------- | ---- | ------------------------------------ |
| zlbytes | 4 字节 | 整个压缩列表的总字节长度                         |
| zltail  | 4 字节 | 表尾元素的偏移量，实现 O (1) 时间复杂度的表尾操作         |
| zllen   | 2 字节 | 列表的元素总数量                             |
| entry   | 不定长  | 每个元素的存储单元，包含前一个元素的长度、当前元素的编码与长度、实际数据 |
| zlend   | 1 字节 | 压缩列表结束标记，固定值 0xFF                    |

- **核心优势**：内存占用极低，无指针冗余，连续内存空间 CPU 缓存命中率高，小元素、短列表场景下性能优异。
- **核心缺陷**：
    
    1. 插入 / 删除操作需重新分配内存，存在**连锁更新问题**：前一个元素的长度变化会导致后续所有 entry 的前导长度字段重新计算，极端场景下性能急剧下降；
    2. 元素查找需遍历整个列表，时间复杂度 O (N)，长列表场景性能极差；
    3. 大元素会导致内存频繁重分配，阻塞主线程。
    

#### 二、linkedlist 双向链表

- **适用场景**：Redis 3.2 之前，当列表不满足 ziplist 的任一条件时，自动从 ziplist 转换为 linkedlist 编码，转换后不可逆。
- **底层结构**：标准的**双向无环链表**，核心组成如下：
    
    1. 链表管理结构`list`：包含头节点指针`head`、尾节点指针`tail`、元素数量`len`、元素复制 / 释放 / 匹配函数指针；
    2. 链表节点`listNode`：包含前驱指针`prev`、后继指针`next`、元素值指针`value`（指向 SDS 结构）。
    
- **核心优势**：
    
    1. 两端增删操作时间复杂度 O (1)，无内存重分配开销，性能稳定；
    2. 无连锁更新问题，支持大元素、长列表场景；
    3. 支持高效的节点遍历，无内存重分配风险。
    
- **核心缺陷**：
    
    1. 每个节点存在两个指针冗余，内存开销极大，小元素场景下内存利用率远低于 ziplist；
    2. 节点内存不连续，CPU 缓存行命中率低，遍历性能差；
    3. 大量节点会产生严重的内存碎片。
    

#### 三、quicklist 快速列表（Redis 3.2+ 默认编码）

- **核心定位**：quicklist 是 ziplist 与 linkedlist 的结合体，将多个 ziplist 作为节点，通过双向链表串联起来，完美平衡了内存占用与操作性能，解决了 ziplist 的连锁更新问题与 linkedlist 的内存冗余问题。
- **底层结构**：
    
    1. 整体为双向链表，每个节点为`quicklistNode`结构；
    2. 每个`quicklistNode`指向一个独立的 ziplist（称为节点压缩列表），存储一部分连续的列表元素；
    3. 整个列表的元素分散存储在多个小 ziplist 中，单个 ziplist 的大小由配置控制，避免了大 ziplist 的连锁更新问题。
    
- **核心配置（redis.conf）**：
    
    表格
    
    |配置项|默认值|核心作用|
    |---|---|---|
    |`quicklist-fill-factor`|-2|每个 ziplist 的最大大小，负数表示字节上限：-1=4KB、-2=8KB、-3=16KB，以此类推；正数表示每个 ziplist 最多存储的元素数量|
    |`quicklist-compress-depth`|0|列表节点压缩深度，0 表示不压缩；设置为 N 表示列表两端各 N 个节点不压缩，中间节点使用 LZF 算法压缩，适合长列表冷数据内存优化|
    
- **核心优势**：
    
    1. 两端增删操作 O (1)，与 linkedlist 一致，仅需操作头尾节点的 ziplist；
    2. 内存占用远低于 linkedlist，ziplist 紧凑存储减少指针冗余，内存利用率提升 50% 以上；
    3. 彻底解决 ziplist 的连锁更新问题，元素分散在多个小 ziplist 中，单个 ziplist 的修改不会影响其他节点；
    4. 连续 ziplist 提升 CPU 缓存命中率，遍历性能远高于 linkedlist；
    5. 支持冷数据节点压缩，长列表场景内存占用进一步降低。
    

#### 生产环境编码优化规范

1. 生产环境必须使用 Redis 3.2 + 版本，优先使用 quicklist 编码，避免 ziplist 的连锁更新与 linkedlist 的内存冗余；
2. `quicklist-fill-factor`默认值 - 2（8KB）为性能与内存的最优平衡，小元素场景可调整为 - 1（4KB），大元素场景可调整为 - 3（16KB），避免单个 ziplist 过大；
3. 长列表冷数据场景开启`quicklist-compress-depth`，建议设置为 1 或 2，仅保留两端高频操作节点不压缩，中间冷数据压缩存储，降低内存占用；
4. 禁止在列表中存储超过 1KB 的超大元素，会导致 ziplist 性能急剧下降，建议拆分存储；
5. 禁止对大列表执行全量 LRANGE、LINDEX 中间索引、全量 LREM 等 O (N) 操作，避免阻塞主线程。

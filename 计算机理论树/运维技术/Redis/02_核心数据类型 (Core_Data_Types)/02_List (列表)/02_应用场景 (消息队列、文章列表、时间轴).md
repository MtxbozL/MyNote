#### 一、轻量级消息队列

- **核心实现**：生产者通过`LPUSH`向列表写入消息，消费者通过`BRPOP`阻塞式获取消息执行，避免轮询开销；可靠队列场景使用`BRPOPLPUSH`实现消息备份，消费成功后删除备份消息，避免丢失。
- **核心命令**：`LPUSH`、`BRPOP`、`RPOPLPUSH`、`BRPOPLPUSH`
- **生产规范**：仅适用于简单异步任务、低并发消息场景；高可靠、高吞吐、多消费者场景优先使用 Redis Stream 类型，List 无法实现消息 ACK 确认、消费者组、消息持久化回溯等企业级能力。

#### 二、时间轴 / 动态列表

- **核心实现**：用户发布动态时，通过`LPUSH`将动态 ID 写入对应用户的时间轴列表（如`timeline:user:1001`）；前端通过`LRANGE`分页读取最新动态，配合`LTRIM`固定保留最新 N 条动态，避免列表无限增长。
- **核心命令**：`LPUSH`、`LRANGE`、`LTRIM`
- **生产规范**：必须配合`LTRIM`限制列表最大长度，分页单次读取不超过 100 条；大 V 用户的海量时间轴需拆分存储，避免单列表过大。

#### 三、栈 / 队列 / 双端队列数据结构模拟

- **栈（先进后出 FILO）**：`LPUSH` + `LPOP`组合，同一端插入弹出；
- **队列（先进先出 FIFO）**：`LPUSH` + `RPOP`组合，两端插入弹出；
- **双端队列**：`LPUSH/RPUSH` + `LPOP/RPOP`组合，两端均支持高效增删，完全覆盖双端队列的所有操作。

#### 四、异步任务队列

- **核心实现**：业务系统通过`LPUSH`提交异步任务（如邮件发送、数据导出），消费者服务通过`BRPOP`持续获取任务执行；通过`RPOPLPUSH`实现任务备份，消费者宕机后可从备份列表恢复未完成的任务，避免任务丢失。
- **核心命令**：`LPUSH`、`BRPOP`、`RPOPLPUSH`

#### 五、固定长度排行榜

- **核心实现**：按插入时间排序的榜单（如最新上线商品、最新评论），通过`LPUSH`新增榜单元素，`LRANGE`读取榜单内容，`LTRIM`固定保留 Top N 条数据，实现定长榜单。
- **核心命令**：`LPUSH`、`LRANGE`、`LTRIM`
- **注意**：按分数 / 权重排序的榜单，优先使用 ZSet 有序集合实现，List 仅适合按插入时间排序的场景。

String 类型覆盖 Redis 绝大多数基础业务场景，核心场景的实现原理、核心命令与生产规范如下：

#### 一、分布式缓存

- **核心实现**：以业务唯一标识为 Key（如`user:info:1001`、`product:detail:2001`），将数据库查询结果、序列化后的对象 / JSON 字符串作为 Value 存储，设置合理的过期时间；业务请求优先查询 Redis，命中则直接返回，未命中则查询数据库并回写 Redis。
- **核心命令**：`SET/GET`、`MSET/MGET`、`SETEX`
- **生产规范**：
    
    1. 必须设置过期时间，避免内存无限增长；热点数据可通过定时任务续期，避免缓存击穿；
    2. Key 命名必须遵循`业务域:对象:唯一标识`的规范，便于管理与排查；
    3. 大对象需拆分存储，禁止单 Value 存储超过 10KB 的序列化对象，避免网络阻塞。
    

#### 二、原子计数器

- **核心实现**：利用 INCR/DECR 系列命令的原子性，实现高并发场景下的计数功能，无需业务层加锁，避免竞态问题。典型场景：文章阅读量、视频播放量、商品库存计数、接口限流计数。
- **核心命令**：`INCR/DECR`、`INCRBY/DECRBY`
- **生产规范**：
    
    1. 库存等有下限的场景，必须通过 Lua 脚本控制数值下限，避免超卖（如库存不能小于 0）；
    2. 高并发计数场景，可通过 Key 分片（如`like:article:1001:0`~`like:article:1001:9`）分散热点 Key 压力，避免单 Key 成为性能瓶颈；
    3. 计数数据需定期持久化到数据库，避免 Redis 重启导致数据丢失。
    

#### 三、分布式锁

- **基础实现**：利用`SET key lock_value NX PX timeout`的原子性，实现分布式互斥锁：Key 为锁的唯一标识（如`lock:order:1001`），Value 为唯一随机值（如 UUID + 线程 ID），PX 设置锁超时时间，避免死锁。仅当 SET 返回 OK 时，代表获取锁成功；执行完业务逻辑后，通过 Lua 脚本校验 Value 匹配后删除锁，避免误删其他线程的锁。
- **核心命令**：`SET NX PX`、`GET`、Lua 脚本 + DEL
- **生产规范**：
    
    1. 必须单命令原子性设置锁 + 过期时间，禁止拆分 SETNX+EXPIRE；
    2. 释放锁必须校验 Value 匹配，避免锁超时过期后，误删其他线程持有的锁；
    3. 生产环境优先使用 Redisson 等成熟框架，内置看门狗机制、可重入锁、锁续期能力，避免自研锁的边界问题。
    

#### 四、Session 共享

- **核心实现**：分布式系统中，将用户登录后的 Session 信息（如用户 ID、权限、登录态）序列化后存储到 Redis，以 SessionID 为 Key，设置合理的过期时间；所有服务节点均从 Redis 中读取 / 更新 Session 信息，实现跨服务、跨实例的登录态共享。
- **核心命令**：`SETEX`、`GET`、`DEL`
- **生产规范**：
    
    1. 必须设置过期时间，与用户登录态有效期保持一致；
    2. 敏感信息需加密后存储，禁止明文存储用户密码、Token 等敏感数据；
    3. 高并发场景可开启 Redis 读写分离，读请求从从节点读取，降低主节点压力。
    

#### 五、全局唯一 ID 生成器

- **核心实现**：利用 INCR 命令的原子递增特性，实现分布式全局唯一 ID 生成；以业务域为 Key（如`id:generator:order`），每次调用 INCR 获取唯一递增 ID，可结合时间戳、机器号拼接，生成符合雪花算法规范的分布式 ID。
- **核心命令**：`INCRBY`
- **生产规范**：
    
    1. 预分配号段，批量获取 ID（如一次 INCRBY 1000），减少 Redis 访问次数，提升性能；
    2. 必须设置持久化，避免 Redis 重启导致 ID 重复；
    3. 高并发场景可按业务分 Key，避免单 Key 热点问题。
    

#### 六、限流控制

- **核心实现**：基于 INCR 命令实现固定窗口限流，以时间窗口为 Key（如`limit:api:user:1001:202604051430`），用户每次请求接口执行 INCR，若计数值超过限流阈值，则拒绝请求；Key 设置过期时间为窗口时长，自动切换新窗口。进阶可通过 Lua 脚本实现滑动窗口、令牌桶算法，提升限流精度。
- **核心命令**：`INCR`、`SETEX`
- **生产规范**：
    
    1. 限流逻辑必须通过 Lua 脚本原子性执行，避免并发场景下的限流穿透；
    2. 必须设置兜底降级策略，避免 Redis 故障导致限流失效。

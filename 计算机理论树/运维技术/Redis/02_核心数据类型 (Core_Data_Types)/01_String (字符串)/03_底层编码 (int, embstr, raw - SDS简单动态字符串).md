String 类型的底层实现基于**SDS（Simple Dynamic String，简单动态字符串）** 结构，Redis 根据 Value 的类型与长度，自动选择三种物理编码格式，实现内存占用与访问性能的最优平衡，编码转换为 Redis 自动完成，用户无感知。

#### 一、SDS 简单动态字符串核心原理

SDS 是 Redis 自定义的字符串结构，完全替代 C 语言原生字符串，是 Redis 所有字符串操作的底层基础，解决了 C 语言字符串的二进制不安全、缓冲区溢出、长度计算 O (N) 等核心问题。

##### 1. SDS 数据结构定义

```c
struct sdshdr {
    // 已使用的字节长度
    int len;
    // 未使用的预留字节长度
    int free;
    // 字符数组，实际存储数据，以'\0'结尾
    char buf[];
};
```

Redis 针对不同长度的字符串，优化了 SDS 的结构头，分为 sdshdr5、sdshdr8、sdshdr16、sdshdr32、sdshdr64，分别对应不同长度的字符串，进一步降低内存占用。

##### 2. SDS 核心特性与优势

1. **O (1) 时间复杂度获取字符串长度**
    
    SDS 结构原生存储`len`属性，获取字符串长度仅需读取属性值，时间复杂度 O (1)；而 C 语言字符串需遍历整个字符数组计算长度，时间复杂度 O (N)，这是 Redis 字符串操作高性能的核心基础。
2. **二进制安全**
    
    SDS 以`len`属性判断字符串结束，而非 C 语言的 '\0' 结束符，可存储包含 '\0' 在内的任意二进制数据（图片、序列化对象、加密数据等），无特殊字符截断问题，实现了二进制安全。
3. **杜绝缓冲区溢出**
    
    C 语言字符串拼接时，若未提前分配足够内存，会导致缓冲区溢出；SDS 执行拼接操作前，会自动检查`free`预留空间，若空间不足则自动扩容，无需用户手动处理，从底层杜绝了缓冲区溢出问题。
4. **内存预分配与惰性释放，减少内存重分配次数**
    
    - 空间预分配：SDS 扩容时，除了分配所需的内存，还会额外分配预留空间（`free`）：字符串长度小于 1MB 时，预留与`len`相等的空间；长度大于 1MB 时，固定预留 1MB 空间，避免频繁的内存重分配操作。
    - 惰性空间释放：SDS 字符串缩短时，不会立即释放多余的内存，而是将其计入`free`属性，预留用于后续的字符串追加操作，避免频繁的内存释放与重分配；可通过 API 手动释放多余内存。
    
5. **兼容 C 语言原生字符串函数**
    
    SDS 的`buf`数组始终以 '\0' 结尾，可直接调用 C 语言 <string.h> 库的原生函数，无需额外适配，降低了开发成本。

#### 二、String 类型的三种底层编码

Redis 会根据 Value 的内容与长度，自动选择最优的编码格式，三种编码的适用场景、结构与转换规则如下：

##### 1. int 编码

- **适用场景**：Value 为可表示为 64 位有符号整数的字符串（范围：-2^63 ~ 2^63-1），如`SET count 100`。
- **底层结构**：直接将整数值存储在 RedisObject 的 ptr 指针字段中，无需额外的 SDS 结构，完全省去了 SDS 结构头的内存开销，内存占用最小，访问性能最高。
- **核心限制**：仅支持整数类型，若对 int 编码的 Key 执行 APPEND、INCRBYFLOAT 等操作，会自动转换为 embstr/raw 编码。

##### 2. embstr 编码

- **适用场景**：Value 为字符串，且字节长度**小于等于 44 字节**（Redis 3.2+，旧版本为 39 字节），且非 int 编码的字符串，如`SET username zhangsan`。
- **底层结构**：嵌入式 SDS 编码，RedisObject 对象头与 SDS 结构在同一块连续的内存空间中，仅需一次内存分配，内存布局紧凑，CPU 缓存行命中率高，访问性能远高于 raw 编码。
- **核心特性**：
    
    1. 只读编码：embstr 编码的字符串为只读，任何修改操作（如 APPEND、SET）都会导致其自动转换为 raw 编码；
    2. 内存优势：连续内存空间，减少内存碎片，内存占用远低于 raw 编码；
    3. 性能优势：仅需一次内存分配 / 释放，而 raw 编码需要两次（分别分配 RedisObject 与 SDS）。
    

##### 3. raw 编码

- **适用场景**：Value 为字符串，且字节长度**大于 44 字节**，如长 JSON 字符串、序列化对象等。
- **底层结构**：RedisObject 对象头与 SDS 结构分别在两块独立的内存空间中，通过指针关联，需两次内存分配 / 释放，支持动态扩容与修改。
- **核心特性**：
    
    1. 支持动态修改，修改操作不会触发编码转换，仅会调整 SDS 的内存空间；
    2. 无长度上限，最大支持 512MB 的字符串；
    3. 内存开销高于 embstr 与 int 编码，访问性能略低。

#### 三、编码转换规则与核心限制

1. **编码转换为 Redis 自动完成，用户无法手动指定**，仅能通过`OBJECT ENCODING key`命令查询当前 Key 的底层编码。
2. **int 编码转换规则**：
    
    - 对 int 编码的 Key 执行非数值操作（如 APPEND、SET 字符串值），会立即转换为 embstr/raw 编码；
    - 数值超过 64 位有符号整数范围，会自动转换为 embstr/raw 编码。
    
3. **embstr 编码转换规则**：
    
    - 对 embstr 编码的 Key 执行任何修改操作，无论长度是否超过 44 字节，都会立即转换为 raw 编码；
    - 字符串长度超过 44 字节，会自动转换为 raw 编码。
    
4. **raw 编码无反向转换**：raw 编码的字符串，即使修改后长度小于 44 字节，也不会自动转换回 embstr 编码，仅能通过重新 SET 触发编码重新选择。

#### 四、生产环境编码优化规范

1. 优先使用 int 编码存储整数数值，最大化内存利用率与访问性能，避免将数字转为带引号的字符串存储，导致无法使用 int 编码。
2. 短字符串（≤44 字节）优先使用 embstr 编码，避免不必要的修改操作触发编码转换，降低内存开销。
3. 禁止存储超长字符串（>100KB），raw 编码的超长字符串会导致内存碎片、网络阻塞、主线程卡顿，生产环境需拆分存储。
4. 定期通过`OBJECT ENCODING`命令检查编码格式，优化不合理的存储方式，提升内存利用率与访问性能。

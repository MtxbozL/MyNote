#### 一、结构化对象存储

- **核心实现**：将业务对象（用户信息、商品详情、订单信息）的每个属性作为 field，属性值作为 value，存储在一个 Hash 中，key 为对象的唯一标识（如`user:info:1001`、`product:detail:2001`）。相比 String 序列化存储，Hash 具备三大核心优势：
    
    1. 按需读写单个字段，无需读取整个对象，大幅减少网络开销；
    2. 支持原子性更新单个字段，无需加锁，无并发竞态问题；
    3. 多个字段聚合存储，减少 Redis 全局 Key 数量，降低内存开销。
    
- **核心命令**：`HSET`、`HGET`、`HMGET`、`HINCRBY`
- **生产规范**：适合属性多、读写频繁的结构化对象，单个 Hash 的 field 数量建议控制在 1000 以内，避免大哈希；超大对象需拆分存储。

#### 二、电商购物车设计

- **核心实现**：以用户 ID 为 key（如`cart:user:1001`），商品 ID 为 field，商品购买数量、规格等信息为 value；核心操作如下：
    
    - 加入购物车：`HSET`/`HINCRBY`更新商品数量；
    - 修改商品数量：`HINCRBY`原子增减；
    - 删除商品：`HDEL`移除对应 field；
    - 全量查询购物车：`HSCAN`/`HGETALL`；
    - 统计商品种类：`HLEN`。
    
- **核心命令**：`HSET`、`HINCRBY`、`HDEL`、`HLEN`、`HMGET`
- **生产规范**：用户购物车 field 数量有限，可安全使用 HGETALL；临时购物车需设置过期时间，永久购物车需定期同步到数据库，避免 Redis 故障导致数据丢失。

#### 三、多维度计数器

- **核心实现**：文章 / 视频的阅读量、点赞量、评论量、收藏量等多维度统计，以内容 ID 为 key，各统计维度为 field，通过`HINCRBY`原子性更新单个维度计数，通过`HMGET`批量读取所有统计数据。相比多个 String 类型计数器，Hash 聚合存储大幅减少 Key 数量，内存利用率更高，便于统一管理。
- **核心命令**：`HINCRBY`、`HMGET`

#### 四、细粒度权限 / 配置管理

- **核心实现**：以角色 / 用户 ID 为 key，权限项 / 配置项为 field，权限值 / 配置值为 value，支持按需查询、修改单个权限项，无需读取整个配置文件，更新操作原子性无竞态。
- **核心命令**：`HGET`、`HSET`、`HDEL`、`HEXISTS`

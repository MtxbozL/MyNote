#### 一、用户 / 内容标签系统

- **核心实现**：以标签为 Key，对应该标签的用户 / 内容 ID 为集合元素（如`tag:user:sports`存储所有体育标签的用户 ID）；用户新增标签时执行 SADD，移除标签时执行 SREM，判断用户是否具备某标签通过 SISMEMBER 实现；多标签交集筛选通过 SINTER 实现（如筛选同时具备「体育」「90 后」标签的用户）。
- **核心命令**：`SADD`、`SREM`、`SISMEMBER`、`SINTER`
- **生产规范**：标签与用户的映射需双向存储，同时维护用户对应的标签集合，便于双向查询；大标签集合的交集运算必须使用 SINTERSTORE，避免阻塞。

#### 二、社交关系系统

- **核心实现**：
    
    1. 关注 / 粉丝列表：以用户 ID 为 Key，关注列表 / 粉丝列表为集合元素，关注操作通过 SADD 实现，取关通过 SREM 实现；
    2. 共同好友 / 共同关注：通过 SINTER 计算两个用户关注集合的交集，得到共同关注列表；
    3. 推荐关注：通过 SDIFF 计算目标用户关注的人、而当前用户未关注的集合，实现推荐关注。
    
- **核心命令**：`SADD`、`SREM`、`SINTER`、`SDIFF`

#### 三、随机抽奖 / 盲盒系统

- **核心实现**：将所有参与抽奖的用户 ID / 奖品 ID 存入集合，通过 SPOP 实现随机抽取中奖用户 / 奖品，抽取后自动从集合中移除，避免重复中奖；通过 SRANDMEMBER 实现不重复的随机名单展示，不修改集合数据。
- **核心命令**：`SADD`、`SPOP`、`SRANDMEMBER`
- **生产规范**：高并发抽奖场景需通过 Lua 脚本保证抽奖与库存扣减的原子性，避免超抽；大数量抽奖名单需分批 SPOP，避免单次弹出元素过多阻塞主线程。

#### 四、IP / 接口黑白名单

- **核心实现**：将黑名单 / 白名单 IP 存入集合，请求到达时通过 SISMEMBER 快速判断 IP 是否在名单中，O (1) 时间复杂度支持百万级 IP 的高频校验，无性能瓶颈。
- **核心命令**：`SADD`、`SREM`、`SISMEMBER`
- **生产规范**：黑白名单需设置过期时间，定期更新；分布式场景下通过集合运算实现多维度名单合并。

#### 五、数据去重与幂等性控制

- **核心实现**：将已处理的请求 ID、订单号、流水号存入集合，重复请求通过 SISMEMBER 判断是否已处理，实现接口幂等性；批量数据处理时通过 SADD 天然去重，无需业务层遍历判断。
- **核心命令**：`SADD`、`SISMEMBER`

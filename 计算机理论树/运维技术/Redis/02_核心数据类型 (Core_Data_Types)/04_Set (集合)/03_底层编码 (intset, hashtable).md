Set 的底层编码分为 **intset（整数集合）**与**hashtable（哈希表）** 两种，Redis 根据集合元素的类型与数量自动选择编码，满足阈值时自动转换，转换后不可逆。

#### 一、intset 整数集合

- **适用场景**：当集合同时满足以下两个条件时，使用 intset 编码：
    
    1. 集合中的所有元素均为 64 位有符号整数；
    2. 集合的元素总数量不超过`set-max-intset-entries`（redis.conf 配置，默认 512）。
    
- **底层结构**：intset 是一段**连续紧凑的内存空间**，结构如下：

| 字段       | 长度   | 核心作用                                                                                |
| -------- | ---- | ----------------------------------------------------------------------------------- |
| encoding | 4 字节 | 编码格式，标识整数的位宽：INTSET_ENC_INT16（16 位整数）、INTSET_ENC_INT32（32 位）、INTSET_ENC_INT64（64 位） |
| length   | 4 字节 | 集合的元素总数量                                                                            |
| contents | 不定长  | 有序整数数组，存储集合所有元素，按从小到大升序排列，数组长度 = length*encoding 位宽                                 |

- **核心特性**：
    
    1. 编码自动升级：当新增的整数超出当前 encoding 的位宽范围时，会自动升级到适配的位宽，所有元素同步转换位宽，升级后不可逆（如 int16 集合新增 int32 整数，整体升级为 int32 编码）；
    2. 有序存储二分查找：contents 数组有序存储，元素查找采用二分查找，时间复杂度 O (logN)，远超无序数组的 O (N)；
    3. 内存极致优化：无指针冗余，连续内存空间，内存占用远低于 hashtable，小整数集合场景内存利用率极高。
    
- **核心缺陷**：
    
    1. 插入 / 删除操作需重新分配内存，元素数量增多时性能下降；
    2. 编码升级会触发全量元素位宽转换，大集合场景会阻塞主线程；
    3. 仅支持整数类型元素，字符串类型元素无法使用。
    

#### 二、hashtable 哈希表编码

- **适用场景**：当集合不满足 intset 的任一条件时，自动从 intset 转换为 hashtable 编码，转换后不可逆。
- **底层结构**：与 Hash 类型的 hashtable 完全一致，基于 Redis 字典（dict）结构实现，核心差异为：字典的 key 存储集合元素，value 固定为 NULL，无需存储额外值，通过哈希表的 key 唯一性实现集合元素的去重。
    
    1. 哈希计算：对元素计算哈希值，通过掩码定位哈希桶索引；
    2. 冲突解决：采用链地址法，同一哈希桶的冲突元素通过单向链表串联；
    3. 渐进式 rehash：负载因子超过阈值时自动扩容，rehash 操作分散到每次增删改查中，无主线程阻塞。
    
- **核心优势**：
    
    1. 元素增删查操作平均时间复杂度 O (1)，性能稳定，不受元素数量与类型限制；
    2. 支持任意字符串类型元素，无类型限制；
    3. 无内存重分配与编码升级的阻塞风险，适合大集合场景。
    
- **核心缺陷**：
    
    1. 内存开销大，每个元素对应一个 dictEntry 节点，存在 3 个指针冗余，小集合场景内存利用率远低于 intset；
    2. 元素无序存储，无法实现范围查找。
    

#### 生产环境编码优化规范

1. 合理设置`set-max-intset-entries`，默认 512 为性能与内存的最优平衡，整数集合场景可适当调高至 1024，避免频繁编码转换；
2. 整数类型的集合优先使用 intset 编码，最大化内存利用率，禁止将整数转为字符串存储，导致无法使用 intset；
3. 大集合必须使用 SSCAN 增量遍历，绝对禁止使用 SMEMBERS 全量读取，避免阻塞主线程；
4. 生产环境集合运算优先使用 STORE 系列命令，避免大结果集网络传输，降低阻塞风险。

### 03_常见问题 (数据延迟、读写分离坑、复制风暴)

#### 一、主从数据延迟

1. **核心原因**
    
    - 命令传播异步性：主节点执行写命令后立即返回客户端，再异步发送给从节点，存在天然的网络传输延迟；
    - 从节点性能不足：从节点 CPU / 内存 / 磁盘性能弱于主节点，命令执行速度慢，导致偏移量差距持续扩大；
    - 主节点写并发过高：主节点每秒写入的命令量超过从节点的处理能力，从节点无法及时同步；
    - 复制链路过长：树状主从架构中，二级从节点的复制延迟逐级放大；
    - 全量复制 / 重写阻塞：从节点加载 RDB、AOF 重写期间，无法处理主节点的增量命令，导致延迟飙升。
    
2. **生产环境解决方案**
    
    - 主从节点采用同规格硬件配置，避免从节点性能短板；
    - 监控主从复制偏移量差值，正常场景应控制在 100ms 以内，超过阈值及时告警；
    - 主节点高并发写场景，优化批量写操作，避免瞬间大量写命令压垮从节点；
    - 树状主从架构控制复制层级不超过 2 层，减少链路延迟；
    - 调整复制积压缓冲区大小，高并发场景建议调至 100MB 以上，避免网络中断后触发全量复制。
    

#### 二、读写分离核心坑点

读写分离是主从架构的核心价值，但存在多个极易踩坑的场景，生产环境必须规避：

1. **从节点过期键读取问题**
    
    - 问题现象：主节点已删除的过期键，从节点仍能读取到，导致脏数据。
    - 根本原因：主从架构下，从节点不会主动删除过期键，仅能同步主节点的 DEL 命令，若主节点的过期键尚未被惰性删除 / 定期删除，从节点仍会返回该键的值，即使已过期。
    - 解决方案：Redis 3.2+ 优化了过期键判断逻辑，从节点读取键时会先检查过期时间，已过期则返回 nil，生产环境必须升级到 3.2 以上版本；业务层避免读取即将过期的键，同时设置合理的过期时间。
    
2. **从节点故障导致的读雪崩**
    
    - 问题现象：多个从节点宕机后，所有读请求全部打到主节点，主节点瞬间被打满，引发服务雪崩。
    - 解决方案：客户端实现读请求的熔断与降级机制，从节点不可用时自动熔断，避免请求转发到主节点；配置从节点监控，宕机后快速扩容补充从节点；核心读请求保留主节点兜底能力，控制主节点的读请求上限。
    
3. **数据延迟导致的读写不一致**
    
    - 问题现象：主节点刚写入的数据，立即从从节点读取不到，导致业务逻辑异常（如下单后立即查询订单不存在）。
    - 解决方案：强一致性读请求直接路由到主节点；非强一致性请求可延迟读取；客户端实现偏移量判断，仅从偏移量不小于写入时偏移量的从节点读取；核心业务避免读写分离，仅将冷数据、非实时数据的读请求路由到从节点。
    
4. **从节点只读配置失效**
    
    - 问题现象：从节点开启`slave-read-only yes`配置后，仍能执行写命令，导致主从数据不一致。
    - 根本原因：`slave-read-only`仅对普通客户端生效，拥有管理员权限的客户端仍可执行写命令；同时 Redis Cluster 模式下从节点的只读配置不生效。
    - 解决方案：生产环境必须开启`slave-read-only yes`；通过 ACL 权限控制，禁止普通业务用户在从节点执行写命令；定期校验主从数据一致性。
    

#### 三、复制风暴

复制风暴是主从架构下的严重故障，指大量从节点同时向主节点发起全量复制，导致主节点 CPU、内存、带宽资源被耗尽，服务完全不可用。

1. **核心触发场景**
    
    - 主节点重启，所有从节点检测到主节点 runid 变化，同时发起全量复制；
    - 网络大面积波动，所有从节点同时断开重连，且偏移量超出复制积压缓冲区范围，触发全量复制；
    - 一主多从架构下，从节点数量过多，主节点带宽不足，单个从节点全量复制未完成，其他从节点又触发全量复制。
    
2. **生产环境解决方案**
    
    - 采用树状主从架构，减少直接连接主节点的从节点数量，主节点仅对接 2-3 个一级从节点，其余从节点复制一级从节点；
    - 调大复制积压缓冲区大小，高并发场景建议设置为 1GB 以上，大幅降低网络中断后全量复制的触发概率；
    - 升级到 Redis 4.0 + 版本，使用 PSYNC2 协议，主节点重启后仍可实现增量复制，避免全量复制；
    - 控制单个主节点的从节点数量，建议不超过 5 个，超过则采用树状架构；
    - 全量复制避开业务高峰时段，手动切换主从、重启节点时，错开从节点的重连时间，避免同时触发全量复制。

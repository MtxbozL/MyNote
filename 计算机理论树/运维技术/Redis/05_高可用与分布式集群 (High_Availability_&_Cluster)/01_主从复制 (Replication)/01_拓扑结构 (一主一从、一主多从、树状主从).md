Redis 单节点架构存在三大核心瓶颈：**单点故障风险**（进程宕机 / 服务器故障导致服务不可用）、**读写性能上限**（单节点无法支撑海量并发读请求）、**内存容量上限**（单节点内存无法支撑 TB 级海量数据存储）。高可用与分布式集群架构正是为解决上述问题设计，分为三个递进的架构层级：主从复制（数据多副本与读写分离）、哨兵模式（主从架构的自动化故障转移）、Redis Cluster 分片集群（分布式数据分片与水平扩展）。

---

## 01_主从复制 (Replication)

主从复制是 Redis 高可用架构的基础，核心是**一主多从的多副本架构**：主节点（Master）负责处理所有写请求，同步写命令到所有从节点（Slave）；从节点只读，负责处理读请求，实现读写分离与数据多副本备份，主节点故障时可手动切换从节点为主节点，避免数据完全丢失。

### 01_拓扑结构 (一主一从、一主多从、树状主从)

Redis 主从复制支持三种核心拓扑结构，适配不同的业务规模与可用性需求，核心差异在于复制链路、性能开销与可用性。

|拓扑类型|结构说明|核心优势|核心缺陷|生产适用场景|
|---|---|---|---|---|
|一主一从|1 个 Master 节点 + 1 个 Slave 节点，Slave 直接复制 Master，是最简单的主从架构|1. 架构极简，运维成本极低；<br><br>2. 数据双副本备份，避免单点数据丢失；<br><br>3. 从节点可承担读请求与数据备份任务，不影响主节点性能|1. 读写分离能力有限，仅 1 个从节点支撑读请求；<br><br>2. 无自动故障转移能力，主节点故障需手动切换；<br><br>3. 从节点宕机后读请求会直接打到主节点，引发性能风险|小型业务、测试环境、冷备场景，对读并发要求不高的业务|
|一主多从|1 个 Master 节点 + N 个 Slave 节点，所有 Slave 直接复制 Master，是工业界最常用的主从拓扑|1. 线性提升读并发能力，多个从节点分担海量读请求；<br><br>2. 多副本数据备份，数据安全性更高；<br><br>3. 架构清晰，运维难度低，可按需横向扩展从节点数量|1. 主节点复制压力随从节点数量增长而上升，每个从节点都需要与主节点建立复制连接，全量复制时会占用主节点大量 CPU/IO/ 带宽资源；<br><br>2. 无自动故障转移能力，主节点故障需手动切换；<br><br>3. 网络波动会导致多个从节点同时触发全量复制，引发复制风暴|中大型业务、读高并发场景，如电商商品详情读、用户信息查询等读多写少业务|
|树状主从（链式主从）|1 个 Master 节点 + 多层级 Slave 节点，从节点分为一级从节点与二级从节点，二级从节点复制一级从节点，而非直接复制 Master，形成树状复制链路|1. 极大降低主节点的复制压力，主节点仅需向少量一级从节点同步数据，二级从节点的复制压力由一级从节点承担；<br><br>2. 跨机房部署场景下，可在备机房部署一级从节点，备机房所有从节点复制该一级节点，避免跨机房带宽占用；<br><br>3. 避免一主多从架构的复制风暴风险|1. 复制链路变长，数据延迟逐级放大，二级从节点的数据延迟远高于一级从节点；<br><br>2. 架构复杂度提升，中间节点故障会导致下游所有从节点复制中断；<br><br>3. 故障切换难度更高，需重新梳理复制链路|超大规模读并发场景、跨机房主从部署、从节点数量极多的业务场景|

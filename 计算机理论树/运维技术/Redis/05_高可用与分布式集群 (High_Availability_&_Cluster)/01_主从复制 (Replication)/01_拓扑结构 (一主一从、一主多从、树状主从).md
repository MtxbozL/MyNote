Redis 单节点架构存在三大核心瓶颈：**单点故障风险**（进程宕机 / 服务器故障导致服务不可用）、**读写性能上限**（单节点无法支撑海量并发读请求）、**内存容量上限**（单节点内存无法支撑 TB 级海量数据存储）。高可用与分布式集群架构正是为解决上述问题设计，分为三个递进的架构层级：主从复制（数据多副本与读写分离）、哨兵模式（主从架构的自动化故障转移）、Redis Cluster 分片集群（分布式数据分片与水平扩展）。

---

## 01_主从复制 (Replication)

主从复制是 Redis 高可用架构的基础，核心是**一主多从的多副本架构**：主节点（Master）负责处理所有写请求，同步写命令到所有从节点（Slave）；从节点只读，负责处理读请求，实现读写分离与数据多副本备份，主节点故障时可手动切换从节点为主节点，避免数据完全丢失。

### 01_拓扑结构 (一主一从、一主多从、树状主从)

Redis 主从复制支持三种核心拓扑结构，适配不同的业务规模与可用性需求，核心差异在于复制链路、性能开销与可用性。

|拓扑类型|结构说明|核心优势|核心缺陷|生产适用场景|
|---|---|---|---|---|
|一主一从|1 个 Master 节点 + 1 个 Slave 节点，Slave 直接复制 Master，是最简单的主从架构|1. 架构极简，运维成本极低；<br><br>2. 数据双副本备份，避免单点数据丢失；<br><br>3. 从节点可承担读请求与数据备份任务，不影响主节点性能|1. 读写分离能力有限，仅 1 个从节点支撑读请求；<br><br>2. 无自动故障转移能力，主节点故障需手动切换；<br><br>3. 从节点宕机后读请求会直接打到主节点，引发性能风险|小型业务、测试环境、冷备场景，对读并发要求不高的业务|
|一主多从|1 个 Master 节点 + N 个 Slave 节点，所有 Slave 直接复制 Master，是工业界最常用的主从拓扑|1. 线性提升读并发能力，多个从节点分担海量读请求；<br><br>2. 多副本数据备份，数据安全性更高；<br><br>3. 架构清晰，运维难度低，可按需横向扩展从节点数量|1. 主节点复制压力随从节点数量增长而上升，每个从节点都需要与主节点建立复制连接，全量复制时会占用主节点大量 CPU/IO/ 带宽资源；<br><br>2. 无自动故障转移能力，主节点故障需手动切换；<br><br>3. 网络波动会导致多个从节点同时触发全量复制，引发复制风暴|中大型业务、读高并发场景，如电商商品详情读、用户信息查询等读多写少业务|
|树状主从（链式主从）|1 个 Master 节点 + 多层级 Slave 节点，从节点分为一级从节点与二级从节点，二级从节点复制一级从节点，而非直接复制 Master，形成树状复制链路|1. 极大降低主节点的复制压力，主节点仅需向少量一级从节点同步数据，二级从节点的复制压力由一级从节点承担；<br><br>2. 跨机房部署场景下，可在备机房部署一级从节点，备机房所有从节点复制该一级节点，避免跨机房带宽占用；<br><br>3. 避免一主多从架构的复制风暴风险|1. 复制链路变长，数据延迟逐级放大，二级从节点的数据延迟远高于一级从节点；<br><br>2. 架构复杂度提升，中间节点故障会导致下游所有从节点复制中断；<br><br>3. 故障切换难度更高，需重新梳理复制链路|超大规模读并发场景、跨机房主从部署、从节点数量极多的业务场景|

# Redis 一主一从 + Sentinel 哨兵 完整搭建操作流程

## 一、核心架构与前置说明

### 1. 标准生产级架构规划

|节点角色|IP 地址|端口|核心作用|
|---|---|---|---|
|Redis 主节点（Master）|192.168.1.100|6379|负责数据写入，主从复制源|
|Redis 从节点（Slave）|192.168.1.101|6379|只读同步主节点数据，故障时可升级为主|
|Sentinel 哨兵节点 1|192.168.1.100|26379|监控主从状态，参与故障判定与选举|
|Sentinel 哨兵节点 2|192.168.1.101|26379|集群投票节点，保障法定票数|
|Sentinel 哨兵节点 3|192.168.1.102|26379|集群投票节点，避免脑裂问题|

### 2. 核心规则与前置条件

- **强制要求**：哨兵集群必须部署**奇数个节点（最少 3 个）**，故障判定、leader 选举需要过半法定票数，2 个节点无法完成选举，会导致故障转移彻底失效。
- **前置检查**：
    
    1. 已完成上一章节的一主一从 Redis 搭建，主从复制状态正常（`master_link_status: up`）
    2. 所有节点防火墙已开放`6379`（Redis）、`26379`（Sentinel）端口，节点间双向网络互通
    3. 所有节点已安装 Redis（`redis-sentinel`随 Redis 原生自带，无需额外部署）
    4. 主节点已配置`requirepass`密码，所有从节点已配置`masterauth`，且密码全局统一

## 二、所有哨兵节点统一环境准备（3 个哨兵节点均执行）

### 1. 创建日志与数据目录

```bash
# 哨兵数据&日志目录，避免权限不足
mkdir -p /var/redis/sentinel
# 配置文件目录
mkdir -p /etc/redis
```

### 2. 统一配置文件说明

3 个哨兵节点的核心配置**必须完全一致**，仅当绑定固定内网 IP 时，`bind`参数可单独调整，其余参数严禁不一致，否则会出现集群发现失败、故障转移异常。

---

## 三、哨兵节点核心配置文件编写

### 1. 编辑哨兵配置文件

在所有 3 个哨兵节点上，创建`/etc/redis/sentinel.conf`配置文件，内容如下：

```bash
# 哨兵服务端口，固定26379（默认）
port 26379

# 允许远程访问，生产环境建议绑定具体内网IP，禁止绑定127.0.0.1
bind 0.0.0.0

# 后台守护进程运行
daemonize yes

# PID文件存放路径
pidfile /var/run/redis-sentinel.pid

# 日志文件路径，故障排查核心依据
logfile "/var/redis/sentinel/sentinel.log"

# 哨兵工作数据目录
dir /var/redis/sentinel

# -------------------------- 核心监控配置 --------------------------
# 1. 定义监控的主节点：别名mymaster（自定义，全局统一）+ 主节点IP+端口 + 法定票数quorum
# quorum规则：判定主节点客观下线，至少需要quorum个哨兵同时同意；3节点集群建议设2，5节点设3
sentinel monitor mymaster 192.168.1.100 6379 2

# 2. 主节点认证密码，必须和主节点的requirepass完全一致，别名必须和上面的mymaster统一
sentinel auth-pass mymaster your_master_password

# 3. 主观下线判定超时（毫秒）：哨兵超过该时长未收到主节点心跳，判定为主观下线SDOWN
# 生产环境建议30000（30秒），避免网络波动误判，测试环境可设5000加快验证
sentinel down-after-milliseconds mymaster 30000

# 4. 故障转移超时时间（毫秒）：超过该时长未完成故障转移，判定本次转移失败
sentinel failover-timeout mymaster 180000

# 5. 故障转移后，并行同步新主节点的从节点数量
# 数字越小，同步对主节点带宽影响越小，同步耗时越长；1主1从架构固定设1即可
sentinel parallel-syncs mymaster 1

# 日志级别，生产环境设notice，排查问题可设debug
loglevel notice
```

### 2. 配置文件权限设置

```bash
# 必须给配置文件写权限，哨兵会在故障转移、节点发现时自动修改配置文件永久保存
chmod 644 /etc/redis/sentinel.conf
```

## 四、启动集群与哨兵节点

### 1. 严格遵循启动顺序

**主节点 → 从节点 → 所有哨兵节点**，严禁颠倒顺序，否则会出现哨兵无法发现主从节点的问题。

### 2. 启动主从节点（已启动可跳过）

```bash
# 主节点192.168.1.100执行
redis-server /etc/redis/redis_master.conf

# 从节点192.168.1.101执行
redis-server /etc/redis/redis_slave.conf
```

### 3. 启动所有哨兵节点（3 个节点均执行）

两种启动方式二选一，推荐第一种原生哨兵启动方式：

```bash
# 方式1：redis-sentinel原生启动（推荐）
redis-sentinel /etc/redis/sentinel.conf

# 方式2：redis-server带哨兵模式启动
redis-server /etc/redis/sentinel.conf --sentinel
```

### 4. 验证进程是否正常启动

```bash
# 查看Redis进程
ps -ef | grep redis-server
# 查看哨兵进程
ps -ef | grep redis-sentinel
```

正常可看到 2 个 Redis 进程、3 个哨兵进程均处于运行状态。

## 五、哨兵集群与主从监控状态验证

### 1. 基础集群状态验证

登录任意一个哨兵节点，执行以下命令：

```bash
# 连接哨兵节点
redis-cli -p 26379

# 查看哨兵核心信息
127.0.0.1:26379> info sentinel
```

**正常输出核心指标**：

```bash
# Sentinel
sentinel_masters:1
sentinel_tilt:0  # 0=正常，1=触发保护模式，不会执行故障转移
sentinel_running_scripts:0
sentinel_scripts_queue_length:0
sentinel_simulate_failure_flags:0
master0:name=mymaster,status=ok,address=192.168.1.100:6379,slaves=1,sentinels=3
```

重点校验：`status=ok`、`slaves=1`（正确识别 1 个从节点）、`sentinels=3`（正确识别集群所有 3 个哨兵节点）。

### 2. 详细监控信息验证

```bash
# 查看监控的主节点完整详情
127.0.0.1:26379> sentinel master mymaster

# 查看识别到的从节点列表与状态
127.0.0.1:26379> sentinel slaves mymaster

# 查看集群内其他哨兵节点信息
127.0.0.1:26379> sentinel sentinels mymaster

# 直接获取当前主节点的IP和端口
127.0.0.1:26379> sentinel get-master-addr-by-name mymaster
```

正常输出会返回当前主节点`192.168.1.100 6379`，说明哨兵监控正常。

## 六、核心功能验证：自动故障转移测试

### 1. 测试前置确认

先确认当前主节点地址，记录下来：

```bash
# 哨兵节点执行
127.0.0.1:26379> sentinel get-master-addr-by-name mymaster
# 正常返回：1) "192.168.1.100" 2) "6379"
```

### 2. 模拟主节点宕机

登录原主节点`192.168.1.100`，手动停止 Redis 进程，模拟宕机：

```bash
# 主节点执行，停止Redis服务
redis-cli -p 6379 -a your_master_password shutdown
```

### 3. 观察故障转移全过程

#### 步骤 1：查看哨兵日志，跟踪转移流程

在任意哨兵节点执行，实时查看日志：

```bash
tail -f /var/redis/sentinel/sentinel.log
```

正常会按顺序输出以下核心事件：

1. `+sdown master mymaster 192.168.1.100 6379`：单个哨兵判定主节点主观下线
2. `+odown master mymaster 192.168.1.100 6379 #quorum 2/2`：达到法定票数，判定主节点客观下线
3. `+new-epoch 1`：更新选举纪元
4. `+vote-for-leader xxxxxxxxxx`：哨兵集群选举出 leader 节点
5. `+failover-state-select-slave master mymaster 192.168.1.100 6379`：筛选合格的从节点
6. `+promoted-slave slave 192.168.1.101:6379`：将从节点升级为新主节点
7. `+failover-end master mymaster 192.168.1.100 6379`：故障转移完成

#### 步骤 2：验证新主节点

等待 30 秒左右（匹配`down-after-milliseconds`配置），在哨兵节点执行：

```bash
127.0.0.1:26379> sentinel get-master-addr-by-name mymaster
```

**正常返回新主节点地址**：`1) "192.168.1.101" 2) "6379"`，说明自动故障转移成功。

#### 步骤 3：验证新主节点读写权限

登录新主节点`192.168.1.101`，验证可正常写入（已自动关闭只读模式）：

```bash
# 连接新主节点
redis-cli -h 192.168.1.101 -p 6379 -a your_master_password

# 写入测试数据
192.168.1.101:6379> set failover_test "success"
OK
```

### 4. 原主节点恢复验证

重新启动原主节点`192.168.1.100`的 Redis 服务：

```bash
# 原主节点执行
redis-server /etc/redis/redis_master.conf
```

**核心特性**：哨兵会自动将原主节点识别为新主节点的从节点，自动修改其配置文件中的`replicaof`指向新主节点，无需手动修改任何配置，等待 10 秒左右即可在哨兵中看到从节点数量恢复为 1。

## 七、业务客户端连接哨兵集群配置示例

哨兵的核心作用是给客户端提供主节点地址发现能力，客户端无需固定主节点 IP，只需连接哨兵集群即可自动获取当前可用主节点。

### 1. Redis-cli 原生连接方式

```bash
# 通过哨兵获取当前主节点地址
redis-cli -p 26379 sentinel get-master-addr-by-name mymaster

# 连接获取到的主节点

```

### 2. Spring Boot 整合配置示例

```yaml
spring:
  redis:
    # 全局密码，和主节点requirepass一致

        - 192.168.1.100:26379

        - 192.168.1.102:26379
    lettuce:
      pool:
        max-active: 8
        max-idle: 8
        min-idle: 2
```

## 八、生产环境核心注意事项与避坑指南

### 1. 集群部署核心规则

- **严禁哨兵节点数为偶数**：最少 3 个节点，生产环境建议 3-5 个，必须保证过半节点存活才能执行故障转移；禁止 2 个哨兵节点部署，主节点故障时无法完成选举。
- **节点分离部署**：生产环境强烈建议哨兵节点与 Redis 节点分开部署，不要所有哨兵都和主节点在同一台服务器，避免主服务器宕机导致哨兵集群过半节点失效。
- **quorum 设置规范**：必须设置为「哨兵节点数 / 2 +1」，3 节点设 2，5 节点设 3，确保必须过半节点同意才能判定主节点下线，从根源避免脑裂。

### 2. 密码配置避坑

- 主节点`requirepass`、所有从节点`masterauth`、哨兵`sentinel auth-pass`三者必须完全一致，否则会出现哨兵无法监控节点、故障转移后从节点无法同步的问题。
- 所有 Redis 节点（主 + 从）必须配置相同的密码，故障转移后新主节点会继承原配置，避免出现无密码导致的同步失败。

### 3. 脑裂问题终极防范

脑裂：主节点发生网络分区，哨兵判定其下线并选举出新主，原主节点恢复后出现双主，导致业务写入双主、数据丢失。

**防范配置（主节点 redis.conf 添加）**：

```bash
# 主节点至少有1个从节点连接，才能接受写入请求
min-replicas-to-write 1
# 从节点的最大复制延迟不能超过10秒
min-replicas-max-lag 10
```

原理：当主节点与从节点、哨兵网络断开后，会自动拒绝写入，避免脑裂期间产生不一致数据。

### 4. 配置文件与运维规范

- 哨兵配置文件必须开启写权限，哨兵会自动修改配置文件永久保存节点信息、故障转移结果，禁止手动修改运行中的哨兵配置文件。
- 重点监控`sentinel_tilt`指标，该指标为 1 时表示哨兵进入保护模式，不会执行任何故障转移，通常由系统时间异常、服务器资源不足、网络大面积波动导致。
- 版本升级顺序：先升级所有从节点 → 升级主节点 → 最后升级所有哨兵节点，避免版本兼容导致的同步失败。
- 禁止同时停止超过半数的哨兵节点，否则集群会失去选举与故障转移能力。

### 5. 网络与持久化规范

- 所有节点之间必须双向互通：主↔从、主↔所有哨兵、从↔所有哨兵、哨兵↔哨兵，所有端口必须开放，生产环境禁止绑定 127.0.0.1。
- 哨兵仅负责监控与故障转移，不负责数据持久化，主从节点必须独立配置 RDB/AOF 持久化，建议主从节点都开启，避免故障转移后新主节点重启数据丢失。
- 跨机房部署时，需调大`down-after-milliseconds`，避免网络延迟导致的下线误判，生产环境不建议跨机房部署哨兵集群。

## 九、常见问题排查

1. **哨兵无法发现从节点**：检查从节点`replicaof`配置是否正确、`masterauth`密码是否匹配、主从复制是否正常（`info replication`查看`master_link_status`是否为 up）。
2. **哨兵无法发现其他哨兵节点**：检查哨兵之间 26379 端口是否互通、配置文件的主节点别名、IP、quorum 是否完全一致。
3. **主节点下线不执行故障转移**：检查 quorum 设置是否正确、是否有过半哨兵同意下线、哨兵节点数是否为奇数、是否触发 tilt 保护模式。
4. **故障转移后从节点无法同步**：检查新主节点`requirepass`是否配置、从节点`masterauth`是否与新主节点密码一致、网络是否互通。

#### 一、Sentinel 领导者选举（基于 Raft 一致性算法）

当主节点被标记为客观下线后，Sentinel 集群需要先选举出一个**领导者 Sentinel 节点**，由该节点全权负责后续的故障转移流程，避免多个 Sentinel 同时执行故障转移导致的集群混乱。选举过程基于 Raft 分布式一致性算法的领导者选举机制，核心流程如下：

1. 每个将主节点标记为 ODOWN 的 Sentinel 节点，都会向其他所有 Sentinel 节点发送请求，要求选举自己为领导者；
2. 收到请求的 Sentinel 节点，若本轮尚未投票给其他节点，会同意该请求，投票给发起者；
3. 若某个 Sentinel 节点，获得的票数**同时满足两个条件**：① 票数≥quorum 法定票数；② 票数≥Sentinel 集群总节点数的半数 + 1，则该节点当选为领导者；
4. 若本轮选举无节点满足当选条件，会进入下一轮选举，直到选出领导者为止；
5. 整个选举过程保证只有一个领导者 Sentinel 节点，由该节点唯一执行故障转移流程。

#### 二、完整故障转移执行流程

领导者 Sentinel 节点选举完成后，会执行完整的自动化故障转移流程，全程无需人工干预，共分为 6 个核心步骤：

1. **筛选合格的从节点**
    
    领导者 Sentinel 会从故障主节点的所有从节点中，筛选出符合资格的候选主节点，过滤规则：
    
    - 过滤掉已下线、断线的从节点；
    - 过滤掉最近 5 秒内未响应 INFO 命令的从节点，保证节点存活；
    - 过滤掉与故障主节点断开连接超过`down-after-milliseconds * 10`时间的从节点，保证数据完整性，避免丢失过多数据；
    - 优先保留优先级更高的从节点（`slave-priority`配置值越小，优先级越高，0 代表永远不会被选为主节点）。
    
2. **选择最优从节点作为新主节点**
    
    从合格的候选节点中，按以下优先级排序，选出最优的从节点提升为新主节点：
    
    1. 第一优先级：`slave-priority`优先级最高的节点；
    2. 第二优先级：复制偏移量最大的节点（与故障主节点的数据最接近，丢失数据最少）；
    3. 第三优先级：runid 最小的节点（兜底规则，保证选举结果唯一）。
    
3. **提升从节点为新主节点**
    
    领导者 Sentinel 向选中的最优从节点发送`SLAVEOF NO ONE`命令，让其脱离原主节点，成为独立的主节点；同时持续发送`INFO`命令监控该节点，直到其角色成功切换为 master。
    
4. **让其他从节点复制新主节点**
    
    领导者 Sentinel 向故障主节点的所有剩余从节点，发送`SLAVEOF <新主节点IP> <新主节点端口>`命令，让这些从节点改为复制新的主节点，建立新的主从复制拓扑。
    
5. **更新故障主节点的配置**
    
    领导者 Sentinel 将原故障主节点标记为下线，更新其配置为新主节点的从节点，待其恢复上线后，自动发送`SLAVEOF`命令，让其成为新主节点的从节点。
    
6. **通知客户端新主节点地址**
    
    故障转移完成后，Sentinel 通过 PUB/SUB 频道发布主节点变更事件，客户端订阅该事件后，自动更新连接的主节点地址，实现业务无感知切换。

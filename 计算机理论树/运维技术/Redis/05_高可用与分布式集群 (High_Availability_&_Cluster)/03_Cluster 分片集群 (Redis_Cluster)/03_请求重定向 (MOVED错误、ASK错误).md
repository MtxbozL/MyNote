Redis Cluster 中，客户端可向任意节点发送请求，节点会先计算 Key 对应的哈希槽，判断该槽是否由自身负责：

- 若槽由自身负责，直接执行命令，返回结果；
- 若槽不由自身负责，会根据槽的状态，向客户端返回**MOVED 错误**或**ASK 错误**，引导客户端重定向到正确的节点。

#### 一、MOVED 错误（永久重定向）

1. **触发场景**：当 Key 对应的哈希槽，由集群中其他固定节点负责，且槽处于稳定状态（无迁移）时，当前节点会向客户端返回 MOVED 错误。
2. **错误格式**：`(error) MOVED <slot> <target_ip>:<target_port>`，包含 Key 对应的槽编号、负责该槽的目标节点地址。
3. **客户端处理逻辑**：
    
    - 客户端收到 MOVED 错误后，会更新本地缓存的槽 - 节点映射表，将该槽对应的节点更新为目标节点；
    - 客户端自动向目标节点重新发送请求，获取执行结果；
    - 智能客户端（如 Redis-Jedis、Lettuce）会自动处理 MOVED 重定向，对上层业务无感知。
    
4. **核心特性**：MOVED 错误代表槽的归属发生了**永久性变更**，客户端需要更新本地缓存的映射表，后续该槽的请求都会直接发送到新的目标节点。

#### 二、ASK 错误（临时重定向）

1. **触发场景**：当 Key 对应的哈希槽，正处于**从源节点向目标节点迁移的过程中**，当前源节点已不存在该 Key，需要到目标节点查询时，源节点会向客户端返回 ASK 错误。
2. **错误格式**：`(error) ASK <slot> <target_ip>:<target_port>`，包含槽编号与迁移的目标节点地址。
3. **客户端处理逻辑**：
    
    - 客户端收到 ASK 错误后，**不会更新本地缓存的槽 - 节点映射表**，因为槽的迁移尚未完成，归属未发生永久变更；
    - 客户端向目标节点先发送`ASKING`命令，打开该连接的槽访问权限；
    - 客户端向目标节点重新发送原请求，获取执行结果；
    - 本次重定向仅对当前请求有效，后续该槽的请求仍会发送到原源节点，直到槽迁移完成，收到 MOVED 错误后再更新映射表。
    
4. **核心特性**：ASK 错误代表槽处于迁移过程中的**临时重定向**，仅对当前单次请求有效，客户端不可更新本地槽映射表，避免迁移过程中请求路由混乱。

#### 三、MOVED 与 ASK 错误的核心区别

|特性|MOVED 错误|ASK 错误|
|---|---|---|
|触发场景|槽归属永久变更，槽处于稳定状态|槽正处于迁移过程中，临时重定向|
|客户端映射表更新|必须更新本地槽 - 节点映射表|不可更新映射表，仅单次请求有效|
|前置命令|无需额外命令，直接重发请求|必须先发送 ASKING 命令，再重发请求|
|生命周期|永久生效，直到槽归属再次变更|仅对当前单次请求有效|

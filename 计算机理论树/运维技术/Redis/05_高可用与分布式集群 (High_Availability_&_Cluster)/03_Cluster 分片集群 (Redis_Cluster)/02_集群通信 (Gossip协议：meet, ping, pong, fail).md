### 02_集群通信 (Gossip 协议: meet, ping, pong, fail)

Redis Cluster 采用去中心化的 P2P 架构，无中心管理节点，集群节点之间通过**Gossip 协议（流言协议）** 进行通信，同步集群状态、节点信息、槽映射关系、故障信息等，实现集群元数据的最终一致性。

Gossip 协议的核心原理是：节点之间定期两两通信，交换各自掌握的集群信息，像流言传播一样，最终所有节点都会获得集群的完整状态信息，即使部分节点网络分区，恢复后也能快速同步最新状态。

#### 一、Gossip 协议核心消息类型

Redis Cluster 定义了 4 种核心 Gossip 消息类型，覆盖集群生命周期的所有通信场景：

1. **MEET 消息**
    
    - 核心作用：用于新节点加入集群，是集群扩容的第一步。
    - 执行流程：当执行`CLUSTER MEET <ip> <port>`命令时，发送方节点会向目标节点发送 MEET 消息，邀请目标节点加入当前集群；目标节点收到 MEET 消息后，会加入集群，开始与其他节点通过 Gossip 协议同步信息。
    
2. **PING 消息**
    
    - 核心作用：集群节点间最核心的心跳消息，用于检测节点存活状态，同时同步集群元数据。
    - 执行流程：每个节点每秒会随机选择 5 个其他节点，向最长时间未通信的节点发送 PING 消息，消息中包含发送方自身的状态、已知的其他节点状态、槽映射信息；节点每隔 100ms 会检查，若有节点超过半数秒未收到 PONG 响应，会再次发送 PING 消息，保证心跳检测的及时性。
    
3. **PONG 消息**
    
    - 核心作用：PING 消息与 MEET 消息的响应消息，用于确认节点存活，同时同步自身的集群信息。
    - 执行流程：节点收到 PING/MEET 消息后，会返回 PONG 消息，消息中包含自身的状态、元数据信息，发送方收到 PONG 消息后，确认目标节点存活，同时更新本地的集群元数据。
    
4. **FAIL 消息**
    
    - 核心作用：用于广播节点故障信息，实现集群故障状态的快速共识。
    - 执行流程：当某个主节点检测到另一个主节点进入 PFail（疑似下线）状态，且收到集群半数以上主节点的 PFail 报告后，会将该节点标记为 Fail（已下线），然后向集群所有节点广播 FAIL 消息；所有节点收到 FAIL 消息后，会立即将该节点标记为已下线，触发故障转移流程。
    

#### 二、Gossip 协议故障检测流程

Redis Cluster 通过 Gossip 协议实现去中心化的故障检测，无需中心节点，核心流程如下：

1. **PFail（疑似下线）标记**：节点 A 向节点 B 发送 PING 消息，若超过`cluster-node-timeout`配置的超时时间，未收到节点 B 的 PONG 响应，节点 A 会将节点 B 标记为**PFail（疑似下线）**，并将该状态通过 Gossip 消息同步给其他节点。
2. **Fail（已下线）共识**：节点 A 通过 Gossip 消息，收集其他主节点对节点 B 的状态报告，若**集群半数以上的主节点**都将节点 B 标记为 PFail，则节点 A 会将节点 B 标记为**Fail（已下线）**，并向集群广播 FAIL 消息。
3. **故障状态同步**：所有节点收到 FAIL 消息后，立即更新节点 B 的状态为已下线，若节点 B 是主节点，其对应的从节点会立即触发故障转移流程，选举新的主节点。

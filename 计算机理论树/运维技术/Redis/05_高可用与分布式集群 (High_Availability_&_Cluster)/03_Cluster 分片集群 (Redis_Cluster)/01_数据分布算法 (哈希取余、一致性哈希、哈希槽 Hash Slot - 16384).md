Redis Cluster 是 Redis 官方提供的**分布式分片集群解决方案**，核心解决了单节点内存容量上限、单节点写并发上限的问题，实现了数据的分布式分片存储、水平弹性扩展、自动化故障转移，是支撑 TB 级海量数据、百万级并发读写的企业级架构。

Redis Cluster 采用去中心化的 P2P 架构，无中心管理节点，所有节点平等，每个节点负责存储一部分数据，同时维护整个集群的元数据信息，最小集群规模为 3 主 3 从 6 个节点，每个主节点配备至少 1 个从节点，实现高可用。

### 01_数据分布算法 (哈希取余、一致性哈希、哈希槽 Hash Slot - 16384)

分布式集群的核心问题是**数据如何均匀分布到多个节点**，同时保证节点扩缩容时数据迁移量最小，Redis Cluster 最终采用**哈希槽（Hash Slot）** 方案，替代了传统的哈希取余与一致性哈希算法。

#### 一、传统数据分布算法的缺陷

1. **哈希取余算法**
    
    - 核心原理：`节点编号 = CRC16(key) % 节点总数`，根据 Key 的哈希值对节点总数取余，决定 Key 存储的节点。
    - 核心缺陷：节点扩缩容时，节点总数发生变化，几乎所有 Key 的映射关系都会改变，需要全量数据迁移，扩缩容成本极高，完全不适合生产环境弹性扩缩容场景。
    
2. **一致性哈希算法**
    
    - 核心原理：构建一个 0~2^32-1 的哈希环，每个节点通过哈希映射到环上的一个位置，Key 通过哈希映射到环上后，顺时针找到第一个节点，即为存储节点。
    - 核心优势：节点扩缩容时，仅影响该节点相邻的一小部分 Key，数据迁移量远小于哈希取余。
    - 核心缺陷：
        
        1. 节点数量较少时，数据分布极不均匀，容易出现数据倾斜；
        2. 节点扩缩容时，会导致环上的映射关系大范围变化，缓存雪崩风险高；
        3. 无法精准控制每个节点的数据量，运维难度高。
        
    

#### 二、Redis Cluster 哈希槽方案

Redis Cluster 将整个数据集划分为**16384 个固定的哈希槽（Hash Slot）**，编号 0~16383，每个 Key 唯一对应一个哈希槽，每个集群节点负责管理一部分哈希槽，是数据分布与路由的核心。

1. **Key 哈希槽计算规则**
    
    - 基础计算公式：`slot = CRC16(key) % 16384`，对 Key 执行 CRC16 算法得到哈希值，对 16384 取余，得到 Key 对应的槽编号；
    - Hash Tag 哈希标签：若 Key 中包含`{tag}`格式的标签，则仅对`{}`内的 tag 部分计算哈希槽，保证具有相同 tag 的 Key 映射到同一个哈希槽，实现跨 Key 的事务、批量操作。例如`order:{1001}:info`与`order:{1001}:item`，tag 均为 1001，会映射到同一个槽。
    
2. **哈希槽与节点的映射**
    
    16384 个哈希槽必须全部分配给集群的主节点，每个主节点负责一部分连续 / 非连续的哈希槽，从节点仅复制主节点的哈希槽数据，不负责槽的写入。例如 3 主节点集群，可分配：
    
    - 主节点 1：负责槽 0~5460
    - 主节点 2：负责槽 5461~10922
    - 主节点 3：负责槽 10923~16383
        
        集群所有节点都会维护完整的**槽 - 节点映射表**，可根据 Key 的槽编号，快速定位到对应的目标节点。
    
3. **为什么是 16384 个哈希槽**
    
    - 哈希槽的映射信息通过 Gossip 协议同步，16384 个槽的映射信息仅占用 2KB 空间（每个槽用 1bit 标记），节点间同步开销极小；
    - 16384 个槽足够支撑最大 1000 个节点的集群规模，槽分配均匀，不会出现数据倾斜；
    - 若槽数量过多，会导致 Gossip 消息过大，占用过多带宽；若槽数量过少，会导致数据分布不均匀，扩缩容时迁移粒度太大。
    

#### 三、哈希槽方案的核心优势

1. **数据分布均匀**：16384 个槽均匀分配到各个节点，保证数据分布均衡，无一致性哈希的倾斜问题；
2. **扩缩容成本极低**：节点扩缩容时，仅需迁移对应哈希槽的数据，无需全量迁移，扩缩容过程中集群可正常提供服务；
3. **路由精准高效**：客户端可提前缓存槽 - 节点映射表，根据 Key 直接计算槽编号，路由到对应节点，无需代理转发，性能损耗极低；
4. **支持批量操作与事务**：通过 Hash Tag 可将相关 Key 映射到同一个槽，实现跨 Key 的批量操作与事务，满足业务需求；
5. **元数据开销极小**：16384 个槽的映射信息仅需 2KB，节点间同步开销极低，集群规模扩展时性能稳定。

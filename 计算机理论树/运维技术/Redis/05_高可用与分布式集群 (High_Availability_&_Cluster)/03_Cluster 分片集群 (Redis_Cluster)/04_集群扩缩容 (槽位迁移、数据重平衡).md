Redis Cluster 的核心优势之一是支持**在线无损水平扩缩容**，扩缩容过程中集群可正常对外提供服务，业务无感知，核心是哈希槽的迁移与数据重平衡。

#### 一、集群扩容完整流程

集群扩容是新增节点到集群，将部分哈希槽从原有节点迁移到新节点，实现集群容量的水平扩展，完整流程如下：

1. **准备新节点**：启动新的 Redis 节点，配置开启集群模式（`cluster-enabled yes`），清空节点数据，确保与集群原有节点网络互通。
2. **新节点加入集群**：在集群任意原有节点上，执行`CLUSTER MEET <新节点IP> <新节点端口>`命令，原有节点向新节点发送 MEET 消息，新节点加入集群，与所有节点建立 Gossip 通信，此时新节点为无槽的主节点，不负责任何数据读写。
3. **分配哈希槽**：根据扩容规划，确定需要从原有主节点迁移到新节点的哈希槽范围，通过`CLUSTER SETSLOT <slot> IMPORTING <源节点ID>`命令，在新节点上标记槽为导入状态；同时在源节点上执行`CLUSTER SETSLOT <slot> MIGRATING <目标节点ID>`命令，标记槽为迁移状态。
4. **批量迁移槽内数据**：
    
    - 源节点执行`CLUSTER GETKEYSINSLOT <slot> <count>`命令，批量获取槽内的 Key，每次最多获取 count 个；
    - 源节点执行`MIGRATE <目标节点IP> <目标节点端口> "" 0 5000 KEYS <key1> <key2> ...`命令，原子性地将批量 Key 迁移到目标节点；
    - 重复上述步骤，直到该槽内的所有 Key 都迁移完成。
    
5. **更新槽归属**：槽内所有 Key 迁移完成后，在集群所有主节点上执行`CLUSTER SETSLOT <slot> NODE <新节点ID>`命令，将槽的归属永久更新为新节点，通过 Gossip 协议同步到整个集群，客户端收到 MOVED 错误后更新映射表。
6. **为新主节点添加从节点**：启动新的从节点，加入集群，执行`CLUSTER REPLICATE <新主节点ID>`命令，成为新主节点的从节点，实现数据备份与高可用。
7. **数据重平衡**：通过`redis-cli --cluster rebalance`命令，自动调整所有主节点的槽数量，保证每个主节点负责的槽数量均匀，数据分布均衡。

#### 二、集群缩容完整流程

集群缩容是下线集群中的节点，将该节点负责的哈希槽迁移到其他存活节点，实现集群容量的收缩，完整流程如下：

1. **确认待下线节点**：确定需要下线的节点，若为从节点，可直接下线；若为主节点，必须先将其负责的所有哈希槽迁移到其他主节点，再下线。
2. **迁移待下线主节点的哈希槽**：
    
    - 规划槽的迁移目标，将待下线主节点的所有槽，均匀分配给其他存活的主节点；
    - 按照扩容流程中的槽迁移步骤，逐个将槽从待下线节点迁移到目标主节点，更新槽的归属信息；
    - 直到待下线节点的所有槽都迁移完成，该节点成为无槽主节点。
    
3. **下线节点**：在集群任意节点执行`CLUSTER FORGET <待下线节点ID>`命令，集群所有节点会移除该节点的信息，停止 Gossip 通信；关闭待下线节点的 Redis 进程，完成缩容。
4. **数据重平衡**：执行集群重平衡命令，调整所有主节点的槽数量，保证数据分布均匀。

#### 三、生产环境扩缩容核心规范

1. **槽迁移必须分批执行**：每次仅迁移少量槽，单批次迁移的 Key 数量不超过 1000 个，避免大 Key 迁移阻塞主线程，影响集群性能；
2. **扩缩容操作必须避开业务高峰**：槽迁移会占用源节点与目标节点的 CPU、带宽资源，必须在低峰时段执行，避免影响业务；
3. **大 Key 拆分后再迁移**：迁移前必须扫描大 Key，拆分后再迁移，避免单个大 Key 迁移导致主线程长时间阻塞；
4. **迁移过程中禁止批量操作跨槽 Key**：槽迁移过程中，跨槽的 MSET、MGET 等批量命令会执行失败，业务层需暂停相关操作，或通过 Hash Tag 保证 Key 在同一槽；
5. **缩容前必须确认槽已全部迁移**：主节点必须将所有槽迁移完成后，才能下线，否则会导致槽数据丢失，集群不可用；
6. **扩缩容前后必须做数据一致性校验**：迁移完成后，校验槽内数据的完整性，确保无数据丢失、无数据不一致。

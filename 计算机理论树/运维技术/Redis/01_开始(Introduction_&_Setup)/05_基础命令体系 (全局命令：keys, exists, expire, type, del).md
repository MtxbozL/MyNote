全局命令是**不依赖特定数据类型、可对任意 Key 执行的通用操作命令**，是 Redis 命令体系的基础。所有全局命令均具备原子性，基于单线程执行模型保证并发安全，执行过程不会被其他命令中断。

本文档严格遵循 Redis 官方定义标注时间复杂度，生产环境必须重点关注高时间复杂度命令的使用风险。

---

#### 一、KEYS 模式匹配 Key 查询

1. **语法**：`KEYS pattern`
    
2. **核心功能**：查找当前数据库中所有匹配给定 glob 风格模式的 Key，返回匹配的 Key 列表。
    
3. **glob 模式匹配规则**
    
| 通配符  | 匹配规则                                   | 示例                                                                         |
| ---- | -------------------------------------- | -------------------------------------------------------------------------- |
| `*`  | 匹配任意数量的任意字符，包括 0 个字符                   | `KEYS *` 匹配所有 Key；`KEYS user:*` 匹配以`user:`开头的 Key                          |
| `?`  | 匹配单个任意字符                               | `KEYS user:?` 匹配`user:1`、`user:a`等单字符后缀的 Key                               |
| `[]` | 匹配括号内的任意单个字符，支持范围匹配                    | `KEYS user:[1-5]` 匹配`user:1`至`user:5`；`KEYS user:[ab]` 匹配`user:a`、`user:b` |
| `\x` | 转义特殊字符，用于匹配`*`、`?`、`[]`等本身作为 Key 内容的场景 | `KEYS user:*` 匹配包含`user:*`字面量的 Key                                         |
    
4. **时间复杂度**：O (N)，N 为当前数据库中 Key 的总数量
    
5. **返回值**：数组类型，匹配模式的所有 Key 列表；无匹配时返回空数组。
    
6. **生产环境铁律与替代方案**
    
    1. **绝对禁止在生产环境执行****`KEYS *`****或大范围匹配的 KEYS 命令**。Redis 单线程模型下，KEYS 会全量遍历数据库所有 Key，当实例 Key 数量达到十万 / 百万级时，会导致主线程长时间阻塞，所有客户端请求排队等待，引发服务雪崩。
        
    2. 生产环境唯一合规替代方案：`SCAN`增量迭代命令。`SCAN`通过游标分批返回结果，单次调用时间复杂度 O (1)，全量遍历总复杂度 O (N)，每次执行仅阻塞极短时间，不会影响服务正常运行。
        
    3. 安全管控：生产环境必须通过`rename-command`禁用 KEYS 命令，防止误操作。

#### 二、EXISTS Key 存在性检查

1. **语法**：`EXISTS key [key ...]`
    
2. **核心功能**：检查给定的一个或多个 Key 是否存在于当前数据库中。
    
3. **时间复杂度**：O (1) 单个 Key；O (N) 多个 Key，N 为传入的 Key 数量。
    
4. **返回值**：整数类型，返回**实际存在的 Key 的数量**；传入重复 Key 时会被多次计数，例如`EXISTS a a b`，若 a、b 均存在，返回 3。
    
5. **核心特性与使用规范**
    
    1. 原子性保证：多 Key 场景下，命令执行全程为原子操作，不会出现中间状态的存在性变化；
        
    2. 过期键处理：若 Key 设置了过期时间但尚未被惰性删除，EXISTS 仍会返回存在，直到 Key 被实际删除；
        
    3. 生产环境适配：O (1) 时间复杂度，无性能风险，可高频调用，是业务层判断 Key 存在性的核心合规命令；
        
    4. 限制：仅能检查当前`SELECT`切换的数据库中的 Key，无法跨库检查。

#### 三、EXPIRE 生存时间设置

1. **语法**：`EXPIRE key seconds [NX | XX | GT | LT]`
    
2. **核心功能**：为指定 Key 设置生存时间（Time To Live, TTL），单位为秒；生存时间耗尽时，Key 会被 Redis 自动删除。
    
3. **核心衍生命令（基础体系必学）** 为覆盖不同精度与场景，Redis 提供完整的过期时间管理命令体系，与 EXPIRE 为同一功能集合：
    
|命令|语法|核心功能|关键说明（参数 / 返回值 / 复杂度）|
|---|---|---|---|
|`PEXPIRE`|`PEXPIRE key milliseconds [NX \| XX \| GT \| LT]`|为指定 key 设置**相对生存时间**，单位为**毫秒**，提供更高时间精度|<ul><li>可选参数：<br><br>• `NX`：仅当 key 当前无过期时间时，才设置<br><br>• `XX`：仅当 key 已存在过期时间时，才更新<br><br>• `GT`：仅当新过期时间 > 当前剩余时间时，才更新<br><br>• `LT`：仅当新过期时间 < 当前剩余时间时，才更新</li><li>时间复杂度：O (1)</li></ul>|
|`EXPIREAT`|`EXPIREAT key unix-timestamp-seconds [NX \| XX \| GT \| LT]`|将 key 的过期时间设置为指定的 **Unix 绝对时间戳（秒级）**，key 在该时间点自动过期|<ul><li>可选参数同 `PEXPIRE`</li><li>时间复杂度：O (1)</li></ul>|
|`PEXPIREAT`|`PEXPIREAT key unix-timestamp-milliseconds [NX \| XX \| GT \| LT]`|将 key 的过期时间设置为指定的 **Unix 绝对时间戳（毫秒级）**，提供更高精度的过期时间点|<ul><li>可选参数同 `PEXPIRE`</li><li>时间复杂度：O (1)</li></ul>|
|`TTL`|`TTL key`|查询指定 key 的**剩余生存时间**，单位为**秒**|<ul><li>返回值规则：<br><br>• `≥0`：key 存在且有过期时间，返回剩余秒数<br><br>• `-1`：key 存在，但无过期时间（永久有效）<br><br>• `-2`：key 不存在</li><li>时间复杂度：O (1)</li></ul>|
|`PTTL`|`PTTL key`|查询指定 key 的**剩余生存时间**，单位为**毫秒**|<ul><li>返回值规则与 `TTL` 完全一致，仅单位不同</li><li>时间复杂度：O (1)</li></ul>|
|`PERSIST`|`PERSIST key`|移除指定 key 的过期时间，将其转为**永久有效**的 key|<ul><li>时间复杂度：O (1)</li><li>仅对有过期时间的 key 生效，对永久 key 执行返回 0</li></ul>|
    
4. **参数说明**
    
    1. `key`：目标 Key，必须为已存在的 Key，对不存在的 Key 执行 EXPIRE 返回 0；
        
    2. `seconds`：生存时间，正整数，单位秒；
        
    3. 可选修饰符（Redis 7.0+ 支持）：
        
        - `NX`：仅当 Key 无过期时间时，才设置过期时间；
            
        - `XX`：仅当 Key 已有过期时间时，才更新过期时间；
            
        - `GT`：仅当新过期时间大于当前过期时间时，才更新；
            
        - `LT`：仅当新过期时间小于当前过期时间时，才更新。
            
5. **时间复杂度**：O(1)
    
6. **返回值**：整数类型，1 = 过期时间设置成功，0 = 设置失败（Key 不存在、不满足修饰符条件等）。
    
7. **核心原理与生产规范**
    
    1. 过期时间存储：Redis 将 Key 的过期时间存储为**绝对 Unix 毫秒级时间戳**，即使实例重启，只要持久化文件保留了过期时间，重启后仍能正确计算剩余 TTL；
        
    2. 过期删除联动：EXPIRE 设置的过期时间，配合 Redis 的**惰性删除 + 定期删除**策略执行，具体原理在后续内存管理章节详解；
        
    3. 主从复制规则：主从架构中，从节点禁止主动删除过期 Key，仅能同步主节点的删除操作，避免数据不一致；过期 Key 的删除动作由主节点执行后同步到从节点；
        
    4. 常见误区纠正：
        
        - 误区：对 Key 执行 SET 等覆盖命令后，过期时间会保留。纠正：SET、GETSET 等覆盖整个 Value 的命令，会移除 Key 原有的过期时间，转为永久有效，必须重新设置 EXPIRE；
            
        - 误区：TTL 返回 - 1 表示 Key 不存在。纠正：TTL 返回 - 2 表示 Key 不存在，-1 表示 Key 存在且为永久有效；
            
    5. 生产环境最佳实践：缓存场景必须为所有 Key 设置合理的过期时间，避免内存无限增长；分布式锁场景必须配合 EXPIRE 设置锁超时时间，防止死锁；禁止设置小于 1 秒的过短过期时间，避免频繁过期删除引发性能波动。

#### 四、TYPE 数据类型查询

1. **语法**：`TYPE key`
    
2. **核心功能**：返回指定 Key 对应的 Value 的**逻辑数据类型**。
    
3. **时间复杂度**：O(1)
    
4. **返回值**：字符串类型，返回对应的数据类型标识，可选值如下：
    
| 返回值      | 对应数据类型  |
| -------- | ------- |
| `string` | 字符串类型   |
| `list`   | 列表类型    |
| `hash`   | 哈希类型    |
| `set`    | 集合类型    |
| `zset`   | 有序集合类型  |
| `stream` | 流数据类型   |
| `none`   | Key 不存在 |
    
5. **核心特性与使用规范**
    
    1. 与`OBJECT ENCODING`的核心区别：TYPE 返回的是**用户层面的逻辑数据类型**，`OBJECT ENCODING`返回的是**底层物理编码结构**（如 int、embstr、raw 等），二者不可混淆，底层编码在后续核心数据类型章节详解；
        
    2. 无性能风险，O (1) 时间复杂度，可高频调用，常用于业务层数据类型校验，避免对错误类型的 Key 执行命令引发报错；
        
    3. 对已设置过期时间但尚未删除的 Key，TYPE 仍能正确返回其数据类型。

#### 五、DEL Key 删除

1. **语法**：`DEL key [key ...]`
    
2. **核心功能**：删除指定的一个或多个 Key 及对应的 Value，无论 Value 为何种数据类型，均可被删除。
    
3. **时间复杂度**：O (1) 单个小 Key；O (N) 多个 Key / 大 Key，N 为删除的 Key 数量，以及大 Key 对应的元素总数量。
    
4. **返回值**：整数类型，返回**实际成功删除的 Key 的数量**，不存在的 Key 不计入。
    
5. **核心特性与生产规范**
    
    1. 原子性保证：多 Key 删除为原子操作，执行过程不会被其他命令中断；
        
    2. 同步阻塞机制：DEL 为**同步阻塞删除**，主线程会执行完整的内存释放流程，直到删除完成才返回结果；若删除包含百万级元素的大 Key，会导致主线程长时间阻塞，引发服务不可用；
        
    3. 生产环境替代方案：
        
        - 大 Key 删除必须使用`UNLINK`命令替代 DEL。UNLINK 为异步非阻塞删除，仅在主线程中移除 Key 的元数据，内存释放操作交给后台 bio 线程异步执行，不会阻塞主线程；
            
        - 批量删除大量小 Key，需使用`SCAN`增量遍历 + 分批 DEL，避免单次 DEL 传入大量 Key 导致主线程阻塞；
            
    4. 安全管控：生产环境禁止直接执行`DEL *`全量删除，必须通过`rename-command`禁用`FLUSHDB`/`FLUSHALL`命令，防止误操作；
        
    5. 限制：仅能删除当前数据库中的 Key，无法跨库删除。

#### 六、RENAME / RENAMENX Key重命名

1. **语法**： `RENAME key newkey` `RENAMENX key newkey`
    
2. **核心功能**： RENAME：将指定Key重命名为newkey，若newkey已存在，会直接覆盖其对应的Value（高危操作）； RENAMENX：仅当newkey不存在时，才将key重命名为newkey，若newkey已存在则不执行任何操作（安全操作）。
    
3. **时间复杂度**：O(1)，无论Key对应的Value大小，重命名仅操作Key的元数据。
    
4. **返回值**： RENAME：始终返回`OK`（即使覆盖已有newkey）； RENAMENX：整数类型，1=重命名成功，0=newkey已存在，重命名失败。
    
5. **核心特性与生产规范**： 覆盖风险：RENAME会强制覆盖已存在的newkey，且操作不可逆，生产环境**禁止使用RENAME**，优先使用RENAMENX避免误覆盖；
    
6. Key不存在处理：对不存在的key执行RENAME/RENAMENX，均返回错误（RENAME返回(error) ERR no such key，RENAMENX返回0）；
    
7. 原子性：两个命令均为原子操作，执行过程中不会被其他命令中断，避免重命名过程中Key状态异常；
    
8. 跨库限制：仅能对当前数据库（SELECT选中的库）中的Key进行重命名，无法跨数据库重命名；若需跨库迁移Key，需使用MOVE命令。

#### 七、DBSIZE 数据库Key数量统计

1. **语法**：`DBSIZE`
    
2. **核心功能**：统计当前数据库中所有未被删除的Key总数（包括设置了过期时间但尚未过期的Key）。
    
3. **时间复杂度**：O(1)，Redis内部维护了当前数据库的Key计数器，无需遍历全量Key，直接返回计数器值。
    
4. **返回值**：整数类型，当前数据库的Key总数。
    
5. **核心特性与生产规范**： 计数范围：仅统计当前数据库的Key，切换数据库（SELECT）后，DBSIZE返回对应库的Key数量；
    
6. 过期键计数：尚未被惰性删除/定期删除的过期Key，仍会被计入统计，直到被实际删除；
    
7. 生产适配：O(1)时间复杂度，无性能风险，可高频调用，常用于实例健康检查、内存管理排查（结合maxmemory判断Key增长趋势）；
    
8. 与KEYS的区别：DBSIZE仅返回Key总数，无需遍历全量Key，无阻塞风险；KEYS返回具体Key列表，有阻塞风险，二者不可混淆。

#### 八、FLUSHDB / FLUSHALL 数据库清空

1. **语法**： `FLUSHDB [ASYNC]` `FLUSHALL [ASYNC]`
    
2. **核心功能**： FLUSHDB：清空当前数据库中的所有Key（无论数据类型、是否过期），仅影响当前SELECT选中的库； FLUSHALL：清空Redis实例中所有数据库的所有Key，影响所有逻辑库（包括0~databases-1），是高危操作； 可选参数`ASYNC`（Redis 4.0+支持）：开启异步清空，主线程仅发起清空请求，实际删除操作由后台bio线程执行，不阻塞主线程。
    
3. **时间复杂度**： 同步模式（无ASYNC）：O(N)，N为当前库（FLUSHDB）或所有库（FLUSHALL）的Key总数，会阻塞主线程； 异步模式（有ASYNC）：O(1)，主线程仅执行元数据清理，实际删除由后台线程完成，无阻塞风险。
    
4. **返回值**：始终返回`OK`，无论是否开启异步模式。
    
5. **核心特性与生产规范**： 高危管控：生产环境必须通过`rename-command`禁用FLUSHDB、FLUSHALL命令，防止误操作导致全量数据丢失；
    
6. 异步模式使用：高并发生产场景，若确需清空数据，必须添加`ASYNC`参数，避免同步清空阻塞主线程引发服务雪崩；
    
7. 主从同步：FLUSHDB/FLUSHALL的操作会同步到所有从节点，主节点执行清空后，从节点会同步清空对应数据，避免主从数据不一致；
    
8. 数据恢复：清空操作不可逆，执行前必须确认数据已备份（如导出RDB文件），避免无法恢复。

#### 九、SELECT 数据库切换

1. **语法**：`SELECT dbid`
    
2. **核心功能**：切换当前客户端连接的逻辑数据库，dbid取值范围为0~databases-1（databases由redis.conf配置），默认连接0号库。
    
3. **时间复杂度**：O(1)，仅切换客户端的数据库指针，无其他耗时操作。
    
4. **返回值**：切换成功返回`OK`；dbid超出范围（如大于等于databases），返回错误信息。
    
5. **核心特性与生产规范**： 集群限制：Redis Cluster集群模式下，仅支持0号逻辑库，SELECT命令无效，执行后会返回错误；
    
6. 权限隔离：逻辑数据库仅为Key的逻辑分组，无独立的权限控制，不同库的Key可重名，不建议用于多业务数据拆分（多业务建议使用不同Redis实例或ACL权限隔离）；
    
7. 客户端隔离：SELECT切换仅对当前客户端有效，其他客户端的数据库连接不受影响；客户端断开连接后，重新连接会默认回到0号库；
    
8. 生产适配：单实例场景可用于开发/测试环境的Key分组，生产环境单实例建议仅使用0号库，避免多库切换导致Key管理混乱。

#### 十、MOVE Key跨库迁移

1. **语法**：`MOVE key dbid`
    
2. **核心功能**：将当前数据库（SELECT选中的库）中的指定Key，迁移到同一Redis实例的dbid对应的目标数据库中。
    
3. **时间复杂度**：O(1) 单个小Key；O(N) 大Key，N为大Key的元素总数量（迁移需复制Value并删除原Key）。
    
4. **返回值**：整数类型，1=迁移成功，0=迁移失败（Key不存在、目标库已存在同名Key、dbid超出范围）。
    
5. **核心特性与生产规范**： 跨库限制：仅能在同一Redis实例的不同逻辑库之间迁移，无法跨Redis实例迁移（跨实例迁移需使用MIGRATE命令，后续进阶命令章节详解）；
    
6. 同名Key处理：若目标库已存在与key同名的Key，MOVE会直接返回0，不执行迁移，避免覆盖目标Key；
    
7. 过期键迁移：Key的过期时间会随Key一起迁移，迁移后过期时间保持不变，过期删除策略正常生效；
    
8. 大Key风险：迁移大Key时，会阻塞主线程（同步操作），生产环境迁移大Key需避开高并发时段，或使用异步迁移方案。

#### 十一、OBJECT 键对象信息查询

1. **语法**：`OBJECT subcommand key`
    
2. **核心功能**：查询指定Key对应的底层对象信息，包括物理编码、内存占用、过期时间等，用于底层原理排查与性能优化（补充TYPE命令的底层细节）。
    
3. **核心子命令（基础必学）**： 子命令核心功能返回值说明encoding返回Key对应的Value的底层物理编码（与TYPE的逻辑类型区分）字符串（如int、embstr、raw、ziplist等）refcount返回Key对应的对象的引用计数（Redis内存回收基于引用计数）整数（通常为1，共享对象可能大于1）idletime返回Key自上次被访问以来的空闲时间（单位秒）整数（空闲时间，0表示刚被访问）freq返回Key的访问频率（仅开启LFU淘汰策略时有效）整数（频率值，越大访问越频繁）
    
4. **时间复杂度**：O(1)，所有子命令均仅查询对象元数据，无耗时操作。
    
5. **核心特性与生产规范**： 与TYPE的区别：TYPE返回用户层面的逻辑数据类型（如string、list），OBJECT encoding返回底层物理编码（如string类型可能对应int、embstr、raw三种编码），用于底层性能优化；
    
6. 生产适配：无性能风险，可用于大Key排查（通过encoding判断是否为高效编码）、内存碎片分析、热点Key辅助定位（结合freq子命令）；
    
7. Key不存在处理：对不存在的Key执行OBJECT命令，返回错误信息。
    

---

#### 全局命令核心总结

全局命令是Redis操作的基础，核心特点为不依赖特定数据类型、原子性执行、时间复杂度以O(1)为主（高危命令除外）。生产环境使用需重点关注：

- 禁用高危命令：KEYS、FLUSHDB、FLUSHALL、MONITOR等，通过rename-command管控，避免误操作；
    
- 规避阻塞风险：大Key操作（DEL、MOVE）优先使用异步/增量方案，避免主线程阻塞；
    
- 关注时间复杂度：O(N)级命令（如KEYS、同步FLUSHDB）禁止在高并发生产环境执行；
    
- 规范权限与隔离：单实例多业务场景优先使用ACL权限隔离，而非逻辑数据库（SELECT），避免Key管理混乱。

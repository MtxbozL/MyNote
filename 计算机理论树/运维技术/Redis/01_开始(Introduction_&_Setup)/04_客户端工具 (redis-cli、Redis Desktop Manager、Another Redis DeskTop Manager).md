Redis 客户端分为**命令行客户端**与**可视化图形客户端**两大类，核心职能是与 Redis 服务端建立 TCP 连接、发送命令、接收响应、执行数据管理与运维操作。其中`redis-cli`为 Redis 官方原生内置的命令行客户端，随服务端安装包同步发布，是生产环境运维、调试、故障排查的核心工具；可视化客户端多用于开发环境数据查看、调试与批量操作，降低使用门槛。

---

#### 一、redis-cli 官方命令行客户端

`redis-cli`是 Redis 生态中最核心、最高效、功能最全面的客户端，无第三方依赖，适配所有 Redis 版本与部署架构，是生产环境唯一合规的运维操作客户端。

##### 1. 核心启动与连接参数

基础执行语法：

```c
redis-cli [OPTIONS] [cmd [arg [arg ...]]]
```

核心连接参数如下：

| 参数                 | 功能说明                                                       | 生产环境使用规范                                        |
| ------------------ | ---------------------------------------------------------- | ----------------------------------------------- |
| `-h <host>`        | 指定 Redis 服务端 IP 地址，默认`127.0.0.1`                           | 生产环境必须指定内网业务 IP，禁止使用默认回环地址                      |
| `-p <port>`        | 指定服务端监听端口，默认`6379`                                         | 匹配服务端自定义端口，多实例部署时必须明确指定                         |
| `-a <password>`    | 直接传入认证密码，对应`requirepass`配置                                 | 禁止在命令行明文传入密码，防止泄露至 shell 历史记录，优先使用交互式`AUTH`命令认证 |
| `-u <uri>`         | 通过标准 URI 格式指定连接信息，格式为`redis://[:password@]host:port/dbnum` | 适合脚本化批量运维场景，密码需做脱敏处理                            |
| `-n <dbid>`        | 指定默认连接的逻辑数据库编号，等效于启动后执行`SELECT <dbid>`                     | 仅单实例场景可用，Redis Cluster 集群模式仅支持 0 号库             |
| `-s <socket-path>` | 指定 Unix 套接字文件路径，用于本地无 TCP 连接                               | 本地进程通信场景优先使用，无 TCP 协议开销，延迟显著低于 TCP 回环连接         |
| `--tls`            | 开启 TLS 加密连接                                                | 跨网传输、公网访问场景必须开启，保障传输层安全                         |

##### 2. 核心运行模式与功能

1. **交互式 REPL 模式**
    
    默认启动模式，进入读取 - 求值 - 输出循环，支持命令 Tab 补全、历史记录回溯、语法高亮，是开发调试、日常运维的核心模式。
2. **单命令执行模式**
    
    启动时直接附加完整命令与参数，执行后返回结果并自动退出，适合 shell 脚本化、自动化运维场景。示例：
    
```bash
    # 远程查询实例内存指标
    redis-cli -h 10.0.0.10 -p 6379 INFO memory
    # 批量设置过期时间
    redis-cli -h 10.0.0.10 -p 6379 EXPIRE user:1001 86400
```
    
3. **管道批量执行模式**
    
    通过标准输入批量传入命令，配合`--pipe`参数开启流水线（Pipeline）模式，合并多次网络 RTT 为单次，批量执行效率提升 10 倍以上，是大规模数据导入、批量命令执行的唯一合规方案。示例：
    
    ```bash
    # 批量执行commands.txt中的命令
    cat commands.txt | redis-cli -h 10.0.0.10 -p 6379 --pipe
    ```
    
4. **运维诊断专用模式**
    
    内置多组生产环境核心运维诊断能力，无需额外工具即可完成实例状态排查：

| 专用参数                | 核心功能                                     | 适用场景                         |
| ------------------- | ---------------------------------------- | ---------------------------- |
| `--stat`            | 实时输出实例核心运行指标，每秒刷新，包括连接数、内存占用、QPS、输入输出流量  | 生产环境实时性能监控、流量波动排查            |
| `--bigkeys`         | 基于 SCAN 增量扫描实例，按数据类型统计最大 Key、平均元素大小、内存占用 | 生产环境大 Key 排查、性能瓶颈定位          |
| `--hotkeys`         | 扫描实例热点 Key，统计访问频次排序（需实例开启 LFU 内存淘汰策略）    | 大促场景热点 Key 定位、缓存倾斜治理         |
| `--scan`            | 基于 SCAN 增量迭代全量 Key，支持模式匹配                | 生产环境全量 Key 遍历，完全替代高危`KEYS`命令 |
| `--latency`         | 持续测试客户端到服务端的网络延迟，输出最小 / 最大 / 平均延迟        | 网络故障排查、跨机房延迟评估               |
| `--rdb <file-path>` | 远程导出实例 RDB 快照文件到本地                       | 远程实例备份、数据迁移                  |


##### 3. 生产环境使用铁律

- 禁止在生产环境执行`KEYS`、`MONITOR`、`FLUSHALL`、`FLUSHDB`等高危命令，生产环境必须通过`rename-command`禁用上述命令；
- 禁止高并发场景下执行长时间阻塞的全量遍历命令（如`HGETALL`、`SMEMBERS`全量执行），避免主线程阻塞引发服务雪崩；
- 批量操作必须使用`--pipe`流水线模式或`SCAN`增量迭代，禁止单次命令传入海量 Key 导致主线程阻塞；
- 密码认证必须使用交互式`AUTH`命令，禁止通过`-a`参数明文传入密码。

    - 基础可视化监控面板，实时展示实例核心运行指标。
    
2. **适用场景与局限性**
    
    - 适用场景：开发环境调试、中小规模实例运维、数据可视化查看；商业版提供企业级权限管控与集群运维能力；

### 05_基础命令体系 (全局命令)

全局命令是**不依赖特定数据类型、可对任意 Key 执行的通用操作命令**，是 Redis 命令体系的基础。所有全局命令均具备原子性，基于单线程执行模型保证并发安全，执行过程不会被其他命令中断。

本文档严格遵循 Redis 官方定义标注时间复杂度，生产环境必须重点关注高时间复杂度命令的使用风险。

---

#### 一、KEYS 模式匹配 Key 查询

1. **语法**：`KEYS pattern`
2. **核心功能**：查找当前数据库中所有匹配给定 glob 风格模式的 Key，返回匹配的 Key 列表。
3. **glob 模式匹配规则**

| 通配符  | 匹配规则                                   | 示例                                                                         |
| ---- | -------------------------------------- | -------------------------------------------------------------------------- |
| `*`  | 匹配任意数量的任意字符，包括 0 个字符                   | `KEYS *` 匹配所有 Key；`KEYS user:*` 匹配以`user:`开头的 Key                          |
| `?`  | 匹配单个任意字符                               | `KEYS user:?` 匹配`user:1`、`user:a`等单字符后缀的 Key                               |
| `[]` | 匹配括号内的任意单个字符，支持范围匹配                    | `KEYS user:[1-5]` 匹配`user:1`至`user:5`；`KEYS user:[ab]` 匹配`user:a`、`user:b` |
| `\x` | 转义特殊字符，用于匹配`*`、`?`、`[]`等本身作为 Key 内容的场景 | `KEYS user\:*` 匹配包含`user:*`字面量的 Key                                        |
    
4. **时间复杂度**：O (N)，N 为当前数据库中 Key 的总数量
5. **返回值**：数组类型，匹配模式的所有 Key 列表；无匹配时返回空数组。
6. **生产环境铁律与替代方案**
    
    - **绝对禁止在生产环境执行`KEYS *`或大范围匹配的 KEYS 命令**。Redis 单线程模型下，KEYS 会全量遍历数据库所有 Key，当实例 Key 数量达到十万 / 百万级时，会导致主线程长时间阻塞，所有客户端请求排队等待，引发服务雪崩。
    - 生产环境唯一合规替代方案：`SCAN`增量迭代命令。`SCAN`通过游标分批返回结果，单次调用时间复杂度 O (1)，全量遍历总复杂度 O (N)，每次执行仅阻塞极短时间，不会影响服务正常运行。
    - 安全管控：生产环境必须通过`rename-command`禁用 KEYS 命令，防止误操作。
    

---

#### 二、EXISTS Key 存在性检查

1. **语法**：`EXISTS key [key ...]`
2. **核心功能**：检查给定的一个或多个 Key 是否存在于当前数据库中。
3. **时间复杂度**：O (1) 单个 Key；O (N) 多个 Key，N 为传入的 Key 数量。
4. **返回值**：整数类型，返回**实际存在的 Key 的数量**；传入重复 Key 时会被多次计数，例如`EXISTS a a b`，若 a、b 均存在，返回 3。
5. **核心特性与使用规范**
    
    - 原子性保证：多 Key 场景下，命令执行全程为原子操作，不会出现中间状态的存在性变化；
    - 过期键处理：若 Key 设置了过期时间但尚未被惰性删除，EXISTS 仍会返回存在，直到 Key 被实际删除；
    - 生产环境适配：O (1) 时间复杂度，无性能风险，可高频调用，是业务层判断 Key 存在性的核心合规命令；
    - 限制：仅能检查当前`SELECT`切换的数据库中的 Key，无法跨库检查。
    

---

#### 三、EXPIRE 生存时间设置

1. **语法**：`EXPIRE key seconds [NX | XX | GT | LT]`
2. **核心功能**：为指定 Key 设置生存时间（Time To Live, TTL），单位为秒；生存时间耗尽时，Key 会被 Redis 自动删除。
3. **核心衍生命令（基础体系必学）**
    
    为覆盖不同精度与场景，Redis 提供完整的过期时间管理命令体系，与 EXPIRE 为同一功能集合：



|命令|语法|核心功能||||
|---|---|---|---|---|---|
|PEXPIRE|`PEXPIRE key milliseconds [NX|XX|GT|LT]`|设置生存时间，单位为毫秒，提供更高时间精度|
|EXPIREAT|`EXPIREAT key unix-timestamp-seconds`|设置 Key 的过期时间为指定的 Unix 时间戳，单位秒||||
|PEXPIREAT|`PEXPIREAT key unix-timestamp-milliseconds`|设置过期时间为指定的 Unix 时间戳，单位毫秒||||
|TTL|`TTL key`|查询 Key 剩余生存时间，单位秒；返回 - 1=Key 存在无过期时间，-2=Key 不存在，≥0 = 剩余秒数||||
|PTTL|`PTTL key`|查询 Key 剩余生存时间，单位毫秒，返回值规则同 TTL||||
|PERSIST|`PERSIST key`|移除 Key 的过期时间，将 Key 转为永久有效，时间复杂度 O (1)||||
    
4. **参数说明**
    
    - `key`：目标 Key，必须为已存在的 Key，对不存在的 Key 执行 EXPIRE 返回 0；
    - `seconds`：生存时间，正整数，单位秒；
    - 可选修饰符（Redis 7.0+ 支持）：
        
        - `NX`：仅当 Key 无过期时间时，才设置过期时间；
        - `XX`：仅当 Key 已有过期时间时，才更新过期时间；
        - `GT`：仅当新过期时间大于当前过期时间时，才更新；
        - `LT`：仅当新过期时间小于当前过期时间时，才更新。
        
    
5. **时间复杂度**：O(1)
6. **返回值**：整数类型，1 = 过期时间设置成功，0 = 设置失败（Key 不存在、不满足修饰符条件等）。
7. **核心原理与生产规范**
    
    - 过期时间存储：Redis 将 Key 的过期时间存储为**绝对 Unix 毫秒级时间戳**，即使实例重启，只要持久化文件保留了过期时间，重启后仍能正确计算剩余 TTL；
    - 过期删除联动：EXPIRE 设置的过期时间，配合 Redis 的**惰性删除 + 定期删除**策略执行，具体原理在后续内存管理章节详解；
    - 主从复制规则：主从架构中，从节点禁止主动删除过期 Key，仅能同步主节点的删除操作，避免数据不一致；过期 Key 的删除动作由主节点执行后同步到从节点；
    - 常见误区纠正：
        
        1. 误区：对 Key 执行 SET 等覆盖命令后，过期时间会保留。纠正：SET、GETSET 等覆盖整个 Value 的命令，会移除 Key 原有的过期时间，转为永久有效，必须重新设置 EXPIRE；
        2. 误区：TTL 返回 - 1 表示 Key 不存在。纠正：TTL 返回 - 2 表示 Key 不存在，-1 表示 Key 存在且为永久有效；
        
    - 生产环境最佳实践：缓存场景必须为所有 Key 设置合理的过期时间，避免内存无限增长；分布式锁场景必须配合 EXPIRE 设置锁超时时间，防止死锁；禁止设置小于 1 秒的过短过期时间，避免频繁过期删除引发性能波动。

#### 四、TYPE 数据类型查询

1. **语法**：`TYPE key`
2. **核心功能**：返回指定 Key 对应的 Value 的**逻辑数据类型**。
3. **时间复杂度**：O(1)
4. **返回值**：字符串类型，返回对应的数据类型标识，可选值如下：
    
| 返回值      | 对应数据类型  |
| -------- | ------- |
| `string` | 字符串类型   |
| `list`   | 列表类型    |
| `hash`   | 哈希类型    |
| `set`    | 集合类型    |
| `zset`   | 有序集合类型  |
| `stream` | 流数据类型   |
| `none`   | Key 不存在 |
    
5. **核心特性与使用规范**
    
    - 与`OBJECT ENCODING`的核心区别：TYPE 返回的是**用户层面的逻辑数据类型**，`OBJECT ENCODING`返回的是**底层物理编码结构**（如 int、embstr、raw 等），二者不可混淆，底层编码在后续核心数据类型章节详解；
    - 无性能风险，O (1) 时间复杂度，可高频调用，常用于业务层数据类型校验，避免对错误类型的 Key 执行命令引发报错；
    - 对已设置过期时间但尚未删除的 Key，TYPE 仍能正确返回其数据类型。

#### 五、DEL Key 删除

1. **语法**：`DEL key [key ...]`
2. **核心功能**：删除指定的一个或多个 Key 及对应的 Value，无论 Value 为何种数据类型，均可被删除。
3. **时间复杂度**：O (1) 单个小 Key；O (N) 多个 Key / 大 Key，N 为删除的 Key 数量，以及大 Key 对应的元素总数量。
4. **返回值**：整数类型，返回**实际成功删除的 Key 的数量**，不存在的 Key 不计入。
5. **核心特性与生产规范**
    
    - 原子性保证：多 Key 删除为原子操作，执行过程不会被其他命令中断；
    - 同步阻塞机制：DEL 为**同步阻塞删除**，主线程会执行完整的内存释放流程，直到删除完成才返回结果；若删除包含百万级元素的大 Key，会导致主线程长时间阻塞，引发服务不可用；
    - 生产环境替代方案：
        
        1. 大 Key 删除必须使用`UNLINK`命令替代 DEL。UNLINK 为异步非阻塞删除，仅在主线程中移除 Key 的元数据，内存释放操作交给后台 bio 线程异步执行，不会阻塞主线程；
        2. 批量删除大量小 Key，需使用`SCAN`增量遍历 + 分批 DEL，避免单次 DEL 传入大量 Key 导致主线程阻塞；
        
    - 安全管控：生产环境禁止直接执行`DEL *`全量删除，必须通过`rename-command`禁用`FLUSHDB`/`FLUSHALL`命令，防止误操作；
    - 限制：仅能删除当前数据库中的 Key，无法跨库删除。

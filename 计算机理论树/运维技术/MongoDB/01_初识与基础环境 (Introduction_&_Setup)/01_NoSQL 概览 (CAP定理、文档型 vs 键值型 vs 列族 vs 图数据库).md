## 1.1 NoSQL 概览

### 1.1.1 NoSQL 的核心定位与诞生背景

NoSQL（Not Only SQL）是为应对传统关系型数据库（RDBMS）在海量高并发、非结构化 / 半结构化数据、分布式水平扩展场景下的架构局限性，而诞生的非关系型数据库体系。其核心设计目标是实现高可用、高可扩展、灵活的数据模型，适配 Web2.0 及之后的互联网业务形态，与 RDBMS 形成互补而非完全替代的关系。

### 1.1.2 CAP 定理

CAP 定理是分布式数据库系统的核心设计准则，由 Eric Brewer 于 1998 年提出，2002 年完成理论证明，其核心结论为：**分布式系统中，一致性、可用性、分区容错性三大属性无法同时满足，最多仅能同时实现其中两项**。

三大核心属性的严格定义如下：

- **一致性（Consistency）**：所有节点在同一时间点的数据视图完全一致，读操作始终能返回最新完成的写操作结果。
- **可用性（Availability）**：系统中每一个非故障节点，总能在有限时间内对用户请求返回合法的非错误响应，无超时或服务拒绝。
- **分区容错性（Partition Tolerance）**：当网络出现分区故障（节点间通信中断）时，系统仍能持续正常运行，不发生整体崩溃。

在实际分布式场景中，网络分区故障为不可避免的客观事件，因此分区容错性是分布式系统的必然前提，实际工程选型仅存在两类架构：

- **CP 系统**：优先保障一致性与分区容错性，牺牲部分可用性，适用于金融交易、账务系统等强数据一致性场景。
- **AP 系统**：优先保障可用性与分区容错性，牺牲强一致性实现最终一致性，适用于社交内容、高并发读、缓存等场景。

### 1.1.3 NoSQL 四大主流数据模型对比

|数据库类型|核心数据模型|核心特性|典型产品|核心适用场景|
|---|---|---|---|---|
|键值型数据库（Key-Value Store）|以唯一键（Key）为索引，值（Value）为数据库无感知的二进制 / 字符串数据|读写性能极致、模型极简、水平扩展能力极强，不支持复杂条件查询|Redis、RocksDB、DynamoDB|分布式缓存、会话管理、高频读写的简单 KV 数据存储|
|列族型数据库（Column-Family Store）|以列族为核心组织单元，列族包含多个列，数据按行键与列族物理集中存储|支持海量数据高并发写入、按列高效查询，水平扩展能力极强，适配宽表与时序场景|HBase、Cassandra、BigTable|物联网时序数据、用户行为日志、大数据离线分析|
|文档型数据库（Document Store）|以结构化文档为最小存储单元，采用类 JSON 格式，支持嵌套结构、数组与复杂数据类型，同集合内文档支持异构结构|平衡了模型灵活性与查询能力，Schema-less 动态模式，支持复杂查询、索引、聚合运算，适配面向对象开发模式|MongoDB、CouchDB|电商商品管理、内容管理系统、企业级通用业务存储、社交平台|
|图数据库（Graph Database）|以节点（实体）、边（实体间关系）、属性为核心存储单元，天然存储关联数据|极致优化多跳关联查询，支持复杂图算法，彻底解决 RDBMS 多表 Join 的性能瓶颈|Neo4j、NebulaGraph、JanusGraph|社交关系网络、金融风控、知识图谱、个性化推荐系统|


# 第六章 并发、事务与存储引擎

本章是 MongoDB 生产环境数据一致性、并发安全、持久化保障的核心基础，覆盖底层存储引擎实现机制、并发控制模型、读写一致性配置、ACID 事务能力四大核心模块。其解决了分布式场景下 “数据会不会丢失、并发写会不会冲突、多操作能否保证原子性” 的核心问题，是从开发进阶到生产运维、架构设计的关键分水岭。

## 6.1 WiredTiger 存储引擎

MongoDB 采用插件化存储引擎架构，历史版本曾使用 MMAPv1 引擎（已在 MongoDB 4.2 版本正式废弃），当前**WiredTiger 为默认存储引擎**，专为高并发、高吞吐量、低延迟的现代业务设计，是 MongoDB 企业级能力的底层基石。

### 6.1.1 引擎核心定位与架构设计

WiredTiger 是一款开源的高性能、事务型 NoSQL 存储引擎，由 MongoDB Inc. 收购后深度定制优化，于 MongoDB 3.2 版本成为默认引擎。其核心设计围绕三大目标：

1. 极致的并发性能：通过细粒度锁与 MVCC 实现读写无阻塞，最大化多核 CPU 的并行处理能力；
2. 可靠的数据持久化：通过预写日志 + 检查点机制，保障数据不丢失，实现快速故障恢复；
3. 高效的资源利用：原生支持数据压缩、多级缓存架构，降低磁盘 IO 与内存占用。

其核心架构分为四层，自下而上为：**磁盘存储层**（数据文件、WAL 日志文件）→ **操作系统页缓存层** → **WiredTiger 内部缓存层** → **MongoDB 服务接口层**，分层设计实现了性能与可靠性的平衡。

### 6.1.2 核心特性与实现原理

#### 1. 文档级锁与并发控制模型

WiredTiger 采用**文档级排他锁**配合**意向锁**的并发控制机制，彻底解决了旧版 MMAPv1 引擎集合级锁的并发瓶颈。

- **锁粒度规则**：写操作的锁粒度为单文档级别，多个并发写操作只要不修改同一个文档，就不会产生锁冲突，可并行执行，大幅提升高并发写入场景的吞吐量。
- **意向锁机制**：在集合级别设置意向共享锁（IS）与意向排他锁（IX），用于快速判断集合内是否存在文档级锁，避免全集合扫描校验锁状态，降低锁管理开销。
- **读写无阻塞特性**：读操作不会对文档加任何共享锁，写操作也不会阻塞读操作，该特性由 MVCC 多版本并发控制实现。

#### 2. MVCC 多版本并发控制

MVCC（Multi-Version Concurrency Control，多版本并发控制）是 WiredTiger 实现高并发读写的核心机制，其核心原理是**写时复制，而非原地覆盖**。

- 执行逻辑：每次修改文档时，不会直接覆盖原数据的磁盘页，而是生成一个新的文档版本，每个版本关联唯一的事务时间戳；读操作启动时会获取一个一致性快照，仅读取快照时间戳之前已提交的最新版本数据，无需加锁等待写操作完成。
- 核心优势：彻底实现读写互不阻塞，读操作永远不会被写操作阻塞，写操作也不会被读操作阻塞，在高并发读写混合场景下，性能表现远优于锁机制实现的并发控制。
- 版本垃圾回收：后台线程会定期清理不再被任何快照引用的旧版本数据，释放内存与磁盘空间，避免版本数据无限膨胀。

#### 3. 双缓存架构与内存管理

WiredTiger 采用**内部缓存 + 操作系统页缓存**的双缓存架构，最大化利用内存资源，减少磁盘 IO 开销。

- **WiredTiger 内部缓存**：
    
    - 核心作用：存储热数据的 B+Tree 索引页、数据页、脏页（已修改未刷盘的数据），是数据库的一级缓存，直接服务于读写请求，避免磁盘随机 IO。
    - 默认配置：占用系统可用内存的 50% 减去 1GB，最小不低于 256MB；生产环境建议根据服务器用途调整，专用 MongoDB 服务器可设置为系统内存的 50%-60%，预留足够内存给操作系统页缓存。
    - 淘汰策略：采用 CLOCK 时钟淘汰算法（类 LRU 改进算法），优先淘汰冷数据页，保留热数据页，保证缓存命中率。
    
- **操作系统页缓存**：
    
    - 核心作用：缓存磁盘上的数据文件与日志文件，优化顺序 IO 性能，服务于冷数据读取、预读、日志刷盘等场景，是数据库的二级缓存。
    - 设计优势：WiredTiger 将冷数据缓存交给操作系统原生页缓存管理，避免重复缓存，同时利用操作系统成熟的页替换、预读机制，降低内核态与用户态的切换开销。
    

#### 4. Checkpoint 检查点持久化机制

Checkpoint 是 WiredTiger 将内存中的脏数据批量持久化到磁盘的核心机制，是数据持久化与故障恢复的关键环节。

- 触发规则：默认每 60 秒执行一次，或当脏数据量达到内部缓存的 5% 时自动触发，也可手动执行`db.fsyncLock()`强制触发。
- 执行流程：
    
    1. 生成当前数据库的一致性快照，暂停新的脏页写入；
    2. 将快照内的所有脏页按顺序刷写到磁盘的数据文件中，更新 B+Tree 元数据；
    3. 写入新的检查点标识到元数据文件，标记本次检查点完成；
    4. 清理检查点之前的旧版本数据与无效 WAL 日志，释放存储空间。
    
- 核心作用：
    
    1. 持久化保障：检查点之前的所有数据已完全刷写到磁盘，即使数据库崩溃，该部分数据也不会丢失；
    2. 快速故障恢复：数据库重启时，仅需从最后一个完整检查点开始重放 WAL 日志，无需全量扫描数据文件，大幅缩短故障恢复时间；
    3. 空间回收：触发旧版本数据的垃圾回收，释放磁盘空间。
    

#### 5. WAL 预写日志（Journal 日志）

WAL（Write-Ahead Logging，预写日志）又称 Journal 日志，是 WiredTiger 实现数据零丢失的核心保障，遵循 “日志先写，数据后写” 的基本原则。

- 核心原理：所有修改操作在写入内存缓存之前，必须先顺序写入 WAL 日志文件；只有 WAL 日志刷盘成功后，该操作才会被标记为可提交，彻底避免服务器断电导致的内存数据丢失。
- 刷盘策略：默认每 100ms 执行一次批量刷盘，平衡性能与持久化安全性；可通过`storage.journal.commitIntervalMs`参数调整，或通过写关注的`j:true`参数强制每次写操作都同步刷盘。
- 生命周期：WAL 日志仅保留最后一个检查点之后的内容，检查点完成后，之前的 WAL 日志会被自动归档或清理，避免日志文件无限膨胀。
- 故障恢复：数据库重启时，会自动校验最后一个检查点之后的 WAL 日志，重放所有已记录但未刷入数据文件的操作，保证数据一致性，无数据丢失。

#### 6. 原生数据压缩

WiredTiger 原生支持多层级数据压缩，在几乎不影响读写性能的前提下，大幅降低磁盘存储空间占用，同时减少磁盘 IO 次数，提升吞吐量。

表格

|压缩类型|作用对象|支持算法|核心特性|
|---|---|---|---|
|块压缩|数据文件 B+Tree 页|Snappy（默认）、zstd、zlib、none|针对数据页的块级压缩，Snappy 算法 CPU 开销极低，压缩比约 3:1，平衡性能与空间效率；zstd 压缩比更高，适合冷数据存储|
|日志压缩|WAL 预写日志文件|Snappy（默认）、none|针对顺序写入的日志文件压缩，CPU 开销极小，可减少 50% 以上的日志文件体积|
|前缀压缩|索引 B+Tree 页|原生内置，不可关闭|针对索引键的前缀重复数据压缩，大幅降低复合索引、字符串索引的存储体积，索引体积可减少 70% 以上|

### 6.1.3 与 MMAPv1 引擎的核心差异

表格

|对比维度|WiredTiger 引擎|MMAPv1 引擎（已废弃）|
|---|---|---|
|锁粒度|文档级锁，并发写入性能极强|集合级锁，并发写入性能极差，易出现锁冲突|
|并发控制|MVCC 多版本并发控制，读写无阻塞|读写互斥锁，读操作会阻塞写，写操作会阻塞读|
|数据压缩|原生支持多层级压缩，空间效率极高|无压缩能力，磁盘空间占用大|
|缓存架构|双缓存架构，缓存命中率高|仅依赖操作系统内存映射，缓存管理能力弱|
|事务支持|原生支持多文档 ACID 事务|仅支持单文档原子性，无多文档事务能力|
|性能表现|多核 CPU 场景下，读写吞吐量提升 5-10 倍|多核扩展性极差，性能瓶颈显著|

## 6.2 读写关注级别

在分布式副本集架构下，MongoDB 通过**写关注（Write Concern）** 与**读偏好（Read Preference）** 两大配置，实现读写操作的一致性、可用性、性能三者的灵活权衡，是生产环境必须正确配置的核心参数，错误配置会直接导致数据丢失、主从数据不一致、业务可用性下降等严重问题。

### 6.2.1 Write Concern 写关注级别

写关注定义了一个写操作（新增、更新、删除、替换）必须满足何种持久化、副本同步条件，才会被数据库确认成功并返回给客户端，核心控制写操作的**数据一致性、持久化安全性、写入延迟**。

#### 标准语法

javascript

运行

```
// 单文档写入配置写关注
db.collection.insertOne(
  { userId: 1001, orderNo: "ORD20260407001" },
  {
    writeConcern: {
      w: "majority",
      j: true,
      wtimeout: 5000
    }
  }
)
```

#### 核心配置参数详解

1. **w 参数：副本确认级别**
    
    定义写操作需要被多少个副本集节点确认写入内存后，才返回成功，是写关注的核心参数。
    
    表格
    
    |取值|核心语义|安全性|性能|适用场景|
    |---|---|---|---|---|
    |w: 0|无确认写，客户端发送请求后立即返回，不等待数据库任何确认|极低，丢数据风险极高|最高|生产环境禁止使用，仅适用于非核心、可丢失的日志数据|
    |w: 1|默认值，仅需 Primary 主节点确认写入内存后即返回成功|中等，主节点宕机存在数据回滚风险|高|开发测试环境、非核心业务、可接受少量数据丢失的场景|
    |w: "majority"|需副本集**大多数投票节点**确认写入内存后才返回成功|极高，主节点宕机无数据回滚风险|中等|生产环境核心业务推荐配置，金融、账务、订单等不可丢失数据的场景|
    |w: <number>|指定具体节点数量，需对应数量的节点确认写入后返回成功|随节点数递增|随节点数递减|多副本固定节点数的集群，需强一致性保障的场景|
    
    - 核心约束：仲裁节点不存储数据，不参与 w 参数的确认计数；w 的取值不能超过副本集的投票节点总数；w: "majority" 的计数基于投票节点的半数以上，而非所有节点。
    
2. **j 参数：Journal 刷盘确认**
    
    定义写操作是否需要被持久化到磁盘的 WAL 日志文件后，才返回成功，核心控制写操作的**断电持久化安全性**。
    
    - j: true：写操作必须被同步刷入磁盘的 WAL 日志后，才返回成功。即使服务器断电、数据库崩溃，该操作也不会丢失，是最高级别的持久化保障。
    - j: false：默认值，仅需写入内存缓存即确认成功，依赖后台线程的批量刷盘机制，服务器断电存在丢数据风险。
    - 最佳实践：核心业务场景必须配置`j: true`配合`w: "majority"`，实现 “大多数节点持久化到磁盘” 的最高安全级别，彻底避免数据丢失。
    
3. **wtimeout 参数：写确认超时时间**
    
    单位为毫秒，定义写操作等待确认的最大超时时间，避免因副本节点不可用导致客户端无限阻塞。超过超时时间未获得足够确认时，写操作会抛出超时错误，已写入的节点不会自动回滚。
    
    - 核心作用：防止写操作长时间挂起，保障业务接口的响应时间可控；
    - 生产环境建议：设置 3000-10000ms 的超时时间，避免无限等待。
    

### 6.2.2 Read Preference 读偏好

读偏好定义了客户端的读请求会被路由到副本集中的哪一类节点执行，是 MongoDB 实现**读写分离、读负载均衡、多机房就近访问**的核心配置，同时决定了读操作的数据一致性级别。

#### 标准配置方式

1. 连接字符串配置（生产环境推荐）：
    
    plaintext
    
    ```
    mongodb://primary-node:27017,secondary-node1:27017,secondary-node2:27017/?readPreference=secondaryPreferred&maxStalenessSeconds=30
    ```
    
2. 客户端会话级别配置：
    
    javascript
    
    运行
    
    ```
    const session = db.getMongo().startSession({ readPreference: { mode: "secondaryPreferred" } });
    ```
    

#### 5 种核心读偏好模式详解

表格

|模式名称|核心路由规则|一致性保障|可用性保障|适用场景|
|---|---|---|---|---|
|primary|默认模式，所有读请求 100% 路由到 Primary 主节点|强一致性，始终读取最新已提交的数据|最低，主节点不可用时读请求直接报错|强一致性要求的核心业务，如账务查询、订单状态查询、支付结果校验|
|primaryPreferred|主节点优先，正常情况路由到主节点；主节点故障转移期间，路由到可用从节点|正常情况强一致性，主节点故障时为最终一致性，可能存在数据延迟|高，主节点故障时仍可提供读服务|需强一致性，同时要求故障期间读可用的核心业务，生产环境平衡模式|
|secondary|仅从节点读，所有读请求 100% 路由到 Secondary 从节点，主节点不处理任何读请求|最终一致性，数据存在主从同步延迟，延迟取决于网络与写入压力|最低，所有从节点不可用时读请求直接报错，即使主节点可用|非核心读业务，如历史数据查询、报表统计、离线数据分析，可接受数据延迟|
|secondaryPreferred|从节点优先，正常情况路由到从节点；所有从节点不可用时，自动路由到主节点|正常情况最终一致性，从节点不可用时为强一致性|最高，只要集群有一个节点可用，读服务就不会中断|生产环境读写分离的推荐模式，分流大部分读请求到从节点，同时保障读可用性|
|nearest|就近节点路由，读请求路由到网络延迟最低的节点，不区分主节点 / 从节点，基于客户端与节点的心跳延迟动态选择|最终一致性，取决于路由到的节点类型|最高，适配多机房部署|跨地域、多机房部署的场景，优先读取就近机房节点，降低网络延迟，对一致性要求低的业务|

#### 配套核心参数：maxStalenessSeconds

最大可接受的从节点数据延迟时间，单位为秒，用于避免读取数据过期太久的从节点。客户端会定期校验从节点与主节点的同步延迟，仅将读请求路由到延迟低于该阈值的从节点。

- 生产环境建议：设置 30-120 秒的阈值，根据业务对数据延迟的容忍度调整；
- 核心约束：最小值为 90 秒，不能低于副本集的心跳检测间隔；仅对 secondary、secondaryPreferred、nearest 模式生效。

## 6.3 ACID 事务支持

MongoDB 从诞生起就提供了**单文档原生原子性**，配合合理的内嵌文档建模，可满足 90% 以上的业务一致性需求；从 v4.0 版本开始支持副本集多文档事务，v4.2 版本扩展支持分片集群分布式事务，实现了完整的 ACID 特性，适配传统关系型数据库迁移的复杂一致性场景。

### 6.3.1 单文档原子性（默认原生保障）

单文档原子性是 MongoDB 文档模型的核心优势，也是业务优先选择的一致性方案。

- 核心定义：针对单个文档的所有写操作（包括新增、更新、删除，以及对内嵌文档、数组元素的修改），都具备完整的原子性 —— 要么全部执行成功，要么完全执行失败，绝不会出现部分修改成功、部分修改失败的中间状态。
- 实现原理：单文档的所有修改都对应单条 WAL 日志记录，且集中在同一个 B+Tree 数据页内完成，天然具备原子性，无需额外的事务开销，性能无任何损失。
- 最佳实践：优先通过内嵌文档的建模方式，将需要原子性修改的关联数据整合到同一个文档中，用单文档原子性替代多文档事务，彻底避免事务的性能开销与复杂度。

### 6.3.2 多文档 ACID 事务

多文档事务支持跨多个文档、多个集合、多个数据库的写操作，保证所有操作的原子性、一致性、隔离性、持久性，完全符合 ACID 规范。

#### 1. ACID 四大特性保障

表格

|特性|核心定义|MongoDB 实现保障|
|---|---|---|
|原子性（Atomicity）|事务内的所有操作是一个不可分割的整体，要么全部提交成功，要么全部回滚失败|事务提交时合并为单条 Oplog 记录，副本集节点要么全量应用，要么完全不应用；异常时可全量回滚，无中间状态|
|一致性（Consistency）|事务执行前后，数据库的数据完整性约束、业务规则始终保持一致|事务执行前后，唯一索引、文档校验规则、分片键约束始终生效；事务提交前的修改对外部不可见|
|隔离性（Isolation）|并发执行的多个事务之间互相隔离，不会互相干扰|默认提供快照隔离级别，避免脏读、不可重复读、幻读；支持读已提交隔离级别配置|
|持久性（Durability）|事务一旦提交成功，其修改的数据就会被永久持久化，不会因任何故障丢失|事务提交时可配置`w: "majority"`+`j: true`的写关注，保证数据被大多数节点持久化到磁盘，断电不丢失|

#### 2. 标准执行语法与流程

javascript

运行

```
// 1. 启动客户端会话（事务必须绑定到独立会话）
const session = db.getMongo().startSession();
// 2. 绑定数据库与集合对象
const testDB = session.getDatabase("test");
const orderColl = testDB.order;
const inventoryColl = testDB.inventory;

try {
  // 3. 启动事务
  session.startTransaction({
    readConcern: { level: "majority" },
    writeConcern: { w: "majority", j: true }
  });

  // 4. 事务内执行多文档操作（必须使用绑定会话的集合对象）
  // 操作1：写入订单数据
  orderColl.insertOne({ orderNo: "ORD20260407001", userId: 1001, amount: 5999, skuId: 10001, quantity: 1 });
  // 操作2：扣减商品库存
  inventoryColl.updateOne({ skuId: 10001 }, { $inc: { stock: -1 } });

  // 5. 提交事务
  session.commitTransaction();
  print("事务提交成功");
} catch (error) {
  // 6. 异常回滚
  session.abortTransaction();
  print("事务执行失败，已回滚，错误信息：", error);
} finally {
  // 7. 结束会话
  session.endSession();
}
```

#### 3. 分布式事务（分片集群场景）

MongoDB v4.2 + 版本支持分片集群环境下的分布式事务，可实现跨分片、跨 mongos 路由节点的多文档原子性操作，基于**两阶段提交协议（2PC）** 实现。

- 执行流程：
    
    1. mongos 路由节点作为事务协调者，接收客户端的事务请求；
    2. 准备阶段：协调者向所有涉及数据修改的分片节点（参与者）发送准备指令，分片节点执行事务操作，写入 WAL 日志，返回准备成功 / 失败；
    3. 提交阶段：只有所有分片节点都返回准备成功，协调者才会向所有分片发送提交指令，完成事务提交；若任意分片返回准备失败，协调者会向所有分片发送回滚指令，全量回滚事务。
    
- 核心约束：分布式事务的性能开销显著高于副本集事务，涉及跨节点网络通信与两阶段提交，仅适用于跨分片的强一致性场景，需严格控制事务范围与执行时间。

### 6.3.3 事务隔离级别

MongoDB 事务支持两种隔离级别，默认采用**快照隔离级别**，可通过`readConcern`参数灵活配置。

1. **快照隔离（Snapshot Isolation）**
    
    - 核心特性：事务启动时，会获取数据库的一致性快照，事务内的所有读操作都基于该快照执行，只会读取快照时间点之前已提交的最新数据；事务提交前，所有修改对其他事务完全不可见。
    - 解决的问题：彻底避免脏读、不可重复读、幻读三大并发问题，是 MongoDB 默认的隔离级别。
    - 适用场景：对数据一致性要求高的核心业务，如金融转账、库存扣减、订单创建等。
    
2. **读已提交（Read Committed）**
    
    - 核心特性：事务内的读操作，只会读取其他事务已提交的数据，不会读取未提交的脏数据；但同一事务内，多次读取同一字段可能返回不同结果（不可重复读），也可能出现幻读。
    - 优势：锁占用时间更短，并发性能更高，快照资源占用更少。
    - 适用场景：对一致性要求较低、追求高并发性能的非核心业务。
    

- 补充说明：MongoDB 不支持读未提交、串行化两种隔离级别。

### 6.3.4 事务核心约束与限制

1. **执行时间限制**：事务的默认最大执行时间为 60 秒，超出会被自动终止回滚，可通过`maxTransactionLockRequestTimeoutMillis`参数调整，生产环境禁止长事务。
2. **数据量限制**：事务内修改的所有数据总大小不能超过 16MB，因为事务提交时会将所有操作合并为单条 Oplog 记录，受限于 BSON 文档 16MB 的硬上限。
3. **锁竞争约束**：事务内的写操作会对修改的文档加排他锁，锁会一直持有到事务提交或回滚；长事务会导致锁长时间占用，引发严重的锁竞争，阻塞其他业务操作，甚至导致数据库雪崩。
4. **操作范围限制**：事务内禁止执行 DDL 操作（创建 / 删除集合、创建 / 删除索引等）；禁止操作 config、admin、local 系统数据库；禁止执行跨分片的 count、mapReduce 操作。
5. **分片集群约束**：分布式事务不能跨多个 mongos 节点执行，所有操作必须在同一个 mongos 节点的同一会话内完成；事务提交后，分片集群的均衡器不会立即迁移事务修改的文档，避免数据不一致。

### 6.3.5 事务最佳实践

1. **优先使用单文档原子性**：这是 MongoDB 最核心的一致性方案，无额外性能开销，90% 以上的业务场景可通过合理建模实现，无需多文档事务。
2. **最小化事务范围**：事务内的操作越少越好，执行时间越短越好，禁止在事务内执行复杂查询、循环操作、接口调用等耗时逻辑，减少锁占用时间。
3. **严格控制事务数据量**：事务内修改的文档数量、数据体积必须严格控制，禁止大批量数据修改使用事务，确保不超过 16MB 的硬限制。
4. **正确配置读写关注级别**：事务必须配置`readConcern: { level: "majority" }`与`writeConcern: { w: "majority", j: true }`，保证事务的一致性与持久化安全性，避免数据回滚与丢失。
5. **完善异常重试机制**：针对事务超时、写冲突、节点不可用等异常，必须实现幂等性重试逻辑；仅对未提交的事务、幂等操作进行重试，禁止重复提交已完成的事务。
6. **避免热点文档事务**：禁止在事务内修改高频并发访问的热点文档，会导致锁竞争加剧，数据库吞吐量急剧下降，甚至引发业务超时雪崩。
7. **低峰期执行大批量事务**：涉及大批量数据修改的事务，必须在业务低峰期执行，避免占用大量数据库资源，影响在线业务的正常运行。

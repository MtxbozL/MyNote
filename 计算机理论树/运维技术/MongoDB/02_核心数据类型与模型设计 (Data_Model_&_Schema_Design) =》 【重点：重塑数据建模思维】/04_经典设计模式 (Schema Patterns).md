
## 2.4 MongoDB 经典 Schema 设计模式

Schema 设计模式是 MongoDB 在海量业务场景中沉淀的标准化建模解决方案，针对特定业务场景的痛点，提供可复用、高性能的建模范式，可有效规避文档模型的常见陷阱。

### 2.4.1 桶模式（Bucket Pattern）

1. **模式定义**
    
    桶模式是针对时序数据场景的专用建模范式，核心是将连续产生的多条离散时序数据，按照固定的时间窗口、数量上限打包存储到单个文档中，每个文档对应一个数据 “桶”，而非每条数据生成一个独立文档。
2. **适用场景**
    
    物联网设备时序采集数据、服务器监控指标、用户行为埋点、系统日志、高频交易流水等按时序产生、按时间范围查询的场景。
3. **设计原理**
    
    时序数据的原生建模方式为单条数据一个文档，会导致集合内文档数量爆炸式增长，索引体积过大，写入与查询性能持续下降。桶模式通过数据打包，将海量小文档合并为少量大文档，实现：
    
    - 大幅减少集合内的文档总数，降低索引的存储开销与维护成本；
    - 优化写入性能，批量写入单桶文档，替代高频次的单条小文档写入；
    - 预聚合数据，可在桶内预计算最大值、最小值、平均值等指标，减少查询时的聚合计算开销；
    - 优化 TTL 过期策略，可按桶维度设置过期索引，替代单条数据的过期清理。
    
4. **核心约束**
    
    桶的时间窗口、数量上限必须严格控制，保证单桶文档的体积远低于 16MB 上限，避免文档超限；同时需设置桶的溢出保护机制，超出数量上限时自动创建新桶。

### 2.4.2 多态模式（Polymorphic Pattern）

1. **模式定义**
    
    多态模式是利用 MongoDB 的 Schema-less 特性，在同一个集合中存储结构相似、共享核心字段，但具备差异化专属字段的多种类型文档的建模范式。所有文档同属一个业务域，共享相同的查询访问模式，同时保留各自类型的结构特性。
2. **适用场景**
    
    电商多品类商品管理（不同品类商品具备差异化属性）、内容管理系统（文章、视频、图片等多类型内容）、企业级多类型业务实体管理、多租户差异化字段场景。
3. **设计原理**
    
    关系型数据库中，不同结构的异构数据必须拆分到不同的数据表中，导致跨表查询复杂度高、业务迭代不灵活。多态模式打破该限制，将同业务域的异构数据统一存储在单个集合中，实现：
    
    - 简化数据模型，同业务域数据统一存储入口，无需拆分多个集合；
    - 简化查询逻辑，统一查询接口，可实现跨类型全局查询，也可按类型过滤查询；
    - 业务迭代零成本，新增业务类型无需修改集合结构，仅需新增专属字段即可上线；
    - 索引优化灵活，可针对通用核心字段创建全局索引，针对差异化专属字段创建部分索引，兼顾查询性能与存储效率。
    

### 2.4.3 属性模式（Attribute Pattern）

1. **模式定义**
    
    属性模式是针对大量动态键值对场景的专用建模范式，核心是将文档中大量扁平的、动态的、同类型的键值对字段，转换为统一结构的数组内嵌文档，每个内嵌文档包含`key`（属性名）、`value`（属性值）及可选元数据，替代原有的独立字段设计。
2. **适用场景**
    
    商品多维度规格参数、用户自定义扩展属性、多语言国际化字段、传感器多维度指标、海量动态键值对的业务场景。
3. **设计原理**
    
    原生扁平建模方式中，每个动态属性对应一个独立字段，当属性数量达到几十甚至上百个时，会出现两大核心痛点：一是需要为每个字段单独创建索引，引发 “索引爆炸” 问题，索引维护成本与存储开销极高；二是针对任意属性的组合查询，需要编写大量冗余的查询条件，灵活性极差。
    
    属性模式通过统一的数组结构重构动态字段，实现：
    
    - 彻底解决索引爆炸问题，仅需创建一个针对`key`与`value`的复合多键索引，即可覆盖所有动态属性的查询场景；
    - 极致的查询灵活性，可针对任意属性组合进行过滤、排序、聚合，无需修改查询语句结构；
    - 动态扩展零成本，新增属性无需修改文档结构与索引，仅需向数组中新增元素即可；
    - 简化数据模型，避免文档字段数量过多导致的可读性、维护性下降。
    
4. **核心约束**
    
    需合理控制数组的元素数量，避免数组过长导致多键索引体积过大、查询性能下降；同时需保证属性结构的标准化，避免数组内元素结构异构。

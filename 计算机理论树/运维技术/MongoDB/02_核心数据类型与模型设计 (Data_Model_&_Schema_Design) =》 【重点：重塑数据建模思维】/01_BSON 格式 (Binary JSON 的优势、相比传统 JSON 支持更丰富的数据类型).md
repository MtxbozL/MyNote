## 2.1 BSON 数据格式

### 2.1.1 BSON 的核心定义与设计目标

BSON（Binary JSON）是 MongoDB 原生的存储与网络传输格式，是一种基于二进制编码的类 JSON 文档结构规范，为 MongoDB 的动态数据模型、高性能读写、丰富数据类型支持提供了底层基础。其核心设计目标是解决原生 JSON 在数据库存储场景中的核心缺陷：文本编解码效率低、存储体积大、数据类型支持有限、无法高效遍历与随机访问。

BSON 遵循三大核心设计原则：

1. **轻量性**：二进制编码开销最小化，在保证功能完备的前提下控制存储体积；
2. **可遍历性**：编码结构支持高效的字段遍历与随机访问，无需完整解码整个文档即可定位目标字段，为查询、索引、更新操作提供性能基础；
3. **类型完备性**：原生支持关系型数据库与业务开发所需的全量数据类型，弥补 JSON 的类型短板。

### 2.1.2 BSON 与原生 JSON 的核心差异

|特性维度|BSON|原生 JSON|
|---|---|---|
|编码形式|二进制编码，机器可直接解析|文本编码，需先完成字符串解码才能解析|
|数据类型|支持日期、二进制、高精度数值、ObjectId、正则表达式等丰富类型|仅支持字符串、数值、布尔、数组、对象、null 六种基础类型|
|编解码性能|编解码效率极高，CPU 开销极低|文本编解码开销大，大规模数据处理性能瓶颈显著|
|存储体积|数值、二进制数据存储体积更小，整体空间效率更高|文本格式存在大量冗余字符，空间效率低|
|随机访问能力|内置字段长度偏移量信息，可直接定位目标字段，无需全量解码|必须完整解析整个 JSON 字符串，才能定位目标字段，无随机访问能力|

### 2.1.3 BSON 的核心约束

MongoDB 对 BSON 文档设置了不可突破的硬约束，是数据建模与操作的核心前提：

1. **单文档最大体积**：单个 BSON 文档的存储体积上限为**16MB**，超出该限制的文档无法写入数据库；
2. **嵌套层级上限**：BSON 文档的嵌套层级最大为**100 层**，超出层级限制的文档无法完成写入；
3. **字段名约束**：字段名禁止包含空字符`\0`，顶级字段名禁止以`$`符号开头，避免与系统操作符冲突。


## 3.2 查询操作（Read）

查询操作是从指定集合中检索匹配过滤条件的文档的操作，是业务开发中最常用的核心操作。MongoDB 提供单文档精准查询与多文档游标查询两类核心 API，支持丰富的查询操作符实现复杂过滤，同时提供游标控制能力实现结果集的排序、分页与截断。

### 3.2.1 基础查询 API

#### 3.2.1.1 findOne 单文档查询

- **标准语法**：`db.collection.findOne(<query filter>, <projection>)`
- **核心特性**：仅返回匹配过滤条件的**第一个**文档，无匹配文档时返回`null`；无需游标处理，直接返回文档对象，适用于按主键、唯一键查询等确定仅返回单条结果的场景。
- **核心参数**：
    
    1. `query filter`：查询过滤条件，空文档`{}`表示匹配集合内所有文档。
    2. `projection`：投影配置，用于控制返回的字段，减少网络传输与内存开销。
    

#### 3.2.1.2 find 多文档查询

- **标准语法**：`db.collection.find(<query filter>, <projection>)`
- **核心特性**：返回匹配过滤条件的所有文档，结果以**游标（Cursor）** 形式返回，而非一次性加载全量文档，避免海量数据导致的内存溢出。游标支持迭代、批量拉取、排序、分页等二次操作。
- **补充说明**：mongosh 中`find`返回的游标会自动迭代前 20 条文档，可通过`it`命令继续迭代后续结果。

#### 3.2.1.3 投影（Projection）规则

投影用于精准控制返回文档的字段，分为包含模式与排除模式，二者不可混用（`_id`字段为唯一例外）。

- **包含模式**：仅返回指定的字段，语法如`{name: 1, age: 1}`，`1`表示包含该字段。默认`_id`字段会自动包含，需手动指定`{_id: 0}`排除。
- **排除模式**：返回除指定字段外的所有字段，语法如`{password: 0, mobile: 0}`，`0`表示排除该字段。
- **核心约束**：除`_id`字段外，同一个投影文档中不能同时出现包含与排除规则，否则会抛出语法错误。

### 3.2.2 核心查询操作符

所有查询操作符均以`$`开头，用于实现复杂的过滤逻辑，核心分为比较操作符、逻辑操作符、元素操作符三大类。

#### 3.2.2.1 比较操作符

比较操作符用于实现字段值与指定值的大小、相等性匹配，是最基础的过滤单元，标准语法为`{<field>: {<operator>: <value>}}`。

表格

|操作符|语义|核心说明|
|---|---|---|
|`$eq`|等于|匹配字段值等于指定值的文档，是默认匹配规则，`{name: "张三"}`与`{name: {$eq: "张三"}}`完全等价|
|`$gt`|大于|匹配字段值严格大于指定值的文档，支持数值、日期、字符串等可排序类型|
|`$gte`|大于等于|匹配字段值大于或等于指定值的文档|
|`$lt`|小于|匹配字段值严格小于指定值的文档|
|`$lte`|小于等于|匹配字段值小于或等于指定值的文档|
|`$ne`|不等于|匹配字段值不等于指定值，或该字段不存在的文档|
|`$in`|包含于|匹配字段值存在于指定数组中的文档，支持多值匹配|
|`$nin`|不包含于|匹配字段值不存在于指定数组中，或该字段不存在的文档|

#### 3.2.2.2 逻辑操作符

逻辑操作符用于组合多个查询条件，实现复杂的逻辑过滤，是多条件查询的核心基础。

表格

|操作符|语义|核心说明|
|---|---|---|
|`$and`|逻辑与|匹配同时满足所有指定条件的文档，逗号分隔的多个条件为隐式`$and`，仅当同字段需重复使用同一操作符、复杂逻辑嵌套时需显式声明|
|`$or`|逻辑或|匹配满足至少一个指定条件的文档，每个分支条件可独立使用索引，MongoDB 会为每个分支执行索引扫描后合并结果集|
|`$not`|逻辑非|匹配不满足指定条件的文档，仅作用于单个字段的操作符表达式，会匹配该字段不存在的文档|
|`$nor`|逻辑或非|匹配所有指定条件都不满足的文档，等价于对所有条件执行`$not`后组合`$and`，会匹配对应字段不存在的文档|

#### 3.2.2.3 元素操作符

元素操作符用于针对字段的存在性、数据类型进行过滤，是 Schema-less 场景下的核心操作符，用于解决同字段异构数据、字段缺失的过滤问题。

- `$exists`：字段存在性匹配，语法`{<field>: {$exists: <boolean>}}`
    
    - `$exists: true`：匹配包含该字段的文档，无论字段值为`null`还是空值；
    - `$exists: false`：匹配不包含该字段的文档；
    - 核心语义区分：`{field: null}`会匹配字段值为`null`或字段不存在的文档；`{field: {$eq: null, $exists: true}}`仅匹配字段存在且值为`null`的文档，二者需严格区分。
    
- `$type`：字段数据类型匹配，语法`{<field>: {$type: <BSON type alias/number>}}`
    
    - 作用：匹配字段值为指定 BSON 数据类型的文档，解决同字段异构数据类型的过滤问题；
    - 常用类型别名：`string`、`int`、`double`、`bool`、`date`、`objectId`、`array`、`object`、`null`。
    

### 3.2.3 游标（Cursor）控制与分页实现

游标是 MongoDB 对`find`查询结果集的抽象封装，本质是指向结果集的迭代指针，不会一次性将全量结果加载到内存中，而是通过批量拉取（batch）的方式迭代数据，是 MongoDB 查询的核心底层机制。

#### 3.2.3.1 核心游标控制方法（分页三剑客）

MongoDB 提供三大核心游标方法，实现结果集的排序、截断与跳过，是分页功能的核心实现基础。

1. **sort () 排序方法**
    
    - 语法：`cursor.sort({<field1>: <order>, <field2>: <order>})`
    - 核心规则：`order`值为`1`表示升序（ASC），`-1`表示降序（DESC）；支持多字段复合排序，按字段顺序优先执行，先按第一个字段排序，值相同的文档再按第二个字段排序。
    - 硬约束：当排序操作无法通过索引完成时，MongoDB 会将结果集加载到内存中排序，内存排序的结果集最大不能超过 32MB，超出会抛出错误，生产环境的排序操作必须有对应索引支撑。
    
2. **limit () 结果集截断方法**
    
    - 语法：`cursor.limit(<number>)`
    - 核心特性：限制游标返回的最大文档数量，`number`为正整数，`0`表示无限制；该方法会在查询优化阶段被下推到存储引擎，仅拉取指定数量的文档，性能极高。
    
3. **skip () 结果集跳过方法**
    
    - 语法：`cursor.skip(<number>)`
    - 核心特性：跳过游标中指定数量的文档，返回后续结果，`number`为非负整数。
    - 性能缺陷：`skip`的实现机制是先扫描并跳过指定数量的文档，再返回后续结果，当`skip`数值极大时（十万、百万级），会产生严重的性能开销，需避免深分页场景下使用。
    

#### 3.2.3.2 分页实现方案

基于游标控制方法，MongoDB 提供两种主流分页实现方案，分别适配不同业务场景。

1. **Offset 分页（skip + limit + sort）**
    
    - 实现逻辑：按页码计算跳过条数，`skip = (pageNum - 1) * pageSize`，配合`limit(pageSize)`与排序条件实现分页。
    - 示例：第 3 页、每页 10 条、按`_id`降序分页：`db.collection.find().sort({_id: -1}).skip(20).limit(10)`
    - 优点：实现简单，支持随机跳页，适配前端传统分页组件；
    - 缺点：深分页性能极差，`skip`值越大性能越差；数据频繁变更时，会出现漏数据、重复数据的问题；
    - 适用场景：小数据量、浅分页场景，页码不超过 100 页的业务。
    
2. **游标分页（Seek 分页，键值分页）**
    
    - 实现逻辑：利用排序字段的唯一值作为游标锚点，通过过滤条件替代`skip`，直接定位到上一页最后一条数据的位置，仅拉取当前页数据，无需扫描跳过大量文档。
    - 前提条件：排序字段必须是唯一、有序、不可变的（如`_id`、唯一时间戳）。
    - 示例：按`_id`降序分页，上一页最后一条文档的`_id`为`660a1b2c3d4e5f67890abcde`，当前页 10 条：`db.collection.find({_id: {$lt: ObjectId("660a1b2c3d4e5f67890abcde")}}).sort({_id: -1}).limit(10)`
    - 优点：彻底解决深分页性能问题，无论分页到多少页，性能始终稳定；数据变更时，不会出现漏数据、重复数据的问题；
    - 缺点：不支持随机跳页，仅支持上一页 / 下一页的滚动分页；
    - 适用场景：大数据量、深分页场景，移动端无限滚动列表、瀑布流等业务，是生产环境推荐方案。


## 3.4 删除操作（Delete）

删除操作是移除集合中指定文档或整个集合的操作，所有删除操作均为不可逆操作，生产环境必须严格控制权限与执行条件，避免数据误删。

### 3.4.1 基础删除 API

#### 3.4.1.1 deleteOne 单文档删除

- **标准语法**：`db.collection.deleteOne(<filter>, <options>)`
- **核心特性**：仅删除匹配过滤条件的**第一个**文档，即使过滤条件匹配多个文档，也仅删除第一条；单文档删除具备原生原子性。
- **适用场景**：按主键、唯一键删除单条文档的场景。

#### 3.4.1.2 deleteMany 批量文档删除

- **标准语法**：`db.collection.deleteMany(<filter>, <options>)`
- **核心特性**：删除匹配过滤条件的**所有**文档，是批量删除数据的标准 API。
- **高危警告**：若过滤条件为空文档`{}`，会删除集合内的所有文档，仅保留集合结构与索引，生产环境必须严格禁止无过滤条件的`deleteMany`操作。

#### 3.4.1.3 drop 集合删除

- **标准语法**：`db.collection.drop(<options>)`
- **核心特性**：彻底删除整个集合，包括集合内的所有文档、所有索引、集合元数据，直接释放磁盘空间。
- **与`deleteMany({})`的核心区别**：`deleteMany({})`仅逐行删除文档，保留集合与索引，会记录全量 Oplog，海量数据删除性能极低；`drop`直接删除集合文件，性能极高，Oplog 仅记录一条`drop`命令，是全量清空集合的推荐方案。
- **高危警告**：`drop`操作不可逆，生产环境必须严格控制该操作的权限，仅管理员可执行。

### 3.4.2 删除操作核心约束与最佳实践

1. **原子性约束**：单文档删除具备原生原子性，`deleteMany`批量删除是多条单文档删除的集合，不具备跨文档原子性，若需批量删除的原子性，必须使用多文档事务。
2. **性能优化**：大批量数据删除时，优先使用`drop`集合，或分批循环执行删除操作，避免单次`deleteMany`删除海量数据导致的数据库锁、Oplog 暴涨、主从延迟问题。
3. **安全规范**：生产环境执行删除操作前，必须先用`find`查询验证过滤条件的匹配结果，确认无误后再执行删除；核心业务数据优先使用软删除（用`status`字段标记删除状态），而非物理删除，避免数据误删无法恢复。
4. **持久化控制**：删除操作支持`writeConcern`配置，生产环境建议设置`w: majority`，保证删除操作写入大多数节点后才返回，避免主节点宕机导致删除操作回滚。


# 第五章 聚合管道框架（Aggregation Framework）

聚合管道是 MongoDB 原生的高性能数据处理与分析框架，是 MongoDB 实现复杂数据转换、多维度统计、跨集合关联、批量数据处理的核心能力，被称为 MongoDB 的 “数据分析杀手锏”。其完全替代了老旧的 Map-Reduce 方案，提供了声明式、可优化、易维护的类 SQL 数据处理能力，同时完美适配 MongoDB 的文档模型与分布式架构，是从基础开发进阶到高级应用的核心分水岭。

## 5.1 聚合管道核心概念与执行模型

### 5.1.1 核心定位与设计目标

聚合管道的核心设计目标是：以流式处理的方式，对集合中的文档进行多阶段、串行化的转换与计算，最终输出结构化的分析结果。其解决了基础`find`查询的能力边界问题：`find`仅能完成数据过滤、投影、排序、分页的基础操作，而聚合管道可实现分组统计、跨集合关联、数据结构重构、复杂数值计算、窗口函数分析等全场景数据处理能力，覆盖 OLTP 场景的复杂查询与 OLAP 场景的轻量数据分析需求。

### 5.1.2 管道执行模型与语法规范

#### 核心执行模型

聚合管道遵循**流式管道执行原则**，其核心逻辑与 Linux 系统的管道符`|`完全一致：

1. 数据输入源为指定集合，初始输入为集合内的全量文档（可通过索引过滤缩小范围）；
2. 管道由一系列有序的**阶段（Stage）** 组成，每个阶段对应一个独立的数据处理逻辑；
3. 上一个阶段的输出文档集，作为下一个阶段的输入文档集，阶段之间严格串行执行；
4. 所有阶段执行完成后，输出最终的结果集，可直接返回给客户端、写入新集合、或通过游标迭代。

#### 标准语法规范

javascript

运行

```
db.collection.aggregate(
  [ <stage1>, <stage2>, ..., <stageN> ],
  <options>
)
```

核心语法约束：

1. 管道的入参为**阶段数组**，数组内的每个元素为一个阶段对象，阶段的执行顺序与数组顺序完全一致；
2. 每个阶段对象**有且仅有一个顶级阶段操作符**（以`$`开头），操作符的值为该阶段的配置参数，禁止一个阶段内包含多个顶级操作符；
3. `options`为可选配置项，用于控制管道的运行时行为，核心包括`allowDiskUse`（是否允许磁盘溢出）、`explain`（执行计划分析）、`cursor`（游标配置）、`maxTimeMS`（最大执行超时时间）。

### 5.1.3 与 find 查询、Map-Reduce 的核心差异

表格

|对比维度|聚合管道|find 查询|Map-Reduce|
|---|---|---|---|
|能力边界|全场景数据处理，支持过滤、分组、关联、转换、计算、排序等全量操作|仅支持基础的过滤、投影、排序、分页，无分组、关联、复杂计算能力|支持自定义 JavaScript 逻辑的分布式计算，能力灵活但受限|
|执行性能|原生 C++ 实现，编译优化执行，性能极高，是 MongoDB 官方推荐方案|原生 C++ 实现，基础查询性能最优，无额外处理开销|基于 JavaScript 引擎执行，需频繁上下文切换，性能极差，已被官方不推荐使用|
|优化能力|内置查询优化器，支持阶段重排、合并、索引下推、分片预聚合等深度优化|支持索引优化、执行计划分析，优化能力覆盖基础查询场景|无自动优化能力，需手动编写优化逻辑，维护成本极高|
|语法特性|声明式 JSON 语法，结构清晰，易读易维护，与 MongoDB 原生操作符完全兼容|极简声明式语法，学习成本最低|命令式 JavaScript 语法，需编写 map、reduce、finalize 三个独立函数，复杂度高|
|分布式支持|原生适配分片集群，支持阶段下推、分片预聚合，水平扩展能力极强|原生适配分片集群，支持路由节点请求分发|适配分片集群，但执行效率低，易出现数据倾斜|

## 5.2 核心 Stage 阶段操作符

阶段操作符是聚合管道的核心组成单元，每个操作符对应一类数据处理能力，以下为生产环境高频使用的核心阶段，覆盖 99% 的业务场景。

### 5.2.1 $match 数据过滤阶段

`$match`是聚合管道的过滤阶段，用于筛选符合条件的文档，仅将匹配的文档传递给下一个阶段，语法与`find`查询的过滤条件完全一致，支持所有 MongoDB 查询操作符。

- **标准语法**：`{ $match: { <query filter> } }`
- **核心优化准则**：
    
    1. **必须放在管道最前端**：`$match`放在管道开头时，可直接利用集合的索引实现`IXSCAN`索引扫描，彻底避免全表扫描，同时最大化减少后续阶段处理的数据量，实现 “提前剪枝”，是聚合管道性能优化的第一准则。
    2. 禁止在`$match`中使用聚合表达式，仅支持查询操作符；聚合表达式的过滤需放在后续的`$project`或`$addFields`阶段配合`$match`实现。
    
- **核心约束**：不支持`$where`、`$function`等需要执行 JavaScript 代码的查询操作符。
- **适用场景**：全量数据的前置过滤，缩小管道处理的数据范围，是所有聚合管道的必备起始阶段。

### 5.2.2 $group 分组聚合阶段

`$group`是聚合管道的核心分组阶段，对应 SQL 的`GROUP BY`子句，用于按指定的分组键将文档划分为多个分组，对每个分组内的文档执行聚合计算，最终每个分组输出一个结果文档。

- **标准语法**：
    
    javascript
    
    运行
    
    ```
    {
      $group: {
        _id: <group key>, // 分组键，必填项
        <output field1>: { <accumulator operator1>: <expression1> },
        <output field2>: { <accumulator operator2>: <expression2> },
        ...
      }
    }
    ```
    
- **核心语法规则**：
    
    1. `_id`字段为必填项，是分组的核心依据：相同`_id`值的文档会被划分到同一个分组，`_id`可以是单个字段、复合字段、常量、聚合表达式；若设置`_id: null`，则将所有文档划分为同一个分组，实现全量数据的聚合统计。
    2. 输出字段必须使用**累加器操作符**定义，累加器仅能在`$group`阶段使用，用于对分组内的文档执行聚合计算。
    
- **核心累加器操作符**：
    
    表格
    
    |累加器|核心语义|典型用法|
    |---|---|---|
    |`$sum`|求和计算|`$sum: 1` 实现分组计数（对应 SQL`COUNT(*)`）；`$sum: "$amount"` 对分组内的`amount`字段求和|
    |`$avg`|平均值计算|`$avg: "$score"` 计算分组内`score`字段的平均值|
    |`$max`|最大值计算|`$max: "$createTime"` 获取分组内最新的创建时间|
    |`$min`|最小值计算|`$min: "$price"` 获取分组内最低的价格|
    |`$push`|数组追加|`$push: "$orderId"` 将分组内的`orderId`拼接为数组，允许重复值|
    |`$addToSet`|去重数组追加|`$addToSet: "$userId"` 将分组内的`userId`拼接为去重数组，无重复值|
    |`$first`|获取分组内第一个文档的指定字段值|配合`$sort`使用，获取分组内排序后的第一条数据|
    |`$last`|获取分组内最后一个文档的指定字段值|配合`$sort`使用，获取分组内排序后的最后一条数据|
    
- **核心约束**：
    
    1. 单个`$group`阶段的内存使用默认上限为 100MB，超出需设置`allowDiskUse: true`允许磁盘溢出，避免内存溢出错误。
    2. `$group`阶段不保证输出结果的有序性，需通过后续`$sort`阶段实现排序。
    3. 分组键的基数过大时，易出现数据倾斜，导致单分组数据量过大，性能下降，需合理设计分组键。
    

### 5.2.3 $project 字段投影与结构重构阶段

`$project`是聚合管道的投影阶段，用于控制输出文档的字段、重构文档结构、创建计算字段，相比`find`的投影能力，其支持聚合表达式实现复杂的字段转换，是数据结构处理的核心阶段。

- **标准语法**：`{ $project: { <field specification> } }`
    
- **核心能力与规则**：
    
    1. **基础字段控制**：与`find`投影规则一致，`1`表示保留该字段，`0`表示排除该字段；除`_id`字段外，同一个`$project`阶段内禁止同时使用包含模式与排除模式。
    2. **计算字段创建**：支持通过聚合表达式创建全新的字段，无需提前在原文档中定义，例如`{ totalAmount: { $multiply: [ "$price", "$quantity" ] } }`，通过单价与数量计算总价。
    3. **文档结构重构**：支持将扁平字段重构为嵌套文档，或将嵌套文档拆解为扁平字段，完美适配业务层的数据结构需求，例如`{ user: { name: "$userName", age: "$userAge" } }`。
    4. **条件字段输出**：配合`$cond`等布尔表达式，实现字段的条件化输出，例如`{ status: { $cond: { if: { $gte: [ "$score", 60 ] }, then: "pass", else: "fail" } } }`。
    
- **与`find`投影的核心差异**：`find`投影仅能实现字段的包含与排除，而`$project`支持通过聚合表达式实现字段的计算、转换、重构，是二者的本质区别。
    

### 5.2.4 $sort / $limit / $skip 排序与分页阶段

该三个阶段对应`find`查询的游标控制方法，在聚合管道中作为独立阶段执行，用于实现结果集的排序、截断与分页，核心规则与`find`完全一致，但存在专属的优化策略。

1. **$sort 排序阶段**
    
    - 标准语法：`{ $sort: { <field1>: 1, <field2>: -1, ... } }`，`1`为升序，`-1`为降序，支持多字段复合排序。
    - 核心优化：当`$sort`阶段紧跟在管道开头的`$match`阶段之后时，可直接利用索引的有序性实现排序，彻底避免内存排序；当`$sort`后紧跟`$limit`阶段时，MongoDB 优化器会自动合并两个阶段，仅维护 Top N 的结果集，无需对全量数据排序，大幅降低内存开销。
    - 核心约束：单个`$sort`阶段的内存使用默认上限为 100MB，超出需设置`allowDiskUse: true`。
    
2. **$limit 结果截断阶段**
    
    - 标准语法：`{ $limit: <positive integer> }`，限制管道传递给下一个阶段的最大文档数量。
    - 核心特性：该阶段会被优化器下推到存储引擎，仅拉取指定数量的文档，性能极高。
    
3. **$skip 结果跳过阶段**
    
    - 标准语法：`{ $skip: <non-negative integer> }`，跳过指定数量的文档，将剩余文档传递给下一个阶段。
    - 性能缺陷：与`find`的`skip`一致，需先扫描并跳过指定数量的文档，数值极大时会出现严重的性能开销，深分页场景推荐使用键值分页替代。
    

### 5.2.5 $unwind 数组解构阶段

`$unwind`是 MongoDB 适配数组模型的专属阶段，核心作用是将包含数组字段的文档，按数组元素拆解为多条独立文档，实现数组的 “扁平化” 处理，是数组数据统计与分析的核心基础。

- **基础语法**：`{ $unwind: "<array field path>" }`，数组字段路径必须以`$`开头，例如`{ $unwind: "$tags" }`。
- **完整进阶语法**：
    
    javascript
    
    运行
    
    ```
    {
      $unwind: {
        path: "<array field path>", // 必填，待解构的数组字段路径
        includeArrayIndex: "<string>", // 可选，输出字段名，用于存储元素在原数组中的索引
        preserveNullAndEmptyArrays: <boolean> // 可选，是否保留数组为空、null、不存在的文档，默认false
      }
    }
    ```
    
- **核心特性与约束**：
    
    1. 解构规则：原文档的所有非数组字段会完整保留到每条拆解后的文档中，仅数组字段替换为对应的单个元素；例如包含 3 个元素的数组，解构后会生成 3 条独立文档。
    2. 空值处理：`preserveNullAndEmptyArrays`默认值为`false`，会自动过滤掉数组为空、null、字段不存在的文档，易导致数据丢失，业务场景中需根据需求显式设置为`true`。
    3. 性能约束：超大数组解构会导致文档数量爆炸式增长，大幅增加后续阶段的处理开销，需提前通过`$match`过滤无效数据，严格控制数组长度。
    
- **适用场景**：数组元素的分组统计、数组内嵌套文档的过滤与聚合、数组元素的排序与去重、数组与主文档的关联分析。

### 5.2.6 $lookup 左外连接阶段

`$lookup`是 MongoDB 实现跨集合关联查询的核心阶段，对应 SQL 的`LEFT OUTER JOIN`，彻底解决了文档模型下多集合关联查询的痛点，是 MongoDB 企业级应用的核心能力。

#### 1. 基础等值匹配语法

适用于简单的等值关联场景，仅支持单字段等值匹配，语法简洁易维护。

javascript

运行

```
{
  $lookup: {
    from: "<target collection name>", // 待关联的目标集合名，必填
    localField: "<local collection field>", // 当前集合的关联字段，必填
    foreignField: "<target collection field>", // 目标集合的匹配字段，必填
    as: "<output array field name>" // 输出的关联结果数组字段名，必填
  }
}
```

- **核心执行逻辑**：当前管道中的每个文档，以`localField`字段的值为匹配条件，去`from`指定的目标集合中，匹配`foreignField`字段值相等的所有文档，将匹配到的文档组装为数组，写入`as`指定的字段中，传递给下一个阶段。
- **左外连接特性**：当前集合的文档一定会被保留，即使目标集合中无匹配数据，`as`字段的值为空数组，不会过滤掉无匹配的文档，与 SQL 的`LEFT JOIN`完全一致。

#### 2. 进阶管道关联语法

MongoDB 3.6 + 版本支持，适用于复杂关联场景，可在关联时对目标集合执行过滤、投影、聚合操作，减少关联返回的数据量，大幅提升关联性能，同时支持非等值匹配、多条件匹配。

javascript

运行

```
{
  $lookup: {
    from: "<target collection name>",
    let: { <variable1>: "<expression1>", <variable2>: "<expression2>", ... }, // 定义当前集合的变量
    pipeline: [ <target collection pipeline> ], // 目标集合的执行管道
    as: "<output array field name>"
  }
}
```

- **核心语法规则**：
    
    1. `let`用于定义变量，将当前集合的字段赋值给变量，变量需以`$$`开头引用，仅能在`pipeline`内部使用。
    2. `pipeline`为目标集合的执行管道，可使用所有阶段操作符，实现对目标集合的过滤、投影、聚合，仅将管道输出的结果作为关联数据返回。
    3. 必须在`pipeline`的`$match`阶段中，显式声明`let`定义的变量匹配条件，否则会导致目标集合全表扫描，出现笛卡尔积关联，性能极差。
    
- **核心约束与优化准则**：
    
    1. 目标集合的`foreignField`字段必须创建索引，否则每次关联都会触发目标集合的全表扫描，海量数据场景下性能会完全不可用，是生产环境的硬性要求。
    2. `from`指定的目标集合不能是分片集合，但当前聚合的集合可以是分片集合，分片集群场景下需严格遵守该约束。
    3. 关联结果以数组形式返回，即使仅匹配到一条文档，也会封装为数组，可通过`$unwind`解构为单文档，或通过`$arrayElemAt`提取第一个元素。
    
- **适用场景**：主从表关联、订单与用户信息关联、商品与分类关联、多表联合查询等所有跨集合关联场景。
    

### 5.2.7 $addFields / $set 字段追加与修改阶段

`$addFields`与`$set`为完全等价的别名操作符（`$set`为 MongoDB 4.2 + 版本引入的简化别名），核心作用是向文档中追加新字段、修改现有字段的值，同时保留文档中的所有原有字段，是`$project`的轻量化替代方案。

- **标准语法**：`{ $addFields: { <new field1>: <expression1>, <new field2>: <expression2>, ... } }`
- **与`$project`的核心优势**：`$project`需要显式指定所有需要保留的字段，否则会自动排除未声明的字段；而`$addFields`仅需指定新增 / 修改的字段，文档中所有未声明的原有字段会自动保留，大幅简化了仅需修改少量字段的场景的语法，避免冗余的字段声明。
- **核心特性**：支持嵌套文档的字段修改、数组元素的更新、聚合表达式创建计算字段，能力与`$project`完全一致，仅字段保留规则不同。
- **适用场景**：仅需新增 / 修改少量字段，无需重构文档结构、无需排除大量字段的场景，是生产环境中高频使用的轻量化字段处理阶段。

## 5.3 聚合表达式与计算体系

聚合表达式是 MongoDB 聚合管道的计算核心，是用于字段值计算、类型转换、逻辑判断、数据处理的专用语法单元，可嵌套使用，支持字段引用、字面量、系统变量，覆盖全场景数据处理需求。

### 5.3.1 表达式基础语法与变量体系

#### 基础语法规则

1. 所有表达式操作符均以`$`开头，操作符的值为操作数数组或单个操作数，例如`{ $add: [ "$price", "$tax" ] }`。
2. 字段引用规则：使用`$`+ 字段名的方式引用文档中的字段，嵌套字段使用全路径引用，例如`$user.address.city`。
3. 字面量规则：非字段引用的常量值直接书写，无需添加`$`，例如数值、字符串、布尔值、日期字面量。
4. 支持无限层级嵌套，一个表达式的输出可作为另一个表达式的操作数，实现复杂的复合计算。

#### 核心系统变量

系统变量以`

开头，用于引用管道的上下文信息，是高级聚合操作的核心基础，高频核心变量如下：系统变量核心语义

### 5.3.2 核心分类表达式操作符

以下为生产环境高频使用的表达式操作符，覆盖 99% 的业务计算需求。

#### 1. 布尔逻辑表达式

用于实现条件判断、逻辑运算，是分支逻辑的核心基础。

表格

|操作符|核心语义|典型用法|
|---|---|---|
|`$cond`|三元条件判断，对应 SQL 的`CASE WHEN`|`{ $cond: { if: <boolean expression>, then: <true value>, else: <false value> } }`|
|`$ifNull`|空值处理，若表达式结果为 null/undefined/ 字段不存在，返回默认值|`{ $ifNull: [ "$phone", "未知手机号" ] }`|
|`$and` / `$or` / `$not`|逻辑与 / 或 / 非运算|`{ $and: [ { $gte: [ "$age", 18 ] }, { $lt: [ "$age", 60 ] } ] }`|

#### 2. 字符串处理表达式

用于实现字符串的拼接、截取、转换、分割等操作，适配文本数据处理需求。

表格

|操作符|核心语义|典型用法|
|---|---|---|
|`$concat`|字符串拼接|`{ $concat: [ "$firstName", " ", "$lastName" ] }`|
|`$substrCP`|按 Unicode 字符截取字符串|`{ $substrCP: [ "$content", 0, 100 ] }` 截取前 100 个字符|
|`$toLower` / `$toUpper`|字符串大小写转换|`{ $toLower: "$email" }` 邮箱转为小写|
|`$trim`|去除字符串首尾的空白字符 / 指定字符|`{ $trim: { input: "$inputStr", chars: " ," } }`|
|`$split`|按分隔符将字符串分割为数组|`{ $split: [ "$tags", "," ] }` 按逗号分割标签字符串|

#### 3. 日期处理表达式

用于实现日期的格式化、提取、计算、转换，是业务中时间维度统计的核心基础。

表格

|操作符|核心语义|典型用法|
|---|---|---|
|`$dateToString`|日期格式化为字符串|`{ $dateToString: { format: "%Y-%m-%d", date: "$createTime" } }` 转为年月日格式|
|`$dateFromString`|字符串解析为日期类型|`{ $dateFromString: { dateString: "$dateStr", format: "%Y-%m-%d" } }`|
|`$year` / `$month` / `$dayOfMonth`|提取日期的年 / 月 / 日部分|`{ $year: "$createTime" }` 提取日期的年份|
|`$dateAdd` / `$dateSubtract`|日期加减计算|`{ $dateAdd: { startDate: "$createTime", unit: "day", amount: 7 } }` 日期加 7 天|
|`$dateDiff`|计算两个日期的差值|`{ $dateDiff: { startDate: "$startTime", endDate: "$endTime", unit: "hour" } }` 计算小时差|

#### 4. 数值计算表达式

用于实现数值的四则运算、取整、取模等数学计算，适配业务数值处理需求。

表格

|操作符|核心语义|典型用法|
|---|---|---|
|`$add` / `$subtract`|数值加法 / 减法|`{ $add: [ "$price", "$tax", "$shippingFee" ] }` 总价计算|
|`$multiply` / `$divide`|数值乘法 / 除法|`{ $multiply: [ "$price", "$quantity" ] }` 金额计算|
|`$mod`|数值取模运算|`{ $mod: [ "$userId", 10 ] }` 用户 ID 按 10 取模，用于分库分表|
|`$round` / `$floor` / `$ceil`|四舍五入 / 向下取整 / 向上取整|`{ $round: [ "$avgScore", 2 ] }` 平均值保留 2 位小数|
|`$abs`|数值绝对值计算|`{ $abs: "$priceDiff" }` 计算价格差的绝对值|

#### 5. 数组处理表达式

用于实现数组的过滤、转换、查找、统计，是文档数组模型处理的核心能力。

表格

|操作符|核心语义|典型用法|
|---|---|---|
|`$filter`|按条件过滤数组元素，返回符合条件的元素数组|`{ $filter: { input: "$orders", as: "order", cond: { $gte: [ "$$order.amount", 1000 ] } } }`|
|`$map`|遍历数组并转换每个元素，返回转换后的新数组|`{ $map: { input: "$items", as: "item", in: { $multiply: [ "$$item.price", "$$item.quantity" ] } } }`|

## 5.4 聚合管道优化机制与最佳实践

### 5.4.1 内置自动优化策略

MongoDB 的查询优化器会对聚合管道执行自动优化，在不改变结果语义的前提下，调整执行计划以提升性能，核心优化策略如下：

1. **阶段重排优化（Pipeline Reordering）**
    
    优化器会自动将过滤、投影类阶段提前，减少后续阶段处理的数据量与字段数量。例如将管道中间的`$match`阶段提前到`$group`之前，将`$project`阶段提前，排除不需要的字段，降低后续阶段的内存与 IO 开销。
    
    注意：优化器不会改变业务语义，仅当阶段重排不影响最终结果时才会执行，手动将`$match`放在管道最前端是最优方案。
    
2. **阶段合并优化（Stage Coalescence）**
    
    优化器会将相邻的同类型阶段合并为一个阶段，减少数据遍历次数与执行开销。例如相邻的两个`$project`阶段会合并为一个，相邻的两个`$match`阶段会合并为一个，相邻的`$sort`+`$limit`阶段会合并为 Top N 排序优化。
    
3. **索引下推优化**
    
    管道开头的`$match`、`$sort`阶段会被优化器直接下推到存储引擎，利用集合的索引实现索引扫描与排序，避免全表扫描与内存排序，是最核心的优化策略。
    
4. **分片集群预聚合优化**
    
    在分片集群中，优化器会将可并行执行的阶段（`$match`、`$project`、`$group`、`$sort`）下推到每个分片节点执行预聚合，仅将预聚合后的结果集发送到 mongos 路由节点合并，大幅减少跨节点数据传输，提升分布式场景下的执行效率。
    

### 5.4.2 核心约束与运行时配置

1. **内存限制**：单个聚合阶段的内存使用默认上限为 100MB，超出该限制会抛出执行错误。对于大数据量的分组、排序操作，需显式设置`allowDiskUse: true`，允许阶段将临时数据写入磁盘文件，突破内存限制。
2. **结果集限制**：聚合管道输出的单个文档不能超过 16MB 的 BSON 文档硬上限；若返回游标迭代结果集，则无该限制，游标会分批拉取数据，避免内存溢出。
3. **执行超时控制**：通过`maxTimeMS`参数设置管道的最大执行时间，避免慢查询长期占用数据库资源，生产环境建议对所有聚合管道设置该参数。
4. **游标配置**：大数据量结果集推荐使用游标迭代，避免一次性加载全量结果到内存，默认`mongosh`会自动创建游标并迭代前 20 条结果。

### 5.4.3 生产环境性能优化最佳实践

1. **前置过滤，最小化数据处理范围**：始终将`$match`阶段放在管道最前端，利用索引过滤无效数据，最大化减少后续阶段处理的文档数量，是聚合管道优化的第一准则。
2. **优先使用索引覆盖**：管道开头的`$match`、`$sort`阶段必须有对应索引支撑，彻底避免全表扫描与内存排序；高频查询场景优先设计覆盖索引，实现仅通过索引即可完成过滤与排序。
3. **提前投影，减少数据传输**：在管道早期通过`$project`或`$addFields`排除不需要的字段，仅保留后续阶段需要的字段，降低内存占用与数据传输开销。
4. **避免大数组解构与笛卡尔积关联**：严格控制`$unwind`的数组长度，避免文档数量爆炸；`$lookup`关联时必须对目标集合的匹配字段创建索引，禁止无匹配条件的全量关联，避免笛卡尔积。
5. **合理拆分复杂管道**：超复杂、多阶段的管道可拆分为多个短管道，配合中间集合存储中间结果，降低单次管道的执行复杂度与内存开销，同时便于问题定位与调优。
6. **禁止在管道中使用 JavaScript 操作符**：`$where`、`$function`等 JavaScript 操作符会频繁触发 JS 引擎与原生引擎的上下文切换，性能极差，且无法利用索引，生产环境必须彻底禁止使用。
7. **业务低峰期执行大数据量聚合**：全量数据的聚合统计、批量数据处理操作，必须在业务低峰期执行，避免占用大量数据库资源，影响在线业务的可用性。


# 第八章 海量数据架构：分片集群（Sharding Cluster）

分片集群是 MongoDB 原生的分布式水平扩展架构，是突破单副本集存储上限、写入性能瓶颈，支撑 PB 级海量数据、百万级高并发读写的核心方案。本章围绕分片集群的架构原理、核心组件、片键设计、数据均衡机制全链路展开，其中片键设计是架构师核心痛点 —— 片键选型错误会导致集群性能雪崩，且几乎无在线修复方案，只能全量重做集群，是生产环境部署与面试考核的绝对重点。

## 8.1 分片集群核心定位与设计目标

### 8.1.1 核心定义与解决的核心痛点

MongoDB 副本集通过多副本冗余实现了高可用，但受限于单主节点架构，存在无法突破的原生瓶颈：

1. **存储上限**：全量数据必须存储在每个副本节点，单机磁盘容量决定了集群的最大存储能力，无法支撑 TB/PB 级海量数据；
2. **写入性能上限**：所有写操作必须路由到主节点，写入吞吐量受单机 CPU、IO 性能限制，无法通过增加节点实现线性扩展；
3. **并发能力瓶颈**：海量数据下，全表扫描、复杂聚合会占用主节点大量资源，导致业务读写延迟升高，无法实现业务与资源的隔离。

分片集群的核心设计，是将集合中的全量数据按**片键（Shard Key）** 拆分为多个连续的数据块（Chunk），并将 Chunk 分散存储到多个独立的分片节点（每个分片为完整副本集）中。通过数据分布式存储、请求分布式处理，实现存储容量与读写性能的无限水平扩展，同时对应用层完全透明，业务代码无需修改即可无缝适配。

### 8.1.2 核心设计目标

1. **无限水平扩展**：存储容量随分片数量线性增长，读写吞吐量随分片数量线性提升，新增分片无需停机，业务无感知；
2. **访问透明化**：通过统一路由节点对外提供服务，访问语法与单副本集完全兼容，应用层无需感知数据分布与集群拓扑；
3. **全链路高可用**：所有核心组件均采用副本集架构，无任何单点故障，单节点 / 单分片故障不影响整个集群的可用性；
4. **灵活的数据分布策略**：支持范围分片、哈希分片、标签分片三种核心分片模式，适配高并发写入、范围查询、多机房合规、冷热分离等全场景业务需求；
5. **自动负载均衡**：内置均衡器后台进程，自动检测分片间的数据分布差异，自动完成 Chunk 迁移，保证数据在集群内均匀分布。

### 8.1.3 分片集群与单副本集核心差异

表格

|对比维度|分片集群|单副本集|
|---|---|---|
|存储上限|无限水平扩展，单分片存储上限即为集群单块容量上限，PB 级数据无压力|受限于单机磁盘容量，通常最大支持 TB 级数据|
|写入性能|多主节点并行写入，吞吐量随分片数量线性提升|单主节点架构，写入性能受单机硬件限制，存在天花板|
|扩展方式|水平扩展，新增分片节点即可提升集群容量与性能|垂直扩展，只能升级单机 CPU、内存、磁盘，扩展上限低|
|业务隔离|支持分片标签实现业务、冷热数据的物理隔离，资源互不干扰|所有业务共享单副本集资源，易出现业务间资源竞争|
|运维复杂度|高，涉及路由、配置、分片三类组件，元数据管理、片键设计、数据均衡门槛高|低，仅需维护副本集节点，运维操作简单|
|适用场景|海量数据、高并发读写、业务线复杂的中大型生产系统|中小规模业务、数据量 TB 级以内、写入并发可控的场景|

## 8.2 分片集群三大核心组件

分片集群采用三层分布式架构，由**Mongos 路由节点、Config Servers 配置服务器、Shards 分片节点**三大核心组件构成，三者协同完成数据的存储、路由、计算与管理，缺一不可。

### 8.2.1 Mongos（路由节点）

Mongos 是分片集群的统一访问入口，是无状态的代理服务节点，充当客户端与底层分片集群之间的桥梁，是整个集群的 “流量网关”。

#### 核心职责

1. **客户端连接管理**：对外暴露与单副本集完全一致的访问协议与接口，接收客户端的所有 CRUD、聚合、管理命令，业务代码无需任何修改即可从副本集迁移到分片集群；
2. **元数据缓存与请求路由**：从 Config Servers 拉取并缓存集群元数据（分片列表、Chunk 分布、片键规则），根据请求中的片键信息，将请求精准分发到对应分片，避免无效的全分片广播；
3. **结果合并与计算**：对于跨分片的查询、聚合、排序请求，先将请求分发到所有目标分片，再对各分片返回的结果进行合并、排序、过滤、二次聚合，最终返回给客户端；
4. **集群管理协调**：作为分片集群管理操作的唯一入口，接收分片集合创建、分片增减、均衡器配置等管理命令，协调 Config Servers 与各分片节点完成执行，保证集群元数据的一致性。

#### 核心特性与生产规范

- **无状态特性**：Mongos 不存储任何业务数据，仅缓存元数据，可无限水平扩展，多个 Mongos 节点之间完全对等，无主从之分；
- **元数据自动刷新**：当 Config Servers 中的元数据发生变更（Chunk 迁移、分片增减）时，Mongos 会自动刷新本地缓存，保证路由的准确性，不会出现路由错误；
- **查询优化能力**：自动将过滤条件、聚合阶段下推到分片节点执行，仅返回必要数据到 Mongos 节点，减少跨节点数据传输与计算开销；
- **生产部署规范**：至少部署 2 个 Mongos 节点避免单点故障，高并发场景下可通过负载均衡器分发客户端连接；Mongos 节点需与应用服务部署在同一可用区，降低网络延迟；禁止将 Mongos 与分片节点部署在同一台服务器，避免资源竞争。

### 8.2.2 Config Servers（配置服务器）

Config Servers 是分片集群的 “大脑”，是集群元数据的唯一持久化存储载体，负责管理整个集群的拓扑、数据分布、权限配置等核心元数据，是集群正常运行的中枢。

MongoDB 3.4 + 版本强制要求 Config Servers 必须采用**3 节点副本集架构（CSRS，Config Server Replica Set）**，禁止使用单节点配置服务器，保证元数据的高可用与一致性。

#### 核心存储的元数据内容

1. 集群分片列表、分片地址、角色、状态信息；
2. 所有分片集合的片键定义、分片类型、索引配置；
3. Chunk 全量元数据：Chunk 的片键范围、所属分片、全局唯一 ID；
4. 集群的用户、角色、权限配置，以及分片标签、均衡器配置规则；
5. 分布式事务元数据、集群锁信息、均衡器迁移状态。

#### 核心职责与生产规范

1. 元数据的持久化存储与原子化修改，保证集群元数据的强一致性，是所有路由、数据迁移操作的唯一依据；
2. 为 Mongos 节点提供元数据查询服务，实时同步元数据变更；
3. 运行均衡器后台进程，协调分片间的 Chunk 迁移，维护数据的均匀分布；
4. 管理分布式事务的提交与回滚，保证跨分片事务的一致性。

**生产环境强制规范**：

- 必须采用 1 主 2 从的 3 节点副本集拓扑，3 个节点必须部署在不同物理服务器、不同可用区，绝对禁止单点部署；
- 禁止向 Config Servers 写入任何业务数据，禁止直接连接 Config Servers 执行业务读写操作；
- 禁止手动修改 Config Servers 中的系统元数据集合，会导致集群路由异常、数据丢失，且无法恢复；
- Config Servers 的硬件配置可低于业务分片节点，但必须保证磁盘的可靠性与 IO 性能，避免元数据写入延迟导致集群整体性能下降。

### 8.2.3 Shards（分片节点）

Shards 是分片集群的实际数据存储与计算单元，**每个分片都是一个独立的 MongoDB 副本集**，负责存储集群中的一部分 Chunk 数据，是集群的 “数据仓库” 与 “计算节点”。

#### 核心职责

1. 存储分配到本分片的 Chunk 数据，执行 Mongos 分发过来的读写请求、聚合计算、索引构建等操作；
2. 配合均衡器完成 Chunk 的迁入与迁出，保证数据的均匀分布，迁移过程中不中断业务读写；
3. 通过副本集机制实现本分片数据的高可用，单分片内的节点故障自动完成故障转移，不影响整个集群的可用性；
4. 执行本地数据的过滤、聚合、排序操作，仅返回必要结果给 Mongos 节点，减少跨节点数据传输。

#### 核心特性与生产规范

- 每个分片都是完整的副本集，具备副本集的所有高可用、数据冗余能力，分片内的节点故障不会导致集群整体不可用；
- 分片之间的数据无重叠，每个 Chunk 仅属于一个分片，保证数据的唯一性；
- 支持无缝水平扩容，新增分片后，均衡器会自动将现有分片的 Chunk 迁移到新分片，实现集群容量与性能的线性扩展；
- 支持分片级别的资源隔离，通过分片标签将特定业务、特定地域的数据绑定到指定分片，实现业务隔离、多机房就近存储、冷热数据分离。

**生产环境强制规范**：

- 每个分片必须采用 1 主 2 从的 PSS 副本集拓扑，禁止使用单节点分片、带仲裁节点的 PSA 拓扑；
- 不同分片的节点必须交叉部署，避免多个分片的主节点集中在同一台物理服务器，防止单机故障导致多个分片同时受损；
- 所有分片的硬件配置必须保持一致，避免出现性能短板，导致集群整体性能受限于最慢的分片；
- 禁止直接连接分片节点执行业务读写操作，所有业务请求必须通过 Mongos 节点，避免数据不一致与元数据异常。

### 8.2.4 分片集群完整请求执行流程

分片集群的所有请求都遵循 “客户端→Mongos→分片节点→Mongos→客户端” 的核心链路，根据请求是否包含片键，分为**精准路由请求**与**广播请求**两类，二者性能差异可达几个数量级。

#### 1. 精准路由请求（最优路径）

当请求的过滤条件包含完整片键时，Mongos 可通过元数据精准定位到目标 Chunk 所在的分片，直接将请求分发到单个分片执行，无跨分片开销。

完整执行流程：

1. 客户端连接 Mongos 节点，发送包含片键的读写请求；
2. Mongos 查询本地缓存的元数据，根据片键值定位到目标 Chunk 所属的分片；
3. Mongos 将请求转发到对应的分片节点；
4. 分片节点执行请求，将结果返回给 Mongos 节点；
5. Mongos 将结果直接返回给客户端，无需二次处理。

#### 2. 全分片广播请求（性能最差路径）

当请求的过滤条件不包含片键，或无法通过片键定位到目标分片时，Mongos 必须将请求广播到所有分片，合并所有分片的结果后再返回给客户端。

完整执行流程：

1. 客户端连接 Mongos 节点，发送不包含片键的查询 / 聚合请求；
2. Mongos 将请求广播到集群内的所有分片节点；
3. 每个分片节点在本地执行请求，返回结果给 Mongos 节点；
4. Mongos 对所有分片的结果进行合并、排序、过滤、二次聚合；
5. Mongos 将最终处理后的结果返回给客户端。

**核心优化准则**：生产环境必须保证 90% 以上的业务请求为精准路由请求，严格控制全分片广播请求的占比，否则分片集群的性能会远低于单副本集。

## 8.3 片键选择（Shard Key）【架构师核心痛点：片键选错，集群重做】

片键是分片集合中用户指定的一个或多个字段，是 MongoDB 拆分数据、路由请求的唯一依据，是分片集群设计的灵魂。分片集合创建后，片键**不可更换、不可删除**，只能通过全量数据导出、重新指定片键、重新导入的方式修改，也就是 “片键选错，集群重做”，是分片集群最核心、最不可逆的设计决策。

### 8.3.1 片键的核心硬约束

创建分片集合时，片键必须满足以下所有硬约束，否则会创建失败，生产环境必须提前校验：

1. **索引约束**：片键必须是分片集合的单键索引，或复合索引的最左前缀，无对应索引的字段无法作为片键；
2. **存在性约束**：片键字段必须存在于集合的每一个文档中，且值不可为`null`；
3. **不可变性约束**：片键的值写入后不可修改（MongoDB 4.2 + 支持有限度修改，但会带来极高的运维开销，生产环境禁止修改）；
4. **类型约束**：片键字段不能是数组类型，禁止使用数组、嵌套文档作为片键；
5. **唯一性约束**：若要创建跨分片唯一索引，索引必须包含完整片键，禁止创建不包含片键的唯一索引（无法跨分片校验唯一性）。

### 8.3.2 三大分片类型与适用场景

MongoDB 提供三种核心分片模式，分别适配不同的业务读写场景，核心差异在于数据分布规则、范围查询支持、写入热点风险。

#### 8.3.2.1 范围分片（Ranged Sharding）

范围分片基于片键的值的连续范围进行数据拆分，将片键值处于同一连续区间的文档分配到同一个 Chunk 中，Chunk 按片键值有序分布在不同分片。

- **标准创建语法**：

javascript

运行

```
// 开启数据库分片
sh.enableSharding("test")
// 为order集合创建范围分片，片键为userId
sh.shardCollection("test.order", { userId: 1 })
```

- **核心优势**：
    
    1. 完美支持片键的范围查询，可直接路由到包含目标数据的少数分片，无需全分片广播，查询性能极高；
    2. 数据按片键有序分布，针对片键的排序、分组聚合可直接利用索引有序性，无需 Mongos 二次排序，性能最优；
    3. 数据分布可控，可配合分片标签实现精细化的数据隔离、多机房就近存储。
    
- **核心劣势**：
    
    1. 极易产生写入热点与数据倾斜：若片键为单调递增 / 递减字段（如自增 ID、时间戳、ObjectId），所有新写入的数据会集中到同一个 Chunk，最终集中到单个分片，导致该分片成为写入热点，写入性能无法水平扩展；
    2. 对片键设计要求极高，若片键基数过低、值分布不均匀，会导致数据无法拆分，出现严重的数据倾斜。
    
- **适用场景**：高频使用片键进行范围查询、排序聚合的业务；片键基数高、值分布均匀，无单调递增特性的场景；多机房部署、多租户隔离需要精细化数据分布控制的场景。

#### 8.3.2.2 哈希分片（Hashed Sharding）

哈希分片通过 MongoDB 内置的哈希算法对片键的值进行计算，根据哈希结果进行 Chunk 拆分与分片分配。哈希算法保证相同片键值的哈希结果一致，不同值的哈希结果均匀随机分布，最终实现数据在所有分片上的均匀分散。

- **标准创建语法**：

javascript

运行

```
// 开启数据库分片
sh.enableSharding("test")
// 为user集合创建哈希分片，片键为userId
sh.shardCollection("test.user", { userId: "hashed" })
```

- **核心优势**：
    
    1. 数据分布极度均匀，彻底解决范围分片的写入热点问题，即使片键是单调递增字段，哈希后的值也会均匀分散，新写入的数据会分布到所有分片，实现写入吞吐量的线性扩展；
    2. 配置简单，无需复杂的片键设计，只要片键基数足够高，即可实现完美的数据均匀分布；
    3. 无单点写入瓶颈，写入压力均匀分散到所有分片，完美适配高并发写入场景。
    
- **核心劣势**：
    
    1. 完全不支持片键的范围查询，所有针对片键的范围查询都必须广播到所有分片，合并结果，查询性能极差；
    2. 无法利用片键的有序性，针对片键的排序操作需要跨分片合并结果，无法利用索引有序性；
    3. 数据分布不可控，无法通过片键实现业务隔离、冷热分离、多机房就近存储。
    
- **适用场景**：高并发写入、极少范围查询的业务场景；片键为单调递增字段、易产生写入热点的场景；核心需求是数据均匀分布、写入性能线性扩展的场景，如用户信息、日志数据、社交内容存储。

#### 8.3.2.3 标签分片（Zoned Sharding / 分区分片）

标签分片是基于范围分片的扩展能力，核心是将片键的特定范围与分片的自定义标签绑定，均衡器会自动将对应范围的 Chunk 迁移到绑定了对应标签的分片上，实现完全可控的精细化数据分布。

- **标准实现流程**：

javascript

运行

```
// 1. 为分片添加地域标签
sh.addShardTag("shard-gz", "GZ-AZ")
sh.addShardTag("shard-bj", "BJ-AZ")
// 2. 创建标签与片键范围的绑定规则
sh.addTagRange(
  "test.order",
  { region: "GZ", userId: MinKey },
  { region: "GZ", userId: MaxKey },
  "GZ-AZ"
)
sh.addTagRange(
  "test.order",
  { region: "BJ", userId: MinKey },
  { region: "BJ", userId: MaxKey },
  "BJ-AZ"
)
// 3. 开启分片集合，片键为region+userId复合片键
sh.shardCollection("test.order", { region: 1, userId: 1 })
```

- **核心优势**：
    
    1. 实现多机房就近存储，将特定地域的数据存储到对应地域的分片，降低业务访问延迟，满足数据属地化合规要求；
    2. 实现业务 / 租户隔离，将不同业务线、不同租户的数据绑定到指定分片，避免业务间的资源竞争；
    3. 实现冷热数据分离，将热数据存储到高性能 SSD 分片，冷数据存储到低成本机械硬盘分片，优化存储成本；
    4. 完全兼容范围分片的所有优势，支持片键的范围查询、精准路由、有序分布。
    
- **核心劣势**：配置复杂度高，需要提前规划片键范围、标签规则与分片拓扑，运维成本高于普通范围分片与哈希分片。
- **适用场景**：跨地域多机房部署、有数据属地化合规要求的业务；多租户、多业务线共享集群，需要业务隔离的场景；冷热数据分离存储，优化成本与性能的场景。

#### 三大分片类型核心对比

表格

|分片类型|数据分布均匀性|范围查询支持|写入热点风险|配置复杂度|核心适用场景|
|---|---|---|---|---|---|
|范围分片|取决于片键设计，易出现不均匀|完美支持，性能极高|高，单调片键极易出现热点|低|高频范围查询、排序聚合场景|
|哈希分片|极度均匀，无需人工干预|不支持，范围查询性能极差|极低，彻底解决写入热点|极低|高并发写入、极少范围查询场景|
|标签分片|完全可控，按标签规则分布|完美支持|极低，可通过标签分散热点|高|多机房部署、多租户隔离、冷热分离场景|

### 8.3.3 片键选择的黄金准则

1. **高基数原则**：片键必须具备极高的基数（不同值的数量），基数越高，数据拆分的粒度越细，越容易实现均匀分布。禁止使用性别、状态、类型等低基数字段作为片键，否则会导致数据无法拆分，出现严重的数据倾斜。
2. **写入分散原则**：片键必须保证新写入的数据均匀分散到所有分片，绝对禁止使用单调递增 / 递减字段作为范围分片的片键，否则会出现单点写入热点，写入性能无法扩展。
3. **查询定向原则**：片键必须完全匹配业务的高频查询模式，90% 以上的业务查询必须包含完整片键，保证请求可精准路由到单个 / 少数分片，避免全分片广播，这是分片集群查询性能的核心保障。
4. **不可变性原则**：片键的值必须是写入后永久不变的，禁止使用会频繁更新的字段作为片键，避免极高的运维开销与数据风险。
5. **原子性保障原则**：需要原子性修改的关联数据，必须保证片键值相同，存储在同一个分片，避免跨分片事务的性能开销与复杂度。

### 8.3.4 片键选择的绝对禁忌

1. 禁止使用低基数字段作为片键；
2. 禁止使用单调递增 / 递减字段作为范围分片的片键；
3. 禁止使用数组、嵌套文档作为片键；
4. 禁止使用会频繁更新的字段作为片键；
5. 禁止选择不匹配业务高频查询模式的字段作为片键；
6. 禁止使用复合片键时，不遵循最左前缀原则，导致查询无法实现精准路由。

## 8.4 Chunks 块与均衡器（Balancer）机制

Chunk 与均衡器是分片集群实现数据均匀分布、自动负载均衡的核心机制，Chunk 是数据拆分与迁移的最小单元，均衡器是维护分片间数据均匀分布的后台进程。

### 8.4.1 Chunks 数据块核心定义与特性

Chunk 是 MongoDB 分片集群中数据拆分的最小单元，是片键值的一个左闭右开的连续区间，每个 Chunk 包含片键值在该区间内的所有文档，是数据迁移、负载均衡的最小执行单位。

#### 核心特性

1. **默认大小**：Chunk 默认大小为 64MB，可通过`chunkSize`参数调整，合法范围为 1MB-1024MB；
2. **范围特性**：所有 Chunk 的范围连续且无重叠，覆盖片键从`MinKey`到`MaxKey`的所有可能值，保证所有文档都能归属到唯一 Chunk；
3. **归属特性**：每个 Chunk 仅属于一个分片，一个分片包含多个 Chunk，分片的总数据量等于其包含的所有 Chunk 的大小之和；
4. **自动分裂**：当 Chunk 的大小超过阈值，或文档数量超过上限时，会自动触发分裂，将一个大 Chunk 拆分为两个连续的小 Chunk，保证 Chunk 大小可控。

#### Chunk 大小配置最佳实践

表格

|Chunk 规格|核心优势|核心劣势|适用场景|
|---|---|---|---|
|小 Chunk（1-16MB）|数据分布更均匀，均衡粒度更细，负载均衡速度快|Chunk 数量过多，Config Servers 元数据压力大，频繁分裂与迁移增加集群开销|数据读写分布极不均匀，需要精细化负载均衡的场景|
|默认 Chunk（64MB）|平衡元数据压力与迁移开销，Chunk 数量可控，适配绝大多数业务|数据分布粒度较粗，极端场景下会出现少量数据不均匀|生产环境默认推荐，绝大多数通用业务场景|
|大 Chunk（128MB-1024MB）|Chunk 数量极少，元数据压力极低，分裂与迁移频率低|单 Chunk 迁移开销极大，易导致集群性能波动，负载均衡速度慢|PB 级海量冷数据、极少更新的归档数据场景|

**生产环境禁忌**：禁止在集群运行过程中随意修改`chunkSize`，修改后会触发全量 Chunk 分裂，导致集群元数据剧烈波动，严重影响业务性能；禁止为不同集合设置不同的`chunkSize`，增加运维复杂度。

### 8.4.2 Chunk 分裂机制

Chunk 分裂是 MongoDB 自动维护 Chunk 大小的核心机制，避免单个 Chunk 过大导致的迁移开销高、查询性能下降问题。

- **触发条件**：当 Chunk 内的文档数据量达到`chunkSize`的 50% 时，会触发分裂检查；当 Chunk 大小超过`chunkSize`阈值时，自动分裂为两个范围连续的 Chunk；
- **核心特性**：Chunk 分裂是纯元数据操作，仅修改 Config Servers 中的 Chunk 元数据，不移动实际业务数据，开销极低，不会影响业务性能；分裂后的 Chunk 仍归属原分片，不会自动迁移；
- **预分裂优化**：对于新创建的分片集合，可提前执行预分裂，创建空 Chunk 并均匀分布到所有分片，避免集合写入初期的频繁分裂与迁移，大幅提升初始写入性能。

### 8.4.3 均衡器（Balancer）核心机制

均衡器是运行在 Config Servers 副本集主节点上的后台进程，核心职责是监控所有分片的 Chunk 数量，当分片间的 Chunk 数量差异达到迁移阈值时，自动将 Chunk 从数量多的分片迁移到数量少的分片，保证数据在所有分片上均匀分布。

#### 8.4.3.1 核心工作流程

1. **负载检测**：均衡器定期检查所有分片的 Chunk 数量，计算分片间的数量差异，判断是否需要触发均衡；
2. **迁移阈值判断**：根据集群总 Chunk 数量动态确定迁移阈值，只有分片间的 Chunk 数量差异达到阈值，才会触发迁移，避免频繁无效迁移：
    
    表格
    
    |集群总 Chunk 数量|迁移阈值（分片间 Chunk 数量差）|
    |---|---|
    |<20|2|
    |20-79|4|
    |80+|8|
    
3. **迁移计划生成**：均衡器生成迁移计划，确定待迁移 Chunk、源分片、目标分片，保证迁移后分片间的 Chunk 差异低于阈值；
4. **原子化迁移执行**：
    
    1. 均衡器向源分片发送迁移指令，源分片锁定待迁移 Chunk，将 Chunk 内的全量文档复制到目标分片；
    2. 全量复制完成后，同步迁移期间的增量数据，保证源与目标分片的数据完全一致；
    3. 数据同步完成后，原子化更新 Config Servers 中的元数据，修改 Chunk 的所属分片为目标分片；
    4. 通知所有 Mongos 节点刷新元数据缓存，后续请求按新的分布路由；
    5. 源分片删除本地已迁移的 Chunk 数据，释放存储空间；
    
5. **均衡完成**：所有分片的 Chunk 数量差异低于阈值后，均衡器停止迁移，进入等待检测状态。

#### 8.4.3.2 核心特性与生产规范

1. **业务无感知**：Chunk 迁移过程中，针对该 Chunk 的读写请求仍路由到源分片，不会中断业务；只有元数据更新完成后，请求才会切换到目标分片，无数据丢失、无业务中断；
2. **流量控制**：默认限制并发迁移数量，避免迁移占用过多网络、IO 资源，可通过参数调整并发数与迁移速率；
3. **标签规则优先**：均衡器严格遵守分片标签规则，只会将 Chunk 迁移到绑定了对应标签的分片，不会破坏用户配置的分区规则；
4. **运行窗口配置**：生产环境必须配置均衡器运行窗口，仅在业务低峰期执行迁移，避免高峰期迁移导致的性能波动，配置示例：

javascript

运行

```
// 配置均衡器仅在凌晨00:00-06:00运行
db.settings.updateOne(
  { _id: "balancer" },
  { $set: { activeWindow: { start: "00:00", stop: "06:00" } } },
  { upsert: true }
)
```

5. **手动启停控制**：集群运维、版本升级、数据迁移期间，可通过`sh.stopBalancer()`停止均衡器，操作完成后通过`sh.startBalancer()`重新启动，避免迁移操作干扰运维流程。

## 8.5 分片集群生产环境核心约束与最佳实践

### 8.5.1 分片集群核心硬约束

1. **片键不可逆约束**：分片集合创建后，片键无法修改，只能通过全量重分片更换，必须提前完成片键设计与验证；
2. **唯一索引约束**：分片集合仅支持两种唯一索引：片键本身的唯一索引、包含完整片键的复合唯一索引，无法实现跨分片的非片键唯一约束；
3. **事务约束**：跨分片分布式事务的性能开销远高于单分片事务，必须严格控制事务范围与执行时间，禁止长事务、大事务；
4. **非分片集合约束**：非分片集合会全部存储在集群的主分片中，易导致主分片成为热点，必须将高频访问的大集合全部分片；
5. **操作约束**：禁止执行无片键过滤条件的全表扫描、更新、删除操作，会广播到所有分片，导致集群性能雪崩；
6. **_id 全局约束**：分片集合的`_id`字段仅在本分片内唯一，若需要全局唯一 ID，必须使用 ObjectId 或分布式 ID 生成器。

### 8.5.2 生产环境最佳实践

1. **先副本集，后分片集群**：业务初期优先使用副本集，当副本集的存储、写入性能达到瓶颈时，再升级为分片集群，避免过早引入分片集群的运维复杂度；
2. **片键设计先行**：分片集群的设计核心是片键设计，必须先梳理业务全量读写模式，完成片键选型与压测验证，再搭建集群，绝对禁止先搭集群再考虑片键；
3. **精准路由优先**：通过片键优化、查询优化，保证 90% 以上的业务请求为精准路由请求，严格控制全分片广播请求的占比；
4. **组件高可用部署**：Mongos、Config Servers、Shards 所有组件都必须采用多节点高可用部署，禁止任何单点故障；不同组件、不同分片的节点必须交叉部署，避免单机故障影响整个集群；
5. **全链路监控告警**：必须监控核心指标：片键路由命中率、全分片广播请求占比、分片数据分布均匀性、Chunk 迁移状态、均衡器运行状态、组件可用性、主从同步延迟；
6. **运维操作规范**：所有集群变更操作必须在业务低峰期执行；禁止直接连接分片节点执行业务读写；禁止手动修改 Config Servers 元数据；新增分片后必须监控迁移进度，保证迁移在低峰期完成。

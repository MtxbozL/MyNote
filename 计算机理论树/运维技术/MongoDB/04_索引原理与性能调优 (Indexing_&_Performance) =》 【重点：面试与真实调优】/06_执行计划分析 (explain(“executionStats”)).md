
## 4.6 执行计划分析

`explain()`是 MongoDB 查询性能诊断的核心工具，可查看查询优化器选择的执行计划，验证索引是否生效、索引效率是否达标，定位全表扫描、内存排序等性能问题，是性能调优的必备手段。

### 4.6.1 explain () 核心执行模式

表格

|执行模式|核心特性|适用场景|
|---|---|---|
|`queryPlanner`（默认）|仅生成候选执行计划，不实际执行查询，无运行时统计数据|快速验证索引是否在候选计划中，语法合法性校验|
|`executionStats`|实际执行查询，返回完整的运行时统计数据，包括扫描行数、执行时间、回表次数等|生产环境性能调优、索引效率验证、问题定位的核心模式|
|`allPlansExecution`|执行所有候选执行计划，返回全量计划的统计数据，可对比优化器的计划选择是否合理|索引选择异常、多索引竞争场景的深度诊断|

- **标准使用语法**：`db.collection.find(<filter>).sort(<sort>).explain("executionStats")`
- 补充说明：`updateOne`、`updateMany`、`deleteMany`等写操作，均可通过`explain()`分析执行计划，验证过滤条件是否命中索引。

### 4.6.2 执行计划核心指标解读

执行计划的核心诊断信息集中在`executionStats`字段，以下为必须掌握的核心指标与判断准则，是性能调优的核心依据。

#### 4.6.2.1 核心执行阶段（stage）

`executionStats.executionStages.stage`字段标识了查询的核心执行阶段，是判断执行计划好坏的第一优先级指标，核心阶段如下：

1. **COLLSCAN**：全表扫描（Collection Scan），**生产环境高危指标，必须彻底避免**。表示 MongoDB 扫描了集合内的所有文档来匹配查询条件，集合数据量越大，性能越差，百万级数据下查询延迟可达秒级甚至分钟级。
2. **IXSCAN**：索引扫描（Index Scan），理想执行阶段。表示 MongoDB 通过索引定位匹配的文档，仅扫描索引中符合条件的条目，性能远高于全表扫描。
3. **FETCH**：回表操作。根据索引扫描得到的`_id`主键，去聚集索引中读取完整文档数据，是查询性能的第二大开销来源，覆盖查询可彻底消除该阶段。
4. **SORT**：内存排序，**生产环境高危指标，必须避免**。表示排序操作无法利用索引的有序性，在内存中执行排序，当排序结果集超过 32MB 时，MongoDB 会直接抛出查询错误。
5. **LIMIT/SKIP**：结果集截断 / 跳过操作，对应`limit()`/`skip()`方法，需关注`SKIP`的数值是否过大，避免深分页性能问题。

#### 4.6.2.2 索引效率核心判断指标

表格

|指标名称|指标含义|理想状态判断准则|
|---|---|---|
|`nReturned`|查询最终返回的文档数量|与业务需求的结果数量一致|
|`totalKeysExamined`|索引扫描的条目总数|约等于`nReturned`，数值越接近，索引过滤效率越高；远大于`nReturned`表示索引设计不合理，大量无效索引扫描|
|`totalDocsExamined`|回表扫描的文档总数|约等于`nReturned`，数值越接近，回表开销越小；等于 0 表示实现了覆盖查询，为最优状态；远大于`nReturned`表示索引无法覆盖过滤条件，大量无效回表|
|`executionTimeMillis`|查询总执行时间，单位毫秒|数值越小越好，生产环境高频查询建议控制在 10ms 以内|
|`indexName`|执行计划实际使用的索引名称|与预期设计的索引一致，避免查询优化器选错索引|

**核心诊断准则**：生产环境的高效查询，必须满足`stage`为`IXSCAN`，无`COLLSCAN`与`SORT`阶段，且`totalKeysExamined`、`totalDocsExamined`与`nReturned`的数值尽可能接近。

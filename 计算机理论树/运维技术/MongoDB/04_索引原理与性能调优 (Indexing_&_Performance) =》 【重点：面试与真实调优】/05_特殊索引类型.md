
## 4.5 特殊索引类型

MongoDB 针对特定业务场景，提供了专用的特殊索引类型，可解决常规 B-Tree 索引无法覆盖的业务需求，以下为生产环境最常用的四类特殊索引。

### 4.5.1 TTL 索引（Time-To-Live Index）

TTL 索引是特殊的单键索引，用于实现文档的自动过期删除，是 MongoDB 原生的临时数据生命周期管理方案。

- **标准创建语法**：`db.collection.createIndex({createTime: 1}, {expireAfterSeconds: 3600})`
    
    - 索引字段必须为`ISODate`日期类型，禁止使用数值、字符串类型字段。
    - `expireAfterSeconds`：文档过期时间，单位为秒，文档过期条件为：当前系统时间 > 字段值 + 过期秒数。
    
- **核心实现原理**：MongoDB 后台启动专用 TTL 线程，每 60 秒执行一次全量扫描，删除索引中匹配过期条件的文档，无需业务层手动执行清理任务。
- **核心约束**：
    
    1. 仅支持单日期字段索引，不支持复合索引、多键索引。
    2. 禁止为`_id`字段、固定集合（Capped Collection）创建 TTL 索引。
    3. 过期删除非精确实时，存在最大 60 秒的延迟，海量过期数据场景延迟会进一步增加。
    4. 过期删除操作记录完整 Oplog，会同步到副本集从节点，主从延迟场景下需关注数据一致性。
    
- **适用场景**：短信验证码、用户 Session、临时日志、缓存数据、一次性临时文件等有明确生命周期的临时数据。

### 4.5.2 唯一索引（Unique Index）

唯一索引用于强制索引字段的值在集合内全局唯一，禁止重复数据写入，是 MongoDB 实现数据唯一性约束的核心方案。

- **标准创建语法**：
    
    - 单键唯一索引：`db.collection.createIndex({userId: 1}, {unique: true})`
    - 复合唯一索引：`db.collection.createIndex({userId: 1, date: 1}, {unique: true})`
        
        - 复合唯一索引约束的是字段组合的全局唯一性，而非单个字段的唯一性。
        
    
- **核心特性与约束**：
    
    1. 对于缺失索引字段的文档，MongoDB 会将其值视为`null`，因此集合内最多只能有一个文档缺失该字段，否则会触发唯一约束冲突。
    2. 数组字段的唯一索引，会同时约束单文档内的数组元素不可重复，以及集合内所有文档的数组元素不可重复。
    3. 唯一索引同时具备普通 B-Tree 索引的查询加速能力，兼顾约束与性能。
    4. 复合唯一索引最多只能包含一个多键（数组）字段。
    
- **适用场景**：用户 ID、手机号、邮箱、订单号、设备编号等唯一标识字段，从数据库层面彻底避免重复数据写入。

### 4.5.3 文本索引（Text Index）

文本索引是针对字符串类型字段创建的特殊索引，用于实现 MongoDB 原生的全文检索功能，支持分词、权重匹配、相关性评分排序。

- **标准创建语法**：
    
    - 单字段文本索引：`db.collection.createIndex({content: "text"})`
    - 多字段加权文本索引：`db.collection.createIndex({title: "text", content: "text"}, {weights: {title: 10, content: 1}})`
        
        - `weights`用于设置字段的权重，权重越高，字段匹配对相关性评分的影响越大。
        
    
- **核心特性与约束**：
    
    1. 一个集合最多只能创建一个文本索引，但可包含多个字符串类型字段。
    2. 内置基于空格与标点符号的分词器，支持英文分词，对中文分词支持极差，中文场景不推荐使用。
    3. 配合`$text`查询操作符，可实现关键词匹配、排除匹配、短语匹配，查询结果默认按相关性评分降序排序。
    4. 无法支持覆盖查询，必须回表读取文档计算相关性评分。
    
- **适用场景**：轻量级英文内容全文检索、站内简单搜索场景；复杂中文全文检索、高并发搜索场景，推荐通过数据同步对接 Elasticsearch 或使用 MongoDB Atlas Search。

### 4.5.4 地理空间索引（2dsphere Index）

2dsphere 地理空间索引是针对 GeoJSON 格式地理空间数据创建的特殊索引，适配地球球面几何模型，用于实现 LBS 场景的精准地理空间查询。

- **标准创建语法**：`db.collection.createIndex({location: "2dsphere"})`
    
    - 索引字段必须为合法的 GeoJSON 格式，核心常用类型为`Point`（点位），格式为`{type: "Point", coordinates: [经度, 纬度]}`，**必须严格遵循先经度后纬度的顺序，否则会导致查询结果完全错误**。
    
- **核心支持能力**：
    
    1. 支持 GeoJSON 全类型：`Point`、`LineString`、`Polygon`、`MultiPoint`、`MultiPolygon`等。
    2. 核心查询操作符：
        
        - `$geoWithin`：查询位于指定多边形区域内的文档，适配城市区域、地理围栏场景。
        - `$geoIntersects`：查询与指定地理图形相交的文档，适配路线、区域覆盖场景。
        - `$nearSphere`：查询距离指定点位最近的文档，按球面距离排序，支持最大 / 最小距离限制，是 “附近的人”、“附近商家” 场景的标准实现方案。
        
    
- **适用场景**：LBS 应用、外卖商家定位、网约车位置匹配、地理围栏、区域风控、位置打卡等地理空间相关业务场景。

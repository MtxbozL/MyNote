
>复合索引是针对文档中多个字段组合创建的索引，是多条件查询、多字段排序场景的核心优化手段，其字段顺序直接决定索引的可用性与效率，是索引设计的核心重点。

### 4.3.1 核心定义与创建规则

- **标准创建语法**：`db.collection.createIndex({field1: 1, field2: -1, field3: 1})`
- **核心规则**：复合索引的字段顺序决定了索引的结构，索引键按照创建时的字段顺序，先按第一个字段排序，第一个字段值相同的文档，再按第二个字段排序，以此类推。索引的有序性仅在字段顺序的前缀上成立，是后续最左前缀原则与 ESR 原则的底层基础。

### 4.3.2 最左前缀原则（Leftmost Prefix Principle）

最左前缀原则是复合索引的核心生效规则，MongoDB 仅能利用复合索引的**最左连续前缀**完成查询匹配、排序操作，只有查询条件中包含索引的最左前缀字段，才能触发索引扫描。

1. **前缀生效规则**：对于复合索引`{a: 1, b: 1, c: 1}`，仅以下连续前缀可被索引完整支持：
    
    - 单字段前缀：`{a}`
    - 双字段前缀：`{a, b}`
    - 全字段前缀：`{a, b, c}`
    
2. **核心约束**：
    
    - 跳过中间字段的查询，仅能利用最左连续前缀，后续字段无法利用索引。例如查询`{a: 1, c: 1}`，仅能利用`a`的前缀过滤，`c`的条件无法通过索引匹配，只能在回表后过滤。
    - 查询条件的字段顺序不影响前缀匹配，MongoDB 查询优化器会自动调整字段顺序匹配索引前缀，例如`{b: 1, a: 1}`与`{a: 1, b: 1}`均可完整利用`{a, b}`前缀。
    - 排序操作必须符合最左前缀规则，才能利用索引的有序性避免内存排序。例如索引`{a:1, b:1, c:1}`，排序`{a:1, b:1}`可利用索引，排序`{b:1}`、`{a:1, c:1}`均无法利用索引完成全量排序。
    

### 4.3.3 ESR 原则：复合索引设计黄金准则

ESR 原则是 MongoDB 官方推荐的复合索引字段排序准则，是生产环境复合索引设计的铁律，可最大化索引的过滤效率与查询性能，彻底避免内存排序。其核心排序规则为：**Equality (等值匹配字段) → Sort (排序字段) → Range (范围匹配字段)**。

#### 4.3.3.1 规则拆解

1. **E - Equality（等值匹配字段）**
    
    - 定义：所有使用`$eq`、`$in`等值操作符的查询字段，必须放在复合索引的最左侧，作为最左前缀。
    - 设计逻辑：等值匹配可最大化索引的过滤效率，先将数据范围缩小到最小集合，减少后续扫描的索引条目数量；多个等值字段的顺序可按过滤效率调整，过滤效率越高的字段越靠前。
    
2. **S - Sort（排序字段）**
    
    - 定义：所有排序字段必须紧跟在等值匹配字段之后，范围匹配字段之前。
    - 设计逻辑：范围匹配会破坏索引后续字段的有序性，将排序字段放在范围字段之前，可保证排序字段在等值匹配的结果集中完全有序，直接利用索引避免内存排序，彻底规避 32MB 内存排序的硬限制。
    
3. **R - Range（范围匹配字段）**
    
    - 定义：所有使用`$gt`、`$gte`、`$lt`、`$lte`、`$nin`等范围操作符的字段，必须放在复合索引的最末尾。
    - 设计逻辑：范围字段之后的所有索引字段，无法满足最左连续前缀规则，索引会完全失效；将范围字段放在最后，可保证前面的等值、排序字段完整利用索引的有序性。
    

#### 4.3.3.2 正反案例对比

- **业务场景**：查询状态为`active`、年龄在 18-30 岁之间的用户，按创建时间降序排序。
    
    - 查询语句：`db.user.find({status: "active", age: {$gte: 18, $lte: 30}}).sort({createTime: -1})`
    
- **错误索引设计**：`{status: 1, age: 1, createTime: -1}`
    
    - 问题：范围字段`age`放在排序字段`createTime`之前，`age`的范围匹配破坏了后续`createTime`的有序性，排序操作无法利用索引，会触发内存排序，海量数据下直接报错。
    
- **正确索引设计**：`{status: 1, createTime: -1, age: 1}`
    
    - 优势：完全符合 ESR 原则，等值字段`status`在前，排序字段`createTime`在中，范围字段`age`在后；查询可完整利用索引完成过滤与排序，无内存排序，索引过滤效率最大化。

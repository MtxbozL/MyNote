
## 4.7 覆盖查询（Covered Query）

覆盖查询是 MongoDB 查询优化的终极手段之一，指**查询的过滤条件、排序字段、投影返回字段，全部包含在索引中**，MongoDB 仅通过索引扫描即可完成整个查询，无需执行回表（FETCH）操作，彻底消除回表的磁盘 IO 开销。

### 4.7.1 覆盖查询的核心生效条件

必须同时满足以下所有条件，才能实现覆盖查询：

1. 查询的所有过滤条件字段，全部包含在索引中，且符合最左前缀原则。
2. 查询的所有排序字段，全部包含在索引中，且与索引的顺序、排序方向完全匹配。
3. 投影返回的所有字段，全部包含在索引中，禁止返回索引以外的字段（`_id`字段除外，二级索引的叶子节点天然存储`_id`值）。
4. 索引不能是多键索引、文本索引、地理空间索引等特殊索引，仅支持普通 B-Tree 单键 / 复合索引。
5. 查询禁止使用`$expr`、`$where`、`$function`等需要访问完整文档的操作符。

### 4.7.2 正反案例对比

- **索引定义**：`db.user.createIndex({name: 1, age: 1, gender: 1})`
- **覆盖查询示例**：
    
    javascript
    
    运行
    
    ```
    db.user.find(
      {name: "张三", age: {$gte: 18}},
      {name: 1, age: 1, gender: 1, _id: 0}
    )
    ```
    
    - 生效说明：过滤条件`name`、`age`在索引中，返回字段全部包含在索引中，排除了`_id`字段，无需回表，`totalDocsExamined`为 0，完全覆盖。
    
- **非覆盖查询示例**：
    
    javascript
    
    运行
    
    ```
    db.user.find(
      {name: "张三"},
      {name: 1, age: 1, address: 1, _id: 0}
    )
    ```
    
    - 失效说明：返回字段`address`不在索引中，必须回表读取完整文档，无法实现覆盖查询。
    

### 4.7.3 核心性能优势

1. **IO 开销最小化**：将两次 IO（索引扫描 + 回表）减少为一次 IO（仅索引扫描），查询性能可提升一个数量级。
2. **缓存利用率最大化**：索引体积远小于完整文档体积，可完全加载到内存缓存中，避免磁盘 IO，高频查询场景下性能优势极其显著。
3. **无锁竞争**：仅访问索引文件，无需访问主数据文件，减少了存储引擎的锁竞争，提升高并发场景下的吞吐量。

## 4.8 索引设计与调优最佳实践

1. **以查询模式为核心设计索引**：先梳理业务的高频查询、排序场景，再针对性设计索引，禁止为低频查询、无查询场景的字段创建无效索引。
2. **严格控制索引数量**：单集合的索引数量建议不超过 10 个，高并发写入场景需进一步压缩；每个索引都会增加写入开销，冗余索引会严重降低写入吞吐量。
3. **优先使用复合索引，避免单键索引泛滥**：MongoDB 一次查询最多只能使用一个索引（`$or`查询的每个分支可使用不同索引），复合索引可通过最左前缀原则覆盖多个查询场景，效率远高于多个单键索引。
4. **严格遵循 ESR 原则设计复合索引**：彻底避免内存排序，最大化索引的过滤效率。
5. **避免低基数字段创建索引**：性别、状态等可选值极少的低基数字段，过滤效率极低，索引扫描的条目数与全表扫描几乎一致，反而会增加写入开销，禁止单独创建索引。
6. **定期清理冗余索引**：例如已有复合索引`{a:1, b:1}`，则单键索引`{a:1}`为冗余索引，其查询场景已被复合索引的最左前缀完全覆盖，需定期清理，减少写入开销。
7. **生产环境规范创建索引**：MongoDB 4.2 + 版本默认采用优化的增量索引构建方式，不会长时间锁集合，但仍需在业务低峰期执行；禁止在业务高峰期创建大集合的索引，避免影响业务可用性。


Python 的垃圾回收（Garbage Collection, GC）是内存管理的核心底层机制，负责自动回收不再使用的对象内存，避免内存泄漏。Python 的 GC 体系遵循**引用计数为主，标记 - 清除与分代回收为辅**的核心架构。

### 5.6.1 Python 对象的内存底层

Python 中「一切皆对象」，所有对象的底层都是 C 语言结构体，核心基础结构体为`PyObject`，其中包含两个核心字段：

1. `ob_refcnt`：引用计数器，记录对象被引用的次数，是引用计数机制的核心；
2. `ob_type`：指向对象类型的指针，记录对象的类型信息。

所有对象的内存分配与回收，都基于该结构体的引用计数器与 GC 机制管理。

### 5.6.2 引用计数：核心回收机制

引用计数是 Python 默认的、实时的垃圾回收机制，贯穿 Python 运行的全流程。

#### 核心原理

每个对象都有一个引用计数器，当对象的引用计数变为 0 时，该对象会被立即销毁，其占用的内存会被释放。

#### 引用计数增减规则

1. **引用计数 + 1 的场景**：
    
    - 对象被创建，如`a = 10`；
    - 对象被其他变量引用，如`b = a`；
    - 对象作为参数传入函数，如`func(a)`；
    - 对象作为容器的元素，如`lst = [a, 20]`。
    
2. **引用计数 - 1 的场景**：
    
    - 变量被显式`del`删除，如`del a`；
    - 变量被重新赋值，如`a = 20`，原对象的引用计数 - 1；
    - 函数执行结束，局部变量被销毁，参数引用计数 - 1；
    - 对象从容器中移除，或容器本身被销毁。
    

可通过`sys.getrefcount(obj)`函数获取对象的当前引用计数，注意：调用该函数时，会临时为对象创建一个引用，因此返回值比实际值大 1。

#### 引用计数的优缺点

- 优点：
    
    1. 实时性：引用计数为 0 时立即回收，内存释放及时；
    2. 实现简单，逻辑清晰；
    3. 最大暂停时间短，无全局扫描的长停顿。
    
- 致命缺陷：**无法处理循环引用（交叉引用）**，这也是需要标记 - 清除与分代回收作为辅助机制的核心原因。

循环引用示例：

```python
# 创建两个对象，互相引用，形成循环引用
a = [1]
b = [2]
a.append(b)
b.append(a)

# 删除两个变量，引用计数仍为1（互相引用），无法被引用计数回收
del a
del b
```

上述代码中，`del a`和`del b`执行后，两个列表对象的引用计数都变为 1（互相引用），永远无法变为 0，引用计数机制无法回收，造成内存泄漏。

### 5.6.3 标记 - 清除：循环引用解决方案

标记 - 清除（Mark and Sweep）是 Python 的辅助 GC 机制，专门用于解决容器对象的循环引用问题，仅针对可能产生循环引用的容器对象（列表、字典、元组、类实例等）。

#### 核心原理

分为两个阶段执行：

1. **标记阶段**：从「根对象」（全局变量、栈中的变量、寄存器中的对象等，永远不会被回收的对象）出发，沿着引用链遍历所有可达的对象，将其标记为「存活对象」；无法被根对象访问到的对象，即使存在循环引用，也被标记为「垃圾对象」。
2. **清除阶段**：遍历所有容器对象，将未被标记为存活的垃圾对象全部销毁，释放其占用的内存。

#### 触发时机

标记 - 清除不会实时执行，当分代回收的阈值达到时，会自动触发，也可通过`gc.collect()`手动触发。

### 5.6.4 分代回收：GC 效率优化

分代回收基于「弱代假说」：绝大多数对象的生命周期都很短，存活时间越久的对象，越不可能是垃圾，越不容易被回收。分代回收的核心作用是**降低标记 - 清除的扫描开销，提升 GC 的整体效率**。

#### 核心原理

Python 将所有容器对象分为 3 个代（Generation），对应 3 个链表，按存活时间从短到长排序：

1. **0 代（年轻代）**：新创建的对象都归入 0 代，生命周期最短，GC 扫描频率最高；
2. **1 代（中年代）**：在 0 代的 GC 中存活下来的对象，会被晋升到 1 代，扫描频率次之；
3. **2 代（老年代）**：在 1 代的 GC 中存活下来的对象，会被晋升到 2 代，扫描频率最低，存放生命周期极长的对象。

#### 回收阈值与触发规则

Python 为每一代设置了回收阈值，可通过`gc.get_threshold()`获取，默认阈值为`(700, 10, 10)`，规则如下：

1. 当 0 代的对象数量达到 700 时，触发 0 代 GC，同时扫描 0 代对象；
2. 每执行 10 次 0 代 GC，触发 1 次 1 代 GC，同时扫描 1 代 + 0 代对象；
3. 每执行 10 次 1 代 GC，触发 1 次 2 代 GC，同时扫描 2 代 + 1 代 + 0 代对象，即全量 GC。

#### gc 模块：手动控制 GC

Python 内置`gc`模块，用于手动管理垃圾回收机制，高频 API 如下：

- `gc.enable()` / `gc.disable()`：开启 / 关闭 GC 机制；
- `gc.isenabled()`：判断 GC 是否开启；
- `gc.set_threshold(threshold0, threshold1, threshold2)`：修改三代 GC 的触发阈值；
- `gc.collect(generation=2)`：手动触发 GC，默认执行全量 GC，返回回收的垃圾对象数量。

### 5.6.5 核心注意事项

循环引用中，若对象实现了`__del__`析构方法，标记 - 清除机制无法处理该循环引用，会将对象放入`gc.garbage`列表中，无法自动回收，造成永久性内存泄漏。因此，应尽量避免在可能产生循环引用的对象中实现`__del__`方法。

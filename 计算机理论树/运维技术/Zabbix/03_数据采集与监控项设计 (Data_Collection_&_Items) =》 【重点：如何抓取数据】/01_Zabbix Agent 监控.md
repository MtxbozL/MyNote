## 3.1 章节核心定位

数据采集是 Zabbix 监控体系的 **数据源入口与核心基础层** ，所有后续的异常判断、告警通知、可视化展示、趋势分析均依赖于高质量、高可靠的采集数据。本章系统阐述 Zabbix 全维度采集能力的技术原理、配置规范、适用场景与运维方法，覆盖从标准化主机采集到无代理异构设备监控、从周期轮询到主动推送、从原生指标到自定义衍生指标的全场景采集方案，是 Zabbix 落地实施的核心技术模块。

监控项（Item）是 Zabbix 采集体系的最小原子单元，其定义了单一指标的采集规则、数据类型、存储策略与更新周期，所有采集能力最终均通过监控项实现标准化落地。


## 3.2 Zabbix Agent 监控

Zabbix Agent/Agent 2 是部署在被监控端的轻量级采集代理，是操作系统级精细化监控的首选方案，兼容被动、主动两种工作模式（详见 1.4 节），分为 **原生内置 Key 采集** 与 **UserParameter 自定义扩展采集** 两大能力体系。

### 3.2.1 内置 Key 采集

#### 3.2.1.1 核心定义与语法规范

内置 Key 是 Zabbix Agent 原生预定义、无需额外开发的标准化采集指标标识符，覆盖操作系统全维度基础指标，具备开箱即用、低资源占用、高可靠性的特点，是主机监控的核心采集载体。

内置 Key 的标准语法格式为：

```shell
key[<parameter1>,<parameter2>,...,<parameterN>]
```

- 核心规则：Key 名称为固定标识符，大小写敏感；参数为可选配置，用于限定采集范围、指标类型、统计周期等；无参数的 Key 可省略方括号。
- 数据返回约束：所有内置 Key 均返回标准化的单行数据，支持数值型、字符型、日志型、文本型四大类数据类型，与监控项的数据类型严格匹配。

#### 3.2.1.2 核心内置 Key 分类与详解

按照监控维度，内置 Key 分为六大类，覆盖 Linux/Windows/Unix 全平台操作系统核心指标，核心高频 Key 如下：

|指标分类|核心 Key|功能说明|关键参数说明|
|---|---|---|---|
|系统通用|`system.uname`|获取操作系统内核版本、主机名等系统信息|无参数|
||`system.hostname`|获取主机名|可选参数指定主机名类型（短名 / 全限定名）|
|CPU 监控|`system.cpu.util`|获取 CPU 利用率百分比|第 1 参数：CPU 核心编号（空为所有核心）；第 2 参数：CPU 状态（user/system/iowait/idle 等）；第 3 参数：统计时间窗口|
||`system.cpu.load`|获取 CPU 平均负载|第 1 参数：负载时间窗口（1m/5m/15m）；第 2 参数：CPU 核心编号|
|内存监控|`vm.memory.size`|获取内存各类状态的容量|参数：内存类型（total/available/used/free/buffers/cached 等），支持百分比后缀|
||`vm.swap.size`|获取交换分区使用情况|参数：swap 类型（total/free/used），支持百分比后缀|
|磁盘文件系统|`vfs.fs.size`|获取文件系统磁盘容量|第 1 参数：挂载点路径；第 2 参数：容量类型（total/free/used/pfree/pused）|
||`vfs.fs.inode`|获取文件系统 inode 使用情况|第 1 参数：挂载点路径；第 2 参数：类型（total/free/used/pfree/pused）|
||`vfs.dev.read`/`vfs.dev.write`|获取磁盘读写吞吐量、IOPS|第 1 参数：设备名；第 2 参数：统计类型（sectors/operations/sps/ops）；第 3 参数：时间窗口|
|网络监控|`net.if.in`/`net.if.out`|获取网卡入 / 出方向流量|第 1 参数：网卡名称；第 2 参数：统计类型（bytes/packets/errors/dropped）|
||`net.tcp.listen`|检测 TCP 端口是否处于监听状态|参数：端口号，返回 1 为监听中，0 为未监听|
|进程监控|`proc.num`|统计指定进程的运行数量|第 1 参数：进程名；第 2 参数：运行用户；第 3 参数：运行状态|
||`proc.mem`|统计指定进程的内存占用|参数与 proc.num 一致，返回进程总内存占用字节数|

#### 3.2.1.3 版本适配说明

- Zabbix Agent（C 语言版）：内置 Key 仅覆盖操作系统基础指标，中间件、容器等场景需通过 UserParameter 扩展；
- Zabbix Agent 2（Golang 版）：在原生内置 Key 基础上，通过插件化体系扩展了 MySQL、Redis、Docker、Kubernetes 等数十种组件的原生采集 Key，无需额外脚本即可实现中间件监控，是当前官方主推的采集代理。

### 3.2.2 UserParameter 自定义参数

#### 3.2.2.1 核心定义与语法规范

UserParameter 是 Zabbix Agent 提供的**标准化自定义采集扩展机制**，允许用户通过 Shell、Python、Perl 等脚本语言，实现非标准化、业务专属指标的采集，是 Agent 采集能力的核心扩展入口。

UserParameter 的标准配置语法分为两种：

1. 无参数固定格式：
    
    plaintext
    
    ```
    UserParameter=<key_name>,<command>
    ```
    
    - `key_name`：自定义监控项 Key，需符合 Zabbix 命名规范，不可与内置 Key 重名；
    - `command`：待执行的系统命令或脚本路径，脚本的标准输出（stdout）作为 Key 的返回值。
    
2. 带参数动态格式：
    
    plaintext
    
    ```
    UserParameter=<key_name[*]>,<command $1 $2 ... $N>
    ```
    
    - `key_name[*]`：支持动态传参的 Key 定义，方括号内的`*`为参数占位符；
    - `$1-$N`：对应 Key 中传入的第 1 到第 N 个参数，支持参数转义与边界校验。
    

#### 3.2.2.2 配置与生效规则

1. 配置位置：可直接写入`zabbix_agent2.conf`主配置文件，或写入`Include`指令指定的独立配置文件（生产环境推荐独立配置，便于管理）；
2. 生效条件：配置修改后，必须重启 Zabbix Agent/Agent 2 服务方可生效；
3. 权限约束：Agent 运行用户必须具备脚本的执行权限与相关资源的访问权限，禁止使用 root 用户运行 Agent，需通过最小权限原则配置脚本权限；
4. 返回值约束：脚本必须仅向 stdout 输出单行有效数据，最大长度不超过 512KB；stderr 输出仅用于故障排查，不影响返回值；脚本退出码非 0 时，Key 返回`ZBX_NOTSUPPORTED`。

#### 3.2.2.3 生产环境最佳实践与安全约束

1. 禁止在脚本中使用高风险命令，禁止直接接收用户传入的未过滤参数，避免命令注入漏洞；
2. 脚本执行超时时间不得超过 Agent 配置的`Timeout`参数最大值（30s），避免阻塞 Agent 采集调度；
3. 脚本需包含异常处理逻辑，异常场景下返回标准化错误值，避免监控项变为不支持状态；
4. 生产环境需对自定义脚本进行版本管理与权限管控，禁止随意修改脚本内容；
5. 解决环境变量问题：Agent 运行的非交互式 shell 环境变量与用户交互式环境不同，脚本中需使用绝对路径引用命令与资源。

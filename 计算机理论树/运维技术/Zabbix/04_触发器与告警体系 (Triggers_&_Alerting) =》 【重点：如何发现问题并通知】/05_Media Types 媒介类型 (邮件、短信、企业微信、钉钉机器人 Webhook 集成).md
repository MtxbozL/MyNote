
## 4.6 Media Types 媒介类型与通知渠道集成

Media Types（媒介类型）是 Zabbix 与外部通知渠道之间的**适配层与协议转换层**，定义了 Zabbix 向外部系统发送告警通知的通信协议、数据格式、认证方式，是实现多渠道告警通知的核心载体。

### 4.6.1 核心定义与分类

媒介类型的核心原理是：将 Zabbix 的事件数据、告警信息，按照通知渠道要求的格式进行封装，通过对应的协议发送至通知渠道，实现告警信息的触达。

Zabbix 媒介类型分为三大类，覆盖全场景通知需求：

1. **内置原生媒介类型**：系统原生支持的标准化媒介类型，无需额外开发，开箱即用，包括邮件、短信、脚本；
2. **Webhook 媒介类型**：基于 HTTP/HTTPS 协议的自定义媒介类型，通过 JavaScript 脚本实现数据格式转换，适配所有支持 Webhook 的通知渠道，是当前企业级场景的主流方案；
3. **第三方插件媒介类型**：通过第三方插件扩展的媒介类型，适配专用的告警通知平台、电话呼叫系统等。

### 4.6.2 内置原生媒介类型

#### 4.6.2.1 邮件媒介类型

邮件是 Zabbix 最基础、最常用的通知渠道，原生支持两种邮件发送模式：

1. **SMTP 模式**：通过标准 SMTP 协议发送邮件，支持 SSL/TLS 加密、身份认证，是生产环境推荐的模式，兼容所有企业邮箱、公共邮箱服务；
2. **Sendmail 模式**：通过服务器本地的 Sendmail/Postfix 服务发送邮件，仅适用于内网邮件服务器场景，生产环境不推荐。

核心配置规范：

- 生产环境必须使用 SSL/TLS 加密传输，禁止使用明文 SMTP 协议，避免邮件内容泄露；
- 配置专用的告警发件邮箱，禁止使用个人邮箱；
- 配置邮件发送重试机制，避免单次发送失败导致的告警遗漏；
- 邮件主题必须标准化，包含异常等级、主机名、异常描述，便于快速识别与过滤。

#### 4.6.2.2 短信媒介类型

短信是高优先级告警的核心通知渠道，原生支持两种实现模式：

1. **GSM Modem 模式**：通过串口连接 GSM 调制解调器，直接发送短信，适用于无互联网的内网隔离环境；
2. **Webhook / 脚本模式**：通过调用运营商、第三方短信服务商的 API 接口发送短信，是当前企业级场景的主流方案。

核心配置规范：

- 仅用于 High 及以上等级的告警通知，禁止低等级告警发送短信，避免短信轰炸；
- 配置短信内容长度限制，单条短信不超过 70 个汉字，避免拆分成多条短信；
- 配置发送频率限制，同一主机同一告警 1 小时内发送短信不超过 3 条，避免重复发送。

#### 4.6.2.3 脚本媒介类型

脚本媒介类型是 Zabbix 的原生扩展能力，通过自定义 Shell/Python/PHP 脚本，实现告警通知的自定义处理，适配各类非标通知渠道。

- 核心原理：Zabbix 将告警信息作为参数传递给自定义脚本，脚本执行对应的逻辑，完成告警发送；
- 配置规范：脚本必须放置在 Zabbix Server 配置的`AlertScriptsPath`指定目录下，具备执行权限；
- 适用场景：非标通知渠道、定制化告警处理、多渠道聚合通知等场景。

### 4.6.3 Webhook 媒介类型

Webhook 媒介类型是 Zabbix 4.0 + 版本主推的通知渠道适配方案，原生支持 HTTP/HTTPS 协议，通过内置的 JavaScript 脚本引擎实现数据格式转换，无需依赖服务器本地脚本，配置便捷、扩展性强，是企业微信、钉钉、飞书、Slack 等即时通讯工具告警集成的首选方案。

#### 4.6.3.1 核心原理与执行流程

1. 告警触发时，Zabbix 将事件数据、宏变量解析后的告警内容，传入 Webhook 对应的 JavaScript 脚本；
2. JavaScript 脚本将 Zabbix 的告警数据，转换为目标通知渠道要求的 JSON/XML 格式；
3. 脚本通过内置的 HTTP 请求函数，向目标渠道的 Webhook 地址发送 HTTP 请求，完成告警通知发送；
4. 脚本处理目标渠道的响应结果，返回发送状态，便于 Zabbix 记录发送日志。

#### 4.6.3.2 企业级主流渠道集成规范

当前国内企业最常用的三大即时通讯工具 Webhook 集成核心要点如下：

表格

|通知渠道|核心配置要点|安全规范|
|---|---|---|
|企业微信|1. 创建企业微信机器人，获取 Webhook 地址；2. 配置消息类型为 markdown，支持富文本格式；3. 支持 @指定人员、@全员功能|配置 IP 白名单，仅允许 Zabbix Server 地址访问机器人 Webhook；禁止在公网暴露 Webhook 地址|
|钉钉|1. 创建钉钉群机器人，获取 Webhook 地址与加签密钥；2. 必须配置签名校验，兼容钉钉的安全机制；3. 支持 markdown 格式、@人员功能|强制开启加签校验，禁止使用关键词白名单的低安全模式；配置 IP 白名单|
|飞书|1. 创建飞书群机器人，获取 Webhook 地址；2. 支持签名校验与富文本消息格式；3. 支持消息卡片、互动按钮等进阶功能|开启签名校验，配置 IP 白名单，严格管控机器人权限|

#### 4.6.3.3 生产环境最佳实践

1. 所有 Webhook 必须配置 HTTPS 协议，禁止使用 HTTP 明文传输，避免告警信息泄露；
2. 优先使用官方提供的标准化 Webhook 模板，禁止自定义脚本中包含高风险代码；
3. 配置超时时间与重试机制，单次请求超时时间不超过 10s，失败重试不超过 3 次；
4. 开启 Webhook 发送日志，记录每次请求的发送时间、响应状态、错误信息，便于故障排查；
5. 告警内容必须标准化，包含异常等级、主机信息、异常时间、异常描述、排查建议，提升处置效率。

### 4.6.4 媒介类型与用户配置关联

媒介类型仅定义了通知渠道的通信规则，告警通知的最终触达，需要将媒介类型与用户配置进行关联，核心配置要点：

1. 每个用户必须配置对应的媒介类型，指定接收通知的邮箱、手机号、Webhook 地址等接收信息；
2. 配置用户的通知时间规则，定义不同媒介类型、不同告警等级的接收时间窗口，如非工作时间仅接收灾难级告警的短信通知；
3. 通过用户组实现批量配置，同一团队的用户加入同一用户组，Action 中直接指定用户组作为接收人，无需单独配置每个用户；
4. 生产环境必须配置用户权限最小化，用户仅能接收其权限范围内的主机告警，避免跨业务线的告警泛滥。

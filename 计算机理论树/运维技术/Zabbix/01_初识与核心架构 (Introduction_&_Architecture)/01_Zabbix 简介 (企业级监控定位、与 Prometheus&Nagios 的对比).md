## 1.1 Zabbix 简介与企业级监控定位

### 1.1.1 Zabbix 核心定义

Zabbix 是一款开源、跨平台、分布式的企业级 IT 基础设施与业务监控系统，由 Alexei Vladishev 于 1998 年发起研发，已形成标准化全栈监控技术体系。其核心面向数据中心、云计算、物联网等场景下的 IT 资源全生命周期监控需求，具备数据采集、存储、分析、可视化、告警闭环、自动化运维的全链路能力。

### 1.1.2 企业级监控定位

Zabbix 的核心定位是 **大一统企业级监控平台** ，解决传统监控工具碎片化、扩展性不足、自动化能力弱、跨场景适配性差的行业痛点，核心满足企业级监控四大刚性诉求：

1. 全栈覆盖：支持从硬件层、操作系统层、网络层、中间件层、应用层到业务层的全维度监控；
2. 高可扩展性：支持分布式架构横向扩展，可适配从数十台到数万台监控节点的规模；
3. 闭环运维：实现从异常发现、告警通知、自动处置到事后复盘的全流程闭环；
4. 标准化与复用：通过模板、宏、自动发现机制实现监控配置的标准化与大规模复用。

### 1.1.3 主流监控系统横向对比

针对企业级监控领域主流开源方案，从核心技术维度进行客观对比，明确 Zabbix 的技术边界与适用场景：

|对比维度|Zabbix|Prometheus|Nagios|
|---|---|---|---|
|核心架构模型|中心化 + 分布式代理架构，原生支持主备高可用|去中心化 Pull/Push 架构，依赖联邦集群实现扩展|中心化插件式架构，原生分布式能力薄弱|
|数据采集能力|原生支持 Agent、SNMP、JMX、IPMI、HTTP、Trapper 等 10 + 种采集方式，无 Agent 场景适配完善|原生基于 Exporter 的 HTTP Pull 采集，Push 能力需额外组件支持，采集方式相对单一|基于插件脚本实现采集，原生采集能力弱，高度依赖第三方插件扩展|
|数据存储模型|以关系型数据库 (MySQL/PostgreSQL) 为核心，支持 TimescaleDB 时序插件扩展，配置数据与时序数据分离存储|原生内置时序数据库 (TSDB)，专为指标数据优化，长期存储依赖远程存储扩展|无原生时序存储能力，仅存储状态与事件数据，历史数据需第三方插件支持|
|告警与自动化|原生支持触发器表达式、告警依赖、告警升级、远程命令自动处置，闭环能力完善|基于 AlertManager 实现告警聚合与路由，原生无告警升级与自动处置能力|原生仅支持基础告警，告警依赖、升级、自动化能力需大量二次开发实现|
|核心适用场景|传统数据中心、混合云、跨机房等稳态 IT 基础设施全栈监控，适配中小规模到超大规模集群|云原生、容器化、微服务等动态 IT 场景的指标监控，深度适配 Kubernetes 生态|小规模基础设施的基础可用性监控，大规模场景下扩展性严重不足|

## 1.4 核心工作模式

Zabbix Agent/Agent 2 与 Zabbix Server/Proxy 之间的通信分为 **被动模式 (Passive)**  与 **主动模式 (Active)**  两种核心工作模式，二者的本质差异在于采集任务的调度权归属与通信发起方，适配不同的监控场景。

### 1.4.1 被动模式 (Passive Check)

被动模式是 Zabbix 的默认工作模式，核心特征是 **Zabbix Server/Proxy 主导采集调度，主动发起数据拉取请求**。

- 工作流程：
    
    1. Zabbix Server/Proxy 根据监控项配置，向被监控端的 Agent 发起数据采集请求；
    2. Agent 接收到请求后，执行对应的指标采集逻辑，将采集结果返回给 Server/Proxy；
    3. Server/Proxy 接收到数据后，完成数据存储与分析处理。
    
- 核心特性：调度权完全归属 Server/Proxy，Agent 仅响应请求，无自主调度能力；每个监控项的采集均需 Server/Proxy 发起一次独立 TCP 请求，大规模监控场景下会给 Server 带来较大的网络与性能压力；
- 适用场景：小规模监控集群、监控节点与 Server 同属一个内网、网络延迟极低的稳定环境。

### 1.4.2 主动模式 (Active Check)

主动模式是 Zabbix 面向大规模、跨网络场景优化的工作模式，核心特征是 **Zabbix Agent 主导采集调度，主动向 Server/Proxy 上报采集数据**。

- 工作流程：
    
    1. Agent 启动后，主动与 Server/Proxy 建立 TCP 连接，请求获取该主机对应的所有主动监控项配置；
    2. Server/Proxy 将该主机的主动监控项配置下发给 Agent，Agent 在本地完成采集任务的调度与执行；
    3. Agent 按照预设的采集周期，将采集到的批量数据主动上报给 Server/Proxy；
    4. Server/Proxy 接收到批量数据后，完成数据存储与分析处理。
    
- 核心特性：调度权归属 Agent，Server/Proxy 仅负责配置下发，无需为每个监控项发起独立请求，大幅降低 Server 的性能与网络压力；支持批量数据上报，网络传输效率更高，适配跨网络、高延迟场景；支持监控配置的自动同步，适配云环境下动态扩容的监控节点；
- 适用场景：大规模监控集群、跨机房 / 跨云监控、网络隔离场景、云环境动态扩容的监控节点。

### 1.4.3 两种模式的本质区别与选型原则

| 核心维度        | 被动模式                | 主动模式              |
| ----------- | ------------------- | ----------------- |
| 调度权归属       | Zabbix Server/Proxy | Zabbix Agent      |
| 通信发起方       | Server/Proxy 主动发起   | Agent 主动发起        |
| 连接模型        | 短连接，单监控项单请求         | 长连接，批量数据上报        |
| Server 性能开销 | 高，随监控项数量线性增长        | 低，仅处理批量上报数据       |
| 网络适配性       | 仅适配低延迟内网环境          | 适配高延迟、跨网络、防火墙隔离环境 |
| 配置生效时效      | 配置变更实时生效            | 配置变更需 Agent 同步后生效 |

选型核心原则：小规模、内网环境可采用被动模式；大规模、跨网络、云原生环境优先采用主动模式，生产环境推荐以主动模式为核心采集模式。

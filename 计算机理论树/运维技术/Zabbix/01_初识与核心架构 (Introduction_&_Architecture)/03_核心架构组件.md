## 1.3 核心架构组件详解

Zabbix 采用松耦合模块化架构，各组件各司其职、协同完成全链路监控能力，核心架构组件分为 5 类，无冗余覆盖监控全流程：

### 1.3.1 Zabbix Server

Zabbix Server 是整个监控系统的 **核心调度与处理中枢** ，采用 C 语言编写，具备极高的处理性能与稳定性。

- 核心职能：负责监控配置管理、采集任务调度、监控数据接收与预处理、异常事件分析判断、告警动作触发、历史数据聚合与存储调度；
- 核心进程模块：内置轮询器 (Poller)、陷阱器 (Trapper)、管家进程 (Housekeeper)、告警发送进程、自动发现进程等核心模块，分别承担对应场景的处理任务；
- 架构定位：整个监控系统的唯一控制中心，所有配置规则、调度逻辑均由 Server 统一管理，是分布式架构的核心节点。

### 1.3.2 Zabbix Database

Zabbix Database 是监控系统的 **持久化存储组件** ，是 Zabbix 架构中唯一的状态存储载体，官方支持 MySQL、PostgreSQL 两种主流关系型数据库，其中 PostgreSQL 可结合 TimescaleDB 时序插件实现存储性能升级。

- 核心存储内容：配置元数据、时序监控数据、事件与审计数据三大类，详见 1.2.2 节；
- 运行机制：Zabbix Server、Web Frontend、Proxy 所有组件的状态同步均依赖数据库，是实现高可用、分布式架构的核心数据底座；
- 架构定位：监控系统的状态存储核心，所有监控数据与配置的唯一持久化载体。

### 1.3.3 Zabbix Web Frontend

Zabbix Web Frontend 是监控系统的 **Web 管理与可视化界面** ，基于 PHP 语言开发，采用 B/S 架构，无状态设计，支持横向扩展部署。

- 核心职能：提供监控配置的可视化管理界面、监控数据的图形化展示、仪表盘定制、事件与告警管理、用户权限管理、系统配置管理等全功能人机交互能力；
- 运行机制：不直接与 Zabbix Agent 通信，仅通过访问 Zabbix Database 获取配置与监控数据，同时调用 Zabbix Server 的 API 接口实现配置管理操作；
- 架构定位：运维人员与监控系统交互的唯一入口，实现监控系统的可操作性与可观测性。

### 1.3.4 Zabbix Agent / Agent 2

Zabbix Agent/Agent 2 是部署在 **被监控对象端的采集代理组件** ，是 Zabbix 最核心的指标采集载体，分为 Agent 与 Agent 2 两个官方版本。

- 核心职能：部署在被监控主机 / 设备上，负责本地指标数据的采集，按照预设规则将采集数据上报给 Zabbix Server/Proxy，同时支持接收 Server 下发的远程命令执行请求；
- 版本核心差异：
    
    1. Zabbix Agent：基于 C 语言编写，轻量、低资源占用，原生支持操作系统基础指标采集，自定义采集能力依赖 UserParameter 扩展；
    2. Zabbix Agent 2：基于 Golang 语言重写，具备高并发、插件化特性，原生支持 Docker、MySQL、Redis 等中间件采集，无需额外脚本扩展，是当前官方主推的采集代理版本；
    
- 架构定位：监控数据的边缘采集端，是实现主机级精细化监控的核心组件。

### 1.3.5 Zabbix Proxy

Zabbix Proxy 是分布式架构中的 **代理采集节点** ，是实现跨网络、大规模监控的核心组件，核心作用是分担 Zabbix Server 的采集与数据处理压力。

- 核心职能：替代 Zabbix Server 完成所辖区域内监控对象的采集任务调度、数据接收与预处理，将采集到的数据批量上报给 Zabbix Server；
- 核心特性：内置本地数据库缓存能力（支持 SQLite/MySQL），当与 Zabbix Server 的网络中断时，可在本地缓存采集的监控数据，网络恢复后自动补传，避免数据丢失；
- 架构定位：分布式监控的边缘汇聚节点，核心解决跨机房、跨云 VPC、防火墙隔离场景的监控适配问题，同时大幅降低中心 Zabbix Server 的性能压力。

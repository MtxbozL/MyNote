# 第八章 分布式监控与高可用集群

## 8.1 章节核心定位

本章是 Zabbix 实现**企业级超大规模、跨地域、高可靠监控部署**的核心架构体系，解决单节点 Zabbix Server 三大核心痛点：单实例性能瓶颈无法支撑万级节点监控、跨机房 / 跨云 / 防火墙隔离场景的网络适配难题、Server 单点故障导致的监控系统整体不可用风险。

本章内容与前序章节深度联动：基于第二章的部署体系、第三章的全场景采集能力、第五章的模板复用机制、第六章的自动化能力，构建 “中心管控 - 边缘采集 - 多活容灾” 的分布式监控架构，完整覆盖三大核心模块：Zabbix Proxy 分布式代理架构、Zabbix Server 高可用集群、Zabbix Agent 2 进阶特性，是 Zabbix 从中小规模测试环境落地到企业级生产环境的核心架构跃迁模块。

本章节核心逻辑链路：**单 Server 性能瓶颈→Proxy 分布式边缘采集分担压力→跨网络隔离场景适配→Server 原生 HA 消除单点故障→Agent 2 进阶能力优化边缘采集效率→大规模集群标准化架构设计**。

## 8.2 Zabbix Proxy 代理架构

### 8.2.1 核心定义与工作原理

#### 8.2.1.1 严谨定义

Zabbix Proxy 是 Zabbix 分布式架构中的**边缘采集与数据转发节点**，是部署在被监控区域、替代 Zabbix Server 完成所辖范围内监控对象的采集任务调度、数据接收预处理、批量上报的专用组件。其核心职能边界为：仅负责数据采集、本地缓存与批量上报，不执行触发器计算、告警触发、动作执行等核心管控逻辑，所有管控能力仍由中心 Zabbix Server 统一管理，是典型的 “管控与采集分离” 分布式架构。

#### 8.2.1.2 完整工作流程

Proxy 与 Server 的协同工作遵循严格的标准化流程，无人工干预即可完成全链路自动化运行：

1. **配置同步阶段**：Proxy 启动后，向中心 Server 请求所辖范围内所有主机的监控配置、采集规则、LLD 发现规则，Server 将对应配置下发给 Proxy，Proxy 在本地完成采集任务的初始化；
2. **数据采集阶段**：Proxy 替代 Server，按照采集规则，向所辖范围内的 Agent / 网络设备 / 监控对象发起采集请求，接收并预处理采集数据，完全替代 Server 的采集调度压力；
3. **本地缓存阶段**：Proxy 将采集到的监控数据写入本地数据库缓存，即使与中心 Server 的网络中断，也不会丢失采集数据；
4. **批量上报阶段**：Proxy 按照预设的上报周期，将本地缓存的采集数据批量压缩上报给中心 Server，Server 接收后完成数据存储、触发器计算、告警触发等后续流程；
5. **配置更新阶段**：Server 端的监控配置发生变更后，会主动同步给对应 Proxy，Proxy 实时更新本地采集规则，保证与中心配置的一致性。

### 8.2.2 核心适用场景

Proxy 是解决大规模、跨网络监控场景的核心方案，核心适用场景如下：

1. **跨机房 / 跨云 VPC 监控场景**：企业多机房、多云部署的架构中，仅需为每个机房 / VPC 部署一台 Proxy，防火墙仅需开通 Proxy 到中心 Server 的单个 10051/TCP 端口，无需为每个被监控主机开通 Server 到内网的访问权限，大幅简化防火墙规则，提升网络安全性；
2. **大规模监控集群性能分担**：当监控节点规模超过 1000 台，单 Server 的采集调度压力达到瓶颈时，通过 Proxy 将监控节点分片，每个 Proxy 负责 500-2000 台节点的采集，Server 仅需处理批量上报的数据与管控逻辑，性能压力降低 80% 以上；
3. **广域网 / 分支机构监控场景**：分支机构与总部之间通过广域网连接，网络延迟高、带宽有限，Proxy 在本地完成数据采集与聚合，批量压缩上报，大幅减少广域网带宽占用，避免高延迟导致的采集超时；
4. **单向网络隔离场景**：等保合规要求的 DMZ 区、生产隔离区，仅允许内网区域向外单向访问，Server 无法主动访问被监控主机，通过 Proxy 部署在隔离区内，主动向中心 Server 上报数据，完美适配单向网络隔离要求；
5. **边缘计算 / 物联网场景**：边缘节点分散、网络不稳定，Proxy 在边缘侧完成数据采集与本地缓存，网络恢复后自动补传数据，避免网络波动导致的监控数据丢失。

### 8.2.3 主动 Proxy vs 被动 Proxy

与 Agent 的工作模式对应，Proxy 与 Server 之间的通信分为**主动 Proxy（Active Proxy）** 与**被动 Proxy（Passive Proxy）** 两种模式，二者的本质差异在于配置同步与数据上报的调度权归属与通信发起方，是 Proxy 部署的核心选型点。

#### 8.2.3.1 核心工作模式详解

1. **主动 Proxy（官方推荐，生产环境首选）**
    
    - 核心特征：**Proxy 主动发起所有通信，调度权归属 Proxy**；
    - 工作流程：Proxy 启动后，主动向 Server 的 10051/TCP 端口发起连接，请求所辖主机的监控配置；配置同步完成后，Proxy 在本地完成采集调度，按照预设周期主动向 Server 批量上报采集数据；
    - 核心优势：仅需 Proxy 到 Server 的单向网络可达，适配防火墙隔离、NAT 场景；批量上报大幅降低 Server 的网络与性能开销；支持网络中断后的本地缓存与补传，数据可靠性更高。
    
2. **被动 Proxy**
    
    - 核心特征：**Server 主动发起所有通信，调度权归属 Server**；
    - 工作流程：Server 按照预设周期，主动向 Proxy 的 10051/TCP 端口发起连接，下发监控配置，拉取采集到的监控数据；Proxy 仅响应 Server 的请求，无主动通信行为；
    - 核心局限性：需要 Server 到 Proxy 的双向网络可达，无法适配单向隔离场景；Server 需为每个 Proxy 维护独立的连接，大规模场景下性能开销大；无主动上报机制，网络中断时无法补传数据，仅适用于 Server 与 Proxy 同属一个内网的小规模场景。
    

#### 8.2.3.2 核心差异对比与选型规则

表格

|对比维度|主动 Proxy|被动 Proxy|
|---|---|---|
|通信发起方|Proxy 主动向 Server 发起连接|Server 主动向 Proxy 发起连接|
|网络要求|仅需 Proxy 到 Server 的 10051/TCP 单向可达|需要 Server 到 Proxy 的 10051/TCP 双向可达|
|调度权归属|Proxy 本地调度采集任务|Server 统一调度所有 Proxy 的采集任务|
|Server 性能开销|极低，仅处理批量上报的数据|高，随 Proxy 数量线性增长|
|数据可靠性|本地缓存，网络中断后自动补传，无数据丢失|无本地缓存补传机制，网络中断期间数据丢失|
|防火墙适配性|极强，仅需开通单个出站端口，适配隔离区、NAT 场景|弱，需开通入站端口，无法适配单向隔离场景|
|广域网适配性|优，批量压缩上报，带宽占用低|差，频繁交互导致带宽占用高，高延迟下易超时|
|生产环境推荐度|最高，官方首选方案|极低，仅适用于内网小规模测试场景|

**选型硬性规范**：生产环境所有场景必须优先使用主动 Proxy，仅当 Proxy 与 Server 同内网、无网络隔离的测试环境可使用被动 Proxy。

### 8.2.4 本地数据库缓存机制

本地数据库缓存是 Proxy 的核心特性，是保障分布式场景下数据可靠性的核心能力，Proxy 必须依赖本地数据库实现配置存储、采集数据缓存，无数据库则无法正常运行。

#### 8.2.4.1 核心原理与缓存生命周期

Proxy 的本地缓存机制遵循严格的生命周期规则：

1. 采集到的监控数据首先写入本地数据库，同时标记为 “未上报” 状态；
2. Proxy 按照预设的上报周期，将 “未上报” 状态的数据批量压缩上报给 Server；
3. Server 接收并成功写入数据库后，向 Proxy 返回确认响应；
4. Proxy 收到确认响应后，将对应数据标记为 “已上报”，按照预设的保留周期清理本地数据；
5. 若与 Server 网络中断，上报失败，数据将持续保留在本地数据库，直到网络恢复后完成上报，最长保留时间由配置参数控制，默认 1 小时，最大可配置 30 天。

#### 8.2.4.2 支持的数据库类型与选型规范

Proxy 官方支持两种本地数据库类型，二者的核心差异与选型规则如下：

表格

|数据库类型|核心优势|核心局限性|生产环境选型规则|
|---|---|---|---|
|SQLite 3|轻量、开箱即用、无需独立部署、资源占用极低、配置简单|不支持高并发写入，大规模场景下性能瓶颈明显，不支持主从高可用|监控节点≤500 台的分支机构、边缘节点、测试环境；Proxy 负载较低的场景|
|MySQL/MariaDB|支持高并发写入、性能强、可扩展性好、支持主从高可用、数据可靠性更高|需要独立部署数据库服务，配置复杂度略高，资源占用略高|监控节点≥500 台的核心机房、大规模集群、生产环境核心 Proxy 节点；官方推荐生产环境首选|

**生产环境硬性规范**：核心机房、大规模场景的 Proxy 必须使用 MySQL/MariaDB 作为本地数据库，禁止使用 SQLite；边缘小规模场景可使用 SQLite，需配置合理的数据保留周期，避免磁盘耗尽。

#### 8.2.4.3 核心配置参数

缓存相关的核心配置在`zabbix_proxy.conf`中，生产环境必须按需调整：

表格

|参数名|核心作用|生产环境配置规范|
|---|---|---|
|DBHost/DBName/DBUser/DBPassword|本地数据库连接配置|与第二章 Server 数据库配置规范一致，必须使用专用用户，最小权限原则|
|ProxyLocalBuffer|已上报数据的本地保留时间，单位小时|默认 0，生产环境建议 1-4 小时，用于临时数据回溯|
|ProxyOfflineBuffer|网络中断时，未上报数据的最长保留时间，单位小时|默认 1 小时，生产环境建议 24-72 小时，根据网络稳定性调整|
|ConfigFrequency|Proxy 向 Server 请求配置更新的频率，单位秒|主动模式默认 3600s，生产环境建议 300-3600s，配置变更频繁的场景可缩短|
|DataSenderFrequency|Proxy 向 Server 批量上报数据的频率，单位秒|主动模式默认 1s，生产环境建议 1-30s，广域网低带宽场景可拉长|
|StartDBSyncers|数据库同步进程数|大规模场景建议 2-4 个，匹配写入性能|

### 8.2.5 标准化部署与核心配置

Proxy 的部署与第二章的 Server 部署体系完全兼容，支持包管理安装与容器化部署两种标准化方式，生产环境优先使用包管理安装。

#### 8.2.5.1 包管理标准化部署流程

1. **环境准备**：配置与 Server 版本完全一致的 Zabbix 官方软件源，关闭 SELinux，配置防火墙放行规则（主动模式仅需放行到 Server 10051/TCP 的出站流量，被动模式需放行 10051/TCP 入站流量），配置 NTP 时间同步，保证与 Server 时间偏差≤1s；
2. **数据库初始化**：MySQL/MariaDB 场景下，创建 Proxy 专用数据库与用户，字符集设置为`utf8mb4`，导入 Proxy 专用的数据库 Schema（与 Server 的 Schema 不同，仅包含 Proxy 所需的表结构）；SQLite 场景无需提前初始化，启动后自动生成数据库文件；
3. **组件安装**：通过系统包管理器安装`zabbix-proxy-mysql`/`zabbix-proxy-sqlite3`，对应选择的数据库类型；
4. **核心配置修改**：编辑`zabbix_proxy.conf`，配置必配参数，详见 8.2.5.2 节；
5. **服务启动与验证**：启动`zabbix-proxy`服务，配置开机自启，查看服务日志确认启动成功，无报错信息；
6. **Server 端配置**：登录 Zabbix Web 前端，进入「管理」→「代理」→「创建代理」，填写 Proxy 名称（必须与配置文件中的`Hostname`参数完全一致），选择 Proxy 模式（主动 / 被动），配置 Proxy 所辖的主机范围，完成 Proxy 与 Server 的联动。

#### 8.2.5.2 核心配置文件必配参数

`zabbix_proxy.conf`的必配参数如下，无有效配置则 Proxy 无法正常启动与联动：

表格

|参数名|核心作用|配置规范|
|---|---|---|
|Server|中心 Zabbix Server 的 IP 地址 / 域名|主动模式必填，配置 Server 的实际地址，多个地址用逗号分隔实现高可用|
|Hostname|Proxy 的全局唯一名称|必填，必须与 Web 前端配置的 Proxy 名称完全一致，大小写敏感，禁止重复|
|DBHost/DBName/DBUser/DBPassword|本地数据库连接参数|必填，与初始化的数据库配置完全匹配|
|ProxyMode|Proxy 工作模式|0 为被动模式，1 为主动模式，生产环境必须设置为 1|
|LogFile|日志文件路径|必填，配置绝对路径，生产环境必须配置日志轮转|
|Timeout|采集超时时间|默认 3s，最大不超过 30s，与所辖 Agent 的超时时间匹配|
|EnableRemoteCommands|是否允许执行远程命令|生产环境默认关闭，确有需要时开启并严格管控|

#### 8.2.5.3 容器化部署

Proxy 支持 Docker 与 Docker Compose 容器化部署，核心规范与第二章的容器化部署一致：

- 必须使用与 Server 版本完全一致的官方 Proxy 镜像，禁止使用第三方镜像；
- 必须配置本地数据库持久化卷，避免容器销毁导致缓存数据丢失；
- 必须配置时区与 Server 一致，保证时间同步；
- 主动模式下，仅需暴露容器到 Server 的网络，无需映射宿主机端口，进一步提升安全性。

### 8.2.6 生产环境最佳实践与安全约束

1. **版本一致性硬性规范**：Proxy 的大版本、小版本必须与中心 Server 完全一致，禁止跨版本混用，否则会出现配置同步失败、数据上报异常、兼容性问题；
2. **分片负载均衡规范**：单台 Proxy 所辖的监控节点数量建议控制在 500-2000 台，NVPS（每秒新增数据量）控制在 5000 以内，超过规模需拆分多台 Proxy 分片部署，避免单 Proxy 性能过载；
3. **网络安全规范**：
    
    - 主动模式下，Proxy 与 Server 之间的通信必须使用 TLS 加密，禁止明文传输；
    - 防火墙仅开放最小必要端口，禁止 Proxy 对公网暴露端口；
    - Server 端配置 IP 白名单，仅允许授权的 Proxy 地址接入，禁止非法 Proxy 接入；
    
4. **高可用规范**：核心机房的 Proxy 必须部署双活节点，搭配本地数据库主从复制，避免 Proxy 单点故障导致所辖区域监控中断；
5. **监控与运维规范**：必须对 Proxy 本身进行监控，包括 Proxy 进程状态、本地数据库状态、与 Server 的连通性、缓存数据量、采集成功率、上报延迟，避免 Proxy 故障未及时发现；
6. **权限管控规范**：Proxy 所辖的主机范围必须严格划分，仅分配对应业务线 / 机房的主机，禁止单 Proxy 拥有全量主机的配置权限，遵循最小权限原则。

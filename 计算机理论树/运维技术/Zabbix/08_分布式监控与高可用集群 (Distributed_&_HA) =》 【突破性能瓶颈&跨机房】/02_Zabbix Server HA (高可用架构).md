
## 8.3 Zabbix Server HA 高可用架构

### 8.3.1 核心定位与故障风险

#### 8.3.1.1 核心定义

Zabbix Server HA 高可用架构，是解决 Zabbix Server 单点故障风险的核心方案，目标是实现监控系统的无单点故障、故障自动切换、数据零丢失、业务无感知，保障监控系统的持续可用性，符合企业级生产系统的等保合规与业务连续性要求。

#### 8.3.1.2 单点故障核心风险

单节点 Zabbix Server 部署模式下，存在以下致命风险：

1. Server 服务器硬件故障、操作系统崩溃，导致整个监控系统完全中断，无法采集数据、触发告警；
2. Server 服务异常、配置错误导致服务不可用，监控体系整体瘫痪；
3. 机房断电、网络中断，导致单节点 Server 完全不可访问；
4. 不符合等保 2.0 三级以上的 “关键系统必须具备高可用架构” 的合规要求。

#### 8.3.1.3 HA 架构核心设计目标

- **RTO（恢复时间目标）**：故障切换时间≤30s，业务无感知；
- **RPO（恢复点目标）**：数据零丢失，切换过程中无监控数据、告警事件丢失；
- **无状态设计**：所有状态数据存储在共享数据库中，节点无本地状态，支持任意扩缩容；
- **自动切换**：故障检测、主备切换完全自动化，无需人工干预；
- **脑裂防护**：完善的集群仲裁机制，避免双主节点导致的数据错乱。

### 8.3.2 传统 HA 方案：Keepalived + 虚拟 VIP

#### 8.3.2.1 架构原理

该方案是 Zabbix 6.0 版本之前的主流高可用方案，基于 Linux 虚拟服务器（LVS）的 Keepalived 组件，通过 VRRP 协议实现虚拟 IP（VIP）的漂移，搭配共享数据库实现主备架构：

1. 部署两台 Zabbix Server 节点，主节点与备节点共用同一个后端数据库；
2. 两台节点均部署 Keepalived 服务，配置同一个虚拟 VIP，主节点持有 VIP，对外提供服务；
3. Agent/Proxy 均配置 VIP 作为 Server 地址，所有采集数据、配置请求均发送到 VIP；
4. Keepalived 持续检测主节点的 Zabbix Server 服务状态，若主节点服务故障、服务器宕机，Keepalived 自动将 VIP 漂移到备节点，备节点启动 Zabbix Server 服务，接管所有业务；
5. 切换完成后，所有 Agent/Proxy 的请求自动转发到备节点，无需修改客户端配置，实现故障无感知切换。

#### 8.3.2.2 优缺点分析

表格

|核心优势|核心局限性|
|---|---|
|兼容性强，支持所有 Zabbix 版本，无版本限制|存在脑裂风险，VRRP 协议在网络分区场景下会出现双主节点，导致数据错乱|
|架构简单，部署与维护成本低，技术成熟|主备节点同一时间仅能有一个节点运行，备节点处于闲置状态，资源利用率低|
|客户端无感知，仅需配置 VIP 地址，无需额外配置|仅支持双节点架构，无法扩展多节点集群，不支持大规模横向扩展|
|无侵入式，无需修改 Zabbix Server 原生配置|依赖 VRRP 协议，仅支持同网段二层网络部署，无法适配跨机房、跨地域部署|
||故障检测机制简单，仅能检测进程存活，无法检测服务假死、数据库连接异常等深层故障|

#### 8.3.2.3 适用场景

- Zabbix 5.0 及以下旧版本环境，无法升级到 6.0+ LTS 版本；
- 同机房、同网段的双节点部署，无跨机房高可用需求；
- 中小规模监控环境，对集群扩展性无要求，仅需解决基础单点故障问题。

### 8.3.3 原生 HA 方案：Zabbix Native HA（6.0+ LTS 官方主推）

Zabbix 6.0 LTS 版本推出了原生内置的高可用集群方案，完全摒弃了 Keepalived 等外挂组件，基于共享数据库实现分布式锁、故障检测、主备切换，是当前官方主推、生产环境首选的高可用方案。

#### 8.3.3.1 核心架构与设计原理

原生 HA 采用 “无状态节点 + 共享数据库” 的分布式架构，所有节点的状态、角色、调度信息均存储在共享数据库中，无本地状态数据，核心架构组成如下：

1. **集群节点**：多个 Zabbix Server 节点组成 HA 集群，节点分为三种角色：
    
    - **主节点（Active）**：集群中唯一的活跃节点，负责全量的核心管控逻辑，包括触发器计算、告警触发、动作执行、配置下发、任务调度，对外提供服务；
    - **备节点（Standby）**：处于待命状态的备用节点，实时同步集群状态，不执行核心管控逻辑，仅启动基础的健康检测进程，主节点故障时自动接管成为新的主节点；
    - **待机节点（Unavailable）**：故障、离线、配置错误的节点，被集群隔离，无法参与主节点选举。
    
2. **共享数据库**：集群的唯一状态存储载体，存储所有配置数据、监控数据、集群状态、分布式锁、节点心跳信息，是集群实现无状态设计的核心底座，必须采用高可用架构部署（主从复制、MGR 集群等）；
3. **分布式锁机制**：基于数据库的排他锁实现，集群中仅能有一个节点持有主节点锁，保证同一时间仅存在一个主节点，从根本上避免脑裂问题；
4. **心跳检测机制**：所有节点按照预设周期（默认 5s）向数据库更新心跳时间戳，主节点超过预设的故障切换阈值（默认 30s）未更新心跳，集群将判定主节点故障，触发新一轮主节点选举。

#### 8.3.3.2 完整工作流程

1. **集群启动流程**：
    
    - 所有集群节点启动后，首先连接共享数据库，注册节点信息，更新心跳时间戳；
    - 节点尝试获取数据库中的主节点排他锁，成功获取锁的节点成为主节点，启动全量服务进程；
    - 其余节点成为备节点，仅启动心跳进程与状态同步进程，等待主节点故障切换。
    
2. **正常运行流程**：
    
    - 主节点正常执行所有核心管控逻辑，持续向数据库更新心跳时间戳；
    - 备节点持续监控主节点的心跳状态，同步集群配置信息，不执行核心业务逻辑；
    - 所有 Agent/Proxy 可向集群中任意节点上报数据，非主节点接收到数据后，自动转发给主节点处理，保证数据不丢失。
    
3. **故障切换流程**：
    
    - 主节点发生故障（服务器宕机、服务崩溃、网络中断、数据库连接失败），无法向数据库更新心跳时间戳；
    - 超过预设的故障切换阈值后，集群判定主节点故障，释放主节点排他锁；
    - 所有备节点发起新一轮主节点选举，最先获取到排他锁的备节点成为新的主节点；
    - 新主节点启动全量服务进程，接管所有业务逻辑，整个切换过程在 30s 内完成，无数据丢失；
    - 原主节点恢复后，自动加入集群，成为备节点，无需人工干预。
    
4. **手动切换流程**：
    
    - 支持通过 Web 前端、CLI 命令手动触发主备切换，用于版本升级、配置变更、服务器维护等计划内操作；
    - 手动切换过程中，主节点会先完成所有待处理任务，平滑移交主节点权限，无业务中断。
    

#### 8.3.3.3 标准化部署与核心配置

原生 HA 的部署基于标准的 Server 包管理安装，仅需额外配置 HA 相关参数，无需额外部署外挂组件，部署流程如下：

1. **共享数据库部署**：部署高可用的 MySQL/PostgreSQL 集群，创建 Zabbix 专用数据库与用户，导入官方 Schema 与初始数据，保证所有集群节点均可访问数据库；
2. **集群节点部署**：在所有集群节点上，安装与版本一致的 Zabbix Server，配置相同的数据库连接参数，保证所有节点均可正常连接共享数据库；
3. **HA 核心参数配置**：编辑所有节点的`zabbix_server.conf`，配置 HA 核心参数：
    
    表格
    
    |参数名|核心作用|生产环境配置规范|
    |---|---|---|
    |HANodeName|节点的全局唯一名称|必填，每个节点的名称必须唯一，大小写敏感|
    |NodeAddress|节点的 IP 地址 / 域名|必填，集群内其他节点可访问的地址，用于数据转发|
    |FailoverDelay|主节点故障后，触发切换的延迟时间，单位秒|默认 30s，生产环境建议 10-30s，避免网络抖动导致的误切换|
    |UnreachablePeriod|节点无心跳后，被标记为不可用的时间，单位秒|默认 45s，需大于 FailoverDelay|
    |DisableHousekeeping|备节点是否禁用管家进程|默认 1，备节点禁用管家进程，仅主节点执行，避免多节点同时执行导致数据库压力飙升|
    
4. **服务启动与集群验证**：依次启动所有节点的 Zabbix Server 服务，配置开机自启；登录 Zabbix Web 前端，进入「管理」→「高可用」，查看集群节点状态、角色、心跳时间，确认集群正常运行；
5. **客户端配置**：Agent/Proxy 的`Server`/`ServerActive`参数配置集群中所有节点的 IP 地址，多个地址用逗号分隔，实现客户端的高可用，单个节点故障时，客户端自动切换到其他节点上报数据。

#### 8.3.3.4 核心优势与局限性

表格

|核心优势|核心局限性|
|---|---|
|原生内置，无任何外挂组件依赖，部署与维护成本极低|仅支持 Zabbix 6.0+ LTS 版本，旧版本无法使用|
|基于数据库分布式锁，从根本上杜绝脑裂问题，数据一致性强|依赖共享数据库的高可用，数据库故障会导致整个集群不可用|
|支持多节点集群（官方推荐奇数节点，最大支持 16 个节点），可横向扩展|同一时间仅能有一个主节点执行核心管控逻辑，备节点仅用于容灾，无法分担主节点的计算压力|
|完善的故障检测机制，可检测服务假死、数据库异常、网络中断等深层故障|跨机房部署时，需保证所有节点到共享数据库的低延迟网络，高延迟会影响集群稳定性|
|支持平滑手动切换，适配计划内维护、版本升级场景||
|官方原生维护，兼容性强，与 Zabbix 所有原生功能完全适配，无兼容性问题||

### 8.3.4 HA 方案选型对比与生产规范

#### 8.3.4.1 方案选型对比表

表格

|对比维度|原生 HA 方案|Keepalived + VIP 传统方案|
|---|---|---|
|版本支持|Zabbix 6.0+ LTS|所有 Zabbix 版本|
|外部依赖|无，原生内置|依赖 Keepalived、VRRP 协议|
|脑裂防护|基于数据库分布式锁，无脑裂风险|存在脑裂风险，需额外配置防护机制|
|集群规模|最大支持 16 个节点，奇数节点推荐|仅支持双节点架构|
|跨机房适配|支持，只要节点到数据库网络可达|不支持，仅能同网段二层网络部署|
|故障检测能力|完善，支持多维度深层故障检测|简单，仅支持进程 / 端口存活检测|
|客户端配置|配置多节点地址，自动切换|配置 VIP，无感知切换|
|资源利用率|备节点仅占用少量资源，可用于其他业务|备节点完全闲置，资源利用率低|
|生产环境推荐度|最高，6.0 + 版本官方唯一推荐|低，仅用于旧版本兼容场景|

#### 8.3.4.2 生产环境硬性规范

1. **版本选型规范**：Zabbix 6.0+ LTS 版本必须使用原生 HA 方案，禁止使用传统 Keepalived 方案；仅 5.0 及以下旧版本可使用传统方案；
2. **数据库高可用规范**：HA 集群的共享数据库必须采用高可用架构部署（MySQL MGR、PostgreSQL 流复制等），禁止使用单节点数据库，否则会成为整个集群的单点故障；
3. **集群节点规范**：原生 HA 集群节点数量必须为奇数，推荐 3 节点（1 主 2 备），最少 2 节点，最大不超过 5 节点；所有节点的硬件配置、操作系统、Zabbix 版本必须完全一致；
4. **网络规范**：所有集群节点到共享数据库的网络延迟必须≤10ms，禁止跨高延迟广域网部署集群；跨机房高可用需采用 “主中心集群 + 异地灾备集群” 架构，而非单集群跨机房部署；
5. **切换验证规范**：生产环境部署完成后，必须进行故障模拟测试，包括主节点宕机、服务崩溃、网络中断等场景，验证切换流程正常、无数据丢失、切换时间符合要求；
6. **集群监控规范**：必须对 HA 集群本身进行监控，包括节点状态、角色、心跳时间、切换事件、数据库连接状态，配置对应的告警，集群故障时第一时间通知运维人员；
7. **升级规范**：集群版本升级必须采用滚动升级方式，先升级备节点，再手动切换主节点，升级原主节点，保证升级过程中业务无中断。

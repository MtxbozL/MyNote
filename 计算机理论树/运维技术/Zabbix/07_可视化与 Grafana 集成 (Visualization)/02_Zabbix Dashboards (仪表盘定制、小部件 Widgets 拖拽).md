
## 7.3 Zabbix Dashboards（仪表盘）

### 7.3.1 核心定义与架构定位

Dashboards 是 Zabbix 5.0 + 版本推出的**新一代企业级可视化聚合平台**，是官方主推的原生可视化方案，全面替代传统 Screens 聚合图形。其核心采用拖拽式、响应式的 Widget（小部件）布局架构，打破了传统 Screens 固定网格的限制，支持全局时间范围联动、动态数据源、自适应分辨率、层级化下钻，同时兼容所有传统可视化组件，实现从单指标到全局业务的全场景可视化聚合。

与传统 Screens 的核心差异对比如下：

表格

|对比维度|传统 Screens|新一代 Dashboards|
|---|---|---|
|布局模式|固定行列网格布局，单元格大小固定|拖拽式自由布局，支持自适应列宽、响应式布局，适配不同分辨率屏幕|
|时间范围控制|每个组件单独配置时间范围，无全局联动|全局统一时间范围控制器，一键调整所有组件的时间窗口，支持临时缩放、时间对比|
|组件生态|组件数量有限，自定义能力弱|丰富的 Widget 组件库，支持第三方扩展，自定义能力极强|
|动态数据源|仅支持固定数据源，无法动态适配|支持基于标签、主机组的动态数据源，一个仪表盘适配所有同类型主机 / 集群|
|交互能力|无交互能力，仅支持静态展示|支持指标缩放、下钻、过滤、hover 详情查看，故障排查效率大幅提升|
|共享能力|仅支持 Zabbix 内部用户查看|支持匿名共享、临时链接分享、嵌入第三方系统|

### 7.3.2 核心 Widget（小部件）体系

Widget 是 Dashboards 的最小组成单元，每个 Widget 对应一个独立的可视化功能模块，Zabbix 原生提供 20 + 类 Widget，覆盖全场景可视化需求，核心高频 Widget 分类详解如下：

表格

|分类|核心 Widget|核心功能|典型适用场景|
|---|---|---|---|
|时序数据类|图形 Widget|兼容传统自定义图形，支持多指标叠加、多坐标轴、样式自定义，支持全局时间联动|指标趋势分析、多指标关联对比|
||顶 N 值 Widget|自动排序展示指定指标的 Top N/Bottom N 主机 / 实例，支持自定义排序规则|集群资源使用率排名、流量 Top N 节点分析|
||趋势预测 Widget|基于历史时序数据，通过线性回归算法预测指标未来趋势，提前预警资源耗尽风险|磁盘容量预测、带宽增长预测、资源扩容预警|
||数据表格 Widget|以表格形式展示多主机、多指标的实时数据，支持排序、过滤、条件格式|集群节点指标汇总、业务数据报表|
|状态监控类|问题 Widget|集中展示当前所有 PROBLEM 状态的异常事件，支持按严重性、主机组、时间范围过滤|运维人员核心监控面板，全局异常事件总览|
||触发器状态 Widget|展示指定触发器的实时状态，支持自定义样式、状态颜色映射|核心业务指标、关键链路状态突出展示|
||主机状态 Widget|展示指定主机组内主机的可用性状态、异常等级统计|机房 / 集群主机存活总览、资产状态监控|
||Web 场景 Widget|展示 Web 业务流程的可用性、响应时间、执行状态|核心业务流程可用性监控|
|业务与 SLA 类|服务可用性 Widget|展示业务服务树的实时状态、可用性百分比|业务服务总览、IT 服务状态监控|
||SLA 状态 Widget|展示指定服务的 SLA 达标情况、目标值、实际可用性、故障时长|业务 SLA 量化监控、管理层汇报面板|
|拓扑与导航类|拓扑图 Widget|嵌入 Maps 拓扑图，支持状态实时更新、下钻导航|全局拓扑总览、机房网络状态展示|
||仪表盘导航 Widget|配置多个仪表盘的跳转链接，实现层级化仪表盘导航|构建全局监控导航体系，实现从总览到详情的快速跳转|
|辅助类|时钟 Widget|展示本地 / UTC 时间，支持多时区配置|跨机房、跨地域监控场景时间同步|
||文本 / HTML Widget|支持纯文本、Markdown、HTML 代码，用于添加说明文档、告警处置指引、公告信息|仪表盘说明、故障排查指引、运维规范展示|
||URL Widget|嵌入第三方系统页面，实现跨系统监控数据聚合|整合日志系统、CMDB、运维平台的页面，实现大一统监控面板|

### 7.3.3 动态仪表盘与模板化能力

动态仪表盘是新一代 Dashboards 的核心进阶能力，通过变量与动态数据源实现 “一个仪表盘，适配全场景”，解决了传统静态仪表盘需要为每个主机、每个集群单独创建的痛点，大幅降低大规模监控场景下的可视化运维成本。

#### 7.3.3.1 核心实现方式

1. **全局变量配置**：仪表盘支持创建自定义变量，变量类型包括主机组、主机、监控项、触发器、文本、正则表达式等，用户可通过变量下拉框快速切换数据源，无需修改仪表盘配置；
    
    标准变量设计示例：
    
    - 变量 1：`$DATACENTER`，类型为数据中心标签，可选值为 “北京机房”“上海机房”“广州机房”
    - 变量 2：`$HOST_GROUP`，类型为主机组，依赖`$DATACENTER`变量，仅展示对应数据中心的主机组
    - 变量 3：`$HOST`，类型为主机，依赖`$HOST_GROUP`变量，仅展示对应主机组内的主机
        
        实现效果：通过三级变量下拉选择，即可快速切换不同数据中心、不同集群、不同主机的监控数据，一个仪表盘适配所有主机。
    
2. **动态数据源**：Widget 支持基于标签、主机组、正则表达式的动态数据源配置，无需指定固定主机 / 监控项，自动匹配符合条件的所有资源。例如：配置 Widget 数据源为 “标签为`app=mysql`的所有主机的 CPU 使用率监控项”，自动适配所有新增的 MySQL 主机，无需修改仪表盘。
3. **仪表盘模板**：支持将标准化仪表盘导出为 JSON 模板，可在不同 Zabbix 集群之间导入复用，同时支持将仪表盘封装到监控模板中，与主机 / 主机组关联，实现监控配置与可视化配置的同步复用。

#### 7.3.3.2 生产环境最佳实践

1. **变量分层设计**：变量必须遵循 “从大范围到小范围” 的层级逻辑，依赖关系清晰，避免变量过多导致的操作复杂；单仪表盘变量数量不超过 5 个；
2. **默认值配置**：所有变量必须配置合理的默认值，保证用户打开仪表盘即可看到有效数据，无需手动选择变量；
3. **权限适配**：变量与用户权限联动，普通用户仅能看到其权限范围内的主机组 / 主机选项，避免越权访问；
4. **模板化复用**：同类型业务、同类型组件的仪表盘必须标准化，导出为模板统一管理，禁止每个运维人员单独创建重复的仪表盘。

### 7.3.4 企业级仪表盘分层设计规范

生产环境仪表盘必须遵循**层级化、下钻式**的设计原则，构建从全局总览到单指标详情的完整排查链路，禁止无层级的碎片化仪表盘设计，标准四层模型如下：

表格

|层级|仪表盘类型|核心受众|核心内容|设计规范|
|---|---|---|---|---|
|L1 全局层|企业监控总览仪表盘|技术总监、运维负责人|全企业业务服务总状态、SLA 达标情况、核心机房可用性、全局异常事件统计、故障数量统计|一屏展示，核心指标优先，仅展示最高优先级的异常与核心 KPI，无冗余细节|
|L2 业务线层|业务线监控仪表盘|业务线运维负责人、应用负责人|对应业务线的服务可用性、核心业务指标、依赖组件状态、集群总览、异常事件列表|按业务模块划分，突出业务服务状态，下钻到对应集群仪表盘|
|L3 集群层|集群 / 组件监控仪表盘|一线运维工程师、DBA、中间件运维|对应集群的节点状态、核心性能指标、资源使用率、异常事件、趋势图形|按技术维度划分，覆盖组件全维度监控指标，下钻到单主机仪表盘|
|L4 主机 / 指标层|主机详情仪表盘、单指标分析仪表盘|一线运维、故障排查人员|单主机全维度监控指标、详细趋势图形、触发器状态、关联事件|指标分类清晰，支持深度时间范围分析、多指标对比，是故障排查的终端页面|

**硬性规范**：所有低层级仪表盘必须配置返回上一层级的导航入口，所有异常状态组件必须配置下钻到详情页面的链接，形成完整的闭环排查链路，避免出现 “信息孤岛” 式的仪表盘。

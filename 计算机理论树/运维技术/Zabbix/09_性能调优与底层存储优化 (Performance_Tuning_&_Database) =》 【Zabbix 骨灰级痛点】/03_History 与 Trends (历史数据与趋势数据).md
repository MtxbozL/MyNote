
## 9.4 历史与趋势数据管理及 Housekeeper 风险

### 9.4.1 历史数据与趋势数据的核心定义

Zabbix 采用**二元时序数据分离存储模型**，是数据生命周期管理的核心基础，二者的核心差异与定位如下：

表格

|数据类型|存储内容|生成规则|典型保留周期|核心用途|
|---|---|---|---|---|
|History（历史数据）|监控项的原始采集数据，每次采集生成一条记录|每个采集周期写入一条，与采集频率完全对应|7-30 天|短时间范围的故障排查、指标细节分析、实时监控|
|Trends（趋势数据）|监控项的小时级聚合数据，包含每小时的 max/min/avg/count 值|每小时对历史数据聚合生成一次，与采集频率无关|1-2 年|长时间范围的趋势分析、容量规划、SLA 统计、报表生成|

**核心优化原则**：

1. 非核心监控项严格缩短历史数据保留周期，从源头减少数据存储量；
2. 核心业务指标可适当延长趋势数据保留周期，用于长期趋势分析；
3. 不同类型监控项设置差异化的保留周期，禁止所有监控项使用统一的长保留周期。

### 9.4.2 Housekeeper 的核心原理与致命风险

#### 9.4.2.1 核心定义

Housekeeper（管家进程）是 Zabbix 原生内置的数据生命周期管理进程，核心职能是按照监控项预设的保留周期，通过`DELETE`语句定期删除数据库中过期的历史数据、趋势数据、事件数据、审计日志，是 Zabbix 默认的数据清理机制。

#### 9.4.2.2 生产环境致命风险

Housekeeper 是 90% 以上大规模 Zabbix 集群性能故障的根因，核心风险如下：

1. **数据库 IO 雪崩**：大规模环境下，history 系列单表数据量可达上亿条，`DELETE`语句删除过期数据时会触发全表扫描，产生海量随机 IO，导致数据库 CPU 与 IO 使用率打满，写入阻塞；
2. **表锁与业务中断**：InnoDB 引擎下，大批量`DELETE`会触发行锁甚至表锁，导致 Zabbix Server 的实时数据写入阻塞，出现采集延迟、告警中断、前端无响应等严重故障；
3. **性能持续恶化**：频繁的大批量删除会导致表空间碎片化，索引效率持续下降，查询与写入性能逐步恶化，形成恶性循环；
4. **不可控的执行时机**：默认每小时执行一次，若在业务高峰期执行，会直接导致监控系统整体瘫痪。

#### 9.4.2.3 临时缓解方案

针对无法立即实施表分区 / TimescaleDB 方案的存量环境，可通过以下配置临时缓解 Housekeeper 风险：

1. 调整`HousekeepingFrequency`为 12-24 小时，仅在凌晨业务低峰期执行；
2. 降低`MaxHousekeeperDelete`为 1000-2000，拆分单次删除操作，减少 IO 峰值；
3. 开启`HousekeepingFrequency`的分时段执行，仅在低峰期启用；
4. 为不同数据类型配置独立的 Housekeeper 规则，分开处理历史数据、趋势数据、事件数据。

**终极解决方案**：关闭针对 history、trends 大表的原生 Housekeeper，采用**数据库表分区技术**或**TimescaleDB 时序数据库方案**，用`DROP PARTITION`替代`DELETE`操作，从根本上解决 IO 雪崩问题。

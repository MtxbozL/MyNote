
## 9.5 数据库表分区技术（传统必备方案）

### 9.5.1 核心原理与优化逻辑

表分区技术是针对 MySQL/MariaDB 数据库的传统标准化优化方案，核心原理是**将 Zabbix 的大时序表按照时间维度拆分为多个独立的物理子表（分区）**，数据写入时按时间落入对应分区，过期数据清理时直接`DROP`整个分区，而非逐行`DELETE`。

核心优化逻辑：

- `DROP`分区是 DDL 操作，直接删除物理数据文件，无需逐行扫描与事务日志记录，IO 开销仅为`DELETE`的万分之一，无锁表风险；
- 时序数据查询时，数据库可直接定位到对应时间范围的分区，避免全表扫描，查询性能提升 10-100 倍；
- 分区表对 Zabbix Server 完全透明，无需修改 Zabbix 源码与配置，无侵入式适配。

### 9.5.2 必须分区的核心表

Zabbix 数据库中，99% 的存储量与 IO 压力集中在 7 张时序大表，是分区的核心目标，其余小表无需分区：

表格

|表名|存储内容|分区粒度建议|
|---|---|---|
|history|浮点型数值历史数据|按天分区|
|history_uint|无符号整型数值历史数据|按天分区|
|history_str|字符串型历史数据|按天分区|
|history_text|文本型历史数据|按天分区|
|history_log|日志型历史数据|按天分区|
|trends|浮点型数值小时级趋势数据|按月分区|
|trends_uint|无符号整型数值小时级趋势数据|按月分区|

### 9.5.3 标准化实施流程

#### 9.5.3.1 前置准备

1. 环境验证：MySQL/MariaDB 版本支持分区功能（5.7 + 全版本支持，8.0 推荐），数据库引擎为 InnoDB；
2. 数据备份：分区操作前必须全量备份数据库，避免操作失误导致数据丢失；
3. 业务低峰期操作：存量数据量大的环境，必须在业务低峰期执行初始分区，避免影响生产业务；
4. 关闭 Housekeeper：分区表对应的历史数据、趋势数据的 Housekeeper 必须关闭，避免重复删除操作。

#### 9.5.3.2 分区实施

生产环境推荐采用**官方标准化存储过程方案**，实现分区的自动创建、自动删除，核心步骤如下：

1. 在 Zabbix 数据库中创建分区管理存储过程，包含三个核心函数：
    
    - 分区创建函数：自动提前创建未来 1-3 个月的分区；
    - 分区删除函数：自动删除超过保留周期的过期分区；
    - 分区校验函数：校验分区完整性，避免分区缺失导致写入失败；
    
2. 对 7 张核心大表执行初始分区，存量数据按时间迁移到对应分区；
3. 配置 Linux 定时任务（Crontab），每天凌晨低峰期执行存储过程，自动维护分区；
4. 验证分区有效性：插入测试数据，确认数据落入正确分区，过期分区可正常自动删除。

#### 9.5.3.3 生产环境配置规范

1. history 系列表按天分区，提前创建 30 天的未来分区，避免分区缺失；
2. trends 系列表按月分区，提前创建 3 个月的未来分区；
3. 分区保留周期与监控项的保留周期严格匹配，预留 1-2 天的缓冲期；
4. 定时任务执行日志必须留存，配置告警通知，分区执行失败时第一时间告警；
5. 主从复制环境下，分区操作必须在主库执行，确保同步到从库；
6. 禁止手动删除正在写入的分区，避免数据写入失败。

### 9.5.4 局限性与适用场景

- 核心优势：无侵入式、兼容所有 Zabbix 版本、技术成熟稳定、性能提升显著，是 MySQL 环境的必备优化方案；
- 核心局限性：需要手动维护分区脚本与定时任务，存在分区缺失的风险；无原生数据压缩能力，存储成本较高；长周期查询性能提升有限；
- 适用场景：Zabbix 6.0 以下旧版本、MySQL/MariaDB 数据库环境、中小规模集群、无法切换到 PostgreSQL 的存量环境。

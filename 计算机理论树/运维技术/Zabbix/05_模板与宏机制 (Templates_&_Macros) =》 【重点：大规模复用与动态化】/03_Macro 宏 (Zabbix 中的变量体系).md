
## 5.4 Macro 宏体系

### 5.4.1 核心定义与学术定位

宏（Macro）是 Zabbix 内置的**标准化变量体系**，是实现监控配置参数化、动态化、场景化适配的核心载体。其核心学术价值在于通过变量替换机制，将监控配置中的固定值抽象为可灵活调整的变量，解耦监控规则与具体参数，解决模板复用过程中的个性化适配问题，同时实现监控配置的统一参数管控。

宏的核心运行原理为：在配置解析、表达式计算、告警通知生成的过程中，Zabbix 会自动将宏变量替换为对应作用域内的实际值，无需修改监控规则本身，仅需调整宏变量的值即可实现配置的个性化适配。

### 5.4.2 宏的语法规范与命名规则

Zabbix 宏体系分为三类核心类型，每类宏有严格的语法格式与命名规范，大小写敏感，格式错误将导致宏无法被正确解析。

#### 通用语法规则

1. 所有宏均使用大括号`{}`包裹，作为宏的边界标识；
2. 宏名称必须使用大写字母、数字、下划线组成，禁止使用特殊字符、空格、小写字母；
3. 不同类型的宏有专属的前缀标识，用于区分宏的类型与作用域，禁止混用前缀格式。

#### 三类核心宏的语法定义

表格

|宏类型|标准语法格式|前缀标识|核心作用域|
|---|---|---|---|
|内置宏|`{<MACRO_NAME>}`|无专属前缀，直接写宏名称|全局通用，Zabbix 系统原生预定义|
|用户自定义宏|`{$<MACRO_NAME>}`|`$`符号，包裹在大括号内|全局、模板、主机三级作用域|
|LLD 宏|`{#<MACRO_NAME>}`|`#`符号，包裹在大括号内|低级别发现规则（LLD）的原型配置内|

### 5.4.3 内置宏

#### 5.4.3.1 核心定义

内置宏（Built-in Macros）是 Zabbix 系统原生预定义的全局变量，无需用户手动创建，可在监控配置、触发器表达式、告警通知模板、远程命令等所有场景中直接调用，用于获取监控对象、事件、系统的元数据与实时状态。

#### 5.4.3.2 核心分类与高频内置宏

内置宏按照用途分为六大类，覆盖全场景使用需求，高频核心宏如下：

表格

|宏分类|高频宏示例|核心功能|典型适用场景|
|---|---|---|---|
|主机相关宏|`{HOST.NAME}`|获取主机的标准化名称|告警通知、触发器名称、图形标题|
||`{HOST.IP}`|获取主机的主 IP 地址|告警通知、故障排查指引、远程命令|
||`{HOST.DNS}`|获取主机的 DNS 名称|资产信息同步、告警上下文补充|
||`{HOST.HOST}`|获取主机的内部唯一标识|自动化脚本、API 调用|
|监控项相关宏|`{ITEM.NAME}`|获取监控项的名称|触发器名称、告警通知、图形标题|
||`{ITEM.KEY}`|获取监控项的采集 Key|故障排查、自动化脚本|
||`{ITEM.VALUE}`|获取监控项的最新采集值|触发器表达式、告警通知、远程命令|
||`{ITEM.LASTVALUE}`|获取监控项的上一个采集值|变化量判断、异常对比|
|触发器相关宏|`{TRIGGER.NAME}`|获取触发器的完整名称|告警通知标题、事件记录|
||`{TRIGGER.SEVERITY}`|获取触发器的严重性等级|告警分级、通知渠道适配|
||`{TRIGGER.EXPRESSION}`|获取触发器的完整表达式|故障排查、审计记录|
||`{TRIGGER.DESCRIPTION}`|获取触发器的描述信息|告警通知内容、排查指引|
|事件相关宏|`{EVENT.ID}`|获取事件的唯一 ID|告警追踪、工单关联、事件确认|
||`{EVENT.DATE}`/`{EVENT.TIME}`|获取事件发生的日期 / 时间|告警通知、故障时间统计|
||`{EVENT.AGE}`|获取事件从发生到当前的持续时长|告警升级、处置进度跟踪|
||`{EVENT.STATUS}`|获取事件的状态（PROBLEM/OK）|恢复通知、事件分类|
|系统相关宏|`{ZABBIX.SERVER.NAME}`|获取 Zabbix Server 的名称|多集群环境告警区分、审计记录|
||`{DATE}`/`{TIME}`|获取当前系统的日期 / 时间|远程命令执行记录、日志生成|
|告警通知相关宏|`{ALERT.SENDTO}`|获取告警通知的接收人|通知审计、发送日志记录|
||`{ALERT.MESSAGE}`|获取告警通知的完整内容|通知审计、第三方系统转发|
||`{ALERT.STATUS}`|获取告警通知的发送状态|发送失败重试、告警审计|

#### 5.4.3.3 使用规范

1. 告警通知模板必须使用内置宏实现内容的动态化填充，禁止硬编码固定内容，保证告警通知的通用性；
2. 触发器名称、图形标题必须使用主机、监控项相关内置宏，实现标准化命名，禁止硬编码固定名称；
3. 远程命令、自动化脚本中优先使用内置宏获取主机、监控项的元数据，避免硬编码导致的脚本通用性差。

### 5.4.4 用户自定义宏

#### 5.4.4.1 核心定义

用户自定义宏（User-defined Macros）是用户根据业务需求创建的自定义变量，是实现模板参数化、阈值动态化适配的核心载体，也是宏体系中最核心、使用最频繁的类型。其核心作用是将模板中硬编码的阈值、路径、端口、名称等可变参数抽象为变量，在不同层级的作用域内设置不同的变量值，实现 “一套模板，多场景适配”。

#### 5.4.4.2 核心适用场景

1. **阈值参数化**：模板内的触发器告警阈值通过宏定义，不同主机、不同业务场景可设置不同的阈值，无需修改模板本身。
    
    标准示例：
    
    - 触发器表达式：`avg("system.cpu.util[,user]",5m) > {$CPU_UTIL_WARN_THRESHOLD}`
    - 模板内默认值：`{$CPU_UTIL_WARN_THRESHOLD} = 80`
    - 核心业务主机个性化值：`{$CPU_UTIL_WARN_THRESHOLD} = 90`
    - 测试环境主机个性化值：`{$CPU_UTIL_WARN_THRESHOLD} = 85`
    
2. **环境参数适配**：不同环境的文件路径、端口、认证信息等参数通过宏定义，实现一套模板适配开发、测试、生产多环境；
3. **标准化文本定义**：告警通知模板、排查指引、标准化描述等文本内容通过宏定义，实现全企业统一管控，一处修改全量同步；
4. **凭证管理**：SNMP 团体名、API 密钥、认证用户名等敏感信息通过加密宏存储，避免明文写入配置，提升安全性。

#### 5.4.4.3 生产环境配置规范

1. **命名标准化规范**：宏名称必须遵循`{$<业务域>_<指标类型>_<场景>_<参数类型>}`的命名规则，示例：`{$DISK_USED_WARN_THRESHOLD}`、`{$MYSQL_CONN_MAX_CRITICAL}`，禁止使用无意义的名称；
2. **无硬编码原则**：模板内所有可变参数必须通过用户自定义宏定义，禁止在监控项、触发器表达式、Web 场景中硬编码固定阈值、路径、端口等参数；
3. **默认值规范**：所有在模板中定义的宏，必须设置通用默认值，保证模板关联主机后，无需额外配置即可正常使用；
4. **敏感信息加密规范**：涉及认证凭证、密钥等敏感信息的宏，必须开启 Zabbix 宏加密功能，禁止明文存储；
5. **注释规范**：所有宏必须添加明确的注释，说明宏的用途、取值范围、适用场景，便于后续维护。

### 5.4.5 LLD 宏

#### 5.4.5.1 核心定义

LLD 宏（Low-Level Discovery Macros，低级别发现宏）是专用于 Zabbix 自动发现机制的动态变量，用于接收自动发现规则返回的监控资源元数据，作为监控项原型、触发器原型、图形原型的参数，实现批量生成监控配置的核心能力。

LLD 宏的核心运行原理为：自动发现规则执行后，返回 JSON 格式的发现结果，其中包含多个`{#MACRO_NAME}`格式的变量与对应的值，Zabbix 将这些值自动填充到原型配置的 LLD 宏中，为每个发现的资源生成独立的监控配置。

#### 5.4.5.2 核心特性与使用规范

1. **语法专属约束**：LLD 宏仅可在低级别发现规则的监控项原型、触发器原型、图形原型中使用，禁止在普通监控项、触发器中使用；
2. **动态生成特性**：LLD 宏的值由自动发现规则的执行结果动态生成，无需用户手动创建，仅需在原型配置中引用；
3. **高频内置 LLD 宏**：Zabbix 原生自动发现规则内置了标准化 LLD 宏，示例：
    
    - `{#FSNAME}`：文件系统挂载点，用于磁盘自动发现；
    - `{#IFNAME}`：网卡名称，用于网络接口自动发现；
    - `{#PROC.NAME}`：进程名称，用于进程自动发现；
    - `{#SNMPINDEX}`：SNMP 设备索引，用于网络设备端口自动发现；
    
4. **自定义 LLD 宏**：用户自定义自动发现脚本可返回自定义 LLD 宏，实现个性化资源的自动发现与监控配置生成，详细用法见第六章自动发现机制。

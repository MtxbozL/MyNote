
## 6.4 低级别发现（LLD, Low-Level Discovery）

### 6.4.1 核心定义与学术定位

LLD 是 Zabbix 内置的、最强大的自动化监控能力，是针对单主机内动态变化的监控资源，实现**自动发现、自动生成监控配置、自动生命周期管理**的核心机制。其核心解决的行业痛点是：单主机内的监控资源（磁盘分区、网卡、CPU 核心、进程、数据库多实例、中间件多实例）是动态变化的，手动为每个资源创建监控项、触发器、图形的运维成本极高，且无法适配资源动态增减的场景，LLD 完全实现了这类资源的监控自动化。

学术定位：LLD 是 Zabbix 监控体系中，实现 “监控配置随资源动态伸缩” 的核心能力，通过**发现规则 - 原型配置 - 动态生成**的三层模型，将监控配置的粒度从主机级下沉到资源实例级，完全适配动态变化的监控场景，是 Zabbix 区别于其他监控系统的核心杀手锏能力，也是监控架构师必须掌握的核心技能。

### 6.4.2 核心工作原理与执行链路

LLD 的完整执行链路分为 5 个核心阶段，严格遵循预设规则执行，无人工干预：

1. **规则执行阶段**：Zabbix Server/Agent 按照 LLD 规则预设的周期，执行发现采集逻辑，获取目标资源的元数据；
2. **数据解析阶段**：将采集到的资源元数据，解析为 Zabbix 标准的 JSON 格式，其中包含 LLD 宏（`{#MACRO_NAME}`）与对应的值，作为后续生成配置的参数；
3. **过滤匹配阶段**：基于预设的 LLD 过滤器规则，过滤掉不需要监控的资源，仅保留符合规则的资源实例；
4. **配置生成阶段**：基于过滤后的资源实例，将 LLD 宏填充到监控项原型、触发器原型、图形原型中，为每个资源实例生成独立的、专属的监控配置；
5. **生命周期管理阶段**：按照预设的资源丢失策略，对不再被发现的资源实例，执行禁用 / 删除对应监控配置的操作，实现监控配置的全生命周期自动化管理。

### 6.4.3 LLD 核心组成组件

LLD 体系由四大核心组件构成，缺一不可，完整覆盖从发现到监控的全流程：

#### 6.4.3.1 发现规则（LLD Rule）

发现规则是 LLD 的入口，定义了 “发现什么资源、怎么发现、多久发现一次” 的核心规则，关键配置项如下：

表格

|配置项|核心作用|生产环境配置规范|
|---|---|---|
|名称|发现规则的语义化标识|必须明确发现的资源类型，如 “文件系统挂载点发现”，禁止模糊命名|
|类型|发现规则的采集类型|与第三章的采集方式完全兼容，支持 Zabbix Agent、SNMP、JMX、HTTP Agent 等所有采集方式|
|发现 Key|采集资源元数据的 Key|原生内置 Key 如`vfs.fs.discovery`，自定义发现规则使用自定义 UserParameter Key|
|更新周期|发现规则的执行周期|绝大多数场景建议 1h-24h，资源变化频繁的场景≥30min，禁止 < 5min 的高频执行|
|资源丢失周期|资源不再被发现后，执行处置操作的等待周期|生产环境建议≥7d，避免资源临时不可达导致监控配置被删除|
|启用状态|控制发现规则是否生效|非必要的发现规则立即禁用，避免无效的性能开销|

#### 6.4.3.2 过滤器（Filter）

过滤器是 LLD 的精准控制组件，用于过滤掉不需要监控的资源实例，避免生成大量无用的监控配置，核心配置规范：

- 过滤类型：支持 “必须匹配”（符合正则的资源保留）、“必须不匹配”（符合正则的资源过滤）；
- 过滤规则：基于 LLD 宏进行正则匹配，如针对`{#FSNAME}`宏，过滤掉`swap`、`/boot`分区，仅保留业务数据分区；
- 逻辑运算：支持多个过滤规则的 AND/OR 组合，实现复杂的过滤逻辑；
- **生产环境硬性要求**：所有 LLD 规则必须配置过滤器，禁止全量保留发现的资源，避免生成过多无效监控项，给 Server 带来性能压力。

#### 6.4.3.3 原型配置（Prototypes）

原型配置是 LLD 的核心模板，定义了为每个发现的资源实例生成什么样的监控配置，是第五章模板机制的延伸，所有原型中均可使用 LLD 宏实现动态参数填充，分为四大类：

1. **监控项原型（Item Prototype）**：为每个资源实例生成的监控项模板，如为每个发现的磁盘分区，生成磁盘使用率、磁盘容量的监控项，Key 中使用`{#FSNAME}`宏填充分区路径，实现每个分区的独立监控；
2. **触发器原型（Trigger Prototype）**：为每个资源实例生成的异常判断规则模板，如为每个磁盘分区生成磁盘使用率超过阈值的触发器，表达式中使用 LLD 宏，实现每个分区的独立告警；
3. **图形原型（Graph Prototype）**：为每个资源实例生成的可视化图形模板，如为每个网卡生成进出流量的趋势图，使用 LLD 宏填充网卡名称；
4. **主机原型（Host Prototype）**：专用于 SNMP 网络设备发现，为每个发现的网络设备节点生成独立的主机，实现网络设备的自动发现与监控。

#### 6.4.3.4 LLD 宏（LLD Macros）

LLD 宏是连接发现规则与原型配置的核心载体，对应第五章的 LLD 宏体系，是发现结果中的动态变量，标准格式为`{#MACRO_NAME}`，必须由大写字母、数字、下划线组成，在原型配置中被替换为每个资源实例的实际值，实现监控配置的动态生成。

- 原生内置 LLD 宏：如`{#FSNAME}`（文件系统挂载点）、`{#FSTYPE}`（文件系统类型）、`{#IFNAME}`（网卡名称）、`{#CPU.NUMBER}`（CPU 核心编号）；
- 自定义 LLD 宏：用户自定义发现脚本中返回的宏，可自定义名称与值，实现个性化资源的动态配置。

### 6.4.4 原生内置 LLD 规则

Zabbix Agent/Agent 2 原生内置了大量标准化 LLD 规则，开箱即用，覆盖操作系统核心资源的自动发现，是生产环境优先使用的规则，核心高频内置规则如下：

表格

|内置发现 Key|核心功能|核心 LLD 宏|适用场景|
|---|---|---|---|
|`vfs.fs.discovery`|自动发现主机上所有挂载的文件系统|`{#FSNAME}`、`{#FSTYPE}`、`{#FSDRIVETYPE}`|磁盘分区、挂载点的自动监控，最常用的内置 LLD 规则|
|`net.if.discovery`|自动发现主机上所有的物理 / 虚拟网卡|`{#IFNAME}`、`{#IFALIAS}`、`{#IFTYPE}`|网卡进出流量、错包、丢包的自动监控|
|`system.cpu.discovery`|自动发现主机上所有的 CPU 逻辑核心|`{#CPU.NUMBER}`、`{#CPU.STATUS}`|单 CPU 核心的使用率、负载的自动监控|
|`proc.discovery`|自动发现主机上运行的指定进程|`{#PROC.NAME}`、`{#PROC.USER}`、`{#PROC.PID}`|进程存活、资源占用的自动监控|
|`net.tcp.listen.discovery`|自动发现主机上所有处于监听状态的 TCP 端口|`{#PORT}`、`{#PROTO}`|端口监听状态的自动监控|
|`snmp.if.discovery`|自动发现 SNMP 网络设备的所有端口|`{#IFINDEX}`、`{#IFNAME}`、`{#IFALIAS}`|交换机、路由器端口的流量、状态自动监控|
|`snmp.discovery`|基于 SNMP OID 的自定义发现|自定义 LLD 宏|网络设备传感器、光模块、硬件状态的自动发现|

### 6.4.5 自定义 LLD 脚本开发与实战

自定义 LLD 是 LLD 能力的核心延伸，允许用户通过自定义脚本，实现任意个性化资源的自动发现与监控，是适配业务专属场景的核心能力。

#### 6.4.5.1 自定义 LLD 脚本硬性规范

自定义脚本的输出必须严格遵循 Zabbix LLD 的标准 JSON 格式，否则 Zabbix 无法解析，标准格式如下：

json

```
{
    "data": [
        {
            "{#MACRO_NAME_1}": "value1",
            "{#MACRO_NAME_2}": "value2"
        },
        {
            "{#MACRO_NAME_1}": "value3",
            "{#MACRO_NAME_2}": "value4"
        }
    ]
}
```

- 顶级必须是 JSON 对象，包含唯一的`data`字段，值为数组；
- 数组中的每个元素是一个 JSON 对象，对应一个发现的资源实例；
- 每个资源实例对象的 Key 必须是 LLD 宏格式（`{#大写名称}`），Value 为对应实例的参数值；
- 脚本必须仅向 stdout 输出标准 JSON，禁止输出其他任何内容，stderr 仅用于错误日志；
- 脚本执行超时时间不得超过 LLD 规则的超时时间，最大不超过 30s；
- 脚本必须具备异常处理能力，异常场景下输出空的`data`数组，避免 LLD 规则变为不支持状态。

#### 6.4.5.2 自定义 LLD 实战示例

**场景**：自动发现主机上运行的所有 Java 进程，为每个 Java 进程自动生成进程存活、CPU 占用、内存占用的监控项与触发器。

步骤 1：编写自定义 LLD 发现脚本`java_process_discovery.sh`

bash

运行

```
#!/bin/bash
# 自定义LLD脚本：发现主机上的Java进程，输出标准LLD JSON格式
set -euo pipefail

# 初始化空数据
data='{"data":[]}'

# 发现Java进程PID，过滤grep进程
java_pids=$(ps -ef | grep java | grep -v grep | awk '{print $2}')

# 遍历PID生成LLD数据
if [ -n "$java_pids" ]; then
    data=$(echo "$java_pids" | while read -r pid; do
        # 提取应用名称，优先读取-Dapp.name参数
        app_name=$(ps -p "$pid" -o cmd= | sed -e 's/.*-Dapp.name=//g' -e 's/ .*//g' | tr -d '\n')
        # 无应用名称时使用PID作为标识
        [ -z "$app_name" ] && app_name="java_${pid}"
        # 输出单实例JSON对象
        echo "{\"{#JAVA_PID}\":\"${pid}\",\"{#JAVA_APP_NAME}\":\"${app_name}\"}"
    done | jq -s '{data: .}')
fi

# 输出最终标准JSON
echo "$data"
```

- 脚本依赖`jq`工具保证 JSON 格式合规，避免手动拼接导致的格式错误；
- 输出两个核心 LLD 宏：`{#JAVA_PID}`（Java 进程 PID）、`{#JAVA_APP_NAME}`（Java 应用名称）；
- 内置异常处理，无 Java 进程时输出合规的空数据结构，避免 LLD 解析失败。

步骤 2：配置 Agent UserParameter

在`zabbix_agent2.conf`或独立配置文件中添加：

plaintext

```
UserParameter=java.process.discovery,/etc/zabbix/scripts/java_process_discovery.sh
```

配置完成后重启 Agent 2 服务，使用`zabbix_get`工具测试 Key 是否正常输出标准 JSON。

步骤 3：在监控模板中创建 LLD 发现规则

- 规则名称：Java 进程自动发现
- 类型：Zabbix Agent
- Key：`java.process.discovery`
- 更新周期：1h
- 资源丢失周期：7d
- 过滤器：可选，基于`{#JAVA_APP_NAME}`正则匹配，过滤掉不需要监控的进程。

步骤 4：创建原型配置

1. **监控项原型 1**：Java 进程 {#JAVA_APP_NAME} 存活状态
    
    - Key：`proc.num[,,,{#JAVA_PID}]`
    - 类型：Zabbix Agent
    - 更新周期：30s
    - 数据类型：数字（无正负）
    
2. **监控项原型 2**：Java 进程 {#JAVA_APP_NAME} CPU 使用率
    
    - Key：`proc.cpu.util[,{#JAVA_PID}]`
    - 类型：Zabbix Agent
    - 更新周期：30s
    - 数据类型：浮点数
    
3. **触发器原型**：Java 进程 {#JAVA_APP_NAME} 异常退出
    
    - 表达式：`last("java.process.discovery.proc.num[,,,{#JAVA_PID}]") = 0`
    - 严重性：High
    - 恢复表达式：`last("java.process.discovery.proc.num[,,,{#JAVA_PID}]") >= 1`
    
4. **图形原型**：Java 进程 {#JAVA_APP_NAME} 资源占用趋势
    
    - 包含 CPU 使用率、内存使用率监控项原型，自动为每个 Java 进程生成趋势图。
    

步骤 5：验证生效

LLD 规则执行后，自动为每个发现的 Java 进程生成对应的监控项、触发器、图形；Java 进程新增 / 退出时，监控配置自动生成 / 禁用，完全无需人工干预。

### 6.4.6 生命周期管理与资源丢失策略

LLD 的核心优势之一是自动管理监控配置的生命周期，针对不再被发现的资源实例，Zabbix 提供四种资源丢失策略，生产环境必须根据场景合理选择：

1. **保留丢失的资源原型，永久启用**：资源不再被发现后，对应的监控配置永久保留并启用，适用于核心固定资源，避免临时不可达导致监控中断，缺点是会产生僵尸监控项；
2. **保留丢失的资源原型，禁用**：资源不再被发现后，对应的监控配置自动禁用，保留历史数据，适用于绝大多数生产场景，官方推荐；
3. **指定周期后删除丢失的资源原型**：资源不再被发现后，保留指定天数，到期后自动删除对应的监控配置与历史数据，适用于容器、临时进程等生命周期极短的资源；
4. **立即删除丢失的资源原型**：资源不再被发现后，立即删除对应的监控配置，仅适用于超短期临时资源，**生产环境禁止使用**，避免临时不可达导致监控配置被删除。

### 6.4.7 生产环境最佳实践与性能优化

1. **执行周期优化**：LLD 规则的更新周期不得短于 30min，资源变化不频繁的场景（磁盘分区、网卡）建议 24h，避免高频执行给 Server 带来性能压力；
2. **过滤器强制配置**：所有 LLD 规则必须配置过滤器，仅保留需要监控的资源，如文件系统发现过滤掉虚拟文件系统、swap 分区，网卡发现过滤掉 loopback 网卡；
3. **原型配置最小化**：仅为必要的核心指标创建监控项原型，禁止为每个资源生成大量无用监控项，控制单主机 LLD 生成的监控项总数；
4. **自定义脚本规范**：自定义 LLD 脚本必须保证 JSON 格式严格合规，内置异常处理，禁止输出非 JSON 内容；脚本执行效率必须达标，避免超时；
5. **性能约束**：单台主机的 LLD 规则数量不超过 10 个，单 LLD 规则单次发现的资源实例不超过 100 个，避免生成过多监控项导致 Server 数据库压力飙升；
6. **与宏体系联动**：触发器原型中的阈值使用用户自定义宏定义，实现不同主机、不同资源的阈值个性化适配，无需修改原型配置；
7. **历史数据管理**：对于频繁变化的容器、临时进程等资源，配置较短的历史数据保留周期，减少数据库存储压力。

### 6.4.8 常见故障排查

1. LLD 规则变为不支持状态：核心原因是发现 Key 无法正常执行，或输出的 JSON 格式不符合规范，使用`zabbix_get`工具测试发现 Key，校验 JSON 格式合规性；
2. 未生成监控配置：核心原因是过滤器配置错误，所有发现的资源都被过滤，或 LLD 宏名称与原型中引用的宏名称大小写不匹配；
3. 监控项生成后变为不支持状态：核心原因是原型中的 Key 使用了错误的 LLD 宏，或宏值填充后 Key 格式错误，检查监控项的最终生成结果；
4. LLD 规则执行超时：核心原因是自定义脚本执行时间过长，或发现的资源实例过多，优化脚本执行效率，调整超时时间，通过过滤器减少资源数量；
5. 资源丢失后监控配置未自动处理：核心原因是资源丢失周期配置过长，或 LLD 规则被禁用，检查对应的配置参数。

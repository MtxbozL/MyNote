
Record（又称 Message）是 Kafka 中数据传输与存储的**最小原子单元**，生产者发送的业务数据会被封装为 Record，写入 Partition 的日志文件中，消费者拉取与处理的也是完整的 Record 对象。本节分为用户可见的逻辑结构与底层物理存储格式两部分，完整覆盖 Record 的全维度设计。

### 2.6.1 Record 逻辑结构（用户可访问字段）

逻辑结构是用户通过客户端 API 可配置、可读取的核心字段，是业务数据的核心载体，所有字段均严格定义如下：

1. **Key**：消息键，字节数组类型，可选字段。核心作用有二：一是用于分区路由，默认分区器会根据 Key 的哈希值计算目标分区，相同 Key 的消息会被路由至同一个 Partition，保障单 Key 的消息顺序性；二是用于日志压缩策略，作为消息去重的唯一键，保留相同 Key 的最新版本消息；
2. **Value**：消息主体，字节数组类型，必选字段，存储用户的业务数据。Kafka 本身不感知 Value 的序列化格式，支持 JSON、Protobuf、Avro、Thrift 等任意序列化方式，业务可按需选择；
3. **Timestamp**：消息时间戳，64 位长整型，单位为毫秒。分为两种类型，可在 Topic 级别配置：
    
    - CreateTime：默认值，生产者创建消息时的本地时间戳，由生产者客户端生成；
    - LogAppendTime：Broker 将消息追加到日志时的服务端时间戳，由 Broker 节点统一生成，适用于对时间一致性要求高的场景；
        
        核心用途包括日志保留策略、消息检索、流处理窗口计算、数据回溯等。
    
4. **Headers**：消息头，键值对集合，可选字段。键为 UTF-8 字符串类型，值为字节数组类型，用于传递消息的元数据，如链路追踪信息、序列化类型标识、业务标签、权限标识等，无需修改 Value 主体即可扩展消息元数据，不影响分区路由与序列化逻辑；
5. **系统元数据字段**：由 Broker 追加消息时自动生成，用户不可修改，包括 Partition 编号、Offset 偏移量、消息大小、CRC 校验码等，用于消息定位、完整性校验与元数据管理。

### 2.6.2 Record 物理存储格式（Record Batch）

Kafka 0.11.0 版本引入了**Record Batch**（记录批次）格式，替代了旧版本的 Message Set 格式，3.x 版本全量沿用该格式。Kafka 并非单条 Record 独立存储，而是将多条 Record 封装为一个 Record Batch，进行批量写入、传输、压缩与存储，是其高吞吐能力的核心优化之一。

Record Batch 是 Kafka 底层存储与传输的最小单元，其二进制结构核心字段如下：

1. **批次 CRC 校验码**：32 位循环冗余校验码，用于校验整个批次数据的完整性，检测网络传输与磁盘存储过程中的数据损坏；
2. **版本号**：标识 Record Batch 的格式版本，用于兼容不同版本的 Kafka 客户端与服务端；
3. **生产者 ID（PID）与批次序列号**：用于实现幂等性生产者与事务机制，保障消息的幂等写入与跨分区原子性，是 Kafka Exactly-Once 语义的底层基础；
4. **压缩类型标识**：标识批次采用的压缩算法，支持 none、gzip、snappy、lz4、zstd 五种类型，批次内的所有 Record 统一压缩，大幅提升压缩比，减少网络传输与磁盘存储开销；
5. **时间戳基准值与类型**：批次内所有 Record 的时间戳采用差值编码，基于基准时间戳存储相对值，大幅减少存储空间占用；
6. **记录数量**：批次内包含的 Record 总数，用于校验批次完整性；
7. **压缩 Record 集合**：批次内的多条 Record，采用差值编码、变长编码进行紧凑存储，进一步降低存储空间占用，提升 IO 效率。

Record Batch 的核心设计优势：批量压缩提升压缩比，降低网络与磁盘开销；批量写入强化了顺序写盘的性能优势；原生支持幂等性与事务特性，是 Kafka 高级功能的底层载体；紧凑的编码格式大幅降低了存储冗余。

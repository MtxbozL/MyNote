
Partition 是 Topic 下属的**物理存储单元**，是 Kafka 实现分布式存储、并行读写、水平扩展的核心载体，也是 Kafka 极致高吞吐能力的核心设计来源。一个 Topic 可划分为一个或多个 Partition，每个 Partition 对应 Broker 节点上的一个独立物理目录，存储该分区的消息日志文件。

### 2.4.1 核心设计原理

1. **分布式分片存储**：同一个 Topic 的不同 Partition 可均匀分布在集群的不同 Broker 节点上，实现了数据的分片存储，突破了单节点的存储、IO、算力瓶颈，Topic 的存储能力可随集群 Broker 节点的扩容线性提升；
2. **全链路并行化处理**：Kafka 的生产与消费并行粒度均为 Partition，是高吞吐的核心来源：
    
    - 生产端：生产者可通过分区路由策略，将消息并行分发到不同 Partition，多个 Partition 可同时接收写入请求，实现写入并行化；
    - 消费端：同一个消费组内，一个 Partition 只能被一个消费者消费，多个 Partition 可被消费组内的多个消费者并行消费，实现消费并行化，消费能力可随分区数量线性扩展；
    
3. **消息顺序性保障**：每个 Partition 是一个严格的**append-only 有序队列**，消息被顺序追加到 Partition 的末尾，分配唯一的、连续递增的 64 位整数 Offset（偏移量），作为消息在该分区内的唯一标识。Kafka 通过 Partition 实现了单分区内的强消息顺序性，这是 Kafka 消息顺序性保障的唯一实现机制。

### 2.4.2 核心特性

- **Offset 偏移量机制**：Offset 是 Partition 内消息的唯一标识，仅在单分区内有效，跨分区不具备可比性。消费者通过 Topic+Partition+Offset 三元组，可精准定位并读取任意消息，同时通过提交 Offset 记录消费进度；
- **不可变日志特性**：消息写入 Partition 后，不可修改、不可随机删除，仅可通过日志清理策略进行批量删除或压缩，保障了日志的 append-only 顺序写入特性，是 Kafka 顺序写盘高性能的核心基础；
- **分区扩容限制**：Kafka 仅支持 Topic 分区数量的扩容，不支持缩容。分区缩容会导致数据丢失、消费偏移量失效、消息顺序性破坏，因此 Kafka 原生不提供该能力；
- **全局有序性约束**：仅当 Topic 的分区数量为 1 时，可实现 Topic 内的全局消息有序；多分区场景下，仅能保障单分区内消息有序，跨分区消息无序，该特性是业务设计中必须关注的核心约束。


Kafka 元数据是集群运行的核心控制数据，包括集群 Broker 节点信息、Topic 与 Partition 配置、分区副本分布、Leader 副本位置、ISR 集合、消费组元数据、事务状态、集群配置等。元数据管理系统是 Kafka 集群的 “大脑”，负责元数据的存储、强一致性保障、选主、故障检测与状态同步，分为 ZooKeeper 时代与 KRaft 架构两个演进阶段。

### 2.7.1 ZooKeeper 时代（Kafka 2.8 以前的核心依赖）

ZooKeeper（简称 ZK）是一套基于 ZAB 协议的分布式一致性协调服务，在 Kafka 2.8 版本之前，是 Kafka 集群的强依赖组件，所有核心控制逻辑均依赖 ZK 实现。

#### 核心职责

1. **集群 Controller 选举**：所有 Broker 启动后会在 ZK 的`/controller`路径抢占创建临时节点，创建成功的 Broker 成为集群唯一的 Active Controller，负责所有控制平面操作；当 Controller 故障时，临时节点自动删除，触发新一轮选举，保障控制平面高可用；
2. **Broker 节点生命周期管理**：所有 Broker 节点在 ZK 的`/brokers/ids`路径创建临时节点，注册自身 IP、端口、版本等信息；Controller 通过监听该路径的节点变化，感知 Broker 上下线事件，触发分区 Leader 重选举、故障转移等操作；
3. **Topic 与 Partition 元数据存储**：所有 Topic 的配置、分区数、副本分布、ISR 集合等元数据，存储在 ZK 的`/brokers/topics`路径下，Controller 通过该路径的元数据管理 Topic 全生命周期；
4. **集群配置管理**：集群全局配置、Topic 级别动态配置、客户端配额配置，均存储在 ZK 的`/config`路径下，支持动态更新，无需重启集群；
5. **消费组元数据管理**：Kafka 0.9 版本之前，消费组的消费偏移量全量存储在 ZK 中；0.9 版本之后，偏移量迁移至内部 Topic `__consumer_offsets`，ZK 仅存储消费组的基础元数据。

#### 核心架构缺陷

ZooKeeper 架构是 Kafka 早期版本的核心依赖，但随着集群规模的扩大，其缺陷逐渐凸显，也是 KRaft 架构推出的核心原因：

1. **外部依赖复杂度高**：Kafka 集群需额外部署、运维一套 ZK 集群，增加了部署、监控、故障排查的复杂度，提升了运维成本；
2. **性能瓶颈**：ZK 的写入性能有限，当集群 Topic、Partition 数量达到上万级别时，ZK 的元数据更新会出现严重性能瓶颈，限制了 Kafka 集群的规模上限；
3. **脑裂风险**：ZK 的 Session 超时机制可能导致 Controller 脑裂，出现双 Active Controller 节点，引发集群元数据不一致；
4. **元数据同步延迟**：元数据读写需经过 ZK 集群，Controller 与 Broker 之间的元数据同步依赖 ZK 监听机制，延迟较高，可能导致客户端请求路由错误；
5. **安全管控割裂**：需同时维护 Kafka 与 ZK 两套集群的安全策略，无法实现统一的身份认证、权限控制与数据加密。

### 2.7.2 KRaft 架构（Kafka 2.8 + 引入，3.x + 主推，去 ZK 化）

KRaft（Kafka Raft）是 Kafka 内置的、基于 Raft 分布式共识协议的元数据管理机制，完全替代了 ZooKeeper 的所有功能，实现了 Kafka 集群的自包含，无需任何外部依赖，是 Kafka 架构的重大演进，3.3 版本之后正式生产可用，3.x 版本成为官方主推架构。

#### 核心角色划分

KRaft 架构中，Broker 进程通过`process.roles`配置，分为三种角色，实现控制平面与数据平面的职责分离：

1. **Controller 节点**：`process.roles=controller`，又称元数据控制器节点，多个 Controller 节点组成 Raft 共识集群，承担控制平面全部职责，替代了原 ZK 与 Active Controller 的功能。Raft 集群中会选举出一个 Leader 节点（Active Controller），其他节点为 Follower，同步元数据日志，保障控制平面高可用；
2. **Broker 节点**：`process.roles=broker`，又称数据节点，仅承担数据平面职责，负责消息存储与生产消费请求处理，不参与元数据共识，通过元数据复制机制从 Active Controller 同步集群全量元数据；
3. **混合节点**：`process.roles=broker,controller`，同时承担 Controller 与 Broker 角色，适用于单机部署、小规模集群，大幅简化部署架构。

#### 核心实现原理

1. **元数据日志机制**：KRaft 架构中，所有集群元数据的变更都被封装为事件，写入内部主题`__cluster_metadata`，该主题是 Raft 协议的日志载体，仅由 Controller 节点管理，采用 append-only 顺序写入模式，复用了 Kafka 的高性能日志存储能力；
2. **Raft 共识协议**：Controller 集群基于标准 Raft 协议实现元数据的强一致性，只有当元数据变更事件被 Raft 集群的多数节点写入成功后，才会被提交生效，保障了元数据的线性一致性；同时 Raft 的 Leader 选举机制，可在 Active Controller 故障时，数百毫秒内完成新 Leader 选举，远快于 ZK 时代的选举速度；
3. **事件驱动的元数据同步**：Active Controller 会将提交后的元数据事件，主动推送给所有 Broker 节点，Broker 在本地维护全量元数据缓存，无需主动轮询，大幅降低了元数据同步延迟，提升了客户端路由的准确性；
4. **无状态控制平面**：所有集群状态都存储在元数据日志中，Controller 节点故障重启后，可通过重放元数据日志快速恢复状态，重新加入 Raft 集群，无需依赖外部存储。

#### 核心架构优势

1. **去外部依赖，简化运维**：无需部署 ZK 集群，Kafka 集群可独立运行，大幅降低了部署、运维、监控的复杂度，减少了集群故障点；
2. **极致的可扩展性**：元数据管理复用了 Kafka 的高性能日志存储机制，支持十万级以上的 Partition 规模，集群扩展能力远超 ZK 时代，解决了大规模集群的性能瓶颈；
3. **更强的一致性保障**：基于标准 Raft 协议实现元数据共识，一致性模型清晰可控，彻底解决了 ZK 时代的 Controller 脑裂问题；
4. **更快的故障恢复**：Controller 故障选举耗时从数秒级降低至数百毫秒级，Broker 节点故障的故障转移速度大幅提升，显著减少了服务不可用时间；
5. **统一的安全管控**：Kafka 原生的 SASL 身份认证、SSL 数据加密、ACL 权限控制，可统一覆盖控制平面与数据平面，实现了全链路的安全管控，无需维护两套安全体系。

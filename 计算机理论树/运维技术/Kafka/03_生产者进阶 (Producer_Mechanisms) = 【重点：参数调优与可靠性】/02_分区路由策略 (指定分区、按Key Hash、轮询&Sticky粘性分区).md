
分区路由策略的核心目标是：在满足业务顺序性要求的前提下，实现消息在 Topic 各分区的均匀分布，最大化利用集群的并行处理能力，保障负载均衡。Kafka 提供 4 种内置路由策略，同时支持自定义分区器实现个性化路由逻辑。

### 3.2.1 路由优先级规则

Kafka 生产者的分区路由存在严格的优先级顺序，高优先级规则生效时，低优先级规则完全失效：

1. 最高优先级：ProducerRecord 中**显式指定分区编号**，直接使用该编号作为目标分区，分区器完全不执行；
2. 次高优先级：自定义分区器，实现`Partitioner`接口的自定义路由逻辑；
3. 最低优先级：内置默认分区器，Kafka 2.4 + 版本的默认分区器为`DefaultPartitioner`，兼容以下 3 种内置策略。

### 3.2.2 内置路由策略详解

#### 1. 显式指定分区策略

该策略为最高优先级路由方式，业务代码在构建`ProducerRecord`时，直接通过`partition`参数指定目标分区编号，生产者直接将消息发送至该分区，不经过分区器计算。

- **核心优势**：完全可控的消息分布，可实现严格的分区级消息顺序性；
- **核心劣势**：强依赖分区数量，Topic 分区扩容后需手动修改路由逻辑，极易出现负载不均，单分区压力过载；
- **适用场景**：对消息全局顺序性有强要求的场景（Topic 分区数固定为 1）、灰度发布、指定分区的消息补发等特殊场景。

#### 2. 按 Key 哈希路由策略

该策略是默认分区器的核心逻辑之一，当消息指定了 Key、未显式指定分区时，自动触发该策略。

- **核心实现原理**：对消息 Key 的字节数组执行 MurmurHash2 哈希算法（非加密型高性能哈希算法），得到哈希值后，对 Topic 的**可用分区总数**取模，最终结果即为目标分区编号。公式为：`targetPartition = Math.abs(Utils.murmur2(keyBytes)) % numPartitions`。
- **核心特性**：相同 Key 的消息，一定会被路由至同一个分区，天然保障单 Key 维度的消息严格顺序性，是 Kafka 最常用的顺序性保障方案。
- **核心风险与约束**：
    
    1. 分区扩容风险：Topic 分区数量扩容后，相同 Key 的哈希取模结果会发生变化，Key 与分区的映射关系被打破，顺序性保障失效，业务设计时需提前规划分区数量，避免中途扩容；
    2. 负载倾斜风险：若 Key 分布不均（如大量消息使用相同 Key），会导致少数分区流量过载，多数分区闲置，出现严重的负载不均，需保证 Key 的均匀分布。
    

#### 3. 轮询策略（Round-Robin）

该策略为 Kafka 2.4 版本之前的默认无 Key 路由策略，当消息未指定 Key、未显式指定分区时，生产者按顺序轮询 Topic 的所有可用分区，将消息均匀分发至每个分区。

- **核心优势**：消息在所有分区完全均匀分布，集群负载均衡效果最优；
- **核心劣势**：无 Key 消息会被分散到所有分区，每个分区的批次很难凑满`batch.size`，产生大量小批次网络请求，大幅降低压缩比与吞吐性能，增加 Broker 的 IO 压力。
- **适用场景**：对消息顺序性无要求、流量极大、单批次可快速凑满的场景，目前已被粘性分区策略替代。

#### 4. Sticky 粘性分区策略

该策略是 Kafka 2.4 + 版本默认分区器的无 Key 核心路由策略，专为解决轮询策略的小批次问题设计，核心目标是：在保障最终负载均衡的前提下，最大化提升批次填充率，减少网络请求次数，提升吞吐性能。

- **核心实现原理**：
    
    1. 生产者启动时，随机选择一个可用分区作为「粘性分区」，持续将所有无 Key 消息追加到该分区的批次中，直到该批次满足`batch.size`或`linger.ms`的就绪条件，被 Sender 线程发送；
    2. 当粘性分区的批次发送完成后，重新选择一个新的粘性分区，重复上述过程，选择时会优先选择负载最低的分区，保证最终的负载均衡。
    
- **核心优势**：
    
    1. 大幅提升批次填充率，减少小批次数量，提升压缩比与网络 IO 效率，吞吐性能相比轮询策略提升 30% 以上；
    2. 保证消息在所有分区的最终均匀分布，不会出现长期的负载倾斜；
    3. 对业务无侵入，无需修改代码，仅需升级客户端版本即可生效。
    
- **适用场景**：无 Key 消息的通用场景，是生产环境无 Key 消息的首选路由策略。

### 3.2.3 自定义分区器

业务可通过实现`org.apache.kafka.clients.producer.Partitioner`接口，自定义分区路由逻辑，满足个性化业务需求。核心需实现 3 个方法：

1. `partition()`：核心路由方法，输入 Topic、Key、Key 字节数组、Value、Value 字节数组、集群元数据，输出目标分区编号；
2. `close()`：分区器关闭时的资源释放逻辑；
3. `configure()`：分区器初始化时的配置加载逻辑。

- **典型适用场景**：基于业务字段（非 Key）的路由、按租户 / 业务线隔离分区、灰度流量路由、优先级消息分区隔离等。

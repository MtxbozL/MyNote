
Kafka 的消息顺序性是业务落地的核心诉求之一，尤其在金融、订单、交易等核心业务场景，消息的处理顺序直接决定业务状态的正确性。**Kafka 并非天生支持全局消息有序，仅能保证单分区内的消息严格有序**，这是由其分区的设计原理决定的。业务的顺序性保障需根据实际需求，选择**全局有序**或**局部有序**方案，避免过度设计，同时保证业务正确性。

### 7.4.1 Kafka 消息顺序性的核心基础：单分区有序原则

Kafka 的消息顺序性基于**分区的 append-only 有序队列**设计，核心原则为：**同一个 Partition 内的消息，严格按照生产者发送的顺序写入，消费者严格按照写入顺序消费，实现单分区内的强顺序性；跨分区的消息，无任何顺序性保障**。

- **底层保障**：Partition 的日志文件是 append-only 的有序队列，消息写入时分配连续递增的 Offset，消费者按 Offset 顺序拉取消息，从物理存储层面保证了单分区有序；
- **生产端保障**：生产者的分区路由策略保证相同 Key 的消息被路由至同一个 Partition，且生产者的幂等性机制保证重试不会导致单分区内的消息乱序；
- **消费端保障**：一个 Partition 只能被一个消费者消费，避免了多消费者并行消费导致的乱序，同时消费端按 Offset 顺序拉取消息，保证消费顺序与写入顺序一致。

该原则是 Kafka 所有顺序性保障方案的基础，所有业务顺序性设计都需围绕该原则展开。

### 7.4.2 消息顺序性的两种业务需求：全局有序 vs 局部有序

生产环境的消息顺序性需求分为**全局有序**与**局部有序**两类，二者的业务诉求、实现方案、性能损耗差异显著，需根据业务实际需求选择，**99% 的业务场景仅需局部有序，无需全局有序**。

#### 1. 全局有序

**定义**：整个 Topic 内的所有消息，严格按照生产者发送的顺序被消费者处理，跨分区也需保证顺序性。

**业务诉求**：适用于对消息顺序性有极致要求的场景，如全局交易流水、全局日志审计、全局事件溯源，要求所有消息的处理顺序与生产顺序完全一致。

**实现代价**：性能损耗极大，Topic 的分区数必须为 1，消费并行度为 1，Topic 的吞吐能力被限制在单分区的处理能力（约 1 万 - 2 万 QPS），无法水平扩展。

#### 2. 局部有序

**定义**：按**业务维度**划分的消息子集，严格按照生产顺序处理，不同子集之间的消息无需保证顺序性，即「分区内有序，分区间无序」。

**业务诉求**：适用于绝大多数业务场景，如订单业务（同一个订单的消息有序）、支付业务（同一个用户的支付消息有序）、物流业务（同一个运单的物流消息有序），仅需保证同一业务维度的消息有序，不同维度之间无顺序要求。

**实现代价**：性能无损耗，Topic 可配置多个分区，实现生产与消费的并行化，吞吐能力可水平扩展，是生产环境的首选方案。

### 7.4.3 全局有序的实现方案：单 Partition 配置

Kafka 实现全局有序的唯一方案是**将 Topic 的分区数设置为 1**，整个 Topic 只有一个 Partition，所有消息都写入该分区，消费者也只有一个实例消费该分区，从根本上保证全局消息有序。

#### 实现步骤

1. **创建单分区 Topic**：`kafka-topics.sh --create --bootstrap-server <broker> --topic <topic> --partitions 1 --replication-factor 3`；
2. **生产者配置**：无需指定 Key，使用默认分区策略即可，所有消息都会写入唯一的 Partition；
3. **消费组配置**：消费组内仅启动一个消费者实例，避免 Rebalance 导致的消费乱序；

#### 核心限制

1. **吞吐能力受限**：单 Partition 的生产与消费吞吐能力有限，约 1 万 - 2 万 QPS，无法应对高吞吐场景；
2. **无并行扩展能力**：分区数无法扩容，消费并行度也无法提升，业务规模增长后会成为性能瓶颈；
3. **高可用风险**：单 Partition 若出现故障，整个 Topic 的服务会中断，需配置副本因子 3，提升高可用能力；

#### 适用场景

仅适用于**低吞吐、对全局顺序性有极致要求**的小众场景，如全局交易流水、全局配置同步，核心高吞吐业务严禁使用。

### 7.4.4 局部有序的实现方案：按业务 Key 分区（生产环境首选）

局部有序是生产环境的主流方案，核心实现原理是：**基于业务唯一标识（如订单 ID、用户 ID、运单 ID）作为消息的 Key，生产者通过按 Key 哈希路由策略，将同一业务维度的消息路由至同一个 Partition，实现单业务维度的局部有序**。该方案兼顾了顺序性与高性能，是 Kafka 顺序性保障的行业最佳实践。

#### 实现步骤（生产环境可直接落地）

1. **创建多分区 Topic**：根据业务吞吐能力配置合理的分区数（如 16、32、64），副本因子 3，保证高可用与并行度；
2. **生产者消息封装**：将**业务唯一标识**作为消息的 Key（如订单 ID 为`order_123456`，则 Key 设置为该字符串），Value 为业务消息体；
3. **生产者配置**：使用 Kafka 默认的分区器（`DefaultPartitioner`），无需自定义分区器，默认的按 Key 哈希路由策略会将相同 Key 的消息路由至同一个 Partition；
    
    - 核心配置：`partitioner.class=org.apache.kafka.clients.producer.internals.DefaultPartitioner`（默认）；
    
4. **消费组配置**：启动与分区数匹配的消费者实例，实现消费并行化，每个消费者消费一个或多个 Partition，保证单 Partition 内的消息被单线程消费，避免乱序；
5. **消费端处理**：消费端无需额外的顺序性处理，按正常逻辑消费即可，单 Partition 内的消息会按生产顺序被处理。

#### 核心保障机制

1. **生产端**：相同 Key 的消息通过哈希路由至同一个 Partition，且生产者开启幂等性（`enable.idempotence=true`），保证重试不会导致单分区内的消息乱序；
2. **Broker 端**：单 Partition 的 append-only 有序队列，保证消息写入顺序与生产顺序一致；
3. **消费端**：一个 Partition 只能被一个消费者消费，且消费者按 Offset 顺序拉取消息，保证消费顺序与写入顺序一致。

#### 关键优化与避坑点

1. **保证 Key 的均匀分布**：Key 的分布直接决定 Partition 的负载均衡，若 Key 分布不均（如大量消息使用相同 Key），会导致少数 Partition 过载，多数 Partition 闲置，需保证 Key 的全局均匀分布；
    
    - 优化方案：若业务 Key 存在热点（如某个用户的消息量极大），可在 Key 后拼接随机数（如`user_001_01`、`user_001_02`），将热点 Key 分散到多个 Partition，同时在消费端通过本地缓存实现顺序性处理；
    
2. **提前规划分区数，避免中途扩容**：按 Key 哈希路由的核心是`hash(key) % numPartitions`，若中途扩容分区数，会导致 Key 与 Partition 的映射关系改变，同一 Key 的消息被路由至不同 Partition，破坏局部有序性；
    
    - 优化方案：根据业务峰值流量的 2 倍配置分区数，提前规划，避免生产环境中途扩容；
    
3. **禁止使用自定义分区器**：默认的`DefaultPartitioner`已实现高效的按 Key 哈希路由，且经过官方大量测试，自定义分区器容易出现哈希不均、顺序性破坏等问题，非必要不自定义；
4. **消费端禁止多线程处理同一个 Partition 的消息**：多线程消费同一个 Partition 的消息会导致乱序，需保证一个 Partition 的消息由一个线程处理，多线程消费仅适用于跨 Partition。

### 7.4.5 特殊场景的顺序性保障：跨分区顺序性

部分业务场景需要**跨分区的局部顺序性**（如同一个商家的多个订单消息有序），但又希望通过多分区实现并行化，此时可采用 **「分区分组 + 本地缓存」** 方案，兼顾并行度与跨分区顺序性。

#### 核心原理

1. 按商家 ID 哈希路由至**分区组**（如将 64 个分区划分为 8 个分区组，每个组 8 个分区），同一个商家的消息路由至同一个分区组；
2. 每个分区组由一个独立的消费线程处理，消费线程拉取分区组内所有 Partition 的消息，通过**本地缓存 + 时间戳排序**实现跨 Partition 的顺序性处理；

#### 优势

兼顾了并行度（分区组数量为并行度）与跨分区顺序性，适用于中等吞吐的跨分区顺序性场景；

#### 劣势

实现复杂，需要自定义分区器与消费端排序逻辑，且消费端有一定的内存开销，普通业务场景不建议使用。


Kafka 的设计目标是处理**小消息（1KB-10KB）** 的高吞吐传输，默认配置对消息大小有严格限制，若直接发送大消息（如 100MB、1GB），会导致 Kafka 性能急剧下降、Broker 端 OOM、网络拥堵等问题。生产环境中，大消息的定义为**超过 1MB 的消息**，针对大消息处理，Kafka 提供了**参数调优直接传输**与**对象存储 + Kafka 传链接**两种方案，分别适配不同的大消息场景，核心原则是**尽量避免直接通过 Kafka 传输大消息**。

### 7.6.1 Kafka 默认的消息大小限制

Kafka 的消息大小限制分布在**生产者端、Broker 端、消费者端**三个环节，任一环节的限制都会导致大消息传输失败，默认配置如下：

1. **Broker 端**：
    
    - `message.max.bytes`：单个消息的最大字节数，**默认 1048576（1MB）**，Broker 端的核心限制，超过该值的消息会被 Broker 拒绝；
    - `replica.fetch.max.bytes`：Follower 副本从 Leader 拉取的最大消息字节数，**默认 1048576（1MB）**，必须大于等于`message.max.bytes`，否则 Follower 无法同步大消息，导致副本同步异常；
    
2. **生产者端**：
    
    - `max.request.size`：单个生产者请求的最大字节数，**默认 1048576（1MB）**，必须小于等于 Broker 端的`message.max.bytes`，否则生产者发送的大消息会被 Broker 拒绝；
    
3. **消费者端**：
    
    - `fetch.max.bytes`：单个消费者拉取请求的最大字节数，**默认 52428800（50MB）**，必须大于等于 Broker 端的`message.max.bytes`，否则消费者无法拉取大消息；
    - `max.partition.fetch.bytes`：单个分区拉取的最大字节数，**默认 1048576（1MB）**，必须大于等于 Broker 端的`message.max.bytes`，否则消费者拉取单个分区的大消息会失败。
    

默认配置下，Kafka 仅支持最大 1MB 的消息传输，超过该值会导致传输失败，这是 Kafka 为保证高吞吐而做的默认限制。

### 7.6.2 方案一：参数调优直接传输大消息（适配小体积大消息，如 1MB-100MB）

对于**1MB-100MB**的小体积大消息，可通过调优 Kafka 的消息大小限制参数，实现直接传输，该方案实现简单、无侵入，无需修改业务逻辑，适配对传输实时性有要求的大消息场景。**注意：该方案仅支持最大 100MB 的消息，超过 100MB 会导致 Kafka 性能急剧下降，严禁使用**。

#### 全链路参数调优配置（生产环境可直接落地）

调优核心原则：**消费者端参数 ≥ Broker 端参数 ≥ 生产者端参数**，保证全链路的消息大小限制一致，无瓶颈。以下配置以支持**100MB**大消息为例，单位为字节，100MB=104857600 字节：

##### 1. Broker 端配置（全局或 Topic 级别，核心）

```properties
# 单个消息最大字节数，设置为100MB
message.max.bytes=104857600
# Follower副本拉取最大字节数，必须≥message.max.bytes
replica.fetch.max.bytes=104857600
# 副本拉取的超时时间，调大至30秒，避免拉取大消息超时
replica.fetch.timeout.ms=30000
```

- 建议在**Topic 级别**配置，仅对需要传输大消息的 Topic 生效，避免全局配置影响其他小消息 Topic 的性能；

##### 2. 生产者端配置

```properties
# 单个请求最大字节数，≤Broker端message.max.bytes
max.request.size=104857600
# 生产者缓冲区大小，调大至256MB，避免大消息占满缓冲区
buffer.memory=268435456
# 批次大小，调大至64KB，适配大消息的批量发送
batch.size=65536
# 发送超时时间，调大至30秒，避免大消息发送超时
request.timeout.ms=30000
```

##### 3. 消费者端配置

```properties
# 单个拉取请求最大字节数，≥Broker端message.max.bytes
fetch.max.bytes=104857600
# 单个分区拉取最大字节数，≥Broker端message.max.bytes
max.partition.fetch.bytes=104857600
# 拉取超时时间，调大至30秒，避免拉取大消息超时
fetch.max.wait.ms=30000
# 单次拉取最大消息数，调小至10，避免拉取过多大消息导致OOM
max.poll.records=10
```

#### 大消息直接传输的性能优化与注意事项

1. **开启消息压缩**：生产者配置`compression.type=lz4`或`zstd`，大消息的压缩比极高（如图片、视频、日志文件可达 10:1），大幅降低网络传输与磁盘存储开销；
2. **单独部署大消息 Topic**：将大消息 Topic 部署在**独立的 Broker 节点与磁盘**上，避免大消息的 IO 与网络开销影响其他小消息 Topic 的性能；
3. **控制大消息的发送频率**：避免高频发送大消息，否则会导致 Broker 的磁盘 IO 与网络带宽过载，影响 Kafka 集群的整体稳定性；
4. **消费端做内存控制**：大消息会占用大量的 JVM 内存，消费端需调大 JVM 堆内存（如 - Xmx4G），并设置`max.poll.records`为较小值，避免拉取过多大消息导致 OOM；
5. **监控大消息的传输指标**：监控大消息 Topic 的生产速度、消费速度、Broker 的磁盘 IO、网络带宽，指标异常时立即触发告警。

### 7.6.3 方案二：对象存储 OSS+Kafka 传链接（适配大体积大消息，如 100MB 以上）

对于**100MB 以上**的大体积大消息（如视频、音频、大文件、备份数据），**严禁通过 Kafka 直接传输**，会导致 Kafka 集群性能急剧下降、Broker OOM、网络拥堵等严重问题。行业内的最佳实践是**基于对象存储（OSS）+ Kafka 传链接**实现，将大文件存储在对象存储中，Kafka 仅传输文件的访问链接与元数据，兼顾了传输效率与 Kafka 的性能。

#### 核心原理

1. **对象存储作为大文件存储载体**：使用对象存储（如阿里云 OSS、腾讯云 COS、MinIO、S3）存储大文件，对象存储专为大文件存储设计，支持高吞吐、高可用、无限扩容，且成本极低；
2. **Kafka 传输链接与元数据**：生产者将大文件上传至对象存储后，将文件的**访问链接、文件名称、文件大小、业务元数据**封装为小消息，发送至 Kafka；
3. **消费者拉取链接并访问对象存储**：消费者从 Kafka 拉取包含文件链接的小消息，通过链接从对象存储下载大文件，执行业务处理。

该方案的核心是**将大文件的传输与存储剥离至对象存储，Kafka 仅承担轻量级的链接与元数据传输**，完全符合 Kafka 的设计目标，保证了 Kafka 集群的高吞吐与稳定性。

#### 实现步骤（生产环境可直接落地）

1. **部署对象存储服务**：根据业务需求选择对象存储，核心业务可使用商业云存储（阿里云 OSS、腾讯云 COS），中小业务可使用开源的 MinIO、Ceph，实现大文件的存储与访问；
2. **生产者实现大文件上传逻辑**：
    
    - 生产者接收到大文件发送请求后，将大文件上传至对象存储，获取文件的**预签名访问链接**（带过期时间，保证安全性）；
    - 将文件链接、文件名称、文件大小、业务元数据（如订单 ID、用户 ID）封装为 JSON 格式的小消息（大小通常 < 1KB）；
    - 将小消息发送至 Kafka 的普通 Topic（无需调优消息大小限制）；
    
3. **Kafka 传输小消息**：Kafka 按正常逻辑传输包含文件链接的小消息，保证高吞吐、低延迟，无任何性能损耗；
4. **消费者实现大文件下载与处理逻辑**：
    
    - 消费者从 Kafka 拉取包含文件链接的小消息；
    - 通过预签名链接从对象存储下载大文件至本地磁盘或内存；
    - 执行业务处理（如解析、转码、存储）；
    - 处理完成后，可根据业务需求删除对象存储中的大文件，或保留一定时间后自动清理；
    
5. **安全与容错兜底**：
    
    - 预签名链接设置**过期时间**（如 1 小时），避免链接泄露导致的文件安全问题；
    - 生产者上传文件与发送 Kafka 消息实现**原子性**（如数据库事务、分布式锁），避免文件上传成功但消息发送失败，或反之；
    - 消费者下载文件失败时，实现**重试机制**，避免大文件处理失败；
    - 对象存储配置**生命周期管理**，自动清理过期的大文件，避免存储资源浪费。
    

#### 优势与劣势

- **优势**：完全不影响 Kafka 集群性能，Kafka 仅传输小消息，保持高吞吐；对象存储专为大文件设计，支持高可用、无限扩容，成本极低；实现简单，无 Kafka 参数调优，业务逻辑清晰；
- **劣势**：引入对象存储中间件，增加了架构复杂度与运维成本；大文件的传输实时性依赖于对象存储的访问速度；
- **适用场景**：所有**100MB 以上**的大体积大消息场景，如视频转码、音频处理、大文件备份、大数据文件传输，是生产环境的行业最佳实践。

### 7.6.4 大消息处理的核心原则与避坑点

1. **核心原则**：**能不用 Kafka 传大消息，就不用**；1MB 以下用默认配置，1MB-100MB 用参数调优直接传输，100MB 以上用对象存储 + Kafka 传链接；
2. **避免高频发送大消息**：即使调优了参数，高频发送大消息也会导致 Kafka 的磁盘 IO 与网络带宽过载，影响集群稳定性，需对大消息的发送频率做限流；
3. **大消息 Topic 与小消息 Topic 隔离**：将大消息 Topic 部署在独立的 Broker 节点与磁盘，避免相互影响；
4. **保证大消息的可靠性**：无论是直接传输还是传链接，都需实现大消息的零丢失保障，如生产者本地落盘、对象存储多副本、消费端幂等性；
5. **监控大消息的全链路指标**：监控对象存储的上传 / 下载速度、Kafka 的小消息传输指标、消费者的文件处理速度，实现全链路监控告警。

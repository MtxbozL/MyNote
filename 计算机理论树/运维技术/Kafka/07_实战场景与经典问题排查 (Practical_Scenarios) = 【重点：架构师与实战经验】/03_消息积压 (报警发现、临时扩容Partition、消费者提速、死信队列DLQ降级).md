
消息积压是 Kafka 生产环境最常见的性能故障，表现为**Kafka Topic 的分区消息数持续增长，消费者的消费速度远低于生产者的生产速度**，若不及时处理，会导致磁盘空间占满、业务延迟飙升、甚至消息丢失。消息积压的处理核心遵循 **「先止损，再排查，最后优化」** 的原则，先通过临时方案快速降低积压，再排查根因，最后从架构层面优化避免再次发生。

### 7.3.1 消息积压的核心发现方式：监控告警体系

消息积压的处理前提是**及时发现**，需搭建完善的监控告警体系，监控 Topic 的积压指标，指标异常时立即触发告警，实现积压的早发现、早处理。

#### 核心监控指标

1. **分区消息积压数**：`分区当前最大Offset - 消费组当前提交Offset`，该值持续增长则表示存在积压，是最核心的积压指标；
2. **消费延迟时间**：根据积压数与消费速度计算的消息处理延迟，如积压 10 万条，消费速度 100 条 / 秒，则延迟 1000 秒；
3. **生产速度（MessagesInPerSec）**：生产者每秒发送到 Topic 的消息数，用于判断生产端是否突发流量；
4. **消费速度（MessagesOutPerSec）**：消费者每秒消费的消息数，用于判断消费端是否处理缓慢；
5. **消费者线程状态**：消费线程是否阻塞、是否有异常，用于判断消费端是否故障；
6. **磁盘使用率**：Topic 所在 Broker 的磁盘使用率，避免积压导致磁盘满。

#### 告警触发策略

1. **阈值告警**：积压数超过阈值（如 1 万条）、消费延迟超过阈值（如 5 分钟），立即触发告警（短信、钉钉、企业微信）；
2. **趋势告警**：积压数连续 5 分钟呈增长趋势，即使未达到阈值，也触发告警，提前预判积压风险；
3. **紧急告警**：磁盘使用率超过 80%、积压数超过 10 万条，触发紧急告警，立即介入处理。

### 7.3.2 消息积压的临时处理方案：先止损，快速降低积压

当发现消息积压后，首先执行**临时应急方案**，快速提升消费速度，降低积压，避免问题扩大，这是处理积压的核心步骤，无需先排查根因，遵循「先解决问题，再分析原因」。

#### 方案一：临时扩容消费者实例，提升消费并行度

**核心原理**：基于 Kafka 消费组的核心铁律（一个分区只能被一个消费者消费），通过增加消费组内的消费者实例数量，提升消费并行度，消费速度随消费者数量线性提升（不超过分区数）。

- **操作步骤**：
    
    1. 快速启动多个消费组相同的消费者实例（临时容器、临时虚拟机），数量不超过 Topic 的总分区数（多余的实例会闲置）；
    2. 监控消费速度与积压数，当积压数开始下降后，持续运行至积压清零；
    3. 积压清零后，逐步缩容临时实例，恢复原有部署架构；
    
- **优势**：操作简单、无侵入、见效快，无需修改代码与配置；
- **适用场景**：因消费端实例数不足、单实例故障导致的积压，是最常用的临时处理方案。

#### 方案二：临时扩容 Topic 分区数，提升消费并行度上限

**核心原理**：若 Topic 的分区数过少，导致消费组的消费者实例数无法继续扩容（已达到分区数），通过临时扩容 Topic 分区数，提升消费并行度的上限，再扩容消费者实例，实现消费速度的二次提升。

- **操作步骤**：
    
    1. 通过 CLI 工具扩容 Topic 分区数：`kafka-topics.sh --alter --bootstrap-server <broker> --topic <topic> --partitions <new_num>`；
    2. 扩容后，消费组会自动触发 Rebalance，将新分区分配给消费者；
    3. 继续扩容消费者实例至新的分区数，提升消费速度；
    
- **注意事项**：
    
    1. Kafka 仅支持分区扩容，不支持缩容，扩容后需评估是否长期保留新分区；
    2. 若生产者使用**按 Key 哈希路由**，分区扩容后会导致 Key 与分区的映射关系改变，可能影响消息顺序性，核心业务需提前规划分区数；
    
- **适用场景**：因 Topic 分区数不足，导致消费并行度达到上限的积压场景。

#### 方案三：消费端提速优化，提升单消费者处理能力

**核心原理**：通过修改消费端配置与代码，提升单个消费者实例的消息处理速度，无需扩容实例，快速见效。

- **核心优化点**：
    
    1. **调大单次拉取数**：配置`max.poll.records`（默认 500），根据业务处理能力调大至 1000-2000，提升单次拉取的消息量，减少网络请求次数；
    2. **开启批量消费**：使用 Spring Kafka 的`BatchListener`或原生批量拉取 API，批量处理消息，提升处理效率；
    3. **关闭自动提交，使用异步提交**：减少 Offset 提交的性能开销，提升消费速度；
    4. **临时关闭非核心业务逻辑**：注释掉消费端的日志上报、数据统计、非核心校验等逻辑，仅保留核心业务处理，临时提升处理速度；
    5. **提升消费线程池核心数**：多线程消费模型中，调大业务线程池的核心线程数，提升单实例的并行处理能力；
    
- **优势**：无需扩容实例，操作简单，快速提升消费速度；
- **适用场景**：因消费端单实例处理能力不足导致的积压场景。

#### 方案四：死信队列（DLQ）降级，隔离异常消息，避免阻塞正常消费

**核心原理**：若积压是因**部分异常消息处理失败**导致（如数据格式错误、业务校验失败），这些异常消息会阻塞整个分区的消费，导致积压持续增长。通过死信队列将异常消息隔离，让消费者继续处理正常消息，快速降低积压。

- **操作步骤**：
    
    1. 为目标 Topic 创建对应的死信队列 Topic（命名规范：`<topic>_dlq`）；
    2. 消费端修改代码，捕获业务处理异常，将异常消息发送至死信队列；
    3. 消费者跳过异常消息，继续处理后续正常消息，积压快速下降；
    4. 积压清零后，单独处理死信队列中的异常消息，排查异常根因；
    
- **优势**：快速隔离异常消息，避免正常消费被阻塞，从根本上解决因异常消息导致的积压；
- **适用场景**：因异常消息处理失败，导致消费线程阻塞的积压场景，这是生产环境最容易被忽视的积压根因。

#### 方案五：临时增加消费组，分流消费压力

**核心原理**：创建新的消费组，订阅目标 Topic，临时分流消费压力，两个消费组同时消费该 Topic，提升整体消费速度。

- **操作步骤**：
    
    1. 创建新的消费组（如原消费组为`test-group`，新消费组为`test-group-temp`）；
    2. 启动新消费组的消费者实例，订阅目标 Topic，开始消费；
    3. 积压清零后，停止新消费组的实例，避免重复消费；
    
- **注意事项**：新消费组会从最新 Offset 开始消费，若需消费历史积压消息，需调用`seek()`方法重置 Offset 到积压起始位置；
- **适用场景**：因生产端突发大流量，导致原有消费组处理能力不足的积压场景。

### 7.3.3 消息积压的根因排查：找到问题本质，避免再次发生

当积压通过临时方案得到控制后，立即执行**根因排查**，找到积压的根本原因，从代码、配置、架构层面优化，避免积压再次发生。消息积压的根因主要分为**生产端、消费端、Broker 端**三类，排查思路清晰，可通过监控指标与日志快速定位。

#### 根因排查思路（按优先级排序）

1. **消费端排查（占比 80%，最核心）**
    
    - 查看消费端日志：是否有异常（如数据库超时、网络异常、业务代码异常）、是否有阻塞（如死循环、慢查询）；
    - 监控消费端指标：消费速度是否下降、消费线程是否存活、CPU / 内存 / 磁盘是否过载；
    - 检查消费端配置：`max.poll.records`是否过小、`max.poll.interval.ms`是否过短、是否开启了不必要的业务逻辑；
    
2. **生产端排查（占比 15%）**
    
    - 监控生产端指标：生产速度是否突发增长（如大促、热点事件）、是否有批量生产操作；
    - 检查生产端配置：是否开启了压缩、是否有批量发送优化，是否存在不合理的高频生产；
    
3. **Broker 端排查（占比 5%）**
    
    - 监控 Broker 指标：是否有 UnderReplicatedPartitions（副本同步异常）、Broker 的 CPU / 内存 / 磁盘 IO 是否过载、网络是否拥堵；
    - 检查 Broker 配置：是否有不合理的参数限制（如`message.max.bytes`过小、限流配额过低）。
    

### 7.3.4 消息积压的长期优化方案：架构层面规避，实现长治久安

针对排查出的根因，从**代码、配置、架构、监控**四个维度进行长期优化，从根本上规避消息积压问题，实现消费链路的稳定性。

1. **消费端优化**：
    
    - 实现**多线程消费模型**（单线程拉取 + 线程池处理），提升单实例处理能力；
    - 实现**异常消息自动重试 + 死信队列**，避免异常消息阻塞正常消费；
    - 对消费端的**远程调用、数据库操作**做超时控制与熔断降级，避免外部依赖故障导致消费阻塞；
    - 核心业务消费端实现**集群化部署**，实例数等于或略小于 Topic 分区数，保证消费并行度；
    
2. **生产端优化**：
    
    - 实现**生产端流量削峰**，避免突发大流量直接写入 Kafka，可使用本地队列、限流等方式；
    - 开启生产端**批量发送 + 压缩**，提升生产效率，同时降低 Broker 的存储与 IO 压力；
    
3. **Topic 与集群优化**：
    
    - 提前规划 Topic**分区数**，根据业务的生产速度与消费速度，设置合理的分区数（建议按峰值流量的 2 倍配置）；
    - 核心业务 Topic 配置**多副本 + 高可靠参数**，避免 Broker 端故障导致的消费速度下降；
    - 对高吞吐 Topic 进行**磁盘隔离**，将其部署在独立的磁盘上，避免磁盘 IO 竞争；
    
4. **监控告警体系优化**：
    
    - 搭建**全链路监控**，覆盖生产端、Broker 端、消费端的所有核心指标；
    - 设置**多级告警**，实现积压的早发现、早处理，避免问题扩大；
    - 实现**消费积压趋势预测**，基于历史数据预测积压风险，提前介入优化。

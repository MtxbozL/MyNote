
Kafka 提供了两种核心的日志清理策略，用于管理磁盘存储空间，避免日志无限膨胀占用磁盘，分别是**基于时间 / 大小的日志删除策略**与**基于 Key 去重的日志压缩策略**，一个 Topic 只能选择一种清理策略，由`cleanup.policy`参数配置。

### 6.7.1 日志删除策略（Log Retention）

日志删除是 Kafka 的默认清理策略，`cleanup.policy=delete`，核心是根据保留规则，删除过期的、非活跃的日志分段文件，是针对整个分段的批量删除，而非单条消息的删除。

#### 核心保留规则

Kafka 提供了三个维度的保留规则，满足任一规则即触发分段删除，优先级：时间规则 > 大小规则 > 起始 Offset 规则。

1. **基于时间的保留规则**
    
    - 核心参数：`log.retention.hours`（默认 168 小时 = 7 天），`log.retention.minutes`、`log.retention.ms`，优先级 ms>minutes>hours；
    - 执行逻辑：非活跃分段的最后一条消息的时间戳，超过保留时间阈值后，该分段会被标记为待删除；
    - 适用场景：绝大多数业务场景，仅需保留一段时间内的热点消息，过期消息无需保留。
    
2. **基于大小的保留规则**
    
    - 核心参数：`log.retention.bytes`，默认 - 1（无限制），为**分区级别**的总日志大小上限；
    - 执行逻辑：当分区的总日志大小超过阈值后，从最旧的非活跃分段开始删除，直到总大小低于阈值；
    - 适用场景：磁盘空间有限，需要严格控制分区日志大小的场景。
    
3. **基于起始 Offset 的保留规则**
    
    - 执行逻辑：当消费者的最小消费 Offset 超过某个非活跃分段的最大 Offset 时，说明该分段的所有消息都已被所有消费组消费完成，可被安全删除；
    - 适用场景：需要保证所有消息都被消费后再删除的场景，避免消息未消费就被清理。
    

#### 清理执行流程

1. Kafka 后台的日志清理线程，每隔`log.retention.check.interval.ms`（默认 5 分钟），扫描所有分区的非活跃分段，匹配保留规则；
2. 符合删除条件的分段，会被重命名为`.deleted`后缀，避免清理过程中仍有读请求访问；
3. 延迟`log.segment.delete.delay.ms`（默认 60 秒）后，后台线程将.deleted 文件真正从磁盘删除，释放存储空间。

### 6.7.2 日志压缩策略（Log Compaction）

日志压缩是 Kafka 独有的清理策略，`cleanup.policy=compact`，核心是**保留同一个 Key 的最新版本的消息，删除旧版本的消息**，而非按时间删除。该策略适用于需要永久保留每个 Key 的最新状态的场景，无论时间多久。

#### 核心设计原理

日志压缩的核心目标，是保证日志中永远存在每个 Key 的最新版本，即使系统宕机重启，也能通过重放日志恢复全量 Key 的最新状态，是 Kafka 实现元数据持久化、事件溯源、配置同步的核心基础。

- 日志分区分为两个部分：`clean`部分（已完成压缩的日志段）与`dirty`部分（未压缩的日志段）；
- 后台的 Compactor 清理线程，每隔固定时间扫描 dirty 部分的日志，按 Key 进行去重，保留每个 Key 的最大 Offset 的消息，删除旧版本的消息；
- 清理完成后，生成新的压缩分段文件，替换旧的分段文件，整个过程对消费者完全透明，消费者仍按 Offset 正常消费。

#### 核心压缩规则

1. 仅对有 Key 的消息生效，无 Key 的消息不会被压缩处理；
2. 保留每个 Key 的最新版本消息，旧版本消息会被清理；
3. 墓碑消息（Value 为 null 的消息）：用于标记 Key 的删除，会在保留`delete.retention.ms`（默认 24 小时）后，连同该 Key 的所有历史消息一起删除；
4. 活跃分段不会被压缩，仅处理非活跃分段；
5. 压缩后的消息 Offset 保持不变，消费者不会感知到消息被清理，仅会跳过已被删除的旧版本消息；
6. 压缩触发阈值：`min.cleanable.dirty.ratio`（默认 0.5），当 dirty 部分的日志占比达到 50% 时，触发压缩操作，避免频繁压缩带来的性能损耗。

#### 典型适用场景

1. Kafka 内置的`__consumer_offsets`主题，采用压缩策略，永久保留每个消费组的最新消费 Offset；
2. 用户画像、会员信息同步场景，保留每个用户 ID 的最新画像数据；
3. 分布式配置中心，保留每个配置 Key 的最新版本；
4. 事件溯源系统的快照存储，保留每个聚合根的最新状态。

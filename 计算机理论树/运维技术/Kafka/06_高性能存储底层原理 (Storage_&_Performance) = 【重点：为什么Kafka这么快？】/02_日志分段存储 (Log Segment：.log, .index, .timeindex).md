
Kafka 并未将一个分区的所有消息存储在一个单一的大文件中，而是采用**日志分段（Log Segment）** 机制，将分区的日志拆分为多个固定大小的、不可变的小文件，每个文件称为一个日志分段。该设计是 Kafka 实现高效读写、过期数据清理、故障快速恢复的核心基础。

### 6.2.1 分段核心规则

1. **分段角色划分**：一个分区同一时间只有一个**活跃分段（Active Segment）**，负责接收新消息的写入，仅该分段可写；其余分段为**非活跃分段（Inactive Segment）**，已写满关闭，只读不可写。
2. **分段命名规则**：每个分段的文件名以该分段内第一条消息的**绝对 Offset**命名，采用 20 位数字补零对齐，例如`00000000000000000000.log`代表该分段的起始 Offset 为 0，`00000000000000100000.log`代表该分段的起始 Offset 为 100000。该命名规则可通过二分查找快速定位目标 Offset 所在的分段文件。
3. **分段滚动触发条件**：当活跃分段满足以下任一条件时，会触发分段滚动，当前活跃分段关闭成为非活跃分段，创建新的活跃分段：
    
    - 大小阈值：分段文件大小达到`log.segment.bytes`配置值，**默认 1GB**；
    - 时间阈值：分段存活时间达到`log.segment.ms`配置值，默认 7 天，优先级高于大小阈值；
    - 索引文件阈值：索引文件大小达到`log.index.size.max.bytes`配置值，默认 10MB，避免索引文件过大。
    

### 6.2.2 分段核心文件详解

每个日志分段由三个一一对应的文件组成，三者文件名完全一致，仅后缀不同，生命周期完全绑定。

#### 1. .log 日志数据文件

.log 文件是消息的实际存储载体，存储的是经过压缩的 Record Batch 二进制数据，而非单条消息，与第三章生产者的批次机制完全对应。文件采用 append-only 顺序写入模式，仅能在文件末尾追加新的 Record Batch，写入完成后不可修改，是 Kafka 顺序写盘特性的物理载体。

#### 2. .index 偏移量索引文件

.index 文件是与.log 文件配套的偏移量稀疏索引，存储消息 Offset 与对应物理文件位置的映射关系，用于快速定位目标 Offset 在.log 文件中的物理位置，避免全文件扫描。

#### 3. .timeindex 时间戳索引文件

.timeindex 文件是与.log 文件配套的时间戳稀疏索引，存储消息时间戳与对应 Offset 的映射关系，是 Kafka 实现按时间戳回溯消费、日志按时间清理的核心基础。

### 6.2.3 分段存储的核心性能优势

1. **过期数据清理高效**：日志清理的最小粒度为分段文件，无需修改正在写入的活跃文件，仅需删除过期的非活跃分段文件，避免了随机 IO 与文件内容修改，清理效率极高，对业务读写无影响；
2. **消息查找速度极快**：通过分段命名的二分查找，可快速锁定目标 Offset 所在的分段，无需扫描整个分区的大文件，查找耗时与分区总数据量无关，仅与单分段大小相关；
3. **内存占用极低**：仅需将当前活跃分段的索引文件加载到内存，非活跃分段的索引仅在查找时加载，极大降低了内存开销；
4. **故障恢复速度快**：Broker 宕机重启时，仅需校验活跃分段的完整性，非活跃分段无需校验，大幅缩短了故障恢复时间，提升了集群可用性。

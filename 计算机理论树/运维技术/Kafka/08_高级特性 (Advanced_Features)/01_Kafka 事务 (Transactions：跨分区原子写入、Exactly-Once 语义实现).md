## 8.1 Kafka 事务（Transactions: 跨分区原子写入、Exactly-Once 语义实现）

Kafka 事务是 0.11.0 版本引入的核心特性，基于前序章节讲解的**幂等性生产者**能力扩展而来，核心解决了幂等性无法覆盖的**跨分区、跨会话、跨 Topic 的原子性写入**问题，同时实现了流处理场景下「消费 - 处理 - 生产」全链路的**Exactly-Once（精确一次）语义**，彻底解决了分布式消息系统中「消息丢失、消息重复、数据不一致」三大核心问题。

### 8.1.1 事务的核心目标与语义边界

#### 1. 消息投递语义的层级定义

在分布式消息系统中，消息投递语义分为三个严格的层级，事务是实现最高层级语义的核心基础：

- **At-Most-Once（最多一次）**：消息最多被投递一次，可能丢失，绝不会重复，对应生产端`acks=0`、消费端先提交 Offset 后处理的模式；
- **At-Least-Once（至少一次）**：消息至少被投递一次，绝不会丢失，可能重复，对应生产端`acks=all`+ 重试、消费端先处理后提交 Offset 的模式，是 Kafka 的默认语义；
- **Exactly-Once（精确一次）**：消息恰好被处理一次，既不丢失，也不重复，是分布式系统的理想语义，仅能通过事务机制实现。

#### 2. 事务解决的核心场景

幂等性生产者仅能保证**单生产者会话内、单分区**的消息不重复、不乱序，无法覆盖以下核心场景，而事务机制完美解决了这些问题：

1. **跨分区 / 跨 Topic 的原子写入**：生产者向多个分区 / 多个 Topic 发送消息，要么全部写入成功，要么全部失败回滚，保证多分区数据的原子一致性，典型场景如分布式事务的消息通知、跨业务的事件联动；
2. **跨会话的幂等性保障**：通过用户指定的`TransactionalId`，实现生产者重启、会话切换后的幂等性，解决了幂等性生产者 PID 重启变更导致的幂等性失效问题；
3. **流处理全链路精确一次**：在「消费 Kafka 消息→业务处理→生产新消息到 Kafka」的典型流处理场景中，实现「消费 Offset 提交与新消息生产」的原子性，要么全部成功，要么全部回滚，彻底避免重复消费、重复生产导致的数据不一致，这也是 Kafka Streams、Flink 与 Kafka 集成实现精确一次语义的底层基础。

### 8.1.2 事务核心架构与组件

Kafka 事务的核心架构与消费组架构高度同源，核心由三大组件构成，所有事务元数据均持久化到 Kafka 内置系统 Topic 中，无需外部依赖。

#### 1. Transaction Coordinator（事务协调者）

事务协调者是运行在每个 Broker 节点上的核心组件，是事务的核心管理者，负责事务的全生命周期管理、事务 ID 与 PID 的映射管理、事务状态持久化。每个事务实例会绑定唯一的事务协调者，绑定规则与消费组绑定 GroupCoordinator 完全一致：根据`TransactionalId`的哈希值，路由到内置 Topic `__transaction_state`的对应分区，该分区的 Leader 副本所在的 Broker，即为该事务的事务协调者。

#### 2. __transaction_state 系统内部 Topic

该 Topic 是事务状态的持久化存储载体，默认 50 个分区、3 副本，采用日志压缩清理策略，仅保留每个`TransactionalId`的最新事务状态。核心存储内容包括：`TransactionalId`与 PID 的映射关系、事务当前状态、事务涉及的分区列表、事务 Epoch 号等，是事务协调者故障恢复、跨会话事务恢复的核心依据。

#### 3. 事务生产者与 TransactionalId

`TransactionalId`是用户为事务生产者指定的、全局唯一的业务标识，是实现跨会话幂等性的核心：

- Kafka 会为每个`TransactionalId`分配一个固定的 PID，无论生产者是否重启、会话是否切换，同一个`TransactionalId`永远绑定同一个 PID，彻底解决了幂等性生产者 PID 随会话变更的问题；
- 同时通过 Epoch 号机制，保证同一个`TransactionalId`同一时间只能有一个活跃的生产者实例，避免多实例并发操作导致的事务混乱，实现了 fencing（隔离）机制。

### 8.1.3 事务完整执行流程

Kafka 事务采用**两阶段提交（2PC）** 协议实现，完整流程分为 6 个核心阶段，全程对业务代码透明，仅需通过事务 API 标记事务的开启、提交与回滚。

#### 阶段 1：事务初始化（InitTransactions）

1. 事务生产者启动后，根据配置的`TransactionalId`，向集群查找对应的事务协调者；
2. 生产者向事务协调者发送初始化请求，协调者为该`TransactionalId`分配 PID 与递增的 Epoch 号，持久化到`__transaction_state`中；
3. 若该`TransactionalId`存在未完成的事务，协调者会先完成上一轮事务的提交 / 回滚，保证事务的幂等性与一致性，避免僵尸事务。

#### 阶段 2：开启事务（beginTransaction）

生产者调用`beginTransaction()`方法开启事务，仅在生产者本地标记事务的开始，不会与协调者发生网络交互，无性能开销。

#### 阶段 3：消息生产与分区注册

1. 生产者向目标分区发送消息，底层复用幂等性生产者的 PID+Sequence Number 机制，保证单分区内的消息不重复、不乱序；
2. 生产者第一次向某个分区发送消息时，会向事务协调者注册该分区，协调者将该分区加入当前事务的分区列表，持久化到`__transaction_state`中，用于后续的提交 / 回滚操作。

#### 阶段 4：消费 Offset 提交（消费 - 生产场景）

在流处理的「消费 - 处理 - 生产」场景中，生产者处理完消费的消息后，可通过`sendOffsetsToTransaction()`方法，将消费组的 Offset 提交与当前事务绑定。协调者会将 Offset 提交请求转发给对应的消费组协调者，仅当事务提交时，Offset 才会真正生效，实现「消费 Offset 提交与新消息生产」的原子性。

#### 阶段 5：事务预提交（Prepare Commit）

生产者调用`commitTransaction()`提交事务时，首先进入预提交阶段：

1. 生产者向事务协调者发送预提交请求；
2. 协调者将当前事务的状态更新为`PREPARE_COMMIT`，持久化到`__transaction_state`中；
3. 一旦进入预提交状态，无论后续发生任何故障，协调者都会保证该事务最终被提交，不会回滚。

#### 阶段 6：事务最终提交 / 回滚

- **提交流程**：
    
    1. 协调者向事务涉及的所有分区的 Leader Broker 发送提交请求，Broker 为该事务对应的消息设置「已提交」标记；
    2. 所有分区提交完成后，协调者将事务状态更新为`COMMITTED`，持久化到`__transaction_state`中，事务完成。
    
- **回滚流程**：
    
    生产者调用`abortTransaction()`回滚事务时，流程与提交一致，仅将事务状态更新为`PREPARE_ABORT`，最终向所有分区发送回滚请求，Broker 标记事务对应的消息为「已回滚」，最终更新事务状态为`ABORTED`。

### 8.1.4 消费者端事务隔离级别

Kafka 为消费者提供了两种事务隔离级别，由消费者端参数`isolation.level`配置，决定了消费者能否读取未提交 / 已回滚的事务消息，这是事务一致性的核心保障。

|隔离级别|配置值|核心行为|适用场景|
|---|---|---|---|
|读已提交|`read_committed`|仅能读取已提交的事务消息，无法读取未提交、已回滚的事务消息；非事务消息不受影响，全部可见|核心业务场景、对数据一致性有严格要求的流处理场景，生产环境推荐配置|
|读未提交|`read_uncommitted`|默认值，可读取所有消息，包括未提交、已回滚的事务消息，无任何隔离限制|非核心场景、对延迟敏感、不要求事务一致性的场景|

> 核心注意：`read_committed`隔离级别下，消费者的拉取请求会被阻塞，直到当前分区中所有未完成的事务完成提交 / 回滚，避免读取到未提交的脏数据，因此会带来轻微的延迟提升，这是一致性的必要代价。

### 8.1.5 核心配置与生产环境最佳实践

#### 1. 全链路核心配置

##### 生产者端配置（事务必填）

```properties
# 全局唯一的事务ID，必填
transactional.id=order_service_transaction_001
# 开启幂等性，事务强制依赖，开启事务后该参数默认自动设置为true
enable.idempotence=true
# acks必须设置为all，事务强制要求，否则会抛出配置异常
acks=all
# 事务超时时间，默认60000ms（1分钟）
transaction.timeout.ms=60000
```

##### Broker 端配置

```properties
# 事务状态Topic的副本因子，生产环境建议≥3
transaction.state.log.replication.factor=3
# 事务状态Topic的最小ISR副本数，生产环境建议≥2
transaction.state.log.min.isr=2
# 事务最大超时时间，默认900000ms（15分钟），生产者配置的超时时间不可超过该值
transaction.max.timeout.ms=900000
```

##### 消费者端配置（一致性要求场景必填）

```properties
# 开启读已提交隔离级别，保证事务一致性
isolation.level=read_committed
```

#### 2. 生产环境最佳实践

1. **严格遵循最小事务范围原则**：事务内仅包含必须原子性执行的操作，避免在事务内执行耗时操作、远程调用，减少事务的执行时长，降低事务阻塞对消费者的影响；
2. **TransactionalId 全局唯一且固定**：同一个业务流程的`TransactionalId`必须固定，不可随机生成，否则无法实现跨会话的幂等性；同时必须全局唯一，避免多实例冲突；
3. **合理设置事务超时时间**：超时时间过短会导致事务频繁超时回滚，过长会导致消费者阻塞、僵尸事务占用资源，建议设置为业务最大执行时长的 2 倍；
4. **异常场景必须显式回滚**：业务处理抛出异常时，必须显式调用`abortTransaction()`回滚事务，避免事务超时自动回滚导致的资源占用与延迟；
5. **避免长事务与大事务**：禁止在一个事务内发送数十万条以上的大批次消息，或执行超过 1 分钟的长事务，否则会导致分区日志无法清理、消费者阻塞、Broker 性能下降；
6. **核心业务必须配合消费端读已提交**：事务的一致性必须由生产端事务与消费端`read_committed`隔离级别共同保障，仅开启生产端事务无法避免脏数据读取。

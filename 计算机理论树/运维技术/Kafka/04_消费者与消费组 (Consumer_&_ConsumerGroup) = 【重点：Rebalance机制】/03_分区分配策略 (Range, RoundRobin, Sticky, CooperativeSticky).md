
分区分配策略是消费组 Rebalance 过程中的核心规则，定义了消费组订阅的 Topic 分区如何分配给组内的各个消费者实例，直接决定了消费组的负载均衡效果、Rebalance 的性能影响与消费稳定性。

分区分配策略为消费组级别配置，对应参数`partition.assignment.strategy`，消费组内所有消费者实例的该配置必须完全一致，否则会导致分配策略协商失败，触发异常 Rebalance。Kafka 提供 4 种内置分配策略，覆盖不同场景需求，随着版本迭代不断优化。

### 4.3.1 前置基础：分配策略的触发时机

分区分配仅在消费组 Rebalance 时执行，触发场景包括：

1. 消费组成员变更（新增 / 下线消费者）；
2. 消费组订阅的 Topic 变更；
3. 订阅 Topic 的分区数量变更。

### 4.3.2 内置分区分配策略详解

#### 1. RangeAssignor（范围分配策略）

RangeAssignor 是 Kafka 2.3 版本之前的默认分配策略，基于单 Topic 维度的分区范围进行分配，实现逻辑极简。

- **核心分配原理**：
    
    1. 对单个 Topic 的所有分区按序号升序排序，对消费组内的消费者按名称字典序排序；
    2. 计算每个消费者的基础分区数：`分区总数 / 消费者总数`，余数为无法均分的剩余分区数；
    3. 前`余数`个消费者，每个多分配 1 个分区，保证所有分区全部分配完毕；
    4. 为每个消费者分配连续的分区范围，完成单 Topic 的分配；
    5. 对消费者订阅的每个 Topic，独立执行上述分配逻辑。
    
- **示例**：TopicA 有 7 个分区（0-6），消费组有 3 个消费者（C1、C2、C3）。7/3=2，余 1，最终分配结果：C1→[0,1,2]，C2→[3,4]，C3→[5,6]。
- **核心优势**：实现简单，单 Topic 维度分配逻辑清晰，无跨 Topic 依赖。
- **核心缺陷**：当消费者订阅多个 Topic 时，极易出现严重的负载不均。例如消费者订阅 2 个 7 分区的 Topic，C1 会分配到 3+3=6 个分区，C2、C3 仅分配到 2+2=4 个分区，订阅的 Topic 越多，负载不均越严重。
- **适用场景**：消费者仅订阅单个 Topic 的简单场景。

#### 2. RoundRobinAssignor（轮询分配策略）

RoundRobinAssignor 基于全局轮询机制实现分配，解决了 RangeAssignor 的多 Topic 负载不均问题。

- **核心分配原理**：
    
    1. 将消费组订阅的**所有 Topic 的全部分区**按序号全局升序排序；
    2. 对消费组内的消费者按名称字典序排序，形成轮询队列；
    3. 按排序后的分区顺序，依次轮询分配给队列中的消费者，一个分区对应一个消费者，循环往复，直到全部分区分配完毕。
    
- **示例**：订阅 TopicA（7 分区）、TopicB（7 分区），共 14 个分区，3 个消费者。全局排序后轮询分配，最终 C1→5 个分区，C2→5 个分区，C3→4 个分区，负载完全均匀。
- **核心优势**：分区分配全局均匀，彻底解决了 RangeAssignor 的多 Topic 负载不均问题，资源利用率最大化。
- **核心缺陷**：
    
    1. Rebalance 时分区变动极大，哪怕仅新增 / 下线一个消费者，几乎所有分区都会被重新分配，导致大量不必要的分区撤销与重建，Rebalance 耗时久、消费中断时间长；
    2. 当消费组内消费者的订阅 Topic 不一致时，轮询分配会失效，再次出现负载不均问题。
    
- **适用场景**：消费组内所有消费者的订阅 Topic 完全一致的场景。

#### 3. StickyAssignor（粘性分配策略）

StickyAssignor 是 Kafka 2.3~3.0 版本的默认分配策略，核心设计目标是**保证分区分配均匀的前提下，最大化减少 Rebalance 时的分区变动**，即 “粘性” 特性。

- **核心分配原理**：
    
    1. 首次分配时，遵循全局均匀分配的原则，保证每个消费者的分区数量差值不超过 1，分配效果与 RoundRobinAssignor 一致；
    2. 后续触发 Rebalance 时，核心规则为：尽可能保留消费者已分配的分区，仅对需要变动的分区进行调整，不做全量重新分配；
    3. 消费者下线：仅将该消费者的分区重新分配给其他健康消费者，其余消费者的分区完全保留；
    4. 消费者上线：仅从现有消费者的分区中，匀出部分分区给新消费者，尽可能减少原有分区的变动，同时保证最终分配均匀。
    
- **核心优势**：
    
    1. 大幅减少 Rebalance 时的分区变动，降低 Rebalance 耗时与消费中断时间，减少不必要的分区元数据重建开销；
    2. 兼容消费者订阅 Topic 不一致的场景，仍能保证分配均匀性，适配复杂订阅场景；
    3. 继承了 RoundRobinAssignor 的均匀分配优势，资源利用率高。
    
- **核心缺陷**：分配算法实现复杂，单次分配的计算开销高于前两种策略，但 Rebalance 的整体耗时远低于前两者，综合收益显著。
- **适用场景**：绝大多数生产环境场景，尤其是订阅多个 Topic、对消费连续性有要求的业务。

#### 4. CooperativeStickyAssignor（协作式粘性分配策略）

CooperativeStickyAssignor 是 Kafka 3.0 + 版本的默认分配策略，是 StickyAssignor 的全面升级版，**彻底解决了传统 Rebalance 的 Stop The World（STW）全组停服问题**，是目前生产环境的最优策略。

- **核心设计突破**：
    
    前三种策略均为**Eager（急切式）Rebalance 模式**：Rebalance 触发时，所有消费者必须先放弃全部已分配的分区，停止消费，等待重新分配，整个过程消费组完全停服，即 STW。
    
    CooperativeStickyAssignor 采用**Cooperative（协作式）Rebalance 模式**：Rebalance 触发时，消费者无需放弃所有分区，仅需释放需要调整的分区，其余分区可继续正常消费，不会出现全组 STW，消费几乎无感知。
- **核心分配原理**：
    
    1. 采用分阶段 Rebalance 协议，第一轮同步消费组成员与订阅信息，Group Leader 计算出需要调整的分区列表；
    2. 第二轮仅对需要调整的分区进行重新分配，无需全量重新分配，未涉及的分区全程保持消费状态；
    3. 继承了 StickyAssignor 的粘性特性，最大化保留原有分区分配，分区变动最小化；
    4. 支持增量 Rebalance，无需一次完成全部分区调整，可通过多轮小范围调整完成最终分配，进一步降低对业务的影响。
    
- **核心优势**：
    
    1. 彻底消除 Rebalance 的全组 STW 问题，Rebalance 期间消费不会中断，业务无感知；
    2. 分区变动最小化，Rebalance 耗时大幅缩短，消费稳定性显著提升；
    3. 分配均匀性优秀，兼容复杂订阅场景，适配所有生产环境。
    
- **核心约束**：不能与其他 Eager 模式的分配策略混用，消费组内所有消费者必须统一配置该策略；版本需升级至 Kafka 2.4+，3.0 + 版本体验最优。
- **适用场景**：所有生产环境核心业务场景，尤其是对消费连续性、延迟稳定性要求极高的金融、实时计算场景。

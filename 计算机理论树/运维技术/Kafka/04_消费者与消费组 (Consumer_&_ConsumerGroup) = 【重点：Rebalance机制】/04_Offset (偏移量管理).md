
Offset（偏移量）是 Kafka 消费者记录消费进度的唯一标识，是实现断点续传、消息回溯、消费状态持久化的核心基础。Kafka 的消费进度管理完全围绕 Offset 展开，其存储机制与提交策略直接决定了消费端的消息丢失与重复消费风险。

### 4.4.1 Offset 核心定义

Offset 是分区内消息的唯一标识，为 64 位单调递增的整数，仅在单分区内有效，跨分区不具备可比性。

- **消费进度的本质**：消费者的消费进度，通过 <消费组，Topic, 分区> 三维度的 Offset 唯一确定，代表该消费组在该分区下，**下一次需要拉取的消息的起始 Offset**。例如：消费组 test-group 在 TopicA 的分区 0，提交的 Offset 为 100，代表该消费组已完成 0-99 号消息的消费，下次拉取将从 100 号消息开始。
- **核心约束**：Offset 的提交必须遵循「先处理，后提交」的铁律 —— 只有当消息的业务逻辑处理完成后，才能提交对应的 Offset，这是避免消息丢失的根本原则。

### 4.4.2 __consumer_offsets 内部 Topic 存储机制

Kafka 0.9 版本之前，消费组的 Offset 存储在 ZooKeeper 中，但 ZK 不适合高频写入场景，在消费组数量多、Offset 提交频繁的场景下会出现严重性能瓶颈。0.9 版本之后，Kafka 将 Offset 的存储完全迁移至集群内置的 Topic——`__consumer_offsets`，实现了 Offset 的高吞吐、高可用持久化存储。

#### 1. 核心特性

- **内置系统 Topic**：集群启动时自动创建，用户不可手动删除或修改，副本因子默认 3，分区数量由`offsets.topic.num.partitions`参数控制，默认 50 个分区；
- **存储结构**：采用 Key-Value 结构存储，Key 为`[消费组ID, Topic名称, 分区编号]`三元组，Value 为提交的 Offset 值、元数据、时间戳、提交状态、消费组元数据等信息；
- **分区路由规则**：消费组的所有 Offset 数据，会被路由到`__consumer_offsets`的固定分区，路由公式为：`partition = Math.abs(groupId.hashCode()) % 分区总数`。同一个消费组的所有元数据与 Offset，均存储在同一个分区中，保证了数据的一致性；
- **日志清理策略**：该 Topic 采用`compact`（日志压缩）清理策略，仅保留同一个 Key 的最新版本数据，删除旧版本的历史提交记录，避免日志无限膨胀，同时保证消费进度的有效性；
- **扩展能力**：除消费 Offset 外，该 Topic 还存储了消费组的成员信息、订阅信息、状态数据，是 Kafka 消费组管理的核心数据载体。

#### 2. 与 GroupCoordinator 的绑定关系

每个消费组会绑定唯一的 GroupCoordinator（消费组协调者，运行在 Broker 上），绑定规则为：消费组对应的`__consumer_offsets`分区的 Leader 副本所在的 Broker，即为该消费组的 GroupCoordinator。所有 Offset 提交、消费组心跳、Rebalance 请求，都必须发送给该 Coordinator，保证了消费组管理的中心化与一致性。

### 4.4.3 Offset 提交策略

Offset 提交策略定义了消费者何时、以何种方式将消费进度提交至`__consumer_offsets`，Kafka 提供 3 种核心提交策略，分别适配不同的可靠性与性能需求。

#### 1. 自动提交（Auto Commit）

自动提交是 Kafka 的默认提交方式，无需业务代码手动处理提交逻辑，完全由消费者客户端自动完成。

- **核心配置参数**：
    
    - `enable.auto.commit=true`：开启自动提交，默认值为 true；
    - `auto.commit.interval.ms=5000`：自动提交的时间间隔，默认 5 秒。
    
- **实现原理**：消费者启动后，会在后台启动一个定时提交线程，每隔`auto.commit.interval.ms`的时间，自动将内存中记录的当前最大 Offset，批量提交至 GroupCoordinator。
- **核心优势**：实现极简，业务代码无侵入，开发成本极低。
- **核心风险**：存在严重的消息丢失与重复消费风险，可靠性极低。
    
    1. **重复消费风险**：消费者消费了一批消息，尚未到自动提交时间，消费者宕机或触发 Rebalance，重启后会从上一次成功提交的 Offset 开始重新消费，导致已处理的消息重复消费；
    2. **消息丢失风险**：消费者拉取到一批消息后，自动提交线程先提交了 Offset，但业务逻辑尚未处理完成，此时消费者宕机，重启后会从新的 Offset 开始拉取，未处理完成的消息永久丢失。
    
- **适用场景**：非核心、可容忍少量重复消费的日志采集、监控数据上报等场景，**核心业务场景严禁使用**。

#### 2. 手动同步提交（CommitSync）

手动同步提交是生产环境最常用的高可靠提交方式，需关闭自动提交（`enable.auto.commit=false`），由业务代码在消息处理完成后，主动调用提交方法完成 Offset 提交。

- **核心实现原理**：业务代码调用`consumer.commitSync()`方法后，消费者会阻塞当前线程，向 GroupCoordinator 发送 Offset 提交请求，直到提交成功或提交失败抛出异常，提交完成后才会继续拉取新的消息。
- **进阶用法**：支持分区粒度的精准提交，可指定每个分区的 Offset 进行提交，而非全量提交所有分区的最大 Offset，适配分批处理、分分区处理的场景。示例：

    ```java
    Map<TopicPartition, OffsetAndMetadata> offsetMap = new HashMap<>();
    offsetMap.put(new TopicPartition("TopicA", 0), new OffsetAndMetadata(100));
    consumer.commitSync(offsetMap);
    ```
    
- **核心优势**：可靠性极高，完全可控，严格遵循「先处理，后提交」的铁律，从根本上避免消息丢失；提交失败会自动重试（不可重试异常除外），保证 Offset 提交的最终成功。
- **核心劣势**：同步阻塞，提交期间消费线程会暂停，消费吞吐会受提交频率影响，尤其是高频提交场景下，性能损耗较为明显。
- **适用场景**：所有核心业务场景，对消息丢失零容忍的金融、订单、交易场景，是生产环境的首选提交方式。

#### 3. 手动异步提交（CommitAsync）

手动异步提交是为高吞吐场景设计的非阻塞提交方式，同样需关闭自动提交，业务代码调用异步提交方法，无需阻塞等待提交结果。

- **核心实现原理**：业务代码调用`consumer.commitAsync()`方法后，立即返回，不会阻塞消费线程，继续拉取处理新消息；Offset 提交的结果通过回调函数异步通知，提交成功或失败都会触发回调，业务代码可在回调中处理异常与重试逻辑。
- **核心优势**：完全无阻塞，不会影响消费线程的运行，消费吞吐性能远高于同步提交，适配高并发、大流量的消费场景。
- **核心缺陷**：
    
    1. 可靠性低于同步提交，提交失败不会自动重试。因为异步提交是无序的，若先提交 Offset 100，未收到响应时又提交了 Offset 200，此时 100 的提交失败，重试会导致 Offset 回滚，引发大量重复消费；
    2. 消费者关闭时，若存在未完成的异步提交，可能会导致 Offset 提交失败，引发重复消费。
    
- **最佳实践**：同步 + 异步组合使用。正常消费过程中使用异步提交，最大化吞吐性能；在消费者关闭、Rebalance 触发前，使用同步提交，保证最后一次 Offset 提交成功，避免消费进度丢失。
- **适用场景**：高吞吐、大流量的日志处理、实时数仓、大数据同步场景，对延迟敏感、可容忍少量重复消费的业务场景。

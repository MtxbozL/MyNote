## 2.2 基础查询语法

基础查询是 SQL 的核心，所有复杂查询均由基础查询子句组合而成。本节严格遵循 SQL 的 **逻辑执行顺序** 展开，区别于语句的书写顺序，是理解 SQL 执行逻辑的核心前提。

### 2.2.1 SQL 语句书写顺序与逻辑执行顺序

SQL 语句的书写顺序与数据库内核的执行顺序存在本质差异，错误理解执行顺序会导致语法错误与逻辑偏差，二者的对应关系如下：

|书写顺序（从上到下）|逻辑执行顺序（从上到下）|核心作用|
|---|---|---|
|SELECT|1. FROM|确定查询的数据源表，包括多表关联的结果集|
|FROM|2. WHERE|对数据源行进行前置过滤，剔除不符合条件的行|
|WHERE|3. GROUP BY|将过滤后的行按指定列进行分组|
|GROUP BY|4. 聚合函数计算|对每个分组执行聚合计算，生成分组结果|
|HAVING|5. HAVING|对分组后的聚合结果进行二次过滤|
|ORDER BY|6. SELECT|提取需要返回的列，执行列表达式计算、别名赋值|
|LIMIT|7. DISTINCT|对结果集进行去重处理|
||8. ORDER BY|对最终结果集进行排序|
||9. LIMIT|限定返回的行数，实现分页|

**核心规则**：只有在`SELECT`子句中定义的别名，才能在`ORDER BY`子句中引用；`WHERE`、`GROUP BY`、`HAVING`子句无法引用`SELECT`中定义的别名，因为其执行顺序早于`SELECT`。

### 2.2.2 基础子句详解

#### 2.2.2.1 SELECT 列选择子句

用于指定查询需要返回的列或表达式，是查询结果的最终输出载体。

- 基础语法：`SELECT [ALL | DISTINCT] 列名1 [AS 别名1], 列名2 [AS 别名2], 表达式 | 常量`
- 核心特性：
    
    1. `*`：通配符，返回表中所有列，生产环境不推荐使用，会增加数据传输开销、暴露敏感字段，且无法适配表结构变更；
    2. `AS`：为列或表达式定义别名，提升可读性，别名仅在当前查询语句内有效；
    3. `DISTINCT`：对结果集进行去重，剔除所有列值完全相同的重复行，可作用于单列或多列；
    4. 支持表达式计算、函数调用，如`SELECT id, price * num AS total_amount FROM order;`。
    

#### 2.2.2.2 WHERE 条件过滤子句

用于对数据源行进行前置过滤，仅保留符合条件的行，是减少后续计算数据量的核心环节。

- 基础语法：`WHERE 条件表达式`
- 条件表达式核心分类：
    
    1. 比较运算符：`=`、`>`、`<`、`>=`、`<=`、`!=`/`<>`，用于数值、字符串、日期的等值与范围比较；
    2. 逻辑运算符：`AND`、`OR`、`NOT`，用于组合多个条件，`AND`优先级高于`OR`，可通过括号调整优先级；
    3. 范围运算符：`BETWEEN 最小值 AND 最大值`，闭区间范围匹配，等价于`>= 最小值 AND <= 最大值`；
    4. 集合运算符：`IN (值1, 值2, ...)`，匹配表达式是否在指定值集合中，等价于多个`OR`条件的组合；
    5. 模糊匹配运算符：`LIKE`，配合通配符`%`（匹配任意长度的任意字符）、`_`（匹配单个任意字符）实现字符串模糊查询；
    6. 空值判断：`IS NULL`、`IS NOT NULL`，用于判断列值是否为 NULL，**禁止使用`= NULL`或`!= NULL`进行空值判断**。
    

**核心理论：三值逻辑**

SQL 的条件运算结果包含三种状态：`TRUE`、`FALSE`、`UNKNOWN`。NULL 表示未知值，任何与 NULL 的直接运算（如`1 + NULL`、`a = NULL`）结果均为`UNKNOWN`，`WHERE`子句仅保留结果为`TRUE`的行，会剔除`FALSE`与`UNKNOWN`的行。

#### 2.2.2.3 GROUP BY 分组聚合子句

用于将过滤后的行按指定的一个或多个列进行分组，相同列值的行归为同一组，配合聚合函数实现分组内的统计计算。

- 基础语法：`GROUP BY 列名1, 列名2, ...`
- 核心规则：
    
    1. SQL 标准规定：`SELECT`子句中，**非聚合函数的列，必须全部出现在`GROUP BY`子句中**。MySQL 8.0 默认开启`ONLY_FULL_GROUP_BY`模式，强制遵守该规则，避免非确定性结果；
    2. 支持多列分组，分组顺序为列的书写顺序，先按第一列分组，再在每个分组内按后续列细分；
    3. 分组列中包含 NULL 值时，所有 NULL 值会被归为同一组。
    

**核心聚合函数**

聚合函数对分组内的行进行计算，每个分组返回单个计算结果，核心特性为：除`COUNT(*)`外，所有聚合函数均会忽略 NULL 值。

|聚合函数|核心作用|注意事项|
|---|---|---|
|COUNT(*)|统计分组内的总行数，包括 NULL 值行|性能最优，不会过滤任何行|
|COUNT (列名)|统计分组内指定列非 NULL 的行数|列存在 NULL 值时，统计结果小于总行数|
|SUM (列名)|计算分组内指定列的数值总和|仅适用于数值类型列，全为 NULL 时返回 NULL|
|AVG (列名)|计算分组内指定列的数值平均值|自动忽略 NULL 值，仅计算非 NULL 行的平均值|
|MAX (列名)|计算分组内指定列的最大值|适用于数值、日期、字符串类型，忽略 NULL 值|
|MIN (列名)|计算分组内指定列的最小值|适用于数值、日期、字符串类型，忽略 NULL 值|

#### 2.2.2.4 HAVING 分组过滤子句

用于对`GROUP BY`分组后的聚合结果进行过滤，仅保留符合条件的分组。

- 基础语法：`HAVING 条件表达式`
- **核心区别：WHERE vs HAVING**
    
| 特性   | WHERE          | HAVING            |
| ---- | -------------- | ----------------- |
| 执行时机 | 分组之前，对原始行过滤    | 分组之后，对聚合结果过滤      |
| 聚合函数 | 禁止使用           | 支持使用，是核心使用场景      |
| 作用对象 | 表中的行           | 分组后的聚合结果集         |
| 索引优化 | 可利用索引过滤数据，性能更优 | 无法直接利用索引，需先完成分组计算 |


**核心规则**：`HAVING`必须与`GROUP BY`配合使用；无`GROUP BY`时，`HAVING`会将整个结果集视为一个分组，仅适用于聚合函数过滤场景。

#### 2.2.2.5 ORDER BY 排序子句

用于对最终的查询结果集按指定列进行升序或降序排序，是结果集有序输出的唯一方式。

- 基础语法：`ORDER BY 列名1 [ASC | DESC], 列名2 [ASC | DESC], ...`
- 核心特性：
    
    1. `ASC`：升序排序，为默认值，从小到大排列；`DESC`：降序排序，从大到小排列；
    2. 支持多列排序，先按第一列排序，第一列值相同时，再按第二列排序，以此类推；
    3. 支持引用`SELECT`子句中定义的列别名，是唯一可引用 SELECT 别名的前置子句；
    4. 无`ORDER BY`子句时，数据库返回的结果集顺序不保证固定，不可依赖数据库的默认存储顺序。
    

#### 2.2.2.6 LIMIT 分页子句

用于限定查询结果集返回的行数，是 MySQL 实现分页查询的核心语法。

- 基础语法：`LIMIT [offset,] row_count`
- 核心参数：
    
    1. `offset`：偏移量，可选参数，表示跳过的行数，从 0 开始计数，默认为 0；
    2. `row_count`：返回的最大行数，必填参数。
    
- 典型分页场景：每页显示 10 条数据，第 N 页的分页语法为`LIMIT (N-1)*10, 10`。
- 核心注意事项：当偏移量`offset`过大时（如`LIMIT 1000000, 10`），会产生严重的性能损耗，具体优化方案在后续性能调优章节详细讲解。

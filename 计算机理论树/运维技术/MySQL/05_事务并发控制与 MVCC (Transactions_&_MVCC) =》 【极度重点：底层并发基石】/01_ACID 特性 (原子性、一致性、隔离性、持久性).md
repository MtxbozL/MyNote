
>事务是关系型数据库区别于文件系统的核心特性，是联机事务处理（OLTP）场景下保障数据一致性、并发安全性的核心载体。InnoDB 作为 MySQL 默认的事务型存储引擎，通过**事务 ACID 特性、隔离级别、锁机制、MVCC 多版本并发控制**四大核心体系，实现了高并发场景下的数据安全与性能平衡。其中 MVCC 是 InnoDB 实现无锁并发读、解决读写冲突的核心底层机制，也是 MySQL 面试的必考核心难点。本章从事务基础理论出发，深入拆解并发事务问题、隔离级别标准，最终深度剖析 MVCC 的底层实现原理与执行逻辑。

## 5.1 ACID 特性

ACID 是事务必须满足的四大核心特性，是事务安全的理论基石，四者并非并列关系：**原子性、隔离性、持久性是实现手段，一致性是最终追求的核心目标**。

### 5.1.1 原子性（Atomicity）

原子性的核心定义是：**一个事务中的所有 DML 操作，要么全部执行成功，要么全部执行失败回滚，不存在中间状态**。事务是一个不可分割的最小执行单元，就像化学中的原子一样，无法被拆分。

- 底层实现支撑：原子性的核心保障是**Undo Log（回滚日志）**。事务执行过程中，所有修改操作都会先写入 Undo Log，记录数据修改前的镜像；当事务执行失败、执行`ROLLBACK`时，InnoDB 会根据 Undo Log 中的记录，将数据回滚到事务开始前的状态，确保操作的原子性。
- 核心约束：原子性仅保障事务操作的 "全有或全无"，不保障事务的并发安全与数据的业务一致性，需配合隔离性共同实现。

### 5.1.2 一致性（Consistency）

一致性的核心定义是：**事务执行前后，数据库的完整性约束（包括实体完整性、参照完整性、用户定义的业务约束）始终保持一致，数据从一个合法的状态转换为另一个合法的状态**。

- 完整性约束的核心范畴：
    
    1. 实体完整性：主键非空且唯一，无重复元组；
    2. 参照完整性：外键取值必须匹配被参照表的主键值，无悬空引用；
    3. 用户定义完整性：业务层面的约束，如账户余额不能为负、库存不能小于 0、字段非空约束等。
    
- 实现逻辑：一致性是事务的最终目标，由原子性、隔离性、持久性共同保障，同时需要业务代码层面保证事务的业务逻辑合法性。数据库层面仅能保障数据约束的一致性，业务层面的一致性需由业务逻辑与事务隔离性共同实现。

### 5.1.3 隔离性（Isolation）

隔离性的核心定义是：**多个并发执行的事务之间相互隔离，每个事务的执行都感觉不到其他事务的存在，并发事务之间不会互相干扰**。隔离性解决的核心问题是并发事务执行过程中的数据竞争问题，是本章的核心讲解内容。

- 底层实现支撑：隔离性的两大核心实现机制是**锁机制**与**MVCC 多版本并发控制**。锁机制通过悲观并发控制，阻止并发事务同时修改同一行数据；MVCC 通过乐观并发控制，实现无锁的一致性读，解决读写冲突，大幅提升并发性能。
- 核心标准：ANSI/ISO SQL 标准定义了四大隔离级别，不同隔离级别对应不同的隔离强度与并发性能，隔离强度越高，并发性能越低，业务需根据场景选择适配的隔离级别。

### 5.1.4 持久性（Durability）

持久性的核心定义是：**一个事务一旦提交成功，其对数据库的修改就会被永久保存，后续的任何操作、系统崩溃、断电等异常情况，都不会导致已提交的数据丢失**。

- 底层实现支撑：持久性的核心保障是 **Redo Log（重做日志）** 与**WAL（预写日志）机制** 。事务提交时，不会立即将 Buffer Pool 中的脏页刷入磁盘，而是先将修改操作写入 Redo Log 并持久化到磁盘；当数据库发生崩溃重启时，InnoDB 会通过 Redo Log 重放已提交但未刷入磁盘的修改，保证数据不丢失，实现崩溃安全（Crash-Safe）。
- 核心约束：持久性仅保障已提交事务的永久生效，未提交的事务不满足持久性要求，崩溃后会通过 Undo Log 回滚。

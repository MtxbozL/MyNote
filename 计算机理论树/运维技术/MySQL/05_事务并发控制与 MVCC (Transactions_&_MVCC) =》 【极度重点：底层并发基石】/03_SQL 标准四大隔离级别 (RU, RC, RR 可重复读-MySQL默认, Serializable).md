
>ANSI/ISO SQL 92 标准定义了四大事务隔离级别，每个级别对应不同的隔离强度，分别解决不同的并发事务问题，隔离强度从低到高依次为：读未提交、读已提交、可重复读、串行化。

### 5.3.1 四大隔离级别核心定义与能力对比

|隔离级别|英文全称|核心定义|脏读|不可重复读|幻读|锁机制与并发特性|
|---|---|---|---|---|---|---|
|读未提交|Read Uncommitted (RU)|一个事务可以读取到其他事务未提交的修改|存在|存在|存在|几乎不加锁，并发性能最高，无任何隔离保障，生产环境禁止使用|
|读已提交|Read Committed (RC)|一个事务只能读取到其他事务已经提交的修改，未提交的修改完全不可见|解决|存在|存在|仅对修改的行加行锁，读操作不加锁（通过 MVCC 实现），并发性能优异，是 Oracle、PostgreSQL 等数据库的默认隔离级别|
|可重复读|Repeatable Read (RR)|保证同一个事务内，多次执行相同的查询，结果始终一致，不受其他并发事务提交的修改影响|解决|解决|SQL 标准允许存在，InnoDB 通过临键锁解决|行锁 + 间隙锁 + MVCC 共同实现，是 MySQL InnoDB 的默认隔离级别，平衡了隔离性与并发性能|
|串行化|Serializable|最高隔离级别，将所有事务串行化执行，完全避免并发事务的互相干扰|解决|解决|解决|所有读操作均加共享锁，写操作加排他锁，读写互相阻塞，完全消除并发冲突，并发性能极差，仅适用于对数据一致性要求极高、并发量极低的金融场景|

### 5.3.2 MySQL 隔离级别核心特性

1. **默认隔离级别**：InnoDB 默认使用**可重复读（RR）** 隔离级别，区别于其他主流数据库的 RC 默认级别。
2. **RR 级别的幻读解决**：SQL 标准的 RR 级别允许幻读存在，但 InnoDB 的 RR 级别通过**Next-Key Lock（临键锁，记录锁 + 间隙锁）** 机制，在当前读场景下彻底解决了幻读问题，实现了比 SQL 标准更严格的隔离性。
3. **隔离级别设置**：
    
    - 全局级别设置：`SET GLOBAL transaction_isolation = 'REPEATABLE-READ';`
    - 会话级别设置：`SET SESSION transaction_isolation = 'READ-COMMITTED';`
    - 查看当前隔离级别：`SELECT @@transaction_isolation;`


>MVCC（Multi-Version Concurrency Control，多版本并发控制），是 InnoDB 在**RC、RR**隔离级别下，实现无锁一致性读、解决读写冲突的核心底层机制。其核心设计思想是：通过维护数据行的多个历史版本，让读操作无需加锁，写操作仅锁定需要修改的行，读写之间互不阻塞，大幅提升数据库的并发读写性能。

### 5.4.1 MVCC 核心机制与适用范围

1. **核心实现逻辑**：InnoDB 为每行数据维护多个历史版本，每个版本对应一个事务 ID；事务读取数据时，根据事务的启动时机与隔离级别，生成对应的读视图（Read View），通过可见性算法判断哪个版本的数据对当前事务可见，无需加锁即可读取到一致性的历史版本数据。
2. **快照读与当前读**：
    
    - **快照读（一致性非锁定读）**：普通的`SELECT`语句，读取的是数据的历史快照版本，通过 MVCC 实现无锁读取，不会阻塞其他事务的写操作，是 MVCC 的核心应用场景。
    - **当前读（锁定读）**：读取数据的最新版本，并对读取的行加锁，包括`SELECT ... FOR UPDATE`、`SELECT ... LOCK IN SHARE MODE`、`INSERT`、`UPDATE`、`DELETE`，当前读不通过 MVCC 实现，依赖锁机制保证并发安全。
    
3. **适用隔离级别**：MVCC 仅在**读已提交（RC）** 和**可重复读（RR）** 两个隔离级别下生效。
    
    - RU 级别：总是读取数据的最新版本，无需历史版本，不使用 MVCC；
    - Serializable 级别：所有读操作均加锁，无需 MVCC，读写互相阻塞。
    

### 5.4.2 InnoDB 行记录隐藏字段

InnoDB 的聚簇索引叶子节点中，每行数据除了用户定义的业务字段外，还会自动添加三个隐藏字段，是 MVCC 实现的核心基础。

|隐藏字段|长度|核心作用|
|---|---|---|
|DB_TRX_ID|6 字节|事务 ID，记录最后一次插入或修改该行数据的事务 ID。InnoDB 会为每个开启的事务分配一个全局唯一、严格递增的事务 ID，是版本判断的核心依据。|
|DB_ROLL_PTR|7 字节|回滚指针，指向该行数据对应的 Undo Log 记录。Undo Log 中存储了该行数据修改前的历史版本，通过回滚指针可将所有历史版本串联成一个版本链。|
|DB_ROW_ID|6 字节|隐藏主键，当表没有定义主键、也没有非空唯一索引时，InnoDB 会自动生成该字段作为聚簇索引的主键；若表已定义主键，该字段不会生成。|

### 5.4.3 Undo Log 版本链

Undo Log 是 MVCC 版本链的物理载体，其核心作用有两个：一是保障事务的原子性，支持事务回滚；二是维护数据的历史版本，支撑 MVCC 的一致性读。

#### 5.4.3.1 Undo Log 分类

1. **Insert Undo Log**：事务执行`INSERT`操作时生成的 Undo Log，仅用于事务回滚。由于`INSERT`操作插入的新数据，仅当前事务可见，其他事务不会访问该数据，因此事务提交后，该 Undo Log 可以被直接删除，无需保留历史版本。
2. **Update Undo Log**：事务执行`UPDATE`或`DELETE`操作时生成的 Undo Log，不仅用于事务回滚，还需要支撑 MVCC 的一致性读。事务提交后，该 Undo Log 不会立即删除，会被保留到 Purge 线程，直到没有任何事务需要引用该历史版本时，才会被清理回收。

#### 5.4.3.2 版本链的生成逻辑

每行数据的所有历史版本，通过`DB_ROLL_PTR`回滚指针串联成一个单向链表，称为版本链，链表的头节点是该行数据的最新版本，尾节点是最早的历史版本。

- 版本链生成示例：
    
    1. 事务 A（trx_id=100）执行`INSERT`插入一行数据，`DB_TRX_ID=100`，`DB_ROLL_PTR`指向 NULL，无历史版本；
    2. 事务 B（trx_id=200）执行`UPDATE`修改该行数据，生成 Undo Log 记录修改前的版本，将当前行的`DB_TRX_ID`更新为 200，`DB_ROLL_PTR`指向刚生成的 Undo Log，版本链形成第一个历史节点；
    3. 事务 C（trx_id=300）再次执行`UPDATE`修改该行数据，生成新的 Undo Log，更新`DB_TRX_ID=300`，`DB_ROLL_PTR`指向事务 B 生成的 Undo Log，版本链新增第二个历史节点；
    4. 最终形成的版本链：最新版本（trx_id=300）→ 历史版本 1（trx_id=200）→ 历史版本 2（trx_id=100）→ NULL。
    

### 5.4.4 Read View（读视图）可见性判断算法

Read View 是事务执行快照读时生成的**数据一致性视图**，记录了当前数据库中活跃的未提交事务的 ID 集合，是 MVCC 判断数据版本可见性的核心依据。可见性算法的核心逻辑是：通过 Read View 中的事务信息，判断版本链中哪个版本的数据对当前事务可见，找到第一个可见的版本后，直接返回该版本的数据，无需加锁。

#### 5.4.4.1 Read View 核心字段

|字段名|核心含义|
|---|---|
|creator_trx_id|创建该 Read View 的当前事务的事务 ID，唯一标识当前事务。|
|m_ids|生成 Read View 时，数据库中所有**活跃的未提交事务**的事务 ID 集合。|
|min_trx_id|生成 Read View 时，活跃事务集合 m_ids 中的最小事务 ID，即当前数据库中未提交事务的最小 ID。|
|max_trx_id|生成 Read View 时，数据库中**下一个要分配的事务 ID**，即已分配的最大事务 ID+1。所有 trx_id >= max_trx_id 的事务，都是在 Read View 生成后才开启的，对当前事务不可见。|

#### 5.4.4.2 核心可见性判断规则

对于版本链中的每个数据版本，按照以下优先级依次判断可见性，找到第一个符合条件的可见版本：

1. **规则 1**：如果版本的`trx_id == creator_trx_id`，说明该版本是当前事务自己修改的，对当前事务完全可见。
2. **规则 2**：如果版本的`trx_id < min_trx_id`，说明生成该版本的事务，在 Read View 生成前就已经提交，对当前事务可见。
3. **规则 3**：如果版本的`trx_id >= max_trx_id`，说明生成该版本的事务，是在 Read View 生成后才开启的，对当前事务完全不可见。
4. **规则 4**：如果版本的`trx_id`在`min_trx_id`和`max_trx_id`之间，判断`trx_id`是否在活跃事务集合`m_ids`中：
    
    - 若在`m_ids`中：说明生成该版本的事务，在 Read View 生成时仍处于活跃未提交状态，该版本对当前事务不可见；
    - 若不在`m_ids`中：说明生成该版本的事务，在 Read View 生成时已经提交，该版本对当前事务可见。
    

若当前版本不可见，则顺着版本链找到下一个历史版本，重复执行上述判断规则，直到找到第一个可见的版本；若遍历完整个版本链都没有可见的版本，则返回空结果集。

#### 5.4.4.3 RC 与 RR 隔离级别下 Read View 的核心差异

RC 和 RR 隔离级别的核心实现差异，本质是**Read View 的生成时机不同**，这也是 RC 允许不可重复读、RR 保证可重复读的底层核心原因。

1. **读已提交（RC）级别**：**事务中每次执行快照读（SELECT），都会生成一个全新的 Read View**。
    
    - 执行逻辑：每次 SELECT 都会获取当前最新的活跃事务集合，因此其他事务在两次 SELECT 之间提交的修改，会被纳入新的 Read View 中，导致当前事务两次 SELECT 读取到的结果不一致，即存在不可重复读。
    - 优势：每次读取都是最新的已提交数据，数据实时性高；
    - 劣势：无法保证事务内的可重复读。
    
2. **可重复读（RR）级别**：**事务中第一次执行快照读（SELECT）时，生成全局唯一的 Read View，整个事务生命周期内复用该 Read View**。
    
    - 执行逻辑：事务内所有 SELECT 都复用同一个 Read View，其他事务在事务执行过程中提交的修改，不会被纳入该 Read View 中，因此当前事务所有 SELECT 读取到的结果始终一致，彻底保证了可重复读。
    - 优势：严格保证事务内的可重复读，数据一致性更高；
    - 劣势：读取的数据不是最新的已提交版本，存在一定的延迟。

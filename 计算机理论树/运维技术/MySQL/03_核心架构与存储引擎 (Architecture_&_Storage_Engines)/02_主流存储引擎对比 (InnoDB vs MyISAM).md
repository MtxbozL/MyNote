
>InnoDB 与 MyISAM 是 MySQL 发展史上最具代表性的两款存储引擎，MyISAM 是 MySQL 5.5 版本之前的默认存储引擎，InnoDB 是 MySQL 5.5 及之后的默认存储引擎，二者的核心特性差异决定了其适用场景的边界。

### 3.2.1 核心特性全维度对比

|对比维度|InnoDB|MyISAM|
|---|---|---|
|事务支持|完全支持 ACID 特性，支持事务提交、回滚、保存点，是 OLTP 场景的核心能力|完全不支持事务，仅支持原子性的单语句操作，无法保证多操作的一致性|
|外键约束|原生支持外键约束，可强制保证数据库的参照完整性，支持级联更新与级联删除|完全不支持外键约束，参照完整性只能在业务代码层实现|
|锁机制|支持表级锁与行级锁，行级锁基于索引实现，仅锁定需要操作的行，并发读写性能优异|仅支持表级锁，读锁共享、写锁排他，写操作会阻塞全表的所有读写操作，并发性能极差|
|MVCC 多版本并发控制|原生支持 MVCC，实现无锁的一致性读，读写互不阻塞，大幅提升高并发场景下的性能|不支持 MVCC，读操作加共享锁，写操作加排他锁，读写互相阻塞|
|索引结构|采用聚簇索引设计，主键索引与数据行存储在一起，二级索引叶子节点存储主键值，支持自适应哈希索引|采用非聚簇索引设计，索引文件与数据文件完全分离，索引叶子节点存储数据行的物理地址|
|COUNT (*) 无过滤查询|因 MVCC 机制，不同事务可见的行范围不同，无法预存总行数，大表无过滤 COUNT (*) 需扫描索引，性能较低|将表的总行数存储在元数据中，无 WHERE 条件的 COUNT (*) 可直接返回，性能极快|
|崩溃恢复（Crash-Safe）|基于 Redo Log 与 Undo Log 实现完整的崩溃安全能力，异常断电重启后可自动恢复数据到一致性状态，无数据丢失风险|无崩溃安全机制，异常断电后极易出现数据文件损坏、索引损坏，需手动执行修复操作，修复成功率低，存在数据丢失风险|
|存储文件结构|每张表对应.ibd 文件（独立表空间模式），存储数据与索引；共享表空间 ibdata1 存储系统元数据、事务日志等|每张表对应三个文件：.frm（表结构元数据）、.MYD（数据文件）、.MYI（索引文件），数据与索引完全分离|

### 3.2.2 选型核心依据

1. **InnoDB 适用场景**：所有需要事务支持、高并发读写、数据一致性要求高的业务场景，包括互联网核心业务、金融交易、订单系统、用户系统等，是现代 MySQL 业务的默认选型。
2. **MyISAM 适用场景**：仅适用于只读、极少更新、对事务无要求、对查询性能要求极高的静态数据场景，如历史归档数据、静态字典表；现代业务中已基本被淘汰，无特殊需求不应使用。

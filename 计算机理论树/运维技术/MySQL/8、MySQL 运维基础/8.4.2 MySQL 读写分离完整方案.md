## 一、开篇：读写分离的核心定位与前置基础

### 1.1 什么是读写分离

读写分离是基于 MySQL 主从复制架构的**核心性能扩展方案**，核心逻辑是：

> **写操作（INSERT/UPDATE/DELETE）全部路由到主库（Master）执行，读操作（SELECT）全部路由到从库（Slave）执行**，通过多从库水平扩展读能力，分散主库的读压力，提升数据库整体吞吐量。

读写分离是高并发 MySQL 架构的标配，完全基于之前的主从复制笔记实现，主从复制是读写分离的基础，没有主从复制就无法实现读写分离。

### 1.2 核心价值

1. **水平扩展读能力**：单主库的读性能有瓶颈，通过增加从库可无限扩展读并发，支撑数万级的读 QPS；
2. **降低主库压力**：主库只负责写操作和核心事务，读压力全部分散到从库，主库 CPU/IO 负载大幅降低；
3. **业务隔离**：报表、数据分析、离线统计等重查询，可路由到专属从库执行，不影响线上业务主库；
4. **高可用兜底**：主库故障时，从库可快速提升为新主库，配合读写分离实现业务无感知切换。

### 1.3 前置基础要求

1. 已搭建稳定的 MySQL 主从复制架构（一主一从 / 一主多从），主从数据延迟可控；
2. 主库开启 `binlog_format=ROW`，保证主从数据一致性；
3. 从库开启 `read_only=1`/`super_read_only=1`，禁止非超级用户写入，避免主从数据不一致；
4. 业务代码已做规范：写操作和读操作可清晰拆分，无强一致读场景的写后立即读。

### 1.4 核心设计原则

1. **写主库唯一原则**：所有写操作必须只走主库，绝对禁止在从库执行写操作；
2. **读从库负载均衡**：多从库场景下，读请求均匀分发到多个从库，避免单从库压力过大；
3. **强一致读兜底**：写后立即读、金融交易等强一致性场景，必须强制路由到主库，避免主从延迟导致的数据不一致；
4. **故障自动剔除**：从库故障、延迟过高时，自动将其从读节点列表中剔除，避免业务报错。

## 二、读写分离方案选型对比

读写分离的实现分为三大类，分别是**客户端层方案**、**代理层方案**、**官方轻量方案**，各有优劣，生产环境需根据业务场景选择。

|方案类型|代表工具|核心原理|优点|缺点|适用场景|
|:--|:--|:--|:--|:--|:--|
|**客户端层（应用层）**|Sharding-JDBC、Spring 动态数据源|在应用的 JDBC 层拦截 SQL，根据 SQL 类型路由到对应的数据库|1. 无代理层，性能无损耗；<br><br>2. 无单点故障风险；<br><br>3. 轻量，无需额外部署服务；<br><br>4. 配置灵活，支持自定义规则|1. 与业务代码耦合，多语言不支持；<br><br>2. 多应用需重复配置；<br><br>3. 数据库拓扑变化需修改应用配置重启|Java 生态业务、中小规模集群、无专职 DBA 的团队|
|**代理层（中间件）**|ProxySQL、MyCat、MaxScale|部署独立的代理中间件，应用连接代理，代理根据 SQL 规则转发到对应数据库|1. 对业务完全透明，无代码侵入；<br><br>2. 支持多语言、多应用统一管理；<br><br>3. 功能丰富，支持 SQL 审计、限流、故障自动剔除；<br><br>4. 数据库拓扑变化无需修改应用|1. 代理层有轻微性能损耗；<br><br>2. 需保障代理层高可用，避免单点；<br><br>3. 有一定的运维成本|大规模集群、多语言 / 多应用架构、有专职 DBA 的团队、生产环境首选|
|**官方轻量方案**|MySQL Router|MySQL 官方推出的轻量路由工具，基于主从状态路由|1. 官方原生，兼容性最好；<br><br>2. 超轻量，部署简单；<br><br>3. 无额外学习成本|1. 功能单一，仅支持基础读写分离，无负载均衡、SQL 审计等高级功能；<br><br>2. 无故障自动剔除能力|测试环境、简单业务场景、主从架构简单的场景|

## 三、方案一：客户端层 - Sharding-JDBC 读写分离实战（Java 生态首选）

Sharding-JDBC 是 Apache 顶级项目，是 Java 生态中最主流的客户端层读写分离方案，完全嵌入 JDBC 层，无额外部署，性能极高。

### 3.1 核心原理

Sharding-JDBC 在应用启动时，加载读写分离配置，创建对应主从库的数据源；业务执行 SQL 时，Sharding-JDBC 拦截 SQL 语句：

- 写语句（INSERT/UPDATE/DELETE）、事务内的语句，强制路由到主库；
- 读语句（SELECT），根据负载均衡策略路由到从库；
- 支持自定义规则，如强制主库路由、从库延迟剔除等。

### 3.2 环境准备

1. 已搭建一主两从架构：
    
    - 主库：192.168.1.100:3306
    - 从库 1：192.168.1.101:3306
    - 从库 2：192.168.1.102:3306
    
2. Spring Boot 2.7+ 项目，JDK 8+
3. 数据库账号：业务账号 `app_user`，主从库通用，拥有读写权限。

### 3.3 完整实战步骤

#### 步骤 1：引入 Maven 依赖

```xml
<!-- Sharding-JDBC 读写分离核心依赖 -->
<dependency>
    <groupId>org.apache.shardingsphere</groupId>
    <artifactId>shardingsphere-jdbc-core-spring-boot-starter</artifactId>
    <version>5.4.1</version>
</dependency>
<!-- MySQL驱动 -->
<dependency>
    <groupId>com.mysql</groupId>
    <artifactId>mysql-connector-java</artifactId>
    <version>8.0.33</version>
</dependency>
<!-- 数据源连接池 -->
<dependency>
    <groupId>com.zaxxer</groupId>
    <artifactId>HikariCP</artifactId>
</dependency>
```

#### 步骤 2：application.yml 读写分离配置

```yaml
spring:
  shardingsphere:
    # 数据源配置
    datasource:
      names: master,slave1,slave2
      # 主库配置
      master:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://192.168.1.100:3306/my_database?useUnicode=true&characterEncoding=utf8mb4&serverTimezone=Asia/Shanghai
        username: app_user
        password: App@20260401
        maximum-pool-size: 20
      # 从库1配置
      slave1:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://192.168.1.101:3306/my_database?useUnicode=true&characterEncoding=utf8mb4&serverTimezone=Asia/Shanghai
        username: app_user
        password: App@20260401
        maximum-pool-size: 20
      # 从库2配置
      slave2:
        type: com.zaxxer.hikari.HikariDataSource
        driver-class-name: com.mysql.cj.jdbc.Driver
        jdbc-url: jdbc:mysql://192.168.1.102:3306/my_database?useUnicode=true&characterEncoding=utf8mb4&serverTimezone=Asia/Shanghai
        username: app_user
        password: App@20260401
        maximum-pool-size: 20
    # 读写分离规则配置
    rules:
      readwrite-splitting:
        data-sources:
          # 自定义读写分离数据源名称
          my-readwrite-db:
            # 写数据源（主库）
            write-data-source-name: master
            # 读数据源列表（从库）
            read-data-source-names:
              - slave1
              - slave2
            # 负载均衡策略：轮询（ROUND_ROBIN）/随机（RANDOM）/权重（WEIGHT）
            load-balancer-name: round_robin
        # 负载均衡配置
        load-balancers:
          round_robin:
            type: ROUND_ROBIN
    # 显示SQL日志，生产环境可关闭
    props:
      sql-show: true
```

#### 步骤 3：业务代码使用

无需修改任何业务代码，正常使用 MyBatis/MyBatis-Plus/JdbcTemplate 即可，Sharding-JDBC 会自动完成 SQL 路由：

```java
// 写操作：自动路由到主库
userMapper.insert(new User("张三", "13800138000"));

// 读操作：自动轮询路由到slave1/slave2
List<User> userList = userMapper.selectList(null);
```

### 3.4 进阶配置

#### 1. 强制主库读（解决写后立即读的主从延迟问题）

通过 `HintManager` 强制当前查询走主库：

```java
// 强制主库路由
try (HintManager hintManager = HintManager.getInstance()) {
    hintManager.setWriteRouteOnly();
    // 此查询强制走主库
    User user = userMapper.selectById(1);
}
```

#### 2. 从库延迟自动剔除

配置从库延迟阈值，超过阈值的从库自动从读列表中剔除：

```yaml
spring:
  shardingsphere:
    rules:
      readwrite-splitting:
        data-sources:
          my-readwrite-db:
            write-data-source-name: master
            read-data-source-names: [slave1, slave2]
            load-balancer-name: round_robin
            # 延迟剔除配置
            query-consistent: true
            delay-milliseconds: 1000 # 从库延迟超过1秒，自动剔除
```

### 3.5 优缺点总结

| 优点                           | 缺点                                |
| :--------------------------- | :-------------------------------- |
| 无代理层，性能损耗几乎为 0，是所有方案中性能最高的   | 仅支持 Java 生态，Python/Go/PHP 等语言无法使用 |
| 无单点故障，每个应用独立配置，不会因为中间件故障影响业务 | 多应用集群需要重复配置，数据库拓扑变化需要修改所有应用配置     |
| 轻量，无需额外部署服务，运维成本极低           | 与业务代码耦合，无法统一管理多应用的读写规则            |
| 配置灵活，支持自定义路由规则、负载均衡策略        | 无法实现 SQL 审计、限流、防火墙等高级功能           |

## 四、方案二：代理层 - ProxySQL 读写分离实战（生产环境首选）

ProxySQL 是一款高性能的 MySQL 代理中间件，是生产环境读写分离的首选方案，对业务完全透明，功能丰富，稳定性极强。

### 4.1 核心原理

ProxySQL 位于应用和 MySQL 集群之间，应用像连接普通 MySQL 一样连接 ProxySQL，ProxySQL 接收 SQL 后：

1. 解析 SQL 语句，识别读写类型；
2. 根据预设的规则，将 SQL 路由到对应的主库 / 从库；
3. 将执行结果返回给应用，全程对业务无感知；
4. 内置连接池、故障检测、SQL 缓存、限流、审计等高级功能。

### 4.2 环境准备

1. 主从架构：主库 192.168.1.100，从库 192.168.1.101、192.168.1.102；
2. ProxySQL 服务器：192.168.1.200，CentOS 7+/Ubuntu 20.04+；
3. 端口规划：ProxySQL 管理端口 6032，业务访问端口 6033。

### 4.3 完整安装搭建步骤

#### 步骤 1：安装 ProxySQL

```bash
# CentOS/RHEL 安装
yum install -y https://github.com/sysown/proxysql/releases/download/v2.5.5/proxysql-2.5.5-1-centos7.x86_64.rpm

# Ubuntu/Debian 安装
wget https://github.com/sysown/proxysql/releases/download/v2.5.5/proxysql_2.5.5-ubuntu20_amd64.deb
dpkg -i proxysql_2.5.5-ubuntu20_amd64.deb

# 启动 ProxySQL 并设置开机自启
systemctl start proxysql
systemctl enable proxysql
```

#### 步骤 2：登录 ProxySQL 管理端

ProxySQL 的所有配置都通过 MySQL 客户端连接管理端完成：

```bash
# 登录管理端（默认账号密码：admin/admin）
mysql -h 127.0.0.1 -P 6032 -u admin -padmin
```

#### 步骤 3：配置主从库节点（mysql_servers 表）

```sql
-- 1. 清空默认配置
DELETE FROM mysql_servers;

-- 2. 添加主库（hostgroup_id=1 表示写组，固定为1）
INSERT INTO mysql_servers(hostgroup_id, hostname, port, weight, comment)
VALUES (1, '192.168.1.100', 3306, 1, '主库-写节点');

-- 3. 添加从库（hostgroup_id=2 表示读组，固定为2）
INSERT INTO mysql_servers(hostgroup_id, hostname, port, weight, comment)
VALUES (2, '192.168.1.101', 3306, 1, '从库1-读节点'),
       (2, '192.168.1.102', 3306, 1, '从库2-读节点');

-- 4. 加载配置到运行时（生效）
LOAD MYSQL SERVERS TO RUNTIME;
-- 5. 保存配置到磁盘（重启不丢失）
SAVE MYSQL SERVERS TO DISK;

-- 查看配置是否生效
SELECT * FROM mysql_servers;
```

#### 步骤 4：配置监控账号（监控主从库状态）

```sql
-- 1. 先在主库创建监控账号（主从库通用）
CREATE USER 'monitor'@'%' IDENTIFIED BY 'Monitor@20260401';
GRANT REPLICATION CLIENT, SELECT ON *.* TO 'monitor'@'%';
FLUSH PRIVILEGES;

-- 2. 在ProxySQL管理端配置监控账号
SET mysql-monitor_username = 'monitor';
SET mysql-monitor_password = 'Monitor@20260401';

-- 加载配置生效
LOAD MYSQL VARIABLES TO RUNTIME;
SAVE MYSQL VARIABLES TO DISK;

-- 查看监控状态
SELECT * FROM monitor.mysql_server_connect_log;
```

#### 步骤 5：配置业务访问账号

```sql
-- 1. 先在主库创建业务账号（主从库通用）
CREATE USER 'app_user'@'%' IDENTIFIED BY 'App@20260401';
GRANT SELECT, INSERT, UPDATE, DELETE ON my_database.* TO 'app_user'@'%';
FLUSH PRIVILEGES;

-- 2. 在ProxySQL管理端配置业务账号，应用通过此账号连接ProxySQL
INSERT INTO mysql_users(username, password, default_hostgroup, transaction_persistent)
VALUES ('app_user', 'App@20260401', 1, 1);
-- transaction_persistent=1：事务内的所有语句都走主库，保证一致性

-- 加载配置生效
LOAD MYSQL USERS TO RUNTIME;
SAVE MYSQL USERS TO DISK;

-- 查看配置
SELECT * FROM mysql_users;
```

#### 步骤 6：配置读写分离规则（核心）

ProxySQL 通过 `mysql_query_rules` 表配置 SQL 路由规则，核心是匹配 SQL 类型，路由到对应的 hostgroup。

```sql
-- 1. 清空默认规则
DELETE FROM mysql_query_rules;

-- 规则1：所有写操作（INSERT/UPDATE/DELETE/REPLACE）路由到主库（hostgroup_id=1）
INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (1, 1, '^INSERT|^UPDATE|^DELETE|^REPLACE', 1, 1);

-- 规则2：SELECT FOR UPDATE 锁查询，路由到主库
INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply)
VALUES (2, 1, '^SELECT.*FOR UPDATE', 1, 1);

-- 规则3：所有SELECT读操作，路由到从库（hostgroup_id=2），排除强制主库的语句
INSERT INTO mysql_query_rules(rule_id, active, match_pattern, destination_hostgroup, apply, negate_match_pattern)
VALUES (3, 1, '^SELECT', 2, 1, 0);

-- 加载配置生效
LOAD MYSQL QUERY RULES TO RUNTIME;
SAVE MYSQL QUERY RULES TO DISK;

-- 查看规则
SELECT * FROM mysql_query_rules;
```

### 4.4 读写分离验证

1. **应用连接 ProxySQL**：应用的数据库连接地址改为 `192.168.1.200:6033`，账号密码为 `app_user/App@20260401`，无需修改任何业务代码；
2. **验证路由结果**：在 ProxySQL 管理端执行以下命令，查看 SQL 路由情况：

```sql
    -- 查看SQL路由统计
    SELECT hostgroup, digest_text, count_star FROM stats_mysql_query_digest ORDER BY count_star DESC;
```

- 写语句的 `hostgroup` 为 1（主库），读语句的 `hostgroup` 为 2（从库），说明读写分离配置成功。

### 4.5 高可用配置

ProxySQL 是单点，生产环境必须部署双节点 ProxySQL + Keepalived 实现 VIP 漂移，避免单点故障：

1. 部署两台 ProxySQL 节点：192.168.1.200（主）、192.168.1.201（备）；
2. 配置 Keepalived，虚拟 IP（VIP）为 192.168.1.202；
3. 应用连接 VIP，主 ProxySQL 故障时，VIP 自动漂移到备节点，业务无感知。

### 4.6 优缺点总结

|优点|缺点|
|:--|:--|
|对业务完全透明，无代码侵入，支持所有编程语言|代理层有轻微的性能损耗（约 5% 以内，生产环境可忽略）|
|功能丰富，支持故障自动剔除、SQL 审计、限流、缓存、防火墙等|需要部署额外的中间件，需保障代理层高可用，有一定运维成本|
|统一管理多应用、多集群的读写规则，数据库拓扑变化无需修改应用|有一定的学习成本，需要掌握 ProxySQL 的配置规则|
|内置连接池，可大幅提升数据库的连接性能||

## 五、方案三：官方轻量方案 - MySQL Router 读写分离实战

MySQL Router 是 MySQL 官方推出的轻量路由中间件，专为 InnoDB Cluster 主从架构设计，配置极简，兼容性最好。

### 5.1 安装与配置

```bash
# 1. 安装MySQL Router（和MySQL版本一致）
yum install -y mysql-router-community

# 2. 生成配置文件（自动发现主从架构）
mysqlrouter --bootstrap root@192.168.1.100:3306 --user=mysqlrouter

# 3. 启动MySQL Router
systemctl start mysqlrouter
systemctl enable mysqlrouter
```

### 5.2 访问方式

- 写端口：6446，所有连接此端口的请求都路由到主库；
- 读端口：6447，所有连接此端口的请求都轮询路由到从库；
- 应用分别配置写数据源和读数据源，连接对应端口即可。

### 5.3 优缺点总结

|优点|缺点|
|:--|:--|
|官方原生，和 MySQL 兼容性 100%，无版本兼容问题|功能单一，仅支持基础读写分离，无高级功能|
|超轻量，部署配置极简，1 分钟即可完成|无 SQL 解析能力，无法根据 SQL 类型自动路由，需要应用分别配置读写端口|
|性能损耗极低|无故障自动剔除、延迟检测能力|

## 六、读写分离核心痛点：主从延迟解决方案

主从延迟是读写分离最大的痛点，写操作在主库执行后，binlog 同步到从库并重放需要时间，此时读从库会读到旧数据，导致业务异常。

### 6.1 主从延迟的根本原因

1. 主库大事务：批量更新、大表 DDL，导致 binlog 过大，从库重放慢；
2. 从库性能不足：从库硬件配置比主库差，无法跟上主库的写入速度；
3. 从库单线程重放：MySQL 5.6 之前 SQL 线程是单线程，重放速度远低于主库多线程写入；
4. 从库有大查询：从库的慢查询占用大量 CPU/IO，导致 SQL 线程阻塞；
5. 网络延迟：主从库之间网络不佳，binlog 传输慢。

### 6.2 7 种生产环境解决方案

#### 1. 强制主库读（兜底方案）

写后立即读、金融交易、订单状态查询等强一致性场景，强制路由到主库，完全避免主从延迟问题，是最稳妥的方案。

#### 2. 半同步复制（数据一致性保障）

开启半同步复制，主库必须等待至少一个从库收到 binlog 并写入 relay log，才提交事务返回成功，保证主库提交的事务，至少有一个从库已经收到，从库数据永远不会落后主库太多。

#### 3. 从库并行复制（提升重放速度）

MySQL 5.7+ 开启逻辑时钟并行复制，大幅提升从库 SQL 线程的重放速度，降低主从延迟：

```bash
# 从库my.cnf配置
slave_parallel_type = LOGICAL_CLOCK
slave_parallel_workers = 8  # 并行线程数，根据CPU核心数设置
slave_preserve_commit_order = ON
```

#### 4. 主库大事务拆分

将大事务拆分为多个小事务，比如批量更新 100 万行数据，拆分为 100 次更新 1 万行，避免 binlog 过大，从库重放阻塞。

#### 5. 业务延迟容忍设计

非核心场景，业务上做延迟容忍设计，比如用户注册后，提示 “注册成功，数据正在同步”，而非立即跳转到个人中心查询数据。

#### 6. 缓存兜底

写操作完成后，同时将数据写入 Redis 缓存，读操作优先从缓存读取，避免读从库，完全规避主从延迟问题。

#### 7. 从库延迟自动剔除

在读写分离规则中配置延迟阈值，从库延迟超过阈值（如 1 秒），自动将其从读节点列表中剔除，只路由到延迟正常的从库，避免读到旧数据。

## 七、读写分离避坑指南

1. **坑 1：从库开启写入**
    
    - 后果：主从数据不一致，从库写入的数据主库没有，导致主键冲突、复制中断；
    - 避坑：从库必须开启 `read_only=1` 和 `super_read_only=1`，禁止任何写入。
    
2. **坑 2：强一致读场景走从库**
    
    - 后果：写后立即读，主从延迟导致读到旧数据，业务逻辑异常（如支付后订单状态未更新）；
    - 避坑：强一致性场景强制走主库，绝对不能走从库。
    
3. **坑 3：多从库负载不均**
    
    - 后果：一个从库压力拉满，其他从库空闲，资源浪费；
    - 避坑：配置合理的负载均衡策略（轮询 / 权重），监控从库的负载状态，动态调整权重。
    
4. **坑 4：读写分离配置后，从库无流量**
    
    - 后果：读请求全部走主库，读写分离失效；
    - 避坑：检查路由规则，事务内的读语句默认走主库，非事务读才会走从库。
    
5. **坑 5：代理层单点故障**
    
    - 后果：ProxySQL/MyCat 单点故障，整个业务数据库访问中断；
    - 避坑：代理层必须部署双节点 + Keepalived VIP，实现高可用。

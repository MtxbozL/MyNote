
>表级锁的锁定粒度为 **整张表** ，加锁速度快，锁开销小，但并发性能极差，分为三类：表读写锁、元数据锁（MDL）、意向锁。其中表读写锁是 Server 层通用锁，元数据锁是 Server 层表结构保护锁，意向锁是 InnoDB 存储引擎层的表级辅助锁。

### 6.2.1 表读写锁

表读写锁是 Server 层实现的通用表级锁，所有存储引擎均支持，分为读锁（共享锁）与写锁（排他锁）两类。

#### 6.2.1.1 核心语法与加锁规则

- 加锁语法：`LOCK TABLES 表名 READ/WRITE;`
- 解锁语法：`UNLOCK TABLES;` 或 会话断开自动释放。

**锁互斥与兼容规则**：

|锁类型|读锁（共享锁 S）|写锁（排他锁 X）|
|---|---|---|
|读锁|兼容|互斥|
|写锁|互斥|互斥|

1. **读锁（READ）**：
    
    - 加锁后，所有会话（包括加锁会话）仅能读取该表数据，无法执行任何写操作；
    - 多个会话可同时对同一张表加读锁，互不阻塞，实现读共享。
    
2. **写锁（WRITE）**：
    
    - 加锁后，只有加锁会话可对该表执行读写操作，其他所有会话的读、写操作均被阻塞；
    - 写锁与所有锁类型完全互斥，同一时间仅能有一个会话持有表的写锁。
    

#### 6.2.1.2 适用场景与限制

- 适用场景：仅适用于不支持事务与行锁的 MyISAM 引擎，或批量更新全表数据的特殊场景；
- 生产环境限制：InnoDB 引擎严禁使用表读写锁，其行级锁可提供更细粒度的并发控制，表锁会完全丧失 InnoDB 的高并发读写能力。

### 6.2.2 元数据锁（MDL - Metadata Lock）

元数据锁是 MySQL Server 层实现的表级锁，**无需手动干预，所有访问表的会话都会自动加锁**，核心作用是保证表元数据（表结构）的一致性，防止 DML 与 DDL 并发执行导致的数据逻辑错乱，是生产环境表操作故障的高频诱因。

#### 6.2.2.1 锁分类与互斥规则

MDL 锁根据操作类型分为两大类，锁的生命周期为**事务提交 / 回滚后才释放**，而非语句执行完成，这是 MDL 锁故障的核心根源。

|MDL 锁类型|对应操作|核心互斥规则|
|---|---|---|
|MDL 读锁（共享锁）|SELECT、INSERT、UPDATE、DELETE 等 DML 操作|读锁之间完全兼容，读锁与写锁完全互斥|
|MDL 写锁（排他锁）|ALTER、DROP、TRUNCATE、CREATE INDEX 等 DDL 操作|写锁与所有锁类型完全互斥|

#### 6.2.2.2 核心故障场景与避坑规则

**高频故障场景**：长事务持有 MDL 读锁，导致 DDL 被阻塞，DDL 又会阻塞后续所有 DML 操作，最终引发整个表的业务雪崩。

- 故障复现流程：
    
    1. 会话 A 开启事务，执行`SELECT * FROM t`，自动加 MDL 读锁，事务未提交，锁持续持有；
    2. 会话 B 执行`ALTER TABLE t ADD COLUMN f INT`，申请 MDL 写锁，被会话 A 的读锁阻塞，进入等待队列；
    3. 会话 C 及后续所有会话执行`SELECT * FROM t`，申请 MDL 读锁，会被队列中等待的 MDL 写锁阻塞，所有对表 t 的访问全部停滞；
    4. 最终表 t 完全不可用，业务请求大量堆积，数据库连接池被打满，引发系统雪崩。
    

**生产环境避坑核心规则**：

1. 严禁在业务高峰期执行 DDL 操作，所有表结构修改必须在低峰期执行；
2. 执行 DDL 前，必须先检查是否存在长事务、未提交的事务，确认无 MDL 读锁持有后再执行；
3. 为 DDL 操作设置锁等待超时，避免长时间阻塞业务，MySQL 8.0 支持`ALTER TABLE t ADD COLUMN f INT WAIT 3;`语法，超时后自动放弃 DDL；
4. 使用 MySQL 8.0 的 Online DDL 特性，最小化 DDL 过程中的 MDL 写锁持有时间，大部分 DDL 操作仅在开始和结束阶段短暂持有 MDL 写锁，执行过程中仅持有 MDL 读锁，不阻塞业务读写。

### 6.2.3 意向锁（Intention Lock）

意向锁是 InnoDB 存储引擎实现的**表级辅助锁**，核心设计目的是**提升表级锁的加锁效率**，避免加表锁时需要遍历全表检查是否存在行锁，大幅降低锁检查的开销。

#### 6.2.3.1 核心定义与加锁规则

意向锁用于声明一个事务即将要对表中的行加的锁类型，分为两类：

1. **意向共享锁（IS）**：事务准备给表中的行加共享行锁（S 锁），必须先给表加 IS 锁；
2. **意向排他锁（IX）**：事务准备给表中的行加排他行锁（X 锁），必须先给表加 IX 锁。

**加锁流程示例**：

事务执行`SELECT * FROM t WHERE id = 1 FOR UPDATE;`，会先给表 t 加 IX 意向排他锁，再给 id=1 的行加 X 排他行锁。

#### 6.2.3.2 兼容与互斥规则

意向锁的核心特性是：**意向锁之间完全兼容，仅与表级读写锁互斥，与行级锁无任何互斥关系**。

|锁类型|意向共享锁（IS）|意向排他锁（IX）|表读锁|表写锁|
|---|---|---|---|---|
|IS|兼容|兼容|兼容|互斥|
|IX|兼容|兼容|互斥|互斥|
|表读锁|兼容|互斥|兼容|互斥|
|表写锁|互斥|互斥|互斥|互斥|

**核心价值示例**：

事务 A 给表 t 的某一行加了 X 行锁，同时持有表的 IX 意向锁；此时事务 B 要给表 t 加表级写锁，无需遍历全表检查每一行是否有行锁，仅需检查表上是否存在意向锁，发现 IX 锁与表写锁互斥，直接阻塞等待，锁检查开销从 O (n) 降为 O (1)。

**常见误区澄清**：意向锁不会阻塞任何行级锁，仅会阻塞全表级别的读写锁，不会影响行级操作的并发性能，是纯辅助优化锁，无任何并发副作用。

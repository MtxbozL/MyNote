
>死锁是指两个或多个并发事务，互相持有对方需要的锁，同时无限等待对方释放锁，形成循环等待，导致事务永远无法执行下去的现象，是高并发写场景下的高频故障。

### 6.4.1 死锁形成的四大必要条件

死锁的形成必须同时满足以下四个条件，缺一不可，规避死锁的核心就是打破其中任意一个条件：

1. **互斥条件**：一个锁同一时间只能被一个事务持有，其他事务必须等待释放，这是锁机制的固有特性，无法打破；
2. **占有且等待条件**：事务已经持有至少一个锁，同时申请新的锁被阻塞，继续持有已有锁不释放；
3. **不可剥夺条件**：事务已持有的锁，只能由事务自己提交 / 回滚释放，其他事务无法强制剥夺；
4. **循环等待条件**：多个事务形成头尾相接的循环等待链，每个事务都在等待链中下一个事务持有的锁。

### 6.4.2 InnoDB 死锁处理机制

InnoDB 内置了两套死锁处理机制，默认同时生效，避免死锁导致事务永久阻塞：

1. **死锁检测机制**
    
    - 由参数`innodb_deadlock_detect`控制，默认开启；
    - InnoDB 后台线程实时监控事务的锁等待关系，一旦检测到循环等待的死锁，会立即选择**回滚代价最小的事务**（修改行数最少的事务），打破循环等待，让其他事务继续执行。
    
2. **锁等待超时机制**
    
    - 由参数`innodb_lock_wait_timeout`控制，默认 50 秒；
    - 作为死锁检测的兜底机制，当死锁检测失效、事务锁等待超过阈值时，自动回滚当前等待的事务，抛出锁等待超时异常。
    

### 6.4.3 死锁排查核心方法

#### 6.4.3.1 实时死锁查看

通过以下 SQL 查看最近一次检测到的死锁详细日志，是死锁排查的核心手段：

```sql
SHOW ENGINE INNODB STATUS;
```

日志中`LATEST DETECTED DEADLOCK`部分，会完整记录死锁的核心信息：

1. 发生死锁的两个事务的 ID、执行的 SQL 语句；
2. 两个事务分别持有的锁类型、锁定的对象；
3. 两个事务分别等待的锁类型；
4. 最终被回滚的事务信息。

#### 6.4.3.2 死锁日志持久化

生产环境推荐开启死锁日志持久化，将所有死锁事件记录到 MySQL 错误日志中，方便事后回溯排查：

```sql
-- 开启死锁全量日志记录，重启后失效，需写入my.cnf永久生效
SET GLOBAL innodb_print_all_deadlocks = ON;
```

### 6.4.4 死锁规避核心方案

1. **统一资源访问顺序**：所有业务事务，必须按照相同的顺序访问表和行记录，彻底打破循环等待条件，这是规避死锁最有效的手段。
    
    示例：避免事务 A 先更新 id=1 再更新 id=2，事务 B 先更新 id=2 再更新 id=1，统一为按 id 升序更新。
2. **大事务拆分为小事务**：减少事务内的操作数量，缩短锁的持有时间，降低锁冲突的概率，生产环境严禁长事务、大事务。
3. **精准命中索引，最小化锁定范围**：所有 DML 语句必须命中主键或唯一索引，避免无索引导致的表锁，避免间隙锁扩大锁定范围，减少锁冲突的可能性。
4. **降低隔离级别**：业务允许的前提下，将隔离级别从 RR 降为 RC，RC 级别无间隙锁，大幅降低锁冲突与死锁概率，同时减少锁的开销。
5. **避免人为锁等待**：严禁在事务内执行人工操作、接口调用等耗时操作，避免事务长时间持有锁不提交。


>行级锁是 InnoDB 独有的锁机制，锁定粒度为**表中的单行索引记录**，锁开销大，加锁速度慢，但并发性能是所有锁类型中最高的，是 InnoDB 支撑高并发读写的核心能力。

> **核心前提**：InnoDB 的行级锁**完全基于索引实现**，如果 SQL 语句无法命中索引，InnoDB 无法定位到具体的行记录，会对全表所有记录加行锁，最终退化为表锁，完全丧失并发能力。

### 6.3.1 行锁的基础分类

行级锁分为共享锁（S 锁）与排他锁（X 锁）两类，是所有行锁算法的基础：

1. **共享锁（S 锁，读锁）**：
    
    - 加锁语法：`SELECT ... LOCK IN SHARE MODE;`
    - 核心规则：多个事务可同时对同一行加 S 锁，实现读共享；持有 S 锁的事务仅能读取该行，无法修改；若某行已有 S 锁，其他事务无法对该行加 X 锁，必须等待所有 S 锁释放。
    
2. **排他锁（X 锁，写锁）**：
    
    - 加锁语法：`SELECT ... FOR UPDATE;`
    - 自动加锁场景：`INSERT`、`UPDATE`、`DELETE`语句会自动给命中的行加 X 锁，事务提交 / 回滚后释放。
    - 核心规则：同一时间仅能有一个事务持有某行的 X 锁，持有 X 锁的事务可读写该行；X 锁与所有 S 锁、X 锁均互斥，彻底避免并发修改冲突。
    

### 6.3.2 Record Lock（记录锁）

记录锁是最基础的行锁，**仅锁定索引中的某一行具体记录**，不会扩大锁定范围，并发性能最优。

- 锁定对象：索引中的具体记录，而非物理行数据，无索引则无法加记录锁；
- 生效场景：主键或唯一索引的**等值查询，且精准命中存在的记录**，InnoDB 会将临键锁退化为记录锁，仅锁定命中的单行记录；
- 示例：用户表主键为 id，执行`SELECT * FROM user WHERE id = 1 FOR UPDATE;`，仅会给 id=1 的主键索引记录加 X 型记录锁，其他事务无法修改 / 删除 id=1 的行，但可正常修改其他所有行，无任何阻塞。

### 6.3.3 Gap Lock（间隙锁）

间隙锁是锁定**索引记录之间的间隙**的锁，不会锁定索引记录本身，核心作用是**防止其他事务在间隙中插入新的记录**，是 InnoDB 在 RR 隔离级别下解决当前读幻读问题的核心机制。

> **生效范围**：间隙锁仅在**可重复读（RR）隔离级别**下生效，读已提交（RC）隔离级别下无间隙锁，仅存在记录锁，因此 RC 级别会存在幻读问题。

#### 6.3.3.1 核心定义与间隙规则

InnoDB 的索引是有序的 B+Tree 结构，相邻两个索引记录之间的区间称为间隙，间隙锁锁定的是这个开区间，防止插入操作。

- 示例：主键索引存在 id=1、4、7 三条记录，索引的间隙分为四个开区间：`(-∞,1)`、`(1,4)`、`(4,7)`、`(7,+∞)`；
- 加锁规则：间隙锁仅会阻塞`INSERT`操作，多个事务可同时给同一个间隙加间隙锁，间隙锁之间完全兼容，无互斥关系；
- 示例：执行`SELECT * FROM user WHERE id BETWEEN 1 AND 4 FOR UPDATE;`，会给`(1,4)`间隙加间隙锁，同时给 id=1、4 的记录加记录锁；其他事务无法插入 id=2、3 的记录，彻底避免幻读，但可正常修改 id=1、4 的已有记录，也可插入 id>4 或 id<1 的记录。

#### 6.3.3.2 核心生效场景

1. 唯一索引的等值查询，命中的记录不存在时，会给查询条件对应的间隙加间隙锁；
    
    示例：`SELECT * FROM user WHERE id = 2 FOR UPDATE;`，id=2 不存在，会给`(1,4)`间隙加间隙锁，防止插入 id=2 的记录。
2. 范围查询命中的索引区间，会给区间内的所有间隙加间隙锁；
3. 普通二级索引的等值查询，即使命中记录，也会给相邻间隙加间隙锁。

### 6.3.4 Next-Key Lock（临键锁）

临键锁是 InnoDB 在**RR 隔离级别下，行锁的默认加锁算法**，是**记录锁 + 间隙锁**的结合体，锁定一个**左开右闭**的索引区间，核心作用是彻底解决 RR 级别下当前读的幻读问题。

#### 6.3.4.1 临键区间规则

InnoDB 会将有序的索引记录，拆分为多个左开右闭的临键区间，每个临键锁锁定的是完整的区间。

- 示例：主键索引存在 id=1、4、7 三条记录，拆分后的临键区间为：
    
    1. `(-∞, 1]`
    2. `(1, 4]`
    3. `(4, 7]`
    4. `(7, +∞]`
    
- 核心规则：临键锁会锁定整个区间，包括区间的右边界记录，既防止区间内的插入操作，也锁定右边界的记录修改。

#### 6.3.4.2 核心加锁退化规则

临键锁是默认算法，在特定场景下会发生退化，最小化锁定范围，提升并发性能：

1. **唯一索引的等值查询，精准命中存在的记录**：临键锁退化为**记录锁**，仅锁定命中的单行记录，不加间隙锁，最小化锁定范围；
2. **唯一索引的等值查询，记录不存在**：临键锁退化为**间隙锁**，仅锁定查询条件对应的间隙，不锁定任何记录；
3. **普通二级索引的等值查询，无论是否命中记录**：均不会退化，完整加临键锁，同时会给主键索引对应的记录加记录锁；
4. **范围查询**：无论主键 / 唯一索引 / 二级索引，均会给命中的所有临键区间加完整临键锁，包括最后一个区间的右边界。

#### 6.3.4.3 幻读问题的完整解决方案

InnoDB 在 RR 隔离级别下，通过**MVCC + 临键锁**的组合，彻底解决了幻读问题：

1. **快照读（普通 SELECT）**：通过 MVCC 的一致性读视图，保证事务内所有快照读的结果集完全一致，不会出现幻读；
2. **当前读（锁定读 / 增删改）**：通过临键锁，锁定查询条件对应的所有索引区间，防止其他事务插入符合条件的新记录，彻底避免幻读。

## 6.1 全局锁

全局锁是作用于 **整个 MySQL 实例** 的锁，加锁后整个实例处于只读状态，所有 DML（增删改）、DDL（表结构修改）、事务提交语句都会被阻塞，是 MySQL 中粒度最粗的锁。

### 6.1.1 核心语法与使用场景

- 加锁语法：`FLUSH TABLES WITH READ LOCK;`（简称 FTWRL）
- 解锁语法：`UNLOCK TABLES;` 或 加锁会话断开后自动释放锁。

**核心唯一适用场景：全库逻辑备份**

对于不支持事务的存储引擎（如 MyISAM），全库备份必须通过 FTWRL 加全局锁，确保备份过程中全库数据处于一致性状态，避免备份期间数据修改导致备份文件出现数据不一致。

- MySQL 官方逻辑备份工具`mysqldump`中，使用`--lock-all-tables`参数时，会执行 FTWRL 加全局锁；
- 对于 InnoDB 引擎，推荐使用`--single-transaction`参数，通过 MVCC 的一致性快照实现无锁备份，无需加全局锁，不会阻塞业务读写，是生产环境的首选方案。

### 6.1.2 核心特性与风险

1. **全局阻塞风险**：主库执行 FTWRL 加全局锁后，会阻塞整个实例的所有写操作，导致业务停写，核心业务系统严禁随意执行；
2. **从库延迟风险**：从库执行 FTWRL 加全局锁后，会阻塞主库同步过来的 Binlog 回放，导致主从延迟急剧扩大，数据同步中断；
3. **锁释放规则**：全局锁与会话绑定，加锁会话异常断开后，锁会自动释放，不会永久阻塞实例；
4. **特殊例外**：FTWRL 加锁后，仅阻塞普通业务的写操作，不会阻塞 MySQL 实例的系统日志写入与 Binlog 写入。

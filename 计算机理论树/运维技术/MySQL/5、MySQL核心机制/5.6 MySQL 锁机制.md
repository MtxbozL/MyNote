## 一、开篇：锁机制的核心定位与价值

MySQL 锁机制是数据库**解决并发事务冲突**的核心手段，是实现事务隔离性、保证数据一致性的底层基石。

简单理解：锁就像数据库给数据加的「保护锁」，多个事务同时操作同一份数据时，通过锁来控制操作的顺序和权限，避免出现脏写、更新丢失、数据不一致等问题。

### 核心认知

1. **锁的本质**：是数据库对共享资源（数据）的访问控制机制，保证同一时刻只有符合条件的事务能操作数据；
2. **锁的分层**：MySQL 锁分为「服务层锁」和「InnoDB 引擎层锁」，InnoDB 锁是生产环境的核心；
3. **锁的目标**：在「并发性能」和「数据一致性」之间找到平衡，锁粒度越细，并发性能越高，锁管理成本也越高。

## 二、锁的核心分类体系

MySQL 锁可从多个维度分类，先建立完整的分类框架，再逐一拆解核心锁类型。

### 2.1 按锁的粒度（最核心分类）

表格

|锁粒度|核心说明|优点|缺点|适用场景|
|:--|:--|:--|:--|:--|
|**表锁**|锁定整张表，操作表时锁定整张表，其他事务无法操作该表的任何数据|锁开销小，加锁解锁快，无死锁|并发性能极差，锁冲突概率极高|MyISAM 引擎（默认）、InnoDB 批量操作（如 ALTER TABLE）|
|**行锁**|仅锁定操作的行数据，其他事务可操作同表的其他行|并发性能极高，锁冲突概率低|锁开销大，加锁解锁慢，可能出现死锁|InnoDB 引擎（默认）、高并发读写场景|
|**页锁**|锁定数据页（InnoDB 数据页默认 16KB），介于表锁和行锁之间|平衡锁开销和并发性能|并发性能不如行锁，锁冲突概率高于行锁|极少使用，仅部分特殊引擎支持|

> 关键：InnoDB 支持表锁和行锁，默认用行锁；MyISAM 仅支持表锁，无行锁。

### 2.2 按锁的兼容性（操作权限）

| 锁类型               | 核心说明                      | 兼容关系          | 典型场景                                           |
| :---------------- | :------------------------ | :------------ | :--------------------------------------------- |
| **共享锁（S 锁 / 读锁）** | 多个事务可同时持有，仅允许读，不允许写       | 与共享锁兼容，与排他锁互斥 | `SELECT ... LOCK IN SHARE MODE`                |
| **排他锁（X 锁 / 写锁）** | 仅一个事务可持有，允许读和写，其他事务无法加任何锁 | 与所有锁互斥        | `UPDATE/DELETE/INSERT`、`SELECT ... FOR UPDATE` |

> 核心规则：**读读共享、读写互斥、写写互斥**。

### 2.3 按锁的算法（InnoDB 行锁细分）

InnoDB 行锁基于索引实现，根据锁定范围不同，分为三种核心算法：

| 锁算法                    | 核心说明              | 锁定范围               | 解决的问题                  | 适用隔离级别   |
| :--------------------- | :---------------- | :----------------- | :--------------------- | :------- |
| **记录锁（Record Lock）**   | 锁定单个索引行           | 仅锁定命中的索引行          | 防止并发更新同一行数据            | 所有隔离级别   |
| **间隙锁（Gap Lock）**      | 锁定索引行之间的间隙，不包含行本身 | 锁定两个索引行之间的空区间      | 防止其他事务插入数据，解决幻读        | RR（可重复读） |
| **临键锁（Next-Key Lock）** | 记录锁 + 间隙锁的组合      | 锁定命中的索引行 + 该行之前的间隙 | 彻底解决幻读，是 InnoDB 默认行锁算法 | RR（可重复读） |

### 2.4 其他常见分类

| 锁类型                     | 核心说明                                    | 典型场景                                 |
| :---------------------- | :-------------------------------------- | :----------------------------------- |
| **意向锁（Intention Lock）** | 表级锁，用于快速判断表中是否有行锁，分为意向共享锁（IS）和意向排他锁（IX） | InnoDB 自动加，无需手动操作                    |
| **自增锁（Auto-inc Lock）**  | 表级锁，用于自增列（AUTO_INCREMENT）的原子性递增         | 插入自增列数据时自动加锁                         |
| **元数据锁（MDL）**           | 表级锁，用于保护表结构，分为读锁和写锁                     | 执行 DDL（ALTER/CREATE）时加写锁，执行 DML 时加读锁 |

## 三、核心锁类型详解

### 3.1 表锁（Table Lock）

#### 核心定义

锁定整张表，加锁后，其他事务对该表的所有操作（读 / 写）都会被阻塞，直到锁释放。

#### 手动加解锁语法

```sql
-- 加读锁（共享锁）
LOCK TABLES table_name READ;
-- 加写锁（排他锁）
LOCK TABLES table_name WRITE;

-- 解锁
UNLOCK TABLES;
```

#### 核心特性

1. **MyISAM 表锁行为**：
    
    - 读锁：多个事务可同时加读锁，允许读，禁止写；
    - 写锁：仅一个事务可加写锁，禁止其他所有读 / 写操作；
    - MyISAM 执行 `SELECT` 自动加读锁，执行 `INSERT/UPDATE/DELETE` 自动加写锁。
    
2. **InnoDB 表锁行为**：
    
    - InnoDB 仅在执行 `ALTER TABLE`、`DROP TABLE` 等 DDL 时自动加表锁；
    - 手动加表锁会覆盖行锁，导致并发性能骤降，生产环境严禁手动加 InnoDB 表锁。
    

#### 适用场景

- MyISAM 引擎的表（仅支持表锁）；
- InnoDB 表的批量数据导入（如 `LOAD DATA INFILE`），临时加表锁提升性能；
- 表结构变更（DDL）。

### 3.2 行锁（Row Lock）

#### 核心定义

InnoDB 核心锁类型，仅锁定操作的索引行，其他事务可正常操作同表的其他行，是高并发场景的基础。

#### 核心前提

InnoDB 行锁**基于索引**实现，若查询 / 更新语句未命中索引（全表扫描），会自动升级为表锁，导致锁冲突！

#### 手动加行锁语法

```sql
-- 加共享锁（S锁），允许读，禁止其他事务加排他锁
SELECT * FROM table_name WHERE id = 1 LOCK IN SHARE MODE;

-- 加排他锁（X锁），禁止其他事务加任何锁
SELECT * FROM table_name WHERE id = 1 FOR UPDATE;
```

#### 核心特性

1. **加锁时机**：
    
    - 执行 `INSERT/UPDATE/DELETE` 时，InnoDB 自动给命中的索引行加排他锁；
    - 执行 `SELECT ... FOR UPDATE/LOCK IN SHARE MODE` 时，手动加行锁；
    - 事务提交 / 回滚后，行锁自动释放。
    
2. **死锁风险**：多个事务互相持有对方需要的锁，导致无限等待，InnoDB 会自动检测死锁，回滚其中一个事务。

#### 实操示例：行锁 vs 表锁（索引影响）

```sql
-- 表结构：user(id INT PRIMARY KEY, name VARCHAR(50))
-- 场景1：命中主键索引，加行锁
BEGIN;
UPDATE user SET name = '张三' WHERE id = 1; -- 仅锁定id=1的行
-- 其他事务可正常操作id=2的行，无锁冲突

-- 场景2：未命中索引（全表扫描），升级为表锁
BEGIN;
UPDATE user SET name = '张三' WHERE name = '李四'; -- name无索引，全表扫描，加表锁
-- 其他事务无法操作该表的任何行，锁冲突
```

### 3.3 间隙锁（Gap Lock）

#### 核心定义

锁定索引行之间的「间隙」，不包含行本身，目的是防止其他事务在间隙中插入数据，解决幻读问题。

#### 核心场景

假设 user 表的 id 有 1、3、5，执行：

```sql
BEGIN;
SELECT * FROM user WHERE id > 2 AND id < 5 FOR UPDATE;
```

InnoDB 会锁定 (2,3)、(3,5) 两个间隙，其他事务无法插入 id=2、4 的数据，防止幻读。

#### 核心特性

1. **仅在 RR 级别生效**：RC（读已提交）级别下，间隙锁自动关闭，仅用记录锁；
2. **仅对非唯一索引生效**：唯一索引（如主键）的等值查询，不会加间隙锁，仅加记录锁；
3. **锁的释放**：事务提交 / 回滚后释放。

### 3.4 临键锁（Next-Key Lock）

#### 核心定义

InnoDB 默认的行锁算法，是「记录锁 + 间隙锁」的组合，锁定范围包含「索引行 + 该行之前的间隙」。

#### 核心场景

还是 user 表 id 有 1、3、5，执行：

```sql
BEGIN;
SELECT * FROM user WHERE id <= 3 FOR UPDATE;
```

InnoDB 会锁定：

- 记录锁：id=3 的行；
- 间隙锁：(-∞,1)、(1,3)；
    
    总锁定范围：(-∞,3]，其他事务无法插入 id=0、2 的数据，也无法更新 id=3 的数据。

#### 核心价值

在 RR 级别下，通过临键锁彻底解决幻读问题，保证事务隔离性。

### 3.5 意向锁（Intention Lock）

#### 核心定义

表级锁，InnoDB 自动加，用于快速判断「表中是否有行锁」，避免逐行检查锁状态，提升锁管理效率。

#### 分类与规则

| 意向锁类型         | 加锁时机                 | 兼容关系                           |
| :------------ | :------------------- | :----------------------------- |
| **意向共享锁（IS）** | 事务给行加共享锁前，自动给表加 IS 锁 | 与 IS、IX 兼容，与表级 S 锁兼容，与表级 X 锁互斥 |
| **意向排他锁（IX）** | 事务给行加排他锁前，自动给表加 IX 锁 | 与 IS、IX 兼容，与表级 S/X 锁互斥         |

#### 核心作用

比如事务 A 给表中某行加了行锁，事务 B 想给整张表加表锁，InnoDB 只需检查表的意向锁，就能快速判断表中有行锁，无需逐行检查，大幅提升效率。

### 3.6 元数据锁（MDL）

#### 核心定义

表级锁，MySQL 服务层自动加，用于保护表结构，防止 DML 和 DDL 并发执行导致表结构不一致。

#### 加锁规则

| 操作类型                                  | MDL 锁类型 | 兼容关系         |
| :------------------------------------ | :------ | :----------- |
| `SELECT/INSERT/UPDATE/DELETE`         | MDL 读锁  | 多个读锁兼容，与写锁互斥 |
| `ALTER TABLE/CREATE INDEX/DROP TABLE` | MDL 写锁  | 与所有锁互斥       |

#### 常见坑点

- **长事务持有 MDL 读锁**：长事务执行 `SELECT` 后未提交，会持有 MDL 读锁，导致后续 `ALTER TABLE` 被阻塞，所有新的 DML 操作也会被阻塞；
- **解决方法**：kill 长事务，或先执行 `KILL QUERY` 终止长事务的查询，再执行 DDL。

### 3.7 自增锁（Auto-inc Lock）

#### 核心定义

表级锁，用于保证自增列（AUTO_INCREMENT）的原子性递增，防止并发插入时自增 ID 重复。

#### 优化机制

MySQL 8.0 引入「可配置的自增锁」，通过 `innodb_autoinc_lock_mode` 参数控制：

| 参数值   | 锁机制                | 优点           | 缺点                           |
| :---- | :----------------- | :----------- | :--------------------------- |
| 0     | 传统表锁，插入时加锁，语句结束释放  | 绝对安全，无 ID 重复 | 并发插入性能差                      |
| 1（默认） | 普通插入用轻量级锁，批量插入用表锁  | 平衡性能和安全      | 批量插入性能略差                     |
| 2     | 完全无锁，自增 ID 由引擎自动分配 | 并发性能最高       | 主从复制用 STATEMENT 格式会导致 ID 不一致 |

## 四、InnoDB 锁机制与隔离级别的关联

不同隔离级别下，InnoDB 锁机制的行为差异是面试核心考点，核心关联如下：

表格

| 隔离级别                  | 锁机制核心行为                                                                    | 解决的问题           |
| :-------------------- | :------------------------------------------------------------------------- | :-------------- |
| **读未提交（RU）**          | 无行锁，所有读操作不加锁，直接读最新数据                                                       | 无，所有并发问题都存在     |
| **读已提交（RC）**          | 1. 仅用记录锁，关闭间隙锁；<br><br>2. 每次快照读生成新的 Read View；<br><br>3. 支持半一致性读，减少锁等待     | 解决脏读，存在不可重复读、幻读 |
| **可重复读（RR）**          | 1. 默认用临键锁（记录锁 + 间隙锁）；<br><br>2. 第一次快照读生成 Read View，后续复用；<br><br>3. 关闭半一致性读 | 解决脏读、不可重复读、幻读   |
| **串行化（Serializable）** | 1. 所有 SELECT 自动加共享锁；<br><br>2. 所有 DML 加排他锁；<br><br>3. 事务完全串行执行             | 解决所有并发问题，性能极差   |

### 核心区别：RC vs RR 锁行为

|对比维度|读已提交（RC）|可重复读（RR）|
|:--|:--|:--|
|间隙锁|关闭|开启（临键锁）|
|幻读|存在|解决|
|锁范围|仅记录锁，范围小|临键锁，范围大|
|死锁概率|低|高|
|并发性能|高|中|
|半一致性读|支持|不支持|

## 五、锁问题排查与实战工具

### 5.1 查看锁状态

#### 1. 查看当前锁等待情况（MySQL 8.0+）

```sql
-- 查看锁等待详情（核心）
SELECT * FROM performance_schema.data_lock_waits;

-- 查看所有锁持有情况
SELECT * FROM performance_schema.data_locks;

-- 简化版锁等待查询
SELECT 
  r.trx_id waiting_trx_id,
  r.trx_mysql_thread_id waiting_thread,
  b.trx_id blocking_trx_id,
  b.trx_mysql_thread_id blocking_thread,
  t.table_name,
  l.lock_index,
  l.lock_type,
  l.lock_mode
FROM information_schema.innodb_lock_waits w
JOIN information_schema.innodb_trx b ON w.blocking_trx_id = b.trx_id
JOIN information_schema.innodb_trx r ON w.requesting_trx_id = r.trx_id
JOIN information_schema.innodb_locks l ON w.lock_id = l.lock_id
JOIN information_schema.tables t ON l.lock_table = t.table_name;
```

#### 2. 查看事务状态（定位长事务）

```sql
-- 查看所有活跃事务，重点关注trx_started（事务开始时间）
SELECT 
  trx_id,
  trx_mysql_thread_id,
  trx_started,
  trx_state,
  trx_query
FROM information_schema.innodb_trx
WHERE trx_state = 'RUNNING';
```

#### 3. 查看 MDL 锁

```sql
-- 查看MDL锁持有情况
SELECT * FROM performance_schema.metadata_locks;
```

### 5.2 死锁排查

#### 1. 开启死锁日志

```sql
-- 临时开启（重启失效）
SET GLOBAL innodb_print_all_deadlocks = ON;

-- 永久开启（修改my.cnf）
[mysqld]
innodb_print_all_deadlocks = 1
```

#### 2. 查看死锁日志

死锁日志会写入 MySQL 错误日志，核心信息包括：

- 死锁涉及的事务 ID；
- 每个事务持有的锁和等待的锁；
- 回滚的事务；
- 死锁发生的 SQL 语句。

#### 3. 死锁解决方法

- 优化 SQL，让事务按相同的顺序访问数据（核心）；
- 减少事务持有锁的时间，事务尽量短；
- 降低隔离级别（如 RR 改为 RC），减少锁范围；
- 手动设置锁等待超时时间（`innodb_lock_wait_timeout`，默认 50 秒）。

### 5.3 核心参数配置

| 参数名                        | 默认值             | 优化建议                   | 说明              |
| :------------------------- | :-------------- | :--------------------- | :-------------- |
| innodb_lock_wait_timeout   | 50              | 核心场景设为 10-30 秒         | 行锁等待超时时间，超时自动回滚 |
| innodb_deadlock_detect     | ON              | 保持开启                   | 自动检测死锁，回滚其中一个事务 |
| innodb_print_all_deadlocks | OFF             | 开启为 ON                 | 记录所有死锁日志到错误日志   |
| innodb_autoinc_lock_mode   | 1               | 保持默认 1                 | 自增锁模式，平衡性能和安全   |
| transaction_isolation      | REPEATABLE-READ | 高并发场景改为 READ-COMMITTED | 隔离级别，影响锁行为      |

## 六、生产环境锁问题避坑指南

### 6.1 行锁升级为表锁

#### 坑点

查询 / 更新语句未命中索引，导致行锁升级为表锁，并发性能骤降。

#### 解决方法

1. 所有 DML 语句的 WHERE 条件必须命中索引；
2. 定期检查慢查询日志，优化全表扫描的 SQL；
3. 给常用查询字段建立合适的索引。

### 6.2 长事务导致锁长期持有

#### 坑点

长事务持有行锁 / 间隙锁 / MDL 锁，导致其他事务阻塞，甚至磁盘空间膨胀。

#### 解决方法

1. 事务尽量短，执行完立即提交 / 回滚；
2. 不在事务中执行非数据库操作（如 RPC 调用、文件读写）；
3. 定期监控长事务，超过阈值自动 kill。

### 6.3 死锁频发

#### 坑点

多个事务按不同顺序访问数据，导致死锁，影响业务稳定性。

#### 解决方法

1. 所有事务按相同的顺序访问表 / 行（如先更新 A 表，再更新 B 表）；
2. 减少事务内的操作数量，拆分大事务为小事务；
3. 降低隔离级别为 RC，减少锁范围；
4. 捕获死锁异常，重试事务。

### 6.4 MDL 锁阻塞 DDL

#### 坑点

长事务持有 MDL 读锁，导致 DDL 操作阻塞，新的 DML 也被阻塞。

#### 解决方法

1. 执行 DDL 前，先 kill 长事务；
2. 使用 Online DDL 工具（如 pt-online-schema-change），避免锁表；
3. 选择业务低峰期执行 DDL。

### 6.5 间隙锁导致插入阻塞

#### 坑点

RR 级别下，间隙锁锁定了插入的区间，导致并发插入阻塞。

#### 解决方法

1. 改为 RC 级别，关闭间隙锁；
2. 使用唯一索引的等值查询，避免间隙锁；
3. 调整业务逻辑，避免在热点区间插入数据。

## 八、核心要点总结

1. **锁的核心价值**：解决并发事务冲突，平衡「数据一致性」和「并发性能」，InnoDB 行锁是高并发场景的核心。
2. **行锁前提**：InnoDB 行锁基于索引，未命中索引会升级为表锁，生产环境必须保证 DML 语句命中索引。
3. **核心锁算法**：RR 级别下默认用临键锁（记录锁 + 间隙锁）解决幻读，RC 级别仅用记录锁，关闭间隙锁提升并发。
4. **隔离级别关联**：隔离级别越高，锁范围越大，隔离性越强，并发性能越低，生产环境仅用 RC/RR。
5. **常见坑点**：行锁升级表锁、长事务持有锁、死锁、MDL 锁阻塞 DDL，核心解决思路是「短事务、命中索引、统一访问顺序」。
6. **排查工具**：通过 performance_schema/information_schema 查看锁等待、事务状态、MDL 锁，开启死锁日志定位死锁原因。

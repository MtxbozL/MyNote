## 一、开篇：为什么要深入学习 InnoDB？

### 1. InnoDB 的核心地位

InnoDB 是 **MySQL 5.5 及之后版本的默认存储引擎**，也是目前工业界 99% 业务场景的首选存储引擎，其核心优势在于：

- **完整支持 ACID 事务**：是核心交易、金融场景的刚需；
- **行级锁**：高并发场景下锁粒度小，性能远优于 MyISAM 的表级锁；
- **崩溃恢复**：通过 redo log/undo log 保证数据不丢失，数据库宕机后可自动恢复；
- **MVCC 多版本并发控制**：读写不阻塞，大幅提升并发性能；
- **聚簇索引**：数据和索引存放在一起，查询性能更优。

### 2. 学习 InnoDB 的核心价值

- **原理理解**：搞懂事务、锁、索引、MVCC 这些核心能力的底层实现；
- **性能优化**：从存储引擎层面做优化，而不是只停留在「加索引」的表层；
- **问题定位**：死锁、幻读、数据丢失、慢查询等问题，能快速定位到 InnoDB 的具体模块；
- **面试刚需**：InnoDB 的架构、事务、锁、MVCC、日志是后端 / 运维 / DBA 岗位的面试必考题。

---

## 二、InnoDB 整体架构全景

InnoDB 的架构分为**内存结构**和**磁盘结构**两大部分，核心是「内存缓冲 + 磁盘持久化 + 日志保障」的设计：

```txt
客户端 → 【MySQL服务层】 → 【InnoDB内存结构】 ←→ 【InnoDB磁盘结构】 ←→ 磁盘文件系统
```

---

## 三、第一部分：InnoDB 内存结构（性能核心）

InnoDB 内存结构的核心设计思想是 **「尽量让读写操作在内存中完成，减少磁盘 IO」**，因为内存读写速度是磁盘的百万倍以上。

### 3.1 缓冲池（Buffer Pool）：最重要的内存组件

#### 核心作用

Buffer Pool 是 InnoDB 最大的一块内存区域（默认 128MB，生产环境建议设置为服务器内存的 50%-70%），本质是一块内存缓存，用来缓存**磁盘上的数据页和索引页**，避免每次查询都访问磁盘。

#### 核心数据结构

- **数据页（Data Page）**：InnoDB 磁盘存储的最小单位，默认 16KB，表中的数据按页存储；
- **缓存页（Buffer Page）**：数据页在内存中的镜像，Buffer Pool 中存储的就是一个个缓存页；
- **控制块**：每个缓存页对应的元数据，记录缓存页的状态、所属表空间、LSN（日志序列号）等信息。

#### LRU 算法：缓存页的淘汰策略

Buffer Pool 内存有限，当缓存页满了之后，需要淘汰「最近最少使用」的缓存页，InnoDB 对传统 LRU 做了优化，采用 **「冷热数据分离」的 LRU 链表 **：

- **链表分为两部分**：热数据区（young list，占 5/8）和冷数据区（old list，占 3/8）；
- **新数据页插入**：新加载的数据页先插入冷数据区的头部，避免全表扫描把热数据淘汰；
- **晋升热数据**：冷数据区的缓存页被访问后，等待一定时间（默认 1s），才会晋升到热数据区头部；
- **淘汰策略**：从冷数据区的尾部开始淘汰。

#### 核心优化建议

- **设置合理的 Buffer Pool 大小**：生产环境建议 `innodb_buffer_pool_size = 服务器内存 × 50%~70%`；
- **开启多实例 Buffer Pool**：如果 Buffer Pool 超过 1GB，建议开启 `innodb_buffer_pool_instances`，分成多个实例，减少锁竞争；
- **监控 Buffer Pool 命中率**：通过 `SHOW ENGINE INNODB STATUS` 查看命中率，目标是 99% 以上，命中率低说明缓存效果差。

---

### 3.2 Change Buffer（写缓冲）：提升写入性能

#### 核心作用

Change Buffer 是 Buffer Pool 的一部分，专门用来缓存**对非唯一辅助索引的修改操作**（INSERT/UPDATE/DELETE），避免每次修改都直接访问磁盘，减少随机 IO，大幅提升写入性能。

#### 核心原理

- 非唯一辅助索引的修改，不影响数据唯一性，不需要立即更新磁盘；
- 先把修改操作缓存在 Change Buffer 中；
- 当后续查询访问到该索引页时，再把 Change Buffer 中的修改合并（Merge）到磁盘的索引页中；
- 后台线程也会定期 Merge Change Buffer 到磁盘。

#### 限制条件

- **仅适用于非唯一辅助索引**：唯一索引需要校验唯一性，必须立即访问磁盘，无法使用 Change Buffer；
- **适用于写多读少的场景**：如果修改后立即查询，Merge 操作会抵消 Change Buffer 的性能优势。

#### 核心优化建议

- 写多读少的场景，可适当增大 `innodb_change_buffer_max_size`（默认占 Buffer Pool 的 25%，最大 50%）；
- 读多写少的场景，可适当减小该值。

---

### 3.3 自适应哈希索引（Adaptive Hash Index）

#### 核心作用

InnoDB 会自动监控 Buffer Pool 中热点索引页的查询，如果发现某个索引页被频繁查询，会自动为该页创建**哈希索引**，利用哈希索引 O (1) 的查询特性，提升热点数据的查询效率。

#### 核心特点

- **自动创建**：无需人工干预，InnoDB 自动判断是否需要创建；
- **仅适用于等值查询**：哈希索引不支持范围查询、排序；
- **默认开启**：可通过 `innodb_adaptive_hash_index` 控制开关。

---

### 3.4 Log Buffer（日志缓冲区）

#### 核心作用

Log Buffer 是用来缓存 **redo log** 的内存区域，避免每次写 redo log 都直接刷盘，减少磁盘 IO，提升写入性能。

#### 核心原理

- 先把 redo log 写入 Log Buffer；
- 当满足一定条件时（如 Log Buffer 满了、事务提交、定时刷盘），再把 Log Buffer 中的 redo log 刷到磁盘的 redo log 文件中；
- 刷盘策略由 `innodb_flush_log_at_trx_commit` 控制（后面详细讲）。

#### 核心优化建议

- Log Buffer 大小默认 16MB，一般足够，无需调太大；
- 大事务场景可适当增大 `innodb_log_buffer_size`，避免频繁刷盘。

---

## 四、第二部分：InnoDB 磁盘结构（持久化核心）

InnoDB 磁盘结构负责数据的持久化存储，保证数据库重启后数据不丢失。

### 4.1 表空间（Tablespace）

表空间是 InnoDB 存储数据的逻辑容器，分为以下几类：

#### 1. 系统表空间（System Tablespace）

- **文件**：默认是 `ibdata1`，可配置多个文件；
- **存储内容**：
    
    - InnoDB 数据字典（表、索引、字段的元数据）；
    - undo log（回滚日志）；
    - Change Buffer 的持久化存储；
    - 双写缓冲区（Doublewrite Buffer）；
    
- **特点**：文件大小会自动增长，但不会自动收缩，删除数据后空间不会释放给操作系统。

#### 2. 独立表空间（File-Per-Table Tablespace）

- **开启方式**：`innodb_file_per_table = ON`（MySQL 5.6+ 默认开启）；
- **文件**：每张表对应一个 `.ibd` 文件，存储在数据库目录下；
- **存储内容**：表的数据页、聚簇索引、辅助索引；
- **特点**：
    
    - 删除表后，空间会立即释放给操作系统；
    - 支持单表的备份、恢复、迁移；
    - 推荐生产环境开启。
    

#### 3. 通用表空间（General Tablespace）

- MySQL 5.7+ 引入，可创建共享的表空间，多个表可存放在同一个通用表空间中；
- 适合需要把多个表存放在一起的场景，使用较少。

#### 4. 临时表空间（Temporary Tablespace）

- 存储临时表的数据，默认文件是 `ibtmp1`；
- 数据库重启后会自动重建，无需担心数据残留。

---

### 4.2 redo log（重做日志）：崩溃恢复的核心

#### 核心作用

redo log 是 InnoDB 独有的物理日志，核心作用是**保证事务的持久性**，实现「WAL（Write-Ahead Logging，预写日志）」机制：

- 先写日志，再写数据；
- 数据库宕机后，可通过 redo log 恢复未刷盘的数据，保证数据不丢失。

#### 核心特点

- **物理日志**：记录「数据页的物理修改」，而不是 SQL 逻辑；
- **循环写入**：文件大小固定（默认 48MB，两个文件各 24MB），写满后会覆盖旧的日志；
- **两阶段提交**：和 binlog 配合，保证两份日志的一致性（之前架构笔记讲过）。

#### 核心参数配置

- `innodb_log_file_size`：单个 redo log 文件的大小，生产环境建议设置为 1GB-4GB，避免频繁切换日志文件；
- `innodb_log_files_in_group`：redo log 文件的数量，默认 2 个，一般无需修改；
- `innodb_flush_log_at_trx_commit`：redo log 刷盘策略，核心参数，三个可选值：
    
    - **0**：每秒刷盘一次，事务提交时不刷盘，性能最高，但宕机会丢失 1s 数据；
    - **1**（默认）：事务提交时立即刷盘，性能最低，但数据最安全，不会丢失；
    - **2**：事务提交时刷到操作系统缓存，每秒刷盘一次，性能和安全性折中，宕机会丢失操作系统缓存中的数据。
    

> 生产环境建议：核心交易场景用 1，普通业务场景用 2，测试环境可用 0。

---

### 4.3 undo log（回滚日志）：事务回滚与 MVCC 的核心

#### 核心作用

undo log 是 InnoDB 独有的逻辑日志，核心作用有两个：

1. **事务回滚**：事务执行失败或手动回滚时，通过 undo log 把数据恢复到修改前的状态，保证事务的原子性；
2. **MVCC 多版本并发控制**：通过 undo log 生成数据的历史版本，实现读写不阻塞（后面详细讲 MVCC）。

#### 核心特点

- **逻辑日志**：记录「数据修改的反向操作」，比如 INSERT 的反向是 DELETE，UPDATE 的反向是 UPDATE 旧值；
- **版本链**：同一行数据的多次修改，会生成多个 undo log，通过 roll_pointer 指针串联成版本链；
- **Purge 回收**：事务提交后，undo log 不会立即删除，会被 purge 线程在后台异步回收，避免影响 MVCC。

---

### 4.4 双写缓冲区（Doublewrite Buffer）：防止数据页损坏

#### 核心作用

解决「部分写（Partial Page Write）」问题：

- InnoDB 数据页默认 16KB，磁盘扇区默认 4KB，一次数据页刷盘需要写 4 个扇区；
- 如果刷盘过程中数据库宕机，可能只写了 2 个扇区，导致数据页损坏；
- 双写缓冲区的作用是：先把数据页写入双写缓冲区（连续的磁盘空间，顺序写入，性能高），再写入数据页的真实位置；
- 如果刷盘时宕机，可从双写缓冲区恢复完整的数据页。

#### 核心特点

- 默认开启，可通过 `innodb_doublewrite` 控制；
- 会有一定的性能损耗，但保证了数据页的完整性，生产环境建议开启。

## 五、InnoDB 核心特性详解

### 5.1 事务（Transaction）：ACID 的 InnoDB 实现

#### 回顾 ACID

- **原子性（Atomicity）**：事务是最小执行单元，要么全部成功，要么全部失败回滚；
- **一致性（Consistency）**：事务执行前后，数据从一个一致状态转移到另一个一致状态；
- **隔离性（Isolation）**：多个并发事务之间相互隔离，互不干扰；
- **持久性（Durability）**：事务提交后，数据永久生效，即使宕机也不丢失。

#### InnoDB 对 ACID 的实现

| ACID 特性 | InnoDB 实现方式                                    |
| :------ | :--------------------------------------------- |
| **原子性** | undo log（回滚日志），事务失败时通过 undo log 恢复数据           |
| **持久性** | redo log（重做日志）+ WAL 机制，宕机后通过 redo log 恢复未刷盘的数据 |
| **隔离性** | 锁机制（行级锁、表级锁）+ MVCC 多版本并发控制                     |
| **一致性** | 原子性 + 持久性 + 隔离性共同保证，同时配合数据库的约束（主键、唯一、外键）       |

---

### 5.2 锁机制：解决并发冲突

InnoDB 的锁机制是保证事务隔离性的核心，支持**行级锁**和**表级锁**，其中行级锁是重点。

#### 1. 锁的分类

| 分类维度     | 锁类型                | 说明                                   |
| :------- | :----------------- | :----------------------------------- |
| **按粒度分** | 表级锁                | 锁定整张表，锁粒度大，并发性能差，InnoDB 很少用          |
|          | 行级锁                | 锁定一行或多行数据，锁粒度小，并发性能高，InnoDB 核心       |
| **按功能分** | 共享锁（S 锁）           | 读锁，多个事务可同时加 S 锁，互不阻塞                 |
|          | 排他锁（X 锁）           | 写锁，一个事务加了 X 锁，其他事务不能加 S 锁或 X 锁，会阻塞   |
| **按实现分** | 记录锁（Record Lock）   | 锁定单行记录                               |
|          | 间隙锁（Gap Lock）      | 锁定索引之间的间隙，不包含记录本身                    |
|          | 临键锁（Next-Key Lock） | 记录锁 + 间隙锁，锁定索引间隙 + 记录本身，InnoDB 默认锁模式 |
| **意向锁**  | 意向共享锁（IS）          | 表级锁，事务准备给行加 S 锁前，先给表加 IS 锁           |
|          | 意向排他锁（IX）          | 表级锁，事务准备给行加 X 锁前，先给表加 IX 锁           |

#### 2. 意向锁的作用

意向锁是表级锁，核心作用是 **「让行级锁和表级锁可以共存」**：

- 如果没有意向锁，一个事务给行加了 X 锁，另一个事务想给表加 S 锁，需要遍历所有行检查是否有 X 锁，效率极低；
- 有了意向锁后，事务给行加 X 锁前，先给表加 IX 锁；另一个事务想给表加 S 锁，只需检查表上是否有 IX 锁，有则直接阻塞，无需遍历行。

#### 3. 锁的兼容性矩阵

| |S 锁|X 锁|IS 锁|IX 锁|
|:--|:--|:--|:--|:--|
|**S 锁**|✅ 兼容|❌ 冲突|✅ 兼容|❌ 冲突|
|**X 锁**|❌ 冲突|❌ 冲突|❌ 冲突|❌ 冲突|
|**IS 锁**|✅ 兼容|❌ 冲突|✅ 兼容|✅ 兼容|
|**IX 锁**|❌ 冲突|❌ 冲突|✅ 兼容|✅ 兼容|

#### 4. 临键锁（Next-Key Lock）：解决幻读

临键锁是 InnoDB 默认的行锁模式，是「记录锁 + 间隙锁」的组合，核心作用是**解决幻读问题**（在 RR 隔离级别下）。

- **记录锁**：锁定索引上的单行记录；
- **间隙锁**：锁定索引之间的间隙，防止其他事务在间隙中插入新记录；
- **临键锁**：锁定「左开右闭」的区间，比如索引值为 1、5、10，临键锁的区间是 (-∞,1]、(1,5]、(5,10]、(10,+∞)。

#### 5. 死锁（Deadlock）

##### 核心定义

两个或多个事务互相等待对方释放锁，导致都无法继续执行，形成死锁。

##### 死锁示例

```sql
-- 事务1
BEGIN;
UPDATE user_info SET age = 26 WHERE user_id = 1; -- 加 X 锁
-- 等待事务2释放 user_id=2 的锁
UPDATE user_info SET age = 27 WHERE user_id = 2;

-- 事务2
BEGIN;
UPDATE user_info SET age = 28 WHERE user_id = 2; -- 加 X 锁
-- 等待事务1释放 user_id=1 的锁
UPDATE user_info SET age = 29 WHERE user_id = 1;
```

##### 死锁检测与处理

- InnoDB 有自动死锁检测机制，发现死锁后，会回滚「代价最小」的事务，让另一个事务继续执行；
- 可通过 `SHOW ENGINE INNODB STATUS` 查看最近的死锁日志。

##### 死锁避免

- 尽量让事务以相同的顺序访问数据（比如都按 user_id 从小到大更新）；
- 尽量减小事务的大小，缩短锁的持有时间；
- 尽量使用索引，避免全表扫描（全表扫描会加表级锁）；
- 合理设置隔离级别，RC 隔离级别下间隙锁会关闭，死锁概率更低。

### 5.3 MVCC（多版本并发控制）：读写不阻塞

#### 核心定义

MVCC（Multi-Version Concurrency Control）是 InnoDB 实现「读写不阻塞」的核心机制，通过**数据的历史版本**实现：

- 读操作：读取数据的历史版本（快照读），不需要加锁；
- 写操作：修改数据的最新版本，加行锁；
- 读写之间互不阻塞，大幅提升并发性能。

#### 核心实现原理

MVCC 的实现依赖三个核心组件：**隐式字段**、**undo log 版本链**、**Read View（读视图）**。

##### 1. 隐式字段

InnoDB 会在每行数据中自动添加三个隐式字段：

- `DB_TRX_ID`：最近一次修改该行数据的事务 ID；
- `DB_ROLL_PTR`：回滚指针，指向 undo log 中该行数据的上一个版本；
- `DB_ROW_ID`：隐藏主键，如果表没有主键，InnoDB 会自动生成这个字段作为聚簇索引。

##### 2. undo log 版本链

同一行数据的多次修改，会生成多个 undo log，通过 `DB_ROLL_PTR` 指针串联成一个**版本链**，版本链的头部是最新版本，尾部是最旧版本。

##### 3. Read View（读视图）

Read View 是事务进行快照读时生成的「读视图」，用来判断版本链中的哪个版本对当前事务可见，包含四个核心信息：

- `m_ids`：生成 Read View 时，活跃的（未提交的）事务 ID 列表；
- `min_trx_id`：`m_ids` 中的最小事务 ID；
- `max_trx_id`：生成 Read View 时，系统应该分配给下一个事务的 ID（即最大事务 ID + 1）；
- `creator_trx_id`：生成 Read View 的事务本身的 ID。

##### 4. 可见性判断规则（核心）

对于版本链中的某一个版本，通过以下规则判断是否对当前事务可见：

1. **如果版本的 `DB_TRX_ID` == `creator_trx_id`**：说明是当前事务自己修改的版本，**可见**；
2. **如果版本的 `DB_TRX_ID` < `min_trx_id`**：说明该版本在生成 Read View 之前就已经提交了，**可见**；
3. **如果版本的 `DB_TRX_ID` >= `max_trx_id`**：说明该版本在生成 Read View 之后才开启的事务修改的，**不可见**；
4. **如果 `min_trx_id` <= `DB_TRX_ID` < `max_trx_id`**：判断 `DB_TRX_ID` 是否在 `m_ids` 中：
    
    - 如果在：说明该版本是活跃事务修改的，未提交，**不可见**；
    - 如果不在：说明该版本已提交，**可见**。
    

如果当前版本不可见，就顺着 `DB_ROLL_PTR` 指针找版本链的上一个版本，重复以上规则，直到找到可见的版本。

#### 快照读 vs 当前读

MVCC 中的读操作分为两种：

| 读类型                    | 说明                   | 是否加锁 | 示例                                                                                 |
| :--------------------- | :------------------- | :--- | :--------------------------------------------------------------------------------- |
| **快照读（Snapshot Read）** | 读取数据的历史版本，基于 MVCC 实现 | 不加锁  | 普通的 `SELECT` 语句                                                                    |
| **当前读（Current Read）**  | 读取数据的最新版本            | 加锁   | `SELECT ... FOR UPDATE`、`SELECT ... LOCK IN SHARE MODE`、`INSERT`、`UPDATE`、`DELETE` |

#### 不同隔离级别下的 MVCC 表现

| 隔离级别                     | MVCC 表现                          | 能否解决幻读        |
| :----------------------- | :------------------------------- | :------------ |
| **READ UNCOMMITTED（RU）** | 不使用 MVCC，直接读取最新版本                | 不能            |
| **READ COMMITTED（RC）**   | 每次快照读都生成新的 Read View             | 不能（只能解决不可重复读） |
| **REPEATABLE READ（RR）**  | 事务第一次快照读时生成 Read View，后续快照读复用同一个 | 能（配合临键锁）      |
| **SERIALIZABLE**         | 不使用 MVCC，所有读操作都加锁                | 能             |

### 5.4 索引：聚簇索引与辅助索引

InnoDB 的索引都是 B+ 树结构，分为**聚簇索引**和**辅助索引**两大类，这是 InnoDB 和 MyISAM 索引的核心区别。

#### 1. 聚簇索引（Clustered Index）

##### 核心定义

聚簇索引是 InnoDB 默认的索引类型，**数据和索引存放在一起**，B+ 树的叶子节点存储的是完整的数据行。

##### 聚簇索引的选择规则

- 如果表有主键，InnoDB 会自动把主键作为聚簇索引；
- 如果表没有主键，InnoDB 会选择第一个非空唯一索引作为聚簇索引；
- 如果既没有主键也没有非空唯一索引，InnoDB 会自动生成一个隐藏的 `DB_ROW_ID` 作为聚簇索引。

##### 核心特点

- 数据和索引在一起，通过聚簇索引查询数据，无需回表，性能极高；
- 一张表只能有一个聚簇索引；
- 建议每张表都显式创建主键，且用自增整数型主键（避免聚簇索引的页分裂）。

#### 2. 辅助索引（Secondary Index）

##### 核心定义

辅助索引也叫二级索引，**数据和索引分开存储**，B+ 树的叶子节点存储的是「索引列的值 + 聚簇索引的键值」。

##### 回表操作

通过辅助索引查询数据时，需要两步：

1. 先通过辅助索引找到聚簇索引的键值；
2. 再通过聚簇索引的键值，去聚簇索引中找到完整的数据行；

- 这第二步操作就叫**回表**，会增加一次 IO，性能比聚簇索引查询低。

##### 覆盖索引（Covering Index）

如果辅助索引的叶子节点中，已经包含了查询需要的所有字段，就不需要回表了，这种索引叫**覆盖索引**，性能极高。

- 示例：`SELECT user_id, user_name FROM user_info WHERE user_name = '张三';`
- 如果给 `user_name` 建一个联合索引 `(user_name, user_id)`，查询的字段都在索引中，无需回表，就是覆盖索引。

## 七、常见问题与避坑指南

### 7.1 幻读问题

- **问题**：RR 隔离级别下，同一个事务两次范围查询，结果不一致（第二次多了新插入的行）；
- **解决**：InnoDB RR 隔离级别下，通过「临键锁（Next-Key Lock）」解决幻读，无需手动干预；
- **避坑**：如果用 RC 隔离级别，间隙锁会关闭，无法解决幻读，需根据业务场景选择。

### 7.2 死锁问题

- **问题**：两个事务互相等待对方释放锁，都无法执行；
- **解决**：
    
    - 查看死锁日志：`SHOW ENGINE INNODB STATUS`；
    - 按相同顺序访问数据；
    - 减小事务大小，缩短锁持有时间；
    - 合理使用索引，避免全表扫描。
    

### 7.3 Buffer Pool 命中率低

- **问题**：命中率 < 99%，查询性能差；
- **解决**：
    
    - 增大 `innodb_buffer_pool_size`；
    - 优化查询，避免全表扫描；
    - 合理设计索引，减少回表。
    

### 7.4 索引失效

- **问题**：加了索引，但查询还是走全表扫描；
- **常见原因**：
    
    - 索引列参与了运算或函数；
    - 使用了 `LIKE '%xxx'`（左模糊匹配）；
    - 隐式类型转换（如字符串字段用数字查询）；
    - 优化器选错了索引（可通过 `FORCE INDEX` 强制指定）。
    

### 7.5 数据页碎片

- **问题**：频繁的 INSERT/UPDATE/DELETE 会导致数据页碎片，占用更多磁盘空间，查询性能下降；
- **解决**：
    
    - 定期重建表：`ALTER TABLE 表名 ENGINE=InnoDB;`；
    - 用 `OPTIMIZE TABLE 表名;` 整理碎片（仅独立表空间有效）。

## 八、核心要点总结

1. **InnoDB 架构**：分为内存结构（Buffer Pool、Change Buffer、Log Buffer）和磁盘结构（表空间、redo log、undo log）；
2. **Buffer Pool**：最重要的内存组件，缓存数据页和索引页，用冷热分离的 LRU 算法淘汰，生产环境建议设为服务器内存的 50%-70%；
3. **redo log**：物理日志，循环写入，保证事务持久性，通过两阶段提交和 binlog 配合；
4. **undo log**：逻辑日志，用于事务回滚和 MVCC 版本链；
5. **锁机制**：行级锁（记录锁、间隙锁、临键锁）+ 表级锁（意向锁），临键锁解决幻读；
6. **MVCC**：通过隐式字段、undo log 版本链、Read View 实现读写不阻塞，RR 隔离级别下第一次快照读生成 Read View，后续复用；
7. **索引**：聚簇索引（数据和索引在一起，一张表一个）+ 辅助索引（需回表），覆盖索引可避免回表；
8. **核心优化方向**：合理设置 Buffer Pool、优化索引、调整 redo log 刷盘策略、避免死锁、减少碎片。

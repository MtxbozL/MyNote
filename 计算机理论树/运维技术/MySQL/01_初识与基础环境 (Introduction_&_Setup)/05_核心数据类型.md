>数据类型是表结构设计的核心，直接决定数据存储的安全性、存储空间占用与查询性能，需遵循 "最小够用" 的选型原则。

### 1.5.1 整数类型

整数类型用于存储整数数据，为固定长度数值类型，支持`UNSIGNED`无符号修饰，核心类型如下：

| 类型      | 存储字节 | 有符号取值范围                                    | 无符号 (UNSIGNED) 取值范围      | 适用场景                 |
| ------- | ---- | ------------------------------------------ | ------------------------ | -------------------- |
| TINYINT | 1    | -128 ~ 127                                 | 0 ~ 255                  | 状态码、枚举值、年龄等极小数值      |
| INT     | 4    | -2147483648 ~ 2147483647                   | 0 ~ 4294967295           | 通用整数场景，如用户 ID、订单编号   |
| BIGINT  | 8    | -9223372036854775808 ~ 9223372036854775807 | 0 ~ 18446744073709551615 | 超大数值、分布式 ID、主键（避免溢出） |

**UNSIGNED 属性核心说明**：

1. 取消负数取值范围，正数上限翻倍，仅适用于明确无负数的字段，如库存、点赞数、访问量。
2. 无符号列运算若出现负数，会直接触发报错；与有符号列联查时，可能触发隐式类型转换导致索引失效，需严格统一字段类型。
3. MySQL 8.0 已废弃整数类型的`(M)`显示宽度参数，该参数不影响存储范围，仅用于`ZEROFILL`时的显示补零，无实际业务意义。

### 1.5.2 字符串类型

#### 1.5.2.1 CHAR 与 VARCHAR 的本质区别

CHAR 与 VARCHAR 是最常用的字符串类型，核心差异如下：

| 特性     | CHAR(M)                        | VARCHAR(M)                                       |
| ------ | ------------------------------ | ------------------------------------------------ |
| 存储方式   | 定长字符串，固定占用 M 个字符对应的字节数         | 变长字符串，占用实际字符字节数 + 1/2 字节长度前缀                     |
| 最大长度   | 255 字符                         | 65535 字节（受字符集、行格式限制，utf8mb4 下最大约 16383 字符）       |
| 尾部空格处理 | 存储时自动填充空格至 M 长度，读取时自动去除尾部空格    | 存储与读取均保留尾部空格，符合 SQL 标准                           |
| 性能特性   | 无长度前缀开销，存取速度快，无行碎片风险           | 需处理长度前缀，频繁更新易产生行碎片，存在行溢出风险                       |
| 行溢出处理  | 固定长度，不会触发行溢出                   | 行总长度超出 16KB InnoDB 页大小时，溢出内容存储至溢出页，行内仅保留 20 字节指针 |
| 适用场景   | 固定长度字符串，如手机号、身份证号、MD5 哈希值、性别编码 | 非固定长度字符串，如用户名、地址、商品名称                            |

#### 1.5.2.2 TEXT 类型存储机制

TEXT 类型用于存储超长文本数据，分为 TINYTEXT、TEXT、MEDIUMTEXT、LONGTEXT 四个层级，核心存储特性如下：

1. 存储机制：InnoDB 中，TEXT 列始终触发行溢出，行内仅存储 20 字节的溢出页指针，完整数据完全存储在溢出页中，不会占用行内 16KB 的默认页空间。
2. 核心限制：无法设置默认值；仅能创建前缀索引，无法创建全列索引；排序、分组操作仅能使用前`max_sort_length`字节（默认 1024 字节），无法全字段排序。
3. 适用场景：超长文本内容，如文章详情、富文本内容、日志信息等；不适合用于高频查询、排序、联查的字段，避免性能损耗。

### 1.5.3 日期时间类型

日期时间类型的核心选型重点为`DATETIME`与`TIMESTAMP`，二者的底层存储与跨时区特性存在本质差异。

#### 1.5.3.1 基础日期时间类型

|类型|存储字节|格式|取值范围|核心特性|
|---|---|---|---|---|
|DATE|3|YYYY-MM-DD|1000-01-01 ~ 9999-12-31|仅存储日期，无时间部分|
|TIME|3|HH:MM:SS|-838:59:59 ~ 838:59:59|仅存储时间，支持时间间隔存储|
|YEAR|1|YYYY|1901 ~ 2155|仅存储年份，存储空间最小|

#### 1.5.3.2 DATETIME 与 TIMESTAMP 核心差异

|特性|DATETIME|TIMESTAMP|
|---|---|---|
|底层存储|存储日期时间字面量，与时区完全无关|存储 Unix 时间戳（1970-01-01 UTC 起的秒数），与时区强相关|
|存储空间|5 字节（MySQL 5.6.4+，不含微秒）|4 字节（不含微秒）|
|取值范围|1000-01-01 00:00:00 ~ 9999-12-31 23:59:59|1970-01-01 00:00:01 UTC ~ 2038-01-19 03:14:07 UTC|
|时区处理|写入与读取完全一致，数据库 / 会话时区修改不改变存储值|写入时转换为 UTC 时间戳存储，读取时转换为当前会话时区的时间显示，时区修改会改变读取结果|
|2038 问题|无|存在，超出取值范围无法存储|
|自动更新|支持 DEFAULT CURRENT_TIMESTAMP 与 ON UPDATE CURRENT_TIMESTAMP|默认第一个 TIMESTAMP 列隐式开启自动更新，8.0 默认关闭该隐式规则|
|适用场景|固定日期时间、跨时区业务需统一时间、生日、合同签订时间等全场景|数据插入 / 更新时间记录、跨时区业务需适配用户本地时间、Unix 时间戳原生适配场景|

### 1.5.4 JSON 类型

MySQL 5.7 版本首次引入原生 JSON 类型，8.0 版本进一步完善了函数与索引能力，实现对半结构化数据的原生支持，符合 RFC 7159 JSON 标准。

1. **核心特性**
    
    - 格式自动校验：写入时自动校验 JSON 文档格式，格式错误直接报错，区别于 TEXT 类型存储的 JSON 字符串。
    - 优化二进制存储：底层以二进制格式存储，而非纯文本，读取时无需解析全文档，可通过键 / 数组索引直接定位目标元素，性能远超 TEXT 存储的 JSON 字符串。
    - 完整函数支持：提供全场景 JSON 操作函数，包括取值（`JSON_EXTRACT()`、`->`、`->>`）、修改（`JSON_SET()`、`JSON_REMOVE()`）、查询（`JSON_CONTAINS()`）、聚合（`JSON_ARRAYAGG()`）等。
    - 索引能力：无法直接对 JSON 列创建索引，可通过**生成列（Generated Column）** 对 JSON 文档内的指定字段创建二级索引，实现 JSON 数据的高性能查询。
    
2. **存储限制**：最大存储容量与 LONGTEXT 一致，最大 4GB，单文档行内存储上限 16MB，超出部分触发溢出页存储。
3. **适用场景**：半结构化数据、动态扩展属性（如商品自定义规格、用户扩展信息）、无需频繁修改的扩展数据；不适合作为核心查询字段、高频更新字段，避免过度使用破坏关系型数据库的范式设计。

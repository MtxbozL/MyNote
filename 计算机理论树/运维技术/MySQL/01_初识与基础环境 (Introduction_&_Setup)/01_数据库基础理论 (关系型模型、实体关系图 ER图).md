## 1.1 数据库基础理论

### 1.1.1 关系型数据模型

关系型模型由 E.F.Codd 于 1970 年在《大型共享数据库的关系模型》中首次提出，是关系型数据库的核心理论基础，其核心定义与约束如下：

1. **核心结构要素**
    
    - 关系：对应数据库中的表，是二维表结构的元组集合，需满足原子性、列同质性、行唯一性、列无序性、行无序性、无重复元组六大特性。
    - 元组：对应表中的行，代表一个实体的完整实例数据。
    - 属性：对应表中的列，代表实体的一个特征，每个属性对应唯一的域（取值范围）。
    - 键：包括超键、候选键、主键（唯一标识元组的非空候选键）、外键（关联两个关系的参照属性）。
    
2. **关系代数核心操作**
    
    是 SQL 语言的理论底层，包括选择（σ）、投影（π）、笛卡尔积（×）、连接（⋈）、并（∪）、差（-）、交（∩）、除（÷）八大基本操作。
3. **三大完整性约束**
    
    - 实体完整性：主键非空且唯一，确保元组可唯一标识。
    - 参照完整性：外键的取值必须匹配被参照关系的主键值，或为 NULL，确保关系间的关联一致性。
    - 用户定义完整性：由业务场景定义的自定义约束，如非空、唯一、CHECK、枚举等。
    

### 1.1.2 实体关系图（ER 图）

ER 图是数据库概念设计的核心工具，用于可视化实体、属性与实体间的关联关系，是逻辑表结构设计的前置依据。

1. **核心组成元素**
    
    - 实体：独立存在的业务对象，用矩形表示，对应数据库中的表。
    - 属性：实体的特征，用椭圆形表示，对应表中的列；其中唯一标识实体的属性为码（主键），用下划线标注。
    - 联系：实体间的关联关系，用菱形表示，对应表间的外键关联或中间表。
    
2. **联系的三种类型**
    
    - 一对一（1:1）：A 实体的一个实例仅对应 B 实体的一个实例，反之亦然，如用户与用户实名认证信息。
    - 一对多（1:N）：A 实体的一个实例对应 B 实体的多个实例，B 实体的一个实例仅对应 A 实体的一个实例，如部门与员工，是最常见的关联类型。
    - 多对多（M:N）：A 实体的一个实例对应 B 实体的多个实例，反之亦然，必须通过中间关联表拆解为两个一对多关系实现，如学生与课程、商品与标签。
    
3. **ER 图映射规则**
    
    概念设计完成后，需将 ER 图转换为关系模式：实体直接映射为表，属性映射为列，1:1 关联可合并表或外键唯一约束，1:N 关联在多端表添加外键，M:N 关联新增中间表存储两端主键。

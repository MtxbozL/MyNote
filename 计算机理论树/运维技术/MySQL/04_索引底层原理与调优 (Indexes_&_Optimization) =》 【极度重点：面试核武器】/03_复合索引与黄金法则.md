
>复合索引（联合索引）是指在多个列上共同创建的索引，其 B+Tree 按照索引列的定义顺序排序：先按第一列排序，第一列值相同时再按第二列排序，以此类推。复合索引是 SQL 优化的核心载体，本节讲解复合索引的三大黄金优化法则。

### 4.3.1 最左前缀法则（Leftmost Prefix Rule）

最左前缀法则是复合索引的核心生效规则，MySQL 查询时仅能匹配复合索引的**最左连续 N 个列**，或字符串列的最左 M 个字符，只有满足最左前缀规则，才能利用索引的有序性实现快速过滤，否则会导致索引失效。

#### 核心执行规则

1. **必须从索引的第一列开始匹配，跳过第一列则索引完全失效**
    
    示例：复合索引`idx_a_b_c(a,b,c)`，查询条件`WHERE b=1 AND c=2`，跳过了第一列`a`，无法利用该索引，最终全表扫描。
2. **中间列中断，后续列无法使用索引**
    
    遇到范围查询（`>`、`<`、`BETWEEN`、`LIKE 'xx%'`）时，范围列后的所有列无法利用索引，仅范围列前的连续列可使用索引。
    
    示例：复合索引`idx_a_b_c(a,b,c)`，查询条件`WHERE a=1 AND b>2 AND c=3`，仅`a`和`b`列可使用索引，`c`列无法使用，因为`b`的范围查询中断了最左前缀的连续性。
3. **等值查询列顺序无要求，优化器自动适配**
    
    对于等值查询条件，MySQL 优化器会自动调整条件的顺序，匹配复合索引的最左前缀，无需严格按照索引定义顺序书写。
    
    示例：复合索引`idx_a_b_c(a,b,c)`，查询条件`WHERE b=2 AND a=1 AND c=3`，优化器会自动调整为`WHERE a=1 AND b=2 AND c=3`，可使用完整的复合索引。
4. **字符串前缀匹配生效，后缀匹配失效**
    
    字符串列的`LIKE 'xx%'`右模糊匹配满足最左前缀规则，可使用索引；`LIKE '%xx'`左模糊、`LIKE '%xx%'`双模糊匹配破坏最左前缀，无法使用索引。

#### 底层原理

复合索引的 B+Tree 仅按索引列的定义顺序排序，只有前序列的值确定后，后续列的值才是有序的；跳过前序列或中断连续匹配，后续列的值处于无序状态，无法利用索引的有序性进行快速查找，只能逐行扫描过滤。

### 4.3.2 覆盖索引（Covering Index）

覆盖索引是指查询的**所有字段都完全包含在复合索引中**，无需通过回表操作访问聚簇索引，是 SQL 优化的终极手段之一，可彻底消除回表带来的性能损耗。

- 生效标识：EXPLAIN 执行计划的`Extra`列会显示`Using index`，表示查询使用了覆盖索引。
- 核心性能优势：
    
    1. 彻底消除回表操作，减少磁盘 IO 次数，查询性能呈数量级提升；
    2. 二级索引的叶子节点仅存储索引列与主键，数据体积远小于聚簇索引，相同内存空间可缓存更多索引页，缓存命中率大幅提升；
    3. 无需访问聚簇索引，减少了高并发场景下的锁竞争，提升系统吞吐量。
    
- 典型示例：
    
    1. 复合索引`idx_a_b_c(a,b,c)`，执行`SELECT a,b,c FROM t WHERE a=1 AND b=2`，所有查询字段均在索引中，无需回表，触发覆盖索引；
    2. 复合索引`idx_name_age(name,age)`，执行`SELECT id,name,age FROM user WHERE name LIKE '张%'`，主键`id`天然存储在二级索引叶子节点中，所有字段均在索引中，触发覆盖索引。
    
- 核心设计规范：生产环境禁止使用`SELECT *`查询，仅查询业务所需的字段，大幅提升覆盖索引的适配概率。

### 4.3.3 索引下推（ICP - Index Condition Pushdown）

索引下推是 MySQL 5.6 版本引入的核心优化特性，核心逻辑是**在存储引擎层遍历索引时，提前对索引中包含的列进行条件过滤**，而非先将所有满足最左前缀的行回表，再在服务层过滤，大幅减少回表次数与磁盘 IO 开销。

#### 核心工作原理

- 无 ICP 的执行流程：存储引擎通过索引找到满足最左前缀的所有行，全部回表获取完整行数据，返回给服务层，服务层再对剩余条件进行过滤；
- 有 ICP 的执行流程：存储引擎在遍历索引的过程中，直接对索引中包含的列的条件进行过滤，仅将满足所有索引条件的行回表，大幅减少无效回表的次数。

#### 典型示例

用户表`user`创建复合索引`idx_name_age(name,age)`，执行 SQL：`SELECT * FROM user WHERE name LIKE '张%' AND age=18`。

- 无 ICP：先找到所有`name`以 "张" 开头的索引记录，全部回表获取完整数据，服务层再过滤`age=18`的行，回表次数等于所有 "张" 开头的记录数；
- 有 ICP：在索引遍历过程中，直接过滤`age=18`的记录，仅将同时满足两个条件的行回表，回表次数大幅减少。

#### 生效规则与配置

1. 仅适用于二级索引，无法用于聚簇索引；
2. 过滤条件的列必须包含在当前使用的索引中；
3. 生效标识：EXPLAIN 执行计划的`Extra`列显示`Using index condition`；
4. MySQL 默认开启，配置参数：`optimizer_switch='index_condition_pushdown=on'`。

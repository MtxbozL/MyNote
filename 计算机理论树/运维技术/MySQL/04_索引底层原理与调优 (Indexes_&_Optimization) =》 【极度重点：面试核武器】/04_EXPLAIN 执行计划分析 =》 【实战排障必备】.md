
>EXPLAIN 是 MySQL 提供的 SQL 执行计划可视化工具，可模拟优化器执行 SQL 语句的完整流程，精准判断 SQL 是否使用索引、索引使用是否合理、是否存在全表扫描、文件排序、临时表等性能问题，是 SQL 调优的核心实战工具。
> 
>	基础用法：在待分析的 SQL 语句前添加`EXPLAIN`关键字，执行后返回执行计划结果集，示例：`EXPLAIN SELECT * FROM user WHERE id = 1;`。

### 4.4.1 type（连接类型 / 访问类型）

`type`字段表示 MySQL 在表中查找目标行的访问方式，是判断 SQL 性能的核心指标，**性能从差到优的完整排序为：ALL < index < range < ref < eq_ref < const < system**。

|访问类型|核心含义|场景与优化要求|
|---|---|---|
|ALL|全表扫描（Full Table Scan），MySQL 遍历整张表的所有数据页匹配目标行|性能最差，生产环境必须杜绝；仅出现在无可用索引、或优化器认为全表扫描成本低于索引扫描的场景|
|index|全索引扫描（Full Index Scan），MySQL 遍历整个索引树匹配目标行|性能仅略优于 ALL，需优化；仅出现在查询字段可被索引覆盖，但无过滤条件需扫描全索引的场景|
|range|索引范围扫描，MySQL 仅扫描索引中指定范围的行|性能良好，是生产环境可接受的最低标准；触发场景：`>`、`<`、`>=`、`<=`、`BETWEEN`、`IN`、`LIKE 'xx%'`等范围查询|
|ref|非唯一索引等值查询，使用非唯一二级索引进行等值匹配，可能返回多行符合条件的记录|性能优秀，是生产环境的理想优化目标；适用于普通二级索引的等值查询场景|
|eq_ref|唯一索引等值查询，使用主键或非空唯一索引进行等值匹配，最多返回 1 行记录|性能极佳，常见于主键或唯一索引的多表联查场景|
|const/system|常量查询，MySQL 优化器将查询优化为常量，仅执行 1 次匹配|性能最优；const 适用于主键 / 唯一索引的等值查询，system 是 const 的特例，仅适用于只有 1 行数据的系统表|

### 4.4.2 key / possible_keys

- `possible_keys`：MySQL 优化器根据查询条件，筛选出的**可能适用的候选索引列表**，仅表示索引可用于条件过滤，不代表最终会被使用。
- `key`：MySQL 优化器最终**实际使用的索引**，是判断索引是否生效的核心标识。若`key`为 NULL，表示查询未使用任何索引。

#### 核心分析要点

1. `possible_keys`非空，但`key`为 NULL：表示优化器认为使用索引的成本高于全表扫描，放弃了索引，通常出现于小表、查询返回表中大部分行、索引选择性极低的场景；
2. `possible_keys`为 NULL：表示无可用的候选索引，必须创建适配查询条件的索引；
3. `key`的长度小于索引定义的总长度：表示仅使用了复合索引的部分列，需结合最左前缀法则分析索引的使用效率。

### 4.4.3 Extra（额外执行信息）

`Extra`字段包含 MySQL 执行 SQL 的核心细节信息，是判断 SQL 是否存在隐性性能损耗的核心指标，重点关注以下三大高频关键项。

|Extra 标识|核心含义|优化方向|
|---|---|---|
|Using index|使用了覆盖索引，查询的所有字段均包含在索引中，无需回表访问聚簇索引|最优状态，是 SQL 优化的核心目标，应尽可能让查询触发覆盖索引|
|Using filesort|文件排序，MySQL 无法利用索引的有序性完成排序，需要在内存或磁盘中对结果集进行额外排序|性能损耗极大，必须优化；核心优化方案是让排序字段符合复合索引的最左前缀法则，利用索引的天然有序性避免排序|
|Using temporary|使用临时表，MySQL 需要创建临时表存储查询的中间结果，常见于`GROUP BY`、`DISTINCT`、`UNION`等操作|性能损耗极大，必须优化；核心优化方案是让分组 / 去重字段符合复合索引的最左前缀法则，利用索引的有序性避免临时表创建|

其他高频标识说明：

- `Using index condition`：使用了索引下推（ICP）优化，减少了回表次数，属于正向优化；
- `Using where`：存储引擎返回的行需要在服务层进行条件过滤，通常表示过滤条件的列未包含在索引中；
- `Impossible WHERE`：WHERE 条件恒为假，无匹配行，无需访问表。

### 4.4.4 其他核心辅助字段

1. `id`：SQL 语句的执行顺序标识，id 值越大越先执行，id 值相同则从上到下依次执行；
2. `rows`：MySQL 优化器预估的、为找到目标行需要扫描的行数，数值越小越好，是判断索引效率的核心辅助指标；
3. `filtered`：返回结果的行数占扫描行数的比例，取值 0-100，数值越高表示索引过滤效率越好，100% 表示扫描的所有行均符合条件。

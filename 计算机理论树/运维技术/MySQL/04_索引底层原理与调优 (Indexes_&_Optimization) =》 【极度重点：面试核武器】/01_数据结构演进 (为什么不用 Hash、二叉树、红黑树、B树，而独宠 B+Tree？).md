
>索引是关系型数据库查询性能的核心基石，其本质是 **帮助 MySQL 高效过滤、排序数据的有序数据结构** ，核心设计目标是将随机磁盘 IO 转为有序的索引扫描，最小化磁盘 IO 次数，彻底解决全表扫描的性能瓶颈。本章深入拆解 MySQL 索引的底层数据结构、存储逻辑、优化法则、排障方法与失效场景，是 SQL 性能调优、MySQL 面试的核心必备知识。

## 4.1 索引数据结构演进

数据库索引的核心痛点在于：数据持久化存储在磁盘，随机磁盘 IO 的性能比内存 IO 低 5 个数量级以上，因此索引设计的核心准则是 **最小化磁盘 IO 次数，降低树的高度，最大化单次 IO 的数据收益** 。本节从数据结构的演进逻辑，拆解 MySQL 为何最终选择 B+Tree 作为索引的底层存储结构。

### 4.1.1 哈希表

哈希表是基于哈希函数实现的键值对存储结构，通过哈希函数将 key 映射到数组的下标，实现 O (1) 时间复杂度的等值查询。

- 核心优势：等值查询性能极致，时间复杂度 O (1)，适用于纯等值匹配场景。
- 致命缺陷，无法适配数据库索引需求：
    
    1. 不支持范围查询、排序、分组、前缀模糊匹配，此类需求只能通过全表扫描实现；
    2. 存在哈希冲突问题，大量冲突会导致查询性能退化为 O (n)；
    3. 无法利用磁盘预读的局部性原理，数据离散存储，无法通过单次 IO 加载批量连续数据；
    4. 无序存储，无法利用索引的有序性优化查询。
    
- MySQL 适配场景：仅在内存中的自适应哈希索引（AHI）中使用，磁盘持久化的索引不采用哈希表结构。

### 4.1.2 二叉搜索树（BST）与平衡二叉树

1. **二叉搜索树（BST）**
    
    核心规则：左子树所有节点值小于根节点，右子树所有节点值大于根节点，中序遍历可得到有序序列。
    
    致命缺陷：极端情况下会退化为单向链表（如顺序插入递增数据），树高等于数据行数，100 万行数据需要 100 万次磁盘 IO，完全无法满足数据库性能需求。
    
2. **平衡二叉树（AVL 树）与红黑树**
    
    AVL 树通过旋转操作保证严格平衡，左右子树高度差不超过 1；红黑树通过染色与旋转实现弱平衡，最长路径不超过最短路径的 2 倍，解决了二叉搜索树的退化问题。
    
    核心缺陷：均为二叉结构，每个节点仅能存储 1 个 key 与对应指针，树高无法有效降低。100 万行数据的红黑树高度约为 20，意味着单次查询最多需要 20 次随机磁盘 IO，性能开销远超数据库可接受范围；同时，二叉节点无法利用磁盘预读机制（磁盘单次 IO 读取一个 4KB/16KB 的页，二叉节点仅存储 1 个 key，造成页空间的极大浪费）。
    

### 4.1.3 B 树（平衡多路查找树）

B 树是为磁盘存储设计的平衡多路查找树，每个节点可存储多个 key、数据指针与子节点指针，子节点数量 = key 数量 + 1，所有节点均存储完整的行数据，整棵树有序平衡。

- 核心优势：多路结构大幅降低树高，16KB 页大小下，单个节点可存储上百个 key，100 万行数据的树高仅需 3 层，单次查询最多 3 次磁盘 IO，性能远超平衡二叉树。
- 无法适配 MySQL 索引的核心缺陷：
    
    1. 非叶子节点存储完整行数据，导致单个节点可存储的 key 数量大幅减少，树高升高，IO 次数增加；
    2. 数据分散在所有节点，范围查询需要中序遍历整棵树，多次回溯产生大量随机 IO；
    3. 查询性能不稳定，根节点命中仅需 1 次 IO，叶子节点命中需要多次 IO；
    4. 节点的分裂、合并成本高，频繁更新易导致树结构频繁调整。
    

### 4.1.4 B+Tree（增强型平衡多路查找树）

B+Tree 是 B 树的优化版本，也是 InnoDB 存储引擎索引的唯一底层数据结构，针对数据库场景做了极致优化，核心设计与 B 树的本质差异如下：

1. **非叶子节点仅存 key 与指针，不存完整数据**：所有完整行数据仅存储在叶子节点，非叶子节点仅存储索引 key 与子节点指针。该设计让单个非叶子节点可存储的 key 数量呈数量级提升，树高大幅降低。
    
    量化计算（InnoDB 默认页大小 16KB）：
    
    - 主键采用 BIGINT 类型（8 字节），子节点指针（6 字节），单个非叶子节点可存储 `16*1024/(8+6) ≈ 1170` 个 key；
    - 叶子节点存储完整行数据，假设单行数据 1KB，单个叶子节点可存储 16 行数据；
    - 3 层 B+Tree 可存储的最大数据量为 `1170 * 1170 * 16 ≈ 2190万行`，意味着千万级数据量的表，主键查询最多仅需 3 次磁盘 IO，根节点常驻内存后仅需 1-2 次 IO。
    
2. **叶子节点通过双向链表有序串联**：所有叶子节点按索引 key 的升序排列，相邻节点通过双向链表连接，范围查询仅需找到起始节点，即可顺着链表顺序遍历，无需回溯树结构，彻底解决了 B 树范围查询的随机 IO 问题，完美适配排序、分组、范围查询等 SQL 核心需求。
3. **查询性能稳定**：所有查询必须遍历到叶子节点才能获取数据，每次查询的 IO 次数固定，性能稳定可预测，适配数据库的执行计划成本估算。
4. **更新维护成本更低**：非叶子节点不存储数据，行数据的更新仅需修改叶子节点，节点分裂、合并的概率大幅降低，高并发更新场景下性能更稳定。

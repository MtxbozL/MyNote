
>索引失效的核心本质是：查询条件破坏了索引的有序性，MySQL 优化器无法利用索引的 B+Tree 结构进行快速查找，最终选择全表扫描。本节拆解生产环境中高频的索引失效场景、底层原理与规避方案。

### 4.5.1 左模糊 / 双模糊匹配

- 失效场景：对索引列使用`LIKE '%xx'`左模糊、`LIKE '%xx%'`双模糊匹配；
- 失效原理：索引按列值的正序排列，左模糊破坏了最左前缀规则，无法利用索引的有序性进行快速定位，只能全表扫描；
- 优化方案：
    
    1. 业务上优先使用右模糊`LIKE 'xx%'`匹配，满足最左前缀规则；
    2. 必须使用左模糊时，通过覆盖索引实现全索引扫描（`index`类型），性能优于全表扫描；
    3. 海量文本数据场景，使用 MySQL 全文索引，或引入 Elasticsearch 搜索引擎实现模糊查询。
    

### 4.5.2 对索引列使用函数 / 表达式计算

- 失效场景：对索引列执行函数调用、算术运算、类型转换等操作，示例：
    
    ```sql
    -- 函数调用失效
    SELECT * FROM user WHERE YEAR(create_time) = 2024;
    -- 算术运算失效
    SELECT * FROM order WHERE id + 1 = 100;
    -- 字符串处理失效
    SELECT * FROM user WHERE UPPER(name) = 'ZHANGSAN';
    ```
    
- 失效原理：索引存储的是索引列的原始值，而非计算后的值，MySQL 无法对计算结果使用索引的有序性，只能逐行计算后过滤，导致索引失效；
- 优化方案：将计算逻辑转移到常量端，保证索引列不被任何函数 / 表达式包裹，示例：
    
    ```sql
    SELECT * FROM user WHERE create_time BETWEEN '2024-01-01' AND '2024-12-31';
    SELECT * FROM order WHERE id = 99;
    ```
    

### 4.5.3 隐式类型转换

- 失效场景：索引列的类型与查询条件的值类型不匹配，MySQL 自动对索引列执行隐式类型转换；
- 高频失效案例：索引列为字符串类型（`VARCHAR`），查询条件使用数值类型，示例：
    
    ```sql
    -- phone列为VARCHAR类型，创建了索引，查询条件使用数值，索引失效
    SELECT * FROM user WHERE phone = 13800138000;
    ```
    
- 失效原理：MySQL 对索引列执行隐式类型转换，相当于对索引列使用了函数，破坏了索引的有序性，导致索引失效；
- 特殊说明：索引列为数值类型，查询条件使用字符串类型时，MySQL 会对字符串常量进行转换，不会修改索引列的值，索引可正常生效；
- 优化方案：严格保证查询条件的值类型与索引列的定义类型完全一致，杜绝隐式类型转换。

### 4.5.4 OR 连接条件包含无索引列

- 失效场景：`OR`连接的多个条件中，有一个条件的列没有索引，示例：
    
    ```sql
    -- id有主键索引，name无索引，整个查询索引失效，全表扫描
    SELECT * FROM user WHERE id = 1 OR name = '张三';
    ```
    
- 失效原理：`OR`的语义是满足任意一个条件即可，无索引的列必须通过全表扫描匹配，MySQL 为避免索引扫描 + 全表扫描的双重开销，直接选择全表扫描；
- 优化方案：
    
    1. 为`OR`连接的所有列创建适配的索引；
    2. 将`OR`查询拆分为两个`UNION ALL`查询，分别利用各自的索引，示例：
        
        ```sql
        SELECT * FROM user WHERE id = 1
        UNION ALL
        SELECT * FROM user WHERE name = '张三';
        ```

### 4.5.5 违反最左前缀法则

- 失效场景：复合索引查询跳过最左列、中间列范围查询中断连续匹配，详细规则见 4.3.1 节；
- 优化方案：调整复合索引的列顺序，将等值查询列、高频过滤列放在索引前端，范围查询列放在后端；拆分范围查询，保证索引的连续匹配。

### 4.5.6 负向查询导致索引失效

- 失效场景：对索引列使用`NOT`、`!=`、`<>`、`NOT IN`、`IS NOT NULL`等负向查询条件；
- 失效原理：负向查询无法利用索引的有序性进行范围匹配，MySQL 优化器通常会选择全表扫描；
- 优化方案：业务上优先使用正向范围查询替代负向查询；必须使用负向查询时，通过覆盖索引实现全索引扫描，避免全表扫描。

### 4.5.7 优化器选错索引

- 失效场景：表存在多个可用索引，但 MySQL 优化器选择了非最优索引，甚至放弃索引选择全表扫描；
- 核心原因：表的统计信息不准确（数据大量增删改后未更新）、多个索引的执行成本估算偏差、数据分布不均匀导致索引选择性判断错误；
- 优化方案：
    
    1. 执行`ANALYZE TABLE 表名;`更新表的统计信息，修正优化器的成本估算；
    2. 谨慎使用`FORCE INDEX`强制指定索引，避免数据分布变化后强制索引反而导致性能下降。

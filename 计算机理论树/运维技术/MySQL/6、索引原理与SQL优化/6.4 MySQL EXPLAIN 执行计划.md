## 一、开篇：EXPLAIN 核心定位与价值

### 1. 什么是 EXPLAIN

EXPLAIN 是 MySQL 内置的**执行计划分析工具**，在 SQL 语句前加上 `EXPLAIN` 关键字，MySQL 不会实际执行该 SQL，而是会模拟优化器的 SQL 执行流程，返回这条 SQL 的**详细执行计划**，让我们清晰看到 MySQL 是如何处理这条 SQL 的。

它是 SQL 优化、索引验证、慢查询排查的**唯一核心工具**，也是我们之前学习的索引、事务、锁等知识的落地载体。

### 2. 学习 EXPLAIN 的核心价值

|核心价值|具体说明|
|:--|:--|
|**验证索引是否生效**|精准判断索引是否被使用、联合索引用到了哪些列、是否出现索引失效|
|**定位慢查询根因**|快速定位 SQL 慢的核心原因：全表扫描、文件排序、临时表、回表次数过多等|
|**优化多表关联查询**|查看表的连接顺序、驱动表选择是否合理，优化 JOIN 性能|
|**校验优化器选择**|查看 MySQL 优化器是否选择了最优的索引和执行计划，必要时可强制指定索引|
|**面试刚需**|后端 / 运维 / DBA 岗位面试必考，是 SQL 优化能力的核心体现|

### 3. 基础语法与扩展用法

#### 基础语法

```sql
-- 核心用法：在SQL前加EXPLAIN
EXPLAIN SELECT * FROM user WHERE name = '张三';

-- 支持的SQL类型：SELECT/INSERT/UPDATE/DELETE/REPLACE
EXPLAIN UPDATE user SET age = 26 WHERE id = 1;
```

#### 扩展用法

```sql
-- 1. JSON格式输出（MySQL 5.7+），展示更详细的执行成本、执行耗时
EXPLAIN FORMAT=JSON SELECT * FROM user WHERE name = '张三';

-- 2. EXPLAIN ANALYZE（MySQL 8.0.18+）：实际执行SQL，返回真实执行耗时、扫描行数、循环次数
EXPLAIN ANALYZE SELECT * FROM user WHERE name = '张三';

-- 3. 查看分区表的执行计划
EXPLAIN PARTITIONS SELECT * FROM order_partition WHERE create_time BETWEEN '2026-01-01' AND '2026-03-01';
```

## 二、EXPLAIN 输出字段总览

MySQL 8.0 版本中，EXPLAIN 执行计划固定输出 **12 个核心字段**，每个字段对应 SQL 执行的一个核心环节，我们按「执行顺序→访问类型→索引使用→性能评估→额外信息」的逻辑拆解。

|字段名|核心含义|重点级别|
|:--|:--|:--|
|id|SQL 查询的执行顺序，标识查询的执行优先级|⭐⭐⭐⭐⭐|
|select_type|查询的类型（普通查询、子查询、联合查询等）|⭐⭐⭐⭐|
|table|当前行对应的表名 / 表别名 / 临时表名|⭐⭐⭐|
|partitions|匹配的分区，非分区表为 NULL|⭐⭐|
|type|数据访问类型，判断 SQL 性能的核心指标|⭐⭐⭐⭐⭐|
|possible_keys|可能用到的索引（优化器候选索引）|⭐⭐⭐|
|key|实际用到的索引，判断索引是否生效的核心|⭐⭐⭐⭐⭐|
|key_len|索引使用的字节长度，判断联合索引用到了哪些列|⭐⭐⭐⭐⭐|
|ref|与索引匹配的列 / 常量|⭐⭐⭐|
|rows|MySQL 预估需要扫描的行数，越少性能越好|⭐⭐⭐⭐⭐|
|filtered|符合查询条件的行占比，0-100，越高越好|⭐⭐⭐|
|Extra|额外执行信息，核心优化提示，覆盖索引 / 文件排序等|⭐⭐⭐⭐⭐|

> 核心重点：**id、type、key、key_len、rows、Extra** 这 6 个字段，是执行计划分析的核心，必须 100% 掌握。

## 三、核心字段详解（从执行顺序到性能评估）

### 3.1 id：SQL 执行顺序标识

id 是查询的序列号，代表 SQL 语句中查询块的执行优先级，**id 值越大，执行优先级越高；id 值相同，从上到下依次执行**。

#### 3 种核心场景

##### 场景 1：id 相同，普通单表 / 多表查询

```sql
EXPLAIN SELECT u.name, o.order_no 
FROM user u
LEFT JOIN order_info o ON u.user_id = o.user_id
WHERE u.user_id = 1;
```

- 执行计划中两条记录的 id 均为 1，执行顺序：从上到下，先查 u 表，再查 o 表。

##### 场景 2：id 不同，包含子查询

```sql
EXPLAIN SELECT * FROM user 
WHERE user_id IN (SELECT user_id FROM order_info WHERE order_amount > 1000);
```

- 子查询的 id 更大，优先执行子查询（order_info 表），再执行外层主查询（user 表）。

##### 场景 3：id 有相同有不同，混合查询

- 先按 id 从大到小执行，id 相同的部分从上到下依次执行；
- id 为 NULL 的行，最后执行（通常是 UNION 结果合并）。

#### 特殊说明

- `DERIVED`（衍生表）：id 对应衍生表的查询，优先执行；
- `UNION RESULT`：id 为 NULL，是 UNION 结果的合并操作，最后执行。

---

### 3.2 select_type：查询类型

select_type 标识当前行对应的查询块的类型，核心作用是区分普通查询、子查询、联合查询、衍生查询等，快速定位需要优化的查询类型。

|select_type 取值|核心含义|优化提示|
|:--|:--|:--|
|**SIMPLE**|简单查询，不包含子查询、UNION、衍生表|最优的查询类型，无需优化|
|**PRIMARY**|主查询，包含子查询的 SQL 中，最外层的查询|-|
|**SUBQUERY**|子查询（WHERE/SELECT 中的子查询，非 FROM 子句）|子查询可能会被优化器改写，注意性能|
|**DERIVED**|衍生表，FROM 子句中的子查询，会生成临时表|高频优化点，衍生表会创建临时表，性能差，尽量避免|
|**UNION**|UNION 语句中，第二个及之后的查询|-|
|**UNION RESULT**|UNION 结果的合并操作|-|
|**DEPENDENT SUBQUERY**|依赖子查询，子查询的结果依赖外层主查询|严重性能问题，子查询会循环执行，扫描行数暴增，必须优化|
|**DEPENDENT UNION**|依赖的 UNION 查询，UNION 中的子查询依赖外层主查询|性能差，必须优化|

> 避坑重点：`DEPENDENT SUBQUERY` 和 `DERIVED` 是高频性能杀手，看到这两个类型，必须优先优化。

---

### 3.3 type：数据访问类型（核心性能指标）

type 是执行计划中**最重要的性能指标**，代表 MySQL 找到目标数据行的访问方式，直接决定了 SQL 的查询效率。

**性能从最优到最差排序**：

> system > const > eq_ref > ref > range > index > ALL

我们重点讲解生产环境最常见的 7 个类型，越靠前性能越好，优化目标是至少达到 `range` 级别，最优是 `ref` 及以上。

|type 取值|核心含义|触发场景|优化级别|
|:--|:--|:--|:--|
|**system**|系统表，表中只有一行数据，是 const 的特例|仅查询 MySQL 系统表，且只有一行数据|极致性能，几乎遇不到|
|**const**|常量查询，通过主键 / 唯一索引的等值查询，最多匹配一行数据|主键 / 唯一索引的等值查询，如 `WHERE id = 1`|最优性能，单值查询的终极目标|
|**eq_ref**|等值关联查询，关联字段是主键 / 唯一非空索引，每次关联只匹配一行数据|多表 JOIN，关联字段是主键 / 唯一索引，如 `user u JOIN order o ON u.id = o.user_id`，o.user_id 是唯一索引|极优性能，JOIN 查询的终极目标|
|**ref**|非唯一索引的等值查询，可能匹配多行数据|普通索引的等值查询，如 `WHERE name = '张三'`、联合索引的前缀等值匹配|优秀性能，普通查询的核心优化目标|
|**range**|索引范围查询，只扫描索引的指定范围|索引列的范围查询：`>`、`<`、`>=`、`<=`、`BETWEEN AND`、`IN()`、`LIKE '前缀%'`|合格性能，范围查询的最低要求|
|**index**|全索引扫描，遍历整个索引树，不扫描全表|查询的字段都在索引中（覆盖索引），但没有 WHERE 条件，需要遍历整个索引|性能差，比 ALL 略好（索引比数据文件小），必须优化|
|**ALL**|全表扫描（Full Table Scan），遍历整张表的所有数据行|没有可用索引，或优化器选择全表扫描|性能极差，必须优化|

#### 核心优化原则

1. 绝对避免 `ALL` 全表扫描，99% 的慢查询都是全表扫描导致的；
2. 普通等值查询，优化目标是 `ref` 级别；
3. 主键 / 唯一索引查询，优化目标是 `const` 级别；
4. JOIN 关联查询，关联字段优化目标是 `eq_ref` 级别；
5. 范围查询，优化目标是 `range` 级别。

### 3.4 key、possible_keys、key_len：索引使用分析三剑客

这三个字段是**验证索引是否生效、索引利用率高低**的核心，也是联合索引最左前缀原则的直接验证工具。

#### 3.4.1 possible_keys：候选索引

possible_keys 表示 MySQL 优化器认为**可能可以用来匹配查询的索引**，是优化器的候选索引列表。

> 关键注意：possible_keys 有值，不代表索引会被使用，最终以 `key` 字段为准；possible_keys 为 NULL，代表没有可用的候选索引，必然全表扫描。

#### 3.4.2 key：实际使用的索引

key 表示 MySQL 优化器**实际执行时使用的索引名称**，是判断索引是否生效的核心字段。

- key 为 NULL：索引完全失效，执行全表扫描 / 全索引扫描；
- key 显示索引名：索引被选中，生效；
- 特殊情况：key 显示 PRIMARY，代表使用了主键索引。

#### 3.4.3 key_len：索引使用的字节长度

key_len 是执行计划中**最容易被忽略，但极其重要的字段**，表示 MySQL 实际使用的索引字节长度，通过这个值，我们可以**精准判断联合索引到底用到了哪些列**，完美验证最左前缀原则。

##### 核心计算规则

key_len 的长度由索引列的类型、长度、是否可为空、字符集共同决定，核心计算公式：

|字段类型|非空|可为空|补充说明|
|:--|:--|:--|:--|
|TINYINT|1 字节|2 字节|可为空额外加 1 字节|
|INT|4 字节|5 字节|可为空额外加 1 字节|
|BIGINT|8 字节|9 字节|可为空额外加 1 字节|
|CHAR(n)|n × 字符集字节数|n × 字符集字节数 +1|utf8mb4 每个字符占 4 字节，utf8 占 3 字节|
|VARCHAR(n)|n × 字符集字节数 +2|n × 字符集字节数 +3|变长字段额外加 2 字节，可为空再加 1 字节|
|DATE|3 字节|4 字节|可为空额外加 1 字节|
|DATETIME|5 字节|6 字节|可为空额外加 1 字节|

##### 示例：联合索引 key_len 计算

表 `user`，联合索引 `idx_name_age_phone (name varchar(50) NOT NULL, age INT NULL, phone char(11) NOT NULL)`，字符集 utf8mb4：

- name 列：50 × 4 = 200 字节（非空 varchar，200+2=202 字节）；
- age 列：4 字节 +1 字节（可为空）=5 字节；
- phone 列：11×4=44 字节（非空 char）；
- 三列总长度：202+5+44=251 字节。

通过 key_len 验证最左前缀：

```sql
-- 1. 仅匹配name列，key_len=202，仅用到第一列
EXPLAIN SELECT * FROM user WHERE name = '张三';

-- 2. 匹配name+age列，key_len=202+5=207，用到前两列
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age = 20;

-- 3. 匹配全部三列，key_len=251，索引全量生效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age = 20 AND phone = '13800138000';

-- 4. 范围查询截断，key_len=207，仅用到name+age列，phone列失效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age > 20 AND phone = '13800138000';
```

> 核心结论：key_len 是验证联合索引匹配列数的唯一精准依据，比 key 字段更能体现索引的利用率。

### 3.5 rows、filtered：扫描行数与匹配率

#### 3.5.1 rows：预估扫描行数

rows 表示 MySQL 优化器预估的，为了找到目标数据，**需要扫描的行数**，是 SQL 性能的核心指标，**值越小越好**。

- 即使索引生效，rows 值很大，也代表索引过滤效果差，需要优化索引（比如区分度低的索引）；
- 全表扫描时，rows 值等于表的总行数，性能极差。

#### 3.5.2 filtered：匹配率

filtered 表示经过索引过滤后，符合最终查询条件的行数占扫描行数的比例，取值范围 0-100，**值越高越好**。

- 核心公式：`最终匹配行数 = rows × filtered / 100`；
- filtered=100，代表索引过滤后的数据 100% 符合条件，无需回表二次过滤；
- filtered 很低，代表索引过滤效果差，需要回表过滤大量数据，需优化索引。

### 3.6 Extra：额外信息（优化提示核心）

Extra 字段会输出 SQL 执行的额外细节信息，里面包含了大量的优化提示和性能问题预警，是执行计划分析的重中之重。

我们按「正向优化标识」「负向问题标识」分类讲解，重点掌握高频出现的标识。

#### 3.6.1 正向优化标识（性能优秀，无需优化）

|Extra 标识|核心含义|说明|
|:--|:--|:--|
|**Using index**|覆盖索引生效|最优标识！查询的所有字段都包含在索引中，无需回表查询整行数据，性能极致|
|**Using index condition**|索引条件下推（ICP）|优化标识！存储引擎层先按索引条件过滤数据，再回表，减少回表次数|
|**Using where**|回表过滤|索引过滤后，在服务层用 WHERE 条件二次过滤，若同时有 Using index，无需优化；无 Using index 时，代表需要回表|
|**Select tables optimized away**|聚合函数优化|无需遍历表 / 索引，直接返回结果，如 `SELECT MAX(id) FROM user`，主键索引直接取最大值|
|**Impossible WHERE**|WHERE 条件不成立|WHERE 条件永远为 false，如 `WHERE 1=2`，无需扫描表|

#### 3.6.2 负向问题标识（性能问题，必须优化）

|Extra 标识|核心含义|优化方案|
|:--|:--|:--|
|**Using filesort**|文件排序|高频性能杀手！MySQL 无法利用索引完成排序，需要在内存 / 磁盘中进行排序，性能极差。<br><br>优化方案：给 ORDER BY/GROUP BY 的字段建立合适的联合索引，利用索引的有序性避免排序|
|**Using temporary**|使用临时表|高频性能杀手！MySQL 需要创建临时表存储中间结果，通常出现在 GROUP BY、DISTINCT、UNION 等场景，性能极差。<br><br>优化方案：给分组 / 去重字段建立联合索引，避免临时表创建|
|**Using join buffer (Block Nested Loop)**|关联查询使用块嵌套循环|JOIN 查询的关联字段没有索引，驱动表的每一行都要遍历被驱动表全表，性能极差。<br><br>优化方案：给 JOIN 关联字段建立索引|
|**Full scan on NULL key**|子查询优化失效|子查询返回 NULL 值，导致全表扫描，必须优化子查询|
|**Range checked for each record**|无可用索引，逐行检查索引|没有合适的索引，每一行都要检查索引使用成本，性能极差，必须加索引|

> 核心避坑：`Using filesort` 和 `Using temporary` 是慢查询的两大元凶，看到这两个标识，必须第一时间优化。

## 四、进阶实操：EXPLAIN 典型场景分析

结合之前学习的索引、联合索引知识，用 EXPLAIN 验证核心场景，形成知识闭环。

### 4.1 场景 1：联合索引最左前缀原则验证

**表结构**：user 表，联合索引 `idx_name_age_gender (name, age, gender)`

```sql
-- 1. 跳过最左列，索引失效，type=ALL，key=NULL
EXPLAIN SELECT * FROM user WHERE age = 20 AND gender = '男';

-- 2. 范围查询截断，key_len=207，仅用到name+age列
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age > 20 AND gender = '男';

-- 3. IN查询不截断，key_len=251，三列索引全生效
EXPLAIN SELECT * FROM user WHERE name = '张三' AND age IN (20,25) AND gender = '男';

-- 4. 覆盖索引生效，Extra=Using index，无需回表
EXPLAIN SELECT name, age FROM user WHERE name = '张三' AND gender = '男';
```

### 4.2 场景 2：JOIN 关联查询优化分析

**核心原则**：小表驱动大表，关联字段必须加索引，被驱动表的关联字段优先加索引。

```sql
-- 表：user（小表，1万行）、order_info（大表，100万行），关联字段 user_id
-- 正确：小表user驱动大表order_info，order_info.user_id加索引，type=eq_ref
EXPLAIN SELECT u.name, o.order_no 
FROM user u
INNER JOIN order_info o ON u.user_id = o.user_id
WHERE u.age > 18;

-- 优化点1：被驱动表order_info的user_id必须加索引，避免Using join buffer
-- 优化点2：驱动表的过滤字段age加索引，减少驱动表的扫描行数
```

### 4.3 场景 3：排序优化分析（Using filesort 解决）

```sql
-- 问题SQL：Extra=Using filesort，文件排序
EXPLAIN SELECT name, age FROM user WHERE name = '张三' ORDER BY age;

-- 优化：创建联合索引 idx_name_age (name, age)，name等值匹配后，age在索引中有序
-- 优化后：Extra无Using filesort，利用索引有序性避免排序
EXPLAIN SELECT name, age FROM user WHERE name = '张三' ORDER BY age;
```

### 4.4 场景 4：索引失效场景验证

```sql
-- 1. 索引列加函数，索引失效，type=ALL，key=NULL
EXPLAIN SELECT * FROM user WHERE LEFT(name,1) = '张';

-- 2. 隐式类型转换，索引失效，name是varchar，用数字查询
EXPLAIN SELECT * FROM user WHERE name = 123456;

-- 3. 左模糊匹配，索引失效
EXPLAIN SELECT * FROM user WHERE name LIKE '%张三';

-- 4. OR连接非索引列，索引失效
EXPLAIN SELECT * FROM user WHERE name = '张三' OR address = '北京';
```

## 五、执行计划分析的标准流程（万能模板）

拿到一条慢 SQL，按以下 6 步分析执行计划，99% 的问题都能定位：

1. **看 type**：是否出现 ALL 全表扫描，优化目标至少达到 range 级别，优先 ref 级别；
2. **看 key**：索引是否生效，key 是否为 NULL，是否选对了索引；
3. **看 key_len**：索引利用率如何，联合索引是否只用到了前缀列，是否有截断；
4. **看 rows、filtered**：扫描行数是否过多，匹配率是否过低，索引过滤效果是否达标；
5. **看 Extra**：是否出现 Using filesort、Using temporary、Using join buffer 等负向标识，优先解决；
6. **看 id、select_type**：是否有依赖子查询、衍生表，优化子查询和 JOIN 顺序。

## 六、高频踩坑与避坑指南

### 6.1 坑 1：possible_keys 有值，就认为索引生效了

- 误区：只看 possible_keys 有值，就认为索引生效了，忽略 key 字段为 NULL；
- 避坑：索引是否生效，**唯一判断标准是 key 字段**，possible_keys 只是候选列表。

### 6.2 坑 2：key 有值，就认为索引完全生效了

- 误区：key 显示了索引名，就认为索引全量生效了，忽略 key_len；
- 避坑：索引的利用率，必须看 key_len，联合索引可能只用到了第一列，后续列完全失效。

### 6.3 坑 3：只关注索引是否生效，忽略扫描行数

- 误区：索引生效了就认为 SQL 优化完成了，忽略 rows 扫描行数依然很大；
- 避坑：索引生效只是基础，还要看索引的过滤效果，rows 越小越好，区分度低的索引（如性别）即使生效，扫描行数依然很多，需要优化。

### 6.4 坑 4：忽略 Extra 字段，只关注 type 和 key

- 误区：只看索引是否生效，忽略 Extra 中的 Using filesort、Using temporary，导致 SQL 依然很慢；
- 避坑：Extra 字段是优化的核心提示，文件排序和临时表的性能损耗，远大于索引失效的损耗，必须优先解决。

### 6.5 坑 5：EXPLAIN 结果和实际执行不一致

- 误区：认为 EXPLAIN 的执行计划就是 MySQL 实际执行的计划；
- 避坑：EXPLAIN 是模拟优化器的执行计划，MySQL 实际执行时可能会有动态优化，MySQL 8.0.18+ 用 `EXPLAIN ANALYZE` 查看真实执行情况。

## 八、核心要点总结

1. **EXPLAIN 核心定位**：是 SQL 优化、索引验证的核心工具，通过模拟优化器执行计划，让我们看到 MySQL 如何处理 SQL。
2. **6 大核心字段**：id（执行顺序）、type（访问类型）、key（实际索引）、key_len（索引长度）、rows（扫描行数）、Extra（额外信息），是执行计划分析的重中之重。
3. **type 优化目标**：绝对避免 ALL 全表扫描，普通查询优化到 ref 级别，范围查询优化到 range 级别，JOIN 查询优化到 eq_ref 级别。
4. **索引验证核心**：key 字段判断索引是否生效，key_len 字段判断索引的利用率，精准验证联合索引的最左前缀原则。
5. **性能杀手预警**：Extra 字段中的 Using filesort（文件排序）、Using temporary（临时表）、Using join buffer（无索引关联），是三大高频性能问题，必须优先优化。
6. **覆盖索引最优解**：Extra 中的 Using index 是最优标识，代表覆盖索引生效，无需回表，是 SQL 查询的终极优化目标。
7. **分析标准流程**：先看 type，再看 key、key_len，然后看 rows、filtered，最后看 Extra，按优先级逐步优化。

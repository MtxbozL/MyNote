
MySQL 高可用架构的核心设计目标是**最小化业务停机时间，保证数据零丢失**，核心衡量指标为：

- **RTO（Recovery Time Objective，恢复时间目标）**：故障发生后，业务恢复正常的最长时间，越短越好；
- **RPO（Recovery Point Objective，恢复点目标）**：故障发生后，允许丢失的最大数据量，核心业务要求 RPO=0。

本节详解生产环境最主流的两套高可用方案：老牌成熟的 MHA 方案，与官方原生的 InnoDB Cluster/Group Replication 方案。

### 9.5.1 MHA（Master High Availability）

MHA 是由日本 DeNA 公司开源的 MySQL 主从高可用管理工具，是过去十年业界最主流的 MySQL 高可用方案，成熟稳定，无侵入性，兼容传统复制与 GTID 复制。

#### 架构组成

MHA 架构分为两个核心组件，无任何额外的中间件依赖，部署简单：

1. **MHA Manager（管理节点）**：独立部署的管理节点，通常部署在单独的服务器上，负责集群的故障检测、主从切换决策、故障切换执行、配置更新、告警通知；
2. **MHA Node（数据节点）**：部署在所有 MySQL 主从节点上，负责故障切换时的 Binlog 差异补全、Relay Log 管理、数据一致性校验，是 Manager 节点执行切换操作的执行代理。

#### 故障自动切换核心流程

MHA 的核心优势是 **保证主从切换时数据零丢失** ，核心切换流程如下：

1. **故障检测**：Manager 节点通过心跳包持续检测主库状态，多次探测失败后，确认主库宕机，进入故障切换流程；
2. **数据补全**：若主库服务器仍可 SSH 访问，Node 节点会读取主库最新的 Binlog 内容，补全到最新的从库，保证所有从库的数据完全一致，RPO=0；
3. **新主库选举**：Manager 节点根据配置的优先级、从库的同步进度，选举出数据最新的从库作为新主库；
4. **从库重定向**：将其他从库重新指向新主库，补全缺失的 Binlog，完成主从关系重建；
5. **VIP 漂移**：将原主库的虚拟 IP（VIP）漂移到新主库，业务层无需修改数据库连接地址，实现业务无感知切换；
6. **告警通知**：切换完成后，发送告警通知运维人员，整个切换流程耗时通常在 10-30 秒，RTO<30 秒。

#### 方案优缺点与适用场景

|优势|劣势|
|---|---|
|1. 成熟稳定，业界有大量十年以上的生产落地案例；<br><br>2. 无侵入性，对 MySQL 原生架构无任何修改，兼容 5.6/5.7/8.0 所有版本；<br><br>3. 性能损耗极低，正常运行时无任何性能开销；<br><br>4. 完美保证数据零丢失，RPO=0；<br><br>5. 部署简单，无复杂的组件依赖，运维成本低|1. 官方社区已停止维护，无新功能更新，仅修复严重 bug；<br><br>2. 仅支持一主多从架构，不支持多主模式、集群自动扩缩容；<br><br>3. 依赖 SSH 免密登录，存在一定的安全风险；<br><br>4. Manager 节点存在单点故障风险，需部署双 Manager 节点；<br><br>5. 跨机房容灾场景支持较弱|

- **适用场景**：传统 MySQL 架构、MySQL 5.7 版本、一主多从的稳定业务架构、对成熟度要求高、无需频繁扩缩容的生产环境。

### 9.5.2 MySQL InnoDB Cluster / Group Replication（MGR）

MySQL Group Replication（简称 MGR，组复制）是 MySQL 官方在 5.7.17 版本推出的原生分布式集群方案，基于 Paxos 分布式共识算法实现，InnoDB Cluster 是基于 MGR 封装的完整高可用解决方案，包含组复制层、路由层、管理工具，是 MySQL 8.0 + 版本官方主推的现代高可用架构。

#### 核心架构组成

InnoDB Cluster 架构分为三个核心组件，完全原生支持，无需第三方依赖：

1. **MySQL Group Replication（MGR，组复制层）**：集群的核心底层，由多个 MySQL 节点组成一个复制组，基于 Paxos 共识算法，事务必须获得组内多数派节点的确认后才能提交，保证集群数据的强一致性；
2. **MySQL Router（路由层）**：官方原生的数据库中间件，部署在业务端，负责集群节点的状态检测、读写请求路由、故障节点自动剔除、负载均衡，业务层仅需连接 Router 的地址，无需感知后端集群的主从切换；
3. **MySQL Shell（管理工具）**：官方提供的集群管理工具，支持集群的一键部署、初始化、扩缩容、故障检测、主从切换，可视化管理集群状态。

#### 核心运行模式

MGR 支持两种运行模式，可根据业务场景选择：

1. **单主模式（Single-Primary Mode，默认推荐）**：集群内仅有一个主节点（Primary），负责所有写业务，其他节点为从节点（Secondary），承担读业务；主节点宕机后，组内自动通过多数派投票选举新主节点，Router 自动将写请求路由到新主节点，RTO<5 秒，RPO=0；
2. **多主模式（Multi-Primary Mode）**：集群内所有节点均可提供写服务，事务在所有节点同步执行，支持多点写入；但对业务代码有严格要求，存在写冲突、死锁风险，生产环境极少使用，仅适用于特定的低并发写入场景。

#### 故障自动切换核心流程

1. **故障检测**：集群内节点通过心跳包持续通信，若某个节点失联超过阈值，组内多数派节点投票确认该节点故障，将其从集群中剔除；
2. **新主库选举**：单主模式下，若故障节点为主节点，组内立即通过多数派投票，选举出数据最新的从节点作为新主节点，更新集群的主节点状态；
3. **路由自动切换**：MySQL Router 实时检测集群状态，自动将写请求路由到新主节点，剔除故障节点，读请求仅路由到健康的从节点，业务完全无感知；
4. **故障节点恢复**：故障节点恢复后，自动重新加入集群，通过增量 Binlog 补全缺失的数据，同步完成后自动加入集群，承担读业务。

#### 方案优缺点与适用场景

|优势|劣势|
|---|---|
|1. MySQL 官方原生支持，持续迭代更新，社区活跃，bug 修复及时；<br><br>2. 基于 Paxos 共识算法，强数据一致性，RPO=0，无数据丢失风险；<br><br>3. 自动故障检测与切换，RTO<5 秒，远快于 MHA；<br><br>4. 原生支持集群水平扩缩容，节点加入 / 退出自动化，无需手动配置；<br><br>5. 内置数据一致性校验、冲突检测、脑裂防护，无需第三方工具；<br><br>6. 完美适配云原生、容器化部署，支持 K8s Operator|1. 有一定的写入性能损耗，相比主从复制，写入性能下降 10%-20%；<br><br>2. 对集群网络要求极高，节点间网络延迟必须 < 1ms，不支持跨城远距离部署；<br><br>3. 有严格的使用限制，仅支持 InnoDB 引擎、必须开启 GTID、表必须有主键等；<br><br>4. 架构复杂度高于 MHA，运维门槛更高，需要掌握分布式共识相关知识；<br><br>5. 集群节点数量必须为奇数，最少 3 节点，最多 9 节点|

- **适用场景**：MySQL 8.0 + 版本、核心交易业务、对高可用与数据一致性要求极高、需要弹性扩缩容、云原生架构的生产环境，是未来 MySQL 高可用架构的主流方向。

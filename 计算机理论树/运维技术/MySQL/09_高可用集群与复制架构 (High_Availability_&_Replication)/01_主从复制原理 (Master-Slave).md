
>单机 MySQL 架构存在三大核心瓶颈：**单点故障风险**（单机宕机导致业务完全不可用）、**高并发读性能瓶颈**（单机 CPU、IO 资源无法支撑海量读请求）、**数据容灾能力缺失**（单机磁盘损坏导致数据永久丢失）。主从复制是解决上述问题的核心基础，通过 Binlog 实现主从节点间的数据同步，支撑读写分离、数据容灾、负载均衡等核心能力；高可用架构则基于复制体系，实现故障自动检测、主从自动切换，最大化降低业务停机时间，是生产环境 MySQL 架构的必备核心能力。本章从主从复制底层原理出发，完整拆解同步模式演进、主从延迟根因排查、GTID 复制体系，最终详解主流高可用架构的选型与实现逻辑。

## 9.1 主从复制原理（Master-Slave）

主从复制是 MySQL 原生支持的数据同步能力，核心基于 Server 层的 Binlog 日志实现，主库将数据修改记录写入 Binlog，从库拉取并回放 Binlog，实现主从节点的数据一致性。

### 9.1.1 主从复制的核心价值

1. **读写分离**：主库承担写业务，多个从库承担读业务，线性提升数据库的读并发能力，解决高并发读性能瓶颈；
2. **数据容灾备份**：从库作为主库的实时副本，主库宕机或数据损坏时，从库可快速接管业务，避免数据永久丢失；
3. **高可用架构基础**：所有 MySQL 高可用方案的底层均依赖主从复制，是故障自动切换的核心前提；
4. **业务隔离**：将报表分析、数据备份、离线统计等重负载业务迁移到从库执行，避免影响主库的核心交易业务；
5. **灰度升级与扩容**：可在从库完成版本升级、硬件更换、架构调整，验证无误后切换为主库，实现业务无感知的架构变更。

### 9.1.2 底层核心三线程模型

主从复制的完整流程由三个核心线程协同完成，其中 1 个线程运行在主库，2 个线程运行在从库，三者共同构成了复制的核心执行模型。

|线程名称|运行节点|核心职责|
|---|---|---|
|Binlog Dump 线程|主库 Master|1. 监听从库的连接请求，为每个连接的从库创建独立的 Dump 线程；<br><br>2. 读取主库 Binlog 的增量内容，按照协议发送给从库的 IO 线程；<br><br>3. 读取 Binlog 时加共享锁，读取完成后立即释放，不阻塞主库业务。|
|IO 线程|从库 Slave|1. 主动与主库建立 TCP 连接，通过认证后向主库发送 Binlog 拉取请求；<br><br>2. 接收主库 Dump 线程发送的 Binlog 内容，实时写入本地的**中继日志（Relay Log）** 文件；<br><br>3. 记录已拉取的 Binlog 文件名与位置点，维护复制进度。|
|SQL 线程|从库 Slave|1. 监听并读取本地 Relay Log 中的增量内容；<br><br>2. 解析 Binlog 中的逻辑操作，在从库中顺序回放执行，保证主从数据一致；<br><br>3. 记录已回放的 Relay Log 位置点，清理已完成回放的中继日志文件。|

> 核心补充：MySQL 5.6 + 引入了多线程复制（MTS），将单 SQL 线程拆分为多个并行回放的工作线程，解决了单线程串行回放的性能瓶颈，详细原理见 9.3 节。

### 9.1.3 主从复制完整执行流程

主从复制的完整数据同步流程分为 7 个核心步骤，全程基于 Binlog 实现，与存储引擎无关：

1. **主库事务提交**：主库执行 DML/DDL 操作，事务提交时，严格按照两阶段提交规则，将修改操作写入 Binlog 并持久化到磁盘；
2. **从库建立连接**：从库 IO 线程主动与主库建立 TCP 连接，完成身份认证后，向主库发送 Binlog 拉取请求，请求中携带从库已同步完成的 Binlog 文件名与偏移量 Position；
3. **主库 Binlog 读取与发送**：主库的 Dump 线程接收到请求后，根据请求的位置点，读取 Binlog 中后续的增量内容，实时发送给从库的 IO 线程；
4. **从库 Relay Log 写入**：从库 IO 线程接收到 Binlog 内容后，实时追加写入本地的 Relay Log 中继日志文件，同时更新本地的`master.info`文件，记录最新的主库 Binlog 同步位置；
5. **从库事务回放**：从库的 SQL 线程监听到 Relay Log 新增内容后，解析 Binlog 中的逻辑操作，在从库中顺序执行，还原主库的修改操作；
6. **回放进度更新**：SQL 线程执行完成后，更新`relay-log.info`文件，记录最新的 Relay Log 回放位置，已完成回放的 Relay Log 会被自动清理；
7. **循环同步**：主库持续产生新的 Binlog，三个线程持续协同工作，实现主从数据的实时同步。

### 9.1.4 复制格式分类

主从复制的格式完全由主库的 Binlog 格式决定，与上一章 Binlog 格式一一对应，生产环境强制使用 ROW 行级格式：

1. **STATEMENT 语句级复制**：主库记录执行的 SQL 语句原文，从库回放相同的 SQL 语句。优点是日志体积小，缺点是包含`NOW()`、`RAND()`、存储过程、触发器的语句会导致主从数据不一致，MySQL 8.0 已不推荐使用；
2. **ROW 行级复制**：主库记录行数据修改前后的镜像，从库基于行镜像回放修改。优点是数据一致性绝对保障，无函数、存储过程导致的同步异常，是生产环境唯一推荐格式；缺点是批量操作会导致日志体积增大；
3. **MIXED 混合模式**：自动在 STATEMENT 和 ROW 格式间切换，格式切换不可控，极端场景仍存在一致性风险，生产环境禁止使用。

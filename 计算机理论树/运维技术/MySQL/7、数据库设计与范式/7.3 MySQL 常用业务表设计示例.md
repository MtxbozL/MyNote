## 一、开篇：示例设计的前置规范

本笔记的所有业务表示例，均严格遵循之前生成的《MySQL 库表设计规范》《三大范式与反范式设计》，核心设计原则如下：

1. **命名规范**：库表字段全小写，下划线分隔，见名知意；
2. **字符集与存储引擎**：统一 `utf8mb4` 字符集，`InnoDB` 存储引擎；
3. **主键设计**：统一自增 `BIGINT UNSIGNED` 做主键，业务唯一字段建唯一索引；
4. **时间字段**：所有表包含 `create_time`、`update_time`、`delete_time`（软删除）；
5. **范式与反范式结合**：基础设计满足 3NF，适度冗余高频低更新字段（如用户名称、商品分类名称），减少多表 JOIN；
6. **清晰注释**：所有表、字段有完整 `COMMENT`，说明业务含义、取值范围、单位等。

## 二、核心场景一：电商核心业务表（最常用）

电商是生产环境中最典型的业务场景，核心表包括用户表、商品表、商品分类表、订单表、订单商品关联表、购物车表，覆盖了用户、商品、订单三大核心模块。

### 2.1 用户表（user_info）

#### 设计思路

- 满足 3NF，用户基础信息独立存储；
- 手机号、邮箱建唯一索引，保证业务唯一性；
- 软删除 `is_deleted`，避免物理删除数据丢失。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS user_info (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    user_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '用户姓名',
    phone CHAR(11) NOT NULL DEFAULT '' COMMENT '手机号（唯一标识）',
    email VARCHAR(100) NOT NULL DEFAULT '' COMMENT '邮箱',
    password VARCHAR(255) NOT NULL DEFAULT '' COMMENT '加密后的密码',
    avatar VARCHAR(255) NOT NULL DEFAULT '' COMMENT '头像URL',
    gender TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '性别：0-未知，1-男，2-女',
    birthday DATE COMMENT '生日',
    province VARCHAR(20) NOT NULL DEFAULT '' COMMENT '省份',
    city VARCHAR(20) NOT NULL DEFAULT '' COMMENT '城市',
    address VARCHAR(255) NOT NULL DEFAULT '' COMMENT '详细地址',
    balance DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '账户余额（单位：元）',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    UNIQUE INDEX uk_phone (phone) COMMENT '手机号唯一索引',
    UNIQUE INDEX uk_email (email) COMMENT '邮箱唯一索引',
    INDEX idx_create_time (create_time) COMMENT '创建时间索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='用户信息表：存储用户的基本信息、账户余额等';
```

### 2.2 商品分类表（goods_category）

#### 设计思路

- 满足 3NF，分类信息独立存储；
- 支持多级分类（通过 `parent_id` 自关联），`parent_id=0` 表示一级分类；
- `sort` 字段用于分类排序，满足业务展示需求。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS goods_category (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    parent_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '父分类ID，0表示一级分类',
    category_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '分类名称',
    category_icon VARCHAR(255) NOT NULL DEFAULT '' COMMENT '分类图标URL',
    sort INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '排序值，越小越靠前',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    INDEX idx_parent_id (parent_id) COMMENT '父分类ID索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='商品分类表：支持多级分类，自关联实现';
```

### 2.3 商品表（goods_info）

#### 设计思路

- 满足 3NF，通过 `category_id` 关联商品分类表；
- **适度反范式**：冗余 `category_name`（分类名称），分类名称更新频率低、查询频率高，避免每次查询商品都关联分类表；
- 库存 `stock` 用 `INT UNSIGNED`，价格 `price` 用 `DECIMAL(10,2)`，保证精度；
- `is_on_sale` 字段控制商品上下架。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS goods_info (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    goods_name VARCHAR(100) NOT NULL DEFAULT '' COMMENT '商品名称',
    goods_image VARCHAR(255) NOT NULL DEFAULT '' COMMENT '商品主图URL',
    goods_desc TEXT COMMENT '商品描述',
    category_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '商品分类ID',
    category_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '商品分类名称（冗余字段，反范式，减少JOIN）',
    price DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '商品价格（单位：元）',
    original_price DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '商品原价（单位：元）',
    stock INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '商品库存',
    sales INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '商品销量',
    sort INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '排序值，越小越靠前',
    is_on_sale TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '是否上架：0-下架，1-上架',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    INDEX idx_category_id (category_id) COMMENT '分类ID索引',
    INDEX idx_is_on_sale (is_on_sale) COMMENT '上架状态索引',
    INDEX idx_sales (sales) COMMENT '销量索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='商品信息表：存储商品的基本信息、价格、库存等，冗余分类名称减少JOIN';
```

### 2.4 订单表（order_info）

#### 设计思路

- 满足 3NF，通过 `user_id` 关联用户表；
- **适度反范式**：冗余 `user_name`、`user_phone`，避免每次查询订单都关联用户表；
- `order_no` 建唯一索引，订单号是业务唯一标识（建议用雪花算法 / 时间戳 + 随机数生成）；
- `order_status` 用 `TINYINT`，有清晰的取值注释；
- 金额字段统一用 `DECIMAL(10,2)`，保证精度；
- 软删除 `is_deleted`，订单数据通常不物理删除。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS order_info (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    order_no VARCHAR(32) NOT NULL DEFAULT '' COMMENT '订单号（业务唯一标识，雪花算法/时间戳+随机数）',
    user_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '用户ID',
    user_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '用户姓名（冗余字段，反范式，减少JOIN）',
    user_phone CHAR(11) NOT NULL DEFAULT '' COMMENT '用户手机号（冗余字段，反范式，减少JOIN）',
    order_amount DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '订单总金额（单位：元）',
    pay_amount DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '实付金额（单位：元）',
    freight DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '运费（单位：元）',
    discount_amount DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '优惠金额（单位：元）',
    order_status TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '订单状态：0-待支付，1-已支付，2-已发货，3-已收货，4-已取消，5-已退款',
    pay_time DATETIME COMMENT '支付时间',
    ship_time DATETIME COMMENT '发货时间',
    receive_time DATETIME COMMENT '收货时间',
    receiver_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '收货人姓名',
    receiver_phone CHAR(11) NOT NULL DEFAULT '' COMMENT '收货人手机号',
    receiver_address VARCHAR(255) NOT NULL DEFAULT '' COMMENT '收货地址',
    remark VARCHAR(255) NOT NULL DEFAULT '' COMMENT '订单备注',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    UNIQUE INDEX uk_order_no (order_no) COMMENT '订单号唯一索引',
    INDEX idx_user_id (user_id) COMMENT '用户ID索引',
    INDEX idx_order_status (order_status) COMMENT '订单状态索引',
    INDEX idx_create_time (create_time) COMMENT '创建时间索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单信息表：存储订单的基本信息、金额、状态等，冗余用户信息减少JOIN';
```

### 2.5 订单商品关联表（order_goods_rel）

#### 设计思路

- 满足 2NF，联合主键 `(order_id, goods_id)`，仅存储完全依赖于联合主键的字段；
- **关键设计**：冗余 `goods_name`、`goods_price`（下单时的商品快照价格），因为商品名称、价格后续可能会变化，必须存储下单时的快照，保证历史订单数据的准确性；
- `goods_num` 存储商品数量，完全依赖于联合主键。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS order_goods_rel (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    order_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '订单ID',
    goods_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '商品ID',
    goods_name VARCHAR(100) NOT NULL DEFAULT '' COMMENT '商品名称（下单时的快照，避免后续商品名称变化影响历史订单）',
    goods_image VARCHAR(255) NOT NULL DEFAULT '' COMMENT '商品主图（下单时的快照）',
    goods_price DECIMAL(10, 2) NOT NULL DEFAULT 0.00 COMMENT '商品价格（下单时的快照价格，单位：元）',
    goods_num INT UNSIGNED NOT NULL DEFAULT 1 COMMENT '商品数量',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    UNIQUE INDEX uk_order_goods (order_id, goods_id) COMMENT '订单ID+商品ID唯一索引',
    INDEX idx_order_id (order_id) COMMENT '订单ID索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='订单商品关联表：存储订单与商品的关联关系、商品快照信息';
```

### 2.6 购物车表（cart_info）

#### 设计思路

- 满足 3NF，通过 `user_id`、`goods_id` 关联用户表、商品表；
- 唯一索引 `uk_user_goods`，保证同一用户同一商品在购物车中只有一条记录，更新数量即可；
- `is_selected` 字段控制购物车商品是否选中，满足结算时的选择需求。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS cart_info (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    user_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '用户ID',
    goods_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '商品ID',
    goods_num INT UNSIGNED NOT NULL DEFAULT 1 COMMENT '商品数量',
    is_selected TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '是否选中：0-未选中，1-选中',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    UNIQUE INDEX uk_user_goods (user_id, goods_id) COMMENT '用户ID+商品ID唯一索引',
    INDEX idx_user_id (user_id) COMMENT '用户ID索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='购物车表：存储用户的购物车商品信息';
```

## 三、核心场景二：RBAC 权限管理表（通用系统）

RBAC（基于角色的访问控制）是绝大多数后台系统的通用权限模型，核心表包括用户表、角色表、权限表、用户角色关联表、角色权限关联表，满足 3NF，通过关联表实现多对多关系。

### 3.1 角色表（sys_role）

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS sys_role (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    role_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '角色名称',
    role_code VARCHAR(50) NOT NULL DEFAULT '' COMMENT '角色编码（唯一标识，如admin、user）',
    role_desc VARCHAR(255) NOT NULL DEFAULT '' COMMENT '角色描述',
    sort INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '排序值，越小越靠前',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    UNIQUE INDEX uk_role_code (role_code) COMMENT '角色编码唯一索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统角色表：存储系统的角色信息';
```

### 3.2 权限表（sys_permission）

#### 设计思路

- 支持多级权限（通过 `parent_id` 自关联），`parent_id=0` 表示一级权限（菜单）；
- `permission_type` 区分菜单权限和按钮权限；
- `permission_code` 是权限的唯一标识，用于代码中权限校验。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS sys_permission (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    parent_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '父权限ID，0表示一级权限',
    permission_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '权限名称',
    permission_code VARCHAR(100) NOT NULL DEFAULT '' COMMENT '权限编码（唯一标识，如user:add、user:list）',
    permission_type TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '权限类型：1-菜单，2-按钮',
    menu_url VARCHAR(255) NOT NULL DEFAULT '' COMMENT '菜单URL（仅菜单类型有效）',
    menu_icon VARCHAR(255) NOT NULL DEFAULT '' COMMENT '菜单图标（仅菜单类型有效）',
    sort INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '排序值，越小越靠前',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    UNIQUE INDEX uk_permission_code (permission_code) COMMENT '权限编码唯一索引',
    INDEX idx_parent_id (parent_id) COMMENT '父权限ID索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统权限表：存储系统的菜单权限和按钮权限，支持多级权限';
```

### 3.3 用户角色关联表（sys_user_role_rel）

    INDEX idx_role_id (role_id) COMMENT '角色ID索引'

## 四、核心场景三：CMS 内容管理表（通用内容系统）

CMS 内容管理系统的核心表包括文章表、文章分类表、标签表、文章标签关联表，支持文章分类、标签、内容管理。

### 4.1 文章表（article_info）

#### 设计思路

- 满足 3NF，通过 `category_id` 关联文章分类表；
- **适度反范式**：冗余 `category_name`，减少 JOIN；
- `article_content` 用 `TEXT` 存储文章内容；
- `view_count`、`like_count` 存储统计数据，可通过定时任务更新，避免实时聚合。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS article_info (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    article_title VARCHAR(200) NOT NULL DEFAULT '' COMMENT '文章标题',
    article_cover VARCHAR(255) NOT NULL DEFAULT '' COMMENT '文章封面URL',
    article_summary VARCHAR(500) NOT NULL DEFAULT '' COMMENT '文章摘要',
    article_content TEXT COMMENT '文章内容',
    category_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '文章分类ID',
    category_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '文章分类名称（冗余字段，反范式，减少JOIN）',
    author VARCHAR(50) NOT NULL DEFAULT '' COMMENT '文章作者',
    view_count BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '浏览量',
    like_count BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '点赞数',
    comment_count BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '评论数',
    is_top TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否置顶：0-否，1-是',
    is_published TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '是否发布：0-草稿，1-已发布',
    is_deleted TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '是否删除：0-未删除，1-已删除',
    create_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
    update_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',
    delete_time DATETIME COMMENT '删除时间',
    INDEX idx_category_id (category_id) COMMENT '分类ID索引',
    INDEX idx_is_published (is_published) COMMENT '发布状态索引',
    INDEX idx_create_time (create_time) COMMENT '创建时间索引',
    INDEX idx_view_count (view_count) COMMENT '浏览量索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='文章信息表：存储文章的基本信息、内容、统计数据等';
```

---

## 五、核心场景四：操作日志表（通用系统）

操作日志表是绝大多数系统的必备表，用于记录用户的操作行为，便于审计和排查问题。

### 5.1 操作日志表（sys_operation_log）

#### 设计思路

- 日志表读多写少，统一用 `InnoDB` 存储引擎（支持崩溃恢复，数据安全）；
- 记录操作人、操作时间、操作类型、操作模块、请求参数、响应结果、IP 地址等核心信息；
- `request_param`、`response_result` 用 `TEXT` 存储，可能较长；
- 索引设计：`user_id`、`operation_time`、`operation_type` 建索引，便于查询。

#### 建表 SQL

```sql
CREATE TABLE IF NOT EXISTS sys_operation_log (
    id BIGINT UNSIGNED PRIMARY KEY AUTO_INCREMENT COMMENT '主键ID',
    user_id BIGINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '操作人ID',
    user_name VARCHAR(50) NOT NULL DEFAULT '' COMMENT '操作人姓名',
    operation_type TINYINT UNSIGNED NOT NULL DEFAULT 0 COMMENT '操作类型：1-新增，2-修改，3-删除，4-查询，5-登录，6-登出',
    operation_module VARCHAR(50) NOT NULL DEFAULT '' COMMENT '操作模块',
    operation_desc VARCHAR(255) NOT NULL DEFAULT '' COMMENT '操作描述',
    request_url VARCHAR(255) NOT NULL DEFAULT '' COMMENT '请求URL',
    request_method VARCHAR(10) NOT NULL DEFAULT '' COMMENT '请求方法：GET、POST、PUT、DELETE',
    request_param TEXT COMMENT '请求参数（JSON格式）',
    response_result TEXT COMMENT '响应结果（JSON格式）',
    ip_address VARCHAR(50) NOT NULL DEFAULT '' COMMENT '操作人IP地址',
    operation_time DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP COMMENT '操作时间',
    cost_time INT UNSIGNED NOT NULL DEFAULT 0 COMMENT '请求耗时（单位：毫秒）',
    is_success TINYINT UNSIGNED NOT NULL DEFAULT 1 COMMENT '是否成功：0-失败，1-成功',
    error_msg VARCHAR(500) NOT NULL DEFAULT '' COMMENT '错误信息（失败时记录）',
    INDEX idx_user_id (user_id) COMMENT '操作人ID索引',
    INDEX idx_operation_time (operation_time) COMMENT '操作时间索引',
    INDEX idx_operation_type (operation_type) COMMENT '操作类型索引'
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='系统操作日志表：记录用户的操作行为，用于审计和排查问题';
```

## 六、示例设计的核心要点总结

1. **命名规范统一**：所有表、字段全小写，下划线分隔，见名知意，避免关键字；
2. **主键设计统一**：自增 `BIGINT UNSIGNED` 做主键，业务唯一字段建唯一索引；
3. **时间字段统一**：所有表包含 `create_time`、`update_time`、`delete_time`，软删除避免物理删除；
4. **范式与反范式结合**：基础设计满足 3NF，适度冗余高频低更新字段（如用户名称、分类名称），减少多表 JOIN；
5. **索引设计合理**：唯一索引保证业务唯一性，普通索引加速查询，联合索引覆盖高频查询场景，避免冗余索引；
6. **数据类型选择合理**：用小不用大，金额用 `DECIMAL`，时间用 `DATETIME`，避免 NULL 值，给合理默认值；
7. **清晰注释完整**：所有表、字段有完整 `COMMENT`，说明业务含义、取值范围、单位、冗余原因等；
8. **多对多关系用关联表**：满足 2NF，关联表仅存储关联关系，无其他冗余字段；
9. **快照数据保障历史准确性**：订单商品表存储下单时的商品快照，避免后续商品变化影响历史订单；
10. **统计数据预计算**：文章的浏览量、点赞数等统计数据，可通过定时任务更新，避免实时聚合查询。

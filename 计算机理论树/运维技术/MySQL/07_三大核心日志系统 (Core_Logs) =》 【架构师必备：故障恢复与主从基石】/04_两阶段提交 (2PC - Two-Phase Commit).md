### 7.4.1 两阶段提交的核心背景

两阶段提交是 MySQL 为了协调**InnoDB 引擎层的 Redo Log**与**Server 层的 Binlog**的写入一致性，实现的内部 XA 事务机制，是保证事务持久性、主从数据一致性的核心基础。

如果没有两阶段提交，两个独立的日志系统会出现严重的一致性问题：

1. 若 Redo Log 写入成功、Binlog 写入失败：崩溃恢复后，主库通过 Redo Log 提交了事务，但 Binlog 无该事务记录，从库无法同步，主从数据不一致；
2. 若 Binlog 写入成功、Redo Log 写入失败：崩溃恢复后，主库回滚了该事务，但 Binlog 已发送给从库，从库执行了该事务，主从数据不一致。

两阶段提交的核心目标是：**保证 Redo Log 与 Binlog 的严格原子性，要么两个日志都完整持久化，要么都不生效，彻底避免主从数据不一致**。

### 7.4.2 两阶段提交完整执行流程

以一个 UPDATE 事务的提交为例，完整的两阶段提交流程如下：

1. **事务执行阶段**：执行 UPDATE 语句，修改 Buffer Pool 中的数据页，生成对应的 Undo Log 和 Redo Log（写入 Log Buffer），同时生成 Binlog（写入 Binlog Cache）。
2. **第一阶段：Prepare 阶段**
    
    - InnoDB 将事务状态设置为`PREPARE`（准备提交）；
    - 将 Log Buffer 中的 Redo Log 写入 OS Page Cache，并调用`fsync()`强制刷入磁盘，保证 Redo Log 持久化；
    - 此时事务已完成所有数据修改，仅等待 Server 层的最终提交指令，崩溃重启后可识别该准备状态的事务。
    
3. **第二阶段：Commit 阶段**
    
    - Server 层将 Binlog Cache 中的 Binlog 写入 OS Page Cache，调用`fsync()`强制刷入磁盘，保证 Binlog 持久化；
    - Server 层向 InnoDB 发送最终的事务提交指令；
    - InnoDB 将事务状态从`PREPARE`修改为`COMMITTED`，在 Redo Log 中写入 Commit 标记，事务正式提交完成；
    - 向客户端返回事务提交成功响应。
    

**核心规则**：Binlog 的持久化是事务最终提交的唯一判定标准，只有 Binlog 成功刷入磁盘，事务才会最终提交；Binlog 持久化失败，事务会直接回滚，保证两个日志的严格一致性。

### 7.4.3 崩溃恢复的一致性判断规则

MySQL 崩溃重启后，会根据 Redo Log 与 Binlog 的状态，自动判断事务的提交或回滚，核心规则如下：

1. 若 Redo Log 中的事务已有完整的`COMMIT`标记：直接提交该事务，无需校验 Binlog；
2. 若 Redo Log 中的事务仅处于`PREPARE`状态，无`COMMIT`标记：
    
    - 校验该事务对应的 Binlog 是否完整持久化到磁盘：若 Binlog 完整存在，提交该事务；
    - 若 Binlog 不存在或不完整，回滚该事务。
    

**一致性保障逻辑**：

- Binlog 完整持久化，说明该事务已记录到归档日志，已发送给从库执行，主库必须提交，保证主从一致；
- Binlog 未持久化，说明从库不会收到该事务，主库回滚，主从数据保持一致；
- Redo Log 已持久化，保证已提交的事务不会因崩溃丢失，实现持久性保障。

### 7.4.4 性能优化：组提交（Group Commit）

两阶段提交中，每次事务提交都需要调用两次`fsync()`（Redo Log 与 Binlog 各一次），高并发场景下，频繁的`fsync()`会成为严重的性能瓶颈。MySQL 5.6 + 引入**组提交（Group Commit）** 机制，解决该性能问题。

组提交的核心逻辑：高并发场景下，多个事务的提交请求会被合并为一个组，一次`fsync()`批量完成多个事务的日志刷盘，大幅减少磁盘 IO 调用次数，提升高并发下的事务提交性能。

- Redo Log 组提交：合并多个事务的 Redo Log 刷盘请求，一次`fsync()`完成批量持久化；
- Binlog 组提交：合并多个事务的 Binlog 刷盘请求，一次`fsync()`完成批量持久化。

组提交机制在双 1 强一致配置下，可将高并发写性能提升一个数量级，生产环境默认开启，无需手动配置。

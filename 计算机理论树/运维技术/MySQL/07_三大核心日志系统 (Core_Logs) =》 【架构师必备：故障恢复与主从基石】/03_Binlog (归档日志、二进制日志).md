### 7.3.1 核心定位与本质特性

- **所属层级**：MySQL Server 层，所有存储引擎通用，无论是否使用 InnoDB，都可开启 Binlog
- **日志类型**：逻辑日志，记录数据库的逻辑修改操作，而非物理页的字节变化
- **两大核心作用**：
    
    1. **数据备份与恢复**：配合全量备份，通过 Binlog 实现时间点恢复（PITR, Point-In-Time Recovery），可恢复到任意一个 Binlog 记录的事务提交时间点；
    2. **主从复制核心载体**：主库将 Binlog 发送给从库，从库重放 Binlog 中的逻辑操作，实现主从数据同步，是 MySQL 高可用架构的核心基础。
    
- **核心写入特性**：**追加写**，而非循环覆盖写。新的日志内容永远追加到 Binlog 文件末尾，文件大小达到`max_binlog_size`阈值后，自动滚动生成新的编号文件，不会覆盖历史日志，可长期保留用于备份与同步。

### 7.3.2 Binlog 日志格式选型

Binlog 支持三种日志格式，由参数`binlog_format`控制，不同格式的一致性、性能、体积存在本质差异，生产环境推荐使用 ROW 格式。

|格式类型|核心实现|优点|缺点|适用场景|
|---|---|---|---|---|
|STATEMENT（语句级）|记录事务执行的 SQL 语句原文|日志体积小，占用磁盘空间少，批量操作仅记录一条 SQL|存在严重的数据一致性风险：包含`NOW()`、`RAND()`、存储过程、自定义函数的语句，主从重放会出现数据不一致；无法准确记录行级修改|MySQL 8.0 已不推荐使用，仅适用于极简静态业务|
|ROW（行级，8.0 默认值，生产推荐）|不记录 SQL 语句，记录行数据修改前后的镜像：<br><br>・before image：修改前的行数据<br><br>・after image：修改后的行数据|数据一致性绝对保障，无论函数、存储过程、触发器，都能准确重放行级修改，主从同步无一致性风险；支持数据闪回恢复|日志体积较大，批量更新、全表更新会记录所有修改行的镜像|所有生产环境核心业务，金融级场景强制使用|
|MIXED（混合模式）|自动在 STATEMENT 和 ROW 格式间切换，简单语句用 STATEMENT，复杂语句用 ROW|兼顾日志体积与一致性|格式切换不可控，极端场景仍存在一致性风险|生产环境不推荐使用|

#### ROW 格式优化配置

通过参数`binlog_row_image`可控制 ROW 格式的镜像记录范围，大幅减少日志体积：

- `FULL`（默认）：记录行所有列的修改前后镜像；
- `MINIMAL`（推荐）：仅记录发生修改的列，以及能唯一标识行的主键列，批量更新场景下可减少 70% 以上的日志体积；
- `NOBLOB`：除 BLOB/TEXT 列外，记录所有列的镜像，适用于包含大文本字段的表。

### 7.3.3 核心刷盘策略与强一致配置

Binlog 的刷盘策略直接决定了日志的持久化安全性，核心由参数`sync_binlog`控制，该参数控制 Binlog 从操作系统 Page Cache 刷入物理磁盘的时机。

|参数值|刷盘执行逻辑|安全性|性能|适用场景|
|---|---|---|---|---|
|0|由操作系统自动控制刷盘时机，MySQL 不主动调用`fsync()`|最低，操作系统宕机时会丢失大量 Binlog 数据|最高|非核心测试环境，生产环境禁止使用|
|1|每次事务提交时，都会调用`fsync()`将 Binlog 强制刷入物理磁盘|最高，事务提交成功后，Binlog 一定持久化到磁盘|较低|核心业务场景，与`innodb_flush_log_at_trx_commit=1`配合，称为**双 1 配置**，是 MySQL 最高级别的数据强一致配置，金融级场景强制开启|
|N（N>1）|每提交 N 个事务，才调用一次`fsync()`批量刷盘|中等，操作系统宕机时最多丢失 N 个事务的 Binlog|中等|非核心业务，可配置为 100/1000 平衡性能与安全性|

### 7.3.4 Binlog 生命周期与管理

#### 写入流程

1. 事务执行过程中，生成的 Binlog 先写入 Binlog Cache（每个连接独享一个 Binlog Cache，大小由`binlog_cache_size`控制）；
2. 事务提交时，将 Binlog Cache 中的内容一次性写入磁盘的 Binlog 文件，事务提交完成后清空 Binlog Cache；
3. 写入完成后，通知 Dump 线程将新的 Binlog 内容发送给连接的从库，实现主从同步。

#### 生命周期管理

1. **文件滚动**：Binlog 文件达到`max_binlog_size`阈值（默认 1GB）、MySQL 重启、执行`FLUSH LOGS`命令时，自动滚动生成新的编号文件，如`mysql-bin.000001`、`mysql-bin.000002`；
2. **自动清理**：通过参数`expire_logs_days`设置 Binlog 自动过期天数，默认 0（不自动清理），生产环境需配置为备份周期的 2-3 倍，确保备份完成前 Binlog 不会被清理；
3. **手动清理**：通过`PURGE BINARY LOGS TO 'mysql-bin.000100';`手动清理指定文件之前的所有 Binlog，严禁直接通过`rm`命令删除 Binlog 文件，会导致元数据不一致，主从同步中断。

### 7.3.5 Redo Log 与 Binlog 核心区别

|对比维度|Redo Log|Binlog|
|---|---|---|
|所属层级|InnoDB 存储引擎层|MySQL Server 层|
|日志类型|物理日志，记录数据页的字节级修改|逻辑日志，记录数据修改的逻辑操作|
|写入方式|固定大小的环形文件，循环覆盖写|追加写，文件滚动生成，无大小上限|
|核心作用|保障事务持久性，实现崩溃恢复|数据备份恢复、主从复制数据同步|
|覆盖范围|仅记录 InnoDB 引擎的事务修改|记录所有存储引擎的 DDL、DML 操作|
|写入时机|事务执行过程中持续写入，提交时完成刷盘|事务执行过程中写入 Cache，提交时一次性写入文件|
|崩溃恢复|重启时自动重放，恢复未刷盘的已提交事务|不参与崩溃恢复，仅用于备份恢复与主从同步|

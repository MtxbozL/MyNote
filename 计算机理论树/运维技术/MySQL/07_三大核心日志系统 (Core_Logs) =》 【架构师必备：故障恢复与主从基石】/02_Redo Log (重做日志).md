
### 7.2.1 核心定位与 WAL 预写日志机制

- **所属层级**：InnoDB 存储引擎层，仅 InnoDB 支持
- **日志类型**：物理日志，记录数据页的字节级物理修改，而非 SQL 逻辑
- **两大核心作用**：
    
    1. 保障事务的**持久性**，实现崩溃安全（Crash-Safe）：事务提交后，即使数据库崩溃、断电，重启后可通过 Redo Log 重放已提交的事务修改，保证已提交数据不丢失；
    2. 性能优化：将随机磁盘 IO 转为顺序磁盘 IO，事务提交时仅需顺序写入 Redo Log，无需随机刷入脏数据页，脏页由后台线程异步批量刷盘，大幅提升高并发写场景的性能。
    

#### WAL（Write-Ahead Logging）预写日志核心原理

WAL 是 InnoDB 实现持久性的核心机制，其核心准则是**日志先写，数据后刷**：

1. 事务修改数据时，仅修改 Buffer Pool 中的内存数据页（脏页），不会立即将脏页刷入磁盘；
2. 事务提交前，必须先将该事务对应的 Redo Log 持久化到磁盘；
3. 只要 Redo Log 持久化成功，事务就具备了持久性，即使数据库崩溃，重启后可通过 Redo Log 重放修改，不会丢失数据；
4. Buffer Pool 中的脏页，由后台 Master 线程异步批量刷入磁盘，将多次随机写合并为顺序写，极大降低磁盘 IO 开销。

### 7.2.2 存储结构与循环写机制

#### 7.2.2.1 物理存储结构

Redo Log 由一组固定大小的日志文件组成，默认包含两个文件`ib_logfile0`和`ib_logfile1`，存储在 MySQL 数据目录下。

- 核心配置参数`innodb_log_file_size`：单个 Redo Log 文件的大小，默认 48MB，最大支持 4GB / 文件，生产环境推荐配置为 1-4GB，平衡刷盘频率与崩溃恢复时间；
- 核心配置参数`innodb_log_files_in_group`：Redo Log 文件组的数量，默认 2 个，最多 100 个。

#### 7.2.2.2 循环写机制

Redo Log 采用**循环覆盖写**的环形结构，文件首尾相连形成一个环形缓冲区，通过两个核心指针控制写入与回收：

1. **Write Pos**：当前日志写入位置，随日志写入顺时针移动；
2. **Checkpoint LSN**：已刷脏页的日志位置，也是可覆盖空间的起始位置，随 Checkpoint 机制触发顺时针移动。

**核心规则**：

- Write Pos 到 Checkpoint LSN 之间的环形区域，是可写入的空闲空间；
- 当 Write Pos 追上 Checkpoint LSN 时，说明 Redo Log 空间已满，必须强制触发 Checkpoint 刷脏页，回收可覆盖空间，此时会阻塞所有写操作，直到空间释放；
- Checkpoint LSN 之前的日志，对应的脏页已全部刷入磁盘，日志内容可被覆盖；之后的日志，对应的脏页尚未刷盘，不可被覆盖，是崩溃恢复时需要重放的内容。

#### 7.2.2.3 LSN（日志序列号）

LSN 是 Redo Log 的唯一单调递增标识，是 InnoDB 日志管理、数据页版本控制、崩溃恢复的核心依据：

- 每个 Redo Log 记录都有唯一的 LSN，代表日志写入的字节偏移量；
- 每个数据页的页头都会记录该页对应的最新 LSN，标识该页最后一次修改对应的 Redo Log 位置；
- 崩溃恢复时，通过对比数据页的 LSN 与 Redo Log 的 Checkpoint LSN，判断该数据页是否需要重放 Redo Log。

### 7.2.3 核心刷盘策略与性能安全权衡

Redo Log 的刷盘策略直接决定了事务的持久性与写入性能，核心由参数`innodb_flush_log_at_trx_commit`控制，该参数有 3 个可选值，对应不同的刷盘逻辑、安全性与性能表现。

> 关键概念澄清：刷盘指调用`fsync()`系统调用，强制将操作系统 Page Cache 中的日志内容写入物理磁盘，而非仅从 Log Buffer 写入 Page Cache。仅写入 Page Cache 的数据，操作系统宕机时会丢失，只有完成`fsync()`刷入磁盘的数据，才具备崩溃安全性。

|参数值|刷盘执行逻辑|数据安全性|写入性能|适用场景|
|---|---|---|---|---|
|1（默认）|事务提交时，立即将 Log Buffer 中的 Redo Log 写入 OS Page Cache，同时调用`fsync()`强制刷入物理磁盘，事务返回成功|最高，只要事务提交成功，即使数据库 + 操作系统崩溃，数据也绝对不会丢失|最低|金融交易、支付、订单等对数据一致性要求极高的核心业务，生产环境推荐配置|
|0|事务提交时，不做任何刷盘操作，仅由后台 Master 线程每秒将 Log Buffer 中的内容写入 Page Cache 并调用`fsync()`刷盘|最低，崩溃时最多丢失 1 秒的已提交事务数据|最高|日志采集、监控数据等允许少量数据丢失的非核心业务，生产环境不推荐核心业务使用|
|2|事务提交时，立即将 Redo Log 写入 OS Page Cache，但不调用`fsync()`刷盘，由后台线程每秒调用`fsync()`批量刷盘|中等，数据库崩溃不会丢失数据，仅操作系统宕机时最多丢失 1 秒的数据|中等|对性能有要求、可接受极端场景下少量数据丢失的非核心业务|

### 7.2.4 Checkpoint 机制

Checkpoint 机制是 Redo Log 空间回收、脏页持久化、崩溃恢复优化的核心机制，其核心作用是将 Buffer Pool 中的脏页批量刷入磁盘，更新 Checkpoint LSN，回收 Redo Log 的可覆盖空间，同时缩短崩溃恢复的时间。

#### Checkpoint 分类与触发时机

1. **Sharp Checkpoint（尖锐检查点）**
    
    - 触发时机：数据库正常关闭时触发；
    - 执行逻辑：将 Buffer Pool 中所有脏页全部刷入磁盘，将 Checkpoint LSN 更新到最新的 Redo Log 位置；
    - 特性：数据库运行期间不会触发，避免阻塞所有业务操作。
    
2. **Fuzzy Checkpoint（模糊检查点）**
    
    - 触发时机：数据库运行期间的异步检查点，是默认的 Checkpoint 方式，仅批量刷部分脏页，不阻塞业务操作；
    - 核心触发场景：
        
        1. Redo Log 空间不足时，强制触发，优先刷最旧的脏页，快速回收日志空间；
        2. Buffer Pool 内存不足时，淘汰脏页前必须先将其刷入磁盘，触发 Checkpoint；
        3. 系统空闲时，后台 Master 线程每秒异步触发，批量刷脏页；
        4. 定时触发，默认每 7 秒触发一次，保证脏页不会长期停留在内存中。
        
    

### 7.2.5 崩溃恢复核心流程

MySQL 崩溃重启后，InnoDB 会自动执行崩溃恢复流程，保证数据最终一致性，完整流程如下：

1. **加载数据页与 Redo Log**：启动时加载 Buffer Pool，读取 Redo Log 文件，定位到最新的 Checkpoint LSN，仅需重放该位置之后的 Redo Log，无需重放全量日志；
2. **Redo Log 重放（Apply）**：遍历 Checkpoint LSN 之后的所有 Redo Log 记录，对比数据页的 LSN 与日志 LSN，若数据页 LSN 小于日志 LSN，说明该页的修改未刷入磁盘，执行 Redo Log 重放，将数据页恢复到崩溃前的最新状态；
3. **未提交事务回滚**：Redo Log 重放完成后，基于 Undo Log，回滚所有仅处于 Prepare 阶段、未最终提交的事务，保证事务的原子性；
4. **恢复完成**：数据库正常对外提供服务。

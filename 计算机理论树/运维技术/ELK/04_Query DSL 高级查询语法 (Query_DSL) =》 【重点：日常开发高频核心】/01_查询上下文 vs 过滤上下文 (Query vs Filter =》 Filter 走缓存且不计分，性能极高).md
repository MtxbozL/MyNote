## 本章导读

本章是 Elasticsearch 日常开发的高频核心章节，承接第三章的映射与分词规则，聚焦 ES 原生结构化查询语言 Query DSL 的完整体系。核心目标是深入理解查询上下文与过滤上下文的本质区别与性能差异，精通全文检索查询、精确值查询的核心语法与适用场景，熟练掌握 Bool 复合查询的多条件组合能力，建立 ES 查询的性能优化思维。本章内容直接决定了检索的精度、召回率与执行效率，是业务开发、日志检索、面试考察的绝对核心，为后续聚合分析、生产级架构设计奠定查询能力基础。

Query DSL（Domain Specific Language）是 ES 基于 JSON 格式定义的结构化查询语言，完全替代了简单的 URI 查询，支持无限层级的条件嵌套、多维度的匹配规则与相关性控制，是 ES 全文检索能力的唯一标准载体，所有复杂查询场景均基于 Query DSL 实现。

## 4.1 查询上下文 vs 过滤上下文（Query vs Filter Context）

ES 的查询执行分为两大核心上下文，二者的执行逻辑、性能开销、适用场景存在本质差异，是理解 ES 查询机制的基础，也是查询性能优化的核心切入点。

### 4.1.1 核心定义与本质区别

|对比维度|查询上下文（Query Context）|过滤上下文（Filter Context）|
|---|---|---|
|核心判断逻辑|「匹配程度」：判断文档与查询条件的**相关性有多高**|「是否匹配」：判断文档与查询条件**是否符合**，仅返回二元结果（是 / 否）|
|相关性计分|会为匹配的文档计算`_score`相关性得分，用于结果排序|不计算任何相关性得分，`_score`固定为 0，不影响结果排序|
|结果缓存|无内置缓存，每次查询均需重新执行匹配与计分逻辑|内置 Bitset 缓存机制，高频过滤条件会被缓存，重复查询性能提升 10~100 倍|
|执行优先级|后置执行，仅对过滤后的结果集执行计分与排序|前置执行，优先过滤掉无效文档，缩小后续查询的处理范围|
|适用场景|全文检索、需要按相关性排序的业务查询场景|固定条件的范围筛选、状态匹配、数据去重等无排序需求的过滤场景|

### 4.1.2 查询上下文（Query Context）

查询上下文的核心是**相关性排序**，ES 会为每个匹配的文档计算相关性得分，得分越高的文档排名越靠前，完全适配全文检索的核心需求 —— 优先返回最符合用户预期的结果。

相关性得分的核心计算逻辑基于**TF-IDF 算法**（词频 - 逆文档频率），8.x + 版本默认采用 BM25 算法，核心考量三个维度：

1. 词频（TF）：查询词条在目标文档中出现的次数，次数越多，相关性越高；
2. 逆文档频率（IDF）：查询词条在整个索引的所有文档中出现的频率，出现次数越少，词条的区分度越高，相关性权重越大；
3. 字段长度归一化：目标字段的长度越短，词条在字段中的权重越高，相关性越高。

查询上下文的触发规则：当查询语句被放置在**需要计算相关性的查询节点**中时，自动进入查询上下文，典型场景包括 Bool 查询的`must`、`should`子句。

### 4.1.3 过滤上下文（Filter Context）

过滤上下文的核心是**精准筛选与极致性能**，仅做二元匹配判断，不计算相关性，同时内置高效的缓存机制，是 ES 查询性能优化的第一抓手。

#### 核心缓存机制原理

ES 会为每个执行过的过滤条件维护一个**Bitset（位图）缓存**，核心逻辑如下：

1. 过滤条件首次执行时，ES 会遍历索引中的所有文档，生成一个与文档 ID 一一对应的二进制位图，匹配的文档标记为 1，不匹配的标记为 0；
2. 该位图会被缓存到内存中，缓存有效期为索引段（Segment）的生命周期，仅当段发生合并时才会重建；
3. 后续相同的过滤条件再次执行时，无需重新遍历文档，直接复用内存中的位图，瞬间完成匹配结果的筛选，性能提升极为显著。

过滤上下文的触发规则：当查询语句被放置在**专用过滤节点**中时，自动进入过滤上下文，典型场景包括 Bool 查询的`filter`、`must_not`子句，以及`constant_score`查询的 filter 节点。

### 4.1.4 生产级最佳实践

1. **过滤优先原则**：所有固定的、无需相关性排序的筛选条件，必须放置在过滤上下文中，优先缩小结果集范围，减少后续查询的计算开销；
2. **最小结果集原则**：将能过滤掉最多文档的条件前置，最大化缩小处理范围，提升整体查询效率；
3. **避免无效计分**：非全文检索场景，无需相关性排序的查询，全部使用过滤上下文，避免无意义的相关性计算；
4. **缓存复用原则**：高频使用的过滤条件保持语句完全一致，确保缓存命中，避免动态参数导致的缓存失效。

### 4.1.5 上下文对比示例

以电商商品查询场景为例，同样的条件分别放置在查询上下文与过滤上下文，执行逻辑与性能完全不同：

```bash
# 1. 全部放在查询上下文：会为所有匹配文档计算相关性得分，无缓存
GET /goods_info/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "goods_name": "华为手机" } },
        { "term": { "status": "on_sale" } },
        { "range": { "price": { "gte": 1000, "lte": 5000 } } }
      ]
    }
  }
}

# 2. 拆分上下文：全文检索放查询上下文，固定过滤条件放过滤上下文，性能最优
GET /goods_info/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "goods_name": "华为手机" } }
      ],
      "filter": [
        { "term": { "status": "on_sale" } },
        { "range": { "price": { "gte": 1000, "lte": 5000 } } }
      ]
    }
  }
}
```

第二个示例中，`status`与`price`的固定条件放在过滤上下文，会被缓存，同时优先过滤掉非上架、价格不符合的文档，仅对剩余的有效文档执行全文检索与相关性计算，性能远高于第一个示例。

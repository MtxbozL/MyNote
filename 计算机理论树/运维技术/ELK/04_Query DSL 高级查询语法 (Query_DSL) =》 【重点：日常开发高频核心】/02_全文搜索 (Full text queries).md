
全文搜索查询是 ES 全文检索能力的核心载体，专门针对`text`类型字段设计，核心特性是：**会对查询文本执行与索引时完全一致的分词处理**，将查询文本拆分为多个词条后，再去倒排索引中执行匹配，完全适配长文本、自然语言的检索需求。

### 4.2.1 match 查询（基础分词查询）

match 查询是最基础、最常用的全文查询类型，是全文检索的入门核心，支持单字段的分词匹配，可通过参数灵活控制匹配规则。

#### 4.2.1.1 核心执行流程

1. 检查字段类型：若目标字段为`text`类型，获取字段配置的分词器；
2. 分词处理：对查询文本执行分词，生成词条列表；
3. 词条匹配：遍历词条列表，在倒排索引中查找包含每个词条的文档；
4. 相关性计分：根据匹配的词条数量、频率等维度，计算每个文档的相关性得分；
5. 结果返回：按得分从高到低返回匹配的文档。

#### 4.2.1.2 标准语法与核心参数

```bash
GET /<索引名称>/_search
{
  "query": {
    "match": {
      "<目标字段名>": {
        "query": "<查询文本>",
        "<参数名>": "<参数值>"
      }
    }
  }
}
```

核心参数详解：

|参数名|可选值|核心作用|默认值|
|---|---|---|---|
|`operator`|`or`/`and`|控制词条的匹配逻辑：`or`表示匹配任意一个词条即可，`and`表示必须匹配所有词条|`or`|
|`minimum_should_match`|数字 / 百分比|控制至少需要匹配的词条数量，可精准控制召回率|与`operator`联动|
|`fuzziness`|`AUTO`/ 数字|开启模糊匹配，允许词条的最大编辑距离（错别字容错），`AUTO`为推荐值|关闭|
|`boost`|数字|字段权重提升，数值越大，该字段匹配对相关性得分的影响越大|1.0|
|`lenient`|`true`/`false`|忽略字段类型不匹配导致的异常，如对数值字段传入字符串查询|`false`|

#### 4.2.1.3 典型示例与避坑指南

1. **基础默认查询（operator=or）**
    
    ```bash
    GET /goods_info/_search
    {
      "query": {
        "match": {
          "goods_desc": "华为 5G 手机"
        }
      }
    }
    ```
    
    分词后生成`[华为,5G,手机]`三个词条，只要文档包含任意一个词条即可匹配，匹配词条越多，得分越高。
    
2. **严格匹配（operator=and）**
    
    ```bash
    GET /goods_info/_search
    {
      "query": {
        "match": {
          "goods_desc": {
            "query": "华为 5G 手机",
            "operator": "and"
          }
        }
      }
    }
    ```
    
    必须同时包含`华为`、`5G`、`手机`三个词条的文档才会被匹配，召回率更低，精度更高。
    
3. **精准控制匹配度（minimum_should_match）**
    
    ```bash
    GET /goods_info/_search
    {
      "query": {
        "match": {
          "goods_desc": {
            "query": "华为 5G 拍照 手机",
            "minimum_should_match": "75%"
          }
        }
      }
    }
    ```
    
    共 4 个词条，75% 即至少匹配 3 个词条，平衡召回率与精度，是生产环境的常用优化手段。
    
4. **模糊匹配容错（fuzziness）**
    
    ```bash
    GET /goods_info/_search
    {
      "query": {
        "match": {
          "goods_name": {
            "query": "华伟手机",
            "fuzziness": "AUTO"
          }
        }
      }
    }
    ```
    
    可容错错别字 “华伟”，匹配到 “华为手机”，`AUTO`模式会根据词条长度自动调整最大编辑距离，是模糊查询的最佳实践，避免固定数值导致的无关结果匹配。
    

#### 核心避坑指南

- 严禁对`keyword`类型字段使用 match 查询，`keyword`类型不分词，仅存储完整词条，match 查询的分词结果无法匹配，导致查询结果异常；
- 模糊匹配`fuzziness`最大允许值为 2，超过 2 会导致召回结果泛滥，性能急剧下降，生产环境严禁设置超过 2 的数值；
- `minimum_should_match`优先级高于`operator`，同时设置时以`minimum_should_match`为准。

### 4.2.2 match_phrase 查询（短语查询）

match_phrase 短语查询是 match 查询的进阶类型，核心解决 match 查询无法控制词条顺序的问题，专门用于精准的短语匹配场景。

#### 4.2.2.1 核心原理与 match 的本质区别

- match 查询仅要求文档包含查询词条，不限制词条的顺序与位置，只要词条存在即可匹配；
- match_phrase 查询要求**文档中的词条必须与查询词条的顺序完全一致，且词条之间的相对位置符合要求**，仅当完整的短语匹配时才会命中，彻底解决跨词条的错误匹配问题。

#### 4.2.2.2 核心参数：slop 容错距离

`slop`是 match_phrase 查询的核心参数，定义了**词条之间允许的最大移动次数（编辑距离）**，用于实现短语的容错匹配，默认值为 0（不允许任何位置偏移，必须严格顺序匹配）。

slop 参数的计算规则：查询词条与目标文档中的词条，要达到顺序一致所需的最小移动次数。例如：

- 查询文本：`大数据 应用`，分词后词条位置为`[0:大数据, 1:应用]`；
- 目标文档：`大数据技术与应用`，分词后词条位置为`[0:大数据, 1:技术, 2:与, 3:应用]`；
- 要让`应用`从位置 1 移动到位置 3，需要移动 2 次，因此`slop`至少设置为 2 才能匹配该文档。

#### 4.2.2.3 标准语法与示例

1. **严格短语匹配（slop=0，默认）**
    
    ```bash
    GET /news_info/_search
    {
      "query": {
        "match_phrase": {
          "content": "中华人民共和国"
        }
      }
    }
    ```
    
    仅当文档中包含连续的、顺序完全一致的 “中华人民共和国” 短语时才会命中，不会匹配到 “中华人民的共和国” 这类拆分后的文本。
    
2. **容错短语匹配（slop 设置）**
    
    ```bash
    GET /goods_info/_search
    {
      "query": {
        "match_phrase": {
          "goods_name": {
            "query": "华为 手机 5G",
            "slop": 2
          }
        }
      }
    }
    ```
    
    可匹配到 “华为 5G 手机”、“华为手机 5G 全网通” 等文本，允许词条之间有 2 次移动的容错空间，平衡匹配精度与灵活性。

#### 适用场景

- 精准的标题匹配、品牌名匹配、专有名词匹配；
- 文章、合同等文本的固定短语检索；
- 日志场景的异常堆栈、固定报错信息的精准检索。

### 4.2.3 multi_match 查询（多字段匹配查询）

multi_match 查询是 match 查询的多字段扩展版本，支持在**多个字段上同时执行全文检索**，并合并计算相关性得分，是多字段检索场景的核心方案，例如商品的标题、副标题、详情同时匹配关键词，文章的标题、内容同时检索等。

#### 4.2.3.1 标准语法与核心参数

```bash
GET /<索引名称>/_search
{
  "query": {
    "multi_match": {
      "query": "<查询文本>",
      "fields": ["<字段1>", "<字段2>^<权重>", "..."],
      "type": "<匹配类型>",
      "<其他参数>": "<参数值>"
    }
  }
}
```

核心参数详解：

1. `fields`：指定需要匹配的字段列表，支持通配符（如`*_name`），支持通过`^`符号设置字段权重，例如`goods_name^3`表示该字段的权重是默认值的 3 倍，匹配对得分的影响更大；
2. `type`：定义多字段的匹配与得分计算模式，是 multi_match 的核心配置，直接影响匹配结果与排序；
3. 其他参数：继承 match 查询的所有核心参数，包括`operator`、`minimum_should_match`、`fuzziness`等。

#### 4.2.3.2 核心 type 匹配类型

生产环境最常用的三种匹配类型，各自的核心逻辑与适用场景如下：

|类型|核心逻辑|适用场景|默认值|
|---|---|---|---|
|`best_fields`|最佳字段模式：分别在每个字段上执行匹配，取**单个字段的最高得分**作为文档的最终得分，其他字段的匹配仅做加分补充|多个字段存在竞争关系，优先返回某个字段匹配度最高的文档，例如标题与内容的检索，标题匹配度高的文档应排在前面|是|
|`most_fields`|最多字段模式：分别在每个字段上执行匹配，将**所有字段的得分相加**作为文档的最终得分|多个字段表达同一语义，例如同一字段配置了多个分词器，匹配的字段越多，文档相关性越高|否|
|`cross_fields`|跨字段模式：将所有指定字段视为一个大字段，在所有字段中匹配查询词条，要求所有词条至少在任意一个字段中匹配|跨字段的短语匹配，例如`first_name`和`last_name`两个字段，查询用户全名，词条分散在两个字段中|否|

#### 4.2.3.3 典型生产示例

1. **基础多字段匹配（best_fields 默认模式）**
    
    ```bash
    GET /goods_info/_search
    {
      "query": {
        "multi_match": {
          "query": "华为5G拍照手机",
          "fields": ["goods_name^3", "goods_subtitle^2", "goods_desc"],
          "operator": "and"
        }
      }
    }
    ```
    
    同时在商品名称、副标题、详情三个字段匹配，商品名称权重最高，副标题次之，详情最低，优先返回商品名称匹配度最高的文档，符合用户检索习惯。
    
2. **跨字段匹配（cross_fields 模式）**
    
    ```bash
    GET /user_info/_search
    {
      "query": {
        "multi_match": {
          "query": "张三 北京",
          "fields": ["user_name", "city", "address"],
          "type": "cross_fields",
          "operator": "and"
        }
      }
    }
    ```
    
    要求 “张三” 和 “北京” 两个词条，至少在任意一个字段中匹配，可命中`user_name=张三`且`city=北京`的文档，解决 best_fields 模式下无法跨字段强制匹配所有词条的问题。


Bool 复合查询是 ES 查询的核心骨架，是唯一支持多条件组合、上下文嵌套的查询类型，生产环境中 99% 的复杂查询均基于 Bool 查询实现。其核心能力是将多个查询子句通过布尔逻辑组合，实现无限层级的条件嵌套，同时精准控制查询上下文与过滤上下文的分离。

### 4.4.1 Bool 查询核心架构

Bool 查询通过四大子句实现布尔逻辑的组合，四大子句分为两类，分别对应查询上下文与过滤上下文，核心结构如下：

```bash
GET /<索引名称>/_search
{
  "query": {
    "bool": {
      "must": [ ... ],          // 查询上下文：必须匹配，计分
      "should": [ ... ],        // 查询上下文：选择性匹配，计分
      "filter": [ ... ],        // 过滤上下文：必须匹配，不计分，可缓存
      "must_not": [ ... ]       // 过滤上下文：必须不匹配，不计分，可缓存
    }
  }
}
```

Bool 查询的核心执行逻辑：

1. 优先执行`filter`与`must_not`过滤子句，通过缓存快速筛选出符合基础条件的文档，缩小结果集范围；
2. 对过滤后的结果集，执行`must`与`should`查询子句，计算相关性得分；
3. 合并所有子句的匹配结果，按最终得分从高到低返回文档。

### 4.4.2 四大子句完整详解

#### 4.4.2.1 must 子句

- 核心逻辑：数组中的所有查询条件**必须全部匹配**，文档才会被命中，等价于逻辑与（AND）；
- 上下文：查询上下文，所有匹配的条件都会参与相关性得分计算，匹配的条件越多，得分越高；
- 适用场景：必须满足的核心业务条件，且需要参与相关性排序，例如全文检索的关键词匹配；
- 示例：
    
    ```bash
    # 必须同时匹配商品名称包含“华为手机”，且商品描述包含“5G”
    GET /goods_info/_search
    {
      "query": {
        "bool": {
          "must": [
            { "match": { "goods_name": "华为手机" } },
            { "match": { "goods_desc": "5G" } }
          ]
        }
      }
    }
    ```

#### 4.4.2.2 filter 子句

- 核心逻辑：数组中的所有查询条件**必须全部匹配**，文档才会被命中，等价于逻辑与（AND）；
- 上下文：过滤上下文，不计算任何相关性得分，结果可缓存，性能极高；
- 适用场景：必须满足的固定过滤条件，无需参与相关性排序，例如状态匹配、时间范围、数值筛选等，是性能优化的核心抓手；
- 示例：
    
    ```bash
    # 必须是上架状态，且价格在1000-5000元之间，不计分，可缓存
    GET /goods_info/_search
    {
      "query": {
        "bool": {
          "filter": [
            { "term": { "status.keyword": "on_sale" } },
            { "range": { "price": { "gte": 1000, "lte": 5000 } } }
          ]
        }
      }
    }
    ```

#### 4.4.2.3 must_not 子句

- 核心逻辑：数组中的所有查询条件**必须全部不匹配**，文档才会被命中，等价于逻辑非（NOT）；
- 上下文：过滤上下文，不计算任何相关性得分，结果可缓存；
- 适用场景：需要排除的无效数据，例如已删除的文档、无效状态、黑名单数据等；
- 示例：
    
    ```bash
    # 排除已删除、已下架的商品，不计分，可缓存
    GET /goods_info/_search
    {
      "query": {
        "bool": {
          "must_not": [
            { "term": { "is_deleted": true } },
            { "term": { "status.keyword": "off_sale" } }
          ]
        }
      }
    }
    ```

#### 4.4.2.4 should 子句

- 核心逻辑：数组中的查询条件**选择性匹配**，等价于逻辑或（OR），匹配的条件越多，文档的相关性得分越高；
- 上下文：查询上下文，参与相关性得分计算；
- 核心规则（最高频踩坑点）：`should`子句的最小匹配数量由`minimum_should_match`参数控制，默认值分两种场景：
    
    1. 【场景 1】Bool 查询中**只有 should 子句**，无`must`/`filter`子句：`minimum_should_match`默认值为 1，即至少匹配一个 should 条件，文档才会被命中；
    2. 【场景 2】Bool 查询中**同时存在 must/filter 子句**：`minimum_should_match`默认值为 0，即无需匹配任何 should 条件，文档也会被命中，should 条件仅影响相关性得分，不影响结果集。
    
- 适用场景：可选的加分条件，用于优化相关性排序，或多条件的或逻辑匹配；
- 示例 1：仅 should 子句，默认至少匹配一个
    
    ```bash
    # 匹配品牌是华为 OR 小米 OR 苹果的商品，至少匹配一个
    GET /goods_info/_search
    {
      "query": {
        "bool": {
          "should": [
            { "term": { "brand.keyword": "华为" } },
            { "term": { "brand.keyword": "小米" } },
            { "term": { "brand.keyword": "苹果" } }
          ]
        }
      }
    }
    ```
    
- 示例 2：与 must 子句共存，显式设置 minimum_should_match
    
    ```bash
    # 必须是上架状态、价格1000-5000，且至少匹配品牌是华为/小米，或内存大于8G
    GET /goods_info/_search
    {
      "query": {
        "bool": {
          "must": [
            { "term": { "status.keyword": "on_sale" } },
            { "range": { "price": { "gte": 1000, "lte": 5000 } } }
          ],
          "should": [
            { "terms": { "brand.keyword": ["华为", "小米"] } },
            { "range": { "ram_gb": { "gte": 8 } } }
          ],
          "minimum_should_match": 1
        }
      }
    }
    ```
    

### 4.4.3 Bool 查询的嵌套用法

Bool 查询支持无限层级的嵌套，每个子句中都可以再次嵌套 Bool 查询，实现任意复杂的布尔逻辑组合，适配所有复杂业务查询场景。

#### 生产级嵌套示例

电商场景复杂查询需求：

1. 必须条件：商品名称包含 “手机”，上架状态，价格在 1000-10000 元；
2. 排除条件：已删除、二手商品；
3. 可选条件：（品牌是华为且处理器是麒麟）OR（品牌是苹果且处理器是 A 系列），满足该条件的文档优先排序；
4. 加分条件：商品描述包含 “5G”、“拍照”，匹配越多得分越高。

对应的嵌套 Bool 查询实现：

```bash
GET /goods_info/_search
{
  "query": {
    "bool": {
      "must": [
        { "match": { "goods_name": "手机" } },
        {
          "bool": {
            "should": [
              {
                "bool": {
                  "must": [
                    { "term": { "brand.keyword": "华为" } },
                    { "match": { "cpu": "麒麟" } }
                  ]
                }
              },
              {
                "bool": {
                  "must": [
                    { "term": { "brand.keyword": "苹果" } },
                    { "match": { "cpu": "A系列" } }
                  ]
                }
              }
            ]
          }
        }
      ],
      "filter": [
        { "term": { "status.keyword": "on_sale" } },
        { "range": { "price": { "gte": 1000, "lte": 10000 } } }
      ],
      "must_not": [
        { "term": { "is_deleted": true } },
        { "term": { "is_second_hand": true } }
      ],
      "should": [
        { "match": { "goods_desc": "5G" } },
        { "match": { "goods_desc": "拍照" } }
      ]
    }
  }
}
```

### 4.4.4 相关性得分计算规则

Bool 查询的最终相关性得分`_score`，由查询上下文中的`must`和`should`子句共同决定，核心规则如下：

1. `filter`和`must_not`子句不参与得分计算，无论多少个过滤条件，都不会影响`_score`值；
2. `must`子句的得分是所有匹配条件的得分之和；
3. `should`子句的得分是所有匹配条件的得分之和，按匹配比例加权；
4. 最终得分是`must`与`should`子句得分的加权总和，可通过`boost`参数调整不同条件的权重占比。

### 4.4.5 生产级最佳实践

1. **上下文严格分离**：全文检索、需要排序的条件放`must/should`查询上下文；固定过滤、状态匹配、范围筛选的条件放`filter/must_not`过滤上下文，最大化利用缓存，提升性能；
2. **should 子句显式控制**：只要 Bool 查询中存在`must/filter`子句，必须显式设置`minimum_should_match`，避免默认值导致的查询结果不符合预期；
3. **避免过度嵌套**：Bool 嵌套层级不宜过深，否则会导致查询解析开销过大，性能下降，超过 3 层嵌套需优化查询逻辑；
4. **大结果集优化**：仅需筛选数据、无需排序的场景，可设置`"track_total_hits": false`关闭总命中数统计，同时设置`size: 0`不返回文档详情，大幅提升大索引的查询性能；
5. **禁用大跨度 OR 查询**：避免在 should 子句中放置数十个以上的 term 条件，否则会导致查询性能急剧下降，应优先使用 terms 查询替代。


ES 字段类型是 Mapping 的核心组成部分，决定了字段的存储格式、索引方式、计算能力与分词规则。ES 提供全场景覆盖的数据类型体系，核心分为四大类：**核心基础类型、复杂结构类型、地理空间类型、专用特殊类型**。本节聚焦目录指定的核心类型，重点讲解面试必考与生产高频使用的类型，明确选型规则与底层实现差异。

### 3.2.1 text 与 keyword 类型（面试必考）

text 与 keyword 是 ES 最核心的字符串类型，二者的本质区别在于**是否执行分词处理**，分别适配全文检索与精确匹配两大核心场景，是数据建模的最基础选型，也是面试高频考点。

#### 3.2.1.1 核心本质区别与选型规则

|对比维度|text 类型|keyword 类型|
|---|---|---|
|核心特性|分词类型，写入时执行文本分析，将字符串拆分为多个独立词条（Term），构建倒排索引|不分词类型，写入时不执行分词，将整个字符串作为单个完整词条构建倒排索引|
|核心能力|支持全文检索，可匹配字符串中的任意关键词|支持精确匹配、前缀匹配、排序、聚合分析，不支持全文检索|
|Doc Values|默认关闭，因分词后词条量级极大，列式存储开销极高，强行开启会导致集群内存溢出|默认开启，原生支持列式存储，排序、聚合性能极高|
|适用场景|文章内容、日志详情、商品描述、用户评论等需要全文检索的长文本场景|用户名、手机号、订单号、状态码、标签、IP 地址等需要精确匹配、排序、聚合的短文本场景|
|索引开销|高，分词后生成多个词条，倒排索引体积大|低，仅生成单个词条，索引体积小，查询性能更高|

#### 3.2.1.2 多字段映射（fields）生产级最佳实践

在实际业务中，同一个字符串字段往往同时存在全文检索与精确匹配 / 聚合的需求，例如商品名称：既需要支持用户输入关键词做全文检索，又需要支持按商品名称做排序、聚合统计。

针对该场景，ES 提供**fields 多字段映射**能力，允许为同一个字段定义多个子字段，分别设置不同的类型与分词规则，无需冗余存储数据，是生产环境的标准用法。

**标准示例**：

```bash
PUT /goods_info
{
  "mappings": {
    "properties": {
      "goods_name": {
        "type": "text",
        "analyzer": "ik_max_word",
        "fields": {
          "keyword": {
            "type": "keyword",
            "ignore_above": 256
          }
        }
      }
    }
  }
}
```

- 主字段`goods_name`为 text 类型，使用 IK 分词器，支持全文检索；
- 子字段`goods_name.keyword`为 keyword 类型，支持精确匹配、排序、聚合；
- `ignore_above: 256`：限制字段最大长度，超过 256 个字符的字符串不会被索引，避免超长文本导致的索引膨胀，是 keyword 类型的推荐配置。

#### 3.2.1.3 核心避坑指南

1. 严禁对 text 类型执行`term`精确匹配查询：text 类型写入时已被分词，`term`查询的关键词不会被分词，无法匹配到分词后的词条，导致查询结果为空或不符合预期；
2. 严禁对超长文本使用 keyword 类型：keyword 类型适合长度小于 256 字符的短文本，超长文本会导致单个词条过大，倒排索引性能急剧下降，甚至触发索引写入失败；
3. 严禁随意开启 text 类型的 fielddata：text 类型默认关闭 Doc Values，若需排序聚合需开启 fielddata，而 fielddata 基于 JVM 堆内存构建，海量数据下会触发频繁 Full GC 甚至 OOM，生产环境强烈不推荐，优先使用多字段映射方案。

### 3.2.2 数值类型

ES 提供全系列数值类型，覆盖整数、浮点数、高精度数值场景，核心选型原则是：**在满足业务取值范围的前提下，优先使用占用空间更小的类型**，以降低存储开销、提升索引与查询性能。

#### 3.2.2.1 整数类型

|类型|占用存储空间|取值范围|适用场景|
|---|---|---|---|
|byte|1 字节|-128 ~ 127|状态码、年龄、等级等极小数值场景|
|short|2 字节|-32768 ~ 32767|商品库存、订单数量等中小数值场景|
|integer|4 字节|-2^31 ~ 2^31-1|用户 ID、商品 ID 等常规数值场景|
|long|8 字节|-2^63 ~ 2^63-1|雪花算法主键、毫秒级时间戳等超大数值场景|

#### 3.2.2.2 浮点与高精度数值类型

|类型|核心特性|适用场景|
|---|---|---|
|float|4 字节单精度浮点数，精度约 6~7 位有效数字|精度要求不高的浮点场景，如温度、湿度等传感器数据|
|double|8 字节双精度浮点数，精度约 15~17 位有效数字|高精度浮点场景，如地理坐标、科学计算数据|
|scaled_float|缩放浮点类型，基于 long 类型存储，通过缩放因子控制精度|【生产推荐】金融金额、商品价格等固定精度场景，如设置`scaling_factor=100`，可将 99.99 存储为 9999 整数，大幅降低存储开销，避免浮点数精度丢失|
|half_float|2 字节半精度浮点数，精度约 3~4 位有效数字|极低精度、海量数据的浮点场景，如机器学习特征值、评分数据|

#### 3.2.2.3 核心最佳实践

1. 禁止无差别使用 long 与 double 类型：大部分业务场景中，integer、short、scaled_float 完全可满足需求，盲目使用大类型会导致存储开销翻倍，查询性能下降；
2. 金融场景强制使用 scaled_float：彻底规避浮点数精度丢失问题，同时提升存储与计算效率；
3. 无需检索的数值字段设置`doc_values: false`：仅需存储、无需排序聚合的数值字段，关闭 Doc Values 可节省大量存储开销。

### 3.2.3 日期类型（date）

ES 的 date 类型用于处理时间相关数据，底层基于 UTC 时间的毫秒级时间戳（long 类型）存储，支持多种时间格式的写入与解析，是日志分析、时序数据统计的核心类型。

#### 3.2.3.1 核心特性与格式配置

1. **核心存储原理**：无论写入何种格式的时间数据，ES 都会将其转换为 UTC 时区的毫秒级时间戳存储到磁盘，查询时再根据指定时区转换为对应格式的时间字符串，彻底解决时区不一致问题。
    
2. **format 格式配置**：通过`format`参数指定支持的时间格式，支持多格式组合，用`||`分隔，ES 会按顺序匹配解析。
    
    - 默认格式：`strict_date_optional_time||epoch_millis`，支持 ISO 标准日期格式（如`yyyy-MM-dd'T'HH:mm:ss.SSSZ`）与毫秒级时间戳；
    - 自定义格式示例：国内业务常用`yyyy-MM-dd HH:mm:ss`格式，标准配置如下：
        
        ```bash
        "create_time": {
          "type": "date",
          "format": "yyyy-MM-dd HH:mm:ss||yyyy-MM-dd||epoch_millis||epoch_second",
          "time_zone": "Asia/Shanghai"
        }
        ```
        
    - 补充格式：`epoch_second`支持秒级时间戳，适配大部分业务系统的时间存储格式。
    
3. **date_nanos 类型**：纳秒级日期类型，支持微秒、纳秒级精度的时间数据，适用于链路追踪、高频交易等高精度时间场景，底层基于纳秒级时间戳存储，常规日志场景无需使用。
    

#### 3.2.3.2 核心避坑指南

1. 时区问题：ES 默认使用 UTC 时区，国内业务需显式指定`time_zone: Asia/Shanghai`，避免时间查询偏移 8 小时；
2. 格式匹配：写入的时间字符串必须严格匹配`format`中定义的格式，否则会触发写入失败，动态映射下会被推导为 text 类型，导致时间范围查询失效；
3. 时间戳类型：秒级时间戳必须指定`epoch_second`格式，否则会被解析为毫秒级时间戳，导致时间异常。

### 3.2.4 复杂类型

ES 支持嵌套结构的 JSON 数据存储，提供`object`与`nested`两种核心复杂类型，分别适配不同的嵌套数据场景，解决 JSON 对象数组的存储与查询问题。

#### 3.2.4.1 object 对象类型

object 是 ES 默认的嵌套对象类型，无需显式指定类型，写入 JSON 嵌套对象时，ES 会自动推导为 object 类型，支持无限层级的嵌套结构。

1. **底层存储原理**：ES 会将嵌套对象扁平化处理，将嵌套字段的完整路径作为字段名，存储为扁平的键值对结构，适配 Lucene 的一维文档存储模型。
    
    - 示例文档：
        
        ```json
        {
          "user": {
            "name": "zhangsan",
            "age": 25,
            "address": {
              "city": "Beijing",
              "district": "Haidian"
            }
          }
        }
        ```
        
    - 底层扁平化存储结构：
        
        ```json
        {
          "user.name": "zhangsan",
          "user.age": 25,
          "user.address.city": "Beijing",
          "user.address.district": "Haidian"
        }
        ```
        
    
2. **适用场景**：单对象嵌套，无数组结构的嵌套数据，如用户基本信息、商品详情等一对一的嵌套场景。
3. **核心缺陷**：无法正确处理**对象数组**场景，扁平化处理会丢失对象内字段的关联关系，导致跨对象的错误匹配。

#### 3.2.4.2 nested 嵌套类型

nested 类型是 object 类型的升级版，专门解决对象数组的关联查询问题，是 ES 处理一对多嵌套数据的唯一方案。

1. **object 类型的核心坑点**：对象数组的扁平化问题
    
    - 示例对象数组文档：
        
        ```json
        {
          "order_list": [
            {
              "order_id": "1001",
              "status": "paid",
              "amount": 100
            },
            {
              "order_id": "1002",
              "status": "unpaid",
              "amount": 200
            }
          ]
        }
        ```
        
    - object 类型底层扁平化结果：
        
        ```json
        {
          "order_list.order_id": ["1001", "1002"],
          "order_list.status": ["paid", "unpaid"],
          "order_list.amount": [100, 200]
        }
        ```
        
    - 错误查询场景：查询`status=unpaid AND amount=100`的文档，object 类型会匹配到该文档，但实际上不存在同时满足这两个条件的订单对象，完全违背查询预期。
    
2. **nested 类型的底层实现原理**
    
    nested 类型会将数组中的每个嵌套对象，作为一个独立的**隐藏文档**单独存储，保留每个对象内字段的关联关系，同时与根文档绑定，保证查询性能。查询时必须通过 nested 专用查询语法，强制在同一个隐藏嵌套对象内完成字段匹配，彻底解决跨对象的错误匹配问题。
    
3. **标准定义语法**
    
    ```bash
    PUT /user_order
    {
      "mappings": {
        "properties": {
          "user_id": {
            "type": "keyword"
          },
          "order_list": {
            "type": "nested",
            "properties": {
              "order_id": { "type": "keyword" },
              "status": { "type": "keyword" },
              "amount": { "type": "integer" },
              "create_time": { "type": "date", "format": "yyyy-MM-dd HH:mm:ss" }
            }
          }
        }
      }
    }
    ```
    
4. **适用场景与限制**
    
    - 适用场景：一对多的对象数组场景，如用户订单列表、商品 SKU 列表、文章评论列表等，需要对嵌套对象内的多个字段做关联查询的场景；
    - 核心限制：
        
        1. 嵌套对象的查询、聚合必须使用专用的 nested 语法，常规查询语法无法生效；
        2. 单个文档的嵌套对象数量默认上限为 10000，单个索引的嵌套字段数量默认上限为 50，避免过度嵌套导致的性能下降；
        3. 嵌套文档的写入、更新会触发整个根文档的重新索引，写入性能低于扁平结构。

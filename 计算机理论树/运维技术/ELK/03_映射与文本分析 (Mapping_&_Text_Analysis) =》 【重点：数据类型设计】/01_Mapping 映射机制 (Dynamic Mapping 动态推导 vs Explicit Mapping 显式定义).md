## 本章导读

本章是 Elasticsearch 数据建模的核心章节，承接第二章的文档 CRUD 操作，聚焦 ES 数据结构的定义规则与全文检索的底层实现逻辑。核心目标是掌握 Mapping 映射机制的设计原理与规范用法，精通 ES 全量核心数据类型的选型规则与底层实现，深入理解文本分析（分词）的三段式架构与执行流程，熟练掌握内置分词器与中文分词器 IK Analyzer 的配置与生产级用法。本章内容直接决定了倒排索引的构建质量、检索的精度与召回率、集群的读写性能与存储成本，是日常开发的高频核心与面试必考重点，为后续 Query DSL 高级查询、聚合分析奠定核心数据模型基础。

## 3.1 Mapping 映射机制

Mapping 是定义索引结构与字段处理规则的元数据集合，等价于关系型数据库中的表结构定义，核心作用是规范索引中字段的名称、数据类型、索引方式、分词规则、存储格式、聚合能力等核心属性，是 ES 数据建模的唯一载体。

Mapping 的核心设计目标是：**在写入阶段规范数据格式，在检索阶段保障匹配精度，在存储阶段优化资源开销，在计算阶段提升执行效率**。ES 提供两种 Mapping 管理模式，分别为动态映射（Dynamic Mapping）与显式映射（Explicit Mapping），二者适用于不同的业务场景。

### 3.1.1 动态映射（Dynamic Mapping）

动态映射是 ES 的原生自动推导机制，当向索引中写入包含未定义字段的文档时，ES 会自动根据字段值的 JSON 数据类型，推导并更新 Mapping 定义，无需用户提前创建索引与字段规则，大幅降低了初期使用门槛。

#### 3.1.1.1 核心类型推导规则

ES 8.x 版本默认的动态类型推导规则如下表所示，严格遵循 JSON 数据类型与 ES 字段类型的映射逻辑：

|JSON 原始数据类型|ES 默认推导的字段类型|补充规则|
|---|---|---|
|布尔值（true/false）|boolean|严格匹配，无额外推导|
|整数（如 100）|long|若数值超出 long 范围，推导为 keyword|
|浮点数（如 99.99）|float|高精度浮点数推导为 double|
|符合日期格式的字符串|date|默认匹配 ISO 标准日期格式、yyyy/MM/dd 等格式，可通过`dynamic_date_formats`自定义|
|普通字符串|text + keyword 多字段|主字段为 text 类型（支持全文检索），自动创建`.keyword`子字段为 keyword 类型（支持精确匹配、排序、聚合）|
|对象（Object）|object|嵌套对象递归执行动态推导|
|数组|按数组内第一个非空元素的类型推导|要求数组内所有元素类型一致，否则推导失败|

#### 3.1.1.2 动态映射的核心配置

通过`dynamic`参数可控制动态映射的行为，是 Mapping 的顶级配置项，支持 4 种取值，覆盖全场景业务需求：

1. **`dynamic: true`（默认值）**
    
    - 核心规则：写入新字段时，自动推导字段类型并更新 Mapping，新字段可被索引、检索、聚合；
    - 适用场景：测试环境、需求快速迭代的原型场景、数据结构不固定的探索性分析场景。
    
2. **`dynamic: false`**
    
    - 核心规则：写入新字段时，不更新 Mapping，新字段不会被索引，无法通过常规查询检索，但会完整保存在`_source`元数据中；
    - 适用场景：需保留原始全量数据，但无需对新增字段做检索的场景，避免无效字段占用索引资源。
    
3. **`dynamic: strict`（严格模式，生产环境推荐）**
    
    - 核心规则：写入新字段时，直接抛出异常，拒绝文档写入，强制要求所有字段必须提前在 Mapping 中显式定义；
    - 适用场景：生产环境核心业务索引，严格规范数据结构，避免脏数据、无效字段写入，保障索引稳定性与可维护性。
    
4. **`dynamic: runtime`（运行时模式）**
    
    - 核心规则：写入新字段时，将其定义为运行时字段（Runtime Field），不做索引存储，仅在查询时动态计算解析；
    - 适用场景：临时新增的低频率查询字段、需动态修改计算逻辑的字段，无需重建索引即可调整字段规则，节省存储开销。
    

#### 3.1.1.3 动态映射的优缺点

- **核心优势**：零配置快速上手，无需提前设计数据结构，适配非结构化、半结构化数据的快速接入，大幅降低初期开发成本。
- **核心缺陷**：
    
    1. 类型推导偏差：如数字格式的字符串会被推导为 text 类型，日期格式不匹配会被推导为 text 类型，导致检索、聚合异常；
    2. 索引膨胀：大量无效字段被自动索引，占用磁盘与内存资源，降低集群读写性能；
    3. 数据不一致：同一字段在不同分片写入时推导类型不一致，导致集群分片分配失败、数据无法检索。
    

### 3.1.2 显式映射（Explicit Mapping）

显式映射是指用户在索引创建时，提前手动定义所有字段的名称、数据类型、处理规则与索引配置，是生产环境的标准规范用法。显式映射完全规避了动态映射的不稳定性，可精准控制每个字段的行为，实现性能、精度、存储的最优平衡。

#### 3.1.2.1 显式映射的标准创建语法

ES 8.x 版本彻底废弃 Type 机制，显式映射的标准创建方式分为两种，严格遵循 RESTful API 规范：

1. **创建索引时同步定义 Mapping**（推荐用法）
    
    ```bash
    PUT /<索引名称>
    {
      "settings": {
        // 索引配置项，如分片数、副本数、刷新间隔等
        "number_of_shards": 3,
        "number_of_replicas": 1
      },
      "mappings": {
        // Mapping核心定义
        "properties": {
          "字段1名称": {
            "type": "字段类型",
            // 字段级配置项，如分词器、是否索引、格式等
          },
          "字段2名称": {
            "type": "字段类型",
            "properties": {
              // 嵌套对象的子字段定义
            }
          }
        }
      }
    }
    ```
    
2. **为已存在的索引新增 / 修改 Mapping**
    
    ```bash
    PUT /<索引名称>/_mapping
    {
      "properties": {
        // 新增字段定义，不可修改已存在字段的类型，否则会导致已有数据无法索引
        "新增字段名称": {
          "type": "字段类型"
        }
      }
    }
    ```

#### 3.1.2.2 核心约束与最佳实践

1. **字段类型不可修改原则**：字段一旦创建并写入数据，其核心`type`类型不可修改。ES 的倒排索引基于字段类型构建，修改类型会导致已有数据的索引完全失效。若需修改字段类型，唯一方案是重建索引并做数据全量重索引（Reindex）。
2. **仅可新增字段**：对已存在的索引，仅可通过`_mapping`接口新增字段，不可删除已有字段、修改已有字段的核心配置。
3. **生产环境强制规范**：核心业务索引必须使用`strict`严格模式，禁用动态映射，所有字段提前显式定义，避免脏数据写入与索引膨胀。
4. **最小字段原则**：仅对需要检索、聚合的字段定义索引，无需检索的字段设置`index: false`，减少存储开销与索引构建成本。

## 本章导读

本章承接第一章的架构认知，核心目标是建立 Elasticsearch（以下简称 ES）面向文档的核心数据模型认知，明确 ES 与关系型数据库的概念映射边界，理解 Type 机制的废弃逻辑，熟练掌握基于 RESTful API 的文档全生命周期 CRUD 操作，精通批量写入的核心优化方案与分布式场景下的乐观锁并发控制机制，为后续映射配置、高级查询等进阶内容奠定操作与理论基础。

## 2.1 ES 核心数据模型与名词映射

ES 是 **面向文档的分布式非关系型数据库** ，以 JSON 格式的文档为最小数据单元，采用层级化的数据模型组织数据，核心层级为：集群（Cluster）→ 节点（Node）→ 索引（Index）→ 文档（Document）→ 字段（Field）。

为降低学习门槛，行业内普遍将 ES 核心概念与关系型数据库（RDBMS）做映射对应，但需明确二者仅为逻辑相似，底层实现与设计理念存在本质差异，具体映射关系与核心定义如下：

| ES 核心概念      | 对应 RDBMS 概念   | 核心定义与本质说明                                                                                      |
| ------------ | ------------- | ---------------------------------------------------------------------------------------------- |
| 索引（Index）    | 数据库（Database） | ES 中数据管理的顶级逻辑单元，是 **相同或相似结构文档的集合** ，承担数据存储、检索、聚合的边界管理，所有操作均基于索引展开。注意：绝非对应 RDBMS 的表，该认知为典型新手误区。 |
| 文档（Document） | 行（Row）        | ES 中最小的数据操作单元，以 JSON 格式序列化存储，对应 RDBMS 中的一行记录。每个文档具备全局唯一的文档 ID（`_id`），同时包含系统内置元数据与用户自定义业务数据。    |
| 字段（Field）    | 列（Column）     | 文档中的单个属性键值对，对应 RDBMS 中的一列。每个字段具备明确的数据类型，决定了该字段的索引方式、存储格式、分词规则与计算能力，是 ES 数据模型的最小组成单元。           |

### 文档核心元数据字段

ES 为每个文档自动生成内置元数据字段，用于标识文档的归属、版本、位置等核心信息，是后续 CRUD 操作、并发控制、路由分发的核心依据，核心元数据如下：

- `_index`：文档所属的索引名称；
- `_id`：文档的唯一标识，可用户自定义或 ES 自动生成；
- `_source`：文档的原始 JSON 业务数据，默认全量存储；
- `_version`：文档的版本号，每次写入 / 更新 / 删除操作自增，为旧版本并发控制字段；
- `_seq_no`：文档的操作序列号，严格递增，7.x + 版本核心并发控制字段；
- `_primary_term`：主分片生命周期标识，主分片切换时自增，与`_seq_no`协同实现分布式乐观锁；
- `_routing`：文档的分片路由值，默认使用`_id`计算。

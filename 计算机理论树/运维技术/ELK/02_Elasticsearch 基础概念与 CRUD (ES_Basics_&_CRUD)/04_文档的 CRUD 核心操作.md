
文档是 ES 数据操作的最小单元，本节全面覆盖文档的创建、读取、更新、删除全生命周期操作，严格遵循 8.x + 版本的 API 规范，明确各操作的语义、语法、底层逻辑与最佳实践。

### 2.4.1 文档创建与全量替换

ES 提供两种文档创建模式，分别为 **指定文档 ID 的 PUT 创建** 与 **自动生成 ID 的 POST 创建**，同时 PUT 操作具备全量替换的语义。

#### 1. 指定文档 ID 的创建与全量替换

- **核心语法**：
    
    ```bash
    PUT /<目标索引名>/_doc/<自定义文档ID>
    {
      "字段1": "值1",
      "字段2": "值2",
      ...
    }
    ```
    
- **语义规则**：
    
    1. 若指定的文档 ID 不存在，执行 **创建操作**，生成新文档；
    2. 若指定的文档 ID 已存在，执行 **全量替换操作**，覆盖整个`_source`文档内容，原文档中未出现在请求体中的字段会被彻底删除，同时文档版本号与`_seq_no`自增。
    
- **幂等性**：严格幂等，多次执行相同 ID 的 PUT 请求，最终文档状态完全一致。
- **适用场景**：具备业务唯一主键的业务数据写入，如用户信息、订单数据等。

#### 2. 自动生成文档 ID 的创建

- **核心语法**：
    
    ```bash
    POST /<目标索引名>/_doc/
    {
      "字段1": "值1",
      "字段2": "值2",
      ...
    }
    ```
    
- **语义规则**：ES 自动生成全局唯一的 22 位 UUID 作为文档 ID，仅执行创建操作，不会出现替换覆盖。
- **幂等性**：非幂等，多次执行相同请求会生成多个不同 ID 的重复内容文档。
- **适用场景**：无业务唯一主键的时序类、事件类数据写入，如系统日志、访问记录、监控数据等。

#### 核心注意事项

1. 全量替换会触发文档的完整重新索引，倒排索引全量更新，若仅需修改部分字段，严禁使用全量替换，应使用局部更新 API；
2. 自定义文档 ID 建议使用业务主键，避免随机 ID 导致的分片路由不均衡；
3. 自动生成的 ID 具备天然的时序性，可提升 Lucene 段合并的效率，更适合高并发写入的日志场景。

### 2.4.2 文档局部更新

局部更新 API 用于仅修改文档中的指定字段，保留未指定的原有字段，彻底规避全量替换的字段丢失与性能损耗问题，是生产环境文档更新的首选方案。

#### 1. 基础局部更新

- **核心语法**：
    
    ```bash
    POST /<目标索引名>/_update/<文档ID>
    {
      "doc": {
        "待更新字段1": "新值1",
        "待更新字段2": "新值2",
        ...
      }
    }
    ```
    
- **底层实现逻辑**：ES 先检索原文档，合并请求体中的字段与原文档字段，再执行全量索引操作，内置并发冲突校验与重试机制。
- **语义规则**：
    
    1. 仅更新`doc`对象中指定的字段，原文档中未指定的字段完全保留；
    2. 若指定的字段不存在，会自动新增该字段，符合 ES 动态映射特性；
    3. 仅当文档 ID 存在时生效，不存在则返回 404 错误。
    
- **幂等性**：对同一字段的相同值更新操作多次执行，结果一致，具备幂等性。
- **示例**：
    
    ```bash
    POST /user_info/_update/1
    {
      "doc": {
        "age": 26,
        "address": "Beijing China"
      }
    }
    ```

- 该操作仅将`age`字段更新为 26，新增`address`字段，原有的`username`、`email`字段完全保留。

#### 2. 脚本化高级更新

ES 内置 Painless 脚本语言，支持在局部更新中实现复杂的条件更新、数值计算、字段操作等逻辑，是局部更新的高级能力。

- **核心语法**：
    
    ```bash
    POST /<目标索引名>/_update/<文档ID>
    {
      "script": {
        "source": "Painless脚本代码",
        "lang": "painless",
        "params": {
          "参数名": "参数值"
        }
      }
    }
    ```
    
- **典型示例 1：数值自增**
    
    ```bash
    POST /user_info/_update/1
    {
      "script": {
        "source": "ctx._source.age += params.increment",
        "lang": "painless",
        "params": {
          "increment": 1
        }
      }
    }
    ```
    
- **典型示例 2：条件更新**
    
    ```bash
    POST /user_info/_update/1
    {
      "script": {
        "source": "if (ctx._source.age < 30) { ctx._source.age += 1 } else { ctx.op = 'none' }",
        "lang": "painless"
      }
    }
    ```

- 其中`ctx.op = 'none'`表示不执行任何操作，避免无意义的版本号自增与索引重建。

#### 核心注意事项

1. 局部更新底层仍会触发文档的重新索引，仅减少了用户侧的查询 - 修改 - 写入的操作成本，并非物理层面的字段级更新；
2. 脚本更新需严格保证幂等性，避免重复执行导致的数据错误；
3. 生产环境需严格控制 Painless 脚本的复杂度，禁止执行高耗时、高 CPU 占用的脚本，避免集群性能下降。

### 2.4.3 文档查询

本节聚焦基于文档 ID 的精确查询，是 ES 中最快的查询方式，直接路由到对应分片，无需分布式检索，具备实时性特性。全文检索相关的高级查询将在后续章节详细讲解。

#### 1. 单文档精确查询

- **核心语法**：
    
    ```bash
    GET /<目标索引名>/_doc/<文档ID>
    ```
    
- **核心特性**：
    
    1. 实时性：不受 ES refresh 间隔影响，会优先读取 translog 中的未刷新数据，确保返回最新版本的文档，区别于全文检索的近实时特性；
    2. 高性能：直接通过文档 ID 路由到对应分片，无需倒排索引检索，延迟在毫秒级；
    3. 响应核心字段：`found`（布尔值，标识文档是否存在）、`_source`（文档原始 JSON 内容）、文档元数据。
    
- **字段过滤**：可通过`_source`参数指定返回的字段，减少网络开销，示例：
    
    ```bash
    # 仅返回username和age字段
    GET /user_info/_doc/1?_source=username,age
    
    # 不返回文档内容，仅返回元数据
    GET /user_info/_doc/1?_source=false
    ```

#### 2. 文档存在性检查

- **核心语法**：
    
    ```bash
    HEAD /<目标索引名>/_doc/<文档ID>
    ```
    
- **语义规则**：仅检查文档是否存在，不返回任何响应体，通过 HTTP 状态码标识结果：
    
    - 200 OK：文档存在；
    - 404 Not Found：文档不存在。
    
- **核心优势**：无响应体，网络开销极低，适合仅需判断文档是否存在的场景。

#### 3. 多文档批量查询（mget API）

mget API 支持在单次 HTTP 请求中批量查询多个不同 ID、不同索引的文档，大幅减少多次请求的网络 IO 开销，提升查询效率。

- **核心语法（跨索引）**：
    
    ```bash
    GET /_mget
    {
      "docs": [
        {
          "_index": "user_info",
          "_id": "1"
        },
        {
          "_index": "user_info",
          "_id": "2"
        },
        {
          "_index": "order_info",
          "_id": "1001"
        }
      ]
    }
    ```
    
- **同索引简化语法**：
    
    ```bash
    GET /user_info/_mget
    {
      "ids": ["1", "2", "3"]
    }
    ```

### 2.4.4 文档删除操作

#### 1. 单文档精确删除

- **核心语法**：
    
    ```bash
    DELETE /<目标索引名>/_doc/<文档ID>
    ```
    
- **语义规则**：
    
    1. 若文档存在，执行删除操作，返回 200 OK，`result`字段为`deleted`，文档版本号与`_seq_no`自增；
    2. 若文档不存在，返回 404 Not Found，无副作用。
    
- **幂等性**：严格幂等，多次删除同一 ID 的文档，最终状态一致。
- **底层实现原理**：ES 的删除操作并非立即从磁盘物理移除文档，而是将文档标记为已删除状态（写入`.del`文件），标记后的文档无法被检索到，仅在后续的 Lucene 段合并（Segment Merge）过程中，才会将标记为删除的文档物理清除，释放磁盘空间。

#### 2. 按查询条件批量删除（_delete_by_query API）

支持通过查询条件批量删除匹配的文档，语法如下：

```bash
POST /<目标索引名>/_delete_by_query
{
  "query": {
    // 匹配条件，与Query DSL语法完全一致
  }
}
```

- **示例**：删除所有年龄大于等于 30 的用户文档
    
    ```bash
    POST /user_info/_delete_by_query
    {
      "query": {
        "range": {
          "age": {
            "gte": 30
          }
        }
      }
    }
    ```
    
- **注意事项**：该 API 基于近实时检索生效，仅能删除已完成 refresh 操作的文档，未刷新的文档无法匹配到。

### 2.4.5 批量操作（_bulk API）

_bulk API 是 ES 提供的批量操作核心接口，支持在单次 HTTP 请求中执行多个创建、更新、删除操作，是**提升写入吞吐量的核心手段**，生产环境大规模写入必须使用_bulk API，严禁循环调用单文档操作接口。

#### 1. 核心语法规范

- **基础语法**：
    
    ```bash
    POST /_bulk
    // 或指定默认索引 POST /<目标索引名>/_bulk
    ```
    
- **请求体格式**：必须采用**NDJSON（换行分隔 JSON）** 格式，严格遵循以下规则：
    
    1. 每组操作由两行组成：第一行为操作元数据行，定义操作类型、目标索引、文档 ID 等信息；第二行为可选的文档内容行，仅`index`、`create`、`update`操作需要，`delete`操作无需内容行；
    2. 每行必须以换行符`\n`结尾，包括最后一行；
    3. 禁止使用格式化的多行 JSON，否则会触发解析错误。
    
- **支持的操作类型**：
    
| 操作类型     | 核心语义                          |
| -------- | ----------------------------- |
| `index`  | 创建或全量替换文档，与 PUT 单文档语义一致       |
| `create` | 仅创建文档，若文档 ID 已存在，返回版本冲突错误，不覆盖 |
| `update` | 局部更新文档，与 POST 单文档_update 语义一致 |
| `delete` | 删除文档，仅需元数据行                   |

#### 2. 完整操作示例

```bash
POST /_bulk
{"index": {"_index": "user_info", "_id": "2"}}
{"username": "lisi", "age": 30, "email": "lisi@example.com"}
{"create": {"_index": "user_info", "_id": "3"}}
{"username": "wangwu", "age": 28, "email": "wangwu@example.com"}
{"update": {"_index": "user_info", "_id": "1"}}
{"doc": {"age": 27}}
{"delete": {"_index": "user_info", "_id": "2"}}
```

#### 3. 核心特性与最佳实践

1. **原子性规则**：_bulk API 中的每个操作独立执行，单个操作的失败不会影响其他操作的执行，响应体中会返回每个操作的执行结果与错误信息；不存在全请求的事务原子性，不会因单个操作失败回滚其他操作。
2. **性能优化核心**：批量操作将多次 HTTP 请求合并为一次，大幅降低网络往返开销与线程调度成本，合理的批量配置可将写入吞吐量提升 10~100 倍。
3. **批量大小规范**：单批次请求体大小建议控制在**5MB~15MB**之间，单批次文档数量根据单文档大小调整，通常建议 1000~5000 条 / 批次，严禁单批次超过 100MB，否则会导致节点内存溢出、GC 压力过大、集群阻塞。
4. **并发优化**：生产环境应采用多线程异步发送 bulk 请求，而非单线程串行发送，充分利用集群的分布式写入能力，同时需控制并发数，避免压垮集群。
5. **路由优化**：若所有操作均针对同一个索引，可在 URL 中直接指定索引名，元数据行中可省略`_index`字段，简化请求体。

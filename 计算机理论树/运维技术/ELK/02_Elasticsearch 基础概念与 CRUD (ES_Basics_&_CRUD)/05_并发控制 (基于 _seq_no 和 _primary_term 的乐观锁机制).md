
ES 作为分布式系统，多客户端同时对同一文档执行写入、更新操作时，会出现典型的 “写覆盖” 并发问题：两个客户端同时读取同一文档，修改不同字段后写回，后写入的操作会覆盖先写入的操作，导致数据丢失。

针对该问题，ES 采用**乐观锁机制**实现并发控制，7.x + 版本全面采用`_seq_no` + `_primary_term`的组合方案，替代旧版本的`_version`方案，彻底解决了主分片切换场景下的版本冲突问题。

### 2.5.1 乐观锁核心设计理念

乐观锁的核心思想是**假设并发冲突概率较低，不进行加锁阻塞**，避免悲观锁带来的性能损耗与死锁风险；仅在写入操作执行时，通过版本号校验判断文档是否被其他线程修改，若校验失败则拒绝操作，由客户端处理冲突重试逻辑，完全适配 ES 分布式高并发的写入场景。

### 2.5.2 核心控制字段定义

1. **`_seq_no`（序列号）**：由主分片分配的严格递增整数，每对文档执行一次写入、更新、删除操作，该文档的`_seq_no`就会自增 1，用于标识文档的修改版本，记录文档最后一次修改的操作序列号。
2. **`_primary_term`（主分片术语）**：递增整数，当主分片发生重新选举、主分片切换时，`_primary_term`会自增 1，用于标识主分片的生命周期，确保版本号在分布式场景下的全局唯一性，彻底解决主分片切换后旧主分片版本号冲突的问题。

### 2.5.3 乐观锁的使用方式

在执行更新、删除操作时，通过 URL 参数`if_seq_no`和`if_primary_term`指定预期的序列号与主分片术语，ES 会执行严格的版本校验：

- 校验通过：文档当前的`_seq_no`与`_primary_term`和传入参数完全一致，执行操作，`_seq_no`自增；
- 校验失败：文档已被其他线程修改，返回`409 Conflict`冲突错误，拒绝执行操作，避免写覆盖。

#### 完整使用示例

1. **查询文档，获取当前版本信息**
    
    ```bash
    GET /user_info/_doc/1
    ```
    
    响应核心元数据：
    
    ```json
    {
      "_index": "user_info",
      "_id": "1",
      "_version": 3,
      "_seq_no": 5,
      "_primary_term": 1,
      "found": true,
      "_source": {
        "username": "zhangsan",
        "age": 27,
        "email": "zhangsan@example.com"
      }
    }
    ```
    
2. **执行带乐观锁校验的更新操作**
    
    ```bash
    POST /user_info/_update/1?if_seq_no=5&if_primary_term=1
    {
      "doc": {
        "age": 28
      }
    }
    ```
    
3. **执行结果**
    
    - 若无其他线程修改该文档，校验通过，更新成功，`_seq_no`自增为 6；
    - 若其他线程已修改该文档，`_seq_no`已更新为 6，请求返回 409 冲突错误，操作被拒绝。
    

#### 批量操作的并发控制支持

_bulk API 同样支持`if_seq_no`与`if_primary_term`参数，实现批量操作的并发控制，示例：

```bash
POST /_bulk
{"update": {"_index": "user_info", "_id": "1", "if_seq_no": 6, "if_primary_term": 1}}
{"doc": {"age": 29}}
```

### 2.5.4 最佳实践与注意事项

1. **冲突处理逻辑**：出现 409 冲突错误时，正确的处理流程是重新读取文档的最新`_seq_no`与`_primary_term`，重新构建请求后重试，需设置最大重试次数，避免无限重试导致的集群压力。
2. **与旧版本`_version`的区别**：`_version`仅能标识单分片内的文档版本，主分片切换时可能出现版本号重复，导致校验失效；`_seq_no` + `_primary_term`实现了全局唯一的版本标识，是 7.x + 版本唯一推荐的并发控制方案。
3. **适用场景**：乐观锁适用于并发冲突概率较低的业务数据更新场景；对于无更新需求的高并发日志、时序数据写入，无需使用乐观锁，避免不必要的性能开销。
4. **幂等性保障**：结合乐观锁机制，可实现更新操作的严格幂等性，避免重复请求导致的数据错误，是生产环境分布式写入的核心保障。

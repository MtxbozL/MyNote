## 本章导读

本章核心目标是建立 Elastic Stack（原 ELK 技术栈）的完整认知框架，明确技术栈的演进逻辑、各核心组件的定位与协同边界，掌握其底层搜索引擎基石 Apache Lucene 的核心数据结构与实现原理，完成标准化学习环境的部署，为后续全链路技术学习奠定理论与实践基础。

## 1.1 Elastic Stack 的定义与演进

### 1.1.1 核心定义

Elastic Stack 是由 Elastic 公司维护的开源分布式数据处理技术栈，面向全场景结构化、半结构化与非结构化数据的**采集、清洗、存储、搜索、分析与可视化**全链路处理，广泛应用于日志实时分析、全链路可观测性、全文检索、安全情报分析、业务数据统计等企业级场景。

### 1.1.2 传统 ELK Stack 架构

传统 ELK Stack 是 Elastic Stack 的前身，名称取自三大核心组件的首字母缩写：**Elasticsearch、Logstash、Kibana**。该架构形成了 “数据清洗 - 存储检索 - 可视化” 的基础闭环，解决了早期日志集中管理与检索的核心需求，但存在原生架构缺陷：

- 数据采集层能力缺失，仅能通过 Logstash 实现数据采集，无专门的边缘采集组件；
- 边缘节点资源开销过大，Logstash 基于 JVM 运行，在业务服务器等边缘节点部署时内存占用高，无法适配轻量化、物联网、容器化等资源受限场景；
- 架构弹性不足，采集与清洗能力耦合，无法针对采集场景做独立的水平扩展。

### 1.1.3 现代 Elastic Stack 架构演进

为解决传统 ELK 架构的核心痛点，Elastic 公司引入**Beats 轻量级数据采集家族**，形成了现代 Elastic Stack 架构，完成了从 “三段式闭环” 到 “采集 - 清洗 - 存储 - 分析 - 可视化” 全链路闭环的范式升级。

相较于传统 ELK 架构，现代 Elastic Stack 的核心升级点为：

1. 解耦数据采集与数据清洗能力，通过 Beats 实现边缘节点的轻量级原位采集，彻底解决 Logstash 在边缘部署的资源开销痛点；
2. 拓展了数据采集场景，Beats 家族支持日志文件、系统指标、网络流量、审计数据、物联网设备数据等多源异构数据的采集，覆盖全场景数据接入需求；
3. 提升了架构弹性与可用性，采集层可独立水平扩展，支持数据中转与削峰设计，适配高并发、大流量的企业级生产场景。

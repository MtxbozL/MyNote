## 本章导读

本章是 Elastic Stack 全链路架构的 **ETL 核心处理层** ，承接第六章 Beats 家族的边缘数据采集能力，衔接 Elasticsearch 的存储检索能力。核心目标是建立 Logstash 作为集中式数据清洗管道的完整认知，掌握其`Input→Filter→Output`三段式核心处理架构，精通四大核心 Filter 插件的生产级用法，重点突破非结构化日志向结构化数据转换的核心痛点，理解持久化队列（PQ）的高可用保障机制与全链路性能优化方案。本章内容是 Logstash 的灵魂所在，解决多源异构数据的标准化、结构化处理难题，是企业级日志架构、可观测性平台落地的核心环节。

### 7.1.1 Logstash 核心定位与架构价值

Logstash 是 Elastic 官方推出的 **开源分布式 ETL（Extract-Transform-Load）数据处理管道** ，基于 JVM 开发，核心定位是 **集中式、可扩展的海量数据清洗与格式转换引擎** 。

其核心架构价值在于实现了 **数据采集与数据清洗的彻底解耦** ，与 Beats 家族形成完美的能力互补：

- Beats 负责边缘端轻量化数据采集，无清洗能力，仅做数据转发；
- Logstash 负责集中式复杂数据清洗，将多源异构、非结构化、半结构化的原始数据，转换为标准化的结构化数据，为 Elasticsearch 的检索、聚合、可视化奠定数据基础。

Logstash 的核心能力边界：

1. **多源数据接入**：支持文件、Beats、Kafka、数据库、TCP/UDP、消息队列等数百种数据源的接入；
2. **全场景数据转换**：提供数百个 Filter 插件，支持正则抽取、字段处理、时间解析、数据过滤、地理信息解析等任意复杂的 ETL 逻辑；
3. **多目标数据路由**：支持将处理后的数据输出到 Elasticsearch、Kafka、文件、对象存储、数据库等数十个下游目标，支持动态路由与多链路输出；
4. **水平扩展能力**：支持集群化部署，可根据数据流量动态扩缩容，适配 TB 级 / 日的海量数据处理场景。

### 7.1.2 三段式管道核心架构与事件流转模型

Logstash 的核心是 **事件驱动的三段式管道架构** ，所有数据均以`Event`（事件）为最小单元在管道中流转，完整处理流严格遵循`Input → Filter → Output`的顺序执行，无任何流程分支例外。

```plaintext
原始数据 → Input（数据接入，生成Event） → Filter（数据清洗转换，修改Event） → Output（数据路由输出，持久化Event）
```

#### 各阶段核心职责

1. **Input 阶段（数据接入层）**
    
    Input 是 Logstash 管道的入口，核心职责是从各类数据源读取原始数据，将每条数据封装为一个标准化的`Event`事件对象，传递给 Filter 阶段。每个 Input 插件负责一类数据源的接入，内置数据解码、协议解析、连接管理、断点续传等能力，是 Logstash 与上游数据源的对接层。
    
2. **Filter 阶段（数据转换层）**
    
    Filter 是 Logstash 管道的核心灵魂，核心职责是对 Input 生成的 Event 执行清洗、转换、过滤、 enrichment（数据丰富）操作。该阶段可串联多个 Filter 插件，按配置顺序依次执行，实现非结构化文本的结构化抽取、字段的增删改查、数据类型转换、时间标准化、无效数据过滤、地理信息补充等复杂 ETL 逻辑，是决定数据质量的核心环节。
    
3. **Output 阶段（数据输出层）**
    
    Output 是 Logstash 管道的出口，核心职责是将 Filter 处理完成的标准化 Event，路由输出到指定的下游目标，完成数据的持久化或转发。每个 Output 插件负责一类下游目标的对接，内置批量提交、负载均衡、重试机制、压缩传输等能力，是 Logstash 与下游存储系统的对接层。

#### 事件流转核心特性

- **全链路可观测**：每个 Event 在流转过程中，可通过标签、字段标记处理状态，支持异常数据的全链路追踪；
- **条件分支处理**：支持在三个阶段中通过条件判断，实现 Event 的分支处理，例如不同来源的日志执行不同的清洗规则；
- **至少一次交付保障**：结合持久化队列机制，保证每个 Event 至少被处理并输出一次，无数据丢失风险；
- **背压控制**：当下游 ES/Kafka 处理延迟时，自动限流上游 Input 的读取速度，避免内存溢出，保障管道稳定性。

### 7.1.3 Logstash 配置文件语法规范

Logstash 的所有管道逻辑均通过`.conf`格式的配置文件定义，配置文件严格按照三段式架构分为三大配置块，语法基于 Ruby 实现，支持注释、变量、条件判断、循环等逻辑，是 Logstash 的核心开发载体。

#### 标准配置文件结构

```ruby
# 1. Input配置块：定义数据源接入规则，必填
input {
  # 一个或多个Input插件配置
  <插件类型> {
    <参数名> => <参数值>
    ...
  }
}

# 2. Filter配置块：定义数据清洗规则，可选（无清洗需求可省略）
filter {
  # 一个或多个Filter插件配置，按书写顺序依次执行
  <插件类型> {
    <参数名> => <参数值>
    ...
  }
}

# 3. Output配置块：定义数据输出规则，必填
output {
  # 一个或多个Output插件配置
  <插件类型> {
    <参数名> => <参数值>
    ...
  }
}
```

#### 核心语法规则

1. **块结构**：`input`/`filter`/`output`三大块为顶级配置块，插件配置嵌套在对应块中，使用`{}`包裹层级结构；
2. **键值对语法**：插件参数采用`key => value`格式，`=>`为赋值符号，区别于 JSON 的冒号；
3. **数据类型**：支持字符串（双引号包裹）、数字、布尔值（true/false，小写）、数组、哈希表等原生数据类型；
4. **字段引用**：通过`%{[字段名]}`格式引用 Event 中的字段值，支持嵌套字段引用`%{[object][filed]}`；
5. **注释**：使用`#`标识单行注释，不支持多行注释，注释行必须以`#`开头；
6. **大小写敏感**：插件名称、参数名、字段名均大小写敏感，必须严格匹配插件官方定义。

### 7.1.4 条件判断与分支处理逻辑

Logstash 支持类 Ruby 的条件判断语法，可在`filter`和`output`块中实现 Event 的分支处理，是实现多源数据差异化清洗、动态路由的核心能力。

#### 标准条件判断语法

```ruby
if <条件表达式1> {
  # 条件1满足时执行的插件配置
} else if <条件表达式2> {
  # 条件2满足时执行的插件配置
} else {
  # 所有条件都不满足时执行的插件配置
}
```

#### 核心比较运算符

|运算符|含义|典型示例|
|---|---|---|
|`==`|等于|`if [env] == "prod"`|
|`!=`|不等于|`if [http_status] != 200`|
|`<`/`<=`|小于 / 小于等于|`if [response_time] > 1000`|
|`>`/`>=`|大于 / 大于等于|`if [bytes] >= 1024`|
|`=~`|正则匹配|`if [message] =~ /^ERROR/`|
|`!~`|正则不匹配|`if [message] !~ /^DEBUG/`|
|`in`|包含于数组|`if [status] in ["400", "404", "500"]`|
|`not in`|不包含于数组|`if [service] not in ["test", "demo"]`|
|`and`/`or`/`!`|逻辑与 / 或 / 非|`if [env] == "prod" and [http_status] >= 500`|

#### 字段存在性判断

通过`[字段名]`直接判断字段是否存在且非空，是生产环境高频用法：

```ruby
# 判断client_ip字段存在且非空
if [client_ip] {
  geoip {
    source => "client_ip"
  }
}

# 判断字段不存在
if ! [user_id] {
  mutate {
    add_tag => ["invalid_data"]
  }
}
```

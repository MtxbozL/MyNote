## 6.1 静态配置的局限性

在第二章中，我们学习了通过`static_configs`手动指定抓取目标的静态配置方式，该方式仅适用于单机、小规模、固定地址的测试环境，在生产环境、云原生场景下存在无法解决的核心局限性：

1. **无法适配动态扩缩容场景**
    
    云原生架构下，Pod、容器实例、弹性云主机的 IP 与生命周期完全动态，扩缩容、重建、漂移是常态。静态配置无法自动同步实例的上下线状态，会导致监控漏采、误采，与服务生命周期完全解耦。
    
2. **大规模集群运维成本极高**
    
    当监控目标规模达到数百、数千个实例时，手动维护静态配置的工作量呈指数级增长，变更易出错，且每次修改都需要触发 Prometheus 热加载，无法实现自动化、无感知的配置更新。
    
3. **无法与基础设施体系联动**
    
    静态配置无法对接 CMDB、服务注册中心、容器编排平台、云厂商 API 等基础设施，无法实现监控配置与服务元数据的自动同步，无法满足企业级的标准化、合规化监控需求。
    
4. **环境一致性难以保障**
    
    开发、测试、预发、生产环境的静态配置差异极大，无法实现配置的模板化、版本化管理，易出现环境间配置不一致导致的监控盲区。
    

静态配置的核心缺陷是**无法适配动态变化的基础设施与服务生命周期**，而动态服务发现正是为解决这一痛点设计，是 Prometheus 成为云原生监控标准的核心能力之一。


Prometheus 以**Pull（拉取）模型为核心，Push（推送）模型为辅**的工作机制，两种模型的核心差异、适用边界与选型策略如下。

### 1.4.1 Pull 模型（核心工作模型）

Pull 模型的核心逻辑是：由 Prometheus Server 主动向监控目标发起 HTTP 请求，拉取暴露的时序指标数据。

#### 核心优势

1. **监控状态可校验**：Prometheus Server 可通过抓取请求的响应状态，直接判断监控目标的存活状态与服务可用性，实现指标采集与存活探测的一体化；
2. **去中心化控制**：抓取策略、频率、目标均由 Prometheus Server 端统一配置，无需在监控目标侧进行复杂的推送配置，降低了客户端的配置复杂度；
3. **故障定位便捷**：当指标采集异常时，可直接通过 curl 等工具访问监控目标的 /metrics 接口，快速验证指标暴露是否正常，降低问题排查成本；
4. **水平扩展友好**：可通过多套 Prometheus Server 实例并行抓取同一批目标，实现采集能力的水平扩展，无推送模型的单点瓶颈问题。

#### 适用场景

适用于长生命周期的监控对象，如主机、容器、中间件、数据库、长期运行的微服务等，此类对象具备固定的网络可达地址与持续运行的 HTTP 指标暴露接口。

### 1.4.2 Push 模型（补充工作模型）

Push 模型的核心逻辑是：由监控目标主动向 Prometheus 生态的 Pushgateway 组件推送指标数据，再由 Prometheus Server 从 Pushgateway 拉取数据。

#### 核心局限性

1. 无法通过推送行为判断监控目标的存活状态，需额外配置存活探测机制；
2. Pushgateway 存在单点瓶颈，若推送网关故障，会导致所有推送类指标的采集中断；
3. 若推送任务异常终止，Pushgateway 会持续缓存过期的指标数据，需手动配置过期清理机制。

#### 适用边界

仅适用于短生命周期、临时运行的批处理任务、定时任务等无法通过 Pull 模型完成采集的场景，不可作为长生命周期服务的核心采集方案。

### 1.4.3 模型选型取舍原则

1. 长生命周期服务、基础设施、中间件等监控对象，**优先采用 Pull 模型**，充分发挥其可靠性与可维护性优势；
2. 仅当监控对象无法适配 Pull 模型时（如运行周期短于抓取间隔、无固定网络地址、防火墙策略限制仅出站访问），采用 Pushgateway 的 Push 模型作为补充；
3. 禁止使用 Push 模型替代 Pull 模型作为核心采集方案，避免引入单点故障与运维复杂度。

## 1.1 监控基础理论

监控是分布式系统、云原生架构可靠性保障的核心基础设施，其核心目标是量化系统状态、提前识别风险、快速定位故障、支撑容量规划与性能优化。本节明确监控体系的核心分类与标准化方法论。

### 1.1.1 白盒监控与黑盒监控

根据监控数据的获取维度与系统可见性，监控体系可分为白盒监控与黑盒监控两类，二者形成互补的监控闭环。

- **白盒监控**：基于被监控系统的内部实现细节，采集系统内部暴露的指标、日志、链路追踪等内部状态数据，实现对系统运行状态的深度观测。其核心优势是可提前暴露潜在风险、定位故障根因，适用于系统内部健康度评估、性能瓶颈分析、业务指标趋势监控。
- **黑盒监控**：以外部用户视角，在不依赖系统内部实现的前提下，对系统对外提供的服务能力进行探测，核心关注服务的可用性、可达性与端到端性能。其核心优势是完全贴合用户真实使用场景，可直接验证服务的 SLA 达成情况，适用于服务存活探测、端口可用性监控、HTTP/HTTPS 接口可用性校验、DNS 解析可用性验证等场景。

### 1.1.2 监控核心方法论与指标体系

行业内形成了三类标准化的监控指标设计方法论，分别适配不同的监控对象与场景，构成了监控体系的理论基石。

#### 1.1.2.1 四个黄金指标

四个黄金指标（Four Golden Signals）由 Google SRE 团队在《Site Reliability Engineering》一书中提出，是分布式服务监控的通用顶层设计规范，覆盖服务运行的核心维度：

1. **延迟（Latency）**：服务处理请求所消耗的时间，需严格区分成功请求与失败请求的延迟，避免错误请求的低延迟拉低整体统计结果，掩盖性能问题。
2. **流量（Traffic）**：系统承载的访问压力，用于量化服务的负载规模，不同场景下的指标载体不同，如 HTTP 服务的 QPS（每秒请求数）、数据库的 TPS（每秒事务数）、消息队列的消息生产 / 消费速率。
3. **错误（Errors）**：请求处理失败的速率，需覆盖显式失败（如 HTTP 5xx 错误码）、隐式失败（如 HTTP 200 但业务逻辑错误）、策略性失败（如请求超时）三类场景，通常以错误率（错误请求数 / 总请求数）作为核心监控指标。
4. **饱和度（Saturation）**：系统资源的满载程度，核心关注资源的使用率与瓶颈，如 CPU 使用率、内存占用率、磁盘空间使用率、连接池耗尽程度，通常需提前设置阈值预警，避免资源耗尽导致服务不可用。

#### 1.1.2.2 USE 方法

USE 方法（Utilization Saturation and Errors Method）由性能优化专家 Brendan Gregg 提出，核心面向**物理资源与基础设施**的监控，适用于主机、服务器、容器、中间件等资源型对象的全维度健康度评估：

1. **使用率（Utilization）**：资源服务于工作负载的时间占比，量化资源的繁忙程度，如 CPU 使用率、磁盘 IO 使用率、网络带宽使用率。
2. **饱和度（Saturation）**：资源无法处理的排队工作负载程度，量化资源的过载状态，如 CPU 平均负载、磁盘 IO 等待队列长度、TCP 连接半连接队列长度。
3. **错误数（Errors）**：资源运行过程中发生的错误事件数量，量化资源的异常状态，如网卡丢包数、磁盘 IO 错误数、内存页错误数。

#### 1.1.2.3 RED 方法

RED 方法由 Weaveworks 团队提出，核心面向**微服务与业务应用**的监控，完全适配云原生微服务架构的可观测性需求，与四个黄金指标形成互补，聚焦服务的用户侧体验：

1. **请求速率（Rate）**：服务每秒接收的请求数量，量化服务的访问规模。
2. **错误率（Errors）**：每秒失败的请求数量，量化服务的异常程度。
3. **请求耗时（Duration）**：请求处理的耗时分布，量化服务的性能表现，通常需关注分位数指标（如 P99、P95、P50），而非仅使用平均值。

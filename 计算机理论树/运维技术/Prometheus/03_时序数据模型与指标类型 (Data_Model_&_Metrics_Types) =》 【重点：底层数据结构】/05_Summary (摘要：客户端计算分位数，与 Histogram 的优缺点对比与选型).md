
Summary 是与 Histogram 互补的分布统计型指标，核心特性为**客户端预计算分位数**，专为高精度分位数统计场景设计，与 Histogram 形成明确的能力边界。

### 3.5.1 核心定义与数据结构

Summary 是一种**客户端预计算分位数**的统计指标，在被监控端的客户端库中直接对原始观测值进行分位数计算，Prometheus 服务端直接抓取预计算结果，无需二次处理。完整的 Summary 指标由 3 组时序构成：

1. `[metric_name]{quantile="φ"}`：**瞬时值（Gauge 类型）**，直接暴露预计算的 φ 分位数的观测值，`quantile`为内置标签，取值为 0 < φ < 1；
2. `[metric_name]_sum`：**累积计数器（Counter 类型）**，记录所有观测值的数值总和；
3. `[metric_name]_count`：**累积计数器（Counter 类型）**，记录所有观测值的总次数。

### 3.5.2 核心特性

1. **客户端分位数计算**：分位数在客户端基于原始观测值直接计算，无插值误差，精度远高于 Histogram；
2. **滑动窗口机制**：Summary 的分位数基于客户端配置的滑动时间窗口计算（默认通常为 10 分钟），仅反映窗口内的观测值分布；
3. **无预定义配置要求**：无需提前预估观测值分布，客户端自动处理任意范围的观测值；
4. **不可聚合特性**：预计算的分位数指标无法跨实例、跨标签维度进行聚合，是 Summary 的核心限制。

### 3.5.3 适用场景

Summary 仅适用于满足以下全部条件的场景：

1. 仅需单实例、单维度的分位数统计，无需跨实例、跨维度聚合；
2. 对分位数精度要求极高，无法接受 Histogram 的线性插值误差；
3. 观测值的分布范围无法提前预估，无法为 Histogram 预定义合理的桶区间。

典型场景包括：单实例离线任务的处理耗时统计、单机应用的高精度性能指标统计、无法预估分布的长尾观测值统计。

## 3.6 Histogram 与 Summary 的深度对比与选型策略

Histogram 与 Summary 均用于观测值的分布统计，但二者的实现原理、能力边界与适用场景存在本质差异，是 Prometheus 学习与面试的核心重点，下表为全维度对比：

|对比维度|Histogram|Summary|
|---|---|---|
|分位数计算位置|Prometheus 服务端，通过`histogram_quantile()`函数计算|被监控端客户端库，预计算完成后暴露|
|计算精度|近似值，基于桶的累积计数线性插值，精度由桶划分决定|精确值，基于原始观测值直接计算，无插值误差|
|跨实例 / 维度聚合|完全支持，可跨多个实例、多个标签维度聚合分位数|完全不支持，预计算的分位数无法进行任何有效聚合|
|预配置要求|必须提前预定义桶的数值区间，需预估观测值分布|无需预定义配置，客户端自动适配任意观测值范围|
|时序基数|由桶的数量决定，桶数量固定则基数固定，开销可控|由预定义的分位数数量决定，通常低于多桶 Histogram|
|存储与计算开销|客户端开销极低，仅需计数；服务端需承担分位数计算开销|客户端需承担分位数计算开销；服务端无额外计算开销|
|时间窗口支持|服务端可灵活指定任意时间窗口计算分位数|仅支持客户端预定义的滑动窗口，服务端无法修改|
|PromQL 适配|支持丰富的聚合与计算操作，灵活性极高|仅可直接查询预计算的分位数，灵活性极低|

### 3.6.1 标准化选型决策树

1. **首要判断条件**：若需要**跨实例、跨标签维度聚合分位数**，**必须选择 Histogram**，Summary 无任何聚合能力，完全不适用该场景；
2. **次要判断条件**：若可**提前预估观测值的分布范围**，优先选择 Histogram，合理划分桶可平衡精度与灵活性，是生产环境 90% 以上场景的首选；
3. **特殊场景选型**：仅当同时满足「无需跨实例聚合」、「对分位数精度要求极高」、「无法预估观测值分布」三个条件时，选择 Summary。

> 生产环境最佳实践：微服务接口性能监控、数据库查询耗时统计等核心场景，**优先使用 Histogram**，其聚合能力是分布式系统监控的核心需求；Summary 仅用于单实例高精度统计的特殊场景。

## 3.1 时序数据核心概念

Prometheus 的本质是一款**面向维度标签的时序数据库（Time Series Database, TSDB）**，其所有采集、存储、查询、告警能力均构建在标准化的时序数据模型之上，该模型是 Prometheus 区别于传统监控系统的核心底层设计。

### 3.1.1 时序数据的数学定义

Prometheus 中，**时序数据（Time Series）** 是一个**按毫秒级 Unix 时间戳单调递增排序的 float64 型数值序列**，数学表达式为：

TS={(t1​,v1​),(t2​,v2​),...,(tn​,vn​)},t1​<t2​<...<tn​

其中：

- tn​ 为数据点的时间戳，精度为毫秒级 Unix 时间戳；
- vn​ 为数据点的观测值，采用 float64 类型，支持`NaN`、`+Inf`、`-Inf`特殊值；
- 每一条时序由**全局唯一的时序标识（Time Series Identifier, TSI）** 确定，不同时序标识对应完全独立的数值序列。

### 3.1.2 时序标识的核心构成

时序标识由**指标名称（Metric Name）** + **标签键值对（Label Key-Value Pairs）** 两部分共同构成，二者的组合全局唯一确定一条时序，表达式为：

TSI=<MetricName,{<LabelKey1​,LabelValue1​>,<LabelKey2​,LabelValue2​>,...,<LabelKeyn​,LabelValuen​>}>

#### 1. 指标名称（Metric Name）

指标名称用于定义指标的核心语义，描述被观测对象的测量目标，必须符合严格的命名规范：

- 仅可包含 ASCII 字母、数字、下划线`_`与冒号`:`；
- 必须以字母或下划线开头，禁止以数字开头；
- 冒号`:`为保留字符，仅用于记录规则（Recording Rule）的预聚合指标命名，禁止在客户端埋点中使用；
- 命名需遵循语义规范，格式为`[应用域]_[测量对象]_[测量维度]_[单位]`，例如`http_request_duration_seconds`、`node_cpu_usage_ratio`。

#### 2. 标签（Label）

标签是 Prometheus 多维度数据模型的核心，用于为指标添加维度属性，实现多视角的聚合、过滤与查询。

- **标签键（Label Key）**：命名规范与指标名称一致，以`__`双下划线开头的标签为系统保留标签，仅在数据处理生命周期内生效，不会持久化存储；
- **标签值（Label Value）**：可为任意 UTF-8 字符串，空字符串的标签会被 Prometheus 自动忽略；
- **核心特性**：标签的笛卡尔积决定了指标的时序数量，例如某指标带有`job`（2 个枚举值）、`instance`（3 个枚举值）、`status`（5 个枚举值）3 个标签，其最大时序数量为 2×3×5=30 条。

### 3.1.3 时序基数（Cardinality）

时序基数指 Prometheus 实例中**全局唯一的时序标识总数量**，是决定 Prometheus 性能与稳定性的核心指标。

- 低基数：时序总数在万级以内，为健康状态；
- 中基数：时序总数在十万级，需优化标签设计；
- 高基数：时序总数在百万级及以上，会导致内存占用激增、TSDB 写入性能下降、PromQL 查询延迟飙升，甚至触发 OOM 进程崩溃。

高基数问题的核心根源为不合理的标签设计，例如在标签中加入用户 ID、订单 ID、请求 ID 等唯一标识类高基数标签值，是生产环境必须严格规避的核心风险。

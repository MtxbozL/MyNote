
PromQL 提供了丰富的内置函数，用于实现时序数据的计算、转换、聚合与分析，本节聚焦生产环境与面试核心的四大类函数，深度拆解其实现原理、数学逻辑与选型规范。

### 4.4.1 速率计算函数

速率计算函数是 PromQL 最核心的函数组，专为**Counter 类型指标**设计，用于将累积的 Counter 值转换为单位时间内的增长速率，是 QPS、错误率、吞吐量等核心指标计算的基础。核心函数包括`rate()`、`irate()`、`increase()`。

#### 4.4.1.1 `rate()` 函数

##### 核心定义与数学原理

`rate()`函数计算区间向量中 Counter 指标在**时间窗口内的平均每秒增长速率**，是生产环境最常用的速率计算函数。

核心数学逻辑：

1. 计算时间窗口内 Counter 的总增长量，自动识别并处理 Counter 重置（进程重启导致 Counter 归零）：当检测到窗口内数值下降时，自动判定为重置，将后续数值加上重置前的最大值，修正增长量；
2. 将总增长量除以时间窗口的总秒数，得到平均每秒增长速率。

##### 语法与标准示例

```bash
rate(<range_vector>)
```

```bash
# 计算HTTP请求的平均每秒QPS（5分钟窗口）
rate(http_requests_total[5m])

# 计算5xx错误请求的每秒错误率
rate(http_requests_total{status=~"5.."}[5m])

# 计算主机CPU用户态使用率
rate(node_cpu_seconds_total{mode="user"}[1m])
```

##### 适用场景与选型规范

- 适用场景：长期趋势监控、告警规则配置、大盘趋势渲染、同比环比分析；
- 核心优势：平滑窗口内的瞬时波动，反映指标的整体变化趋势，避免毛刺触发误告警；
- 窗口选型规范：时间窗口大小必须至少为**2 倍的抓取间隔**，推荐设置为抓取间隔的 2~10 倍，例如 15s 抓取间隔对应 30s~2.5m 窗口，生产环境常用 5m 窗口。

#### 4.4.1.2 `irate()` 函数

##### 核心定义与数学原理

`irate()`函数计算区间向量中 Counter 指标在**时间窗口内最后两个连续数据点的瞬时每秒增长速率**，是瞬时峰值监控的专用函数。

核心数学逻辑：仅使用窗口内最新的两个数据点，计算二者的差值与时间差的比值，得到瞬时每秒速率，同样会处理 Counter 重置。

##### 语法与标准示例

```bash
irate(<range_vector>)
```

```bash
# 计算HTTP请求的瞬时QPS（5分钟窗口）
irate(http_requests_total[5m])

# 实时监控接口的瞬时错误率
irate(http_requests_total{status=~"5.."}[1m])
```

##### 适用场景与选型规范

- 适用场景：实时大盘、峰值性能监控、高频波动指标的瞬时状态展示；
- 核心优势：精准反映指标的最新瞬时状态，不会被历史峰值平滑，可捕捉到瞬间的流量波动；
- 核心限制：不适合用于告警规则与长期趋势分析，瞬时波动易触发误告警，无法反映整体趋势；
- 窗口选型规范：时间窗口需足够覆盖至少 2 个数据点，无需过大，推荐设置为抓取间隔的 3~5 倍。

#### 4.4.1.3 `increase()` 函数

##### 核心定义与数学原理

`increase()`函数计算区间向量中 Counter 指标在**时间窗口内的总增长量**，本质是`rate() * 窗口总秒数`，同样会自动处理 Counter 重置。

##### 语法与标准示例

```bash
increase(<range_vector>)
```

```bash
# 统计5分钟内的HTTP总请求数
increase(http_requests_total[5m])

# 统计1小时内的5xx错误请求总数
increase(http_requests_total{status=~"5.."}[1h])

# 统计一天内的磁盘读写字节总数
increase(node_disk_read_bytes_total[1d])
```

##### 适用场景与选型规范

- 适用场景：周期内事件总数统计、业务量统计、SLA 达成情况计算；
- 核心约束：仅用于 Counter 类型指标，禁止用于 Gauge 类型指标；
- 窗口选型规范：与`rate()`一致，窗口需至少覆盖 2 个抓取间隔，避免计算结果失真。

#### 4.4.1.4 高频使用误区与禁忌

1. **绝对禁止**对 Gauge 类型指标使用`rate()`/`irate()`/`increase()`函数，该类函数专为单调非递减的 Counter 设计，用于 Gauge 会产生完全错误的结果；
2. **禁止**对`rate()`的输出结果再次使用`rate()`函数，会导致双重速率计算，逻辑完全错误；
3. 禁止使用小于 2 倍抓取间隔的时间窗口，会因数据点不足导致计算无结果；
4. 禁止在告警规则中使用`irate()`，瞬时波动易触发误告警，生产环境告警优先使用`rate()`。

### 4.4.2 聚合运算函数

聚合运算函数用于对瞬时向量中的时序数据进行分组聚合，将多条时序合并为一条或多条聚合后的时序，是多维度统计分析的核心工具。所有聚合函数均支持`by()`与`without()`子句，用于指定分组规则。

#### 4.4.2.1 分组规则：`by()` 与 `without()`

- `by(<label_list>)`：按指定的标签列表进行分组聚合，**保留指定的标签**，移除未指定的标签；
- `without(<label_list>)`：按指定标签列表之外的标签进行分组聚合，**移除指定的标签**，保留其余标签。

示例：

```bash
# 按instance分组，计算每个实例的CPU使用率平均值，保留instance标签
avg by (instance) (node_cpu_usage_ratio)

# 移除instance标签，按job和env分组，计算每个任务的总请求数
sum without (instance) (http_requests_total)
```

#### 4.4.2.2 核心聚合函数详解

|函数名|语义|适用场景|标准示例|
|---|---|---|---|
|`sum()`|对分组内的时序数值求和|总请求数、总吞吐量、集群总资源使用量统计|`sum by (job) (rate(http_requests_total[5m]))`|
|`avg()`|对分组内的时序数值求平均值|平均 CPU 使用率、平均响应耗时、集群平均负载统计|`avg by (cluster) (node_load1)`|
|`max()`|取分组内时序数值的最大值|峰值 CPU 使用率、最大响应耗时、单实例峰值负载统计|`max by (api_path) (http_request_duration_seconds)`|
|`min()`|取分组内时序数值的最小值|最小可用内存、最低磁盘使用率、最小响应耗时统计|`min by (instance) (node_filesystem_avail_bytes)`|
|`count()`|统计分组内的时序总数量|在线实例数、活跃 Pod 数、异常指标数量统计|`count by (job) (up == 0)`|
|`topk()`|取分组内数值最大的前 K 条时序|峰值 TOP10 实例、请求量 TOP5 接口、资源占用 TOP 节点统计|`topk(10, sum by (instance) (rate(http_requests_total[5m])))`|
|`bottomk()`|取分组内数值最小的前 K 条时序|资源剩余最少的节点、请求量最低的接口、成功率最低的服务统计|`bottomk(5, avg by (instance) (node_memory_MemAvailable_bytes))`|
|`count_values()`|统计分组内相同数值出现的次数|版本分布统计、状态码分布统计、配置一致性校验|`count_values("version", node_exporter_build_info)`|

#### 4.4.2.3 核心使用规范

1. 聚合函数的输入必须为瞬时向量，禁止直接输入区间向量；
2. `topk()`/`bottomk()`返回的是原始时序数据，而非聚合值，仅用于排序筛选，不适合用于告警规则；
3. 聚合顺序规范：**先速率计算，后聚合**，禁止先聚合 Counter 原始值，再计算速率，会导致结果完全失真。

### 4.4.3 变化趋势函数

变化趋势函数专为**Gauge 类型指标**设计，用于计算时序数据的变化率与未来趋势预测，核心函数包括`deriv()`与`predict_linear()`，二者均基于最小二乘法线性回归实现。

#### 4.4.3.1 `deriv()` 函数

##### 核心定义与数学原理

`deriv()`函数计算区间向量中 Gauge 指标在时间窗口内的**每秒变化率（导数）**，基于线性回归计算，反映指标的增长 / 下降趋势。

##### 语法与标准示例

```bash
deriv(<range_vector>)
```

```bash
# 计算CPU使用率的每秒变化率，判断增长/下降趋势
deriv(avg by (instance) (node_cpu_usage_ratio)[5m:10s])

# 计算内存占用的变化趋势
deriv(node_memory_MemTotal_bytes - node_memory_MemAvailable_bytes[10m:15s])
```

##### 适用场景与规范

- 适用场景：指标变化趋势分析、增长 / 下降速率监控、异常波动检测；
- 核心约束：仅用于 Gauge 类型指标，禁止用于 Counter 类型指标；
- 窗口选型规范：窗口需覆盖足够多的数据点，保证线性回归的准确性，推荐至少覆盖 10 个抓取间隔。

#### 4.4.3.2 `predict_linear()` 函数

##### 核心定义与数学原理

`predict_linear()`函数基于时间窗口内的 Gauge 指标数据，通过线性回归预测**未来指定时间（秒）后的指标数值**，是资源耗尽预警的核心工具。

##### 语法与标准示例

```bash
predict_linear(<range_vector>, <duration_seconds>)
```

```bash
# 预测磁盘空间将在24小时（86400秒）内耗尽的实例
predict_linear(node_filesystem_avail_bytes[6h:1m], 24*3600) < 0

# 预测内存将在1小时内耗尽的服务
predict_linear(container_memory_usage_bytes[1h:15s], 3600) < 0
```

##### 适用场景与规范

- 适用场景：磁盘空间耗尽预警、内存溢出预警、资源容量规划、趋势预警；
- 核心约束：仅用于 Gauge 类型指标，禁止用于 Counter 类型指标；
- 窗口选型规范：预测窗口越大，历史数据窗口需同步增大，例如预测 24 小时后的数值，推荐使用 6 小时以上的历史数据窗口，保证预测准确性。

### 4.4.4 分位数计算函数

`histogram_quantile()`是 Prometheus 专为 Histogram 类型指标设计的分位数计算函数，用于在服务端基于 Histogram 的桶数据计算 P99、P95、P90 等分位数，是接口性能耗时分布统计的核心工具。

#### 核心定义与数学原理

`histogram_quantile()`函数基于 Histogram 的累积桶计数，通过线性插值计算指定分位数的近似值，核心逻辑：

1. 根据指定的分位数 φ，计算对应的累积计数阈值；
2. 找到包含该阈值的桶区间；
3. 在桶区间内通过线性插值计算分位数的近似值。

#### 语法规范

```bash
histogram_quantile(<φ>, <range_vector>)
```

- 参数`φ`：分位数阈值，取值范围为`0 < φ < 1`，例如`0.99`对应 P99 分位数，`0.95`对应 P95 分位数；
- 参数`<range_vector>`：必须是包含`le`标签的 Histogram 桶时序的瞬时向量，通常为`rate()`计算后的桶速率向量。

#### 标准示例与执行规范

**必须严格遵循「先 rate，后聚合，再计算分位数」的执行顺序**，错误的顺序会导致分位数计算结果完全失真。

```bash
# 标准用法：计算HTTP请求耗时的P99分位数（5分钟窗口）
histogram_quantile(
  0.99,
  sum by (le, service, api_path) (rate(http_request_duration_seconds_bucket[5m]))
)

# 计算跨实例聚合的P95耗时
histogram_quantile(
  0.95,
  sum by (le, instance) (rate(http_request_duration_seconds_bucket[10m]))
)

# 计算P50（中位数）耗时
histogram_quantile(
  0.5,
  sum by (le, job) (rate(http_request_duration_seconds_bucket[5m]))
)
```

#### 核心使用规范与禁忌

1. 输入向量必须包含`le`标签，且必须保留`le`标签在聚合分组中，否则函数无法正常计算；
2. 必须先对每个桶的 Counter 值计算`rate()`，再进行聚合，禁止直接使用桶的原始累积值计算分位数；
3. 禁止对 Summary 类型的预计算分位数使用该函数，Summary 的分位数无法聚合，直接查询即可；
4. 桶的划分越贴合观测值分布，线性插值的精度越高，若绝大多数观测值落入`+Inf`桶，分位数计算结果完全失效。

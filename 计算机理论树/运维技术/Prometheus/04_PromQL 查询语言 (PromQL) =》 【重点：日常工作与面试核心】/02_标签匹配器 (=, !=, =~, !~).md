
标签匹配器是 PromQL 实现多维度数据过滤的核心能力，用于在指标选择器中对标签值进行条件过滤，支持 4 类标准匹配器，可组合使用实现复杂的多维度过滤逻辑。

### 4.2.1 核心匹配器类型与语义

|匹配器|语义|严格匹配规则|
|---|---|---|
|`=`|完全相等匹配|仅保留标签值与指定字符串**完全相等**的时序|
|`!=`|反向相等匹配|仅保留标签值与指定字符串**不相等**的时序，包含无该标签的时序|
|`=~`|正则正向匹配|仅保留标签值**完全匹配**指定 RE2 正则表达式的时序|
|`!~`|正则反向匹配|仅保留标签值**不匹配**指定 RE2 正则表达式的时序，包含无该标签的时序|

### 4.2.2 核心语法规范与示例

1. **匹配器可组合使用**：多个标签匹配器之间为逻辑与（AND）关系，仅当所有匹配条件均满足时，时序才会被保留；
2. **正则匹配为全量匹配**：`=~`与`!~`的正则表达式为**全标签值匹配**，而非子串包含匹配。若需实现子串匹配，需通过`.*`包裹匹配内容；
3. **空标签值处理**：`label=""`会匹配标签值为空字符串的时序，或无该标签的时序；`label!=""`会匹配存在该标签且标签值非空的时序。

标准示例：

```bash
# 完全相等匹配：查询prod环境、job为api-server的请求总数
http_requests_total{env="prod", job="api-server"}

# 反向相等匹配：排除GET方法的请求总数
http_requests_total{method!="GET"}

# 正则正向匹配：匹配4xx和5xx错误状态码的请求总数
http_requests_total{status=~"4..|5.."}

# 正则反向匹配：排除test和dev环境的主机CPU指标
node_cpu_seconds_total{env!~"test|dev"}

# 子串匹配：匹配路径包含/api/user的请求总数
http_requests_total{path=~".*/api/user.*"}

# 多匹配器组合：查询prod环境、POST方法、非200状态码的请求总数
http_requests_total{env="prod", method="POST", status!="200"}
```

### 4.2.3 底层实现与性能规范

1. **指标名称的本质**：Prometheus 中，指标名称本质是名为`__name__`的内置标签的值。因此`http_requests_total`等价于`{__name__="http_requests_total"}`，可通过正则匹配实现多指标的批量查询；
2. **性能优先级**：相等匹配（`=`/`!=`）的性能远高于正则匹配（`=~`/`!~`），生产环境优先使用相等匹配，减少正则匹配的使用；
3. **高基数过滤禁忌**：禁止对高基数标签（如用户 ID、订单 ID、请求 ID）使用正则匹配，会导致查询负载激增，触发 Prometheus 性能瓶颈。

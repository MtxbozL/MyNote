Git_GitLab_Learning_Roadmap/
│
├── 01_Git 基础原理与环境准备 (Foundation_&_Setup)
│   ├── 01_版本控制演进 (本地控制 -> 集中式 SVN -> 分布式 Git)
│   ├── 02_Git 核心设计思想 (直接记录快照，而非差异比较；几乎所有操作都在本地执行)
│   ├── 03_环境配置 (git config --global user.name/email, core.autocrlf 换行符处理)
│   ├── 04_SSH 密钥配置 (ssh-keygen 生成，配置公钥至 GitLab/GitHub)
│   └── 05_三大核心区域 => 【重点：Git 模型基石】
│       ├── Working Directory (工作区：你在电脑上看到的目录)
│       ├── Staging Area / Index (暂存区：保存下次将要提交的文件列表信息)
│       └── Git Repository (本地仓库：.git 目录，保存所有历史版本快照)
│
├── 02_Git 本地基础操作 (Local_Operations)
│   ├── 01_仓库初始化 (git init)
│   ├── 02_文件状态流转 (Untracked 未跟踪 -> Modified 已修改 -> Staged 已暂存 -> Unmodified 未修改)
│   ├── 03_核心流转命令 (git add, git commit -m)
│   ├── 04_查看状态与差异 (git status, git diff 工作区与暂存区, git diff --cached 暂存区与仓库)
│   ├── 05_查看提交历史 (git log, git log --oneline --graph --all, git show)
│   └── 06_忽略文件 (.gitignore 语法规则，如何清除已经被跟踪的忽略文件)
│
├── 03_撤销与时光倒流 (Undoing_&_Time_Travel) => 【重点：后悔药】
│   ├── 01_工作区/暂存区撤销 (git restore, git rm --cached)
│   ├── 02_修改最后一次提交 (git commit --amend - 漏提文件或改注释神器)
│   ├── 03_git reset (重置) => 【极其重要】
│   │   ├── --soft (仅回退本地仓库历史，保留暂存区和工作区)
│   │   ├── --mixed (默认：回退历史和暂存区，保留工作区代码)
│   │   └── --hard (核弹：历史、暂存区、工作区全部回退，未提交代码会丢失！)
│   ├── 04_git revert (反转) (生成一个新提交来撤销之前的修改，团队协作时取代 reset 的安全做法)
│   └── 05_git reflog (引用日志) (Git 的终极救命稻草：找回被 hard reset 删掉的孤儿提交)
│
├── 04_分支管理与合并 (Branching_&_Merging) => 【重点：多人协作核心】
│   ├── 01_分支本质 (分支只是一个指向某次提交的轻量级可移动指针，HEAD 指向当前分支)
│   ├── 02_基础操作 (branch 创建, checkout / switch 切换, branch -d 删除)
│   ├── 03_git merge (合并)
│   │   ├── Fast-forward (快进合并：无分叉直接指针前移)
│   │   └── 3-way merge (三方合并：有分叉，会产生一个新的 Merge Commit)
│   ├── 04_冲突解决 (Conflict Resolution：识别 <<<< ==== >>>> 标记，手动修复后 add + commit)
│   └── 05_git rebase (变基) => 【高级必考】
│       ├── 原理 (将某分支上的所有修改都移至另一分支上，重写历史，打造纯线性提交历史)
│       ├── rebase vs merge 的使用场景与利弊
│       ├── 交互式变基 (git rebase -i：合并多次零碎提交 squash、修改历史提交)
│       └── 黄金法则 (绝不要在公共的/已推送到远程的分支上执行 Rebase！)
│
├── 05_远程仓库与协作机制 (Remote_Repositories)
│   ├── 01_关联远程仓库 (git clone, git remote add, git remote -v)
│   ├── 02_拉取机制 (git fetch 下载远程最新状态但不合并)
│   ├── 03_git pull (实质 = git fetch + git merge) 
│   │   └── 优化做法：git pull --rebase (拉取并变基，避免产生冗余的 Merge Commit)
│   └── 04_推送到远程 (git push, git push -u, 强制推送 git push -f 慎用)
│
├── 06_Git 高级生产力工具 (Advanced_Tools)
│   ├── 01_git stash (储藏：临时保存当前工作区未完成的代码，以便切换分支修 Bug)
│   ├── 02_git cherry-pick (摘取：精准挑拣某一个/几个 Commit 合并到当前分支)
│   ├── 03_git bisect (二分查找：自动化快速定位引入 Bug 的那一次 Commit)
│   ├── 04_Tag 标签管理 (打轻量标签、附注标签 git tag -a，通常用于版本发布如 v1.0.0)
│   └── 05_Submodule 子模块 (在一个 Git 仓库中嵌套引用的另一个独立 Git 仓库)
│
├── 07_企业级分支开发工作流 (Workflows)
│   ├── 01_Git Flow (经典/最复杂：包含 master, develop, feature, release, hotfix)
│   ├── 02_GitHub Flow (极其简单：master 永远可部署 + 各种 feature 分支直接提 PR)
│   └── 03_GitLab Flow (环境驱动：结合环境分支如 master, pre-production, production，最适合 CI/CD)
│
├── 08_GitLab 基础与团队协作 (GitLab_Collaboration)
│   ├── 01_GitLab 层级架构 (Group 群组 -> Subgroup -> Project 项目)
│   ├── 02_权限与角色控制 (Guest, Reporter, Developer, Maintainer, Owner)
│   ├── 03_Issues 缺陷跟踪与看板 (Boards, Labels 标签体系, Milestones 里程碑)
│   ├── 04_Merge Request (MR 代码合并请求) => 【团队 Code Review 核心】
│   │   ├── 创建 MR、Draft (WIP) 状态、Assignees (指派给)与 Reviewers (审查者)
│   │   ├── 解决 MR 冲突、Squash commits (压缩提交保持主干干净)
│   │   └── 保护分支设置 (Protected Branches：禁止直接 Push master，必须经过 MR 及审批通过)
│   └── 05_Webhooks (将代码 Push / MR 事件推送到钉钉/企业微信群发通知)
│
├── 09_GitLab CI/CD 流水线核心 (GitLab_CI_CD_Core) => 【重点：DevOps 灵魂】
│   ├── 01_概念概览 (CI 持续集成，CD 持续交付/部署，Pipeline 流水线)
│   ├── 02_GitLab Runner (执行者)
│   │   ├── Shared Runners (共享) vs Specific Runners (项目专用)
│   │   └── Executor (执行器选型：Shell, Docker 推荐, Kubernetes)
│   ├── 03_.gitlab-ci.yml 基础语法结构
│   │   ├── stages (定义流水线阶段顺序：build -> test -> deploy)
│   │   └── job (任务：定义属于哪个 stage、执行的 script 脚本)
│   ├── 04_环境准备指令 (image, before_script, after_script)
│   ├── 05_Variables 环境变量 (UI 配置的隐蔽变量 vs yml 内定义，预定义变量如 $CI_COMMIT_SHA)
│   └── 06_Artifacts 与 Cache (工件与缓存)
│       ├── Artifacts (用于在不同的 Stage 之间传递编译产物，并供页面下载)
│       └── Cache (用于缓存 npm/maven 依赖包，极大加速下次流水线构建)
│
├── 10_GitLab CI/CD 高级特性 (GitLab_CI_CD_Advanced)
│   ├── 01_规则与流转控制 (rules, only/except) 
│   │   └── (例如：只在合并到 master 分支，或者打了 tag 时才触发发布 Job)
│   ├── 02_Environments 与 Deployments (环境部署追踪，UI 上一键 rollback 回滚)
│   ├── 03_Manual 动作与审批 (when: manual 人工点击确认部署到生产环境)
│   ├── 04_Needs (有向无环图 DAG：打破 Stage 顺序限制，让无依赖的任务提前并行运行)
│   ├── 05_include (流水线模板复用：将公共的 CI 配置拆分引入)
│   └── 06_Auto DevOps / DevSecOps (自动代码质量检查、SAST 静态安全扫描集成)
│
└── 11_GitLab 企业级底层探秘与运维 (GitLab_Administration_&_Internals) => 【高阶架构师】
    ├── 01_Git 底层对象模型探秘 (直接打开 .git 目录一探究竟)
    │   ├── Blob (存储文件内容，不存文件名)
    │   ├── Tree (存储目录结构与文件名，指向 Blob)
    │   └── Commit (指向顶层 Tree，包含作者信息、时间戳与父 Commit)
    ├── 02_Omnibus 官方包部署与架构体系 (Nginx + Ruby on Rails + PostgreSQL + Redis + Gitaly)
    ├── 03_GitLab 备份与恢复 (gitlab-backup create/restore)
    └── 04_LDAP 与企业认证集成 (对接企业内部的活动目录域统一登录)

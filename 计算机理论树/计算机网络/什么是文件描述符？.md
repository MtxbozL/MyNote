#### 核心标准答案

文件描述符（File Descriptor，简称 fd）是**Linux/Unix 系统内核为进程分配的、用于标识已打开资源的非负整数索引**，是进程访问内核资源的句柄。

系统为每个进程维护独立的文件描述符表，默认 0 对应标准输入 stdin、1 对应标准输出 stdout、2 对应标准错误 stderr，进程每打开一个文件、套接字、管道、设备等资源，都会分配一个新的、唯一的 fd，资源关闭后 fd 会被回收复用。

#### 专家级拓展

- 核心限制：系统级有最大 fd 上限（可通过`/proc/sys/fs/file-max`查看），进程级也有默认上限（通常 1024，可通过`ulimit -n`调整），运维高频故障`too many open files`，本质就是进程 fd 耗尽。
- 底层本质：fd 是进程文件描述符表的数组下标，指向内核维护的全局文件表，最终映射到 inode 节点，实现了「进程 - 内核 - 磁盘资源」的隔离与统一管理。
- 落地场景：IO 多路复用（select/poll/epoll）的核心就是监听多个 fd 的事件，Nginx、Redis 等高并发服务，核心优化点之一就是调大进程 fd 上限。

#### 面试避坑指南

- 严禁只说「fd 是一个数字」，不说清它的作用、归属（进程级）、核心场景；
- 避免混淆系统级和进程级的 fd 上限，面试被问到相关故障时，无法给出解决方案。

### 3. 请说明 IO 多路复用的核心原理

#### 核心标准答案

IO 多路复用是一种**单线程 / 单进程同时监听多个文件描述符的 IO 事件，当任意 fd 就绪（可读 / 可写 / 异常）时，内核通知进程处理对应事件**的 IO 机制。

它解决了阻塞 IO 单线程只能处理一个 fd、非阻塞 IO 需要频繁轮询的痛点，用极小的线程开销实现高并发 IO 处理，是 Linux 高并发服务的核心基础。

Linux 下主流实现有三种：select、poll、epoll。

#### 专家级拓展

- 三种实现的核心差异：
    
    1. select：用固定长度的位图存储 fd，有 1024 个 fd 上限，每次调用都需要把 fd 集合从用户态拷贝到内核态，遍历全量 fd 找就绪事件，效率随 fd 数量增加线性下降。
    2. poll：用链表存储 fd，无 1024 上限，解决了 select 的 fd 数量限制，但仍存在全量遍历、用户态内核态频繁拷贝的问题。
    3. epoll：红黑树管理 fd + 就绪链表存储就绪事件，无 fd 数量上限，仅需一次 fd 拷贝到内核，只返回就绪的 fd，无需全量遍历，性能远超 select/poll，是高并发场景的唯一选择。
    
- 触发模式：epoll 支持水平触发 LT（默认，fd 就绪后会持续通知，直到事件处理完成）和边缘触发 ET（仅在 fd 状态变化时通知一次，性能更高，要求一次性处理完数据）。
- 落地场景：Nginx、Redis、MySQL、K8s 组件等所有 Linux 高并发服务，底层均基于 epoll 实现 IO 多路复用。

#### 面试避坑指南

- 严禁把 IO 多路复用和异步 IO 混淆，IO 多路复用属于**同步 IO**，事件就绪后需要进程主动拷贝数据；
- 避免只说概念，不说三种实现的差异，面试官大概率会追问 epoll 的优势。

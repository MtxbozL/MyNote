#### 核心标准答案

Ceph 是一款开源的分布式统一存储系统，支持块存储、文件存储、对象存储三大存储类型，采用**去中心化的分布式架构**，无单点故障，可水平扩展至数千个节点，核心组件分为底层核心存储组件和上层存储网关组件，具体如下：

##### 一、底层核心存储组件（Ceph 集群的基础，所有存储类型共用）

1. **MON（Monitor，监控节点）**：集群的控制平面核心，负责维护集群的全局状态 map（包括 OSD map、MON map、PG map、CRUSH map），处理集群的认证授权、配置管理，为客户端和其他组件提供集群元数据信息。MON 集群采用奇数节点部署，通过 Paxos 算法保证数据一致性，组成法定仲裁，半数以上节点在线集群才可用。
2. **OSD（Object Storage Device，对象存储设备）**：集群的数据平面核心，负责实际的数据存储、读写、复制、恢复、重平衡，以及数据的健康检查。每个 OSD 对应一块磁盘 / 一个 RAID 组，OSD 之间会互相心跳检测，上报状态给 MON 节点。生产环境通常一个磁盘对应一个 OSD，最大化磁盘性能。
3. **MDS（Metadata Server，元数据服务器）**：仅用于 CephFS 文件存储，负责管理文件系统的元数据（目录结构、文件权限、inode 信息），将文件的元数据和实际数据分离存储，提升文件系统的读写性能，避免客户端直接遍历 OSD 获取元数据。

##### 二、上层存储网关组件（对外提供不同类型的存储服务）

1. **RBD（Rados Block Device，RADOS 块设备）**：提供块存储服务，将分布式的 RADOS 存储池封装为块设备，可挂载到服务器使用，支持快照、克隆、精简配置，常用于虚拟机磁盘、容器持久化存储，是 K8s 环境最常用的 Ceph 存储类型。
2. **RGW（Rados Gateway，RADOS 网关）**：提供对象存储服务，兼容 S3、Swift 协议，对外提供 RESTful API，可用于静态资源存储、备份归档、大数据存储，实现分布式的对象存储能力。
3. **CephFS（Ceph File System）**：提供 POSIX 兼容的分布式文件系统，支持多客户端同时挂载读写，常用于日志存储、大数据共享存储、容器共享存储等场景。

补充：Ceph 的底层核心是**RADOS（Reliable Autonomic Distributed Object Store）**，即可靠的自治分布式对象存储，所有上层存储服务都是基于 RADOS 实现的，所有数据最终都会以对象的形式存储在 RADOS 集群中。

#### 专家级拓展

- 核心数据读写流程：客户端写入数据时，先通过 MON 节点获取集群的 CRUSH map，通过 CRUSH 算法计算出数据对应的 PG 和主 OSD，直接将数据写入主 OSD，主 OSD 再将数据复制到副本 OSD，所有副本写入完成后，返回客户端写入成功，全程无需经过中心节点，实现了去中心化的读写，性能随节点数量线性扩展。
- CRUSH 算法：是 Ceph 的核心创新，替代了传统分布式存储的中心化元数据表，通过算法计算数据的存储位置，无需查询中心节点，避免了元数据节点的性能瓶颈和单点故障，同时支持自定义故障域，可实现跨主机、跨机架、跨机房的数据副本隔离，提升集群的容灾能力。
- 生产环境部署最佳实践：
    
    1. MON 节点：至少 3 个，奇数部署，使用 SSD 磁盘，单独部署，不与 OSD 共用节点，保证控制平面的稳定性；
    2. OSD 节点：数据盘使用 SSD/NVMe 磁盘，系统盘单独配置，每个磁盘对应一个 OSD，开启日志分离，提升性能；
    3. 网络规划：分离集群网和数据网，集群网用于 OSD 之间的数据复制、心跳，数据网用于客户端读写，避免网络拥堵；
    4. 副本配置：生产环境默认 3 副本，核心业务可配置 4 副本，跨机架分布，保证单机架故障不丢失数据。
    

#### 面试避坑指南

- 严禁混淆各个组件的作用，比如把 MDS 说成块存储的组件，MDS 仅用于 CephFS，RBD 不需要 MDS，这是面试高频错误；
- 避免只说组件名称，不说核心职责，面试问这个问题，核心是考察你对 Ceph 架构的理解，能不能根据业务场景选择合适的存储类型；
- 不要遗漏 RADOS 和 CRUSH 算法这两个核心，这是 Ceph 架构的灵魂，提到这两个点，会被面试官判定为「理解了 Ceph 的底层原理，不是只会搭集群」。

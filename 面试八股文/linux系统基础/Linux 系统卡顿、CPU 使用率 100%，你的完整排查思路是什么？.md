#### 核心标准答案

我会遵循「从整体到局部，从系统到进程，从现象到根因」的分层排查思路，快速定位问题，完整流程如下：

1. **第一步：确认整体负载与 CPU 状态**
    
    执行`uptime`查看系统平均负载，判断是突发负载还是长期高负载；执行`top`查看整体 CPU 使用率，确认是用户态 CPU 高、内核态 CPU 高，还是 IO 等待占比高，明确排查方向。
2. **第二步：定位占用 CPU 最高的进程**
    
    在`top`界面按`P`，按 CPU 使用率排序，找到占用 CPU 最高的业务进程，记录进程 PID、进程名、CPU 使用率，确认是业务进程、系统进程，还是恶意进程。
3. **第三步：定位进程内占用 CPU 最高的线程**
    
    执行`top -Hp 进程PID`，查看该进程下的所有线程，按 CPU 使用率排序，找到占用 CPU 最高的线程 ID（TID），确认是业务线程、GC 线程，还是其他异常线程。
4. **第四步：定位线程的具体执行逻辑，找到根因**
    
    - Java 进程：将线程 ID 转为 16 进制，执行`jstack 进程PID | grep 16进制线程ID -A 20`，查看线程的堆栈信息，定位是死循环、频繁 GC、锁竞争，还是接口调用耗时过高。
    - Go 进程：执行`go tool pprof http://ip:port/debug/pprof/profile`，采集 CPU profile，定位占用 CPU 高的函数。
    - 其他进程：执行`strace -p 线程TID`，查看线程的系统调用，定位是频繁的 IO 操作、系统调用异常，还是其他问题。
    
5. **第五步：针对性解决问题，验证恢复**
    
    根据根因处理，比如优化死循环代码、调整 JVM 参数优化 GC、重启异常进程、查杀恶意进程、扩容 CPU 资源，处理完成后，再次查看 CPU 使用率，确认系统恢复正常。
6. **第六步：事后复盘与预防**
    
    记录故障根因、处理过程，优化程序代码，配置 CPU 使用率监控告警，避免同类问题再次发生。

#### 专家级拓展

- 不同 CPU 高场景的专项排查方向：
    
    1. 用户态 CPU 高（% us>80%）：90% 以上是业务代码问题，优先排查死循环、频繁 GC、正则表达式回溯、大量计算、接口并发量突增。
    2. 内核态 CPU 高（% sy>30%）：优先排查频繁的系统调用、大量的上下文切换、锁竞争、内核参数配置不合理、网络 / 磁盘 IO 频繁。
    3. IO 等待 CPU 高（% wa>30%）：优先排查磁盘 IO 瓶颈（大量读写、磁盘损坏）、swap 分区频繁读写（物理内存不足）、网络 IO 瓶颈。
    
- 高频隐藏场景：CPU 上下文切换频繁，会导致内核态 CPU 高，可用`vmstat 1`查看`cs`指标，上下文切换次数突增，大概率是进程 / 线程创建过于频繁，比如程序 bug 导致无限创建线程 / 协程。
- 应急处理：生产环境业务受影响时，优先重启异常进程，恢复业务，再事后排查根因，避免故障扩大。

#### 面试避坑指南

- 严禁只说「用 top 看哪个进程 CPU 高，杀掉就行」，会被面试官判定为「只会暴力处理，无定位根因的能力」，面试核心考察的是完整的排查逻辑，不是处理结果；
- 避免排查流程混乱，东一榔头西一棒子，必须遵循从整体到局部的分层逻辑，体现专业的故障排查能力；
- 不要只讲通用流程，不说不同场景的专项排查方向，无法体现生产环境的实战经验。

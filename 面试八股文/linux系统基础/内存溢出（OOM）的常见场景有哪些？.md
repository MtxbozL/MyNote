#### 核心标准答案

内存溢出（OOM，Out Of Memory）指程序申请内存时，系统 / 进程可用内存无法满足需求，导致程序崩溃、被系统终止的故障，核心分为**进程级 OOM**和**系统级 OOM**两大类，常见场景如下：

1. **堆内存溢出**：最常见场景，无限创建对象 / 实例，且对象被持续引用无法被 GC 回收，堆内存耗尽，比如无界集合无限添加数据、循环创建大对象。
2. **栈内存溢出**：方法递归调用过深、无限递归，或方法内创建过大的局部变量，导致线程栈内存超出上限，比如未设置终止条件的递归函数。
3. **直接内存 / 堆外内存溢出**：程序申请的堆外内存超出系统限制，且未手动释放，比如 NIO 的 DirectBuffer、JNI 调用分配的 native 内存。
4. **资源泄漏导致的内存溢出**：文件句柄、网络连接、数据库连接、流等资源打开后未关闭，底层关联的内存无法释放，长期运行累积导致内存耗尽。
5. **系统级 OOM**：系统整体物理内存 + 交换分区耗尽，Linux 内核触发 OOM Killer 机制，按照 OOM Score 优先级杀掉进程释放内存，通常是高内存占用的业务进程。

#### 专家级拓展

- 分岗位排查与解决方案：
    
    - 运维侧：
        
        1. 系统级 OOM：通过`dmesg | grep -i oom`查看 OOM 日志，确认被 kill 的进程与原因；调整`/proc/[pid]/oom_score_adj`降低核心进程的 OOM 优先级，避免被误杀；监控系统内存使用率、swap 使用率，提前扩容。
        2. 进程级 OOM：通过监控系统查看进程内存趋势，定位内存泄漏的进程，配合开发排查问题。
        
    - 开发侧：
        
        1. Java：通过`jmap`生成堆快照，用 MAT、JProfiler 分析堆内存中的大对象、泄漏对象；优化 JVM 堆内存参数（Xmx/Xms），避免堆内存设置过小。
        2. Go：通过`pprof`采集堆内存 profile，定位内存泄漏的代码；排查 goroutine 泄漏（无限创建 goroutine 且未退出），这是 Go 最常见的 OOM 场景。
        
    
- 高频隐藏场景：缓存无淘汰策略（比如本地 map 做缓存，无过期时间、无最大容量限制）、大文件一次性加载到内存、流量突增导致对象创建速度远超 GC 回收速度。

#### 面试避坑指南

- 严禁只泛泛说「创建对象太多」，不分类、不给出具体场景，会被判定为无实际故障排查经验；
- 避免混淆内存溢出（OOM）和内存泄漏（Memory Leak）：内存泄漏是原因，内存溢出是结果，长期的内存泄漏最终一定会导致 OOM。

#### 核心标准答案

服务器巡检的核心目标是**提前发现服务器的硬件、系统、应用、安全隐患，保证服务器的稳定运行**，核心巡检指标分为 6 大类，以及对应的采集方式如下：

##### 一、硬件与基础设施层巡检指标

核心是检查服务器硬件的健康状态，避免硬件故障导致系统崩溃，核心指标：

1. **服务器硬件状态**：电源状态、风扇转速、CPU / 内存 / 主板温度、RAID 卡状态、硬盘健康状态（SMART 信息）、硬件告警灯状态；
2. **物理机 / 虚拟机状态**：宿主机的 CPU / 内存 / 磁盘使用率、虚拟化平台的运行状态、虚拟机的资源分配情况；
3. **存储设备状态**：磁盘阵列的状态、LUN 状态、存储池使用率、存储控制器状态、存储端口状态。

##### 二、操作系统层巡检指标

核心是检查系统的运行状态，资源使用情况，避免系统资源耗尽导致服务异常，核心指标：

1. **CPU 指标**：CPU 整体使用率、用户态 / 内核态 / IO 等待 / 空闲占比、系统平均负载（1/5/15 分钟）、CPU 上下文切换次数、中断次数、CPU 核心温度；
2. **内存指标**：总内存、已用内存、可用内存、缓存 / 缓冲占用、swap 使用率、swap 换入换出速率、内存 OOM 事件；
3. **磁盘指标**：磁盘分区使用率、inode 使用率、磁盘读写吞吐量（MB/s）、IOPS、IO 等待时间（await）、磁盘使用率（% util）、坏道 / 只读错误；
4. **网络指标**：网卡入 / 出带宽、数据包收发量、错包 / 丢包率、TCP 连接数（各状态分布）、网络延迟、DNS 解析状态；
5. **系统状态**：系统开机时长、内核版本、系统时间同步状态、运行的进程数、僵尸进程数量、系统日志中的错误告警。

##### 三、应用服务层巡检指标

核心是检查业务服务的运行状态，保证业务可用，核心指标：

1. **服务基础状态**：服务进程状态、端口监听状态、服务运行时长、CPU / 内存占用、服务启动 / 重启次数；
2. **服务性能指标**：服务 QPS/RT、请求成功率、错误率、并发连接数、线程池状态、GC 情况（Java 服务）、协程数量（Go 服务）；
3. **日志状态**：服务日志的错误 / 异常关键字、日志轮转状态、日志磁盘占用。

##### 四、中间件层巡检指标

核心是检查中间件的运行状态，核心指标：

1. **Web 服务（Nginx/Apache）**：进程状态、端口监听、QPS、请求成功率、错误率、连接数、日志错误、配置文件语法；
2. **数据库（MySQL/PostgreSQL）**：进程状态、端口监听、连接数、慢查询数量、QPS、TPS、主从同步状态、锁等待、缓存命中率、表空间使用率；
3. **缓存（Redis）**：进程状态、端口监听、内存使用率、命中率、QPS、主从同步状态、持久化状态、过期 key 数量、慢查询数量；
4. **消息队列（Kafka/RabbitMQ）**：进程状态、端口监听、消息生产 / 消费速率、消息堆积量、副本同步状态、分区状态、日志错误。

##### 五、安全与合规巡检指标

核心是检查服务器的安全状态，避免安全漏洞和违规风险，核心指标：

1. **账号与权限**：系统用户列表、sudo 权限配置、空密码用户、过期账号、SSH 登录配置、密钥登录状态；
2. **防火墙与安全**：防火墙规则状态、开放端口、SELinux 状态、恶意进程、root 用户远程登录限制、登录失败次数；
3. **安全事件**：SSH 暴力破解记录、异常登录 IP、系统日志中的安全告警、恶意文件检测；
4. **合规配置**：密码复杂度策略、日志审计开启状态、系统补丁更新情况、漏洞修复情况。

##### 六、业务与容灾巡检指标

1. **业务可用性**：核心业务接口的可用性、响应时间、成功率；
2. **数据备份**：备份任务执行状态、备份文件完整性、备份保留周期、备份恢复演练情况；
3. **容灾状态**：高可用集群状态、主备切换状态、负载均衡状态、机房 / 可用区容灾能力。

##### 二、指标采集方式

分为**人工命令行采集**和**自动化工具采集**两种方式，分别适用于临时巡检和日常自动化巡检：

###### 1. 人工命令行采集（临时巡检 / 故障排查）

通过 Linux 原生命令，手动采集巡检指标，核心命令对应指标：

表格

|巡检维度|核心采集命令|
|:--|:--|
|CPU / 负载|`top`、`htop`、`uptime`、`vmstat`、`mpstat`|
|内存|`free -h`、`top`、`vmstat`、`cat /proc/meminfo`|
|磁盘|`df -h`、`df -i`、`iostat -x 1`、`smartctl`、`fdisk`|
|网络|`ss`、`netstat`、`ip addr`、`ping`、`telnet`、`tcpdump`、`iftop`|
|系统状态|`uname -a`、`dmesg`、`ps aux`、`systemctl status`、`chronyc tracking`（时间同步）|
|硬件状态|`ipmitool`（硬件传感器）、`megacli`（RAID 卡状态）、`smartctl`（硬盘 SMART 信息）|
|日志审计|`tail`、`grep`、`less`查看`/var/log/messages`、`/var/log/secure`等系统日志|

###### 2. 自动化工具采集（日常生产巡检，首选方式）

通过自动化工具，实现指标的自动采集、可视化、告警，无需人工操作，是生产环境的标准方案，核心工具：

1. **监控系统采集**：
    
    - **Prometheus + Node Exporter + Grafana**：最主流的方案，Node Exporter 采集服务器的系统指标，Prometheus 存储时序数据，Grafana 实现可视化巡检看板，自动采集所有系统指标，设置告警阈值，异常自动告警；
    - **Zabbix**：全栈监控工具，通过 Agent 采集服务器、硬件、应用、中间件的所有指标，自带可视化、告警、巡检报表功能，适合中小规模集群；
    - 其他监控工具：Nagios、Prometheus+Alertmanager、云厂商的云监控服务。
    
2. **自动化运维脚本采集**：
    
    编写 Shell/Python 巡检脚本，批量执行采集命令，自动生成巡检报告，通过邮件 / 企业微信 / 钉钉发送，适合自定义巡检场景，配合 crontab 定时执行，实现每日自动巡检。
3. **自动化运维平台采集**：
    
    基于 Ansible、SaltStack 等批量运维工具，批量在多台服务器上执行巡检任务，采集指标，汇总生成统一的巡检报告，适合大规模集群的批量巡检。
4. **安全巡检工具**：
    
    - 漏洞扫描：Nessus、OpenVAS，自动扫描系统漏洞、安全配置问题；
    - 基线检查：Lynis、OpenSCAP，自动检查系统的安全合规基线，生成合规报告；
    - 日志审计：ELK/Loki，集中采集系统和应用日志，自动分析异常日志、安全事件。
    

#### 专家级拓展

- 生产环境巡检最佳实践：
    
    1. 制定标准化的巡检流程，分为日常巡检（每日）、周巡检、月巡检、季度巡检，不同巡检周期覆盖不同的指标，日常巡检关注核心可用性指标，季度巡检关注安全合规、容灾备份、性能优化；
    2. 优先使用自动化监控工具实现 7*24 小时的指标采集和告警，替代人工巡检，人工巡检只做定期的复核和深度检查，避免人工巡检的遗漏和低效；
    3. 巡检结果必须闭环，发现的问题必须记录、跟踪、整改、复盘，形成完整的巡检闭环，避免只巡检不整改；
    4. 核心业务服务器，必须配置硬件监控，通过 IPMI/iDRAC/iLO 远程管理卡，监控服务器的硬件状态，提前发现硬件故障隐患；
    5. 定期做备份恢复演练，验证备份的可用性，这是巡检中最容易遗漏，但最重要的环节。
    
- 巡检报告规范：
    
    巡检完成后，必须生成标准化的巡检报告，包含：巡检时间、巡检范围、巡检指标、正常项、异常项、问题整改建议、上一次问题的整改情况，同步给相关负责人，跟踪整改进度。

#### 面试避坑指南

- 严禁只说 CPU、内存、磁盘等基础指标，不说硬件、应用、安全、容灾等维度的指标，会被面试官判定为只会基础巡检，无体系化的生产环境巡检经验；
- 避免只说人工命令采集，不说自动化工具采集，生产环境大规模集群，人工巡检完全不现实，必须重点讲自动化巡检方案，体现生产环境实战经验；
- 不要只说指标采集，不说巡检的闭环管理，巡检的最终目的是发现问题、解决问题，而不只是采集指标，体现你的运维闭环思维。

#### 核心标准答案

K8s 集群采用**控制平面（Master 节点）+ 节点平面（Node 节点）** 的分布式架构，核心组件职责清晰，共同实现集群的容器编排、调度、管理能力，具体如下：

##### 一、控制平面组件（集群的大脑，负责全局决策、调度、状态管理）

1. **kube-apiserver**：集群的唯一入口，所有组件的通信中枢，提供 K8s API 的 HTTP/HTTPS 服务，负责认证、授权、准入控制、API 注册与发现，所有对集群的操作，都必须经过 apiserver，最终写入 etcd。
2. **etcd**：集群的分布式键值存储数据库，是 K8s 唯一的数据存储组件，存储集群所有的资源状态、配置信息，保障数据的一致性和高可用。
3. **kube-controller-manager**：集群的控制器管理器，运行所有的内置控制器，负责维护集群的期望状态，比如 Deployment 控制器、StatefulSet 控制器、Node 控制器、Endpoint 控制器，持续监听资源的状态，当实际状态与期望状态不一致时，执行调谐操作，恢复到期望状态。
4. **kube-scheduler**：集群的调度器，负责 Pod 的调度决策，监听未绑定节点的 Pod，根据调度策略、节点资源、亲和性 / 反亲和性、污点 / 容忍等规则，为 Pod 选择最合适的 Node 节点，绑定 Pod 到对应节点。
5. **cloud-controller-manager**：可选组件，用于对接云厂商的 API，实现集群与云厂商基础设施的集成，比如负载均衡、云硬盘、VPC 网络的管理，解耦集群与底层云平台。

##### 二、节点平面组件（运行在每个 Node 节点上，负责节点的容器运行、Pod 管理）

1. **kubelet**：节点的核心代理，运行在每个 Node 节点上，负责接收 apiserver 下发的 Pod 配置，确保节点上的 Pod 按照期望状态运行，包括容器的创建、启动、停止、健康检查，同时向控制平面上报节点的状态、资源使用率、Pod 的运行状态。
2. **kube-proxy**：节点的网络代理，运行在每个 Node 节点上，负责实现 K8s Service 的网络规则，通过维护节点上的 iptables/ipvs 规则，实现 Service 的四层负载均衡、集群内 Pod 的网络互通、节点端口转发。
3. **容器运行时（Container Runtime）**：负责容器的生命周期管理，是 kubelet 与容器之间的中间层，遵循 CRI（容器运行时接口）标准，常见实现有 containerd、CRI-O，早期为 Docker（已废弃）。

#### 专家级拓展

- 组件通信机制：K8s 所有组件都只和 kube-apiserver 通信，组件之间不直接通信，apiserver 是唯一的通信中枢，所有数据都通过 apiserver 写入 etcd，保障了集群的安全性和一致性。
- 高可用架构核心：生产环境控制平面必须部署 3 个及以上节点，实现 apiserver 的负载均衡，etcd 采用奇数节点的集群部署，保障数据的一致性和容灾能力，单个控制平面节点故障，集群依然可用。
- 进阶知识点：kube-proxy 有两种模式，iptables 模式和 ipvs 模式，iptables 模式是默认模式，实现简单，适合小规模集群；ipvs 模式基于内核的 IPVS 模块，性能更高，支持更多的负载均衡算法，适合大规模集群、高并发场景。

#### 面试避坑指南

- 严禁遗漏核心组件，或把组件的作用说反，比如把 kube-scheduler 的作用说成 kube-controller-manager，这是基础扣分点；
- 避免只说组件名称，不说核心职责，会被判定为死记硬背，无实际集群运维能力；
- 注意 Docker 已经不是 K8s 的默认容器运行时，1.24 版本之后已经移除了 dockershim，不要再说 Docker 是核心组件，会被判定为知识过时。

#### 核心标准答案

HTTP 是明文传输的应用层协议，运行在 TCP 之上，安全性极差；HTTPS 本质是**HTTP + TLS/SSL**，在 HTTP 和 TCP 之间增加了 TLS/SSL 安全层，相比 HTTP，额外使用的核心协议如下：

1. **核心协议：TLS（传输层安全协议）**
    
    这是 HTTPS 最核心的额外协议，前身是 SSL（安全套接层），目前主流版本是 TLS 1.2、TLS 1.3，TLS 1.0/1.1 已被废弃。
    
    - 核心作用：实现数据的加密传输、身份认证、数据完整性校验，解决 HTTP 明文传输、数据被窃听、篡改、伪造的三大安全问题。
    - 核心能力：
        
        1. 身份认证：通过 CA 数字证书，确认服务端的身份，防止访问到钓鱼网站；
        2. 对称加密：握手完成后，使用对称加密算法（AES、ChaCha20）加密传输的 HTTP 数据，保证只有客户端和服务端能解密；
        3. 非对称加密：握手过程中使用非对称加密算法（RSA、ECDH），协商对称加密的密钥，保证密钥传输的安全性；
        4. 数据完整性：通过摘要算法（SHA256）和数字签名，保证传输的数据不被篡改。
        
    
2. **辅助协议：PKI/CA 相关协议**
    
    HTTPS 的身份认证依赖 PKI（公钥基础设施）体系，额外使用了相关协议：
    
    - **X.509 协议**：定义了数字证书的标准格式，HTTPS 使用的 SSL 证书都是遵循 X.509 标准的；
    - **OCSP 协议（在线证书状态协议）**：用于实时查询 SSL 证书的吊销状态，避免使用已被吊销的证书，保障安全；
    - **CRL 协议（证书吊销列表）**：用于下载 CA 机构发布的已吊销证书列表，验证证书的有效性。
    
3. **可选扩展协议**
    
    - **HSTS（HTTP 严格传输安全协议）**：通过响应头`Strict-Transport-Security`，强制浏览器使用 HTTPS 访问网站，禁止 HTTP 访问，防止降级攻击；
    - **ALPN/NPN 协议**：用于 TLS 握手过程中协商应用层协议，比如协商 HTTP/2、HTTP/3，提升传输效率；
    - **SNI（服务器名称指示）协议**：TLS 握手时，客户端传递要访问的域名，让服务端可以返回对应域名的 SSL 证书，实现一个 IP 地址托管多个 HTTPS 站点。
    

#### 专家级拓展

- HTTPS 的完整握手流程（TLS 1.2）：
    
    1. 客户端发送 Client Hello，携带支持的 TLS 版本、加密套件、随机数；
    2. 服务端返回 Server Hello，确认 TLS 版本、加密套件、随机数，同时发送服务器的 SSL 证书；
    3. 客户端验证证书的有效性，通过 CA 公钥验证证书的数字签名，确认服务端身份；
    4. 客户端生成预主密钥，用服务端证书的公钥加密，发送给服务端；
    5. 服务端用自己的私钥解密，得到预主密钥；
    6. 客户端和服务端通过预主密钥和两个随机数，生成相同的会话密钥（对称加密密钥）；
    7. 客户端发送 Change Cipher Spec，通知服务端后续使用对称加密传输；
    8. 服务端返回 Change Cipher Spec，握手完成，开始加密传输 HTTP 数据。
    
- 生产环境最佳实践：
    
    1. 禁用 SSL 3.0、TLS 1.0/1.1，仅支持 TLS 1.2、TLS 1.3，提升安全性；
    2. 禁用不安全的加密套件，仅支持安全的 AES-GCM、ChaCha20-Poly1305 等加密套件；
    3. 开启 HSTS，防止降级攻击和中间人攻击；
    4. 配置 OCSP Stapling，将证书吊销状态随证书一起发送给客户端，提升握手速度和安全性。
    

#### 面试避坑指南

- 严禁只说「HTTPS 用了 SSL/TLS」，不说 TLS 的核心作用和其他辅助协议，会被判定为理解不全面；
- 避免混淆对称加密和非对称加密的作用，非对称加密仅用于握手阶段协商密钥，实际数据传输用的是对称加密，这是面试高频考点；
- 不要说「HTTPS 用了 SSL 协议」，SSL 已经被废弃，现在用的是 TLS 协议，说 SSL 会被判定为知识过时。

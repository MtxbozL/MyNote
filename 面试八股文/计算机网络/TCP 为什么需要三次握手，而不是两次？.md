#### 核心标准答案

TCP 是面向连接的、可靠的传输层协议，三次握手的核心目的是**确认客户端和服务端的发送能力、接收能力都正常，同时协商初始序列号 ISN，防止历史失效连接被初始化，避免资源浪费**。两次握手无法满足 TCP 的可靠传输要求，核心问题如下：

1. **无法防止历史失效连接导致的资源浪费**
    
    客户端发送的 SYN 请求因为网络延迟，在连接释放后才到达服务端。如果是两次握手，服务端收到这个失效的 SYN 后，会直接建立连接，分配资源，而客户端并不会认可这个连接，也不会发送任何数据，服务端会一直维护这个无效连接，浪费大量服务器资源。
    
    三次握手中，服务端收到 SYN 后，会回复 SYN+ACK，客户端会判断这个序列号是否是历史连接的，如果是，会发送 RST 报文终止连接，只有客户端确认是新连接，回复 ACK，服务端才会正式建立连接，避免无效资源占用。
2. **无法确认双方的收发能力都正常**
    
    - 两次握手只能确认：客户端的发送能力、服务端的接收能力正常；
    - 无法确认：服务端的发送能力、客户端的接收能力是否正常。
        
        三次握手中，客户端收到服务端的 SYN+ACK 并回复 ACK，才能证明服务端的发送能力、客户端的接收能力正常，最终完成双向能力的确认，保证后续的双向数据传输可靠。
    
3. **无法安全协商初始序列号 ISN**
    
    TCP 的序列号是可靠传输的核心，用于保证数据的有序性、不重复、不丢失。两次握手只能完成一方的序列号协商，无法完成双向的序列号确认；三次握手可以让客户端和服务端互相确认对方的初始序列号，保证双向数据传输的有序性和可靠性。

#### 专家级拓展

- 三次握手的核心报文流转：
    
    1. 第一次握手：客户端发送 SYN 报文，携带客户端的初始序列号 ISN (c)，进入 SYN_SENT 状态；
    2. 第二次握手：服务端收到 SYN 后，回复 SYN+ACK 报文，携带服务端的初始序列号 ISN (s)，同时确认号为 ISN (c)+1，进入 SYN_RCVD 状态；
    3. 第三次握手：客户端收到 SYN+ACK 后，回复 ACK 报文，确认号为 ISN (s)+1，客户端进入 ESTABLISHED 状态；服务端收到 ACK 后，也进入 ESTABLISHED 状态，连接正式建立。
    
- 进阶知识点：SYN 洪水攻击，就是利用了两次握手的缺陷，攻击者伪造大量源 IP 的 SYN 报文，服务端收到后会回复 SYN+ACK，并分配资源，攻击者不回复 ACK，导致服务端维护大量半连接，耗尽服务器资源。Linux 中可通过`tcp_syncookies`参数防御 SYN 洪水攻击，原理是服务端不保存半连接状态，而是通过 SYN 报文计算出一个 cookie，放在 SYN+ACK 中，只有客户端回复的 ACK 携带正确的 cookie，才会建立连接，避免资源浪费。

#### 面试避坑指南

- 严禁只说「三次握手是为了建立可靠连接」，不说两次握手的核心缺陷，会被面试官判定为死记硬背，不理解底层原理；
- 避免混淆「三次握手的双向能力确认」，必须明确两次握手无法确认服务端的发送能力和客户端的接收能力，这是核心考点；
- 不要遗漏「防止历史失效连接」这个核心原因，这是 TCP 设计者最优先考虑的问题，也是面试的加分细节。

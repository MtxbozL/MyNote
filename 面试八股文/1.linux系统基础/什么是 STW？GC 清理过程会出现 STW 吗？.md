#### 核心标准答案

STW 全称**Stop The World**，指垃圾回收（GC）过程中，内核暂停所有用户业务线程的执行，仅保留 GC 线程运行，直到 GC 的关键阶段完成后，再恢复用户线程的执行。

**所有基于追踪式的 GC 算法，都会出现 STW**，包括 Java、Go 等主流语言的 GC，区别仅在于现代 GC 算法通过优化，将 STW 的时间压缩到了毫秒级甚至微秒级，对业务几乎无感知。

#### 专家级拓展

- STW 的核心目的：解决 GC 并发标记阶段的「漏标问题」，保证垃圾标记的准确性，避免出现存活对象被误回收的严重故障。
- 不同语言 GC 的 STW 差异：
    
    1. Go 语言：采用三色标记法 + 混合写屏障，STW 仅发生在 GC 的标记开始和标记结束两个极短阶段，通常 STW 时间在 1ms 以内，与堆内存大小无关。
    2. Java 语言：
        
        - 串行 GC/Parallel GC：STW 时间与堆内存大小、存活对象数量正相关，堆内存越大，STW 时间越长，可达数百毫秒甚至秒级。
        - G1 GC：通过增量回收、Region 分区，将 STW 时间控制在用户设定的阈值内（默认 200ms）。
        - ZGC/Shenandoah GC：通过读屏障、并发转移，几乎实现了无 STW，最大 STW 时间不超过 1ms。
        
    
- 落地场景：运维 / 后端排查服务卡顿、接口超时问题时，核心排查点之一就是 GC 的 STW 时间 ——Go 语言可通过`go tool trace`、`prometheus metrics`查看 STW 耗时；Java 可通过`jstat`、`GC日志`、Arthas 等工具排查。

#### 面试避坑指南

- 严禁说「现代 GC 没有 STW」，这是原则性错误，只能说 STW 时间被极大优化，无法完全消除；
- 避免只说 STW 的定义，不说清它的作用、不同 GC 的差异，无法体现实际排查经验。

#### 核心标准答案

##### 一、日常编写的 Shell 自动化运维脚本

日常工作中，Shell 脚本主要用于实现**高频、重复、标准化的运维操作自动化**，提升运维效率，减少人工操作失误，核心编写的脚本分为 9 大类，具体如下：

1. **服务器日常巡检脚本**
    
    最常用的脚本，自动采集服务器的 CPU、内存、磁盘、网络、进程、硬件、日志等核心巡检指标，自动生成标准化的巡检报告，通过邮件 / 企业微信 / 钉钉自动发送，替代人工巡检。
    
    - 核心功能：自动执行巡检命令、汇总指标、判断异常、生成 HTML / 文本格式的巡检报告、自动发送给运维团队；
    - 执行方式：通过 crontab 定时执行，每日 / 每周自动巡检，异常指标标红告警。
    
2. **日志清理与轮转脚本**
    
    用于自动清理应用日志、系统日志、临时文件，避免日志占满磁盘，导致服务异常。
    
    - 核心功能：按文件大小、保留时间清理过期日志，压缩归档历史日志，删除临时文件、core dump 文件，记录清理日志，异常告警；
    - 适用场景：Nginx、MySQL、Tomcat 等应用的日志清理，/tmp、/var/log 目录的定期清理。
    
3. **数据备份与恢复脚本**
    
    用于业务数据、数据库、配置文件的自动备份，保证数据安全，实现备份的自动化、标准化。
    
    - 核心功能：自动备份 MySQL/PostgreSQL 数据库、业务数据、系统配置文件，备份文件压缩加密，自动上传到对象存储 / 备份服务器，自动清理过期备份，备份结果通知，备份完整性校验；
    - 执行方式：crontab 定时执行，全量备份 + 增量备份结合，定期做备份恢复验证。
    
4. **服务批量部署与更新脚本**
    
    用于批量部署、更新、重启业务服务，替代人工逐台服务器操作，保证多台服务器的部署一致性。
    
    - 核心功能：批量上传安装包、解压部署、修改配置、重启服务、验证服务状态，批量执行部署操作，记录执行日志，失败自动告警；
    - 适用场景：中小规模的业务服务部署、版本更新，替代 Ansible 的轻量级批量部署。
    
5. **服务健康检查与故障自愈脚本**
    
    用于监控业务服务的运行状态，服务异常时自动重启、自愈，减少业务中断时间。
    
    - 核心功能：定期检查服务进程状态、端口监听状态、接口可用性，服务异常时自动重启服务，记录故障日志，发送告警通知，多次重启失败则触发升级告警；
    - 适用场景：非核心业务服务、边缘服务的故障自愈，提升服务可用性。
    
6. **批量服务器操作脚本**
    
    用于批量在多台服务器上执行命令、修改配置、分发文件，实现批量运维操作自动化。
    
    - 核心功能：基于 SSH 密钥登录，批量在目标服务器上执行命令，批量上传 / 下载文件，批量修改配置文件，记录执行结果，失败的服务器单独汇总；
    - 适用场景：批量修改系统配置、批量安装软件、批量更新 hosts 解析，中小规模集群的批量操作。
    
7. **监控告警辅助脚本**
    
    用于补充监控系统的自定义监控、告警处理，实现个性化的监控需求。
    
    - 核心功能：采集监控系统不支持的自定义指标，推送到 Prometheus/Zabbix；监控到异常时，执行自定义的故障处理动作，比如自动扩容、重启服务、切换流量；
    - 适用场景：个性化业务指标监控、异常自动处理。
    
8. **系统初始化与安全加固脚本**
    
    用于新服务器的系统初始化、安全加固，标准化新服务器的配置，提升部署效率，保证所有服务器的配置一致。
    
    - 核心功能：关闭 SELinux / 防火墙、配置内核参数、优化系统配置、配置时间同步、配置 SSH 安全、创建运维用户、配置 yum 源、安装基础工具、安全基线加固；
    - 适用场景：新服务器上架、虚拟机初始化，替代人工逐台配置。
    
9. **故障排查与信息收集脚本**
    
    用于服务器故障时，自动收集系统、应用的故障信息，快速定位故障根因。
    
    - 核心功能：自动收集系统日志、应用日志、CPU / 内存 / 磁盘 / 网络状态、进程信息、内核报错、硬件状态，汇总生成故障信息报告，方便运维人员快速定位故障。
    

##### 二、是否常用 Shell 进行服务部署？

**会根据场景选择性使用，不是所有场景都用 Shell 做服务部署**，具体如下：

1. **常用 Shell 部署的场景**
    
    - 中小规模、简单的单体服务部署：比如单节点 / 少量节点的 Nginx、静态服务、简单的后端服务，Shell 脚本可以快速完成部署、配置、启动，无需复杂的工具，效率高；
    - 无自动化运维平台的环境：没有 Ansible、Jenkins、CICD 平台的环境，Shell 脚本是最便捷、最原生的部署方式，无需额外安装依赖；
    - 定制化的部署流程：有特殊的部署逻辑、前置检查、后置处理的部署场景，Shell 脚本可以灵活实现定制化的逻辑，适配个性化需求；
    - 应急部署、故障恢复场景：线上故障需要快速部署、回滚服务时，Shell 脚本可以快速执行，无需依赖其他工具，应急响应速度快。
    
2. **不使用 Shell 部署的场景**
    
    - 大规模、多节点的集群部署：几十上百台服务器的集群部署，Shell 脚本的批量执行能力、幂等性、错误处理能力不足，优先使用 Ansible、SaltStack 等批量运维工具；
    - 复杂的微服务部署：微服务有大量的依赖、配置、服务发现、滚动更新需求，Shell 脚本无法实现复杂的编排、灰度发布、回滚逻辑，优先使用 CICD 平台、K8s、Docker Compose 等工具；
    - 生产环境标准化部署：生产环境的正式部署，需要严格的版本控制、审计、权限控制、回滚机制，Shell 脚本的可追溯性、可审计性不足，优先使用 Jenkins、GitLab CI 等 CICD 流水线，实现部署的标准化、可追溯、可审计；
    - 有状态服务的部署：数据库、中间件等有状态服务的部署，需要复杂的初始化、主从配置、数据持久化逻辑，Shell 脚本无法保证幂等性和数据一致性，优先使用 Ansible Playbook、Operator 等工具。
    

#### 专家级拓展

- Shell 脚本编写的生产环境最佳实践：
    
    1. **规范编写**：脚本开头添加`#!/bin/bash`解释器，添加作者、时间、功能说明、使用方法；使用`set -euxo pipefail`，开启错误检测，脚本出现错误立即退出，避免错误继续执行；
    2. **幂等性设计**：脚本必须支持重复执行，重复执行不会导致异常，比如创建目录前先判断是否存在，修改配置前先判断是否已经修改，避免重复执行出现问题；
    3. **日志记录**：脚本必须记录执行日志，包括执行时间、执行步骤、执行结果、错误信息，方便后续审计和故障排查；
    4. **异常处理**：脚本必须添加异常处理，命令执行失败时，有对应的错误提示、告警、回滚逻辑，不能静默失败；
    5. **安全规范**：脚本中禁止硬编码密码、密钥等敏感信息，通过环境变量、配置文件读取；脚本必须设置合理的权限，禁止 777 权限，避免越权执行；
    6. **版本控制**：所有脚本都必须提交到 Git 仓库，做版本控制，记录变更历史，方便追溯和回滚。
    
- 进阶优化：
    
    1. 复杂的脚本，拆分为函数、模块化编写，提升脚本的可读性和可维护性；
    2. 脚本编写完成后，必须做全面的测试，在测试环境验证无误后，再到生产环境执行；
    3. 对于复杂的批量操作，先执行 dry run，模拟执行，确认无误后，再正式执行；
    4. 优先使用系统原生命令，避免使用不兼容的扩展命令，保证脚本在不同的 Linux 发行版上都能正常运行。
    

#### 面试避坑指南

- 严禁只说「会写 Shell 脚本」，不说具体的脚本类型和适用场景，会被面试官判定为只会写简单脚本，无实际的自动化运维经验；
- 避免说「所有服务部署都用 Shell」，会被判定为对运维部署体系理解不全面，生产环境复杂场景下，Shell 不是最优方案，必须分场景说明，体现你的选型能力；
- 不要遗漏 Shell 脚本的最佳实践，比如幂等性、异常处理、日志记录，这些是生产环境脚本的核心要求，提到会大幅加分，体现你有生产环境的脚本编写经验。

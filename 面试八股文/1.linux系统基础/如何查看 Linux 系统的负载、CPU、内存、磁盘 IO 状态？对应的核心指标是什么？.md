#### 核心标准答案

我会通过对应的专用命令，分维度查看系统状态，核心命令与关键指标如下：

1. **系统平均负载**：使用`uptime`、`top`命令查看，核心指标是**1 分钟、5 分钟、15 分钟的平均负载值**，代表单位时间内，系统中处于可运行状态和不可中断状态的平均进程数。
2. **CPU 使用率与状态**：使用`top`、`htop`、`vmstat`命令查看，核心指标：
    
    - `%us`：用户态 CPU 使用率，业务进程消耗的 CPU
    - `%sy`：内核态 CPU 使用率，系统调用消耗的 CPU
    - `%wa`：IO 等待 CPU 占比，磁盘 / 网络 IO 等待导致的 CPU 空闲
    - `%id`：CPU 空闲率
    
3. **内存使用状态**：使用`free -h`、`top`、`vmstat`命令查看，核心指标：
    
    - `total`：总物理内存
    - `used`：已使用内存
    - `available`：真正可用的内存（包含空闲内存 + 可回收的缓存 / 缓冲）
    - `buff/cache`：页缓存和缓冲区内存，用于提升磁盘 IO 性能，可被回收
    - `swap`：交换分区使用量，swap 占用过高代表物理内存不足
    
4. **磁盘空间与使用率**：使用`df -h`命令查看，核心指标是磁盘总容量、已用容量、使用率、挂载点，重点关注使用率超过 80% 的磁盘分区。
5. **磁盘 IO 性能**：使用`iostat -x 1`、`vmstat`、`dstat`命令查看，核心指标：
    
    - `%util`：磁盘的 IO 使用率，超过 80% 代表磁盘处于高负载状态
    - `rMB/s`、`wMB/s`：每秒读 / 写的数据量
    - `r/s`、`w/s`：每秒 IO 请求次数（IOPS）
    - `await`：IO 请求的平均等待时间，正常应小于 10ms，超过 30ms 代表磁盘 IO 性能严重不足
    

#### 专家级拓展

- 负载的核心判断标准：平均负载的合理值，最理想的是等于 CPU 核心数；如果 1 分钟负载远高于 15 分钟负载，代表系统近期出现突发负载，需要紧急排查；如果 15 分钟负载长期高于 CPU 核心数，代表系统 CPU 资源不足，需要扩容或优化程序。
- 生产环境故障排查逻辑：
    
    1. 系统卡顿，先看`uptime`的负载，再看`top`的 CPU 使用率：如果`%us`过高，定位占用 CPU 高的业务进程，优化代码；如果`%sy`过高，排查系统调用、内核参数问题；如果`%wa`过高，排查磁盘 IO 问题。
    2. 内存相关故障，优先看`available`，而不是`free`，Linux 会尽可能利用空闲内存做缓存，`free`低不代表内存不足，`available`低才是真正的内存紧张。
    3. 磁盘 IO 故障，`%util`100% 不代表磁盘性能到顶，比如 SSD 支持并发 IO，需要结合`await`、`IOPS`综合判断。
    
- 进阶工具：生产环境常用`prometheus + node_exporter + grafana`做这些指标的长期监控与告警，提前发现系统性能瓶颈。

#### 面试避坑指南

- 严禁把「平均负载」和「CPU 使用率」混淆，这是面试最高频的错误 —— 负载高不一定 CPU 使用率高，比如大量 IO 等待会导致负载高，但 CPU 空闲；CPU 使用率高也不一定负载高，比如单进程 CPU 跑满，多核 CPU 的负载可能很低。
- 避免说「free 内存低就是内存不足」，这是新手常见误区，会被面试官判定为无实际 Linux 运维经验。
- 不要只说命令名称，不说核心指标的含义，无法体现故障排查能力。

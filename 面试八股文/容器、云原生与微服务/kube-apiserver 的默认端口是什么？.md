#### 核心标准答案

kube-apiserver 是 K8s 集群的唯一入口，所有组件和客户端都通过 apiserver 交互，它提供了两个核心端口，分别用于不同的访问场景：

1. **安全端口（HTTPS 端口，默认值：6443）**
    
    - 这是 kube-apiserver 的**核心默认端口**，生产环境唯一推荐使用的端口；
    - 核心特性：基于 HTTPS 协议，开启了 TLS 加密、身份认证、鉴权、准入控制，所有集群组件（kubelet、controller-manager、scheduler、kubectl）、客户端都通过这个端口访问 apiserver；
    - 配置参数：通过`--secure-port`参数修改，默认 6443，绑定在所有网卡接口上。
    
2. **不安全端口（HTTP 端口，默认值：8080）**
    
    - 这是废弃的端口，K8s 1.20 + 版本已经默认禁用，生产环境严禁使用；
    - 核心特性：基于 HTTP 协议，无 TLS 加密、无身份认证、无鉴权，任何能访问这个端口的请求，都拥有 apiserver 的管理员权限，存在严重的安全风险；
    - 配置参数：通过`--insecure-port`参数开启，默认 0（禁用），生产环境必须保持禁用。
    

补充：旧版本 K8s 中，不安全端口默认绑定在本地回环地址 127.0.0.1，仅本地可访问，即使如此，依然存在安全风险，新版本已完全废弃。

#### 专家级拓展

- 生产环境最佳实践：
    
    1. 严格禁用不安全端口 8080，即使是测试环境也不建议开启，避免安全漏洞；
    2. 安全端口 6443，必须通过防火墙、安全组严格限制访问源 IP，只允许集群节点、运维管理 IP 访问，禁止对公网开放；
    3. 生产环境建议修改默认的 6443 端口，降低被扫描攻击的风险；
    4. 配置 apiserver 的 IP 白名单、RBAC 权限控制，最小化访问权限，避免越权访问。
    
- 进阶知识点：apiserver 的 6443 端口，是 K8s 集群高可用的核心端口，集群高可用部署时，负载均衡器需要转发流量到所有 apiserver 节点的 6443 端口，实现 apiserver 的负载均衡和高可用。

#### 面试避坑指南

- 严禁只说 8080 端口，不说 6443 端口，这是原则性错误，8080 已经被废弃禁用，6443 才是默认的核心端口；
- 避免混淆两个端口的安全特性，必须明确说明 6443 是加密认证的安全端口，8080 是无认证的不安全端口，生产环境禁用；
- 不要遗漏新版本的废弃说明，体现你对 K8s 版本迭代的了解，会大幅加分。

### 6. 有了 Docker，为什么还需要使用 K8s？

#### 核心标准答案

Docker 解决了应用的「打包和运行」问题，实现了「一次打包，到处运行」，解决了应用运行环境不一致的痛点；而 K8s 解决的是**大规模 Docker 容器的集群管理、自动化运维、高可用保障**问题，是 Docker 的上层编排平台，二者是互补关系，而非替代关系。

核心原因分为 6 个维度，单靠 Docker 无法解决，必须依赖 K8s：

1. **单靠 Docker 无法实现大规模容器的自动化编排与生命周期管理**
    
    Docker 只能管理单台服务器上的容器，当应用需要部署到几十、上百台服务器，需要上百个容器实例时，纯 Docker 完全无法应对：
    
    - 纯 Docker 场景下，需要人工逐台服务器执行 docker run 命令部署容器，无法实现批量部署、版本统一管理；
    - 应用更新时，需要人工逐台服务器停止旧容器、启动新容器，无法实现无感知的滚动更新、一键回滚，更新过程中业务会中断，且极易出现版本不一致的问题；
    - 应用扩缩容时，需要人工判断服务器资源，手动创建 / 删除容器，无法根据流量、负载自动扩缩容，应对流量峰值的能力极差。
        
        而 K8s 通过声明式 API，一键实现应用的批量部署、滚动更新、回滚、自动扩缩容，完全解决了大规模容器的编排问题。
    
2. **单靠 Docker 无法实现应用的故障自愈，高可用能力为零**
    
    这是生产环境最核心的痛点，Docker 本身没有任何故障自愈能力：
    
    - 纯 Docker 场景下，容器崩溃、应用进程挂掉，除非人工配置监控和重启脚本，否则容器不会自动重启；
    - 服务器宕机，该服务器上的所有容器都会停止，Docker 无法自动将这些容器迁移到其他健康的服务器上，业务会直接中断；
    - 应用出现无响应、健康检查失败，Docker 无法感知，也不会自动重启容器，会导致业务请求持续失败。
        
        而 K8s 通过健康检查、故障重启、节点驱逐、自动重建，实现了应用的故障自愈，无需人工干预，大幅提升了业务的可用性。
    
3. **单靠 Docker 无法实现统一的服务发现与负载均衡**
    
    微服务架构下，应用之间需要互相调用，纯 Docker 场景下，没有统一的服务发现能力：
    
    - 容器的 IP 是动态变化的，重启、重建后 IP 会改变，纯 Docker 场景下，需要人工维护服务的 IP 地址，极易出现配置错误；
    - 一个应用部署了多个容器实例，Docker 无法实现请求的负载均衡，需要额外部署 Nginx，人工配置上游服务器，容器扩缩容时需要手动修改 Nginx 配置，运维成本极高。
        
        而 K8s 通过 Service 资源 + CoreDNS，内置了服务发现和负载均衡，容器的 IP 变化、扩缩容完全不影响业务访问，无需人工干预。
    
4. **单靠 Docker 无法实现集群资源的统一调度与高效利用**
    
    纯 Docker 场景下，容器的部署完全依赖人工，无法实现资源的智能调度：
    
    - 人工部署容器时，很容易出现部分服务器资源被打满，部分服务器资源闲置的情况，资源利用率极低；
    - 无法限制容器的 CPU、内存资源上限，单个容器占用所有服务器资源，会导致其他容器被 OOM 杀死，业务故障；
    - 无法实现容器的智能调度，比如将高 IO 的容器调度到 SSD 磁盘的服务器，将相关应用调度到同一个机房，提升性能。
        
        而 K8s 的调度器会智能调度容器，平衡集群的资源使用率，实现资源的最大化利用，同时通过资源限制保证集群的稳定性。
    
5. **单靠 Docker 无法实现企业级的多租户、权限管控、安全隔离**
    
    企业级场景下，需要多团队共享集群，纯 Docker 没有任何多租户和权限管控能力：
    
    - 纯 Docker 场景下，任何能访问 Docker daemon 的用户，都可以操作所有容器，没有细粒度的权限控制，存在严重的安全风险；
    - 无法实现不同团队、不同环境的网络隔离，测试环境的容器可以直接访问生产环境的容器，极易出现安全事故；
    - 没有审计日志，无法追溯容器的操作记录，无法满足合规审计要求。
        
        而 K8s 通过 Namespace、RBAC 权限控制、网络策略、审计日志，实现了企业级的多租户隔离和安全管控，满足合规要求。
    
6. **单靠 Docker 无法实现存储、网络的统一编排管理**
    
    容器化应用需要持久化存储、跨主机网络通信，纯 Docker 的原生能力极弱：
    
    - 纯 Docker 的本地卷，只能在单台服务器上使用，容器迁移到其他服务器后，无法访问之前的存储数据，无法支持有状态服务；
    - 跨主机的容器网络通信，需要人工配置端口映射、路由规则，运维成本极高，无法支持大规模集群的网络互通。
        
        而 K8s 通过 CSI 存储插件、CNI 网络插件，实现了持久化存储、跨主机容器网络的统一编排，支持有状态服务的部署，适配各种存储和网络环境。
    

#### 专家级拓展

- 场景边界说明：
    
    1. 如果是单台服务器、少量容器的测试环境、单体应用，纯 Docker+Docker Compose 完全足够，不需要使用 K8s，反而会增加复杂度；
    2. 如果是多服务器集群、微服务架构、大规模容器部署、生产环境高可用要求，必须使用 K8s，纯 Docker 完全无法满足生产环境的需求。
    
- 补充：Docker Swarm 是 Docker 官方的容器编排工具，试图解决 Docker 的集群管理问题，但它的功能、生态、社区活跃度远不如 K8s，已经基本被市场淘汰，K8s 已经成为容器编排的事实标准。

#### 面试避坑指南

- 严禁只说「K8s 可以编排容器」，不说具体的痛点和 Docker 的局限性，会被面试官判定为只会背概念，不理解二者的核心差异；
- 避免否定 Docker 的价值，必须明确二者是互补关系，Docker 是 K8s 的底层容器运行时，K8s 是 Docker 的上层编排平台，不是替代关系；
- 不要只讲技术功能，不说生产环境的业务痛点，比如高可用、故障自愈、运维成本，这些才是企业选择 K8s 的核心原因，体现你对生产环境的理解。

#### 心标准答案

我会遵循「**先恢复业务，再定位根因，最后复盘优化**」的运维故障处理原则，分 5 个标准化步骤处理：

1. **故障发现与确认**
    
    通过监控告警（服务健康状态、接口错误率、实例心跳）确认故障实例，登录监控平台查看实例的 CPU / 内存 / 磁盘状态、服务进程状态、错误日志，确认实例是否真的宕机、是否可自动恢复。
2. **故障隔离与业务恢复（最高优先级）**
    
    - 自动隔离：如果服务接入了注册中心（Nacos/Eureka/Consul），注册中心会通过健康检查自动摘除故障实例，流量会自动转发到其他健康实例，无需人工干预；
    - 手动隔离：如果自动隔离失效，手动在注册中心 / 负载均衡器中下线故障实例，禁止流量继续打入，避免用户请求失败；
    - 实例恢复：如果是 K8s 部署的服务，K8s 会自动重启故障 Pod，或重新创建新的 Pod 实例；如果是物理机 / 虚拟机部署，手动重启服务进程，或快速扩容新的实例，补充服务能力，保证业务可用。
    
3. **根因定位与排查**
    
    业务恢复后，深入排查故障根因，核心排查方向：
    
    - 程序层面：查看服务日志，是否有 OOM、死锁、空指针、依赖服务调用超时等代码问题；
    - 资源层面：查看服务器的 CPU / 内存 / 磁盘 IO / 网络带宽，是否有资源耗尽、磁盘满、端口占用等系统问题；
    - 依赖层面：排查数据库、Redis、MQ 等中间件是否异常，是否是依赖服务故障导致的级联宕机；
    - 环境层面：排查服务器是否宕机、机房网络是否中断、容器是否被 OOM Kill 等基础设施问题。
    
4. **故障修复与验证**
    
    根据根因针对性修复，比如修复代码 bug、优化 JVM 参数、扩容服务器资源、修复依赖服务故障；修复完成后，重新上线实例，接入流量，持续监控服务的健康状态、接口成功率、资源使用率，确认服务完全恢复正常。
5. **事后复盘与优化**
    
    记录故障的时间、影响范围、根因、处理过程，组织复盘会，制定优化措施，比如：优化服务的健康检查机制、增加服务的熔断降级能力、完善监控告警、优化代码避免 OOM、配置实例的自动扩缩容策略，避免同类故障再次发生。

#### 专家级拓展

- 生产环境自动自愈方案（核心加分项）：
    
    生产环境优先通过自动化机制实现故障的自动处理，无需人工干预，核心实现：
    
    1. 健康检查：注册中心开启 TCP/HTTP 接口健康检查，故障实例自动摘除；K8s 配置存活探针（livenessProbe）、就绪探针（readinessProbe），故障 Pod 自动重启、重建；
    2. 熔断降级：通过 Sentinel/Hystrix 配置服务熔断，故障实例的请求快速失败，避免级联故障；
    3. 自动扩缩容：基于 CPU / 内存使用率、请求量配置 HPA 自动扩缩容，实例故障时自动补充新的实例；
    4. 故障自愈脚本：配置自定义脚本，检测到服务进程宕机时，自动重启服务，同时上报告警。
    
- 进阶优化：核心业务服务配置多机房、多可用区部署，避免单机房故障导致所有实例宕机；同时配置限流策略，避免流量突增导致实例被打垮。

#### 面试避坑指南

- 严禁只说「重启实例」，不说完整的故障处理流程，会被面试官判定为「只会暴力处理，无体系化故障排查能力」，面试核心考察的是你的运维故障处理规范和风险管控能力；
- 避免遗漏「先隔离恢复业务，再排查根因」的核心原则，生产环境故障处理的第一优先级是恢复业务，减少用户影响，而不是先定位根因；
- 不要只讲手动处理，不说自动化自愈方案，这是区分普通运维和高级运维的核心点，提到自动自愈会大幅加分。

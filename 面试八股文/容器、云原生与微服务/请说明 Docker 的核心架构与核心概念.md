#### 核心标准答案

Docker 是一个开源的容器化引擎，基于 Linux 内核的 Namespace、Cgroups、UnionFS 等技术，实现了应用的打包、分发、运行，核心是**一次打包，到处运行**，解决了应用运行环境不一致的问题。

##### 一、Docker 的核心架构

Docker 采用**C/S（客户端 - 服务端）架构**，分为三大核心组件，完整架构如下：

1. **Docker Client（客户端）**
    
    是用户和 Docker 交互的入口，我们日常使用的`docker`命令就是 Docker Client，负责接收用户的命令，发送给 Docker Daemon，同时展示服务端返回的结果。Docker Client 可以和本地的 Docker Daemon 通信，也可以和远程的 Docker Daemon 通信，实现远程管理。
2. **Docker Daemon（守护进程）**
    
    是 Docker 的核心后台服务，运行在宿主机的后台，负责监听 Docker Client 的请求，管理 Docker 的镜像、容器、网络、数据卷等所有资源，执行容器的创建、启动、停止、删除等所有操作。Docker Daemon 通过 API 接口对外提供服务，是整个 Docker 架构的大脑。
3. **containerd（容器运行时）**
    
    是 Docker Daemon 和内核之间的中间层，负责容器的生命周期管理，包括容器的创建、启动、停止、删除、镜像管理、存储管理、网络管理等，遵循 OCI（开放容器倡议）标准，是 Docker 真正执行容器操作的组件。Docker 1.11 版本之后，containerd 从 Docker Daemon 中拆分出来，成为独立的开源项目，现在已经是云原生容器运行时的标准，被 K8s 等容器编排平台广泛使用。
4. **runc**
    
    是 OCI 运行时规范的参考实现，是最底层的容器运行时，负责和 Linux 内核交互，通过 Namespace、Cgroups 等技术，创建和隔离容器的运行环境，启动容器进程。containerd 通过调用 runc，实现容器的最终创建和运行。

##### 二、Docker 的核心概念

1. **镜像（Image）**
    
    镜像是容器的模板，是一个只读的、分层的文件系统，包含了应用运行所需的代码、依赖、库、环境变量、配置文件、启动命令等所有内容。镜像不包含任何动态数据，构建完成后不会被修改，是容器运行的基础。
    
    - 核心特性：分层存储，镜像由多个镜像层组成，每个层对应 Dockerfile 中的一条指令，相同的层可以复用，减少存储空间和传输带宽；镜像可以通过 Dockerfile 构建，也可以从镜像仓库拉取。
    
2. **容器（Container）**
    
    容器是镜像的运行实例，是一个独立的、隔离的运行环境，由镜像创建而来。镜像相当于类，容器相当于类的实例，一个镜像可以创建多个容器。
    
    - 核心特性：基于 Namespace 实现资源隔离（PID、网络、挂载、用户等隔离），基于 Cgroups 实现资源限制（CPU、内存、磁盘 IO 等限制），容器之间互相隔离，互不影响；容器是临时的，生命周期结束后，容器内的修改会丢失，需要通过数据卷持久化数据。
    
3. **仓库（Repository）/ 镜像仓库**
    
    镜像仓库是存储和分发 Docker 镜像的地方，相当于镜像的 "代码仓库"，分为公共镜像仓库和私有镜像仓库。
    
    - 公共镜像仓库：Docker Hub、阿里云镜像仓库、网易云镜像仓库等，提供大量的官方镜像，比如 Nginx、MySQL、Redis 等，用户可以免费拉取，也可以上传自己的镜像；
    - 私有镜像仓库：Harbor、Nexus 等，企业内部搭建的私有镜像仓库，用于存储企业内部的业务镜像，保证镜像的安全和权限管控。
    
4. **Dockerfile**
    
    Dockerfile 是一个文本文件，包含了构建 Docker 镜像的一系列指令，比如 FROM、RUN、COPY、CMD、ENTRYPOINT 等，通过`docker build`命令，读取 Dockerfile 中的指令，一步步构建出镜像，是实现镜像标准化、可重复构建的核心。
5. **数据卷（Volume）**
    
    数据卷是 Docker 实现数据持久化的机制，是宿主机上的一个目录或文件，可以挂载到容器中，容器内的数据写入数据卷，会直接写入宿主机的目录，容器删除后，数据卷中的数据不会丢失，实现了容器数据的持久化和共享。
6. **网络（Network）**
    
    Docker 提供了多种网络驱动，实现容器之间、容器和宿主机之间的网络通信，核心网络模式包括：bridge（桥接模式，默认）、host（主机模式）、none（无网络模式）、container（容器共享网络）、overlay（跨主机容器通信）。

#### 专家级拓展

- 容器的底层核心技术：
    
    1. **Namespace（命名空间）**：实现容器的资源隔离，包括 PID Namespace（进程隔离）、Net Namespace（网络隔离）、Mnt Namespace（挂载隔离）、UTS Namespace（主机名隔离）、IPC Namespace（进程间通信隔离）、User Namespace（用户权限隔离），每个容器都有自己独立的 Namespace，实现了容器之间的完全隔离。
    2. **Cgroups（控制组）**：实现容器的资源限制，限制容器可以使用的 CPU、内存、磁盘 IO、网络带宽等资源，避免单个容器占用宿主机的所有资源，保证宿主机和其他容器的稳定运行。
    3. **UnionFS（联合文件系统）**：实现镜像的分层存储，将多个镜像层联合挂载，形成一个统一的文件系统，镜像层是只读的，容器运行时会在顶部添加一个可写层，容器的所有修改都在可写层，不会修改底层的镜像层，实现了镜像的复用和容器的快速创建。
    
- 生产环境最佳实践：
    
    1. 镜像优化：最小化镜像体积，使用 Alpine 等基础镜像，多阶段构建，减少镜像层，清理无用依赖，提升镜像拉取速度，减少攻击面；
    2. 安全规范：镜像不使用 root 用户运行，禁止在镜像中硬编码敏感信息，镜像构建完成后做安全扫描，修复漏洞；
    3. 数据持久化：容器内的业务数据必须通过数据卷持久化，禁止将数据存储在容器内，避免容器删除导致数据丢失；
    4. 容器生命周期：容器是无状态的，一次性的，有状态的应用（数据库、中间件）不建议直接用 Docker 部署，优先使用 K8s 的 StatefulSet 部署。
    

#### 面试避坑指南

- 严禁只说镜像、容器两个概念，不说 Docker 的核心 C/S 架构，会被面试官判定为对 Docker 的底层架构理解不全面；
- 避免混淆 containerd 和 runc 的作用，containerd 是容器生命周期管理，runc 是和内核交互创建容器，不能说反，这是 Docker 架构的核心细节，也是面试高频考点；
- 不要遗漏容器底层的 Namespace、Cgroups 技术，这是 Docker 容器隔离和限制的核心，提到会大幅加分，体现你对容器底层原理的理解。

#### 核心标准答案

延迟双删是 Redis 缓存场景中，**解决「更新数据库 + 更新缓存」场景下，并发操作导致的缓存与数据库数据不一致问题**的经典方案，核心是通过两次缓存删除 + 延迟等待，保证最终一致性。

1. **核心解决的问题**
    
    常规的「先删除缓存，再更新数据库」方案，在高并发场景下会出现脏数据：
    
    - 线程 A 先删除缓存，准备更新数据库；
    - 此时线程 B 发起查询，缓存为空，从数据库读取到旧数据，写入缓存；
    - 线程 A 更新数据库完成，最终缓存中的是旧数据，数据库是新数据，出现数据不一致，且缓存过期前一直是脏数据。
    
2. **延迟双删的完整执行流程**
    
    1. **第一次缓存删除**：更新数据前，先删除 Redis 中的对应缓存；
    2. **更新数据库**：执行数据库的更新操作，修改数据；
    3. **延迟等待**：等待一段固定的时间（通常几百毫秒，比如 300ms-500ms）；
    4. **第二次缓存删除（延迟删除）**：延迟时间到后，再次删除 Redis 中的对应缓存，清除并发场景下可能写入的脏数据。
    
3. **核心原理**
    
    延迟等待的时间，必须大于「线程读取数据库 + 写入缓存」的耗时，保证第二次删除时，并发线程写入的脏缓存已经完成，第二次删除可以把脏缓存清理掉，后续的查询会重新从数据库读取最新数据，写入缓存，保证最终一致性。

#### 专家级拓展

- 延迟时间的设置依据：
    
    延迟时间必须大于「业务读请求的平均耗时 + Redis 写入的耗时」，生产环境通常设置为 300ms-1s，具体根据业务的接口耗时调整；如果是分布式场景，还要加上网络传输的耗时。
- 生产环境优化方案：
    
    1. **异步延迟删除**：第二次删除不要阻塞主线程，通过线程池、MQ 异步执行延迟删除，避免延迟等待影响主线程的接口响应时间；
    2. **重试机制**：第二次删除可能因为 Redis 故障、网络问题失败，需要配置重试机制，比如删除失败后，通过 MQ 重试几次，保证删除成功；
    3. **兜底方案**：给缓存设置合理的过期时间，即使延迟双删失败，缓存过期后也会重新读取最新数据，避免脏数据长期存在；
    4. **终极方案：监听 binlog 异步删除**：基于 Canal/Debezium 监听 MySQL 的 binlog，数据变更后，异步删除缓存，完全解耦业务代码，保证数据一致性，是生产环境的首选方案。
    
- 方案对比：
    

| 方案          | 优点                         | 缺点                     | 适用场景          |
| :---------- | :------------------------- | :--------------------- | :------------ |
| 延迟双删        | 实现简单，无额外组件依赖，解决绝大多数并发脏数据问题 | 延迟时间不好把控，极端场景仍有不一致风险   | 中小规模、并发量不高的业务 |
| binlog 监听删除 | 完全解耦业务，一致性高，无侵入性           | 需要引入 Canal 等额外组件，架构更复杂 | 高并发、大规模生产环境   |
| 更新缓存        | 缓存命中率高                     | 并发场景下脏数据概率高，容易出现更新覆盖   | 读多写少、无并发更新的场景 |


#### 面试避坑指南

- 严禁只说延迟双删的步骤，不说解决的核心问题，会被面试官判定为「只会背方案，不懂背后的原理」；
- 避免说「延迟双删可以保证强一致性」，延迟双删只能保证最终一致性，极端并发场景下，仍有短暂的不一致窗口，不能说强一致性；
- 不要遗漏延迟时间的设置依据，面试会大概率追问「延迟时间应该设置多久」，必须提前准备好。

#### 核心标准答案

ACID 是关系型数据库事务的四大核心特性，是保证事务正确执行、数据一致性的根本准则，具体定义如下：

1. **原子性（Atomicity）**：事务是一个不可分割的最小执行单元，事务中的所有操作，要么全部执行成功，要么全部执行失败回滚，不会出现部分成功、部分失败的情况。
2. **一致性（Consistency）**：事务执行前后，数据库的完整性约束、业务规则保持一致，数据的状态始终符合业务预期。比如转账事务，A 账户扣钱、B 账户加钱，执行前后两个账户的总金额始终保持一致。
3. **隔离性（Isolation）**：多个并发执行的事务之间，相互隔离，互不干扰，一个事务的中间状态对其他事务不可见，避免并发事务之间互相影响，导致数据不一致。
4. **持久性（Durability）**：一个事务一旦提交成功，对数据库中数据的修改就是永久的，后续的任何操作、系统宕机、硬件故障，都不会导致提交的数据丢失。

#### 专家级拓展

- ACID 四大特性的底层实现（以 MySQL InnoDB 引擎为例）：
    
    1. 原子性：核心靠**undo log（回滚日志）** 实现。事务执行的所有修改，都会记录对应的 undo log，当事务执行失败、执行 rollback 时，通过 undo log 反向回滚已经执行的操作，恢复到事务开始前的状态，保证原子性。
    2. 一致性：是事务的最终目标，靠原子性、隔离性、持久性共同保障，同时配合数据库的约束（主键约束、唯一约束、外键约束）、业务层的逻辑校验，共同保证数据的一致性。
    3. 隔离性：核心靠**锁机制 + MVCC（多版本并发控制）** 实现。通过锁机制解决并发写的冲突问题，通过 MVCC 实现读写分离，避免读写互相阻塞，同时通过不同的事务隔离级别，控制隔离的程度，平衡并发性能与数据一致性。
    4. 持久性：核心靠**redo log（重做日志）** 实现。事务提交时，不会立即将数据修改刷入磁盘，只会先写入 redo log，之后通过后台线程异步刷盘；当系统宕机重启时，通过 redo log 恢复已经提交但未刷入磁盘的数据，保证持久性。
    
- 核心关系：原子性、隔离性、持久性是手段，一致性是最终目的，三大手段共同保障数据的一致性。

#### 面试避坑指南

- 严禁只背诵四个单词的中文翻译，不说每个特性的核心含义，会被判定为死记硬背，无实际理解；
- 避免混淆四大特性的底层实现，比如把 undo log 说成持久性的实现，redo log 说成原子性的实现，这是面试的核心扣分点；
- 不要只说定义，不说 MySQL InnoDB 的底层实现，面试问这个问题，90% 会追问底层实现，提前准备能大幅加分。

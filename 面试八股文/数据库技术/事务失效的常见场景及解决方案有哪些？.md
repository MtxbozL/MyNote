#### 核心标准答案

事务失效是 Spring 框架中 Java 后端开发的高频问题，仅当事务满足 Spring 的事务管理规则时才会生效，常见的失效场景与对应解决方案如下：

1. **场景 1：非 public 修饰的方法**
    
    Spring 的声明式事务基于 AOP 动态代理实现，只有 public 修饰的方法，才会被 Spring 的事务代理拦截，private、protected、default 修饰的方法，事务会直接失效。
    
    - 解决方案：将事务方法改为 public 修饰。
    
2. **场景 2：类内部方法调用，非代理对象调用**
    
    同一个类内部，一个非事务方法直接调用本类的事务方法，是通过 this 对象调用，不是通过 Spring 的代理对象调用，事务拦截器无法拦截到这个调用，事务失效。
    
    - 解决方案：
        
        1. 将事务方法放到单独的类中，通过注入的代理对象调用；
        2. 在当前类中注入自身的代理对象，通过代理对象调用事务方法；
        3. 使用`AopContext.currentProxy()`获取当前类的代理对象，通过代理对象调用事务方法。
        
    
3. **场景 3：异常被捕获，没有抛出**
    
    Spring 的事务回滚规则，默认只在方法抛出未捕获的运行时异常（RuntimeException）和 Error 时，才会回滚；如果方法内用 try-catch 捕获了异常，没有继续抛出，事务切面无法感知到异常，不会触发回滚，事务失效。
    
    - 解决方案：
        
        1. 捕获异常后，在 catch 块中继续抛出异常，让事务切面感知到；
        2. 手动触发回滚，在 catch 块中执行`TransactionAspectSupport.currentTransactionStatus().setRollbackOnly()`。
        
    
4. **场景 4：事务传播属性设置错误**
    
    Spring 的事务传播属性，控制事务在方法调用之间的传递，如果传播属性设置错误，会导致事务失效。比如设置为`Propagation.NOT_SUPPORTED`（以非事务方式运行）、`Propagation.NEVER`（非事务方式运行，存在事务则抛出异常），会直接导致事务不生效。
    
    - 解决方案：使用默认的`Propagation.REQUIRED`传播属性（当前存在事务则加入，不存在则新建事务），仅在特殊场景下调整传播属性。
    
5. **场景 5：异常类型不匹配，rollbackFor 设置错误**
    
    Spring 默认只回滚 RuntimeException 和 Error，不会回滚受检异常（比如 IOException、SQLException 等自定义受检异常）；如果方法抛出了受检异常，但是没有在`@Transactional`注解中指定`rollbackFor`，事务不会回滚，看似失效。
    
    - 解决方案：在`@Transactional`注解中明确指定`rollbackFor = Exception.class`，覆盖所有异常类型，避免遗漏受检异常。
    
6. **场景 6：方法被 final/static 修饰**
    
    Spring 的 AOP 动态代理，是通过继承类、重写方法实现的，如果方法被 final/static 修饰，无法被重写，代理对象无法拦截该方法，事务失效。
    
    - 解决方案：移除事务方法的 final/static 修饰符。
    
7. **场景 7：数据库不支持事务**
    
    事务是数据库层面实现的，如果使用的 MySQL 存储引擎是 MyISAM，MyISAM 不支持事务，无论 Spring 怎么配置，事务都不会生效。
    
    - 解决方案：将数据库表的存储引擎改为 InnoDB。
    
8. **场景 8：多线程场景下的事务失效**
    
    Spring 的事务是通过 ThreadLocal 绑定到当前线程的，多线程场景下，子线程的事务和主线程的事务不在同一个线程上下文中，互相独立，子线程抛出异常，不会导致主线程的事务回滚，反之亦然。
    
    - 解决方案：多线程场景下，避免跨线程的事务控制，将需要事务控制的逻辑放到同一个线程中；如果必须分布式场景，使用分布式事务（Seata、XA）。
    

#### 专家级拓展

- 核心底层原理：Spring 的声明式事务，基于 AOP 动态代理实现，目标方法被调用时，Spring 的事务拦截器（TransactionInterceptor）会在方法执行前开启事务，方法正常执行完成后提交事务，方法抛出异常时回滚事务。所有事务失效的场景，本质都是「事务拦截器没有拦截到方法的执行」或者「没有触发回滚规则」。
- 生产环境最佳实践：
    
    1. 所有`@Transactional`注解，必须加上`rollbackFor = Exception.class`，这是避免事务失效的最基础规范。
    2. 避免在同一个类内部调用事务方法，尽量将事务逻辑放到单独的 Service 类中，通过注入的对象调用，从根本上避免内部调用导致的失效。
    3. 事务方法尽量精简，不要在事务方法中写大量的非 DB 操作、远程调用，避免长事务导致的数据库连接占用、锁等待超时问题。
    4. 尽量使用编程式事务（TransactionTemplate），替代声明式事务，完全控制事务的开启、提交、回滚，避免声明式事务的各种失效场景，同时更灵活。
    
- 排查技巧：事务失效时，优先查看方法的修饰符、是否内部调用、异常是否被捕获，这三个场景占了事务失效的 90% 以上；同时开启 Spring 的事务 debug 日志，查看事务是否开启、是否触发回滚，快速定位问题。

#### 面试避坑指南

- 严禁只说零散的场景，不说底层原理，会被面试官判定为「只会背场景，不理解本质，遇到新的失效场景无法排查」；
- 避免遗漏`rollbackFor`的场景，这是生产环境最高频的事务失效问题，也是面试必问的核心场景；
- 不要说「try-catch 一定会导致事务失效」，只有捕获异常后不抛出、不手动回滚，才会失效，说的太绝对会被面试官判定为理解不透彻。

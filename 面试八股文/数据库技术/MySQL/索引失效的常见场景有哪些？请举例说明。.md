#### 核心标准答案

MySQL 索引失效的核心原因是：**SQL 语句的写法，导致 MySQL 优化器无法使用索引，只能选择全表扫描**，常见的失效场景及示例如下，所有示例均基于表`user`，包含索引`idx_name_age_create_time(name, age, create_time)`（联合索引，遵循最左前缀原则）、主键索引`PRIMARY KEY(id)`、普通索引`idx_phone(phone)`。

1. **场景 1：联合索引不遵循最左前缀原则**
    
    联合索引必须从最左侧的字段开始匹配，跳过左侧字段、中间字段，都会导致索引失效，或只能匹配部分前缀。
    
    - 失效示例：
        
        sql
        
        ```
        -- 跳过最左侧的name字段，直接用age查询，完全无法使用联合索引
        SELECT * FROM user WHERE age = 20;
        -- 跳过中间的age字段，只能使用name字段的索引，无法使用create_time的索引
        SELECT * FROM user WHERE name = '张三' AND create_time = '2024-01-01';
        ```
        
    - 生效示例：
        
        sql
        
        ```
        -- 按最左前缀顺序匹配，联合索引完全生效
        SELECT * FROM user WHERE name = '张三' AND age = 20 AND create_time = '2024-01-01';
        -- 匹配最左前缀的前N个字段，索引部分生效
        SELECT * FROM user WHERE name = '张三' AND age = 20;
        ```
        
    
2. **场景 2：在索引字段上使用函数、表达式计算**
    
    在索引字段上执行任何函数、表达式计算，都会导致索引失效，因为 MySQL 无法通过索引树匹配计算后的值。
    
    - 失效示例：
        
        sql
        
        ```
        -- 索引字段使用函数，失效
        SELECT * FROM user WHERE LEFT(name, 2) = '张三';
        SELECT * FROM user WHERE YEAR(create_time) = 2024;
        -- 索引字段使用表达式计算，失效
        SELECT * FROM user WHERE age + 1 = 21;
        SELECT * FROM user WHERE id * 2 = 20;
        ```
        
    - 生效示例：
        
        sql
        
        ```
        -- 把计算放在值上，索引字段不做任何处理，索引生效
        SELECT * FROM user WHERE create_time >= '2024-01-01' AND create_time < '2025-01-01';
        SELECT * FROM user WHERE age = 20;
        SELECT * FROM user WHERE id = 10;
        ```
        
    
3. **场景 3：隐式类型转换**
    
    索引字段的类型和查询值的类型不匹配，MySQL 会自动做隐式类型转换，导致索引失效。最常见的是字符串字段用数字查询。
    
    - 失效示例：
        
        sql
        
        ```
        -- phone是varchar字符串类型，查询值是数字，发生隐式转换，索引失效
        SELECT * FROM user WHERE phone = 13800138000;
        ```
        
    - 生效示例：
        
        sql
        
        ```
        -- 查询值和字段类型一致，索引生效
        SELECT * FROM user WHERE phone = '13800138000';
        ```
        
    
4. **场景 4：模糊查询以通配符 % 开头**
    
    模糊查询中，% 在前面，MySQL 无法通过 B + 树的前缀匹配索引，导致索引失效。
    
    - 失效示例：
        
        sql
        
        ```
        -- %在开头，索引失效
        SELECT * FROM user WHERE name LIKE '%张三';
        SELECT * FROM user WHERE name LIKE '%张三%';
        ```
        
    - 生效示例：
        
        sql
        
        ```
        -- %在结尾，匹配前缀，索引生效
        SELECT * FROM user WHERE name LIKE '张三%';
        ```
        
    
5. **场景 5：使用 OR 连接非索引字段**
    
    OR 条件中，只要有一个字段没有索引，整个 SQL 的索引都会失效，MySQL 会全表扫描所有条件，避免漏数据。
    
    - 失效示例：
        
        sql
        
        ```
        -- name有索引，address无索引，OR连接后，索引失效
        SELECT * FROM user WHERE name = '张三' OR address = '北京市';
        ```
        
    - 生效示例：
        
        sql
        
        ```
        -- OR两边的字段都有索引，索引生效（会走index_merge索引合并）
        SELECT * FROM user WHERE name = '张三' OR phone = '13800138000';
        -- 拆分为两个UNION ALL查询，索引生效
        SELECT * FROM user WHERE name = '张三'
        UNION ALL
        SELECT * FROM user WHERE address = '北京市';
        ```
        
    
6. **场景 6：使用反向查询，导致索引失效**
    
    使用`!=`、`<>`、`NOT IN`、`IS NOT NULL`等反向查询，大概率会导致索引失效，MySQL 优化器会认为全表扫描比走索引更快。
    
    - 失效示例：
        
        sql
        
        ```
        -- 反向查询，索引失效
        SELECT * FROM user WHERE name != '张三';
        SELECT * FROM user WHERE age NOT IN (20, 21, 22);
        SELECT * FROM user WHERE id <> 10;
        ```
        
    - 补充：主键的反向查询，大概率还是会走索引，比如`id != 10`，普通索引的反向查询基本都会失效。
    
7. **场景 7：MySQL 优化器判断全表扫描比走索引更快**
    
    当表的数据量很小，或者查询的结果集占表的大部分数据时，MySQL 优化器会认为全表扫描比走索引更高效，主动放弃使用索引。
    
    - 失效示例：
        
        sql
        
        ```
        -- 表只有几百条数据，全表扫描比走索引更快，索引失效
        SELECT * FROM user WHERE name = '张三';
        -- 查询结果集占表的80%，全表扫描更高效，索引失效
        SELECT * FROM user WHERE age > 10; -- 表中90%的age都大于10
        ```
        
    
8. **场景 8：索引字段允许为 NULL，使用 IS NULL/IS NOT NULL 查询**
    
    索引字段设置为允许 NULL 时，使用`IS NULL`/`IS NOT NULL`查询，可能会导致索引失效，具体取决于数据分布和优化器判断。
    
    - 失效示例：
        
        sql
        
        ```
        -- name字段允许为NULL，IS NOT NULL查询，索引失效
        SELECT * FROM user WHERE name IS NOT NULL;
        ```
        
    - 最佳实践：索引字段设置为 NOT NULL，设置默认值，避免 NULL 值导致的索引失效。
    

#### 专家级拓展

- 索引失效的底层原理：
    
    MySQL 的索引是 B + 树结构，数据是有序排列的，只能通过「前缀匹配、等值匹配、范围匹配」的方式，快速定位索引树中的数据。上述所有失效场景，本质都是破坏了 B + 树的匹配规则，MySQL 无法通过索引树快速定位数据，只能选择全表扫描。
- 验证索引是否生效的方法：
    
    必须通过`explain`命令查看执行计划，核心看`type`字段（是否为 ALL）、`key`字段（是否为 NULL）、`rows`字段（扫描行数），不能仅凭 SQL 写法判断索引是否生效，MySQL 优化器可能会有特殊的优化。
- 生产环境最佳实践：
    
    1. 联合索引必须遵循最左前缀原则，查询频率高、区分度高的字段放在最左侧；
    2. 禁止在索引字段上使用函数、表达式计算，所有计算都放在查询值上；
    3. 严格匹配字段类型，避免隐式类型转换，尤其是字符串类型的字段，查询值必须加引号；
    4. 索引字段尽量设置为 NOT NULL，设置默认值，避免 NULL 值带来的索引失效问题；
    5. 避免使用 OR 连接非索引字段，尽量拆分为 UNION ALL 查询。
    

#### 面试避坑指南

- 严禁只说场景名称，不举 SQL 示例，面试时必须能写出对应的失效和生效 SQL，才能证明你真的理解；
- 避免说「所有反向查询都会导致索引失效」，主键的反向查询通常还是会走索引，要区分场景，不能说的太绝对；
- 不要遗漏优化器主动放弃索引的场景，很多时候不是索引不能用，是优化器认为全表扫描更快，这是面试的加分细节。

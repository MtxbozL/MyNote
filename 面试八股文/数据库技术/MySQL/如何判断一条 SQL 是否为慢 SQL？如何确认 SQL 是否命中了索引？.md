#### 核心标准答案

##### 一、如何判断一条 SQL 是否为慢 SQL？

判断慢 SQL 分为**技术层面的标准判断**和**业务层面的自定义判断**，核心方法如下：

1. **技术层面：基于慢查询日志的标准判断**
    
    MySQL 内置了慢查询日志机制，可自动识别慢 SQL，核心判断标准：
    
    - **核心阈值：long_query_time**：MySQL 通过`long_query_time`参数设置慢 SQL 的时间阈值，单位是秒，默认值 10 秒，生产环境通常设置为 1 秒，高并发核心业务建议设置为 0.5 秒甚至 0.2 秒；
    - **判断规则**：SQL 的实际执行时间超过`long_query_time`设置的阈值，就会被判定为慢 SQL，自动记录到慢查询日志中；
    - **补充规则**：开启`log_queries_not_using_indexes`参数后，没有使用索引的 SQL，即使执行时间低于阈值，也会被记录到慢查询日志中，属于潜在的慢 SQL。
    
2. **业务层面：基于业务容忍度的自定义判断**
    
    即使 SQL 没有达到慢查询日志的阈值，只要不符合业务的性能要求，也会被判定为慢 SQL：
    
    - 接口响应超时：业务接口的响应时间超过了业务容忍的阈值（比如 500ms），其中 SQL 执行占了大部分时间，即使 SQL 执行时间只有 200ms，也属于需要优化的慢 SQL；
    - 高频执行的 SQL：单次执行时间很短（比如 10ms），但每秒执行上万次，累计占用大量数据库 CPU 资源，也属于需要优化的慢 SQL；
    - 扫描行数过多：SQL 的扫描行数远大于返回行数，即使执行时间短，也是潜在的慢 SQL，数据量增长后会快速变慢。
    
3. **实时判断方法**
    
    登录 MySQL，执行`show processlist;`，查看`Time`字段，就是 SQL 的执行时间，超过业务阈值的，就是正在执行的慢 SQL。

##### 二、如何确认 SQL 是否命中了索引？

核心方法是通过`EXPLAIN`命令查看 SQL 的执行计划，这是最权威、最常用的方法，核心看执行计划中的 5 个关键字段：

1. **核心字段 1：type（访问类型）**
    
    表示 MySQL 在表中找到所需行的方式，是判断索引是否生效、索引效率的核心指标，性能从优到劣依次为：
    
    `system > const > eq_ref > ref > range > index > ALL`
    
    - 只要`type`的值不是`ALL`，就代表 SQL 使用了索引（`ALL`代表全表扫描，索引完全失效）；
    - `range`及以上级别（`range`/`ref`/`eq_ref`/`const`/`system`），代表索引高效生效；
    - `index`级别：代表全索引扫描，虽然遍历了整个索引树，但比全表扫描快一点，属于索引低效使用，本质还是需要优化。
    
2. **核心字段 2：key（实际使用的索引）**
    
    表示 MySQL 实际执行时，真正使用的索引名称，是判断索引是否命中的直接依据：
    
    - 如果`key`字段不为 NULL，显示了索引名称，代表 SQL 命中了对应的索引；
    - 如果`key`字段为 NULL，代表 SQL 没有命中任何索引，索引完全失效。
    
3. **核心字段 3：key_len（使用的索引长度）**
    
    表示 MySQL 使用的索引的字节长度，可用于判断联合索引命中了多少个字段：
    
    - 比如联合索引`idx_name_age(name varchar(20), age int)`，name 的长度是 20*4+2=82 字节，age 的长度是 4 字节；
    - 如果 key_len=82，代表只命中了 name 字段；如果 key_len=86，代表命中了 name+age 两个字段，联合索引完全生效。
    
4. **核心字段 4：rows（扫描行数）**
    
    表示 MySQL 为了找到所需数据，预计需要扫描的行数，数值越小，索引效率越高：
    
    - 命中索引的 SQL，扫描行数会接近返回行数；
    - 如果扫描行数远大于返回行数，即使 key 字段不为 NULL，也代表索引效率极低，需要优化。
    
5. **核心字段 5：Extra（额外信息）**
    
    补充说明 SQL 的执行细节，可辅助判断索引的使用情况：
    
    - `Using index`：使用了覆盖索引，SQL 需要的所有字段都在索引中，无需回表查询，是索引使用的最优状态；
    - `Using where`：使用了 WHERE 条件过滤，但没有使用索引，或者索引过滤不充分；
    - `Using filesort`：无法通过索引完成排序，索引失效或未覆盖排序字段；
    - `Using temporary`：使用了临时表，通常是 GROUP BY 未命中索引。
    

#### 专家级拓展

- 进阶验证方法：
    
    1. `show profile`：开启后，可查看 SQL 执行的完整生命周期，包括是否有索引扫描、是否有全表扫描，以及每个阶段的耗时；
    2. `optimizer_trace`：开启优化器追踪，可查看 MySQL 优化器的完整执行过程，为什么选择这个索引，为什么放弃了其他索引，定位索引失效的根本原因；
    3. `show status like 'Handler_read%';`：执行 SQL 前后查看该状态，可查看 SQL 实际读取的行数、索引读取的行数，精准判断索引的使用效率。
    
- 生产环境最佳实践：
    
    1. 上线前必须用`explain`分析 SQL 的执行计划，保证`type`字段至少达到`range`级别，禁止`type=ALL`的 SQL 上线；
    2. 定期用`pt-query-digest`分析慢查询日志，批量检查慢 SQL 的索引命中情况，提前优化；
    3. 不要只看`key`字段不为 NULL 就认为索引生效了，必须结合`type`、`rows`、`key_len`综合判断，`index`级别的全索引扫描，和全表扫描性能差距不大，必须优化；
    4. 优先使用覆盖索引，避免回表查询，大幅提升 SQL 性能。
    

#### 面试避坑指南

- 严禁只说「看 key 字段是否为 NULL」，不说`type`字段的判断，这是新手常见错误，`key`不为 NULL 不代表索引高效，`index`级别就是典型的低效索引；
- 避免混淆「命中索引」和「高效使用索引」，面试问这个问题，核心是考察你能不能判断索引的使用效率，而不只是有没有命中；
- 不要只说 explain 命令，不说慢查询日志的判断方法，慢 SQL 的判断和索引命中的判断是两个问题，必须都覆盖到。

#### 核心标准答案

缓存击穿和缓存穿透是 Redis 缓存场景中最常见的两个问题，二者的触发场景、根因、解决方案有本质区别，具体定义与解决方案如下：

##### 一、缓存穿透

**核心定义**：客户端请求的数据，**在 Redis 缓存和后端数据库中都不存在**，导致每次请求都会穿透缓存，直接打到数据库上。如果有大量这种无效请求，会把数据库压垮，甚至宕机。

**常见触发场景**：

- 恶意攻击：用大量不存在的 ID / 参数发起请求，比如 ID 为负数、超大数值的无效请求；
- 业务代码 bug：请求参数校验不严格，传入了非法的、不存在的参数；
- 缓存与数据库同步异常：数据被物理删除后，缓存没有同步清理，或者出现了无效的 key。

**核心解决方案（按优先级从高到低）**：

1. **前置参数校验**：在接口入口层做严格的参数合法性校验，过滤掉非法参数，比如 ID 为负数、格式错误的请求，直接返回，避免走到缓存和数据库层，这是成本最低、最有效的方案。
2. **空值 / 默认值缓存**：对于数据库中不存在的 key，在 Redis 中缓存一个空值（或默认值），设置较短的过期时间（比如 30 秒 - 5 分钟），避免同一个无效 key 反复请求数据库。
3. **布隆过滤器拦截**：将数据库中所有存在的 key，提前写入布隆过滤器，请求到来时，先经过布隆过滤器校验，如果布隆过滤器判断 key 不存在，直接返回，无需查询缓存和数据库，适合数据量较大、变更不频繁的场景。
4. **接口限流与黑名单**：针对恶意攻击的场景，对接口做限流，同时监控异常请求 IP，将高频发起无效请求的 IP 加入黑名单，禁止访问。

##### 二、缓存击穿

**核心定义**：客户端请求的某个**热点 key，在缓存中过期失效了**，此时刚好有大量并发请求打到这个 key 上，所有请求都会穿透缓存，直接打到数据库上，导致数据库压力瞬间暴增，甚至宕机。

**核心特点**：key 在数据库中是存在的，只是缓存过期了，且是高并发访问的热点 key。

**常见触发场景**：

- 秒杀活动的热点商品 key 过期，大量用户同时请求；
- 热门新闻、首页数据的缓存过期，高峰期大量用户访问；
- 缓存的过期时间设置不合理，热点 key 在业务高峰期过期。

**核心解决方案（按优先级从高到低）**：

1. **热点 key 永不过期**：对于核心热点 key，不设置过期时间，由后台异步线程定时更新缓存，保证缓存始终有效，从根源上避免 key 过期导致的击穿问题，这是生产环境最常用的方案。
2. **互斥锁（分布式锁）控制**：当缓存失效时，不是所有请求都去查数据库，而是第一个请求先获取分布式锁（Redis 的 SETNX），只有拿到锁的请求才能去查询数据库并更新缓存，其他请求等待锁释放后，重新查询缓存，保证同一时间只有一个请求访问数据库，避免并发打满数据库。
3. **热点 key 过期时间打散**：给热点 key 的过期时间增加随机值，比如原本过期时间是 1 小时，设置为 50-70 分钟随机，避免大量热点 key 在同一时间同时过期，导致集中式的缓存击穿。
4. **缓存预热**：在业务高峰期到来之前，提前将热点数据加载到 Redis 中，设置合理的过期时间，避免高峰期缓存失效。

#### 专家级拓展

- 两个问题的核心区别：
    
    表格
    
    |问题|核心根因|数据是否存在|并发特征|
    |:--|:--|:--|:--|
    |缓存穿透|数据在缓存和数据库都不存在|不存在|大量不同的无效 key 请求|
    |缓存击穿|热点 key 缓存过期，数据库有数据|存在|大量相同的热点 key 并发请求|
    
- 生产环境避坑点：
    
    1. 布隆过滤器的局限性：布隆过滤器存在误判率，只能判断「key 一定不存在」，不能判断「key 一定存在」，误判会导致正常请求被拦截，同时数据新增 / 删除时，需要同步更新布隆过滤器，不适合数据频繁变更的场景；
    2. 互斥锁的优化：使用分布式锁时，必须设置锁的超时时间，避免锁无法释放导致所有请求阻塞；同时可采用双检锁模式，获取锁之后，再次查询缓存，避免重复更新缓存；
    3. 空值缓存的风险：如果有大量不同的无效 key，缓存空值会占用大量 Redis 内存，导致 Redis 内存溢出，必须配合短过期时间和参数校验，避免这个问题。
    
- 关联问题区分：缓存雪崩，指的是大量缓存 key 在同一时间集中过期，或者 Redis 集群宕机，导致所有请求都打到数据库上，数据库压力暴增，是比击穿、穿透更严重的故障，解决方案主要是过期时间打散、Redis 集群高可用、服务熔断降级、多级缓存。

#### 面试避坑指南

- 严禁混淆缓存穿透和缓存击穿的定义，把二者的根因和解决方案说反，这是面试最高频的错误，会直接被判定为对缓存原理理解不透彻；
- 避免只说解决方案，不说每个方案的适用场景和局限性，面试问这个问题，核心是考察你能不能根据业务场景选择合适的方案，而不是只会背解决方案；
- 不要只讲技术方案，不说前置的参数校验、业务优化，比如缓存击穿的核心是热点 key，提前做好热点 key 的识别和永不过期，比事后加锁更有效，体现业务视角会大幅加分。

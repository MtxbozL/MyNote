#### 核心标准答案

我会遵循「**从整体到局部，从进程到业务，从现象到根因**」的分层排查思路，完整流程如下：

1. **第一步：确认整体状态，锁定异常进程**
    
    - 执行`top`命令，确认 MQ Broker 进程的 CPU 使用率（用户态 % us / 内核态 % sy）、内存使用率、负载情况，确认是 MQ 进程导致的 CPU 占用过高，还是服务器其他进程导致；
    - 执行`top -Hp 进程PID`，查看进程内的线程 CPU 占用，找到占用 CPU 最高的线程 ID，定位是业务处理线程、IO 线程、GC 线程，还是其他异常线程。
    
2. **第二步：排查 JVM / 虚拟机层面的问题（Java/Erlang 实现的 MQ）**
    
    绝大多数 MQ（RocketMQ、RabbitMQ、Kafka）都是基于 JVM/Erlang VM 实现的，CPU 高大概率和 VM 相关，优先排查：
    
    - **GC 问题排查**：查看 MQ 的 GC 日志，确认是否频繁 Full GC/YGC，GC 停顿时间过长，会导致 CPU 占用飙升；如果频繁 GC，排查是否是内存泄漏、堆内存设置过小、消息堆积导致的对象无法回收；
    - **线程异常排查**：将占用 CPU 最高的线程 ID 转为 16 进制，通过 jstack（Java）/erl（Erlang）查看线程的堆栈信息，定位线程是在执行什么逻辑，比如死循环、锁竞争、频繁的系统调用。
    
3. **第三步：排查 MQ 核心业务层面的根因**
    
    这是 CPU 高的最常见场景，核心排查方向：
    
    - **消息堆积问题**：查看 MQ 的 Topic / 队列的消息堆积量，是否有大量消息未被消费，Broker 需要频繁管理堆积的消息、持久化、索引更新，导致 CPU 占用飙升；
    - **生产端压力过大**：查看生产端的 TPS，是否有突发的大量消息生产，频繁的消息写入、持久化、副本同步，导致 CPU 占用过高；
    - **消费端异常**：消费端频繁重试、消费失败，导致消息重复投递、死信队列处理，增加 Broker 的 CPU 开销；
    - **分区 / 队列过多**：Topic 的分区 / 队列数量过多，Broker 需要管理大量的队列、索引、元数据，增加 CPU 开销，尤其是 RabbitMQ，队列过多会导致 CPU 占用显著上升。
    
4. **第四步：排查配置与持久化层面的问题**
    
    - **持久化配置不合理**：比如开启了同步刷盘，每条消息都强制刷盘，导致频繁的磁盘 IO 和系统调用，CPU 占用升高；或者刷盘策略设置不合理，频繁触发刷盘操作；
    - **副本同步压力过大**：开启了多副本同步，副本之间频繁的数据同步、一致性校验，导致 CPU 占用过高；
    - **参数配置不合理**：比如网络线程、业务处理线程数设置过多，导致频繁的上下文切换；或者开启了过多的插件、消息过滤、消息追踪等功能，增加了 CPU 开销。
    
5. **第五步：排查系统与网络层面的问题**
    
    - **网络 IO 频繁**：查看服务器的网络带宽、网络连接数，是否有大量的网络连接、频繁的网络读写，导致内核态 CPU 占用过高；
    - **磁盘 IO 瓶颈**：通过`iostat`查看磁盘 IO 使用率、await，是否磁盘 IO 性能不足，导致 Broker 线程阻塞等待 IO，CPU 占用升高；
    - **系统参数配置不合理**：比如文件句柄数设置过小、TCP 参数配置不合理，导致频繁的连接异常、重传，增加 CPU 开销。
    
6. **第六步：根因修复与验证**
    
    根据定位的根因针对性修复，比如优化消费速度解决消息堆积、调整 JVM 堆内存解决频繁 GC、调整刷盘策略优化 IO、限制生产端 TPS 削峰、清理无效的 Topic / 队列，修复完成后，持续监控 MQ 的 CPU 使用率，确认恢复正常。

#### 专家级拓展

- 不同 MQ 的专项排查重点：
    
    1. Kafka：重点排查分区数量、副本同步机制、日志段清理、消费者重平衡，Kafka 的 CPU 高通常和批量消息处理、分区 Leader 切换、日志压缩相关；
    2. RabbitMQ：重点排查队列数量、Erlang VM 的调度、消息持久化、镜像队列同步，RabbitMQ 对队列数量非常敏感，单节点超过 1000 个队列，CPU 占用会显著上升；
    3. RocketMQ：重点排查刷盘策略、CommitLog/ConsumeQueue 的索引更新、消息重试、死信队列，JVM GC 是 RocketMQ CPU 高的常见原因。
    
- 生产环境最佳实践：
    
    1. 搭建 MQ 监控体系，监控 Broker 的 CPU 使用率、TPS、消息堆积量、GC 情况、刷盘耗时，提前发现异常；
    2. 合理规划 Topic 的分区 / 队列数量，避免创建大量无用的 Topic 和队列；
    3. 生产端开启消息批量发送，消费端开启批量消费，减少网络 IO 和 Broker 的处理开销；
    4. 合理设置 JVM 堆内存和 GC 策略，避免频繁 Full GC。
    

#### 面试避坑指南

- 严禁只说「查看进程 CPU 占用」，不说分层排查流程，会被面试官判定为无体系化排查能力；
- 避免遗漏 JVM GC 问题，这是 Java 系 MQ CPU 高的最常见原因，不提会被认为无实际排查经验；
- 不要只排查 Broker 端，忽略生产端和消费端的问题，很多时候 Broker CPU 高是业务端的异常使用导致的。

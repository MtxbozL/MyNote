#### 核心标准答案

Redis 哨兵模式（Sentinel）是 Redis 官方提供的**高可用解决方案**，解决 Redis 主从架构的单点故障问题，实现主节点的自动故障转移、监控、配置通知、配置管理，保证 Redis 服务的高可用。

1. **核心作用**
    
    - **监控**：持续监控 Redis 主节点和从节点的运行状态，确认节点是否正常工作；
    - **自动故障转移**：当主节点故障时，自动执行故障转移，从从节点中选举一个新的主节点，将其他从节点切换到新的主节点，同时通知客户端更新主节点地址，整个过程无需人工干预；
    - **通知（告警）**：当监控的 Redis 节点出现异常时，哨兵可以通过 API、脚本、邮件等方式，通知运维人员；
    - **配置中心**：客户端连接哨兵集群，获取当前 Redis 主节点的地址，主节点发生故障转移后，哨兵会通知客户端最新的主节点地址，客户端无需修改配置。
    
2. **核心架构**
    
    哨兵模式由两部分组成：
    
    - **Redis 主从集群**：1 个主节点 + N 个从节点，主节点负责写操作，从节点负责读操作，从节点从主节点同步数据；
    - **哨兵集群**：至少 3 个哨兵节点（奇数个），每个哨兵节点都是独立的进程，哨兵之间会互相通信、投票，共同监控 Redis 节点的状态，保证故障判断的准确性。
    
3. **核心原理与故障转移流程**
    
    1. **主观下线（SDOWN）**：每个哨兵节点会定期向 Redis 主从节点发送 PING 命令，如果节点在`down-after-milliseconds`时间内没有返回有效响应，该哨兵节点会将这个节点标记为「主观下线」。
    2. **客观下线（ODOWN）**：当哨兵将主节点标记为主观下线后，会向其他哨兵节点发送命令，询问其他哨兵是否也认为该主节点下线；当收到超过半数（quorum）的哨兵节点的确认后，会将主节点标记为「客观下线」，正式触发故障转移。
    3. **哨兵 Leader 选举**：哨兵集群会通过 Raft 算法选举出一个 Leader 哨兵，由这个 Leader 哨兵负责执行后续的故障转移操作，保证只有一个哨兵执行故障转移，避免混乱。
    4. **新主节点选举**：Leader 哨兵会从所有健康的从节点中，选举一个新的主节点，选举优先级：
        
        - 优先选择`replica-priority`（副本优先级）最高的从节点；
        - 优先级相同，选择复制偏移量最大的从节点（和主节点数据最同步，数据最完整）；
        - 前两个都相同，选择 run_id 最小的从节点。
        
    5. **执行故障转移**：
        
        - Leader 哨兵将选举出的从节点升级为主节点，向该节点发送`SLAVEOF no one`命令；
        - 向其他所有从节点发送命令，让它们切换到新的主节点，从新的主节点同步数据；
        - 将原来的故障主节点标记为从节点，等它恢复后，让它从新的主节点同步数据；
        - 将最新的主节点地址通知给所有客户端。
        
    

#### 专家级拓展

- 核心设计要点：
    
    1. **哨兵集群必须是奇数节点**：至少 3 个，保证投票时能形成多数派，避免脑裂问题；如果只有 2 个哨兵节点，其中一个宕机，就无法形成多数派，无法执行故障转移；
    2. **主观下线和客观下线的区别**：主观下线是单个哨兵的判断，可能存在误判；客观下线是多数哨兵的共同判断，保证故障判断的准确性，只有主节点会有客观下线，从节点只有主观下线；
    3. **脑裂问题的规避**：通过`min-replicas-to-write`参数，配置主节点必须至少有 N 个从节点连接，否则拒绝写操作，避免网络分区时，主节点脑裂，出现两个主节点同时写入的情况。
    
- 生产环境最佳实践：
    
    1. 哨兵节点和 Redis 节点分开部署，不要和 Redis 主从节点部署在同一台服务器上，避免服务器宕机导致 Redis 和哨兵同时故障；
    2. 3 个哨兵节点部署在不同的可用区 / 机房，避免单机房故障导致哨兵集群不可用；
    3. 合理设置`down-after-milliseconds`参数，避免网络抖动导致的误判，同时保证故障发现的及时性；
    4. 客户端必须通过哨兵集群获取主节点地址，不要硬编码主节点 IP，否则故障转移后，客户端无法连接到新的主节点。
    

#### 面试避坑指南

- 严禁只说哨兵模式的作用，不说故障转移的完整流程，会被面试官判定为「只会用，不懂底层原理」；
- 避免混淆主观下线和客观下线，这是哨兵模式的核心概念，也是面试必问的细节；
- 不要说「单节点哨兵可以实现高可用」，单节点哨兵本身就是单点故障，生产环境必须部署奇数个哨兵节点，至少 3 个，这是核心原则。

#### 核心标准答案

Redis 分布式锁是基于 Redis 的原子操作，实现的跨进程、跨服务的分布式互斥锁，解决分布式系统中多个服务并发操作共享资源的问题，核心是保证**互斥性、防死锁、可重入、安全释放、容错性**五大核心特性。

1. **核心实现原理**
    
    基于 Redis 的**SETNX（SET if Not Exists）+ 过期时间**的原子命令实现，完整的加锁、解锁流程如下：
    
    - **加锁核心逻辑**：
        
        使用 Redis 的原子 SET 命令，一次性完成「不存在才设置 + 设置过期时间」，命令格式：
        
        plaintext
        
        ```
        SET lock_key unique_value NX PX expire_time
        ```
        
        各参数含义：
        
        1. `lock_key`：锁的唯一 key，对应要锁定的共享资源；
        2. `unique_value`：唯一值，比如客户端 ID + 线程 ID，保证只有加锁的客户端才能释放锁；
        3. `NX`：只有 key 不存在时，才会设置成功，保证互斥性，只有一个客户端能加锁成功；
        4. `PX expire_time`：设置 key 的过期时间（毫秒），避免客户端宕机后，锁无法释放，导致死锁。
            
            只有 SET 命令执行成功，才算加锁成功；执行失败，代表锁被其他客户端持有，加锁失败。
        
    - **解锁核心逻辑**：
        
        解锁必须保证「只有加锁的客户端才能释放锁」，不能直接用 DEL 命令删除 key，否则会误删其他客户端的锁，必须通过 Lua 脚本实现原子操作：
        
        lua
        
        ```
        if redis.call('get', KEYS[1]) == ARGV[1] then
            return redis.call('del', KEYS[1])
        else
            return 0
        end
        ```
        
        先判断锁的 value 是否和当前客户端的唯一值一致，一致才删除锁，整个 Lua 脚本在 Redis 中原子执行，避免判断和删除之间的并发问题。
    
2. **核心特性保障**
    
    - **互斥性**：通过 NX 参数保证，同一时间只有一个客户端能加锁成功；
    - **防死锁**：通过过期时间保证，即使客户端宕机，锁到期后会自动释放；
    - **安全释放**：通过唯一 value+Lua 脚本保证，只有加锁的客户端才能释放锁；
    - **原子性**：加锁用单条 SET 命令，解锁用 Lua 脚本，保证操作的原子性，避免并发问题。
    

#### 专家级拓展

- 生产环境进阶实现：
    
    1. **可重入锁**：基于 Redis 的 Hash 结构 + Lua 脚本实现，记录加锁的客户端和加锁次数，同一个客户端多次加锁时，次数 + 1，解锁时次数 - 1，次数为 0 时才删除锁，适配嵌套加锁的场景；
    2. **锁续期（看门狗机制）**：如果业务执行时间超过了锁的过期时间，会导致锁提前释放，出现并发问题。通过后台守护线程，在锁即将过期时，自动续期，延长锁的过期时间，保证业务执行完成前，锁不会被释放，Redisson 等客户端已经内置了看门狗机制；
    3. **红锁（Redlock）**：解决 Redis 单节点故障导致的锁失效问题，在多个独立的 Redis 主节点上同时加锁，超过半数节点加锁成功，才算加锁成功，即使单个节点宕机，锁依然有效，保证分布式锁的高可用；
    4. **阻塞锁**：加锁失败时，不是直接返回，而是通过自旋 + 订阅锁释放的通知，等待锁释放后再次尝试加锁，减少无效的自旋请求，提升性能。
    
- 生产环境最佳实践：
    
    1. 优先使用成熟的客户端，比如 Redisson，不要自己手动实现分布式锁，避免出现并发安全问题；
    2. 锁的粒度要尽可能小，锁定的资源范围越小越好，加锁的时间越短越好，提升并发性能；
    3. 必须设置合理的过期时间，避免死锁，同时配合看门狗续期，避免锁提前释放；
    4. 加锁和解锁必须在同一个客户端、同一个线程中，避免跨线程释放锁。
    

#### 面试避坑指南

- 严禁只说「用 SETNX 实现」，不说过期时间和原子操作，这是新手最常见的错误，SETNX 和 EXPIRE 分两条命令执行，不是原子操作，中间客户端宕机，会导致锁无法过期，出现死锁；
- 避免解锁时直接用 DEL 命令，必须判断锁的归属，否则会误删其他客户端的锁，导致并发问题，这是面试高频扣分点；
- 不要说「Redis 分布式锁可以保证强一致性」，Redis 是 AP 系统，极端场景下（主从切换）会出现锁丢失，需要用红锁机制提升可靠性，不能夸大 Redis 分布式锁的一致性。

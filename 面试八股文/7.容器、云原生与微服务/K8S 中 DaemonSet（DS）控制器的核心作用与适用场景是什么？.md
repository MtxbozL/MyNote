#### 核心标准答案

DaemonSet 是 K8s 内置的工作负载控制器，核心作用是**确保集群中符合条件的每个 Node 节点上，都运行且仅运行一个指定的 Pod 副本**。

当集群中有新的 Node 节点加入时，DaemonSet 会自动在新节点上创建对应的 Pod；当 Node 节点从集群中移除时，该节点上的 DaemonSet Pod 会被自动清理，无需人工干预。

#### 专家级拓展

- 核心特性：
    
    1. 节点级独占：默认每个节点仅运行一个 Pod 副本，无法通过调整副本数扩缩容，副本数等于符合条件的节点数量。
    2. 自动扩缩容：节点新增 / 删除时，自动创建 / 清理 Pod，无需人工操作。
    3. 节点筛选：可通过`nodeSelector`、`nodeAffinity`、`污点与容忍`，控制 Pod 只在指定的节点上运行，而不是全节点部署。
    4. 滚动更新：支持滚动更新策略，可控制 DaemonSet Pod 的更新速度，避免更新过程中影响节点功能。
    
- 核心适用场景（生产环境高频）：
    
    1. 监控数据采集：在每个节点上部署监控代理，采集节点的指标数据，比如 Prometheus 的 node-exporter、Datadog Agent。
    2. 日志采集：在每个节点上部署日志采集组件，采集节点和容器的日志，比如 Filebeat、Fluentd、Fluent Bit。
    3. 集群网络插件：每个节点都必须运行的网络组件，比如 Calico、Flannel、Cilium 的节点代理。
    4. 集群存储插件：每个节点都必须运行的存储组件，实现容器的持久化存储挂载，比如 Rook、Ceph 的节点客户端、Longhorn 的节点组件。
    5. 安全合规组件：每个节点都必须运行的安全审计、入侵检测组件，比如 Falco、节点安全扫描工具。
    
- 与 Deployment 的核心区别：Deployment 的副本数是固定的，Pod 会被调度到任意有资源的节点，一个节点可以运行多个副本；DaemonSet 的副本数等于节点数量，每个符合条件的节点仅运行一个副本，调度不受节点资源的限制（除非设置了资源限制）。

#### 面试避坑指南

- 严禁说「DaemonSet 可以调整副本数」，DaemonSet 的副本数由节点数量决定，无法手动设置 replicas，这是和 Deployment 最核心的区别；
- 避免只说 DaemonSet 的定义，不说生产环境的适用场景，会被判定为无实际使用经验。

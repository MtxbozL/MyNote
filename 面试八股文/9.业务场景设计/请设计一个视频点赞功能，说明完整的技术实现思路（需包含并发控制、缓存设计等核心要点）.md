#### 核心标准答案

视频点赞功能是典型的**读多写少、高并发、数据一致性要求高**的场景，核心设计目标是：保证高并发下的点赞 / 取消点赞不重复、点赞数准确、接口响应快、服务高可用，完整设计方案如下：

##### 一、需求分析与核心约束

1. **核心功能**：用户点赞视频、用户取消点赞、查看视频点赞总数、查看用户是否点赞过该视频、查看用户的点赞列表；
2. **核心约束**：
    
    - 并发控制：同一个用户不能重复点赞同一个视频，取消点赞后可重新点赞；
    - 性能要求：点赞 / 取消点赞接口响应时间 < 100ms，点赞数查询接口响应时间 < 50ms，支持每秒上万次的点赞请求；
    - 一致性要求：最终数据一致，点赞数不能出现长期错误，极端场景下允许短暂的不一致。
    

##### 二、整体架构设计

采用「**前端→网关→应用服务→缓存层 Redis→持久化层 MySQL**」的分层架构，核心是**缓存为主，数据库为辅，异步持久化，最终一致性**，高并发下优先保证读写性能，通过异步机制保证数据最终一致。

##### 三、核心数据结构设计

1. **MySQL 持久化层**
    
    设计两张核心表，保证数据的持久化和一致性：
    
    - **视频点赞记录表（user_video_like）**：记录用户的点赞明细，保证幂等性，避免重复点赞
        
        表格
        
        |字段名|类型|说明|
        |:--|:--|:--|
        |id|bigint|主键 ID|
        |user_id|bigint|用户 ID，联合唯一索引|
        |video_id|bigint|视频 ID，联合唯一索引|
        |like_status|tinyint|点赞状态：1 = 已点赞，0 = 已取消点赞|
        |create_time|datetime|创建时间|
        |update_time|datetime|更新时间|
        |核心设计：`user_id + video_id`建立联合唯一主键，保证同一个用户对同一个视频只有一条记录，从数据库层面避免重复点赞。|||
        
    - **视频表（video）**：冗余点赞数字段，避免每次都统计点赞表
        
        表格
        
        |字段名|类型|说明|
        |:--|:--|:--|
        |video_id|bigint|视频 ID，主键|
        |title|varchar|视频标题|
        |like_count|bigint|点赞总数，默认 0|
        |...|...|其他视频信息|
        
    
2. **Redis 缓存层设计**
    
    核心是用 Redis 实现高并发的读写，避免请求直接打到数据库，采用两种核心数据结构：
    
    - **点赞用户集合：Set 结构**
        
        - Key 设计：`video:like:user:set:{video_id}`
        - Value：点赞用户的 user_id 集合
        - 核心作用：O (1) 时间复杂度判断用户是否点赞过该视频，同时支持获取视频的所有点赞用户，是点赞 / 取消点赞的核心数据结构。
        
    - **点赞总数：String 结构**
        
        - Key 设计：`video:like:count:{video_id}`
        - Value：视频的点赞总数
        - 核心作用：O (1) 时间复杂度获取视频点赞数，避免每次都统计 Set 集合或数据库。
        
    

##### 四、核心业务流程设计

###### 1. 点赞 / 取消点赞核心流程（写操作）

采用「**Redis 原子操作优先，异步同步到数据库**」的方案，保证高并发下的性能和幂等性：

1. **接口前置校验**：校验用户登录状态、视频是否存在，过滤非法请求；
2. **Redis 原子操作执行点赞 / 取消点赞**：
    
    - **点赞操作**：通过 Redis 的`SADD`命令，将 user_id 加入对应视频的 Set 集合，`SADD`的返回值为 1，代表点赞成功（之前没点赞过）；返回值为 0，代表已经点赞过，重复点赞，直接返回成功，保证幂等性；点赞成功后，执行`INCR`命令，点赞总数 + 1。
    - **取消点赞操作**：通过 Redis 的`SREM`命令，将 user_id 从 Set 集合中移除，返回值为 1，代表取消成功；返回值为 0，代表已经取消过，直接返回成功；取消成功后，执行`DECR`命令，点赞总数 - 1。
    - 核心优化：将点赞 / 取消点赞的两个操作，封装到 Lua 脚本中原子执行，避免`SADD/SREM`和`INCR/DECR`之间出现并发问题，保证 Redis 操作的原子性。
    
3. **异步同步到 MySQL 数据库**：
    
    点赞 / 取消点赞操作 Redis 成功后，直接给用户返回成功，同时将点赞事件发送到 MQ（Kafka/RocketMQ），通过异步消费程序，更新数据库的点赞记录表和视频点赞数。
    
    - 核心优势：削峰填谷，避免高并发下数据库压力过大，同时保证 Redis 宕机后，数据库有完整的点赞数据，可恢复缓存。
    
4. **兜底幂等性保障**：
    
    数据库的`user_id + video_id`联合唯一索引，即使异步消费出现重复消息，也不会出现重复记录，保证数据的最终一致性。

###### 2. 查看点赞数、是否点赞流程（读操作）

采用「多级缓存，优先读缓存，缓存 miss 再查数据库」的方案，保证读操作的高性能：

1. **查看用户是否点赞过视频**：
    
    - 先查询 Redis 的 Set 集合，执行`SISMEMBER video:like:user:set:{video_id} {user_id}`，返回 1 代表已点赞，0 代表未点赞；
    - 如果缓存 miss（Set 集合不存在），查询数据库的点赞记录表，确认用户的点赞状态，同时将点赞用户全量写入 Redis 的 Set 集合，回填缓存，下次直接读缓存。
    
2. **查看视频点赞总数**：
    
    - 先查询 Redis 的 String 结构，获取点赞总数，直接返回；
    - 如果缓存 miss，查询视频表的 like_count 字段，回填到 Redis 缓存，同时设置过期时间，避免缓存穿透。
    

##### 五、并发控制核心方案

1. **Redis 原子操作**：所有点赞 / 取消点赞的写操作，都通过 Lua 脚本封装为原子操作，避免并发下的计数错误、重复点赞问题，Redis 的单线程模型保证了 Lua 脚本的原子性，无需加锁，性能极高。
2. **数据库唯一索引兜底**：`user_id + video_id`联合唯一索引，从数据库层面保证同一个用户对同一个视频只有一条点赞记录，即使 Redis 出现异常，也不会出现重复点赞的数据。
3. **分布式锁兜底**：极端场景下（比如缓存重建），通过 Redis 分布式锁控制并发，避免缓存击穿、重复写入数据库的问题。
4. **MQ 消息幂等性**：异步消费时，通过消息 ID 做幂等处理，避免重复消费导致的点赞数错误。

##### 六、缓存异常问题解决方案

针对缓存三大问题，做针对性处理：

1. **缓存穿透**：
    
    - 问题：用户请求不存在的视频 ID，缓存和数据库都没有，请求直接打到数据库；
    - 解决方案：接口前置校验视频 ID 的合法性；对不存在的视频 ID，在 Redis 中缓存一个空值，设置短过期时间（比如 5 分钟）；布隆过滤器，将所有存在的视频 ID 写入布隆过滤器，请求先经过布隆过滤器，不存在的视频 ID 直接返回。
    
2. **缓存击穿**：
    
    - 问题：热点视频的缓存过期，大量并发请求同时打到数据库；
    - 解决方案：热点视频的点赞数据不设置过期时间，通过后台异步定时更新缓存；缓存重建时，通过分布式锁控制，只有一个线程去查询数据库重建缓存，其他线程等待缓存重建完成，避免并发打垮数据库。
    
3. **缓存雪崩**：
    
    - 问题：大量视频的缓存同时过期，大量请求打到数据库；
    - 解决方案：不同视频的缓存过期时间增加随机值，避免同时过期；热点数据永不过期；搭建 Redis 集群，保证缓存服务的高可用，避免 Redis 整体宕机。
    

#### 专家级拓展

- 高并发场景进阶优化：
    
    1. **本地缓存二级缓存**：在应用服务层增加 Caffeine 本地缓存，缓存热门视频的点赞数、用户点赞状态，减少 Redis 的访问压力，进一步降低接口响应时间，本地缓存设置短过期时间（比如 1 秒），保证数据最终一致。
    2. **点赞数合并更新**：异步消费时，不是每次点赞都更新数据库的 like_count 字段，而是将一段时间内的点赞数变更合并，批量更新数据库，减少数据库的写操作，提升性能。
    3. **冷热数据分离**：热门视频的点赞数据常驻 Redis，冷门视频的点赞数据按需加载，设置过期时间，节省 Redis 内存，提升缓存命中率。
    4. **分库分表**：当点赞记录数据量达到亿级时，对点赞记录表按 user_id 哈希分库分表，提升查询性能；视频表按 video_id 分库分表。
    5. **防刷限流**：接口层增加限流，限制单个用户的点赞频率，比如每秒最多点赞 10 次，避免恶意刷赞；IP 限流，避免恶意攻击打垮服务。
    
- 容灾与数据一致性保障：
    
    1. **Redis 宕机容灾**：Redis 集群宕机时，服务降级，直接读写数据库，保证点赞功能可用；Redis 恢复后，通过数据库数据全量重建缓存。
    2. **定时数据校验**：后台定时任务，对比 Redis 和数据库的点赞数、点赞记录，修复不一致的数据，保证最终一致性。
    3. **点赞历史归档**：用户取消点赞的历史记录，定期归档到冷存储，减少点赞记录表的数据量，提升查询性能。
    

#### 面试避坑指南

- 严禁只说功能实现，不讲高并发下的并发控制和缓存设计，这是面试的核心考察点，面试官问这个问题，不是问你 CRUD 怎么写，而是问你高并发场景下的设计能力；
- 避免用「先更新数据库，再更新缓存」的方案，这种方案在并发场景下会出现严重的数据不一致问题，必须优先操作缓存，异步同步数据库；
- 不要遗漏幂等性设计，重复点赞是点赞功能最常见的问题，必须从 Redis 和数据库两个层面保证幂等性；
- 不要只讲正常流程，不讲缓存异常、容灾降级的方案，体现你对生产环境风险的把控能力，这是区分普通开发和高级开发的核心点。

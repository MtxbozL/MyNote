### 核心考察点

候选人对事务并发问题的理解，对 InnoDB 隔离机制的掌握程度，是 MySQL 面试的超高频核心题，区分中级与高级工程师。

### 专家级满分回答模板

1. **前置说明**：事务隔离级别是为了解决并发事务执行中的 3 种核心数据异常：**脏读、不可重复读、幻读**，SQL 标准定义了 4 种隔离级别，从低到高依次为读未提交、读已提交、可重复读、串行化。
2. **4 种隔离级别与解决的问题**

| 隔离级别                      | 核心定义                                 | 脏读  | 不可重复读 | 幻读   |
| ------------------------- | ------------------------------------ | --- | ----- | ---- |
| 读未提交（Read Uncommitted）    | 一个事务可以读取到另一个未提交事务的修改数据               | 存在  | 存在    | 存在   |
| 读已提交（Read Committed, RC）  | 一个事务只能读取到另一个事务已经提交的修改数据              | 解决  | 存在    | 存在   |
| 可重复读（Repeatable Read, RR） | 一个事务内，多次读取同一行数据的结果始终一致，不受其他事务提交的修改影响 | 解决  | 解决    | 部分解决 |
| 串行化（Serializable）         | 所有事务串行执行，完全禁止并发读写，最高隔离级别             | 解决  | 解决    | 解决   |

3. **核心异常定义**
    
    - 脏读：事务 A 读取了事务 B 未提交的修改，若事务 B 回滚，事务 A 读取到的就是无效脏数据。
    - 不可重复读：事务 A 内多次读取同一行数据，期间事务 B 修改并提交了这行数据，导致事务 A 两次读取的结果不一致，核心是**update/delete 操作**。
    - 幻读：事务 A 内多次执行范围查询，期间事务 B 插入了符合该范围的新数据并提交，导致事务 A 两次查询的行数不一致，核心是**insert 操作**。
    
4. **InnoDB 默认隔离级别（RR）的底层实现**
    
    InnoDB 默认使用**可重复读（RR）** 隔离级别，也是 SQL 标准中唯一实现了 RR 级别的主流数据库，核心通过两大机制实现：
    
    1. **快照读（普通 select 查询）**：通过**MVCC（多版本并发控制）** 实现。事务启动时生成全局唯一的 Read View（数据版本快照），整个事务生命周期内复用这个 Read View，保证事务内每次读取的都是同一个数据版本，彻底解决不可重复读，同时避免了幻读。
    2. **当前读（select ... for update/lock in share mode、insert、update、delete）**：通过**Next-Key Lock（临键锁）** 实现。临键锁是「记录锁 + 间隙锁」的组合，不仅锁定查询到的记录本身，还会锁定记录之间的间隙，禁止其他事务在间隙中插入符合条件的数据，彻底解决了当前读场景下的幻读问题。
    

### 加分亮点（专家级区分度）

1. 补充 RC 与 RR 的核心差异：RC 级别下，每次查询都会生成新的 Read View，因此只能避免脏读，无法避免不可重复读；RR 级别下，事务启动时生成一次 Read View，全程复用，因此实现了可重复读。
2. 补充实战细节：InnoDB 的 RR 级别并非完全杜绝幻读，仅在快照读场景下解决，若事务内混合使用快照读和当前读，仍可能出现幻读现象，这是生产环境的高频踩坑点。
3. 补充行业现状：Oracle、PostgreSQL 等数据库的默认隔离级别是 RC，因为 RC 级别锁冲突更少、并发性能更高，仅在需要严格可重复读的场景使用 RR。

### 踩坑避雷区

- 错误表述：InnoDB 的 RR 级别完全解决了幻读。（仅解决快照读场景的幻读，当前读场景靠临键锁解决，混合读写场景仍有幻读风险）
- 错误表述：隔离级别越高，性能越好。（隔离级别越高，并发性能越差，串行化级别几乎无并发能力，生产环境极少使用）

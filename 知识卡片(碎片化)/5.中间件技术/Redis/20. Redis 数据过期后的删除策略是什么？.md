## 第一部分：标准化答题模板（极致精简・口语化・零晦涩）

### Level 1：初级（校招 / 初级岗，5 分钟背完，基础分全拿）

针对这个问题，我先纠正一个面试高频核心误区：**过期删除≠内存淘汰**—— 过期删除是针对**设置了 TTL/EXPIRE/PEXPIRE/EXPIREAT/PEXPIREAT 的 key**，内存淘汰是针对**内存达到 maxmemory 阈值的所有 key**，两者完全不同但有联动（内存淘汰时可能优先淘汰已过期的 key）。

---

#### 一、过期删除的两大核心策略（面试必背）

Redis 采用 **「惰性删除 + 定期删除」的混合策略 **，兼顾性能与内存利用率：

1. **惰性删除（Lazy Expiration）—— 被动触发，性能优先**
    
    - **触发时机**：只有当客户端**主动访问**（GET/HGET/LRANGE 等）已过期的 key 时，Redis 才会检查过期时间，确认过期后立即删除该 key；
    - **核心优势**：无需额外的后台线程 / 进程，无额外的 CPU 开销，性能极高；
    - **核心劣势**：如果已过期的 key 长期不被访问，会一直占用内存，造成内存浪费。
    
2. **定期删除（Active Expiration）—— 主动触发，内存兜底**
    
    - **触发时机**：Redis 后台有一个**定时任务**（`serverCron`），默认**每 100ms 执行一次**（可通过`hz`配置调整，范围 1-500），每次执行时会：
        
        1. 从所有设置了过期时间的 key 的字典（`expires`字典）中，**随机抽取 20 个 key**；
        2. 检查这 20 个 key 的过期时间，删除其中已过期的；
        3. 如果已过期的 key 占比**超过 25%**，则**立即重复执行**步骤 1-2，直到占比≤25% 或单次执行时间**超过 25ms**（避免阻塞主线程）；
        
    - **核心优势**：主动清理长期不被访问的已过期 key，避免内存浪费；
    - **核心劣势**：随机抽取可能漏删部分已过期的 key，单次执行时间有限，无法保证 100% 清理。
    

---

#### 二、过期时间的存储方式（面试必背的基础考点）

Redis 的过期时间存储在**独立的`expires`字典**中，而非和数据一起存储在`db->dict`字典中：

- `db->dict`：存储所有 key-value 数据，key 是 Redis 的 key，value 是 Redis 的 value；
- `db->expires`：仅存储设置了过期时间的 key，key 是 Redis 的 key，value 是**过期时间的毫秒级时间戳**（UNIX 时间戳）；
- **核心优势**：独立存储避免了遍历整个`db->dict`字典检查过期时间，大幅提升了过期删除的效率。

---

#### 三、核心常用命令（面试必背）

表格

|命令|核心作用|核心说明|
|---|---|---|
|`EXPIRE key seconds`|设置 key 的过期时间（秒级）|返回 1 表示设置成功，返回 0 表示 key 不存在或设置失败|
|`PEXPIRE key milliseconds`|设置 key 的过期时间（毫秒级）|精度更高，适合需要精确过期的场景|
|`EXPIREAT key timestamp`|设置 key 的过期时间（秒级 UNIX 时间戳）|适合需要在指定时间点过期的场景|
|`PEXPIREAT key timestamp`|设置 key 的过期时间（毫秒级 UNIX 时间戳）|精度更高，适合需要精确过期的场景|
|`TTL key`|获取 key 的剩余过期时间（秒级）|返回 - 2 表示 key 不存在，返回 - 1 表示 key 存在但未设置过期时间，返回正数表示剩余过期时间|
|`PTTL key`|获取 key 的剩余过期时间（毫秒级）|精度更高，适合需要精确监控的场景|
|`PERSIST key`|取消 key 的过期时间|返回 1 表示取消成功，返回 0 表示 key 不存在或未设置过期时间|

### Level 2：进阶（中高级岗 / 大厂面试，精简有亮点，可承接深挖）

针对这个问题，我的核心理解是：**Redis 的过期删除混合策略，是「性能优先、内存兜底、避免阻塞主线程」三大设计原则的完美结合**—— 惰性删除保证了访问时的高性能，定期删除保证了内存的合理利用，单次执行时间限制保证了不会阻塞 Redis 单线程主线程。

---

#### 一、定期删除的完整源码级执行流程（面试必背的进阶考点）

Redis 的定期删除任务是`serverCron`函数的一部分，完整执行流程如下：

1. **遍历所有数据库**：Redis 默认有 16 个数据库（可通过`databases`配置调整），定期删除会遍历所有数据库；
2. **对每个数据库执行以下操作**：
3. **检查数据库是否有设置了过期时间的 key**：如果`db->expires`字典为空，跳过该数据库；
4. **初始化单次执行的参数**：

- `max_deleted_keys`：单次执行最多删除的 key 数，默认 20；
- `max_time`：单次执行最多占用的时间，默认 25ms（`hz=10`时，100ms 执行一次，每次最多 25ms，避免阻塞主线程）；
- `start_time`：记录单次执行的开始时间；

3. **进入循环，直到满足退出条件**：
4. **从`db->expires`字典中随机抽取 20 个 key**：Redis 的字典是哈希表，随机抽取的时间复杂度 O (1)；
5. **遍历这 20 个 key**：
6. **检查 key 是否过期**：对比当前时间戳和`db->expires`字典中存储的过期时间戳；
7. **如果过期**：
    
    - 从`db->dict`字典中删除该 key-value；
    - 从`db->expires`字典中删除该 key；
    - 增加已删除的 key 数；
    - 触发 key 的过期通知（如果开启了`notify-keyspace-events`的`Ex`事件）；
    
8. **检查退出条件**：
9. **已删除的 key 数≥max_deleted_keys**：退出；
10. **当前时间 - start_time ≥ max_time**：退出；
11. **已过期的 key 占比≤25%**：退出；
12. **完成所有数据库的遍历**：定期删除任务结束。

---

#### 二、过期删除与内存淘汰的联动（面试必背的高级考点）

过期删除和内存淘汰是两个完全不同的机制，但有联动：

1. **内存淘汰时优先淘汰已过期的 key**：当内存达到 maxmemory 阈值时，Redis 会先检查是否有已过期的 key，如果有，优先删除已过期的 key，避免删除未过期的业务数据；
2. **内存淘汰策略中包含过期相关的策略**：Redis 的 8 种内存淘汰策略中，有 4 种是针对已过期的 key 的：

- `volatile-lru`：仅淘汰设置了过期时间的 key，采用 LRU 算法；
- `volatile-lfu`：仅淘汰设置了过期时间的 key，采用 LFU 算法；
- `volatile-random`：仅淘汰设置了过期时间的 key，采用随机算法；
- `volatile-ttl`：仅淘汰设置了过期时间的 key，优先淘汰剩余过期时间最短的 key。

---

#### 三、过期通知机制（面试必背的高级特性考点）

Redis 支持**过期通知机制**，当 key 过期时，会向订阅了对应频道的客户端发送通知：

1. **开启过期通知**：通过`notify-keyspace-events`配置开启，需要包含`Ex`事件（`E`表示键空间事件，`x`表示过期事件）；
2. **通知类型**：

- **键空间通知**：频道格式为`__keyspace@<db>__:<key>`，通知内容为`expired`；
- **键事件通知**：频道格式为`__keyevent@<db>__:expired`，通知内容为`<key>`；

3. **典型业务场景**：

- 分布式锁的自动释放通知；
- Session 过期通知；
- 缓存过期预热通知。

---

#### 四、生产级最佳实践

1. **合理设置过期时间**：

- 避免设置过长的过期时间，造成内存浪费；
- 避免设置过短的过期时间，造成缓存雪崩；
- 可以设置随机的过期时间（比如在基础过期时间上 ±10%），避免缓存雪崩；

2. **合理调整`hz`配置**：

- `hz`默认 10，每 100ms 执行一次定期删除；
- 如果内存中已过期的 key 较多，可以适当调大`hz`（比如 20-50），但不要超过 50，避免后台任务占用过多 CPU；

3. **合理开启过期通知**：

- 过期通知会占用一定的 CPU 和内存，仅在需要时开启；
- 优先使用键事件通知，而不是键空间通知，键事件通知的频道更少，占用资源更少；

4. **合理设置 maxmemory 和内存淘汰策略**：

- 必须设置 maxmemory，避免 Redis 内存占满宿主机内存，触发操作系统 OOM；
- 优先使用`volatile-lru`或`volatile-lfu`内存淘汰策略，仅淘汰设置了过期时间的 key，避免删除未过期的业务数据；

5. **避免在过期 key 上执行耗时的命令**：

- 虽然过期 key 会被惰性删除，但在删除前执行耗时的命令（如`KEYS *`、`LRANGE 0 -1`）会阻塞主线程，生产环境必须避免。

## 第二部分：学术深度解析

### 一、混合策略的设计依据

Redis 的过期删除混合策略，严格遵循 **「性能优先、内存兜底、避免阻塞主线程」三大设计原则 **，核心学术依据是：

1. **性能优先原则**：惰性删除无需额外的后台开销，访问时的检查时间复杂度 O (1)，性能极高；
2. **内存兜底原则**：定期删除主动清理长期不被访问的已过期 key，避免内存浪费；
3. **避免阻塞主线程原则**：定期删除单次执行时间限制在 25ms 以内，随机抽取 20 个 key，避免遍历整个`expires`字典，保证不会阻塞 Redis 单线程主线程。

### 二、随机抽取的概率分析

定期删除每次随机抽取 20 个 key，已过期的 key 占比超过 25% 则重复执行，核心概率分析是：

- 假设已过期的 key 占比为 p，随机抽取 20 个 key，其中至少有 1 个已过期的概率为 1-(1-p)^20；
- 当 p=25% 时，至少有 1 个已过期的概率为 1-(0.75)^20≈99.7%，几乎可以保证每次抽取都能删除已过期的 key；
- 当 p≤25% 时，至少有 1 个已过期的概率较低，重复执行的概率较低，避免占用过多 CPU。

### 三、过期时间的存储优化

Redis 的过期时间存储在独立的`expires`字典中，而非和数据一起存储，核心存储优化是：

1. **避免遍历整个`db->dict`字典**：独立存储`expires`字典，过期删除时仅需遍历`expires`字典，大幅提升了效率；
2. **哈希表的随机抽取优化**：Redis 的字典是哈希表，随机抽取的时间复杂度 O (1)，非常适合定期删除的随机抽取策略；
3. **毫秒级时间戳的精度优化**：过期时间存储为毫秒级 UNIX 时间戳，精度更高，适合需要精确过期的场景。

## 第四部分：避坑红线 & 高频踩坑点（面试绝对不能犯的错误）

### 一、核心概念红线（一旦说错，直接暴露基础不牢，扣基础分）

1. **绝对禁止说 “过期删除就是内存淘汰”**：必须明确，过期删除和内存淘汰是两个完全不同的机制，过期删除是针对设置了 TTL 的 key，内存淘汰是针对内存满了的所有 key，两者完全不同但有联动，这是面试最核心的概念红线；
2. **绝对禁止说 “Redis 只有惰性删除策略”**：必须明确，Redis 采用「惰性删除 + 定期删除」的混合策略，惰性删除保证性能，定期删除保证内存，这是面试最高频的考点；
3. **绝对禁止说 “定期删除会遍历整个`expires`字典”**：必须明确，定期删除每次仅随机抽取 20 个 key，单次执行时间限制在 25ms 以内，避免遍历整个`expires`字典，避免阻塞主线程，这是高频认知错误；
4. **绝对禁止说 “过期时间和数据一起存储在`db->dict`字典中”**：必须明确，过期时间存储在独立的`expires`字典中，避免遍历整个`db->dict`字典检查过期时间，大幅提升了效率，这是高频认知错误。

### 二、生产环境认知避坑

1. **绝对禁止生产环境不设置过期时间**：必须明确，不设置过期时间的 key 会一直占用内存，除非手动删除，生产环境必须给所有缓存 key 设置合理的过期时间；
2. **绝对禁止生产环境设置过长的过期时间**：必须明确，设置过长的过期时间会造成内存浪费，生产环境必须根据业务场景设置合理的过期时间；
3. **绝对禁止生产环境设置过短的过期时间**：必须明确，设置过短的过期时间会造成缓存雪崩，生产环境必须根据业务场景设置合理的过期时间，同时可以设置随机的过期时间（±10%）；
4. **绝对禁止生产环境调大`hz`超过 50**：必须明确，`hz`调大超过 50 会导致后台任务占用过多 CPU，影响 Redis 的性能，生产环境`hz`建议设置为 10-50；
5. **绝对禁止生产环境在不需要时开启过期通知**：必须明确，过期通知会占用一定的 CPU 和内存，仅在需要时开启，优先使用键事件通知，而不是键空间通知。

## 第一部分：问题本质与核心前提

### 一、高频误区纠正

首先必须明确：**没有 100% 完美的强一致性方案**，只能在**一致性、可用性、性能**三者之间做平衡，这是 CAP 定理的限制 —— 分布式系统无法同时满足强一致性、高可用性、分区容错性，只能根据业务场景做取舍。

---

### 二、核心问题拆解

缓存与数据库不一致的核心场景有 3 个：

1. **并发脏读**：两个线程同时读写，导致缓存加载到旧数据；
2. **操作失败**：更新数据库成功，但删除缓存失败，导致缓存残留旧数据；
3. **主从延迟**：读写分离架构下，读从库读到旧数据，写入缓存，导致缓存脏数据。

---

## 第二部分：分层次解决方案

从**简单临时方案**到**成熟生产方案**，再到**通用架构兜底方案**，每个方案都讲**适用场景、优缺点、实现要点**，方便用户根据自身情况选择。

---

### 一、基础方案：Cache Aside Pattern 变种（简单场景首选）

#### 1. 最常用的标准方案：先更新数据库，再删除缓存

- **为什么是删除缓存而不是更新缓存？**
    
    更新缓存会有**并发安全问题**：比如线程 A 更新数据库为 v1，线程 B 更新数据库为 v2，线程 B 先更新缓存为 v2，线程 A 后更新缓存为 v1，最终缓存是 v1，数据库是 v2，脏数据；
    
    删除缓存更简单，下次读取时再从数据库加载最新的，避免并发脏数据。
- **为什么是先更新数据库再删除缓存，而不是先删除再更新？**
    
    先删除再更新会有**并发脏读问题**：线程 A 先删除缓存，然后去更新数据库（耗时较长）；线程 B 此时读取，缓存没有，从数据库读到旧数据，写入缓存；最终缓存是旧数据，数据库是新数据，脏数据。
- **实现流程**：
    
    1. 写请求：先更新数据库，更新成功后，立即删除 Redis 缓存；
    2. 读请求：先读 Redis 缓存，缓存没有则读数据库，读取成功后写入 Redis 缓存；
    
- **优点**：
    
    实现非常简单，不需要引入额外中间件；
    
    并发脏读风险比先删除再更新低；
- **缺点**：
    
    更新数据库成功，删除缓存失败，导致缓存残留旧数据；
    
    删除缓存后，大量请求同时读取，都打到数据库，引发缓存击穿；
    
    有短暂的不一致窗口（更新数据库成功但还没删除缓存的瞬间）。

#### 2. 优化方案：先删除缓存，再更新数据库，最后延迟二次删除缓存

- **解决的问题**：先删除再更新的并发脏读问题；
- **实现流程**：
    
    1. 写请求：先删除 Redis 缓存；
    2. 再更新数据库；
    3. 等待一段**业务执行时间的 2-3 倍**（比如 1 秒）；
    4. 再删除一次 Redis 缓存（二次删除），确保即使中间有并发脏读，二次删除也能清理掉；
    
- **优点**：
    
    大幅降低先删除再更新的并发脏读风险；
- **缺点**：
    
    延迟时间不好控制，太短了没用，太长了影响一致性；
    
    依然存在删除缓存失败的问题；
    
    有短暂的不一致窗口。

### 二、进阶生产方案：保证最终一致性（高并发场景首选）

#### 1. 基于消息队列的异步重试删除缓存（解决删除缓存失败问题）

- **解决的问题**：更新数据库成功，但删除缓存失败，导致缓存残留旧数据；
- **实现流程**：
    
    1. 写请求：先更新数据库，更新成功后，**先尝试同步删除 Redis 缓存**；
    2. 如果同步删除成功，直接返回；
    3. 如果同步删除失败，把「删除缓存 key」的请求发送到消息队列（Kafka/RabbitMQ）；
    4. 消费消息队列，异步重试删除缓存，直到成功（最多重试 3 次，超过 3 次告警）；
    
- **优点**：
    
    保证删除缓存最终成功，最终一致性；
    
    同步删除 + 异步重试，兼顾性能和可靠性；
- **缺点**：
    
    引入了消息队列，增加了系统复杂度；
    
    有短暂的不一致窗口。

#### 2. 基于 Binlog 的异步更新缓存（Canal/Maxwell，阿里系生产环境首选）

这是**生产环境最推荐、最成熟**的方案，阿里、美团等大厂都在用，完全解耦缓存和数据库，保证最终一致性。

- **核心原理**：
    
    应用只操作数据库，不操作缓存；通过订阅 MySQL 的 Binlog（二进制日志），解析数据变更，异步删除对应的缓存。
- **实现流程**：
    
    1. **应用层**：
        
        读请求：先读 Redis 缓存，缓存没有则读数据库，读取成功后写入 Redis 缓存；
        
        写请求：**只更新数据库，完全不操作 Redis 缓存**；
    2. **Binlog 订阅层**：
        
        使用**Canal（阿里开源）**或**Maxwell（Zendesk 开源）**，伪装成 MySQL 从库，订阅 MySQL 的 Binlog；
    3. **缓存更新层**：
        
        Canal/Maxwell 解析 Binlog，得到数据变更（INSERT/UPDATE/DELETE）；
        
        解析到数据变更后，**异步删除对应的 Redis 缓存 key**；
    
- **优点**：
    
    1. **应用代码极简**：应用只需要操作数据库，不需要关心缓存，解耦非常彻底；
    2. **数据变更不丢失**：基于 Binlog，数据变更持久化在 MySQL，不会丢失，保证最终一致性；
    3. **架构清晰**：缓存和数据库完全解耦，便于维护和扩展；
    4. **性能高**：应用只操作数据库，性能损耗极低；
    
- **缺点**：
    
    1. **引入额外组件**：需要部署维护 Canal/Maxwell，增加了系统复杂度；
    2. **有短暂不一致窗口**：不一致时间取决于 Binlog 解析和消费的延迟，通常在几十毫秒到几百毫秒之间；
    
- **适用场景**：
    
    高并发、对一致性有较高要求的生产环境，尤其是阿里系技术栈的公司。

### 三、通用架构兜底方案（不管用什么方案都可以用，作为补充）

#### 1. 缓存设置合理的过期时间（终极兜底）

- 作为所有方案的兜底，即使所有一致性方案都失效，缓存过期后也会重新从数据库加载最新数据，保证最终一致性；
- 过期时间设置：
    
    根据业务数据的更新频率设置，更新频繁的设置短一点（比如 5 分钟），更新不频繁的设置长一点（比如 1 小时）；
    
    可以设置随机过期时间，避免大量缓存同时过期，引发缓存雪崩。

#### 2. 并发控制：缓存击穿 / 缓存雪崩 / 缓存穿透的防护

- **缓存击穿**：
    
    使用**Redisson 分布式互斥锁**，缓存没有时，只有一个线程能去数据库加载数据，其他线程等待，避免大量请求打到数据库；
- **缓存雪崩**：
    
    缓存过期时间设置随机值（比如基础时间 ±10%），避免大量缓存同时过期；
- **缓存穿透**：
    
    使用**布隆过滤器**，拦截不存在的 key，避免打到数据库。

#### 3. 幂等性设计（分布式系统的终极兜底）

即使缓存和数据库有短暂的不一致，也要保证**业务操作是幂等的**—— 同一请求多次执行，结果和执行一次完全相同，避免数据重复；

- 实现方式：
    
    用**唯一业务 ID**（比如订单号、支付流水号）作为数据库的唯一索引，或者作为 Redis 的 key，重复请求会被拒绝。

## 第三部分：总结与选择路径

### 一、清晰的选择路径

1. **简单场景，并发不高**：
    
    先更新数据库，再删除缓存，加缓存过期时间兜底；
2. **中等场景，并发较高**：
    
    先删除缓存，再更新数据库，最后延迟二次删除缓存，加 Redisson 互斥锁防缓存击穿；
3. **生产环境，高并发，对一致性有较高要求**：
    
    基于 Binlog 的 Canal/Maxwell 方案，加消息队列异步重试，加缓存过期时间兜底；
4. **金融级场景，对一致性要求极高**：
    
    优先考虑使用 CP 系统的数据库，或者使用分布式事务（Seata/TCC），但性能会下降。

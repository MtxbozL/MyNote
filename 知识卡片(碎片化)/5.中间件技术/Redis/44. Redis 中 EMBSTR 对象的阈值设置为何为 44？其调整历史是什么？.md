## 一、EMBSTR 阈值的完整调整历史

EMBSTR 是 Redis String 类型的短字符串专用编码，其阈值并非一直固定为 44 字节，核心调整完全绑定**SDS（简单动态字符串）的结构重构**，完整演进分为 4 个阶段：

|Redis 版本|EMBSTR 阈值|核心调整背景|
|---|---|---|
|Redis 2.x|无 EMBSTR 编码|该版本未引入 EMBSTR 编码，所有字符串均使用 RAW 编码，分两次分配`redisObject`和 SDS 的内存|
|Redis 3.0|39 字节|首次引入 EMBSTR 编码，基于经典 SDS 结构设计，阈值为 39 字节，实现`redisObject`+SDS 的连续内存分配|
|Redis 3.2 ~ 7.x|44 字节|对 SDS 结构做了颠覆性精细化重构，拆分出 5 种不同头部的 SDS 结构，短字符串使用`sdshdr8`替代经典 SDS，头部从 8 字节压缩到 3 字节，阈值同步调整为 44 字节，该阈值稳定沿用至今|
|Redis 8.x|适配调整|引入`kvobj`统一键值结构，EMBSTR 的缓存行适配逻辑略有调整，但核心设计逻辑不变，主流生产环境仍以 44 字节为标准阈值|

> 源码佐证：Redis 源码中通过宏定义`OBJ_ENCODING_EMBSTR_SIZE_LIMIT`控制阈值，3.0 版本定义为 39，3.2 + 版本修改为 44，源码注释明确说明：**44 字节的选择，是为了让最大的 EMBSTR 对象刚好能塞进 jemalloc 内存分配器的 64 字节分配单元中**。

## 二、44 字节阈值的核心由来

### 1. EMBSTR 编码的核心设计目标

EMBSTR（Embedded String，嵌入式字符串）的设计初衷，是为短字符串做极致的内存与性能优化：

- RAW 编码会分**两次内存分配**，分别创建`redisObject`（通用对象头）和 SDS 结构，两者内存不连续，需要两次内存寻址；
- EMBSTR 编码会**一次性分配连续的内存块**，同时包含`redisObject`和 SDS 结构，内存完全连续，仅需一次内存分配 / 释放，且能一次性加载到 CPU 缓存行中，最大化 CPU 缓存命中率，访问性能比 RAW 编码高 2-3 倍。

而 44 字节的阈值，本质是**为了让整个 EMBSTR 对象刚好填满 x86 架构 CPU 的 64 字节标准缓存行**，同时适配 jemalloc 内存分配器的 64 字节最小分配单元，不产生内存浪费。

### 2. 44 字节的精准内存计算（64 位系统，Redis 3.2+）

64 位系统下，整个 EMBSTR 对象的总内存上限为 64 字节，我们逐行拆解固定开销，最终得到可用于存储字符串内容的空间：

表格

|结构组成|占用内存|说明|
|---|---|---|
|`redisObject`（通用对象头）|16 字节|64 位系统下固定大小，结构为：<br><br>`type:4bit + encoding:4bit + lru:24bit`（合计 4 字节）<br><br>`refcount:int`（4 字节）<br><br>`void *ptr`（8 字节指针）<br><br>总计：4+4+8=16 字节|
|`sdshdr8`（SDS 头部）|3 字节|Redis 3.2 + 短字符串专用 SDS 结构，无内存对齐，结构为：<br><br>`uint8_t len`（1 字节，已用长度）<br><br>`uint8_t alloc`（1 字节，总分配长度）<br><br>`unsigned char flags`（1 字节，类型标志）<br><br>总计：1+1+1=3 字节|
|字符串末尾`\0`结束符|1 字节|为了兼容 C 语言原生字符串函数，SDS 的 buf 数组末尾会自动追加`\0`，必须占用 1 字节|
|**可用于存储字符串内容的空间**|**44 字节**|总上限 64 字节 - 16 字节 (redisObject) - 3 字节 (sdshdr8) - 1 字节 (\0) = 44 字节|

这就是 44 字节阈值的核心由来：超过 44 字节的字符串，整个对象就无法塞进 64 字节的缓存行，失去了 EMBSTR 编码的设计意义，Redis 会自动将其转换为 RAW 编码。

## 三、3.2 版本之前 39 字节阈值的由来

Redis 3.0 版本首次引入 EMBSTR 编码时，SDS 还是经典的单一结构，头部占用 8 字节，因此阈值为 39 字节，计算逻辑完全一致：

表格

|结构组成|占用内存|说明|
|---|---|---|
|`redisObject`|16 字节|64 位系统下固定大小，无变化|
|经典`sdshdr`头部|8 字节|3.2 之前的 SDS 结构：<br><br>`unsigned int len`（4 字节）<br><br>`unsigned int free`（4 字节）<br><br>总计 8 字节|
|字符串末尾`\0`|1 字节|兼容 C 函数的结束符，无变化|
|可存储字符串内容|39 字节|64 - 16 - 8 - 1 = 39 字节|

Redis 3.2 对 SDS 做了精细化重构，针对不同长度的字符串拆分出 5 种不同的头部结构，短字符串使用`sdshdr8`将头部从 8 字节压缩到 3 字节，因此阈值同步增加了 5 字节，从 39 字节调整为 44 字节。

## 四、EMBSTR 编码的核心特性与设计细节

1. **只读特性**
    
    EMBSTR 编码是**只读**的：因为其内存是连续分配的，没有预留空闲空间，任何修改操作（如`APPEND`、`SETRANGE`）都会导致内存重新分配，Redis 会先将其转换为 RAW 编码，再执行修改操作，修改后不会自动切回 EMBSTR 编码。
    
2. **为什么不用 sdshdr5？**
    
    Redis 3.2 新增的`sdshdr5`头部仅占 1 字节，理论上可以让阈值更大，但实际 EMBSTR 编码并未使用它：
    
    - `sdshdr5`没有`alloc`字段，无法记录总分配长度，无法管理空闲空间，不支持字符串修改；
    - `sdshdr5`仅能存储长度 < 32 字节的字符串，收益有限，因此 Redis 统一使用`sdshdr8`作为 EMBSTR 的底层 SDS 结构，保证兼容性和扩展性。
    
3. **CPU 缓存局部性优化**
    
    x86 架构的 CPU 缓存行固定为 64 字节，EMBSTR 的连续内存设计，能让整个对象一次性加载到 CPU 缓存中，访问`redisObject`和字符串内容仅需 1 次内存读取；而 RAW 编码的两个结构内存不连续，大概率不在同一个缓存行，需要两次内存读取，访问延迟更高。

## 五、面试高频误区纠正

1. **误区 1：44 字节是整个 EMBSTR 对象的总大小**
    
    纠正：44 字节是**字符串内容本身的最大长度**，而非整个对象的总大小。整个 EMBSTR 对象的总大小为 16+3+44+1=64 字节，刚好填满一个缓存行。
    
2. **误区 2：EMBSTR 和 RAW 的核心区别是字符串长度**
    
    纠正：核心区别是**内存分配方式**，EMBSTR 是一次连续内存分配，RAW 是两次独立内存分配，长度阈值只是这个设计带来的结果。
    
3. **误区 3：小于 44 字节的字符串一定会用 EMBSTR 编码**
    
    纠正：只有无法用 int 编码的字符串才会用 EMBSTR 编码。如果字符串内容是 long 范围内的整数值（如`SET key 12345`），Redis 会直接使用 int 编码，将数值存在`redisObject`的 ptr 字段中，无需分配 SDS 内存，比 EMBSTR 更高效。
    
4. **误区 4：Redis 3.2 + 阈值可以手动修改**
    
    纠正：阈值是源码中的宏定义硬编码，无法通过配置文件修改，只能通过修改源码重新编译调整，官方不建议修改该值。

## 六、拓展：Redis 8.x 的适配调整

Redis 8.x 版本引入了`kvobj`结构，将键和值统一封装，EMBSTR 的缓存行适配逻辑略有调整：新结构会将键和值的总长度纳入考量，优先保证整个键值对能塞进缓存行。但核心设计逻辑不变，短字符串的阈值仍以 44 字节为基准，主流生产环境（Redis 6.x/7.x）仍以 44 字节为标准阈值。

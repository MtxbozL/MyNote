## 第一部分：标准化答题模板（极致精简・口语化・零晦涩）

### Level 1：初级（校招 / 初级岗，5 分钟背完，基础分全拿）

针对这个问题，我先纠正一个面试高频核心误区：**内存淘汰≠过期删除**—— 过期删除是针对**设置了 TTL/EXPIRE 的 key**，内存淘汰是针对**内存达到`maxmemory`阈值的所有 key**，两者完全不同但有联动（内存淘汰时优先淘汰已过期的 key）。

---

#### 一、8 种核心内存淘汰策略（面试必背）

Redis 4.0 + 支持 8 种内存淘汰策略，分为 **「不淘汰」「淘汰所有 key」「仅淘汰设置了过期时间的 key」** 三大类：

|策略分类|策略名称|核心说明|核心时间复杂度|典型业务场景|
|---|---|---|---|---|
|**不淘汰**|`noeviction`|内存达到`maxmemory`阈值时，**直接拒绝所有写命令**，返回`OOM command not allowed when used memory > 'maxmemory'`错误|O(1)|核心数据存储、绝对不能丢失数据的场景|
|**淘汰所有 key**|`allkeys-lru`|从**所有 key**中，优先淘汰 ** 最近最少使用（LRU）** 的 key|O (1)（近似 LRU，非严格 LRU）|纯缓存场景、所有 key 都可淘汰的场景|
||`allkeys-lfu`|从**所有 key**中，优先淘汰 ** 最近最少访问（LFU）** 的 key（Redis 4.0 + 新增）|O (1)（近似 LFU，非严格 LFU）|纯缓存场景、访问频率差异大的场景（比如热点商品缓存）|
||`allkeys-random`|从**所有 key**中，**随机淘汰**key|O(1)|纯缓存场景、所有 key 访问频率差异极小的场景|
|**仅淘汰设置了过期时间的 key**|`volatile-lru`|从**设置了过期时间的 key**中，优先淘汰 ** 最近最少使用（LRU）** 的 key|O (1)（近似 LRU，非严格 LRU）|核心数据 + 缓存混合场景、核心数据不设置过期时间、缓存设置过期时间的场景|
||`volatile-lfu`|从**设置了过期时间的 key**中，优先淘汰 ** 最近最少访问（LFU）** 的 key（Redis 4.0 + 新增）|O (1)（近似 LFU，非严格 LFU）|核心数据 + 缓存混合场景、核心数据不设置过期时间、缓存访问频率差异大的场景|
||`volatile-random`|从**设置了过期时间的 key**中，**随机淘汰**key|O(1)|核心数据 + 缓存混合场景、核心数据不设置过期时间、缓存访问频率差异极小的场景|
||`volatile-ttl`|从**设置了过期时间的 key**中，优先淘汰 ** 剩余过期时间最短（TTL）** 的 key|O(1)|核心数据 + 缓存混合场景、核心数据不设置过期时间、缓存需要快速过期的场景|

#### 二、核心配置（面试必背）

表格

|配置项|核心作用|核心说明|
|---|---|---|
|`maxmemory <bytes>`|设置 Redis 的最大内存阈值|单位支持 b/k/m/g，生产环境必须设置，避免 Redis 内存占满宿主机内存，触发操作系统 OOM|

### Level 2：进阶（中高级岗 / 大厂面试，精简有亮点，可承接深挖）

针对这个问题，我的核心理解是：**Redis 的内存淘汰策略，是「近似算法优先、性能优先、避免阻塞主线程」三大设计原则的完美结合**—— 近似 LRU/LFU 避免了严格 LRU/LFU 的内存和 CPU 开销，性能优先保证了淘汰操作的高效，避免阻塞主线程保证了 Redis 的高可用性。

---

#### 一、近似 LRU 的实现原理（面试必背的源码级考点）

Redis 的 LRU 不是严格 LRU（严格 LRU 需要维护一个双向链表，每次访问 key 都要移动到链表头部，内存和 CPU 开销极大），而是**近似 LRU**，核心实现原理如下：

1. **采样机制**：当内存达到`maxmemory`阈值时，Redis 会从**所有 key**（或仅设置了过期时间的 key，取决于策略）中，**随机抽取`maxmemory-samples`个 key**（默认 5）；
2. **LRU 字段**：每个 key 的`redisObject`结构中，有一个`lru`字段（24 位），存储的是**key 最后一次被访问的时间戳的低 24 位**（秒级，Redis 4.0 + 之前是秒级，4.0 + 之后是毫秒级，但`lru`字段还是 24 位，存储的是毫秒级时间戳的低 24 位，约 194 天循环一次）；
3. **淘汰机制**：Redis 会比较这`maxmemory-samples`个 key 的`lru`字段，**淘汰`lru`字段最小的 key**（即最后一次被访问时间最早的 key）；
4. **重复执行**：如果淘汰一个 key 后，内存仍未降到`maxmemory`阈值以下，则**立即重复执行**步骤 1-3，直到内存降到阈值以下。

---

#### 二、近似 LFU 的实现原理（面试必背的源码级考点）

Redis 4.0 + 新增了 LFU（最近最少访问）策略，核心实现原理如下：

1. **LFU 字段**：每个 key 的`redisObject`结构中，原来的`lru`字段（24 位）被拆分为两部分：

- `ldt`（Last Decrement Time）：16 位，存储的是**key 最后一次被访问的时间戳的低 16 位**（秒级，约 18 小时循环一次）；
- `logc`（Logarithmic Counter）：8 位，存储的是**key 的访问频率的对数近似值**（范围 0-255，值越大，访问频率越高）；

2. **访问频率更新机制**：

- 每次访问 key 时，Redis 会：
    
    1. 检查当前时间戳的低 16 位与`ldt`的差值，如果差值超过`lfu-decay-time`（默认 1 分钟），则将`logc`减 1（衰减访问频率）；
    2. 将`ldt`更新为当前时间戳的低 16 位；
    3. 以一定的概率（`logc`越大，概率越高）将`logc`加 1（增加访问频率）；
    

3. **采样机制**：当内存达到`maxmemory`阈值时，Redis 会从**所有 key**（或仅设置了过期时间的 key，取决于策略）中，**随机抽取`maxmemory-samples`个 key**（默认 5）；
4. **淘汰机制**：Redis 会比较这`maxmemory-samples`个 key 的`logc`字段，**淘汰`logc`字段最小的 key**（即访问频率最低的 key）；
5. **重复执行**：如果淘汰一个 key 后，内存仍未降到`maxmemory`阈值以下，则**立即重复执行**步骤 3-4，直到内存降到阈值以下。

#### 三、近似 LRU/LFU vs 严格 LRU/LFU 的核心差异（面试必背的性能对比考点）

表格

|对比维度|近似 LRU/LFU|严格 LRU/LFU|
|---|---|---|
|内存开销|极低，仅需每个 key 的 24 位字段|极高，严格 LRU 需要维护一个双向链表，每个 key 需要两个指针（prev/next），内存开销是近似 LRU 的数倍|
|CPU 开销|极低，仅需随机抽取、比较字段、淘汰 key|极高，严格 LRU 每次访问 key 都要移动到链表头部，锁竞争（如果是多线程）、内存拷贝开销极大|
|淘汰准确性|较高，采样数 5-10 时，近似 LRU/LFU 的淘汰准确性接近严格 LRU/LFU 的 95% 以上|极高，100% 淘汰最近最少使用 / 访问的 key|
|适用场景|所有 Redis 场景，生产环境强烈推荐|仅适用于内存和 CPU 资源极其充足、淘汰准确性要求 100% 的场景，生产环境几乎不使用|

#### 四、生产级最佳实践

1. **必须设置`maxmemory`和`maxmemory-policy`**：

- 生产环境必须设置`maxmemory`，避免 Redis 内存占满宿主机内存，触发操作系统 OOM；
- 生产环境必须根据业务场景调整`maxmemory-policy`，默认`noeviction`几乎不适用；

2. **优先使用`volatile-lru`或`volatile-lfu`**：

- 核心数据 + 缓存混合场景，优先使用`volatile-lru`或`volatile-lfu`，仅淘汰设置了过期时间的缓存 key，避免删除未过期的核心数据；
- 访问频率差异大的场景（比如热点商品缓存），优先使用`volatile-lfu`或`allkeys-lfu`；
- 访问频率差异极小的场景，优先使用`volatile-random`或`allkeys-random`；

3. **合理调整`maxmemory-samples`**：

- `maxmemory-samples`默认 5，采样数越多，近似 LRU/LFU 越接近严格 LRU/LFU，但 CPU 开销越大；
- 生产环境建议设置为 5-10，采样数 10 时，近似 LRU/LFU 的淘汰准确性接近严格 LRU/LFU 的 99% 以上；

4. **合理调整`lfu-decay-time`**：

- `lfu-decay-time`默认 1 分钟，访问频率的衰减时间；
- 访问频率变化快的场景（比如秒杀活动缓存），可以适当调小`lfu-decay-time`（比如 30 秒）；
- 访问频率变化慢的场景（比如用户信息缓存），可以适当调大`lfu-decay-time`（比如 5 分钟）；

5. **避免在内存淘汰时执行耗时的命令**：

- 虽然内存淘汰是近似算法，性能极高，但在内存淘汰时执行耗时的命令（如`KEYS *`、`LRANGE 0 -1`）会阻塞主线程，生产环境必须避免。

## 第二部分：学术深度解析

### 一、近似算法的设计依据

Redis 的近似 LRU/LFU，严格遵循 **「性能优先、准确性次之」三大设计原则 **，核心学术依据是：

1. **性能优先原则**：严格 LRU/LFU 的内存和 CPU 开销极大，在 Redis 这种高性能内存数据库场景下，性能优先于 100% 的淘汰准确性；
2. **近似算法的准确性保证**：随机抽取 5-10 个 key，近似 LRU/LFU 的淘汰准确性接近严格 LRU/LFU 的 95% 以上，完全满足绝大多数业务场景的需求；
3. **避免阻塞主线程原则**：近似 LRU/LFU 的淘汰操作时间复杂度 O (1)，单次执行时间极短，避免阻塞 Redis 单线程主线程。

### 二、近似 LRU 的采样概率分析

近似 LRU 每次随机抽取`maxmemory-samples`个 key，核心概率分析是：

- 假设最近最少使用的 key 在所有 key 中的占比为 p，随机抽取`maxmemory-samples`个 key，其中至少有 1 个最近最少使用的 key 的概率为 1-(1-p)^maxmemory-samples；
- 当 p=10%，maxmemory-samples=5 时，至少有 1 个最近最少使用的 key 的概率为 1-(0.9)^5≈41%；
- 当 p=10%，maxmemory-samples=10 时，至少有 1 个最近最少使用的 key 的概率为 1-(0.9)^10≈65%；
- 当 p=20%，maxmemory-samples=5 时，至少有 1 个最近最少使用的 key 的概率为 1-(0.8)^5≈67%；
- 当 p=20%，maxmemory-samples=10 时，至少有 1 个最近最少使用的 key 的概率为 1-(0.8)^10≈89%；
- 当 p=50%，maxmemory-samples=5 时，至少有 1 个最近最少使用的 key 的概率为 1-(0.5)^5≈97%；
- 当 p=50%，maxmemory-samples=10 时，至少有 1 个最近最少使用的 key 的概率为 1-(0.5)^10≈99.9%；
- 由此可见，采样数 5-10 时，近似 LRU 的淘汰准确性已经非常高。

### 三、近似 LFU 的对数计数器设计

近似 LFU 的`logc`字段是 8 位的对数计数器，核心设计依据是：

- 访问频率的增长不是线性的，而是对数的，即访问频率越高，增长越慢；
- 8 位的对数计数器可以覆盖 0-255 的范围，完全满足绝大多数业务场景的需求；
- 对数计数器的概率增长机制，避免了访问频率的无限增长，同时保证了访问频率高的 key 增长更快。

## 第四部分：避坑红线 & 高频踩坑点（面试绝对不能犯的错误）

### 一、核心概念红线（一旦说错，直接暴露基础不牢，扣基础分）

1. **绝对禁止说 “内存淘汰就是过期删除”**：必须明确，内存淘汰和过期删除是两个完全不同的机制，内存淘汰是针对内存满了的所有 key，过期删除是针对设置了 TTL 的 key，两者完全不同但有联动，这是面试最核心的概念红线；
2. **绝对禁止说 “Redis 的 LRU 是严格 LRU”**：必须明确，Redis 的 LRU 是近似 LRU，不是严格 LRU，严格 LRU 的内存和 CPU 开销极大，生产环境几乎不使用，这是面试最高频的考点；
3. **绝对禁止说 “Redis 的 LFU 是 Redis 3.0 + 新增的”**：必须明确，Redis 的 LFU 是 Redis 4.0 + 新增的，不是 3.0+，这是高频认知错误；
4. **绝对禁止说 “`maxmemory-samples`越大越好”**：必须明确，`maxmemory-samples`越大，近似 LRU/LFU 越接近严格 LRU/LFU，但 CPU 开销越大，生产环境建议设置为 5-10，这是高频认知错误。

### 二、生产环境认知避坑

1. **绝对禁止生产环境不设置`maxmemory`**：必须明确，不设置`maxmemory`会导致 Redis 内存占满宿主机内存，触发操作系统 OOM，导致 Redis 进程被杀死，生产环境必须设置`maxmemory`；
2. **绝对禁止生产环境使用`noeviction`策略**：必须明确，`noeviction`策略内存达到阈值时直接拒绝所有写命令，生产环境几乎不适用，除非是绝对不能丢失数据的核心数据存储场景；
3. **绝对禁止生产环境调大`maxmemory-samples`超过 10**：必须明确，`maxmemory-samples`调大超过 10 会导致后台任务占用过多 CPU，影响 Redis 的性能，生产环境建议设置为 5-10；
4. **绝对禁止生产环境在不需要时使用`allkeys-lru`或`allkeys-lfu`**：必须明确，`allkeys-lru`或`allkeys-lfu`会淘汰所有 key，包括未过期的核心数据，生产环境核心数据 + 缓存混合场景必须使用`volatile-lru`或`volatile-lfu`。

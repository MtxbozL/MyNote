## 第一部分：问题本质与高频误区

### 一、高频误区纠正

首先必须明确：**Set 存用户 ID 仅适用于小量 UV（≤10 万），大量 UV（百万级、千万级、亿级）必须用 HyperLogLog（HLL）或 Bitmap，否则会有 Big Key 问题，内存占用爆炸，性能下降**。

---

## 第二部分：分层次解决方案

从**简单临时方案**到**成熟生产方案**，再到**极端场景优化方案**，每个方案都讲**适用场景、优缺点、核心命令、实现要点、内存占用对比**，方便用户根据自身情况选择。

---

### 一、简单临时方案：Set（仅适用于小量 UV，≤10 万）

#### 1. 核心原理

Set 是 Redis 5 种基础数据类型中唯一的**无序不重复集合**，天然适合存唯一用户 ID，天然去重。

#### 2. 核心命令

表格

|命令|核心作用|核心时间复杂度|核心说明|
|---|---|---|---|
|`SADD key member [member ...]`|添加用户 ID 到 Set，天然去重|O (1)，单次添加；O (n)，批量添加（n 是批量添加的数量）|重复添加 member 会返回 0，不会新增元素|
|`SCARD key`|获取 Set 的元素总数（即 UV）|O(1)|直接读取 Set 的头部字段，无需遍历|
|`SISMEMBER key member`|判断用户 ID 是否在 Set 中|O(1)|适合判断用户是否已访问|
|`SREM key member [member ...]`|删除 Set 中的用户 ID|O (1)，单次删除；O (n)，批量删除|适合清理过期的用户 ID|

#### 3. 典型实现流程（以每日 UV 为例）

bash

运行

```
# 添加用户1001到今日UV Set
SADD uv:20260404 user:1001
# 添加用户1002到今日UV Set
SADD uv:20260404 user:1002
# 获取今日UV
SCARD uv:20260404
# 判断用户1001是否已访问今日
SISMEMBER uv:20260404 user:1001
```

#### 4. 优缺点与适用场景

表格

|维度|优点|缺点|适用场景|
|---|---|---|---|
|准确性|100% 准确，天然去重|无|小量 UV（≤10 万），对准确性要求极高的场景|
|内存占用|小量 UV 时内存占用低|大量 UV 时内存占用爆炸：100 万 UUID（36 字节）约占 36MB，1000 万约占 360MB，1 亿约占 3.6GB，Big Key 问题严重|小量 UV 场景|
|性能|小量 UV 时性能极高|大量 UV 时性能下降：Big Key 的添加、删除、遍历都会阻塞主线程|小量 UV 场景|

### 二、成熟生产方案：HyperLogLog（HLL）（生产环境首选，大厂必问，亿级 UV 首选）

#### 1. 核心原理

HyperLogLog 是 Redis 2.8.9 + 新增的**概率性数据结构**，专门用于**统计大量唯一元素的基数（即 UV）**，核心原理是**基于概率统计的基数估算算法**，通过记录元素哈希值的前导零个数，估算唯一元素的总数。

#### 2. 核心特性（面试必背）

表格

|特性|说明|
|---|---|
|准确性|不是 100% 准确，**标准误差率≤0.81%**，完全满足绝大多数业务场景的 UV 统计需求|
|内存占用|极低，**单个 HLL 固定占用 12KB 内存**，无论存 1 个还是 1 亿个唯一元素，内存占用都是 12KB，彻底解决了 Big Key 问题|
|性能|极高，所有操作的时间复杂度都是 O (1)，不会阻塞主线程|
|支持合并|支持多个 HLL 合并成一个，适合统计周 UV、月 UV、跨页面 UV 等聚合场景|
|不支持判断元素是否存在|仅支持统计基数，不支持判断用户 ID 是否已访问，这是 HLL 的核心边界|

#### 3. 核心命令（面试必背）

表格

|命令|核心作用|核心时间复杂度|核心说明|
|---|---|---|---|
|`PFADD key element [element ...]`|添加用户 ID 到 HLL，天然去重|O (1)，单次添加；O (n)，批量添加（n 是批量添加的数量）|重复添加 element 会返回 0，不会新增元素；如果 HLL 的基数估算值发生变化，会返回 1|
|`PFCOUNT key [key ...]`|获取单个 HLL 的基数（即 UV），或合并多个 HLL 后获取基数|O (1)，单个 HLL；O (m)，合并 m 个 HLL|合并多个 HLL 时，不会修改原 HLL，会返回合并后的基数估算值|
|`PFMERGE destkey sourcekey [sourcekey ...]`|合并多个 HLL 到目标 HLL|O (m)，合并 m 个 HLL|合并后，目标 HLL 的基数估算值是所有源 HLL 的并集的基数估算值|

#### 4. 典型实现流程（以每日 / 每周 / 月 UV 为例）

bash

运行

```
# 添加用户1001到今日UV HLL
PFADD uv:20260404 user:1001
# 添加用户1002到今日UV HLL
PFADD uv:20260404 user:1002
# 获取今日UV（标准误差率≤0.81%）
PFCOUNT uv:20260404
# 合并本周7天的UV HLL到本周UV HLL
PFMERGE uv:2026w14 uv:20260401 uv:20260402 uv:20260403 uv:20260404 uv:20260405 uv:20260406 uv:20260407
# 获取本周UV
PFCOUNT uv:2026w14
```

#### 5. 优缺点与适用场景

表格

|维度|优点|缺点|适用场景|
|---|---|---|---|
|准确性|标准误差率≤0.81%，完全满足绝大多数业务场景|不是 100% 准确，对准确性要求极高的场景（如金融级 UV 统计）不适用|大量 UV（百万级、千万级、亿级），对准确性要求≤1% 的场景|
|内存占用|极低，单个 HLL 固定占用 12KB，彻底解决了 Big Key 问题|无|大量 UV 场景|
|性能|极高，所有操作的时间复杂度都是 O (1)，不会阻塞主线程|无|大量 UV 场景|
|支持合并|支持多个 HLL 合并，适合统计聚合 UV|无|周 UV、月 UV、跨页面 UV 等聚合场景|
|不支持判断元素是否存在|仅支持统计基数，不支持判断用户 ID 是否已访问|仅需要统计 UV，不需要判断用户是否已访问的场景||

### 三、进阶生产方案：Bitmap（中等规模 UV，百万级 - 千万级，需要判断用户是否已访问的场景首选）

#### 1. 核心原理

Bitmap 是 Redis 2.2 + 新增的**位操作数据结构**，本质是 String 类型的底层二进制数组，每个位（bit）代表一个用户 ID 的访问状态（0 = 未访问，1 = 已访问），天然去重，天然支持判断用户是否已访问。

#### 2. 核心特性（面试必背）

表格

|特性|说明|
|---|---|
|准确性|100% 准确，天然去重|
|内存占用|中等，取决于最大用户 ID：100 万用户 ID 约占 122KB，1000 万约占 1.2MB，1 亿约占 12MB，比 Set 小得多，但比 HLL 大|
|性能|极高，所有操作的时间复杂度都是 O (1)，不会阻塞主线程|
|支持判断元素是否存在|天然支持判断用户 ID 是否已访问|
|支持聚合|支持位运算（AND/OR/XOR/NOT），适合统计周 UV、月 UV、跨页面 UV 等聚合场景|
|核心边界|最大用户 ID 不能太大，否则内存占用会爆炸：10 亿用户 ID 约占 120MB，100 亿约占 1.2GB，虽然比 Set 小，但比 HLL 大得多|

#### 3. 核心命令（面试必背）

表格

|命令|核心作用|核心时间复杂度|核心说明|||
|---|---|---|---|---|---|
|`SETBIT key offset value`|设置用户 ID 对应的位（offset = 用户 ID-1，避免 offset=0 的问题），value=0/1|O(1)|重复设置 value=1 会返回原来的位值（0 或 1），不会改变位值|||
|`GETBIT key offset`|获取用户 ID 对应的位值（0 = 未访问，1 = 已访问）|O(1)|适合判断用户是否已访问|||
|`BITCOUNT key [start end [BYTE|BIT]]`|获取 Bitmap 中 1 的个数（即 UV）|O (n)，n 是 Bitmap 的字节数 / 位数|`start end`是可选的，默认统计整个 Bitmap；`BYTE|BIT` 是可选的，默认按字节统计|
|`BITOP operation destkey key [key ...]`|对多个 Bitmap 执行位运算（AND/OR/XOR/NOT），结果存到目标 Bitmap|O (m*n)，m 是 Bitmap 的个数，n 是 Bitmap 的字节数|AND = 交集（同时访问多个页面的 UV），OR = 并集（周 UV、月 UV），XOR = 对称差（只访问一个页面的 UV），NOT = 补集（未访问的 UV）|||

#### 4. 典型实现流程（以每日 / 周 UV，判断用户是否已访问为例）

bash

运行

```
# 设置用户1001已访问今日（offset=1000，避免offset=0）
SETBIT uv:20260404 1000 1
# 设置用户1002已访问今日
SETBIT uv:20260404 1001 1
# 判断用户1001是否已访问今日
GETBIT uv:20260404 1000
# 获取今日UV
BITCOUNT uv:20260404
# 合并本周7天的UV Bitmap到本周UV Bitmap（OR=并集）
BITOP OR uv:2026w14 uv:20260401 uv:20260402 uv:20260403 uv:20260404 uv:20260405 uv:20260406 uv:20260407
# 获取本周UV
BITCOUNT uv:2026w14
```

#### 5. 优缺点与适用场景

表格

|维度|优点|缺点|适用场景|
|---|---|---|---|
|准确性|100% 准确，天然去重|无|中等规模 UV（百万级 - 千万级），对准确性要求极高，且需要判断用户是否已访问的场景|
|内存占用|中等，比 Set 小得多，但比 HLL 大|最大用户 ID 不能太大，否则内存占用会爆炸|中等规模 UV，最大用户 ID≤1 亿的场景|
|性能|极高，所有操作的时间复杂度都是 O (1)，不会阻塞主线程|无|中等规模 UV 场景|
|支持判断元素是否存在|天然支持判断用户是否已访问|无|需要判断用户是否已访问的场景|
|支持聚合|支持位运算，适合统计聚合 UV|无|周 UV、月 UV、跨页面 UV 等聚合场景|

### 四、极端场景优化方案

#### 1. 亿级 UV，需要判断用户是否已访问的场景

当 UV 达到亿级，且需要判断用户是否已访问时，Bitmap 的最大用户 ID 不能太大，可通过**用户 ID 哈希分片**拆分 Bitmap，将不同的用户 ID 哈希到不同的 Bitmap：

bash

运行

```
# 用户ID哈希分片：按用户ID的最后一位哈希到不同的Bitmap
# 用户ID=1001，最后一位=1，哈希到uv:20260404:1
SETBIT uv:20260404:1 100 1
# 用户ID=1002，最后一位=2，哈希到uv:20260404:2
SETBIT uv:20260404:2 100 1
# 获取今日UV：聚合所有分片的Bitmap的BITCOUNT
BITCOUNT uv:20260404:0 + BITCOUNT uv:20260404:1 + ... + BITCOUNT uv:20260404:9
```

#### 2. 亿级 UV，对准确性要求极高的场景

当 UV 达到亿级，且对准确性要求极高（如金融级 UV 统计）时，HLL 的 0.81% 误差率不满足，可通过**HLL+Set 兜底**方案：

- 用 HLL 统计每日 UV，误差率≤0.81%；
- 用 Set 存每日 Top 10000 的用户 ID，100% 准确；
- 每日 UV=HLL 的基数估算值 + Set 的 Top 10000 的用户 ID 中未被 HLL 统计到的数量（可通过 Set 和 HLL 的对比计算）。

## 第三部分：总结与选择路径

### 一、清晰的选择路径

1. **小量 UV（≤10 万），对准确性要求极高**：
    
    Set 方案；
2. **大量 UV（百万级、千万级、亿级），对准确性要求≤1%，不需要判断用户是否已访问**：
    
    HyperLogLog（HLL）方案，生产环境首选；
3. **中等规模 UV（百万级 - 千万级），对准确性要求极高，且需要判断用户是否已访问**：
    
    Bitmap 方案；
4. **亿级 UV，需要判断用户是否已访问**：
    
    Bitmap + 用户 ID 哈希分片方案；
5. **亿级 UV，对准确性要求极高**：
    
    HLL+Set 兜底方案。

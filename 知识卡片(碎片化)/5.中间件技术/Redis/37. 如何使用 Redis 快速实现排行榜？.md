## 第一部分：问题本质与高频误区

### 一、高频误区纠正

首先必须明确：**Redis 实现排行榜的方案不止一种，但 ZSET（有序集合）是最标准、最常用、性能最高的方案**，其他方案（如 List、Hash+Lua）仅适用于极端小众场景，生产环境优先选择 ZSET。

### 一、简单标准方案：ZSET（生产环境首选，大厂必问）

#### 1. ZSET 为什么适合做排行榜？（面试必背核心原因）

ZSET 是 Redis 5 种基础数据类型中唯一的**有序集合**，完美适配排行榜的核心需求：

表格

|排行榜核心需求|ZSET 的对应特性|
|---|---|
|元素唯一|每个元素（member）唯一，重复添加会更新分数（score）|
|按分数排序|自动按分数（score）从小到大排序，可通过`ZREVRANGE`/`ZREVRANK`快速获取倒序（从高到低）的排名和元素|
|快速获取排名|O (logN) 时间复杂度获取单个元素的排名|
|快速获取 Top N|O (logN + M) 时间复杂度获取 Top M 元素（M≤N）|
|快速更新分数|O (logN) 时间复杂度更新单个元素的分数|
|支持分数相同的元素排序|分数相同时，按元素（member）的字典序排序，可通过分数拼接时间戳 / 用户 ID 实现自定义排序|

#### 2. 核心命令（面试必背，倒序为主，因为排行榜通常从高到低）

表格

|命令|核心作用|核心时间复杂度|核心说明|
|---|---|---|---|
|`ZADD key score member [score member ...]`|添加 / 更新排行榜元素的分数|O (logN)，N 是 ZSET 的元素总数|重复添加 member 会更新 score，而非新增元素|
|`ZREVRANGE key start stop [WITHSCORES]`|获取倒序（从高到低）的 Top N 元素|O (logN + M)，M 是获取的元素数量|`start=0, stop=9`获取 Top10，`WITHSCORES`同时返回分数|
|`ZREVRANK key member`|获取单个元素的倒序排名（从 0 开始，Top1 是 0）|O(logN)|元素不存在返回`nil`|
|`ZSCORE key member`|获取单个元素的分数|O(1)|元素不存在返回`nil`|
|`ZINCRBY key increment member`|原子性增加单个元素的分数|O(logN)|适合实时更新的排行榜（如点赞数、阅读数、积分）|
|`ZCOUNT key min max`|获取分数在 [min, max] 范围内的元素数量|O(logN)|适合统计特定分数段的用户数|
|`ZREMRANGEBYRANK key start stop`|删除倒序排名在 [start, stop] 范围内的元素|O(logN + M)|适合清理排行榜的尾部元素（如只保留 Top10000）|

#### 3. 典型实现流程（以实时积分排行榜为例）

1. **初始化排行榜**：
    
    bash
    
    运行
    
    ```
    # 添加初始用户积分
    ZADD leaderboard:integral 1000 user:1001
    ZADD leaderboard:integral 2000 user:1002
    ZADD leaderboard:integral 1500 user:1003
    ```
    
2. **实时更新用户积分**：
    
    bash
    
    运行
    
    ```
    # 用户1001点赞，积分+10
    ZINCRBY leaderboard:integral 10 user:1001
    ```
    
3. **获取用户的排名和积分**：
    
    bash
    
    运行
    
    ```
    # 获取用户1001的倒序排名（Top1是0）
    ZREVRANK leaderboard:integral user:1001
    # 获取用户1001的积分
    ZSCORE leaderboard:integral user:1001
    ```
    
4. **获取 Top10 排行榜**：
    
    bash
    
    运行
    
    ```
    # 获取Top10，同时返回积分
    ZREVRANGE leaderboard:integral 0 9 WITHSCORES
    ```
    
5. **清理排行榜尾部元素**：
    
    bash
    
    运行
    
    ```
    # 只保留Top10000，删除排名10000及以后的元素
    ZREMRANGEBYRANK leaderboard:integral 10000 -1
    ```
    

#### 4. 基础优化

1. **分数相同的自定义排序**：
    
    分数相同时，ZSET 默认按 member 的字典序排序，可通过**分数拼接时间戳 / 用户 ID / 权重**实现自定义排序：
    
    bash
    
    运行
    
    ```
    # 分数拼接时间戳（倒序时间戳，保证最新的同分数用户排名更高）
    # 时间戳用倒序：2^63-1 - 当前时间戳（毫秒）
    ZADD leaderboard:integral 2000000000000000000 user:1001
    ZADD leaderboard:integral 2000000000000000000 user:1002
    ```
    
2. **排行榜分片**：
    
    当排行榜的元素总数超过 100 万时，可通过**时间分片**（如每日 / 每周 / 每月排行榜）或**业务分片**（如按地区 / 按等级）拆分 ZSET，避免单 ZSET 过大导致的性能下降：
    
    bash
    
    运行
    
    ```
    # 每日排行榜
    ZADD leaderboard:integral:20260404 1000 user:1001
    # 每周排行榜
    ZADD leaderboard:integral:2026w14 1000 user:1001
    ```
    
3. **排行榜缓存**：
    
    当 Top N 的访问量极高时，可将 Top N 的结果缓存到 Redis String 类型，设置较短的过期时间（如 1 秒），避免频繁访问 ZSET：
    
    bash
    
    运行
    
    ```
    # 缓存Top10，过期时间1秒
    SET leaderboard:integral:top10 "[{\"user\":\"user:1002\",\"score\":2000},...]" EX 1
    ```

### 二、进阶生产方案：ZSET+Lua 脚本 + Redisson（高并发场景首选）

#### 1. 解决的问题

简单 ZSET 方案在高并发场景下存在以下问题：

1. **并发更新分数的原子性问题**：虽然`ZINCRBY`是原子的，但如果需要**先获取分数，再根据业务逻辑更新分数**（如积分翻倍活动，先判断用户是否符合条件，再翻倍），就需要 Lua 脚本保证原子性；
2. **排行榜的实时性与性能平衡问题**：高并发场景下，频繁更新 ZSET 会导致性能下降，可通过**批量更新**+**Redisson 分布式锁**+**Lua 脚本**平衡；
3. **排行榜的持久化问题**：虽然 ZSET 是内存结构，但可通过**Redis 持久化**（RDB/AOF）+**数据库兜底**保证数据不丢失。

#### 2. 核心实现：先判断再更新的原子性 Lua 脚本

lua

```
-- Lua脚本：先判断用户是否符合积分翻倍条件，再原子性更新分数
-- KEYS[1]：排行榜ZSET的key
-- ARGV[1]：用户ID
-- ARGV[2]：当前时间戳（毫秒）
-- ARGV[3]：活动开始时间戳（毫秒）
-- ARGV[4]：活动结束时间戳（毫秒）
-- ARGV[5]：积分翻倍倍数
local current_score = redis.call("ZSCORE", KEYS[1], ARGV[1])
if not current_score then
    return 0 -- 用户不在排行榜
end
if tonumber(ARGV[2]) < tonumber(ARGV[3]) or tonumber(ARGV[2]) > tonumber(ARGV[4]) then
    return 0 -- 不在活动时间范围内
end
local new_score = tonumber(current_score) * tonumber(ARGV[5])
redis.call("ZADD", KEYS[1], new_score, ARGV[1])
return 1 -- 更新成功
```

调用方式：

bash

运行

```
EVAL "lua脚本内容" 1 leaderboard:integral user:1001 1743772800000 1743686400000 1743859200000 2
```

#### 3. Redisson 的 RScoredSortedSet（生产环境首选 Java 客户端）

Redisson 是 Redis 官方推荐的 Java Redis 客户端，它封装了 ZSET 的所有操作，提供了更高级的 API，同时保证了原子性和性能：

java

运行

```
// 获取Redisson的RScoredSortedSet
RScoredSortedSet<String> leaderboard = redisson.getScoredSortedSet("leaderboard:integral");
// 原子性添加/更新用户积分
leaderboard.add(1000, "user:1001");
// 原子性增加用户积分
leaderboard.addScore("user:1001", 10);
// 获取用户的倒序排名（Top1是0）
Integer rank = leaderboard.revRank("user:1001");
// 获取用户的积分
Double score = leaderboard.getScore("user:1001");
// 获取Top10，同时返回积分
Collection<ScoredEntry<String>> top10 = leaderboard.entryRangeReversed(0, 9);
```

### 三、极端场景优化方案

#### 1. 超大规模排行榜（元素总数超过 1000 万）

当排行榜的元素总数超过 1000 万时，单 ZSET 的性能会下降，可通过**Redis Cluster 的哈希标签**拆分 ZSET，将不同的元素分散到不同的哈希槽：

bash

运行

```
# 哈希标签：{leaderboard:integral:0}、{leaderboard:integral:1}、...、{leaderboard:integral:9}
# 按用户ID的最后一位哈希到不同的ZSET
ZADD {leaderboard:integral:1} 1000 user:1001
ZADD {leaderboard:integral:2} 2000 user:1002
```

但需要注意：哈希标签拆分后，无法直接获取全局 Top N，需要在应用层聚合所有分片的 Top N，再排序得到全局 Top N。

#### 2. 实时性要求极高的排行榜（如秒杀活动的实时排名）

当排行榜的实时性要求极高（如毫秒级更新）时，可通过**Redis Stream**批量收集更新请求，再通过**Lua 脚本**批量更新 ZSET，减少 ZSET 的访问次数：

bash

运行

```
# 批量收集更新请求到Stream
XADD leaderboard:integral:stream * user:1001 10
XADD leaderboard:integral:stream * user:1002 20
# 消费Stream，批量更新ZSET
XREAD COUNT 100 BLOCK 1000 STREAMS leaderboard:integral:stream 0
```

## 第三部分：总结与选择路径

### 一、清晰的选择路径

1. **简单场景，元素总数≤100 万**：
    
    简单 ZSET 方案，加基础优化（分数自定义排序、时间 / 业务分片、Top N 缓存）；
2. **高并发场景，需要先判断再更新**：
    
    ZSET+Lua 脚本 + Redisson 方案，保证原子性和性能；
3. **超大规模场景，元素总数 > 1000 万**：
    
    ZSET+Redis Cluster 哈希标签方案，应用层聚合全局 Top N；
4. **实时性要求极高的场景**：
    
    ZSET+Redis Stream 批量更新方案，减少 ZSET 的访问次数。

### --> 线上一台 CentOS 服务器根分区磁盘使用率突然涨到 100%，请写出你的完整排查和处理流程。

>面试官您好，我会按照**从大到小、从常见到罕见**的顺序排查，具体流程如下：

### 第一步：快速定位大目录（最常用，90% 的问题在这里）

1. 先执行 `df -h` 确认根分区确实满了，同时看一下其他分区的使用情况
2. 然后用 `du -sh /* | sort -hr` 从根目录开始，找出占用空间最大的一级目录
3. 逐层深入，比如发现`/var`占了 80G，就继续`du -sh /var/* | sort -hr`，直到找到具体的大文件或大目录

### 第二步：处理常见的大文件

1. **日志文件**：最常见的是`/var/log`下的日志，尤其是没有做日志切割的服务日志，可以先清空大日志（注意：不要直接 rm 正在写入的日志，会导致空间不释放，应该用`> 文件名`清空）
2. **临时文件**：`/tmp`和`/var/tmp`下的临时文件，可以删除 7 天前的文件
3. **旧的镜像和备份**：Docker 镜像、数据库备份文件、应用打包文件，删除不需要的旧版本

### 第三步：排查已删除但未释放的文件（高频坑）

如果删了大文件空间还是没释放，大概率是文件被进程占用着：

1. 用 `lsof | grep deleted` 找出所有已删除但未释放的文件
2. 找到对应的进程 PID，重启进程释放空间（如果是核心进程不能重启，可以用`> /proc/PID/fd/文件描述符`清空）

### 第四步：紧急扩容（如果以上都处理完还是不够）

1. 临时扩容：添加新的磁盘，挂载到占用大的目录（比如`/var`）
2. 永久扩容：如果是 LVM 分区，可以直接扩展根分区的大小

> 补充：我一般会在服务器上配置日志自动切割和定时清理脚本，避免磁盘突然满的情况。


### _你写的一个 Shell 自动化备份脚本，运行时偶尔会出现 "文件不存在" 的错误，你会如何排查和优化这个脚本，让它更健壮？

> 面试官您好，这种偶发错误比必现错误难排查，我会先加日志定位问题，再针对性优化脚本。

### 第一步：排查问题原因

1. 首先给脚本加**详细的执行日志**，记录每一步的时间、执行的命令、返回值和变量值，尤其是文件路径相关的变量
2. 常见的偶发原因有：
    
    - 并发问题：多个脚本同时操作同一个文件，一个删了另一个还在读
    - 路径问题：脚本中用了相对路径，不同用户或不同目录执行时路径不对
    - 权限问题：脚本执行用户偶尔没有文件的读取权限
    - 文件被其他进程删除或移动：比如定时清理脚本刚好在备份脚本执行时删除了文件

### 第二步：脚本健壮性优化

1. **所有文件操作前先判断存在性**：`if [ -f "$FILE" ]; then ... else ... fi`
2. **使用绝对路径**：脚本中所有文件和目录都用绝对路径，避免相对路径的问题
3. **加锁防止并发**：用 flock 给脚本加文件锁，同一时间只能有一个实例运行
4. **异常处理**：每个关键命令都加返回值判断，出错时记录日志并退出
5. **原子操作**：文件操作尽量用原子操作，比如先写临时文件，再重命名到目标文件

> 补充：我之前遇到过一个类似的问题，就是备份脚本和日志清理脚本同时执行，清理脚本刚好把要备份的日志删了，后来我给两个脚本加了时间差，并且加了文件锁，就再也没出现过这个问题。


### _线上 Redis 服务器内存使用率达到 99%，触发了内存淘汰策略，导致大量缓存失效。你会按照什么顺序排查原因？有哪些临时和永久的解决方案？

> 面试官您好，我会先紧急处理避免服务崩溃，再排查根本原因，最后给出永久解决方案。

### 第一步：排查原因（按优先级）

1. 先执行 `redis-cli info memory` 查看内存使用详情，重点看`used_memory`（已用内存）、`used_memory_peak`（峰值内存）、`maxmemory`（最大内存限制）
2. 再看 `redis-cli dbsize` 查看 key 的总数，判断是 key 太多还是单个 value 太大
3. 用 `redis-cli --bigkeys` 扫描大 key，找出占用内存最大的 key
4. 查看内存淘汰策略：`config get maxmemory-policy`，确认是否是合理的淘汰策略

### 第二步：临时解决方案（紧急处理）

1. 手动清理过期 key：`redis-cli expireat key 1` 强制过期不需要的 key
2. 删除无用的大 key：注意不要直接 del 大 key，会阻塞 Redis，应该用`unlink`异步删除
3. 临时调大 maxmemory：如果服务器还有剩余内存，可以临时调大 Redis 的最大内存限制
4. 开启 RDB 持久化，把冷数据落地，释放内存

### 第三步：永久解决方案

1. 优化缓存策略：给所有 key 设置合理的过期时间，避免永不过期的 key
2. 拆分大 key：把大的 Hash、List、Set 拆分成多个小 key
3. 升级 Redis 集群：采用主从集群 + 分片，把数据分散到多个节点
4. 优化业务代码：避免把不必要的数据存入 Redis，控制 value 的大小


### _Nginx 出现`504 Gateway Timeout`错误，和`502 Bad Gateway`有什么本质区别？请写出 504 的完整排查流程。

> 面试官您好，这两个错误的本质区别是：
> 
> - **502 Bad Gateway**：Nginx 把请求转发给后端后，后端直接拒绝了连接或者没有返回任何响应
> - **504 Gateway Timeout**：Nginx 把请求转发给后端后，后端在规定的超时时间内没有返回结果，Nginx 主动断开了连接

### 504 错误的完整排查流程

1. **第一步：检查 Nginx 的超时配置**
    
    - 查看 Nginx 配置文件中的`proxy_connect_timeout`（连接超时）、`proxy_read_timeout`（读取超时）、`proxy_send_timeout`（发送超时），默认都是 60 秒
    - 如果后端处理请求确实比较慢，可以适当调大这些超时参数
    
2. **第二步：检查后端服务的性能**
    
    - 登录后端服务器，用`top`、`free`查看 CPU、内存使用率，看是否有资源耗尽的情况
    - 用`netstat -an | grep ESTABLISHED | wc -l`查看后端服务的连接数，是否超过了最大连接数限制
    - 查看后端服务的日志，看是否有慢请求或者报错
    
3. **第三步：检查网络情况**
    
    - 在 Nginx 服务器上用`ping`和`mtr`测试到后端服务器的网络延迟和丢包
    - 用`telnet 后端IP 后端端口`测试端口连通性，看是否有网络波动
    
4. **第四步：检查后端服务的业务逻辑**
    
    - 直接在后端服务器本地测试接口：`curl http://127.0.0.1:端口/接口`，看响应时间是否过长
    - 如果是数据库查询慢导致的接口响应慢，需要优化数据库的慢查询

### _你在简历中提到熟悉 Ansible，如何使用 Ansible 批量修改 100 台服务器的 root 密码？如何确保操作的安全性？

> 面试官您好，我会用 Ansible 的 user 模块来批量修改 root 密码，同时做好安全措施，避免操作失误。

### 具体操作步骤

1. **生成加密的密码哈希**（Ansible 不支持明文密码，必须用哈希）
    
    ```bash
    # 安装passlib库
    pip install passlib
    # 生成SHA512加密的密码哈希
    python -c "from passlib.hash import sha512_crypt; print(sha512_crypt.hash('你的新密码'))"
    ```
    
2. **编写 Ansible Playbook**
    
    ```yaml
    - name: 批量修改root密码
      hosts: all
      become: yes
      tasks:
        - name: 修改root用户密码
          user:
            name: root
            password: "{{ 上面生成的密码哈希 }}"
            update_password: always
    ```
    
3. **执行 Playbook**
    
    ```bash
    # 先在1台测试服务器上执行，验证没问题再全量执行
    ansible-playbook -i hosts change_root_password.yml --limit test-server
    # 全量执行
    ansible-playbook -i hosts change_root_password.yml
    ```
    

### 安全性保障措施（重点，面试官最关心）

1. **先测试后全量**：永远先在 1-2 台测试服务器上执行，验证密码修改成功，能正常登录后再全量执行
2. **分批执行**：100 台服务器分 10 批执行，每批 10 台，避免同时出问题
3. **备份旧密码**：修改前备份所有服务器的旧密码，万一出问题可以回滚
4. **记录操作日志**：执行时加`-v`参数，记录详细的执行日志，方便排查问题
5. **权限控制**：Playbook 和密码哈希文件要严格控制权限，只有 root 用户能访问
6. **验证结果**：执行完成后，用 Ansible 批量验证密码是否修改成功：
    
    ```bash
    ansible all -m ping -u root -k
    ```

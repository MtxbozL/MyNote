### 核心考察点

候选人对 MySQL 高可用架构的理解，生产环境主从问题的排查与解决能力，是 DBA / 运维工程师的必考题，后端开发高频题。

### 专家级满分回答模板

1. **核心定位**：MySQL 主从复制是实现数据库高可用、读写分离、负载均衡、数据容灾的核心基础，基于 binlog 日志实现，将主库的修改操作同步到从库，保证主从数据一致性。
2. **主从复制的核心原理**
    
    MySQL 主从复制是**基于 binlog 的逻辑复制**，核心是 3 个线程的协作：主库的`Binlog Dump线程`、从库的`IO线程`、从库的`SQL线程`。
3. **主从复制的完整流程**
    
    1. 主库写入数据：所有的修改操作（insert、update、delete、DDL）都会按顺序写入主库的 binlog 日志；
    2. 从库连接主库：从库执行`change master to`配置主库信息后，IO 线程主动连接主库，向主库发送 binlog 同步请求，指定同步的 binlog 文件名和位置；
    3. 主库推送 binlog：主库的 Binlog Dump 线程接收到请求后，会读取 binlog 日志，将新增的 binlog 事件推送给从库的 IO 线程；
    4. 从库写入中继日志：从库的 IO 线程接收到 binlog 事件后，将其写入本地的`relay log（中继日志）`中，同时记录同步到的 binlog 文件名和位置，保存在[master.info](https://master.info)文件中；
    5. 从库重放日志：从库的 SQL 线程实时读取中继日志中的事件，解析并执行对应的 SQL 语句，将修改应用到从库的数据库中，最终实现主从数据一致。
    
4. **主从复制的核心模式**
    
    - 异步复制：MySQL 默认模式，主库提交事务后，无需等待从库接收 binlog，直接返回客户端成功，性能最高，但主库宕机有数据丢失风险。
    - 半同步复制：主库提交事务后，必须等待至少一个从库接收到 binlog 并写入中继日志，再返回客户端成功，大幅降低数据丢失风险，兼顾性能与数据安全，生产环境推荐使用。
    - 全同步复制：主库提交事务后，必须等待所有从库都执行完成事务，再返回客户端成功，数据一致性最高，但性能极差，生产环境极少使用。
    
5. **主从延迟的常见原因与解决方案**
    
    主从延迟指的是从库执行完成的事务，落后于主库提交的事务的时间差，通过`show slave status`中的`Seconds_Behind_Master`字段查看，是生产环境的高频问题。
    
    - **常见原因 1：从库硬件性能差于主库**
        
        原因：主库使用 SSD、高配置 CPU / 内存，从库使用低配置服务器，IO、CPU 性能不足，无法及时重放中继日志。
        
        解决方案：从库服务器配置与主库保持一致，甚至更高；使用 SSD 硬盘，提升磁盘 IO 性能；优化从库的`innodb_buffer_pool_size`等配置，提升缓存命中率。
    - **常见原因 2：大事务导致的延迟**
        
        原因：主库执行了超大事务，比如一次性更新百万行数据、大批量导入数据，binlog 写入了大量事件，从库 SQL 线程单线程重放，需要很长时间，导致延迟飙升。
        
        解决方案：拆分大事务，将大批量操作拆分为多个小事务分批执行；大批量数据导入使用`load data infile`，或在业务低峰期执行；开启从库并行复制，提升重放效率。
    - **常见原因 3：SQL 线程单线程瓶颈**
        
        原因：MySQL 5.6 之前，SQL 线程是单线程的，主库高并发写入时，从库单线程重放跟不上，必然出现延迟。
        
        解决方案：升级到 MySQL 5.7 + 版本，开启**基于 GTID 的并行复制**，或基于库的并行复制，让 SQL 线程多线程并行重放中继日志，大幅提升重放效率。
    - **常见原因 4：从库表无主键或无有效索引**
        
        原因：主库的 update/delete 语句，在主库可以通过索引快速执行，但从库的表无主键、无有效索引，重放时需要全表扫描，执行效率极低，导致延迟。
        
        解决方案：所有表必须设置主键，保证 update/delete 语句的 where 条件字段有索引；避免无 where 条件的全表更新 / 删除。
    - **常见原因 5：主库大 DDL 操作**
        
        原因：主库对大表执行 DDL 操作（比如加字段、加索引），执行时间长，binlog 写入后，从库需要串行执行同样的 DDL，期间无法执行其他事务，导致延迟飙升。
        
        解决方案：使用`pt-online-schema-change`、`gh-ost`等工具执行无锁 DDL，避免锁表和长事务；在业务低峰期执行 DDL；拆分大表，降低 DDL 执行时间。
    - **常见原因 6：从库承担了大量读请求**
        
        原因：从库承担了大量的复杂查询、报表统计、慢 SQL，CPU、IO、内存被占满，SQL 线程无法获得足够的资源执行重放，导致延迟。
        
        解决方案：搭建多从库，拆分读压力，将报表统计、离线分析请求分流到专属的从库；优化从库的慢 SQL，避免复杂查询占用过多资源；限制从库的最大连接数，避免连接数打满。
    

### 加分亮点（专家级区分度）

1. 补充 GTID 复制的优势：MySQL 5.6 + 支持 GTID（全局事务 ID）复制，每个事务对应一个唯一的 GTID，无需指定 binlog 文件名和位置，主从切换、故障恢复更简单，避免传统位点复制的同步错位问题，生产环境推荐使用。
2. 补充主从一致性校验：使用`pt-table-checksum`工具定期校验主从数据一致性，使用`pt-table-sync`工具修复数据不一致的问题，避免主从数据长期不一致。
3. 补充延迟优化最佳实践：核心是提升从库的重放效率，优先开启并行复制，保证所有表有主键，拆分大事务，这三个手段能解决 90% 以上的主从延迟问题。

### 踩坑避雷区

- 错误表述：Seconds_Behind_Master 为 0，代表主从数据完全一致。（该值仅代表 SQL 线程重放的延迟，若 IO 线程同步中断，该值仍可能为 0，实际主从已经停止同步）
- 错误表述：主从复制可以完全替代备份。（主从复制只能同步数据，若主库执行误删操作，从库也会同步执行，无法恢复数据，必须配合全量备份 + binlog 增量备份）

### -> 编写一个 Shell 脚本，实现每天凌晨 2 点自动清理`/var/log`目录下 7 天前的`.log`文件，同时记录清理的文件数量和总大小到`/var/log/clean_log.log`。

> 面试官您好，这个需求我分两步实现：先写健壮的独立脚本，再用 crontab 配置定时任务。

#### 核心实现思路

1. 用变量统一定义清理目录、保留天数、日志路径，便于后续维护
2. 先检查目标目录是否存在，避免脚本异常
3. 先统计待清理的文件数量和总大小，再执行删除
4. 用`-delete`替代`rm -rf`，更安全高效
5. 所有操作记录到日志文件，便于排错

#### 核心脚本代码（只记关键行）

```bash
#!/bin/bash
LOG_DIR="/var/log"
RETENTION_DAYS=7
LOG_FILE="/var/log/clean_log.log"

echo "===== $(date '+%Y-%m-%d %H:%M:%S') 开始 =====" >> $LOG_FILE
[ ! -d "$LOG_DIR" ] && echo "目录不存在" >> $LOG_FILE && exit 1

FILE_COUNT=$(find "$LOG_DIR" -type f -name "*.log" -mtime +7 | wc -l)
FILE_SIZE=$(find "$LOG_DIR" -type f -name "*.log" -mtime +7 | xargs du -ch | tail -1 | awk '{print $1}')

[ $FILE_COUNT -gt 0 ] && find "$LOG_DIR" -type f -name "*.log" -mtime +7 -delete && echo "清理$FILE_COUNT个文件，大小$FILE_SIZE" >> $LOG_FILE
echo "===== $(date '+%Y-%m-%d %H:%M:%S') 结束 =====" >> $LOG_FILE
```

#### 定时任务配置

1. 加执行权限：`chmod +x /usr/local/bin/clean_log.sh`
2. 编辑 root 的 crontab：`sudo crontab -e`
3. 添加任务：`0 2 * * * /usr/local/bin/clean_log.sh`

#### 一、4 个核心分支的使用场景

1. **`master`**：**生产环境分支**，只能合并不能直接提交，存放随时可上线的稳定代码，每次合并都要打版本标签
2. **`develop`**：**开发环境主分支**，集成所有开发完成的功能，是下一个版本的代码基础
3. **`feature`**：**功能开发分支**，从`develop`拉取，每个新功能单独建一个，开发完成后合并回`develop`
4. **`hotfix`**：**紧急修复分支**，从`master`拉取，修复线上紧急 bug，修复完成后同时合并回`master`和`develop`

#### 二、Git 合并冲突的具体解决过程

1. 先拉取远程最新代码：`git pull origin develop`，此时 Git 会提示冲突文件
2. 打开冲突文件，找到`<<<<<<< HEAD`（本地代码）、`=======`（分隔线）、`>>>>>>> 远程分支名`（远程代码）标记的冲突区域
3. 对比两边代码，保留正确的部分，删除所有冲突标记
4. 保存文件后，执行`git add 冲突文件`标记冲突已解决
5. 最后执行`git commit -m "解决合并冲突"`完成提交

#### 补充得分点

> 我一般会在合并前先拉取远程最新代码，尽量减少冲突；如果冲突比较复杂，会用 VSCode 的 Git 插件可视化对比，效率更高。

### -> MySQL 主从同步的完整原理是什么？主从同步延迟超过 10 分钟，你会按照什么顺序排查原因？至少说出 3 种常见解决方案。

>理解主从同步的 **三个核心线程**

#### 一、主从同步完整原理（核心：三个线程）

1. **Master 节点**：有一个`dump 线程`，当 slave 连接后，负责读取 binlog 日志，推送给 slave 的 IO 线程
2. **Slave 节点**：有两个线程
    
    - `IO线程`：接收 master 推送的 binlog，写入本地的`relay log（中继日志）`
    - `SQL线程`：读取中继日志，重放里面的 SQL 命令，实现数据同步

#### 二、主从延迟 10 分钟的排查顺序（从易到难）

![[Pasted image 20260417163902.png]]

1. 先看 **从节点 SQL 线程状态** ：`show slave status\G`，看`Seconds_Behind_Master`和`SQL_Running`是否正常
2. 再看**主节点负载**：用 top 看 CPU、IO、内存，是否有大事务、慢查询导致 binlog 生成过慢
3. 再看**从节点负载**：是否有大量读请求占用资源，导致 SQL 线程执行慢
4. 最后看**网络情况**：用 mtr 测试主从之间的网络延迟和丢包

#### 三、3 种常见解决方案

1. 优化主节点：避免大事务，拆分长 SQL 为小 SQL；开启 binlog 行模式
2. 优化从节点：关闭从节点的慢查询日志；增加从节点硬件配置
3. 架构优化：采用一主多从分担读压力；使用 **半同步复制** 减少数据丢失风险

### -> Redis 的 RDB 和 AOF 持久化有什么本质区别？如果线上 Redis 服务器突然宕机，如何保证数据尽可能不丢失？

>**数据完整性、恢复速度、性能影响**三个核心维度
#### 一、RDB 与 AOF 的核心区别

| 维度    | RDB                 | AOF                  |
| ----- | ------------------- | -------------------- |
| 存储内容  | 某一时刻的 **全量数据快照**    | 所有 **写操作命令** 的日志     |
| 数据完整性 | 差，会丢失最后一次快照后的数据     | 好，可配置每秒刷盘，最多丢失 1 秒数据 |
| 恢复速度  | 快，适合全量恢复            | 慢，需要重放所有命令           |
| 性能影响  | 小，fork 子进程执行，不阻塞主进程 | 大，每次写操作都要刷盘          |

#### 二、线上 Redis 宕机保证数据不丢失的方案

![[Pasted image 20260417162902.png]]

1. **开启混合持久化**（Redis 4.0 + 默认推荐）：结合 RDB 和 AOF 的优点，用 RDB 做全量备份，AOF 记录增量命令，恢复速度快且数据丢失少
2. **配置持久化参数**：AOF 设置`appendfsync everysec`（每秒刷盘，性能和安全平衡）；RDB 设置合理的自动快照频率
3. **部署主从集群**：主节点宕机后，从节点可以快速切换，保证服务不中断
4. **定期手动备份**：每天凌晨备份 RDB 文件到异地存储，防止极端情况


### -> 什么是缓存击穿、缓存雪崩、缓存穿透？请分别说明它们的触发场景和你在实际工作中使用过的解决方案。

> 面试官您好，这三个问题的触发场景和我实际用过的解决方案如下：

7. **缓存穿透**
    
    - 触发场景：用户查询 **数据库和缓存中都不存在** 的 key，导致所有请求直接打数据库
    - 解决方案：我用过 **布隆过滤器** ，把所有存在的 key 提前存入布隆过滤器，请求先过过滤器，不存在的直接拦截；同时对数据库返回的空结果也缓存 5 分钟
    
8. **缓存击穿**
    
    - 触发场景：**某个热点 key 突然过期**，大量并发请求同时打数据库
    - 解决方案：我用过 **互斥锁**，当缓存失效时，只有第一个请求能获取锁去查数据库并更新缓存，其他请求等待缓存更新完成；另外对热点 key 设置 **永不过期** 
    
9. **缓存雪崩**
    
    - 触发场景：① 大量 key **同一时间集体过期** ；
	           ② Redis 服务**整体宕机**
    - 解决方案：① 给 key 的过期时间加 **随机值**，避免同时过期；
               ② 部署 Redis 主从集群 + 哨兵，实现自动故障转移；
               ③ 服务端加 **熔断和降级**，Redis 宕机时暂时返回默认值

### -> 如何定位 MySQL 的慢查询？请说出至少 3 种慢查询优化方法，并说明各自的适用场景。

![[Pasted image 20260417174552.png]]

> 面试官您好，我定位和优化 MySQL 慢查询的方法如下：

#### 一、定位慢查询的 3 种方法

1. **开启慢查询日志**：设置`long_query_time=1`，让 MySQL 自动记录执行超过 1 秒的 SQL，这是生产环境最常用的方法
2. **工具分析**：用`pt-query-digest`批量分析慢查询日志，按总耗时和执行次数排序，快速找出 Top 慢 SQL
3. **实时排查**：紧急故障时用`show full processlist`查看当前正在执行的 SQL，找到耗时最长的进程

#### 二、3 种核心优化方法及适用场景

1. **加索引**：给`WHERE`、`JOIN`、`ORDER BY`后面的字段加合适的索引，这是成本最低、效果最好的优化，适用于绝大多数没有索引的慢查询
2. **优化 SQL**：避免`SELECT *`、避免字段上用函数、拆分长 SQL，适用于 SQL 写法不规范导致的慢查询
3. **架构优化**：当 SQL 和索引优化到极致后，采用读写分离、Redis 缓存、分库分表等架构方案，适用于高并发、大数据量的场景

#### 补充得分点（一句话带过，体现实操经验）

> 我一般会在生产环境开启慢查询日志，每天凌晨用定时任务自动分析并生成报告，定期和开发一起优化 Top 慢 SQL；同时会避免给更新频繁的字段加索引，因为索引会降低写性能。

### Nginx 如何实现反向代理和负载均衡？常用的负载均衡策略有哪些？什么场景下适合使用 ip_hash 策略？
>1. 不能只说明使用了哪个模块，需要说清楚反向代理和负载均衡是如何配合工作的。
>2. 需要说明哪个是默认策略，同时讲清楚每个策略的核心特点。

> 面试官您好，Nginx 反向代理和负载均衡的实现、常用策略及 ip_hash 的适用场景如下：

#### 一、反向代理和负载均衡的实现

Nginx 通过两个核心模块配合实现：

1. 用`upstream`模块定义后端服务器池，把所有提供相同服务的后端节点配置在一起
2. 用`server`模块的`location`匹配请求，通过`proxy_pass`指令把请求转发到 upstream 定义的服务器池，从而实现反向代理和负载均衡

#### 二、常用的 5 种负载均衡策略

![[Pasted image 20260417170226.png]]

1. **轮询（默认）**：请求按顺序依次分配到不同的后端服务器
2. **加权轮询**：权重越高的服务器，分配到的请求越多，适用于后端服务器性能不均的场景
3. **ip_hash**：根据客户端 IP 的哈希结果分配服务器，同一个客户端的请求始终分配到同一个后端服务器 **(会话保持)**
4. **url_hash**：根据请求 URL 的哈希结果分配服务器，适用于静态资源缓存场景
5. **least_conn**：请求分配到当前连接数最少的服务器，适用于长连接较多的场景

#### 三、ip_hash 的适用场景

**核心场景：需要会话保持的业务**，比如用户登录状态、购物车等。因为同一个用户的请求始终打到同一台后端服务器，就不需要在多台服务器之间同步会话数据了。

### 配置 Nginx HTTPS 需要哪些步骤？如何强制将 HTTP 请求跳转到 HTTPS？

> 1. **步骤完善**： 除了 listen ssl 还需要提及 提证书申请、证书配置、SSL 协议优化等核心步骤。
> 2. **没有实操细节**：强制跳转不能只说了指令，要给生产环境标准的配置示例。
> 3. **遗漏关键注意事项**：比如 443 端口开放、证书权限等常见坑

> 面试官您好，配置 Nginx HTTPS 和强制跳转的步骤如下：

#### 一、HTTPS 完整配置步骤

1. **申请 SSL 证书**：可以从阿里云、腾讯云等平台申请免费的 Let's Encrypt 证书，得到`证书文件(.crt)`和`私钥文件(.key)`
2. **上传证书到服务器**：一般放在`/etc/nginx/ssl/`目录下，设置证书权限为 600（避免私钥泄露）
3. **修改 Nginx 配置文件**，在 server 块中添加以下配置：
    
    ```nginx
    server {
        listen 443 ssl;
        server_name yourdomain.com;
        
        # 配置证书路径
        ssl_certificate /etc/nginx/ssl/yourdomain.crt;
        ssl_certificate_key /etc/nginx/ssl/yourdomain.key;
        
        # 优化SSL协议和加密套件（提升安全性和性能）
        ssl_protocols TLSv1.2 TLSv1.3;
        ssl_ciphers ECDHE-RSA-AES128-GCM-SHA256:HIGH:!aNULL:!MD5:!RC4:!DHE;
        
        # 其他常规配置
        location / {
            proxy_pass http://backend;
        }
    }
    ```
    
4. 检查配置并重启 Nginx：`nginx -t && systemctl restart nginx`
5. 开放服务器 443 端口防火墙

#### 二、HTTP 强制跳转 HTTPS（生产环境标准写法）

单独写一个 80 端口的 server 块，用`return 301`永久重定向（比 rewrite 性能更好）：

```nginx
server {
    listen 80;
    server_name yourdomain.com;
    return 301 https://$server_name$request_uri;
}
```

### Nginx 出现`502 Bad Gateway`错误，可能有哪些原因？请写出你的完整排查流程。

> 排查思路应该为 **从易到难、从常见到罕见**。

![[Pasted image 20260417171259.png]]

> 面试官您好，Nginx 出现 502 Bad Gateway，本质是 Nginx 转发请求后，没有收到后端服务的正常响应。我会按照以下顺序排查：

1. **第一步：先确认后端服务是否正常运行**（90% 的 502 都是这个原因）
    
    - 登录后端服务器，用`ps -ef | grep 服务名`查看进程是否存在
    - 本地测试后端服务：`curl http://127.0.0.1:端口号`，看是否能正常返回结果
    
2. **第二步：检查 Nginx 与后端服务器的网络连通性**
    
    - 在 Nginx 服务器上，用`telnet 后端IP 后端端口`或`nc -zv 后端IP 后端端口`测试端口是否能通
    - 检查两端服务器的防火墙和安全组，是否开放了后端服务的端口
    
3. **第三步：检查 Nginx 配置是否正确**
    
    - 用`nginx -t`检查配置文件语法是否正确
    - 确认`proxy_pass`指向的后端地址和端口是否正确
    - 检查是否配置了错误的负载均衡策略或健康检查
    
4. **第四步：检查后端服务资源是否不足**
    
    - 用`top`、`free`、`df`查看后端服务器的 CPU、内存、磁盘使用率
    - 如果资源耗尽，后端服务会无法响应新的请求，导致 502
    
5. **第五步：检查 Nginx 的超时设置**
    
    - 如果后端服务处理请求过慢，超过了 Nginx 的超时时间，也会返回 502
    - 可以适当调大`proxy_connect_timeout`、`proxy_read_timeout`等超时参数

### 你是如何实现 Nginx 动静分离的？相比不做动静分离，能带来哪些具体的性能提升？

> 面试官您好，我是通过 Nginx 的 location 匹配规则实现动静分离的，具体如下：

#### 一、动静分离的具体实现

静态资源（图片、CSS、JS、HTML 等）直接由 Nginx 处理，动态请求（.php、.jsp、接口等）转发给后端应用服务器。配置示例：

```nginx
server {
    listen 80;
    server_name yourdomain.com;
    
    # 静态资源由Nginx直接处理
    location ~* \.(jpg|jpeg|png|gif|css|js|html)$ {
        root /usr/share/nginx/static;  # 静态资源存放目录
        expires 7d;  # 设置7天缓存，提升二次访问速度
    }
    
    # 动态请求转发给后端Tomcat
    location / {
        proxy_pass http://tomcat_backend;
    }
}
```

#### 二、动静分离带来的具体性能提升

![[Pasted image 20260417171711.png|632]]

1. **大幅降低后端应用服务器的压力**：静态资源占网站请求的 80% 以上，全部由 Nginx 处理，后端只需要处理 20% 的动态请求
2. **提高静态资源的响应速度**：Nginx 处理静态资源的性能远高于 Tomcat、Node.js 等应用服务器
3. **利用 Nginx 的缓存能力**：静态资源可以设置长时间缓存，用户二次访问时直接从浏览器缓存或 Nginx 缓存返回，减少网络请求
4. **提升网站整体并发能力**：Nginx 可以轻松支持几万甚至几十万的并发连接，比应用服务器的并发能力强得多

## 模块 3：Docker 与 Kubernetes

### Dockerfile 中`RUN`、`CMD`、`ENTRYPOINT`三个指令的核心区别是什么？如果需要容器启动时执行多个命令，应该如何配置？

> 面试官您好，这三个指令的 **执行时机** 和 **作用** 完全不同：

1. **RUN**：**构建镜像时** 执行，用于安装软件、配置环境，每一个 RUN 都会生成一个新的镜像层

2. **CMD**：**容器启动时** 执行，是默认命令和参数，可以被`docker run`后面的命令完全覆盖

3. **ENTRYPOINT**：**容器启动时** 执行，是主程序入口，`docker run`后面的参数会作为它的参数，不会被覆盖

- 我一般会组合使用：ENTRYPOINT 指定主程序，CMD 指定默认参数，这样既灵活又规范

### Kubernetes 中`Pod`、`Deployment`、`Service`、`Ingress`的作用分别是什么？它们之间是如何协同工作的？

>面试官您好，这四个核心组件的作用和协同工作流程如下：

1. **Pod**：K8s 最小部署单元，封装容器，提供网络和存储
2. **Deployment**：管理 Pod 的生命周期，定义 Pod 的副本数和更新策略，保证 Pod 的高可用
3. **Service**：为一组相同的 Pod 提供固定的集群内访问地址，实现负载均衡
4. **Ingress**：集群对外的 HTTP/HTTPS 入口，根据域名和路径将请求转发到不同的 Service

协同流程：用户请求先到 Ingress，Ingress 转发到对应的 Service，Service 再把请求负载均衡到后端的多个 Pod 上，Deployment 负责保证这些 Pod 始终有指定数量在运行。

### Docker 多阶段构建的核心优势是什么？请结合你简历中提到的前端项目，写出一个简单的多阶段构建 Dockerfile 示例。

> 面试官您好，多阶段构建的核心优势是 **大幅减小最终镜像的体积**，同时避免把构建工具和源码带到生产环境，更安全。

我在之前的 CI/CD 项目中，给前端项目写的多阶段构建 Dockerfile 示例如下：

```dockerfile
# 第一阶段：构建阶段，用node镜像打包前端代码
FROM node:16-alpine AS builder
WORKDIR /app
COPY package*.json ./
RUN npm install
COPY . .
RUN npm run build  # 生成dist目录，里面是编译后的静态资源

# 第二阶段：运行阶段，只保留nginx和编译后的静态资源
FROM nginx:alpine
COPY --from=builder /app/dist /usr/share/nginx/html
EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
```

> 这样最终的镜像只有几十 MB，而单阶段构建会有几百 MB，不仅传输快，部署也更快。

    如果 Pod 使用了 hostPort，确认该端口在目标节点上没有被占用

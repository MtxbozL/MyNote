### 核心考察点

候选人对 MySQL 存储引擎的底层认知、技术选型能力，区分基础入门级与有实战经验的候选人。

### 专家级满分回答模板

1. **核心定位**：InnoDB 是 MySQL 5.5 之后的默认存储引擎，面向**高并发、事务安全**的 OLTP 场景；MyISAM 是 MySQL 原生的非事务引擎，面向只读为主、低并发的 OLAP / 静态数据场景。
2. **核心区别（6 个核心维度）**
    
    - **事务支持**：InnoDB 完整支持 ACID 事务、4 种隔离级别，具备崩溃恢复能力；MyISAM 完全不支持事务，无原子性保障。
    - **锁机制**：InnoDB 默认支持**行级锁**（基于索引实现），仅锁定操作的行，并发性能极强；MyISAM 仅支持**表级锁**，写入时会锁定全表，读写互斥，高并发写入性能极差。
    - **MVCC 支持**：InnoDB 通过 MVCC（多版本并发控制）实现读写不阻塞，大幅提升高并发场景性能；MyISAM 不支持 MVCC，读写完全互斥。
    - **索引结构**：InnoDB 采用**聚簇索引**，主键索引与数据绑定，二级索引存储主键值；MyISAM 采用**非聚簇索引**，索引与数据文件分离，索引存储数据的物理地址。
    - **数据可靠性**：InnoDB 通过 redo log+undo log 实现**crash-safe（崩溃安全）**，宕机后可自动恢复数据，无数据丢失风险；MyISAM 不支持崩溃安全，宕机后极易出现数据损坏，需手动修复。
    - **附加能力**：InnoDB 支持外键约束、外键级联操作；MyISAM 不支持外键。
    
3. **适用场景**
    
    - InnoDB：互联网核心业务系统、订单 / 支付 / 用户系统、高并发读写场景、需要事务保障的所有业务。
    - MyISAM：静态数据查询（如历史归档数据、只读报表）、极低写入的静态站点、对事务无要求且极致要求读性能的极简场景。
    

### 加分亮点（专家级区分度）

1. 补充性能细节：MyISAM 的`count(*)`不带 where 条件时性能极高，因为其在元数据中存储了全表行数；而 InnoDB 需要全表扫描，需通过`count(1)`或覆盖索引优化。
2. 补充实战限制：MyISAM 仅支持表级锁，哪怕是单行更新也会锁定全表，哪怕是读操作也会被写入阻塞，因此生产环境几乎不再使用。
3. 补充底层差异：InnoDB 的数据文件是`.ibd`，MyISAM 分为`.MYD`数据文件和`.MYI`索引文件。

### 踩坑避雷区

- 错误表述：MyISAM 的查询性能一定比 InnoDB 好。（仅无写入的单表全量 count 场景有优势，带 where 条件、有并发写入的场景，InnoDB 性能远超 MyISAM）
- 错误表述：InnoDB 的行锁是绝对的。（无有效索引时，InnoDB 会升级为表锁）

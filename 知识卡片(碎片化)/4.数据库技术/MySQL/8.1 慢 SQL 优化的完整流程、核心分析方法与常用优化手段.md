### 核心考察点

候选人的 SQL 优化实战能力，生产环境问题排查与解决能力，是开发、DBA、运维工程师的通用必考题。

### 专家级满分回答模板

1. **慢 SQL 优化的完整闭环流程**
    
    慢 SQL 优化不是单纯的加索引，而是一套标准化的闭环流程，分为 4 个核心步骤：**定位慢 SQL → 分析执行计划 → 针对性优化 → 验证优化效果**。
2. **步骤 1：定位慢 SQL（找到优化目标）**
    
    核心手段：
    
    - 开启慢查询日志：配置`slow_query_log=on`，`long_query_time=1`（超过 1 秒的 SQL 记录，可根据业务调整），`log_queries_not_using_indexes=on`（记录未使用索引的 SQL），通过慢查询日志获取所有慢 SQL。
    - 日志分析工具：使用`mysqldumpslow`、`pt-query-digest`工具分析慢查询日志，按执行次数、总耗时、锁等待时间排序，找到 TopN 慢 SQL，优先优化高频、高耗时的 SQL。
    - 实时排查：通过`show processlist`查看当前正在执行的慢 SQL，通过`information_schema.processlist`定位长事务、慢查询会话。
    
3. **步骤 2：分析慢 SQL 的根因（核心工具 explain）**
    
    拿到慢 SQL 后，使用`explain`命令查看 SQL 的执行计划，重点分析 6 个核心字段，定位根因：
    
    - **type**：访问类型，性能从好到差依次为 system > const > eq_ref > ref > range > index > ALL，核心优化目标：至少达到 range 级别，最优是 ref 级别，杜绝 ALL（全表扫描）。
    - **key**：实际命中的索引，若为 NULL，说明未使用索引，是核心优化点。
    - **rows**：MySQL 预估需要扫描的行数，数值越大性能越差，优化目标是大幅降低扫描行数。
    - **Extra**：额外信息，核心优化点：① 避免出现`Using filesort`（文件排序，无法利用索引排序）；② 避免出现`Using temporary`（使用临时表，常见于 group by、order by）；③ 避免出现`Using where`（全表扫描后过滤）；④ 优先出现`Using index`（使用覆盖索引，无需回表，性能最优）。
    - **possible_keys**：可能命中的索引，与 key 对比，可判断索引失效的问题。
    - **ref**：与索引比较的列或常量，可判断索引的匹配效率。
    
4. **步骤 3：针对性优化手段（核心优化方案）**
    
    根因不同，优化方案不同，核心分为 5 大类优化手段，优先级从高到低：
    
    - **索引优化（最高优先级，性价比最高）**
    
    1. 为 where、join on、order by、group by 后的字段创建合适的索引，优先创建联合索引，而非多个单列索引；
    2. 遵循联合索引的最左匹配原则，避免索引失效；
    3. 构建覆盖索引，让查询的所有字段都包含在联合索引中，彻底避免回表查询；
    4. 避免索引失效的场景：索引字段上使用函数、运算、隐式类型转换、模糊查询 % 前缀、使用 or 连接非索引字段、is not null 等。
    
    - **SQL 语句改写优化**
    
    1. 避免 select *，只查询业务需要的字段，减少数据传输和回表概率；
    2. 避免大表 join，控制 join 的表数量不超过 3 张，小表驱动大表，join 字段必须创建索引；
    3. 优化深分页问题：比如`limit 100000,20`，优化为`where id>100000 limit 20`，利用主键索引避免全表扫描；
    4. 避免 in 子查询嵌套过深，避免使用 not in，可改为 left join + is null；
    5. 避免大事务、批量操作，将大批量的 update/delete 拆分为小批量执行，减少锁持有时间和慢查询。
    
    - **表结构优化**
    
    1. 选择合适的数据类型，比如用 tinyint 代替 int，用 varchar 代替 char，用 datetime 代替 timestamp，避免大字段类型；
    2. 表必须设置主键，优先使用自增主键，避免无序 UUID 导致的页分裂；
    3. 避免 null 字段，设置非空约束和默认值，null 值会导致索引失效、统计复杂；
    4. 单表数据量超过千万行，或单表体积超过 10GB，考虑分库分表、冷热数据分离。
    
    - **架构与配置优化**
    
    1. 读写分离，读请求走从库，写请求走主库，分散主库压力；
    2. 热点数据加入 Redis 缓存，减少数据库查询压力；
    3. 优化 MySQL 配置：调整连接池、缓冲池`innodb_buffer_pool_size`（建议设置为物理内存的 50%-70%）、排序缓冲、join 缓冲等；
    4. 优化磁盘 IO：使用 SSD 硬盘，优化 redo log、binlog 的刷盘策略，平衡性能与安全。
    
    - **业务逻辑优化**
    
    1. 避免过度查询，比如循环查询数据库，改为批量查询；
    2. 避免非必要的实时统计，改为离线统计、定时更新；
    3. 拆分复杂业务逻辑，减少单条 SQL 的复杂度。
    
5. **步骤 4：验证优化效果**
    
    优化完成后，重新执行 explain 查看执行计划，确认索引命中、扫描行数降低、无文件排序和临时表；重新执行 SQL，查看执行耗时，确认优化效果；上线后持续监控慢查询日志，确认慢 SQL 不再出现，无新的慢 SQL 产生。

### 加分亮点（专家级区分度）

1. 补充实战踩坑：隐式类型转换是高频索引失效场景，比如 varchar 类型的字段用数字查询`where phone=13800138000`，会导致索引完全失效，必须加引号`where phone='13800138000'`。
2. 补充进阶分析工具：使用`show profile`查看 SQL 执行的每个阶段的耗时，定位是 CPU、IO、锁等待哪个环节耗时高；使用`optimizer_trace`查看优化器的执行计划选择过程，解决索引选错的问题。
3. 补充最佳实践：优先优化高频慢 SQL，一条每秒执行 1000 次的 100ms SQL，优化到 10ms，带来的性能提升，远大于优化一条每天执行 1 次的 10s SQL。

### 踩坑避雷区

- 错误表述：只要给字段加了索引，SQL 就一定会走索引。（MySQL 优化器会根据成本估算，选择全表扫描还是走索引，若索引区分度极低，优化器会选择全表扫描）
- 错误表述：索引越多越好。（索引会大幅降低 insert、update、delete 的性能，因为需要同步维护所有索引的 B + 树，单表索引数量建议不超过 5 个）
